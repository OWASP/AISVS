# C9.9: Architectural Data-Flow Isolation and Origin Enforcement

[Back to C09 Index](C09-Orchestration-and-Agents.md)

## Purpose

Prevent data-flow attacks that exploit agentic tool-calling pipelines by manipulating tool arguments without altering the control flow, through architectural separation of planning from untrusted data processing and origin-aware policy enforcement at the tool-call boundary. These controls complement C2.1 (input-level filtering) and C9.7 (intent verification gates), which do not address attacks where the correct tools are called in the correct sequence but with attacker-controlled arguments derived from untrusted data. Any isolation mechanism that achieves the stated outcome satisfies the requirement.

---

## 2024-2026 Research Context

Indirect prompt injection is no longer only a prompt-filtering problem. The hard case for agent systems is that untrusted content can influence a later tool argument while the overall plan still looks legitimate. OWASP LLM01:2025 calls out external-content prompt injection, and OWASP LLM06:2025 ties the impact to excessive agency: functions, permissions, and autonomy exposed to manipulated model outputs. MITRE ATLAS techniques relevant to this section include AML.T0051.001 (indirect prompt injection), AML.T0053 (AI agent tool invocation), and AML.T0086 (exfiltration via AI agent tool invocation).

The most directly relevant academic work is [AgentDojo](https://arxiv.org/abs/2406.13352), which evaluates agents operating over untrusted data and external tools, and [CaMeL](https://arxiv.org/abs/2503.18813), which defends by extracting control flow from the trusted user request, treating retrieved or tool-returned content as data, and enforcing capabilities on data flows before tool calls. CaMeL is a research artifact rather than a drop-in production control, but it is useful as a reference architecture: do not let untrusted data decide which tools run, and do not let data with the wrong provenance cross into sensitive tool arguments.

Several 2026 papers are pushing this from prompt filtering toward runtime evidence. [AdapTools](https://arxiv.org/abs/2602.20720) shows why static prompt-injection test strings are insufficient: adaptive attacks can choose stealthier tools and vary payloads to evade task-relevance checks. [AttriGuard](https://arxiv.org/abs/2603.10749) frames the key question as causal attribution: would the same tool call still be justified if the untrusted observation were control-attenuated? [IPIGuard](https://arxiv.org/abs/2508.15310), accepted to EMNLP 2025, argues for the same architectural separation that C9.9 evaluates: the agent must commit to a Tool Dependency Graph derived from the trusted user instruction before any external interaction, then execute tools in strict topological order so injected instructions arriving in tool responses cannot create new edges or reorder calls. The authors also describe a fake tool invocation pattern that absorbs hostile follow-on directives in a spoofed call rather than letting them rewrite later arguments — useful as a concrete test pattern for "same plan, unsafe arguments" failures. [ClawGuard](https://arxiv.org/abs/2604.11790), last revised on May 11, 2026, takes a boundary-enforcement angle by deriving task-specific constraints before external tool use and checking later tool calls against that rule set. [LogJack](https://arxiv.org/abs/2604.15368) shows why this matters operationally: cloud logs can become the indirect prompt-injection channel for debugging agents that later execute remediation commands. [Ghost in the Agent / NeuroTaint](https://arxiv.org/abs/2604.23374), released April 25, 2026, argues that classical program-memory taint analysis fundamentally fails on LLM agents because data propagation is governed by probabilistic natural-language reasoning, and proposes taint propagation that captures semantic transformation, causal influence on later decisions, and cross-session persistence through memory. [Causality Laundering](https://arxiv.org/abs/2604.04035), April 5, 2026, identifies a denial-feedback leakage pattern in tool-calling agents where adversaries probe a protected action, learn from the denial outcome, and exfiltrate inferred information through a later seemingly benign tool call, and introduces the Agentic Reference Monitor that mediates each tool invocation against a provenance graph covering tool calls, returned data, field-level provenance, and denied actions. [ARGUS](https://arxiv.org/abs/2605.03378), released May 5, 2026, builds an influence provenance graph that tracks how untrusted context propagates into agent decisions and verifies whether each tool call is justified by trustworthy evidence before execution; on the accompanying AgentLure benchmark of context-dependent tasks and adaptive attacks across four agentic domains, the defense drops attack success rate to 3.8% while preserving 87.5% task utility, which is closer to a deployable trade-off than most prompt-only mitigations. [Sleeper Channels and Provenance Gates](https://arxiv.org/abs/2605.13471), published May 13, 2026, names a class of attacks that C9.9 reviewers should carry forward into always-on autonomous-agent deployments: untrusted input persists across memory, skills, scheduled jobs, and filesystem patches and executes later through a different entry point (e.g., a confused-deputy cron path) without the attacker present. The proposed defense pairs a canonical action-instance digest with one-shot owner attestations and provenance gates around the scheduled-execution hooks, with a stated soundness argument against paraphrase laundering, multi-input grant reuse, and replay. The runtime adapter and 42-test artifact are useful as a regression model for any deployment where the agent owns a memory store, a scheduler, or an autonomous filesystem. [Open Agent Passport](https://arxiv.org/abs/2603.20953), submitted March 21, 2026, adds a synchronous pre-action authorization layer that intercepts each tool call, evaluates it against declarative policy at a median latency of 53 ms, and emits cryptographically signed audit records; adversarial testing reported social-engineering attacks succeeding 74.6% of the time under permissive policies and 0% under the restrictive policy across 879 attempts, giving reviewers a concrete benchmark for "deterministic enforcement before the call" rather than retrospective monitoring. The SoK on [Prompt Injection Attacks on Agentic Coding Assistants](https://arxiv.org/abs/2601.17548), January 24, 2026, reframes the problem for IDE-class agents as text-to-action rather than text-to-output, and reinforces that defenses degrade sharply under adaptive attackers across skills, tools, and protocol ecosystems. These are not mature production standards, but they give auditors concrete test patterns for "same plan, unsafe arguments" failures and remind reviewers that block decisions, denials, and policy traces are themselves a side channel worth modeling.

[Operationalizing CaMeL](https://arxiv.org/abs/2505.22852), submitted May 28, 2025, is a useful bridge between the CaMeL reference architecture and enterprise rollout. It adds prompt screening for initial inputs, output auditing for instruction leakage, a tiered-risk access model with a "Red Tier" that reserves full capability checks and multi-factor approval for high-risk irreversible or externally visible operations, and a verified intermediate language to support static guarantees. For C9.9 reviewers, this is concrete evidence that capability-based data-flow enforcement can be staged for production while preserving fail-closed behavior on the most consequential argument origins.

On April 30, 2026, the joint NSA/CISA/Five Eyes guidance [Careful Adoption of Agentic AI Services](https://media.defense.gov/2026/Apr/30/2003922823/-1/-1/0/CAREFUL%20ADOPTION%20OF%20AGENTIC%20AI%20SERVICES_FINAL.PDF) reinforced the same architectural theme from an operational risk angle. The guidance highlights privilege, design/configuration, structural, behavior, and accountability risks in agentic systems, including stale authorization checks, weak segmentation, excessive privileges, and audit logs that appear legitimate after a trusted agent is misused. For C9.9, the practical takeaway is to evaluate origin-aware authorization at each tool invocation, bind agent identities and credentials tightly to intended duties, and preserve enough dependency evidence to explain why a tool call happened after the fact.

The December 2025 [OWASP Top 10 for Agentic Applications 2026](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/) and OWASP's April 2026 exploit round-up make the same point in incident language: agent goal hijack, tool misuse, identity abuse, and human-agent trust exploitation often show up as ordinary-looking actions with bad data provenance. C9.9 therefore treats policy evaluation as a per-invocation data-flow decision, not only a prompt-safety or workflow-design review.

Production implementations usually need several layers working together: explicit orchestration roles for planners versus extractors, policy-as-code at the tool boundary, origin metadata on every value that can reach a tool argument, and durable telemetry that lets reviewers reconstruct which data sources influenced each invocation. As of May 2026, frameworks such as LangChain provide useful tool approval middleware, Pydantic AI supports deferred approval/external-execution patterns, the OpenAI Agents SDK exposes tool guardrails and tracing hooks for function-tool calls, and policy engines such as OPA/Rego or Cedar can enforce machine-readable decisions. Complete per-value provenance and taint-aware tool execution still usually require custom wrapper code around the agent runtime.

MCP deployments make the same data-flow issue concrete. OWASP's [MCP Tool Poisoning](https://owasp.org/www-community/attacks/MCP_Tool_Poisoning) write-up describes a runtime trust gap where a tool response enters the model context as trusted material, and [MCPTox](https://arxiv.org/abs/2508.14925) measured tool-metadata poisoning across real MCP servers and agents. AgentSeal's March 2026 scan of 5,125 MCP servers found [toxic data flows](https://agentseal.org/blog/toxic-data-flows-mcp-servers) where individually plausible tools combined into exfiltration, destructive-action, or privilege-escalation paths, and the follow-up [Runtime Exploitation of MCP Servers](https://agentseal.org/blog/runtime-exploitation-mcp-servers) work in 2026 turned several of those static findings into working exploits including command injection, output-instruction injection, and path traversal. On May 4, 2026, the Cloud Security Alliance's [MCP Security Crisis](https://labs.cloudsecurityalliance.org/research/csa-research-note-mcp-security-crisis-20260504-csa-styled/) note tied these composition risks to STDIO command execution, optional authorization, rug-pull changes, and cross-server tool shadowing. The MCP transport origin boundary is now a recurring failure mode across language SDKs: [CVE-2026-35568](https://nvd.nist.gov/vuln/detail/CVE-2026-35568) is a DNS rebinding flaw in the MCP Java SDK before 1.0.0 (CWE-346 Origin Validation Error) that lets a malicious website reach a local or network-private MCP server through a victim browser and invoke tools as if it were a locally running AI client; [CVE-2026-33252](https://vulnerability.circl.lu/vuln/cve-2026-33252) is a cross-site tool execution flaw in the MCP Go SDK Streamable HTTP transport before 1.4.1, where the server accepted browser-generated cross-site POSTs without Origin header validation and without requiring `Content-Type: application/json`; [CVE-2026-42559](https://nvd.nist.gov/vuln/detail/CVE-2026-42559), disclosed May 14, 2026, is a CVSS 8.8 DNS rebinding flaw in the official Rust SDK ([rmcp](https://github.com/modelcontextprotocol/rust-sdk/security/advisories/GHSA-89vp-x53w-74fx)) Streamable HTTP server transport before 1.4.0 that omitted Host header validation; [CVE-2025-66414](https://github.com/modelcontextprotocol/typescript-sdk/security/advisories/GHSA-w48q-cv73-mx4w), fixed in the TypeScript SDK 1.24.0, shipped DNS rebinding protection disabled by default on localhost HTTP transports; and [CVE-2026-35577](https://nvd.nist.gov/vuln/detail/CVE-2026-35577), published April 9, 2026 with CVSS 8.1 (CWE-346), is the same DNS rebinding pattern in the Apollo MCP Server StreamableHTTP transport before 1.7.0, which exposes GraphQL operations as MCP tools and let any visited site invoke those tools against a victim's localhost instance until Host header validation was added. The November 2025 MCP authorization specification adds better OAuth resource binding, scope challenge, and confused-deputy guidance for HTTP transports, but it also leaves authorization optional and STDIO credentials environment-dependent. For C9.9, these findings reinforce that tool review must cover transport-level origin and Host checks, compositions, configuration provenance, argument origins, and token audience boundaries, not just whether each tool looks safe in isolation.

The system-level framing tightened further in 2026. [Architecting Secure AI Agents](https://arxiv.org/abs/2603.30016), a March 31, 2026 position paper from authors at NVIDIA, MIT, CISPA, and the University of Wisconsin, argues that indirect prompt injection cannot be solved purely at the model layer and outlines three system-level positions that map directly onto C9.9: dynamic replanning and policy updates are required for realistic environments, security-relevant LLM decisions must operate inside system designs that strictly constrain what the model can observe and decide, and ambiguous cases require personalized human interaction rather than silent automation. The May 17, 2026 paper [AI Agents May Always Fall for Prompt Injections](https://arxiv.org/abs/2605.17634) by Sahar Abdelnabi and Eugene Bagdasarian reframes the problem through Contextual Integrity theory and proves an impossibility result: an adversary can always construct a context where a blocked flow looks legitimate, or a defender who tightens norms blocks legitimate ones. The authors group attacks into misrepresenting information flow, manipulating norms, and mixing multiple flows. For C9.9 reviewers, the practical message is that data-instruction filtering and content classifiers will not converge on a clean separation, so durable controls must live in the runtime — origin-tagged values, machine-interpretable policy, pre-dispatch denial, and CI-aware norms that consider sender, recipient, attributes, and transmission principle for each tool call.

October 2025 brought two complementary frames that C9.9 reviewers should know. [The Attacker Moves Second](https://arxiv.org/abs/2510.09023), a joint study from authors at Google DeepMind, OpenAI, Anthropic, and Stanford, broke 12 published prompt-injection and jailbreak defenses with adaptive attacks (gradient descent, reinforcement learning, random search, and human-guided exploration), pushing originally near-zero success rates above 90% across most defenses. The practical implication is that prompt-only or content-classifier guardrails should not be load-bearing controls for tool-call decisions, which makes architectural separation, machine-interpretable policy, and origin-aware enforcement at the dispatcher the durable controls C9.9 actually evaluates. Meta's [Agents Rule of Two](https://ai.meta.com/blog/practical-ai-agent-security/), published October 31, 2025, takes the opposite end of the spectrum: a deliberately simple session-scoping rule that an agent should hold at most two of (A) processing untrusted inputs, (B) access to sensitive systems or private data, and (C) the ability to change state or communicate externally. The rule is a useful capability-budget framing for reviewers because it maps cleanly to C9.9's data-flow concerns. Treat it as a complementary architectural budget rather than a defense, because the rule itself notes that least-privilege, sandboxing, and per-invocation enforcement remain non-negotiable underneath it.

Late May and early June 2026 added ecosystem-scale measurement evidence for the same failure modes. [VIPER-MCP](https://arxiv.org/abs/2605.21392), submitted May 20, 2026, ran automated taint-style auditing across 39,884 open-source MCP server repositories, evolving natural-language proof-of-concept prompts toward shell, network, and filesystem sinks; it confirmed 106 zero-day flaws with end-to-end exploit traces, with 67 CVE IDs assigned at publication — the "same plan, attacker-controlled arguments" failure mode demonstrated at catalog scale rather than as isolated CVEs. A companion [measurement study of real-world remote MCP servers](https://arxiv.org/abs/2605.22333) found 40.55% of 7,973 live remote servers exposing tools with no authentication at all, and every one of 119 testable OAuth-enabled servers had at least one authorization flaw (325 flaws total, 9 new CVE IDs), while Censys's May 27, 2026 [internet scan](https://censys.com/blog/mcp-servers-on-the-internet/) counted 12,520 internet-accessible MCP services across 8,758 hosts, including roughly 90 servers advertising shell-execution tools and 1,776 exposing direct database query interfaces without authentication. The defensive proposals are arriving in the same window: [Attested Tool-Server Admission](https://arxiv.org/abs/2605.24248), revised June 1, 2026, specifies offline-signed clearance assertions verified against a pinned trust root before any tool dispatch, deny-by-default per-server tool allowlists, and enforcement modes that convert warnings into hard denials with tamper-evident audit logging; [MCPShield](https://arxiv.org/abs/2605.11053), revised May 22, 2026, classifies per-session tool-call graphs enriched with argument and response embeddings (content embeddings reach above 0.89 AUROC versus roughly 0.64 for metadata-only features) and usefully warns that random-split evaluation inflates reported detection results by up to 26 points versus task-disjoint splits. In the same month, NSA's Artificial Intelligence Security Center released the CSI [Model Context Protocol (MCP): Security Design Considerations for AI-Driven Automation](https://www.nsa.gov/Portals/75/documents/Cybersecurity/CSI_MCP_SECURITY.pdf) (May 2026, U/OO/6030316-26), which frames the agentic environment as a continuum where "misaligned assumptions or subtle inconsistencies at any stage can propagate and compound into exploitable conditions," and recommends validation of tool invocations, signed MCP messages with expiration timestamps and replay protection, tool-execution sandboxing, tool name-collision and drift detection, outbound filtering with data-loss prevention, and detailed audit logs suitable for SIEM integration — government guidance that maps closely onto C9.9's per-invocation validation and dependency-evidence requirements.

Governance context also moved in May-June 2026. OWASP's June 1, 2026 [State of Agentic AI Security and Governance](https://genai.owasp.org/resource/state-of-agentic-ai-security-and-governance/) update notes that nearly every threat in the 2025 edition now has a CVE, vendor advisory, or production incident attached, and introduces a maturity ladder whose top level expects risk-tiered autonomy, drift dashboards, kill switches, and "governance-as-code" — machine-interpretable policy as the expected end state rather than a research aspiration, which is exactly C9.9.3's position. On the regulatory side, the EU's Digital Omnibus agreement (provisional political agreement May 6-7, 2026, Council confirmation May 13, 2026) [defers the Annex III high-risk obligations](https://www.gibsondunn.com/eu-ai-act-omnibus-agreement-postponed-high-risk-deadlines-and-other-key-changes/) — including the Article 12 automatic event-recording duties often cited as the compliance driver for agent traceability — from August 2, 2026 to December 2, 2027. Reviewers should treat Article 12 as direction-of-travel rather than an imminent deadline, while noting that incident-response needs justify the same dependency-graph evidence regardless of the regulatory timeline. The [MCP draft specification](https://modelcontextprotocol.io/specification/draft/changelog) (post-2025-11-25) is also relevant to origin enforcement: it moves toward stateless servers with per-request identity and capabilities carried in `_meta`, documents OpenTelemetry trace-context propagation conventions (`traceparent`, `tracestate`, and `baggage` keys), adds RFC 9207 issuer validation, and deprecates Dynamic Client Registration in favor of Client ID Metadata Documents — a standardized substrate that origin labels and session dependency graphs can ride on, though it still says nothing about per-argument provenance.

Use AgentDojo-style tasks, CaMeL-inspired tests, AttriGuard-style counterfactual checks that ask whether the tool call survives when untrusted observations are control-attenuated, and adaptive attack pressure (gradient descent, RL, random search, human-guided exploration) following the methodology from The Attacker Moves Second. | As of May 2026, this is an architectural property more than a library checkbox. Production adoption still requires application-specific decomposition of planning, extraction, and execution, and the defense must survive adaptive attackers, not just static prompt-injection test sets. |

| **9.9.3** | **Verify that** security policies for tool execution are expressed as auditable, versioned, machine-interpretable code or configuration, not solely as natural language instructions within prompts. | 2 | **Prompt-only policy bypass.** Natural-language instructions can be ignored, reinterpreted, or contradicted by injected content. OPA decision logs and Cedar schema validation are useful starting points, but agent-specific origin fields must be modeled explicitly and rejected when missing. |

Without origin metadata, the runtime cannot distinguish user-approved parameters from untrusted content. NeuroTaint argues this gap is structural for LLM agents: probabilistic natural-language reasoning means classical program-memory taint analysis underapproximates how source labels should propagate through summaries, translations, and chain-of-thought rewrites. | Trace sample values from user request through retrieval, parsing, summarization, transformation, and tool-call construction. | **9.9.6** | **Verify that** tool invocations where argument origin violates the applicable security policy are blocked before execution. | 3 | **Fail-open execution after detecting tainted arguments.** A system may log risky provenance but still call the tool, or it may ask the same compromised model to "try again" without changing the trust boundary. Verify the block event records policy version, decision ID, tool, redacted arguments, origin chain, user/session, and reason without surfacing distinguishing detail back into the model context. LangChain HITL middleware can pause selected tool calls, Pydantic AI deferred tools can require approval and preserve tool-call metadata, OpenAI Agents SDK tool guardrails can wrap custom function tools, [Open Agent Passport](https://arxiv.org/abs/2603.20953)-style synchronous pre-action authorization can produce cryptographically signed audit records at sub-100 ms latency for declarative policy evaluation against each call, and Operationalizing CaMeL's Red Tier pattern reserves capability checks and multi-factor approval for high-risk irreversible operations. Include adaptive-attack regression tests so the block point keeps holding under iterative attacker effort, not just static prompt-injection strings. | A useful control has a hard enforcement point outside the model's text generation path. Re-prompting is acceptable for recovery only after the unsafe call is blocked. ClawGuard's May 2026 revision is a helpful test reference for deterministic tool-call boundary enforcement, and Causality Laundering's Agentic Reference Monitor is a useful pattern for treating denied actions as graph nodes rather than as transient errors. Production systems still need credential scoping, destination allowlists, and tool-side validation so an orchestrator bug does not become the only barrier. Watch framework boundaries closely: OpenAI Agents SDK tool guardrails apply to custom function tools, while hosted MCP, file search, code-interpreter, shell, computer-use, and similar built-in tools need separate enforcement wrappers or policy gates. For tool-server admission specifically, the June 2026 [Attested Tool-Server Admission](https://arxiv.org/abs/2605.24248) proposal is a useful review reference: offline-signed clearance assertions checked against a pinned trust root before dispatch, deny-by-default per-server allowlists, and an RFC-2119 normative spec with conformance vectors that auditors can borrow as test cases. |
| **9.9.7** | **Verify that** data-flow integrity is enforced such that untrusted data cannot modify tool arguments beyond what the security policy explicitly permits, even when the control flow (sequence of tool calls) remains as intended. | 3 | **Same-plan, different-argument attacks.** The attacker does not need to change the tool sequence if they can alter a recipient, file path, SQL predicate, webhook URL, prompt template, amount, remediation command, or access-control filter inside a permitted call. This is where prompt injection becomes a data-flow integrity failure rather than a planning failure. AdapTools and LogJack both reinforce that adaptive payloads and operational-log channels can bypass narrow static filters while still steering consequential arguments, and VIPER-MCP's May 2026 sweep of 39,884 MCP server repositories (106 confirmed zero-days, 67 CVEs) showed that natural-language input routinely flows untainted into shell, network, and filesystem sinks across the deployed tool ecosystem. | Compare the original trusted intent, approved plan, and final tool arguments. Verify immutable or policy-protected fields cannot be filled from untrusted sources unless explicitly permitted. Use taint tests on recipient fields, URLs, file paths, query filters, command arguments, remediation commands, and free-text bodies. Include LogJack-style payloads in cloud logs or observability events when debugging or remediation agents are in scope. For MCP-style tool catalogs, build pairwise and multi-step tests for untrusted-content-to-public-sink, private-data-to-public-sink, untrusted-content-to-destructive-action, privileged-output-to-upload, rug-pull definition changes, and cross-server tool-shadowing flows. For high-impact operations, bind user approval to exact arguments using a signed approval token or equivalent manifest so later model output cannot rewrite approved values. | Data-flow integrity policies need field-level granularity. A single tool call may legitimately combine trusted and untrusted data, such as a user-approved recipient with an externally sourced summary. The policy should express which fields may contain untrusted content, which require trusted origin, and which require sanitization or human review. "Sanitize then execute" behavior is still unsafe if the remaining command or argument is attacker-controlled. Large tool catalogs increase the number of possible tool pairs quickly, so sampling only the most obvious read/write pairings can miss the path that attackers use. Treat tool definition drift as part of the data-flow problem: a previously approved manifest can become an untrusted source if it changes without re-approval. |
| **9.9.8** | **Verify that** the system's data-flow dependency graph is maintained per session and is available for review by authorized security and operations personnel for post-hoc audit, enabling identification of which data sources influenced each tool invocation and its arguments. | 3 | **Forensic blind spots after agent misuse.** Without a dependency graph, responders cannot tell whether a bad tool call came from user intent, retrieved content, a tool response, memory, another agent, or a compromised connector. EU AI Act Article 12 reinforces the need for traceability and automatic event recording for high-risk AI systems (note the May 2026 Digital Omnibus agreement defers the Annex III high-risk obligations from August 2, 2026 to December 2, 2027, so this is a forward obligation, not an imminent deadline), and the April 2026 joint agentic AI guidance calls out accountability gaps when trusted agents obscure who or what caused an action. | Review session telemetry for a complete graph: user request, model turns, retrieved documents, tool outputs, derived values, policy decisions, approvals, denied actions, tool invocations, agent identities, credentials used, and side effects. Confirm authorized reviewers can answer "which sources influenced this argument?" and "which identity executed it?" without reading raw secrets. Use OpenTelemetry traces for causal execution flow, OPA decision IDs or equivalent policy logs for enforcement, OpenAI Agents SDK-style spans for model/tool/guardrail events where applicable, and lineage/provenance records such as W3C PROV, OpenLineage-style events, or PROV-AGENT-style agent-aware extensions for data dependencies. Where OpenTelemetry baggage is used, verify that sensitive or security-critical origin labels are not blindly propagated to third-party services and are not trusted without integrity checks. Spot-check at least three recent sessions and verify graph completeness across memory writes, summarization steps, and agent-to-agent handoffs. The session graph can also feed detection: MCPShield (May 2026) shows that per-session tool-call graphs with sequential and data-flow edges plus argument/response embeddings separate attacked from benign sessions far better than metadata alone, and NSA's May 2026 MCP CSI expects detailed tool-invocation audit logs suitable for SIEM integration. | Full payload logging can violate privacy and leak secrets, so store hashes, classifications, redacted snippets, and secure references where possible. The graph should survive session cleanup and tool container deletion for the retention period required by incident response and compliance. Gaps are common around memory retrieval, summarization steps, agent-to-agent handoffs, guardrail-only decisions, tool calls executed through shared service credentials, and always-on agents that own memory stores, schedulers, or filesystem patches where Sleeper Channels-style persistent attacks can detonate later through a confused-deputy cron path long after the original session ended. OpenLineage-style custom facets and PROV-AGENT-style agent activities are useful for durable lineage, but auditors should check that facet schemas are version-pinned, that denied-action nodes are first-class graph elements, and that parent-child relationships survive asynchronous jobs and external tool execution. |

## Implementation Guidance

### Implementation Maturity

Emerging to early mature. Policy engines, approval middleware, tracing, and tool-call guardrails are available, but few platforms provide complete per-value provenance, taint-aware argument construction, cross-tool composition analysis, and tool-definition integrity out of the box. As of June 2026, deterministic per-invocation policy at the tool boundary has shipped in managed form (AWS AgentCore Policy, Microsoft Agent Governance Toolkit, Cloudflare MCP portals), but per-argument origin metadata (9.9.4) and origin-conditional policy (9.9.5) still have no mainstream off-the-shelf implementation — the closest open-source attempt, a CaMeL-style taint-wrapper RFC in a popular agent runtime, was closed unadopted in 2026. Treat vendor guardrails as useful enforcement plumbing only after confirming where they run, which tool types they cover, whether they see structured origin labels, and whether they fail closed before side effects.

### Notable Incidents and Case Studies

- **October 3, 2025: Amp DNS exfiltration via prompt injection.** MCPSec demonstrated that untrusted repository content could steer an agent into running a DNS command that encoded an API key in a subdomain without asking for approval. The useful C9.9 test case is not the exact command; it is the untrusted-file-to-shell-argument path and whether network-capable commands are denied when their arguments derive from project content.
- **November 19, 2025: ServiceNow Now Assist second-order prompt injection.** AppOmni reported that agent discovery and team-based collaboration settings could let a low-privilege, data-reading agent recruit more powerful agents to perform sensitive actions. This is a cross-agent version of the same origin problem: the task looks like normal delegation unless reviewers can see which untrusted field influenced the later action.
- **March 20, 2026: MCP toxic data-flow scan.** AgentSeal reported that 555 of 5,125 scanned MCP servers had tool combinations that could create exfiltration, destructive-action, privilege-escalation, or untrusted-relay paths. Treat this as a review pattern for installed tool catalogs: identify source tools, sink tools, and the policy that prevents untrusted data from connecting them.
- **March 2026: MCP ecosystem CVE wave.** Independent reporting and a [community CVE round-up](https://www.heyuan110.com/posts/ai/2026-03-10-mcp-security-2026/) tallied roughly 30 MCP-related CVEs in 60 days, dominated by exec/shell injection (~43%), tooling-infrastructure flaws in clients/inspectors/proxies (~20%), and authentication bypass (~13%). A scan of 560 MCP servers reported 38% with no authentication and a survey of 2,614 implementations reported 82% exposing path-traversal-prone file operations. The C9.9 takeaway is that even when origin-aware dispatch is enforced upstream, the catalog itself routinely ships without input validation, authentication, or trustworthy tool descriptions — review tool catalogs for these classes before relying on dispatcher policy.
- **April 14, 2026: OWASP GenAI exploit round-up.** OWASP's Q1 2026 report mapped recent agentic incidents to ASI01 Agent Goal Hijack, ASI02 Tool Misuse, and ASI09 Human-Agent Trust Exploitation, and noted that many events are architectural or configuration failures rather than CVE-style bugs. For C9.9 reviews, that is a reminder to preserve provenance and policy evidence even when no single vulnerable package explains the misuse.
- **May 4, 2026: CSA MCP systemic design-risk note.** The Cloud Security Alliance summarized MCP attack patterns spanning STDIO command execution, optional authorization, rug-pull tool changes, cross-server tool shadowing, and unauthenticated deployments. For C9.9, the practical review point is that data-flow isolation has to include tool configuration provenance, re-approval on definition changes, per-server session isolation, and behavioral monitoring of MCP server processes.
- **April-May 2026: MCP transport origin CVEs.** [CVE-2026-35568](https://nvd.nist.gov/vuln/detail/CVE-2026-35568) (CVSS 7.6, CWE-346) reports that the MCP Java SDK before 1.0.0 omitted Origin validation, enabling DNS rebinding so a malicious site could invoke local MCP tools through a victim's browser as if it were a locally running AI client. [CVE-2026-33252](https://vulnerability.circl.lu/vuln/cve-2026-33252) (CVSS 7.1) reports the MCP Go SDK Streamable HTTP transport before 1.4.1 accepting cross-site POSTs without Origin validation and without requiring `Content-Type: application/json`, allowing arbitrary sites to drive sessionless local servers. [CVE-2026-42559](https://nvd.nist.gov/vuln/detail/CVE-2026-42559) (CVSS 8.8, CWE-346/CWE-350), disclosed May 14, 2026, reports the official Rust SDK ([rmcp](https://github.com/modelcontextprotocol/rust-sdk/security/advisories/GHSA-89vp-x53w-74fx)) Streamable HTTP server transport before 1.4.0 not validating the Host header, so DNS rebinding from a malicious public site could invoke any tool exposed by a locally-running rmcp server. [CVE-2025-66414](https://github.com/modelcontextprotocol/typescript-sdk/security/advisories/GHSA-w48q-cv73-mx4w) (CVSS 7.6) reports that the MCP TypeScript SDK before 1.24.0 shipped with DNS rebinding protection disabled by default on localhost HTTP transports, fixed by Host header validation that returns 403 when the Host header does not match an expected localhost value. [CVE-2026-35577](https://nvd.nist.gov/vuln/detail/CVE-2026-35577) (CVSS 8.1, CWE-346), published April 9, 2026, reports the same DNS rebinding pattern in the [Apollo MCP Server](https://github.com/apollographql/apollo-mcp-server) before 1.7.0, which exposes GraphQL operations as MCP tools and accepted arbitrary Host headers on StreamableHTTP until a Host allowlist was added. For C9.9, these are concrete reminders that the MCP transport origin boundary is now an active attack surface across language ecosystems and server implementations, and that argument-origin policy at the dispatcher is not a substitute for transport-level Origin, Host, and Content-Type checks on the server itself.
- **May 13, 2026: MCP back-end database CVEs.** Akamai researcher Tomer Peled disclosed [three back-end vulnerabilities](https://www.akamai.com/blog/security-research/one-fluke-3-pattern-mcp-back-end-vulnerabilities) across popular MCP database servers, summarized publicly in [The Register](https://www.theregister.com/security/2026/05/13/bug-hunter-tracks-down-three-serious-mcp-database-flaws-one-left-unpatched/5238916). [CVE-2025-66335](https://github.com/advisories/GHSA-qhfq-gvvc-5q6q) in Apache Doris MCP before 0.6.1 lets an attacker inject SQL through an unvalidated `db_name` parameter in the `exec_query` tool, leading to arbitrary command execution against the Doris instance (patched in December 2025). Apache Pinot MCP (StarTree integration before v2.0.0) shipped without authentication on its HTTP transport, allowing remote callers to invoke MCP tools and run SQL with full takeover potential; StarTree added an optional OAuth path but the SQL injection in the underlying code remained as of disclosure. The Alibaba RDS MCP server invoked its RAG tool without authenticating the caller, exposing table names and schema metadata to anyone able to reach the endpoint, and Alibaba declined to patch. For C9.9, these are reminders that the data-flow policy at the agent dispatcher cannot rescue a back-end MCP server that ships without authentication or input validation; tool catalog review must cover server-side auth, query construction, and unauthenticated tool surfaces in addition to client-side origin labels.
- **March 30, 2026: nginx-ui "MCPwn" missing-auth on /mcp_message, CVE-2026-33032.** Endor Labs and Pluto Security disclosed [CVE-2026-33032](https://nvd.nist.gov/vuln/detail/CVE-2026-33032) (CVSS 9.8, CWE-306) in nginx-ui ≤ 2.3.5: the `/mcp` endpoint correctly required `AuthRequired()` middleware, but the parallel `/mcp_message` endpoint that routed to identical MCP handlers was registered without it, and the IP allowlist defaulted to empty — which the middleware treated as allow-all. Any network-reachable attacker could invoke MCP tools to restart nginx, write or delete configuration files, and trigger automatic reloads, intercepting traffic by inserting attacker-controlled proxy directives. [Recorded Future flagged it as actively exploited](https://thehackernews.com/2026/04/critical-nginx-ui-vulnerability-cve.html) in March 2026. The C9.9 takeaway is that argument-origin policy at the dispatcher does not help when the *tool server itself* has a missing-auth or fail-open default; tool-catalog review must include per-endpoint auth coverage and the default behavior of any allowlist used as a security boundary.
- **April 24, 2026: Splunk MCP Server token disclosure in observability logs, CVE-2026-20205.** Splunk patched [CVE-2026-20205](https://nvd.nist.gov/vuln/detail/CVE-2026-20205) ([SVD-2026-0407](https://advisory.splunk.com/advisories/SVD-2026-0407), CVSS 7.2, CWE-532) in Splunk MCP Server app prior to 1.0.3: session and authorization tokens were written to the Splunk `_internal` index in clear text, where any user holding the `mcp_tool_admin` capability or read access to `_internal` could harvest them. This matters for C9.9 even though it is not a code-execution flaw: the same `_internal` index is frequently consumed by debugging or incident-response agents, so leaked credentials can re-enter the agent loop as observation data and be replayed as authenticated tool calls. Treat tool-server logs as both forensic evidence and a potential data-flow source, mask secrets before they reach storage, and inventory which agents currently have read access to telemetry indexes that may contain tokens. This is a real-world LogJack-style channel rather than only a research scenario.
- **March 10, 2026: Azure MCP Server SSRF, CVE-2026-26118.** Microsoft's March 2026 Patch Tuesday addressed [CVE-2026-26118](https://msrc.microsoft.com/update-guide/vulnerability/CVE-2026-26118) (CVSS 8.8, CWE-918) in Azure MCP Server prior to 2.0.0, including all 2.0.0-beta builds. An MCP tool that accepted user-provided Azure resource identifiers forwarded the value into an outbound HTTP request without restricting destinations, so a low-privileged caller could substitute a URL pointing to cloud metadata services or internal endpoints and have the server attach its managed identity token to the request. This was reported as the first CVE to directly target the MCP layer in a production system and is a textbook same-plan, unsafe-argument case: the agent calls the documented tool with documented-looking parameters, but the destination field is attacker-controlled. The C9.9 takeaway is that origin metadata on URL-shaped arguments and per-invocation destination policy must run before any outbound dispatch, not after, and that managed identities attached to MCP servers need scope and audience controls so SSRF cannot escalate into broader Azure resource access.
- **April 2, 2026: Azure MCP Server missing authentication, CVE-2026-32211.** Microsoft disclosed [CVE-2026-32211](https://nvd.nist.gov/vuln/detail/CVE-2026-32211) (CWE-306, CVSS 9.1 per Microsoft / 7.5 per NIST) where an Azure MCP Server function exposed sensitive configuration to unauthenticated callers over the network. For C9.9, this reinforces that tool catalogs must be treated as data-flow sources whose authentication state is part of the origin label — a server that does not authenticate the caller cannot meaningfully attest to the origin of values it returns, and any downstream policy decision based on that data inherits the gap.
- **April 15-20, 2026: MCP STDIO command-injection ecosystem advisory.** [OX Security's joint advisory](https://www.ox.security/blog/mcp-supply-chain-advisory-rce-vulnerabilities-across-the-ai-ecosystem/) traced a recurring command-injection pattern in MCP STDIO transports across Cursor, VS Code, Windsurf, Claude Code, Gemini-CLI, and GitHub Copilot, with [CVE-2026-30615](https://nvd.nist.gov/vuln/detail/CVE-2026-30615) (Windsurf 1.9544.26, CVSS 8.0, CWE-77) the only case requiring zero user interaction: when the assistant processed attacker-controlled HTML, an injected payload wrote a new STDIO server entry into the local MCP configuration, which the runtime then launched with arbitrary commands and no further approval. OX reported roughly 7,000 publicly accessible MCP servers and over 150 million cumulative downloads across the Python, TypeScript, Java, and Rust implementations covered by the broader pattern, with [The Hacker News write-up](https://thehackernews.com/2026/04/anthropic-mcp-design-vulnerability.html) noting more than a dozen assigned CVEs (e.g., CVE-2026-30615 Windsurf, CVE-2026-30623 LiteLLM, CVE-2026-30624 Agent Zero, CVE-2026-30625 Upsonic, CVE-2026-40933 Flowise). Anthropic characterized the STDIO execution model as expected behavior and declined to modify the reference protocol, framing sanitization and origin checks as integrator responsibility. For C9.9, this is a direct test of whether untrusted-content paths can rewrite the *tool catalog itself* — not just an argument — and whether the planner re-evaluates origin, authorization, and policy after configuration changes; treat any unsanitized HTML, markdown, or repository content that reaches an agent that also has write access to MCP configuration as a same-plan, unsafe-tool-registration failure mode that argument-level policy alone will not catch.
- **May 7, 2026: Microsoft Semantic Kernel RCE through agent tool arguments, CVE-2026-26030 and CVE-2026-25592.** Microsoft Security disclosed [two vulnerabilities](https://www.microsoft.com/en-us/security/blog/2026/05/07/prompts-become-shells-rce-vulnerabilities-ai-agent-frameworks/) in Semantic Kernel where prompt-injected content flowed directly into tool parameters that drove code execution. CVE-2026-26030 (fixed in Python `semantic-kernel` 1.39.4) involved unsafe string interpolation in the In-Memory Vector Store filter, letting attacker text become part of a lambda expression that then traversed Python's `__class__` chain to `BuiltinImporter` and executed shell commands. CVE-2026-25592 (fixed in .NET SDK 1.71.0) mistakenly marked `DownloadFileAsync` with `[KernelFunction]`, exposing it to the model so a prompt could ask the agent to write attacker-controlled scripts to the Windows Startup folder for persistence. The Microsoft write-up frames the root cause as untrusted data being used as input for high-risk operations — i.e., the C9.9 failure mode where the plan looks legitimate but at least one argument origin is attacker-controlled. For C9.9 reviewers, the practical evidence is to inventory which tools accept free-form text into filters, file paths, or download URLs; require origin-tagged arguments for those fields; and remove or scope-restrict `KernelFunction`-style attributes (and equivalents in other frameworks) that expose sensitive operations to the planner without an explicit policy gate.
- **May 11, 2026: PraisonAI authentication bypass exploited within four hours, CVE-2026-44338.** The PraisonAI multi-agent framework's legacy Flask API server hardcoded `AUTH_ENABLED = False`, leaving `GET /agents` and `POST /chat` unauthenticated (CVSS 7.3, affecting 2.5.6 through 4.6.33, fixed in 4.6.34). [Sysdig's honeypot telemetry](https://www.sysdig.com/blog/cve-2026-44338-praisonai-authentication-bypass-in-under-4-hours-and-the-growing-trend-of-rapid-exploitation) recorded a scanner probing the exact vulnerable endpoint 3 hours 44 minutes after the advisory published. The C9.9 relevance is the blast radius: an unauthenticated caller could enumerate the agent configuration and drive whatever tools, databases, and cloud resources the agents were wired to — an entire origin-enforcement stack upstream is moot when the agent endpoint itself accepts anonymous instructions.
- **May 20-27, 2026: MCP exposure measured at ecosystem scale.** [VIPER-MCP](https://arxiv.org/abs/2605.21392) confirmed 106 taint-style zero-days (67 CVE IDs assigned) across 39,884 scanned MCP server repositories by evolving natural-language prompts toward shell, network, and filesystem sinks; a parallel [remote-server measurement study](https://arxiv.org/abs/2605.22333) found 40.55% of 7,973 live remote MCP servers expose tools with no authentication and all 119 testable OAuth deployments had flaws (9 new CVEs); and [Censys](https://censys.com/blog/mcp-servers-on-the-internet/) counted 12,520 internet-accessible MCP services, including ~90 advertising shell-execution tools and 1,776 exposing direct database query interfaces. For C9.9 reviews, this is the strongest evidence yet that tool-catalog review cannot assume the catalog's own baseline hygiene: the deployed ecosystem ships unauthenticated, taint-prone tool surfaces by default.
- **Q4 2025 / 2026 ecosystem signals.** [The Attacker Moves Second](https://arxiv.org/abs/2510.09023) (October 2025, OpenAI/Anthropic/Google DeepMind/Stanford authors) bypassed 12 published prompt-injection and jailbreak defenses with adaptive attacks at over 90% success, reinforcing that prompt-layer or content-classifier guardrails cannot substitute for architectural separation and origin-aware enforcement. Meta's [Agents Rule of Two](https://ai.meta.com/blog/practical-ai-agent-security/) (October 31, 2025) gave operators a compact capability budget — process untrusted input, access sensitive data, or change external state, pick at most two per session — that is useful as a session-design heuristic complementing C9.9's value-level origin checks. The [Manifold/Manifest](https://siliconangle.com/2026/05/12/manifold-scores-7700-mcp-servers-manifest-expansion-aimed-agent-security-teams/) expansion published May 12, 2026 scored roughly 7,700 MCP servers using a Lineage Score for publisher provenance and a Safety Score for behavioral risk, surfacing that catalog-scale provenance scoring is becoming a procurement input rather than a research curiosity.

### Origin-Aware Tool Boundary

Treat the tool dispatcher as a policy enforcement point. Before any tool call runs, the dispatcher should receive a structured request containing the tool name, argument object, user or delegated agent identity, side-effect classification, destination, origin metadata for each argument, and the policy version to evaluate. The dispatcher should return an allow, deny, or review decision with a durable decision ID. The model can propose a call, but it should not be the component that authorizes the call.

### MCP Authorization Boundary

For MCP HTTP transports, verify OAuth resource indicators, audience validation, and scope challenges before treating a server as an approved tool source. These checks reduce token confusion and overbroad grants, but they answer "may this client call this server?" rather than "may this untrusted value populate this argument?" Keep the value-origin decision at the dispatcher or tool wrapper even when MCP transport authorization is configured correctly. Also verify transport-level Origin header and Host header validation, Content-Type enforcement, and bind locally to 127.0.0.1 rather than 0.0.0.0 — CVE-2026-35568 (Java), CVE-2026-33252 (Go), CVE-2026-42559 (Rust rmcp), CVE-2025-66414 (TypeScript), and CVE-2026-35577 (Apollo MCP Server) all show that an SDK or server that skips any of these checks turns the victim's browser into a remote MCP client and an argument-origin policy higher up the stack will not see those calls. Pin SDK versions, track upstream advisories, and treat any local or private-network MCP server as DNS-rebinding-exposed unless transport-level Origin and Host validation are confirmed.

Server-side argument handling matters as much as transport authorization. CVE-2026-26118 in Azure MCP Server prior to 2.0.0 showed that a tool accepting an Azure-resource-shaped URL parameter could be steered into outbound SSRF that attached the server's managed identity token to attacker-chosen destinations. CVE-2026-32211 (CWE-306) showed that an MCP-exposed function shipped without authentication, leaking configuration over the network. Treat URL-typed and identifier-typed arguments as high-sensitivity destination fields, require origin metadata that distinguishes user-approved identifiers from model-generated or retrieved values, and gate any outbound network egress (including managed-identity-bound HTTP calls) through the same dispatcher policy that authorizes other side effects. Validate that managed identities, OAuth audiences, and back-end scopes attached to an MCP server cannot be reused implicitly when the tool argument changes provenance.

For STDIO MCP servers, treat the process launch configuration, inherited environment, command path, package version, and server manifest as part of the trust boundary. The November 2025 authorization specification explicitly leaves STDIO outside the HTTP OAuth flow, so auditors should look for separate controls: pinned binaries, minimal environment variables, per-server working directories, no shared high-value credentials, and re-approval when tool definitions or command lines change.

The [MCP draft specification](https://modelcontextprotocol.io/specification/draft/changelog) that follows 2025-11-25 is worth tracking for this boundary. It moves toward stateless servers with identity and capabilities carried per-request in `_meta`, adds RFC 9207 `iss` validation and binds credentials to the issuing authorization server, deprecates Dynamic Client Registration in favor of Client ID Metadata Documents, documents OpenTelemetry trace-context propagation conventions, and tightens tool input/output contracts with JSON Schema 2020-12. None of that supplies per-argument origin labels, but it standardizes the request-level identity and trace plumbing that a dispatcher policy can anchor to. NSA's May 2026 MCP CSI adds the operational expectations reviewers can cite: signed MCP messages with expiration and replay protection, tool-execution sandboxing, tool name-collision and drift detection, outbound filtering with DLP, and audit logs detailed enough for SIEM-based reconstruction.

### Runtime Enforcement Checks

At a minimum, the pre-dispatch gate should validate three things before a tool executes: the proposed call is still attributable to trusted user intent, each protected argument field has an allowed origin, and the current policy version permits the destination and side effect. If any of those checks cannot be evaluated because origin metadata is missing or malformed, the safer default is deny or human review, not silent downgrade to prompt-only judgment.

As of June 2026, packaged options for pre-dispatch enforcement are starting to ship beyond research artifacts. Microsoft's [Agent Governance Toolkit](https://opensource.microsoft.com/blog/2026/04/02/introducing-the-agent-governance-toolkit-open-source-runtime-security-for-ai-agents/) (MIT, [microsoft/agent-governance-toolkit](https://github.com/microsoft/agent-governance-toolkit)) ships an Agent OS policy engine that intercepts every agent action with sub-millisecond latency (the project README reports 0.012 ms p50 for a single rule and roughly 35 K ops/sec concurrent), pairs it with AgentMesh identity, an Agent Runtime sandbox, and an Agent Compliance verifier, and bundles integrations for LangChain, CrewAI, LlamaIndex, OpenAI Agents SDK, LangGraph, and PydanticAI. The project's May 8 through June 9, 2026 [release train](https://github.com/microsoft/agent-governance-toolkit/releases) (v3.5.0 to v4.1.0) is directly relevant to C9.9 evidence collection: a "Decision BOM" reconstructible action-lineage view, intent-based authorization with drift detection, execution-ring privilege isolation, wire-protocol-aware policy evaluation for SQL and Kubernetes across TypeScript/Rust/Go/.NET, a LangGraph governance adapter with stale-authorization fingerprinting, Cedar-based policy packages, and Ed25519-signed agent registration — plus over a dozen authorization-bypass fixes in its own code, a reminder that enforcement layers are attack surface too. Treat the latency and OWASP-coverage claims as vendor benchmarks rather than independent measurements, and require local testing that confirms it sees structured origin metadata and fails closed on policy-engine outage before treating it as load-bearing. The same review approach applies to other gateway/runtime products in this space — usefulness depends on whether they receive the structured origin labels described in 9.9.4 and enforce per-invocation rather than per-session decisions.

Managed and gateway-layer options are filling in alongside it. [AWS Bedrock AgentCore Policy](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/policy.html) (generally available since March 2026) intercepts every agent-to-tool request at the AgentCore Gateway and evaluates Cedar policies — with conditions over user identity and tool input parameters — deterministically outside the model loop, supports natural-language policy authoring validated by automated reasoning, and offers enforce-versus-log modes with CloudWatch decision logs. Cloudflare's [enterprise MCP reference architecture](https://blog.cloudflare.com/enterprise-mcp/) (April 2026) combines AI Gateway, MCP Server Portals, and DLP guardrails so that agents reach tools only through a mediated portal with per-user-group tool exposure and centralized logging. Early open-source gateways such as [mcp-firewall](https://github.com/ressl/mcp-firewall) (pre-1.0 as of June 2026) add YAML/OPA policy, inbound injection and tool-chain checks, outbound secret/PII scanning, and hash-chained signed audit events, but have no tagged releases yet — verify maturity before depending on them. None of these products consume per-argument origin labels yet, so they enforce identity, destination, and parameter-value policy rather than full 9.9.4-style provenance policy.

For testing, keep a regression pack that mutates arguments without changing the tool sequence. Useful cases include hostile web text changing an email recipient, log text changing a remediation shell command, retrieved documents changing a database predicate, tool output changing a webhook URL, and mixed-origin text flowing into a public upload. These tests catch the failure mode where an agent appears to follow the approved plan while attacker-controlled data rewrites the consequential fields.

### Evidence Auditors Should Request

- Architecture diagrams showing the separation between planner, untrusted-data processors, policy engine, and tool executor.
- Policy-as-code repositories or managed policy snapshots with review history and tests.
- Tool-call traces showing argument origins, policy inputs, policy decisions, trace/span IDs, approvals, and execution outcomes.
- Negative-test results for indirect prompt injection, origin laundering, and same-plan argument manipulation.
- MCP or plugin inventory showing approved server versions, configuration provenance, tool-definition hashes, OAuth resource indicators, scope grants, token-audience validation, and re-approval records after manifest changes.
- Incident response queries that reconstruct which sources influenced a specific tool invocation.
- API and tool manifests that declare agent trust requirements, such as OpenAPI `x-agent-trust` metadata or MCP authorization scopes, mapped to runtime checks rather than accepted as documentation-only controls.

### Practical Starting Patterns

- Use a no-tool or read-only extractor for untrusted documents, web pages, emails, tickets, and tool outputs.
- Pass extracted values as typed data objects with source IDs and content hashes, not as unstructured text pasted into the planner context.
- Keep sensitive destinations, recipients, paths, SQL predicates, and mutating parameters under trusted-origin or explicit-approval rules.
- Route policy decisions through OPA/Rego, Cedar, or equivalent application code that produces replayable decision logs.
- Validate policy schemas so origin, trust class, side-effect level, destination, and agent identity are required fields for protected tools.
- Use runtime middleware and guardrails as enforcement plumbing, but treat them as incomplete unless they receive structured origin labels and fail closed before tool side effects.
- For MCP and large tool catalogs, review toxic combinations as a system: untrusted input tools, private-data tools, public sinks, destructive tools, privileged execution tools, and any server that combines several of those roles.
- Apply Meta's Agents Rule of Two as a session-scoping budget: an agent session that processes untrusted input, holds sensitive data access, and can change external state should require human approval or be split before C9.9's value-level origin policy is invoked.
- Pin MCP server packages and tool definitions; require re-approval when descriptions, schemas, command lines, scopes, or server origins change. Track upstream SDK advisories — Host/Origin validation has shipped late or off-by-default across the Java, Go, Rust, and TypeScript SDKs.
- Use OpenTelemetry traces for execution causality and a separate provenance store for per-value or per-object data dependencies.
- Redact or hash sensitive argument values in logs, but keep enough metadata to support incident reconstruction.
