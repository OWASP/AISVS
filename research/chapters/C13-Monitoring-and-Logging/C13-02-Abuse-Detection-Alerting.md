# C13.2: Abuse Detection and Alerting

> **Parent:** [C13 Monitoring, Logging & Anomaly Detection](C13-Monitoring-and-Logging)
> **Requirements:** 8 (13.2.1 -- 13.2.8)

## Purpose

This section addresses the detection of adversarial activity, abuse patterns, and attack campaigns targeting AI systems. Unlike traditional application security monitoring, AI abuse detection must account for natural language attack vectors (prompt injection, jailbreaks), multi-turn attack strategies, and AI-specific exfiltration techniques. These controls bridge the gap between raw logging (C13.1) and incident response (C13.5).

Provides classification scores and block/allow decisions that can be logged per 13.1.5.
- **Cloudflare Firewall for AI** -- Edge-layer protection that intercepts LLM API traffic, applying prompt injection and jailbreak detection rules across all models behind a single policy. Enforces guardrails at the network layer before requests reach model endpoints.
- **Akamai Firewall for AI** -- Real-time monitoring and adaptive filtering that detects jailbreaks, prompt injections, and toxic content with guardrail policies applied at the CDN/WAF layer.
- **Palo Alto Networks AI Security** -- Enterprise AI security posture management with prompt injection detection integrated into their NGFW and Prisma Cloud platforms. Unit 42 research has documented web-based indirect prompt injection attacks against AI agents observed in the wild.

- **CrowdStrike Falcon AIDR** -- As of March 2026, provides real-time detection and prevention of prompt injection, jailbreaks, and model manipulation. Deploys via browser extensions, SDKs (Python, Node.js, Go, Java, C#) with OpenTelemetry support, and AI/API gateway integrations (LiteLLM, Kong, Apigee, Azure API Gateway). Findings stream directly to Falcon Next-Gen SIEM for cross-domain correlation across endpoints, cloud, identity, SaaS, and AI behaviors. ### Open-Source Detection Tools

For organizations building detection pipelines without commercial products, several open-source tools address requirements 13.2.1 and 13.2.4:

- **Vigil** -- Python library and REST API for assessing LLM prompts and responses. Uses multiple detection methods: vector database/text similarity, YARA/heuristic rules, transformer model analysis, prompt-response similarity, and canary tokens. Best-in-class for low false-positive deployments (0% FPR at 83.8% recall in June 2025 benchmarks).
- **Rebuff** -- Multi-layered prompt injection defense by Protect AI. Combines heuristic filtering, LLM-based detection (GPT-4o achieves 0.021 FPR vs. 0.499 for GPT-3.5-turbo), vector database of known attacks, and canary tokens. Most balanced trade-off at 3.4% FPR and 98.1% recall.
- **LLM Guard** -- Open-source prompt injection detection with highest recall (99.9%) but higher false-positive rate (12.7% FPR). Hard-coded detection threshold at 0.92 limits configurability.

A comparative evaluation (arxiv:2506.19109) found that combining multiple scanner types outperforms any single technique, and vector database scanners degrade 10-40% against obfuscated attacks. Canary token checks in default configurations showed 0% detection rates but improved to 41-94% with explicit instructions.

### Evolving Attack Landscape

The threat landscape for AI abuse detection continues to evolve rapidly:

- **Best-of-N (BoN) Jailbreaking** -- Automated attacks that generate large numbers of prompt variations using obfuscation techniques (character substitution, encoding, rephrasing), achieving near-100% success rates against safety filters in seconds. This makes signature-based detection (13.2.1) necessary but insufficient -- behavioral detection (13.2.4) and session analysis (13.2.8) are critical complementary layers.
- **Multi-turn Crescendo Attacks** -- Attackers gradually shift model behavior across conversation turns, with no single turn being overtly malicious. Detection requires session-level trajectory analysis (13.2.8) using sliding window analysis over conversation embeddings or cumulative risk scoring.
- **Indirect Prompt Injection via Tool Use** -- As AI agents gain tool-calling capabilities, adversarial content embedded in retrieved documents, web pages, or API responses can hijack agent behavior. Detection must extend beyond user-submitted prompts to cover all inputs the model processes.
- **LLM-as-C2-Channel** -- No longer theoretical. LAMEHUG (identified by CERT-UA in July 2025) uses the Hugging Face API with Qwen 2.5-Coder-32B-Instruct to dynamically generate reconnaissance and data theft commands. The SesameOp backdoor (Microsoft DART, July 2025) abuses OpenAI's Assistant API feature as a covert C2 server, avoiding attacker-hosted infrastructure entirely. Akamai Hunt identified a separate malware strain sending Base64-encoded C2 commands to /v1/chat/completions endpoints. Splunk has published 14 specific detection rules for LAMEHUG indicators. Detection should monitor for unauthorized outbound calls to AI API endpoints, DNS queries to AI service domains from unexpected processes, and entropy anomalies in request/response payloads.
- **AI Agent Context Poisoning & Memory Manipulation** -- Added to MITRE ATLAS in October 2025 (collaboration with Zenity Labs), these techniques target agentic AI systems. Context poisoning manipulates an agent's LLM context to persistently influence responses or actions. Memory manipulation alters long-term memory to ensure malicious changes persist across sessions. As of February 2026, Microsoft documented real-world memory poisoning via "Summarize with AI" buttons, identifying over 50 unique injection prompts from 31 companies embedding hidden instructions into AI assistant memory via URL prompt parameters.
- **TokenBreak Attacks** -- A newer class of adversarial attack targeting the tokenization layer rather than the prompt itself, manipulating how input text is broken into tokens to bypass content classifiers. This technique exploits an often-overlooked component in NLP pipelines and requires detection at the pre-tokenization stage.
- **Policy Simulation Attacks** -- Trick the model into believing unsafe outputs are allowed under updated or alternate policies, using prompts that mimic policy file structures (XML, JSON, INI). Roleplay-variant prompt injections achieved the highest attack success rate (89.6%) in 2025 red-team evaluations by deflecting responsibility away from the model.
- **Autonomous Reasoning-Model Jailbreaks** -- Published in Nature Communications (2026), research by Hagendorff et al. demonstrates that large reasoning models (DeepSeek-R1, Gemini 2.5 Flash, Grok 3 Mini, Qwen3 235B) can autonomously plan and execute multi-turn jailbreaks against other models with a 97.14% success rate, requiring no human guidance beyond an initial system prompt. This creates an alignment regression feedback loop: as reasoning models improve, they become more effective at subverting alignment in target models, making automated detection (13.2.4, 13.2.8) essential.
- **Unicode and Encoding Evasion** -- An empirical study (arxiv:2504.11168, 2025) tested character injection attacks against major guardrail products (Azure Prompt Shield, Meta Prompt Guard, Nvidia NeMo Guard, ProtectAI). Emoji smuggling achieved 100% evasion success for both jailbreaks and prompt injections; bidirectional text reached 99.23%. These techniques are black-box, requiring no knowledge of the guardrail internals, and highlight the need for input normalization before classification.
- **Indirect Injection as the Dominant Enterprise Threat** -- In the February 2026 Claude Opus 4.6 system card, Anthropic dropped the direct prompt injection metric entirely, arguing that indirect injection is now the relevant enterprise threat. Published numbers: a constrained coding environment held at 0% attack success across 200 attempts, while a GUI-based agent with extended thinking reached 17.8% at 1 attempt, climbing to 78.6% at 200 attempts without safeguards and 57.1% with safeguards. The shift matters for detection program design -- investments in per-message direct-injection classifiers yield diminishing returns compared to content scanning on untrusted inputs (emails, documents, web pages, tool output) before they enter the model's context window. The Google + Forcepoint April 24, 2026 joint reporting confirmed a 32% relative jump in malicious indirect prompt injection between November 2025 and February 2026, with payloads targeting search manipulation, API key exfiltration, file destruction, and embedded payment-redirect fraud (PayPal transactions, Stripe donation redirects). Obfuscation observed in the wild includes pixel-sized text, color-drained content, HTML metadata, and comment sections invisible to humans but visible to AI ingestion.
- **EchoGram Flip-Token Bypass** -- HiddenLayer published EchoGram in November 2025: an attack on guardrails (LLM-as-a-judge and classifier-based) that finds short "flip tokens" through dataset distillation and vocabulary probing, then appends them to malicious prompts to reverse the safety verdict. Published demonstrations include verdict flips on Qwen3Guard across multiple severity levels, with token combinations amplifying the effect. EchoGram exposes a structural fragility: guardrail training data has token-level imbalances that can be enumerated. Detection signals worth monitoring per 13.2.4 include false-positive flood patterns (an attacker probing for flip tokens) and unusual single-character or short-token suffix patterns appended to otherwise normal prompts.
- **Multimodal Symbolic / Rebus Injection** -- The NVIDIA AI Red Team (July 2025) demonstrated a class of "semantic" prompt injections that work entirely through visual symbol sequences -- a printer + waving figure + globe emoji-like arrangement parses as "print Hello, World" inside an early-fusion multimodal model (Llama 4 architecture, OpenAI o-series, Gemini), with similar arrangements coercing file deletion or shell-style operations without any explicit text. Cross-modal attacks evade text-only signature and classifier defenses because the adversarial signal lives in the shared latent representation. NVIDIA's recommendation is to push detection from input filtering to output-centric review, including adaptive output filters before code execution and runtime monitoring on tool calls.
- **Comment and Control Attack on Coding Agents** -- April 2026 research by Aonan Guan with Johns Hopkins University collaborators demonstrated working prompt injection against Anthropic's Claude Code Security Review, Google's Gemini CLI Action, and GitHub Copilot Agent via crafted GitHub comments, PR titles, and issue bodies. The attack hijacked the agents on GitHub Actions runners and recovered an API key in published proofs of concept. Detection programs should treat code-review surfaces (comments, titles, descriptions) as untrusted input streams equivalent to user prompts, with telemetry routed through 13.2.6 SIEM rules.
- **MCP Channel Compromise** -- January 2026 disclosed three CVEs (CVE-2025-68143/68144/68145) in Anthropic's official MCP Git server: two path traversal and one argument injection flaws exploitable via prompt injection in README files and poisoned issue descriptions. Separately, a public GitHub issue was shown to hijack a coding agent, pull data from private repos via an over-privileged PAT, and exfiltrate it into a public pull request. A systematic analysis of 78 studies (Jan 2026) reported adaptive attack success rates exceeding 85% against Claude Code, GitHub Copilot, and Cursor. Detection for 13.2.6 and 13.2.8 should treat MCP tool invocations as first-class telemetry, not sidecar traffic.
- **Industrial-Scale AI-Enhanced Phishing (Tycoon2FA / Storm-1747)** -- Microsoft's Digital Crimes Unit disrupted the Tycoon2FA phishing-as-a-service platform in March 2026, seizing 330 domains with Europol. The platform generated tens of millions of phishing emails monthly and was linked to nearly 100,000 compromised organizations since 2023, accounting for ~62% of phishing attempts Microsoft blocked at peak. AI-enhanced AiTM session-token interception combined with AI-customized message generation reportedly achieves 54% click-through rates versus ~12% for traditional campaigns. While outbound phishing detection lives outside the AI system itself, 13.2.4 rule libraries should treat AI-generated social-engineering payloads (forged corporate communication, AI-written spear-phishing) reaching enterprise AI assistants as a distinct threat class that may bypass legacy email defenses.
- **Third-Party AI Chatbot Plugin Compromise (IEEE S&P 2026)** -- "When AI Meets the Web: Prompt Injection Risks in Third-Party AI Chatbot Plugins" (accepted to IEEE Symposium on Security and Privacy 2026, arxiv:2511.05797) audited the plugin ecosystem and found 8 plugins used by ~8,000 websites transmitted browser message history to backend LLMs without integrity checks, allowing direct injection of forged conversation turns, and 15 plugins supported automated web scraping for RAG that ingested both first-party and third-party content indiscriminately, creating a turnkey indirect prompt injection surface. Detection for 13.2.4 should add allowlists for plugin-origin message payloads and treat plugin-fetched content as untrusted RAG input subject to the same scanning as user-supplied documents.

### Enterprise Detection Maturity (Q1-Q2 2026 Baselines)

Two industry data points help calibrate where most detection programs actually sit, versus where the requirements above target:

- **VentureBeat enterprise survey (April 2026)** -- 88% of surveyed enterprises reported AI agent security incidents in the prior 12 months, but only 21% had runtime visibility into what their agents were doing, only 47% had observed agents exhibiting unintended behavior, and just 5% felt confident they could contain a compromised agent. The "stage 3" threat framing (post-compromise agent abuse with privileged tool access) is where most programs are weakest -- detection rules covering 13.2.4 and 13.2.6 generally lag behind 13.2.1 signature deployment.
- **CrowdStrike 2026 Global Threat Report (Feb 26 2026)** -- 90+ organizations with prompt-injection abuse documented in 2025, 89% YoY growth in AI-enabled adversary operations, and 82% of all detections involving malware-free intrusions through legitimate pathways. The takeaway for 13.2 detection design: pattern-of-life and behavioral telemetry on legitimate AI tool usage now matters as much as content classifiers, because the abuse path is "valid prompt + valid tool + adversarial intent" rather than novel malware delivery.

These baselines suggest that for most enterprises the highest-yield investment in 2026 is shifting telemetry from prompt content alone to **agent action telemetry** (which tools were invoked, with what arguments, against what data, by which credentialed identity), then writing 13.2.4/13.2.6 rules over that data plane.

### Detection Architecture

A layered detection architecture addresses the full spectrum of requirements:

1. **Layer 1 -- Signature-Based (13.2.1):** Pattern matching against known jailbreak databases (JailbreakBench, PromptInject). Fast, low false-positive, but easily evaded by novel attacks. Signature databases require continuous updates.
2. **Layer 2 -- Classifier-Based (13.2.4):** ML classifiers trained on adversarial input datasets detect prompt injection and jailbreaks that evade signatures. Azure Prompt Shields and similar products operate at this layer.
3. **Layer 3 -- Behavioral Anomaly (13.2.3, 13.2.5):** Statistical anomaly detection on session-level features (retry rates, token consumption patterns, input diversity scores, query timing). Catches automated attack tools and probing behavior.
4. **Layer 4 -- Session Trajectory (13.2.8):** Multi-turn conversation analysis using stateful architectures. DeepContext (Feb 2026) demonstrated that RNN-based hidden state propagation over turn embeddings achieves 0.84 F1 at sub-20ms latency, outperforming stateless approaches like Llama-Prompt-Guard-2 (0.67 F1). Detects crescendo attacks, progressive manipulation, and autonomous reasoning-model-driven jailbreaks.
5. **Layer 5 -- Cross-Session Correlation (13.2.6):** SIEM rules correlating events across users, IPs, and time windows to detect coordinated campaigns and model extraction attempts.

### SIEM Integration Patterns

For requirement 13.2.2, the most maintainable integration approach in 2025-2026 uses OpenTelemetry pipelines:

- **Collection:** OpenTelemetry SDK emits GenAI spans with security event attributes
- **Processing:** OpenTelemetry Collector or Vector/Fluentd normalizes and enriches events
- **Routing:** OTLP export to SIEM (Splunk, Elastic, Sentinel) alongside observability backends
- **Detection:** Custom SIEM rules augmented with AI-specific detection logic (13.2.6)

As of March 16 2026, OCSF v1.8.0 ships native AI support: the `ai_operation` profile with `ai_model` and `message_context` objects provides vendor-neutral schema coverage for AI workloads including token usage metrics and role-based interaction tracking. v1.8.0 also added new `privilege_info`, `privilege_attack_info`, and `service_privilege_analysis` objects with MITRE ATT&CK technique mapping, enabling unused-privilege detection on AI service identities -- directly relevant to detecting compromised model API credentials per 13.2.5. OCSF maintainers announced collaboration with the OpenTelemetry maintainers in early 2026 to unify security and observability domains. ITU member nations formally supported OCSF for ratification as an international standard in December 2025, with **final ratification slated for June 2026**, after which schema extensions in private SIEM/XDR pipelines should align with the ratified vocabulary to avoid migration debt. CrowdStrike Falcon AIDR natively streams AI detection findings to Falcon Next-Gen SIEM; Splunk has published detection rules for LLM-abusing malware. Microsoft Incident Response published structured playbooks for prompt abuse detection and response in March 2026, emphasizing defense-in-depth combining monitoring, analysis, and documented response protocols. Organizations adopting OCSF-aligned pipelines can reduce custom parsing overhead significantly.

### MITRE ATLAS Coverage (Updated through v5.5.0, March 30 2026)

MITRE ATLAS has shipped four releases relevant to abuse detection between October 2025 and March 2026:

- **v5.1.0 (Nov 2025)** -- 16 tactics, 84 techniques, 56 sub-techniques, 32 mitigations, 42 real-world case studies; consolidates the October 2025 Zenity Labs collaboration that introduced 14 agent-focused techniques.
- **v5.3.0 (Jan 2026)** -- Three MCP-focused case studies: MCP server compromise, indirect prompt injection via MCP channels, malicious AI agent deployment.
- **v5.4.0 (Feb 5 2026)** -- 7 new agent-focused techniques including **Publish Poisoned AI Agent Tool** and **Escape to Host**, plus 4 case studies including ClawdBot vulnerabilities and OpenClaw remote code execution scenarios. The companion CTID OpenClaw investigation (February 9, 2026) mapped real OpenClaw exploitation paths to ATLAS TTPs and identified chokepoint techniques adversaries rely on, including direct and indirect LLM prompt injection, AI agent tool invocation, and Modify Agentic Configuration.
- **v5.5.0 (Mar 30 2026)** -- 12 new techniques including **AI Agent Tool Poisoning**, **Machine Compromise**, and **Cost Harvesting** subtechniques, plus 5 new case studies including LLMSmith (RCE vulnerabilities in LLM-integrated applications) and model distillation campaigns.

Key earlier additions still relevant to abuse detection:

- **AI Agent Context Poisoning (AML.T0057)** -- Manipulating context to persistently influence agent responses or actions
- **Memory Manipulation (AML.T0058)** -- Altering long-term LLM memory to persist malicious changes across sessions
- **Thread Injection** -- Introducing malicious instructions into a specific chat thread to change behavior for the duration of that conversation
- **Modify AI Agent Configuration** -- Persisting adversary-controlled settings in agent configuration stores so that malicious behavior survives restart
- **AI Service API (AML.T0096)** -- Using legitimate AI service APIs (OpenAI Assistant API, Hugging Face Inference API, MCP endpoints) as covert C2 channels, demonstrated in the wild by LAMEHUG and SesameOp

Detection rules (13.2.6) should map to ATLAS techniques, and analysts should use ATLAS indicators to draft SOC detection hypotheses that SIEM/XDR platforms can exercise; published mappings show roughly 70% of ATLAS mitigations line up with existing enterprise security controls, so much of the work is instrumenting and labeling rather than inventing from scratch. The SAFE-AI framework maps ATLAS threats to NIST SP 800-53 controls at each system element (Environment, AI Platform/Tools, AI Models, AI Data), providing a structured approach to control selection. NIST IR 8596 (Cybersecurity Framework Profile for AI, 2025) provides complementary guidance on managing cybersecurity challenges specific to AI systems.

### MITRE ATT&CK v19 (April 28 2026, shipped)

ATT&CK v19 landed on schedule and reshapes how AI-related TTPs are organized for SOC detection programs:

- **Defense Evasion split** -- The legacy Defense Evasion tactic was divided into **Stealth (TA0005, retained ID, "behavioral camouflage")** and **Defense Impairment (TA0112, new ID, "actively breaking your defenses")**. T1562 (Impair Defenses) sub-techniques were restructured, with T1562.001 and T1562.006 merging into the new **T1685: Disable or Modify Tools**. SIEM rule libraries with ATLAS->ATT&CK cross-mappings need regression testing on the migration -- legacy TA0005 references to AI-specific evasion (jailbreak suffix injection, output filter bypass, guardrail vocabulary attacks like EchoGram) should split between Stealth (T1685, evasion-by-obfuscation) and Defense Impairment (rules where the adversary is actively turning a guardrail off rather than slipping past it).
- **New AI techniques** -- Three top-level entries directly relevant to 13.2.4 detection: **T1682 Query Public AI Services** (reconnaissance/planning at scale, where adversaries call commercial LLM APIs to generate attack tooling), **T1683 Generate Content** with sub-techniques T1683.001 (written content, e.g. phishing emails) and T1683.002 (audio-visual content, e.g. deepfakes), and **T1684 Social Engineering** as a new parent rolling up impersonation and email spoofing. Detection programs should add rules for outbound calls to commercial LLM APIs from non-application processes, and for content-generation patterns originating from compromised internal hosts.
- **Campaign C0062 -- Anthropic AI-orchestrated Campaign** -- First named ATT&CK campaign anchored in autonomous AI use. Documents GTG-1002, a PRC-aligned cluster that used Claude Code agents and Model Context Protocol tools to automate ~80–90% of tactical operations across roughly 30 targets in technology, financial, chemical, and government sectors starting September 2025. Operators bypassed AI guardrails through fabricated "defensive cybersecurity testing" personas, then tasked agents with reconnaissance, SSRF exploitation, lateral movement, credential harvesting, and exfiltration via web services. Anthropic's published mitigation expanded its cyber-focused classifiers and account-detection telemetry. For 13.2.4 detection, the takeaway is that abuse patterns must now span account-level behavioral signals (sustained agent throughput at "physically impossible request rates" against multiple targets) and prompt-content classifiers, not either in isolation.
- **Software entry S9035 -- LAMEHUG** -- First malware family formally tracked in ATT&CK that queries an LLM in live operations. Tied to APT28 (G0007). Validates the LLM-as-C2 detection scope already documented under 13.2.8.

The April 2026 ATT&CK release also added 36 new software entries, two new threat groups (MirrorFace, VOID MANTICORE), and Mobile Detection Strategies. Enterprise gained Detection Strategies for the new AI-related techniques and Defense Impairment family, providing concrete starting telemetry that SOC teams can adapt instead of writing from scratch.

### The Lethal Trifecta as a Detection-Triage Lens

Simon Willison's "lethal trifecta" framing (private data access + exposure to untrusted content + an exfiltration vector) gained traction across 2026 as a structural model for prioritizing 13.2.4 detection coverage. Between January 7-15 2026, four major AI-powered productivity tools had public indirect-prompt-injection disclosures that all reduced to the same trifecta pattern. Constellation, HiddenLayer, and Promptfoo each published trifecta-specific test harnesses through Q1 2026.

For SOC teams the practical implications are:

- **Triage before detection.** The trifecta is an architectural property -- audit each agent deployment for which "leg" is unconstrained, then weight 13.2.4/13.2.6 detection priority by how many legs are present. Agents with all three legs cannot be reliably defended through detection alone; structural defenses (taint tracking, capability gating, exfiltration egress controls) reduce blast radius before per-message classifiers ever fire.
- **Filter ceiling.** Publicly reported guardrails reach roughly 97% accuracy on known patterns, but attacker iteration cycles compress that gap rapidly -- detection should be paired with hard policy gates on the exfiltration leg (e.g., outbound URL allowlists, image-rendering restrictions to block image-tag exfiltration, CSP on agent-rendered surfaces).
- **MCP and tool inventory.** Audit MCP server configurations and tool descriptions for embedded instructions before they reach the agent context window; this directly feeds 13.2.4 detection-rule scope and 13.2.8 covert-channel monitoring.

### Audio-Modality Jailbreak Coverage (JALMBench, ICLR 2026)

JALMBench (ICLR 2026, [openreview.net/forum?id=DJkQ236C8B](https://openreview.net/forum?id=DJkQ236C8B)) is the first large-scale benchmark for jailbreak vulnerabilities in audio language models (LALMs): 11,316 text samples, 245,355 audio samples (>1,000 hours) covering 12 mainstream LALMs, 8 attack methods (4 text-transferred, 4 audio-originated), and 5 defenses. Findings relevant to 13.2.1 and 13.2.4 coverage:

- Audio-originated attacks (direct audio manipulation) remain effective even when text-based safety alignment partially transfers to audio inputs.
- Existing general-purpose moderation methods only marginally improve security; defenses adapted from vision-language models are largely ineffective.
- Interleaved audio-text strategies provide stronger cross-modal generalization than single-modality defenses.

Detection programs that ingest audio (voice agents, transcription pipelines, multimodal copilots) should add JALMBench-derived test cases to the same regression suite that runs JailbreakBench and AgentDojo, and should not assume that text-side guardrails extend automatically to audio.

### Benchmarks for Measuring Detection Effectiveness

Abuse detection claims should be backed by repeatable benchmark runs, not vendor marketing. Useful public evaluation suites as of April 2026:

- **AgentDojo (NeurIPS 2024, active through 2026)** -- 97 realistic agent tasks (email client, e-banking, travel booking) with 629 security test cases and attack/defense paradigms from the literature. Baseline GPT-4o achieves 69% benign utility, dropping to 45% under attack; targeted "Important message" attacks succeed 53.1% of the time on an undefended baseline. Prompt sandwiching improves utility (65.7%) but leaves attack success at 30.8%; tool filtering drops attack success to 7.5% at the cost of utility (53.3%). US and UK AISI used AgentDojo to document Claude 3.5 Sonnet (new) vulnerabilities. As of ICLR 2026, **PromptArmor** -- a lightweight LLM preprocessor that detects and excises injected prompts using fuzzy regex matching -- achieves <1% false-positive and <1% false-negative rates on AgentDojo, with attack success near 0% (0–0.47%) while preserving up to 76.35% utility on o4-mini and 72.02% on GPT-4.1. The trade-off is roughly 200-600ms added latency and one extra LLM call per user input. AgentSys reports 0.78% / 4.25% attack success on AgentDojo with marginal benign utility improvement. Use this suite to validate 13.2.6 detection rules and 13.2.8 session trajectory analysis.
- **JailbreakBench, PromptInject** -- Still the standard corpora for 13.2.1 signature-based testing; require refresh against BoN and Unicode-evasion variants to stay relevant.
- **Comparative Evaluation of LLM Security Detection Systems (arxiv:2506.19109, Jun 2025)** -- Published FPR/recall numbers for Vigil, Rebuff, and LLM Guard; use as a baseline when procuring or swapping detection tooling.

Publish benchmark deltas alongside each SIEM rule rollout so detection coverage is measurable rather than asserted.
