# C1.5: Data Lineage and Traceability

> [Back to C01 Index](C01-Training-Data.md)
> **Last Researched:** 2026-04-15 (refreshed)

## Purpose

When a model produces unexpected or harmful outputs, the first question in incident response is "what data trained this model, and where did it come from?" Without complete lineage records — tracking every transformation, augmentation, merge, and synthetic data generation step — that question is unanswerable. Lineage is also a regulatory survival requirement: the FTC's algorithmic disgorgement pattern (enforcement actions against Everalbum in 2021, Weight Watchers in 2022, Ring and Edmodo in 2023, and Rite Aid and Avast in 2024 — all requiring deletion of models trained on improperly collected data) means organizations that cannot reconstruct a dataset's full journey from source to model input face existential regulatory risk. As the FTC has made clear, if you cannot trace where everything came from and what transformations were applied, the enforcement agency is unlikely to be charitable.

This section requires recording the full provenance chain of each dataset, storing lineage records immutably for audit, and extending lineage tracking to cover synthetic data so that it remains clearly distinguishable from real data throughout the pipeline. As organizations increasingly generate synthetic training data from other models, the risk of inheriting and amplifying upstream poisoning (demonstrated by the Virus Infection Attack and the January 2026 "Basilisk Venom" incident where hidden prompts in GitHub code comments poisoned a fine-tuned model) makes synthetic data lineage tracking especially urgent. The March 2026 LiteLLM supply chain attack — where compromised PyPI packages were live for ~40 minutes before detection — reinforced how quickly tainted dependencies can enter ML pipelines and why lineage must capture dependency provenance, not just data provenance.

EU AI Act Article 10 (Data and Data Governance), entering enforcement on 2 August 2026, will make data lineage a hard compliance requirement for all high-risk AI systems. With the deadline now roughly four months away, organizations deploying high-risk AI face fines up to €35 million or 7% of global revenue for non-compliance — making lineage infrastructure an urgent operational priority, not a future aspiration. ISO/IEC 42001 Annex A.7.5 (Data Provenance) independently requires continuous, automated provenance capture — including origin, consent, transformation logs with script versions and operator IDs, and tamper-resistant storage — making it a useful operational checklist even for organizations not yet subject to the EU AI Act. NIST AI 600-1 (GenAI Profile) similarly identifies provenance as a core risk management action mapped across the Govern, Map, Measure, and Manage functions of the AI RMF.

| 1.5.2 | **Verify that** lineage records are immutable, securely stored, and accessible for audits. | 2 | Tampering with lineage records to cover tracks after data poisoning; loss of audit trail due to storage failure; disputed provenance during regulatory investigation. FTC algorithmic disgorgement actions (6 cases 2021-2024) have required model deletion when provenance could not be demonstrated — immutable records are the defense against this. With the EU AI Act enforcement deadline now under four months away, organizations that cannot produce immutable audit trails face fines up to €35M or 7% of global revenue. | Verify lineage metadata is stored in append-only or immutable storage (S3 Object Lock in compliance mode, Azure Immutable Blobs, GCS retention policies). Check access controls on lineage databases — pipeline operators should not have write or delete access to lineage stores. Test that lineage records cannot be modified or deleted via API or console. Verify backup and retention policies align with EU AI Act Article 10 documentation requirements (enforcement begins August 2026) and [ISO 42001 Annex A.7.5](https://www.isms.online/iso-42001/annex-a-controls/a-7-data-for-ai-systems/a-7-5-data-provenance/) (which requires tamper-resistant storage with point-in-time evidence and exportable audit records). Audit log retention should cover the full expected model lifetime plus regulatory holding periods. For cryptographic integrity, verify that lineage events carry SHA-256 content hashes computed over canonical event payloads — this enables post-hoc tamper detection without modifying stored records. Consider an [OpenTelemetry-based immutable audit pipeline](https://oneuptime.com/blog/post/2026-02-06-immutable-audit-log-pipeline-otel/view) pattern: instrument lineage events via OTel SDK, process through OTel Collector with integrity hashing, and dual-export to both immutable storage (S3 Object Lock) and a queryable backend (Elasticsearch) for search. | Immutable storage backends are mature (S3 Object Lock, Azure Immutable Blobs, WORM-compliant storage). Gap: lineage metadata systems (Atlas, DataHub, Marquez) typically use mutable databases (MySQL, PostgreSQL, Elasticsearch) and do not enforce immutability at the storage layer by default. Achieving true immutability requires architectural layering — for example, writing lineage events to an append-only log (Kafka topic with compaction disabled, or a dedicated ledger) and using the lineage UI as a read-only projection. The OpenTelemetry-based pattern (Feb 2026) provides a practical reference architecture: OTel SDK instrumentation → OTel Collector with SHA-256 hashing → dual export to S3 Object Lock (immutable) + Elasticsearch (queryable), satisfying SOC 2, HIPAA, and EU AI Act audit requirements. Delta Lake and Apache Iceberg provide table-level time-travel and snapshot lineage as a complementary layer — Iceberg's snapshot history is queryable and supports point-in-time audits without redundant data copies. No major open-source lineage tool ships with built-in immutability as of April 2026, so the architectural layering approach remains necessary. |
| 1.5.3 | **Verify that** lineage tracking covers synthetic data generated via augmentation, synthesis, or privacy-preserving techniques. | 2 | Synthetic data contamination obscuring real data quality issues; inability to trace model behaviors to real vs. synthetic training examples; privacy violations if synthetic data retains real-data characteristics without proper tracking. The VIA (Virus Infection Attack) demonstrated how poisoned content propagates through synthetic data pipelines, making lineage of synthetic generation steps critical for containment. Model Autophagy Disorder (MAD) — where models trained recursively on their own synthetic outputs degrade — is another risk that lineage tracking helps detect. LoRA adapter poisoning (emerging 2025-2026) adds a new vector: malicious fine-tuning adapters are small, easy to distribute, and can introduce backdoors indistinguishable from legitimate fine-tuning — lineage must capture adapter provenance alongside data provenance. | Review lineage records for synthetic data generation steps. Verify each synthetic dataset records: the generator model/tool identity and version, the seed/source data used, generation parameters and prompts, and a timestamp. Verify synthetic data carries metadata tags distinguishing it from real data. Check that downstream consumers can filter by data origin type. Confirm lineage survives merge operations — when synthetic and real datasets are combined, per-record origin tags should persist. For media-type synthetic data, verify [C2PA Content Credentials](https://c2pa.org/) (v2.2, May 2025) are attached to generated assets — the spec now covers AI/ML workflows explicitly. For LoRA and adapter artifacts, verify that SLSA provenance attestations record the base model, training data snapshot, and hyperparameters used to produce each adapter. | Synthetic data generation tools (Gretel, MOSTLY AI, SDV, and LLM-based generators) produce data but rarely integrate with lineage systems. Gap: no widely adopted standard metadata schema for marking data as synthetic exists as of April 2026 — the IPTC Photo Metadata Standard 2025.1 (ratified November 2025) introduced four XMP fields for AI-generated media (`AISystemUsed`, `AISystemVersionUsed`, `AIPromptInformation`, `AIPromptWriterName` in the `Iptc4xmpExt` namespace), but nothing equivalent exists for tabular or text training data. C2PA Content Credentials (v2.2, May 2025) provide cryptographic tamper-evident provenance for media content and now explicitly cover AI/ML workflows, but adoption in ML training pipelines is still nascent. When synthetic and real data are merged in dataframes or parquet files, distinguishability is typically lost unless explicit column-level origin tags are maintained. OpenDataDiscovery (ODD) is one of the few platforms that gives ML workflows (models, experiments, feature stores) first-class lineage support, though its community is smaller than OpenLineage's. |
| **1.5.4** | **Verify that** synthetic data is clearly labeled and distinguishable from real data throughout the pipeline. | 2 | Unlabeled synthetic data mixed with real data creates compounding risks: model quality degradation from recursive training on synthetic outputs (Model Autophagy Disorder); inability to isolate contaminated synthetic batches during incident response; regulatory non-compliance when auditors cannot determine what fraction of training data was real vs. generated; and inherited bias amplification when synthetic data silently inherits and magnifies biases from the source model. The early 2026 "Grok 4" incident, where social media training data saturation created a universal backdoor trigger, illustrates how unlabeled synthetic-origin data can introduce systemic vulnerabilities. Every major AI content regulation now converging in 2026 — EU AI Act Article 50, California SB 942, and IAB disclosure standards — shares a common thread: AI-generated content must carry machine-readable disclosure metadata. | Inspect dataset schemas for a mandatory origin-type field (e.g., `data_origin: "synthetic" | "real" | "augmented"`). Verify the field is populated for every record, not just at the dataset level. Test that merge and ETL operations preserve origin labels — sample records before and after joins, unions, and feature engineering steps. Verify downstream training scripts can filter by origin type and that training logs record the synthetic-to-real ratio per batch. Check that data catalogs (Atlan, Alation, Collibra) surface origin-type metadata in search and discovery views. For media training data, verify [C2PA Content Credentials](https://c2pa.org/) (v2.2) manifests are preserved through the pipeline — the JSON-LD documents with cryptographic signatures create tamper-evident provenance records. Verify IPTC 2025.1 XMP fields (`AISystemUsed`, `AISystemVersionUsed`, `AIPromptInformation`, `AIPromptWriterName`) are populated at generation time and survive downstream processing for image/media assets. | This is one of the most significant gaps in the current tooling landscape. No widely adopted standard schema for synthetic data labeling exists as of April 2026. Practical approaches include: (1) adding a `data_origin` column to all datasets at creation time, enforced by pipeline validation checks; (2) using metadata sidecar files (JSON/YAML) that travel with datasets through storage and compute layers; (3) leveraging Databricks Unity Catalog tags or Delta Lake table properties to mark synthetic tables; (4) using Apache Iceberg's snapshot lineage or Delta Lake's JSON change logs for table-level provenance that supports "as of" queries for any historical state. The [World Privacy Forum's 2025 technical review](https://worldprivacyforum.org/posts/privacy-identity-and-trust-in-c2pa/) of C2PA raises important privacy and identity concerns with the framework that organizations should evaluate before adoption. |

## Implementation Guidance

| Tool | Scope | Strengths | Limitations |
|------|-------|-----------|-------------|

| [Apache Atlas](https://atlas.apache.org/) v2.4.0+ | Hadoop-centric governance | Deep Hadoop lineage; graph-based metadata | Slower release cadence; CVE-2024-46910 (XSS) in v2.3.0 and earlier requires upgrade |

### Immutable Storage Architecture

For organizations that need to satisfy both EU AI Act Article 10 documentation requirements and FTC algorithmic disgorgement defense, the recommended layered approach is:

1. **Event capture**: Instrument pipeline steps with OpenLineage or OpenTelemetry SDK to emit structured lineage events with SHA-256 content hashes
2. **Immutable persistence**: Write events to S3 Object Lock (compliance mode), Azure Immutable Blobs, or GCS with retention policies — pipeline operators must not have write/delete access
3. **Queryable projection**: Dual-export to Elasticsearch or a graph database for search and visualization
4. **Table-level snapshots**: Enable Delta Lake or Iceberg time-travel on all training data tables for point-in-time reconstruction
5. **Attestation**: Generate SLSA v1.1+ provenance attestations for model artifacts and training data snapshots

### Synthetic Data Labeling Strategies

Until a universal standard emerges for non-media synthetic data labeling, practical approaches include:

- **Column-level tagging**: Add a mandatory `data_origin` field (`synthetic | real | augmented`) to every record at creation time, enforced by pipeline validation
- **Metadata sidecars**: JSON/YAML files that travel with datasets through storage and compute
- **Table properties**: Databricks Unity Catalog tags or Delta Lake table properties for dataset-level markers
- **Media assets**: Apply C2PA v2.2 Content Credentials + IPTC 2025.1 XMP fields at generation time; verify preservation through pipeline
- **Batch-level tracking**: Log synthetic-to-real ratios per training batch in experiment tracking systems (MLflow, W&B)

### Poisoning Detection Techniques Enabled by Lineage

When lineage is comprehensive, incident response teams can apply targeted detection at specific pipeline stages:

- **Pre-training**: Per-class feature drift monitoring, label entropy checks, cosine similarity baselines against known-good data distributions
- **During training**: Gradient norm outlier detection (flags samples exerting outsized influence on weights), per-example influence function estimation, suspicious performance gains on narrow data slices
- **Post-training**: Backdoor activation probes with targeted trigger patterns, red-team prompt variants testing semantic preservation across formatting changes
- **Supply chain**: Transparency log monitoring via sigstore for unauthorized model signing events, signed dataset manifests for immutable dataset versioning
