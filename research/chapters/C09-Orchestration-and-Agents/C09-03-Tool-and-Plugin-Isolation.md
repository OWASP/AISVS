# C9.3: Tool and Plugin Isolation and Safe Integration

[Back to C09 Index](C09-Orchestration-and-Agents.md)

## Purpose

Agents extend their capabilities through tools and plugins -- code execution, file access, API calls, database queries. Each tool represents an attack surface: a compromised or malicious tool can access the host system, exfiltrate data, or escalate privileges. This section requires that tools run in isolated sandboxes with least-privilege permissions, that their outputs are validated before being trusted, and that their supply chain integrity is verified.

## 2024-2026 Research Context

The explosion of tool-using AI agents -- particularly through the Model Context Protocol (MCP) -- has made tool isolation one of the most active areas of agentic security research. Real-world attacks in 2025 demonstrated that tool-based vectors are not theoretical: supply chain compromises, tool poisoning, and sandbox escapes have all been observed in production systems.

### MCP Tool Poisoning and Real-World Attacks

Tool Poisoning Attacks (TPA), catalogued as OWASP MCP03:2025, involve embedding malicious instructions in MCP tool descriptions that are invisible to users but processed by AI models. This is particularly dangerous because the full tool description is part of the model's context but is often not displayed in user-facing UIs. Attackers can craft tool metadata that causes agents to exfiltrate data, execute unauthorized commands, or steal credentials without user awareness.

CyberArk's research (published early 2026) expanded the attack surface further with **Full-Schema Poisoning (FSP)**: the entire tool schema -- not just the description field -- is a potential injection point. Parameter names, type definitions, required field arrays, default values, and even non-standard custom fields can carry malicious instructions. In one demonstration, a parameter named `content_from_reading_ssh_id_rsa` (with a clean description) successfully triggered credential exfiltration based solely on the identifier. CyberArk also documented **Advanced Tool Poisoning Attacks (ATPA)**, which exploit tool *outputs* rather than schemas: a tool returns a fake "error" requesting sensitive file contents, the LLM interprets it as a legitimate resolution step, and data is exfiltrated within what looks like normal retry logic. ATPA requires no schema modifications -- only internal tool logic changes -- making it particularly hard to detect with static analysis.

Systematic analysis of MCP servers (arXiv, 2025) found that 5.5% of surveyed servers exhibited tool poisoning characteristics, and 33% allowed unrestricted network access -- confirming that the ecosystem's default-open trust model is a structural weakness, not an edge case.

Real-world incidents in 2025 confirmed these risks:

- **CVE-2025-6514 (mcp-remote):** A command injection flaw that allowed a malicious MCP server to execute arbitrary code on connected clients by passing server-provided configuration data directly to the system shell, achieving full remote code execution.
- **CVE-2025-49596 (MCP Inspector):** A CSRF vulnerability in a popular MCP developer utility that enabled remote code execution simply by visiting a crafted webpage.
- **Postmark Incident (September 2025):** A supply chain compromise demonstrating operational impact through a compromised MCP tool distribution channel.
- **Smithery Supply Chain Attack (October 2025):** Affected 3,000+ hosted applications and their API tokens through a compromised tool registry.

As of April 2026, the pace of MCP vulnerability disclosure has accelerated dramatically. Between January and March 2026, security researchers filed over 30 CVEs targeting MCP servers, clients, and infrastructure -- ranging from path traversals to CVSS 9.6+ remote code execution flaws. Notable additions to the incident record:

- **CVE-2025-53109 / CVE-2025-53110 (Anthropic Filesystem MCP Server, CVSS 8.4/7.3):** Cymulate Research Labs discovered that Anthropic's own Filesystem MCP Server used naive prefix-matching for path validation (e.g., `/tmp/allowed_dir_evil` passed validation for `/tmp/allowed_dir`) and failed to resolve symlinks before access checks. Chained together, an attacker could escape the sandbox, read/write arbitrary files, and achieve code execution via macOS Launch Agents. Fixed in version 0.6.3/2025.7.1.
- **CVE-2025-54136 (Cursor Trust Bypass / MCPoison):** Once a user approved an MCP server configuration, Cursor never re-validated it -- allowing malicious logic to be injected silently after initial approval. This is the rug pull attack pattern at the IDE level.
- **CVE-2026-39421 (MaxKB ToolExecutor, CVSS 6.3):** A sandbox escape in MaxKB's Python ToolExecutor where the LD_PRELOAD-based sandbox intercepted standard functions (execve, system, connect) but failed to block `pkey_mprotect`. Attackers used Python's ctypes to execute raw kernel syscalls, bypassing the entire interception layer. Fixed in version 2.8.0.
- **CVE-2026-39888 (PraisonAI, CVSS 9.9):** A critical sandbox escape in PraisonAI's `execute_code()` function. The AST-based blocklist in subprocess mode contained only 11 attribute names (versus 30+ in direct mode), leaving the exception frame traversal chain (`__traceback__` -> `tb_frame` -> `f_back` -> `f_builtins`) unblocked. Attackers triggered a caught exception, traversed frames to the subprocess wrapper's real builtins dict, retrieved `exec`, and achieved arbitrary code execution. Fixed in version 1.5.115.
- **CVE-2025-53372 (node-code-sandbox-mcp, CVSS 7.5):** Command injection bypassing Docker sandbox protections via unsanitized `execSync` calls in the MCP server itself -- the sandbox was intact but the server process running *outside* the sandbox was vulnerable.
- **CVE-2026-0755 (gemini-mcp-tool, CVSS 9.8):** Unsanitized user input passed directly to shell commands with no authentication required -- a zero-day in a widely-used MCP tool.
- **CVE-2025-53967 (Framelink Figma MCP Server, CVSS 7.5):** A command injection in one of the most widely-deployed MCP servers (600,000+ downloads, 10,000+ GitHub stars). The `fetch-with-retry.ts` module fell back to executing `curl` via `child_process.exec` without sanitizing the URL -- an attacker could inject arbitrary system commands through Figma design file names, text layers, or component descriptions via prompt injection. The fix (v0.6.3) switched to `child_process.execFile` with argument arrays, avoiding shell interpretation entirely. This vulnerability is instructive because the attack chain crosses domains: a design artifact in Figma becomes a code execution vector on the developer's machine through the MCP server.
- **CVE-2025-68143 / CVE-2025-68144 / CVE-2025-68145 (Anthropic mcp-server-git, January 2026):** Three vulnerabilities in Anthropic's official Git MCP server. CVE-2025-68143 and CVE-2025-68145 allowed the `git_init` tool to create repositories at arbitrary filesystem paths because configured repository boundaries were never validated. CVE-2025-68144 passed user-controlled arguments directly to the Git CLI without sanitization, enabling argument injection. These are notable because they affect a first-party server from the MCP protocol's creator -- illustrating that even well-resourced maintainers ship classical AppSec bugs (CWE-22, CWE-78) in MCP tooling.
- **CVE-2026-5023 (codebase-mcp, unpatched as of March 2026):** OS command injection in the `getCodebase`, `getRemoteCodebase`, and `saveCodebase` functions. User-supplied parameters are concatenated directly into command strings without escaping shell metacharacters. No official patch has been released despite the maintainers being notified.

A systematic survey found that 82% of 2,614 MCP implementations were vulnerable to path traversal, 43% of CVEs involved exec/shell injection, and 38-41% of servers lacked authentication entirely. Endor Labs' independent analysis found similar patterns: 67% of MCP implementations use sensitive APIs related to code injection (CWE-94) and 34% are exposed to command injection (CWE-78). Their core conclusion is that MCP servers inherit well-documented AppSec vulnerability classes -- command injection, SSRF, path traversal, missing authentication -- concentrated in AI infrastructure that processes untrusted LLM-generated inputs.

These incidents underscore that MCP's open ecosystem requires defense-in-depth: integrity verification (9.3.4), manifest-declared privileges (9.3.5), and automated containment (9.3.6) are not optional hardening but essential baseline controls.

### Rug Pull Attacks and Post-Approval Tool Mutation

A distinct class of supply chain attack gaining attention as of early 2026 is the **rug pull** -- a tool that behaves legitimately when first approved but is later modified to include malicious logic. The fundamental weakness is architectural: MCP implementations typically ask users to approve tool access at connection time based on the tool's current description and schema, but nothing in the base protocol prevents the server from changing that definition afterward. A malicious server can modify a tool's required parameters to include environment variables like `AWS_ACCESS_KEY_ID`, and the LLM treats this as a legitimate API constraint, extracting credentials from its context and passing them to the server. The server logs the stolen data while continuing to execute the original query normally, making detection difficult.

The OWASP MCP Security Cheat Sheet (published 2025) formalizes the defense: **tool definition pinning** using cryptographic hashes. Pin tool definitions at discovery time by computing SHA-256 over the canonical JSON of the tool name, description, and input schema. Before each execution, re-hash the current definition and compare against the pinned value -- any mismatch indicates post-deployment mutation. The open-source **mcp-scan** tool from Invariant Labs implements this approach, hashing descriptions on first scan and alerting on changes.

The **Enhanced Tool Definition Interface (ETDI)**, proposed in academic research (arXiv, 2025), takes this further by integrating cryptographic identity verification, immutable versioned tool definitions, and OAuth 2.0-based permission management directly into MCP. Tools are digitally signed by their providers; MCP clients verify signatures using provider public keys. Any definition change -- functionality, schema, permissions, or API contract hash -- mandates a new signed version requiring explicit user re-approval. While ETDI is not yet part of the core MCP specification, its principles (definition pinning, cryptographic signing, version immutability) can be applied today using existing tooling.

### Sandbox Escapes: From Theoretical to Production Impact

As of early 2026, sandbox escapes are no longer theoretical concerns -- they are actively exploited in production agent systems:

- **CVE-2026-25049 (n8n, CVSS 10.0):** Disclosed in February 2026, two critical sandbox escape vulnerabilities in n8n's JavaScript expression evaluator allowed authenticated users to execute arbitrary system commands and decrypt all stored credentials, including AI API keys and cloud provider tokens. The attack chained three flaws: template literal notation bypassing AST sanitizer checks, arrow functions circumventing `this` binding restrictions, and V8's `prepareStackTrace` hook providing stack frame access. The initial patch was bypassed within 48 hours using `Object.defineProperty()`. On n8n Cloud, this enabled multi-tenant data exposure and AI pipeline hijacking.
- **Snowflake Cortex CLI (March 2026):** A malicious instruction embedded in a GitHub repository README exploited Snowflake's Cortex Code CLI agent -- the agent fetched the README as part of its workflow, and the hidden instruction caused command execution outside the sandbox, bypassing the human approval gate entirely.
- **November 2025 runC vulnerabilities (CVE-2025-31133, CVE-2025-52565, CVE-2025-52881):** Three container escape flaws affecting Docker, Kubernetes, containerd, and CRI-O. These exploit symlink replacement of `/dev/null`, timing attacks during container initialization, and writes to critical system files. Requires runC 1.2.8+, 1.3.3+, or 1.4.0-rc.3+ to remediate.
- **CVE-2024-1086 (netfilter use-after-free):** A Linux kernel vulnerability confirmed by CISA in October 2025 as actively exploited in ransomware campaigns. Allows container escape via the kernel's netfilter subsystem.

The n8n platform was hit again in January 2026 with **CVE-2026-1470 (CVSS 9.9)** and **CVE-2026-0863 (CVSS 8.5)**, discovered by JFrog Security Research. CVE-2026-1470 exploits JavaScript's deprecated `with` statement to bypass the AST sanitizer -- by scoping a decoy `constructor` variable inside a `with` block, the attacker reaches `Function.prototype.constructor` without triggering the regex that blocks `.constructor` patterns, achieving RCE in n8n's main process. CVE-2026-0863 targets the Python Code Node in "Internal" mode, leveraging Python 3.10's new `AttributeError.obj` attribute to bypass the blocked `getattr` function and reach frame builtins via exception handling. Both vulnerabilities demonstrate a systemic problem: **language evolution invalidates static sandbox designs** -- new syntax features and runtime behaviors create bypass paths that existing AST sanitizers cannot anticipate.

AI development tools are also affected. **CVE-2026-25725 (Claude Code, CVSS 7.7)** disclosed in February 2026 demonstrated a sandbox escape via persistent configuration injection: the bubblewrap sandbox failed to protect `.claude/settings.json` when the file didn't exist at startup. While the parent directory was mounted writable and `settings.local.json` had read-only protections, `settings.json` lacked equivalent safeguards. Malicious code running inside the sandbox could create this missing file and inject persistent hooks (e.g., SessionStart commands) that would execute with host privileges when Claude Code restarted. This is classified as CWE-501 (Trust Boundary Violation) -- a reminder that sandbox configuration files themselves are part of the attack surface.

Cloud provider sandbox infrastructure has also proven vulnerable. Palo Alto Networks Unit 42 published research in April 2026 ("Cracks in the Bedrock") demonstrating a complete sandbox escape chain against **AWS Bedrock AgentCore's Code Interpreter**. The sandbox's "network isolation mode" -- marketed as providing complete isolation with no external access -- left DNS resolution functional, enabling data exfiltration via DNS tunneling through a covert bidirectional channel. More critically, the AgentCore Runtime's microVM Metadata Service (MMDS) lacked session token enforcement (using legacy MMDSv1), allowing SSRF-based extraction of IAM credentials from `http://169.254.169.254/latest/meta-data/iam/security-credentials/execution_role`. A follow-up disclosure ("Agent God Mode") revealed that AgentCore's starter toolkit auto-generates IAM roles with wildcard permissions across ECR, memory stores, Code Interpreter instances, and agent invocations -- meaning a single compromised agent could read every other agent's conversation history, download any container image, and invoke any agent in the account. AWS remediated by enforcing MMDSv2 as default for new agents as of February 14, 2026, and updated documentation to clarify that the default roles are intended for development only. This incident is particularly instructive because it shows that even when the sandbox itself (a Firecracker microVM) is sound, the surrounding infrastructure -- metadata services, IAM policies, DNS resolution -- can undermine the isolation guarantees.

Beyond workflow platforms, sandbox escapes are appearing across the AI agent ecosystem:

- **CVE-2026-22709 (vm2, CVSS 9.8):** A critical sandbox escape in the widely-used Node.js vm2 library, caused by V8 engine async stack trace optimizations in Node.js 24/25 that expose internal object states during exception handling. vm2 cannot sanitize these states, enabling unauthenticated remote code execution.
- **CVE-2026-27952 (Agenta-API, CVSS 8.8):** A Python sandbox escape in the Agenta LLMOps platform (disclosed February 2026). Attackers exploited improper allowlisting of `numpy` within RestrictedPython -- `numpy.ma.core.inspect` provided access to `sys.modules`, enabling arbitrary `os` module imports. Agenta v0.60+ removed RestrictedPython entirely in favor of alternative execution models.
- **CVE-2026-27002 (OpenClaw):** A configuration injection flaw in OpenClaw's Docker tool sandbox allowed attackers to specify dangerous Docker options (host networking, unconfined Seccomp/AppArmor profiles, sensitive bind mounts) through the sandbox configuration, enabling container escape and host compromise. The system validated data types but failed to block insecure values.

**CVE-2026-25520 (SandboxJS, CVSS 10.0)** disclosed in February 2026 showed yet another JavaScript sandbox escape -- `Object.values()` and `Object.entries()` return Arrays containing references to host objects without proper wrapping, enabling access to the host Function constructor via `Array.prototype.at()`. This is the same fundamental pattern as vm2's escape: JavaScript's object model makes it extremely difficult to completely isolate host objects from sandboxed code.

Additional sandbox escapes disclosed in April 2026 reinforce the pattern:

- **CVE-2026-39888 (PraisonAI, CVSS 9.9):** PraisonAI's `execute_code()` sandbox used an AST blocklist in subprocess mode that was a strict subset of the direct-execution blocklist (11 names vs. 30+). The omitted `__traceback__`, `tb_frame`, `f_back`, and `f_builtins` attributes formed a complete frame-traversal chain that allowed attackers to escape the sandbox through caught exceptions. This demonstrates a recurring antipattern: **inconsistent security boundaries between code paths** -- the same function had different blocklists depending on execution mode.
- **CVE-2026-34955 (PraisonAI SubprocessSandbox, CVSS 8.8):** A second PraisonAI sandbox escape, affecting all sandbox modes (BASIC, STRICT, NETWORK_ISOLATED). The SubprocessSandbox calls `subprocess.run()` with `shell=True` and relies on string-pattern matching to block dangerous commands, but omits `sh` and `bash` as standalone executables -- trivial bypass via `sh -c '<command>'`. Fixed in version 4.5.97. PraisonAI being hit twice in rapid succession with fundamentally different bypass mechanisms (frame traversal in `execute_code()` and shell injection in SubprocessSandbox) illustrates the whack-a-mole problem: fixing one sandbox component does not secure others.
- **CVE-2026-39421 (MaxKB, CVSS 6.3):** MaxKB's LD_PRELOAD sandbox intercepted standard libc functions but forgot to block `pkey_mprotect`, allowing attackers to use Python's ctypes for direct kernel syscalls. This is the **interposition bypass** antipattern: sandboxes that hook user-space functions can always be circumvented by going directly to the kernel.
- **CVE-2026-25904 (Pydantic-AI MCP Run Python):** An SSRF vulnerability where the Deno sandbox was configured with overly permissive network settings, allowing sandboxed Python code to access the host's localhost interface and reach internal services. The mcp-run-python project has since been archived without a fix -- illustrating the risk of sandbox components that become unmaintained while still deployed.

### Quantifying Escape Capability: SandboxEscapeBench

The UK AI Safety Institute and University of Oxford published **SandboxEscapeBench** in March 2026 (arXiv:2603.02277), the first systematic benchmark measuring whether frontier LLMs can autonomously escape container sandboxes. The benchmark defines 18 escape scenarios across five difficulty levels spanning orchestration misconfigurations, runtime vulnerabilities, and kernel exploits. Results are sobering: GPT-5 and Claude Opus 4.5 achieved ~40-49% success rates on level 3 (intermediate) scenarios, with the largest models reliably escaping common misconfigurations like exposed Docker sockets and privileged containers. No model succeeded on level 4-5 kernel exploits, but the cost per escape attempt is approximately $1 at current pricing, and success scales log-linearly with token budget -- meaning that as models become more capable and inference becomes cheaper, the attack economics will only improve. The benchmark is open-source, enabling defenders to track escape capability progression over time. Key takeaway: organizations should not rely on plain Docker isolation for containing capable agents, and deployment parameters like token budgets and retry policies can move a system across capability thresholds.

These incidents collectively demonstrate that shared-kernel container isolation is insufficient for untrusted AI agent code execution. The recurring n8n vulnerabilities are particularly instructive: even application-layer sandboxes (AST sanitizers, expression evaluators) face systematic bypass risks because language runtimes evolve faster than sanitizer rules can be updated. The Agenta, OpenClaw, PraisonAI, and MaxKB cases show that the problem extends beyond JavaScript -- Python sandboxing via RestrictedPython, AST blocklists, and LD_PRELOAD interposition all have fundamental limitations when facing adversarial inputs. The emerging consensus as of April 2026 is that application-layer sandboxing should be treated as defense-in-depth only, never as a primary isolation boundary.

### Sandboxing Strategies: A Tiered Approach

NVIDIA's 2025 practical security guidance for sandboxing agentic workflows establishes a clear hierarchy of isolation strength. As of March 2026, the industry consensus is converging: shared-kernel containers alone are not sufficient for untrusted AI agent code execution. The hierarchy, with updated performance data:

1. **Full Virtualization / MicroVMs (strongest):** Firecracker microVMs boot in ~125ms with less than 5 MiB memory overhead per VM, supporting up to 150 VMs per second per host -- making them practical for high-throughput agent workloads. Kata Containers (~200ms boot) provide the same hardware-level isolation with native Kubernetes integration. NVIDIA recommends running agentic tools "within a fully virtualized environment isolated from the host kernel at all times." An attacker must escape both the guest kernel and the hypervisor, which is a fundamentally harder problem than container escape.
2. **User-Space Kernel Mediation:** gVisor intercepts system calls via a separate user-space kernel (the Sentry process), drastically reducing the host kernel attack surface. Performance overhead is 10-30% on I/O-heavy workloads but minimal on compute-heavy tasks. It offers a practical middle ground for multi-tenant SaaS and CI/CD pipelines -- stronger than containers, lighter than full VMs -- but with "different and potentially weaker security guarantees than full virtualization."
3. **Container-Based Isolation:** Docker, Bubblewrap, macOS Seatbelt, and Windows AppContainer provide process-level isolation but share the host kernel, creating residual vulnerability exposure. The November 2025 runC CVEs and the CVE-2024-1086 netfilter exploit illustrate the concrete risk. Hardening measures (drop all capabilities with `--cap-drop=ALL`, run as non-root, `--security-opt no-new-privileges`, read-only filesystems, Seccomp deny-by-default profiles, AppArmor/SELinux) can raise the bar but cannot eliminate shared-kernel risk.
4. **WASM Sandboxes:** Wasmtime and Wasmer provide lightweight execution limits (fuel-based CPU metering) with deny-by-default capability-based security. As of early 2026, CPython targets wasm32-unknown-wasip1 as a Tier 2 supported platform, and Pyodide ports CPython to WASM with NumPy/pandas/SciPy support. **Microsoft's Wassette** (announced August 2025) bridges WASM and MCP directly: it runs WebAssembly Components as MCP tools via the Wasmtime runtime, with components distributed through OCI registries and cryptographically signed using Notation/Cosign. Wassette implements a deny-by-default model where components cannot access system resources without explicit user-granted permissions -- making it one of the first production-grade implementations of WASM-based tool isolation for AI agents. Wasmtime's defense-in-depth includes 2GB guard regions for linear memories and automatic zeroing of memory between instances. The primary escape vector remains JIT compiler bugs, making runtime version hygiene critical.
5. **V8 Isolate-Based Sandboxing:** Cloudflare's **Dynamic Workers** (open beta March 2026) provide an alternative lightweight isolation model using V8 JavaScript isolates. Startup is in the low single-digit milliseconds -- 100x faster than containers -- with 10-100x better memory efficiency. The security model relies on nearly a decade of V8 isolate hardening, custom second-layer sandboxing with dynamic tenant cordoning, hardware-leveraged defenses (Memory Protection Keys), and Spectre mitigations, with V8 patches deployed within hours of upstream release. The tradeoff is language restriction (TypeScript/JavaScript only) and a fundamentally different isolation model than hardware virtualization.
6. **Confidential Computing (emerging):** Hardware-encrypted memory isolation using AMD SEV-SNP, Intel TDX, or ARM OP-TEE ensures that even the host OS cannot read sandbox data. Primarily relevant for high-sensitivity workloads where the infrastructure operator is not fully trusted.

The sandbox platform ecosystem has matured rapidly as of March 2026. Beyond the established Firecracker-based options (E2B, Modal), several notable entrants are worth tracking:

- **Microsandbox** (Zerocore AI, YC X26): An open-source, self-hosted sandbox using **libkrun** microVMs (leveraging KVM on Linux, Hypervisor.framework on macOS) rather than Firecracker. Sub-200ms startup with ~5 MiB RAM per instance and native MCP server integration for direct use with AI agents. The self-hosted model avoids vendor lock-in while maintaining hardware-level isolation. Uses Transparent Socket Impersonation (TSI) for networking rather than traditional virtual network interfaces.
- **Daytona**: Pivoted to AI code execution infrastructure in early 2025, offering sub-90ms sandbox creation with Docker isolation by default and optional Kata Containers for enhanced isolation. Provides stateful workspaces with RESTful API access, useful for agent workflows needing persistent environments across multiple tool invocations.
- **Stacklok ToolHive**: An enterprise MCP platform built for production tool management. Runs MCP servers inside containers with OIDC/OAuth SSO integration (Okta, Entra ID, Google), per-request identity propagation, tool provenance verification, and OpenTelemetry-based audit logging aligned with the MCP semantic conventions merged in January 2026. Circuit breaker protection added February 2026. Multi-namespace Kubernetes isolation supports multi-tenant deployments.

The MCP CVE wave has continued into April 2026, now extending to major cloud provider MCP servers:

- **CVE-2026-32211 (Azure MCP Server, CVSS 9.1):** Disclosed April 3, 2026 -- the Azure MCP Server shipped without authentication entirely, allowing unauthorized access to sensitive data. A CVSS 9.1 vulnerability in a first-party Microsoft cloud integration server underscores that even major cloud providers are shipping MCP implementations with basic security controls missing.
- **CVE-2026-5058 (aws-mcp-server, CVSS 9.8):** Published April 11, 2026 -- a critical command injection vulnerability in the AWS MCP server requiring no authentication for exploitation, providing a direct path to arbitrary code execution on the server.
- **CVE-2026-39884 (Kubernetes MCP Server):** An argument injection vulnerability in the `port_forward` tool where a kubectl command is constructed via string concatenation with user-controlled input, allowing arbitrary kubectl flag injection -- potentially enabling an attacker to forward traffic from any pod or service.
- **CVE-2026-20205 (Splunk MCP Server):** In Splunk MCP Server app versions below 1.0.3, users with access to the `_internal` index could view session and authorization tokens in clear text.
- **CVE-2026-35577 (Apollo MCP Server):** An authentication bypass via DNS rebinding in versions before 1.7.0, allowing attackers to access local MCP servers remotely.

Additional cloud-provider and enterprise MCP vulnerabilities disclosed in early 2026:

- **CVE-2026-26118 (Azure MCP Server SSRF, CVSS 8.8):** Patched in Microsoft's March 10, 2026 Patch Tuesday. An authorized attacker could submit a malicious URL in place of a normal Azure resource identifier through an MCP Server tool that accepts user parameters. The server sends an outbound request to that URL and includes its managed identity token, allowing the attacker to capture the token and elevate privileges without administrative access. This is a distinct vulnerability from CVE-2026-32211 (missing authentication) -- the same Azure MCP Server had two high-severity flaws within weeks of each other.
- **CVE-2026-30624 (Agent Zero, CVSS 8.6):** Published April 15, 2026 -- a remote code execution vulnerability in Agent Zero 0.9.8's External MCP Servers configuration feature. Insufficient validation of user-provided MCP server configurations allows attackers to execute arbitrary OS commands through the agent framework.

These cloud-provider MCP CVEs represent a qualitative shift: the vulnerability pattern is no longer limited to community-built tools but has reached enterprise MCP integrations from AWS, Azure, Splunk, and Kubernetes -- infrastructure that organizations are deploying into production with implicit trust.

Two notable additions as of April 2026:

- **Docker Sandboxes** (launched March 2026): Docker's official microVM-based sandbox for AI coding agents. Each sandbox runs in an isolated microVM with its own dedicated Linux kernel, Docker daemon, filesystem, and network stack -- eliminating the shared-kernel attack surface that container-based isolation suffers from. Supports Claude Code, Gemini CLI, Codex, GitHub Copilot CLI, and others out of the box. Ships as a standalone CLI (`docker sandbox run <agent>`) for macOS and Windows. The move from container-based to microVM-based isolation by Docker itself signals that the industry's largest container platform recognizes shared-kernel isolation is insufficient for AI agent workloads.
- **Kubernetes agent-sandbox** (kubernetes-sigs, v0.3.10 as of April 2026): A Kubernetes-native CRD (`Sandbox`) for managing isolated, stateful, singleton workloads designed for AI agent runtimes. Each sandbox gets a stable hostname, persistent storage that survives restarts, and lifecycle management (creation, scheduled deletion, pausing, resuming). Supports pluggable runtimes including gVisor and Kata Containers. Still pre-1.0 but gaining traction in the Kubernetes SIG Apps community.

**Cisco DefenseClaw** (announced RSA Conference, March 2026) takes a different approach: rather than providing a sandbox runtime, it is an open-source framework that automates the security verification pipeline for AI agents. It integrates Skills Scanner (scanning and sandboxing agent skills), MCP Scanner (verifying MCP server integrity), AI BoM (automatic inventory of all AI assets), and CodeGuard into a single framework. The integration plan with NVIDIA OpenShell aims to provide automated security at the runtime level, extending verification from development into production. DefenseClaw is notable as the first vendor-backed open-source framework that treats MCP server verification and agent skill scanning as a unified security problem.

On the process-level sandboxing front, **Sandlock** (Multikernel Technologies, April 2026) takes a different approach: instead of container or VM isolation, it applies per-tool-invocation confinement using three unprivileged Linux kernel interfaces -- Landlock (filesystem ACLs and TCP port restrictions), seccomp-bpf (syscall filtering), and user namespaces. The result is sub-millisecond overhead per tool call (vs. ~100ms for container startup), with no daemon, no root requirement, and a single static binary. The tradeoff is Linux-only and reliance on kernel interface maturity (Landlock requires Linux 6.12+ for full feature parity).

The **AgentBox** approach (described in MCP security research) encapsulates each MCP server inside an isolated container enforcing a declared manifest -- the server starts with zero privileges and only receives access explicitly granted by both the manifest and the user. This maps directly to requirement 9.3.5's manifest enforcement.

### Network Egress Controls and Runtime Enforcement

Network exfiltration is one of the highest-risk tool capabilities. NVIDIA's guidance is unambiguous: "Network connections created by sandbox processes should not be permitted without manual approval." Defense layers include:

- Tightly scoped allowlists enforced through HTTP proxy, IP, or port-based controls
- DNS resolution limited to designated trusted resolvers to prevent DNS-based exfiltration
- Per-tool egress quotas enforced via sidecar proxies or eBPF-based controls (per requirement 9.3.2)

As of March 2026, **eBPF-based runtime enforcement** is emerging as the preferred kernel-level mechanism for AI agent sandboxing. eBPF-PATROL (arXiv, November 2025) demonstrates a lightweight agent that intercepts syscalls (execve, open, ptrace, mount, socket) with context-aware filtering -- examining arguments and process metadata rather than just syscall names. It achieves less than 2.5% CPU overhead with 23-microsecond response latency, making it practical for production agent workloads. Policies are defined declaratively in YAML, with enforcement actions ranging from syscall denial to process termination.

In Kubernetes environments, **ARMO's KubeArmor** provides native eBPF/LSM-based enforcement for AI agent workloads, implementing per-container syscall filtering, file system restriction, and network segmentation with observe/audit versus enforce modes. This approach enables sandboxing without requiring code modifications to the agent or tool, and integrates with existing Kubernetes network policies for egress containment.

The convergence of eBPF enforcement with MCP gateway architectures is creating a layered defense model: the gateway handles authentication, rate limiting, and policy-as-code enforcement (using frameworks like Open Policy Agent), while the kernel-level eBPF layer catches any bypass attempts at the syscall level.

### Tool Output Validation and Cross-Tool Contamination

Tool outputs are a vector for indirect prompt injection: a compromised tool can return schema-valid but semantically malicious content that manipulates downstream agent reasoning. In multi-server MCP deployments, **cross-tool contamination** allows a malicious server to leverage its position in the agent's context to access data from other legitimate servers -- particularly dangerous in multi-tenant environments. The MITRE ATLAS October 2025 update formalized this as a distinct attack technique: **Exfiltration via AI Agent Tool Invocation**, where adversaries use prompts to invoke tools with write capabilities (sending emails, creating documents, updating CRM records) to exfiltrate sensitive data encoded in tool input parameters.

CyberArk's ATPA research demonstrates a particularly insidious variant: a tool returning a fake error message that requests sensitive data as part of "troubleshooting." The LLM interprets this as a legitimate resolution step, reads the requested files, and retries the tool call with the sensitive data included -- all within what appears to be normal retry logic. This bypasses both schema validation and sandboxing because the tool operates within its authorized communication channel.

Defense requires both structural validation (schema checking, type enforcement, size limits per requirement 9.3.3) and semantic validation (detecting manipulative natural language content in tool outputs). Recommended defenses as of early 2026 include runtime auditing for tools returning prompts that request sensitive data, flagging unexpected secondary tool calls following error responses, and treating all external tool information -- schema and output -- as potentially adversarial (zero-trust model). Fully automated semantic validation remains an active research area without a mature solution.

### Tool Manifest and Privilege Declaration

MCP tool definitions include basic metadata (name, description, input schema) but as of early 2026 do not natively declare privilege requirements, side-effect levels, or resource limits. Requirement 9.3.5 anticipates the need for Android-style permission manifests for AI tools. The AgentBox approach provides a reference implementation: JSON manifest files specify description and required permissions (filesystem mounts, iptables rules for network scopes, environment variable whitelists), enforced at runtime through containerization primitives. Critically, servers start with no privileges by default -- only generic permissions explicitly specified in the manifest can be instantiated as runtime permissions at execution time, creating an auditable, enforceable operating-system-level policy that cannot be circumvented through prompt manipulation.

The Microsoft Agent Governance Toolkit implements a related concept through its 4-tier privilege ring system, where tools are classified by their privilege level and the runtime enforces boundaries between rings.

Several authorization frameworks are emerging to address dynamic tool permission enforcement. Cerbos and Strata's AI Identity Gateway provide dynamic, fine-grained authorization that scopes permissions at runtime for each action the agent attempts, using central policies. This approach moves beyond static manifests toward context-aware privilege enforcement -- checking not just what the tool is authorized to do in general, but what it should be allowed to do given the current user, task, and data sensitivity.

### MCP Protocol Evolution: Tool Annotations and Authorization

The MCP specification has evolved significantly through 2025-2026, adding security-relevant capabilities that partially address the manifest and privilege gaps described above.

**Tool Annotations** (shipped in the 2025-03-26 spec revision) allow servers to attach behavioral metadata to tool definitions via a `ToolAnnotations` interface with four boolean properties: `readOnlyHint`, `destructiveHint`, `idempotentHint`, and `openWorldHint`. These enable clients to implement confirmation workflows for destructive tools, detect dangerous capability combinations (following Simon Willison's "lethal trifecta" pattern of private data access, untrusted content exposure, and external communication), and feed authorization engines with risk signals. Critically, annotations are **hints, not guarantees** -- the spec explicitly states they do not faithfully describe tool behavior. An untrusted server can lie about its annotations. They provide no enforcement mechanism and cannot replace network isolation or sandboxing for actual containment. Five proposals are under active discussion to extend the annotation vocabulary beyond the initial four hints, based on production experience from GitHub, OpenAI, and AWS.

**OAuth 2.1 Authorization** (2026-03-15 spec update) introduces a formal authorization framework where the MCP server acts as an OAuth 2.1 Resource Server. The spec now mandates RFC 8707-compliant resource indicators to prevent token mis-redemption attacks -- access tokens are strictly scoped to specific MCP servers, preventing malicious servers from using tokens intended for other services. Incremental scope consent allows clients to request minimal access per operation, and fine-grained scoping maps permissions to specific tools (e.g., `database.query` for read operations, `database.write` restricted to administrators). Per-tool invocation should validate the token's signature, issuer, audience, expiry, and required scope -- not just at session establishment.

However, the 2026 MCP roadmap positions deeper security and authorization work as "on the horizon" rather than active priority. Two SEPs are in review (SEP-1932 for DPoP, SEP-1933 for Workload Identity Federation), but substantive authorization enforcement is not expected in the core spec during 2026. Organizations requiring robust tool permission enforcement will need to implement this through external layers (gateways, policy engines, runtime sandboxes) rather than relying on protocol-level guarantees.

### MCP Gateway Architecture

As of March 2026, the emerging best practice for production MCP deployments is to place every remote MCP server behind an **API gateway** that serves as the single enforcement point for authentication, authorization, rate limiting, and logging. Because MCP does not yet include a native permissions model, enterprises must apply external policy enforcement -- **Open Policy Agent (OPA)** and similar policy-as-code frameworks define declarative rules for which tools can be called, by whom, and under what conditions. The gateway pattern provides centralized policy application regardless of which AI agents or MCP servers are involved, and enables kill switches at both the client and gateway level for rapid tool revocation when threats are detected. Workload identity verification (where agents and tools authenticate using runtime environment-derived credentials) ensures every MCP participant proves its identity before gaining access, with runtime signals such as authentication failures, burst traffic, and unusual geographies triggering automated alerts.

### Structural Credential Isolation: The "Brain and Hands" Pattern

As of April 2026, a new architectural pattern for tool isolation is emerging that addresses credential exposure structurally rather than through sandbox hardening alone. Anthropic's Managed Agents (public beta, April 8 2026) splits every agent into three mutually distrusting components: a **brain** (the model and its orchestration harness), **hands** (disposable Linux containers where code executes), and a **session** (an append-only event log outside both). The key security property is that credentials never enter the sandbox where agent-generated code runs -- instead, OAuth tokens and API keys are stored in a credential vault, and tool invocations go through a dedicated proxy that fetches the real credential from the vault and makes the API call on behalf of the sandbox. The sandbox never sees the credential string; it only sees a pre-configured proxy endpoint.

This pattern addresses a fundamental weakness in traditional tool isolation: even a perfect sandbox is insufficient if credentials live inside the sandboxed environment. A sandbox escape or prompt injection that exfiltrates environment variables can steal tokens regardless of process isolation. By making credentials unreachable by construction, the blast radius of a sandbox compromise is limited to a disposable container with no tokens and no persistent state.

Anthropic's Claude Code sandboxing applies a related approach on the client side: bubblewrap (Linux) and seatbelt (macOS) provide filesystem and network isolation, with network traffic routed through a Unix domain socket to an external proxy that enforces domain allowlists. Git credentials and signing keys are kept outside the sandbox, with a custom proxy transparently attaching scoped tokens to git operations after validating branch names and repository destinations.

However, this pattern is not without limitations. Independent security analysis of the Managed Agents architecture (as of April 2026) identified several gaps: runtime monitoring tools (Falco, eBPF agents) cannot be deployed inside the sandbox, OAuth token scopes cannot be restricted at the vault level (only at client configuration), and the confused deputy problem remains -- anyone with API key access can specify another user's credential vault when creating sessions, collapsing cross-user permission boundaries. Additionally, default networking allows package manager access, creating supply chain exposure through slopsquatting and dependency confusion attacks.

The "brain and hands" separation pattern is conceptually similar to the separation of concerns in HSM-backed cryptographic operations (where keys never leave the hardware) and represents a meaningful architectural advance over the traditional model of sandboxing a process that holds its own credentials.

### AI Agent Governance Gaps: Industry Survey Data

A CSA Lab Space research note published April 3, 2026 quantified the gap between AI agent deployment velocity and security governance maturity. Key findings from an enterprise survey: 92% of large-enterprise security leaders lack full visibility into AI agent identities operating in their environments, only 16% effectively govern AI agent access to core business systems (ERP, CRM, financial platforms), and just 17% continuously monitor agent-to-agent interactions. Meanwhile, over 8,000 MCP servers are publicly exposed without authentication -- a number that has grown alongside MCP adoption.

The report notes that existing security frameworks (NIST AI RMF 1.0, ISO/IEC 42001:2023, EU AI Act) predate agentic systems and contain structural gaps. Recommended immediate actions include establishing comprehensive AI agent inventories with identity, permissions, connected tools, and business owners; applying least-privilege credentials with time-bound, just-in-time access; extending SIEM/UEBA to capture agent-to-agent interactions and tool call sequences; and developing agent-specific incident response procedures. The CSA MAESTRO framework (Multi-Agent Environment, Security, Threat Risk, and Outcome) provides a seven-layer threat model from foundation models through ecosystem integration, enabling systematic identification of cross-boundary attack surfaces that emerge when agents compose tools, memory, and external integrations across organizational trust boundaries.

## Open Research Questions

- What is the right granularity for tool permission manifests -- per-tool, per-invocation, or per-task? The AgentBox JSON manifest approach operates per-server; finer granularity (per-invocation) would increase security but also overhead. Dynamic authorization frameworks (Cerbos, Strata) are exploring per-action scoping, but the performance and complexity implications for high-throughput agent workloads are not yet well understood.
- How do you securely handle tools that need stateful sessions (e.g., database connections, authenticated API sessions) within sandbox isolation boundaries? Full VM isolation complicates connection pooling and session persistence.
- Can tool output validation be automated using a secondary LLM as a "tool output auditor," or does this introduce new attack surface? Cross-tool contamination research suggests that any component with access to tool outputs is itself a potential manipulation target. CyberArk's ATPA research shows that fake error messages requesting sensitive data bypass both schema validation and sandboxing -- what detection approach can catch these without unacceptable false positive rates?
- How should dynamic tool discovery (agent finds and loads tools at runtime) be secured without making the system too rigid? The Smithery supply chain attack (October 2025) demonstrated the risk of open tool registries.
- How can "sampling attacks" -- which operate through legitimate protocol features rather than exploiting vulnerabilities -- be detected and mitigated? These bypass both tool integrity checks and sandboxing because they use authorized channels.
- What is the performance/security tradeoff between full virtualization (strongest isolation, highest overhead) and gVisor/WASM approaches for high-throughput agent workloads? Firecracker's ~125ms boot time and <5 MiB overhead make microVMs practical, but the operational complexity of managing VM fleets at agent scale remains a barrier.
- How should tool manifests handle tools whose privilege requirements change dynamically based on the task context (e.g., a database tool that sometimes needs write access)?
- Full-Schema Poisoning demonstrates that any part of a tool schema can carry malicious instructions. Should MCP clients enforce strict schema allowlisting (rejecting unknown fields), or would this break backward compatibility with the ecosystem? What is the right balance between schema flexibility and security?
- The n8n sandbox escape series (CVE-2026-25049, CVE-2026-1470, CVE-2026-0863) showed that application-layer sandboxes (AST sanitizers, expression evaluators) are systematically brittle because language evolution creates bypass paths faster than sanitizer rules can be updated. Should agent platforms rely exclusively on OS/hypervisor-level isolation, or is there a defensible architecture for application-layer sandboxing? The vm2 CVSS 9.8 escape (CVE-2026-22709) caused by V8 async stack trace optimizations suggests even mature sandbox libraries cannot anticipate runtime behavior changes.
- How should organizations layer eBPF-based runtime enforcement with MCP gateway controls? eBPF catches syscall-level escapes with microsecond latency but requires kernel access; gateway enforcement operates at the protocol level without kernel privileges. What is the right division of responsibility, and how do you ensure the layers reinforce rather than conflict with each other?
- As WASM-based tool isolation matures (Wassette, Wasmtime), what is the right model for dependency management? Full CPython-in-WASM passes security vectors but cannot run pip-installed packages or C-extensions -- is this an acceptable tradeoff for high-security environments, or does it limit WASM adoption to simple tools?
- How should the MCP ecosystem standardize tool definition integrity? The ETDI proposal adds cryptographic signing and versioned immutability, but adoption requires ecosystem-wide changes. Tool definition pinning (mcp-scan) works today but only catches mutations after the fact -- it cannot prevent a malicious server from executing a modified tool before the client detects the change. What is the right balance between protocol-level enforcement and client-side detection?
- V8 isolate-based sandboxing (Cloudflare Dynamic Workers) provides millisecond startup and strong memory efficiency but is limited to JavaScript/TypeScript. As AI agents increasingly need to execute Python, shell scripts, and multi-language toolchains, how should isolation strategies be composed across different runtime environments within a single agent workflow?
- The "brain and hands" credential isolation pattern (vault-proxy separation) prevents credential theft even if the sandbox is compromised, but introduces new attack surfaces: confused deputy attacks where one user's credentials are accessed through another's session, and supply chain exposure through package managers that remain network-accessible inside the sandbox. How should credential vault access be scoped to prevent cross-session credential access, and is there a viable model for fully air-gapping sandboxes from package registries while still supporting tool installation?
- The MCP vulnerability wave (now 40+ CVEs, January-April 2026, including cloud-provider servers from AWS, Azure, Splunk, and Kubernetes) revealed that 82% of implementations are vulnerable to path traversal and 43% to shell injection. Is the MCP ecosystem's security posture fundamentally a maturity problem (that will improve as implementations harden), or does the protocol's design -- which delegates security entirely to implementations rather than enforcing it at the protocol level -- create a structural weakness that cannot be addressed without protocol changes?
- The AWS AgentCore research ("Cracks in the Bedrock" and "Agent God Mode") demonstrated that sandbox isolation and infrastructure security are orthogonal concerns: a sound Firecracker microVM is undermined by unprotected metadata services, overly permissive IAM roles, and functional DNS resolution. How should cloud provider AI sandbox offerings be evaluated? Should there be a standard security benchmark that tests not just the sandbox boundary itself but the entire surrounding infrastructure (metadata services, IAM scoping, network egress including DNS, credential exposure)?
- Application-layer sandbox escapes keep recurring across different languages and techniques (n8n AST sanitizer bypass, PraisonAI frame traversal, PraisonAI shell injection, MaxKB LD_PRELOAD bypass, Agenta RestrictedPython escape). Each bypass exploits a different mechanism, but the root cause is the same: the sandbox cannot anticipate all runtime behaviors. Should security standards explicitly recommend against application-layer sandboxing for untrusted code, or is there a defensible architecture that combines application-layer restrictions with mandatory OS/hypervisor-level isolation?
- SandboxEscapeBench shows that frontier LLMs can escape container misconfigurations at 40-49% success rates for ~$1 per attempt, and success scales log-linearly with token budget. As models become more capable and inference costs drop, the attack economics will shift further in the attacker's favor. How should organizations factor LLM escape capability into their sandbox hardening decisions? Should container escape benchmarking become a standard part of AI deployment security assessments?
- MCP tool annotations (readOnlyHint, destructiveHint, etc.) are explicitly "hints, not guarantees" with no enforcement mechanism. Five proposals are under discussion to extend the vocabulary, but no enforcement path has been proposed. Is there a viable architecture where annotations can be trusted (e.g., through attestation or verified execution), or are they fundamentally limited to UX hints for human operators? The gap between what annotations promise and what requirement 9.3.5 demands (enforced privilege declarations) suggests the ecosystem may need a parallel enforcement layer that treats annotations as advisory inputs to a policy engine, not as security controls themselves.
- The Figma MCP RCE (CVE-2025-53967) and Anthropic mcp-server-git vulnerabilities (CVE-2025-68143/68144/68145) show that even first-party and widely-trusted MCP servers ship with classical AppSec bugs. Should MCP tool integrity verification (9.3.4) extend beyond supply chain tamper detection to include mandatory security scanning (SAST/DAST) of the tool implementation itself? Cisco DefenseClaw's integrated scanning approach suggests the industry is moving in this direction.
