# C4.2: AI Hardware Security

[Back to C04 Index](C04-Infrastructure.md)

## Purpose

Secure AI-specific hardware components including GPUs, TPUs, and specialized AI accelerators. AI hardware presents unique security challenges absent from traditional computing: GPU memory is shared across workloads by default, accelerator interconnects carry sensitive model data at high bandwidth, firmware vulnerabilities can compromise entire GPU clusters, and side-channel attacks on accelerators can leak model weights or training data. This section addresses hardware-level security controls specific to AI accelerator infrastructure.

The draft explicitly scopes out pipeline chaining, multi-inference receipts, and any proof of data deletion or model correctness, so it does not replace existing attestation stacks but adds a portable, audit-friendly wrapper for per-request evidence. (1) **GDDRHammer** (UNC Chapel Hill, Georgia Tech, MBZUAI -- gddr.fail) bypasses Target Row Refresh (TRR) defenses in GDDR6, averaging 129 bit flips per bank on RTX A6000. (2) **GeForge** (Purdue, Rochester, Western Australia, HydroX AI, Clemson) produced 1,171 bit flips on a consumer RTX 3060 and 202 on A6000. (3) **GPUBreach** (University of Toronto, April 8, 2026, gpubreach.ca; formal paper arxiv 2605.03812) goes further: it chains GPU page-table corruption with a memory-safety flaw in the NVIDIA kernel driver to bypass IOMMU entirely, eliminating the primary defensive recommendation against GPU DMA attacks. The paper quantifies the primitive on an RTX A6000 (48 GB GDDR6): of 34 bit flips observed across six DRAM banks, 9 were usable for privilege escalation by flipping the Page Frame Number field of a GPU PTE to redirect a virtual address onto another page-table page, with roughly 3 hours of profiling per exploitable flip. The IOMMU bypass works because unsanitized GPU-produced values flowing through the GSP (GPU System Processor) message queue reach kernel operations, yielding out-of-bounds writes in host kernel memory even with IOMMU enabled. The authors disclosed to NVIDIA on November 11, 2025, and recommend statically isolating page tables from data pages, sanitizing GSP-sourced values in driver code, MAC-based integrity Rowhammer defenses, and in-DRAM secure refresh counters. Tested without bit flips: RTX 3080 (GDDR6X), RTX 4060/4060 Ti and RTX 6000 Ada (GDDR6 with newer controllers), RTX 5050 (Blackwell, GDDR7); datacenter Hopper/Blackwell (H100/H200/B100/B200) have ECC enabled by default and are not demonstrated vulnerable. As of early April 2026 no CVEs were assigned and NVIDIA had not issued a security bulletin specific to these attacks. **Attack efficiency varies sharply across GPU models even with identical memory standards:** Kaspersky's April 2026 analysis highlighted that GeForge efficiency "dropped by more than five times" on RTX A6000 compared to consumer RTX 3060 despite both using GDDR6, meaning attackers would need device-specific profiling before exploitation. This reduces the practical risk of indiscriminate cloud mass-exploitation but does not eliminate targeted attacks where the adversary knows the assigned hardware. **Consumer GDDR6 has no known mitigation:** the researchers state plainly that for desktop/laptop GPUs running GDDR6, ECC is unavailable at the hardware level and no patch closes the driver memory-safety path GPUBreach exploits -- a concerning asymmetry for workstations running local AI inference on gaming cards. Mitigation stack (server-grade): enable ECC (`nvidia-smi -e 1` + reboot, ~5-15% inference overhead, 6.25% VRAM loss), enable IOMMU (blocks GDDRHammer/GeForge DMA escalation but not GPUBreach), restrict GPU co-tenancy in multi-tenant environments, prioritize Hopper/Blackwell/GDDR7 platforms for sensitive workloads, and add cryptographic model-integrity verification on model load. Note that ECC is not a universal defense: if attack patterns produce more than two bit flips per word (demonstrated feasible on DDR4/DDR5 host memory systems), single-bit ECC cannot correct and may silently mask corruption. The papers were presented at IEEE S&P 2026 (May 18-20, San Francisco); as of early June 2026, no CVEs had been assigned and NVIDIA had not issued a bulletin specific to these attacks. NVIDIA MIG (available on A100, H100, Blackwell) creates up to 7 isolated GPU instances with separate memory, cache, and compute resources. Without MIG, CUDA applications on the same GPU can potentially access each other's memory via `cudaIpcGetMemHandle`. MIG partitions have their own error isolation -- a fault in one partition does not affect others. Limitation: MIG reduces flexibility in GPU memory allocation. **NVIDIAScape (CVE-2025-23266, July 2025):** Wiz Research discovered a critical container escape vulnerability (CVSS 9.0) in the NVIDIA Container Toolkit. SR-IOV partitions GPU resources at the PCIe level, offering stronger isolation guarantees than software-based partitioning. **Confidential computing as complementary isolation:** On Blackwell with CC mode and TDISP/IDE support, direct communication between GPU and Confidential Virtual Machine uses inline encryption, eliminating the bounce buffer latency seen on H100. This provides hardware-enforced isolation beyond MIG partitioning. For multi-GPU workloads, NVLink pathways are also encrypted on Blackwell, preventing cross-partition snooping at the interconnect level. **Behind Bars MIG cache side channel (USENIX Security 2026):** "Behind Bars: A Side-Channel Attack on NVIDIA MIG Cache Partitioning Using Memory Barriers" -- by Yanan Guo, Cheng Gu (University of Rochester), Tyler Sorensen (Microsoft Research / UC Santa Cruz), Reese Levine (UC Santa Cruz), and Zhenkai Zhang (Clemson) -- is now a published USENIX Security 2026 paper, not just a vendor-acknowledged disclosure. That is a confidentiality (workload/metadata) leak, not yet a weight-exfiltration channel, but it directly contradicts the assumption that MIG gives full confidentiality isolation between tenant instances. |

## Implementation Guidance

### Confidential Computing Evolution (2024-2026)

NVIDIA Confidential Computing has progressed through three generations, each expanding the hardware trust boundary:

| Generation | Platform | Key Capabilities | Limitations |
|-----------|----------|-----------------|-------------|
| **Gen 1** | H100 Hopper (2023-2024) | On-die hardware RoT, TEE with encrypted PCIe via bounce buffers, GPU attestation SDK, CC-On/CC-Off/CC-DevTools modes | ~4 GB/s PCIe encryption throughput (CPU-limited), NVLink unencrypted in non-CC mode, performance counters disabled in CC-On |
| **Gen 2** | B200/GB200 Blackwell (2025) | TDISP/IDE inline encryption (no bounce buffers), encrypted NVLink, GPU memory encryption in CC mode, dual attestation (Intel TDX + NVIDIA), near-native performance | Requires compatible host CPUs (AMD SEV-SNP or Intel TDX) |
| **Gen 2.5** | B300/GB300 Blackwell Ultra (shipped January 2026) | Inherits Blackwell Gen 2 CC architecture (TEE-I/O, CC mode, encrypted NVLink, dual attestation), scales to 288 GB HBM3e per GPU and 14 PFLOPS dense FP4, GB300 NVL72 rack integrates 18 BlueField-3 DPUs (attestable via SPDM/Redfish) and ConnectX-8 SuperNICs at 800 Gb/s per GPU. PCIe Gen6 host link | NVIDIA's public B300 documentation does not yet detail whether the 288 GB HBM3e memory controller introduces new on-die ECC or Rowhammer mitigations beyond what B200 shipped with -- independent validation not yet public as of April 2026 |
| **Gen 3** | Vera Rubin NVL72 (full production May 2026) | Rack-scale confidential computing across 72 GPUs + 36 CPUs + full NVLink fabric, unified TEE across entire rack, NRAS integration verifying CPUs/GPUs/NICs/firmware/drivers/workloads against known-good measurements | Ramped into full production (NVIDIA newsroom, May 31, 2026); launch CC adopters include CoreWeave, Firmus, GMI Cloud, IBM Cloud, IREN, Lambda, Microsoft Azure, Nebius, Nscale, SpaceXAI, Vultr |

**Practical deployment note:** CUDA applications should run without code changes in CC mode -- encryption is handled transparently. The primary overhead on H100 is CPU-GPU data transfer bandwidth; applications with high compute-to-data ratios see minimal impact. Blackwell eliminates this bottleneck with inline encryption. As of early 2026, confidential AI inference using CPU+GPU TEE combinations is in production at major cloud providers including Alibaba, ByteDance, Google, and Oracle, confirming that bounce buffer architectures are operationally viable at scale.

**Corvex verified production CC on HGX B200 (March 3, 2026):** Corvex publicly announced verified production deployment of confidential computing on NVIDIA HGX B200 systems, validating encrypted GPU-to-GPU traffic across NVSwitch and NVLink fabrics using Intel TDX on the host and NVIDIA Confidential Computing on the GPU, with end-to-end remote attestation via Intel Trust Authority (ITA) covering both CPU and GPU. Performance reporting on Llama-3.1-70B showed near-native throughput: because production-scale LLM serving is dominated by GPU compute and HBM bandwidth, the marginal cost of TDISP/IDE inline encryption rounded down to "essentially zero" at typical batch sizes. This is one of the first independent, vendor-neutral validations that the Blackwell CC architecture is operationally viable for regulated LLM inference workloads outside hyperscaler-managed offerings, and a useful reference for security teams arguing that confidential-AI mode does not require a measurable performance trade-off on production-class models. Alpha Compute separately reported a 504-chip B200 cluster targeted for AI Confidential Computing customer hand-off on May 8, 2026, after testing B300/B200/H200 hardware with CC enabled from November 2025 onward -- evidence that mid-size compute providers are also building B200/B300 deployments around CC mode rather than offering it as an opt-in.

**Jetson AGX Thor (Blackwell edge, 2026):** NVIDIA's developer documentation confirms that Confidential Computing and TDISP support are being enabled on Jetson AGX Thor, the Blackwell-architecture edge/robotics platform. This extends the Hopper/Blackwell CC model beyond data center SKUs to the edge for the first time -- relevant for robotics, autonomous systems, and regulated on-prem AI deployments where the workload handles sensitive data but is not housed in a hyperscale facility. Operationally, this means organizations deploying Thor-based inference at the edge can expect the same attestation-rooted CC posture (NRAS-verifiable quotes, TEE-I/O with inline encryption on the host link) that is available in H100/B200 cloud instances, subject to host CPU TEE support. For regulated edge deployments (medical imaging appliances, industrial control AI) this is the first credible hardware path to attestation-backed AI inference off a hyperscaler. As of April 2026 the Thor CC enablement is tracked in NVIDIA developer forums rather than a formal product announcement, so production readiness should be validated per deployment.

**Critical caveat -- TEE.Fail (October 2025):** All three generations of GPU confidential computing depend on CPU-side TEE attestation (Intel TDX or AMD SEV-SNP) as their trust anchor. The TEE.Fail attack demonstrated that a sub-$1,000 DDR5 memory-bus interposer can extract attestation keys from these CPU TEEs, enabling forged attestation quotes that pass verification. This means GPU CC protections can be bypassed by an adversary with physical access to the server's memory bus. Organizations relying on GPU confidential computing for high-value workloads should assess whether their threat model includes physical-access adversaries and consider complementary controls (physical security, tamper-evident server chassis, restricted datacenter access).

**GPU-CC implementation-level security gaps (IBM Research / Ohio State, July 2025):** A systematic analysis of NVIDIA's GPU Confidential Computing implementation (arxiv 2507.02770) identified several architectural weaknesses beyond the TEE.Fail physical-access threat. Key findings: (1) RPC queue headers and physical address tables remain unencrypted, allowing attackers to locate all TX/RX queue elements and infer RPC operation types from plaintext metadata; (2) self-referential memory address tables enable adversaries to scan physical memory in 4KB strides to identify critical data structures; (3) memory transfer operations exhibit a bimodal timing distribution that reveals transfer sizes, creating a timing side-channel; (4) approximately 0.02% of BAR0 register fields remain accessible after firewall activation despite encryption, with no public rationale for the exposed registers; (5) GPU memory faults transit through unprotected shadow buffers, leaking fault metadata; (6) the SEC2 security co-processor lacks encryption capability (decryption-only), leaving pushbuffers and semaphores unprotected during memory scrubbing operations. All findings were responsibly disclosed to NVIDIA PSIRT. These gaps do not break the overall CC threat model (which excludes certain physical and firmware-level attacks) but reveal that the implementation has a larger residual attack surface than the architecture suggests.

### GPU Side-Channel and Hardware Attack Landscape (2023-2026)

The threat landscape for GPU-specific attacks has expanded significantly, spanning software-exploitable side channels, physical fault injection, and TEE bypass attacks:

**Software-exploitable (remote/co-tenant):**
- **LeftoverLocals (CVE-2023-4969):** Co-resident attacker reads uncleared GPU local memory to reconstruct LLM responses. Affected: Apple, AMD, Qualcomm, Imagination GPUs. NVIDIA GPUs not affected. Requires only user-level GPU compute access.
- **GPU.zip (IEEE S&P 2024):** Browser-based pixel-stealing attack exploiting GPU data compression. Affects all major GPU vendors. Currently unpatched across the industry. Exploitable via malicious web pages in Chrome.
- **GPUHammer (USENIX Security 2025):** First demonstrated Rowhammer attack on GPU GDDR6 memory. Induces bit flips via multi-warp parallel CUDA hammering on NVIDIA RTX A6000, degrading DNN model accuracy from 80% to 0.1% with a single bit flip. Requires only user-level CUDA access, making it viable in shared cloud environments. ECC mitigates single-bit flips but incurs ~10% performance overhead. Newer HBM3 and GDDR7 architectures have on-die ECC that should resist current attack patterns.
- **GDDRHammer / GeForge / GPUBreach (April 2026, presented at IEEE S&P 2026, May 18-20 San Francisco):** A coordinated trio of disclosures elevated GPU Rowhammer from "accuracy degradation" to "full host compromise." GDDRHammer (UNC Chapel Hill, Georgia Tech, MBZUAI -- gddr.fail) bypasses GDDR6 Target Row Refresh via "memory massaging," averaging 129 flips per bank on an RTX A6000. GeForge (Purdue, Rochester, Western Australia, HydroX AI, Clemson) produced 1,171 bit flips on a consumer RTX 3060 and 202 on A6000. Both exploit an allocator placement flaw where GPU page tables and user data share the same GDDR6 region, then corrupt a page-table entry to redirect a GPU virtual address through the PCIe BAR1 aperture into CPU physical memory -- yielding arbitrary read/write to host RAM and a root shell. GPUBreach (University of Toronto, April 8, 2026) chains the same page-table corruption with a memory-safety bug in the NVIDIA kernel driver to escape even an IOMMU-protected configuration. IOMMU blocks GDDRHammer/GeForge but not GPUBreach. Tested and not reproduced on: GDDR6X (RTX 3080), newer GDDR6 controllers (RTX 4060/4060 Ti, RTX 6000 Ada), GDDR7 (Blackwell RTX 5050), and datacenter Hopper/Blackwell (H100/H200/B100/B200 ship with ECC on by default). As of mid-April 2026 no CVEs had been assigned. NVIDIA's position, communicated after the researchers' November 2025 disclosure, is that these findings reinforce the existing "Security Notice: Rowhammer - July 2025" rather than constituting new vulnerabilities; NVIDIA stated it "may" update the July 2025 notice but as of May 12, 2026 (the day NVIDIA's May GPU Display Drivers bulletin shipped) had still not issued a targeted bulletin or driver patch for GPUBreach/GDDRHammer/GeForge, and the May bulletin did not silently include such a patch either. The papers were presented at IEEE S&P 2026 on May 18-20 in San Francisco, and as of early June 2026 NVIDIA had still not assigned CVEs or issued a targeted bulletin -- security teams that relied on the "no CVE, no bulletin" pattern as a prioritization signal should treat this as a confirmed gap, since the vendor has effectively categorized these as known-issue variants despite the new full-system-compromise primitives.
- **Not so Refreshing (USENIX Security 2025):** Ravan Nazaraliyev, Yicheng Zhang, Sankha Baran Dutta, Andres Marquez, Kevin Barker, and Nael Abu-Ghazaleh (UC Riverside / PNNL) showed that Refresh Management (RFM) -- the very feature added to GDDR6 to *mitigate* Rowhammer -- introduces a timing side channel: repeated same-bank accesses trigger refresh events whose latency is observable. They built a covert channel exceeding 50 KBps per bank at 0.03% error and two end-to-end side channels (GPU application fingerprinting at up to 95% F1, Blender 3D-object-rendering fingerprinting at up to 98% F1) on a shared discrete GDDR6 GPU. The lesson is directly relevant to memory-isolation requirements: a hardware Rowhammer countermeasure can itself become a confidentiality leak, so verifying ECC/RFM is enabled does not by itself establish that co-tenant workloads are isolated from timing-based fingerprinting.
- **Behind Bars (USENIX Security 2026):** A side-channel attack on NVIDIA MIG cache partitioning using memory barriers, by Yanan Guo, Cheng Gu (University of Rochester), Tyler Sorensen (Microsoft Research / UC Santa Cruz), Reese Levine (UC Santa Cruz), and Zhenkai Zhang (Clemson), now published in the USENIX Security 2026 proceedings. It demonstrates cross-MIG-partition timing leakage on H100 despite MIG's dedicated per-instance L2 cache groups, memory controllers, and DRAM address buses, with the concrete primitive being cross-instance inference of a victim's kernel launch pattern (which correlates with LLM-inference workload characteristics) rather than direct memory reads. AMD's response bulletin AMD-SB-6026 (February 12, 2026) concluded Instinct MI3XX XCD partitioning is not affected; NVIDIA had issued no bulletin or CVE as of June 2026. First public, peer-reviewed evidence that MIG -- the primary multi-tenant isolation control on datacenter NVIDIA GPUs -- has an exploitable microarchitectural side channel.

**Physical proximity (edge/embedded):**
- **BarraCUDA (USENIX Security 2025):** Electromagnetic side-channel extraction of neural network weights from NVIDIA Jetson devices running TensorRT inference. Demonstrates that edge AI deployments face IP theft risks from physical access.
- **Power/EM analysis research (ongoing):** Multiple academic groups have demonstrated DNN architecture extraction from power and electromagnetic traces, though practical exploitation requires physical proximity or co-tenancy.

**TEE and attestation bypass (physical access):**
- **TEE.Fail (October 2025):** Georgia Tech, Purdue, and Synkhronix researchers demonstrated a DDR5 memory-bus interposition attack using under $1,000 in off-the-shelf components to extract attestation keys from Intel TDX and AMD SEV-SNP. Because NVIDIA GPU Confidential Computing depends on CPU CVM attestation as its trust anchor, forged attestation quotes can undermine the entire GPU TEE chain. Intel and AMD consider physical memory-bus attacks out of scope for their threat models. This is a fundamental limitation: hardware confidential computing currently assumes the memory bus is trusted.

**Browser-based GPU attacks:**
- **WebGPU Tint CVE-2026-3062 (2026):** Critical memory corruption in Chromium's WebGPU Tint shader compiler allows arbitrary code execution through malicious WGSL shader code. Affects all Chromium-based browsers (Chrome, Edge, Brave, Opera). Particularly concerning for browser-based AI inference applications and GPU-accelerated web tools. Patched in Chrome 130.0.6723.118+. This extends the GPU.zip attack surface -- browsers remain a viable vector for GPU-level exploitation without requiring local access.

**Container escape and virtualization escape:**
- **NVIDIAScape (CVE-2025-23266, CVSS 9.0, July 2025):** Critical container escape in NVIDIA Container Toolkit allowing full host root access via a three-line Dockerfile exploiting `LD_PRELOAD` in the `nvidia-ctk` OCI hook. NVIDIA's bulletin fixes the broader affected component set in Container Toolkit 1.17.8, Kubernetes Device Plugin 0.17.3, MIG Manager 0.12.2, and GPU Operator 25.3.2.
- **vGPU guest-to-host escape (CVE-2025-33220, CVSS 7.8, January 2026):** Use-after-free in NVIDIA Virtual GPU Manager enables guest VM escape on VMware vSphere, XenServer, and KVM. Combined with the NVIDIAScape container escape from six months earlier, this represents a pattern of isolation boundary failures in GPU virtualization infrastructure.

**MITRE ATLAS coverage gap (as of v5.6.0, May 2026):** The February 2026 v5.4.0 release of ATLAS added `Publish Poisoned AI Agent Tool` and `Escape to Host` techniques, v5.5.0 added local-machine compromise techniques for AI agents and artifacts, and v5.6.0 added techniques such as AI agent configuration acquisition, code-repository search, and deepfake-assisted phishing. However, ATLAS still does not catalog GPU-specific attack techniques -- LeftoverLocals, GPU.zip, GPUHammer, GPUBreach, GDDRHammer, GeForge, BarraCUDA, and TEE.Fail are not represented with their own technique IDs. When mapping GPU incidents to ATLAS, the closest current primitives are `AML.T0018 Manipulate AI Model` (weight corruption via Rowhammer), `AML.T0025 Exfiltration via Cyber Means` (GPU memory side-channel leakage), and `Escape to Host` (NVIDIAScape CVE-2025-23266 container escape, vGPU CVE-2025-33220 guest-to-host escape, and GPUBreach IOMMU-bypass root shell). Security teams using ATLAS for reporting should explicitly document this mapping gap and treat GPU attacks as cross-cutting infrastructure controls rather than assuming equivalent ATLAS coverage.

**Mitigation hierarchy:** (1) Use hardware confidential computing where available -- NVIDIA CC mode on Hopper/Blackwell/Vera Rubin, or AMD SEV-SNP with Instinct MI350X DICE attestation. (2) Enable hardware-level multi-tenant isolation (NVIDIA MIG, AMD SR-IOV). (3) Enforce memory sanitization between workloads. (4) Enable ECC on GDDR6 GPUs in shared environments -- note that the April 2026 GDDRHammer/GeForge/GPUBreach disclosures make ECC plus IOMMU the new baseline for any RTX A-series or consumer GDDR6 device exposed to untrusted workloads, and datacenter Hopper/Blackwell (H100/H200/B100/B200) already ship with ECC on. (5) Deploy telemetry monitoring for anomaly detection. (6) Patch GPU drivers, container toolkits, and schedulers aggressively -- treat as kernel-mode or cluster-control-plane components (NVIDIA January 2026: 5 CVEs; NVIDIA May 2026 GPU Display Drivers bulletin a_id/5821: 12 CVEs including CVE-2026-24187 CVSS 8.8 on Linux Display Driver and CVE-2026-24197 affecting MIG partition management specifically -- fixed in R595 595.71.05 for RTX/Quadro/Tesla/NVS, GeForce 596.36+; AMD March 2026: CVE-2026-23213; Container Toolkit: CVE-2025-23266 NVIDIAScape and CVE-2025-23267 symlink attack -- fixed in Toolkit 1.17.8 / Device Plugin 0.17.3 / MIG Manager 0.12.2 / GPU Operator 25.3.2; KAI Scheduler: CVE-2026-24176 and CVE-2026-24177 fixed in v0.13.0; AMD May 13, 2026 patch day: 15 advisories covering 45 CVEs including CVE-2026-0481 CVSS 9.2 in the ROCm Device Metrics Exporter and the AMD-SB-6027 graphics driver batch, plus upstream Linux amdgpu kernel CVEs CVE-2026-46197 and CVE-2026-46199 in late May 2026; Intel May 2026: CVE-2026-20794 CVSS 9.3 buffer overflow in the Data Center Graphics Driver for VMware ESXi). (7) For browser-exposed GPU workloads, monitor WebGPU attack vectors (CVE-2026-3062). (8) For edge devices, apply physical security controls and consider model obfuscation. (9) For BlueField DPU deployments, treat DPU firmware as part of the attestation trust chain and track vendor bulletins (Dell DSA-2026-119 for BlueField/ConnectX/DOCA); where BlueField is used to enforce agent policy (e.g., EQTY Verifiable Runtime at GTC 2026), include DPU attestation in the remote attestation flow. (10) For AI framework risk, pin and patch the full stack (Apex, NeMo, Megatron Bridge, DALI, Triton r26.03, BioNeMo commit e5e58c8+, NVFlare 2.7.2) -- framework deserialization and auth-bypass CVEs are now the highest-volume entry vector for compromising GPU hosts.

### Kubernetes GPU Scheduling Isolation (2026)

GPU partitioning controls can be undermined above the hardware layer if the Kubernetes scheduler, device plugin, or operator can assign resources across the wrong tenant boundary. This is now a first-class assurance issue because large AI clusters increasingly rely on dynamic GPU sharing, fractional allocation, and topology-aware placement rather than static one-tenant-per-node assignment.

**NVIDIA KAI Scheduler vulnerabilities (April 2026):** NVIDIA's April 21, 2026 bulletin for KAI Scheduler disclosed CVE-2026-24177 (CVSS 7.7, missing authentication for API endpoints) and CVE-2026-24176 (CVSS 4.3, improper authorization through cross-namespace pod references). Both affect Linux deployments before v0.13.0. In a shared GPU cluster, these are not just scheduler bugs: exposed scheduling metadata can reveal tenant workload placement, model size, queue pressure, and accelerator topology, while cross-namespace references can tamper with resource-binding state. Verification should include confirming KAI Scheduler v0.13.0 or later, restricting access to the scheduler namespace, auditing Queue and PodGroup objects for cross-namespace references, and testing that low-privilege tenants cannot query scheduler APIs or borrow reserved GPU slices outside their namespace.

**GPU Operator privilege boundary:** NVIDIA's GPU Operator security guidance still requires several operands to run with `privileged: true`, `hostPID: true`, and `hostIPC: true` because they load kernel modules, restart container runtime services, and access host GPU devices. That is operationally understandable, but it means the operator namespace is effectively part of the hardware trust boundary. Auditors should review RBAC on the operator namespace, admission controls that block tenant workloads from running there, image provenance for driver and toolkit containers, and upgrade cadence for the Container Toolkit/GPU Operator CVE list. **CDI/NRI operational check (GPU Operator 25.10+ / 26.3):** CDI is now the default device-injection path for Kubernetes GPU workloads, and the NRI plugin can remove the need for an `nvidia` runtime class or containerd config rewrites. That reduces legacy runtime-hook exposure, but it changes the evidence auditors should request: confirm CDI is enabled, confirm GPU management containers are the only workloads allowed to use broad `NVIDIA_VISIBLE_DEVICES`, and verify NRI is supported by the actual runtime version (containerd 1.7.30, 2.1.x, or 2.2.x; not CRI-O).

**Dynamic Resource Allocation and ComputeDomains:** NVIDIA donated its Dynamic Resource Allocation driver for GPUs to the Kubernetes community in March 2026, and its ComputeDomains abstraction bridges NVLink/IMEX fabric configuration into Kubernetes scheduling. This is useful for secure high-bandwidth multi-node jobs, but it also means topology and interconnect policy can move from hand-configured infrastructure into declarative Kubernetes objects. The November 2025 GB200 NVL72 guidance describes ComputeDomains dynamically forming, managing, and tearing down per-workload IMEX domains so multi-node NVLink reachability follows a scheduled job rather than a static fabric carve-out. Verification should check that ComputeDomains are namespace-scoped where appropriate, that multi-node NVLink domains are created only for approved tenants, that resource claims and `nvidia.com/gpu.clique` placement rules match the approved topology, and that current limitations (one pod per node and at most one ComputeDomain per node in the documented v25.8.0 flow) do not weaken isolation assumptions.

### Non-NVIDIA Accelerator Security Landscape (2025-2026)

As of March 2026, the GPU security ecosystem is no longer NVIDIA-only. AMD and Google have introduced meaningful hardware security capabilities, though maturity varies:

| Vendor | Platform | Attestation | Memory Encryption | Interconnect Encryption | Multi-Tenant Isolation |
|--------|----------|-------------|-------------------|------------------------|----------------------|
| **NVIDIA** | H100 / Blackwell / Vera Rubin | NRAS + Attestation SDK (ECC-384 RoT) | CC mode (HBM encrypted on Blackwell+) | NVLink encrypted (Blackwell+), TDISP/IDE | MIG (up to 7 instances) |
| **AMD** | Instinct MI350X/MI355X (CDNA 4) | Platform-Level DICE Identity | SEV-SNP (CPU-side) | Infinity Fabric Link encryption | SR-IOV |
| **Google** | TPU v7 Ironwood | Integrated hardware RoT, secure boot | Not publicly documented | Not publicly documented | Pod-level isolation |
| **Google** | A3 Confidential VMs (H100) | Intel TDX + NVIDIA CC composite attestation | Bounce buffer encryption (CPU-GPU) | PCIe encrypted via bounce buffers | Confidential VM isolation (spot/flex-start) |

**Key gaps:** AMD's GPU-side confidential computing does not yet match NVIDIA's CC mode -- AMD relies on CPU-side SEV-SNP for memory encryption rather than on-GPU encryption. The upcoming MI400 series (HBM4, 432 GB, late 2026) has not disclosed whether it will add GPU-side memory encryption comparable to NVIDIA's Blackwell CC mode. Google TPU attestation and encryption capabilities are not exposed as public APIs, limiting independent verification, though Google's A3 Confidential VMs with H100 GPUs provide an alternative path via NVIDIA CC mode on Google Cloud infrastructure. Intel's Gaudi 3 is available on IBM Cloud paired with Intel TDX at the CPU layer, but Intel has not published a GPU-side confidential computing spec comparable to NVIDIA CC mode or AMD's Platform-Level DICE -- the accelerator itself is not a TEE, so the trust chain terminates at the TDX-protected host.

**AMD MI355X production readiness (April 2026):** AMD's MLPerf Inference 6.0 submission (April 1, 2026) reported over 1 million tokens per second on Llama 2 70B and GPT-OSS-120B at multinode scale on MI355X platforms. This is significant from a security-architecture perspective: production deployment at that scale means AMD's DICE attestation + Infinity Fabric Link encryption + SR-IOV isolation stack is now operationally validated for regulated workloads, not just demonstrated. Organizations evaluating AMD for regulated AI inference now have a credible alternative to NVIDIA CC mode for workloads whose threat model is satisfied by CPU-anchored confidentiality (SEV-SNP) rather than requiring GPU-resident memory encryption.

**Driver vulnerability cadence:** Both NVIDIA and AMD disclosed high-severity GPU driver CVEs in Q1 2026 (NVIDIA: 5 CVEs in January 2026 including vGPU escape; AMD: CVE-2026-23213 SMU race condition in March 2026). Organizations should treat GPU driver patching with the same urgency as OS kernel patching -- these are kernel-mode components with direct memory access.

### CUDA Toolkit and Development Tool Supply Chain

The GPU software supply chain extends beyond drivers and firmware to the development toolchains used to build and deploy AI workloads. In January 2026, NVIDIA disclosed four vulnerabilities in CUDA Toolkit Nsight tools (CVE-2025-33228, CVE-2025-33229, CVE-2025-33230, CVE-2025-33231), including OS command injection and arbitrary code execution flaws with CVSS scores up to 7.3. Earlier, Palo Alto Unit 42 discovered nine vulnerabilities in cuobjdump and nvdisasm (CVE-2024-53870 through CVE-2024-53878) via fuzzing, demonstrating that CUDA binary parsing tools contain exploitable memory safety issues.

**Why this matters for AI security:** Compromised development tools can inject malicious code into CUDA kernels that ultimately execute on production GPUs with full hardware access. Unlike driver vulnerabilities (which affect the runtime), toolkit vulnerabilities affect the build pipeline -- a supply chain attack vector that is harder to detect because the malicious payload is baked into the compiled model inference code. Organizations should:

- Pin CUDA Toolkit versions and verify checksums before installation
- Isolate GPU development environments from production inference clusters
- Monitor for toolkit CVEs with the same urgency as driver and firmware CVEs
- Consider that CUDA Toolkit 13.1 (January 2026) addresses the latest batch, but the cadence of discovery suggests ongoing attention is needed

### NVIDIA AI Framework Supply Chain Vulnerabilities (Q1 2026)

The GPU supply chain attack surface extends well beyond drivers and CUDA tools to the AI training and inference frameworks that run on top of them. In late February and March 2026, NVIDIA disclosed a significant batch of vulnerabilities across its core AI software stack:

- **NVIDIA Apex (CVE-2025-33244, CVSS 9.0 Critical, March 2026):** Deserialization of untrusted data (CWE-502) in the Apex mixed-precision training library on Linux. Affects environments using PyTorch versions earlier than 2.6. An unauthorized attacker could achieve code execution, privilege escalation, and full system compromise. Fixed in Apex commit db8e053+.
- **NVIDIA NeMo Framework (8 CVEs, up to CVSS 8.0, February 2026):** CVE-2025-33245 (CVSS 8.0) enables remote code execution via malicious training data. CVE-2025-33236, CVE-2025-33241, CVE-2025-33243 (all CVSS 7.8) cover code injection and RCE in distributed training environments. Additional CVEs (CVE-2025-33246, CVE-2025-33249, CVE-2025-33250 through CVE-2025-33253) target ASR evaluator and voice-preprocessing components with command injection and code injection flaws. All fixed in NeMo 2.6.1.
- **NVIDIA Megatron Bridge (CVE-2025-33239, CVE-2025-33240, both CVSS 7.8, February 2026):** Code injection in data merging and shuffling tutorials. While labeled "tutorials," these components are routinely used in production data preprocessing pipelines for large-scale language model training. Fixed in Megatron-Bridge 0.2.2. Reported by TencentAISec.
- **NVIDIA Triton Inference Server (5 CVEs, April 2026):** NVIDIA's April 7, 2026 bulletin disclosed five server-side vulnerabilities in Triton. CVE-2026-24146 (CWE-789, CVSS 7.5) is a memory-allocation-with-excessive-size flaw where insufficient input validation on output counts crashes the server. CVE-2026-24173 (CWE-190, integer overflow), CVE-2026-24174 (CWE-681, numeric conversion error), and CVE-2026-24175 (CWE-248, uncaught exception) are all remote-unauthenticated DoS flaws triggered by malformed requests or request headers (each CVSS 7.5). CVE-2026-24147 (CWE-22, CVSS 4.8) is a path-traversal / information-disclosure in model-configuration upload that also enables DoS. All affect Triton versions prior to r26.02 -- fixed in r26.02. Triton is the default inference serving runtime for many cloud GPU deployments, so these bulk-scan DoS primitives are particularly dangerous when Triton endpoints are exposed to untrusted networks or multi-tenant ingress. Organizations should upgrade to r26.02 and verify that model-configuration upload endpoints require authentication.
- **NVIDIA DALI (CVE-2026-24156, CVSS 7.3, April 2026):** Deserialization of untrusted data (CWE-502) in the NVIDIA Data Loading Library used across CUDA-accelerated data preprocessing pipelines. Reported by Jaeyeong Lee; patched in DALI 2.0 (bulletin released April 7, 2026, updated April 14, 2026). Same pattern as the earlier Apex CVE-2025-33244 deserialization bug -- a malicious pickled artifact processed by DALI can yield arbitrary code execution on the training host. DALI is embedded in many reference training pipelines for computer-vision and audio workloads, so teams that pinned to older container images need to refresh.
- **Downstream advisories amplify triage pressure (mid-April 2026):** CERT-In issued CIVN-2026-0186 and Thailand's ThaiCERT issued a matching bulletin covering CVE-2026-24146, CVE-2026-24147, CVE-2026-24156, CVE-2026-24173, CVE-2026-24174, and CVE-2026-24175 (HIGH severity), mapping NVIDIA's Triton and DALI bulletins into regional regulator tracking. For organizations with procurement or regulatory mandates tied to national CERT advisories (financial services in India, critical infrastructure operators in APAC), the national-CERT rebroadcasts accelerate the patch SLA clock even when the upstream NVIDIA bulletins have been available for a week. Worth tracking because these downstream listings occasionally include additional impact context (e.g., information-disclosure framing on CVE-2026-24147) that the original vendor bulletin treats more narrowly.

These vulnerabilities are particularly concerning because they target the training pipeline rather than inference runtime -- a compromised training framework can poison model weights at the source, making the resulting model a persistent backdoor regardless of how securely the inference environment is configured. The cadence (at least 17 CVEs across 5 frameworks -- Apex, NeMo, Megatron Bridge, DALI, and Triton -- between February and April 2026) suggests that NVIDIA's AI framework code has not undergone the same security scrutiny as its driver and firmware code. Organizations running large-scale training or managed inference on NVIDIA infrastructure should treat framework updates with the same urgency as kernel-mode driver patches, and should consider network-level isolation for Triton endpoints until they are confirmed on r26.02.

**Triton May 2026 follow-up bulletin (a_id/5828, r26.03 fix):** NVIDIA's second Triton Inference Server bulletin (May 18-20, 2026) is materially worse than the April batch: it covers eight CVEs headlined by **CVE-2026-24207, a CVSS 9.8 Critical unauthenticated authentication bypass (CWE-288)** -- network-exploitable with low complexity and no user interaction, enabling code execution, privilege escalation, data tampering, DoS, and information disclosure on the inference host. A second authentication bypass (CVE-2026-24206, CVSS 7.3) ships in the same bulletin, alongside three DALI backend flaws (CVE-2026-24213 out-of-bounds read enabling code execution, CVE-2026-24214 integer overflow, CVE-2026-24215 uncontrolled resource consumption -- up to CVSS 8.0) and path-traversal/integer-overflow DoS issues (CVE-2026-24208, CVE-2026-24209, CVE-2026-24210). All are fixed in Triton Server r26.03. The original April r26.02 fix did not close the full set, and CERT-In and ThaiCERT have continued rebroadcasting the Triton CVE stream into national advisories. The escalation from April's DoS-class primitives to a May unauthenticated 9.8 auth bypass on the serving plane means network-level isolation (no public internet exposure, mTLS or service-mesh authentication in front of every Triton endpoint, per-tenant rate limiting on inference requests) is a required compensating control, and any Triton endpoint that was reachable from untrusted networks before the r26.03 upgrade should be treated as potentially compromised and investigated, not just patched. No active exploitation had been reported at publication time.

**NVIDIA BioNeMo CVE-2026-24164 (March 31, 2026):** A high-severity (CVSS 8.8) insecure-deserialization vulnerability in the BioNeMo training framework, exploitable over the network with no authentication, enabling remote code execution, DoS, information disclosure, and data tampering on the training host. BioNeMo is widely used for drug-discovery and protein-modeling training pipelines, so the practical risk is a malicious dataset or checkpoint compromising a long-running HPC training job and the GPU host underneath it. Fixed at commit e5e58c8+ in the NVIDIA/BioNeMo Framework GitHub repository.

**NVIDIA NVFlare CVE-2026-24178 (April 28, 2026, Critical):** A CVSS 9.8 unauthenticated authorization-bypass vulnerability (CWE-639) in the NVIDIA FLARE Dashboard's user-management and authentication system allows an attacker to manipulate authentication parameters and gain unauthorized access to protected resources, leading to privilege escalation, data tampering, information disclosure, code execution, and DoS. A second CVE in the same disclosure window, CVE-2026-24186 (High), is an insecure-deserialization flaw in the FLARE SDK's FOBS (Federated Object Binary Serialization) component, where a malicious FOBS-encoded message can yield arbitrary code execution. Both are fixed in NVIDIA FLARE 2.7.2. For organizations running federated-learning workloads on GPU clusters, this is the highest-severity FLARE finding to date and means any FLARE Dashboard reachable from the network (including from trusted internal subnets) should be treated as compromised until upgraded and audited. The combination of an unauthenticated CVSS 9.8 federation-control-plane bug and a deserialization RCE in the federated-object protocol is particularly bad because successful exploitation can poison the shared model across all participating sites.

**Pattern across Q1-Q2 2026:** With BioNeMo, NVFlare, Apex, NeMo, Megatron Bridge, DALI, and Triton now all subject to high- or critical-severity CVEs in three months, the AI-framework layer has emerged as the dominant entry point for compromising GPU infrastructure -- exceeding driver, firmware, and even container-toolkit CVE volume. Auditors should add framework-CVE patch currency to their hardware-security checklist alongside driver patch level, because a compromised training framework can corrupt model weights before any hardware-attestation flow ever sees the artifact.

### Confidential GPU Marketplace and Third-Party Attestation

A new category of infrastructure is emerging: third-party confidential GPU marketplaces that provide hardware-signed attestation for multi-tenant GPU workloads. As of early 2026, NEAR AI launched the Confidential GPU Marketplace at NEARCON 2026, offering TEE-secured compute where every request returns hardware-signed attestation in under 30 seconds. Their IronClaw runtime (open-source, Rust-based) deploys inside encrypted TEEs with credential isolation and WebAssembly-sandboxed tools.

This model shifts GPU attestation from a cloud-provider-managed service to a verifiable, third-party-auditable primitive. For organizations evaluating multi-cloud or sovereign AI deployments, third-party attestation marketplaces may offer an alternative to vendor-locked confidential computing stacks. However, the security guarantees depend entirely on the underlying hardware TEE (currently Intel TDX + NVIDIA CC), which inherits the same limitations (such as TEE.Fail's memory-bus attack surface) as direct cloud provider offerings.

### Confidential Containers and Zero-Trust AI Factories (2026)

A practical deployment pattern is emerging for GPU confidential computing: confidential containers (CoCo) running on Kubernetes with hardware TEE isolation. This approach wraps standard Kubernetes pods in lightweight utility VMs using Kata Containers, eliminating the host OS and hypervisor from the trust boundary without requiring application code changes.

The attestation flow follows the IETF RATS (Remote Attestation Procedures) architecture: an Attestation Agent inside the CoCo collects cryptographic hardware evidence (CPU quotes from Intel TDX or AMD SEV-SNP, GPU attestation reports from NVIDIA CC), sends them to an Attestation Service that validates against known-good reference measurements via a Reference Value Provider Service, and a Key Broker Service releases decryption keys only after successful verification. Secrets are exposed exclusively inside the protected memory boundary via a Confidential Data Hub. The guest VM runs a distro-less, minimal OS with a chiseled root filesystem, and NVIDIA's Runtime Container (NVRC) serves as the secure init system to minimize the in-VM attack surface.

**Red Hat OpenShift support (2025-2026):** Red Hat OpenShift sandboxed containers v1.10.0 shipped with generally available confidential container support for Intel TDX and AMD SEV-SNP on Azure, with bare metal and additional cloud support planned. RHEL has committed to day-zero support for NVIDIA's Rubin GPUs and their confidential computing framework, providing cryptographic proof of workload protection across the full AI lifecycle.

**Cisco Secure AI Factory (GTC March 2026):** Cisco expanded its Secure AI Factory partnership with NVIDIA to enable zero-trust AI infrastructure from data center to edge. Key security features include: Cisco Hybrid Mesh Firewall enforcing policy on NVIDIA BlueField DPUs embedded in GPU servers (blocking threats at the server level before they reach data), Cisco AI Defense providing automated vulnerability testing and model security, and integration with NVIDIA NeMo Guardrails for agent security at the edge. The architecture compresses AI deployment from months to weeks while embedding security from the start. This represents the first major enterprise networking vendor integrating GPU-level confidential computing into a turnkey infrastructure product.

### AI Chip Supply Chain Security

As AI accelerators become critical infrastructure, supply chain integrity is an emerging concern:

- **Hardware trojans:** Malicious modifications embedded during chip design or fabrication can exfiltrate data or weaken security. AI-driven detection methods using LLMs to scan chip designs are being researched (University of Missouri, 2025).
- **Hardware-Enabled Mechanisms (HEMs):** CNAS proposes embedding governance features directly into AI chips -- location verification, compute metering, offline licensing, and bandwidth controls -- to enable post-export enforcement without remote kill switches.
- **Domestic manufacturing:** CHIPS Act mega-fabs beginning high-volume production in 2026 aim to reduce dependency on overseas fabrication, but supply chain verification from design to deployment remains critical.
- **Standards development:** NIST is recommended to develop AI hardware security standards extending FIPS 140-3, in coordination with the Open Compute Project and semiconductor industry consortia. **NIST IR 8596 Cyber AI Profile draft (preliminary, December 2025):** NIST released a preliminary draft of the Cybersecurity Framework Profile for Artificial Intelligence, overlaying three AI focus areas (Secure, Detect, Thwart) on CSF 2.0 outcomes. The 45-day public comment period ran through January 30, 2026, with an initial public draft expected later in 2026. Hardware-relevant content includes AI-specific considerations within the supply chain (Govern) and protection (Protect) functions and explicit language on using AI to "forecast hardware failures and system degradation" within the Detect function. Organizations operating GPU fleets for regulated AI workloads should track the initial public draft for mapped hardware security controls.

### Compliance Framing for GPU Memory and Multi-Tenant Isolation (April 2026)

The CSA Research Note on GPUBreach (April 8, 2026) is the first major compliance-oriented analysis to treat GPU memory architecture as an auditable security property. Key positions worth citing in internal risk registers and auditor conversations:

- **GPU co-tenancy is no longer presumed isolated.** CSA states that "co-tenant GPU workloads should not be treated as inherently isolated from one another absent verified ECC enforcement and confirmed patch status." This invalidates the common deployment assumption that IOMMU plus container/VM boundaries is sufficient for multi-tenant GPU inference.
- **Memory family as a procurement security criterion.** GDDR6 is now categorized by CSA as a distinct security risk compared to GDDR7 and HBM3e. Procurement specifications for regulated AI workloads should call out memory generation and default ECC posture, not just raw performance.
- **Runtime model-integrity verification as a compensating control.** Where migration off GDDR6 is infeasible, CSA recommends loading model weights with a cryptographic hash at startup and periodically re-hashing against a known-good baseline to detect silent corruption caused by Rowhammer-induced flips. This does not prevent corruption but provides detection with modest operational overhead.
- **Explicit mapping to SOC 2, ISO 42001, and NIST AI RMF.** Organizations operating under these frameworks should document GPU Rowhammer (GPUHammer, GDDRHammer, GeForge, GPUBreach) as a residual risk in their compliance registry until their GPU fleet is confirmed on ECC-enabled Hopper/Blackwell or HBM3e/GDDR7 hardware. CSA has proposed that future STAR for AI assessments should incorporate attestation requirements covering GDDR6 ECC status, GPU driver patch currency, and multi-tenant isolation policies as first-class controls -- signalling that GPU hardware posture may soon move from "infrastructure hygiene" to an explicit cloud-security assurance deliverable.
- **Cloud-customer verification responsibility.** Default ECC enablement varies across cloud providers and instance types. CSA guidance is that cloud customers must verify ECC enforcement directly with their provider rather than relying on defaults -- a reasonable question to add to any cloud GPU vendor security questionnaire going forward.

## Open Research Questions

- [ ] How effective are GPU power/EM side-channel attacks against production AI workloads in data center environments (as opposed to edge devices), and what countermeasures beyond physical isolation are practical at scale?
- [ ] What is the real-world performance overhead of NVIDIA Blackwell Confidential Computing with TDISP/IDE inline encryption across different model architectures, batch sizes, and multi-GPU training workloads?
- [ ] Will the Vera Rubin NVL72 rack-scale confidential computing model become the baseline expectation for multi-tenant AI infrastructure, and how will competing platforms (AMD, Intel) respond?
- [ ] How should GPU.zip-class browser-based side channels be mitigated -- at the GPU hardware level (disabling compression), browser level (restricting GPU access), or application level (preventing cross-origin iframe rendering)?
- [ ] Can Hardware-Enabled Mechanisms (HEMs) for AI chip governance be designed to resist sophisticated adversaries with physical chip access while avoiding the civil liberties concerns of government backdoors?
- [ ] How will on-chip export control mechanisms (compute metering, location verification, offline licensing) interact with AI hardware security requirements, and could they introduce new attack surfaces?
- [ ] What monitoring strategies can compensate for the loss of performance counter visibility when GPUs operate in CC-On mode, where telemetry is intentionally restricted?
- [ ] As of early 2026, AMD (DICE attestation, Infinity Fabric encryption) and Google (TPU Ironwood integrated RoT) have introduced hardware security primitives -- but can they achieve parity with NVIDIA's end-to-end CC mode, and will Intel Gaudi publish comparable security specifications?
- [ ] Given TEE.Fail's demonstration that DDR5 memory-bus interposition can forge attestation quotes, what architectural changes (e.g., integrity trees, non-deterministic encryption modes) would be needed to harden CPU TEEs against physical-access adversaries, and how would this affect GPU CC trust chains?
- [ ] Will GPUHammer-class Rowhammer attacks evolve to produce multi-bit flips that bypass on-die ECC in newer GPU architectures (HBM3, GDDR7), and should cloud providers mandate ECC enablement for all shared GPU workloads despite the performance penalty?
- [ ] How should organizations balance the systemic risk of NVIDIA Container Toolkit vulnerabilities (CVE-2025-23266) against the operational dependency on containerized GPU workloads, and are micro-VM or unikernel isolation approaches mature enough for production AI workloads?
- [ ] As confidential GPU compute becomes available via third-party attestation marketplaces (e.g., NEAR AI's Confidential GPU Marketplace), how should organizations evaluate the trust chain differences between cloud-provider-managed CC and third-party-brokered CC, particularly regarding attestation quote provenance and key management?
- [ ] The CUDA Toolkit supply chain (Nsight tools, cuobjdump, nvdisasm) has shown a pattern of memory safety and command injection vulnerabilities -- should critical AI inference pipelines require formally verified or memory-safe GPU development toolchains, and what would the migration cost be?
- [ ] Will AMD's MI400 series (HBM4, late 2026) introduce GPU-side memory encryption comparable to NVIDIA's Blackwell CC mode, or will AMD continue to rely on CPU-side SEV-SNP as the primary confidential computing mechanism for GPU workloads?
- [ ] Given the implementation-level security gaps identified in NVIDIA GPU-CC (unencrypted RPC metadata, timing side-channels on memory transfers, accessible BAR0 registers), what residual information leakage risks remain for organizations deploying confidential computing for high-sensitivity AI workloads, and will NVIDIA address these in the Vera Rubin generation?
- [ ] The Q1 2026 wave of NVIDIA AI framework CVEs (11 vulnerabilities across Apex, NeMo, and Megatron Bridge) highlights a pattern of insecure deserialization and code injection in training pipelines. Should organizations require sandboxed execution environments for all AI training framework components, and is the overhead of doing so practical at scale?
- [ ] The April 2026 GDDRHammer / GeForge / GPUBreach disclosures collapsed the gap between "theoretical GPU Rowhammer" and "arbitrary host read/write via PCIe BAR1." GPUBreach further showed that IOMMU is not a sufficient mitigation when combined with a driver memory-safety flaw. Should cloud providers mandate ECC and restrict GDDR6 GPU co-tenancy outright for multi-tenant inference, and what is the minimum acceptable firmware/driver cadence for any GPU running shared tenant workloads?
- [ ] As hardware-attested AI agent runtimes (EQTY Verifiable Runtime, NVIDIA Confidential Computing on BlueField DPUs) move policy enforcement into the DPU, how should auditors verify the combined trust chain across CPU TEE + GPU TEE + DPU attestation, and are the tooling and reference measurements mature enough to make this operationally feasible outside hyperscale environments?
- [ ] The IETF RATS Attested Inference Receipt draft (March 2026) defines profiles for AWS Nitro Enclaves and Intel TDX but no GPU-specific measurement profile. What would a portable AIR profile for NVIDIA Hopper/Blackwell and AMD MI355X DICE attestation look like, and who should drive that standardization -- the CCC Attestation SIG, the IETF RATS working group, or GPU vendors directly?
- [ ] Given the cadence of NVIDIA AI framework CVEs in Q1-Q2 2026 (Apex, NeMo, Megatron Bridge, DALI, Triton -- at least 17 CVEs in three months), how should organizations architect their inference serving so that Triton-class DoS primitives (CVE-2026-24146/24173/24174/24175) cannot be used to bulk-crash a multi-tenant GPU fleet from a single malformed request?
- [ ] When a GPU vendor (NVIDIA, in this case) publicly classifies a new attack class as a "reinforcement of existing mitigations" rather than issuing a CVE and a targeted bulletin -- as happened with GPUBreach/GDDRHammer/GeForge against the July 2025 Rowhammer notice -- how should enterprise patch-management and vulnerability-scanning workflows detect and prioritize these "no-CVE" GPU vulnerabilities? Is there a need for a third-party (CSA, MITRE) to mint tracking identifiers for attacks that vendors decline to assign CVEs to?
- [ ] If GPU Dynamic Resource Allocation, KAI Scheduler queues, and ComputeDomains become the standard Kubernetes abstraction for multi-node accelerators, what admission-control and audit evidence should prove that a tenant cannot request unauthorized GPU slices, PodGroups, or NVLink/IMEX domains across namespace boundaries?
- [ ] With Jetson AGX Thor enabling Blackwell-class Confidential Computing at the edge (April 2026), what reference measurements and attestation verification infrastructure are required for regulated edge AI (medical appliances, industrial control, autonomous systems) to claim equivalent assurance to hyperscaler H100/B200 confidential inference, and who is responsible for operating the Reference Value Provider Service outside a hyperscale context?
- [ ] Should CSA STAR for AI attestation requirements for GPU hardware (ECC enablement, driver patch currency, multi-tenant isolation posture) be aligned with the IETF RATS Attested Inference Receipt profile, and what minimum set of per-request attested evidence should a cloud GPU tenant expect their provider to emit?
- [ ] MITRE ATLAS v5.5.0 (March 2026) catalogs agent-focused AI techniques including `Escape to Host` but does not yet assign individual technique IDs to GPU-specific attacks (LeftoverLocals, GPU.zip, GPUHammer, GPUBreach, GDDRHammer, GeForge, BarraCUDA, TEE.Fail, NVIDIAScape). Should a GPU/accelerator sub-technique family be added to ATLAS, and if so, should it sit under AML.TA0004 Initial Access (for co-tenant attacks), AML.TA0009 Collection (for side-channel extraction), or a new hardware-attack tactic? - [ ] With BioNeMo (CVE-2026-24164, CVSS 8.8 RCE) and NVFlare (CVE-2026-24178, CVSS 9.8 unauthenticated auth bypass; CVE-2026-24186 deserialization RCE) joining the Q1-Q2 2026 framework CVE wave, federated learning and bio-AI training now share the same insecure-deserialization and auth-bypass pattern as the rest of the NVIDIA AI stack. - [ ] The "Behind Bars" disclosure (2026) demonstrated a cache-partitioning side channel across H100 MIG boundaries via memory barriers, yet the research is so far only visible through AMD's response bulletin (AMD-SB-6026) -- NVIDIA has published neither a bulletin nor a CVE.
