# C3.5: Hosted and Provider-Managed Model Controls

> [Back to C03 Index](C03-Model-Lifecycle-Management.md)
> **Last Researched:** 2026-05-13

## Purpose

Hosted and provider-managed models may change behavior without notice. These controls help ensure visibility, reassessment, and safe operation when the organization does not control the model weights.

As of May 2026, the practical risk is no longer hypothetical. OpenAI's GPT-4o sycophancy rollback showed that an incremental model-behavior update can materially change safety posture even when the product name remains familiar; OpenAI's follow-up postmortem also said subtle model changes need more proactive communication and should treat behavior issues as launch-blocking concerns. Anthropic's Claude Code changelog records a March 4, 2026 configuration change where Opus 4.6 defaulted to `medium` effort for Max and Team subscribers, illustrating that hosted-model behavior can shift through configuration as well as model weights.

Provider lifecycle transparency has improved in some places, but it is still uneven. Microsoft Foundry publishes lifecycle stages and retirement tables, Amazon Bedrock exposes `modelLifecycle` through `GetFoundationModel` and `ListFoundationModels`, Google Vertex AI publishes stable-model retirement dates and auto-updated aliases, OpenAI and Anthropic expose model identifiers in API responses, and observability tools increasingly support OpenTelemetry GenAI attributes. The remaining control gap is that no cross-provider protocol guarantees immutable model identity, routing transparency, or advance notification for configuration-level changes.

## Verification Notes

For audits, the strongest evidence is a joined trail across four sources: a hosted-model inventory, production inference logs, gateway route history, and the evaluation record that approved the model/configuration for the workflow. The auditor should be able to pick one high-risk output and answer: which provider handled it, which model identifier the provider returned, which route or fallback policy selected it, which request configuration was active, which eval bundle approved that configuration, and what would have happened if the provider failed to expose enough identity data.

Requested model names are not reliable evidence by themselves. Treat aliases, deployment names, and gateway route labels as local pointers that must resolve to provider-returned identity, provider request IDs, and routing history at the time of the request.

## Provider Signal Checklist

For each production hosted-model call path, retain enough evidence to reconstruct these fields:

- Requested provider, endpoint, region, tenant, and local deployment or gateway route.
- Requested model alias and provider-returned model, deployment, or version identifier.
- Provider request ID, response ID where available, and backend fingerprint or service tier where exposed.
- Lifecycle state, retirement date, replacement model, and notification source for hosted catalog entries.
- Gateway fallback decision, policy rule, and approved fallback target when routing changes.
- Safety-relevant request configuration, including tools, system prompt hash, effort/reasoning mode, temperature, and top-p.
- Evaluation bundle, approval record, and exception expiry tied to the exact hosted-model configuration.
