# C05: Access Control & Identity for AI Components & Users

> **Source:** [`1.0/en/0x10-C05-Access-Control-and-Identity.md`](https://github.com/OWASP/AISVS/blob/main/1.0/en/0x10-C05-Access-Control-and-Identity.md)
> **Requirements:** 11 | **Sections:** 3

## Control Objective

AI systems introduce access control challenges beyond traditional application security: classification labels must follow data through AI-specific transformations (embeddings, caches, model outputs), multi-tenant inference infrastructure creates novel side channels, and retrieval-augmented pipelines must enforce caller entitlements at every stage. This chapter addresses AI-specific access control and identity concerns, including runtime isolation of the policy decision point from agent execution and authorization-aware output filtering where entitlements vary per caller.

## Requirement Research Snapshot

| # | Requirement | Level | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|-----------------|----------------------|--------------|
| 5.1.1 | Verify that high-risk AI operations (model deployment, weight export, training data access, production configuration changes) require step-up authentication. | 3 | Stolen operator sessions, prompt-driven deployment changes, and compromised admin consoles turning into model or data exfiltration. | Inspect IdP and PAM policies for model deployment, weight export, training-data access, and production config paths; test WebAuthn/passkey or equivalent step-up prompts; confirm audit logs record the human approver and target asset. | CI/CD, notebooks, and agent service accounts often bypass human step-up paths unless privileged actions are routed through the same approval gateway. |
| 5.1.2 | Verify that AI agents in federated or multi-system deployments authenticate using short-lived, minimal-scoped, cryptographically signed tokens. | 3 | Bearer-token theft, static API key replay, agent impersonation, and confused-deputy delegation across tools or tenants. | Review OAuth/OIDC, SPIFFE/SVID, WIMSE, or token-exchange configuration; verify token lifetime, audience, issuer, scope, JWKS or x5c validation, DPoP or mTLS proof where used, and replay rejection at each resource server. | IETF agent-auth drafts are active but still work in progress; many deployments still use long-lived service-account keys or shared API tokens. |
| 5.2.1 | Verify that every AI resource (datasets, endpoints, vector collections, embedding indices, compute instances) enforces access controls with explicit allow-lists and default-deny policies. | 2 | Public model endpoints, overbroad vector-store namespaces, orphaned datasets, and compute resources reachable without explicit entitlement. | Inventory AI resources from cloud IAM, vector DB admin APIs, model registries, and Kubernetes; sample deny-by-default behavior with a non-entitled user; review allow-list ownership and approval history. | AI assets are frequently created by pipelines outside the CMDB, so the control depends on discovery as much as policy syntax. |
| 5.2.2 | Verify that retrieval pipelines (e.g., RAG queries, embedding lookups) enforce the end-user's authorization context at each retrieval and assembly stage, rather than relying solely on the service account's permissions. | 2 | RAG cross-tenant leakage where the service account can retrieve documents the requester cannot see. | Trace the caller identity through retriever, reranker, metadata filter, chunk assembly, and citation generation; run negative tests with same query text under users with different entitlements; inspect server-side filters, not client-side filtering alone. | Most RAG frameworks make retrieval easy but authorization context propagation remains application-specific, especially across hybrid search and reranking. |
| 5.2.3 | Verify that sensitive data is retrieved via retrieval pipelines (e.g., RAG queries, embedding lookups) to prevent permanent storage in models. | 2 | Sensitive data memorized through fine-tuning, prompt-cache reuse, or training corpus contamination instead of staying behind authorization-aware retrieval. | Review training and fine-tuning manifests, DLP scans, and data-source allow-lists; confirm sensitive collections are referenced through retrieval connectors; test that fine-tune jobs reject restricted datasets. | Proving a model did not memorize data is hard; verification should focus on provenance, ingestion gates, and retrieval-only handling for known sensitive sources. |
| 5.2.4 | Verify that post-inference filtering mechanisms prevent responses from including data that the requestor is not authorized to receive. | 2 | Prompt-injection-driven exfiltration, unauthorized citations, and model synthesis that reveals restricted facts through otherwise permitted channels. | Run red-team prompts against users with partial access; validate output chunks and cited sources against caller entitlements before delivery; review DLP, policy engine, and response-filter logs for block decisions. | DLP false negatives and agent-initiated channels remain weak points; filters need source-level authorization evidence, not just pattern matching. |
| 5.2.5 | Verify that the policy decision point for agent authorization is isolated from the agent's execution environment. | 2 | A compromised agent runtime bypassing or rewriting its own authorization checks before invoking tools. | Confirm PDP execution in an external gateway, sidecar, or service; verify signed policy bundles and immutable decision logs; test tool calls where the agent tampers with local policy state or sends unstructured reasoning instead of an action request. | Many frameworks still enforce policy inline with tool code, so isolation often requires additional gateway or service-mesh architecture. |
| 5.2.6 | Verify that privileged access to model weights, training pipelines, and production AI configuration is granted just in time, with a defined maximum session duration and automatic expiry. Zero Standing Privilege (ZSP) to these resources is encouraged. | 3 | Persistent admin tokens, abandoned pipeline privileges, and long-lived access to model weights or production config after the task ends. | Inspect PAM/PIM grants, approval records, maximum session durations, revocation events, and break-glass reviews; test that expired sessions cannot export weights or change deployment config. | Pipeline credentials and emergency accounts commonly outlive the work item unless expiration is enforced by the identity plane. |
| 5.2.7 | Verify that data classification labels propagate to downstream resources (embeddings, prompt caches, model outputs). | 3 | Label stripping during chunking, embedding, cache reuse, summarization, or generated output that lets restricted data lose its handling requirements. | Trace labels from source documents through ETL, vector metadata, cache keys, prompts, response objects, and logs; sample derived artifacts for retained sensitivity metadata; test deny decisions based on propagated labels. | Label fidelity in embeddings and natural-language outputs is still imprecise; systems need conservative defaults when lineage is missing. |
| 5.3.1 | Verify that shared model serving infrastructure prevents one tenant's fine-tuning, inference, or embedding operations from influencing or observing another tenant's operations. | 2 | Cross-tenant adapter influence, shared model-state contamination, embedding index bleed, and KV-cache side channels in shared serving. | Review tenant isolation for LoRA/adapters, embedding indices, model routing, and prefix/KV-cache reuse; test tenant-specific cache salts or namespace separation; inspect scheduler and cache metrics for cross-tenant hits. | Performance optimizations such as prefix caching create pressure to share state; isolation needs explicit trust-group boundaries. |
| 5.3.2 | Verify that one tenant cannot influence or observe another tenant's operations through shared compute resources. Satisfying this requirement typically requires hardware partitioning, confidential computing, or dedicated per-tenant compute allocation. | 3 | GPU memory leakage, timing channels, TLB/NVLink contention, and other co-residency side channels across shared inference hardware. | Validate tenant placement policy, MIG or SR-IOV configuration, confidential-computing attestation, firmware/driver patch status, and dedicated allocation for hostile tenants; run co-residency tests where feasible. | No single software-only control is sufficient for hostile co-tenancy; MIG, TEEs, and dedicated compute each have operational and residual-risk tradeoffs. |

- Privilege escalation through prompt injection causing the model to act with elevated permissions
- Cross-tenant data leakage in shared AI infrastructure (e.g., 2025 Salesloft-Drift breach affecting 700+ orgs)
- Agent identity spoofing in multi-agent systems; interagent trust gaps across organizational boundaries
- Confused deputy attacks where AI tools act with the user's credentials beyond intended scope

- **[2026-05]** IETF agent-authorization work expanded: `draft-araut-oauth-transaction-tokens-for-agents-01` adds `actor` and `principal` context fields to OAuth Transaction Tokens so the agent and the human principal are both carried into each call; `draft-embesozzi-oauth-agent-native-authorization-00` defines a Human-to-Agent (H2A) OAuth flow with structured elicitation so a first-party agent client can request and re-request consent without falling back to redirect-based flows; `draft-song-oauth-ai-agent-collaborate-authz-00` adds an `applier_id` claim for multi-agent collaboration so a lead agent can request tokens for downstream sub-agents that the authorization server attributes correctly.
- **[2026-05]** AIMS (Agent Identity Management System) IETF Internet-Draft (March 2026) composes WIMSE, SPIFFE, and OAuth 2.0 into an end-to-end agent identity framework — relevant to 5.1.2 because it specifies how SVIDs feed into the signed-token assertions that resource servers must validate against issuer-controlled keys.
- **[2026-05]** CVE-2026-32173 (Azure SRE Agent, April 2026; Microsoft CVSS 8.6 / NVD 7.5) showed that "signature valid" plus "audience correct" is not enough at the resource server. The agent's SignalR `/agentHub` endpoint accepted any token issued by any Entra ID tenant because the underlying app registration was configured multi-tenant. Once connected, the hub broadcast every user prompt, internal reasoning trace, command, and command output to all clients with no per-tenant or per-resource role filter; a ~15-line PoC was enough to eavesdrop on every deployed SRE Agent. This is why 5.1.2 evidence should include tenant and resource binding, not just JWT signature, before federated agent tokens are honored.
- **[2026-05]** Cisco Outshift's "MemoryTrap" (April 14 2026) showed that a poisoned `MEMORY.md` / `CLAUDE.md` could persist across every session, project, and reboot of Claude Code. The poisoned agent then advised users to commit API keys into source files and disabled `.env` usage. Anthropic removed the capability from the system prompt in Claude Code v2.1.50. The incident pulls agent memory and scratchpad files into the label-propagation evidence expected by 5.2.7 and the output-entitlement scope of 5.2.4.
- **[2026-05]** Capsule Security's "ShareLeak" and "PipeLeak" research demonstrated that Microsoft's January patch for CVE-2026-21520 (Copilot Studio indirect prompt injection, CVSS 7.5) did not stop data exfiltration in their PoC: a public-facing SharePoint form field smuggled instructions into the agent context, and the agent then exfiltrated customer data through its authorized SharePoint Lists → Outlook chain. A parallel finding in Salesforce Agentforce ("PipeLeak") has no CVE assigned. The takeaway is that post-inference filtering under 5.2.4 must cover agent-initiated channels, not just the final chat transcript.
- **[2026-05]** CERT/CC VU#221883 (publicly disclosed Mar 26 2026) chained four CrewAI flaws into agent host takeover: CVE-2026-2275 (Code Interpreter Tool fallback to SandboxPython enabling RCE via arbitrary C calls), CVE-2026-2287 (insecure fallback when Docker is unavailable at runtime), CVE-2026-2286 (RAG search SSRF via unvalidated URLs), and CVE-2026-2285 (arbitrary local file read in the JSON loader). - **[2026-05]** Lyrie's Agent Trust Protocol (ATP, public release May 11 2026) proposes an open cryptographic standard for AI agent trust organized around five primitives — Identity (who the agent is), Scope (what it is authorized to do), Attestation (whether the agent or its instructions have been tampered with), Delegation (who delegated authority), and Revocation (whether that authority has been withdrawn). The reference implementation is MIT-licensed (github.com/OTT-Cybersecurity-LLC/lyrie-ai), the protocol is royalty-free, and IETF submission is planned. ATP targets the same scope-attenuation and cascade-revocation gaps that AATs, applier_id, and ZeroID address, but pulls attestation in as a first-class primitive — a piece largely absent from the OAuth-extension drafts.

- **[2026-06]** Shared-compute side channels have no software-only mitigation in current shared inference servers — hardware partitioning (NVIDIA MIG, SR-IOV virtual functions), confidential computing (NVIDIA Confidential Computing on H100+, AWS Nitro Enclaves), or dedicated per-tenant compute is the expected evidence. The attack record backs this up: LeftoverLocals (CVE-2023-4969) recovered ~5.5 MB per GPU invocation of another process's local memory on AMD/Apple/Qualcomm GPUs — enough to reconstruct LLM responses; "TunneLs for Bootlegging" (ACM CCS 2023) built a 31 kbps covert channel across NVIDIA MIG partitions on A100/A30 because the last-level TLB is not partitioned; NVBleed (2025) demonstrated cross-VM side channels over NVLink interconnects on GCP; and TDXRay (IEEE S&P 2026) reconstructed user prompts word-for-word from an encrypted Intel TDX VM by observing which cache lines the tokenizer touched — TEEs narrow but do not eliminate the co-residency problem.
- **[2026-06]** BadHost (CVE-2026-48710, disclosed Jun 1 2026): Starlette — 325M weekly downloads, the framework underneath much of the Python AI serving stack — reconstructs `request.url` by concatenating the unvalidated Host header with the path, letting malformed Host values bypass path-based access controls. Found during a vLLM source audit; MCP servers are singled out because the MCP spec mandates unauthenticated OAuth discovery endpoints, giving attackers a reliable exploitation path. Researchers demonstrated chains to authentication bypass, SSRF, and RCE in popular projects gating security decisions on `request.url`. Fixed in Starlette 1.0.1.
- **[2026-06]** MCP exposure measured at scale: Censys counted 12,520 internet-accessible MCP services, most unauthenticated, many exposing database-query and command-execution tools; the first large-scale study of remote MCP servers found ~40% expose tools with no authentication and traced nine CVEs to broken OAuth flows; the VIPER-MCP framework swept ~40,000 server repositories and produced 106 zero-days / 67 CVEs; Akamai disclosed three database-MCP flaws (SQL injection in Apache Doris MCP, unauthenticated metadata exfiltration in Alibaba RDS MCP, takeover in Apache Pinot MCP — one vendor declined to patch); Trend Micro found 1,467 exposed servers plus CVSS 9.8 command injection in unofficial AWS and Azure MCP implementations; and the NSA published design-considerations guidance for locking MCP down.
- **[2026-06]** CSA's "Attribution Gap" analysis (May 26 2026) maps eight regulatory frameworks — SOX, CCPA, SEC disclosure rules, GDPR, NIS2, DORA, the Colorado AI Act (effective Jun 30 2026), and the EU AI Act high-risk tier (Dec 2 2027) — back to the same five agent capabilities: identify the acting agent, scope its access, trace authorization to a named human, verify both agent and user permissions before data moves, and log immutably. Case law is already filling the gap: Air Canada's chatbot defense was rejected, Garcia v. Character Technologies proceeded on product liability, and Italy fined Replika EUR 5M over lawful-basis failures.
- **[2026-06]** Production agent identity is converging on per-agent cryptographic identifiers: Google's Gemini Enterprise Agent Platform now assigns every agent a unique cryptographic ID referenced on every action and mapped to traceable authorization policies, while IBM (watsonx), Auth0 (CIBA), and Yubico demonstrated a human-in-the-loop authorization pattern requiring hardware-backed, cryptographically verified human approval for high-stakes agent actions. - **[2026-06]** IETF `draft-klrc-aiagent-auth-02` updates the agent-authentication profile around WIMSE/SPIFFE workload identifiers, short-lived SVID/WIT credentials, OAuth token exchange, Transaction Tokens, OpenID SSF, and runtime observability. - **[2026-06]** Google Cloud Agent Identity documentation now frames agent identity as a distinct workload identity with automatic credential rotation, SPIFFE identifiers, OAuth on-behalf-of brokering, and principal access boundaries. That gives auditors concrete artifacts for 5.1.2: agent principal URI, certificate/token lifetime, scope boundaries, and per-action attribution.
- **[2026-06]** vLLM's automatic prefix-caching design now documents `cache_salt` for shared environments and notes that cryptographically weak hash choices can leak private information in multi-tenant settings. For 5.3.1, auditors should verify tenant/session-derived salts, SHA-256 or deterministic SHA-256/CBOR hash choices, and cache-hit metrics segmented by trust group.

## Tooling & Implementation

Current tools, frameworks, and libraries that help implement these controls:

- **Policy engines:** OPA (Open Policy Agent, CNCF graduated), Cedar (AWS, CNCF Sandbox since Jan 2026; integrated into Amazon Bedrock AgentCore for agent-level policy with static analysis), Casbin, OpenID AuthZEN, Permit.io (fine-grained RBAC/ABAC/ReBAC across apps, APIs, data layers, AI agents; powers OPA/Cedar-style policies via OPAL)
- **Policy administration:** OPAL (Open Policy Administration Layer) for keeping policies consistent across microservices and AI agents
- **Identity:** SPIFFE/SPIRE for workload identity, OAuth 2.0 for user-facing AI apps, OIDC-A (OpenID Connect for Agents) 1.0 proposal, ARIA (Agent Relationship-Based Identity and Authorization), Microsoft OAuth 2.0 on-behalf-of token exchange for agent delegation. For 5.1.2, signed agent tokens should be bound to issuer-controlled keys through JWKS/x5c or sender-constrained mechanisms such as RFC 9449 DPoP and RFC 8705 OAuth mTLS, with resource servers checking `aud`, `exp`, replay controls, and proof-of-possession before honoring the token.
- **Agent credential management:** Grantex (scoped JWT grant tokens, per-agent DIDs, delegation chains), ScaleKit (delegated agent access with on-behalf-of auth), Nango (secure AI agent API authentication), Auth0 Token Vault (RFC 8693 token exchange with connected-account token storage and per-user external API access), Microsoft Entra Agent ID (preview; agent identities as service principals with blueprint-scoped federated identity credentials, sponsor metadata, and token acquisition on behalf of the agent identity), HashiCorp Vault Enterprise 1.21 (May 2026; native SPIFFE auth method for non-human identities) and Vault 2.0 (SPIFFE secrets engine — issues X.509 and JWT SVIDs directly for AI agent workloads), Google Cloud Agent Identity (GA on Agent Runtime, preview on Gemini Enterprise Agent Platform, May 2026; SPIFFE-based identifier and 24-hour X.509 SVID per deployed agent with automatic rotation, no long-lived static keys, distinct identity type rather than reused service account; Agent Identity Auth Manager preview brokers OAuth on-behalf-of), ZeroID (open-source, Apr 2026; RFC 8693 delegation chains with automatic permission narrowing, OpenID SSF/CAEP cascade revocation, LangGraph/CrewAI/Strands integrations), Lyrie Agent Trust Protocol (ATP, May 2026; open cryptographic standard with five primitives — identity, scope, attestation, delegation, revocation — royalty-free, MIT-licensed reference implementation, IETF submission planned, brings attestation in as a first-class primitive missing from OAuth-extension drafts), Aembit MCP Authorization Server / Identity Gateway (blended user + workload identity, OAuth 2.1 dynamic client registration, short-lived per-session tokens, credential isolation, per-user audit trails)
- **Verification evidence for 5.1.2:** Capture issuer JWKS/x5c metadata, token lifetime policy, DPoP proof replay logs or mTLS certificate-thumbprint checks, resource-server rejection tests for stolen bearer tokens, and key rotation records.
- **DLP:** Google DLP API, Microsoft Purview, Presidio, custom regex/NER-based PII filters
- **Multi-tenant isolation:** Kubernetes namespaces, Calico/Cilium network policies, database row-level security, Pinecone namespaces (up to 100K per index), Weaviate tenant-isolated collections, Nutanix Agentic AI multi-tenancy framework (Apr 2026; per-tenant GPU allocation, tenant-specific security/networking policies across GPU-aaS, K8s-aaS, VectorDB-aaS, Models-aaS), Blaxel (microVM architecture with hardware-enforced tenant boundaries, sub-25ms resume, SOC 2 Type II / HIPAA / ISO 27001 compliant)
- **Vector DB access control:** Pinecone (document-level filtering, role-based retrieval, SOC 2/HIPAA/ISO 27001), Weaviate (tenant isolation, collection-level access), Qdrant (managed multi-tenant support)
- **Agent authorization:** LangChain tool permissions, OpenAI function calling constraints, Anthropic tool use scoping, Model Context Protocol (MCP) for authorization context negotiation, IETF `draft-oauth-ai-agents-on-behalf-of-user` (OAuth 2.0 agent delegation), IETF Secure Intent Protocol (`draft-goswami-agentic-jwt-00`), IETF `draft-klrc-aiagent-auth-02` (June 2026; WIMSE/SPIFFE-based agent auth; treats agents as workload identities with identifiers, short-lived credentials, attestation, provisioning, authorization, observability, remediation, and compliance), IETF `draft-niyikiza-oauth-attenuating-agent-tokens-00` (AATs — cryptographic scope attenuation for delegation chains, Mar 2026), IETF `draft-aap-oauth-profile-00` (Agent Authorization Profile — task binding, capability constraints, oversight claims, Feb 2026), IETF `draft-araut-oauth-transaction-tokens-for-agents-02` (June 2026; adds agent and principal context to Transaction Tokens so agent and human principal are both carried into each downstream call), IETF `draft-embesozzi-oauth-agent-native-authorization-00` (Apr 2026; Human-to-Agent OAuth flow with structured elicitation for first-party agent clients), IETF `draft-song-oauth-ai-agent-collaborate-authz-00` (May 2026; `applier_id` claim for multi-agent collaboration so lead agents request tokens for sub-agents the AS attributes correctly), AIMS IETF Internet-Draft (Mar 2026; composes WIMSE, SPIFFE, and OAuth 2.0 into a unified agent identity framework)
- **Runtime authorization fabric:** Microsoft Authorization Fabric (PEP/PDP with RBAC+ABAC+approval injection, Apr 2026), Entra Agent Identity for workload identity governance, Microsoft Agent Governance Toolkit (open-source, Apr 2026; seven-package runtime security with Agent OS sub-millisecond policy engine, Agent Mesh cryptographic identity via DIDs, and Agent Runtime execution rings with emergency kill switches; supports OPA Rego + Cedar + YAML policies)
- **Vector DB RBAC research:** HoneyBee (SIGMOD 2026) -- dynamic partitioning framework for RBAC in vector databases; Milvus native RBAC with fine-grained collection/partition-level permissions
- **KV-cache isolation:** vLLM cache salting (`cache_salt` parameter) for per-tenant prefix isolation, plus SHA-256 or SHA-256/CBOR cache-block hashing for multi-tenant safety; SafeKV (arxiv 2508.08438) selective sharing with ChunkGuard real-time sensitivity detection; KV-Cloak (arxiv 2508.09442) reversible matrix obfuscation for plaintext cache protection (~5% overhead); NVIDIA BlueField-4 ICMSP (CES 2026) for hardware-accelerated line-rate encryption and DPU-enforced tenant isolation of KV-cache flows; MIRAGE (arxiv 2507.11507, 2025 preprint) tenant-aware parameter remapping for KV-cache optimization that preserves cross-tenant separation
- **GPU TEE confidential computing:** NVIDIA H100 Confidential Compute (<5% throughput overhead for typical LLM inference), AMD SEV-SNP and Intel TDX for VM-level isolation; production deployments available on OpenRouter via Phala Network; provides hardware-enforced tenant isolation for the entire inference pipeline, not just KV-cache
- **Hardware partitioning for 5.3.2:** NVIDIA MIG (up to 7 isolated instances per GPU since Ampere; Hopper and Blackwell add per-instance confidential computing at the hardware/hypervisor level), SR-IOV virtual functions, Kata Containers with VFIO passthrough for full kernel isolation when running untrusted inference code, AWS Nitro Enclaves (network-free, storage-free child VMs with hypervisor-signed attestation — sidesteps the shared-silicon side channels that affect SGX/TDX), and dedicated per-tenant GPU allocation. Operational caveats: MIG partition changes require a node drain (no hot reconfiguration), and the shared last-level TLB means MIG alone does not close covert channels (TunneLs, CCS 2023) — pair it with confidential computing or dedicated allocation for hostile-tenant threat models.
- **Multi-tenant inference isolation:** Token Pools (arxiv 2603.00356) — inference-native capacity isolation decomposing resources into token throughput, KV-cache capacity, and request concurrency with 5-tier priority hierarchy and admission control at API gateway level
- **API gateway security:** API7.ai (RBAC with OPA integration), Palo Alto Prisma for AI API security

### Implementation Maturity (updated 2026-06-22)

| Control Area | Tooling Maturity | Notes |
|--------------|:---:|-------|
| C5.1 Authentication | High | OIDC/SAML and MFA are mature for human users; step-up auth for high-risk AI operations is well understood. The current 5.1.2 wording emphasizes short-lived, minimal-scoped, cryptographically signed agent tokens. Auditors should verify issuer JWKS or x5c trust anchors, scope and audience checks, DPoP proof validation or mTLS certificate-bound tokens where used, short expirations, and replay detection at every federated agent boundary. CVE-2026-32173 (Azure SRE Agent, Apr 2026) is the failure mode this evidence is meant to catch: "token is valid" plus "audience is correct" without a tenant or per-resource role check leaves multi-tenant hubs eavesdroppable. Google Cloud Agent Identity, Microsoft Entra Agent ID, HashiCorp Vault SPIFFE support, and IETF draft-klrc-aiagent-auth-02 now give workload-identity-first implementation paths, while CSA and SANS data show that static and long-lived machine credentials remain common in production. |
| C5.2 AI Resource Authorization & Classification | Medium-High | OPA and Cedar production-ready; Cedar now integrated into Amazon Bedrock AgentCore for agent-level policy with static analysis. Microsoft's Agent Governance Toolkit (Apr 2026) provides open-source multi-engine policy enforcement (OPA Rego + Cedar + YAML) with sub-millisecond latency. JIT adoption remains low — only 1% of orgs have fully adopted JIT privileged access (CyberArk), and CSA found 74% of organizations report agents often receive more access than necessary. Label propagation for AI-specific derived assets is still largely custom. The May 2026 cluster — PraisonAI CVE-2026-44338, Azure AI Foundry CVE-2026-35435, and Microsoft's "exploitable misconfigurations" report — keeps reinforcing that the dominant failure mode is "no authentication on the agent control plane" rather than "policy engine bugs". **Query-time authorization** is maturing: SQL RLS is mature and vector database access control has significantly improved — Pinecone namespaces, Weaviate tenant isolation, Qdrant filtering, and HoneyBee-style RBAC research all help, but every retrieval and assembly stage must still evaluate the caller, not only the service account. **Output entitlement enforcement** remains less mature: DLP tools exist but false negatives remain high for AI-generated content, and citation-level entitlement validation is still largely custom. **Policy decision point isolation** remains emerging: out-of-process policy enforcement patterns such as OPA sidecars, remote Cedar evaluators, AgentCore Policy gateways, MCP identity gateways, and structured action requests are the reference architecture, but many agent frameworks still evaluate policy inline with agent execution. |
| C5.3 Multi-Tenant Isolation | Medium-High | Network and storage isolation is mature; Pinecone/Weaviate namespace isolation is production-ready. Nutanix Agentic AI (Apr 2026) adds per-tenant GPU allocation with dynamic isolation across GPU-aaS, K8s-aaS, and VectorDB-aaS. KV-cache isolation is advancing on multiple fronts: vLLM `cache_salt` and SHA-256 cache keys, SafeKV selective sharing, NVIDIA BlueField-4 ICMSP for DPU-enforced tenant isolation, and MIRAGE parameter remapping. GPU TEE confidential computing (NVIDIA H100 CC, AMD SEV-SNP, Intel TDX) now has production paths, but no software-only mitigation exists for hostile co-located execution. Auditors should treat MIG as necessary-but-insufficient for hostile tenants: TunneLs showed a shared last-level TLB covert channel across MIG partitions, LeftoverLocals recovered GPU local memory across processes, NVBleed used NVLink contention, and TDXRay reconstructed prompts through tokenizer cache-line access inside Intel TDX. |

## Open Research Questions

- [ ] How should authorization policies account for the non-deterministic nature of AI outputs?
- [ ] What's the right model for agent identity in multi-agent orchestration? **[2026-03]** OIDC-A, ARIA, and NIST NCCoE are actively working on this; NIST listening sessions scheduled Apr 2026.
- [x] How do you enforce query-time access control in vector databases that lack native RBAC? **[2026-03]** Partially resolved: Pinecone, Weaviate, and Qdrant now offer native document-level filtering and tenant-isolated namespaces. Complex cross-collection queries still need application-layer enforcement.
- [ ] Should AI systems have their own identity distinct from the calling user? **[2026-03]** NIST's concept paper and ISACA's ARIA framework both point toward yes -- agents need enterprise-grade identities, not just inherited API keys or service accounts.
- [ ] How can GPU memory isolation be enforced between tenants sharing inference hardware? **[2026-03]** KV-cache specifically is being addressed: vLLM cache salting, SafeKV selective sharing, and KV-Cloak obfuscation each tackle different threat vectors. Broader GPU memory isolation (activations, gradient residuals) remains unsolved.
- [ ] What are effective strategies for propagating data classification labels through embedding transformations?
- [ ] **[2026-03]** How should delegation chains be modeled for multi-hop agent-to-agent authorization with scope narrowing and cascade revocation?
- [ ] **[2026-03]** What is the appropriate trust model for peer-to-peer agent trust establishment across organizational boundaries?
- [ ] **[2026-03]** How can continuous per-action authorization scale to agents executing hundreds of actions per session without creating latency bottlenecks?
- [ ] **[2026-03]** How should organizations handle the "agentic authorization bypass" pattern where shared agents evaluated against agent identity (not requester identity) enable indirect privilege escalation?
- [ ] **[2026-03]** What is the right architecture for output filtering when agentic browsers can exfiltrate data through their own authorized channels, bypassing traditional DLP controls?
- [ ] **[2026-04]** How can organizations enforce JIT privileged access for AI workloads when only 1% have fully adopted it and 91% still rely on always-on standing privileges?
- [ ] **[2026-04]** What is the right data classification taxonomy for AI-specific assets (embeddings, model weights, prompt templates, RAG assemblies, fine-tuning datasets, agent tool schemas) — and how should classification labels propagate through embedding transformations?
- [ ] **[2026-04]** How should authorization policy expiration be enforced for AI agent tool access as available tools and capabilities change, without causing operational disruption from premature revocation?
- [ ] **[2026-04]** Can hardware-accelerated KV-cache isolation (e.g., NVIDIA BlueField-4 ICMSP) provide sufficient tenant separation to eliminate software-level side-channel mitigations, or do both layers remain necessary?
- [ ] **[2026-04]** How should inference capacity isolation (token pools, priority-aware admission control) interact with authorization policy to prevent resource-level denial of service across tenants sharing GPU infrastructure?
- [ ] **[2026-04]** How should the MCP specification address its optional-authentication gap, given that CVE-2026-32211 (CVSS 9.1) and CVE-2025-59528 (CVSS 10.0) both stem from MCP servers shipping without authentication?
- [ ] **[2026-04]** What is the appropriate "reasonable care" standard for AI agent identity and authorization under the Colorado AI Act (effective June 2026), and how should organizations demonstrate compliance through adopted standards?
- [ ] **[2026-04]** How should AI-native social platforms secure agent-to-agent credentials, given the Moltbook breach demonstrated that plaintext credential exchange in agent DMs creates cascading exposure across interconnected services?
- [ ] **[2026-04]** How should multi-engine policy frameworks (e.g., Microsoft Agent Governance Toolkit supporting OPA + Cedar + YAML simultaneously) handle policy conflict resolution when overlapping rules from different engines produce contradictory authorization decisions for the same agent action?
- [ ] **[2026-05]** How should enterprises inventory, classify, and decommission unknown or abandoned agents when CSA reports that 82% of organizations have unknown agents and only 21% have formal decommissioning processes?
- [ ] **[2026-05]** Should localhost MCP servers be treated as remotely reachable security boundaries by default, given DNS rebinding can convert browser access into unauthorized tool invocation unless authentication, Host header validation, or stdio-only transport is enforced?
- [ ] **[2026-05]** What evidence should auditors require to prove a federated agent token is bound to the issuing system's identity, rather than merely signed by an untracked or replayable key?
- [ ] **[2026-05]** How should agent frameworks default tool-execution mode (auto-invoke vs. manual approval) when the May 2026 Semantic Kernel CVEs show that auto-invocation plus prompt injection equals RCE — and what auditing controls demonstrate that least-agency defaults are enforced in shipped products?
- [ ] **[2026-05]** What configuration hardening baseline should ship by default with cloud-native AI deployment templates (Ollama, vLLM dashboards, Flowise, AnythingLLM, MLflow) so that internet-reachable agent control planes are not the single most common initial access vector identified in Microsoft's May 2026 misconfiguration research?
- [ ] **[2026-05]** Will the IETF's parallel agent-authorization drafts (Transaction Tokens for Agents, Agent Native Authorization via Structured Elicitation, Applier-On-Behalf-Of, AAP, AIMS, AATs, on-behalf-of-user) converge into a single profile, or do operators need to pick a draft per use case (workload identity vs. delegation chain vs. multi-agent collaboration)?
- [ ] **[2026-05]** How should the relying-party side of agent identity be validated — is workload-identity-first attribution (CSA "Solved Backwards") materially different from credential-vault attribution at runtime, and what audit trail proves the relying party can distinguish them?
- [ ] **[2026-05]** Given the SANS 2026 baseline that 92% of organizations do not rotate machine credentials at 90 days, how should AISVS verification artifacts treat short-lived (24-hour SVIDs) versus long-lived tokens — should presence of any long-lived token in the agent fleet automatically fail 5.1.2?
- [ ] **[2026-05]** How should the Lyrie Agent Trust Protocol's five-primitive model (identity, scope, attestation, delegation, revocation) interoperate with the parallel IETF agent-auth drafts (WIMSE/SPIFFE, AATs, AAP, applier_id, on-behalf-of-user) — should attestation become a shared primitive across those drafts, or is it specific to ATP's cryptographic-first design?
- [ ] **[2026-05]** What evidence should auditors collect to prove that multi-tenant agent endpoints (Entra ID app registrations, OIDC apps, GitHub Apps acting as agents) actually verify tenant and per-resource role at the resource server, not just JWT signature and audience? CVE-2026-32173 is the canonical "signature valid, tenant ignored" failure mode and should drive concrete 5.1.2 verification artifacts.
- [ ] **[2026-05]** Should 5.2.7's label-propagation evidence explicitly enumerate agent memory and scratchpad files (`MEMORY.md`, `CLAUDE.md`, project rules) as labeled assets with provenance tracking? - [ ] **[2026-05]** When the patched form of an indirect prompt injection (e.g., CVE-2026-21520) still allows exfiltration through the agent's own authorized channels (Capsule Security's ShareLeak / PipeLeak), what additional 5.2.4 evidence beyond post-inference content filtering is required — and should AISVS introduce explicit "agent-initiated channel" entitlement gates separate from the existing output filtering wording?

- [ ] **[2026-06]** With 5.3.2 now requiring hardware partitioning, confidential computing, or dedicated compute for shared-infrastructure tenancy, what combination should auditors accept for a hostile-tenant threat model? MIG's shared last-level TLB (TunneLs), NVLink contention (NVBleed), and TDX cache-line leakage (TDXRay) each undermine a single-layer answer — is MIG + GPU TEE the practical floor, or does true adversarial co-tenancy require dedicated GPUs?
- [ ] **[2026-06]** Should the MCP specification's mandated unauthenticated OAuth discovery endpoints be revisited now that BadHost (CVE-2026-48710) demonstrates them as a reliable exploitation path, and Censys/VIPER-MCP measurements show most internet-reachable MCP services run without authentication at all?
- [ ] **[2026-06]** As regulations converge on attribution (CSA's "Attribution Gap": eight frameworks, Colorado AI Act effective Jun 30 2026), what minimum audit evidence proves the five capabilities — agent identification, scoped access, traceable authorization to a named human, dual permission checks, immutable logging — for systems built before agent identity platforms existed?
