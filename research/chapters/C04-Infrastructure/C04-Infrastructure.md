# C04: Infrastructure, Configuration & Deployment Security

> **Source:** [`1.0/en/0x10-C04-Infrastructure.md`](https://github.com/OWASP/AISVS/blob/main/1.0/en/0x10-C04-Infrastructure.md)

## Control Objective

AI-specific infrastructure components must be hardened against model theft, data leakage, and cross-tenant contamination. This chapter covers AI workload sandboxing, AI accelerator hardware security, and edge/distributed AI deployment security. General infrastructure security controls (container hardening, network segmentation, secrets management, CI/CD pipeline security) are covered by ASVS, CIS Benchmarks, and NIST SP 800-53, and are not repeated here. AI-specific supply chain controls are covered in C6.

> **2025-2026 Highlights:** The AI supply chain saw its largest cascading compromise in March 2026: the TeamPCP campaign hit Trivy, Checkmarx KICS, and LiteLLM simultaneously (CVE-2026-33634, CVSS 9.4), exfiltrating cloud tokens and Kubernetes secrets from over 20,000 repositories — with LiteLLM present in 36% of monitored cloud environments. Days later, Langflow's unauthenticated RCE (CVE-2026-33017, CVSS 9.3) was exploited within 20 hours of disclosure, giving attackers full access to AI pipeline credentials. April 2026 added a Kubernetes-native model-serving case: KubeAI CVE-2026-34940 let users with Model CRD create/update rights inject shell commands into Ollama startup probes, fixed in v0.23.2. Inference servers themselves continued to burn: Oligo's ShadowMQ disclosure (Nov 2025) showed Meta's Llama Stack pickle-over-ZMQ RCE propagated by direct copy-paste into NVIDIA TensorRT-LLM, vLLM, SGLang, and Modular Max; Wiz's CVE-2025-23319/23320/23334 chain against NVIDIA Triton Inference Server completed the picture of AI serving stacks as an untreated attack surface. NVIDIA then shipped March and April 2026 Triton security updates for eight additional network-reachable denial-of-service and model-configuration exposure flaws, pushing the operational baseline to r26.02 or later. Pwn2Own Berlin 2025 made AI infrastructure a first-class competition target, and Trend Micro now projects 2,800–3,600 AI CVEs in 2026 (up 31–69% from 2,130 in 2025). On the defensive side, KubeCon Europe 2026 brought major moves: Kubernetes Dynamic Resource Allocation (DRA) reached stable in v1.35, NVIDIA donated its DRA GPU driver to the CNCF, shipped Kata Containers GPU support (still early-access per NVIDIA's own docs), and Harvester 1.7 added native NVIDIA MIG vGPU with hardware-level spatial partitioning; Microsoft shipped DRA to GA with Cilium-based security for AKS, and Edera announced KVM support for its zone-based container isolation. As of May 13, 2026, the EU AI Act high-risk timeline is in flux: the Commission service desk still lists August 2, 2026 for Annex III high-risk rules, but Parliament and Council negotiators reached a May 7 provisional deal to move those obligations to December 2, 2027 and product-embedded high-risk systems to August 2, 2028, pending formal adoption.

## Threat Landscape

Known attacks, real-world incidents, and threat vectors relevant to this chapter:

- **AI framework RCE via workflow platforms** — CVE-2026-33017 (CVSS 9.3, March 2026) in Langflow allows unauthenticated remote code execution via a single HTTP POST to the public flow build endpoint. Attacker-controlled Python code is passed directly to `exec()` without sandboxing, granting access to environment variables, API keys for OpenAI/Anthropic/AWS, and connected databases. CISA added it to KEV within 8 days. Exploited in the wild within 20 hours of disclosure.
- **Cascading supply chain compromise (TeamPCP)** — In March 2026, the TeamPCP threat actor compromised Aqua Security's Trivy scanner (CVE-2026-33634, CVSS 9.4) by force-pushing 76 malicious version tags in the trivy-action GitHub Action and publishing poisoned Docker Hub images. The attack cascaded to LiteLLM (malicious PyPI packages 1.82.7/1.82.8) and Checkmarx KICS. Wiz Research found LiteLLM present in 36% of monitored cloud environments. Microsoft published emergency detection guidance; BerriAI engaged Mandiant for incident response.
- **Kubernetes-native AI operator command injection** — CVE-2026-34940 in KubeAI showed how model-serving control planes can become execution surfaces. Unsanitized model URL components from a `Model` custom resource were interpolated into an Ollama `bash -c` startup probe, letting users with create/update rights on Model CRDs execute commands inside model server pods. The practical controls are to upgrade to KubeAI 0.23.2 or later, restrict Model CRD writers, enforce admission validation for model URLs, and review startup/readiness/liveness probes as part of C4.1 sandbox validation.
- **Container escape via GPU toolkits** — Critical vulnerabilities in NVIDIA Container Toolkit (CVE-2024-0132, CVE-2025-23266 "NVIDIAScape") allow AI workloads to break container isolation and gain root on the host. As of mid-2025, 37% of cloud AI environments were vulnerable.
- **GPU side-channel attacks** — NVBleed (March 2025) demonstrated cross-VM covert channels via NVLink performance counters at 70+ Kbps, while BarraCUDA showed electromagnetic side-channel model extraction from NVIDIA Jetson edge devices. Separate research found four leakage sources bypassing MIG isolation under MPS mode.
- **GPU memory bit-flip attacks** — GPUHammer (USENIX Security 2025) demonstrated practical Rowhammer attacks against GDDR6 memory on NVIDIA GPUs. A single bit flip to a model weight's exponent can degrade accuracy from 80% to under 1%. In shared cloud GPU environments, a malicious tenant can target adjacent workloads' model weights. ECC memory (enabled by default on Hopper and Blackwell data center GPUs) mitigates this, but consumer and edge GPUs remain vulnerable.
- **Supply chain compromise of ML frameworks** — CVE-2025-32434 (CVSS 9.3) proved that PyTorch's `weights_only=True` could be bypassed for RCE. Three PickleScan zero-day bypasses (CVE-2025-10155/56/57) allowed undetected malicious model uploads to Hugging Face.
- **Secrets leakage from AI pipelines** — A November 2025 study found 65% of Forbes AI 50 companies had leaked API keys on GitHub, primarily through Jupyter notebooks and .env files. One leaked Hugging Face token exposed ~1,000 private models.
- **TEE bypass attacks** — TEE.Fail (October 2025) broke Intel SGX, TDX, and AMD SEV-SNP using DDR5 memory bus interposition hardware costing under $1,000, undermining confidential computing assumptions.
- **Container runtime breakout (runC)** — Three critical runC vulnerabilities disclosed in November 2025 (CVE-2025-31133, CVE-2025-52565, CVE-2025-52881) affect all Docker, Kubernetes, containerd, and CRI-O deployments. CVE-2025-31133 exploits a mount race condition in maskedPaths to redirect writes to `/proc/sys/kernel/core_pattern`, achieving container escape. CVE-2025-52565 abuses a timing window where `/dev/console` is mounted before security protections are applied. CVE-2025-52881 bypasses Linux Security Module labels via shared mount race conditions. All three are deliverable through untrusted container images — a realistic attack vector for AI workloads that routinely pull models and dependencies from external registries. Fixed in runC 1.2.8 / 1.3.3 / 1.4.0-rc.3.
- **ShadowMQ — copy-paste RCE across inference frameworks** — In November 2025, Oligo Security disclosed ShadowMQ, a systemic pattern where Meta's Llama Stack RCE (CVE-2024-50050) propagated by direct copy-paste into NVIDIA TensorRT-LLM, vLLM, SGLang, and Modular Max Server. All of them called ZeroMQ's `recv_pyobj()` over unauthenticated TCP sockets, triggering `pickle` deserialization on attacker input. Exploitation yields arbitrary code execution on GPU clusters — exfiltration of model weights, lateral movement, or GPU cryptomining. Fixed in Llama Stack ≥ 0.0.41, TensorRT-LLM 0.18.2, vLLM 0.8.0, and Modular Max v25.6. This is a direct C4.1.3 case study: the frameworks that serve models were themselves loading attacker input through unsafe deserialization paths.
- **Inference server takeover chain** — CVE-2025-23319, CVE-2025-23320, and CVE-2025-23334 in NVIDIA Triton Inference Server (disclosed Aug 2025, discovered by Wiz Research following Pwn2Own Berlin 2025) chain into unauthenticated RCE against a production model-serving platform. The first Pwn2Own AI-infrastructure category included Triton, Redis, and Chroma as targets — seven of twenty-eight competition vulnerabilities came from the AI bracket. Trend Micro's 2026 projection puts AI CVEs at 2,800–3,600 (up from 2,130 in 2025), with inference servers, model-hosting platforms, and agentic systems driving the growth.
- **Inference server availability and model-config exposure** — NVIDIA's March and April 2026 Triton bulletins added eight more network-reachable flaws after the August 2025 takeover chain. March fixed CVE-2025-33238, CVE-2025-33254, and CVE-2026-24158 (large compressed HTTP payload denial of service) in 26.01; April fixed CVE-2026-24146/24173/24174/24175 denial-of-service flaws and CVE-2026-24147 model-configuration upload information disclosure/DoS in r26.02. Treat inference endpoints as service infrastructure: cap request and output sizes, gate model repository writes, restrict model-configuration upload paths, and rehearse rollback before disclosure windows.
- **Lateral movement** — Attackers have used malicious registry URLs to force exposed LLM inference servers into making outbound requests, probing for paths to training infrastructure.
- **Edge model extraction** — Studies show 34–76% success rates extracting ML models from mobile apps using standard reverse-engineering tools when models lack encryption.

## Tooling & Implementation

Current tools, frameworks, and libraries that help implement these controls:

- **Container isolation:** [gVisor](https://gvisor.dev/) (user-space kernel, used by Modal for AI sandboxing), [Kata Containers](https://katacontainers.io/) (lightweight VMs, 2M+ AI workloads/month at Northflank; as of March 2026, NVIDIA ships GPU support for Kata enabling hardware-isolated AI inference), [Firecracker](https://firecracker-microvm.github.io/) (Rust-based microVMs, ~100ms boot, used by E2B for AI agent sandboxing), [Edera](https://edera.dev/) (Type-1 hypervisor with per-container micro-VMs called "zones" — each gets its own kernel, eliminating shared-kernel container escapes; announced KVM support at KubeCon EU 2026 alongside a new Rust-based OCI image library, `ocirender`), [VCluster](https://www.vcluster.com/) (virtual Kubernetes clusters for multi-tenant GPU isolation — each tenant gets a full control plane while sharing the GPU fleet; integrates Karpenter auto-scaling, Netris network isolation, and vNode Runtime for container sandboxing)
- **GPU security & orchestration:** [NVIDIA MIG](https://www.nvidia.com/en-us/technologies/multi-instance-gpu/) (up to 7 hardware-isolated GPU instances on A100/H100/H200), [Kubernetes Dynamic Resource Allocation](https://kubernetes.io/docs/concepts/scheduling-eviction/dynamic-resource-allocation/) (stable in v1.35 — DeviceClasses and ResourceClaims support attribute-based accelerator selection, sharing, and centralized hardware classes), [NVIDIA H100 Confidential Computing](https://developer.nvidia.com/blog/confidential-computing-on-h100-gpus-for-secure-and-trustworthy-ai/) (on-die root of trust, encrypted CPU-GPU transfers, remote attestation via NRAS), [NVIDIA Vera Rubin NVL72](https://www.nvidia.com/en-us/data-center/vera-rubin-nvl72/) (announced January 2026 — first rack-scale confidential computing spanning 72 GPUs, 36 Vera CPUs, and NVLink interconnects in a unified trust domain; end-to-end encryption across CPU-GPU, GPU-GPU, and I/O paths with near-native performance), [NVIDIA DRA Driver](https://blogs.nvidia.com/blog/nvidia-at-kubecon-2026/) (donated to CNCF at KubeCon EU 2026 — community-owned GPU resource allocation for Kubernetes with MIG/MPS sharing, multi-node NVLink support, and runtime reconfiguration; backed by AWS, Google Cloud, Microsoft, Red Hat, and 4 others), [NVIDIA Grove](https://thenewstack.io/kubecon-vclusters-k8s-platform-to-manage-gpus-as-a-service/) (open-source K8s API for declarative GPU cluster management, integrates with llm-d inference stack), AMD SEV-SNP (VM-level memory encryption on Genoa/Milan CPUs), Intel TDX (VM isolation with attestation, expanding on Emerald Rapids)
- **Confidential containers:** [Confidential Containers (CoCo)](https://github.com/confidential-containers) (CNCF sandbox project — pod-level confidential computing on Kubernetes via Kata Containers CVMs with hardware-backed TEEs; developing end-to-end attestation for NVIDIA confidential GPUs), [NVIDIA Confidential Containers Reference Architecture](https://docs.nvidia.com/datacenter/cloud-native/confidential-containers/latest/overview.html) (Kata + GPU Operator + Trustee/NRAS composite attestation; supports single-GPU passthrough and multi-GPU passthrough on NVSwitch/HGX systems, but requires all GPUs on the host to be configured for confidential computing and containerd runtime support), [NVIDIA GPU Operator with Kata + CoCo](https://docs.nvidia.com/datacenter/cloud-native/gpu-operator/25.10/confidential-containers.html) (25.10 early-access docs still warn the feature is not production-supported; AMD SEV-SNP only in that EA path), [Google Confidential GKE Nodes](https://docs.cloud.google.com/kubernetes-engine/docs/how-to/gpus-confidential-nodes) (encrypt GPU workload data in-use), [Red Hat OpenShift + NVIDIA CoCo](https://www.redhat.com/en/blog/power-confidential-containers-red-hat-openshift-nvidia-gpus) (confidential GPU containers on OpenShift; still experimental as of early 2026)
- **Model signing & build security:** [OpenSSF Model Signing v1.0](https://github.com/sigstore/model-transparency) (released April 2025 — sign/verify ML models of any format via Sigstore), [Sigstore/cosign](https://www.sigstore.dev/) (keyless artifact signing, SLSA provenance), [GitHub artifact attestations](https://github.blog/changelog/2026-01-20-strengthen-your-supply-chain-with-code-to-cloud-traceability-and-slsa-build-level-3-security/) (January 2026 — SLSA Build Level 3, cryptographic provenance binding artifacts to source repos and build workflows, code-to-cloud traceability), [Syft](https://github.com/anchore/syft) (SBOM generation for containers, supports SPDX 3 and CycloneDX)
- **Secrets management:** [HashiCorp Vault](https://www.hashicorp.com/en/blog/integrating-secret-hygiene-into-ai-and-ml-workflows) (dynamic AI agent identity, Vault Radar for dataset secret scanning), AWS Secrets Manager, Azure Key Vault, Google Secret Manager
- **Network security:** [Calico](https://www.tigera.io/blog/securing-ai-workloads-in-kubernetes-why-traditional-network-security-isnt-enough/) (microsegmentation, FQDN-based egress filtering, WireGuard encryption), Cilium (eBPF-based network policies), [Kubernetes Pod Certificates](https://v1-35.docs.kubernetes.io/docs/concepts/storage/projected-volumes/#podcertificate-projected-volumes) (beta in K8s v1.35 for native pod mTLS; disabled by default and requires the PodCertificateRequest feature gate), [Cisco Hybrid Mesh Firewall on BlueField DPUs](https://newsroom.cisco.com/c/r/newsroom/en/us/a/y2026/m03/cisco-secure-ai-factory-with-nvidia-GTC-2026.html) (March 2026 — 400G line-rate stateful inspection on GPU servers, policy enforcement at the DPU without consuming CPU/GPU resources)
- **Runtime threat detection:** [Falco](https://falco.org/) (CNCF graduated — eBPF-based runtime security for containers and K8s; detects privilege escalation, container escape, and anomalous system calls in real time), [Sysdig](https://sysdig.com/) (ML-enhanced threat detection and drift prevention built on Falco's runtime data)
- **Hardware monitoring:** [NVIDIA DCGM](https://github.com/NVIDIA/DCGM) (GPU health monitoring, telemetry export to Prometheus/Grafana), [NVIDIA Remote Attestation Service](https://developer.nvidia.com/blog/announcing-confidential-computing-general-access-on-nvidia-h100-tensor-core-gpus) (GPU identity and firmware validation)
- **Edge & mobile deployment:** [Apple Core ML encryption](https://developer.apple.com/documentation/coreml/encrypting-a-model-in-your-app) (compile-time model encryption, decrypted only at runtime), [ONNX Server for Open Enclave](https://github.com/microsoft/onnx-server-openenclave) (confidential inference with attestation), Android Keystore / iOS Secure Enclave for model decryption key protection
- **AI workload sandboxing:** [E2B](https://e2b.dev/) (Firecracker-based AI agent sandboxing), [IOValve](https://www.microsoft.com/en-us/research/wp-content/uploads/2025/08/iovalve-ccs25.pdf) (Microsoft Research, leakage-free I/O sandbox, ACM CCS 2025), [UK AISI Inspect Sandboxing Toolkit](https://www.aisi.gov.uk/blog/the-inspect-sandboxing-toolkit-scalable-and-secure-ai-agent-evaluations)

### Inference Server Hardening Checks

- Track inference-server PSIRT bulletins separately from base container and GPU driver patching; Triton moved from the August 2025 RCE chain to March/April 2026 availability and model-configuration fixes within two release trains.
- Put explicit limits in front of model-serving HTTP/gRPC endpoints: compressed request body size, output tensor count, decompression ratio, request concurrency, and per-tenant queue depth.
- Treat model repository writes, model-control APIs, Python backend loading, and configuration upload paths as administrative interfaces; keep them authenticated, network-restricted, logged, and separated from public inference traffic.
- Exercise rollback and canary procedures for serving-stack upgrades before disclosure windows; a broken inference rollback can become an availability incident even when the security patch is correct.
- Subscribe to the [Kubernetes official CVE feed](https://kubernetes.io/docs/reference/issues-security/official-cve-feed/) as part of AI platform operations; ingress-nginx configuration injection and CSI path traversal issues still become AI infrastructure incidents when model-serving ingress, shared datasets, or mounted model repositories depend on those components.

### AI Operator Hardening Checks

- Treat `Model`, `InferenceService`, runtime, and model-repository CRDs as code-execution surfaces; restrict create/update rights, log changes, and require review for external model URLs.
- Reject model URLs with shell metacharacters through admission policy, and prefer exec probes that pass arguments as arrays rather than interpolating model references into `bash -c`.
- Test startup, readiness, liveness, and sidecar hooks with malicious model names, PVC query parameters, and registry references before onboarding model-serving operators.
- Pin and patch AI operators such as KubeAI, KServe, and Triton management components separately from the underlying Kubernetes and GPU driver patch cycle.

## Open Research Questions

- [x] **What's the maturity of confidential computing for GPU workloads?** — As of early 2026, NVIDIA H100 CC is GA with on-die root of trust, encrypted CPU-GPU transfers, and remote attestation (NRAS), but throughput is limited to ~4 GB/s. The Vera Rubin NVL72 (announced January 2026) extends confidential computing to rack-scale — 72 GPUs, 36 CPUs, and NVLink interconnects form a unified trust domain with end-to-end encryption and near-native performance, a significant leap over H100's per-GPU model. However, the TEE.Fail attack (Oct 2025) showed DDR5 memory bus interposition can break underlying CPU TEEs with <$1K hardware, and GPUHammer demonstrated practical Rowhammer attacks against GDDR6 GPU memory (USENIX Security 2025). The hardware threat surface continues to expand alongside defensive capabilities.
- [x] **How do you isolate multi-tenant GPU workloads without significant performance overhead?** — NVIDIA MIG provides hardware partition-level isolation (up to 7 instances on A100/H100) with dedicated memory, cache, and SMs per tenant. Performance isolation is strong but MIG does not encrypt — separate research showed four side-channel leakage sources bypassing MIG under MPS mode, and GPUHammer showed cross-tenant Rowhammer attacks are practical on GPUs without ECC. Full confidential multi-tenancy requires H100 CC mode (with throughput constraints) or the upcoming Vera Rubin NVL72, which promises rack-scale encryption with near-native performance.
- [ ] **What are the best practices for securing AI inference at the edge with limited compute?** — Apple Core ML offers compile-time model encryption (decrypted only at runtime), but this is platform-specific. Android lacks a native equivalent; teams typically use Android Keystore for key protection with custom encryption wrappers. No universal cross-platform model encryption standard exists yet. The ONNX Server for Open Enclave project provides TEE-based inference but targets server-class hardware.
- [ ] **How should hardware security requirements differ for training vs. inference infrastructure?** — Training requires higher throughput and multi-node coordination (NVLink, InfiniBand), where the NVBleed side-channel and RDMA protocol weaknesses (NeVerMore) are relevant. Inference is more deployable in TEEs due to lower data transfer volumes but faces model extraction risks. Edge inference faces unique physical access threats (BarraCUDA EM side-channel).
- [ ] **How will post-quantum cryptography affect model signing and TEE attestation chains?** — OpenSSF Model Signing v1.0 and Sigstore currently use classical cryptographic primitives. As PQC standards finalize, the model signing and TEE attestation ecosystems will need migration paths.
- [ ] **Can GPU telemetry anomaly detection reliably identify side-channel attacks?** — NVIDIA DCGM exports power, temperature, and error counters, but no published production system uses this telemetry specifically for side-channel detection (C4.2.8). Kubernetes DRA is now stable in v1.35 and the NVIDIA DRA driver is moving into CNCF/community ownership, which gives operators richer allocation metadata; the detection models that turn that metadata into reliable side-channel alerts remain an open gap.
- [ ] **How should organizations defend against supply chain attacks that compromise security tooling itself?** — The TeamPCP campaign (March 2026) showed that attackers can poison the very scanners and CI/CD actions that enforce supply chain integrity. Pinning GitHub Actions by commit SHA rather than mutable tags is a minimum countable defense, but the broader question — how to maintain trust in the verification chain when the verifiers are compromised — needs architectural answers beyond current SLSA and Sigstore capabilities.
- [ ] **When will Confidential Containers (CoCo) with GPU support reach production readiness?** — As of March 2026, the CNCF CoCo project supports pod-level confidential computing on Kubernetes via Kata Containers CVMs, with end-to-end attestation including CPU and GPU. NVIDIA's GPU Operator integrates with CoCo for confidential GPU workloads, but the combination is still experimental and not recommended for production. Red Hat OpenShift and Google Confidential GKE Nodes offer managed paths, but full GPU CC on vanilla Kubernetes remains a gap. The answer matters because C4.1.5-C4.1.7 requirements assume TEE-based workload isolation that currently lacks a turnkey K8s solution.
- [ ] **How should inference servers themselves be hardened as part of C4.1?** — ShadowMQ (Nov 2025), the Triton RCE chain (Aug 2025), and KubeAI CVE-2026-34940 (Apr 2026) showed that the biggest C4.1 gap in practice is not just running untrusted user code — it is the inference servers, AI gateways, and model-serving operators that load pickle-over-ZMQ, parse untrusted protobuf, expose management endpoints without auth, or turn model URLs into shell probes. Pwn2Own Berlin 2025 made AI infrastructure a first-class category, and Trend Micro projects 2,800–3,600 AI CVEs in 2026 concentrated in model-serving stacks. Current requirements (particularly 4.1.3) cover the model loader, but the inference server's own IPC, admin, control plane, and Kubernetes operator surfaces need explicit guidance — network-policy defaults, authenticated-only IPC, pickle-free deserialization inside the server process, CRD admission controls, and shell-free probe construction — not just around it.
