# C5.2: AI Resource Authorization & Classification

> [Back to C05 Index](C05-Access-Control.md)
> **Last Researched:** 2026-06-09 (11th pass, added the actively exploited LiteLLM pre-auth RCE chain CVE-2026-42271/CVE-2026-48710 placed on CISA KEV on June 8 2026, the MLflow registry authorization bypasses CVE-2026-2651/2652/2734, HashiCorp Vault's native AI agent identity support, Weaviate 1.38 RBAC-gated namespaces, and Azure AI Search ACL and sensitivity-label propagation onto vector chunks)

## Purpose

Implement access controls for all AI resources with explicit permission models and audit trails. AI systems introduce authorization challenges that go beyond traditional application security: vector databases, embedding indices, model registries, prompt caches, agent tool schemas, and derived outputs all need explicit access control. Data classification labels must propagate through AI pipelines to derived resources, and authorization decisions should be externalized to dedicated policy decision points where possible to prevent scattered, inconsistent logic across AI services.

> **Scope note:** Controls in this section govern authorization policy definition, enforcement, classification, and audit for AI resources. Identity and authentication are addressed in C5.1; query-time enforcement at the data layer is in C5.3; output-level filtering is in C5.4. Per ASVS v5 14.1.1, all sensitive data should be identified and classified into protection levels; C5.2.3 and C5.2.4 extend this to AI-specific data types and derived resources.

| **5.2.1** | **Verify that** every AI resource (datasets, models, endpoints, vector collections, embedding indices, compute instances) enforces access controls (e.g., RBAC, ABAC) with explicit allow-lists and default-deny policies. | 1 | Unauthorized access to training data, model weights, inference endpoints, vector collections, prompt caches, or agent tools. OWASP's Agentic Top 10 maps this directly to ASI03 Identity & Privilege Abuse, where agents inherit, reuse, or misuse privileges beyond the user they represent, with task-scoped tokens called out as the primary mitigation pattern. MITRE's February 2026 OpenClaw investigation showed the same pattern in practice: exposed agent control interfaces let researchers read connected-application credentials and invoke agent skills to reach root in the container, and the OpenClaw case studies `AML.CS0048` through `AML.CS0051` repeatedly point back to missing tool and credential authorization boundaries. MITRE ATLAS v5.3.0 in January 2026 then added three new case studies covering MCP server compromise, indirect prompt injection over MCP, and malicious AI agent deployment - each one assumes resource access controls are weak or absent. EchoLeak, tracked as CVE-2025-32711, is another useful test case: an AI system with broad retrieval authority can become a data-disclosure path even when the attacker never directly authenticates to the underlying documents. CVE-2026-33032 in nginx-ui, fixed in 2.3.4 on March 15, 2026 and added to VulnCheck's Known Exploited Vulnerabilities list on April 13, 2026, shows the same default-deny gap from a different angle: the `/mcp_message` MCP endpoint applied only IP allowlisting, the default allowlist was empty, and the middleware treated an empty allowlist as "allow all", which let any unauthenticated network caller invoke the full set of nginx-ui MCP tools, including configuration writes and service restarts. CVE-2026-30623 in Anthropic's MCP SDK, disclosed by OX Security on April 15, 2026, makes the same point at the protocol layer: the stdio transport's `StdioServerParameters` accepts whatever command string the caller supplies and spawns it as a subprocess on the host, so any path that lets an attacker register or edit an MCP server entry becomes a local command execution primitive. Anthropic treats the stdio execution model as a documented default, so the practical mitigation has shifted to the host-side PEP - LiteLLM's v1.83.6-nightly fix is illustrative: it restricts stdio commands to a fixed allowlist (`npx`, `uvx`, `python`, `python3`, `node`, `docker`, `deno`) and limits the `/mcp-rest/test/connection` and `/mcp-rest/test/tools/list` preview endpoints to the `PROXY_ADMIN` role. OX Security's audit found similar exposure in LangChain, LangFlow, Flowise, LettaAI, and LangBot, and estimated about 200,000 publicly reachable MCP servers shipped against this design. The sequel is a caution about treating admin-gated test endpoints as resolved: those same LiteLLM `/mcp-rest/test/connection` and `/mcp-rest/test/tools/list` endpoints accept a full MCP server config and spawn the supplied `command` as a subprocess, and chained with the Starlette TrustedHost bypass CVE-2026-48710 the result was pre-authentication remote code execution, tracked as CVE-2026-42271 against LiteLLM 1.74.2 through 1.83.6 and fixed in 1.83.7. CISA added it to the Known Exploited Vulnerabilities catalog on June 8, 2026 with a June 22 remediation deadline; exploitation hands over the provider credentials and API keys the gateway holds. Exposure measurements make the default-deny half of this requirement concrete: Censys counted roughly 12,500 internet-accessible MCP services in mid-2026, and a remote-MCP authentication study found about 40% expose tools with no authentication at all. | Build an inventory of datasets, model artifacts, endpoints, vector databases, embedding indices, prompt/template stores, agent tools, and compute targets. For each resource, test anonymous access, low-privilege user access, stale API keys, service-account access, and agent-mediated access. Confirm explicit grants are required, deny is the fallback, and audit logs show the principal, action, resource, decision, and policy version. For vector stores, inspect native controls: Pinecone project/data-plane roles, API-key permissions, and the December 2025 Dedicated Read Nodes that isolate read workloads from writes per index; Weaviate RBAC for collections, objects, tenants, and backups; Qdrant API keys, read-only keys, JWT RBAC, and managed-cloud granular API keys. Validate that scoped credentials actually scope: CVE-2026-44221, published May 12, 2026 against ArcadeDB versions before 2.6.4, let authenticated tokens scoped to a single database read, write, and mutate schema on any other database on the same server because `ServerSecurityUser.getDatabaseUser` returned an uninitialized `fileAccessMap` that downstream checks treated as allow-all, and `ArcadeDBServer.createDatabase` omitted `factory.setSecurity(...)` for databases created through the REST `create database` command. For MCP-mediated agent traffic, route requests through a Cedar-aware PEP such as ToolHive or Amazon Bedrock AgentCore Gateway, exercise tools across tenants - runZero's CVE-2026-5374, fixed in 4.0.260202.0 in April 2026, let MCP agents read remediation and asset records for organizations outside their authorized scope because the data-access layer did not re-check the organization id supplied with each query - and test through the policy enforcement point rather than only through the application UI. Retest model registries after upgrades: MLflow fixed three authorization gaps in spring 2026 - CVE-2026-2652 left the Job API and OpenTelemetry trace-ingestion routes unauthenticated under uvicorn even with basic auth enabled (fixed in 3.11.0), CVE-2026-2734 let any authenticated user enumerate model versions across all registered models because `SearchModelVersions` was missing from the authorization middleware (fixed in 3.10.0), and CVE-2026-2651 exposed multipart artifact-upload endpoints when `--serve-artifacts` was enabled. On shared vector clusters, exercise Weaviate 1.38.0's namespace isolation (June 5, 2026, preview): namespaces require RBAC to be enabled, scope OIDC and user management per namespace, and strip cross-namespace metadata from responses and errors. | Native authorization depth still varies by AI resource. Pinecone and Weaviate now have useful RBAC primitives, while Qdrant's own documentation still warns that self-hosted instances are open by default unless authentication, binding, and TLS are configured. Qdrant managed-cloud granular database keys are available for clusters on v1.11.0 and later, but older admin keys lack granular access. When vector databases cannot enforce the needed row, tenant, or purpose boundary themselves, the application must store labels in metadata and enforce them consistently through a PDP or gateway. Amazon Bedrock AgentCore Policy reached GA on March 3, 2026 with an external Cedar policy engine for agent-tool traffic and now supports natural-language policy authoring with automated-reasoning safety checks; Cerbos, OPA, Cedar/AWS Verified Permissions, ToolHive, and AuthZEN-compatible PDPs are practical choices for centralizing these checks, but they only help if every resource path is routed through the PEP. Two June 2026 releases tighten that PEP story: ToolHive v0.29.0 binds virtual MCP sessions to the OIDC identity rather than raw token bytes, blocking session replay by a different token holder, and OPA v1.17.0 (May 28, 2026) emits rule labels in decision logs so authorization decisions can be audited per classification tier or service. |
| **5.2.2** | **Verify that** privileged access to model weights, training pipelines, and production AI configuration is granted just in time, with a defined maximum session duration and automatic expiry. Permanent standing privileged access to these resources is not permitted. | 2 | Persistent privileged access to model weights enables insider or account-takeover exfiltration. Always-on access to training pipelines enables poisoning, unauthorized fine-tuning, or production configuration changes that persist after the original task is finished. Agentic systems make the risk sharper because a compromised agent runtime can use any standing service-account privilege it holds; MITRE's OpenClaw report and the Zenity Labs additions to MITRE ATLAS in October 2025 call out credential harvesting from agent configuration, cached SSH or OAuth material in agent memory, and unrestricted tool invocation as recurring attack paths. GitGuardian's 2026 report adds scale: more than 24,000 unique secrets were sitting in MCP configuration files on public GitHub, over 2,100 of them still valid - standing agent credentials with no expiry. Splunk's MCP server (CVE-2026-20205, fixed in 1.0.3 in April 2026) logged session and authorization tokens in clear text, letting users with internal-index access hijack other users' privileged sessions. | Enumerate every principal with write, admin, export, deploy, or key-management access to model registries, training pipelines, feature stores, production configuration, prompt-template repositories, and vector database administration. Confirm privileged roles are eligible rather than permanently active, require MFA or workload identity proof, have a maximum activation duration, and expire automatically. Test that access fails after expiry and that refresh tokens or long-lived API keys cannot silently recreate the session. Review Entra PIM, HashiCorp Boundary plus Vault dynamic credentials, CyberArk, BeyondTrust, Teleport, or equivalent PAM configuration for approval, session recording, and audit export. For agents, verify delegated access uses a session-bound token tied to the initiating user or task instead of a reusable privileged service account, and that the session token is scoped to a single tool call where possible. Where Vault is deployed, check for the Agent Registry and policy-intersection model from the native AI agent support announced May 12, 2026 (public beta expected summer 2026); in Entra environments, review the May 2026 agent-identity wave - Conditional Access templates for agent identities, time-bound access packages for on-behalf-of and autonomous agents, and lifecycle workflows that reassign sponsorship when an agent's owner leaves - as concrete JIT controls for agent principals. | Mature PAM tools handle human JIT access better than agent or pipeline identities. Microsoft Entra PIM supports time-based and approval-based activation, and Boundary can broker sessions with fixed limits and Vault-backed dynamic credentials, but AI training and deployment systems often still rely on IAM roles, static cloud keys, or shared CI secrets. April 2026 analyses of cloud PAM for AI agents argue that traditional session-based PAM was not built for the token-and-API-call pattern agents use, so JIT for agents typically means short-lived workload identities, capability tokens, and Slack or Teams approval workflows rather than RDP-style session brokering. Break-glass access should be time-bound, ticketed, and reviewed. A useful pattern is token exchange into a short-lived capability for a single model export, deployment, or tool call, with no persistent credential material stored in the agent runtime. The vendor and standards landscape started converging on that pattern in mid-2026: HashiCorp Vault's AI agent support computes effective permissions as the intersection of the human's policy, the agent baseline, and a ceiling policy, with ephemeral per-request authorization via OAuth 2.0 Rich Authorization Requests so permissions are bound to token lifecycle rather than standing identity, and the IETF individual draft `draft-araut-oauth-transaction-tokens-for-agents-02` (May 22, 2026, not yet working-group adopted) adds an `agentic_ctx` claim and `txn_path` lineage to OAuth transaction tokens so multi-agent call chains can carry per-hop trust downgrades. |
| **5.2.3** | **Verify that** a documented data classification taxonomy covering AI-specific data types (embeddings, model weights, prompt templates, RAG context assemblies, fine-tuning datasets, agent tool schemas) is defined. Verify that all AI production assets are labeled in accordance with this taxonomy. | 2 | AI assets are often misclassified because they do not look like ordinary documents or databases. Embeddings can encode sensitive source data; model weights can expose proprietary behavior or memorized data; prompt templates can contain system instructions and guardrails; RAG context assemblies and fine-tuning sets can combine records with different handling rules; agent tool schemas can reveal high-impact capabilities. Without an AI-aware taxonomy, these assets are stored, shared, or indexed at protection levels below their source data, and EU AI Act conformity assessments cannot reliably map a system to the Article 6 high-risk categories that drive the August 2, 2026 and December 2, 2027 obligations. | Request the classification taxonomy and verify that it explicitly covers embeddings, vector collections, model weights, adapters, evaluation sets, fine-tuning datasets, prompt templates, system prompts, RAG source corpora, RAG context assemblies, prompt and response logs, agent tool schemas, tool credentials, and API keys. Sample assets from each category and confirm labels are applied in metadata, catalog entries, storage tags, and policy inputs. Cross-check the inventory against NIST IR 8596's Cyber AI Profile (preliminary draft comment period closed January 30, 2026), the in-progress NIST SP 800-53 Control Overlays for Securing AI Systems (COSAiS) use cases for generative, predictive, single-agent, multi-agent, and developer-tooling controls, the European Commission's May 2026 draft guidelines on classification of high-risk AI systems under Article 6 (open for consultation through June 23, 2026), and against C5.2.4 for derived-label propagation. CSA's AI Security Maturity Model, released May 20, 2026 as a layered companion to the AI Controls Matrix, adds deployment-type fields (self-hosted, PaaS, API/SaaS) that help scope which assets the taxonomy must cover per deployment model. | General data-governance platforms can help, but none should be assumed to classify every AI artifact automatically. Microsoft Purview supports DSPM for AI, data classification, sensitivity labels, DLP, audit, eDiscovery, and compliance coverage for supported AI interactions, and Purview's Restricted tier now gates Microsoft 365 Copilot grounding regardless of file permissions; Google Sensitive Data Protection can discover and profile sensitive data across supported stores; Atlan, Collibra, and Informatica can attach governance metadata and lineage to common data assets. The remaining gap is AI-specific coverage: derived embeddings, prompt caches, agent tool schemas, and runtime context bundles usually require custom metadata fields, custom classifiers, ingestion hooks, and periodic sampling. That gap is starting to narrow: Google Sensitive Data Protection added a `Conversation` content type on June 3, 2026 for inspecting and de-identifying prompt and response transcripts - one of the first classifier features purpose-built for an AI-specific data type - and Purview's DSPM agent observability reached GA in May 2026 with an inventory of deployed Microsoft and non-Microsoft agents and per-agent data-risk posture, though it requires Microsoft 365 E7 or Agent 365 licensing. The expanded `BlockContentAnalysisServices` label setting (May 5, 2026) now blocks all connected AI content analysis for labeled Office files, not just Copilot grounding, but is PowerShell-only configuration. |
| **5.2.4** | **Verify that** data classification labels (PII, PHI, proprietary, etc.) automatically propagate to derived resources (embeddings, prompt caches, model outputs). | 3 | Classification loss during transformation lets sensitive data move into embeddings, prompt caches, model outputs, evaluation logs, or analytics tables without the controls applied to the source. This creates unauthorized retrieval, cross-tenant leakage, data-subject-rights failures, and retention-policy failures because derived artifacts look less sensitive than the records used to create them. The Spring AI ChatMemory advisory CVE-2026-41712 shows the same failure mode at runtime: when conversation IDs default implicitly, memory rows from different users can be served back as if they were a single tenant's data. | Select source records labeled PII, PHI, confidential, export-controlled, or proprietary and trace them through ingestion, chunking, embedding generation, vector storage, prompt assembly, response caching, evaluation logging, and output persistence. Verify the derived record carries the source label, lineage pointer, tenant or purpose tag, retention rule, and access policy input. Attempt retrieval as a user or agent that lacks the source entitlement and confirm both pre-retrieval filtering and post-retrieval enforcement deny access. Review pipeline tests that fail builds when label metadata is missing from embeddings, caches, or outputs, and treat retrieval auditing as part of the AI logging requirements in C13. Azure AI Search's 2026-05-01-preview API is a useful propagation reference implementation: SharePoint ACLs, including site groups with incremental ACL re-sync, are projected via `indexProjections` onto every vector chunk so document permissions survive chunking and embedding, and Purview sensitivity labels can be preserved into the index - test that `x-ms-query-source-authorization` enforcement actually trims results per caller rather than trusting the application tier. | Tooling is improving, but propagation still depends heavily on pipeline design. Atlan can propagate tags to downstream assets through lineage when enabled, with column-level lineage updating in real time for supported sources; Collibra can carry classifications through stewardship workflows; Microsoft Purview sensitivity labels persist with supported files and are recognized by Microsoft 365 Copilot and agents when those labels and usage rights are available. Those controls do not automatically label every vector, prompt cache entry, model output, or third-party retrieval store. Vector stores can only enforce classification if labels are carried into metadata and every query path includes the correct filters or PDP decision. Treat missing labels on derived artifacts as a security defect, not as incomplete documentation. The first mainstream automatic derived-label propagation is now shipping: Teams meeting transcripts, recordings, and Loop notes inherit the meeting's sensitivity label (GA expected June 2026, off by default and admin-enabled per tenant), and Copilot-generated content in Word, PowerPoint, and Outlook inherits the highest-priority source label. These cover Microsoft 365 artifacts only - embeddings, prompt caches, and third-party retrieval stores still need pipeline-level labeling. |

## Current Research Notes

- The LiteLLM exploitation chain is the cautionary tale of early June 2026 for C5.2.1: the `/mcp-rest/test/*` endpoints that the April 2026 CVE-2026-30623 fix restricted to `PROXY_ADMIN` turned out to be reachable pre-auth when chained with the Starlette TrustedHost bypass CVE-2026-48710, and CISA placed CVE-2026-42271 on the Known Exploited Vulnerabilities catalog on June 8, 2026 with a June 22 remediation deadline. The lesson generalizes: an AI gateway is a credential concentrator, so "admin-only" endpoints that spawn subprocesses deserve the same default-deny scrutiny as anonymous ones, and KEV-listed AI infrastructure should be tracked in the same patch SLA process as everything else.
- MITRE's Secure AI v2 release on May 6, 2026 added more than 45 techniques and sub-techniques, 10+ mitigations, and 20+ case studies to ATLAS, introduced a Technique Maturity filter and the first ATLAS Rapid Response Report, and moved the project to a monthly release cadence - worth re-checking monthly when mapping C5.2 threats to technique IDs.
- MCP exposure measurements firmed up in June 2026: Censys counted 12,520 internet-accessible MCP services (most unauthenticated), a remote-MCP authentication study found roughly 40% expose tools with no authentication, and Akamai disclosed database-MCP flaws including SQL injection in the Apache Doris MCP server and unauthenticated metadata exfiltration in the Alibaba RDS MCP server. Default-deny for MCP endpoints remains the exception, not the norm.
- Model registries deserve a standing line item in C5.2.1 test plans. MLflow shipped three authorization fixes in spring 2026 (CVE-2026-2651 critical multipart-upload exposure, CVE-2026-2652 unauthenticated Job API and trace-ingestion routes, CVE-2026-2734 cross-model version enumeration), all in the registry and pipeline layer where model weights and lineage live.
- On the standards side, the EU digital omnibus on AI cleared the European Parliament's IMCO and LIBE committees on June 2, 2026 with the plenary vote expected in the June session; GPAI provider enforcement still begins August 2, 2026 with fines up to 3% of global turnover. NIST's COSAiS overlays and the IR 8596 Cyber AI Profile initial public draft remained unpublished as of June 9, 2026, so the January 2026 preliminary materials are still the working references.
- The OpenID AuthZEN working group is developing an MCP Tool Authorization profile (proposal issue #429) that maps MCP JSON-RPC tool, resource, and prompt calls to AuthZEN evaluation requests with an `agent` subject type, positioning MCP servers and gateways as standard PEPs - the cleanest emerging answer to "how do I put a PDP in front of agent tool calls" raised throughout this section.
- As of May 2026, vector-store authorization needs explicit testing below the application UI. Milvus CVE-2026-26190 exposed unauthenticated business APIs on a management port, while Spring AI CVE-2026-22729 and CVE-2026-22730 showed that metadata filters used for tenant and role isolation can be bypassed when filter expressions are built from unescaped input.
- Spring AI 1.0.7 and 1.1.6 added fresh security fixes on May 8, 2026. CVE-2026-41705 is especially relevant to C5.2.1 because unsanitized document IDs in `MilvusVectorStore#doDelete(List)` could alter vector-store deletion filters; CVE-2026-41712 is relevant to C5.2.4 because an implicit default conversation ID could mix chat memory across users unless applications pass an explicit conversation identifier.
- AuthZEN 1.0 reached Final specification status on January 11, 2026 and the Authorization API 1.0 was published as a Standards Track document later in Q1 2026. It gives teams a practical interoperability target for PDP and PEP integration: request attributes should include subject, resource, action, and context, and AI gateways should preserve enough context for policy decisions to be replayed during audit. Cerbos, Aserto, SGNL, AWS, and Axiomatics demonstrated live AuthZEN interoperability at the 2025 Gartner IAM Summit events.
- The EU Council and Parliament reached political agreement on the AI Act omnibus on May 7, 2026. High-risk obligations in biometrics, critical infrastructure, education, employment, migration, asylum, and border control now apply from December 2, 2027, but the agreement reinstates the obligation for providers to register systems they self-exempt from high-risk classification - which only works if AI assets and their derived artifacts carry consistent classification labels.
- NIST is layering several artifacts on top of CSF 2.0 and the AI RMF: the Cyber AI Profile (NIST IR 8596) closed its 45-day preliminary-draft comment window on January 30, 2026, an AI RMF Profile concept note for trustworthy AI in critical infrastructure landed on April 7, 2026, and SP 800-53 Control Overlays for Securing AI Systems (COSAiS) are in progress with five use cases including single-agent, multi-agent, and AI developer controls.
- Asset-governance products are starting to cover AI-specific resources, but coverage is uneven. Microsoft Purview supports classification, sensitivity labels, audit, and DSPM for Entra-registered AI apps, with the Restricted tier now gating Microsoft 365 Copilot grounding regardless of user permissions; Collibra's AI Governance operating model adds assets such as AI agents, AI endpoints, model deployments, and agent-governance records. Neither removes the need to label embeddings, prompt caches, memory rows, and vector metadata at ingestion time.
- MITRE ATLAS v5.3.0, released in January 2026 with contributions from Zenity Labs, added 14 new agent and generative-AI techniques in October 2025 and three case studies for MCP server compromise, indirect prompt injection over MCP, and malicious AI agent deployment - all of which assume some combination of missing tool authorization, missing identity scoping, or shared agent credentials.
- Pinecone introduced Dedicated Read Nodes in December 2025, which isolate read workloads from writes per index and can be combined with API-key roles to harden multi-tenant retrieval paths. Pair them with caller-context filters in C5.3 to avoid degenerating into a "shared service account, everyone wins" pattern.
- ArcadeDB CVE-2026-44221 (CVSS 9.0, published May 12, 2026, fixed in 2.6.4) is a useful reference defect for C5.2.1: a multi-model DBMS often pressed into service as a vector store let API tokens scoped to a single database read, write, and modify schema on any other database on the same server. The defect was a defaulted-to-allow-all access map combined with a missing security factory on REST-created databases, which is exactly the failure mode the "default-deny" half of C5.2.1 is meant to catch during cross-resource testing.
- The NCCoE published "Accelerating the Adoption of Software and AI Agent Identity and Authorization" as an initial public-draft concept paper on February 5, 2026, with comments closing April 2, 2026. It treats AI agents as distinct identities, applies OAuth 2.0-style authorization to their rights, and ties access delegation back to the initiating user identity for accountability - a useful reference when arguing internally for capability tokens and per-call delegation rather than shared agent service accounts. NIST's Center for AI Standards and Innovation formally launched the AI Agent Standards Initiative on February 17, 2026 to coordinate the broader interoperability and security work, including OAuth 2.0, OpenID Connect, SPIFFE/SPIRE, Next Generation Access Control, and Model Context Protocol profiles for agent identity.
- The Anthropic MCP stdio command-injection design issue, tracked as CVE-2026-30623 and disclosed by OX Security on April 15, 2026, is the most consequential authorization story of the quarter for C5.2.1. Anthropic confirmed the stdio behavior is "by design", which moves the burden squarely onto the deploying organization: anything that lets a caller register an MCP server entry must enforce strict allow-listing of executables, run the launcher in a sandbox with no access to host credentials, and route every tool invocation through a PEP that can deny new commands without a code deploy. Treat the LiteLLM patch pattern (fixed command allowlist plus admin-only preview endpoints) as a baseline rather than a ceiling.
- The European Commission published draft guidelines on the classification of high-risk AI systems under Article 6 in May 2026 with a public consultation open through June 23, 2026. These guidelines pair with the May 7, 2026 omnibus to define which Annex III categories trigger the December 2, 2027 obligations, so the C5.2.3 taxonomy should be able to map labeled AI assets to specific Annex III rows and capture the documented justification when a provider claims the Article 6(3) exemption.
