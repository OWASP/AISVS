# C5.5: Policy Decision Point Isolation

> [Back to C05 Index](C05-Access-Control.md)
> **Last Researched:** 2026-06-10 (added the May 27 Starlette "BadHost" Host-header authentication bypass CVE-2026-48710 that undercuts path-based PEP middleware across FastAPI-built MCP gateways, CERT Polska's still-unpatched Code Runner MCP no-auth RCE CVE-2026-5029, Adversa AI's TrustFall (May 7) and SymJack (May 26) coding-agent trust-dialog and symlink config-overwrite chains, the Azure MCP managed-identity SSRF CVE-2026-26118, and three new authorization building blocks: the Open Agent Passport pre-action authorization study (arXiv 2603.20953), the AIP invocation-bound capability-token protocol (arXiv 2603.24775), and the April 30 IETF OAuth Actor Profile draft formalizing `act` delegation chains with entity-type claims. ## Purpose

Ensure that the authorization decision infrastructure for AI agents is protected from compromise or manipulation by the agents it governs. A policy decision point (PDP) that shares an execution surface with the agent runtime cannot be trusted to enforce authorization, because a compromised agent can influence evaluation inputs, alter policy state, or observe decision logic in ways that a segregated PDP would not permit. This section covers the runtime isolation boundary between agent execution and the PDP that evaluates each agent action, complementing the agent-specific authorization controls in C9.6.

> **Scope note:** Agent authorization policy design (scoped capabilities, delegation, continuous evaluation) lives in C9.6. Single-system agent identity is in C9.4.1. This section focuses narrowly on the runtime isolation of the PDP itself so that the surface a compromised agent can reach does not include the decision engine.

| **5.5.2** | **Verify that** the policy decision point receives structured action descriptions (e.g., action type, target resource, parameters) rather than the agent's raw reasoning context. | 3 | Passing raw chain-of-thought or prompt text to the PDP re-exposes the decision to prompt injection — the same class of attack (`AML.T0051`) the PDP is supposed to contain. MITRE ATLAS' 2025 additions `AI Agent Context Poisoning` and `Memory Manipulation` show how adversary-controlled text embedded in tool outputs, documents, or long-term memory can steer an agent's reasoning; if that reasoning is what the PDP sees, the PDP inherits the compromise. OWASP's 2025 LLM Top 10 places Prompt Injection (LLM01) at #1, noting it appears in over 73% of audited production AI deployments (obsidiansecurity.com, genai.owasp.org). The Trail of Bits October 2025 disclosures (Claude Code, Cursor, Amazon Q) all trace back to the same root cause: the enforcement layer saw only a command name like `find` or `grep` — not the full typed argument tree — so prompt-injected flags such as `-exec 'bash -c …'` rode through the allowlist. The Authensor gateway pattern (authensor.com, 2026) and the Cerbos MCP guide (cerbos.dev, 2025) both emphasize that the PDP must see a typed action envelope — never free-form text — with undeclared actions defaulting to DENY. The October 2025 AgentSentry paper (arxiv.org/abs/2510.26212, SaTS @ CCS 2025) goes further, arguing that even typed envelopes are insufficient if the allowed action set is static; it proposes task-centric access control that derives a temporary, minimal policy per task and revokes it when the task completes. The April 22, 2026 IETF individual draft **draft-chen-oauth-rar-agent-extensions-01** (expires October 24, 2026) updates this point at the protocol layer by extending OAuth 2.0 Rich Authorization Requests (RFC 9396) with an `intent_request` authorization-details type plus `policy_context` (so a client can demand a specific assurance level — e.g., `"financial_grade_v1"` — preventing a prompt-injected agent from silently opting into a weaker policy set) and `lifecycle_binding` (so the token is tied to an external task state and revoked the moment the task completes, fails, or is cancelled, closing the window where a leaked token outlives the work it was granted for). The companion `draft-brossard-oauth-rar-authzen-03` (July 2024) uses AuthZEN as the on-the-wire shape for `authorization_details`, making the RAR payload the same structured Subject-Action-Resource-Context envelope the PDP already consumes. The OWASP Agentic Top 10 2026 (**ASI03 Identity and Privilege Abuse**) specifically calls out agents that reuse user sessions or self-assert identity in authorization calls; the PEP must construct the subject from a workload identity, not trust the agent's serialized output. | Auditors should capture a sample of PDP request payloads and confirm they conform to a typed schema: a subject (agent identity), an action with a discrete `name` plus typed `properties`, a resource with `type` and `id`, and a context object with PEP-supplied environment attributes. OpenID Foundation members voted to approve **AuthZEN Authorization API 1.0 as a Final Specification on January 6, 2026** (81-1-25, openid.net/authorization-api-1-0-final-specification-approved); the final spec was published on **March 11, 2026** and refreshed in an editor's build dated April 29, 2026 (openid.github.io/authzen) — it is now the vendor-neutral shape for PDP payloads. Verify AuthZEN compliance against the official spec editors' reference implementations (Aserto/Topaz, Axiomatics, SGNL); PDPs such as Cerbos (v0.51.0, February 2026 — see cerbos.dev/authzen), Axiomatics, Styra, AWS Verified Permissions, and PlainID now ship AuthZEN-compatible endpoints, and the OPA community has an open tracking issue (open-policy-agent/opa#8449) plus a community plugin that turns OPA into an AuthZEN-compatible PDP. Validate via JSON Schema on the PEP side that no free-form reasoning, system prompt, or raw user utterance leaks into the payload. For OPA-based stacks, review the Rego input contract and confirm `input.action` is an enum or tagged union, not a string pulled from the LLM's output; run `opa test` with adversarial inputs. For Cedar, the schema file enforces this at compile time — AWS Bedrock AgentCore Policy (GA March 2026) uses this exact pattern to bind tool calls to typed Cedar actions before dispatch, injecting `context.toolName` from the OpenAPI `operationId` so the agent cannot rename the action in its output. Fuzz the PEP with prompt-injected agent outputs and confirm the PEP refuses to forward anything that does not match the schema; the Trail of Bits post-mortem explicitly recommends argument separators (`--`), disabling shell invocation, and drastically reducing allowlists over regex-based filtering. The Open Agent Passport study (arXiv 2603.20953, March 2026) gives auditors a measured benchmark for what deterministic pre-action authorization buys: a synchronous tool-call interceptor evaluating declarative policies and emitting cryptographically signed audit records held median enforcement latency to 53 ms, and across 1,151 bounty-incentivized adversarial sessions the social-engineering success rate fell from 74.6% under permissive policies to 0% across 879 attempts under restrictive ones — independent confirmation of the deterministic-enforcement numbers Microsoft's Agent Governance Toolkit reported in April 2026. | Many homegrown agent frameworks still serialize the model's tool-call JSON directly to the PDP, which works when the framework parses the tool call strictly but fails when it forwards the raw `function_call.arguments` string, trusting the model to have produced valid structure. OpenAI and Anthropic function-calling produces structured JSON, but agents that synthesize tool calls via free-form text (ReAct-style) are at higher risk. AuthZEN 1.0 was approved as an OpenID Final Specification on January 6, 2026 and published March 11, 2026, the MCP profile draft addresses parameter-level mapping, the April 22 `draft-chen-oauth-rar-agent-extensions-01` begins filling the agent-attribute gap at the OAuth layer with `intent_request`, `policy_context`, and `lifecycle_binding`, and the IETF informational draft **`draft-klrc-aiagent-auth-01`** (Kasselman/Lombardo/Rosomakho/Campbell/Steele, published March 30, 2026; expires September 3, 2026) sketches how WIMSE workload identity plus OAuth 2.0 Token Exchange with the `act` claim can carry an agent's delegator/task/tenant/session envelope through multi-hop agent chains — but the spec ecosystem remains deliberately agnostic about delegation chains, model identifiers, and capability tokens; those go in the extensible `properties` object for now and are a likely extension point for a future AuthZEN Agent Profile. Two newer pieces start filling that gap. The April 30, 2026 **draft-mcguinness-oauth-actor-profile-00** makes (`act.iss`, `act.sub`) a canonical actor identifier, adds a `sub_profile` entity-type claim so a PDP can distinguish user, service, AI-agent, and workload actors, and standardizes nested `act` objects for delegation chains across JWT assertion grants, JWT access tokens, and Transaction Tokens (the companion **draft-araut-oauth-transaction-tokens-for-agents-01** covers the transaction-token leg). On the research side, the **AIP** proposal (arXiv 2603.24775, March 2026) binds capability tokens to individual invocations — compact JWTs for single-hop calls, Biscuit tokens with Datalog policies for multi-hop delegation — and reports sub-millisecond verification (0.049 ms Rust / 0.189 ms Python) with a 100% rejection rate across 600 attack attempts, including delegation-depth violations that JWT-only schemes miss; its finding that roughly 2,000 scanned internet-reachable MCP servers all lacked authentication is the systemic backdrop for this entire control. Neither has production deployments yet. No current tool automatically detects that a PEP is leaking raw reasoning into the PDP — this is still a code-review finding. Tool recommendations for the PEP boundary: `pydantic` or `zod` for the typed envelope, `jsonschema` on the PDP side, and `promptfoo`, `DeepTeam`, or `Garak` to red-team the boundary with MITRE ATLAS-aligned prompt-injection payloads. |

## Community Notes

A recurring point of confusion in 2025–2026 architecture reviews: teams conflate "the PDP is in a separate file" with "the PDP is isolated." Running a Rego policy inside the agent's Python process, or evaluating Cedar policies via an SDK linked into the agent runtime, gives the agent read and write access to the policy store, the decision cache, and often the PDP's memory. That is not isolation in the sense this control requires — a compromised agent can mutate policy before the next call, tamper with cached decisions, or simply return ALLOW from the in-process client stub. The bar for 5.5.1 is a separate trust zone: a different process with a distinct workload identity, a network boundary, and a PEP that mediates all agent-to-PDP traffic.

A second recurring issue: using the agent's own output tokens as the authorization request body. If the agent is what constructs `{"action": "transfer_funds", "amount": 50000}` and the PEP forwards that verbatim, then a prompt-injected agent that invents an `action` string the policy silently allows (or a numeric edge case the policy didn't consider) bypasses the PDP even though the PDP "ran." The PEP, not the agent, must construct the request payload from validated inputs — the agent's role is to propose, the PEP's role is to describe the proposal to the PDP in typed form.

A third pattern, surfaced by the April 2026 CrewAI and PraisonAI CVE clusters: a fallback path that silently re-enables an unsafe execution mode. CrewAI's code interpreter fell back from Docker to an in-process Python sandbox when Docker was unavailable (CVE-2026-2287), and PraisonAI's AST allowlist could be defeated by a `str` subclass with an overridden `startswith()` (CVE-2026-34938). Both are PDP-shaped failures even when no explicit PDP is wired in: the "policy" was the sandbox's allowlist, and the enforcement layer shared a process with the attacker-controlled code it was meant to constrain. Treat any in-process allowlist as advisory only — the authoritative decision must live behind a process boundary the agent cannot rewrite.

A fourth pattern, driven home by the Kyverno disclosures of February–April 2026: the PDP itself is a privileged workload and must be hardened the same way any other privileged component is. CVE-2026-22039's policy `apiCall` authorization bypass and CVE-2026-4789's unrestricted CEL `http.Get()` both let an attacker turn the PDP's own admission controller into an SSRF or cross-namespace pivot — the decision engine's broad RBAC, intended for evaluating cluster-wide policy, became the attacker's capability. "Isolated" therefore means more than "separate process": the PDP's own request path needs its own authorization model, its outbound network calls need an egress policy, and user-authored policy inputs (CEL, Rego, Cedar fragments) must be treated as untrusted data evaluated inside a sandbox, not as configuration the engine will faithfully execute with elevated credentials. Similarly, the April 2026 nginx-ui MCPwn incident (CVE-2026-33032) and the Azure MCP Server disclosure (CVE-2026-32211) are reminders that an MCP server cannot double as its own PDP; pair MCP with an external PDP and verify every tool route carries the auth middleware, not just the headline one.

A fifth pattern, visible in the late-April OpenClaw advisories, is inconsistent PEP coverage across routes that appear to perform the same sensitive action. CVE-2026-35660 described a Gateway `agent` path that accepted `/reset` and `/new` from `operator.write` callers even though the direct `sessions.reset` RPC required `operator.admin`; CVE-2026-41298 then showed read-scoped callers terminating subagent sessions through `POST /sessions/:sessionKey/kill` before 2026.4.2. Both cases are small in implementation terms but important architecturally: an isolated PDP only helps when every route that can trigger an action is forced through the same decision contract and the PEP cannot quietly reinterpret read, write, and admin scopes.

An eighth pattern, surfaced by Akamai's May 13, 2026 MCP back-end research, is treating the MCP server as the security perimeter while its back-end speaks in raw SQL or exposes raw metadata. Apache Pinot MCP, Apache Doris MCP (CVE-2025-66335), and Alibaba RDS MCP all shipped with the assumption that whatever fronts the MCP transport would handle authentication and the SQL layer would validate inputs — none of which held in practice. For 5.5.1 and 5.5.2, the practical advice is to draw the trust boundary at the MCP-tool-to-data-source edge: the PEP that authorizes a tool call must derive the canonical resource, the typed argument set, *and* the back-end query shape before the back-end ever sees it, then validate that the back-end driver enforces parameterization. A vendor's "we'll add OAuth as an option" — without a default-on configuration, without query parameterization, and without a separate PDP evaluating the resulting query — should not be treated as compliance with this control.

The Five Eyes joint guidance *Careful Adoption of Agentic AI Services* (Australian Cyber Security Centre, CISA, NSA, Canadian Centre for Cyber Security, NCSC-NZ, NCSC-UK, published April 30, 2026; companion CISA release May 4, 2026) names the C5.5 design choice directly: "the decision about when human approval is required should be made by system designers, not delegated to the agent itself." The same document repeats the least-privilege rule that "agents should operate on strict least-privilege principles, with strong identity management, cryptographically anchored credentials, and clearly defined roles" and warns against granting broad permissions that "allow compromised components to inherit dangerous capabilities across interconnected systems." This is the first multinational government guidance to encode the externalized-PDP pattern in plain language; cite it during audits when the in-process toolkit camp pushes back on out-of-process enforcement.

AgentROA draft-01, published April 16, 2026, is worth tracking as an experimental direction rather than a deployable standard. It proposes a signed authorization envelope, per-hop attestation for delegated agent chains, and an execution receipt generated at the enforcement boundary. The useful idea for C5.5 is the topology-independent "border gateway" model: put the enforcement process outside the agent context, then cover MCP, service-mesh, egress-gateway, and domain-specific execution boundaries with the same policy envelope semantics.

A sixth pattern, visible in the April-May 2026 MCP STDIO disclosures, is treating connector configuration as harmless metadata when it is really executable policy surface. OX Security's April 15 advisory traced a family of command-execution findings across LangFlow, GPT Researcher (CVE-2025-65720), LiteLLM (CVE-2026-30623), Agent Zero (CVE-2026-30624), Upsonic (CVE-2026-30625), Flowise (CVE-2026-40933), Windsurf (CVE-2026-30615), and related tools where MCP STDIO `command` / `args` fields became a route around the intended enforcement point. Cloud Security Alliance's April 23 research note framed the same issue as an architectural supply-chain risk: once a compromised agent or exposed UI can influence the local MCP configuration, the "tool call" has already crossed into process launch. OX's May 12 follow-up added kubectl-mcp-server (CVE-2025-65719), Archon OS (CVE-2025-69443), and MarkItDown file-read exposure as further evidence that the integration layer itself needs PDP coverage. The practical lesson for 5.5.1 and 5.5.2 is straightforward: an MCP server definition, local connector manifest, or IDE tool registry must be treated like privileged configuration, not like agent-editable context. Adversa AI's May 2026 disclosure pair pushed the same point into the IDE-agent mainstream. **TrustFall** (May 7, 2026; helpnetsecurity.com) showed that Claude Code, Gemini CLI, Cursor CLI, and GitHub Copilot CLI all auto-execute project-scoped MCP helpers the moment a developer reflexively accepts a generic trust dialog — before the model performs any reasoning, and with no dialog at all on CI runners — with the researchers noting that each tool "read project-scoped MCP configuration" behind materially different approval language. **SymJack** (May 26, 2026; radar.offseq.com) achieved RCE across six coding agents by disguising a config overwrite as a benign file copy through symlinks, silently installing an attacker-controlled MCP server that runs with full user privileges on the next restart. Both are textbook `AML.T0073 Modify AI Agent Configuration` plays, and both confirm that a consent prompt rendered by the same process that will execute the attacker's manifest is not a PDP. Claude Code's centrally managed "Managed scope" settings — which let administrators disable project-scoped MCP auto-approval fleet-wide — are the right shape of fix: an enforcement decision moved out of the agent's own UI and into configuration the agent cannot rewrite.

A seventh pattern, surfaced by the May 2026 Microsoft "When prompts become shells" series (microsoft.com/security/blog, May 7, 2026) and the PraisonAI auth-disabled-by-default incident, is treating the agent runtime's *own* extension surface as policy-equivalent. Semantic Kernel's `[KernelFunction]` annotation (CVE-2026-25592, CVSS 10.0) and its `InMemoryVectorStore` filter parser (CVE-2026-26030) both exposed in-process code paths to an LLM-controlled string, and PraisonAI's `api_server.py` shipped with `AUTH_ENABLED = False` (CVE-2026-44338) — internet scans hit the vulnerable endpoint within three hours forty-four minutes of the May 11 advisory (sysdig.com, csoonline.com). The takeaway for 5.5.1: an isolated PDP only helps when there is no parallel path inside the agent process that accepts agent-influenced inputs and turns them into actions. Every framework annotation, default-on dev server, and "convenience" endpoint needs the same authorization contract, or none of them does. The April 21, 2026 Spring Security advisory **CVE-2026-22752** (CVSS 9.6, spring.io/security/cve-2026-22752) makes the corollary point on the PDP side: DCR endpoints that accept attacker-influenced client metadata can themselves become an SSRF / XSS / privilege-escalation surface (a `javascript:` `redirect_uri`, an `http://169.254.169.254` `jwks_uri`, or a crafted `logo_uri` that forces an outbound IMDS request), so the PDP and its surrounding IAM stack must validate every author-supplied URI scheme and host the same way they validate policy syntax.

## Implementation Guidance

**Deployment pattern for 5.5.1 (isolated PDP):**

1. Run the PDP as a dedicated workload — a Kubernetes Deployment in its own namespace with a distinct ServiceAccount, an Azure Function behind Entra as in the Microsoft reference (techcommunity.microsoft.com, April 2026), or an AWS Verified Permissions policy store accessed via IAM-scoped API calls. Do not co-locate with the agent.
2. Apply network isolation: Istio or Linkerd AuthorizationPolicy that allows only the PEP's ServiceAccount to reach the PDP port; deny-all default for everything else. For cloud-native deployments, use VPC Service Controls (GCP), PrivateLink (AWS), or Private Endpoints (Azure).
3. Use workload identity (SPIFFE/SPIRE, Azure Managed Identity, AWS IRSA, GCP Workload Identity) for PEP-to-PDP authentication. The agent must not hold this credential.
4. Store policy as immutable, signed artifacts (OCI images, Git-backed with commit signing, or AWS Verified Permissions policy templates). The agent's filesystem must be read-only with respect to policy. Use OPAL or Styra DAS for controlled policy distribution.
5. Check route coverage, not just the main PEP. Exercise alternate RPCs, MCP routes, background job handlers, session-control endpoints, and admin convenience paths to confirm they all call the same PDP with the same subject/action/resource/context contract.
6. Treat MCP STDIO configuration as privileged code. Disable agent- or prompt-writable connector manifests, pin executable paths, block arbitrary `command` / `args` registration through web UIs, and run every local MCP server inside a sandbox with a narrow filesystem and network profile. TrustFall and SymJack (May 2026) are the live exploits for this step: deploy centrally managed configuration scopes (e.g., Claude Code managed settings) so project-scoped MCP auto-approval is disabled fleet-wide, and treat any symlinked file copy into an agent's config directory as a write to privileged policy.
7. Log every decision with attribution to the PEP identity and the structured action, not the agent's reasoning. Ship to an append-only store (S3 Object Lock, Azure immutable blobs, Loki behind write-only credentials).

**Deployment pattern for 5.5.2 (structured action envelopes):**

1. Define a schema. AuthZEN 1.0 (openid.github.io/authzen) gives a vendor-neutral shape — `subject`, `action`, `resource`, `context` — that works across Cerbos, Axiomatics, Styra, AWS Verified Permissions, and PlainID. For Cedar, define an action-entity schema. For OPA, publish a Rego input contract plus a JSON Schema validated at the PEP boundary.
2. Construct the envelope in the PEP from validated fields: the tool name (from a closed enum of declared tools), the resource identifier (canonicalized, not a free-form URL), the action parameters (typed and range-checked), and agent identity (workload token, not a self-asserted string).
3. Reject any PEP input that does not match the schema. Do not coerce. Do not forward extra fields from the agent's output.
4. Default-deny undeclared actions. Cerbos, Cedar, and the Authensor gateway pattern (authensor.com, 2026) all recommend a final DENY rule catching anything not explicitly allowed.
5. For MCP tool calls, do not stop at OAuth scopes or the top-level tool name. The PEP should derive the canonical resource, route, action, and typed arguments from the tool registry and runtime request, then reject undeclared parameters, implicit STDIO commands, and connector manifests that do not match the approved schema.
6. Red-team with MITRE ATLAS-aligned payloads. `promptfoo`, `DeepTeam`, and `Garak` each carry prompt-injection suites; route their outputs through your agent and verify the PEP refuses malformed envelopes.
7. For OAuth-backed delegation, track draft-chen-oauth-rar-agent-extensions-01: it adds an `intent_request` authorization-details type and keeps `policy_context` plus `lifecycle_binding` as explicit inputs, which gives auditors concrete fields to test for policy downgrade and stale-token reuse.

**Engine selection notes (as of May 2026):**

- **Cedar / AWS Verified Permissions** — strongest isolation story, deterministic, type-safe, formally verifiable. Good fit when the authorization model is primarily RBAC/ABAC.
- **OPA / Rego** — most flexible, widest ecosystem, but Apple's August 2025 acquisition of the maintainers and Styra's enterprise sunset have created roadmap uncertainty; factor that in. Still the right choice for complex cross-cutting policy.
- **Oso Cloud / Polar** — opinionated for ReBAC and relationship-based agent delegation; pairs well with multi-tenant agent platforms.
- **OpenFGA / Zanzibar-style** — best when authorization is primarily about relationship graphs (agent-owns-resource, user-delegates-to-agent).
- **Cerbos** — purpose-built PDP, ships with AuthZEN support, strong MCP integration story (cerbos.dev, 2025).
- **Border-gateway enforcement** — not a PDP language, but a useful deployment pattern for MCP/A2A and egress-heavy agents. AgentROA draft-01 is still work in progress, yet its signed envelope and receipt model is a good checklist for route coverage and auditability.

## Open Research Questions

- [ ] What is the right latency budget for out-of-process PDP evaluation on an agent hot path? Microsoft's in-process toolkit targets <0.1 ms p99; an isolated PDP typically runs 1–10 ms; the Open Agent Passport study measured 53 ms median for full pre-action interception with signed audit records, while AIP reports 0.22 ms for token verification alone. Where does the security/latency tradeoff land for different agent classes (synchronous chat vs long-horizon autonomous)?
- [ ] How should PDPs handle delegation chains? If agent A spawns agent B, does B carry A's authorization token, a derived capability, or re-authenticate from scratch? AuthZEN 1.0 has no agent-delegation primitives yet; draft-mcguinness-oauth-actor-profile-00 (nested `act` chains with entity typing) and AIP's invocation-bound capability tokens are the first concrete answers, but neither has production deployments, and AgentROA is still an individual Internet-Draft.
- [ ] What is the correct threat model when the PDP itself runs a policy authored (or suggested) by an LLM? "AI-generated policy" is appearing in vendor demos but has no audit guidance.
- [ ] Is there a reference architecture for PDP evaluation against long-term agent memory stores (vector DBs, episodic memory) where the data the policy reads may itself be poisoned (MITRE ATLAS `AI Agent Context Poisoning`)?
- [ ] How do organizations reconcile the MCP model (where the MCP server is often itself the PEP) with out-of-process PDP evaluation? MCP-native authorization patterns are still converging.
- [ ] Can execution receipts become practical evidence for auditors without creating a second sensitive log store full of agent actions, resource IDs, and business intent?
