# C10.1: Component Integrity & Supply Chain Hygiene

> **Parent:** [C10 MCP Security](C10-MCP-Security)
> **Requirements:** 2 | **IDs:** 10.1.1–10.1.2

## Purpose

This section ensures that MCP server and client software is obtained from trusted sources, that configuration artifacts do not leak credentials, and that only approved MCP servers are permitted in production. MCP components are typically distributed as npm packages, Python packages, Docker images, or standalone binaries — all of which are susceptible to supply-chain attacks including typosquatting, dependency confusion, and build-time tampering. Because MCP servers execute arbitrary code with access to sensitive resources (databases, APIs, file systems), compromised components have outsized blast radius compared to typical library dependencies.

The severity of these risks was demonstrated in real-world incidents throughout 2025-2026. The Postmark MCP supply chain breach (2025) saw attackers backdoor an npm package used in AI automation pipelines, inserting a single line of malicious code that directed compromised MCP servers to BCC every outgoing email to an attacker-controlled address. The Smithery supply chain attack (October 2025) affected 3,000+ hosted applications and their API tokens. Research using the MCPTox benchmark showed that tool poisoning attacks — where malicious instructions are embedded in tool metadata invisible to human users but processed by AI models — pass seamlessly into agent contexts with alarming frequency. A 2026 survey of 2,614 MCP implementations found 82% use file operations vulnerable to path traversal, two-thirds have code injection risks, and over a third are susceptible to command injection. AgentSeal's April 2026 scan of 1,808 MCP servers (sourced from GitHub repositories, npm and PyPI packages, and public registries including Smithery and MCP.run) found 66% had at least one security finding — 43% involved shell or command injection, 20% targeted tooling infrastructure, 13% were authentication bypasses, and 10% exploited path traversal. These findings underscore why component integrity and supply-chain hygiene are foundational to MCP security. The ecosystem's governance is also maturing: the Linux Foundation launched the Agentic AI Foundation in December 2025 to provide open governance over MCP standard development, and the MCP specification's November 2025 revision (2025-11-25) introduced formal governance structures including working groups and SDK tiering — though as of April 2026 the spec still stops short of mandating integrity verification or supply-chain controls for servers, and Ox Security's April 15, 2026 disclosure of a systemic STDIO-execution flaw in the official MCP SDKs (Python, TypeScript, Java, Rust — covering roughly 150 million cumulative downloads and an estimated 200,000 vulnerable instances) demonstrated that architectural assumptions baked into the reference SDKs can inherit through to every downstream server, with Anthropic declining to patch on the grounds that STDIO is a "secure default" and sanitization is the developer's responsibility.

## Real-World Incidents & Research (2025-2026)

- **WhatsApp MCP Tool Poisoning (April 2025):** A malicious MCP server silently exfiltrated a user's entire WhatsApp chat history through tool poisoning, exposing personal conversations and business data. One of the earliest documented cases of MCP-specific data exfiltration.
- **GitHub MCP Prompt Injection (May 2025):** A prompt-injection attack against a GitHub MCP integration enabled theft of private repository contents, internal project details, and personal financial information. Overly broad API tokens amplified the blast radius.
- **Asana MCP Cross-Tenant Leak (June 2025):** An access control flaw in an Asana MCP integration permitted cross-tenant data visibility between different customer organizations.
- **Anthropic MCP Inspector RCE (June 2025):** An unauthenticated remote code execution vulnerability in the MCP Inspector development tool exposed developer workstations' filesystems, API keys, and environment secrets.
- **CVE-2025-6514 — mcp-remote Command Injection (July 2025):** Critical command-injection bug (CVSS 10.0) in the `mcp-remote` npm package affected 437,000+ downloads. The package's OAuth proxy passed malicious endpoints directly to system shell handlers (PowerShell on Windows, shell commands on Unix), enabling credential theft, Git repository exfiltration, and full system compromise. The first documented case of full system compromise achieved through MCP infrastructure.
- **Anthropic Filesystem MCP Sandbox Escape (August 2025):** Sandbox escape vulnerabilities allowed arbitrary file access through symlink bypass and containment flaws in the official filesystem MCP server.
- **Postmark MCP Supply Chain Breach (September 2025):** Attackers injected a single line of malicious code into an npm package used for sending transactional emails through Postmark in AI automation pipelines. Compromised MCP servers silently BCC'd every outgoing email to the attackers.
- **Smithery Registry Path Traversal (October 2025):** A path-traversal bug in the Smithery.ai build configuration exposed Docker credentials controlling 3,000+ hosted MCP servers and their API tokens.
- **Figma MCP Command Injection (October 2025):** A command-injection flaw via unsanitized input in the Figma/Framelink MCP integration enabled shell execution and design data access.
- **Supabase Cursor Agent Exploit (mid-2025):** Supabase's Cursor agent, running with privileged service-role access, processed support tickets containing user-supplied input as commands. Attackers embedded SQL instructions to exfiltrate sensitive integration tokens by leaking them into a public support thread.
- **CVE-2026-21852 — Claude Code Project File RCE (January 2026):** Critical vulnerability allowing remote code execution and API credential theft through malicious project configuration files.
- **MCP CVE Surge (early 2026):** 30+ CVEs filed against MCP implementations in a 60-day period, covering SSRF, command injection, path traversal, and authentication bypass. Over 40% of public MCP servers tested were susceptible to command injection.
- **8,000+ Exposed MCP Servers (2026):** Research identified over 8,000 publicly exposed MCP servers, with 492 lacking basic authentication or encryption.
- **GitGuardian Secrets Report (2026):** 24,008 unique secrets found in MCP-related configuration files across public GitHub, including 2,117 valid credentials. AI coding tools correlated with a doubling of secret leak rates.
- **mcp-server-git CVE Chain (CVE-2025-68145/68143/68144):** Three chained vulnerabilities in Anthropic's own `mcp-server-git` package combined with the Filesystem MCP server to achieve full remote code execution via malicious `.git/config` files. Demonstrated that even vendor-maintained MCP components can harbor critical security flaws.
- **CoSAI MCP Threat Analysis (early 2026):** The Coalition for Secure AI identified 40+ distinct MCP threat categories. Most organizations were addressing fewer than two categories, suggesting widespread underestimation of MCP attack surface.
- **Tool Poisoning Prevalence (2026):** Invariant Labs research found 5.5% of MCP servers in the wild exhibit tool poisoning characteristics — malicious instructions hidden in tool descriptions that are invisible to human users but processed by AI models.
- **Shadow MCP Servers (2026):** Developers increasingly connect MCP servers to IDEs without IT oversight, granting file system and shell access outside traditional security monitoring perimeters. These "shadow MCP" deployments bypass organizational security controls entirely.
- **MCPTox Benchmark (2025-2026):** Research benchmark evaluating how often malicious or manipulated tool definitions pass into AI agent contexts, finding tool poisoning attacks frequently result in unauthorized execution or data leakage.
- **CVE-2026-30856 — WeKnora Tool Execution Hijacking (2026):** A naming collision flaw (CWE-706) in Tencent's WeKnora framework allowed attackers to register malicious MCP servers with tool identifiers that overwrote legitimate tool references in memory, enabling indirect prompt injection and data exfiltration. Fixed in v0.3.0 with UUID-based tool identifiers and "first-wins" registration policy.
- **Unit 42 MCP Sampling Attack Vectors (2026):** Palo Alto Networks' Unit 42 research identified three novel attack vectors through MCP's sampling feature: resource theft via hidden prompts (servers inject covert instructions causing LLMs to generate unauthorized content), persistent conversation hijacking (injected directives alter AI behavior across multiple turns), and covert tool invocation (servers manipulate prompts to trigger unauthorized file system operations). These attacks exploit the bidirectional control reversal inherent in MCP sampling, where servers become active requesters of LLM completions with minimal oversight.
- **CVE-2026-23744 — MCPJam Inspector RCE (March 2026):** Critical RCE vulnerability (CVSS 9.8) in the MCPJam Inspector development platform (versions ≤1.4.2). The service bound to `0.0.0.0` instead of `127.0.0.1`, allowing remote attackers to send crafted HTTP requests that triggered MCP server installation and arbitrary code execution without authentication. Fixed in v1.4.3.
- **CVE-2026-26118 — Azure MCP Server SSRF (March 2026):** An SSRF vulnerability (CVSS 8.8) in Azure MCP Server allowed authenticated attackers to submit malicious URLs as tool parameters, coercing the server into making outbound requests that exposed its managed identity token. Captured tokens granted the attacker the full privileges of the MCP Server's Azure identity. Patched in Microsoft's March 2026 Patch Tuesday.
- **Qualys TotalAI Shadow MCP Discovery (March 2026):** Qualys research found MCP servers frequently evade traditional security tools by binding to localhost, using random high ports, or hiding behind proxies. Over 10,000 public MCP servers appeared within one year of the protocol's introduction, with many operating outside IT governance perimeters.
- **NeighborJack MCP Server Exposure (2026):** Hundreds of MCP servers found defaulting to `0.0.0.0` binding instead of `127.0.0.1`, exposing command injection paths to adjacent network hosts and enabling complete host system compromise when internet-accessible — even for servers intended to be local-only.

The incident is particularly relevant to MCP security because LiteLLM is a common transitive dependency in MCP server implementations and agent orchestration frameworks.

Google Threat Intelligence attributed the operation to the North Korea–nexus Sapphire Sleet cluster; the attack window lasted 179 minutes before the packages were yanked. Notably, for agentic workflows where Claude Code or Cursor run `npm install` autonomously, the traditional detection signal — a developer watching terminal output — is absent, making pre-install provenance gating the only reliable defense. [Huntress](https://www.huntress.com/blog/supply-chain-compromise-axios-npm-package), [Wiz](https://www.wiz.io/blog/axios-npm-compromised-in-supply-chain-attack), and [Microsoft](https://www.microsoft.com/en-us/security/blog/2026/04/01/mitigating-the-axios-npm-supply-chain-compromise/) published mitigation guidance within 24 hours.
- **CVE-2026-32211 — Azure DevOps MCP Server Missing Authentication (April 3, 2026):** Microsoft disclosed an information disclosure vulnerability (CVSS 9.1) in the `@azure-devops/mcp` npm package. The server lacked any authentication mechanism, exposing work items, repositories, pipelines, pull requests, API keys, and project metadata to unauthenticated callers. No patch was available at disclosure — Microsoft provided only mitigation guidance (restrict network access, use firewalls/reverse proxies). Independent monitoring flagged two pre-incident supply-chain risk signals: a `preinstall` script that rewrites the npm registry configuration on the installing host (the same vector abused in the axios campaign days earlier) and publication from a personal npm account without GitHub Actions trusted publishing, leaving no verifiable source-to-artifact provenance link. The incident underscores that allowlisting must evaluate both malware presence and architectural security controls — "does this server implement the security controls it should?" not only "does this package contain malware?"
- **Ox Security MCP STDIO Systemic Flaw (April 15, 2026):** Ox Security disclosed a critical architectural vulnerability in the official MCP SDKs (Python, TypeScript, Java, Rust): the STDIO transport executes launch commands even when the child process fails to start, providing no toolchain-level sanitization warning, and enabling arbitrary command execution on any system that connects to a malicious or compromised MCP server configuration. Ox estimated 150 million cumulative downloads across the affected SDKs, 7,000+ publicly accessible servers, and up to 200,000 vulnerable instances, spanning 200+ open-source projects. Ox issued 30+ responsible disclosures and identified 10+ high/critical CVEs in individual downstream projects. Anthropic declined to modify the protocol, characterizing STDIO as a "secure default" where input sanitization is the developer's responsibility. The implication for C10.1 is that cryptographic attestation of the server binary is necessary but not sufficient — every server built on an affected SDK inherits the transport-level exposure, so allowlisting must pin SDK versions in addition to server versions, and organizations operating MCP fleets should track SDK-level CVEs as a first-class dependency.
- **AgentSeal Scan of 1,808 MCP Servers (April 2026):** AgentSeal performed what is, to date, the largest systematic security audit of the MCP ecosystem — scanning 1,808 servers discovered through GitHub repositories, npm and PyPI packages, and public MCP registries including Smithery and MCP.run. 66% had at least one security finding: 43% shell or command injection, 20% tooling infrastructure issues, 13% authentication bypasses, and 10% path traversal. The detection pipeline combined regex signatures, deobfuscation, semantic embedding-based comparison against known attack patterns, and Claude Opus–based classification for context-aware severity. The result confirms that the distribution of MCP server vulnerabilities has not materially shifted from the early-2026 CVE surge despite the emergence of multiple scanner ecosystems.
- **Stacklok ToolHive Registry Update (April 13, 2026):** ToolHive's Registry Server added claim-based authorization: read, write, and admin operations are gated on JWT claims so only entries the caller's identity is authorized for are visible (unauthorized resources are hidden entirely rather than surfaced as permission errors). Per-entry access control for Kubernetes sources allows individual MCP server and skill entries to carry their own authorization claims, and immutable entry visibility prevents subsequent versions from broadening access. A `/me` endpoint exposes authenticated identity and roles in the pattern of GitHub and Microsoft Graph. MCPServers in Kubernetes now enforce per-user, per-tool call rate limiting on `tools/call` (shared, per-user, or hybrid) backed by Redis, returning HTTP 429 with `Retry-After` on exceeded limits and failing open if Redis is unreachable. The virtual MCP Server gained an embedded OAuth authorization server with Dynamic Client Registration and upstream identity-provider support (GitHub, Google, Okta). This release materially closes the gap between registry-level allowlisting and runtime tool-invocation enforcement.

## Open Research Questions

- [ ] Should the MCP ecosystem adopt a standardized package-signing mechanism? MCPS proposes ECDSA P-256 "Agent Passports" with five trust levels, but adoption is early-stage — is this the right foundation, or should the MCP spec itself incorporate signing requirements?
- [ ] How should MCP configuration files handle secret references in a cross-platform, cross-SDK way? Only ~8.5% of MCP implementations currently use OAuth 2.1 — what would it take to make secretless access the default?
- [ ] Is there a viable model for an MCP server registry with enforced integrity metadata? GitHub, AWS Q Developer, and Kiro now offer registry-based enforcement, but all rely on name-based matching rather than cryptographic attestation — can the ecosystem converge on a standard that ties server identifiers to verified binaries?
- [ ] How do you verify integrity of MCP servers that are dynamically discovered at runtime rather than pre-configured? MCPS's signed tool definitions are one approach, but runtime discovery introduces trust bootstrapping challenges.
- [ ] Can MCPTox-style automated benchmarking be integrated into CI/CD pipelines to detect tool poisoning before deployment?
- [ ] The MCP gateway debate: centralized gateways offer unified enforcement but create single points of failure. Should per-server enforcement (signed envelopes, per-connection authentication) replace or complement gateway architectures?
- [ ] How should organizations handle the "contextual secret leakage" problem, where credentials persist in agent context memory across sessions?
- [ ] Should AI-BOM (AI Bill of Materials) standards extend to MCP server dependencies? Traditional SBOMs miss model versions, training data lineage, and agentic dependency chains — is CycloneDX/SPDX sufficient, or does the MCP ecosystem need purpose-built transparency formats?
- [ ] How should organizations detect and govern "shadow MCP" deployments where developers connect servers to IDEs without IT oversight, bypassing security monitoring? Qualys TotalAI's three-layer detection (network/host/supply-chain) is one approach — are there lighter-weight alternatives?
- [ ] MCP sampling creates a bidirectional control reversal where servers can request LLM completions — should the protocol require explicit user consent for sampling requests, or can capability declarations and prompt sandboxing suffice?
- [ ] Tool shadowing via namespace collisions (demonstrated by CVE-2026-30856) is a gap in the MCP spec. Should tool identifiers be globally unique (UUID-based) by default, or is registry-backed namespace isolation sufficient?
- [ ] NIST's AI Agent Standards Initiative (February 2026) is collecting input on agent security and identity — should the MCP community align with NIST's forthcoming framework for AI agent identity and authorization, or develop MCP-specific standards independently?
- [ ] Enterprise platforms (GitHub Copilot, AWS Q, Kiro, Agentforce, Microsoft) each implement MCP governance differently. Can the ecosystem converge on a standard registration and verification workflow — identity verification, automated scanning, human review, cryptographic signing — or will fragmentation persist?
- [ ] ToolHive demonstrates container-based provenance verification using Sigstore, but most MCP servers are distributed as npm/PyPI packages rather than container images. How should provenance verification extend to non-containerized MCP components?
- [ ] The MCP gateway market is fragmenting (MintMCP, Kong, Bifrost, Lunar.dev, Docker, Lasso) with different enforcement models. Should the MCP spec define a standard gateway API or enforcement interface, or will market consolidation handle this?
- [ ] Lockfile-based integrity verification (MCPShield's SHA-512 `mcp.lock.json`, MCPTrust's capability-pinned lockfiles) provides deterministic snapshots enforceable in CI — but lockfiles verify capability contracts, not binary integrity. Can lockfile verification be composed with cryptographic attestation to close the gap, or are these fundamentally different trust layers?
- [ ] The Claude DXT architecture runs extensions unsandboxed with full system privileges, and Anthropic's position that this "falls outside our current threat model" leaves users exposed. Should MCP hosts be required to sandbox MCP server extensions, or is this an acceptable tradeoff for local development tooling? How should this be reflected in verification standards?
- [ ] DPoP (SEP-1932) and Workload Identity Federation (SEP-1933) are under review for the MCP spec. If ratified, these could eliminate static client secrets and bind tokens to specific client instances — but adoption requires coordinated changes across all MCP SDKs. What is the realistic timeline for this?
- [ ] The Proofpoint Agent Integrity Framework proposes a five-phase maturity model for agent verification. Should MCP server registries require attestation of a minimum maturity level before listing, similar to how app stores require security reviews?
- [ ] The TeamPCP campaign demonstrated that transitive dependencies (e.g., LiteLLM as an AI gateway library) can propagate supply chain compromises silently into MCP servers and agent frameworks. Should MCP integrity verification require recursive dependency scanning, and how deep should the scan go? Current SBOM tooling captures direct dependencies but often misses build-time and runtime transitive chains.
- [ ] The MCP security scanning landscape now includes at least five major tools (Cisco MCP Scanner, ToolTrust, MCPShield, mcpscan.ai, Snyk agent-scan) with overlapping but non-identical coverage. Should the ecosystem converge on a common vulnerability taxonomy and reporting format for MCP-specific findings, analogous to CWE for traditional software?
- [ ] The Ox Security April 2026 STDIO disclosure demonstrates that vulnerabilities baked into the official MCP SDKs propagate to every downstream server built on them. Should allowlisting and component-integrity verification require SDK version pinning as a first-class check alongside server name and version — and should MCP registries track SDK-level CVE exposure per listed server, analogous to how container registries track base-image CVEs?
- [ ] When a vendor declines to patch a disclosed architectural vulnerability (as Anthropic did for both Claude DXT and the STDIO STDIO-execution flaw), what is the correct posture for a verification standard? Should AISVS treat vendor-affirmed architectural risks as implementer-level requirements (e.g., mandate sandboxing at the host layer), accept them as residual risk disclosed in threat models, or recommend replacement of the affected component entirely?
- [ ] The Axios compromise affected transitive dependencies that most MCP server authors never explicitly declared. Traditional SBOM and provenance checks capture direct dependencies well, but transitive chains — particularly for autonomous agent workflows that run `npm install` without human review — remain a blind spot. Is pre-install provenance gating (SLSA Level 3+ verification at install time, not scan time) a realistic requirement for MCP fleets, or is the tooling not yet mature enough to enforce it without breaking developer workflows?
