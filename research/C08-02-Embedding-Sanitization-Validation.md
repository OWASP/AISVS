# C8.2: Embedding Sanitization & Validation

> [Back to C08 Index](C08-Memory-and-Embeddings.md)
> **Last Researched:** 2026-04-16

## Purpose

Pre-screen content before vectorization; treat memory writes as untrusted inputs; prevent ingestion of unsafe payloads. Embedding ingestion pipelines are a critical trust boundary — content that passes through them enters the system's long-term memory and influences all future retrieval. As of mid-April 2026, PII detection and encoding validation are well-supported by mature tooling (Microsoft Presidio 2.2.x, AWS Comprehend, Google Cloud DLP, and NVIDIA NeMo Guardrails' retrieval rails), but outlier detection and contradiction checking still require custom implementations tuned to the corpus. The ACM AISec "Hidden Threat in Plain Text" study (PhantomText toolkit, October 2025) remains the baseline for ingestion red-teaming with a 74.4% ASR across 357 scenarios and 100% ASR for font poisoning and homoglyphs against every loader tested — including LangChain, LlamaIndex, Docling, Haystack, and LLMSherpa. The landscape continued to worsen through Q1 2026: CorruptRAG demonstrated single-document poisoning at 98.2% ASR, the "Overcoming the Retrieval Barrier" paper (arXiv 2601.07072, January 2026) showed near-100% retrieval of malicious content across 11 benchmarks and 8 embedding models for ~$0.21 per target query, and Microsoft Defender's February 2026 disclosure catalogued 50 distinct "AI Recommendation Poisoning" campaigns from 31 companies across 14 industries — now formally tracked as MITRE ATLAS AML.T0080. Inference-time defenses like A-MemGuard (consensus validation + dual-memory, 95%+ ASR reduction), hybrid BM25+dense+cross-encoder reranking (Databricks reports 35% hallucination reduction), and OWASP Agent Memory Guard project guidance are the most promising complementary layers.

| **8.2.1** | **Verify that** regulated data and sensitive fields are detected prior to embedding and are masked, tokenized, transformed, or dropped based on policy. | 1 | PII/PHI/PCI data leakage via embeddings — once sensitive text is embedded, the original content can often be reconstructed via inversion attacks, and the vector store becomes a regulated data repository subject to GDPR/CCPA deletion requirements. Maps to [OWASP LLM02:2025 (Sensitive Information Disclosure)](https://genai.owasp.org/llmrisk/llm02-sensitive-information-disclosure/) and [LLM08:2025 (Vector and Embedding Weaknesses)](https://genai.owasp.org/llmrisk/llm082025-vector-and-embedding-weaknesses/), which formally recognizes that embedding inversion attacks can recover 50-70% of original input words if vectors are compromised. Vec2Text reports reconstruction of up to 92% of 32-token inputs from T5-based embeddings. The 2026 reproducibility study ["Rethinking the Privacy of Text Embeddings"](https://arxiv.org/html/2507.07700v1) confirmed full-name recovery from clinical notes is practical. [MITRE ATLAS](https://atlas.mitre.org/) tracks related activity under the "RAG Credential Harvesting" technique added in the October 2025 Zenity Labs collaboration. | Review the ingestion pipeline for DLP/PII scanning stages before the embedding step. Test with synthetic documents containing known PII patterns (SSNs, credit card numbers, email addresses, MRNs, API keys). Use [Microsoft Presidio](https://github.com/microsoft/presidio)'s Evaluator to measure precision/recall on a labeled test set. Verify sensitive fields are masked or dropped in both the stored text chunks and metadata. Confirm handling of structured attachments (DOCX tables, PDF form fields) where parsers commonly drop spans. For healthcare, cross-check against IHE's FHIR De-Identification Implementation Guide (draft February 2026). | [Microsoft Presidio](https://microsoft.github.io/presidio/) (open-source, 2.2.x stream), AWS Comprehend, Google DLP API, Azure AI Language, and Microsoft Fabric's `ai.extract`/`ai.generate_response` functions all cover ingestion-time scanning. NER-based detection still has recall gaps for domain-specific sensitive data (medical record numbers, internal project codes); custom recognizers are typically required. Masking can hurt retrieval quality for legitimate queries. New defense options to consider: [DPPN (Detecting and Perturbing Privacy-Sensitive Neurons)](https://openreview.net/forum?id=DF5TVzpTW0) reduces privacy leakage 5-78% while improving task performance 14-40%; [SPARSE (concept-aware privacy)](https://arxiv.org/html/2602.07090); and embedding quantization, which measurably degrades Vec2Text reconstruction while preserving top-k retrieval performance. [CrowdStrike Falcon AIDR + NVIDIA NeMo Guardrails](https://www.crowdstrike.com/en-us/blog/secure-homegrown-ai-agents-with-crowdstrike-falcon-aidr-and-nvidia-nemo-guardrails/) ships 75+ built-in PII/sensitive-data rules that can be wired into retrieval rails. |
| **8.2.2** | **Verify that** embedding ingestion rejects or quarantines inputs that violate required content constraints (e.g., non-UTF-8, malformed encodings, oversized payloads, invisible Unicode characters, or executable content intended to poison retrieval). | 1 | Indirect prompt injection via RAG — attackers inject adversarial instructions into documents that get embedded and later retrieved into the model's context window. Invisible Unicode (zero-width joiners, RTL overrides, BOMs) hides instructions from human reviewers while remaining effective against LLMs. The January 2026 paper ["Overcoming the Retrieval Barrier"](https://arxiv.org/abs/2601.07072) showed API-only adversaries can achieve near-100% retrieval across 11 benchmarks and 8 embedding models for as little as $0.21 per target query, and a single poisoned email coerced GPT-4o into exfiltrating SSH keys with 80%+ success in a multi-agent workflow. The ACM AISec ["Hidden Threat in Plain Text"](https://dl.acm.org/doi/full/10.1145/3733799.3762976) study (Castagnaro et al., October 2025) tested 19 stealthy techniques across five loaders (LangChain, LlamaIndex, Docling, Haystack, LLMSherpa) and found 74.4% ASR across 357 scenarios — font poisoning and homoglyphs hit 100% against every loader, zero-size font attacks 81% against LlamaIndex, out-of-bound text 83% against Docling. Attacks validated against black-box services including NotebookLM and OpenAI Assistants. [MITRE ATLAS AML.T0080](https://atlas.mitre.org/) now formally tracks this family. | Submit test documents containing: non-UTF-8 byte sequences, invisible Unicode (U+200B, U+200E, U+FEFF, U+202E), oversized payloads, documents with embedded scripts or encoded executables, font poisoning payloads (white-on-white, 0pt), homoglyph substitutions (Cyrillic/Greek lookalikes), and diacritical-overflow combining characters. Verify each is rejected or quarantined. Inspect quarantine storage for review capability. Scan for injection patterns ("ignore previous directives," "you must respond with") using regex and LLM classifiers. Test across every loader in the ingestion pipeline — the AISec study showed loader-specific vulnerabilities. Re-run PhantomText-style red-team scripts whenever a loader is upgraded. Use [NeMo Guardrails](https://github.com/NVIDIA-NeMo/Guardrails) retrieval rails or [Lakera Guard](https://www.lakera.ai/blog/indirect-prompt-injection) (sub-30ms) as an inline classifier and verify block/allow decisions are logged. | Encoding and size checks are straightforward; semantic attacks are harder. Invisible Unicode requires explicit character-class filtering (Python's `unicodedata.category` for Cf/Mn/Cc or libraries like `anyascii`). Font poisoning and homoglyph attacks target the document parsing layer before embeddings exist, so format-aware sanitization (DOCX, HTML, PDF) must live alongside text normalization. Commercial classifiers — Rebuff, Lakera Guard, Prompt Security, CrowdStrike Falcon AIDR — catch many semantic injections, but Sunil et al. (2025) note "no single signal is sufficient": composite trust scoring across multiple signals is needed. The AISec PhantomText toolkit is the practical red-team checklist for ingestion pipelines. [Snyk Labs' RAGPoison PoC](https://labs.snyk.io/resources/ragpoison-prompt-injection/) and [Prompt Security's RAG_Poisoning_POC](https://github.com/prompt-security/RAG_Poisoning_POC) are useful exploit repros for CI regression tests. CVE-2024-31621 (Flowise) — 438 servers leaking Pinecone/OpenAI credentials — is a reminder that the vector store itself must be secured alongside ingestion controls. |
| **8.2.3** | **Verify that** vectors that fall outside normal clustering patterns are flagged and quarantined before entering production indices. | 2 | Adversarial embedding injection — an attacker crafts vectors positioned in unusual regions of embedding space to manipulate retrieval behavior. Out-of-distribution vectors may be designed to be retrieved for a wide range of queries (high-centrality poisoning) or to target specific query clusters. [PoisonedRAG](https://www.usenix.org/system/files/usenixsecurity25-zou-poisonedrag.pdf) (USENIX Security 2025) demonstrated that carefully positioned vectors dominate retrieval for target queries. MINJA (Dong et al., NeurIPS 2025) showed that bridging steps create vectors that look individually reasonable while advancing attacker objectives, making pure outlier detection insufficient. [CorruptRAG](https://arxiv.org/abs/2504.03957) (updated January 2026) achieves 98.2% ASR with a single optimized document, meaning detectors must flag individual vectors, not just clusters. [Vector Collision Attacks](https://instatunnel.my/blog/vector-collision-attacks-hijacking-the-nearest-neighbor) (InstaTunnel, 2026) exploit nearest-neighbor geometry directly rather than text content. | Review the outlier detection mechanism in the ingestion pipeline. Verify it uses statistical methods (z-score on cosine distance to cluster centroids, isolation forest, DBSCAN, autoencoder reconstruction error, or [cleanlab's out-of-distribution scoring](https://cleanlab.ai/blog/learn/outlier-detection/)). [Qdrant](https://qdrant.tech/), Milvus/Zilliz, Weaviate, and SingleStoreDB all support similarity-based anomaly queries that can back such a pipeline. Inject synthetic outlier vectors and confirm they are quarantined. Confirm quarantine does not block legitimate novel content without review. Adopt write-ahead validation with a secondary model that evaluates proposed entries before commitment ([Schneider 2025, Layer 2 defense](https://christian-schneider.net/blog/persistent-memory-poisoning-in-ai-agents/)). Validate detection catches collision attacks by injecting documents crafted to match a specific target query vector. | Defining "normal clustering patterns" requires a baseline — cold-start or rapidly evolving knowledge bases make this hard. False positive rates in high-dimensional space can be high. Dimensionality reduction (UMAP/t-SNE) before outlier detection is common but lossy. Adaptive baselines that update from verified-clean batches are the current best practice. MINJA's progressive shortening strategy means poisoned entries increasingly resemble legitimate content over iterative refinement, challenging static thresholds. CorruptRAG's single-vector success rate means cluster-level detection alone is inadequate. Emerging complementary defenses: perplexity filtering (flagging statistically unnatural text), [cross-encoder re-ranking](https://zeroentropy.dev/articles/ultimate-guide-to-choosing-the-best-reranking-model-in-2025/) (Databricks testing shows 35% hallucination reduction vs raw similarity; 50-document rerank is the 2026 sweet spot for chat, 100-200 for search), hybrid BM25+dense retrieval (Pinecone, Vespa, Weaviate native support), [RevPRAG](https://aclanthology.org/2025.findings-emnlp.698.pdf) (Findings of EMNLP 2025), and [Traceback-of-Poisoning](https://dl.acm.org/doi/abs/10.1145/3696410.3714756) (WebConf 2025) for post-compromise forensics. |
| **8.2.4** | **Verify that** an agent's own outputs are not automatically written back into its trusted memory without explicit validation (such as content-origin checks or write-authorization controls that verify the content's source before committing writes). | 2 | Memory poisoning feedback loops — if an agent can write its own (potentially hallucinated or manipulated) outputs back into its memory, an attacker can use a single successful prompt injection to permanently corrupt the agent's knowledge base, creating a persistent backdoor that survives conversation resets. [OWASP Top 10 for Agentic Applications 2026](https://www.paloaltonetworks.com/blog/cloud-security/owasp-agentic-ai-security/) lists Memory and Context Poisoning as ASI06. The January 2026 "ZombieAgent" PoC (Radware) showed ChatGPT's connector + memory features can chain indirect prompt injection into persistent, cross-session compromise spreading through email attachments. Unit 42 (October 2025) showed Amazon Bedrock Agents' session summarization could be hijacked to inject persistent instructions via forged XML tags. [Microsoft Defender's February 10, 2026 "AI Recommendation Poisoning" disclosure](https://www.microsoft.com/en-us/security/blog/2026/02/10/ai-recommendation-poisoning/) documented 50 campaigns from 31 companies across 14 industries using hidden URL prompt parameters in "Summarize with AI" buttons to inject persistent preferences — formally catalogued as MITRE ATLAS AML.T0080. | Review agent architecture for every write-back path from agent output to memory stores. Verify content-origin tagging distinguishes user-provided content, retrieved content, and agent-generated content (e.g., `source=user\|retrieved\|agent\|tool`). Test that agent-generated content requires explicit approval or a separate validation pipeline before memory write. Test for delayed tool invocation attacks (Rehberger, January 2025) where conditional instructions embedded in chat context trigger memory writes on natural affirmations. Exercise the [OWASP Agent Memory Guard project's](https://owasp.org/www-project-agent-memory-guard/) reference tests against your stack. For LangGraph/CrewAI deployments, confirm a memory firewall exists between agent output and persistent store ([Roborhythms pattern, 2026](https://www.roborhythms.com/how-to-prevent-ai-agent-memory-poisoning/)). | This is critical for autonomous agents (AutoGPT-style) with self-modification capabilities. LangChain's `ConversationBufferMemory` and `ConversationSummaryMemory` write back by default without origin checks; [Mem0.ai's 2026 best-practices guide](https://mem0.ai/blog/ai-memory-security-best-practices) explicitly recommends write-ahead validation. The distinction between "agent output" and "validated knowledge" must be architecturally enforced. MINJA (NeurIPS 2025) demonstrated >95% injection success through query-only interaction without direct memory access — even read-only interfaces can lead to poisoning. [A-MemGuard](https://arxiv.org/abs/2510.02373) (NTU + Oxford + Max Planck + Ohio State) cuts ASR by 95%+ (from 100% to ~2% on EHRAgent) via consensus-based validation across related memories plus a dual-memory structure where detected failures become "lessons" consulted before future actions. [Microsoft Copilot Studio's March 30, 2026 guidance](https://www.microsoft.com/en-us/security/blog/2026/03/30/addressing-the-owasp-top-10-risks-in-agentic-ai-with-microsoft-copilot-studio/) maps Dataverse memory isolation and tenant-scoped connectors to ASI06. |
| **8.2.5** | **Verify that** new content written to memory is checked for contradictions with what is already stored and that conflicts trigger alerts. | 3 | Knowledge base corruption and slow-burn poisoning — an attacker gradually introduces contradictory information to degrade the reliability of RAG responses. Without contradiction detection, conflicting facts coexist silently and the model may retrieve either version unpredictably, undermining trust in the system. [Microsoft's February 10, 2026 "AI Recommendation Poisoning" report](https://www.microsoft.com/en-us/security/blog/2026/02/10/ai-recommendation-poisoning/) documented 50 distinct examples deployed by 31 real companies across 14 industries in a 60-day review, confirming slow-burn poisoning is actively exploited in the wild. Context manipulation (Zealynx 2025) gradually establishes false assumptions that influence AI behavior over time — distinct from direct prompt injection. Multiple 2026 analyses report that 68% of enterprises adopting AI have experienced at least one data or context leakage incident, and industry red-team write-ups note that as few as five crafted documents can sway AI responses ~90% of the time. | Review contradiction detection implementation (NLI-based entailment, semantic similarity with polarity analysis, GraphRAG-style triple consistency checks). Test by ingesting documents that directly contradict existing stored content. Verify alerts fire and conflicting content is flagged for human review rather than silently merged. Confirm ingestion pipelines enforce provenance metadata (source, author, timestamp, approval status) and that tags on documents that explicitly state something outrank LLM-inferred tags (validated tags ~99.2% accuracy vs 78% for unvalidated). Confirm versioned corpora and rollback are available — "secure RAG must be governable and recoverable after compromise" ([Securing RAG taxonomy, 2026](https://arxiv.org/html/2604.08304)). | Automated contradiction detection at scale is an unsolved NLP problem. Current approaches using NLI models (e.g., DeBERTa fine-tuned on MNLI, RoBERTa-ANLI) catch explicit contradictions but miss subtle or contextual conflicts. Cross-document consistency is O(n^2) in the worst case; practical implementations compare new content against a curated "golden" knowledge set. Memory poisoning is temporally decoupled — poison planted today may execute weeks later when semantically triggered, making real-time contradiction detection the only viable prevention point. [A-MemGuard's](https://arxiv.org/html/2510.02373v1) consensus-based validation (comparing reasoning paths from multiple related memories) is the strongest published approach to date; even so, A-MemGuard authors report that advanced LLM-based memory classifiers miss ~66% of poisoned entries without the full consensus framework. Complementary tooling: RevPRAG (EMNLP 2025 Findings) for poisoning detection, Traceback-of-Poisoning (WebConf 2025) for post-incident forensics, and the [OWASP Agent Memory Guard project](https://owasp.org/www-project-agent-memory-guard/) for reference test suites. |
