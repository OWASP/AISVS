# C3.4: Secure Development Practices

> [Back to C03 Index](C03-Model-Lifecycle-Management.md)
> **Last Researched:** 2026-06-20

## Purpose

Model development and training processes must follow secure practices to prevent compromise. This section covers AI-specific runtime component isolation across environment boundaries, version-controlled configuration artifacts with review gates, and training environment separation from production services.

The attack surface during model development is significant and growing. The Clinejection incident (February 2026) demonstrated a new class of agent-mediated lateral movement: a poisoned GitHub issue title was processed by a Claude AI coding agent, which executed a malicious bash command that poisoned GitHub Actions caches, leading to npm publish token theft and an 8-hour supply chain compromise distributing unauthorized packages. Christian Schneider's research on [AI agent lateral movement](https://christian-schneider.net/blog/ai-agent-lateral-movement-attack-pivots/) (2026) identifies this as a distinct third class of pivot — agents bridge isolated systems via delegated authority and tool access without new network paths or stolen credentials, and traditional SIEM rules miss the activity because every step uses legitimate permissions. The OpenClaw crisis (February–March 2026) saw 138 CVEs disclosed in 63 days across an AI agent platform with 135,000+ exposed instances, with Antiy CERT confirming 1,184 malicious skills across ClawHub — underscoring the cascading risks when development tooling lacks integrity controls. The [Flowise CVE-2025-59528](https://github.com/FlowiseAI/Flowise/security/advisories/GHSA-3gcm-f6qx-ff7p) max-severity (CVSS 10.0) code-injection vulnerability in the CustomMCP node entered active exploitation on April 7, 2026 with 12,000+ instances still exposed to the internet and exploitation traffic originating from a single Starlink IP; the flaw executes user-supplied JavaScript with full Node.js privileges (including `child_process` and `fs`), and organizations running unpatched Flowise below 3.0.6 face opportunistic mass RCE. Palo Alto Unit 42 ([Double Agents in Vertex AI](https://unit42.paloaltonetworks.com/double-agents-vertex-ai/), March 2026) showed that Google's Vertex AI Agent Engine leaks the Per-Project, Per-Product Service Agent (P4SA) credentials through the metadata service on every agent invocation — with excessive default scoping, these credentials provided unrestricted read access to all GCS buckets in the customer project and let researchers pull Google-owned Artifact Registry images for the Vertex Reasoning Engine. Microsoft disclosed [CVE-2026-32211](https://cvefeed.io/vuln/detail/CVE-2026-32211) (CVSS 9.1) on April 3, 2026, an unauthenticated information-disclosure flaw in the Azure MCP Server (`@azure-devops/mcp` on npm) that exposes configuration, API keys, and auth tokens without credentials — the package was also published from a personal npm account with no provenance attestations, layering a supply chain concern on top of the auth gap. The LiteLLM TeamPCP supply chain attack (March 24, 2026) backdoored versions 1.82.7 and 1.82.8 of a package drawing ~3.4M daily downloads: attackers first compromised Trivy (pulled unpinned from apt in LiteLLM's own CI), stole the `PYPI_PUBLISH` token, and shipped a malicious `litellm_init.pth` file that auto-executes on every Python process startup, harvesting SSH keys, cloud tokens, Kubernetes secrets, and `.env` files, deploying privileged pods across cluster nodes, and installing a systemd persistent backdoor ([Datadog Security Labs analysis](https://securitylabs.datadoghq.com/articles/litellm-compromised-pypi-teampcp-supply-chain-campaign/), [Trend Micro](https://www.trendmicro.com/en_us/research/26/c/inside-litellm-supply-chain-compromise.html)). Palo Alto Unit 42 (December 2025) disclosed RCE vulnerabilities in three ML frameworks — NVIDIA NeMo ([CVE-2025-23304](https://nvd.nist.gov/vuln/detail/CVE-2025-23304)), Salesforce Uni2TS, and Apple FlexTok — all stemming from unsafe Hydra `instantiate()` calls in model config files. The MCP attack surface expanded dramatically: Trend Micro found 492 MCP servers exposed to the internet with zero authentication, the AI Accelerator Institute's review of 281 MCP servers flagged 92% as high security risk, a prompt-injection attack against the GitHub MCP server exfiltrated private repository contents and salary information via a public pull request, and researchers demonstrated unauthenticated RCE through Anthropic's MCP Inspector tool. Check Point's disclosure of [CVE-2025-59536](https://research.checkpoint.com/2026/rce-and-api-token-exfiltration-through-claude-code-project-files-cve-2025-59536/) (CVSS 8.7) and [CVE-2026-21852](https://www.mintmcp.com/blog/claude-code-cve) in Anthropic's Claude Code illustrates what happens when repository-controlled configuration bypasses approval gates: malicious `.claude/settings.json` hooks execute before any trust dialog renders, and the `enableAllProjectMcpServers` / `enabledMcpjsonServers` overrides auto-approve all MCP servers on launch. MITRE ATLAS v5.1.0 (November 2025) now tracks 84 techniques including agent-specific lateral movement patterns. The [OWASP Top 10 for Agentic Applications 2026](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/) (peer-reviewed with input from 100+ contributors) centers on goal misalignment, tool misuse, delegated trust, inter-agent communication, persistent memory, and emergent autonomous behavior — with "Agent Goal Hijack" ranked as the top risk and direct relevance to every control in this section. The AWS Agentic AI Security Scoping Matrix (2026) provides a framework mapping security controls across six dimensions — identity context, data/memory protection, audit logging, agent controls, agency perimeters, and orchestration — with escalating isolation requirements at each autonomy level. NIST's AI Agent Standards Initiative (February 2026) is developing SP 800-53 control overlays (COSAiS project) for five AI use cases, with two directly addressing agentic deployments, and on April 7, 2026 NIST released a concept note for an AI RMF Profile on Trustworthy AI in Critical Infrastructure. The CoSAI [MCP Security Guide](https://www.coalitionforsecureai.org/securing-the-ai-agent-revolution-a-practical-guide-to-mcp-security/) recommends mandatory code signing for all MCP servers, container sandboxing with gVisor or Kata Containers, and Trusted Execution Environments with remote attestation for high-security deployments. These controls establish the baseline security posture for the environments and processes where models are built.

## May 2026 Research Notes

Recent disclosures reinforce that C3.4 is not only a conventional environment-separation control.
It now covers agent runtime configuration, MCP stdio launch paths, prompt and policy promotion, and managed-service identities.

- **Agent-framework RCE epidemic (May 2026):** Microsoft's [When prompts become shells](https://www.microsoft.com/en-us/security/blog/2026/05/07/prompts-become-shells-rce-vulnerabilities-ai-agent-frameworks/) post (May 7, 2026) consolidates a pattern that previously read as isolated incidents.
  [CVE-2026-25592](https://nvd.nist.gov/vuln/detail/CVE-2026-25592) (CVSS 10.0) in the Semantic Kernel .NET SDK exposed `DownloadFileAsync` and `UploadFileAsync` to the LLM through an accidental `[KernelFunction]` annotation with no path validation, letting a single prompt-injected agent write outside its Azure Container Apps Python sandbox; companion [CVE-2026-26030](https://nvd.nist.gov/vuln/detail/CVE-2026-26030) (CVSS 9.9) is a code-injection flaw in the Python SDK's `InMemoryVectorStore` that bypasses a blocklist validator with `__class__.__bases__[0].__subclasses__()` to reach `os.system` (patched in `Microsoft.SemanticKernel` ≥1.71.0 and `semantic-kernel` Python ≥1.39.4).
  The CrewAI advisory [VU#221883](https://kb.cert.org/vuls/id/221883) (April 2, 2026) bundles four CVEs — [CVE-2026-2275](https://nvd.nist.gov/vuln/detail/CVE-2026-2275), [CVE-2026-2285](https://nvd.nist.gov/vuln/detail/CVE-2026-2285), [CVE-2026-2286](https://nvd.nist.gov/vuln/detail/CVE-2026-2286), [CVE-2026-2287](https://nvd.nist.gov/vuln/detail/CVE-2026-2287) — that chain through the Code Interpreter Tool: when Docker is unreachable, CrewAI silently falls back to an in-process `SandboxPython` environment, and from there prompt injection escalates to host RCE, JSON-loader file read, and RAG-tool SSRF.
  [CVE-2026-34070](https://nvd.nist.gov/vuln/detail/CVE-2026-34070) (CVSS 7.5) is a path traversal in LangChain's prompt-loading API that returns arbitrary host files (Docker secrets included), and [CVE-2025-67644](https://nvd.nist.gov/vuln/detail/CVE-2025-67644) (CVSS 7.3) is SQL injection in the LangGraph SQLite checkpoint via unvalidated metadata filter keys.
  These are not adversarial-ML novelties; they are the classic appsec catalog reappearing inside agent runtimes, which means the same secure-development controls — input validation review, sandbox integrity tests, environment isolation — must extend over the agent layer.
- **Production prompt-injection-to-data-loss chains:** [DarkReading's Microsoft/Salesforce write-up](https://www.darkreading.com/cloud-security/microsoft-salesforce-patch-ai-agent-data-leak-flaws) covers two now-patched chains.
  Microsoft "ShareLeak" ([CVE-2026-21520](https://nvd.nist.gov/vuln/detail/CVE-2026-21520), CVSS 7.5) used a SharePoint form input to drive Copilot into emailing customer data to attacker-controlled addresses; Salesforce "PipeLeak" exploited Agent Flows that processed unauthenticated lead-form fields as trusted instructions.
  Both are configuration-artifact failures (the configuration-governance concern that backs C3.4.1 runtime isolation) — the failing surface is a routing graph or agent skill definition, not the model itself.
- **MCP "by design" disclosure (April–May 2026):** OX Security's [MCP supply-chain advisory](https://www.ox.security/blog/mcp-supply-chain-advisory-rce-vulnerabilities-across-the-ai-ecosystem/) and follow-up [systemic-vulnerability analysis](https://www.ox.security/blog/the-mother-of-all-ai-supply-chains-critical-systemic-vulnerability-at-the-core-of-the-mcp/) frame stdio MCP as an architectural rather than implementation flaw, with [Hacker News reporting](https://thehackernews.com/2026/04/anthropic-mcp-design-vulnerability.html) the impact across 7,000+ exposed servers and 150M+ SDK downloads.
  The named CVEs include [CVE-2025-49596](https://nvd.nist.gov/vuln/detail/CVE-2025-49596) (MCP Inspector), [CVE-2026-22252](https://nvd.nist.gov/vuln/detail/CVE-2026-22252) (LibreChat), [CVE-2026-22688](https://nvd.nist.gov/vuln/detail/CVE-2026-22688) (WeKnora), [CVE-2025-54994](https://nvd.nist.gov/vuln/detail/CVE-2025-54994) (`@akoskm/create-mcp-server-stdio`), and [CVE-2025-54136](https://nvd.nist.gov/vuln/detail/CVE-2025-54136) (Cursor).
  CSA's [companion research note](https://labs.cloudsecurityalliance.org/research/csa-research-note-mcp-by-design-rce-ox-security-20260420-csa/) and [Authzed's MCP breach timeline](https://authzed.com/blog/timeline-mcp-breaches) reinforce that the practical mitigation is the C3.4 one: do not share stdio MCP hosts across environments, treat MCP entries as code under version control, and inventory every command/args/env tuple.
- **Azure MCP SSRF ([CVE-2026-26118](https://nvd.nist.gov/vuln/detail/CVE-2026-26118)):** Microsoft's March 10, 2026 patches addressed an SSRF in Azure MCP Server Tools where authenticated users could redirect server-side requests to Azure instance metadata; the captured managed-identity token then granted lateral access to the service account's resources.
  Combined with the earlier [CVE-2026-32211](https://cvefeed.io/vuln/detail/CVE-2026-32211) unauthenticated information disclosure in `@azure-devops/mcp`, the pattern is the same: shared managed MCP endpoints concentrate identity and become high-value targets when they span environment boundaries.
- **npm "Shai-Hulud" worm waves:** Palo Alto Unit 42 has updated its [npm threat landscape monitor](https://unit42.paloaltonetworks.com/monitoring-npm-supply-chain-attacks/) through May 1, 2026, tracking two major campaigns: "Shai-Hulud: The Third Coming" (April 22, 2026) and "Mini Shai-Hulud" (April 29, 2026) targeting the SAP developer ecosystem.
  The April 22 wave included a fake `@bitwarden/cli` 2026.4.0 package whose post-install payload harvested cloud credentials, CI tokens, and developer workstation secrets.
  Both campaigns now prioritize npm token and GitHub Personal Access Token theft to enable automated republishing of legitimate packages — the same self-propagating-via-credentials pattern as TeamPCP, applied to JavaScript training tooling, agent SDKs, and front-end RAG stacks.
- **Flowise [CVE-2026-40933](https://github.com/advisories/GHSA-c9gw-hvqq-f33r) (CVSS 10.0):** GitHub's advisory documents an authenticated stdio MCP injection in Flowise ≤3.0.13 — the input sanitization in the Custom MCP node had a `validateCommandInjection` check and an allowlist of "safe" commands, but commands like `npx` combined with `-c <payload>` passed the allowlist and reached the shell.
  Fixed in 3.1.0 (paired with the [CVE-2025-59528](https://github.com/FlowiseAI/Flowise/security/advisories/GHSA-3gcm-f6qx-ff7p) CustomMCP JavaScript-execution flaw fixed in 3.0.6).
  Treat any shared Flowise instance as a cross-environment hot zone until both fixes are deployed and the management UI is off the public internet.
- **Zero-interaction MCP RCE chain (April–May 2026):** [CVE-2026-30615](https://nvd.nist.gov/vuln/detail/CVE-2026-30615) (CVSS 8.0, OX Security disclosure April 15, 2026) is a Windsurf IDE 1.9544.26 flaw where rendered HTML — from a web page, repository README, or remote tool description — silently rewrites the local `mcp.json` to register an attacker-controlled stdio MCP server, and the MCP SDK then launches the binary the next time the project loads.
  OX Security's [policy-layer write-up](https://policylayer.com/mcp-incidents/windsurf-zero-click-mcp-rce-cve-2026-30615) emphasizes that Windsurf was the only IDE where exploitation required zero approval clicks; Cursor, Claude Code, and Gemini-CLI still gated the same trick behind at least one user prompt.
  In parallel, [CVE-2026-23744](https://nvd.nist.gov/vuln/detail/CVE-2026-23744) (CVSS 9.8) in [MCPJam Inspector](https://github.com/advisories/GHSA-232v-j27c-5pp6) ≤1.4.2 binds to `0.0.0.0` by default and exposes `/api/mcp/connect` without auth — a single POST with `command`/`args` runs arbitrary shell on the host, fixed in 1.4.3.
  Together these reinforce that the practical C3.4.1 runtime-isolation and configuration-governance mitigations are inseparable: MCP host registration must be treated as a code-promotion event (no auto-registration from rendered content), and MCP-related scanning, inspector, and orchestration UIs must never be exposed on shared interfaces across environments.
- **Five Eyes "Careful Adoption of Agentic AI Services" (May 1, 2026):** [CISA's joint guidance](https://www.cisa.gov/resources-tools/resources/careful-adoption-agentic-ai-services) with NSA, the Australian Signals Directorate, the Canadian Centre for Cyber Security, NZ NCSC, and UK NCSC ([CyberScoop coverage](https://cyberscoop.com/cisa-nsa-five-eyes-guidance-secure-deployment-ai-agents/), [Crowell client alert](https://www.crowell.com/en/insights/client-alerts/american-and-allied-cyber-agencies-issue-first-joint-guidance-on-securing-agentic-ai)) frames four agentic risks — expanded attack surface, privilege compromise and scope creep, behavior misalignment, and obscure event records — and lands directly on the C3.4 controls: "Train agents in simulated, controlled environments to evaluate consequences of actions safely before real-world deployment" (3.4.2 training/fine-tuning isolation), "Deploy AI agents in phase, restricting APIs and confining agents to 'sandboxes'" and "segment high-risk agents into specific domains" (3.4.1 runtime isolation), and "maintain a trusted registry of authorized agents and regularly reconcile it with active systems" alongside least-privilege scoping (supports both controls).
  This is the first international cyber-agency consensus on the controls this section already requires, so it is now a defensible compliance citation alongside NIST SP 800-218A and EU AI Act Article 15.

- **3.4.1 runtime isolation:** GitHub's advisory for Flowise [CVE-2026-40933](https://github.com/advisories/GHSA-c9gw-hvqq-f33r) (published April 15, 2026) documents authenticated command execution in MCP adapters for Flowise <=3.0.13, fixed in 3.1.0.
  OX Security's [MCP STDIO advisory](https://www.ox.security/blog/mcp-supply-chain-advisory-rce-vulnerabilities-across-the-ai-ecosystem) and related [architectural analysis](https://www.ox.security/blog/the-mother-of-all-ai-supply-chains-critical-systemic-vulnerability-at-the-core-of-the-mcp) report command execution through user-controlled stdio MCP configuration across multiple agent platforms.
  Treat every stdio MCP server definition as executable code, and do not share MCP hosts, Flowise instances, model registries, prompt stores, or orchestration runtimes across dev, staging, and production.
- **Configuration-artifact governance (supports 3.4.1 runtime isolation):** Store prompt labels, MCP manifests, tool schemas, routing graphs, and agent skill definitions in reviewed repositories — keeping these under version control is how shared runtime components stay free of unreviewed, poisoned definitions.
  Langfuse [prompt version control](https://langfuse.com/docs/prompt-management/features/prompt-version-control) supports version IDs, environment labels, rollbacks, prompt diffs, and protected `production` labels; use equivalent controls even when prompts are stored outside source code.
  MLflow's [Prompt Registry](https://mlflow.org/docs/latest/genai/prompt-registry/) adds immutable prompt versions, commit-style change history, aliases such as `production`, side-by-side diffs, and model-configuration metadata, which makes it useful as an auditable control point when prompts are promoted outside normal code deployment.
  Snyk [Agent Scan](https://github.com/snyk/agent-scan) inventories and scans Claude, Cursor, Windsurf, Gemini CLI, Amp, Amazon Q, MCP servers, and skills for prompt injection, tool poisoning, tool shadowing, toxic flows, malware payloads, and credential-handling issues.
  Invariant [MCP-Scan](https://invariantlabs-ai.github.io/docs/mcp-scan/) adds local MCP configuration discovery, tool-description prompt-injection checks, tool poisoning and cross-origin escalation detection, runtime proxying, tool pinning, PII/secrets guardrails, and JSON output for CI review gates.
  Because scanner execution can start stdio MCP commands, run third-party configuration scans in a disposable container or VM and require explicit review of the command, arguments, network reachability, and data shared with scanner services before execution.
- **3.4.2 training isolation:** Datadog's [TeamPCP analysis](https://securitylabs.datadoghq.com/articles/litellm-compromised-pypi-teampcp-supply-chain-campaign/) is the current reference incident for training and build environments.
  Compromised `litellm` 1.82.7 and 1.82.8 releases harvested cloud, SSH, Kubernetes, CI/CD, and `.env` credentials and installed persistence; affected hosts, containers, CI jobs, and developer workstations should be treated as full credential-exposure events.
  Datadog's response guidance also points to package-install telemetry, Kubernetes audit searches, and filesystem indicators such as `litellm_init.pth`, `sysmon.service`, and `node-setup-*` pods; that maps directly to training isolation: training jobs should use locked dependency mirrors, hash-pinned packages, isolated service accounts, no production RAG endpoints, and egress policies that block direct package installation from public registries unless mediated by quarantine.
- **Assurance and compliance:** NIST [SP 800-218A](https://csrc.nist.gov/pubs/sp/800/218/a/final) remains the clearest secure-development profile for generative and dual-use foundation model work.
  NIST's April 2026 [AI RMF Critical Infrastructure Profile concept note](https://www.nist.gov/programs-projects/concept-note-ai-rmf-profile-trustworthy-ai-critical-infrastructure) explicitly calls out AI agents and tools in critical infrastructure risk management.
  The European Commission's May 7, 2026 [Digital Omnibus political agreement](https://digital-strategy.ec.europa.eu/en/news/eu-agrees-simplify-ai-rules-boost-innovation-and-ban-nudification-apps-protect-citizens) now sets high-risk AI system obligations for Annex III use cases to apply from December 2, 2027, and product-embedded high-risk systems from August 2, 2028, with the Commission's [standardisation page](https://digital-strategy.ec.europa.eu/en/policies/ai-act-standardisation) tying those dates to the availability of harmonised standards and support tools.
  [Article 15](https://ai-act-service-desk.ec.europa.eu/en/ai-act/article-15) remains the cybersecurity benchmark for high-risk AI systems, including controls for data poisoning, model poisoning, adversarial examples, confidentiality attacks, and model flaws.

Practical audit checks to add in this cycle:

- Treat every stdio MCP entry as code execution: collect `command`, `args`, working directory, environment variables, network reachability, and owning service account before allowing promotion.
- Require protected `production` labels or aliases for prompts, policies, and routing graphs, and verify that only approved maintainers can move those labels.
- Diff raw MCP schemas and tool descriptions, not only rendered UI text, because hidden instructions and schema-poisoning changes may be invisible to reviewers.
- Run Snyk Agent Scan, Invariant MCP-Scan, or equivalent tooling from a disposable container when evaluating third-party agent configs, and record whether scanner startup executed any stdio server.
- For training and fine-tuning jobs, block direct package downloads from public registries unless they pass quarantine, signature or hash validation, and newly published package delay rules.
- Hunt for TeamPCP-style persistence in build and training environments after dependency incidents, including `.pth` startup files, unexpected user systemd units, Kubernetes privileged pod creation, and outbound credential exfiltration.
- Audit agent-framework versions in every environment: `Microsoft.SemanticKernel` ≥1.71.0 / `semantic-kernel` Python ≥1.39.4 (covers [CVE-2026-25592](https://nvd.nist.gov/vuln/detail/CVE-2026-25592) and [CVE-2026-26030](https://nvd.nist.gov/vuln/detail/CVE-2026-26030)), CrewAI with Docker enforcement and the [VU#221883](https://kb.cert.org/vuls/id/221883) patches, LangChain past [CVE-2026-34070](https://nvd.nist.gov/vuln/detail/CVE-2026-34070), LangGraph past [CVE-2025-67644](https://nvd.nist.gov/vuln/detail/CVE-2025-67644), Flowise ≥3.1.0, and Azure MCP Server with the [CVE-2026-26118](https://nvd.nist.gov/vuln/detail/CVE-2026-26118) March 10, 2026 patch.
- Validate that lead-form, ticket, document, and other untrusted text fields are never wired directly into Copilot/Agentforce-style routing graphs as trusted prompts — Microsoft "ShareLeak" and Salesforce "PipeLeak" both turned standard form inputs into data-exfiltration commands.

## June 2026 Research Notes

The four weeks since the last refresh produced the first NSA-authored MCP security guidance, a clear pattern of unauthenticated MCP database back-ends, and hard numbers on how much of the MCP ecosystem is exposed.

- **NSA Cybersecurity Information Sheet on MCP (May 20, 2026):** The NSA's Artificial Intelligence Security Center released [Model Context Protocol (MCP): Security Design Considerations for AI-Driven Automation](https://www.nsa.gov/Portals/75/documents/Cybersecurity/CSI_MCP_SECURITY.pdf) (U/OO/6030316-26, v1.0; [Intelligence Community News coverage](https://intelligencecommunitynews.com/nsa-releases-security-design-considerations-for-ai-driven-automation/), [ExecutiveGov summary](https://www.executivegov.com/articles/nsa-model-context-protocol-deployment)).
  It is the first agency-level design document dedicated to MCP, and it lands squarely on C3.4 territory: the named failure modes include poor approval workflows, inadequate token lifecycle management, inconsistent agent behavior, and arbitrary code execution where user-provided logic runs inside MCP environments.
  Its nine recommendations include filtering outgoing proxies and DLP for external MCP connections with pinned resource URLs, sandboxing and resource constraints on tool execution, MCP message signing with expiration timestamps and replay protection, tool inventory auditing to detect name collisions and drift, audit logs feeding SIEM, and local MCP server discovery scanning to find unsafe listeners before deployment.
  The CSI explicitly frames these as environment-level controls that "cannot be patched at isolated endpoints" — the same position C3.4.1 takes on shared runtime components.
- **MCP database back-ends: "one is a fluke, 3 is a pattern" (May 13, 2026):** Akamai disclosed three back-end MCP flaws ([Akamai research post](https://www.akamai.com/blog/security-research/one-fluke-3-pattern-mcp-back-end-vulnerabilities), [Register coverage](https://www.theregister.com/security/2026/05/13/bug-hunter-tracks-down-three-serious-mcp-database-flaws-one-left-unpatched/5238916)): SQL injection in the Apache Doris MCP server ([CVE-2025-66335](https://www.sentinelone.com/vulnerability-database/cve-2025-66335/), patched December 2025), an authentication bypass in Apache Pinot's MCP that uses plain HTTP transport with no authentication and can lead to full database takeover (StarTree added an OAuth option, but the underlying SQL injection remains), and an unauthenticated information-disclosure flaw in Alibaba's RDS MCP where any client that can reach the endpoint can invoke the RAG tool with no query validation — which Alibaba has declined to patch.
  Unpatched, unauthenticated MCP back-ends are exactly the components C3.4.1 forbids sharing across environments: a dev-reachable RDS MCP that fronts production data is a standing exfiltration path.
- **Ecosystem exposure numbers (May–June 2026):** [Adversa AI's June 2026 MCP security roundup](https://adversa.ai/blog/top-mcp-security-resources-june-2026/) aggregates the measurement work: the VIPER-MCP static-plus-dynamic analysis framework swept ~40,000 MCP server repositories and surfaced 106 zero-day vulnerabilities yielding 67 CVEs; Censys counted 12,520 internet-accessible MCP services, predominantly unauthenticated; Trend Micro found 1,467 exposed servers carrying CVSS 9.8 command-injection flaws; and academic measurement puts roughly 40% of remote MCP servers at zero authentication.
  These figures supersede the earlier "492 exposed servers" datapoint and make discovery scanning (the NSA CSI's ninth recommendation) a baseline audit step rather than a nice-to-have.
- **PraisonAI auth-off-by-default ([CVE-2026-44338](https://github.com/advisories/GHSA-6rmh-7xcm-cpxj)):** The PraisonAI agent framework's legacy Flask API server shipped with `AUTH_ENABLED = False` and no token, letting unauthenticated attackers enumerate agents via `GET /agents` and trigger workflows via `POST /chat` — [exploitation began within hours of the May 2026 disclosure](https://cybersecuritynews.com/praisonai-vulnerability-exploited/).
  Fixed in 4.6.34; the maintainers recommend migrating to the newer serve command, which binds to `127.0.0.1` and requires authentication by default.
  Insecure-by-default orchestration endpoints are a C3.4.1 audit item: any agent API reachable across an environment boundary should be treated as compromised-by-design until authentication is verified.
- **npm dependency-confusion wave (May 28–29, 2026):** Microsoft documented [45 malicious npm packages across three coordinated personas](https://www.microsoft.com/en-us/security/blog/2026/05/29/33-malicious-npm-packages-abuse-dependency-confusion-profile-developer-environments/) that registered scoped packages matching internal corporate namespaces (including data-science/ML platform scopes and Sberbank's payment widget) with inflated version numbers like 99.0.7 so they win resolution over legitimate internal packages.
  The postinstall stager downloaded reconnaissance payloads from `oob.moika[.]tech`, harvested environment variables and developer context, and — notably — detected CI/CD environments to avoid triggering alerts in monitored pipelines.
  For training and build isolation (3.4.2), that last detail matters: pipeline-only telemetry misses a campaign that deliberately goes quiet inside CI, so the controls have to be preventive — `.npmrc` scope-locking so internal scopes never fall back to the public registry, `--ignore-scripts` in build environments, and pinned known-good versions.
- **Malicious MCP servers as a packaged supply chain product:** Kaspersky's [Securelist analysis](https://securelist.com/model-context-protocol-for-ai-integration-abused-in-supply-chain-attacks/117473/) built a proof-of-concept malicious MCP server (`devtools-assistant` on PyPI) that harvested SSH keys, AWS credentials, `.env` files, and browser secrets while returning legitimate-looking output, with exfiltration disguised as GitHub API traffic.
  It catalogs the protocol-level abuse paths — name spoofing, tool poisoning, server shadowing in multi-server setups, and rug-pull updates — and recommends exactly the C3.4.2 workflow: allow-listed servers, scanning before deployment, container isolation, and centralized kill switches.
- **Training-isolation incidents (June 2026):** C3.4 covers two requirements — 3.4.1 (runtime components not shared across environment boundaries) and 3.4.2 (training and fine-tuning environments isolated from production); the configuration-artifact-governance material above is supporting context for 3.4.1. The clearest current evidence for 3.4.2 is the **ShadowRay 2.0** campaign (Oligo Security, November 2025): [CVE-2023-48022](https://nvd.nist.gov/vuln/detail/CVE-2023-48022) (CVSS 9.8) is an unauthenticated RCE on the Ray Jobs API that was never directly patched, and exposed Ray instances grew to 230,000+ while the IronErn440 actor (MITRE Campaign C0045) exfiltrated training data and built a self-propagating cryptojacking botnet from internet-reachable training clusters. The **Mercor/Meta breach** ([March 27, 2026](https://thenextweb.com/news/meta-mercor-breach-ai-training-secrets-risk)) is the shared-vendor counterpart — ~4 TB exposed across multiple frontier labs that all used the same data-labeling vendor, including proprietary labeling protocols and training strategies. NIST [SP 800-218A](https://csrc.nist.gov/pubs/sp/800/218/a/final) remains the controlling standard: separate execution environments, restrict development to approved environments, keep production data only within them, and apply least privilege to training pipelines and registries.
- **Digital Omnibus status check (June 2026):** The May 7 political agreement is still pending formal adoption — [Gibson Dunn's analysis](https://www.gibsondunn.com/eu-ai-act-omnibus-agreement-postponed-high-risk-deadlines-and-other-key-changes/) notes it needs a Parliament plenary vote, Council adoption, and Official Journal publication, expected between June and July 2026, ahead of the original August 2, 2026 deadline.
  Until publication, the December 2, 2027 (Annex III) and August 2, 2028 (Annex I) dates remain provisional; Article 15 cybersecurity expectations are unchanged either way.

New audit checks from this cycle:

- Run local MCP server discovery scans (the NSA CSI's recommendation) in every environment and reconcile findings against the approved MCP inventory; unexplained listeners are an isolation failure even before exploitability is assessed.
- Verify every database-fronting MCP server (Doris, Pinot, RDS, and equivalents) authenticates clients before invoking tools; treat the unpatched Alibaba RDS MCP as unsafe to deploy where any cross-environment network path exists.
- Check agent-framework API surfaces for insecure defaults: PraisonAI ≥4.6.34 with the legacy Flask server disabled, and any orchestration API bound to `0.0.0.0` flagged for review.
- Confirm `.npmrc` (and pip equivalent) scope-locks internal namespaces to internal registries with no public fallback, and that build/training environments run package installs with lifecycle scripts disabled.
- Require MCP message signing with expiry and replay protection (per the NSA CSI) for any MCP traffic that crosses a trust boundary, and log tool-inventory drift — new tools, renamed tools, changed descriptions — as review-gate events.
