# C5.2: Authorization & Policy

> [Back to C05 Index](C05-Access-Control.md)
> **Last Researched:** 2026-04-15 (3rd pass)

## Purpose

Implement access controls for all AI resources with explicit permission models and audit trails. AI systems introduce authorization challenges that go beyond traditional application security: vector databases, embedding indices, model registries, and prompt caches all require explicit access control. Data classification labels must propagate through AI pipelines to derived resources, and authorization decisions should be externalized to dedicated policy decision points to prevent scattered, inconsistent logic across AI microservices.

> **Scope note:** Controls in this section govern authorization policy definition, enforcement, and audit for AI resources. Identity and authentication are addressed in C5.1; query-time enforcement at the data layer is in C5.3; output-level filtering in C5.4. Per ASVS v5 14.1.1, all sensitive data should be identified and classified into protection levels; C5.2.11 extends this to AI-specific data types not enumerated in ASVS.

| 5.2.1 | **Verify that** every AI resource (datasets, models, endpoints, vector collections, embedding indices, compute instances) enforces access controls (e.g., RBAC, ABAC) with explicit allow-lists and default-deny policies. | 1 | Unauthorized access to training data, model weights, or inference endpoints; privilege escalation through permissive default policies. | Attempt access to each resource type with unauthenticated and under-privileged credentials; confirm default-deny by testing access without explicit grants; review ACL configurations for all AI resource types. | Vector databases (Pinecone, Weaviate, Milvus) have varying RBAC maturity. Some require application-layer enforcement rather than native database-level controls. Embedding indices are often overlooked in access control audits. **[2026-03]** Google Cloud API keys default to "Unrestricted" status, automatically gaining access to all enabled APIs including Gemini — a clear violation of default-deny. Pinecone and managed Weaviate now support document-level filtering and role-based retrieval natively. Permit.io provides fine-grained policy-based access control spanning apps, APIs, data layers, and AI agents. **[2026-04]** Cerbos now supports end-to-end authorization for AI agents and MCP servers, controlling which documents agents can retrieve from vector stores (Pinecone, Weaviate, Chroma, Qdrant) based on requesting user entitlements. IBM's July 2025 Cost of a Data Breach report found that 13% of organizations experienced breaches of AI models or applications, and of those breached, 97% lacked proper AI access controls — underscoring how common default-allow postures remain. Shadow AI breaches (employees using unsanctioned AI tools) averaged $670K more than traditional incidents, with 65% compromising PII and 40% exposing IP. The March 2026 Mercor/LiteLLM supply chain breach demonstrated cascading credential failure: attackers compromised LiteLLM (an AI API proxy used by thousands of organizations), harvested credentials from an estimated 500,000 machines within 40 minutes, and exfiltrated 4TB of data including source code, user databases, and API keys from the $10B AI recruiting startup — a direct consequence of permissive default-allow access policies across AI service dependencies. A CSA/Strata Identity survey (February 2026) found that 84% of organizations cannot pass a compliance audit focused on agent access controls, and only 23% have a formal agent identity strategy. |
| 5.2.2 | **Verify that** all access control modifications are logged with timestamps, actor identities, resource identifiers, and permission changes. | 1 | Undetected privilege escalation; inability to perform forensic analysis after a breach; insider threats modifying permissions without accountability. | Make permission changes and verify log entries contain all required fields; confirm logs capture both successful and failed modification attempts; validate log completeness across all AI resource types. | Standard audit logging practice. The AI-specific concern is ensuring coverage of non-traditional resources like vector store collections, model registry entries, and compute quota assignments. **[2026-04]** IBM's 2025 report found only 34% of organizations with AI governance policies perform regular audits for unauthorized AI use, and just 37% have policies to detect shadow AI — meaning the majority of organizations lack the audit trail coverage needed to detect unauthorized AI resource access. The June 2025 Microsoft Copilot EchoLeak vulnerability (CVE-2025-32711, CVSS 9.3) demonstrated how zero-click prompt injection could exfiltrate data from OneDrive, SharePoint, and Teams without user interaction, highlighting the need for access control change logging that covers AI-integrated services. Meta's March 2026 internal Sev 1 incident involved an AI agent that posted responses and exposed user data to unauthorized engineers — the agent was not compromised, it simply had overly broad permissions and acted in unexpected ways, illustrating that audit trails must capture both the agent identity and the originating authorization grant to enable root-cause analysis. |
| 5.2.3 | **Verify that** access control audit logs are stored immutably and are tamper-evident. | 2 | Log tampering to conceal unauthorized access or data exfiltration; destruction of forensic evidence after compromise. | Attempt to modify or delete existing log entries; verify cryptographic chaining or append-only storage; confirm log integrity verification mechanisms are in place and tested. | Append-only storage (e.g., AWS CloudTrail with S3 Object Lock, immutable Azure Blob) provides tamper-evidence. For on-prem deployments, cryptographic hash chaining or write-once media may be needed. **[2026-04]** Google's Trillian provides an open-source, cryptographically verifiable append-only log backed by Merkle tree proofs — originally built for Certificate Transparency but now applicable to AI audit trails. Trillian's architecture stores only hashes on-chain (reducing storage by up to 90%) while enabling third-party auditors to verify log integrity without accessing raw payloads. For AI systems, this addresses the specific challenge of proving that access control logs for model registries or vector databases haven't been retroactively altered after an incident. |
| 5.2.4 | **Verify that** data classification labels (PII, PHI, proprietary, etc.) automatically propagate to derived resources (embeddings, prompt caches, model outputs). | 2 | Data classification loss during AI pipelines; sensitive data in embeddings or caches accessed without appropriate controls because labels were dropped during transformation. | Tag a source dataset with a classification label; trace label propagation through embedding generation, prompt caching, and output storage; confirm derived resources inherit the source classification. | This is an AI-specific challenge with limited tooling. Embeddings are mathematical transformations that lose explicit metadata. Label propagation typically requires custom pipeline instrumentation. Microsoft Purview and Google DLP have partial support. **[2026-04]** Atlan (a Leader in the 2026 Gartner Magic Quadrant for Data & Analytics Governance) now automatically propagates tags, classifications, sensitivity labels, and policy labels along lineage paths to downstream assets — including derived datasets and transformations. Atlan's MCP Server allows AI agents to query the lineage graph before using data, returning provenance, quality score, governance policy, and ownership in a single call. Column-level lineage stitches together dbt, Airflow, and warehouse query history into a unified graph. For AI pipelines specifically, ML teams can track full model lineage from training sets through features, transformations, model versions, and predictions. Collibra supports manual and automated sensitive data classification but its lineage capabilities remain limited to cloud editions. |
| 5.2.5 | **Verify that** unauthorized access attempts and privilege escalation events trigger real-time alerts with contextual metadata. | 2 | Delayed detection of active attacks; brute-force credential attacks against model endpoints; privilege escalation going unnoticed during active exploitation. | Generate unauthorized access attempts and privilege escalation scenarios; confirm alerts fire within defined SLA; verify alerts include actor identity, target resource, action attempted, and timestamp. | Requires SIEM integration (Splunk, Sentinel, Elastic) with AI platform event sources. Alert fatigue is a risk; tuning thresholds for AI-specific resources (e.g., repeated vector DB queries) is an ongoing challenge. **[2026-03]** A critical blind spot: when actions execute through an AI agent, logging systems attribute activity to the agent's identity rather than the requesting user, obscuring who initiated the action. Wing Security documented this "agentic authorization bypass" pattern where security teams lose the ability to attribute intent during investigations. Organizations should ensure audit logs capture both the agent identity and the originating user context. Noma Security identifies "shadow permissions" — where prompt manipulation or unexpected tool interactions cause agents to access resources indirectly — as a pattern invisible to conventional alerting. **[2026-04]** Microsoft Sentinel's UEBA (User and Entity Behavior Analytics) behaviors layer is now GA, aggregating and sequencing related events into normalized behaviors so analysts can understand who-did-what-to-whom without manually correlating raw logs — relevant for detecting unusual AI resource access patterns. Harmonic Security provides inline detection and control over all AI tools (approved or shadow) with direct Sentinel and Splunk integration for actionable AI-specific alerts. The August 2025 Salesloft/Drift OAuth token compromise (UNC6395) affected 700+ organizations because the attacker "used legitimate third-party access that looked routine" — highlighting how AI integrations with broad OAuth scopes can bypass conventional alerting when access patterns appear normal. The CSA Agentic Trust Framework (February 2026) prescribes behavioral anomaly detection that triggers mid-execution policy re-evaluation as agents progress through maturity levels — from mandatory human sign-off at Level 1 through conditional auto-approval at Level 3-4, with continuous trust re-evaluation and circuit breakers for autonomous operation cessation at every tier. |
| 5.2.6 | **Verify that** authorization decisions are externalized to a dedicated policy decision point (e.g., OPA, Cedar, or equivalent). | 3 | Scattered, inconsistent authorization logic embedded across AI microservices; difficulty auditing or updating policies; policy drift between environments; confused deputy attacks where agents with broad capabilities are manipulated into unauthorized actions (MITRE ATLAS AML.T0056). | Confirm a centralized PDP exists; verify that AI service components call the PDP for authorization rather than implementing local checks; test policy changes propagate to all enforcement points; verify OPA/Rego or Cedar policies evaluate tool access, device access (ABAC), and command authorization in layered defense-in-depth. | OPA (Rego), Cedar (AWS), Cerbos, and Casbin are mature options. Externalized policy is best practice but adds latency to inference paths. Caching strategies (see 5.2.8) are needed to offset performance impact. **[2026-03]** Cedar joined CNCF as a Sandbox project (Jan 2026), signaling growing adoption for fine-grained authorization. Cedar's static analysis tooling can model AI tools, RAG datasets, and operations as Cedar resources/actions, enabling pre-deployment policy impact analysis. OPAL (Open Policy Administration Layer) keeps policies consistent across microservices and AI agents. OpenID AuthZEN enables fine-grained, context-aware policy evaluation atop existing OAuth infrastructure. **[2026-04]** OPA can be deployed as an AI agent guardrail — checking Rego policy before each tool/API call, securing vector queries, and validating operation type, user role, and risk band at the LLM API gateway. Practical deployments report ~30 lines of Rego for multi-tenant ABAC policies, but require 60+ unit tests to prevent policy contradictions. Sidecar OPA deployments avoid single points of failure but introduce 10-20 second propagation delays for policy updates; centralized deployments guarantee instant consistency but need HA infrastructure. Cerbos provides declarative YAML policies with MCP server integration, dynamically enabling only permitted tools per agent session. Topaz (open-source, Aserto) combines OPA's Rego engine with a Google Zanzibar-inspired directory, delivering sub-millisecond authorization decisions via an embedded BoltDB store — supporting RBAC, ABAC, and ReBAC in a single evaluator relevant for latency-sensitive inference paths. Strata's Maverics AI Identity Gateway (GA November 2025) places an embedded OPA engine directly in the MCP request path, evaluating every tool invocation with full context (tool name, arguments, agent identity, delegating user) before any request reaches upstream services — producing per-tool scoped tokens with 5-second TTLs and delegation chains tracing every action to the authorizing human. The NCCoE published a concept paper in February 2026 ("Accelerating the Adoption of Software and AI Agent Identity and Authorization") acknowledging that existing IAM frameworks were designed for human users and static services, and proposing adapted authorization models for agents that autonomously access tools, query databases, and execute operations across multiple systems. |
| 5.2.7 | **Verify that** policies evaluate dynamic attributes at runtime including user role or group, resource classification, request context, tenant isolation, and temporal constraints. | 3 | Static RBAC failing to account for contextual risk (e.g., unusual access times, cross-tenant requests); overly permissive access when context changes after initial grant. | Test access decisions under varying contexts (different times, locations, tenant contexts); confirm that changing a dynamic attribute (e.g., revoking group membership) immediately affects authorization; verify temporal constraint enforcement. | Full ABAC requires rich attribute sources (IdP, CMDB, classification service). NIST SP 800-162 provides the reference framework. Integration complexity increases with the number of attribute sources. **[2026-03]** ISACA (2025) identifies that modern agent security blends RBAC for coarse boundaries with ABAC for parameter-level constraints, with policies enabling conditional logic allowing actions only when roles, attributes, and contexts align. NIST AI Agent Standards Initiative emphasizes least privilege, just-in-time access, and task-scoped privileges as core authorization principles for agents. **[2026-04]** Oso documents four complementary permission strategies for AI agents: JIT access (short-lived credentials discarded after task completion), human-in-the-loop approval for sensitive operations, context-aware restrictions incorporating device trust/geolocation/time-of-day, and behavioral guardrails with automated token revocation for anomalous patterns like bulk downloads or off-hours access. Microsoft's Zero Trust for AI initiative (March 2026) adds a dedicated AI pillar to its Zero Trust Workshop covering 700+ security controls across 116 logical groups, with a Zero Trust Assessment for AI tool launching summer 2026 — tests are derived from NIST, CISA, and CIS standards and include agent identity verification, AI pipeline data protection, and enterprise-wide AI usage monitoring. |
| 5.2.8 | **Verify that** policy cache TTL values are defined based on resource sensitivity, with shorter TTLs for high-sensitivity resources, and that cache invalidation capabilities are available. | 3 | Stale cached permissions allowing access after revocation; delayed enforcement of emergency access revocations during incidents. | Review cache TTL configurations per resource sensitivity tier; test that cache invalidation propagates within defined SLA; verify high-sensitivity resources use TTLs of seconds rather than minutes. | Balances security (short TTL) vs. performance (long TTL). For inference-time authorization, even milliseconds of policy evaluation latency can impact throughput. Event-driven cache invalidation is preferred over TTL-only approaches. **[2026-04]** Topaz demonstrates a practical architecture: all authorization data is stored locally in an embedded BoltDB database so decisions are evaluated without network round-trips, achieving sub-millisecond latency. OPAL (sitting atop OPA) delivers real-time policy and data updates to distributed authorization instances, enabling event-driven cache invalidation rather than relying solely on TTL expiry. For high-sensitivity AI resources (model weights, training pipeline configs), OPAL's push-based updates ensure revocations propagate within seconds rather than waiting for cache expiry. |
| **5.2.9** | **Verify that** privileged access to model weights, training pipelines, and production AI configuration is provisioned on a just-in-time basis with a defined maximum session duration and automatic expiry, and permanent standing privileged access to these resources is not permitted. | 2 | Persistent standing privileges to model weights enable insider exfiltration; always-on access to training pipelines allows undetected data poisoning or unauthorized fine-tuning; compromised service accounts with permanent access provide attackers dwell time for lateral movement across AI infrastructure. | Verify no permanent privileged accounts exist for AI resources — enumerate all accounts with write access to model registries, training pipelines, and production configs; confirm each uses JIT provisioning with maximum session durations; test that access automatically expires; attempt to access resources after session expiry; review PAM tool configurations (CyberArk, HashiCorp Vault/Boundary, Azure PIM) for JIT enforcement. | **[2026-04]** CyberArk's January 2026 study found only 1% of organizations have fully adopted JIT privileged access, while 91% report at least half of their privileged access is always-on. 54% discover unmanaged privileged accounts weekly. HashiCorp Vault/Boundary supports dynamic short-lived credentials that are created at access time and destroyed after session completion. Azure PIM provides time-bound role activation with approval workflows. The gap is AI-specific: training pipelines carry invisible privileges (unrestricted data access, automated model updates, cross-cloud execution) that most PAM tools were not designed to inventory or govern. Break-glass access with ticketing and time-bound approvals is the recommended fallback for vendor support scenarios. |
| **5.2.10** | **Verify that** authorization policies governing AI agent tool access and action permissions include defined validity periods, and that expired policies are automatically revoked or excluded from evaluation rather than persisting as default-allow, ensuring that permission scope remains bounded as available tools and capabilities evolve. | 3 | Unbounded agent permissions that persist after tool decommissioning or capability changes; "shadow permissions" where prompt manipulation or unexpected tool interactions cause agents to access resources indirectly; privilege creep as new tools are added without revoking stale grants; confused deputy attacks exploiting overly broad agent capabilities (MITRE ATLAS AML.T0056). | Review all agent authorization policies for explicit validity periods (start/end dates or TTLs); confirm the policy engine rejects or ignores expired policies rather than defaulting to allow; add a new tool to the agent's available set and verify no access is granted without explicit policy creation; remove a tool and verify its associated permissions are revoked; test with Cerbos, OPA, or Permit.io policy evaluation to confirm expired grants return deny. | **[2026-04]** Oso, Cerbos, and Permit.io all support fine-grained agent authorization, but none of the major open-source policy engines natively enforce temporal constraints or automatic policy expiry — this must be implemented at the application or orchestrator layer. Cerbos uses declarative YAML policies mapping agent actions to roles, with MCP integration that dynamically enables only permitted tools per session. Permit.io's Four-Perimeter Framework (Prompt Filtering, RAG Data Protection, External Access, Response Enforcement) covers agent authorization across AI stacks. Delegated access patterns where agents inherit user identity through token exchange are fundamentally safer than application-wide service accounts. Short-lived, scoped capability tokens should be discarded after task completion. The EU AI Act (enforcement starting August 2026) will scrutinize agent access patterns for compliance. The CSA Agentic Trust Framework (February 2026) defines an Intern-to-Principal maturity model where promotion between autonomy levels requires passing five gates: demonstrated accuracy, security audit at target level, measurable impact, clean operational history, and explicit stakeholder approval — no agent is promoted without human sign-off. The OpenClaw AI agent crisis (early 2026, 135K+ GitHub stars) exposed 21,000+ instances and 1,184 malicious skills in its marketplace, demonstrating how unbounded agent tool permissions without validity periods enable rapid exploitation at scale. |
| **5.2.11** | **Verify that** a documented data classification taxonomy covering AI-specific data types (embeddings, model weights, prompt templates, RAG context assemblies, fine-tuning datasets, agent tool schemas) is defined, and that AI assets are labeled in accordance with this taxonomy. | 2 | Unclassified AI assets treated with inappropriate protection levels; model weights stored alongside non-sensitive data without access restrictions; prompt templates containing system instructions or guardrails exposed as public artifacts; embeddings that encode sensitive source data treated as non-sensitive because their mathematical nature obscures the original content; fine-tuning datasets containing PII processed without privacy controls. | Request the organization's data classification taxonomy and confirm it enumerates AI-specific data types (embeddings, model weights, prompt templates, RAG context assemblies, fine-tuning datasets, agent tool schemas); sample AI assets from each category and verify classification labels are applied; confirm classification labels propagate through pipelines (cross-reference with 5.2.4); verify that protection controls match the assigned classification tier. | **[2026-04]** NIST IR 8596 (Cybersecurity Framework Profile for AI, December 2025 draft) urges organizations to maintain inventories covering models, agents, APIs/keys, datasets/metadata, and embedded AI integrations, along with maps of end-to-end AI data flows. NIST AI 100-2 E2025 provides an adversarial ML taxonomy that implicitly defines asset sensitivity (training data as poisoning targets, model weights as exfiltration targets). Microsoft Purview supports sensitivity labels with autolabeling policies and trainable classifiers, and its Fabric APIs now allow programmatic label management — but coverage of AI-specific artifacts like embeddings and prompt templates remains partial. No off-the-shelf tool comprehensively classifies all six AI data types listed in this requirement; organizations typically extend existing DLP taxonomies with custom categories. The ODI published a "Data for AI" taxonomy addressing data provenance and fitness considerations. The initial public draft of NIST IR 8596 (December 2025) explicitly expands asset inventory requirements to include models, system prompts and templates, training and evaluation datasets, RAG sources (vector databases), embeddings, agent tool permissions, and API keys — moving closer to a standardized taxonomy that aligns with this requirement's scope. |
