# C12: Monitoring, Logging & Anomaly Detection

> **Source:** [`1.0/en/0x10-C12-Monitoring-and-Logging.md`](https://github.com/OWASP/AISVS/blob/main/1.0/en/0x10-C12-Monitoring-and-Logging.md)
> **Requirements:** 21 | **Sections:** 5

## Control Objective

Deliver real-time and forensic visibility into what the model and other AI components see, do, and return, so AI-specific threats can be detected, triaged, and learned from. The chapter scope is deliberately AI-specific: log content unique to AI systems (model identifier, token usage, safety filter outcomes, prompt/response handling), AI-specific abuse and attack detection (jailbreak, prompt injection, extraction, multi-turn trajectory, covert channels over LLM endpoints), model and data drift detection, AI telemetry signals (token attribution, output/input ratio anomalies), proactive agent behavior monitoring, and training-data and model-lifecycle audit logging.

## Threat Landscape

Known attacks, real-world incidents, and threat vectors relevant to this chapter:

- **Shadow AI and unmonitored usage** — 98% of organizations report unsanctioned AI use (Knostic 2025). 63% of employees paste sensitive data into personal chatbot accounts, with the average enterprise running an estimated 1,200 unofficial AI applications and 86% of organizations reporting no visibility into AI data flows. Shadow AI adds $670K to average breach costs and takes 247 days to detect on average.
- **AI-specific failure detection gaps** — AI failures take an average of 4.5 days to detect with current tooling. Traditional metrics (latency, error rates) appear healthy while output quality silently degrades. Only 62% of organizations use detailed tracing; only 72% of those with production agents have tracing. As of early 2026, 85% of organizations plan LLM observability but only 8% have actually implemented it (Elastic Observability Report 2026).
- **Multi-turn abuse trajectories** — Crescendo attacks (USENIX Security 2025) escalate through benign-seeming turns; Deceptive Delight achieves 65% success within three turns. Per-request monitoring misses these cross-turn patterns entirely.
- **Log tampering and forensic integrity** — PoisonedRAG showed 5 documents corrupt 90% of RAG outputs. Medical training data poisoning (0.001% of tokens) produces harmful models undetectable by standard benchmarks. Beyond data poisoning, AI agent logs are themselves attack targets — agents that make autonomous decisions without reliable logging create blind spots that attackers exploit to hide data exfiltration or unauthorized actions. Adversaries may also spoof agent identities in multi-agent systems to perform actions under another persona.
- **Training telemetry as a poisoning signal** — C12.3.10 treats training runtime, loss trajectory, convergence rate, validation divergence, resource utilization, and artifact hashes as security telemetry, not just MLOps hygiene. [NIST AI 100-2e2025](https://www.nist.gov/publications/adversarial-machine-learning-taxonomy-and-terminology-attacks-and-mitigations-0) keeps data poisoning and backdoor attacks in the core adversarial ML taxonomy, and [MITRE ATLAS](https://atlas.mitre.org/) maps poisoning, extraction, and evasion to detection use cases; the practical gap is attribution, since natural data quality failures and adversarial poisoning can create similar curves.
- **AI-assisted attack operations** — GTG-1002 campaign (November 2025) targeted ~30 organizations with AI handling 80-90% of operations including reconnaissance, exploit development, and lateral movement. Only 4-6 human decision points per campaign. Traditional IR playbooks had no framework for this.
- **Cascading failures in agentic systems** — Research shows a single compromised agent can poison 87% of downstream decision-making within 4 hours. OWASP's Top 10 for Agentic Applications (2026) dedicated ASI08 specifically to cascading failures. Memory injection attacks (Lakera AI research) demonstrate how indirect prompt injection via poisoned data sources can corrupt an agent's long-term memory, creating persistent false beliefs that propagate through multi-agent workflows.
- **Limited availability of standardized AI-specific detection rule packs in SIEM platforms** — Standard SIEM platforms have limited support for detecting AI-specific attack patterns (e.g., prompt injection, multi-turn jailbreaks). Most AI-in-SIEM features focus on assisting investigation of traditional events rather than detecting attacks targeting AI systems.
- **Prompt injection dominates real incidents** — Adversa AI's 2025 AI Security Incidents Report found 35% of all real-world AI security incidents were caused by simple prompt injection, with some leading to $100K+ in real losses. GenAI was involved in 70% of incidents, but agentic AI caused the most dangerous failures including crypto thefts, API abuse, and supply chain attacks. System prompt extraction was the most common attacker objective in Q4 2025.
- **Indirect prompt injection is now observable in the wild** — CSA's April 26, 2026 research note reported that Google observed a 32% relative increase in malicious indirect prompt injection content from November 2025 to February 2026 across 2-3 billion crawled pages per month, while Unit 42 documented real web-based IDPI payloads for SEO poisoning, ad-review evasion, data destruction, unauthorized transactions, sensitive information leakage, and system-prompt leakage. Monitoring needs prompt-source provenance, rendered-vs-raw content inspection, and intent analysis rather than simple keyword signatures.
- **CI/CD and SDLC agents create new exfiltration loops** — Comment and Control (April 15, 2026) demonstrated that PR titles, issue bodies, issue comments, and hidden HTML comments can hijack AI coding agents running in GitHub Actions. Claude Code Security Review, Gemini CLI Action, and GitHub Copilot Agent all leaked credentials or tokens through GitHub-native channels (comments, logs, commits), showing that "no external callback" attacks can still exfiltrate secrets if agents share a runtime with build credentials.
- **Alert fatigue and observability-action gap** — 59% of organizations are drowning in telemetry but cannot get answers when needed. 36% are buried in alert fatigue. 39% have integration gaps between monitoring tools and workflows. The monitoring system itself can be an attack target — since LLM-based monitoring inherits the same vulnerabilities it aims to detect, an attacker who compromises monitoring could blind the organization to ongoing attacks or create false alerts masking real threats.
- **AI-specific SIEM rules emerging but immature** — MITRE ATLAS Splunk app (2025) was the first production SIEM with AI attack detection rules. As of December 2025, ANY.RUN released Sigma rules for threat detection. SigmaGen builds ATT&CK-mapped rules using LLMs. However, no major SIEM vendor ships comprehensive AI-attack-specific rule packs — the gap between "AI assists SOC analysts" and "SIEM detects attacks on AI" persists. RSAC 2026 announcements (April 2026) narrowed but did not close this gap: Splunk's Agentic SOC ships six agent-specialized detection/triage agents embedded in Enterprise Security, and Datadog's Bits AI Security Analyst (now GA) handles alert investigation inside Cloud SIEM.
- **LLM routing-layer compromise** — Help Net Security (April 16, 2026) detailed research on 28 paid and 400 free LLM routers: 9 routers (1 paid, 8 free) actively injected malicious tool calls into sessions; one drained funds from an exposed Ethereum key; 17 free routers misused AWS canary credentials. Some routers applied tampering only after a warm-up period or specifically when the router inferred autonomous "YOLO mode" with no human approval — evading evaluation and test-harness detection. Critically: no widely deployed mechanism verifies that the tool-call payload received by the client matches what the model originally produced. Client-side defenses (fail-closed policy gates, response-side anomaly screening, append-only transparency logs) remain DIY.
- **AI supply-chain pipeline compromise** — On March 24, 2026 malicious LiteLLM releases (v1.82.7, v1.82.8) went live on PyPI for roughly 40 minutes before quarantine. The packages dropped a `litellm_init.pth` loader (and an injected `litellm/proxy/proxy_server.py` payload in 1.82.7) that harvested environment variables, SSH keys, cloud credentials, Kubernetes data, Docker configs, shell history, database credentials, wallet files, and CI/CD secrets, AES-256/RSA-4096 encrypted them, and exfiltrated to `models.litellm[.]cloud` with header `X-Filename: tpcp.tar.gz`. Persistence was installed via `~/.config/sysmon/sysmon.py` and `~/.config/systemd/user/sysmon.service`, with a follow-on beacon to `https://checkmarx[.]zone/raw` writing `/tmp/pglog` and `/tmp/.pg_state`; where service-account tokens were available, privileged `node-setup-*` pods spread the payload across Kubernetes. The incident is stage 4 of a TeamPCP cascade (Trivy v0.69.4 on March 19, an npm worm across 44+ packages March 20-22, Checkmarx KICS / OpenVSX March 23, Telnyx PyPI 4.87.1/4.87.2 backdoored March 27). Docker images that pinned dependencies in `requirements.txt` were unaffected; pip installs during the window must be treated as full-credential exposure events, not just package removal. Monitoring lessons: package-install telemetry, outbound DNS to typosquat domains (`models.litellm[.]cloud`, `checkmarx[.]zone`, `83.142.209.203:8080`), `*.pth` file creation in `site-packages`, and unexplained `node-setup-*` pods all remain undervalued controls for AI-gateway supply chains.
- **Prompt-to-shell RCE in agent frameworks** — Microsoft Security Research's [May 7, 2026 disclosure](https://www.microsoft.com/en-us/security/blog/2026/05/07/prompts-become-shells-rce-vulnerabilities-ai-agent-frameworks/) showed that prompt injection now compounds into native code execution when AI agent frameworks expose unsafe primitives. [CVE-2026-26030](https://nvd.nist.gov/vuln/detail/CVE-2026-26030) (Semantic Kernel Python < 1.39.4) used `eval()` inside a default in-memory vector store filter, letting attacker-controlled prompts traverse AST and class hierarchies to RCE. [CVE-2026-25592](https://nvd.nist.gov/vuln/detail/CVE-2026-25592) (Semantic Kernel .NET < 1.71.0) exposed `DownloadFileAsync` as a `[KernelFunction]` without path validation, enabling arbitrary file writes — including startup folder persistence — directly from model invocations. The defensive guidance is explicit: "LLM is not a security boundary." Detection moves to **host-level** telemetry around agent processes (suspicious `cmd.exe`/`powershell.exe`/`bash.exe`/`curl.exe`/`certutil.exe` children, file writes to startup or scheduled-task paths, sandbox-to-host file movement), via Defender for Endpoint `DeviceProcessEvents` hunts on Semantic Kernel parent processes.
- **Telemetry manipulation as a first-class attack** — A [May 20, 2026 Help Net Security analysis](https://www.helpnetsecurity.com/2026/05/20/agentic-ai-security-llm-research/) of agentic AI in production operations elevated four overlapping attack categories: prompt injection via operational artifacts (tickets, wiki pages), retrieval poisoning of runbooks and incident histories, retrieval jamming that floods the knowledge base with blocker documents to trigger refusal loops, and telemetry manipulation where attackers shape the metrics and logs an agent reads to steer mitigation decisions without ever touching the model. The recommended architecture is propose-commit: models draft proposals only, non-bypassable policy-as-code gates enforce approvals, and integrity-protected audit logs let post-incident forensics reconstruct what the agent considered, which tools it called, what data it accessed, and why it chose its path. Monitoring metrics shift toward tool-call traces, gate-violation rates, refusal rates under jamming, and rollback completeness — not just latency or token counts.

- **LLM agents now run the intrusion itself** — Sysdig TRT's [May 10, 2026 observation](https://www.sysdig.com/blog/ai-agent-at-the-wheel-how-an-attacker-used-llms-to-move-from-a-cve-to-an-internal-database-in-4-pivots) is the first well-documented in-the-wild case of an LLM agent driving post-exploitation: Marimo notebook RCE (CVE-2026-39987) → credential harvest → AWS Secrets Manager replay through a Cloudflare Workers egress pool (12 redundant `GetSecretValue` calls across 11 edge IPs in 22 seconds, defeating IP-based detection) → SSH bastion → full PostgreSQL exfiltration in a two-minute window. Four command-telemetry signatures gave the agent away: improvised targeting (dumping a `credential` table that didn't exist in the inferred schema), a Chinese-language planning comment leaked mid-execution at sub-second cadence across six worker IPs, machine-consumption formatting (`echo '---'` delimiters, quoted-EOF HEREDOCs bundling six SQL statements, `| head -30`, `2>/dev/null`), and output-driven handoffs reusing values from earlier command output. The monitoring lesson is new: **command-stream stylometry is now detection telemetry** — formatting choices that mark output as destined for LLM parsing rather than human eyes are alertable signal.
- **N-day weaponization of AI inference infrastructure in hours** — [Sysdig's LMDeploy analysis](https://www.sysdig.com/blog/cve-2026-33626-how-attackers-exploited-lmdeploy-llm-inference-engines-in-12-hours) caught CVE-2026-33626 (SSRF in LMDeploy's vision-model image URL loader, fixed in v0.12.3) being exploited 12 hours 31 minutes after the GitHub advisory published — an 8-minute session probing AWS IMDS, Redis, MySQL, and LMDeploy's `/distserve/p2p_drop_connect` admin endpoint via out-of-band DNS callbacks. Vision-language endpoints that fetch attacker-supplied image URLs are SSRF primitives by design; Falco's instance-metadata-contact rules and egress monitoring on inference processes (link-local, RFC 1918, loopback destinations) are the practical detection layer.

## Tooling & Implementation

Current tools, frameworks, and libraries that help implement these controls:

- **LLM observability:** [LangSmith](https://smith.langchain.com/) (~0% overhead, $39/seat/month), [Langfuse](https://langfuse.com/) (open-source MIT, self-hostable, prompt versioning, agent graph visualization), [Arize Phoenix](https://arize.com/) (open-source EL2, embedding drift monitoring, LLM-as-judge), [MLflow Tracing](https://mlflow.org/docs/latest/genai/tracing/) (OpenTelemetry-compatible tracing for LLM and agent calls, with production traces feeding evaluation datasets), [W&B Weave](https://docs.wandb.ai/weave/concepts/what-is-weave) (LLM traces, prompt/data/model versioning, production guardrails and scorers), [WhyLabs](https://whylabs.ai/) (Apache 2.0 since Jan 2025, real-time drift + prompt injection detection), [Helicone](https://www.helicone.ai/) (gateway-based, cost tracking across 100+ LLMs)
- **OpenTelemetry GenAI SIG:** [Semantic conventions](https://opentelemetry.io/docs/specs/semconv/gen-ai/) remain in "Development" status and define attributes including `gen_ai.operation.name`, `gen_ai.usage.input_tokens`, `gen_ai.tool.name`, provider-specific conventions for Anthropic/OpenAI/AWS Bedrock. Prompt content is opt-in/sensitive by design. As of April 28, 2026, [semantic-conventions v1.41.0](https://changelog.opentelemetry.io/) introduced a breaking `execute_tool` span-naming change requiring the tool name, so teams should pin schema versions and test downstream SIEM/APM parsers before upgrading. [Agent span conventions](https://opentelemetry.io/docs/specs/semconv/gen-ai/gen-ai-agent-spans/) define create-agent, invoke-agent, invoke-workflow, and execute-tool span types with `gen_ai.conversation.id` tracking. Opt-in via `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental`. Datadog [natively supports](https://www.datadoghq.com/blog/llm-otel-semantic-convention/) OTel GenAI conventions for LLM observability. The project's 2026 ["Inside the LLM Call" walkthrough](https://opentelemetry.io/blog/2026/genai-observability/) demonstrates the three-span pattern (`invoke_agent` → `chat` → `execute_tool`), confirms content capture stays metadata-only unless explicitly enabled, and notes that VS Code Copilot, OpenAI Codex, and Claude Code now emit OTel telemetry — meaning coding-agent activity can land in the same trace pipeline as application LLM calls.
- **Drift detection:** [Evidently AI](https://www.evidentlyai.com/) (20+ drift detection methods, now supports LLM evaluations), [NannyML](https://www.nannyml.com/) (performance estimation without ground truth), WhyLabs (embedding drift for LLMs). Key gap: LLM drift requires embedding drift + LLM-as-judge, not just statistical distribution shifts.
- **Training-pipeline telemetry:** C12.3.10 is best implemented by treating ML training runs as auditable security events: log dataset and code commit hashes, model-card/version IDs, dependency snapshots, runtime duration, loss and validation curves, convergence rate, GPU/CPU utilization, and checkpoint integrity. MLflow, W&B, Comet ML, Kubeflow Pipelines, and Airflow can capture much of this, but poisoning attribution still needs adversarial review because benign data drift, label noise, dependency changes, and targeted backdoors can all look like abnormal training behavior.
- **SIEM integration:** [Datadog LLM Observability](https://www.datadoghq.com/) (LiteLLM integration, 800+ OOTB detection rules with MITRE mapping, native OTel GenAI convention support; [Bits AI Security Analyst](https://www.datadoghq.com/blog/bits-ai-security-analyst/) GA, Cloud SIEM-resident alert investigator), [Microsoft Sentinel](https://azure.microsoft.com/en-us/products/microsoft-sentinel) (RSAC 2026: [MCP entity analyzer GA](https://techcommunity.microsoft.com/blog/microsoftsentinelblog/what%E2%80%99s-new-in-microsoft-sentinel-rsac-2026/4503971) for reasoned risk assessments, A365 Observability connector preview for AI agent telemetry, graph tools for relationship-based hunting), Splunk ([Agentic SOC](https://openclawai.io/blog/splunk-agentic-soc-rsac-2026) at RSAC 2026 for triage/correlation/investigation; Cisco's pending acquisition of [Galileo](https://blogs.cisco.com/news/cisco-announces-the-intent-to-acquire-galileo) (Apr 9, 2026) adds agent development lifecycle observability), Elastic. [MITRE ATLAS Splunk App](https://splunkbase.splunk.com/app/8527) — first production SIEM with 2-tier AI attack detection rules. [Sigma](https://github.com/SigmaHQ/sigma) open standard enables SIEM-agnostic detection rules convertible to 40+ platforms; [SigmaGen](https://blogs.night-wolf.io/sigmagen-ai-powered-attck-mapped-threat-detection-with-sigma-rules) automates ATT&CK-mapped rule generation.

Note: 73% of deployments are vulnerable to prompt injection but only 34.7% have defenses deployed. The [CSA April 2026 note](https://labs.cloudsecurityalliance.org/research/csa-research-note-indirect-prompt-injection-in-the-wild-2026/) and [Unit 42 IDPI telemetry](https://unit42.paloaltonetworks.com/ai-agent-prompt-injection/) make a stronger case for browser/web-crawler-side detection that classifies raw, rendered, and hidden content before an agent ingests it.
- **Inference-host runtime detection:** The Sysdig LMDeploy and Marimo cases make a concrete tooling argument: [Falco](https://falco.org/) rules for instance-metadata-service contact from containers/hosts ("Contact EC2/GCP/Azure Instance Metadata Service") plus egress monitoring on inference processes (alerting on connections to link-local, RFC 1918, or loopback destinations) catch SSRF-to-IMDS chains that LLM-layer observability never sees. For LLM-agent-driven intrusions, Sysdig recommends alerting on credential-file access from notebook environments, cloud API calls fanned across multiple edge-network IPs within seconds, schema enumeration followed by rapid table dumps, and command formatting that targets machine consumption (delimiter echoes, HEREDOC bundling, output suppression) in SSH sessions.
- **Agent runtime and CI/CD hardening:** Comment and Control shows that GitHub Actions agents need runtime isolation, not just prompt filtering: avoid `pull_request_target` on untrusted PRs where possible, keep API keys out of the same process tree as agent tools, disallow shell/process-inspection tools, redact Actions logs, block credential-shaped content in comments and commits, and monitor GitHub audit logs for AI-agent authored exfiltration paths. Microsoft's [May 7, 2026 Semantic Kernel disclosure](https://www.microsoft.com/en-us/security/blog/2026/05/07/prompts-become-shells-rce-vulnerabilities-ai-agent-frameworks/) extends this to agent SDKs: ship Defender for Endpoint hunts on `DeviceProcessEvents` filtered to Semantic Kernel parent processes, alert on unexpected `cmd.exe`/`powershell.exe`/`bash.exe`/`curl.exe`/`certutil.exe` children, and treat writes to startup and scheduled-task paths from agent hosts as detection-grade telemetry. Emerging runtime-security products such as [Capsule Security](https://www.capsulesecurity.io/) and open-source experiments such as ClawGuard focus on pre-tool-use enforcement, action-chain policy, and agent security graphs, but this space is still early.
- **Propose-commit and gate telemetry:** The [Help Net Security May 20, 2026 survey](https://www.helpnetsecurity.com/2026/05/20/agentic-ai-security-llm-research/) argues that production agents need a strict propose-commit split — models draft proposals, non-bypassable policy-as-code gates enforce approvals and rollback, and audit logs are integrity-protected so post-incident reconstruction is possible. The corresponding monitoring metrics — tool-call traces, gate-violation rates, refusal rates under retrieval-jamming, and rollback completeness — are still custom dashboards in most deployments rather than shipped features in observability platforms.
- **LLM routing-layer defenses:** As of April 2026, commercial hardening for the router tier is effectively nonexistent — [Help Net Security research](https://www.helpnetsecurity.com/2026/04/16/llm-router-security-risk-agent-commands/) recommends a DIY stack: client-side fail-closed policy gates, response-side anomaly screening that diffs the router's tool-call payload against the model's original output, append-only transparency logs that let downstream auditors reconstruct the request chain, and high-risk shell command blocklists before tool execution. No open or commercial tool yet implements end-to-end router-payload integrity verification.
- **Agent monitoring:** [AgentOps](https://www.agentops.ai/) (session replay, time-travel debugging, 400+ LLMs, 12% overhead), [Lunary](https://lunary.ai/) (conversation replay, topic classification), Weights & Biases Weave (hierarchical multi-agent tracing), [AG2](https://docs.ag2.ai/latest/docs/blog/2026/02/08/AG2-OpenTelemetry-Tracing/) (Feb 2026, native OTel tracing for multi-agent systems — structured hierarchical traces of conversations, agent turns, tool executions across local and distributed deployments), [OpenObserve](https://openobserve.ai/blog/product-update-march-2026/) (Mar 2026, v0.70.0 — LLM observability with DAG trace visualization, session-level multi-turn inspection, per-user token/latency analytics), [Arize AX](https://arize.com/blog/best-ai-observability-tools-for-autonomous-agents-in-2026/) (Agent Graph visualization renders execution as a tree rather than linear trace — shows sub-agent delegations, tool fires, and state changes; built on Phoenix + OpenInference), [Galileo](https://futurumgroup.com/insights/cisco-to-acquire-galileo-ai-agent-observability-cant-run-at-human-speed/) (April 2026 pending Cisco acquisition; hallucination/bias detection, real-time multi-agent guardrails, cost + security metrics across the Agent Development Lifecycle)
- **Training-data & model-lifecycle audit (C12.7):** [MLflow Model Registry](https://mlflow.org/docs/latest/model-registry/) remains the most widely adopted open-source option for model versioning, stage transitions, and lineage display from experiment to deployment. [OpenLineage](https://openlineage.io/) (Linux Foundation, vendor-neutral lineage event standard with native dbt, Airflow, and Spark integrations) paired with [Marquez](https://marquezproject.ai/) captures dataset-to-model lineage events for C12.7.1 reconstruction. [DVC](https://dvc.org/) with content-hash dataset versioning, and the [Atlas ML-lifecycle provenance framework](https://arxiv.org/pdf/2502.19567) (2025), support C12.7.5 change tracking and forensic rollback. A practical pattern logs DVC-versioned datasets and SageMaker training jobs to MLflow so every artifact has a reproducible content hash. Gap: write-time per-document tagging (C12.7.4) and immutable, tamper-evident model-change records (C12.7.3) usually require gluing a data catalog, registry, and append-only store together rather than a single audit-grade product.

Identity-bound logging ties each entry to agent identifier, tenant context, authorization scope, and delegation status. Automated tamper detection monitors for missing sequences, invalid hash chains, or signature verification failures. ## Open Research Questions

- [ ] **When should full prompt/response content be logged?** — OTel GenAI marks content as opt-in/sensitive. C12.1.3 excludes content by default, while C12.1.6 allows full content on security events or with explicit consent and documented legal basis. But MITRE ATLAS Tier 2 detection (jailbreaks, prompt injection) *requires* content inspection. NIST IR 8596 recommends PII redaction. EU AI Act Art 12 requires logging events that identify risk. The tension between detection capability and privacy is unresolved.
- [ ] **Will OpenTelemetry GenAI conventions stabilize in time?** — Core GenAI conventions remain in "Development" status, and semantic-conventions v1.41.0 introduced an April 2026 `execute_tool` span naming change. Agent spans now cover create-agent, invoke-agent, invoke-workflow, and execute-tool types, but the transition plan for stable status has not been defined. OTel production deployment doubled year-over-year to 11% (Elastic 2026), and 89% of production users consider vendor OTel compliance critical. Schema changes could still break existing instrumentation, though the `OTEL_SEMCONV_STABILITY_OPT_IN` mechanism provides a migration path.
- [ ] **How do you detect multi-turn abuse trajectories at scale?** — Crescendo and Deceptive Delight attacks are invisible to per-request monitoring. Single-turn defenses show single-digit ASR while multi-turn exceeds 70%. Session-level analysis requires correlating across requests — expensive and architecturally complex.
- [ ] **Can AI-specific SIEM detection rules become standard?** — MITRE ATLAS Splunk app was the first step, and Sigma/SigmaGen now enable SIEM-agnostic rule generation convertible to 40+ platforms. Microsoft Sentinel's MCP entity analyzer (GA Apr 2026) adds AI-powered risk assessment. But comprehensive AI-attack-specific rule packs remain absent from major SIEM vendors — the gap between "AI assists SOC analysts" and "SIEM detects attacks on AI" is narrowing but persists.
- [ ] **How do you distinguish adversarial drift from natural concept drift?** — Both manifest as degraded outputs. Adversarial drift may be intentional poisoning (PoisonedRAG, VIA). Current tools detect drift but don't attribute cause. CSA (Sep 2025) warns incidents are often misattributed to natural drift rather than active attack.
- [ ] **How should shadow AI be governed?** — 98% of orgs have unsanctioned AI; 47% through personal accounts. Agent-based shadow AI (MCP servers, OAuth tokens) creates elevated-privilege access chains invisible to IT. No standard governance framework exists.
- [ ] **Can monitoring systems protect themselves from the attacks they detect?** — LLM-based monitoring inherits the vulnerabilities it aims to detect. An attacker who compromises the monitoring layer via prompt injection or adversarial inputs could blind the organization to ongoing attacks or generate false alerts masking real threats. No standard architecture for self-protecting monitoring exists.
- [ ] **How should monitoring preserve prompt provenance across raw, rendered, and hidden content?** — CSA and Unit 42's April 2026 findings show attackers embedding indirect prompts in HTML attributes, hidden comments, CSS-suppressed text, multilingual payloads, and Base64-like encodings. Current logs often store only the final text sent to the model, not the source, rendering state, or trust boundary that explains why the agent treated attacker data as instructions.
- [ ] **How do you close the observability-implementation gap?** — 85% of organizations plan LLM observability, but only 8% have implemented it (Elastic 2026). OTel production deployment doubled to 11% but remains low. The gap between intent and deployed monitoring creates a window where most AI systems run effectively unmonitored in production.
- [ ] **How do you detect cascading failures across multi-agent systems before downstream damage?** — OWASP ASI08 (2026) highlights that a single compromised agent can poison 87% of downstream decisions within 4 hours. Current monitoring tools observe individual agents but lack cross-agent propagation detection. The 4.5-day average detection gap means cascading failures can complete their damage long before detection fires.
- [ ] **Which training-run anomalies should trigger a security escalation?** — C12.3.10 now asks teams to alert on abnormal training duration, loss trajectory, and convergence. That is useful, but thresholds are model- and dataset-specific: a benign curriculum change, new optimizer, larger batch, mislabeled data burst, poisoned training data, or dependency compromise can all present as unstable loss curves. The open problem is turning experiment-tracking telemetry into a defensible poisoning triage workflow.
- [ ] **Can anyone monitor integrity across the LLM routing layer?** — April 2026 research on 428 routers found 9 silently rewriting tool calls, with some only activating in autonomous "YOLO mode" — evading evaluation harnesses by design. No widely deployed mechanism verifies that the tool-call payload received by the client matches what the model originally produced. Open questions: Should routers be required to sign payloads? Should clients maintain a verifier that replays the model request through an independent path and diffs the tool call? What telemetry schema should downstream systems consume to reconstruct the full request chain?
- [ ] **Can SOC-speed AI agents outpace the AI agents they monitor?** — RSAC 2026 marked the transition from "AI-assisted SOC" to purpose-built agentic SOC (Splunk's six Enterprise Security agents, Datadog Bits AI Security Analyst, Cisco/Galileo). But the detection layer now inherits every risk it watches for: if a Splunk SOC agent can be prompt-injected via an alert field, the attacker inherits the SOC's privileges. No public work yet measures prompt-injection resilience of SOC agents against the attacks they are deployed to catch.
- [ ] **Can SDLC agents safely read untrusted collaboration text while holding build credentials?** — Comment and Control showed the conflict directly: AI coding agents must read PR and issue text to work, but the same runtime often holds API keys, GitHub tokens, and commit privileges. The open problem is not just input filtering; it is how to enforce hard separation between untrusted workflow text, agent tools, build secrets, comments, commits, and Actions logs.
- [ ] **How do you detect telemetry manipulation against the agent's own observation surface?** — The May 20, 2026 Help Net Security survey treats telemetry manipulation as a peer of prompt injection: attackers shape metrics, logs, runbooks, and ticket text to steer agent decisions without touching the model. Current SIEM and APM stacks were built to trust their own telemetry, not to detect when it has been weaponized. Open questions: should agent-consumed telemetry be cryptographically attested? Should retrieved runbooks carry provenance and integrity tags? And how do you distinguish "the agent was prompt-injected" from "the agent was fed manipulated metrics" in post-incident triage?
- [ ] **Can command-stream stylometry reliably identify LLM-agent intrusions?** — Sysdig's May 10, 2026 Marimo case surfaced four telltale signatures (improvised targeting, leaked planning comments, machine-consumption output formatting, output-driven value handoffs) — all observable in standard shell/SSH telemetry, and all clustered within a 113-second window. But these are one campaign's habits, not validated detection science: a careful operator can strip delimiters and suppress comments, and legitimate automation (including defensive AI agents) produces similar formatting. Open questions: which signatures survive adversarial adaptation, what false-positive rate machine-consumption heuristics carry in DevOps-heavy environments, and whether EDR or SIEM vendors will ship LLM-operator detections before the pattern becomes commonplace.
- [ ] **What is the right detection layer for prompt-to-shell RCE in agent SDKs?** — CVE-2026-26030 and CVE-2026-25592 show that vulnerabilities in agent-framework primitives (`eval()` in default filters, `[KernelFunction]`-exposed file write) collapse the gap between model output and host code execution. Microsoft's recommendation is host-level process and file-system telemetry around agent processes, but most LLM observability stacks today still focus on prompt, response, token, and tool-call traces inside the SDK. The open question is who owns this layer — EDR, AI observability, or a new agent-runtime category — and how SOC-resident AI agents are expected to consume it.
