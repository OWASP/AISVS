# C4.2 Secure Build & Deployment Pipelines

[Back to C04 Index](C04-Infrastructure.md)

## Purpose

Ensure cryptographic integrity and supply chain security through reproducible builds and signed artifacts. AI systems depend on complex build pipelines that pull model weights, training data, framework dependencies, and custom operators from multiple sources. A compromised build pipeline can inject backdoors into models or serving infrastructure that persist undetected through deployment. This section ensures that every artifact is traceable, verified, and tamper-evident.

| **4.2.2** | **Verify that** build artifacts are cryptographically signed with build-origin metadata (source repository, build pipeline, commit hash) that can be independently verified. | 2 | Artifact tampering between build and deployment. Rogue builds injecting backdoored model-serving code. Lack of provenance making it impossible to trace a production artifact to its source. The **Ultralytics supply chain attack (Dec 2024)** demonstrated this threat: attackers exploited a `pull_request_target` vulnerability in GitHub Actions to inject cryptominer payloads into ultralytics v8.3.41 (60M+ PyPI downloads, 30K+ GitHub stars). More recently, the **Clinejection attack (Feb 2026)** showed how an AI-powered issue triage bot could be prompt-injected to poison GitHub Actions caches ("Cacheract"), ultimately publishing an unauthorized npm package to 5M+ Cline users for ~8 hours. The **hackerbot-claw campaign (Feb-Mar 2026)** demonstrated autonomous, bot-driven exploitation at scale: an automated agent systematically scanned public repositories for exploitable `pull_request_target` workflows, targeting Microsoft, Datadog, Aqua Security, and CNCF projects. It exfiltrated a write-privileged GitHub token from the awesome-go repository (140K+ stars) and stole a PAT from the Trivy project that was subsequently used to privatize the repository and delete release history. The **TeamPCP cascading campaign (March 2026)** showed how a single stolen credential chains across the ecosystem: Trivy GitHub Actions → Trivy Docker Hub images → Checkmarx KICS actions → LiteLLM PyPI packages → Telnyx PyPI packages, each compromised tool yielding tokens for the next target. The Cloud Security Alliance's [April 2026 analysis](https://labs.cloudsecurityalliance.org/research/csa-research-note-teampcp-supply-chain-cascade-20260402-csa/) confirmed that credential chaining across shared CI/CD infrastructure enables lateral movement across vendor boundaries -- mutable GitHub tags (76 of 77 Trivy tags silently repointed) are the root enabler. | Verify artifacts are signed using **Sigstore/cosign v3** (GA as of early 2026 -- standardized bundle format now default, mandatory `--certificate-identity` and `--certificate-oidc-issuer` for verification, trusted timestamping support), Notary, or GPG. Check that SLSA provenance metadata is attached. Validate signatures independently using public keys. Confirm signing occurs in a hardened build environment (not developer laptops). Verify that GitHub Actions workflows use network egress monitoring (e.g., [StepSecurity Harden-Runner](https://www.stepsecurity.io/)) to detect unauthorized outbound connections during builds, and that workflows are statically analyzed for dangerous `${{ }}` expression injection patterns. For ML model files specifically, verify adoption of the **[OpenSSF Model Signing (OMS) specification](https://openssf.org/blog/2025/06/25/an-introduction-to-the-openssf-model-signing-oms-specification/)**, which uses a detached signature format covering model weights, configuration files, tokenizers, and datasets as a single verifiable unit. OMS is PKI-agnostic, supporting private enterprise systems, self-signed certificates, bare keys, and keyless signing with public or private Sigstore instances. As of March 2026, OMS has production adoption: **NVIDIA signs all NGC catalog models**, and **Google's Kaggle** automatically signs models during upload using OIDC-backed keyless Sigstore certificates, with verification displayed directly in the model hub UI. For Python package publishing, verify adoption of **[PyPI Trusted Publishers](https://docs.pypi.org/trusted-publishers/)** (OIDC-based authentication eliminating long-lived API tokens) -- as of late 2025, over 50,000 projects use Trusted Publishing and ~25% of all PyPI uploads use this mechanism. The CSA's TeamPCP analysis specifically recommends Trusted Publishers as a primary mitigation against credential-theft supply chain attacks. | SLSA Level 2+ provenance is the target. **SLSA v1.2** (released Nov 2025) introduces a new **Source Track** alongside the existing Build Track, covering threats from source code authoring, review, and management -- Level 2 focuses on history and provenance, Level 3 on continuous enforcement of technical controls. This multi-track approach is directly relevant to AI build pipelines where source integrity (training scripts, configuration, data preprocessing code) is as critical as build integrity. Container images should use **cosign v3** with keyless signing via Fulcio/Rekor for transparency log -- v3 makes the standardized bundle format default (all verification material in one file), requires explicit identity flags for verification, and deprecates legacy `triangulate` and Rekor entry-type workflows. **OMS solves the large-model signing problem** by employing a detached format that avoids repackaging original content -- the manifest includes file listings with cryptographic hashes and optional annotations for domain-specific fields. The **[model-signing v1.0 CLI](https://blog.sigstore.dev/model-transparency-v1.0/)** (April 2025) makes model signing accessible via `pip install model-signing`, providing sign and verify commands backed by Sigstore with integrations for Kaggle, HuggingFace, TensorFlow, and PyTorch -- supporting keyless OIDC-based signing, PKI, and self-signed certificates. The [Coalition for Secure AI (CoSAI)](https://www.coalitionforsecureai.org/building-trust-in-ai-supply-chains-why-model-signing-is-critical-for-enterprise-security/) now positions model signing as critical enterprise infrastructure. Adapting SLSA to AI supply chains requires addressing the long, resource-intensive nature of model training; SLSA provenance for training data and RAG data remains challenging since dynamic data can render signatures outdated rapidly. Tools like cosign, Syft, Kyverno, and GUAC now make **SLSA Level 2 achievable in an afternoon** for teams with existing CI/CD pipelines, thanks to GitHub's built-in attestation support and maturity of cosign and slsa-github-generator. As of April 2026, **GitHub Artifact Attestations** can be enforced at the Kubernetes boundary via Kyverno's `ImageValidatingPolicy` or OPA Gatekeeper, validating SLSA v1.0/v2.0 provenance predicates, SPDX SBOMs, and OIDC issuer identity as part of admission control -- moving from "we generate security signals" to "we enforce them." Future GUAC integration is planned for generating AI-BOMs and inspecting ML supply chains during incident response. |
| **4.2.3** | **Verify that** build artifact signatures and build-origin metadata are validated at deployment admission, and unverified artifacts are rejected. | 2 | Deploying tampered or unauthorized artifacts to production. Bypassing the build pipeline by pushing images directly to registries. **Malicious pickle models on Hugging Face (Feb 2025)**: researchers found ML models using 7z compression (instead of standard ZIP) to evade Picklescan detection ("NullifAI" technique), containing reverse-shell payloads targeting Windows/Linux/macOS. Some models went undetected for over 8 months. **JFrog discovered 3 zero-day vulnerabilities in Picklescan** that allowed further evasion of Hugging Face's scanning. **[Model Namespace Reuse](https://unit42.paloaltonetworks.com/model-namespace-reuse/)** (Palo Alto Unit 42, Sept 2025): attackers can re-register deleted Hugging Face account namespaces and upload backdoored models under the original trusted name. Orphaned model references were found in Google Vertex AI Model Garden and Microsoft Azure AI Foundry, achieving remote code execution on cloud inference endpoints. **LiteLLM `.pth` persistence (March 2026)**: TeamPCP's malicious `litellm==1.82.8` used a Python `.pth` file that auto-executes on every Python interpreter startup without any import, bypassing static analysis -- a technique that standard admission scanning would miss entirely. | Configure admission controllers (Kyverno, OPA Gatekeeper, Sigstore Policy Controller) to enforce signature verification. Test by attempting to deploy an unsigned artifact and confirming rejection. Verify admission policies cover all deployment paths (Kubernetes, serverless, edge). On **Kubernetes 1.35+**, leverage the new `imagePullCredentialsVerificationPolicy` (beta) which re-verifies registry credentials even for cached images, preventing unauthorized image reuse. The new PodCertificateRequest API simplifies workload mTLS certificate provisioning for pod-to-pod supply chain verification. For model serving frameworks, verify that **Seldon Core** (with Istio integration for automatic mTLS and JWT-based RequestAuthentication), **BentoML** (structured APIs with Docker containerization), **KServe**, and **NVIDIA Triton Inference Server** enforce artifact validation before model loading. Verify that all model references in code, notebooks, and configuration files are pinned to specific commit hashes (e.g., `revision="commit_hash"`) rather than fetched by name alone -- the Model Namespace Reuse attack demonstrated that name-based trust is exploitable across Google Vertex AI and Azure AI Foundry. For organizations managing model weights at scale, consider using **Harbor** as an AI model registry: models packaged as OCI artifacts with immutable tags, SHA256 digests, Cosign/Notation signature verification enforced before distribution, and RBAC separating push (training) from pull (inference) roles. For Python dependencies, enforce `pip install --require-hashes` in CI/CD to reject packages that don't match pre-computed digests -- the CSA's April 2026 TeamPCP analysis identifies this as a key mitigation against credential-theft supply chain attacks targeting PyPI. | Admission control for model weight files is less standardized than container images. Organizations serving models from object storage (S3, GCS) need custom validation hooks. Consider integrating signature checks into model loading code as defense-in-depth. The Hugging Face pickle incidents demonstrate that format-level scanning (Picklescan) is insufficient alone -- deployment admission should combine signature verification, format validation, and sandboxed model loading. The **OpenSSF Secure MLOps whitepaper (Aug 2025)** recommends extending DevOps admission controls to cover datasets and models using SLSA, Sigstore, and OpenSSF Scorecard. The **[NIST IR 8596 Cyber AI Profile](https://nvlpubs.nist.gov/nistpubs/ir/2025/NIST.IR.8596.iprd.pdf)** (preliminary draft, Dec 2025) maps CSF 2.0 to AI systems, recommending that organizations inventory AI models, APIs, keys, agents, and data alongside their permissions, and establish threat detection processes for supplier-provided AI models -- reinforcing the need for admission control at every boundary where AI artifacts enter the deployment pipeline. The **CanisterWorm cascade** (March 2026) demonstrated a new escalation pattern: credential theft from a compromised CI/CD tool (Trivy) spawned a self-propagating worm across 47 npm packages, using ICP blockchain C2 to resist takedown -- admission control that validates package signatures would have blocked the unauthorized publications. The **LiteLLM `.pth` persistence technique** (March 2026) bypasses standard import-time scanning entirely by injecting a Python `.pth` file that auto-executes on interpreter startup -- admission control must extend beyond format scanning to include `.pth` file detection and Python environment integrity verification. The **Telnyx WAV steganography** (March 2026) concealed malware inside audio files, demonstrating that file format diversity in attack payloads continues to expand. |
| **4.2.4** | **Verify that** builds are reproducible, producing identical output from identical source inputs, enabling independent verification of build integrity. | 3 | Compromised build infrastructure silently inserting backdoors (Ken Thompson attack). Inability to verify that a production artifact matches its claimed source. The 2025 State of Pipeline Security report documented "Pipeline Parasitism" as a systemic pattern: attackers living off CI/CD using shell injections, over-privileged tokens, invisible Unicode payloads, blockchain C2, and wiper failsafes. The **Trivy/TeamPCP supply chain attack (March 2026)** escalated the threat: attackers force-pushed 75 of 76 version tags in the `aquasecurity/trivy-action` GitHub Action, injecting the "TeamPCP Cloud stealer" that dumped Runner.Worker process memory, harvested SSH/cloud/K8s secrets, and exfiltrated them via AES-256+RSA-4096 encryption -- all within a security scanning tool trusted by thousands of CI/CD pipelines. Root cause: incomplete credential rotation from an earlier incident left a valid token accessible. The **CodeBreach vulnerability (disclosed Jan 2026)** revealed that AWS CodeBuild webhook actor-ID filters for four AWS repositories (including aws-sdk-js-v3) used a regex without start/end anchors, allowing any GitHub user with a superstring user ID to trigger builds and obtain admin-privileged PATs. Wiz estimated a successful exploit could have enabled platform-wide compromise of AWS accounts via the SDK. AWS fixed the issue in Sept 2025 and added a global Pull Request Comment Approval build gate. | Rebuild from the same source commit and compare output hashes. Verify build environment is hermetic (pinned toolchains, no network access during build). Check that non-deterministic elements (timestamps, random seeds) are controlled. For ML serving infrastructure, verify that container images and serving code produce reproducible builds even if model weights themselves are non-deterministic. Validate that all CI/CD workflows use pinned action versions (not mutable tags) and restrict `pull_request_target` trigger usage. As of August 2025, GitHub Actions organization policies can enforce SHA pinning at the platform level -- administrators should enable this so that any workflow referencing an action by mutable tag is automatically rejected. After any credential compromise, perform **atomic credential rotation** -- revoke all tokens simultaneously before issuing replacements to prevent the incomplete-rotation attack pattern seen in both the Clinejection and Trivy incidents. | Full reproducibility for ML artifacts is extremely challenging. Model training is inherently non-deterministic on GPUs (floating-point non-associativity, cuDNN algorithm selection). This requirement is more practical for serving infrastructure builds than for model weights. PyTorch `torch.use_deterministic_algorithms(True)` helps but has significant performance penalties and doesn't cover all operations. The Trivy/TeamPCP and Clinejection attacks (both early 2026) reinforce that even "trusted" CI/CD pipelines and security tools can be weaponized -- hermetic builds with no network access and pinned dependencies are necessary but not sufficient without hardened workflow trigger configurations, atomic credential rotation, and GitHub Actions tag immutability enforcement via [StepSecurity Harden-Runner](https://www.stepsecurity.io/). The CodeBreach incident (Jan 2026) adds another layer: even managed CI/CD services (AWS CodeBuild) can have webhook filter flaws that bypass build authorization entirely. Recent reproducibility research found that only 2.7% of Dockerfiles produce bitwise-reproducible images without infrastructure modifications, rising to 21.3% with configuration changes -- underscoring the practical difficulty of this requirement. |
| **4.2.5** | **Verify that** automated builds produce a software bill of materials (SBOM) listing all components, versions, and their origins. | 1 | Untracked dependencies entering production without visibility into what components are actually deployed. As of 2026, only 21% of organizations have complete dependency visibility (Datadog DevSecOps 2026), and 62% of security practitioners cannot tell where LLMs are in use across their organization (VentureBeat, 2026). The Sonatype 2026 report found open-source malware grew 75% year-over-year to 1.2M+ malicious packages across 9.8T yearly downloads, with ~27% of AI-generated dependency suggestions being hallucinations -- meaning AI coding assistants actively introduce phantom dependencies that no scanner tracks. The **Axios npm supply chain attack (March 31, 2026)** demonstrated the downstream consequences: OpenAI's macOS code-signing workflow pulled compromised `axios@1.14.1` because it used a floating tag rather than a pinned commit hash, exposing signing certificates for ChatGPT Desktop, Codex, and Atlas. Without an SBOM correlating the exact dependency versions consumed at build time, tracing blast radius during such incidents requires ad hoc forensics. The **Mercor incident (April 2026)** showed the same pattern at scale: the company was among thousands affected by the LiteLLM compromise, with no fast way to determine exposure without a comprehensive component inventory. MITRE ATLAS [AML.T0010](https://atlas.mitre.org/techniques/AML.T0010) (ML Supply Chain Compromise). | Confirm that CI/CD pipelines generate an SBOM in CycloneDX or SPDX format on every build. Verify the SBOM includes all dependency layers: ML framework versions (PyTorch, TensorFlow, JAX), CUDA/cuDNN runtime versions, Python packages with pinned hashes, native libraries, and container base image components. For AI-specific coverage, verify the SBOM or a supplementary AI-BOM captures model architecture metadata, training dataset references, fine-tuning lineage, and adapter/LoRA provenance. Validate SBOM generation using tools like **[Syft](https://github.com/anchore/syft)** (v1.42+, supports SPDX and CycloneDX), **[cdxgen](https://github.com/cdxgen/cdxgen)** (v12+, produces SBOM/CBOM/OBOM/SaaSBOM/CDXA with reachability analysis), or **[Trusera AI-BOM](https://github.com/Trusera/ai-bom)** (auto-discovers LLM providers, agent frameworks, and model references). Feed generated SBOMs into **[OWASP Dependency-Track](https://dependencytrack.org/)** or **[GUAC](https://guac.sh/)** for continuous vulnerability correlation against NVD, GitHub Advisories, OSV, and Sonatype OSS Index. Verify that SBOMs are cryptographically signed alongside build artifacts (4.2.2) and stored as attestations linked to the build provenance chain. Confirm SBOM completeness by cross-referencing against the actual runtime dependency tree (e.g., `pip freeze` output vs. SBOM contents). | SBOM generation tooling for traditional software dependencies is mature, but AI-specific component coverage remains a gap. CycloneDX 1.7 ML-BOM fields and SPDX 3.0 AI/Dataset profiles provide format support, but as of early 2026 fewer than half of open-source projects include SBOMs in version control or release artifacts, and most generated SBOMs lack recommended data fields like hashes or supplier information. A June 2025 survey found 48% of security professionals admit their organizations are falling behind on SBOM requirements, with ML-BOM adoption significantly lower. **Regulatory pressure is now concrete**: the EU Cyber Resilience Act mandates SBOM generation for all products with digital elements sold in the EU, with vulnerability reporting obligations effective September 2026 and full compliance by December 2027 -- organizations that lack automated SBOM generation in their build pipelines cannot comply. The knowledge/expertise gap is the top adoption barrier (cited by 69% of developers). Key remaining challenges: (1) representing model weights, training data provenance, and fine-tuning lineage in standard SBOM formats, (2) capturing AI-generated code and hallucinated dependency suggestions in component inventories, (3) maintaining SBOM freshness across long-running training jobs with dynamic data dependencies. The **Brainjacked research (Mitiga, April 2026)** demonstrated how shared IAM credentials between web applications and ML training infrastructure can allow path traversal attacks to poison training data in production S3 buckets, triggering automated SageMaker retraining -- SBOM coverage of the training pipeline's data dependencies could have flagged the unauthorized dataset modifications. |

## Implementation Maturity

| Requirement | Maturity | Notes |
|-------------|:---:|-------|
| 4.2.1 Automated builds with SBOM | Mature | CI/CD automation is standard. CycloneDX 1.7 ML-BOM and SPDX 3.0.1 AI/Dataset profiles provide AI-specific SBOM formats. OWASP AIBOM Generator produces CycloneDX output from HF model references. The AIBoMGen platform (Jan 2026) demonstrates automated AIBOM generation during training with in-toto attestations. The EU Cyber Resilience Act now legally mandates SBOM generation (vulnerability reporting from Sept 2026, full compliance by Dec 2027). Gap: only 21% of orgs have complete dependency visibility (Datadog DevSecOps 2026), and the CanisterWorm cascade (March 2026) showed that a single compromised CI/CD tool can spawn self-propagating malware across 47+ npm packages -- SBOM tracking is essential but insufficient without signature verification. |
| 4.2.2 Cryptographic signing with provenance | Mature | **Cosign v3** is now GA with the standardized bundle format as default, mandatory identity verification flags, and trusted timestamping support -- a notable hardening over v2.x defaults. OpenSSF Model Signing (OMS) has real-world adoption: NVIDIA signs all NGC catalog models and Google's Kaggle signs models automatically during upload using OIDC-backed keyless Sigstore. The **model-signing v1.0 CLI** (April 2025) makes this accessible via `pip install model-signing` with integrations for Kaggle, HuggingFace, TensorFlow, and PyTorch. The Coalition for Secure AI (CoSAI) now positions model signing as critical enterprise infrastructure. **SLSA v1.2** (Nov 2025) adds a Source Track alongside the Build Track, extending provenance to source code authoring and review. SLSA Level 2 is achievable in an afternoon with GitHub's built-in attestation support and cosign + slsa-github-generator. **PyPI Trusted Publishers** (OIDC-based, eliminating long-lived API tokens) now cover 50K+ projects and ~25% of all uploads -- the LiteLLM project [filed issue #24542](https://github.com/BerriAI/litellm/issues/24542) post-compromise to migrate to Trusted Publishers, and the Cloud Security Alliance's April 2026 analysis recommends this as a primary mitigation. Gap: SLSA provenance for training data and long-running training jobs remains challenging. The **TeamPCP cascading campaign** demonstrates in devastating detail why signed provenance is essential: a single stolen Trivy credential chained across Checkmarx KICS (35 tags hijacked), LiteLLM (malicious PyPI packages in 36% of cloud environments), and Telnyx (WAV steganography evasion), with each compromise yielding tokens for the next target. Without signed provenance, each unauthorized publication was indistinguishable from legitimate releases. |
| 4.2.3 Admission control rejecting unsigned artifacts | Maturing | **Kyverno reached CNCF Graduated status (March 16, 2026)**, a significant maturity milestone for policy-as-code admission control; Kyverno 1.16 introduces CEL-based policies alongside existing YAML-native definitions. OPA Gatekeeper and Sigstore Policy Controller also enforce container image signature verification. Kubernetes 1.35 adds `imagePullCredentialsVerificationPolicy` (beta) for re-verifying registry credentials on cached images, and PodCertificateRequest API for workload mTLS. **Kubernetes 1.36 (April 22, 2026)** graduates **MutatingAdmissionPolicy to GA** -- admission mutations can now be defined as native Kubernetes objects using CEL expressions, eliminating the need for separate webhook servers, TLS certificates, and availability concerns. K8s 1.36 also graduates **external ServiceAccount token signing** to stable, allowing delegation to cloud KMS or HSMs for enterprise-grade signing infrastructure. Harbor can now serve as an AI model registry with OCI artifact packaging and Cosign/Notation signature enforcement. **[Kubescape 4.0](https://kubescape.io/)** (announced KubeCon EU 2026) introduces AI agent scanning with 15 controls covering 42 security-critical configuration points in KAgent CRDs, and promotes runtime threat detection (CEL-based detection rules against Application Profiles) to GA. NIST IR 8596 Cyber AI Profile (draft Dec 2025) formalizes admission control as a CSF 2.0 expectation for AI systems. As of April 2026, **GitHub Artifact Attestations** can be enforced at the Kubernetes boundary via Kyverno's `ImageValidatingPolicy` or OPA Gatekeeper, validating SLSA provenance predicates, SPDX SBOMs, and OIDC issuer identity during pod admission -- closing the gap between attestation generation and runtime enforcement. Gap: admission control for model weight files loaded from object storage (S3, GCS, HF Hub) remains less standardized. The **Model Namespace Reuse** attack (Unit 42, Sept 2025) demonstrated that name-based model trust without commit pinning is exploitable across Google Vertex AI and Azure AI Foundry. The TeamPCP campaign expansion to **Docker Hub**, **PyPI** (LiteLLM with `.pth` auto-execution, Telnyx with WAV steganography), and **OpenVSX** (Checkmarx extensions) shows that admission control must cover all distribution channels. |
| 4.2.5 Automated SBOM generation | Maturing | SBOM generation tooling for traditional dependencies is mature: Syft (v1.42+), cdxgen (v12+), and CycloneDX CLI integrate directly into CI/CD pipelines. 76% of US/UK enterprise organizations maintain SBOMs (Sonatype), and 92% of large enterprises have implemented or plan to adopt SBOMs. However, adoption quality lags: fewer than half of open-source projects include SBOMs in releases, and most lack recommended fields like hashes or supplier info. AI-specific SBOM coverage (model weights, training data, fine-tuning lineage) remains a significant gap -- 48% of security professionals admit falling behind on SBOM requirements (June 2025), with ML-BOM adoption much lower. The EU CRA (vulnerability reporting from Sept 2026, full compliance Dec 2027) is now the strongest forcing function. OWASP Dependency-Track and GUAC provide mature SBOM consumption and continuous vulnerability correlation. **NIST SP 1800-44 Volume B** (public comment March-April 2026) provides a practical DevSecOps reference architecture with 14 technology partners, including SBOM generation as a core practice across Plan-Develop-Build-Test-Release-Deploy-Operate phases. The Contagious Interview campaign (April 2026) underscores a new gap: SBOM tooling must cover polyglot dependency ecosystems (npm, PyPI, Go, Rust, Packagist) simultaneously, as 1,700+ malicious packages were deployed across all five registries by a single threat actor cluster. |
| 4.2.4 Reproducible builds | Emerging | Practical for serving infrastructure container images, though research shows only 2.7% of Dockerfiles produce bitwise-reproducible output without infrastructure changes (21.3% with configuration). Extremely challenging for GPU-trained model weights due to floating-point non-determinism. `torch.use_deterministic_algorithms(True)` helps but has performance penalties. As of April 2026, the **TeamPCP/UNC6780 campaign** has escalated further beyond the March 2026 timeline: Unit 42 confirmed 300+ GB of data and 500,000 credentials exfiltrated across ~500,000 machines, and **Cisco disclosed that stolen credentials led to the cloning of 300+ private GitHub repositories** containing unreleased AI products and customer data from banks and US government agencies. Multiple threat groups (LAPSUS$, ShinyHunters, Vect ransomware) are now actively monetizing the stolen credentials. Concurrently, the **Axios npm attack** (March 31) was attributed by Google GTIG to **DPRK state-sponsored group UNC1069 (Sapphire Sleet)**, deploying the WAVESHAPER.V2 RAT across Windows/macOS/Linux -- marking the first nation-state attribution for a major npm supply chain compromise. The convergence of financially motivated (UNC6780/TeamPCP) and state-sponsored (UNC1069/DPRK) supply chain campaigns in the same month underscores that hermetic builds, pinned dependencies, tag immutability, network egress monitoring, atomic credential rotation, `--ignore-scripts` flags, `minimumReleaseAge` policies, and isolated per-project publication credentials are all necessary layers. GitHub Actions organization-level SHA pinning enforcement (Aug 2025) would have mitigated the initial tag hijacking entry point. **Kubernetes 1.36** (April 22, 2026) removes the gitRepo volume plugin entirely (KEP-5040), closing a long-standing root-code-execution risk on nodes. |

## Open Research Questions

- [ ] How should SBOMs represent model weights, training data dependencies, and fine-tuning lineage? CycloneDX 1.7 ML-BOM and OWASP AIBOM are converging but not yet comprehensive -- gaps remain for RAG data sources, adapter/LoRA lineage, and quantization provenance.
- [ ] What is the practical path to reproducible builds for GPU-trained models given floating-point non-determinism? Current approaches (deterministic algorithms, fixed seeds) incur significant performance penalties and do not cover all operations.
- [ ] How can SLSA provenance be extended to cover the full ML pipeline (data ingestion, preprocessing, training, evaluation, packaging)? SLSA v1.1 does not yet address dynamic data or long-running training jobs natively.
- [ ] What signing infrastructure works well for multi-gigabyte model weight files stored in object storage? The OpenSSF OMS specification (2025) addresses this with detached signatures, and as of early 2026 NVIDIA NGC and Google Kaggle have production deployments -- but broader ecosystem adoption beyond these early movers needs tracking.
- [ ] How can admission control be extended beyond container images to validate model files loaded from object storage (S3, GCS, Hugging Face Hub) at serving time? Current Kubernetes admission controllers only cover container image deployment.
- [ ] Given the demonstrated insufficiency of format-based scanning (Picklescan evasions), what layered defenses should be standard for model artifact validation before deployment?
- [ ] How should organizations secure AI-powered CI/CD automation (issue triage bots, code review agents) against prompt injection attacks that can cascade into supply chain compromises? The Clinejection attack (Feb 2026) demonstrated this risk concretely, but defensive patterns are still ad hoc.
- [ ] What is the right approach to credential rotation in CI/CD pipelines to prevent the incomplete-rotation attack pattern seen in the Clinejection and Trivy/TeamPCP incidents? Atomic rotation, short-lived OIDC tokens, and separate publication scopes are emerging best practices but lack standardized implementation guidance.
- [ ] How should organizations defend against autonomous bot-driven CI/CD exploitation (e.g., hackerbot-claw)? Network egress monitoring and workflow static analysis help, but the scale and speed of automated scanning suggests that reactive defenses are insufficient -- what proactive architectural patterns (isolated build identities, ephemeral runners, mandatory PR approval gates) should be baseline?
- [ ] As managed CI/CD services (AWS CodeBuild, Azure DevOps, GCP Cloud Build) are shown to have their own authorization vulnerabilities (CodeBreach), how should defense-in-depth strategies account for trusted-platform compromise? Multi-vendor signing and independent verification may be needed even within a single cloud provider's ecosystem.
- [ ] With OMB M-26-05 (Jan 2026) rescinding the standardized federal software attestation Common Form in favor of agency-specific risk-based approaches, how will this fragmentation affect AI/ML supply chain security requirements for government contractors? Organizations may need to track divergent requirements across agencies rather than a single SBOM/attestation standard.
- [ ] How should AIBOMs capture the provenance of AI-generated code and AI-suggested dependencies, given that Sonatype's 2026 data shows ~27% of AI dependency suggestions are hallucinations? Standard SBOM tools cannot currently distinguish human-selected from AI-suggested components.
- [ ] How should supply chain defenses adapt to decentralized C2 infrastructure (ICP blockchain canisters, IPFS, etc.) that resists traditional takedown? The CanisterWorm (March 2026) demonstrated that self-propagating malware using blockchain dead-drop resolvers cannot be disrupted by domain seizure or server takedowns, requiring a shift from reactive blocklisting to proactive signature verification and anomaly detection at build-time.
- [ ] How should SBOM generation tools handle the Axios-style attack pattern where a floating tag pulls in a compromised version between SBOM generation runs? Real-time SBOM validation at install time (not just build time) and `minimumReleaseAge` policies for new package versions are emerging mitigations, but no standard CI/CD configuration enforces both simultaneously. The OpenAI incident (March 2026) showed that even leading AI companies can miss this gap.
- [ ] How should CI/CD security models adapt to the convergence of nation-state (UNC1069/DPRK targeting Axios) and financially motivated (UNC6780/TeamPCP targeting Trivy/LiteLLM) supply chain campaigns operating simultaneously in the same ecosystem? The March-April 2026 period saw both threat categories active against npm, PyPI, and GitHub Actions concurrently, with shared infrastructure (stolen credentials from one campaign enabling the other) blurring the line between state-sponsored and criminal operations. Traditional threat modeling that separates these categories may underestimate the compounded risk.
- [ ] With PyPI Trusted Publishers now covering ~25% of uploads and 50K+ projects, what is the adoption trajectory for high-value AI/ML libraries specifically? The TeamPCP campaign demonstrated that credential-theft attacks against PyPI maintainers are the primary vector for AI supply chain compromise, yet many critical AI libraries (including LiteLLM pre-compromise) still relied on long-lived API tokens. How can the ecosystem accelerate migration, and should package registries enforce Trusted Publishers for packages exceeding a download threshold?
- [ ] As supply chain attacks demonstrate sustained multi-phase campaigns (hackerbot-claw scanning -> TeamPCP credential theft -> Trivy tag hijacking -> Docker Hub image poisoning -> Checkmarx KICS compromise -> LiteLLM PyPI backdoor -> Telnyx PyPI backdoor -> IDE extension weaponization -> CanisterWorm npm propagation over February-March 2026), what organizational structures and incident response playbooks are needed to track and respond to attack chains that span CI/CD, container registries, PyPI, IDE extensions, and developer environments simultaneously? The TeamPCP campaign is now the longest-documented cascading supply chain attack, with at least 7 distinct compromise phases across 5 different distribution channels.
- [ ] How should Python package registries and consumers defend against `.pth` file persistence, where malicious code auto-executes on every Python interpreter startup without any import? TeamPCP's LiteLLM 1.82.8 demonstrated this technique -- standard static analysis and import-time scanning miss it entirely. Should `.pth` file auditing become a standard CI/CD security check?
- [ ] How should model hubs (Hugging Face, Kaggle, NGC) handle namespace lifecycle to prevent Model Namespace Reuse attacks? Namespace reservation, tombstoning deleted accounts, and mandatory commit-hash pinning in downstream integrations are all proposed mitigations, but no hub has implemented comprehensive protections yet. Cloud platforms that auto-fetch models by name (Google Vertex AI Model Garden, Azure AI Foundry) remain particularly exposed.
- [ ] How should SBOM tooling and admission control adapt to cross-ecosystem supply chain campaigns like Contagious Interview (April 2026), where a single threat actor deploys 1,700+ malicious packages simultaneously across npm, PyPI, Go, Rust, and Packagist? Current SBOM generation and dependency scanning typically cover one or two ecosystems per pipeline, but polyglot AI/ML projects often pull dependencies from all five. The runtime-trigger evasion technique (malicious code embedded in normal function calls rather than install hooks) bypasses `--ignore-scripts` mitigations and requires behavioral analysis or sandboxed execution to detect.
- [ ] How should CI/CD plugin security be assessed and enforced given the structural vulnerabilities in plugin-centric build systems? JetBrains' March 2026 analysis documents that abandoned plugins can be silently taken over, many plugins lack formal security oversight, and pipelines often auto-update plugins without review. Jenkins alone had 70+ security flaws in 2025, with 45,000+ servers still vulnerable to CVE-2024-23897 months after patching. Should organizations maintain a plugin SBOM and enforce plugin signing analogous to container image signing?
