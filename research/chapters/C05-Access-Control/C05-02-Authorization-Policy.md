# C5.2: AI Resource Authorization & Classification

> [Back to C05 Index](C05-Access-Control.md)
> **Last Researched:** 2026-05-13 (7th pass, updated with May 2026 Spring AI vector-store and memory advisories, Milvus management-port exposure, AuthZEN 1.0, Bedrock AgentCore Policy GA, and current AI asset-governance tooling)

## Purpose

Implement access controls for all AI resources with explicit permission models and audit trails. AI systems introduce authorization challenges that go beyond traditional application security: vector databases, embedding indices, model registries, prompt caches, agent tool schemas, and derived outputs all need explicit access control. Data classification labels must propagate through AI pipelines to derived resources, and authorization decisions should be externalized to dedicated policy decision points where possible to prevent scattered, inconsistent logic across AI services.

> **Scope note:** Controls in this section govern authorization policy definition, enforcement, classification, and audit for AI resources. Identity and authentication are addressed in C5.1; query-time enforcement at the data layer is in C5.3; output-level filtering is in C5.4. Per ASVS v5 14.1.1, all sensitive data should be identified and classified into protection levels; C5.2.3 and C5.2.4 extend this to AI-specific data types and derived resources.

## Requirements

| # | Requirement | Level | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|-----------------|----------------------|--------------|
| **5.2.1** | **Verify that** every AI resource (datasets, models, endpoints, vector collections, embedding indices, compute instances) enforces access controls (e.g., RBAC, ABAC) with explicit allow-lists and default-deny policies. | 1 | Unauthorized access to training data, model weights, inference endpoints, vector collections, prompt caches, or agent tools. OWASP's Agentic Top 10 maps this directly to ASI03 Identity & Privilege Abuse, where agents inherit, reuse, or misuse privileges beyond the user they represent. MITRE's February 2026 OpenClaw investigation showed the same pattern in practice: exposed agent control interfaces let researchers read connected-application credentials and invoke agent skills to reach root in the container, and the OpenClaw case studies `AML.CS0048` through `AML.CS0051` repeatedly point back to missing tool and credential authorization boundaries. EchoLeak, tracked as CVE-2025-32711, is another useful test case: an AI system with broad retrieval authority can become a data-disclosure path even when the attacker never directly authenticates to the underlying documents. | Build an inventory of datasets, model artifacts, endpoints, vector databases, embedding indices, prompt/template stores, agent tools, and compute targets. For each resource, test anonymous access, low-privilege user access, stale API keys, service-account access, and agent-mediated access. Confirm explicit grants are required, deny is the fallback, and audit logs show the principal, action, resource, decision, and policy version. For vector stores, inspect native controls: Pinecone project/data-plane roles and API-key permissions; Weaviate RBAC for collections, objects, tenants, and backups; Qdrant API keys, read-only keys, JWT RBAC, and managed-cloud granular API keys. For agent tool access, test through the policy enforcement point rather than only through the application UI. | Native authorization depth still varies by AI resource. Pinecone and Weaviate now have useful RBAC primitives, while Qdrant's own documentation still warns that self-hosted instances are open by default unless authentication, binding, and TLS are configured. Qdrant managed-cloud granular database keys are available for clusters on v1.11.0 and later, but older admin keys lack granular access. When vector databases cannot enforce the needed row, tenant, or purpose boundary themselves, the application must store labels in metadata and enforce them consistently through a PDP or gateway. Amazon Bedrock AgentCore Policy reached GA on March 3, 2026 with an external Cedar policy engine for agent-tool traffic; Cerbos, OPA, Cedar/AWS Verified Permissions, and AuthZEN-compatible PDPs are practical choices for centralizing these checks, but they only help if every resource path is routed through the PEP. |
| **5.2.2** | **Verify that** privileged access to model weights, training pipelines, and production AI configuration is provisioned on a just-in-time basis with a defined maximum session duration and automatic expiry, and permanent standing privileged access to these resources is not permitted. | 2 | Persistent privileged access to model weights enables insider or account-takeover exfiltration. Always-on access to training pipelines enables poisoning, unauthorized fine-tuning, or production configuration changes that persist after the original task is finished. Agentic systems make the risk sharper because a compromised agent runtime can use any standing service-account privilege it holds; MITRE's OpenClaw report calls out credential harvesting from agent configuration and unrestricted tool invocation as recurring attack paths. | Enumerate every principal with write, admin, export, deploy, or key-management access to model registries, training pipelines, feature stores, production configuration, prompt-template repositories, and vector database administration. Confirm privileged roles are eligible rather than permanently active, require MFA or workload identity proof, have a maximum activation duration, and expire automatically. Test that access fails after expiry and that refresh tokens or long-lived API keys cannot silently recreate the session. Review Entra PIM, HashiCorp Boundary plus Vault dynamic credentials, CyberArk, BeyondTrust, Teleport, or equivalent PAM configuration for approval, session recording, and audit export. For agents, verify delegated access uses a session-bound token tied to the initiating user or task instead of a reusable privileged service account. | Mature PAM tools handle human JIT access better than agent or pipeline identities. Microsoft Entra PIM supports time-based and approval-based activation, and Boundary can broker sessions with fixed limits and Vault-backed dynamic credentials, but AI training and deployment systems often still rely on IAM roles, static cloud keys, or shared CI secrets. Break-glass access should be time-bound, ticketed, and reviewed. A useful pattern is token exchange into a short-lived capability for a single model export, deployment, or tool call, with no persistent credential material stored in the agent runtime. |
| **5.2.3** | **Verify that** a documented data classification taxonomy covering AI-specific data types (embeddings, model weights, prompt templates, RAG context assemblies, fine-tuning datasets, agent tool schemas) is defined, and that AI assets are labeled in accordance with this taxonomy. | 2 | AI assets are often misclassified because they do not look like ordinary documents or databases. Embeddings can encode sensitive source data; model weights can expose proprietary behavior or memorized data; prompt templates can contain system instructions and guardrails; RAG context assemblies and fine-tuning sets can combine records with different handling rules; agent tool schemas can reveal high-impact capabilities. Without an AI-aware taxonomy, these assets are stored, shared, or indexed at protection levels below their source data. | Request the classification taxonomy and verify that it explicitly covers embeddings, vector collections, model weights, adapters, evaluation sets, fine-tuning datasets, prompt templates, system prompts, RAG source corpora, RAG context assemblies, prompt and response logs, agent tool schemas, tool credentials, and API keys. Sample assets from each category and confirm labels are applied in metadata, catalog entries, storage tags, and policy inputs. Cross-check the inventory against NIST IR 8596's Cyber AI Profile, which calls out AI system components and data flows as cybersecurity-relevant assets, and against C5.2.4 for derived-label propagation. | General data-governance platforms can help, but none should be assumed to classify every AI artifact automatically. Microsoft Purview supports DSPM for AI, data classification, sensitivity labels, DLP, audit, eDiscovery, and compliance coverage for supported AI interactions; Google Sensitive Data Protection can discover and profile sensitive data across supported stores; Atlan and Collibra can attach governance metadata and lineage to common data assets. The remaining gap is AI-specific coverage: derived embeddings, prompt caches, agent tool schemas, and runtime context bundles usually require custom metadata fields, custom classifiers, ingestion hooks, and periodic sampling. |
| **5.2.4** | **Verify that** data classification labels (PII, PHI, proprietary, etc.) automatically propagate to derived resources (embeddings, prompt caches, model outputs). | 3 | Classification loss during transformation lets sensitive data move into embeddings, prompt caches, model outputs, evaluation logs, or analytics tables without the controls applied to the source. This creates unauthorized retrieval, cross-tenant leakage, data-subject-rights failures, and retention-policy failures because derived artifacts look less sensitive than the records used to create them. | Select source records labeled PII, PHI, confidential, export-controlled, or proprietary and trace them through ingestion, chunking, embedding generation, vector storage, prompt assembly, response caching, evaluation logging, and output persistence. Verify the derived record carries the source label, lineage pointer, tenant or purpose tag, retention rule, and access policy input. Attempt retrieval as a user or agent that lacks the source entitlement and confirm both pre-retrieval filtering and post-retrieval enforcement deny access. Review pipeline tests that fail builds when label metadata is missing from embeddings, caches, or outputs. | Tooling is improving, but propagation still depends heavily on pipeline design. Atlan can propagate tags to downstream assets through lineage when enabled, and Microsoft Purview sensitivity labels persist with supported files and are recognized by Microsoft 365 Copilot and agents when those labels and usage rights are available. Those controls do not automatically label every vector, prompt cache entry, model output, or third-party retrieval store. Vector stores can only enforce classification if labels are carried into metadata and every query path includes the correct filters or PDP decision. Treat missing labels on derived artifacts as a security defect, not as incomplete documentation. |

## Current Research Notes

- As of May 2026, vector-store authorization needs explicit testing below the application UI. Milvus CVE-2026-26190 exposed unauthenticated business APIs on a management port, while Spring AI CVE-2026-22729 and CVE-2026-22730 showed that metadata filters used for tenant and role isolation can be bypassed when filter expressions are built from unescaped input.
- Spring AI 1.0.7 and 1.1.6 added fresh security fixes on May 8, 2026. CVE-2026-41705 is especially relevant to C5.2.1 because unsanitized document IDs in `MilvusVectorStore#doDelete(List)` could alter vector-store deletion filters; CVE-2026-41712 is relevant to C5.2.4 because an implicit default conversation ID could mix chat memory across users unless applications pass an explicit conversation identifier.
- AuthZEN 1.0, published April 29, 2026, gives teams a practical interoperability target for PDP and PEP integration: request attributes should include subject, resource, action, and context, and AI gateways should preserve enough context for policy decisions to be replayed during audit.
- Asset-governance products are starting to cover AI-specific resources, but coverage is uneven. Microsoft Purview supports classification, sensitivity labels, audit, and DSPM for Entra-registered AI apps; Collibra's AI Governance operating model adds assets such as AI agents, AI endpoints, model deployments, and agent-governance records. Neither removes the need to label embeddings, prompt caches, memory rows, and vector metadata at ingestion time.
