# C1.5: Data Lineage and Traceability

> [Back to C01 Index](C01-Training-Data.md)
> **Last Researched:** 2026-03-29

## Purpose

When a model produces unexpected or harmful outputs, the first question in incident response is "what data trained this model, and where did it come from?" Without complete lineage records — tracking every transformation, augmentation, merge, and synthetic data generation step — that question is unanswerable. Lineage is also a regulatory survival requirement: the FTC's algorithmic disgorgement pattern (6 enforcement actions from 2019-2025 requiring deletion of models trained on improperly collected data) means organizations that cannot reconstruct a dataset's full journey from source to model input face existential regulatory risk.

This section requires recording the full provenance chain of each dataset, storing lineage records immutably for audit, and extending lineage tracking to cover synthetic data so that it remains clearly distinguishable from real data throughout the pipeline. As organizations increasingly generate synthetic training data from other models, the risk of inheriting and amplifying upstream poisoning (demonstrated by the Virus Infection Attack, NeurIPS 2025) makes synthetic data lineage tracking especially urgent.

| **1.5.1** | **Verify that** the lineage of each dataset and its components, including all transformations, augmentations, and merges, is recorded and can be reconstructed. | 1 | Inability to trace poisoned data back through the pipeline; undetectable introduction of malicious transformations; failure to reproduce training conditions for incident investigation. | Review lineage tracking system (Apache Atlas, DataHub, MLflow, OpenLineage/Marquez, custom metadata stores). Verify that each transformation step is logged with input/output dataset references. Test reconstruction of a sample dataset's full history. The Atlas framework (Feb 2025) demonstrated fully attestable ML pipelines with <8% overhead but remains a research prototype. Gap: fine-grained per-sample lineage (tracking individual records through transformations) is computationally expensive and rarely implemented. Most tools track dataset-level, not record-level lineage. |
| **1.5.2** | **Verify that** lineage records are immutable, securely stored, and accessible for audits. | 2 | Tampering with lineage records to cover tracks after data poisoning; loss of audit trail due to storage failure; disputed provenance during regulatory investigation. | Verify lineage metadata is stored in append-only or immutable storage. Check access controls on lineage databases. Test that lineage records cannot be modified or deleted by pipeline operators. Verify backup and retention policies. | Immutable storage options exist (S3 Object Lock, Azure immutable blobs). Gap: lineage metadata systems (Atlas, DataHub) typically use mutable databases. Achieving true immutability requires architectural decisions that most lineage tools do not enforce by default. |
| **1.5.3** | **Verify that** lineage tracking covers synthetic data generated via augmentation, synthesis, or privacy-preserving techniques. | 2 | Synthetic data contamination obscuring real data quality issues; inability to trace model behaviors to real vs. synthetic training examples; privacy violations if synthetic data retains real-data characteristics without proper tracking. | Review lineage records for synthetic data generation steps. Verify synthetic data carries metadata tags distinguishing it from real data. Check that downstream consumers can filter by data origin type. | Synthetic data generation tools (Gretel, MOSTLY AI, SDV) produce data but rarely integrate with lineage systems. Gap: no standard metadata schema for marking data as synthetic. When synthetic and real data are merged, distinguishability is often lost. This is an emerging best practice without mature tooling support. |
