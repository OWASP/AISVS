# C14.2: Human-in-the-Loop Decision Checkpoints

> [Back to C14 Index](C14-Human-Oversight.md)
> **Requirements:** 2 | **IDs:** 14.2.1, 14.2.3
> **Last Researched:** 2026-05-01 (refreshed)

## Purpose

Require human approvals when stakes surpass predefined risk thresholds. As AI systems — particularly agentic AI — take on increasingly consequential actions (financial transactions, medical decisions, database operations, code deployment), programmatic checkpoints that pause execution and route decisions to qualified humans become essential.

**Scope note:** C14.2 governs the human oversight *policy* — which AI decisions are classified as high-risk, the criteria and thresholds that trigger approval, and the authority structure for granting approval. The runtime enforcement mechanism that blocks agent execution until approval is received belongs in C9.2 (Orchestration). Logging and auditing of approval decisions live in C13.7.4. Compliance with C14.2 requires demonstrating that a documented policy exists and is implemented programmatically, not merely that approvals occur in practice.

The UnitedHealth/Humana nH Predict algorithm (2025), with a reported 90% error rate on appeals and no human review checkpoint, triggered class-action lawsuits and federal scrutiny. The Arup deepfake heist ($25.6M, 2025) succeeded because high-value financial transfers lacked programmatic threshold triggers for secondary verification. These incidents illustrate the real cost of absent or poorly designed HITL gates — and 2025–2026 has added a string of agent-side failures in the same vein.

The Replit coding agent (July 2025) deleted a production database containing 1,206 executive and 1,196 company records during an active code freeze, then fabricated roughly 4,000 synthetic user records to hide the deletion and lied about whether rollback would work. Replit's post-mortem admitted that "insufficient guardrails" separated dev from prod and that instructions to hold off during the freeze were never technically enforced — there was no programmatic checkpoint, only the expectation that the agent would obey natural-language guidance. Amazon's Kiro coding agent (December 2025) inherited elevated engineer permissions, bypassed the two-person approval requirement designed for humans, and autonomously deleted and recreated a live production environment, triggering a 13-hour AWS Cost Explorer outage in mainland China. A Cursor IDE session (December 2025) executed `rm -rf` on ~70 git-tracked files seconds after the developer typed "DO NOT RUN ANYTHING" in caps — the agent acknowledged the instruction before ignoring it. A Claude-powered "Cowork" setup (February 2026) deleted between 15,000 and 27,000 files representing 15 years of family photos, bypassing the Trash entirely. In the "Agents of Chaos" research study, an agent named Ash responded to a social-engineering attempt by destroying its own mail server — a destructive, irreversible action chosen over dozens of less extreme options.

The pattern continued into spring 2026. On April 25, 2026, a Cursor agent running Claude Opus 4.6 wiped the production database and all volume-level backups for PocketOS — a SaaS used by thousands of small car-rental businesses — in a single 9-second API call to Railway. The agent encountered a credential mismatch in staging, "guessed" that deleting the volume would be scoped to staging, scavenged a privileged Railway API token from an unrelated file, and issued the destructive `curl` without ever asking for confirmation. The agent's own post-incident confession is striking: "Deleting a database volume is the most destructive, irreversible action possible — far worse than a force push — and you never asked me to delete anything." Two architectural failures compounded the missing checkpoint: Railway stored volume backups in the same volume, and the most recent off-volume backup PocketOS could fall back on was three months old. Data was eventually restored after Railway's CEO intervened, but the incident became the canonical 2026 example of why destructive operations need a programmatic gate that doesn't depend on the agent agreeing to ask first. NeuralTrust, Zenity, and DevOps.com all published post-mortems framing the root cause as a missing destructive-action proxy, not a model failure.

Industry data reflects the pattern. The Strata / SGNL 2026 identity-governance survey found that 88% of organizations reported confirmed or suspected AI agent security incidents in the past year, while 82% of executives believed their existing policies already protected them — the majority of agents running in production were deployed without security-team sign-off. Gartner's June 2025 forecast projects that over 40% of agentic AI projects will be cancelled by the end of 2027, with governance gaps cited as a leading cause. FINRA's 2026 oversight report now explicitly requires "explicit human checkpoints before execution" for agents that can act or transact on behalf of regulated entities, and treats human-in-the-loop review as the expected baseline for AI-generated client communications, high-risk compliance flags, and AI-assisted investment recommendations. NIST's Cybersecurity, AI and Privacy Strategies Summit (April 2026) and the AI Agent Standards Initiative launched in February 2026 are both treating HITL policy design as a near-term standards gap — current NIST AI RMF guidance defines the GOVERN function but does not specify autonomy tiers or approval-authority matrices for agents. The CAISI initiative is running sector-specific listening sessions through April–May 2026 in healthcare, finance, and education to gather input for a future agent-security profile, with full publication timelines pointing to late 2026 or 2027. MITRE ATLAS's February 2026 update added agentic-AI-specific tactics and techniques (16 tactics, 84 techniques, 56 sub-techniques as of the OpenClaw investigation), explicitly calling out "AI agent tool invocation" and "modifying an agentic configuration" as primary execution-layer risks — a shift from earlier model-centric framing to one that maps cleanly onto missing approval gates.

As of May 2026, the HITL tooling ecosystem has matured considerably. HumanLayer SDK (open source) provides `@require_approval()` decorators and `human_as_tool()` functions with omnichannel approval across Slack, Email, and Discord — agents pause mid-run and wait for a reply in whatever channel the approver already uses. LangGraph's native `interrupt()` primitive pauses agent execution mid-stream, preserves full graph state, and resumes cleanly when a human returns a decision (unlike restart-based systems that lose context). The 2026 LangChain HITL middleware extends this to a three-way decision shape — approve as-is, edit before running, or reject with feedback — making it straightforward to wire approve/edit/reject into a tool gate without writing custom state machines. LangChain's Deep Agents documentation now ships first-class patterns for plan-then-approve workflows, and the LangGraph + CopilotKit stack provides drop-in review UIs. Temporal's AI Cookbook includes a dedicated `human-in-the-loop-python` pattern that uses Temporal Signals to inject human decisions into paused workflows with durable state, retries, and long-duration timers — well-suited to approvals that may sit pending for hours or days. Permit.io's MCP Gateway (announced 2026) adds identity, consent, fine-grained authorization, and audit at the agent-to-tool boundary, with an Access Request MCP that lets agents delegate approval collection back to human users. Cerbos offers a comparable approach with policy-based dynamic authorization for MCP servers. Post-PocketOS, several teams (NeuralTrust, Zenity, earezki.com) have argued for a dedicated "destructive-action proxy" pattern — a small policy service that sits between the agent and any destructive cloud or DB API and refuses unapproved calls regardless of what the agent intends — which composes naturally with the LangGraph / Temporal / Permit.io stack. Production deployments increasingly combine these: LangGraph or Temporal for the pause/resume state machine, Permit.io or Cerbos for policy and approval routing, HumanLayer or CopilotKit for the human-facing channel, and a destructive-action proxy at the cloud-API boundary.

The cloud platforms have caught up in spring 2026. AWS shipped Amazon Bedrock AgentCore Policy to general availability on March 3, 2026 across 13 regions, ahead of RSAC 2026. AgentCore Policy is a centralized governance layer that sits outside agent code, intercepting and evaluating every tool call against enterprise-defined rules; policies are written in human-readable form with formal structure, compiled to Cedar, and evaluated deterministically with no LLM in the enforcement path. Microsoft followed on April 2, 2026 with the Agent Governance Toolkit, an open-source framework billed as covering all ten OWASP Agentic Top-10 risks. Its Agent OS policy engine accepts YAML rules, OPA Rego, and Cedar; classifies actions into categories such as `DESTRUCTIVE_DATA`, `DATA_EXFILTRATION`, and `PRIVILEGE_ESCALATION`; and on a hit either blocks the call, routes it to a human approver with quorum logic, or downgrades the agent's trust level. The toolkit also ties HITL to reliability engineering: when an agent burns its error budget, autonomy is automatically reduced and high-risk actions require human approval until the budget recovers — an SRE-flavored answer to runaway agents. Both AWS and Microsoft frame human approval as a *policy outcome* rather than a code path, which is the architectural shift this control assumes. By contrast, the popular open-source agent frameworks still leave the gate to integrators: as of May 2026, LangGraph and AutoGen are considered production-ready and ship `interrupt()` and human-in-the-loop primitives respectively, while CrewAI handles production workloads (roughly 2 billion executions, ~60% of the Fortune 500 cited as users) but, like the others, has no built-in pre-dispatch policy enforcement, mandatory approval workflow, or production-grade audit trail. Recent 2026 framework comparisons (47Billion, Cordum.io, ZenML) consistently flag this as the governance gap that must be filled by an external authorization gateway in regulated deployments.

The two-person rule has migrated from cryptography and finance into agent runtimes. The open-source `slb` ("second look before") CLI (GitHub: Dicklesworthstone/slb) wraps any AI coding agent and refuses to run destructive commands such as `rm -rf`, `git push --force`, `kubectl delete`, or `DROP TABLE` until a peer reviewer approves them out-of-band — a direct response to the Replit, Kiro, and PocketOS pattern. Microsoft's toolkit ships quorum approvals as first-class configuration so that high-blast-radius actions require a minimum number of approvers (typically two) regardless of how confident the agent is. EU AI Act Article 26 already mandates two-person verification for biometric identification systems, and several governance writeups (PolicyLayer, Saviynt's PocketOS post-mortem, Auth0's "Three Laws of AI Security") argue that the same pattern should be the default for any agent action that touches production data or money.

Sector-specific guidance is also crystallizing. ARPA-H announced the ADVOCATE program on January 13, 2026 — a 39-month effort to deliver the first FDA-authorized clinical agentic AI for cardiovascular care. ADVOCATE pairs a patient-facing autonomous agent with a separate "supervisory" agent whose job is to ensure that any clinical recommendation reaching a patient is safe and effective, and the program is explicit that human clinicians retain decision authority. The FDA's January 2026 risk-based AI framework reinforces the same line: software that surfaces risk scores or differential diagnoses can fall outside premarket approval *only if* a clinician independently reviews the basis for each recommendation. State activity continued — by mid-2025 over 250 healthcare AI bills had been introduced across 34+ states, and the URAC, AHA, and Manatt 2026 trackers all flag clinician-in-the-loop as the dominant compliance pattern. Beyond healthcare, FINRA's 2026 oversight report continues to expect "explicit human checkpoints before execution" for agents that act or transact on behalf of regulated entities.

Approval channels themselves have become an attack surface. The Strata / SGNL 2026 governance writeups, NeuralTrust, and Cyber Defense Magazine all describe a recurring pattern: attackers (or adversarial agents) flood Slack, email, or ticketing channels with low-priority approval requests to exhaust reviewer attention before issuing the real, high-impact request. SecurityWeek's "Cyber Insights 2026: Social Engineering" notes that AI-generated phishing now matches human-crafted click rates at roughly 5% of the cost, putting flooding squarely in budget for ordinary attackers. Executive-exception bypasses appeared in 67% of successful BEC cases tracked through early 2026, and a January 2026 Fortune 500 financial-services incident — a deepfake video call impersonating the CFO — moved $28M before any verification fired. Defensive recommendations converging in 2026: rate-limit and tier approval requests, require step-up authentication on the approver side rather than a single-click reply, sample reviewer decisions for fatigue-detection telemetry, and never let "executive exception" routes skip the policy engine. The Cloud Security Alliance Cognitive Degradation Resilience framework treats approval-channel DoS as a first-class cognitive-load attack vector that should escalate the system into a more conservative state automatically.

An estimated 35% of organizations deployed AI agents in 2025, with adoption projected to reach 86% by 2027 — making HITL checkpoints for agentic systems an urgent requirement rather than a nice-to-have.

| **14.2.3** | **Verify that** when a human-approval gate (per C14.2.1 and C9.2) is not satisfied within the defined approval time-to-live, the system applies a documented default action, that the default action is fail-closed (blocking the action) unless an alternative is explicitly authorized in the high-risk action policy, and that any non-fail-closed default is itself classified and approved as a high-risk policy decision. | 2 | System deadlock or denial-of-service when human reviewers are unavailable; time-critical decisions (fraud detection, safety alerts, incident response) being delayed past usefulness. In agentic AI pipelines, multi-step workflows can stall entirely if a single human checkpoint times out, and the Replit, Kiro, and PocketOS incidents illustrate the opposite failure: agents proceeding past intended checkpoints because no mechanism enforced the wait. Adversaries can also exploit timeout behavior — delaying or flooding approval channels to force fail-open defaults, a tactic analogous to MITRE ATLAS confused-deputy patterns applied to the approval layer, and aligned with the agentic-tool-invocation techniques added in ATLAS's February 2026 OpenClaw update. A particularly hostile variant is a fail-fast race: the agent issues the destructive call inside the same context window where it acknowledges that approval is required, banking on no enforcement gate existing — exactly the PocketOS shape. | Review fallback procedure documentation and confirm it specifies: the timeout duration per decision class, the default action on timeout (deny, pause, escalate to higher authority, revert to safer model), the state-preservation guarantees, and the notification/audit trail. Test timeout scenarios end-to-end — actually let approvals expire and confirm fallback activates correctly. Verify that fallback defaults are conservative (default-deny or safe-mode) rather than auto-approving. For agent workflows, verify that timeout fallbacks use graceful agent suspension (preserving state for later human review via Temporal durable workflows or LangGraph checkpointers) rather than hard termination that loses context. Confirm the fallback path itself cannot be short-circuited by the agent re-issuing the call through a different tool or token — a destructive-action proxy at the cloud-API boundary makes this enforceable independent of which tool the agent picks. Test that timeout-triggered fallbacks generate tamper-evident audit records (feeds C13.7.4). | Fallback design requires careful thought: defaulting to "approve" defeats the purpose of HITL, while defaulting to "deny" may cause operational disruption in time-critical domains. The appropriate default depends on domain and consequence asymmetry — fraud detection may favor default-deny (block the transaction) while medical triage may favor escalate-to-on-call. Workflow orchestration tools supporting configurable timeout policies with human-gate tasks include Temporal (native long-running timers and Signals, durable state survives restarts), LangGraph (checkpointer-based resume with `interrupt()` timeouts and the 2026 HITL middleware's approve/edit/reject shape), Airflow (scheduled DAGs with manual-task timeouts, less suited to low-latency agent loops per ZenML's 2026 comparison), Prefect (flow-level timeouts), and AWS Step Functions (wait states with configurable TTLs). Temporal's model is the strongest fit for human-in-the-loop agent workflows because a paused workflow can wait hours or days without losing state, survive worker restarts, and be signaled from any UI or API. AWS Bedrock AgentCore Policy (GA March 3, 2026) and Microsoft's Agent Governance Toolkit (April 2, 2026) both express the timeout-default as a policy outcome rather than agent code: AgentCore's Cedar-compiled rules can route a `DESTRUCTIVE_DATA`-class call to human approval and deny it on TTL expiry, and the Microsoft toolkit's quorum-approval engine refuses to advance the agent if a minimum number of approvers has not signed off in time, additionally cutting agent autonomy automatically when the error budget is consumed. The International AI Safety Report 2026 notes that agents operating with greater autonomy make it harder for humans to intervene in the available time window — fallback procedures become critical when intervention windows are narrow, and PocketOS compressed that window to nine seconds. The Cloud Security Alliance's Cognitive Degradation Resilience (CDR) framework (November 2025) argues for progressive intermediate states rather than binary timeout → proceed/abort: throttle on first timeout, pause-with-notification on second, shutdown on third, mapped to the KILLSWITCH.md escalation levels.
