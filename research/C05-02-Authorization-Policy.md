# C5.2: AI Resource Authorization & Classification

> [Back to C05 Index](C05-Access-Control.md)
> **Last Researched:** 2026-04-16 (5th pass)

## Purpose

Implement access controls for all AI resources with explicit permission models and audit trails. AI systems introduce authorization challenges that go beyond traditional application security: vector databases, embedding indices, model registries, and prompt caches all require explicit access control. Data classification labels must propagate through AI pipelines to derived resources, and authorization decisions should be externalized to dedicated policy decision points to prevent scattered, inconsistent logic across AI microservices.

> **Scope note:** Controls in this section govern authorization policy definition, enforcement, and audit for AI resources. Identity and authentication are addressed in C5.1; query-time enforcement at the data layer is in C5.3; output-level filtering in C5.4. Per ASVS v5 14.1.1, all sensitive data should be identified and classified into protection levels; C5.2.11 extends this to AI-specific data types not enumerated in ASVS.

| **5.2.1** | **Verify that** every AI resource (datasets, models, endpoints, vector collections, embedding indices, compute instances) enforces access controls (e.g., RBAC, ABAC) with explicit allow-lists and default-deny policies. | 1 | Unauthorized access to training data, model weights, or inference endpoints; privilege escalation through permissive default policies. | Attempt access to each resource type with unauthenticated and under-privileged credentials; confirm default-deny by testing access without explicit grants; review ACL configurations for all AI resource types. | Vector databases (Pinecone, Weaviate, Milvus) have varying RBAC maturity. Some require application-layer enforcement rather than native database-level controls. Embedding indices are often overlooked in access control audits. **[2026-03]** Google Cloud API keys default to "Unrestricted" status, automatically gaining access to all enabled APIs including Gemini — a clear violation of default-deny. Pinecone and managed Weaviate now support document-level filtering and role-based retrieval natively. Permit.io provides fine-grained policy-based access control spanning apps, APIs, data layers, and AI agents. **[2026-04]** Cerbos now supports end-to-end authorization for AI agents and MCP servers, controlling which documents agents can retrieve from vector stores (Pinecone, Weaviate, Chroma, Qdrant) based on requesting user entitlements. IBM's July 2025 Cost of a Data Breach report found that 13% of organizations experienced breaches of AI models or applications, and of those breached, 97% lacked proper AI access controls — underscoring how common default-allow postures remain. Shadow AI breaches (employees using unsanctioned AI tools) averaged $670K more than traditional incidents, with 65% compromising PII and 40% exposing IP. The March 2026 Mercor/LiteLLM supply chain breach demonstrated cascading credential failure: attackers compromised LiteLLM (an AI API proxy used by thousands of organizations), harvested credentials from an estimated 500,000 machines within 40 minutes, and exfiltrated 4TB of data including source code, user databases, and API keys from the $10B AI recruiting startup — a direct consequence of permissive default-allow access policies across AI service dependencies. A CSA/Strata Identity survey (February 2026) found that 84% of organizations cannot pass a compliance audit focused on agent access controls, and only 23% have a formal agent identity strategy. Milvus now supports row-level RBAC using bitmap-indexed `security_group` columns, enabling fine-grained per-row access control within vector collections — a significant improvement over collection-level-only controls. The January 2026 Step Finance breach ($40M+ drained from treasury) demonstrated cascading failure when AI trading agents operated with excessive permissions: attackers compromised executive devices and the connected agents executed unauthorized large-scale transfers because their protocols lacked proper authorization boundaries and isolation. As of April 2026, vector databases now power roughly 68% of enterprise AI applications, but RBAC depth still varies: Pinecone provides organization/project/member RBAC plus SOC 2 Type II and GDPR alignment, Weaviate offers document-level filtering and tenant isolation, and Milvus's row-level `security_group` controls remain unique among the open-source options. The OWASP Top 10 for Agentic Applications 2026 (released December 2025, peer-reviewed by 100+ contributors) elevates "Identity & Privilege Abuse" and "Tool Misuse" to top-tier risks specifically because organizational agents often inherit broader permissions than the users they serve, and a default-allow posture lets a low-privilege user trigger high-privilege actions through agent indirection. |
| **5.2.2** | **Verify that** privileged access to model weights, training pipelines, and production AI configuration is provisioned on a just-in-time basis with a defined maximum session duration and automatic expiry, and permanent standing privileged access to these resources is not permitted. | 2 | Persistent standing privileges to model weights enable insider exfiltration; always-on access to training pipelines allows undetected data poisoning or unauthorized fine-tuning; compromised service accounts with permanent access provide attackers dwell time for lateral movement across AI infrastructure. | Verify no permanent privileged accounts exist for AI resources — enumerate all accounts with write access to model registries, training pipelines, and production configs; confirm each uses JIT provisioning with maximum session durations; test that access automatically expires; attempt to access resources after session expiry; review PAM tool configurations (CyberArk, HashiCorp Vault/Boundary, Azure PIM) for JIT enforcement. | **[2026-04]** CyberArk's January 2026 study found only 1% of organizations have fully adopted JIT privileged access, while 91% report at least half of their privileged access is always-on. 54% discover unmanaged privileged accounts weekly. HashiCorp Vault/Boundary supports dynamic short-lived credentials that are created at access time and destroyed after session completion. Azure PIM provides time-bound role activation with approval workflows. The gap is AI-specific: training pipelines carry invisible privileges (unrestricted data access, automated model updates, cross-cloud execution) that most PAM tools were not designed to inventory or govern. Break-glass access with ticketing and time-bound approvals is the recommended fallback for vendor support scenarios. CyberArk's 2026 guidance pairs JIT with explicit Zero Standing Privileges (ZSP) — agents and humans alike receive permissions only for the specific task and session, after a risk assessment, and lose them on completion — and recommends pre-approval policies for routine elevation requests so operators are not bottlenecked on manual review. The pattern that concretely closes 5.2.9 for AI workloads is a delegated-access flow: agents inherit user identity through token exchange (RFC 8693) rather than running under a long-lived service account, and the resulting capability tokens are scoped to a single tool call and a short TTL — Strata's documented Maverics flow uses 5-second tokens for this purpose, and AWS Bedrock AgentCore Policy enforces the same model at the Cedar gateway. |
| **5.2.3** | **Verify that** data classification labels (PII, PHI, proprietary, etc.) automatically propagate to derived resources (embeddings, prompt caches, model outputs). | 3 | Data classification loss during AI pipelines; sensitive data in embeddings or caches accessed without appropriate controls because labels were dropped during transformation. | Tag a source dataset with a classification label; trace label propagation through embedding generation, prompt caching, and output storage; confirm derived resources inherit the source classification. | This is an AI-specific challenge with limited tooling. Embeddings are mathematical transformations that lose explicit metadata. Label propagation typically requires custom pipeline instrumentation. Microsoft Purview and Google DLP have partial support. **[2026-04]** Atlan (a Leader in the 2026 Gartner Magic Quadrant for Data & Analytics Governance) now automatically propagates tags, classifications, sensitivity labels, and policy labels along lineage paths to downstream assets — including derived datasets and transformations. Atlan's MCP Server allows AI agents to query the lineage graph before using data, returning provenance, quality score, governance policy, and ownership in a single call. Column-level lineage stitches together dbt, Airflow, and warehouse query history into a unified graph. For AI pipelines specifically, ML teams can track full model lineage from training sets through features, transformations, model versions, and predictions. Collibra supports manual and automated sensitive data classification but its lineage capabilities remain limited to cloud editions. In practice, classification propagation failures remain the most pervasive AI data protection gap: lineage records rarely link embeddings back to source records, and missing classification tags on derived embeddings create compliance blind spots — particularly for GDPR/DSR erasure obligations where derived artifacts persist after source deletion. Atlan's active metadata platform now automates lineage across 100+ connectors with bi-directional tag synchronization to Snowflake and Databricks, though embedding-specific lineage still requires custom instrumentation. Microsoft Purview's DSPM for AI (rebranded from AI Hub at RSA 2026) extends sensitivity-label inheritance to SharePoint document libraries via auto-labeling defaults, so files at rest pick up the library's label without user action — but Microsoft's own documentation acknowledges the inheritance does not yet flow into derived embedding stores or prompt caches. The "breach-by-exhaust" pattern flagged in industry 2026 predictions describes precisely this gap: forgotten vector databases hold derivatives of sensitive source records that were never tagged, look like normal infrastructure, and only surface as a breach vector once attackers query them. Cyberhaven and zvelo both report that more than 80% of exfiltrated AI-derived data is fragmented across embeddings, prompt caches, and model outputs, making lineage-based tagging — not content inspection — the only viable detection path. |
| **5.2.4** | **Verify that** a documented data classification taxonomy covering AI-specific data types (embeddings, model weights, prompt templates, RAG context assemblies, fine-tuning datasets, agent tool schemas) is defined, and that AI assets are labeled in accordance with this taxonomy. | 2 | Unclassified AI assets treated with inappropriate protection levels; model weights stored alongside non-sensitive data without access restrictions; prompt templates containing system instructions or guardrails exposed as public artifacts; embeddings that encode sensitive source data treated as non-sensitive because their mathematical nature obscures the original content; fine-tuning datasets containing PII processed without privacy controls. | Request the organization's data classification taxonomy and confirm it enumerates AI-specific data types (embeddings, model weights, prompt templates, RAG context assemblies, fine-tuning datasets, agent tool schemas); sample AI assets from each category and verify classification labels are applied; confirm classification labels propagate through pipelines (cross-reference with 5.2.4); verify that protection controls match the assigned classification tier. | **[2026-04]** NIST IR 8596 (Cybersecurity Framework Profile for AI, December 2025 draft) urges organizations to maintain inventories covering models, agents, APIs/keys, datasets/metadata, and embedded AI integrations, along with maps of end-to-end AI data flows. NIST AI 100-2 E2025 provides an adversarial ML taxonomy that implicitly defines asset sensitivity (training data as poisoning targets, model weights as exfiltration targets). Microsoft Purview supports sensitivity labels with autolabeling policies and trainable classifiers, and its Fabric APIs now allow programmatic label management — but coverage of AI-specific artifacts like embeddings and prompt templates remains partial. No off-the-shelf tool comprehensively classifies all six AI data types listed in this requirement; organizations typically extend existing DLP taxonomies with custom categories. The ODI published a "Data for AI" taxonomy addressing data provenance and fitness considerations. The initial public draft of NIST IR 8596 (December 2025) explicitly expands asset inventory requirements to include models, system prompts and templates, training and evaluation datasets, RAG sources (vector databases), embeddings, agent tool permissions, and API keys — moving closer to a standardized taxonomy that aligns with this requirement's scope. NIST IR 8596 remains in initial preliminary draft status as of April 2026; the public comment period closed January 30, 2026 and a final version is expected later in 2026 with expanded resource mappings. Microsoft Purview's DSPM for AI extends sensitivity-label inheritance to SharePoint document libraries and integrates with Copilot and third-party LLM apps via the Microsoft Purview portal, but coverage of derived embeddings, RAG context assemblies, and agent tool schemas still requires custom autolabeling rules and trainable classifiers that organizations must author themselves. The OWASP Top 10 for Agentic Applications 2026 implicitly expands this taxonomy by treating agent tool permissions, delegation chains, and identity context as first-class assets that must be inventoried alongside the data they touch — a labeling dimension few existing data-classification taxonomies cover. |
