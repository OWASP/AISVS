# C04: Infrastructure, Configuration & Deployment Security

> **Source:** [`1.0/en/0x10-C04-Infrastructure.md`](https://github.com/OWASP/AISVS/blob/main/1.0/en/0x10-C04-Infrastructure.md)
> **Requirements:** 21 | **Sections:** 3

## Control Objective

AI-specific infrastructure components must be hardened against model theft, data leakage, and cross-tenant contamination. This chapter covers AI workload sandboxing, AI accelerator hardware security, and edge/distributed AI deployment security. General infrastructure security controls (container hardening, network segmentation, secrets management, CI/CD pipeline security) are covered by ASVS, CIS Benchmarks, and NIST SP 800-53, and are not repeated here. AI-specific supply chain controls are covered in C6.

> **2025-2026 Highlights:** June 2026 shifted the spotlight from inference servers to the AI middleware around them. CISA added LiteLLM CVE-2026-42271 (CVSS 8.8) to the Known Exploited Vulnerabilities catalog on June 8, 2026 — the proxy's MCP server preview endpoints accepted attacker-supplied stdio `command`/`env` fields and spawned them as subprocesses, so any valid API key of any role yields OS command execution (fixed in 1.83.7; KEV remediation due June 22). vLLM shipped three advisories in nine days: CVE-2026-48746 (CVSS 9.1, June 2) lets crafted Host headers desync URL parsing from FastAPI routing and reach protected endpoints without the configured `VLLM_API_KEY` (fixed in 0.22.0); a dependency-confusion hole in the official Dockerfile let a squatted PyPI package execute as root during image builds (fixed in 0.22.1, June 9); and "Artifact Pin Decay" (CVE-2026-47155, June 10) showed `--revision` pins silently failing to propagate to dynamic code modules, GGUF loaders, and cross-repo weights. Hugging Face Transformers CVE-2026-4372 (disclosed June 4 after a silent March patch) bypassed `trust_remote_code=False` via an `_attn_implementation_internal` key in `config.json`, fetching and executing attacker kernel code at model load (fixed in 5.3.0). May 2026 remains the worst month yet for inference servers proper: NVIDIA's May 20 Triton bulletin (CVE-2026-24207, CVSS 9.8 unauthenticated authentication bypass plus seven companions; baseline now r26.03), SGLang's pickle/dill-over-ZMQ RCEs (CVE-2026-7301/7304, both CVSS 9.8, fixes still incomplete as of June 10), and the Linux kernel Dirty Frag root primitive (CVE-2026-43284/43500) on virtually every GPU-cluster distribution; April's LMDeploy SSRF (CVE-2026-33626) was exploited within 12.5 hours of disclosure. The defensive headline is a status flip: NVIDIA's Confidential Containers Reference Architecture 1.0.0 went GA — H100 single-GPU and H200 Protected PCIe multi-GPU confidential containers are now a supported configuration with GPU Operator 26.3 (the 25.10 line was early-access only), Confidential Containers v0.21.0 (May 29) builds on Kata 3.31.0 with composite CPU+GPU attestation, the kubernetes-sigs DRA driver for NVIDIA GPUs shipped its first post-CNCF-donation release (v0.4.0, May 15), and Red Hat documented bare-metal CoCo on OpenShift with NRAS GPU attestation (June 4). MITRE ATLAS restructured onto a monthly release cadence (v2026.05, May 27) with platform designations — Predictive, Generative, Agentic AI, Enterprise — across roughly 16 tactics and 84 techniques. On regulation: Parliament's IMCO and LIBE committees approved the EU AI Act omnibus on June 2, 2026 (plenary vote expected by early July, racing the original August 2, 2026 trigger; Annex III high-risk obligations move to December 2, 2027), and a June 2, 2026 White House Executive Order stood up a Treasury-led AI cybersecurity clearinghouse with NSA/CISA/ONCD and directed AI-enabled cyber defense of national security systems. The March 2026 TeamPCP cascade (Trivy, Checkmarx KICS, LiteLLM — CVE-2026-33634) remains the largest AI supply chain compromise on record, and Trend Micro still projects 2,800–3,600 AI CVEs in 2026.

## Threat Landscape

Known attacks, real-world incidents, and threat vectors relevant to this chapter:

- **AI gateway exploited in the wild — LiteLLM MCP command injection (June 2026 KEV)** — CVE-2026-42271 (CVSS 8.8, fixed in LiteLLM 1.83.7) let any caller holding a valid API key — any role — POST an MCP stdio-transport configuration with attacker-controlled `command` and `env` fields to the proxy's MCP preview endpoints, which spawned the command as a subprocess with the proxy's privileges. CISA added it to the [Known Exploited Vulnerabilities catalog](https://www.cisa.gov/known-exploited-vulnerabilities-catalog) on June 8, 2026 with a June 22 remediation deadline — confirmation that AI gateways sitting in front of model-serving fleets are being exploited, not just researched. See the [GitHub advisory](https://github.com/BerriAI/litellm/security/advisories/GHSA-v4p8-mg3p-g94g). For the second time in 2026 (after TeamPCP), LiteLLM's ubiquity — 36% of monitored cloud environments per Wiz — turns a single proxy flaw into fleet-wide exposure.
- **vLLM June 2026 advisory cluster — auth bypass, build-time dependency confusion, pin decay** — Three advisories in nine days. [CVE-2026-48746](https://github.com/vllm-project/vllm/security/advisories/GHSA-94f4-hr76-p5j6) (CVSS 9.1, June 2): the API-key middleware reconstructs URLs from the untrusted Host header, so special characters desync the checked path from the FastAPI-routed path and requests reach protected endpoints without `VLLM_API_KEY` (fixed in 0.22.0; RFC-conforming reverse proxies like nginx block the malformed Host header). [GHSA-jrf6-vqxq-pjv2](https://github.com/vllm-project/vllm/security/advisories/GHSA-jrf6-vqxq-pjv2) (CVSS 8.8, June 9): the official Dockerfile installed `flashinfer-jit-cache` via `--extra-index-url` with `UV_INDEX_STRATEGY=unsafe-best-match` while the name was unregistered on PyPI — a squatted package (one was actually deposited and later quarantined) would execute as root during image build (fixed in 0.22.1). [CVE-2026-47155](https://github.com/vllm-project/vllm/security/advisories/GHSA-3ww4-5jv9-j5gm) "Artifact Pin Decay" (June 10): `--revision`/`--code-revision` pins are not propagated to secondary artifacts — dynamic code modules, GGUF loaders, image processors, cross-subfolder weights — so "pinned" deployments silently load unpinned code and weights from default branches.
- **Model-config code execution despite `trust_remote_code=False` (June 2026)** — Hugging Face Transformers [CVE-2026-4372](https://www.csoonline.com/article/4181094/hugging-face-transformers-rce-flaw-enables-stealthy-compromise-via-ai-model-configs.html) (affects 4.56.0–5.2.x, fixed in 5.3.0, silently patched March 3 and publicly disclosed June 4 by Pluto Security): injecting `_attn_implementation_internal` into a model's `config.json` makes Transformers — with the optional `kernels` package installed — fetch and execute code from an attacker-controlled kernel repo during model load, bypassing `trust_remote_code=False`. OpenMed CVE-2026-47117 (CVSS 9.8, June 3, fixed in 1.5.2) made the same class pre-auth: its PII-filter substring-matched the user-supplied `model_name` and loaded the matching Hugging Face repo with `trust_remote_code` enabled. Both reinforce C4.1.3's point that the *configuration* and *loader path* of a model artifact are code-execution surfaces, not just the serialized weights.
- **Container-escape KEV additions (May–June 2026)** — CISA added CVE-2022-0492 (Linux cgroups `release_agent` container escape) to KEV on June 2, 2026, and Langflow CVE-2025-34291 (origin validation, distinct from the March CVE-2026-33017) on May 21, 2026 — both confirmed exploited in the wild and both squarely in C4.1 sandbox-isolation territory: a four-year-old kernel escape is actively used against containerized workloads.
- **Triton authentication bypass and DALI flaws (May 2026)** — NVIDIA's [May 2026 Triton bulletin](https://nvidia.custhelp.com/app/answers/detail/a_id/5828/~/security-bulletin%3A-nvidia-triton-inference-server---may-2026) patches eight issues, headlined by CVE-2026-24207 (CVSS 9.8, CWE-288) — an unauthenticated authentication-bypass that gives remote code execution, privilege escalation, data tampering, denial of service, and information disclosure against a network-reachable Triton instance. Three high-severity issues land in the DALI backend (CVE-2026-24213/24214/24215, two at CVSS 8.0), tightening the noose on the pre-processing pipeline that sits in front of model inference. Fixed in Triton r26.03; older r26.02 deployments need an urgent upgrade.
- **SGLang pickle-over-ZMQ revival (May 2026)** — Published May 18, 2026, CVE-2026-7301 (CVSS 9.8) shows the SGLang multimodal scheduler ROUTER socket binding to `0.0.0.0` and calling `pickle.loads()` on inbound messages — unauthenticated RCE against any internet-reachable SGLang deployment. CVE-2026-7304 (CVSS 9.8) chains in when `--enable-custom-logit-processor` is on, this time via `dill.loads()`. CERT/CC's [VU#777338](https://kb.cert.org/vuls/id/777338) flags incomplete maintainer fixes — still incomplete as of June 10, 2026, with no completed remediation published; the safest position remains keeping SGLang off the network or fronting it with a network policy that blocks ZMQ ports. This is the third ShadowMQ-pattern bug in SGLang in twelve months — the C4.1.3 deserialization control is doing nothing if the framework calls `pickle.loads()` on raw socket bytes.
- **AI framework RCE via workflow platforms** — CVE-2026-33017 (CVSS 9.3, March 2026) in Langflow allows unauthenticated remote code execution via a single HTTP POST to the public flow build endpoint. Attacker-controlled Python code is passed directly to `exec()` without sandboxing, granting access to environment variables, API keys for OpenAI/Anthropic/AWS, and connected databases. CISA added it to KEV within 8 days. Exploited in the wild within 20 hours of disclosure.
- **LMDeploy SSRF as inference-engine pivot (April 2026)** — CVE-2026-33626 in InternLM's LMDeploy let attackers turn the vision-language image loader (`load_image()` in `lmdeploy/vl/utils.py`) into a generic HTTP SSRF primitive. No hostname resolution check, no private-network blocklist. Sysdig saw the first in-the-wild exploitation 12 hours 31 minutes after the GitHub advisory landed — over an eight-minute window the attacker enumerated AWS IMDS, Redis on 127.0.0.1:6379, MySQL on 127.0.0.1:3306, an internal admin HTTP service, and an out-of-band DNS exfiltration endpoint, all through a vision-LLM chat completions endpoint. Fixed in v0.12.3. SSRF defenses (`169.254.0.0/16` egress block, allowlisted hostname pattern, separate egress identity from the inference service identity) belong in every model server deployment guide.
- **Cascading supply chain compromise (TeamPCP)** — In March 2026, the TeamPCP threat actor compromised Aqua Security's Trivy scanner (CVE-2026-33634, CVSS 9.4) by force-pushing 76 malicious version tags in the trivy-action GitHub Action and publishing poisoned Docker Hub images. The attack cascaded to LiteLLM (malicious PyPI packages 1.82.7/1.82.8) and Checkmarx KICS. Wiz Research found LiteLLM present in 36% of monitored cloud environments. Microsoft published emergency detection guidance; BerriAI engaged Mandiant for incident response.
- **Kubernetes-native AI operator command injection** — CVE-2026-34940 in KubeAI showed how model-serving control planes can become execution surfaces. Unsanitized model URL components from a `Model` custom resource were interpolated into an Ollama `bash -c` startup probe, letting users with create/update rights on Model CRDs execute commands inside model server pods. The practical controls are to upgrade to KubeAI 0.23.2 or later, restrict Model CRD writers, enforce admission validation for model URLs, and review startup/readiness/liveness probes as part of C4.1 sandbox validation.
- **Container escape via GPU toolkits** — Critical vulnerabilities in NVIDIA Container Toolkit (CVE-2024-0132, CVE-2025-23266 "NVIDIAScape") allow AI workloads to break container isolation and gain root on the host. As of mid-2025, 37% of cloud AI environments were vulnerable.
- **GPU side-channel attacks** — NVBleed (March 2025) demonstrated cross-VM covert channels via NVLink performance counters at 70+ Kbps, while BarraCUDA showed electromagnetic side-channel model extraction from NVIDIA Jetson edge devices. Separate research found four leakage sources bypassing MIG isolation under MPS mode.
- **GPU memory bit-flip attacks** — GPUHammer (USENIX Security 2025) demonstrated practical Rowhammer attacks against GDDR6 memory on NVIDIA GPUs. A single bit flip to a model weight's exponent can degrade accuracy from 80% to under 1%. In shared cloud GPU environments, a malicious tenant can target adjacent workloads' model weights. ECC memory (enabled by default on Hopper and Blackwell data center GPUs) mitigates this, but consumer and edge GPUs remain vulnerable.
- **Supply chain compromise of ML frameworks** — CVE-2025-32434 (CVSS 9.3) proved that PyTorch's `weights_only=True` could be bypassed for RCE. Three PickleScan zero-day bypasses (CVE-2025-10155/56/57) allowed undetected malicious model uploads to Hugging Face.
- **Secrets leakage from AI pipelines** — A November 2025 study found 65% of Forbes AI 50 companies had leaked API keys on GitHub, primarily through Jupyter notebooks and .env files. One leaked Hugging Face token exposed ~1,000 private models.
- **TEE bypass attacks** — TEE.Fail (October 2025) broke Intel SGX, TDX, and AMD SEV-SNP using DDR5 memory bus interposition hardware costing under $1,000, undermining confidential computing assumptions.
- **Linux kernel Dirty Frag (May 2026)** — CVE-2026-43284 (CVSS 7.8) and CVE-2026-43500 (CVSS 7.8) are paged page-cache write primitives in the IPsec ESP (`xfrm`) and RxRPC subsystems. Disclosed May 7-8, 2026 after a broken responsible-disclosure embargo, the public exploit yields root "in a single command" on essentially every distribution running modern AI clusters — Ubuntu, RHEL, CentOS Stream, AlmaLinux, Fedora, openSUSE, OpenShift. Microsoft observed in-the-wild SSH-to-LPE activity within 48 hours. Hardened Kubernetes nodes with default seccomp profiles raise the bar (the trigger needs `CAP_NET_ADMIN`), but AI workloads that grant elevated network capabilities — sidecars handling RDMA setup, model fetchers, training-job init containers — are still in scope. Patch availability is staggered: AlmaLinux, Ubuntu, and SUSE shipped kernels by May 8; KernelCare live-patches followed.
- **Container runtime breakout (runC)** — Three critical runC vulnerabilities disclosed in November 2025 (CVE-2025-31133, CVE-2025-52565, CVE-2025-52881) affect all Docker, Kubernetes, containerd, and CRI-O deployments. CVE-2025-31133 exploits a mount race condition in maskedPaths to redirect writes to `/proc/sys/kernel/core_pattern`, achieving container escape. CVE-2025-52565 abuses a timing window where `/dev/console` is mounted before security protections are applied. CVE-2025-52881 bypasses Linux Security Module labels via shared mount race conditions. All three are deliverable through untrusted container images — a realistic attack vector for AI workloads that routinely pull models and dependencies from external registries. Fixed in runC 1.2.8 / 1.3.3 / 1.4.0-rc.3.
- **ShadowMQ — copy-paste RCE across inference frameworks** — In November 2025, Oligo Security disclosed ShadowMQ, a systemic pattern where Meta's Llama Stack RCE (CVE-2024-50050) propagated by direct copy-paste into NVIDIA TensorRT-LLM, vLLM, SGLang, and Modular Max Server. All of them called ZeroMQ's `recv_pyobj()` over unauthenticated TCP sockets, triggering `pickle` deserialization on attacker input. Exploitation yields arbitrary code execution on GPU clusters — exfiltration of model weights, lateral movement, or GPU cryptomining. Fixed in Llama Stack ≥ 0.0.41, TensorRT-LLM 0.18.2, vLLM 0.8.0, and Modular Max v25.6. This is a direct C4.1.3 case study: the frameworks that serve models were themselves loading attacker input through unsafe deserialization paths.
- **Inference server takeover chain** — CVE-2025-23319, CVE-2025-23320, and CVE-2025-23334 in NVIDIA Triton Inference Server (disclosed Aug 2025, discovered by Wiz Research following Pwn2Own Berlin 2025) chain into unauthenticated RCE against a production model-serving platform. The first Pwn2Own AI-infrastructure category included Triton, Redis, and Chroma as targets — seven of twenty-eight competition vulnerabilities came from the AI bracket. Trend Micro's 2026 projection puts AI CVEs at 2,800–3,600 (up from 2,130 in 2025), with inference servers, model-hosting platforms, and agentic systems driving the growth.
- **Inference server availability and model-config exposure** — NVIDIA's March and April 2026 Triton bulletins added eight more network-reachable flaws after the August 2025 takeover chain. March fixed CVE-2025-33238, CVE-2025-33254, and CVE-2026-24158 (large compressed HTTP payload denial of service) in 26.01; April fixed CVE-2026-24146/24173/24174/24175 denial-of-service flaws and CVE-2026-24147 model-configuration upload information disclosure/DoS in r26.02. Treat inference endpoints as service infrastructure: cap request and output sizes, gate model repository writes, restrict model-configuration upload paths, and rehearse rollback before disclosure windows.
- **Lateral movement** — Attackers have used malicious registry URLs to force exposed LLM inference servers into making outbound requests, probing for paths to training infrastructure.
- **Edge model extraction** — Studies show 34–76% success rates extracting ML models from mobile apps using standard reverse-engineering tools when models lack encryption.

### Inference Server Hardening Checks

- Track inference-server PSIRT bulletins separately from base container and GPU driver patching; Triton moved from the August 2025 RCE chain through March, April, and May 2026 bulletins — each one network-reachable, each one shifting the operational baseline (now Triton r26.03 after May 2026's CVE-2026-24207 authentication bypass).
- Put explicit limits in front of model-serving HTTP/gRPC endpoints: compressed request body size, output tensor count, decompression ratio, request concurrency, and per-tenant queue depth.
- Treat model repository writes, model-control APIs, Python backend loading, and configuration upload paths as administrative interfaces; keep them authenticated, network-restricted, logged, and separated from public inference traffic.
- Block direct internet exposure of internal IPC sockets used by serving frameworks. SGLang's May 2026 CVE-2026-7301 fired because the scheduler ROUTER socket bound to `0.0.0.0` and the framework called `pickle.loads()` on inbound messages. Default network policies should deny ingress to ZeroMQ ports; mutual authentication or even simple `--host 127.0.0.1` binding would have prevented the RCE.
- Treat image and document fetchers inside vision-language servers as SSRF risk. LMDeploy CVE-2026-33626 (April 2026) had no IP-allowlist on `load_image()`, so a `image_url=http://169.254.169.254/...` body retrieved IMDS credentials through the model server. Block link-local, RFC1918, and metadata-service ranges at the egress firewall and run the inference service under an IAM role that holds no useful metadata.
- Do not rely on framework-native API-key middleware as the only authentication layer. vLLM CVE-2026-48746 (June 2026) bypassed `VLLM_API_KEY` entirely with a crafted Host header because the middleware reconstructed URLs from untrusted input; an RFC-conforming reverse proxy (nginx, Envoy) in front of the server blocked the attack. Authenticate at the proxy and treat the framework check as defense-in-depth.
- Verify that model and code revision pinning actually covers every artifact the server loads. vLLM's "Artifact Pin Decay" (CVE-2026-47155) showed `--revision` pins silently skipping dynamic code modules, GGUF loaders, image processors, and cross-repo weights — audit what the process fetches at startup (egress logs, `strace`/eBPF on first load) rather than trusting the flag.
- Treat AI gateway and proxy configuration endpoints — MCP server registration, tool config, provider setup — as administrative command-execution surfaces. LiteLLM CVE-2026-42271 (KEV, June 2026) turned an MCP stdio config POST into OS command execution for any API-key holder. Scope these endpoints to admin roles, disable preview/experimental config APIs in production, and alert on subprocess spawns from gateway processes.
- Audit serving-container build pipelines for dependency confusion: pin hashes, avoid `--extra-index-url` with permissive index strategies (vLLM's Dockerfile shipped `UV_INDEX_STRATEGY=unsafe-best-match` against an unregistered PyPI name until 0.22.1), and build from lockfiles with `--require-hashes` where possible.
- Exercise rollback and canary procedures for serving-stack upgrades before disclosure windows; a broken inference rollback can become an availability incident even when the security patch is correct.
- Subscribe to the [Kubernetes official CVE feed](https://kubernetes.io/docs/reference/issues-security/official-cve-feed/) as part of AI platform operations. Note that the [ingress-nginx project formally retired on March 24, 2026](https://kubernetes.io/blog/2026/04/22/kubernetes-v1-36-release/) and now receives no further updates or security patches — any AI platform still routing model-serving ingress through it needs a migration plan to `ingate`, Istio Gateway API, or a cloud-managed alternative.

### AI Operator Hardening Checks

- Treat `Model`, `InferenceService`, runtime, and model-repository CRDs as code-execution surfaces; restrict create/update rights, log changes, and require review for external model URLs.
- Reject model URLs with shell metacharacters through admission policy, and prefer exec probes that pass arguments as arrays rather than interpolating model references into `bash -c`.
- Test startup, readiness, liveness, and sidecar hooks with malicious model names, PVC query parameters, and registry references before onboarding model-serving operators.
- Pin and patch AI operators such as KubeAI, KServe, and Triton management components separately from the underlying Kubernetes and GPU driver patch cycle.

## Open Research Questions

- [x] **What's the maturity of confidential computing for GPU workloads?** — As of early 2026, NVIDIA H100 CC is GA with on-die root of trust, encrypted CPU-GPU transfers, and remote attestation (NRAS), but throughput is limited to ~4 GB/s. The Vera Rubin NVL72 (announced January 2026) extends confidential computing to rack-scale — 72 GPUs, 36 CPUs, and NVLink interconnects form a unified trust domain with end-to-end encryption and near-native performance, a significant leap over H100's per-GPU model. However, the TEE.Fail attack (Oct 2025) showed DDR5 memory bus interposition can break underlying CPU TEEs with <$1K hardware, and GPUHammer demonstrated practical Rowhammer attacks against GDDR6 GPU memory (USENIX Security 2025). The hardware threat surface continues to expand alongside defensive capabilities.
- [x] **How do you isolate multi-tenant GPU workloads without significant performance overhead?** — NVIDIA MIG provides hardware partition-level isolation (up to 7 instances on A100/H100) with dedicated memory, cache, and SMs per tenant. Performance isolation is strong but MIG does not encrypt — separate research showed four side-channel leakage sources bypassing MIG under MPS mode, and GPUHammer showed cross-tenant Rowhammer attacks are practical on GPUs without ECC. Full confidential multi-tenancy requires H100 CC mode (with throughput constraints) or the upcoming Vera Rubin NVL72, which promises rack-scale encryption with near-native performance.
- [ ] **What are the best practices for securing AI inference at the edge with limited compute?** — Apple Core ML offers compile-time model encryption (decrypted only at runtime), but this is platform-specific. Android lacks a native equivalent; teams typically use Android Keystore for key protection with custom encryption wrappers. No universal cross-platform model encryption standard exists yet. The ONNX Server for Open Enclave project provides TEE-based inference but targets server-class hardware.
- [ ] **How should hardware security requirements differ for training vs. inference infrastructure?** — Training requires higher throughput and multi-node coordination (NVLink, InfiniBand), where the NVBleed side-channel and RDMA protocol weaknesses (NeVerMore) are relevant. Inference is more deployable in TEEs due to lower data transfer volumes but faces model extraction risks. Edge inference faces unique physical access threats (BarraCUDA EM side-channel).
- [ ] **How will post-quantum cryptography affect model signing and TEE attestation chains?** — OpenSSF Model Signing v1.0 and Sigstore currently use classical cryptographic primitives. As PQC standards finalize, the model signing and TEE attestation ecosystems will need migration paths.
- [ ] **Can GPU telemetry anomaly detection reliably identify side-channel attacks?** — NVIDIA DCGM exports power, temperature, and error counters, but no published production system uses this telemetry specifically for side-channel detection (C4.2.8). Kubernetes DRA is now stable in v1.35 and the NVIDIA DRA driver is moving into CNCF/community ownership, which gives operators richer allocation metadata; the detection models that turn that metadata into reliable side-channel alerts remain an open gap.
- [ ] **How should organizations defend against supply chain attacks that compromise security tooling itself?** — The TeamPCP campaign (March 2026) showed that attackers can poison the very scanners and CI/CD actions that enforce supply chain integrity. Pinning GitHub Actions by commit SHA rather than mutable tags is a minimum countable defense, but the broader question — how to maintain trust in the verification chain when the verifiers are compromised — needs architectural answers beyond current SLSA and Sigstore capabilities.
- [x] **When will Confidential Containers (CoCo) with GPU support reach production readiness?** — It happened in spring 2026. NVIDIA's Confidential Containers Reference Architecture 1.0.0 is GA: H100 single-GPU passthrough and H200 Protected PCIe multi-GPU passthrough on AMD SEV-SNP or Intel TDX, with Kata, GPU Operator 26.3 (which makes Kata/CoCo a supported deployment and deprecates the standalone Kata Manager), and Trustee/NRAS composite attestation gating key release. CoCo v0.21.0 (May 29, 2026) added gRPC attestation plugins and Hopper-based CI; Red Hat documented bare-metal OpenShift CoCo with combined CPU+GPU attestation in June 2026. Remaining constraints: every GPU on a host must run in CC mode (no mixed-mode nodes), the supported hardware matrix is narrow (H100/H200), and Blackwell-generation confidential computing is only in cloud preview (Google Confidential G4 VMs, April 2026). C4.1.5–C4.1.7 now have a turnkey-adjacent Kubernetes answer, at the cost of hardware homogeneity.
- [ ] **How should inference servers themselves be hardened as part of C4.1?** — ShadowMQ (Nov 2025), the Triton RCE chain (Aug 2025), KubeAI CVE-2026-34940 (Apr 2026), LMDeploy CVE-2026-33626 (Apr 2026, exploited in 12h31m), the SGLang ZMQ pickle/dill RCEs (CVE-2026-7301/7304, May 2026), and the Triton authentication-bypass bulletin (CVE-2026-24207, May 20, 2026) showed that the biggest C4.1 gap in practice is not just running untrusted user code — it is the inference servers, AI gateways, and model-serving operators that load pickle-over-ZMQ, expose management endpoints without auth, fetch attacker-supplied URLs as a feature, turn model URLs into shell probes, or ship with `--enable-custom-logit-processor` deserializing dill objects from the network. Pwn2Own Berlin 2025 made AI infrastructure a first-class category, and Trend Micro projects 2,800–3,600 AI CVEs in 2026 concentrated in model-serving stacks. Current requirements (particularly 4.1.3) cover the model loader, but the inference server's own IPC, admin, control plane, Kubernetes operator surfaces, and user-supplied URL fetchers need explicit guidance — network-policy defaults, authenticated-only IPC, pickle-free deserialization inside the server process, CRD admission controls, shell-free probe construction, and SSRF egress filtering — not just around it. June 2026 added two more surfaces to that list: framework-native authentication middleware (vLLM CVE-2026-48746's Host-header bypass) and gateway configuration APIs (LiteLLM CVE-2026-42271, the first AI-gateway flaw in CISA KEV with confirmed in-the-wild exploitation).
