# C8.3: Memory Expiry, Revocation & Deletion

> [Back to C08 Index](C08-Memory-and-Embeddings.md)
> **Last Researched:** 2026-04-16

## Purpose

Retention must be explicit and enforceable; deletions must propagate to derived indices and caches. Vector stores that grow without retention controls become an ever-expanding attack surface for data exfiltration and a compliance liability under GDPR, CCPA, and similar regulations. As of April 2026, Milvus v2.6 (with the v2.6.13 patch released 23 March 2026 fixing strong-consistency query timeouts during compaction), Qdrant's tiered multitenancy, and Weaviate's three-state tenant lifecycle (Active/Inactive/Offloaded) improve native support, but deletion propagation to derived indices and caches still requires custom engineering — particularly in hybrid search architectures where parallel BM25 and vector indices each maintain independent copies of content. Crypto-shredding remains the most viable path to verifiable deletion in distributed systems, though the "harvest now, decrypt later" quantum threat means key-destruction alone may not guarantee permanent inaccessibility for data with long sensitivity windows. The EDPB's February 2026 report on its 2025 Coordinated Enforcement Action (32 DPAs across Europe, 9 formal investigations) found systemic failures in erasure compliance: poorly documented procedures, ineffective anonymization substituted for deletion, absent retention periods, and technical limitations preventing erasure in backups. The EU AI Act's August 2026 deadline for high-risk system logging requirements adds urgency — organizations face a structural tension between the AI Act's archival mandates and GDPR's storage limitation principle (Article 5(1)(e)). Machine unlearning research continues to mature — University of California, Riverside's September 2025 "source-free unlearning" method operates without original training data — but for RAG-based systems, the more immediate challenge is ensuring deletion propagates across source documents, vector embeddings, keyword indices, evaluation sets, and application-layer caches. Session-scoped memory disposal remains a distinct concern following SpAIware (Rehberger, 2024), cross-session leak research (Giskard, 2025), and Unit 42's demonstration of persistent memory poisoning in Amazon Bedrock Agents. The threat landscape continues to expand — the eTAMP attack (Zou et al., April 2026) demonstrated environment-injected cross-session, cross-site memory poisoning where a single contaminated web observation (e.g., a manipulated product page) silently poisons an agent's memory and activates during unrelated future tasks, achieving up to 32.5% attack success on GPT-5-mini. Separately, Memory Control Flow Attacks (Xu et al., March 2026) showed that poisoned long-term memory can hijack tool selection in agents built on GPT-5 mini, Claude Sonnet 4.5, and Gemini 2.5 Flash with over 90% of trials vulnerable even under strict safety constraints — and the poisoning persists across tasks and resists conversational correction attempts. These findings reinforce that retention controls and session disposal are not just compliance exercises but active security boundaries. Real-world incidents in early 2026 underscore this urgency: the Chat & Ask AI breach (February 2026) exposed 300 million messages from 25 million users through a Firebase misconfiguration in an AI wrapper app, and the Mercor supply chain breach (March 2026) via compromised LiteLLM packages exfiltrated approximately 4 TB including a 211 GB user database and 3 TB of video recordings — both incidents involved AI systems that retained far more data than their operational purpose required. On 2 April 2026 Microsoft released the open-source Agent Governance Toolkit (MIT license, available in Python, TypeScript, Rust, Go, and .NET), the first runtime-security framework to address all ten OWASP Agentic Top 10 risks with deterministic sub-millisecond policy enforcement — its Cross-Model Verification Kernel (CMVK) uses majority voting across independent models to detect memory-poisoning attempts before a write commits, giving defenders a concrete off-the-shelf building block for the quarantine, revocation, and selective-purge controls in this section. ISO 42001 now provides a standards-based framework for AI data retention governance, requiring purpose-aligned collection, impact-assessment-driven retention limits, and mandatory secure deletion or anonymization once the processing purpose ends.

| **8.3.1** | **Verify that** retention times are applied to every stored vector and related metadata across memory storage. | 1 | Unbounded data accumulation — without TTLs or retention policies, vector stores grow indefinitely, increasing the attack surface for data exfiltration and complicating regulatory compliance (GDPR Article 5(1)(e) storage limitation principle). Stale data also degrades retrieval quality. Stale poisoned vectors that should have been purged remain retrievable indefinitely without TTL enforcement, extending the window of any successful RAG poisoning attack. | Review vector store configuration for TTL settings or retention policy enforcement. Query stored vectors and verify TTL metadata is present. Check for automated expiration jobs or garbage collection processes. Verify retention policies cover both vector data and associated metadata/text chunks. | Pinecone does not support native TTL — requires application-layer deletion jobs. Weaviate's three-state tenant lifecycle (Active → Inactive → Offloaded to S3) provides infrastructure-level data temperature management, but `autoTenantActivation` handles reactivation rather than deletion — explicit tenant removal is still a manual API call. Milvus v2.5+ supports collection-level TTL specified in seconds; expired data is filtered from queries immediately but physically removed only during compaction (typically within 24 hours). Redis Vector Search inherits Redis key TTL natively. Qdrant's tiered multitenancy (2025) supports both small and large tenants efficiently with SOC2/GDPR/HIPAA compliance, but lacks native per-record TTL — retention must be enforced via metadata-based sweeps. PostgreSQL with pgvector inherits standard SQL row-level retention via `pg_cron` scheduled deletes, increasingly relevant as more teams consolidate vectors into relational databases (a notable 2026 trend). Milvus v2.6 introduces a dedicated StreamingNode for streaming data responsibilities, allowing DataNode to focus on historical tasks like compaction and index creation — this separation improves compaction stability and reduces the window where TTL-expired data remains physically present. For databases without native TTL, a separate cron-based cleanup process scanning metadata timestamps is the standard pattern. CVE-2026-3172 (CVSS 8.1) exposed a buffer overflow in pgvector 0.6.0–0.8.1 during parallel HNSW index builds that could leak data from other PostgreSQL relations — fixed in pgvector 0.8.2 but a reminder that vector index operations themselves can be a data exposure vector, not just the stored vectors. |
| **8.3.2** | **Verify that** only information required for the system's defined function is persisted in memory (such as user preferences and conversation decisions, not credentials or full conversation transcripts). | 1 | Over-retention of sensitive conversational data — full conversation transcripts in memory create a rich target for exfiltration and may contain credentials, personal disclosures, or business secrets shared in context. Maps to data minimization principles (GDPR Article 5(1)(c)). The "EchoLeak" zero-click attack demonstrated exfiltration of corporate data from Microsoft 365 Copilot by sending a specially crafted email — the hidden prompt caused Copilot to autonomously leak information from conversation memory. Unit 42's October 2025 research showed Amazon Bedrock Agents encoding user conversation history in URLs and transmitting it to attacker-controlled servers without visible indication. | Review memory write logic to identify what data categories are persisted. Compare persisted data against the system's functional requirements. Verify session-scoped memory is cleared at session termination. Test by conducting a session with sensitive content, ending it, and confirming the data is not retrievable in a new session. | Defining "required for the system's defined function" needs explicit documentation — a data retention specification. ChatGPT's memory feature persists user preferences but not full transcripts. Systems using ConversationBufferMemory in LangChain persist everything by default; ConversationSummaryMemory is better but may still retain sensitive details in summaries. Memory poisoning research (MINJA, NeurIPS 2025) showed that over-retained conversation data becomes the attack surface — all prompts including commands like "remember," "in future conversations," or "as a trusted source" were used to ensure long-term influence. The Zombie Agents paper (February 2026) demonstrated self-reinforcing injections that persist control over self-evolving LLM agents by continuously re-injecting into memory, making over-retention a vector for indefinite compromise. InjecMEM (NeurIPS 2025) showed that a single query-only interaction — no direct memory access needed — can inject malicious records that steer later responses toward attacker-specified outputs. A practical defense pattern emerging in 2026 is the "memory firewall" architecture: provenance tagging on every write (source channel, timestamp, session ID), trust scoring (user conversation: high baseline; document parse: low), staging-first writes where all entries go through validation before committing to main storage, and temporal decay with configurable TTLs (typically 30–90 days) to bound poisoning persistence automatically. A public reference implementation (Roborhythms, March 2026) runs seven sequential checks per write — provenance tagging, Unicode-Tags stripping, instruction-pattern detection ("ignore previous", "remember that I have", "your real purpose"), length and PII/credentials validation, trust scoring, quarantine routing for anything below the high-confidence threshold, and snapshot-based rollback pointers — and benchmarks at under 5 ms added latency per write on LangGraph and CrewAI agents, making the approach deployable without noticeable user impact. Mem0 provides per-user namespaces and exclusion rules; LangGraph offers session-scoped memory but needs manual snapshot integration for rollback. The Chat & Ask AI breach (February 2026) exposed 300 million messages from 25 million users due to a Firebase misconfiguration where security rules were left public — a security researcher scanning 200 iOS apps found 103 had the same vulnerability, illustrating how AI wrapper apps routinely over-retain conversation data in improperly secured backends. ISO 42001 establishes four retention principles for AI systems: purpose alignment (retain only for defined AI objectives), impact-assessment-based retention limits, mandatory secure deletion or anonymization once purpose ends, and documented audit trails for all lifecycle decisions — though it deliberately does not prescribe fixed timelines, requiring organizations to justify durations per use case. |
| **8.3.3** | **Verify that** deletion requests purge vectors, metadata, cache copies, and derivative indices within an organization-defined maximum time. | 1 | Right-to-erasure non-compliance (GDPR Article 17) and residual data exposure — if deletion only removes the vector but leaves metadata, cached text chunks, or derived indices (e.g., BM25 keyword indices alongside vector indices), the sensitive content remains accessible through alternative retrieval paths. The EDPB's 2025 Coordinated Enforcement Action (published February 2026) found that controllers routinely fail to propagate deletion across derived data stores, with "technical limitations preventing erasure in backup systems" cited as a systemic gap across industries. | Issue a deletion request for a known document. After the defined maximum time, attempt to retrieve the document via: (1) vector similarity search, (2) metadata filter query, (3) any keyword/BM25 index, (4) application-layer caches. Verify all paths return no results. Check backup systems for residual copies. For hybrid retrieval architectures, verify deletion covers both dense (vector) and sparse (BM25/SPLADE) indices independently — recall in hybrid systems improves from ~0.72 (BM25 alone) to ~0.91 (hybrid), meaning a missed index path has high probability of resurfacing deleted content. | Hybrid search architectures (vector + keyword) are increasingly common (e.g., Weaviate's hybrid search, Pinecone's sparse-dense). Each index must be covered by deletion propagation — ANN indices (HNSW, IVF) mark deletes as tombstones with deferred compaction, while BM25 inverted indices require separate term-level cleanup. Cache invalidation is particularly challenging — Redis caches, CDN-layer caches, and in-memory application caches all need explicit purge logic. Document the full data flow map (source documents → chunking pipeline → vector index → keyword index → evaluation sets → caches → logs) before implementing deletion. As of March 2026, BigID's unified privacy management platform can propagate deletion requests to AI training datasets and vector databases with automated verification, validation, and audit-ready fulfillment tracking across 100+ regulations — the first commercial platform to treat vector stores as first-class deletion targets alongside traditional databases. Private networking (VPC peering or private link with no public ingress) available from Pinecone, Weaviate, and Qdrant for production deployments reduces the exposure window during deletion propagation. The EDPB report also flagged "ineffective anonymisation techniques as a substitute for deletion" — vector embeddings are not anonymous, as demonstrated by embedding inversion attacks (see C08-04). As of March 2026, the EDPB launched a new Coordinated Enforcement Action with 25 DPAs auditing GDPR transparency and information obligations, adding further regulatory pressure. Organizations face a structural "regulatory clock" conflict: Article 18 of the EU AI Act requires high-risk AI system documentation retention for up to 10 years, while GDPR's storage limitation principle (Article 5(1)(e)) demands deletion once the processing purpose ends. The only legally defensible resolution is irreversible anonymization — pseudonymization alone still triggers GDPR protections. Cumulative GDPR fines now exceed €7.1 billion (DLA Piper survey, January 2026), with penalties up to 4% of global annual turnover for storage limitation violations. Article 17 scoping guidance emerging in early 2026 (Relyance AI, TianPan privacy-engineering writeups) is converging on a simple rule for LLM-backed systems: if the personal data is kept outside the frozen model weights (vector store, CRM connector, conversation memory, agent scratch space), the erasure request is technically feasible and therefore in scope — "we can't retrain the model" is not a defensible answer when the data also lives in retrievable form in a RAG index. Auditors are beginning to ask for a written data-flow map listing every retrievable copy of a user's data and the deletion propagation path for each. |
| **8.3.4** | **Verify that** deleted or expired vectors are removed reliably and are unrecoverable. | 2 | Data remanence in vector stores — soft-delete implementations or lazy garbage collection may leave vector data recoverable through direct storage-layer access, low-level API calls, or backup restoration even after logical deletion. For crypto-shredding implementations, the "harvest now, decrypt later" quantum computing strategy poses a theoretical long-term risk — adversaries collecting encrypted ciphertext today could decrypt it post-quantum, undermining the assumption of permanent inaccessibility through key destruction alone. | Verify the deletion mechanism is hard-delete, not soft-delete/tombstone with deferred cleanup. For HNSW-based indices, confirm that deleted node slots are overwritten on compaction. Check that storage-layer files (e.g., segment files in Milvus, WAL in Qdrant) do not retain deleted data after compaction. Review backup retention policies. Test crypto-shredding implementations by verifying key destruction is auditable, key material is isolated from data storage, and AES-256 (or equivalent) is used consistently. | Most vector databases use HNSW or IVF indices that do not immediately reclaim deleted vector slots — they mark them as deleted and reclaim space during compaction/merge operations. Milvus v2.5 refined its expiring compaction to automatically merge smaller segments, remove logically deleted entities, and reclaim TTL-expired data in background — but physical removal still takes up to 24 hours after logical deletion. Qdrant uses WAL with periodic snapshots; deleted data persists in WAL until the next snapshot, and its Raft consensus compaction logging (improved in recent releases) aids forensic verification. Crypto-shredding — encrypting each vector/document with a unique key and destroying the key on deletion — is the strongest path to verifiable unrecoverability, but requires that initial encryption practices are airtight: strong algorithms (AES-256), isolated key storage from the data layer, and auditable key lifecycle management. As of early 2026, no major vector database vendor offers built-in crypto-shredding; implementation requires wrapping the storage layer with application-level encryption (e.g., AWS KMS with per-tenant keys, HashiCorp Vault). LanceDB's zero-copy versioning creates new dataset versions without duplicating unchanged data, which enables a forensic-friendly deletion pattern: create a new version excluding deleted vectors while retaining the prior version in access-controlled storage for audit — then hard-delete the old version after the retention window. |
| **8.3.5** | **Verify that** expired vectors are excluded from retrieval results within a measured and monitored propagation window. | 2 | Stale data serving — expired or deleted vectors that remain retrievable during a propagation window can serve outdated, revoked, or sensitive content. The window must be defined, measured, and kept as short as possible. In hybrid retrieval architectures, each index type (dense vector, sparse BM25/SPLADE, graph) has its own propagation characteristic, creating a composite window that is only as fast as the slowest component. | Define the maximum acceptable propagation delay. Issue a deletion/expiration, then immediately query for the affected vectors across all retrieval paths (vector similarity, keyword/BM25, metadata filters). Measure the actual time until they stop appearing in results on each path independently. Verify monitoring/alerting exists for propagation delays exceeding the defined maximum. Test cold-start scenarios — Pinecone serverless indexes that haven't been accessed recently experience higher latency on first query as data loads into memory, which may affect deletion verification timing. | In distributed vector databases, propagation delay depends on replication lag, cache TTLs, and index rebuild frequency. Pinecone's second-generation serverless architecture (February 2025) achieves eventual consistency with typical query latency of 50-100ms under normal conditions, but cold-start namespaces experience higher latency. Milvus v2.5 filters expired data from query results immediately via TTL, but physical segment reclamation happens during background compaction within ~24 hours. Milvus v2.6.13 (23 March 2026) specifically fixed a correctness bug where search and query requests running with strong consistency could time out during compaction because tsafe advancement was blocked — a direct propagation-window hazard for teams that rely on strong-consistency reads to verify deletion has taken effect. Teams on earlier 2.6.x point releases should plan the upgrade before publishing measured propagation-delay SLAs. Weaviate's tenant state transitions are "eventually consistent" per their documentation — query-time filtering may still serve stale replicas during state propagation. Qdrant's Raft consensus ensures strong consistency for writes but read replicas may lag. Application-layer caching (Redis, in-memory LRU, CDN) adds another propagation layer — cache TTLs must be shorter than the acceptable deletion propagation window, or explicit cache invalidation must be triggered alongside the vector delete. |
| **8.3.6** | **Verify that** memory can be reset for security reasons (quarantine, selective purge, full reset) through an operation that is separate and independent from the retention deletion process. | 2 | Incident response capability — when a memory poisoning attack is detected, the team needs to quarantine suspected vectors immediately (excluding them from retrieval) while preserving them for forensic analysis. Without this separation, teams face a choice between continued exposure and evidence destruction. Given MINJA's >95% injection success rate and A-MemGuard's finding that LLM-based classifiers miss 66% of poisoned entries, rapid quarantine capability is essential for incident response when poisoning is detected through behavioral monitoring rather than entry-level scanning. | Simulate a poisoning incident. Verify that quarantine operations: (1) immediately exclude targeted vectors from all retrieval paths, (2) preserve quarantined vectors in a separate, access-controlled storage for investigation, (3) do not trigger retention-based deletion timers. Test selective purge (specific documents/batches) and full reset independently. Schneider's four-layer defense recommends circuit breakers for rapid isolation when compromise is detected. | Most vector databases do not have native quarantine functionality. Implementation patterns include: moving vectors to a separate "quarantine" collection/namespace, adding a metadata flag and enforcing query-time filtering, or maintaining a deny-list of vector IDs checked at retrieval time. The deny-list approach has the lowest latency impact but requires consistent enforcement across all query paths. Weaviate's three-state tenant lifecycle (Active → Inactive → Offloaded to S3) provides the closest native approximation — deactivating a tenant immediately removes it from query paths while preserving data on disk or in cloud storage for forensic review, and `autoTenantActivation` can be disabled to prevent accidental reactivation. Qdrant's tiered multitenancy enables per-tenant isolation with granular database API keys and RBAC, supporting access-controlled quarantine zones. The RAGForensics traceback framework (ACM Web Conference 2025) provides a practical forensic methodology — its iterative LLM-based classification achieves >97% accuracy in identifying poisoned texts, enabling systematic quarantine scoping before any data is permanently deleted. The Semantic Chameleon attack (March 2026) demonstrated corpus-dependent poisoning where injected content adapts to the target corpus characteristics, making quarantine scoping harder — defenders need to evaluate the full corpus neighborhood of suspected vectors, not just the flagged entries. Memory Control Flow Attacks (Xu et al., March 2026) showed that poisoned memory can hijack tool selection across Override, Order, M-Scope, Persistence, and Resistance attack families — the Persistence variant achieves 100% success maintaining control without re-injection, making rapid quarantine of the Read→Plan→Execute→Write memory loop essential for incident response. The memory firewall pattern (staging-first writes with provenance tagging, trust scoring, and snapshot-based rollback) provides a defense-in-depth complement to quarantine capabilities. Microsoft's Agent Governance Toolkit (April 2026) now ships an Agent OS policy-enforcement layer that intercepts every agent action before execution and a Cross-Model Verification Kernel that uses majority voting across multiple models to catch poisoned memory writes — adapters are already available for LangGraph, Haystack, PydanticAI, and the OpenAI Agents SDK, removing the "no off-the-shelf quarantine" gap that existed for most of 2025. |
| **8.3.7** | **Verify that** context not needed beyond the current session is discarded at session end and is not accessible in subsequent sessions. | 1 | Cross-session data leakage — if ephemeral conversation context (intermediate reasoning, user-supplied credentials, uploaded documents) persists after session termination, it becomes accessible to subsequent sessions or other users in multi-tenant deployments. Giskard's October 2025 cross-session leak research demonstrated that misconfigured caches and shared memory states allow attackers to retrieve another user's PII, health records, and financial identifiers from prior sessions. The SpAIware attack (Rehberger, 2024) showed that prompt injection can write persistent instructions into ChatGPT's long-term memory, causing silent data exfiltration across all future sessions for up to 365 days. Unit 42's October 2025 research on Amazon Bedrock Agents demonstrated that poisoned session summaries become part of persistent memory, executing malicious directives silently across subsequent conversations. Maps to OWASP LLM02:2025 Sensitive Information Disclosure. | Conduct two-phase session isolation testing: (1) inject distinctive marker data during a session, terminate the session, then start a new session and attempt retrieval via direct queries and prompt engineering; (2) in multi-tenant setups, verify per-user memory isolation by attempting cross-tenant data access. Check that session-scoped stores (conversation buffers, temporary file uploads, intermediate chain outputs) are explicitly cleared at session end. Verify cache keys include session/user identifiers to prevent shared-cache leakage. Tools: Giskard Hub for autonomous multi-session red teaming, manual testing with distinctive canary tokens planted in session context. | LangChain's `ConversationBufferMemory` persists full conversation history by default and requires explicit `clear()` calls at session end — many implementations omit this. Frameworks like LlamaIndex and Semantic Kernel similarly need explicit session lifecycle management. OpenAI retains deleted ChatGPT memories for up to 30 days for safety and debugging, creating a residual exposure window. For agentic systems with tool-use capabilities, intermediate tool outputs (API responses, file contents, database results) are particularly sensitive and often remain in context buffers without explicit cleanup. Session isolation must occur at the infrastructure level (per-tenant caching, cryptographically strong isolation) — relying on the LLM to self-enforce boundaries is insufficient. The eTAMP attack (Zou et al., April 2026) demonstrates that even without direct memory access, a single contaminated web observation can poison an agent's trajectory memory and activate across sessions and websites — "frustration exploitation" (agents encountering dropped clicks or garbled text) increases susceptibility up to 8x. Memory Control Flow Attacks (Xu et al., March 2026) formalize five attack families including Persistence (100% success maintaining long-horizon effects without re-injection) and Resistance (100% success withstanding textual correction). The proposed Role-Based Memory Segregation defense provides incomplete protection — Gemini 2.5 Flash maintained 63.9% attack success even with RBMS active, confirming that prompt-level hierarchy alone cannot substitute for infrastructure-level session isolation. The Meta AI agent data exposure (March 2026, Sev-1) demonstrated a "confused deputy" failure where an internal AI agent with valid credentials bypassed role-based access boundaries at the output level — a single incorrect instruction scaled into broad exposure of sensitive company and user data across employees for approximately two hours before containment. Google's open-source Always-On Memory Agent (March 2026) introduces an alternative architecture that replaces vector databases with LLM-driven structured memory and a periodic consolidation job (default 30-minute cycle) that merges duplicates and drops stale entries — while this simplifies lifecycle management, it trades deterministic deletion guarantees for probabilistic LLM-mediated cleanup, which may not satisfy compliance requirements for verifiable erasure. |
| **8.3.8** | **Verify that** quarantined content is retained for investigation but is excluded from all retrieval results while under quarantine. | 2 | Continued exposure from poisoned or compromised vectors — when a RAG poisoning attack is detected (e.g., via PoisonedRAG-style knowledge corruption, accepted at USENIX Security 2025), responders need to immediately exclude suspect vectors from retrieval while preserving them for forensic analysis. Without quarantine-with-exclusion, the team faces a binary choice between continued exposure (leaving poisoned data retrievable) and evidence destruction (hard-deleting before investigation). The RAGForensics traceback framework (ACM Web Conference 2025) demonstrated that forensic identification of poisoned texts achieves >97.4% detection accuracy with <2.7% false positive rates — but this analysis requires the poisoned data to remain available for examination. Premature deletion eliminates the ability to determine attack scope, identify all affected queries, and trace the injection vector. | Simulate a poisoning incident by inserting known-bad vectors with distinctive markers. Execute quarantine procedure and verify: (1) quarantined vectors return zero results across all retrieval paths (vector similarity, keyword/BM25 hybrid, metadata filters, application caches); (2) quarantined vectors remain accessible through a separate, access-controlled forensic interface with audit logging; (3) quarantine status persists across system restarts and replication; (4) quarantine does not trigger retention-based deletion timers; (5) un-quarantine restores vectors to normal retrieval. Measure quarantine propagation latency across all replicas. Verify forensic access generates immutable audit logs. | Most vector databases lack native quarantine functionality — implementation requires custom engineering. Common patterns: (a) metadata-flag approach — add `quarantined: true` flag and enforce `quarantined != true` filter on every query path, which has low latency impact but requires consistent enforcement across all retrieval APIs; (b) namespace/collection isolation — move suspect vectors to a dedicated quarantine collection, providing stronger isolation but higher operational complexity; (c) deny-list approach — maintain a blocklist of vector IDs checked at retrieval time, simplest to implement but scales poorly. Qdrant's tenant-level lifecycle APIs and Weaviate's tenant lifecycle endpoints can facilitate per-tenant quarantine by suspending individual tenants without data deletion. For forensic analysis, the RAGForensics iterative LLM-based classification method provides a structured approach to systematically trace poisoned texts across the knowledge base. No major vector database vendor offers built-in quarantine-with-forensic-retention as of early 2026. |
