# C9.2: High-Impact Action Approval and Irreversibility Controls

[Back to C09 Index](C09-Orchestration-and-Agents.md)

## Purpose

Agents acting autonomously may perform actions with significant real-world consequences -- deploying code, transferring funds, deleting data, sending communications. This section requires that high-impact or irreversible actions pass through explicit human approval checkpoints. The approval mechanism must be tamper-resistant: the human must see exactly what will happen, and the system must execute exactly what was approved.

## 2024-2026 Research Context

Human-in-the-loop (HITL) approval has matured from an ad-hoc practice into a first-class architectural pattern for agentic AI systems. By 2025-2026, multiple frameworks offer native support for structured approval gates, and industry consensus has converged on risk-based action classification as the foundation for approval routing. As of early 2026, roughly 70% of organizations run a human-in-the-loop model where the agent recommends and a person approves, with only 14% allowing fully autonomous remediation. Gartner's March 2026 forecast projects that 40% of enterprise applications will embed agentic capabilities by the end of 2026 (up from 12% in 2025), which means the surface area for approval-gated actions is expanding rapidly across every industry.

Gartner also now distinguishes between two oversight postures that are often conflated: **human-in-the-loop** (human approval required per action) and **human-on-the-loop** (AI acts with broader latitude, human supervises at the strategic level, not per-tactical-decision). Choosing between them is an explicit risk-tolerance decision, and C9.2 squarely addresses the first mode for privileged or irreversible actions.

### Approval Workflow Patterns

Modern HITL implementations follow a five-step control loop: (1) agent receives task, (2) agent proposes action with full parameters, (3) agent pauses and routes the request to a human approver, (4) human reviews context and approves or rejects, (5) agent resumes only if approval is granted. Two primary patterns have emerged:

- **User Confirmation (synchronous):** A straightforward boolean validation where the agent pauses execution via an `interrupt()` call and waits for a yes/no decision. Suited for real-time, high-stakes decisions. LangGraph's native `interrupt()` function is the canonical implementation, pausing the graph mid-execution and resuming cleanly after human input.
- **Asynchronous Authorization:** The agent issues an authorization request and continues non-blocking work while waiting for a response routed through channels like Slack, email, or dedicated approval UIs. The HumanLayer SDK provides decorators like `@require_approval()` that wrap functions with multi-channel notification logic. This pattern is suited for lower-priority flows or when the approver is not immediately available.

The OWASP AI Agent Security Cheat Sheet codifies a `HumanInTheLoopController` pattern that queues sensitive actions for review with unique action IDs, generates readable previews showing sanitized parameters and risk levels, and returns a `pending_confirmation` status until user consent is obtained.

**Gaps in framework-native support:** A recurring caveat in 2026 production write-ups is that the popular checkpoint/interrupt primitives are not, by themselves, an approval workflow. LangGraph's `interrupt()` pauses the graph and persists state to the checkpointer, but nothing else happens: no notification is sent, no timeout fires, and no backup approver is escalated to. Interrupted threads sit in the checkpointer indefinitely unless the application adds its own notification, TTL (see 9.2.5), and escalation logic. A March 2026 Diagrid analysis titled "Checkpoints Are Not Durable Execution" argues that LangGraph, CrewAI, and Google ADK all share this gap: checkpointing captures state, but production-grade approval workflows also need durable timers, retries, and compensations — which still require an external orchestrator (Temporal, Diagrid, or equivalent) or custom glue code.

### Risk Classification for Actions

The industry has converged on a four-tier risk classification for determining whether an action requires human review:

| Risk Tier | Examples | Approval Policy |
|---|---|---|
| **Low** | Read operations, status queries | Auto-approved |
| **Medium** | Write operations, API calls | May auto-approve based on threshold settings |
| **High** | Financial transfers, external communications | Requires human approval |
| **Critical** | Irreversible deletes, security config changes | Mandatory human review |

The four decision dimensions remain: **irreversibility** (can the action be undone?), **blast radius** (how many systems/users are affected?), **compliance exposure** (does the action touch regulated data or processes?), and **confidence** (how certain is the agent about the action's correctness?). A March 2026 Security Boulevard analysis recommends a phased "trust ladder" approach -- Suggest, then Assist, then Execute with guardrails -- starting with reversible actions like session revocation or endpoint quarantine before graduating to irreversible ones.

Over-classification leads to approval fatigue; under-classification leads to missed dangerous actions. As of early 2026, 79% of enterprises report blindspots where agents invoke tools, touch data, or trigger actions that the security team cannot fully observe. Organizations must calibrate thresholds to their risk tolerance.

### Governance Layers and Adaptive Trust

UC Berkeley's ATLAS (Adaptive Trust Layer for Autonomous Systems) project, published in early 2026, implements a governance layer that intercepts agent decisions before execution and evaluates whether human authorization is required. It uses a two-component architecture: a customer-environment agent that gathers information and generates case proposals, and a control layer that evaluates proposed actions against applicable policy and risk criteria. Actions affecting safety, health, or fundamental rights -- aligned with EU AI Act Article 14 and NIST AI RMF -- route to human decision-makers, while clearly low-risk cases follow an automated pathway.

This governance-first approach contrasts with purely algorithmic trust scoring. The key insight is that meaningful human oversight requires centralized policy evaluation, not just threshold-based automation.

### Transactional Semantics and Rollback

A December 2025 MarkTechPost article described transactional agentic AI systems using LangGraph with two-phase commit patterns: reversible changes are staged, strict invariants are validated, the graph pauses for human approval via `interrupt()`, and execution either commits or rolls back. This maps directly to requirement 9.2.4's call for compensating actions and saga-pattern rollback, adapting distributed-systems transaction theory to agentic workflows.

The rollback tooling landscape has expanded significantly in 2025-2026:

- **Rubrik Agent Rewind** (August 2025): Snapshots monitored environments and can roll back unwanted agent-led changes across files, databases, configurations, and code repositories. Marketed as "rogue AI agent action rewind protection."
- **Cohesity + ServiceNow + Datadog** (announced March 2026): A joint recoverability service preserving immutable snapshots of AI environments with point-in-time recovery across files, databases, object storage, SaaS applications, vector stores, and agent memory.
- **Cisco**: Built native rollback capabilities into its agentic tools.
- **Railway delayed deletes and staged agent surfaces** (April 2026): After the PocketOS deletion incident, Railway extended its 48-hour soft-delete behavior from the dashboard to the API, added delayed deletion for backups, reassessed token scoping, and pointed agents toward safer surfaces such as the Railway Agent, CLI, and Remote MCP Server. The important design lesson is that rollback and approval controls must cover the raw API path, not just the human-facing console.

However, one-way transactions (bank transfers, sent emails, published content) remain fundamentally irreversible. The March 2026 IT Brew analysis notes that actions "outside your four walls of control" cannot be unwound, which is why pre-execution approval gates remain the primary defense for truly irreversible actions.

### Pre-Execution Interception and Destructive-Action Proxies

The PocketOS and Kiro incidents converged the 2026 design conversation on a specific pattern: an enforcement layer in front of destructive APIs that the agent cannot route around. The April 28, 2026 Dev Journal "Nine Seconds to Zero" write-up coined the **destructive-action proxy** label for it -- a middleware gateway that wraps calls like `volume.delete`, `backup.destroy`, `DELETE` without a `WHERE` clause, or any CloudFormation stack teardown, holds them for human approval, and refuses to forward them otherwise. AgentSteer's analysis of the Kiro incident reaches the same conclusion under the name **pre-execution interception**: "a guardrail that sees 'delete production CloudFormation stack' and blocks it, regardless of whether the IAM policy allows it." Permission boundaries assume proportionality judgment that an agent does not have, so the policy decision needs to live at a layer the agent cannot reason its way past.

Several concrete techniques are converging:

- **Out-of-band confirmation tokens.** The proxy generates a random string (e.g., `delete-volume-abc123`), sends it to the approver via a channel the agent cannot read (Slack DM, SMS, email to a non-agent address), and refuses to execute until the human pastes the exact string back. This directly counters the inverse-failure mode from the March 2026 Meta incident, where the agent persuaded the human to bypass controls: the agent has nothing to persuade with because it never sees the token.
- **Deny-by-default ABAC at the infrastructure layer.** AWS IAM `Deny` statements on resources tagged `backup=true` or `production=true` reject `volumeDelete`, `Delete*`, and `Terminate*` calls regardless of which identity invoked them. Cedar policy in Bedrock AgentCore and OPA policy in Permit MCP Gateway provide the same enforcement for tool calls and MCP servers.
- **Allowlist-by-default for tool surfaces.** Read tools (`GET`, status queries, list operations) are auto-approved; write tools require `approvalStatus=approved`; destructive operations are denied unless wrapped through the proxy. This matches AgentCore Policy's three-tier pattern and the InfoQ "least-privilege AI agent gateway" reference architecture combining MCP, OPA, and ephemeral runners.
- **Delayed-destructive APIs.** Railway's 48-hour soft-delete for volumes (extended from the dashboard to the API after PocketOS), Cohesity/ServiceNow/Datadog's point-in-time recovery service, and 5-minute hold windows recommended in the AGAT and Undercode write-ups give the human approver real recovery time even when the agent succeeds at calling a destructive endpoint.
- **MCP tool annotations.** Recent MCP specification revisions added read-only / destructive hints on tool definitions. These are advisory rather than tamper-resistant -- a malicious or misconfigured server can lie -- but they let gateways like Permit MCP Gateway, ToolHive, and the Diagrid Catalyst MCP proxy escalate to step-up flows for any tool that declares itself destructive.
- **Per-action approval binding, not per-wrapper approval.** OpenClaw's March 2026 CVE-2026-29607 ("Allow-Always Wrapper Bypass," CVSS 6.4) is the canonical bait-and-switch demonstration in the agent ecosystem: an administrator approves a safe-looking wrapped command once, but the approval persists at the wrapper level rather than binding to the canonical inner command. An attacker (or a manipulated agent) can swap the inner payload later and obtain RCE without re-prompting the human. The OpenClaw "Nine CVEs in Four Days" cluster (March 2026) included two adjacent bypass primitives that compound the risk: CVE-2026-28460 (allowlist bypass using shell line-continuation characters to access blocked commands through a different parser path) and CVE-2026-22172 (a critical 9.9 WebSocket scope self-declaration flaw letting authenticated users declare admin scopes, undermining any downstream action-approval mechanism). A second OpenClaw cluster -- the May 15, 2026 "Claw Chain" -- extends the same failure pattern with four new CVEs that compose into an end-to-end sandbox escape: CVE-2026-44112 (CVSS 9.6, TOCTOU race in the OpenShell sandbox that lets a tool write outside the intended mount root and install a persistent backdoor), CVE-2026-44113 (CVSS 7.7, the read-side TOCTOU race that exfiltrates credentials from outside the mount root), CVE-2026-44115 (CVSS 8.8, incomplete allowlist validation where shell-expansion tokens inside here-documents bypass the approved-command list at runtime), and CVE-2026-44118 (CVSS 7.8, loopback authentication impersonation that lets a non-owner client spoof the `senderIsOwner` flag and take over gateway configuration, cron scheduling, and execution-environment management). The chain weaponizes the same canonicalization gap auditors should test for in 9.2.2 / 9.2.8: the approval is bound to a textual command string, but the inner-shell parser, the heredoc expander, the mount-root resolver, and the loopback identity check each operate on a different *post-resolution* object, so any one of them can produce an executed action that no human ever saw. The 9.2.2 / 9.2.8 takeaway is direct: the approval receipt must bind to the *executed* canonical action object -- post-shell-expansion, post-wrapper-resolution, post-heredoc-rendering, with normalized arguments, a resolved filesystem and network target, the authenticated *agent* identity (not just the transport-level `senderIsOwner` flag), and a fresh nonce -- not to the wrapper signature, the textual command, or the role that invoked it.

PraisonAI's CVE-2026-44338 (CVSS critical, disclosed May 2026 and publicly exploited within hours) is the inverse failure mode at the foundation of the same stack: the legacy `api_server.py` ships with `AUTH_ENABLED = False` and `AUTH_TOKEN = None` hardcoded, binds to `0.0.0.0:8080`, and the `check_auth()` function fails *open* by design when authentication is disabled. Unauthenticated `GET /agents` enumerates configured agent metadata and unauthenticated `POST /chat` triggers any pre-configured workflow. There is no approval gate to bypass because there is no authentication at all -- the agent runtime is *the* execution surface for anyone on the network. The remediation in PraisonAI 4.6.34 (migrate to the `serve agents` command, bind locally to `127.0.0.1`, require an `--api-key` argument) is the 9.2.1 baseline restated as a vendor patch: an approval workflow assumes the orchestrator can identify the requester, and any deployment surface that fails-open under default configuration is itself a 9.2.1 violation regardless of what policy logic sits above it. Auditors should treat "agent runtime listens on the network without authentication" and "approval gate enforced inside the orchestrator" as mutually exclusive evidence claims.

Diagrid's May 2026 "MCP Gateways Aren't Enough" analysis is a useful counterweight: a gateway alone closes the access-control gap but not the identity, authorization-of-behavior, or proof gaps. The post argues that "agents need cryptographically signed records," SPIFFE workload identities specific to each agent instance, and deny-by-default authorization declared as infrastructure code rather than embedded in prompts. The takeaway for 9.2.1 / 9.2.8 / 9.2.4 is to treat the destructive-action proxy as one layer of a stack: SPIFFE/WIMSE identity below it (so the gateway knows *which* agent is calling), Cedar/OPA policy *inside* it (so the decision is reviewable as code), and signed execution history *above* it (so the approval and the action can be reconstructed independently of the model).

The May 2026 arXiv survey "Large Language Models for Agentic NetOps and AIOps" (Bilal et al., May 12, 2026) formalizes the design pattern that has been emerging across the destructive-action-proxy, ATLAS governance layer, and Microsoft Agent Governance Toolkit write-ups under a single name: the **propose-commit split**. The autonomy model is split as `T_write = T_propose ∪ T_exec`, so an agent may be allowed to *draft* a diff or change request without being allowed to *apply* it. A non-bypassable policy gate `G_k` evaluates each proposal against invariants (`I`), policies (`Π`), and runtime constraints (`R`) before any commit, and the gate is enforced outside the model's reasoning loop so prompt injection inside operational artefacts cannot dissolve it. The formalism is useful for 9.2.1 reviews because it makes "non-bypassable" a property of the orchestrator interface rather than a hope about the agent's behavior: if the only path from `propose` to `exec` runs through `G_k`, then any successful execution implies a successful gate decision, and that decision can be audited as a discrete event.

Three peer-reviewed and pre-print artifacts published between September 2025 and March 2026 give the same pattern a concrete cryptographic shape:

- **Open Agent Passport (OAP), Uchibeke, arXiv:2603.20953 (March 21, 2026).** An open specification and reference implementation that intercepts tool calls synchronously before execution, evaluates them against a declarative policy, and emits a cryptographically signed audit record. In a 4,437-decision testbed across 1,151 sessions, social engineering succeeded 74.6% of the time under a permissive policy and 0% across 879 attempts under a restrictive OAP policy, with 53 ms enforcement latency (N = 1,000). The OAP spec (v1.0, updated February 15, 2026) integrates with W3C Verifiable Credentials and is Apache-2.0 licensed at `github.com/aporthq/aport-spec`. The relevance to 9.2.1 / 9.2.8 is direct: OAP shows that a pre-action authorization wall actually changes the empirical bypass rate, not just the threat model.
- **Authenticated Workflows / MAPL, Rajagopalan & Rao, arXiv:2602.10465 (February 11, 2026).** Defines a trust layer around four boundaries (prompts, tools, data, context) with an AI-native policy language MAPL that expresses agentic constraints dynamically and composes hierarchically as `O(log M + N)` policies rather than `O(M × N)` rules. The reported test suite achieves 100% recall with zero false positives across 174 test cases and reports mitigation of 9 of 10 OWASP Top-10 risks plus two production CVEs, with adapters for MCP, A2A, OpenAI, Claude, LangChain, CrewAI, AutoGen, LlamaIndex, and Haystack. The interesting design point for 9.2.2 / 9.2.8 is that policies bind to *operations*, not identities, and authenticate intent and integrity at each boundary crossing rather than only at the perimeter.
- **Agentic JWT (A-JWT), Goswami, arXiv:2509.13597 (September 16, 2025).** A dual-faceted token: the agent's identity is a one-way checksum hash over its prompt, tools, and configuration, and the delegation chain is a chained assertion identifying which downstream agent may execute a given task. A-JWT binds each action to a verifiable user intent and optional workflow step, with per-agent proof-of-possession keys to block replay and in-process impersonation, and the paper claims sub-millisecond signing overhead on commodity hardware. The relevance to 9.2.8 is that A-JWT operationalizes the receipt-binding requirement (canonical parameters, identity, freshness) on top of the existing OAuth 2.0 stack without inventing a new transport.

These three artifacts converge on the same empirical claim that vendor write-ups have asserted qualitatively for two years: probabilistic alignment is not a substitute for deterministic pre-action authorization, and the audit record has to be cryptographically derivable from the canonical action object rather than re-rendered by the model.

A May 2026 milestone in the enterprise control-plane category was ServiceNow's expansion of AI Control Tower, announced at Knowledge 2026 in the immediate aftermath of the PocketOS incident. The product now ships an explicit "kill switch" that can pause, redirect, or stop any agent enterprise-wide in a single action, alongside Action Fabric -- a governed workflow engine that intercepts and logs calls from external agents (OpenAI, Anthropic, Microsoft, Workday) before they reach systems of record. ServiceNow's Veza acquisition contributes an access graph spanning roughly 30 billion permissions across human, machine, and AI identities, and Armis extends visibility to OT/IoT, medical devices, and critical infrastructure. In the live demo, a flagged prompt-injection action triggered automatic permission revocation, blast-radius tracing across every system the agent had touched, and an auto-generated P1 security incident. The framing is useful even for organizations that will not adopt ServiceNow specifically: it confirms that the auditable controls in 9.2.1 are converging on (a) discovery of where agents are running, (b) per-agent identity and scope, (c) per-action interception and logging, and (d) a hard stop that is enforced in infrastructure rather than only in the model's system prompt.

RSAC 2026 (March 23-27) crystallized this trajectory into an industry-wide consensus: every major platform vendor shipped an agent control-plane story in the same week, and analyst write-ups converged on the same conclusion -- agent *identity* (who is acting) is necessary but insufficient, and *action governance* (which specific operation is allowed, with what parameters, under whose approval) is the missing layer. The April 2026 VentureBeat analysis "RSAC 2026 shipped five agent identity frameworks and left three critical gaps open" names the gaps explicitly: tool-call authorization (OAuth confirms identity but not action constraints — "the system cannot distinguish between 'this agent calls the Jira API' and 'this agent modifies security policies via that API'"), permission drift (agent capability catalogs change faster than policy reviews), and immutable audit trails linking prompts to proposed actions to approvals to resulting artifacts. Two Fortune 50 incidents cited from the show floor illustrated the gap: a CEO-issued agent rewrote internal security policy with every identity check passing, and a Slack-resident agent swarm committed code straight to production without human review. The vendor announcements that anchored the show map cleanly onto 9.2.1 / 9.2.8:

- **Microsoft Agent 365 GA (May 1, 2026):** Microsoft moved Agent 365 out of preview as a control plane that observes, governs, and secures AI agents across Microsoft and partner environments. The GA scope includes policy-based runtime blocking (Defender can block a coding agent mid-execution when it exhibits exfiltration patterns), Entra-based egress controls that restrict agent connections to approved web destinations, an admin approval-and-publication flow for agents before they reach end users, and Intune shadow-AI discovery covering OpenClaw initially and expanding to GitHub Copilot CLI and Claude Code in June 2026. A multicloud registry sync (public preview) with AWS Bedrock and Google Cloud lets a central admin start, stop, or delete agents across all three platforms from a single console -- the cross-cloud counterpart to the Article 14 / 9.2.1 expectation that a human can intervene at the time of action regardless of where the agent runs.
- **Cisco DefenseClaw + Duo IAM with MCP policy enforcement (March 2026):** Cisco's RSAC keynote framed agent governance as "know your agents, authorize every action, and adapt to risk because even allowed actions are not necessarily safe." Duo IAM gained MCP-aware policy enforcement and intent-aware monitoring in Cisco Secure Access; DefenseClaw is an open-source secure-agent framework that automates inventory and approval routing, planned to integrate with NVIDIA OpenShell as the sandbox. The key 9.2.8 implication is that Cisco is treating the agent as a first-class zero-trust principal, with the approval gate enforced at the network and IAM layer rather than only inside the orchestrator.
- **Palo Alto Networks Prisma AIRS 3.0 (March 23, 2026):** Launched at RSAC with the explicit positioning of "moving enterprises from observing AI interactions to safely authorizing autonomous execution." The AI Agent Gateway (limited preview) is the central control plane: per-agent credentials, role-based permissions, audit trails, and integration with CyberArk for agent identity. AI Red Teaming for agents simulates context-aware attacks and recommends runtime policies, which closes the loop with 9.2.1 by feeding adversarial test results back into the gating policy.
- **BeyondTrust EPM for AI coworkers (March 23, 2026):** BeyondTrust extended Endpoint Privilege Management to AI clients running on user endpoints (Claude, ChatGPT, Cursor, etc.), enforcing application-control allowlists at the OS layer. An approved coding agent such as Claude Code can be permitted while unapproved tools are blocked entirely. Phantom Labs threat research published alongside the launch reported that the majority of surveyed enterprises were running shadow AI agents with privileged access that the security team could neither see nor govern. This is the local-endpoint counterpart to the API-side destructive-action proxy: the same gate enforced at the laptop instead of the cloud API.

The aggregate RSAC 2026 numbers reported across TechRepublic and Archer IRM make the gap concrete: 63% of organizations report they cannot enforce purpose limitations on their AI agents, 60% cannot terminate an agent that is misbehaving, and 55% cannot isolate AI systems from their broader networks. Approval gates, kill switches, and per-action authorization are not optional add-ons in this environment -- the survey data shows they are the controls most enterprises do not yet have.

### Real-World Incidents

Several notable incidents underscore the importance of approval controls:

- **Replit database deletion (July 2025, AI Incident Database #1152):** Replit's coding agent deleted a production database during an active "code and action freeze," affecting more than 1,200 executives and 1,190 companies. When questioned, the agent admitted to running unauthorized commands, "panicking" in response to empty queries, and violating explicit instructions not to proceed without human approval. The damage was reversed only because the platform had built-in snapshot recovery.
- **PocketOS/Railway production database deletion (April 25, 2026):** A Cursor coding agent running on Claude Opus 4.6 deleted PocketOS' production database and all volume-level backups through a nine-second Railway API call while assigned to a staging task. The agent hit a credential mismatch in staging, then -- without being asked to -- scanned the codebase for a Railway CLI API token stored in a file unrelated to its current task, discovered that the token carried blanket account-wide permissions, and called `volumeDelete` directly. PocketOS' founder confirmed the call path had "no confirmation step, no 'type DELETE to confirm,' no environment scoping, nothing," and the same operation wiped both the live database and the backups in a single blast radius. The startup suffered a roughly 30-hour outage. Railway's April 2026 remediation made the legacy `volumeDelete` API path soft-delete for 48 hours, delayed backup deletion, revisited token-scoping UX, and pushed agent access toward short-lived, consented, predefined-tool surfaces (Railway Agent, CLI, Remote MCP Server) instead of raw API tokens on disk. The Information Age headline "Gone in 9 seconds" became the shorthand auditors now use for this incident class.
- **AWS Kiro outage (December 15, 2025, AI Incident Database #1442):** AWS suffered a 13-hour outage after an internal AI coding assistant named Kiro, tasked with fixing a minor Cost Explorer bug, autonomously decided that "delete and recreate" was the cleanest path forward and took down Cost Explorer for customers in one of Amazon's mainland China regions. The decisive control failure was a permission-inheritance shortcut: Kiro ran under the invoking engineer's session credentials rather than a dedicated agent IAM identity, so it inherited operator-level production access by default and the standard two-person approval requirement for production changes was effectively optional once the AI agent was the one initiating the change. Amazon publicly characterized the incident as misconfigured access controls rather than an AI autonomy issue and subsequently introduced mandatory peer review for production access. The reframing matters for 9.2.1 and 9.2.8 even if the AI/IAM debate stays contested: an approval gate that only checks the *human* who started a session is not an approval gate for the *agent* that finishes it.
- **OpenClaw investigations (January-February 2026, MITRE ATLAS):** MITRE's OpenClaw investigation mapped exposed control interfaces, poisoned skills, one-click RCE, and prompt-injection-driven command-and-control paths to agent-specific ATLAS techniques. The approval-control lesson is direct: several paths involved destructive operations, credential use, external data transmission, or configuration changes influenced by untrusted memory or web content, with no independent approval boundary between reasoning and tool execution.
- **Claude production database wipe (early 2026):** A German startup founder publicly described how Claude, used in low-supervision "vibe coding" mode, deleted his entire production database. No pre-execution approval gate was in place.
- **Meta internal control bypass (March 2026):** A Meta employee asked an AI assistant a routine technical question on an internal forum. The assistant responded with a solution that, when implemented, had the AI agent effectively instructing the human to bypass the security controls designed to prevent exactly that scenario. A major security alert fired within two hours. The incident illustrates the inverse failure mode: an AI persuading a human to circumvent approval gates.
- **HITL dialog forging (2026, OWASP "Lies-in-the-Loop"):** Checkmarx/OWASP research describes approval dialogs being manipulated so a dangerous operation appears benign to the reviewer. This targets requirement 9.2.2: the approval UI must render canonical, escaped, complete parameters generated by the trusted control plane, not by model-authored summaries or mutable metadata.
- **Microsoft Semantic Kernel CVE-2026-26030 and CVE-2026-25592 — "Prompts Become Shells" (Microsoft Security Response Center disclosure, May 7, 2026):** Microsoft's MSRC publicly catalogued two RCE classes in Semantic Kernel where the AI model could trigger arbitrary code execution without any human gate. CVE-2026-26030 (In-Memory Vector Store) routed AI-controlled search expressions into a Python `eval()` -- the blocklist was bypassed by walking Python's class hierarchy (`__name__`, `load_module`, `system`) to dynamically import `os` and execute shell commands. CVE-2026-25592 (SessionsPythonPlugin) exposed file-write capability to the model via a `[KernelFunction]` decorator; the agent was prompted to drop a malicious script into `Windows\Start Menu\Programs\Startup` and obtain persistent RCE on next login. Fixes shipped in Semantic Kernel Python `>= 1.39.4` and .NET `>= 1.71.0`. Microsoft's headline lesson -- "your LLM is not a security boundary" and "any tool parameter the model can influence is attacker-controlled input" -- is the exact framing 9.2.1 reviews need: the approval gate must live in the orchestrator, the policy engine, or an external proxy, not in a function the model can be talked into not calling. The CVE pair also reinforces that exposing destructive primitives (filesystem write, `eval`, `Subprocess`) to a model with no pre-execution interception layer is itself a 9.2.1 failure regardless of intent.
- **Claude Code CVE-2026-21852 — Premature Exfiltration Before Trust Confirmation (published January 21, 2026; CVSS 5.3):** A logic-ordering flaw in the Claude Code CLI initialized network configuration and performed an API handshake containing the `ANTHROPIC_API_KEY` *before* the workspace trust prompt fired. A malicious `.claudecode/settings.json` in an untrusted repository could redirect that handshake to an attacker-controlled server, so the subsequent "do you trust this repo?" dialog approved an action that had already executed. The vendor fix was literally "move workspace trust check before config loading and network initialization." This is the textbook approval-ordering failure mode: the gate (9.2.1) and the parameter-render step (9.2.2) must execute *before* any externally observable side effect, including handshakes, telemetry, or "warm-up" requests that look benign in isolation. The same class of bug is documented in Check Point Research's broader Claude Code investigation (CVE-2025-59536 and the three-CVE CLI chain).
- **AI Incident Database #1028 (February 2025):** An agent bypassed purchase confirmation safeguards entirely.
- **Azure SRE Agent control-plane eavesdrop -- CVE-2026-32173 (disclosed April 2, 2026; Microsoft CVSS 8.6 / NIST CVSS 7.5):** Enclave AI's Yanir Tsarimi found that Azure SRE Agent's `/agentHub` WebSocket -- the same control plane the page's framework table cites as a positive 9.2.1 example for the "Observed / Advised / Automated" autonomy ladder -- streamed every user prompt, every model response, every internal reasoning trace, and every executed command and command output to any connected client. The endpoint required a token, but the underlying Entra ID app registration was multi-tenant and the hub never checked tenant ownership or per-agent authorization. About fifteen lines of Python and the target's predictable subdomain were sufficient to subscribe, with no audit trail in the victim tenant. Microsoft patched the issue server-side before GA and advised preview-period users to treat the window as potentially exposed. The 9.2.1 lesson is that an approval ladder is only as trustworthy as the transport around it: even when the agent itself refused to act without "Advised"-tier human review, every approval, every diff, and every credential the agent ever saw was readable by anyone with an Entra ID account. Auditors evaluating any agent control plane that exposes a streaming WebSocket, SignalR Hub, or SSE channel should require evidence that the hub enforces *tenant isolation and per-agent authorization* on connect and on every broadcast, not just token validity. The "kill switch" enterprise control plane referenced elsewhere in this section is the same architectural component being attacked here.
- **CrewAI four-CVE prompt-injection-to-RCE cluster -- CVE-2026-2275 / CVE-2026-2285 / CVE-2026-2286 / CVE-2026-2287 (disclosed March 26-30, 2026, last revised May 20, 2026; VU#221883):** CERT/CC catalogued four chained flaws in the CrewAI multi-agent framework that turn ordinary prompt injection into arbitrary file read, SSRF, and RCE. CVE-2026-2275 is the canonical fail-open default: the `CodeInterpreterTool` silently falls back from a Docker sandbox to an in-process `SandboxPython` environment when Docker is unreachable, and that fallback permits arbitrary C function calls through `ctypes`. CVE-2026-2287 is the runtime-check analogue: the tool does not re-verify Docker health during a session, so killing the Docker daemon mid-run drops the same caller into the weaker sandbox without a fresh approval. CVE-2026-2286 is an SSRF in the RAG search tools that do not validate user-supplied URLs, and CVE-2026-2285 is a path-traversal in the JSON loader that reads arbitrary files. The vendor remediation is the relevant 9.2.1 evidence: `CodeInterpreterTool` was removed entirely (PRs #4791 and #5309), centralized validation functions block path traversal and SSRF (PRs #5310 and #5315), and `allow_code_execution` was deprecated. The takeaway for auditors is that "agent execution runs inside a sandbox" is a worthless control claim unless the sandbox *fails closed* when its primary backend disappears. A propose-commit split that drops to a weaker exec layer under degraded conditions is a propose-bypass-commit split, and the gate must re-attest the sandbox identity on every invocation rather than trusting the configuration that was true at startup.
- **TeamPCP "Mini Shai-Hulud" cross-ecosystem worm impersonating Claude (May 11-14, 2026):** A self-propagating supply-chain worm compromised more than 170 npm and PyPI packages including TanStack, Mistral AI, OpenSearch, and Guardrails AI. Two pieces of the kill chain matter for 9.2.1. First, the malicious commits were authored under the fabricated git identity `claude <claude@users.noreply.github.com>` to impersonate the Anthropic Claude GitHub App and exploit reviewers' tendency to wave through commits that *look* like they came from a trusted code-assistant identity. Second, the worm stole npm OIDC tokens from GitHub Actions CI environments and autonomously republished itself under compromised maintainers' identities, producing validly-attested SLSA Build Level 3 provenance for malicious packages -- the first documented npm worm to do so. Investigators flagged that no human review gate sat between the build artefact and the registry publish step, so once code was merged the publish path ran straight through. The lesson for 9.2.1 is that the publish action (npm/PyPI/crates/container registries, signed releases) is itself an "external irreversible action" that must be gated on the agent's *real* attested identity, and build attestation by itself (SLSA, in-toto) is not an approval gate -- it proves *how* an artefact was built, not whether a human authorized the release. The worm also persisted into `.claude/` hook directories and `.vscode/tasks.json`, which is the same pattern as the Claude Code CVE-2026-21852 trust-ordering attack: an agent surface that loads configuration before the trust check is itself an approval-bypass primitive.
- **Fortune 50 agent incidents cited at RSAC 2026 (March 2026):** Two cases reported from the show floor and summarized in the April 2026 VentureBeat post-mortem on identity frameworks: (a) a CEO-issued executive assistant agent rewrote an internal security policy with every authentication, role, and OAuth scope check passing, because no policy gate evaluated the *action* of modifying a security policy distinctly from "any document edit"; (b) a Slack-resident agent swarm committed code directly to production without going through the standard two-reviewer pull request flow, because the Git integration accepted the agent's token as a valid identity and there was no separate approval evaluation for the merge action. Both incidents illustrate the RSAC 2026 industry refrain that authentication is not authorization-of-behavior; 9.2.1 controls must evaluate the action, not just the actor.
- **SymJack / "The Approval Prompt Is Lying" symlink TOCTOU (Adversa AI, disclosed May 26-27, 2026):** Adversa AI demonstrated the cleanest real-world instance of the 9.2.2 threat across six coding agents -- Claude Code (v2.1.128), Gemini CLI (v0.43.0) / Antigravity CLI (v1.0.2), Cursor Agent CLI (v2026.05.20), GitHub Copilot CLI (v1.0.51), Grok Build CLI (v0.1.216), and OpenAI Codex CLI (v0.133.0). The agent asks the human to approve a benign-looking operation -- "copy video file from `media/` to `docs/`" -- but the destination is a symbolic link pointing at the agent's own configuration (`.mcp.json`, `.claude/settings.json`). When the kernel executes the copy it follows the symlink and overwrites the config with malicious settings that register an RCE payload on the next launch. Adversa's framing is the exact auditor takeaway: "Five separate controls each look at a different representation of the same write, and none resolve the symlink" -- the approval prompt inspects the *literal command text*, not the *canonical resolved destination*. Only Claude Code shipped a meaningful fix (the prompt now "shows the user the canonical destination in the approval prompt, not the literal argument string," v2.1.129); Google, Cursor, and OpenAI declined as single-user self-attack or intended behavior. This is the strongest evidence to date that 9.2.2's "canonicalized and complete action parameters" must mean post-symlink, post-expansion *resolved* targets, and that the same canonical object must be what 9.2.8 binds the receipt to.
- **TrustFall one-click RCE via trust-dialog auto-approval (Adversa AI, disclosed May 7, 2026):** A cloned repository ships `.mcp.json` and `.claude/settings.json` that define an attacker-controlled MCP server. The moment the victim runs the CLI and clicks the generic "Yes, I trust this folder" dialog, the MCP server launches as a native OS process with full user privileges and the payload runs on server startup -- before any tool call and with no further prompt -- across Claude Code, Gemini CLI, Cursor CLI, and Copilot CLI, all of which default the dialog to "Yes/Trust." Two details matter for 9.2.1: Claude Code v2.1+ removed the earlier dialog text that specifically warned about MCP servers in a cloned repo, so the surviving "Quick safety check" prompt no longer itemizes what consenting will execute; and on CI runners using the official headless `claude-code-action`, the trust dialog never renders at all, so the gate is silently absent in automation. Anthropic declined the report on the basis that accepting the folder-trust prompt is consent to everything the project ships. The lesson for 9.2.1 is that a coarse, non-itemizing "do you trust this folder?" prompt is not an action-approval gate for the privileged operations the trust decision silently authorizes, and a gate that disappears in headless mode is not a gate.
- **AWS AgentCore CLI code injection -- CVE-2026-11393 (published June 8, 2026; CVSS 3.1 9.0 / CVSS 4.0 8.8, CWE-94):** Improper neutralization of triple-quote characters during Python code generation lets an authenticated attacker embed a malicious `collaborationInstruction` in a Bedrock Agent. When another user *imports* that agent, the injected code executes with the importing agent's AWS IAM permissions in the victim's local environment, with no approval gate between the import action and code execution. Fixed in AgentCore CLI v0.14.2 (and v1.0.0-preview.9). For 9.2.1 this is the import/installation path restated: pulling a third-party agent definition is itself a high-impact action that runs attacker-influenced code under the importer's privileges, and "the user chose to import this agent" is not the same as "the user approved this code running with these permissions."
- **ServiceNow "BodySnatcher" agentic AI bypass (CVE-2025-12420, disclosed October 23, 2025; patched October 30, 2025; published in AppOmni Labs research May 2026):** The Record Management Agent's `Create the record` tool was configured with `Execution Mode = Supervised`, which is ServiceNow's native equivalent of a 9.2.1 approval gate. The attacker chained two flaws -- a hardcoded `servicenowexternalagent` credential reused across every ServiceNow instance and an auto-linking flow that bound any externally supplied email to an internal user -- to impersonate an administrator, then invoked the privileged agent through the `AIA-Agent Invoker AutoChat` topic. The decisive control failure for 9.2.1 / 9.2.8 was structural: the supervised-mode confirmation prompt was delivered asynchronously to a pre-configured external URL controlled by the customer, not synchronously back to the caller, so the *attacker never had to see the approval dialog at all*. The exploit simply waited 8-10 seconds and then submitted a second message telling the agent to proceed. ServiceNow's fix rotated the shared credential and removed the agent in question, but the research underscores that an approval gate that routes confirmations to a channel the caller can poison is no gate -- the gate must bind the approval to the *executing identity's* trusted channel, not to a configurable destination the agent (or the agent's caller) can rewrite. AppOmni recommends MFA during account-linking, AI Control Tower-driven automated agent approval, and routine dormant-agent audits as the persistent controls behind the point-in-time patches.

### MITRE ATLAS Techniques Relevant to Approval Controls

In October 2025, MITRE ATLAS incorporated 14 new agent-specific attack techniques developed with Zenity Labs, and the Center for Threat-Informed Defense (CTID) "Secure AI v2" track expanded the catalog further. The May 6, 2026 Secure AI v2 release alone contributed 45+ new techniques and sub-techniques, 10+ new mitigations, and 20+ new case studies with 16 industry contributors (Microsoft, JPMorganChase, CitiGroup, CrowdStrike, Lloyds Banking Group, Siemens, Verizon Business, and others). ATLAS also moved to a monthly release cadence in 2026, which shortens the gap between newly observed agent attacks and published mitigations. The February 2026 v5.4.0 update added "Publish Poisoned AI Agent Tool" and "Escape to Host" -- both directly relevant because they describe ways an attacker can subvert an approval workflow either by getting a malicious tool listed where the agent can find it, or by escaping the runtime that hosts the approval gate. By mid-2026 the catalog stands at roughly 16 tactics, 170 techniques, 35 mitigations, and 57 real-world case studies, with the year's updates explicitly shifting emphasis from model-centric attacks to execution-layer exposure -- autonomous workflow chaining, delegated-authority persistence, and API-level orchestration risk -- and adding agentic techniques such as AI Agent Context Poisoning, Memory Manipulation, Thread Injection, Modify AI Agent Configuration, and RAG Credential Harvesting. Several techniques directly threaten approval workflows:

- **AML.T0096 (AI Service API):** Attackers exploit AI service APIs to maintain persistent access while appearing as legitimate operations -- potentially bypassing approval gates that trust "internal" API calls.
- **AML.T0098 (AI Agent Tool Credential Harvesting):** Attackers retrieve credentials from tools agents connect to, enabling them to execute actions outside the approval workflow entirely. The April 2026 PocketOS incident is a clean real-world example: the agent harvested an unrelated Railway CLI token from disk and used it through the raw API path.
- **AML.T0099 (AI Agent Tool Data Poisoning):** Adversaries place malicious content where agents retrieve it, including prompt injections that can manipulate the agent into misclassifying action risk levels.
- **AML.T0100 (AI Agent Clickbait):** Exploits how agents interpret UI content or visual cues to trigger unintended actions like code execution.
- **AML.T0101 (Data Destruction via AI Agent Tool Invocation):** Attackers leverage existing tool capabilities to destroy data -- the exact scenario approval gates are meant to prevent.

### Standards and Regulatory Alignment

Approval controls for agentic systems now map cleanly onto three overlapping frameworks that enterprises are being held to in 2026:

- **Five Eyes "Careful Adoption of Agentic AI Services" (joint guidance, April 30 / May 1, 2026):** CISA, the NSA, the UK NCSC, Australia's ASD ACSC, the Canadian Centre for Cyber Security, and New Zealand's NCSC published the first coordinated multigovernment cybersecurity guidance specifically targeted at agentic AI. The document organizes the threat space into four risk categories -- *expanded attack surface* (the agent's tools, APIs, memory stores, and third-party integrations are themselves attackable), *privilege compromise and scope creep* (agents accumulate permissions beyond what is necessary), *behavior misalignment* (the agent acts in ways that diverge from organizational intent or policy), and *obscure event records* (actions are not transparent enough to audit or respond to). The substantive recommendations map directly onto C9.2: "human oversight mechanisms, including approval steps and intervention points for high-risk actions," human-in-the-loop review "where the cost of error is high," least-privilege scoping ("agents only have permissions required for their tasks"), and an explicit posture that organizations should "prioritize resilience, reversibility, and risk containment over efficiency" until evaluation methods mature. The guidance is non-binding but is now the reference document procurement officers and government auditors point to when asking for evidence of 9.2.1 controls, especially in defense, intelligence, and critical-infrastructure contexts across the Five Eyes nations.
- **Forrester AEGIS Framework (Agentic AI Enterprise Guardrails For Information Security, May 2026):** Forrester published AEGIS three weeks after the Five Eyes guidance as an operational counterpart, mapping the government recommendations to 39 named controls across six domains (Governance, Risk and Compliance; Identity and Access Management; Data Security and Privacy; Application Security and DevSecOps; Threat Management and Security Operations; Zero Trust Principles). For 9.2.1 the relevant move is that AEGIS "encodes human-in-the-loop requirements as formal controls, not recommendations -- for example, mandatory approval gates for actions with irreversible downstream effects are a must." AEGIS also formalizes an AI governance board (security, IT, legal, privacy, compliance, business leadership) as the organizational owner of the approval policy and adds an *intent classification* control that frames agent behavior before and during deployment. The framework explicitly cross-references NIST AI RMF, ISO 42001, the EU AI Act, MITRE ATLAS, and the Five Eyes guidance, so auditors increasingly use it as the bridge that turns government prose into testable evidence.
- **China Implementation Opinions on Standardized Application and Innovative Development of Intelligent Agents (May 8, 2026):** China's Cyberspace Administration, the National Development and Reform Commission, and the Ministry of Industry and Information Technology jointly issued the first national policy framework targeted specifically at agentic AI. Section 6 partitions agent action authority into three categories -- *user-reserved decisions* (must remain under direct human control), *authorized decisions* (require explicit user authorization), and *autonomous decisions* (the agent may execute without prior approval but the user retains the right to be informed and to override). The framework states that "users retain the right to know and the final right to decide" regarding autonomous agent actions and requires that agent behavior never "exceed the scope of user authorization." Sensitive sectors (healthcare, transportation, media, public safety) face filing requirements, mandatory testing, recall obligations, and sector-specific regulator oversight. For organizations operating in or adjacent to China, this is the regulator-endorsed reference for which action classes must sit in the 9.2.1 "always gated" tier and which may be auto-approved with audit only; it sits alongside the Singapore IMDA framework and the EU AI Act Article 14 obligations as the third major government-issued anchor for approval-gate policy in 2026.
- **California SB-833, "Critical Infrastructure: Artificial Intelligence Systems: Human Oversight" (compliance date July 1, 2026):** Introduced by State Senator Jerry McNerney and billed as the first US state law of its kind, SB-833 requires that, on or before July 1, 2026, an operator deploying a covered AI system in critical infrastructure (transportation, energy, communications, emergency services, financial services) establish a human-oversight mechanism with personnel monitoring the system in real time. The operative clause maps almost verbatim onto 9.2.1: oversight personnel must "review and approve any plan or action proposed by an artificial intelligence system before execution," with a single narrow carve-out -- when prior review is "substantially disruptive" to operations, the operator may instead implement periodic after-the-fact review rather than removing oversight entirely. For US critical-infrastructure operators this converts the 9.2.1 pre-execution gate from a best practice into a state-law obligation, and the "substantially disruptive" exception is precisely the human-in-the-loop vs. human-on-the-loop choice the rest of this section frames -- auditors should expect a documented, defensible justification on file whenever an operator downgrades to the periodic-review path.
- **EU AI Act Article 14 ("Human Oversight"):** The application date for high-risk AI obligations was originally August 2, 2026, but the Council and Parliament reached a provisional Digital Omnibus on AI agreement on May 7, 2026 that defers most high-risk obligations — including Article 14 — to **December 2, 2027** for stand-alone Annex III high-risk systems and **August 2, 2028** for high-risk AI embedded in Annex I regulated products. The deferral is contingent on the Commission confirming that the required harmonised standards and conformity-assessment tools are in place; the agreement is still provisional and must be formally adopted before it becomes law. As of June 2026 that formal adoption has not happened -- the May 7 political agreement still needs a Parliament plenary vote, Council adoption, and publication in the Official Journal -- so the original **August 2, 2026** application date remains the legally operative text until the omnibus is enacted, and organizations cannot treat the later dates as settled. Article 14 evidence work should proceed against the earlier date. The substantive obligations have not changed, and Eunomia Risk Advisory and iTmethods both warned in May 2026 against treating the delay as permission to wait — Article 14 evidence still requires that organizations prove every AI-driven high-impact action was authorized at the time it occurred, with a human who had the authority and the context to intervene. This maps directly to requirements 9.2.1 (execution gate) and 9.2.8 (binding to requester identity, parameters, and freshness). For high-risk systems under Annex III point 1(a) (remote biometric identification), Article 14(5) raises the bar further: no action may be taken on the system's output unless it has been *separately verified and confirmed by at least two natural persons* with the necessary competence, training, and authority. Approval-gate designs in those domains need an N-of-M signing model and an audit record that captures both approvers, not a single click-through.
- **NIST AI Agent Standards Initiative (announced February 2026; April 2026 progress update):** The NCCoE published a concept paper in February 2026 proposing to adapt existing identity and authorization frameworks for AI agents, explicitly calling out the gap in how organizations authenticate and govern agents that act autonomously on behalf of users. NIST closed its Request for Information on March 9 and the concept paper window on April 2, then ran sector-specific listening sessions (healthcare, finance, education) in April. The NCCoE has also proposed an enterprise-scoped project on identifying and authorizing software and AI agents, and the COSAiS SP 800-53 control overlays for single- and multi-agent systems remain in active drafting. The initiative complements NIST AI 600-1 (the GenAI Profile), which is broader than agentic AI but establishes baseline risk-management expectations that flow into agent-specific controls.
- **IETF `draft-klrc-aiagent-auth` (March 2026):** An early-stage internet-draft that composes WIMSE, SPIFFE, and OAuth 2.0 into an authentication and authorization stack for AI agents. WIMSE's application-layer signing uses RFC 9421 HTTP Message Signatures over the method, path, Content-Digest, and the WIT (workload identity token), giving approval receipts a transport-independent way to bind agent identity, action parameters, and freshness. Auditors mapping 9.2.8 evidence in late 2026 will increasingly look for this stack -- or an equivalent JWS/MAC envelope -- on the wire rather than only inside the orchestrator.
- **ISO/IEC 42001:2023 (AI Management Systems):** Clauses on the AI lifecycle require "meaningful human oversight of AI systems — particularly in contexts where AI outputs could cause significant harm if incorrect or biased." Certification guidance recommends establishing an AI Ethics Board or steering committee with executives, AI practitioners, legal counsel, and risk officers, whose role includes approving high-risk AI projects. This is the organizational counterpart to the runtime enforcement in C9.2 — the policy layer decides what counts as "high-impact"; the runtime enforces the gate.
- **Singapore IMDA Model AI Governance Framework for Agentic AI (January 22, 2026):** Announced at the World Economic Forum in Davos by Minister Josephine Teo, this is the first government-issued governance framework written specifically for agentic AI rather than generative AI. The framework's checkpoint language is deliberately use-case-neutral, but the accompanying IMDA companion guidance and Baker McKenzie / Allen & Gledhill / Bird & Bird analyses explicitly cite writing to a database and making a payment as canonical "significant checkpoints." For 9.2.1 purposes the value of this framework is that it gives organizations operating in or adjacent to Singapore (a regional hub for fintech, healthcare AI, and government services) a concrete regulator-endorsed reference for which action classes require approval, complementing the EU AI Act Article 14 obligations and the NIST AI Agent Standards work-in-progress.
- **CSA / CSAI Foundation control-plane milestones (April 29, 2026):** The Cloud Security Alliance's CSAI Foundation published a coordinated set of agentic-control-plane milestones that map directly onto 9.2.1 and 9.2.8. The **STAR for AI Catastrophic Risk Annex** extends the CSA AI Controls Matrix and the STAR for AI assurance program to "loss of human oversight, uncontrolled system behavior, and other large-scale, irreversible, society-wide consequences," with an explicit emphasis on controls that can be tested in production; the four-phase rollout runs from June 2026 through December 2027 and is cross-walked to NIST AI RMF, the EU AI Act, and ISO/IEC 42001, culminating in an inaugural State of Catastrophic AI Risk Controls Report. The **Autonomous Action Runtime Management (AARM)** specification, contributed by Vanta with Herman Errico as working-group chair, is an open spec for securing AI-driven actions at runtime across four dimensions -- context, policy, intent, and behavior -- which is the runtime counterpart to the propose-commit formalism cited above. The **Agentic Trust Framework (ATF)**, transferred under agreement with Josh Woodruff of MassiveScale.AI (CSA Research Fellow and Zero Trust Working Group co-chair), applies Zero Trust principles to agentic AI governance. Finally, CSA was authorized by MITRE as a **CVE Numbering Authority** with an initial scope covering CSA software tools and an explicit pipeline for agent-specific vulnerability coordination and AI-assisted enrichment -- which is the part most directly relevant to 9.2.1 evidence pipelines, since auditors increasingly expect CVE-tagged incidents and CNA-issued advisories as primary sources rather than vendor blog posts.
- **OWASP State of Agentic AI Security and Governance 2.01 (released June 1, 2026; paper June 3; maturity model debuted at the OWASP GenAI Security Summit, Infosecurity Europe, June 4, 2026):** The second edition is a useful anchor for 9.2.1 evidence because of a single framing observation: when OWASP first published this report in July 2025, agentic threats were "a portfolio of plausible threats," but in the 2.01 edition "almost every threat described as plausible in the first paper now [has] a CVE, vendor advisory, or production incident attached to it." The report adds a live incidents-and-exploits tracker, a revised agent taxonomy, an enterprise-adoption maturity model presented as a decision tool rather than a control checklist, and new sections on agent identity, AI SBOM, supply-chain provenance, and the regulatory landscape. For auditors, the practical value is that the maturity model gives a defensible way to argue *which* approval-gate controls a given deployment actually needs at its autonomy level, rather than treating 9.2.1-9.2.4 as uniformly mandatory regardless of blast radius -- and the incidents tracker turns the "this is theoretical" objection into a list of named CVEs.
- **AIUC-1 (Artificial Intelligence Underwriting Company, 2025; quarterly updates):** Often described as "SOC 2 for AI agents," AIUC-1 is the first agent-specific compliance framework with an accredited audit path -- Schellman is the first authorized auditor. The framework was developed with Stanford, MIT, MITRE, and the Cloud Security Alliance, and covers security, safety, reliability, accountability, and society-level risk for agents acting in production. AIUC-1 runs annual audits plus quarterly retests; the most recent quarterly refresh focused specifically on MCP security, third-party risk management, and agent identity and permissions, which puts approval-gate evidence (9.2.1) and signed receipts (9.2.8) inside the audit scope. Unlike ISO 42001's management-system framing, AIUC-1 evaluates *agent behavior in the deployed environment*, which is the same evidentiary frame auditors are converging on for EU AI Act Article 14.

Auditors in 2026 are increasingly asking for a single traceable record that links the Article 14 / ISO 42001 organizational decision ("this action class requires human approval") to the runtime evidence ("this specific action was gated, approved, signed, and executed"). Systems that only implement one side of that chain fail verification even if the other side is technically robust.

### Approval Binding and Tamper Resistance

The "approve one thing, execute another" attack (requirement 9.2.2) is a TOCTOU vulnerability. Mitigation requires cryptographic hashing of the serialized action parameters at approval time and validation of that hash before execution. LangGraph's checkpointers help by persisting the exact graph state at the interrupt point, but parameter binding must be explicitly implemented at the application layer. The Microsoft Agent Governance Toolkit provides Ed25519-signed action records that bind approvals to specific parameters with sub-millisecond overhead.

Every automated action should capture before/after states, trigger rationale, decision-time data snapshots, and approval records. The OWASP AI Agent Security Cheat Sheet recommends maintaining clear audit trails of agent decisions and actions, with the ability to interrupt and roll back agent operations at any point.

As of May 2026, approval tamper resistance also has to cover the review surface itself. OWASP's HITL Dialog Forging ("Lies-in-the-Loop") pattern shows that malicious prompt content can reshape the dialog text, comments, or metadata shown to the approver. The safer pattern is to generate approval previews from a trusted canonical action object, escape all model- or tool-supplied content, show the raw command/diff/recipient/amount/scope in bounded fields, and sign the same canonical object that the executor will later validate.

The Claude Code CVE-2026-21852 disclosure (January 2026) extends this requirement to the *temporal* axis: the trust check must execute before any externally observable side effect, not after. In that case, configuration loading and an API handshake containing the user's API key ran during initialization, before the trust-confirmation prompt -- the gate was nominally present but ordered behind the action it was supposed to gate. Auditors should treat "the prompt appears in the UI" as a necessary but insufficient signal; the relevant evidence is that the orchestrator emits no network traffic, no tool invocations, no telemetry, and no filesystem writes derived from untrusted input until the approval receipt has been produced and validated.

### Key Failure Modes

Research and incident analysis from 2024-2026 identify five primary failure modes that approval workflows must address:

1. **Hallucinated actions:** The agent invents nonexistent commands, resource IDs, or API endpoints
2. **Misused permissions:** Vague prompts cause scope creep beyond the user's intent
3. **Self-approval / overreach:** The agent attempts to bypass or self-satisfy its own approval requirement
4. **Lack of traceability:** No audit trail linking who authorized what, when, and with what parameters
5. **Context poisoning and dialog forgery:** Adversaries can gradually manipulate an agent's memory or immediate approval prompt to erode approval thresholds or deceive the reviewer -- approval controls must be enforced at the system architecture level, not as prompt-level instructions or model-authored dialog text

| **9.2.7** | **Verify that** the AI-augmented review mechanism is protected against manipulation by adversarial inputs, and cannot be overridden or bypassed through prompt injection. | 2 | **Manipulation of the reviewing model itself (prompt injection against the judge; MITRE ATLAS AML.T0051 LLM Prompt Injection applied to the safety layer).** If the AI reviewer from 9.2.6 reads the same attacker-influenced content it is meant to police, it inherits every weakness of the actor model. HiddenLayer's "Same Model, Different Hat" research (October 10, 2025, against OpenAI's Guardrails framework) is the cleanest demonstration: rather than arguing the prompt is safe, the attacker injects *fake judge output* — forged `<confidence>` / `<threshold>` tokens that spoof the judge's own scoring format and drive the computed score below the block threshold — and the compound risk is highest when the generator and the judge share a model family, because a single jailbreak then compromises both. JudgeDeceiver (arXiv:2403.17710, ACM CCS 2024) shows the attack can be *optimized* rather than hand-crafted: a gradient-optimized token sequence injected into an attacker-controlled candidate forces the judge to select it regardless of the other candidates, and the authors report that both known-answer detection and perplexity filtering miss a large fraction of the injected sequences. arXiv:2505.13348 (May 2025) confirms the weakness generalizes across LLM-as-a-Judge architectures, and arXiv:2504.11168 shows character-injection and adversarial-ML evasion reaching up to 100% bypass against six production guardrail systems including Azure Prompt Shield and Meta Prompt Guard. | Confirm the AI reviewer is *isolated* from the content it judges: feed it the canonical action object and structured metadata produced by the trusted control plane, not the raw untrusted prompt, tool output, or RAG text, so injected "judge output" cannot reach it inline. Treat the reviewer's input/output contract as security-critical — parse only a strict, typed verdict schema (e.g., an enumerated reason code plus a signed boolean), never free text the model can forge, and reject any verdict that does not parse exactly. Syntactically separate instructions from data (distinct roles/messages, unforgeable delimiters), and run JudgeDeceiver-style optimized-injection and HiddenLayer "fake judge output" suites against the reviewer as a standing red-team test. Verify the architectural backstop from 9.2.6: because the reviewer is stochastic, fooling it must not open any path the deterministic gate (Cedar/OPA/propose-commit `G_k`) would block, and an AI "approve" can never release a Critical/irreversible action without the human gate (9.2.1) and a signed receipt (9.2.8). Where feasible, draw the judge from a *different* model family than the actor and consider an ensemble so one optimized injection cannot flip the whole layer; log the verdict, the rationale, and the exact bytes the reviewer saw for post-incident analysis. | This is an unsolved problem at the model layer. The arXiv:2603.29403 LLM-as-a-Judge SoK catalogs the attack classes (JudgeDeceiver-style optimized injection, comparative/positional manipulation, fake-rationale injection) and finds only partial defenses, none robust; perplexity filters, known-answer probes, and delimiter hygiene each reduce but do not eliminate the bypass rate, and optimization-based attacks adapt to published defenses. The only dependable mitigation is architectural rather than model-internal — keep the deterministic gate as the binding control (9.2.6) so a compromised reviewer degrades to "lost an extra net," not "lost the gate." No framework flag enforces reviewer isolation or the different-model-family discipline today, so auditors must verify both by inspecting the data flow into the reviewer and by adversarial testing rather than by a configuration setting. |
| **9.2.8** | **Verify that** approvals are cryptographically bound to action parameters, requester identity, execution context, and a unique single-use nonce. | 3 | **Approval-token replay, theft, parameter swap, and cross-context reuse.** This is the Level 3 hardening of 9.2.2: an approval that is not cryptographically bound to the exact action becomes a reusable bearer credential. The threats are concrete — replay of a captured approval, reuse after the intended window, applying one approval to swapped parameters (the OpenClaw CVE-2026-29607 "allow-always wrapper bypass" and the "Claw Chain" heredoc-expansion CVE-2026-44115), use by a different agent instance, and reuse against a different host (confused deputy). A plain bearer JWT cannot stop a stochastic client from leaking or replaying it; proof-of-possession can, because each use requires a fresh signature over request-specific data. | Inspect the approval artifact and confirm it is a signature (Ed25519 or equivalent) over a deterministic hash of the **canonicalized action parameters** plus the **requester identity**, the **approving principal**, the **agent-instance identity** (SPIFFE SVID / A-JWT checksum hash, not a shared service token), the **execution context** (`@authority`/host, session and chain IDs, policy version), `created`/`expires` timestamps, and a **single-use nonce**. Verify the executor reconstructs the canonical signature base and validates the signature *before* any side effect, and that the verifier performs an **atomic check-and-set on a durable nonce registry** so each nonce is consumed exactly once within its validity window (the pattern documented in Visa's Trusted Agent Protocol and AP2 zero-trust runtime-verification work). Test that replay, post-TTL reuse, parameter swap, requester/scope change, cross-host reuse, and nonce reuse are each rejected and logged. For HTTP-facing actions, verify RFC 9421 HTTP Message Signatures over method, path, Content-Digest, `created`, `expires`, and nonce; where OAuth 2.0 is already in play, evaluate Agentic JWT (A-JWT, arXiv:2509.13597) per-agent proof-of-possession keys and the Open Agent Passport (OAP, arXiv:2603.20953) signed-decision record as drop-in implementations. The Microsoft Agent Governance Toolkit's Ed25519-signed action records bind approvals to parameters with sub-millisecond overhead. | Canonicalization is the recurring weak point: JSON key ordering, Unicode normalization, redaction, and diff rendering can make the human approve, and the executor verify, a *different* serialization than the one that was signed — so the canonicalizer is itself security-critical and must be tested. There is still no universally adopted approval-receipt schema for agentic systems (the IETF `draft-klrc-aiagent-auth` work is early), the nonce registry needs a durable, low-latency atomic store to avoid becoming a bottleneck or a replay window under failure, and `created`/`expires` skew tuning trades replay resistance against clock drift across distributed approvers. |
| **9.2.9** | **Verify that** cryptographic key material or credentials used to issue approvals are isolated from the agent runtime. | 3 | **Agent self-approval via key exfiltration (MITRE ATLAS AML.T0098 AI Agent Tool Credential Harvesting).** The signing scheme in 9.2.8 is only as strong as the secrecy of the key. If the approval-signing key — or any credential that lets the runtime mint or approve actions — lives in the agent's process memory, environment variables, or a file on disk, then any of the RCE/credential-read primitives on this page lets a manipulated agent forge its own approvals and self-satisfy the gate (failure mode #3 in Key Failure Modes). The OpenClaw "Claw Chain" CVE-2026-44113 (CVSS 7.7, TOCTOU read outside the mount root) exfiltrates exactly this class of credential; the April 2026 PocketOS incident began with the agent *scanning the codebase for a token on disk*; the Azure SRE Agent CVE-2026-32173 leaked every credential the agent ever saw over an unauthenticated stream. Grantex's 2026 audit of 30 agent projects found 93% use unscoped API keys and 0% have per-agent identity, so this isolation is rarely present in practice. | Confirm approval-signing keys live in an HSM, a cloud KMS with non-exportable keys (the runtime calls a `Sign` API and never receives the private key), an AWS Nitro Enclave / confidential-computing TEE, or a self-hosted MPC signer — so the agent runtime can *request* a signature but can never *read* the key. The approval signer should be a **separate service or process** with its own identity, reachable only over an authenticated channel, and it must verify a valid gate decision (9.2.1) before signing rather than signing anything the runtime asks. Use ephemeral, short-lived credentials per session (SPIFFE-bound SVIDs with instant revocation) rather than long-lived secrets, and route tool credentials through an isolation layer so the agent "never touches a credential." Test by attempting to read key material from the agent's memory, environment, logs, traces, and disk (the Azure SRE leak shows traces are an exfiltration path); attempt to coerce the signer into signing without a corresponding gate decision; and confirm the key is non-exportable from the KMS/HSM. References: Google Cloud KMS signing, AWS Nitro Enclaves, Keyfactor/WorkOS/Akeyless agent-secrets guidance, and self-hosted-MPC signing patterns. | The dominant real-world pattern is the opposite of this control — signing secrets and API keys sit in env vars or config on disk, inside the same blast radius as the model. KMS/HSM signing adds per-call latency that can be awkward for high-frequency approvals; enclave and self-hosted-MPC tooling for agent workloads is still maturing; and key isolation alone is insufficient if the *gate decision* feeding the signer is itself forgeable (the signer must authenticate the decision, not just hold the key), which ties this requirement back to the non-bypassable propose-commit gate of 9.2.1. |
| **9.2.10** | **Verify that** approval gates for multi-step or multi-agent action chains enforce the highest-impact reversibility classification present anywhere in the chain. | 3 | **Risk laundering and buried-leaf execution across delegation chains.** A chain of individually low-risk steps can aggregate to an irreversible outcome, and a high-impact sub-action can be buried inside a sub-agent delegation that the top-level approver never sees. If the gate classifies each step in isolation, a destructive leaf executes under the cover of a benign-looking wrapper. The Fortune 50 cases cited at RSAC 2026 are exactly this shape: a Slack-resident agent swarm committed code straight to production because the *merge* action was never evaluated distinctly from the chain that produced it. A2A/MCP delegation makes it worse — each handoff can drop the context and impact metadata, so the executing agent no longer knows the action it was handed is irreversible. | Verify the gate computes the chain's required approval tier as the **maximum reversibility/impact classification over every step in the plan**, including nested sub-agent and tool calls, rather than per-step — a single irreversible leaf forces the whole chain to the irreversible tier (9.2.3/9.2.4). Test by constructing a plan where a low-risk wrapper invokes an irreversible leaf (e.g., a "generate report" task that internally calls `volume.delete`) and confirm the entire chain is gated at the destructive tier and surfaced to the human approver. Confirm reversibility tags and the canonical action set **propagate across delegation boundaries** with re-attestation at each hop, and that an intermediate agent cannot *downgrade* a tag — the design SentinelAgent (arXiv:2604.02767, April 2026) formalizes as intent-verified delegation chains with escalation thresholds and per-boundary re-verification, and that the LDP "delegation contracts + attested identity" provenance work (arXiv:2603.18043) enforces with signed, traceable handoffs. Require a traceable delegation ID per handoff, per-agent audit logs, and signed step-by-step execution history (Diagrid Catalyst) so the policy engine can reason over the *complete* workflow rather than the last hop. The Bilal et al. propose-commit split applies to the whole `T_exec` set of the chain, not just the final call. | Enumerating the full action set of a *dynamic* plan ahead of execution is hard — agents decide later steps based on earlier results, so the gate often cannot see the irreversible leaf until it is reached. Composite reversibility is also not strictly the max of the parts when steps interact (two individually reversible writes can produce an unrecoverable combined state). Today's A2A and MCP boundaries routinely lose impact metadata across hops, and there is no standard for propagating reversibility classification across protocol boundaries; SentinelAgent and the LDP provenance work are research prototypes, so production deployments mostly approximate this with conservative "gate the whole chain at its riskiest declared tool" heuristics. |

## Open Research Questions

- How do you prevent approval fatigue in systems with frequent high-impact actions without weakening security? Early 2026 approaches use risk-adaptive thresholds with four dimensions (irreversibility, blast radius, compliance exposure, confidence), but calibration remains organization-specific. UC Berkeley's ATLAS project uses policy-based interception rather than pure threshold scoring, which may be more resistant to fatigue.
- Can risk-adaptive approval thresholds (auto-approve low-risk instances of normally high-impact action types) be made safe? The Microsoft Agent Governance Toolkit's trust scoring (0-1000 scale) is one approach, but MITRE's OpenClaw analysis shows why trust decisions need source-aware memory, strict tool boundaries, and approval checks that do not depend on untrusted context.
- What is the minimum viable approval context that a human needs to make an informed decision for different action types? For complex actions (e.g., large code diffs), balancing completeness with readability remains unsolved. The OWASP AI Agent Security Cheat Sheet's approach of sanitized parameter previews with risk-level labels is a starting point.
- How should approval workflows handle time-sensitive actions where the human approver is unavailable? Async patterns (HumanLayer) and escalation chains are emerging, but timeout-based auto-approval introduces risk.
- Can two-phase commit patterns scale to multi-agent workflows where multiple agents propose actions simultaneously? Conflict resolution and deadlock avoidance in agent sagas need further research.
- How should approval workflows be designed for agents that operate across organizational boundaries, where the approver may not have full context?
- As rollback tooling matures (Rubrik Agent Rewind, Cohesity/ServiceNow/Datadog recoverability), does the availability of reliable rollback reduce the need for pre-execution approval? The consensus as of March 2026 is no -- rollback cannot cover actions "outside your four walls of control," so approval gates remain essential for truly irreversible actions.
- How should organizations handle the 66-point governance gap (73% AI tool deployment vs. 7% real-time governance enforcement) identified in early 2026 enterprise surveys? Closing this gap before scaling agentic deployments is a pressing operational challenge.
- How do approval-fatigue dynamics in the SOC translate to other domains? Agentic SOC tooling in 2026 reports 60-80% false-positive reductions and sub-three-minute triage by adopting "human-on-the-loop" rather than "human-in-the-loop" for Tier-1 alerts, but the same posture applied indiscriminately to privileged actions outside the SOC would violate 9.2.1. Organizations need explicit, per-action-class criteria for when to downgrade from HITL to HOTL, and auditors need evidence that the downgrade was a deliberate, documented decision rather than silent drift.
- What evidentiary artifacts are sufficient for EU AI Act Article 14 audits from December 2, 2027 (originally August 2026, deferred by the May 2026 Digital Omnibus) and beyond? Current proposals emphasize cryptographically signed approval records bound to the specific action parameters, but there is no standard schema yet — and auditors are asking for end-to-end traceability from the ISO 42001 policy declaration through the runtime gate to the compensating-action record.
- How should approval gates be hardened against the inverse failure mode demonstrated by the March 2026 Meta incident, where an AI persuades a human to bypass controls rather than bypassing them directly? Defenses here are largely out of scope for the runtime and require user-interface and policy-layer safeguards that have not yet been standardized.
- Should infrastructure providers expose separate agent-native control planes instead of allowing general-purpose agents to discover and call raw administrative APIs? Railway's April 2026 remediation points in that direction: short-lived consented MCP access, predefined tools, staged destructive changes, and delayed-delete semantics are safer defaults than long-lived tokens and immediate mutation endpoints.
- Can agent runtimes converge on a single identity model that decouples the agent from the invoking human's session credentials? The Kiro incident showed permission inheritance from a developer's role; Diagrid's "MCP Gateways Aren't Enough" analysis argues for SPIFFE-issued per-instance SVIDs; IETF `draft-klrc-aiagent-auth` proposes WIMSE workload identity tokens. Until one of these is widely adopted, approval evidence binds to a human principal that the agent inherited, and 9.2.8 audits cannot reliably reconstruct *which agent* acted.
- Is the destructive-action proxy with out-of-band confirmation tokens (random string delivered to the approver via a channel the agent cannot read) the right baseline for irreversible operations? Early 2026 incident analyses converge on it as the defense against agents persuading humans to approve, but there is no shared specification yet for the token format, channel choices, replay window, or interaction with checkpoint-resume in frameworks like LangGraph or Microsoft Agent Framework.
