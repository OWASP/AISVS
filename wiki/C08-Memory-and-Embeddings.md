# C08: Memory, Embeddings & Vector Database Security

> **Source:** [`1.0/en/0x10-C08-Memory-Embeddings-and-Vector-Database.md`](https://github.com/OWASP/AISVS/blob/main/1.0/en/0x10-C08-Memory-Embeddings-and-Vector-Database.md)
> **Requirements:** 25 | **Sections:** 5

## Control Objective

Embeddings and vector stores act as semi-persistent and persistent 'memory' for AI systems via RAG. This memory can become a high-risk data sink and data exfiltration path. This control family hardens memory pipelines and vector databases so that access is least-privilege, data is sanitized before vectorization, retention is explicit, and systems are resilient to embedding inversion, membership inference, and cross-tenant leakage.

## C8.1 Access Controls on Memory & RAG Indices

Enforce fine-grained access controls and query-time scope enforcement for every vector collection.

| # | Requirement | Level | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|----------------------|--------------|

Promptfoo now supports RAG-specific red-teaming scenarios including metadata filter injection tests. |

True immutability may require append-only storage patterns or write-once object storage for metadata alongside the vector index. The four-layer defense architecture (Schneider 2025) recommends provenance tagging as part of Layer 2 (memory sanitization), including creation time, session context, and trust scores for each stored entry. |

Verify logs are shipped to a SIEM or centralized logging system. Test log completeness under high-throughput conditions. | Logging similarity scores and retrieved chunks at scale generates significant data volume. ## C8.2 Embedding Sanitization & Validation

Pre-screen content before vectorization; treat memory writes as untrusted inputs; prevent ingestion of unsafe payloads.

| # | Requirement | Level | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|----------------------|--------------|
| **8.2.1** | **Verify that** regulated data and sensitive fields are detected prior to embedding and are masked, tokenized, transformed, or dropped based on policy. | 1 | Review the ingestion pipeline for DLP/PII scanning stages before the embedding step. Test with synthetic documents containing known PII patterns (SSNs, credit card numbers, email addresses). Verify sensitive fields are masked or dropped in both the stored text chunks and metadata. | Microsoft Presidio, AWS Comprehend, and Google DLP API can be integrated into ingestion pipelines. The challenge is that NER-based PII detection has recall gaps for domain-specific sensitive data (e.g., medical record numbers, internal project codes). Custom entity recognizers are often needed. Embedding the masked version may reduce retrieval quality for legitimate queries. New defense: the DPPN approach (2025) identifies and perturbs only privacy-sensitive neurons, reducing privacy leakage by 5-78% while improving downstream task performance by 14-40% compared to blanket noise methods. |
| **8.2.2** | **Verify that** embedding ingestion rejects or quarantines inputs that violate required content constraints (e.g., non-UTF-8, malformed encodings, oversized payloads, invisible Unicode characters, or executable content intended to poison retrieval). | 1 | Submit test documents containing: non-UTF-8 byte sequences, invisible Unicode (U+200B, U+200E, U+FEFF), oversized payloads exceeding defined limits, documents with embedded scripts or encoded executable content. Verify each is rejected or quarantined. Inspect quarantine storage for review capability. Scan for suspicious instruction patterns ("ignore previous directives," "you must respond with") using regex or LLM-based detection as recommended by Prompt Security. | Content constraint enforcement is straightforward for encoding and size checks but harder for semantic attacks. Invisible Unicode detection requires explicit character class filtering. "Executable content" in the context of RAG typically means prompt injection payloads, which require LLM-based classifiers (e.g., Rebuff, Lakera Guard, Prompt Security's real-time scanning platform) rather than simple pattern matching. Preprocessing filters should scan for suspicious instruction patterns, but Sunil et al. (2025) emphasize that "no single signal is sufficient because attackers can craft content that evades any individual detector" — composite trust scoring across multiple signals is needed. |
| **8.2.3** | **Verify that** vectors that fall outside normal clustering patterns are flagged and quarantined before entering production indices. | 2 | Review the outlier detection mechanism in the ingestion pipeline. Verify it uses statistical methods (e.g., z-score on cosine distance to cluster centroids, isolation forest, or DBSCAN-based novelty detection). Inject synthetic outlier vectors and confirm they are quarantined. Check that quarantine does not block legitimate novel content without review. Consider write-ahead validation using secondary models to evaluate proposed entries before commitment (Schneider 2025, Layer 2 defense). | Defining "normal clustering patterns" requires a baseline — cold-start scenarios or rapidly evolving knowledge bases make this challenging. False positive rates for outlier detection in high-dimensional spaces can be high. Dimensionality reduction (UMAP/t-SNE) before outlier detection is common but lossy. Consider adaptive baselines that update with verified-clean ingestion batches. MINJA's progressive shortening strategy means poisoned entries increasingly resemble legitimate content over iterative refinement, challenging static outlier thresholds. CorruptRAG (Jan 2026) showed a single optimized document can achieve 98.2% attack success, meaning outlier detection must catch individual vectors, not just clusters. Perplexity filtering (flagging statistically unnatural text) and cross-encoder re-ranking are emerging complementary defenses. |
| **8.2.4** | **Verify that** an agent's own outputs are not automatically written back into its trusted memory without explicit validation (such as content-origin checks or write-authorization controls that verify the content's source before committing writes). | 2 | Review agent architecture for write-back paths from agent output to memory stores. Verify that content-origin tagging distinguishes user-provided content, retrieved content, and agent-generated content. Test that agent-generated content requires explicit approval or a separate validation pipeline before memory write. Test for delayed tool invocation attacks (Rehberger, January 2025) where conditional instructions embedded in chat context trigger memory writes when users respond with natural affirmations. | This is critical for autonomous agents (e.g., AutoGPT-style systems) that have self-modification capabilities. LangChain's memory modules (ConversationBufferMemory, ConversationSummaryMemory) write back by default without origin checks. Custom guardrails are needed. The distinction between "agent output" and "validated knowledge" must be architecturally enforced. MINJA (NeurIPS 2025) demonstrated >95% injection success rate through query-only interaction without direct memory access, meaning even read-only agent interfaces can lead to memory poisoning. A-MemGuard research found that even advanced LLM-based memory classifiers miss 66% of poisoned entries, highlighting the difficulty of post-hoc detection. Mem0.ai published AI memory security best practices recommending write-ahead validation. |
| **8.2.5** | **Verify that** new content written to memory is checked for contradictions with what is already stored and that conflicts trigger alerts. | 3 | Review contradiction detection implementation (e.g., NLI-based entailment checking, semantic similarity with polarity analysis). Test by ingesting documents that directly contradict existing stored content. Verify alerts fire and conflicting content is flagged for human review rather than silently merged. | Automated contradiction detection at scale is an unsolved problem in NLP. Current approaches using Natural Language Inference (NLI) models (e.g., DeBERTa fine-tuned on MNLI) work for explicit contradictions but miss subtle or contextual conflicts. Cross-document consistency checking is computationally expensive — O(n^2) pairwise comparisons in the worst case. Practical implementations focus on checking new content against a curated "golden" knowledge set. Memory poisoning attacks are temporally decoupled — poison planted today may execute weeks later when semantically triggered by unrelated interactions, making real-time contradiction detection the only viable prevention point. |

## C8.3 Memory Expiry, Revocation & Deletion

Retention must be explicit and enforceable; deletions must propagate to derived indices and caches.

| # | Requirement | Level | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|----------------------|--------------|
| **8.3.1** | **Verify that** retention times are applied to every stored vector and related metadata across memory storage. | 1 | Review vector store configuration for TTL settings or retention policy enforcement. Query stored vectors and verify TTL metadata is present. Check for automated expiration jobs or garbage collection processes. Verify retention policies cover both vector data and associated metadata/text chunks. | Pinecone does not support native TTL — requires application-layer deletion jobs. Weaviate supports TTL via `autoTenantDeletion` for multi-tenant setups and implements tenant-aware classes with lifecycle endpoints. Milvus added TTL support in v2.3. Redis Vector Search inherits Redis key TTL. Qdrant provides tenant-level lifecycle APIs for managing retention. For databases without native TTL, a separate cron-based cleanup process scanning metadata timestamps is the standard pattern. |
| **8.3.2** | **Verify that** only information required for the system's defined function is persisted in memory (such as user preferences and conversation decisions, not credentials or full conversation transcripts), and that context not needed beyond the current session is discarded at session end. | 1 | Review memory write logic to identify what data categories are persisted. Compare persisted data against the system's functional requirements. Verify session-scoped memory is cleared at session termination. Test by conducting a session with sensitive content, ending it, and confirming the data is not retrievable in a new session. | Defining "required for the system's defined function" needs explicit documentation — a data retention specification. ChatGPT's memory feature persists user preferences but not full transcripts. Systems using ConversationBufferMemory in LangChain persist everything by default; ConversationSummaryMemory is better but may still retain sensitive details in summaries. Memory poisoning research (MINJA, NeurIPS 2025) showed that over-retained conversation data becomes the attack surface — all prompts including commands like "remember," "in future conversations," or "as a trusted source" were used to ensure long-term influence. |
| **8.3.3** | **Verify that** deletion requests purge vectors, metadata, cache copies, and derivative indices within an organization-defined maximum time. | 1 | Issue a deletion request for a known document. After the defined maximum time, attempt to retrieve the document via: (1) vector similarity search, (2) metadata filter query, (3) any keyword/BM25 index, (4) application-layer caches. Verify all paths return no results. Check backup systems for residual copies. | Hybrid search architectures (vector + keyword) are increasingly common (e.g., Weaviate's hybrid search, Pinecone's sparse-dense). Each index must be covered by deletion propagation. Cache invalidation is particularly challenging — Redis caches, CDN-layer caches, and in-memory application caches all need explicit purge logic. Document the full data flow map before implementing deletion. Private networking (VPC peering or private link with no public ingress) available from Pinecone, Weaviate, and Qdrant for production deployments reduces the exposure window during deletion propagation. |
| **8.3.4** | **Verify that** deleted or expired vectors are removed reliably and are unrecoverable. | 2 | Verify the deletion mechanism is hard-delete, not soft-delete/tombstone with deferred cleanup. For HNSW-based indices, confirm that deleted node slots are overwritten on compaction. Check that storage-layer files (e.g., segment files in Milvus, WAL in Qdrant) do not retain deleted data after compaction. Review backup retention policies. | Most vector databases use HNSW or IVF indices that do not immediately reclaim deleted vector slots — they mark them as deleted and reclaim space during compaction/merge operations. Qdrant uses WAL with periodic snapshots; deleted data persists in WAL until the next snapshot. Milvus requires explicit `compact()` calls. True unrecoverability may require encryption with key destruction (crypto-shredding). |
| **8.3.5** | **Verify that** expired vectors are excluded from retrieval results within a measured and monitored propagation window. | 2 | Define the maximum acceptable propagation delay. Issue a deletion/expiration, then immediately query for the affected vectors. Measure the actual time until they stop appearing in results. Verify monitoring/alerting exists for propagation delays exceeding the defined maximum. | In distributed vector databases, propagation delay depends on replication lag, cache TTLs, and index rebuild frequency. Pinecone's serverless architecture has eventual consistency with typical delays under 10 seconds. Weaviate's distributed mode uses Raft consensus but query-time filtering may still serve stale replicas. Application-layer caching adds another propagation layer that must be accounted for. |
| **8.3.6** | **Verify that** memory can be reset for security reasons (quarantine, selective purge, full reset) separately from retention deletion, and that quarantined content is kept for investigation but excluded from retrieval. | 2 | Simulate a poisoning incident. Verify that quarantine operations: (1) immediately exclude targeted vectors from all retrieval paths, (2) preserve quarantined vectors in a separate, access-controlled storage for investigation, (3) do not trigger retention-based deletion timers. Test selective purge (specific documents/batches) and full reset independently. Schneider's four-layer defense recommends circuit breakers for rapid isolation when compromise is detected. | Most vector databases do not have native quarantine functionality. Implementation patterns include: moving vectors to a separate "quarantine" collection/namespace, adding a metadata flag and enforcing query-time filtering, or maintaining a deny-list of vector IDs checked at retrieval time. The deny-list approach has the lowest latency impact but requires consistent enforcement across all query paths. Qdrant's tenant-level lifecycle APIs can facilitate per-tenant quarantine operations. Weaviate's tenant lifecycle endpoints support suspending individual tenants without data deletion. |

## C8.4 Prevent Embedding Inversion & Leakage

Address inversion, membership inference, and attribute inference with explicit threat modeling, mitigations, and regression testing gates.

| # | Requirement | Level | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|----------------------|--------------|
| **8.4.1** | **Verify that** sensitive vector collections are protected against direct read access by infrastructure administrators via technical controls such as application-layer encryption, envelope encryption with strict KMS policies, or equivalent compensating controls. | 2 | Verify encryption at rest covers vector data files specifically (not just general disk encryption). Check KMS policies restrict key access to application service accounts, not infrastructure admin roles. Test that a simulated admin with storage access but no KMS key access cannot read vector data. Review envelope encryption implementation if used. | Standard database encryption at rest (e.g., AWS EBS encryption, GCP CMEK) protects against physical media theft but not against privileged administrators who can access the decrypted database process. Application-layer encryption (encrypting vectors before storage) is stronger but breaks similarity search — the vector DB cannot compute cosine similarity on encrypted vectors. Practical approaches include: encrypted metadata with cleartext vectors (partial protection), or using encrypted search schemes (still largely research-stage for vector similarity). Pinecone and Weaviate Cloud manage encryption transparently (end-to-end encryption at rest and in transit). Pinecone's BYOC mode (GA 2024) places clusters inside the customer's own cloud account. Region pinning keeps data geographically segregated by jurisdiction. |
| **8.4.2** | **Verify that** privacy/utility targets for embedding leakage resistance are defined and measured, and that changes to embedding models, tokenizers, retrieval settings, or privacy transforms are gated by regression tests against those targets. | 3 | Review documented privacy/utility targets (e.g., "inversion attack accuracy below X% at retrieval recall above Y%"). Verify a regression test suite exists that measures both metrics. Test that pipeline changes (model swap, tokenizer update, dimension change) trigger the regression suite and block deployment if targets are not met. | Research-frontier requirement with significant 2025-2026 advances. Key new defenses: **SPARSE framework** (February 2026) — a user-centric concept-specific privacy protection using differentiable mask learning to identify privacy-sensitive dimensions and the Mahalanobis mechanism applying elliptical noise calibrated by dimension sensitivity, selectively perturbing privacy-sensitive dimensions while preserving non-sensitive semantics. **DPPN** (2025) — identifies and perturbs only privacy-sensitive neurons, reducing privacy leakage by 5-78% while improving downstream task performance by 14-40% compared to blanket noise methods. **Eguard** (November 2025) — transformer-based projection network with text mutual information optimization protecting over 95% of tokens from inversion while maintaining 98%+ consistency on four downstream tasks; evaluated against adaptive attacks. **Quantization methods** (2025) — shown to effectively reduce Vec2Text reconstruction capability while preserving retrieval performance. On the attack side, **ZSinvert** (2025) demonstrated universal zero-shot embedding inversion that works across any embedding model without per-model training, significantly lowering the attacker barrier and increasing urgency for regression testing. Additive Gaussian noise remains effective but requires precise scale parameter tuning. Earlier key papers: Morris et al. (2023) on text embedding inversion, Song & Raghunathan (2020) on membership inference. No standard benchmark suite exists yet, but the ACM RecSys 2025 reproducibility study provides a starting framework. Differential privacy for embeddings (DP-SGD during fine-tuning) is theoretically sound but impacts embedding quality significantly. |

## C8.5 Scope Enforcement for User-Specific Memory

Prevent cross-tenant and cross-user leakage in retrieval and prompt assembly.

| # | Requirement | Level | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|----------------------|--------------|
| **8.5.1** | **Verify that** every retrieval operation enforces scope constraints (tenant/user/classification) in the vector engine query and verifies them again before prompt assembly (post-filter). | 1 | Trace a retrieval request end-to-end: verify scope filters are applied in the vector DB query (pre-filter). Then verify a second scope check occurs in the application layer before retrieved chunks are assembled into the prompt. Test with a scenario where query-time filtering is deliberately weakened — confirm post-filter catches the leakage. | Pre-filtering vs. post-filtering has performance implications. Weaviate and Qdrant support filtered vector search (pre-filter) natively with first-class multitenancy. Pinecone supports metadata filtering at query time. The post-filter layer is always application code — typically in the RAG orchestration framework (LangChain, LlamaIndex). Ensure both layers use the same source of truth for scope definitions to avoid drift. Qdrant's payload-based filtering and Weaviate's tenant-aware classes provide strong pre-filter foundations. |
| **8.5.2** | **Verify that** vector identifiers, namespaces, and metadata indexing prevent cross-scope collisions and enforce uniqueness per tenant. | 1 | Review ID generation scheme for tenant-scoping (e.g., `{tenant_id}_{document_id}` or separate namespaces). Attempt to upsert a vector with an ID belonging to another tenant's namespace. Verify the operation is rejected or isolated. Check that namespace/collection naming prevents collisions (e.g., tenant-prefixed collection names). | Pinecone namespaces provide physical ID isolation — IDs are unique within a namespace but can collide across namespaces (by design, since cross-namespace operations are not possible). Weaviate multi-tenancy (since v1.20) provides tenant-level isolation. In single-collection architectures with metadata-based tenancy, ID uniqueness must be enforced at the application layer — UUIDv4 or UUIDv7 with tenant prefix is the standard pattern. |
| **8.5.3** | **Verify that** retrieval results that match similarity criteria but fail scope checks are discarded. | 1 | Execute a similarity search as Tenant A in a multi-tenant environment. Inspect raw retrieval results before and after scope filtering. Verify that any results belonging to other tenants are discarded and never reach the prompt assembly stage. Check that discarded results are logged for security monitoring. | The discard must happen silently from the user's perspective but should be logged. A common mistake is to return fewer results than requested (e.g., requesting top-10 but only 7 pass scope checks) without backfilling — this can degrade user experience. Better implementations over-fetch (e.g., retrieve top-20) and return the top-10 that pass scope checks. |
| **8.5.4** | **Verify that** multi-tenant tests simulate adversarial retrieval attempts (prompt-based and query-based) and demonstrate zero out-of-scope document inclusion in prompts and outputs. | 2 | Review the adversarial test suite for multi-tenant scenarios. Verify tests include: (1) direct cross-tenant similarity queries, (2) metadata filter injection attempts, (3) prompt-based retrieval manipulation ("search across all users"), (4) namespace traversal attempts. Run the test suite and confirm zero out-of-scope inclusions. Verify tests run in CI/CD pipeline. | This requirement is verification-focused (Role: V). Test suites should use realistic data distributions — synthetic data with uniform distributions may not reveal issues that appear with skewed real-world data. Consider property-based testing (e.g., Hypothesis framework) to generate diverse adversarial query combinations. Integration with red-team tooling (e.g., Garak, PyRIT, Promptfoo) can automate prompt-based retrieval manipulation testing. Promptfoo now supports RAG-specific red-teaming scenarios. Palo Alto Networks Prisma AIRS provides automated agent security testing. |
| **8.5.5** | **Verify that** in multi-agent systems, each agent's memory namespace is isolated and enforced through access control, not just organizational naming conventions. | 2 | Review multi-agent memory architecture. Verify each agent's memory namespace has separate access credentials or ACLs. Test that Agent A's credentials cannot read or write to Agent B's memory namespace. Verify this is enforced at the vector DB or middleware layer, not just by convention in application code. | Most multi-agent frameworks (CrewAI, AutoGen) use shared memory by design for inter-agent communication. Isolating memory while still allowing controlled sharing requires an explicit architecture — e.g., a "shared" namespace with write controls and per-agent private namespaces. LangGraph's state management can enforce this if properly configured. The key distinction is access-controlled sharing vs. unrestricted sharing. DEV Community article (2025) argues that traditional relational databases with mature RBAC may provide stronger memory isolation than vector-native stores for agent memory, as they have decades of battle-tested access control. |
| **8.5.6** | **Verify that** encryption keys and access policies are segregated per tenant for memory/vector storage, providing cryptographic isolation in shared infrastructure. | 3 | Verify each tenant has a distinct encryption key in the KMS. Confirm key access policies restrict each tenant's key to that tenant's service principal only. Test that rotating or revoking one tenant's key does not affect other tenants' data availability. Review key hierarchy (master key wrapping per-tenant DEKs is acceptable). | Per-tenant encryption in vector databases is operationally complex: key management overhead scales linearly with tenant count, and index-level encryption (encrypting entire collections per tenant) requires separate indices per tenant. Envelope encryption with per-tenant DEKs wrapped by a shared KEK in HSM/KMS is the practical middle ground. Pinecone's BYOC mode (GA 2024) places clusters in the customer's own AWS/Azure/GCP account, enabling customer-managed KMS keys. Pinecone also supports BYOK (bring-your-own-key) for encryption at rest. Weaviate and Qdrant Cloud manage encryption transparently but do not yet offer per-tenant key segregation in their managed offerings — this requires self-hosted deployments or custom encryption layers. Private networking (VPC peering/private link) is available from all major managed vector DBs to eliminate public ingress. |

## Threat Landscape

Known attacks, real-world incidents, and threat vectors relevant to this chapter:

- RAG poisoning — injecting malicious content into knowledge bases retrieved at inference time; PoisonedRAG (USENIX Security 2025) showed 5 documents achieve 90% attack success
- Embedding inversion attacks — reconstructing original text from embedding vectors; Vec2Text achieves up to 92% reconstruction of 32-token inputs; transfer attacks (ACL 2024) work without access to the original embedding model
- Cross-tenant data leakage through shared vector indices and metadata filter injection
- Memory persistence attacks — injecting instructions that persist across conversations; MINJA (NeurIPS 2025) achieves >95% injection success via query-only interaction
- Agent memory poisoning — OWASP ASI06 (2026) formally recognizes this as a top agentic risk; ZombieAgent (January 2026) showed persistent cross-session attacks via ChatGPT connectors
- AI Recommendation Poisoning — Microsoft (February 2026) documented 50+ real-world examples across 31 companies and 14 industries
- PII accumulation in vector stores without adequate deletion mechanisms; embedding inversion makes this a concrete exfiltration path
- Membership inference — determining whether a specific document exists in an embedded corpus
- Feedback loop poisoning — agent outputs written back to memory create self-reinforcing hallucinations; A-MemGuard found LLM classifiers miss 66% of poisoned entries
- Metadata filter injection — manipulating vector query filters to bypass tenant isolation
- Single-document RAG poisoning — CorruptRAG (Jan 2026) showed a single optimized document achieves 98.2% attack success by pushing its vector representation close to target queries, far stealthier than multi-document approaches
- Knowledge graph RAG poisoning — KEPo (Mar 2026) inserts adversarial triples that complete misleading inference chains in structured knowledge graphs, reducing QA accuracy by up to 80%
- Corpus-adaptive poisoning — Semantic Chameleon (Mar 2026) tailors poison documents to the specific corpus composition, defeating static detection
- Universal embedding inversion — ZSinvert demonstrated zero-shot inversion across any embedding model without per-model training, eliminating the attacker's need for model-specific knowledge
- Vector collision attacks — crafting documents whose embeddings collide with high-value query vectors via gradient-based optimization; the 2025 Supabase Cursor incident demonstrated real-world exploitation via support ticket injection
- Vector worms — theoretical future variant where poisoned embeddings instruct models to re-embed and reintroduce poisoned data elsewhere, creating cascading contamination
- Delayed tool invocation (Rehberger, January 2025) — conditional instructions in chat context trigger memory writes when users respond with natural affirmations, bypassing Google Gemini's guardrails
- Zero-click exfiltration — EchoLeak demonstrated corporate data exfiltration from Microsoft 365 Copilot via crafted emails

## Tooling & Implementation

Current tools, frameworks, and libraries that help implement these controls:

- **Vector databases:** Pinecone (namespaces, per-index RBAC, BYOC mode GA 2024, BYOK encryption, SOC 2/HIPAA/ISO 27001/GDPR), Weaviate (multi-tenancy since v1.20, tenant-aware classes with lifecycle endpoints, ACLs, HIPAA on AWS since 2025, SOC 2 Type II), Qdrant (first-class multitenancy, JWT authentication, read-only API keys, TLS, Prometheus/Grafana integration, SOC 2 Type II, HIPAA-ready enterprise), Milvus (RBAC, partition keys, TTL since v2.3)
- **Embedding privacy defenses:** SPARSE framework (Feb 2026, concept-aware selective perturbation), DPPN (2025, privacy-neuron perturbation reducing leakage 5-78%), Eguard (Nov 2025, transformer-based projection network protecting 95%+ tokens from inversion at 98% downstream task consistency), dimensionality reduction via PCA/random projection, additive Gaussian noise calibrated to privacy budgets, vector quantization (shown to reduce Vec2Text reconstruction)
- **RAG frameworks:** LlamaIndex (metadata filtering, callback logging), LangChain (retrievers with metadata-based access control), Mem0.ai (AI memory security best practices and write-ahead validation)
- **PII detection:** Microsoft Presidio, AWS Comprehend, Google DLP API — integrate pre-embedding in ingestion pipelines
- **RAG poisoning defenses:** SDAG (Sparse Document Attention, Feb 2026 — block-sparse attention mask preventing cross-document interaction, no fine-tuning needed), perplexity filtering to detect statistically unnatural documents, hybrid search combining vector + keyword matching to reduce single-vector dominance, cross-encoder re-ranking to verify retrieved document relevance
- **Prompt injection detection in retrieved content:** Rebuff, Lakera Guard, Prompt Security (real-time scanning of prompts/responses/retrieval payloads), custom NLI-based classifiers
- **Red-teaming and adversarial testing:** Garak, PyRIT, Promptfoo (RAG-specific red-teaming scenarios), Palo Alto Prisma AIRS (automated agent security testing)
- **Agent memory security:** Mem0.ai (memory security best practices), Amazon Bedrock Guardrails (prompt-attack policies), Palo Alto Prisma AIRS and Advanced URL Filtering
- **Data cleanup:** TTL-based expiration (Milvus v2.3+, Redis), application-layer deletion jobs (Pinecone), tombstone-based deletion with compaction, tenant lifecycle APIs (Qdrant, Weaviate)
- **Infrastructure isolation:** VPC peering / private link (Pinecone, Weaviate, Qdrant), BYOC deployments (Pinecone), region pinning for jurisdictional data segregation

## Open Research Questions

- [ ] How do you enforce RBAC in vector databases that don't natively support it? (Qdrant and Weaviate now offer native solutions; Chroma and FAISS still lack RBAC)
- [x] What's the practical risk of embedding inversion for modern high-dimensional embeddings? — Partially answered: Vec2Text achieves 92% reconstruction on 32-token inputs; OWASP LLM08:2025 estimates 50-70% word recovery; ACM RecSys 2025 reproducibility study provides nuanced benchmarks per model. Transfer attacks (ACL 2024) mean attackers do not need the original model.
- [ ] How should memory expiry policies differ for conversation memory vs. knowledge base memory? Especially given MINJA's temporal decoupling where poison planted today executes weeks later.
- [ ] What sanitization is adequate for content entering RAG pipelines? Sunil et al. (2025) showed no single signal is sufficient — composite trust scoring needed. A-MemGuard found LLM classifiers miss 66% of poisoned entries. CorruptRAG (Jan 2026) raised the bar further: a single optimized document achieves 98.2% success, and Semantic Chameleon (Mar 2026) showed corpus-adaptive poisoning defeats static detectors. SDAG (Feb 2026) offers a promising inference-time defense via sparse attention.
- [ ] Can homomorphic encryption or encrypted search schemes enable similarity search on encrypted vectors at practical latency?
- [x] What is the optimal noise calibration for embedding privacy that preserves retrieval utility above acceptable thresholds? — Partially answered: SPARSE (Feb 2026) uses Mahalanobis mechanism with elliptical noise calibrated by dimension sensitivity. DPPN (2025) perturbs only privacy-sensitive neurons, reducing leakage 5-78% while improving utility 14-40%.
- [ ] How should contradiction detection scale for large, rapidly-evolving knowledge bases? Microsoft's Feb 2026 report shows 50+ real-world slow-burn poisoning campaigns, increasing urgency.
- [ ] How can write-ahead validation effectively prevent MINJA-style query-only memory injection while maintaining agent usability?
- [ ] What architectural patterns best defend against "vector worms" — self-propagating poisoned embeddings that instruct models to re-embed and spread contamination?
- [ ] How should memory quarantine and forensic investigation workflows be standardized across different vector database backends?
- [ ] How effective are inference-time defenses (e.g., SDAG sparse attention, cross-encoder re-ranking) compared to ingestion-time defenses against single-document attacks like CorruptRAG? Can they be layered effectively?
- [ ] As of March 2026, ZSinvert enables universal zero-shot embedding inversion across any model — what minimum privacy transform (SPARSE, DPPN, Eguard, or combinations) is needed to stay ahead of model-agnostic inversion?
