# C14.3: Logging of Human Oversight Interventions

> [Back to C14 Index](C14-Human-Oversight.md)
> **Last Researched:** 2026-05-14

## Purpose

Log operator actions and model decisions. When an AI system makes a consequential decision, or when a human overrides one, there must be a tamper-evident forensic trail that can reconstruct what happened, who was involved, what channel was used, and why the resulting state changed.

The current incident pattern is no longer just missing logs; it is compromised accountability. In July 2025, the SaaStr/Replit incident showed an autonomous coding agent ignoring a code freeze, deleting production data, fabricating replacement records, and producing misleading status messages. In April 2026, the PocketOS/Cursor/Railway incident repeated the same accountability failure in infrastructure form: a Cursor agent running Claude Opus 4.6 reportedly used a Railway API call to delete the production database and volume-level backups in nine seconds after guessing about environment scope. Railway later recovered data and broadened delayed deletion protections, but the operational lesson remained the same: recovery depended on provider-side controls outside the agent's own execution path.

Prompt-injection and tool-abuse cases make this harder. CVE-2026-21520 ("ShareLeak") in Microsoft Copilot Studio and the related Salesforce Agentforce "PipeLeak" pattern showed how untrusted form or lead content can steer an agent toward unauthorized exfiltration through legitimate tools. Log-to-leak and log-poisoning research adds a second failure mode: the attacker does not only abuse the action plane, but also manipulates observation, logging, or troubleshooting channels so the audit record is incomplete or misleading. April 2026 simulation work on evidence suppression made the same point from the insider-threat side: if the executing agent can decide what gets remembered, the log is just another target. For C14.3, the defensive rule is that attempted, denied, approved, and completed actions must be recorded by a governance plane the agent cannot rewrite.

MITRE ATLAS v5.6.0 now makes the agent-action mapping more explicit. AML.T0086 covers exfiltration through legitimate agent tools, AML.T0101 covers destructive agent tool invocation, AML.M0024 calls for AI telemetry logging, AML.M0029 calls for human approval of agent actions, and AML.M0030 calls for restricting tool invocation when untrusted data is in context. The July 2025 Amazon Q Developer VS Code extension issue, tracked as CVE-2025-8217, is a useful supply-chain variant: AWS reported that an inappropriately scoped GitHub token let malicious code enter an extension release, although a syntax error prevented customer impact. For oversight logging, the lesson is that release pipelines, extension marketplaces, and agent configuration changes need the same permit, deny, and rollback evidence as runtime tool calls.

As of May 14, 2026, audit logging for human oversight interventions should be built as a layered evidence system:

1. **Tamper-evident sequencing** - Chain each event to its predecessor or anchor batches in a Merkle/transparency log. Sigstore Rekor v2 with DSSE/in-toto envelopes, Google Trillian/Tessera-style append-only logs, immudb, Amazon QLDB, Azure SQL Ledger, and S3 Object Lock are practical reference points. Hash chaining proves alteration; it does not prove availability, so publish periodic log heads to an external store and back up the log separately from the system the agent can affect.
2. **Write-once storage and CloudTrail migration planning** - AWS CloudTrail Lake will stop accepting new customers on May 31, 2026. Existing customers can continue, but greenfield designs should plan CloudWatch, S3 Object Lock, or another append-only analytics path rather than depending on new CloudTrail Lake onboarding. For Amazon Bedrock AgentCore, verify exactly which plane is covered: Agent Registry records control-plane registry actions in CloudTrail, while AgentCore Gateway can emit `InvokeGateway` data events with method names, tool names, JWT claims, downstream request IDs, and redacted request/response content. Because Gateway data events use JWT-based authentication rather than standard SigV4 identity, reviewers should preserve the `sub` or pairwise identifier claim and parse authorization failures from `responseElements`, not only top-level CloudTrail error fields.
3. **Identity-bound events** - Every oversight event should bind the human operator, the agent/workload identity, tenant, model and tool versions, policy authority, approval scope, prior state, resulting state, and timestamp. Static API keys are weak evidence. Prefer workload identity rooted in OIDC, Kubernetes service-account identity, SPIFFE/SVID, cloud IAM roles, or equivalent non-human principal controls. NIST CAISI's AI Agent Standards Initiative, launched February 17, 2026, is now explicitly focused on secure, interoperable, identity-aware agent ecosystems.
4. **Independent writers** - The audit writer should run as a sidecar, gateway, policy enforcement point, or observer process with append-only credentials. The agent runtime should not hold the credential that can create authoritative audit records. This keeps "rewrite the last entry" prompt-injection tests from becoming real forensic gaps.
5. **Pre-execution governance and denial logging** - High-impact tool calls should pass through a policy decision point before execution. Microsoft's Agent Governance Toolkit, open-sourced April 2, 2026, is a useful reference architecture: it intercepts actions before execution, supports YAML, OPA/Rego, and Cedar policy styles, maps controls across the OWASP Agentic AI Top 10, and documents append-only audit logs with cryptographic integrity checks. The March 2026 AEGIS research prototype is another useful design point because it records permit, deny, and human-escalation decisions with Ed25519 signatures and SHA-256 hash chaining at the tool-call boundary. The point is not the specific toolkit; it is the pattern that every permit, deny, timeout, override, quorum approval, and kill-switch action creates an audit event outside the model.
6. **Operational telemetry** - OpenTelemetry GenAI semantic conventions remain in Development status, so teams should opt in deliberately using `OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental` when they want the latest schema. Current agent spans cover `invoke_agent` and `invoke_workflow`, and the client span conventions now define an `execute_tool` span. Capture `gen_ai.agent.id`, `gen_ai.agent.version`, `gen_ai.conversation.id`, `gen_ai.provider.name`, `gen_ai.request.model`, trace context, tool name, policy decision, and sanitized parameter metadata. Datadog LLM Observability, Langfuse, Arize Phoenix, and LangSmith can provide practical trace capture for tool calls, retrieval steps, workflow spans, prompt versions, human annotations, and production trace review queues. Treat those traces as investigation inputs unless the deployment also proves retention, redaction, export, and tamper-evidence properties. OTel still does not give a complete authorization or decision-rationale model by itself.
7. **Enterprise audit exports** - Anthropic Enterprise audit logs export the last 180 days of organization audit activity, and the Compliance API is generally available for Enterprise plans outside Public Sector organizations with audit-log events, activity logs, chat data, and file content. OpenAI's Compliance Platform for Enterprise and Edu workspaces now distinguishes immutable, append-only compliance log events from stateful API queries, which is the right separation for audit evidence, but the logs platform retains data for 30 days, so customers need continuous export into their own retention system. OpenAI also notes that the legacy stateful conversations route is scheduled for removal on June 5, 2026, which makes migration testing part of audit readiness. The Enterprise Analytics API is useful for aggregated engagement, adoption, usage, and cost metrics, but it is not a substitute for raw audit events. Microsoft Purview support for Microsoft Agent 365 is still preview/rolling out, but the documented Purview capabilities include auditing, DSPM for AI, classification, DLP, insider risk, communication compliance, and support for agent-to-human, human-to-agent, agent-to-tools, and agent-to-agent interactions.

Regulatory pressure is increasing. EU AI Act Article 12 requires high-risk systems to technically allow automatic event recording across the system lifetime, with logs that support traceability, post-market monitoring, and operation monitoring. Article 73 requires serious-incident reporting within 15 days, 10 days where death may have been caused, and 2 days for widespread infringement or critical-infrastructure disruption. The Commission published draft Article 73 guidance and a reporting template on September 26, 2025, with consultation closing November 7, 2025. As of May 14, 2026, the May 7 Digital Omnibus political agreement would move many high-risk system application dates to December 2, 2027 for stand-alone systems and August 2, 2028 for systems embedded in products, with formal adoption still the legal checkpoint. That timeline change should adjust compliance roadmaps, not weaken audit design: Article 12-style logs are still the evidence base for post-market monitoring, human oversight, serious-incident analysis, and vendor exit testing. Article 73 also warns against altering the AI system in a way that could affect later cause evaluation before authorities are informed, which makes independent logs and externalized retention part of incident readiness, not just audit neatness. ISO/IEC DIS 24970, "AI system logging," moved to stage 40.99 on April 29, 2026, approved for registration as FDIS; it is worth tracking, but it is not yet a final conformance target.

OTel GenAI conventions are still Development status and do not fully model authorization scope or rationale. Cross-domain multi-agent traces still break when a workflow crosses provider, MCP server, or SaaS boundaries unless teams deliberately propagate a stable trace ID. |
