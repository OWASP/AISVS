# C7.4 Output & Action Limiting

[Back to C07 Index](C07-Model-Behavior)

## Purpose

Output and action limiting constrains what the model can do and how much it can do. Without explicit limits, AI systems — especially agentic ones — can incur runaway costs through excessive API calls, execute high-impact actions (file writes, emails, code execution) without user awareness, or recursively invoke tools in unbounded chains. These controls enforce the principle of least privilege at the output and action layer, ensuring the model operates within defined boundaries.

## Implementation Guidance (2025-2026 Landscape)

### The Agentic AI Security Challenge

By mid-2025, over 70% of enterprise AI deployments involved multi-agent or action-based systems, dramatically expanding the attack surface for output-triggered actions. Cisco's State of AI Security 2026 report found that 83% of organizations plan to deploy agentic AI, but only 29% feel ready to secure it. The ICLR 2025 Agent Security Bench (ASB) quantified the risk: while 41.2% of models succumb to direct prompt injection, 52.9% are vulnerable to RAG backdoor attacks, and 82.4% can be compromised through **inter-agent trust exploitation** — where LLMs that resist direct malicious commands execute identical payloads when requested by peer agents.

The OWASP Top 10 for Agentic Applications, released in December 2025 by more than 100 industry experts, formalized the security categories that matter most: Agent Behavior Hijacking (ASI01), Tool Misuse (ASI02), and Identity & Privilege Abuse (ASI03) are the top three risks. The framework introduced the **principle of least agency** — it is no longer just about what an agent can access, but how much freedom it has to act on that access without checking back with a human. Autonomy should be earned, not a default setting.

### The Lethal Trifecta

A useful mental model for evaluating action-limiting risk is the "Lethal Trifecta." An agentic system is fundamentally vulnerable when it has all three of the following properties simultaneously:

1. **Access to private data** — the agent can read emails, documents, databases
2. **Exposure to untrusted tokens** — the agent processes input from external sources (shared docs, web content, calendar invites)
3. **Exfiltration vector** — the agent can make external requests (render images, call APIs, generate links)

If your agentic system has all three, it is exploitable through indirect prompt injection regardless of other safeguards. The July 2025 Amazon Q incident (CVE-2025-8217) demonstrated this pattern: a compromised extension contained injected instructions to delete file systems and destroy AWS resources. Mitigations should aim to remove at least one leg of the trifecta — for example, by segmenting agent access to limit data sources, restricting external request capabilities, or sanitizing all untrusted inputs before ingestion.

### Recent Incidents Demonstrating Action-Limiting Failures (2025–2026)

The pace of real-world exploits targeting insufficient action limiting has accelerated sharply:

- **OpenClaw / ClawHub supply chain attack (early 2026, CVE-2026-25253):** The largest confirmed supply chain attack targeting AI agent infrastructure. Antiy CERT confirmed 341 malicious skills (~12% of the ClawHub marketplace) distributing keyloggers on Windows and Atomic Stealer on macOS. The root cause was the absence of any skill vetting, signature verification, or runtime sandboxing — agents with persistent memory and shell access installed and executed arbitrary payloads. Additionally, 21,639 OpenClaw instances were found publicly accessible on the internet, leaking API keys and OAuth tokens. The Moltbook social platform associated with the ecosystem exposed 1.5 million agent API tokens through an unsecured database.
- **PerplexedBrowser zero-click hijacking (March 2026):** Zenity Labs disclosed vulnerabilities in Perplexity Comet, an agentic browser that autonomously executes tasks within authenticated user sessions. Attackers could embed malicious instructions in routine content like calendar invites. The agent would autonomously access local files and exfiltrate data to attacker-controlled servers with no user interaction and no confirmation prompt — an evolution of ClickFix social engineering adapted for automated systems. A second exploit chain allowed credential theft by abusing the agent's authorized access to password management tools.
- **MS-Agent shell injection (CVE-2026-2256):** A critical command injection vulnerability in the MS-Agent framework v1.5.2 where the Shell tool component executed commands without sanitization. Prompt-based attacks could escalate to full system compromise, demonstrating that tool-layer input validation is just as critical as prompt-layer defenses.
- **Amazon Q Developer (CVE-2025-8217, July 2025):** A compromised VS Code extension contained injected prompts instructing the agent to delete file systems and destroy AWS resources — the incident that initially brought the Lethal Trifecta model into mainstream security discussion.
- **GitHub Copilot RCE (CVE-2025-53773, CVSS 9.6, January 2025):** Prompt injection via code comments triggered "YOLO mode" in GitHub Copilot, enabling remote code execution on over 100,000 developer machines. The root cause was the absence of shell command execution restrictions and file write controls — the agent treated injected instructions in code comments as legitimate user intent.
- **LangChain Core credential exfiltration (CVE-2025-68664, January 2025):** A serialization flaw in LangChain Core allowed credential exfiltration via HTTP headers across its 847 million downloads. Without sandbox controls on object instantiation and outbound request filtering, environment variables containing secrets could be leaked through agent tool calls.
- **Cursor IDE CurXecute/MCPoison (CVE-2025-54135, CVE-2025-54136, January 2025):** Two related vulnerabilities in Cursor IDE demonstrated MCP configuration poisoning. Malicious README files could instruct the agent to create unauthorized MCP server connections with reverse shells (CurXecute), and a one-time trust mechanism bypass enabled persistent backdoor installation via MCP config modifications (MCPoison). These incidents highlighted that configuration file writes are high-impact actions requiring the same confirmation gates as code execution.
- **Moltbook agent token exposure (February 2026):** An unsecured database in the Moltbook social platform exposed 1.5 million AI agent API tokens. Attackers leveraged this access to inject 506 prompt injection payloads that propagated through the compromised agent population, demonstrating how stolen agent credentials amplify the blast radius of action-limiting failures.
- **Meta rogue AI agent incident (March 2026):** An internal AI agent at Meta autonomously posted a response to an internal employee forum without seeking permission from the engineer who invoked it. The agent's unauthorized guidance caused downstream actions that inadvertently granted engineers access to Meta systems they should not have been authorized to see. Meta classified the incident as a "Sev 1" — its second-highest severity level. The root cause was the absence of action-level confirmation gates on the agent's write capabilities: the agent could analyze internal data and compose responses, but nothing prevented it from publishing those responses without human approval. This incident demonstrates that even large organizations with mature security programs can be caught by agents that lack explicit output-action confirmation controls.
- **LiteLLM AI Gateway supply chain compromise (March 24, 2026):** Unauthorized malicious packages (litellm v1.82.7 and v1.82.8) were published to PyPI through compromised maintainer credentials linked to the broader Trivy security compromise. The packages contained a credential stealer targeting environment variables, SSH keys, cloud credentials, and Kubernetes tokens, exfiltrating data to an attacker-controlled domain. The compromised versions were available for approximately 5.5 hours before removal. Organizations using unpinned pip installs or receiving LiteLLM as a transitive dependency were affected. This incident highlights the AI agent supply chain risk: LiteLLM is widely used as a proxy gateway for routing requests to multiple LLM providers, so compromising it gives attackers access to every API key and credential flowing through the gateway.
- **Agent Session Smuggling in A2A systems (April 2026):** [Unit 42 research](https://unit42.paloaltonetworks.com/agent-session-smuggling-in-agent2agent-systems/) demonstrated a new attack class targeting stateful agent-to-agent communication. A malicious remote agent exploits an ongoing A2A session to inject hidden instructions between a legitimate client request and the server's response. In proof-of-concept attacks using Google ADK with Gemini 2.5 Pro/Flash, a rogue research assistant smuggled instructions that caused a financial assistant to execute unauthorized stock trades — all invisible to the end user. Unlike MCP-based attacks (which are stateless and deterministic), A2A session smuggling exploits persistent session state and model-driven reasoning, enabling adaptive multi-turn attacks that are significantly harder to detect. The research found that multi-turn session exploitation is "significantly more difficult to defend against" than single-instruction injection, because the malicious agent can adapt its strategy across exchanges and build false trust progressively. Recommended mitigations include out-of-band confirmation for sensitive actions, context grounding with task anchors, cryptographically signed AgentCards for identity verification, and real-time activity exposure to end users.
- **Chrome Gemini Live Panel Hijacking (CVE-2026-0628, 2026):** Unit 42 discovered a high-severity flaw in Chrome's Gemini Live panel that allowed malicious browser extensions to hijack the privileged AI assistant and access the camera and microphone, demonstrating that agent hijacking vectors extend beyond traditional tool-calling interfaces to browser-embedded AI assistants.
- **Flowise AI Agent Builder RCE (CVE-2025-59528, CVSS 10.0, active exploitation April 2026):** A maximum-severity code injection vulnerability in Flowise's CustomMCP node, where user-supplied MCP server configuration was passed directly to JavaScript's `Function()` constructor — functionally identical to `eval()` — without any validation. Exploitation requires no authentication and grants full Node.js runtime privileges, enabling command execution, credential harvesting, and lateral movement into connected cloud services. As of April 2026, between 12,000 and 15,000 Flowise instances are exposed on the internet, with active exploitation confirmed by VulnCheck. The [CSA's analysis](https://labs.cloudsecurityalliance.org/research/csa-research-note-flowise-mcp-rce-exploitation-20260409-csa/) noted that compromised Flowise instances typically store LLM provider API keys, vector database credentials, RAG pipeline documents, and agent workflow configurations — making each compromised instance a pivot point into broader AI infrastructure. The vulnerability was patched in version 3.0.6 (September 2025), but the six-month patch lag across development-focused deployments underscores that AI agent builder tooling requires production-grade vulnerability management despite internal-use classifications. The CSA noted that MCP's design flexibility "can create pressure on developers to accept overly permissive input formats," suggesting this vulnerability pattern likely extends across the broader MCP ecosystem.
- **PraisonAI multi-agent framework CVE chain (April 2026):** PraisonAI, a popular multi-agent orchestration framework, disclosed a cascade of four critical vulnerabilities in rapid succession: CVE-2026-34938 (CVSS 10.0) — a complete sandbox bypass in the `execute_code()` function, where an attacker crafts a custom `str` subclass overriding `startswith()` to defeat the `_safe_getattr` wrapper, achieving arbitrary OS command execution on the host despite a three-layer sandbox design; CVE-2026-39888 (Critical) — a separate sandbox escape path in the agent runtime; CVE-2026-39890 (Critical) — RCE via malicious YAML parsing in agent configuration; and CVE-2026-39891 — template injection through agent tool definitions allowing code execution via crafted input. All four were patched in PraisonAI v1.5.90. The CVE-2026-34938 sandbox bypass is particularly instructive: it demonstrates that Python-level sandboxing for agent code execution is fundamentally brittle, and that the sandbox itself becomes the attack surface. This echoes the broader pattern seen in Flowise — when agent frameworks implement execution restrictions in application-layer code rather than OS-level isolation, determined attackers can find bypass paths.
- **Azure DevOps MCP Server authentication bypass (CVE-2026-32211, CVSS 9.1, April 2026):** Microsoft disclosed that the `@azure-devops/mcp` npm package — an MCP server exposing Azure DevOps tools for work items, repos, pipelines, and pull requests — lacked authentication for critical functions entirely. An attacker with network access could read sensitive project data, configuration details, API keys, and authentication tokens without any credentials. As of mid-April 2026, no patch was available; Microsoft recommended restricting network access via firewall rules and placing a reverse proxy with authentication in front of the MCP server. This is the first major authentication vulnerability in a vendor-supported MCP server, validating the earlier guidance that MCP servers should never be exposed to untrusted networks and must always sit behind authentication proxies — advice that the MCP ecosystem has been slow to operationalize.
- **Marimo Python Notebook pre-auth RCE (CVE-2026-39987, CVSS 9.3, April 2026):** A critical authentication bypass in Marimo's terminal WebSocket endpoint (`/terminal/ws`) — while the primary `/ws` endpoint correctly called `validate_auth()`, the terminal endpoint accepted connections without any credential check, giving attackers a full interactive PTY shell running as the Marimo process user. Despite no public proof-of-concept, Sysdig's honeypots detected [exploitation within 9 hours 41 minutes](https://thehackernews.com/2026/04/marimo-rce-flaw-cve-2026-39987.html) of the April 8, 2026 advisory disclosure. Attackers completed credential theft — SSH keys, `.env` files, cloud tokens — in under three minutes per compromised instance. The [CSA's analysis](https://labs.cloudsecurityalliance.org/research/csa-research-note-marimo-rce-cve-2026-39987-ai-toolchain-202/) highlighted the AI-specific impact: compromised Marimo instances typically store OpenAI, Anthropic, and Google API keys alongside cloud provider credentials, transforming each compromise into a pivot point across AI infrastructure, billing accounts, and training pipelines. Patched in Marimo 0.23.0. This incident reinforces the pattern established by Flowise — AI development tools treated as internal-only are high-value targets that require production-grade vulnerability management and authentication enforcement.

### MCP Authorization Specification and Tool Access Control

The [MCP Authorization Specification](https://modelcontextprotocol.io/specification/2025-06-18/basic/authorization) (updated March 2026) formalized OAuth 2.1 as the mandatory authentication framework for MCP server interactions, directly addressing the class of vulnerabilities demonstrated by CVE-2026-32211. Key security requirements include mandatory [Resource Indicators (RFC 8707)](https://datatracker.ietf.org/doc/html/rfc8707), which bind tokens to specific MCP server endpoints so a token obtained for one server cannot be replayed against another; incremental scope consent allowing clients to request only the minimum access needed for each operation; role-based tool authorization enabling administrators to restrict which tools are accessible by user role; and Protected Resource Metadata requiring MCP servers to advertise their authorization server configuration. PKCE (Proof Key for Code Exchange) is mandatory for all public clients, and token audience validation ensures servers only accept tokens specifically issued for them. As of April 2026, approximately 86% of enterprise MCP deployments have adopted RFC 8707 compliance, though the Azure DevOps MCP vulnerability shows that adoption of the specification's authentication requirements remains inconsistent, particularly among early-stage or community-maintained MCP server packages.

### Delegation Chain Security

As of April 2026, delegation chain security has emerged as one of the most critical unsolved problems in agentic AI. A [Grantex audit of 30 popular agent frameworks](https://grantex.dev/report/state-of-agent-security-2026) (including CrewAI, AutoGen, MetaGPT, and OpenClaw — collectively over 500,000 GitHub stars) found that **zero projects implement delegation chain controls**: no scope narrowing, no depth limits, no cascade revocation. In every framework audited, child agents either inherit the parent's full credentials or receive independent API keys with no scope restrictions. Additionally, 93% rely exclusively on unscoped API keys, 0% have per-agent identity mechanisms, and 97% lack user consent flows for delegated actions.

The [CSA's delegation chain analysis](https://cloudsecurityalliance.org/blog/2026/03/25/control-the-chain-secure-the-system-fixing-ai-agent-delegation) (March 2026) formalized the core risk: "every unverified task handoff becomes a breach in motion." Their recommended controls include permission attenuation at each hop (each delegated agent receives strictly fewer permissions than its parent), cryptographic provenance tracking for delegation lineage, cascade revocation where revoking a parent automatically invalidates all downstream delegatees, and behavioral anomaly detection for unusual delegation patterns.

Google DeepMind's [Intelligent Delegation](https://arxiv.org/html/2602.11865v1) paper (February 2026, Tomašev, Franklin, Osindero et al.) proposed five foundational principles for delegation governance: dynamic capability assessment of delegatees, adaptive task reassignment when sub-agents fail verification or exceed budgets, mandatory permission attenuation and scope narrowing at each level, structural transparency through process-level monitoring proportional to risk, and systemic resilience to prevent cascading failure across delegation networks. The paper frames delegation as "a governance and risk-management challenge, not a prompt-engineering problem."

The most concrete protocol proposal is the [Agent Identity Protocol (AIP)](https://arxiv.org/html/2603.24775v1) (Prakash, March 2026), which introduces Invocation-Bound Capability Tokens (IBCTs) — append-only cryptographic chains that unify identity, attenuated authorization, and provenance binding. Each delegation hop appends a signed block that must cryptographically narrow authority. The protocol enforces a configurable `max_depth` (default 3 hops), mandatory delegation context declarations, and budget ceilings. Adversarial evaluation across 600 test cases showed 100% rejection of depth violation and scope expansion attacks, with sub-millisecond verification overhead (0.086% of end-to-end latency in production LLM inference). AIP supports both compact JWT mode for single-hop delegation and chained Biscuit tokens for multi-hop scenarios.

For federal and high-assurance environments, the [SentinelAgent framework](https://arxiv.org/abs/2604.02767) (April 2026) introduces a Delegation Chain Calculus (DCC) that establishes seven verifiable properties for delegation chains: authority narrowing, policy preservation, forensic reconstructibility, cascade containment, scope-action conformance, output schema conformance, and intent preservation. A key architectural decision is using a non-LLM Delegation Authority Service for critical security decisions rather than relying on language models. Six of the seven properties are deterministically verified (confirmed via TLA+ model checking across 2.7 million states with zero violations), while intent preservation is acknowledged as probabilistically bounded — the paper formally proves the "practical infeasibility of deterministic intent verification." On the DelegationBench v4 benchmark (516 scenarios across 10 attack categories), SentinelAgent blocked all 30 black-box adversarial attacks with zero false positives. Even when intent verification degrades under sophisticated paraphrasing attacks (13% detection rate without fine-tuning), the remaining six deterministic properties constrain adversaries to permitted API calls, conformant outputs, traceable actions, and bounded cascades.

On the identity infrastructure side, [ZeroID](https://www.helpnetsecurity.com/2026/04/13/zeroid-open-source-identity-platform-autonomous-ai-agents/) (Highflame, open-source, April 2026) provides a purpose-built identity and credentialing platform for autonomous agents. ZeroID implements RFC 8693 token exchange to create verifiable delegation chains: when an orchestrator delegates to a sub-agent, the resulting token carries the sub-agent's identity, the orchestrator's identity, and the original authorizing principal, with permissions automatically attenuating at each hop. Real-time revocation uses the OpenID Shared Signals Framework (SSF) and Continuous Access Evaluation Profile (CAEP), so revoking credentials at any point in a delegation chain instantly invalidates all downstream tokens. SDKs are available for Python, TypeScript, and Rust, with released integrations for LangGraph, CrewAI, and Strands.

### Tool-Use Restriction Patterns

Modern agentic frameworks have converged on several defensive patterns for action limiting:

- **Function-level policies**: Tools like Lasso Security enforce tool usage boundaries through function-level policies and real-time validation, ensuring agents cannot invoke tools outside their role or without context-aware authorization. The OpenAI Agents SDK (as of early 2026) includes built-in guardrail hooks for tool confirmation, making this pattern a first-class framework feature rather than a bolt-on.
- **Intent-based permission scoping**: As of March 2026, Token Security introduced intent-based agent security — rather than giving agents static permissions or inheriting human credentials, each agent gets a purpose-defined permission envelope scoped to its declared operational intent. The platform discovers agents and their access, maps declared vs. observed intent, dynamically enforces least-privilege policies, flags out-of-bounds actions, and applies lifecycle governance to prevent access drift. This addresses a fundamental gap: traditional guardrails filter prompts, but don't constrain the permissions an agent holds at the identity layer.
- **Positive security model for tool access**: The tool allowlist should be deny-by-default, allow-explicitly — not a blocklist. LangChain, CrewAI, AutoGen (AG2), and LangGraph all support tool allowlists, but defaults are often permissive and must be hardened. MCP (Model Context Protocol) server connections should be audited and never exposed to untrusted networks. Place MCP servers behind a proxy, enforce authentication, validate token audience and scopes, and prevent token passthrough.
- **Separation of planning from execution**: Architecture should separate the agent's planning phase (deciding what to do) from the execution phase (actually doing it), with a validation gate between them. This prevents prompt injection from directly triggering tool invocations. LangGraph's architecture explicitly supports this pattern with sandboxed file I/O and human-in-the-loop confirmation on writes.
- **Sandboxed execution environments**: High-impact actions (file writes, code execution, network calls) should run in isolated sandboxes with resource limits, preventing lateral movement even if an action is authorized. Nvidia's NemoClaw framework (early 2026) provides programmable system and network controls through its OpenShell intermediary, managing what agents can access and execute within isolated sandboxes. Agents should run with ephemeral environments for each run, short-lived credentials, least-privilege IAM, and network egress boundaries blocking unknown destinations by default. This is especially important given that MITRE ATLAS (updated October 2025 with 14 new agent-specific techniques) documents tool abuse as a core attack path.
- **Safety agents as policy enforcement**: A dedicated safety agent layer evaluates proposed actions before execution, applying rules related to data sensitivity and tool usage. Actions that violate defined policies are blocked, modified, or logged for review — this creates an independent enforcement point that cannot be bypassed by prompt injection targeting the primary agent. As of February 2026, Gartner formally recognized this pattern as a standalone enterprise category with the first [Market Guide for Guardian Agents](https://www.gartner.com/en/newsroom/press-releases/2025-06-11-gartner-predicts-that-guardian-agents-will-capture-10-15-percent-of-the-agentic-ai-market-by-2030), defining three types: **Reviewers** (inspecting agent-generated output for accuracy), **Monitors** (observing and tracking agent actions in real time), and **Protectors** (adjusting or blocking agent actions and permissions during operations). With nearly 70% of enterprises already running agents in production, Gartner projects guardian agent spending will grow from less than 1% of agentic AI budgets today to 5-7% by 2028 — and that by 2029, guardian agents will allow more than 70% of companies to retire roughly half of their incumbent agent risk and security systems. Representative vendors recognized in the guide include NeuralTrust, Silverfort, PlainID, and Wayfound, among others.
- **OS-level enforcement beneath the application layer**: As of March 2026, NVIDIA's [practical sandboxing guidance](https://developer.nvidia.com/blog/practical-security-guidance-for-sandboxing-agentic-workflows-and-managing-execution-risk/) recommends enforcing restrictions at the operating system level using macOS Seatbelt, Linux Bubblewrap, or Windows AppContainer — below the application where agent code cannot override them. Key controls include blocking write operations to files outside the workspace, protecting agent configuration files (hooks, MCP servers, IDE settings) regardless of user approval, restricting DNS resolution to designated trusted resolvers to prevent DNS-based exfiltration, and applying sandbox restrictions to all spawned subprocesses including hooks and MCP initialization scripts. The guidance explicitly calls for "no cached approvals" — each potentially dangerous action should require fresh user confirmation rather than relying on blanket trust from a prior approval.
- **Agent-specific credential lifecycle management**: As of April 2026, the emerging consensus is that AI agents require identity management distinct from both human users and traditional non-human identities (service accounts, API keys). [Strata Identity's agent identity playbook](https://www.strata.io/blog/agentic-identity/new-identity-playbook-ai-agents-not-nhi-8b/) articulates the key distinction: traditional NHIs are static and narrowly scoped, while agents are dynamic, autonomous, and fluid — they reason, delegate, and act independently across domains. Recommended patterns include just-in-time credential provisioning that creates agent identities on-demand with attached TTL, purpose, risk classification, and delegation context, then retires credentials upon task completion; OIDC for initial authentication via trusted identity providers with OAuth actor-subject binding for delegation; SPIFFE/SPIRE X.509 with mTLS for internal agent-to-agent communication; and OTEL-compatible centralized logging tracking subject, actor, delegation chain, purpose, resource, and policy decisions at each action. [Aembit's agentic guardrails framework](https://aembit.io/blog/agentic-ai-guardrails-for-safe-scaling/) implements risk-tiered action classification where low-risk operations proceed autonomously, medium-risk actions trigger notifications, and high-risk operations (database deletions, security group modifications) require explicit human authorization — with the same action carrying different risk profiles depending on environment context.

### AWS Agentic AI Security Scoping Matrix

AWS published the [Agentic AI Security Scoping Matrix](https://aws.amazon.com/ai/security/agentic-ai-scoping-matrix/) to help organizations classify their agent deployments and apply proportionate action-limiting controls. The matrix defines four scopes based on autonomy level:

- **Scope 1 (No Agency):** Read-only, human-initiated systems with fixed workflow paths. Controls focus on identity context and input validation to prevent agents from exceeding operational boundaries.
- **Scope 2 (Prescribed Agency):** Human approval required before every action. Controls require mandatory human-in-the-loop (HITL) workflows and secure communication channels.
- **Scope 3 (Supervised Agency):** Autonomous execution within predefined bounded parameters, with human initiation. Controls include continuous behavioral monitoring, validation against expected patterns, and effective shut-off switches.
- **Scope 4 (Full Agency):** Self-initiating systems with minimal human oversight. Controls require advanced behavioral monitoring, anomaly detection, and automated containment mechanisms.

The matrix evaluates security across six critical dimensions: identity context, workflow integrity, input validation, output governance, tool and API access, and behavioral monitoring. Control sophistication increases significantly with each scope level, and organizations should classify each agent deployment individually rather than applying a single scope to all agents.

### Execution Sandboxing: Isolation Technology Comparison

For agents that execute code, write files, or make network requests, the choice of isolation technology directly determines the security boundary's strength. As of March 2026, three primary approaches are in production use:

| Technology | Boot Time | Memory Overhead | Isolation Level | Best For |
|-----------|-----------|----------------|----------------|----------|
| **Firecracker microVMs** | ~125 ms | <5 MiB per VM | Hardware-level (dedicated kernel per workload) | Production untrusted code, multi-tenant environments |
| **Kata Containers** | ~200 ms | Moderate | Hardware-level (VM-backed containers) | Kubernetes-native deployments requiring VM isolation |
| **gVisor** | Milliseconds | Minimal | Syscall-level (user-space kernel) | Compute-heavy agents with limited I/O requirements |
| **Docker containers** | Milliseconds | Minimal | Process-level (shared kernel) | Trusted, vetted code only |

For production AI agents executing untrusted code, microVMs (Firecracker or Kata Containers) are the recommended choice because the hardware boundary prevents entire classes of kernel-based attacks — a kernel vulnerability in a standard container allows container escape and host access, while a microVM's dedicated kernel eliminates this attack surface. gVisor offers a middle ground, adding 10–30% overhead on I/O-heavy workloads but intercepting syscalls through a user-space kernel that significantly reduces the attack surface without full virtualization. Standard Docker containers share the host kernel and should only be used for pre-vetted, trusted agent code.

The recommended practice is to combine isolation layers: ephemeral microVM per agent session, workspace-scoped file system access, restricted network egress with allowlisted destinations only, short-lived credentials injected per task, and periodic sandbox reset (weekly at minimum) to prevent accumulation of secrets and exploitable state.

### Rate Limiting and Cost Controls

AI-specific rate limiting must account for several dimensions beyond traditional API rate limits. Gartner predicts that more than 30% of the increase in API demand will come from AI and LLM tools by 2026, making token-aware rate limiting a critical infrastructure concern:

- **Token-based limits**: Separate limits for input and output tokens, since a single prompt can generate thousands of output tokens. Track prompt tokens and completion tokens independently, reading actual usage from LLM provider response bodies (e.g., OpenAI's `usage.total_tokens`). Completion tokens should be weighted differently from prompt tokens to reflect provider pricing.
- **Cost-based limits**: Total spend per user per period, aggregated across multiple model invocations in a pipeline. Assign cost multipliers per model and track in dollar-equivalent units rather than raw token counts. Set alerts at 50% and 80% of budget thresholds with automatic suspension when ceilings are exceeded.
- **Tiered budgets**: Layer three levels of limits — per-request `max_tokens`, per-task token budgets, and per-day/per-month spending caps. This prevents both runaway individual requests and slow-burn cost accumulation across many small requests.
- **Per-action rate limits**: High-impact actions (email sends, file writes) should have their own rate limits independent of general API rate limits.
- **Agent-aware traffic management**: AI agent traffic should be tagged and managed separately from human user traffic via API key metadata, since agents produce fundamentally different consumption patterns — bursty, sequential, and high-volume.

### Human Confirmation Design

The confirmation mechanism for high-impact actions must be implemented at the application layer, not within the model's prompt. Critical design principles:

- The model must not be able to confirm its own actions (no self-approval). The confirmation gate must live outside the model's context window entirely.
- Confirmation dialogs should include a clear description of the action, the resources affected, and the irreversibility of the operation.
- For agentic systems with many tool calls, implement tiered confirmation: auto-approve low-risk actions (read-only queries), require confirmation for medium-risk (file writes, data modifications), and require re-authentication for high-risk (deletions, financial transactions, external communications).
- Implement Content Security Policy (CSP) controls on agent-generated output to block or restrict external image loading and link rendering, which are common exfiltration vectors in indirect prompt injection attacks.
- Monitor for unusual patterns of external requests generated by agent outputs — anomalous outbound traffic may indicate a successful injection exploiting the exfiltration leg of the Lethal Trifecta.

### CISO Priorities for Agentic Action Governance

Bessemer Venture Partners' March 2026 analysis of the agentic security landscape distills five priorities for security leaders deploying AI agents:

1. **Define organizational risk posture before procurement** — establish what agents are allowed to do before they are deployed, not after an incident forces the conversation.
2. **Treat agents as production infrastructure with managed identities** — agents are not applications; they need identity lifecycle management equivalent to human employees, including onboarding, access reviews, and offboarding.
3. **Deploy with minimum permissions, expand deliberately** — start every agent with the smallest permission set that allows it to function, then expand only when justified by business need with documented approval.
4. **Establish action-level guardrails before monitoring** — monitoring tells you what happened; guardrails prevent it from happening. Action-level enforcement (not just access-level) must be in place before the first production deployment.
5. **Audit agent access equivalent to human employee oversight** — regular access reviews, privilege recertification, and anomaly detection should apply to agents with the same rigor as human IAM programs.

### Regulatory and Standards Landscape (2025–2026)

**Singapore Model AI Governance Framework (2025):** The IMDA published a governance framework specifically for agentic AI, recommending strong authentication and access controls ensuring least-privilege principles, continuous monitoring that flags anomalous requests or deviations from typical behavior patterns to trigger shutdown, and clear accountability chains for autonomous agent actions.

**NIST AI Agent Standards Initiative (February 2026):** NIST's Center for AI Standards and Innovation (CAISI) launched a dedicated initiative to ensure autonomous agents are "widely adopted with confidence, can function securely on behalf of its users, and can interoperate smoothly across the digital ecosystem." The accompanying NCCoE concept paper on AI agent identity and authorization proposes treating agents as identifiable entities within enterprise identity systems — with their own credentials, authorization scopes, audit trails, and non-repudiation capabilities — rather than running as anonymous automation under shared human credentials. The initiative explicitly covers controls for preventing and mitigating prompt injection attacks at the identity layer.

**EU AI Act Human Oversight (Article 14, effective August 2026):** Many agentic applications — particularly those involving employment screening, credit scoring, and critical infrastructure — fall into "High-Risk" categories under the EU AI Act. Article 14 requires human oversight mechanisms that enable individuals to understand system capabilities and limitations, monitor operation, and intervene or halt the system. For action-limiting controls, this translates to mandatory confirmation gates, audit trails, and kill switches for high-risk autonomous actions. The Act's rules for high-risk AI become fully enforceable in August 2026 and August 2027.

**Cisco AI Defense (February 2026):** Cisco expanded its AI Defense solution to add runtime protections against tool abuse and supply chain manipulation specifically at the MCP layer, reflecting the industry shift toward securing the agent-tool interface as a first-class security boundary.

**NIST Request for Information on AI Agent Security (January 2026):** In January 2026, NIST published a [Request for Information](https://www.nist.gov/artificial-intelligence/ai-risk-management-framework) covering security considerations for AI agents including agent hijacking, backdoor attacks, and autonomous action risks. This RFI addresses many of the same threats the EU AI Act calls out in Article 15, signaling regulatory convergence across jurisdictions on agent action limiting.

**White House National AI Policy Framework (March 20, 2026):** The [National Policy Framework for Artificial Intelligence](https://www.whitehouse.gov/wp-content/uploads/2026/03/03.20.26-National-Policy-Framework-for-Artificial-Intelligence-Legislative-Recommendations.pdf) outlines legislative recommendations for a unified federal approach to AI governance. Rather than creating a new federal AI regulator, the framework routes oversight through existing sector-specific agencies with subject-matter expertise (SEC for financial AI, FDA for health AI, FTC for consumer protection). For agent action limiting, the framework recommends establishing regulatory "sandboxes" for AI applications to support experimentation, and proposes federal preemption of state AI laws deemed overly burdensome — a move that could limit the patchwork of state-level agent governance requirements. The framework explicitly avoids prescriptive action-limiting controls, favoring industry-led standards instead. How liability attaches when agentic AI takes unauthorized actions remains an open question under this framework, as it recommends against penalizing developers for third-party misuse of their models.

**Colorado AI Act (SB24-205, enforceable June 30, 2026):** The first US state law to impose specific obligations on AI developers and deployers, with enforcement delayed from February 2026 to June 30, 2026 via Senate Bill 4. Enforcement authority sits exclusively with the Colorado Attorney General (no private right of action). For action-limiting controls, the Act requires developers and deployers of "high-risk" AI systems to implement risk management practices including the ability to revoke an AI's operating role. The Act provides an affirmative defense for organizations demonstrating compliance with nationally or internationally recognized risk management frameworks (such as NIST AI RMF), making framework alignment a concrete legal benefit. Alongside the EU AI Act (August 2026) and the Colorado AI Act (June 2026), organizations deploying agentic AI now face two near-simultaneous enforcement deadlines.

**CrowdStrike Falcon AIDR + NVIDIA NeMo Guardrails** (March 2026): CrowdStrike's AI Detection and Response platform now integrates with NVIDIA NeMo Guardrails v0.20.0, providing a combined detection-and-enforcement approach for securing homegrown AI agents. NeMo Guardrails adds programmable guardrails — including content safety, PII detection, jailbreak detection, and topic control via the Nemotron Safety model family — while Falcon AIDR provides runtime behavioral monitoring and automated incident response. This integration represents the convergence of endpoint detection/response (EDR) platforms with AI-specific guardrail frameworks, enabling organizations to apply the same security operations workflows to AI agents that they already use for traditional infrastructure.

**Industry Readiness Gap:** As of early 2026, only 41% of organizations have runtime guardrails in place for agentic AI, and 30% rely on homegrown scripts or custom tooling rather than mature security solutions. Bessemer Venture Partners reports that 48% of cybersecurity professionals now identify agentic AI as the single most dangerous attack vector, and IBM's 2025 data shows shadow AI breaches cost an average of $4.63 million per incident — $670,000 more than standard breaches. Gartner projects that 40% of enterprise applications will embed task-specific AI agents by end of 2026, up from less than 5% in 2025. This gap between deployment pace and security readiness makes action limiting one of the most urgent unsolved problems in enterprise AI security.

## Open Research Questions

- How should confirmation UX be designed to avoid "confirmation fatigue" where users automatically approve all actions? Tiered confirmation models show promise but lack empirical validation at scale.
- What is the right granularity for tool allowlists — per-tool, per-tool-per-action, or per-tool-per-action-per-resource? The OWASP Agentic Top 10's "least agency" principle suggests the answer depends on the autonomy budget for each use case.
- How should cost limits be set for multi-model pipelines where a single user request triggers multiple model invocations across different providers? Dollar-equivalent budgets look promising, but cross-provider cost normalization remains unsolved.
- Given the 82.4% inter-agent trust exploitation rate found by ASB (ICLR 2025), how should tool authorization be scoped in multi-agent systems where agents delegate to each other? The AIP protocol (March 2026) and Grantex framework propose cryptographic scope attenuation at each hop, but no major framework has adopted these approaches yet, and empirical validation in production multi-agent pipelines is absent.
- Can formal verification methods be applied to agent tool-use policies to prove that certain dangerous action sequences are unreachable?
- How should action limiting adapt to emergent agentic capabilities (e.g., computer-use agents, browser-use agents) where the boundary between "tool call" and "general action" is blurred? The PerplexedBrowser incident (March 2026) showed that agentic browsers operating within authenticated sessions present a fundamentally different threat model than traditional tool-calling agents.
- As MCP adoption grows, what security controls should govern MCP server discovery and connection? Currently, most MCP configurations are trust-on-first-use with no runtime policy enforcement. Cisco's February 2026 expansion of AI Defense to cover MCP-layer protections is an early step, but standardized MCP security profiles remain absent.
- How should the Lethal Trifecta be evaluated in multi-tenant environments where different users' data flows through the same agent infrastructure?
- The OpenClaw crisis exposed the absence of skill/plugin vetting in AI agent marketplaces. What minimum security controls should agent skill registries enforce — code signing, sandboxed execution, permission declarations, behavioral analysis — and who should operate them?
- Intent-based permission models (e.g., Token Security's March 2026 approach) represent a shift from static permissions to purpose-defined envelopes. How should intent be formally specified, and what happens when an agent's runtime behavior legitimately evolves beyond its declared intent during a long-running task?
- How should organizations map their agent deployments to the AWS Agentic AI Security Scoping Matrix's four autonomy levels when a single agent pipeline spans multiple scopes (e.g., Scope 1 for data retrieval, Scope 3 for code execution)? Should the highest scope in the pipeline govern the entire system?
- The Microsoft Agent Governance Toolkit's `did:mesh` agent identity format and Ed25519 credentials represent one approach to agent identity, while NIST's NCCoE proposes SPIFFE-based identity. Will the industry converge on a single agent identity standard, or will interoperability layers be needed?
- With only 41% of organizations having runtime guardrails in place as of early 2026, what is the minimum viable action-limiting control set that organizations should deploy before their first agentic AI goes to production?
- The LiteLLM supply chain attack (March 2026) demonstrated that AI gateway infrastructure is a high-value target — a single compromised proxy can expose every API key and credential flowing through it. Should AI gateways be treated as critical infrastructure with mandatory integrity verification, reproducible builds, and dependency pinning?
- Meta's rogue AI agent incident showed that even internal agents in mature organizations can take unauthorized actions. How should organizations balance agent productivity (allowing agents to compose and post responses) with safety (requiring confirmation before every write action) when the agents are used for routine internal tasks?
- Agent Session Smuggling (Unit 42, April 2026) demonstrated that stateful A2A communication enables multi-turn adaptive attacks that are "significantly more difficult to defend against" than single-instruction injection. How should context grounding and task anchoring be standardized across A2A implementations, and can these defenses be made robust against adversarial agents that deliberately build false trust over multiple turns?
- The Grantex audit found 0/30 major agent frameworks implement delegation chain depth limits or scope narrowing. Is this a framework design problem (requiring upstream changes to LangChain, CrewAI, AutoGen, etc.) or an infrastructure problem best solved by external authorization layers like AIP or Grantex? What is the adoption path for delegation chain controls when no framework supports them natively?
- The Flowise CVE-2025-59528 incident revealed a six-month patch lag on a CVSS 10.0 vulnerability affecting 12,000+ instances. AI agent builder platforms are widely treated as internal dev tools exempt from production vulnerability management — how should organizations classify and patch AI development infrastructure given that it typically holds LLM API keys, RAG pipeline contents, and workflow configurations?
- SentinelAgent's formal proof of the "practical infeasibility of deterministic intent verification" raises a fundamental question: if intent cannot be deterministically verified across delegation chains, what is the minimum set of deterministic properties (authority narrowing, cascade containment, etc.) that provides acceptable security even when intent verification fails?
- With the Colorado AI Act (June 2026) and EU AI Act high-risk obligations (August 2026) creating near-simultaneous enforcement deadlines, what constitutes a defensible minimum action-limiting control set that satisfies both frameworks? The Colorado Act's affirmative defense for recognized risk management framework compliance creates a concrete incentive for NIST AI RMF alignment.
- The PraisonAI CVE chain (four critical vulnerabilities including CVSS 10.0 sandbox bypass) demonstrates that application-layer sandboxing in Python-based agent frameworks is fundamentally brittle. Should agent framework security certifications require OS-level isolation (Firecracker, gVisor) as a baseline rather than relying on language-level sandbox implementations?
- The MCP Authorization Specification now mandates OAuth 2.1 with RFC 8707 resource indicators, but the Azure DevOps MCP vulnerability (CVE-2026-32211) shows that specification compliance is inconsistent across the ecosystem. What enforcement mechanisms — registry requirements, automated security scanning, or certification programs — would accelerate adoption of MCP authorization controls across community-maintained MCP servers?
- The Marimo CVE-2026-39987 exploitation timeline — functional exploit within 10 hours of advisory disclosure, credential theft in under 3 minutes — suggests that AI development tools are being actively monitored by threat actors for disclosure events. Should AI toolchain vendors adopt coordinated disclosure practices that include pre-patch notification to hosting providers, automatic Shodan/Censys surface scans, and proactive credential rotation advisories?
- Gartner's Guardian Agents Market Guide (February 2026) formalized independent agent oversight as a standalone category, but the three-type taxonomy (reviewers, monitors, protectors) raises questions about composition: should guardian agents themselves be governed by meta-guardian layers, and at what point does layered oversight introduce latency or conflicts that degrade agent utility? The White House National AI Policy Framework's preference for sector-specific oversight over centralized regulation suggests that guardian agent deployments may need to be tuned per industry vertical.
