# C11.5: Model-Extraction Defense

> **Chapter:** [C11 Adversarial Robustness & Attack Resistance](C11-Adversarial-Robustness)
> **Requirements:** 5 | **IDs:** 11.5.1--11.5.5

## Purpose

Detect and deter unauthorized model cloning through API abuse. Model extraction (model stealing) attacks create a functionally equivalent copy of a target model by querying it systematically and training a surrogate on the input-output pairs. Successful extraction compromises intellectual property, enables white-box adversarial attacks against the original model, and bypasses access controls. The primary defenses are rate limiting, query-pattern analysis, and post-hoc identification via watermarking.

## Recent Research (2024--2026)

### Expanded LLM Extraction Threat Model

A comprehensive survey on model extraction attacks and defenses for LLMs (June 2025) identifies three primary attack categories beyond traditional functionality extraction: **training data extraction** (exploiting memorization to recover sensitive training examples via crafted prompts), **prompt-targeted attacks** (stealing system prompts, few-shot examples, and other prompt IP), and **parameter recovery** (reverse-engineering internal weights). This expanded threat model means that rate limiting (requirement 11.5.1) and query-pattern analysis (11.5.3) must account for all three attack vectors, not just functionality cloning.

### Adaptive Watermarking: ModelShield

**ModelShield** (IEEE TIFS, January 2025) introduces adaptive and robust watermarking that dynamically adjusts protection strength based on query patterns. Unlike static watermarking schemes that can be characterized and removed, ModelShield strategically perturbs responses while maintaining high utility for legitimate users. This addresses a key gap in requirement 11.5.4: static watermarks are increasingly vulnerable to removal through fine-tuning and pruning, while adaptive approaches resist these attacks more effectively.

### Explanation-Based Watermarking (EaaW)

**Explanation as a Watermark (EaaW)** (2024) introduces a novel paradigm that embeds multi-bit watermarks into feature attribution explanations rather than model predictions. This means verification behaviors are implanted in the explanation layer rather than the output layer, making the watermark invisible to standard extraction pipelines that only replicate input-output mappings. EaaW supports multi-bit encoding, enabling richer ownership claims than single-bit backdoor watermarks.

### Watermark Vulnerability Research

USENIX Security 2024 research ("How to Break White-Box DNN-Watermarking Schemes") systematically demonstrated attacks against existing white-box watermarking schemes, showing that many can be removed or forged. This directly validates requirement 11.5.5's emphasis on protecting watermark verification keys and trigger sets -- if an attacker gains access to verification materials, watermark schemes are fundamentally compromised. Organizations implementing watermarking should assume adversaries will attempt both extraction-then-removal and direct watermark attacks.

### Semantically Conditioned Watermarks for LLMs

A May 2025 paper introduces **Semantically Conditioned Watermarks (SCW)**, a paradigm shift in LLM fingerprinting. Instead of teaching a model to memorize fixed query-key pairs (which break under fine-tuning and quantization), SCW uses an entire semantic domain (e.g., a specific language or topic area) as the trigger. The model diffuses a statistical watermarking signal throughout all responses generated from that domain. This makes the fingerprint both stealthier and more robust — it survives standard deployment modifications that compromise traditional approaches. Model owners verify ownership by querying the predetermined semantic domain and detecting the watermark statistically.

### FingerMarks: Multi-Bit Edge Model Watermarking

**FingerMarks** (MDPI Electronics, 2025) addresses a gap that earlier edge-deployment protections (TransLinkGuard, CoreGuard) left open: combining model attribution with user traceability. FingerMarks embeds multi-bit watermarks using a progressive adversarial training strategy, enabling organizations to identify both which model was stolen and which edge node user was responsible. Evaluation across multiple datasets confirms robustness against common removal attacks.

### Edge Deployment Protection

Earlier work on **TransLinkGuard** and **CoreGuard** addresses model extraction defense for edge-deployed models where rate limiting and API-level defenses are not available. These techniques embed architectural-level watermarks that persist even when models are deployed on untrusted hardware, extending the protection scope beyond API-served models.

### Operation Bizarre Bazaar: Real-World LLMjacking at Scale

In December 2025–January 2026, Pillar Security documented **Operation Bizarre Bazaar**, the first publicly attributed systematic campaign targeting exposed LLM and MCP endpoints. Across 35,000 attack sessions averaging 972 daily attempts, attackers scanned for Ollama (port 11434) and OpenAI-compatible APIs (port 8000), validated access with placeholder API keys, then resold access through silver.inc — a marketplace offering discounted access to 30+ LLM providers without authorization. The threat actor ("Hecker") operated from Netherlands-based bulletproof hosting. This incident validates that model extraction and API abuse are not theoretical — unprotected inference endpoints are actively hunted and monetized.

### Falling Inference Costs and Extraction Economics

As of early 2026, LLM inference costs have dropped approximately 50x per year (per Epoch AI analysis), with providers like DeepSeek offering processing at $0.70 per million tokens total. This cost collapse changes the extraction threat model: surrogate training becomes cheaper, making even large-scale extraction economically rational for attackers who previously would have trained from scratch. Rate limits (11.5.1) and query-pattern detection (11.5.3) become more important as the economic barrier to extraction falls.

### Defense Taxonomy

The 2025 LLM extraction survey organizes defenses into three categories that map well to AISVS requirements:
1. **Model Protection** (architectural defenses, output control) -- maps to requirements 11.5.1, 11.5.3, 11.5.4
2. **Data Privacy Protection** (preventing training data extraction, output sanitization) -- connects to C11.3 and C11.4
3. **Prompt Protection** (watermarking prompts, query monitoring for prompt theft) -- an emerging concern not yet explicitly covered in the current requirements

### ModelGuard: Information-Theoretic Output Perturbation

**ModelGuard** (USENIX Security 2024) introduces a principled approach to output perturbation that goes beyond ad-hoc noise injection. Rather than simply adding random noise to model predictions, ModelGuard formulates extraction defense as an optimization problem that minimizes information leakage to attackers while maximizing utility for legitimate queries. The framework resists adaptive attackers who adjust their strategies based on observed perturbations — a critical advancement since naive perturbation schemes can be reverse-engineered. ModelGuard supports multiple defense modes (perturbation-based and quantization-based) and has been evaluated against six different extraction attack strategies including KnockoffNet and JBDA-TR across eight datasets. An open-source implementation is available on GitHub. This directly strengthens requirements 11.5.1 and 11.5.3 by providing a concrete, implementable output-level defense.

### Dynamic Neural Fortresses: Architectural Extraction Defense

**Dynamic Neural Fortresses (DNF)** (ICLR 2025) represents the first application of early-exit neural networks to model extraction defense. DNF dynamically routes queries through the network: benign queries proceed to later layers for full-fidelity predictions, while suspected extraction queries are diverted to early-exit points that return deliberately degraded responses. This architectural approach achieves a 2x improvement in computational efficiency (since attack queries consume fewer resources) and reduces clone model accuracy by up to 12% compared to prior state-of-the-art defenses. DNF also provides protection against neural architecture theft — an attack vector where adversaries attempt to reverse-engineer the target model's structure, not just its behavior. This is notable because most existing defenses focus solely on functionality extraction.

### GreyNoise Intelligence: Systematic LLM Infrastructure Targeting

Between October 2025 and January 2026, GreyNoise's Ollama honeypot infrastructure captured **91,403 attack sessions** revealing two distinct campaigns targeting AI infrastructure. The first campaign exploited SSRF vulnerabilities in Ollama's model pull functionality, peaking at 1,688 sessions in 48 hours over Christmas 2025. A single JA4H fingerprint appeared in 99% of traffic, suggesting shared automation tooling (likely Nuclei). The second campaign, starting December 28, 2025, was a methodical LLM reconnaissance operation: two IPs generated 80,469 enumeration requests across 73+ model endpoints in eleven days, probing for OpenAI, Anthropic, Meta, DeepSeek, Google, Mistral, Alibaba, and xAI models. Together with Operation Bizarre Bazaar, this data confirms that exposed inference endpoints face sustained, professional-grade scanning — organizations should treat unprotected model endpoints as equivalent to exposed databases.

### Practical Extraction Economics: Low-Cost Demonstrations

A January 2026 Praetorian demonstration quantified just how accessible model extraction has become: using standard knowledge distillation via PyTorch, they achieved 80.1% accuracy and 80.8% prediction agreement on a Fashion-MNIST model after only 1,000 API queries. Simple categories like trousers and sneakers reached over 90% agreement. Separately, Model Leeching research extracted 73% similarity from ChatGPT-3.5-Turbo for approximately $50 in API costs over 48 hours. These results confirm that extraction is no longer a theoretical concern requiring massive resources — it is a practical attack within reach of individual actors with modest budgets.

### EU AI Act: Regulatory Dimensions of Model Protection

As of March 2026, the EU AI Act's obligations for general-purpose AI (GPAI) model providers have been in effect since August 2025, with full enforcement (including fines up to EUR 15M or 3% of global turnover) starting August 2026. GPAI providers must maintain detailed technical documentation, publish training data summaries using the EU AI Office template, and comply with copyright law. While the Act does not explicitly mandate extraction defenses, its documentation and transparency requirements create an implicit expectation that providers can demonstrate model provenance and ownership. Watermarking (11.5.4) and fingerprinting techniques may become compliance tools for proving ownership in IP disputes. The Code of Practice provides a grace period through August 2026, but organizations should align extraction defense documentation with EU AI Act requirements now.

### EU AI Act Code of Practice: Multilayered Watermarking Mandate

The EU AI Act's second draft Code of Practice on marking and labelling AI-generated content (March 2026) takes the position that no single marking technique is sufficient and mandates a multilayered approach for GPAI providers: (1) metadata embedding with digital signatures, (2) interwoven watermarking with imperceptible marks that survive compression and cropping, and (3) fingerprinting/logging facilities for output verification. Providers must also deploy detectors via API or user interface capable of identifying both marked and unmarked synthetic content. For open-weight models, structural marking must be encoded during training. The Code is expected to be finalized by June 2026, with full Article 50 enforcement and penalty powers (up to EUR 15M or 3% of global turnover) effective August 2, 2026. This regulatory trajectory strengthens requirement 11.5.4 by making watermarking effectively mandatory for EU-market GPAI providers rather than a best-practice recommendation.

### Watermarking Adoption Gap in Practice

Despite regulatory momentum, a March 2026 study analyzing 50 popular AI image generation systems found that only 38% implement any form of machine-readable watermark, with just 8 systems using robust invisible watermarking. Visible disclosures appear in only 18% of systems. Adoption is heavily concentrated among well-funded end-to-end platforms (Google, Adobe, OpenAI) — smaller providers and those using APIs from upstream model developers rarely maintain watermark protections. Even when base model developers embed watermarks, downstream API consumers often fail to preserve them through their pipeline. Only 3 of the 40 analyzed systems originate from EU countries, raising questions about enforcement reach. This gap between regulatory expectation and industry reality means that requirement 11.5.4 verification must account for supply-chain watermark integrity — confirming that watermarks survive the full deployment pipeline, not just that the base model supports them.

### Adversarial Robustness of LLM Fingerprints

A September 2025 paper by Nasery et al. ("Are Robust LLM Fingerprints Adversarially Robust?") exposed a fundamental weakness in the current generation of model fingerprinting: while these schemes are designed to survive benign modifications like fine-tuning and quantization, they fail catastrophically against adaptive adversaries. The researchers tested ten fingerprinting methods — including Chain-and-Hash, FPEdit, Implicit Fingerprints, EditMF, Perinucleus, Instructional FP, MergePrint, RoFL, ProFLingo, and Domain Specific Watermarks — and achieved 94–100% attack success rates against memorization-based schemes and 100% against GCG-based intrinsic fingerprints using simple perplexity filtering. Even statistical watermarks were compromised at 65% success while retaining 90% model utility. The attacks exploit three weaknesses: overconfidence in memorized responses (detectable and suppressible), unnatural query patterns (filterable by perplexity thresholds), and learnable statistical signatures (stealable via bigram analysis). This directly challenges the assumptions behind requirement 11.5.4 — organizations should treat current fingerprinting as a deterrent rather than a reliable ownership proof, and evaluate hybrid approaches combining multiple verification strategies.

### ATOM: RL-Based Extraction Detection for MLaaS

**ATOM** (KDD 2025) introduces a real-time extraction detection framework for graph-based Machine Learning as a Service platforms. Unlike static detectors like PRADA, ATOM uses reinforcement learning with Proximal Policy Optimization (PPO) to dynamically adapt its detection policy as attackers refine their query strategies. The framework's differential query feature encoding mechanism tracks changes across consecutive interactions, while k-core embedding captures structural properties of query graphs. While designed for GNNs rather than LLMs, the architectural approach — combining sequential modeling with adaptive RL-based detection — is transferable to any API-served model and represents the direction extraction detection is heading. Source code is available at https://github.com/LabRAI/ATOM.

### Embedding Service Protection: GuardEmb and SemMark

As embedding-as-a-service (EaaS) becomes a major deployment model, two complementary watermarking techniques have emerged. **GuardEmb** (EMNLP Findings 2024) selects special tokens and perturbs their embeddings to inject watermarks, while simultaneously training a verifier to detect them. When an attacker trains a surrogate on stolen embeddings, the watermark transfers to the clone, and verification is conducted by querying the suspect service with constructed test texts. **SemMark** (2025) takes a more sophisticated approach: it partitions embedding space using locality-sensitive hashing to identify semantic regions, then injects adaptive watermarks with weights determined by local outlier factor analysis. SemMark achieves verification p-values below 10⁻¹⁰ even under clustering-based, dimensionality-reduction, and detect-sampling attacks, while maintaining 94%+ similarity to original embeddings. These techniques fill a gap left by requirement 11.5.4, which focuses on model outputs rather than embedding endpoints.

### LiteLLM Supply Chain Compromise (March 2026)

On March 24, 2026, the widely used LiteLLM Python package (3.4 million daily PyPI downloads) was compromised when threat actor TeamPCP exfiltrated PyPI publishing credentials from the project's CI/CD pipeline via a poisoned Trivy GitHub Action. The backdoored versions (1.82.7 and 1.82.8) harvested SSH keys, cloud tokens, Kubernetes secrets, and .env files, then attempted lateral movement across Kubernetes clusters. While primarily a supply-chain attack (see C06), the incident is directly relevant to extraction defense: LiteLLM serves as a unified proxy for 100+ LLM providers, meaning a compromised proxy could silently intercept and exfiltrate all inference traffic — effectively enabling model extraction without querying the model API at all. The attack was live for approximately three hours before PyPI quarantine. This underscores that extraction defense must extend beyond API-level controls to include supply-chain integrity of inference infrastructure.

### Distributed Computing and Cross-Environment Extraction Threats

A February 2025 survey on model extraction in distributed computing environments (arXiv:2502.16065) highlights that extraction threats vary significantly by deployment context. Cloud-hosted models face API query exploitation with rich confidence score outputs. Edge-deployed models face physical side-channel attacks (power analysis, electromagnetic emanations). Federated learning models face gradient leakage during iterative training rounds. The survey identifies emerging cross-environment defenses including fully homomorphic encryption (FHE) for encrypted inference, hardware-assisted security features (secure enclaves, TPMs), and differential privacy frameworks with formal privacy budgets. As of March 2026, organizations deploying models across hybrid cloud-edge environments should evaluate environment-specific extraction risks rather than applying a one-size-fits-all defense.

### MITRE ATLAS and SAFE-AI Framework Alignment

As of February 2026, MITRE ATLAS has expanded to v5.4.0 with 16 tactics, 84 techniques, and 56 sub-techniques — a significant growth from the October 2025 baseline (15 tactics, 66 techniques). The October 2025 update added 14 agent-focused techniques in collaboration with Zenity Labs, followed by the February 2026 release adding techniques for AI agent exploitation including "Publish Poisoned AI Agent Tool" (AML.T0104), "Escape to Host" (AML.T0105), and "AI Agent" (AML.T0108). MITRE's **SAFE-AI** framework provides organizational controls mapped to the ATLAS taxonomy, including specific guidance on model extraction defense through output anomaly monitoring and access restriction to training environments. Model extraction falls under AML.T0024 with sub-techniques for training data membership inference (AML.T0024.000) and model inversion/extraction (AML.T0024.001).

### Edinburgh Study: Fingerprint Removal at Scale (IEEE SaTML 2026)

A March 2026 University of Edinburgh study — the largest independent evaluation of AI fingerprinting to date — tested 14 detection methods across 12 generative models and found that fingerprints can be eliminated with striking effectiveness: over 80% success rate with full-access attacks, and over 50% success using simple techniques like JPEG compression or image resizing. Critically, fingerprints could not only be removed but **forged to falsely attribute content to a generator that never produced it**. The peer-reviewed findings, presented at IEEE SaTML 2026, establish that fingerprint-based detection represents a "structural weakness" rather than a fixable software flaw — none of the 14 techniques examined guarantees both high accuracy and robustness across all scenarios. This complements the Nasery et al. (Sept 2025) findings on LLM fingerprint bypass and further challenges the reliability assumptions behind requirement 11.5.4. Together, these results suggest that watermarking should be treated as one layer in a defense-in-depth strategy rather than a standalone ownership proof.

### ProbeSpec: Behavioral Fingerprinting via Perturbation Response (Feb 2026)

**ProbeSpec** (MDPI Electronics, February 2026) introduces a novel black-box model fingerprinting approach based on dynamic perturbation response spectrum analysis. Rather than relying on memorized query-key pairs (which the Nasery et al. and Edinburgh studies have shown to be vulnerable), ProbeSpec captures a model's unique "behavioral spectrum" — the response patterns under multi-level perturbations that originate from implicit decision mechanisms learned during training. The method uses three probe types (Gabor textures, boundary samples, and cross-class mixup samples), applies multi-level perturbations to construct dynamic trajectory response matrices, and extracts spectral fingerprints via 2D-DCT compression. ProbeSpec identifies stolen models with only 80 queries, achieving AUC greater than 0.97 across six architectures while maintaining extremely low false positive rates for independently trained models — even under label-only output restrictions. While currently demonstrated on vision models, the principle of dynamic perturbation response analysis provides a foundation for extending to LLM fingerprinting that may prove more robust than memorization-based schemes.

### CoSAI AI Incident Response Framework (2026)

The Coalition for Secure AI (CoSAI) released the **AI Incident Response Framework v1.0** in early 2026, the first comprehensive open-source framework addressing AI-specific incident response. The framework identifies model extraction as a distinct incident archetype and maps five common AI architecture patterns to component-specific vulnerabilities, MITRE ATLAS techniques, and response procedures. Aligned with the NIST SP 800-61r3 incident response lifecycle and the OASIS CACAO standard for playbook documentation, the framework defines four AI-adapted phases: preparation (AI asset inventory, specialized response teams, AI-specific telemetry), detection and analysis (monitoring for model drift, suspicious prompt patterns, unusual retrieval behaviors), containment/eradication/recovery (model version rollback, poisoned memory purging), and post-incident activities. Separately, the GenAI-IRF framework (MDPI, January 2026) formalizes six recurrent GenAI incident archetypes with pre-defined decision gates for SOAR integration. These frameworks directly support requirement 11.5.6's mandate for extraction-specific incident response playbooks.

### LoRD: Alignment-Aware LLM Extraction (ACL 2025)

**Locality Reinforced Distillation (LoRD)**, published at ACL 2025, marks a step change in LLM extraction efficiency. Prior extraction methods adapted techniques from smaller DNNs and ignored the alignment training that makes LLMs useful — LoRD closes that gap by treating extraction as a policy-gradient reinforcement learning task, using the victim model's responses as implicit reward signals. The result: a pre-trained 8B-parameter local model can approximate a 175B-parameter commercial LLM with as few as 100 API queries. LoRD also explicitly defeats watermark protections through exploration-based stealing that avoids the fixed query patterns that watermark verification relies on. Source code is available at https://github.com/liangzid/LoRD-MEA. This directly challenges the assumptions behind requirements 11.5.1 (rate limits set for thousands-of-queries budgets may be far too generous) and 11.5.4 (static watermarks may not survive alignment-aware extraction).

### RADEP: Resilient Adaptive Defense Framework (IEEE IWCMC 2025)

**RADEP** addresses limitations of earlier defense frameworks like AMAO (IEEE TDSC 2023) by combining four interlinked layers: progressive adversarial training to weaken extraction effectiveness, uncertainty-quantified malicious query detection using behavioral pattern analysis, an adaptive response mechanism that degrades output quality proportional to suspicion scores, and embedded ownership verification via watermarking. Where AMAO's label-flipping poisoning was a coarse-grained response to detected attackers, RADEP's graduated degradation preserves utility for borderline-suspicious users while maintaining strong defense against confirmed extraction campaigns. Testing shows minimal impact on legitimate users — a practical concern that hampered adoption of earlier all-or-nothing defense approaches.

### Hardware Side-Channel Extraction: NVBleed and BarraCUDA

Two research lines demonstrate that model extraction threats extend well beyond API queries into the physical layer. **NVBleed** (March 2025) reverse-engineers NVIDIA's NVLink interconnect and identifies two leakage sources: contention-based timing variations and accessible performance counters that disclose communication patterns. Using these channels, researchers achieved 97.78% F1 score in fingerprinting deep learning models across multi-GPU systems — and critically, the attack works remotely across VM instances in cloud environments like GCP (F1 > 88%), meaning co-tenants on shared GPU clusters can identify which models neighbors are running. **BarraCUDA** (2024) takes a complementary approach on edge devices: it uses correlation electromagnetic analysis (CEMA) to recover neural network weights and biases directly from NVIDIA Jetson hardware, requiring about five minutes of trace collection per weight. NVIDIA's official mitigation advice — prevent physical access — is inadequate for shared cloud and edge deployments. Together, these attacks mean that extraction defense must consider not just API-level controls but also infrastructure isolation: dedicated GPU instances, side-channel-resistant scheduling, and hardware performance counter access restrictions.

### Dual-Layer VLM Side-Channel Attacks (March 2026)

A March 2026 paper (arXiv:2603.25403) demonstrates that local Vision-Language Models using dynamic high-resolution preprocessing (like LLaVA-NeXT's AnyRes) leak information through two side channels simultaneously. The first tier exploits execution-time variation: square images require ~110 seconds (five patches) while rectangular images need ~48 seconds (three patches), revealing input geometry with a 2.3x timing difference. The second tier monitors Last-Level Cache (LLC) misses via hardware performance counters to distinguish visual density — dense structures like medical X-rays create more cache contention than sparse content. Combined, these channels classify input content with 84% accuracy. While not full model extraction, this demonstrates that multimodal architectures create novel side-channel surfaces that traditional API-level defenses do not cover.

### On-Device Inference: The Emerging Extraction Blind Spot

As of April 2026, a growing concern in model extraction defense is the proliferation of on-device and local AI inference — developers running models locally via Ollama, llama.cpp, and similar tools outside of API-controlled environments. When models are deployed on developer workstations or edge devices, traditional API-level defenses (rate limiting, query-pattern analysis, watermarking) are entirely bypassed. ### Industrial-Scale Distillation: The Anthropic and Google Disclosures (February 2026)

The campaigns used over 24,000 fraudulent accounts to generate more than 16 million exchanges, with each lab targeting distinct capabilities: DeepSeek focused on foundational logic and alignment across 150,000+ exchanges, Moonshot AI targeted agentic reasoning and tool use across 3.4 million exchanges, and MiniMax concentrated on agentic coding capabilities across 13 million exchanges. Anthropic attributed each campaign through IP address correlation, request metadata, infrastructure indicators, and corroboration from industry partners. Defense measures included classifiers and behavioral fingerprinting systems that identify distillation patterns in API traffic, strengthened verification for educational and research accounts, and model-level safeguards designed to reduce the efficacy of outputs for illicit distillation without degrading legitimate use. Anthropic emphasized that "no company can solve this alone," calling for coordinated action across AI labs, cloud providers, and policymakers.

### Malicious LLM API Router Intermediaries: "Your Agent Is Mine" (CCS 2026)

An April 2026 paper accepted at ACM CCS ("Your Agent Is Mine: Measuring Malicious Intermediary Attacks on the LLM Supply Chain") presents the first systematic analysis of security risks from third-party LLM API routers. The researchers analyzed 428 routers across Taobao, Shopify, and free markets, finding real-world malicious operators: 1 of 28 paid routers injected malicious code, 8 of 400 free routers deployed adaptive evasion triggers, and 17 routers abused researcher-owned AWS credentials. The study identifies four attack classes: response-side payload injection (AC-1, 100% rewrite compatibility across four major agent frameworks), passive secret exfiltration (AC-2, 100% coverage of secrets in payloads), dependency-targeted injection (AC-1.a, 99.6% compatibility), and conditional delivery (AC-1.b, evasion triggers activating only after N benign requests or during YOLO-mode sessions). Critically, "no end-to-end integrity mechanism binds the provider's tool-calling output to what the client finally receives" — a compromised router at any layer in a chain can silently intercept all inference traffic, enabling model extraction without querying the model API directly. The study found that 401 of 440 observed Codex sessions operated in YOLO mode, making sophisticated evasion unnecessary. Defense recommendations include high-risk tool policy gates (100% detection at 1.0% FP for basic attacks), response-side anomaly screening (89% detection at 6.7% FP), and append-only transparency logging. Long-term, the authors propose provider-signed response envelopes analogous to DKIM signatures — currently no major provider offers this capability. This directly extends extraction defense beyond API-level controls to supply-chain integrity of the inference routing layer.

### Neural Honeytrace: Training-Free Watermarking (January 2025)

**Neural Honeytrace** introduces a training-free, plug-and-play watermarking framework built on an information-theoretic model of watermark transmission through extraction attacks. Unlike prior approaches (DAWN, EWE, Composite Backdoor, MEA-Defender) that require model retraining, Neural Honeytrace operates at the inference layer by calculating feature similarity between input queries and predefined watermark samples, then encoding watermark information into output predictions through logit mixing. Its multi-step transmission strategy encodes watermark information into the label distribution itself via probability-based label flipping, making the watermark transfer to attacker-trained surrogates through the extraction process. The framework reduces samples required for copyright claims from approximately 12,000 to 200, achieves 36.6% minimum watermark success rate under worst-case adaptive attacks (versus 2.1% for MEA-Defender), and is effective even in hard-label scenarios — the most common MLaaS configuration where competing methods fail. The framework classifies attackers into three categories (naive, adaptive, oracle) and maintains robustness against pruning-based removal. Open-source implementation: https://github.com/NeurHT/NeurHT. This is particularly relevant for organizations that cannot retrain production models to embed watermarks, directly addressing a practical barrier to adopting requirement 11.5.4.
