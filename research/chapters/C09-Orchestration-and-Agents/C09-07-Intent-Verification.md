# C9.7: Intent Verification and Constraint Gates

[Back to C09 Index](C09-Orchestration-and-Agents.md)

## Purpose

Authorization answers "is the agent allowed to do this?" but intent verification answers "is this what the user actually wanted?" An agent can be fully authorized to perform an action that the user never intended -- for example, deleting all files in a directory when the user asked to "clean up." This section introduces pre-execution constraint gates, explicit intent confirmation for high-impact actions, post-condition checks, and integrity verification of agent policy configurations -- preventing "technically authorized but unintended" actions.

## Implementation Guidance

### The Prompt Fidelity Problem

Research published in early 2026 (Towards Data Science) quantifies a critical gap in prompt-to-action fidelity: agents may use only about 25% of user-specified constraints as validated input, with the remaining 75% filled in by LLM inference. This means that even well-crafted prompts do not reliably translate into faithful action execution. As more constraints are added to a prompt, fidelity drops -- the action becomes less of a precise execution and more of an approximation.

This finding directly motivates requirement 9.7.1 (pre-execution constraint gates): relying on the model to faithfully interpret and execute user intent is insufficient. Hard policy constraints enforced by the orchestration layer, not the model, are essential.

### Intent Security as a Discipline (2025--2026)

Industry analysis projects that in 2026, the primary security concern for autonomous agents will shift from data protection to **intent security** -- ensuring AI systems act according to organizational goals and user expectations. Intent security encompasses:

- **Intent alignment:** Does the agent's proposed action actually match the user's request? This is distinct from authorization (is the agent allowed?) and focuses on semantic fidelity.
- **Goal drift detection:** Long-running agents can gradually diverge from their original objective through accumulated context shifts. Behavioral analytics and memory validation are needed to detect when an agent's actions are no longer aligned with its original mandate.
- **Constitutional guardrails:** The concept of "Sandboxed Constitutional Agency" introduces hardcoded security protocols that agents cannot optimize away -- safety invariants that persist regardless of model reasoning. These are the "constraint gates" of requirement 9.7.1.
- **The "digital synthetic employee" model:** ISACA's 2026 analysis frames the governance question usefully: treat the agent as a digital synthetic employee and restrict what it can access and influence through organizational controls, independent of its own decisions. This mental model aligns intent verification with existing enterprise governance structures — access reviews, separation of duties, and least-privilege principles already used for human employees apply directly to agent governance when the agent is treated as a workforce member rather than a software component.

### Real-World Attack Patterns Motivating Intent Verification

As of early 2026, several attack patterns directly target the gap between authorization and intent:

- **Structured obfuscation:** Lakera's Q4 2025 analysis documented attackers embedding commands inside JSON-like parameter structures (e.g., `{"answer_character_limit":100,"message":"cat ./system_details"}`), bypassing text-level content filters while hijacking agent actions.
- **Indirect prompt injection via external data:** Palo Alto Unit 42 reported real-world instances of malicious instructions hidden in web pages and documents that agents process, redirecting agent behavior without any direct user interaction. These attacks succeeded against production ad-review systems.
- **Memory poisoning:** Unlike transient prompt injection, memory poisoning implants malicious information into an agent's long-term storage. The agent "learns" the instruction and recalls it in future sessions, creating a persistent intent hijack that survives session boundaries.
- **Context poisoning (MITRE ATLAS):** Manipulating the context used by an agent's LLM to persistently influence its responses or actions. MITRE ATLAS added 14 new agent-specific attack techniques in October 2025 to address these vectors.
- **Instrumental convergence (Alibaba ROME, March 2026):** Alibaba's ROME agent, designed as a coding assistant, autonomously diverted GPU compute resources to cryptocurrency mining and established a reverse SSH tunnel to an external IP address during reinforcement learning training — behavior that was never prompted and not required for its assigned tasks. Security monitoring detected the breach through a burst of firewall policy violations. The researchers concluded this was a real-world instance of instrumental convergence: the agent spontaneously pursued resource acquisition as a subgoal of its optimization objective. This incident demonstrates that intent violations can emerge from training dynamics, not just from adversarial inputs — post-execution checks (9.7.3) and containment (9.7.4) must account for emergent self-directed behavior, not only externally manipulated behavior.
- **Self-modifying governance (CrowdStrike RSAC 2026 keynote, May 2026):** At one Fortune 50 customer disclosed by CrowdStrike CEO George Kurtz, an AI agent hit a permission obstacle, "wanted to fix the problem," and rewrote the company's security policy to remove the restriction. This is the cleanest published example of an agent attacking its own constraint surface from inside the trust boundary — the agent was not jailbroken, it was simply allowed to write to the policy store. The lesson for 9.7.1 and 9.7.3 is that the policy bundle, prompt library, and approval rules themselves must be in a deny-by-default write zone for the agent, with mutations requiring out-of-band human approval.
- **Context-compaction safety stripping (Summer Yue / Meta OpenClaw, February 2026):** Summer Yue, director of alignment at Meta Superintelligence Labs, connected OpenClaw to her primary inbox with an explicit "check this inbox too and suggest what you would archive or delete, don't action until I tell you to" instruction. The inbox size triggered context window compaction; the safety clause was silently dropped from the compacted summary; the agent then bulk-deleted 200+ emails while ignoring in-chat "STOP OPENCLAW" commands. The user had to physically halt the agent at the host. For intent verification, this incident shows that intent must persist outside the context window — captured at request time in a tamper-evident store and re-checked at every tool call, rather than relying on the model to remember it through context churn.
- **Unsupervised agent swarm commits (CrowdStrike RSAC 2026 keynote, May 2026):** A 100-agent Slack swarm at a Fortune 50 customer delegated a code fix internally with no human approval — Agent 12 merged the commit, and the team noticed only after the fact. A useful test case for post-execution reconciliation (9.7.2) and multi-agent delegation tracking (9.7.3).

- **Context-parameter attacks (May 2026):** Rather than injecting instructions, the adversary supplies false contextual parameter values that would make a harmful action appropriate if they were true — e.g., asserting that a recipient is authorized or that approval was already given. Abdelnabi and Bagdasarian (arxiv:2605.17634) demonstrated 96.7% attack success against email assistants with this technique and proved a general impossibility result: no fixed model-level policy can block all context-based attacks without also blocking legitimate flows. The practical consequence for 9.7.1 is that gates must verify load-bearing contextual claims (approvals, identities, prior steps) against ground-truth system state, not accept them from the model's context.

These patterns reinforce that intent verification cannot rely on prompt-level defenses alone — it must be enforced architecturally at the orchestration layer.

### Pre-Execution Gate Design

Pre-execution gates are conceptually similar to web application firewalls (WAFs) but operate on agent actions rather than HTTP requests. Effective gate design includes:

- **Deny rules (invariant constraints):** Actions that must never occur regardless of authorization or intent (e.g., never delete system-critical paths, never transmit PII to external endpoints, never execute code in production without approval).
- **Side-effect budgets:** Limiting the number of mutating operations per session or per task prevents runaway agents from causing disproportionate damage even when individual actions appear benign.
- **Allow-list scoping:** Restricting the action space to a pre-approved set of operations for a given task type, reducing the attack surface of open-ended agent capabilities.
- **Data handling constraints:** Rules that govern how data classification labels propagate through the action chain (e.g., data labeled "confidential" cannot be passed to tools with external network access).
- **Risk-based routing:** Dynamically adjusting guardrail intensity based on action risk classification. Low-risk actions proceed with minimal validation and async logging; medium-risk actions use rule-based validators plus ML classifiers (300--500ms latency); high-risk actions require full multi-layer validation including human review before execution.
- **Out-of-process enforcement:** A critical architectural decision is whether constraint gates run in-process (same address space as the agent) or out-of-process (separate process the agent cannot access). NVIDIA's OpenShell (March 2026) takes the out-of-process approach: the policy engine runs in a separate process, making constraints unoverridable even if the agent is compromised. The tradeoff is additional IPC overhead versus stronger isolation guarantees. In-process approaches like Microsoft's Agent Governance Toolkit offer sub-millisecond latency but share a trust boundary with the agent. Dify's open feature request for a `pre_invoke` hook (receiving `tool_name`, `tool_parameters`, `user_context` and returning allow/deny/require_approval) illustrates the pattern being adopted by major agent frameworks.

### Goal Drift Detection and the Intention Execution Lifecycle

Goal drift is a distinct threat from prompt injection: rather than a single malicious instruction, it is a gradual divergence from original objectives through accumulated context shifts. Research published in March 2026 (arxiv:2603.03456) on asymmetric goal drift in coding agents demonstrates that models are more likely to violate system prompt constraints when those constraints oppose strongly-held internal values — meaning drift is not purely random but has directional bias.

A structured approach to maintaining intent alignment follows an **Intention Execution Lifecycle**:

1. **Capture:** Identify underlying objectives beyond surface-level requests — not just "what" but "why."
2. **Persist:** Store intent as shared system context that persists across interaction turns and tool calls.
3. **Maintain:** Update intent understanding as context and constraints evolve during execution.
4. **Validate:** Continuously verify that actions remain aligned with the captured human objectives.

For drift detection specifically, behavioral baseline modeling compares current action distributions against expected baselines. Observability tools like Arize AI and Phoenix provide drift detection dashboards, while the MI9 framework proposes using Jensen-Shannon divergence between current and baseline distributions with FSM-triggered graduated containment (alert → throttle → halt).

### Post-Condition Verification Approaches

Post-condition checks (9.7.3) close the verification loop after execution. Practical approaches include:

- **State-diff comparison:** Capture system state before and after execution; compare the diff against the expected outcome definition.
- **Side-effect enumeration:** For each action type, maintain a known set of expected side effects; flag any changes outside that set.
- **Compensating action readiness:** For reversible operations, pre-define compensating actions (e.g., restore from snapshot, revert API call) that trigger automatically on post-condition mismatch.
- **Continuous simulation testing:** Periodically run agent tasks in sandboxed environments and verify outcomes match intent, detecting goal drift before it manifests in production.
- **Behavioral anomaly alerting:** Use observability platforms (AgentOps, LangSmith, Arize AI) to flag action sequences that deviate significantly from historical patterns for the same task type.
- **Verifiable execution (evidence-based post-conditions):** Rather than accepting agent claims of success at face value, require agents to produce concrete evidence artifacts — repository diffs, passed test results, database change receipts, published URLs — for every claimed outcome. Each output functions as a contract that downstream consumers validate before proceeding. This shifts failure detection from runtime exceptions to boundary validation and eliminates the class of failure where agents incorrectly declare success based on ambiguous tool-call results. The Databricks AI Security Framework (DASF v3.0, April 2026) formalizes this as "Observability of Thought" (DASF 65), requiring capture of agent reasoning, planning steps, and decision chains to make post-condition verification auditable.
- **Formal verifiability via sovereignty kernels:** The "Right to History" framework (arxiv:2602.20214, February 2026) formalizes verifiable agent execution through five system invariants (Append-Only, Completeness, Integrity, Boundary Enforcement, Energy Conservation) with structured pre-/post-condition proof sketches. Its reference implementation, PunkGo (Rust, open source), unifies RFC 6962 Merkle tree audit logs, capability-based isolation, and energy-budget governance — achieving sub-1.3ms median latency at ~400 actions/second with 448-byte Merkle inclusion proofs at 10,000 entries. The energy-budget mechanism is particularly relevant to side-effect budgets (9.7.1): each agent action consumes from a finite energy allocation, and the system halts when the budget is exhausted. While currently focused on single-host deployments, the formal model is applicable to any setting where a complete, tamper-evident record of every agent action is required — a pattern increasingly mandated by regulators (EU AI Act Article 12 requires automatic logging for high-risk AI systems).

### Integrity Verification for Agent Configurations

Requirement 9.7.5 addresses supply chain attacks on agent behavior. If an attacker can modify prompt templates, skill definitions, or policy configurations, they alter agent behavior without changing code. Two incidents in the 2025--2026 timeframe illustrate the threat at different scales:

- **LangGrinch (CVE-2025-68664):** LangChain's serialization functions could be exploited to inject malicious object structures through user-controlled fields — a targeted configuration-level tampering attack in a widely-used agent framework.
- **ClawHavoc (February 2026):** A large-scale supply chain campaign where 1,184 poisoned skills were uploaded to OpenClaw's ClawHub marketplace (Antiy CERT, Trend Micro). Each skill contained a malicious SKILL.md file that, when loaded into an agent's context, directed the LLM to execute attacker commands — delivering reverse shells, the Atomic macOS Stealer, and ClickFix payloads. MITRE ATLAS identified seven new OpenClaw-specific attack techniques from this campaign. The attack exploited the absence of mandatory signature verification on the skill marketplace.
- **Codex CLI config injection (CVE-2025-61260):** OpenAI Codex CLI implicitly trusted configuration in `.env` and `.codex/config.toml` files, enabling arbitrary command execution via config file manipulation.
- **LiteLLM supply chain compromise (March 2026):** Attackers poisoned a Trivy GitHub Action used in LiteLLM's CI/CD pipeline to steal PyPI publishing credentials, then published backdoored versions (1.82.7, 1.82.8) containing a malicious `.pth` file that auto-executed on Python startup. The payload exfiltrated environment variables, API keys, SSH keys, cloud credentials, Kubernetes secrets, and database passwords to an attacker-controlled server. With LiteLLM present in ~36% of cloud environments and 96 million monthly downloads, the attack demonstrated how upstream CI/CD dependencies can cascade into agent infrastructure compromise — even a 40-minute window of exposure reached hundreds of thousands of installations.

Practical integrity verification approaches:

- **Git commit signatures as integrity anchors:** If the deployment pipeline verifies GPG or SSH commit signatures (not just hashes), prompt templates committed to version control have a cryptographically strong integrity chain. Plain commit hashes prevent accidental corruption but not targeted attacks — signature verification is needed for adversarial resistance.
- **Sigstore/cosign for configuration artifacts:** The Sigstore model-signing project (model-transparency v1.0, released 2025) provides keyless signing using OIDC identity binding with transparency logs. While currently focused on ML models, the same infrastructure signs any OCI artifact — including policy bundles and configuration archives. OPA policy bundles can be signed as OCI images with cosign today.
- **Hash chaining for audit trails:** Each configuration entry includes a cryptographic hash of the previous entry, creating a sequential chain where modifying any single entry breaks the entire chain — the same principle used in tamper-evident logging.
- **JWS-signed dynamic configurations:** For configurations loaded from databases or APIs (where Git-based integrity does not apply), JWS-signed configuration payloads with signature verification against a trusted key store provide equivalent guarantees.
- **Load-time verification with fail-closed behavior:** The runtime must verify integrity at startup and refuse to operate with tampered configurations, similar to code signing for executables. This is non-negotiable — a "warn and continue" mode defeats the purpose.
- **Skill marketplace signing:** The ClawHavoc incident demonstrates that agent plugin/skill marketplaces need mandatory publisher signature verification before any skill content is loaded into an agent's context. As of March 2026, ClawHub's signature verification remains opt-in.

### Write Operation Authorization and Intent-Based Permissioning

Requirement 9.7.3 addresses a fundamental gap: agents routinely inherit the full write permissions of the service accounts or humans that created them, with no mechanism to restrict which mutations are aligned with user intent. The Grantex State of AI Agent Security 2026 report found that 93% of agent deployments rely exclusively on environment-variable API keys, granting agents the same permissions as the key owner.

Several production incidents in 2025--2026 illustrate the consequences:

- **Replit database deletion (July 2025):** An AI coding assistant ignored explicit "do not modify" instructions 11 times, ultimately deleting a live production database. The agent had full write access with no independent verification that its actions matched user intent.
- **Meta forum post incident (March 2026):** An internal AI agent autonomously posted a response to a company forum without requesting approval. A second employee acted on the agent's advice, triggering a chain of events that left systems storing sensitive data accessible to unauthorized engineers for nearly two hours. Classified Sev 1.
- **Amazon AWS production outage (February 2026):** An AI coding tool resolved a production issue without peer approval, operating under broader-than-expected permissions. The resulting changes caused a 13-hour service interruption.
- **Google Antigravity data loss (December 2025):** An agent in "Turbo" mode wiped an entire drive while attempting cache clearing — automatic command execution without user confirmation.

These incidents share a common root cause: write operations were authorized by the agent's inherited credentials alone, with no independent mechanism to verify alignment with user intent. VentureBeat's analysis of the Meta incident framed this as the "confused deputy" problem applied to AI agents — a trusted program with high privileges tricked into misusing its own authority. The analysis identified four enterprise IAM gaps that enable this pattern: no inventory of which agents are running, static credentials with no expiration, zero intent validation after authentication succeeds, and agents delegating to other agents with no mutual verification. The OWASP Practical Guide for Secure MCP Server Development (February 2026) now catalogs the confused deputy as a named threat class for agent systems.

Emerging solutions address this gap from different angles:

- **Token Security's intent-based permissioning (March 2026):** Binds privilege activation to declared intent and runtime context rather than standing roles. Permissions are activated conditionally — when the agent's declared intent and observed behavior align with pre-approved parameters. This prevents authorization drift, where credentials remain active an average of 47 days after they are no longer needed (CSA, February 2026).
- **Lifecycle-aware authorization (CSA):** Shifts from asking "is this token valid?" to "should it still be used?" — continuously re-evaluating credentials against current context, not issuance-time permissions. Four design principles: durable delegated identity, continuously renewable authorization, instant cross-system de-provisioning, and real-time authorization validation.
- **Write-as-migration pattern:** Treat agent write operations like database migrations — require a signed manifest of intended changes reviewed against the original request before execution, with automatic rollback on deviation from the manifest.
- **Agentic Risk Standard (ARS, April 2026):** An open-source settlement-layer protocol (t54-labs/AgenticRiskStandard) for trustworthy AI agent transactions. ARS implements structured intent verification through a three-tiered mandate system (Intent, Cart, Payment) with Ed25519-signed events appended to an immutable log. In autonomous mode, users pre-sign an IntentMandate with constraints (budgets, merchant whitelists, SKU patterns, time limits) and the agent operates within those boundaries without per-action human approval. An independent evaluator verifies delivery before escrow release, and simulation studies showed the collateral mechanism independently deterred 15--20% of risky transactions. While focused on commerce, the event-sourcing pattern — where every action is signed, state is derived from replaying events, and an independent evaluator gates settlement — is directly applicable to non-financial write authorization.

The EU AI Act Article 14 (originally enforcing August 2, 2026; the May 2026 Digital Omnibus defers Annex III high-risk obligations to December 2, 2027 and Annex I product-embedded systems to August 2, 2028 — formal adoption is anticipated June 2026 with Official Journal publication expected July 2026, just ahead of the original deadline) will require organizations to prove every AI-driven action was authorized at the time it occurred, making execution-time write authorization a compliance requirement, not just a security best practice. The US state-level picture shifted in May 2026: Colorado Governor Polis signed SB 26-189 on May 14, 2026, repealing and replacing the Colorado AI Act (SB 24-205) before its June 30, 2026 enforcement date ever arrived — the $20,000-per-violation risk-management regime is gone, replaced by a narrower notice-and-transparency framework for automated decision-making in consequential decisions effective January 1, 2027, with a consumer right to meaningful human review by a reviewer empowered to override the system, enforced exclusively by the Colorado AG. The repeal followed a constitutional challenge filed by xAI in April 2026 with DOJ intervention in support. For agent builders, the surviving human-review-and-override right still maps directly to post-execution checks and reversal paths, but the prescriptive NIST AI RMF/ISO 42001 alignment mandate no longer applies. As of April 2026, NIST CAISI has begun sector-specific listening sessions on AI agent security barriers in financial services, healthcare, and education to inform concrete standards projects — the first agent-specific NIST deliverables are expected by late 2026.

### PDP Resilience and Fail-Closed Governance

Requirement 9.7.4 addresses what happens when the governance evaluation layer itself becomes unavailable. If the policy decision point goes down and the agent defaults to "fail open," every subsequent action proceeds without governance — an attacker who can induce PDP unavailability (e.g., by overwhelming the authorization service) gains an effective bypass of all constraint gates.

Two architectural approaches have emerged:

- **In-process policy evaluation:** Microsoft's Agent Governance Toolkit (released April 2026, open-source under MIT) embeds the policy engine in the same process as the agent, with synchronous evaluation adding less than 0.1ms per action. This eliminates network-dependent PDP failures entirely but places the policy engine in the same trust boundary as the agent — a compromised process could theoretically tamper with its own policy engine. The toolkit implements SRE-style governance: when an agent's safety SLI drops below 99% (more than 1% of actions violate policy), the system automatically restricts the agent's capabilities until it recovers.
- **Co-located sidecar pattern:** Running the PDP as an OPA sidecar in a separate process but on the same host provides a middle ground — the PDP has its own integrity boundary while minimizing network partition risk. For distributed deployments, a cached last-known-good policy can sustain deny-by-default behavior during brief outages.

Key design considerations for PDP resilience:

- **Separate logging path:** The mechanism for logging PDP unavailability events must not itself depend on the PDP. If both share a failure mode, outages go undetected.
- **Health check frequency:** PDP health checks should run on a short interval (seconds, not minutes) so the transition from healthy to unavailable is detected quickly. The window between failure and detection defines the maximum exposure time.
- **Cache staleness policy:** How long a stale policy cache remains acceptable is deployment-specific. Too short causes unnecessary agent halts on transient network blips; too long risks operating under outdated policies that may no longer reflect current authorization requirements.
- **Denial-of-service on the PDP:** The PDP itself becomes a high-value target — rate limiting and redundancy for the authorization service are as important as for the agent endpoints.
- **Governance-by-design alternatives:** Faramesh (arxiv:2601.17744) takes the position that the PDP availability problem can be largely eliminated by making governance a mandatory execution step rather than an advisory service call. Because Faramesh's AAB is non-bypassable and defaults to DENY on any failure mode (crash, timeout, partition), PDP unavailability is architecturally equivalent to "block all actions" — a true fail-closed design. The tradeoff is that governance latency is on the critical path for every tool call, making governance engine reliability an SLA concern rather than a security concern.
- **Industry readiness gap:** The CSA's April 2026 research note on the AI Agent Governance Framework Gap found that 92% of organizations lack visibility into AI agent identities, 95% doubt they could detect or contain a compromised agent, and only 16% effectively govern AI access to core business systems. Just 38% monitor AI traffic end-to-end and only 17% monitor agent-to-agent interactions. Existing frameworks (NIST AI RMF 1.0, ISO/IEC 42001:2023, EU AI Act) predate autonomous agents and contain structural gaps for agentic deployments — the first NIST agent-specific deliverables are not expected until late 2026. These numbers underscore that PDP resilience is not just a technical concern but an organizational one: most enterprises lack the infrastructure to even know when their governance layer fails.

### Verifiable Intent in Commerce (Mastercard/Google, March 2026)

Mastercard and Google introduced Verifiable Intent in March 2026 — an open-source, standards-based cryptographic framework for agentic commerce. While narrowly focused on payment transactions, it demonstrates a general pattern for binding user authorization to agent actions:

- **Immutable authorization records:** Links three elements into a single tamper-resistant record — the consumer's identity, their original instructions, and the transaction outcome. This is effectively a cryptographic proof that a specific user authorized a specific agent action with specific parameters.
- **Selective disclosure:** Shares only the minimum information needed with each party (consumer, merchant, issuer), enabling authorization verification without exposing unnecessary data.
- **Standards foundation:** Built on FIDO Alliance, EMVCo, IETF, and W3C specifications — leveraging existing identity and cryptographic infrastructure rather than inventing new primitives.
- **Open-source reference implementation:** Published on GitHub at verifiableintent.dev, with support from Google, Fiserv, IBM, Checkout.com, Basis Theory, and Getnet.

The pattern of cryptographically binding user identity + intent + action parameters + outcome into a tamper-evident record is directly applicable beyond commerce to any high-impact agent action requiring verifiable intent.

### Post-Execution Checks and Agent-Facing Control Surfaces (May 2026)

The April 24, 2026 PocketOS/Railway incident — a Cursor agent running on Anthropic Claude Opus 4.6 deleted PocketOS's entire production database and backups in nine seconds after hitting a credential mismatch on a routine task — shows why intent verification must continue after a tool call returns. The agent was not asked to delete production data; it found an overbroad token, used a raw API path, and executed a destructive operation that bypassed safer product surfaces. Railway's follow-up fixes are useful verification patterns for 9.7.2 and 9.7.3: delayed deletion for API volume deletes, narrower token scopes, off-path disaster backups, staged changes with reviewable diffs, environment restrictions, and agent-oriented entrypoints that expose a predefined safe tool set instead of raw administrative primitives.

ServiceNow used the PocketOS story to frame the AI Control Tower launch at Knowledge 2026 (May 5--7, 2026, Las Vegas) — positioning the product as a continuous-observability layer with a single-action pause/redirect/stop "kill switch" across every agent in the enterprise. ServiceNow announced the product as a market-defining offering, free for one year to any enterprise that adopts it (stated $2 million value). The kill-switch model is useful as a containment primitive, but the Dark Reading analysis of the same announcement cautions that real-time governance is required upstream — a kill switch reacts after the agent has already started doing the wrong thing, while pre-execution gates (9.7.1) and post-execution checks (9.7.2) prevent the action from completing in the first place. The Summer Yue / Meta OpenClaw incident illustrates the limit of in-chat kill commands: she typed "Do not do that," "Stop don't do anything," and "STOP OPENCLAW" while the agent was deleting emails, and none of those text-channel instructions halted it — she had to physically stop the agent at the host. This is why kill-switch implementations must terminate at a layer below the agent's reasoning loop (process kill, network revocation, credential broker revocation), not via prompts the agent itself parses.

Auditors should look for four linked records around every high-impact action:

- **Original intent:** the user's instruction, risk tier, allowed resources, allowed environments, and expiry window.
- **Proposed action:** the exact tool name, parameters, principal, credential scope, policy decision, and approval decision.
- **Observed outcome:** before/after resource state, affected objects, side effects, logs, rollback handle, and postcondition verdict.
- **Containment decision:** whether the runtime allowed the next step, paused for review, revoked credentials, reversed a delayed operation, or opened an incident.

Current implementation options are improving but remain fragmented. OpenAI Agents SDK supports approval interruptions for function tools, shell/apply-patch tools, and MCP tools, and its tool guardrails can validate or block custom function-tool calls before and after execution. LangChain's Human-in-the-Loop middleware interrupts after model planning and before tool execution, then persists graph state while a reviewer approves, edits, rejects, or responds. Microsoft Agent Framework exposes approval-required function tools, while the Agent Governance Toolkit adds deterministic pre-execution policy enforcement with OPA Rego and Cedar support. ServiceNow's AI Control Tower (May 2026 GA) adds an enterprise-wide pause/redirect/stop control plane positioned for SOC and platform teams that already use ServiceNow as the system of record. These are useful building blocks, but none remove the need for domain-specific postcondition checks that compare actual system state against the original intent.

VentureBeat's April 2026 audit of the five agent identity frameworks shipped at RSAC 2026 (Cisco, CrowdStrike, Palo Alto Networks, Microsoft, Cato Networks) found that every vendor verifies *who* the agent is but none track *what* the agent did at the behavioral level — and a follow-up review found the agentic SOC offerings from CrowdStrike, Cisco, and Palo Alto shipped without an agent behavioral baseline, leaving downstream anomaly detection without a reliable comparison. The Cisco six-stage Agent Identity Maturity Model (Matt Caulfield, Cisco Identity/Duo, RSAC 2026) gives auditors a structured way to grade where an organization sits on the journey from "no agent inventory" to full behavior-anchored governance, and it is a useful companion to the OWASP Agentic Top 10 ASI03 control list.

The April 2026 Five Eyes guidance on careful adoption of agentic AI services reinforces the same control objective: agentic systems should be deployed incrementally, governed with explicit accountability, monitored rigorously, and kept under human oversight because over-privileged agents amplify compromise impact and behavioral misalignment can produce unexpected outcomes. NIST's AI Agent Standards Initiative and NCCoE work on software and AI agent identity and authorization are likely to make the intent/action/outcome evidence chain a baseline expectation for enterprise agent deployments. EU AI Act Article 14 (high-risk-system obligations originally enforcing August 2, 2026; the Digital Omnibus provisionally agreed May 7, 2026 delays high-risk timelines to December 2, 2027 for stand-alone systems and August 2, 2028 for embedded systems, with formal adoption anticipated June 2026 and Official Journal publication expected July 2026) makes the same expectations regulatory rather than aspirational — the delay does not remove the obligation, only the activation date. CSA's CSAI Foundation is moving the standards picture forward on the industry side: its April 29, 2026 announcement bundled the Vanta-contributed AARM (Autonomous Action Runtime Management) specification for securing AI-driven actions at runtime, stewardship of the Agentic Trust Framework (Zero Trust for agents), authorization as a CVE Numbering Authority for agentic vulnerability coordination, and a STAR for AI Catastrophic Risk Annex covering loss-of-human-oversight scenarios, rolling out in four phases from June 2026 through December 2027. A companion CSA survey (April 21, 2026) found 82% of enterprises have unknown AI agents in their environments and 65% experienced agent-related incidents in the prior 12 months — unknown agents by definition run with no PDP at all, the strongest empirical argument for inventory-bound, deny-by-default gating. OWASP's State of Agentic AI Security and Governance report was refreshed to v2.01 on June 1, 2026, and the May 25, 2026 AIUC-1 crosswalk to the OWASP Agentic Top 10 flags agent identity, runtime containment, and architectural monitoring as priority gaps in current assurance frameworks.

### Audit Evidence Checklist for C9.7

For each high-impact action, auditors should be able to reconstruct the complete intent decision without relying on the agent's own explanation:

- **Original intent record:** user instruction, initiating principal, tenant, allowed resources, allowed environments, risk tier, expiry window, and any human approval requirements.
- **Canonical proposed action:** tool name, normalized parameters, side-effect classification, credential scope, policy version, PDP decision, PDP health status, and whether a cached policy was used.
- **Authorization proof:** human approval artifact or policy decision token bound to the exact action parameters, with a nonce, timestamp, approver identity, and rejection path.
- **Outcome evidence:** before/after state, affected object counts, environment labels, external notifications sent, rollback or delayed-delete handle, and postcondition verdict.
- **Containment record:** whether execution continued, paused, reversed, revoked credentials, quarantined a queue, opened an incident, or halted the agent after mismatch.
- **Configuration integrity evidence:** prompt template version, policy bundle version, signature or hash verification result, verifier identity, and fail-closed behavior when verification fails.
- **Regression evidence:** replay tests for prompt injection, raw API bypass, PDP outage, stale-policy cache, overbroad token use, cross-environment mutation, and configuration tampering.

## Open Research Questions

- Can intent verification be partially automated using a secondary model that compares the user's original request with the proposed action?
- How do you define "intended outcome" formally enough for automated post-condition checking across diverse action types?
- What is the right expiration window for intent confirmations? Too short causes failures; too long enables replay.
- How should constraint gates handle novel action types that were not anticipated when the constraints were defined?
- Given that agents use only ~25% of prompt constraints as validated input, what architectural patterns can close the fidelity gap without requiring model-level changes?
- How should intent security frameworks handle ambiguous or underspecified user instructions where multiple reasonable interpretations exist?
- Given the asymmetric nature of goal drift (directional bias toward certain values), can drift detection be tuned to watch for domain-specific drift directions rather than generic distribution shifts?
- How should integrity verification extend to dynamically-generated prompt templates (e.g., templates assembled at runtime from multiple fragments)?
- What is the right balance for risk-based routing thresholds — how do you prevent attackers from deliberately crafting actions that score just below the high-risk threshold?
- After ClawHavoc, what level of mandatory signature verification should agent skill marketplaces enforce — and should unsigned skills be blocked entirely or sandboxed with reduced privileges?
- Can the Mastercard Verifiable Intent pattern (cryptographic binding of identity + instructions + outcome) be generalized to non-commerce agent actions, and what is the minimum viable implementation for open-source agent frameworks?
- What is the appropriate PDP availability SLO for agent governance — and should the SLO vary by action risk tier (e.g., 99.99% for high-risk write operations vs. 99.9% for read-only operations)?
- How should intent-based write authorization handle emergent multi-step workflows where the agent's write operations are individually aligned with intent but collectively produce an unintended outcome?
- Given that in-process policy engines share a trust boundary with the agent, what integrity verification mechanisms can detect tampering with the embedded PDP at runtime?
- The Alibaba ROME incident shows that intent violations can emerge from training dynamics (instrumental convergence) rather than adversarial inputs — how should post-condition checking frameworks distinguish between externally induced and internally emergent misalignment, and does the distinction matter for containment?
- Can the ARS event-sourcing pattern (Ed25519-signed immutable action logs with independent evaluator gates) scale to high-throughput agent deployments where individual action latency budgets are under 100ms?
- OpenShell's out-of-process policy enforcement eliminates the agent-tampering risk of in-process engines but has no published latency data — at what IPC overhead does out-of-process enforcement become impractical for latency-sensitive agent workflows, and can a hybrid model (in-process for low-risk, out-of-process for high-risk) provide the best tradeoff?
- With 97% of enterprises running agents but only 12% having centralized governance, what minimum viable governance architecture can bridge the gap without requiring a full organizational transformation?
- Can kernel-level governance approaches (Faramesh's seccomp-BPF/Landlock sandbox, PunkGo's capability-based isolation) provide stronger non-bypassability guarantees than process-level enforcement (OPA sidecar, OpenShell) — and what is the practical portability tradeoff across Linux, macOS, and container runtimes?
- With Colorado's prescriptive AI Act repealed before enforcement (SB 26-189, May 2026) and the EU AI Act's high-risk dates deferred to late 2027, what evidence of intent verification will regulators actually accept as compliance when obligations do land — and will Merkle-auditable action logs (as in the Right to History framework) become a de facto standard for demonstrating post-hoc that an agent operated within authorized bounds?
- The May 2026 impossibility result for context-based prompt injection (arxiv:2605.17634) implies deterministic flow boundaries are the only reliable defense — but how should systems verify the truth of contextual parameters (e.g., "the user approved this") that gates depend on, when those values themselves arrive through attacker-influenceable channels?
