# C8.3: Memory Expiry, Revocation & Deletion

> [Back to C08 Index](C08-Memory-and-Embeddings.md)
> **Last Researched:** 2026-05-02

## Purpose

Retention and revocation for memory and RAG systems must be measurable operations, not policy statements. A delete, expiry, quarantine, or reset request has only taken effect when the content is excluded from every retrieval path that can influence a response: dense vectors, sparse or keyword indexes, rerankers, graph memories, tool caches, application caches, backups, and restored snapshots. For systems with long-term or cross-session memory, deletion is also a security boundary because poisoned entries can persist after the initiating interaction has ended.

As of May 2026, the practical control pattern is query-time exclusion first, physical reclamation second, and audit evidence throughout. Pinecone documents log sequence numbers for checking whether deletes are reflected in serverless query responses, Milvus TTL excludes expired entities from query results before later compaction, Qdrant exposes `wait` and `ordering` controls on point deletion, and Weaviate tenant states can make a tenant unavailable but are eventually consistent. These vendor semantics are useful, but they do not replace application-level deletion orchestration across hybrid search, caches, evaluation copies, exported files, and backup media.

Regulatory pressure is also concrete. The EDPB's 18 February 2026 right-to-erasure report found recurring problems with internal procedures, ineffective anonymisation used instead of deletion, uncertain retention periods, and backup deletion limits. The EU AI Act requires high-risk systems to support automatic event logging under Article 12 and requires providers to retain covered logs for at least six months under Article 19, so teams need a written conflict-resolution model for evidence retention, incident response, and privacy erasure rather than ad hoc deletion exceptions.

## Recent Research Notes

- Microsoft reported AI Recommendation Poisoning on 10 February 2026: more than 50 prompt-based attempts from 31 companies tried to plant persistent recommendations through pre-filled AI-assistant URLs. Microsoft maps the activity to MITRE ATLAS `AML.T0051` and `AML.T0080.000`, making memory reset and review a practical incident-response requirement, not only a privacy feature: https://www.microsoft.com/en-us/security/blog/2026/02/10/ai-recommendation-poisoning/
- MINJA, revised 12 February 2026, shows query-only memory injection against LLM agents without direct access to the memory bank; the January 2026 follow-up found that realistic memory state and trust-aware sanitization reduce attack effectiveness but require careful threshold tuning: https://arxiv.org/abs/2503.03704 and https://arxiv.org/abs/2601.05504
- MemoryGraft, published 18 December 2025, demonstrates persistent compromise through poisoned experience retrieval where small sets of malicious procedure memories can influence later tasks through combined lexical and embedding retrieval: https://arxiv.org/abs/2512.16962
- RAGForensics, published at The Web Conference 2025, gives responders a practical reason to retain quarantined material under restricted access: traceback needs the suspect chunks to remain available for analysis while they are excluded from production retrieval: https://papersgraph.com/research-papers/traceback-of-poisoning-attacks-to-retrieval-augmented-generation
- OWASP LLM08:2025 treats vector and embedding weaknesses as a RAG security risk class covering unauthorized access, cross-context leakage, embedding inversion, poisoning, and retrieval behavior alteration. The same page recommends fine-grained access controls, data validation, classification, and immutable retrieval logs: https://genai.owasp.org/llmrisk/llm082025-vector-and-embedding-weaknesses/
- The EDPB's 2026 right-to-erasure findings and the EU AI Act's official Articles 12 and 19 text should be read together when setting memory retention policy: erasure must work in practice, but security and high-risk-system logs may have minimum retention duties: https://www.edpb.europa.eu/our-work-tools/our-documents/other/coordinated-enforcement-action-implementation-right-erasure_en and https://eur-lex.europa.eu/eli/reg/2024/1689/oj?locale=en

| **8.3.2** | **Verify that** memory can be reset for security reasons (quarantine, selective purge, full reset) through an operation that is separate and independent from the retention deletion process. | 2 | Incident response failure after memory poisoning or context poisoning. Microsoft observed real-world AI Recommendation Poisoning attempts in February 2026 using pre-filled assistant URLs to plant persistent "trusted source" memories, mapped to `AML.T0051` and `AML.T0080.000`. MINJA and MemoryGraft show that memory compromise can be query-only, indirect, and durable across later tasks, so responders need a reset path that does not wait for ordinary retention timers. | Run a poisoning drill. Seed benign and malicious memories, detect the malicious entries, and exercise three independent operations: immediate quarantine, selective purge by document/batch/user/session, and full memory reset. Verify each operation excludes the target entries from vector, lexical, hybrid, and agent-memory retrieval without deleting forensic evidence unless purge is explicitly selected. Hunt for AI-assistant URLs and prompts containing `remember`, `trusted source`, `authoritative`, `future conversations`, `cite`, or similar persistence language, as recommended in Microsoft's campaign analysis. Review user-visible saved memories where the product exposes them, and verify resets are auditable but do not require the ordinary privacy-erasure workflow to run first. | Most vector databases provide deletes, filters, collections, namespaces, and tenant controls, but not a complete incident-response state machine. A reliable implementation usually needs a staging area for proposed memory writes, provenance on every memory record, a deny-list or quarantine flag enforced by every query path, and a separate forensic store. The January 2026 memory-poisoning defense study found that trust-aware sanitization can help, but thresholds are operationally fragile: too strict blocks legitimate memories, too loose leaves subtle poisoned entries available: https://arxiv.org/abs/2601.05504 |
| **8.3.3** | **Verify that** quarantined content is retained for investigation but is excluded from all retrieval results while under quarantine. | 2 | Continued exposure or evidence destruction during RAG and memory-poisoning response. If suspect content remains retrievable, poisoned context can keep influencing outputs; if it is hard-deleted immediately, responders may lose the ability to scope affected queries, identify neighboring poisoned chunks, or prove which source document introduced the payload. RAGForensics-style traceback depends on retaining suspect chunks under controlled access while removing them from production retrieval. | Insert known-bad content with unique markers, run the quarantine operation, and verify zero production results through dense vector, keyword/BM25, hybrid, metadata, reranker, cache, and agent-memory paths. Then verify the same content remains available only through a restricted forensic interface with immutable audit logging. Restart services, force replica failover where possible, restore from a recent backup/snapshot in a test environment, and confirm the quarantine state survives. Test un-quarantine separately so the team can show that forensic retention, production exclusion, purge, and restoration are distinct operations. | Metadata-flag quarantine is easy to implement but fails if any query path omits the filter. Namespace or collection isolation is stronger but requires synchronized moves and can complicate restoring normal service. Qdrant snapshots include the vectors, IDs, payloads, and indexes needed to restore a collection efficiently, so restored snapshots must be checked for stale quarantine/deletion state before rejoining production: https://qdrant.tech/documentation/tutorials-operations/create-snapshot/ Weaviate backups do not include inactive or offloaded tenants unless they are activated first, which can surprise forensic workflows that use tenant state as a quarantine proxy: https://docs.weaviate.io/weaviate/manage-data/multi-tenancy |
