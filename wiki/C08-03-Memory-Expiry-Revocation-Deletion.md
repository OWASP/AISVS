# C8.3: Memory Expiry, Revocation & Deletion

> [Back to C08 Index](C08-Memory-and-Embeddings.md)
> **Last Researched:** 2026-03-29

## Purpose

Retention must be explicit and enforceable; deletions must propagate to derived indices and caches. Vector stores that grow without retention controls become an ever-expanding attack surface for data exfiltration and a compliance liability under GDPR, CCPA, and similar regulations. As of early 2026, Milvus v2.3+ TTL, Qdrant tenant lifecycle APIs, and Weaviate tenant lifecycle endpoints improve native support, but deletion propagation to derived indices and caches still requires custom engineering. Crypto-shredding support remains rare among vector database vendors.

| **8.3.1** | **Verify that** retention times are applied to every stored vector and related metadata across memory storage. | 1 | Unbounded data accumulation — without TTLs or retention policies, vector stores grow indefinitely, increasing the attack surface for data exfiltration and complicating regulatory compliance (GDPR Article 5(1)(e) storage limitation principle). Stale data also degrades retrieval quality. Stale poisoned vectors that should have been purged remain retrievable indefinitely without TTL enforcement, extending the window of any successful RAG poisoning attack. | Review vector store configuration for TTL settings or retention policy enforcement. Query stored vectors and verify TTL metadata is present. Check for automated expiration jobs or garbage collection processes. Verify retention policies cover both vector data and associated metadata/text chunks. | Pinecone does not support native TTL — requires application-layer deletion jobs. Weaviate supports TTL via `autoTenantDeletion` for multi-tenant setups and implements tenant-aware classes with lifecycle endpoints. Milvus added TTL support in v2.3. Redis Vector Search inherits Redis key TTL. Qdrant provides tenant-level lifecycle APIs for managing retention. For databases without native TTL, a separate cron-based cleanup process scanning metadata timestamps is the standard pattern. |
| **8.3.2** | **Verify that** only information required for the system's defined function is persisted in memory (such as user preferences and conversation decisions, not credentials or full conversation transcripts). | 1 | Over-retention of sensitive conversational data — full conversation transcripts in memory create a rich target for exfiltration and may contain credentials, personal disclosures, or business secrets shared in context. Maps to data minimization principles (GDPR Article 5(1)(c)). The "EchoLeak" zero-click attack demonstrated exfiltration of corporate data from Microsoft 365 Copilot by sending a specially crafted email — the hidden prompt caused Copilot to autonomously leak information from conversation memory. Unit 42's October 2025 research showed Amazon Bedrock Agents encoding user conversation history in URLs and transmitting it to attacker-controlled servers without visible indication. | Review memory write logic to identify what data categories are persisted. Compare persisted data against the system's functional requirements. Verify session-scoped memory is cleared at session termination. Test by conducting a session with sensitive content, ending it, and confirming the data is not retrievable in a new session. | Defining "required for the system's defined function" needs explicit documentation — a data retention specification. ChatGPT's memory feature persists user preferences but not full transcripts. Systems using ConversationBufferMemory in LangChain persist everything by default; ConversationSummaryMemory is better but may still retain sensitive details in summaries. Memory poisoning research (MINJA, NeurIPS 2025) showed that over-retained conversation data becomes the attack surface — all prompts including commands like "remember," "in future conversations," or "as a trusted source" were used to ensure long-term influence. |
| **8.3.3** | **Verify that** deletion requests purge vectors, metadata, cache copies, and derivative indices within an organization-defined maximum time. | 1 | Right-to-erasure non-compliance (GDPR Article 17) and residual data exposure — if deletion only removes the vector but leaves metadata, cached text chunks, or derived indices (e.g., BM25 keyword indices alongside vector indices), the sensitive content remains accessible through alternative retrieval paths. | Issue a deletion request for a known document. After the defined maximum time, attempt to retrieve the document via: (1) vector similarity search, (2) metadata filter query, (3) any keyword/BM25 index, (4) application-layer caches. Verify all paths return no results. Check backup systems for residual copies. | Hybrid search architectures (vector + keyword) are increasingly common (e.g., Weaviate's hybrid search, Pinecone's sparse-dense). Each index must be covered by deletion propagation. Cache invalidation is particularly challenging — Redis caches, CDN-layer caches, and in-memory application caches all need explicit purge logic. Document the full data flow map before implementing deletion. Private networking (VPC peering or private link with no public ingress) available from Pinecone, Weaviate, and Qdrant for production deployments reduces the exposure window during deletion propagation. |
| **8.3.4** | **Verify that** deleted or expired vectors are removed reliably and are unrecoverable. | 2 | Data remanence in vector stores — soft-delete implementations or lazy garbage collection may leave vector data recoverable through direct storage-layer access, low-level API calls, or backup restoration even after logical deletion. | Verify the deletion mechanism is hard-delete, not soft-delete/tombstone with deferred cleanup. For HNSW-based indices, confirm that deleted node slots are overwritten on compaction. Check that storage-layer files (e.g., segment files in Milvus, WAL in Qdrant) do not retain deleted data after compaction. Review backup retention policies. | Most vector databases use HNSW or IVF indices that do not immediately reclaim deleted vector slots — they mark them as deleted and reclaim space during compaction/merge operations. Qdrant uses WAL with periodic snapshots; deleted data persists in WAL until the next snapshot. Milvus requires explicit `compact()` calls. True unrecoverability may require encryption with key destruction (crypto-shredding). |
| **8.3.5** | **Verify that** expired vectors are excluded from retrieval results within a measured and monitored propagation window. | 2 | Stale data serving — expired or deleted vectors that remain retrievable during a propagation window can serve outdated, revoked, or sensitive content. The window must be defined, measured, and kept as short as possible. | Define the maximum acceptable propagation delay. Issue a deletion/expiration, then immediately query for the affected vectors. Measure the actual time until they stop appearing in results. Verify monitoring/alerting exists for propagation delays exceeding the defined maximum. | In distributed vector databases, propagation delay depends on replication lag, cache TTLs, and index rebuild frequency. Pinecone's serverless architecture has eventual consistency with typical delays under 10 seconds. Weaviate's distributed mode uses Raft consensus but query-time filtering may still serve stale replicas. Application-layer caching adds another propagation layer that must be accounted for. |
| **8.3.6** | **Verify that** memory can be reset for security reasons (quarantine, selective purge, full reset) through an operation that is separate and independent from the retention deletion process. | 2 | Incident response capability — when a memory poisoning attack is detected, the team needs to quarantine suspected vectors immediately (excluding them from retrieval) while preserving them for forensic analysis. Without this separation, teams face a choice between continued exposure and evidence destruction. Given MINJA's >95% injection success rate and A-MemGuard's finding that LLM-based classifiers miss 66% of poisoned entries, rapid quarantine capability is essential for incident response when poisoning is detected through behavioral monitoring rather than entry-level scanning. | Simulate a poisoning incident. Verify that quarantine operations: (1) immediately exclude targeted vectors from all retrieval paths, (2) preserve quarantined vectors in a separate, access-controlled storage for investigation, (3) do not trigger retention-based deletion timers. Test selective purge (specific documents/batches) and full reset independently. Schneider's four-layer defense recommends circuit breakers for rapid isolation when compromise is detected. | Most vector databases do not have native quarantine functionality. Implementation patterns include: moving vectors to a separate "quarantine" collection/namespace, adding a metadata flag and enforcing query-time filtering, or maintaining a deny-list of vector IDs checked at retrieval time. The deny-list approach has the lowest latency impact but requires consistent enforcement across all query paths. Qdrant's tenant-level lifecycle APIs can facilitate per-tenant quarantine operations. Weaviate's tenant lifecycle endpoints support suspending individual tenants without data deletion. |
