# C13.2: Abuse Detection and Alerting

> **Parent:** [C13 Monitoring, Logging & Anomaly Detection](C13-Monitoring-and-Logging)
> **Requirements:** 10 (13.2.1 -- 13.2.10)

## Purpose

This section addresses the detection of adversarial activity, abuse patterns, and attack campaigns targeting AI systems. Unlike traditional application security monitoring, AI abuse detection must account for natural language attack vectors (prompt injection, jailbreaks), multi-turn attack strategies, and AI-specific exfiltration techniques. These controls bridge the gap between raw logging (C13.1) and incident response (C13.5).

## Implementation Guidance

### AI Firewall and Guardrail Products (2025-2026)

A new category of "AI Firewalls" has emerged to address requirements 13.2.1 and 13.2.4-13.2.6 at the infrastructure layer:

- **Azure AI Content Safety -- Prompt Shields** -- Microsoft's unified API that analyzes prompts and documents before content generation, detecting both direct jailbreak attempts and indirect prompt injection from retrieved documents. Provides classification scores and block/allow decisions that can be logged per 13.1.5.
- **Cloudflare Firewall for AI** -- Edge-layer protection that intercepts LLM API traffic, applying prompt injection and jailbreak detection rules across all models behind a single policy. Enforces guardrails at the network layer before requests reach model endpoints.
- **Akamai Firewall for AI** -- Real-time monitoring and adaptive filtering that detects jailbreaks, prompt injections, and toxic content with guardrail policies applied at the CDN/WAF layer.
- **Palo Alto Networks AI Security** -- Enterprise AI security posture management with prompt injection detection integrated into their NGFW and Prisma Cloud platforms. Unit 42 research has documented web-based indirect prompt injection attacks against AI agents observed in the wild.

- **CrowdStrike Falcon AIDR** -- As of March 2026, provides real-time detection and prevention of prompt injection, jailbreaks, and model manipulation. Deploys via browser extensions, SDKs (Python, Node.js, Go, Java, C#) with OpenTelemetry support, and AI/API gateway integrations (LiteLLM, Kong, Apigee, Azure API Gateway). Monitors MCP server communications and detects harmful content including toxic material. Findings stream directly to Falcon Next-Gen SIEM for cross-domain correlation across endpoints, cloud, identity, SaaS, and AI behaviors.

- **CalypsoAI** -- Inference-time AI security platform with agentic red-teaming, real-time defense, and continuous observability. Covers prompt injection, jailbreaks, data leakage, and model manipulation for both standalone models and AI agents.
- **Lasso Security** -- Secured LLM gateway with browser and application integrations for real-time detection, logging, masking, or blocking of risky interactions including data leaks and prompt injection.
- **Check Point GenAI Protect + CloudGuard WAF (with Lakera, Sept 2025)** -- Check Point's ~$300M acquisition of Lakera (announced September 2025, closed Q4 2025) is being integrated into Check Point's Infinity Platform. First integrations target CloudGuard WAF (protecting AI-enabled apps) and GenAI Protect (securing user traffic to SaaS GenAI). Lakera's classifiers report 98%+ detection across prompt injection, jailbreaks, and data leakage with sub-50ms latency and <0.5% false positive rates across 100+ languages. Lakera now forms Check Point's Global Center of Excellence for AI Security in Zurich.
- **Portkey AI Guardrails** -- LLM gateway with plug-in guardrails that the vendor reports reduce prompt injection vulnerability by 95% with sub-50ms overhead; wraps existing LLM calls in three lines of code. Real-world adoption reported at 300% growth through early 2026, making it a common gateway option for teams that cannot deploy inline SDKs on every workload.
- **Meta LlamaFirewall + PromptGuard 2** -- Open-source guardrail framework (May 2025) with three layers: PromptGuard 2 (BERT-style jailbreak/injection classifier in 86M and 22M variants), Agent Alignment Checks, and CodeShield for static analysis. On the AgentDojo benchmark, PromptGuard 2 (86M) alone reduced attack success rate from 17.6% to 7.5% with minimal task-utility loss. The 22M variant cuts latency by ~75% vs. 86M for latency-sensitive inline deployments.

These products enable requirement 13.2.2 (SIEM integration) by emitting structured security events via standard protocols (CEF, syslog, OTLP) that flow into existing SOC workflows.

### Open-Source Detection Tools

For organizations building detection pipelines without commercial products, several open-source tools address requirements 13.2.1 and 13.2.4:

- **Vigil** -- Python library and REST API for assessing LLM prompts and responses. Uses multiple detection methods: vector database/text similarity, YARA/heuristic rules, transformer model analysis, prompt-response similarity, and canary tokens. Best-in-class for low false-positive deployments (0% FPR at 83.8% recall in June 2025 benchmarks).
- **Rebuff** -- Multi-layered prompt injection defense by Protect AI. Combines heuristic filtering, LLM-based detection (GPT-4o achieves 0.021 FPR vs. 0.499 for GPT-3.5-turbo), vector database of known attacks, and canary tokens. Most balanced trade-off at 3.4% FPR and 98.1% recall.
- **LLM Guard** -- Open-source prompt injection detection with highest recall (99.9%) but higher false-positive rate (12.7% FPR). Hard-coded detection threshold at 0.92 limits configurability.

A comparative evaluation (arxiv:2506.19109) found that combining multiple scanner types outperforms any single technique, and vector database scanners degrade 10-40% against obfuscated attacks. Canary token checks in default configurations showed 0% detection rates but improved to 41-94% with explicit instructions.

### Evolving Attack Landscape

The threat landscape for AI abuse detection continues to evolve rapidly:

- **Best-of-N (BoN) Jailbreaking** -- Automated attacks that generate large numbers of prompt variations using obfuscation techniques (character substitution, encoding, rephrasing), achieving near-100% success rates against safety filters in seconds. This makes signature-based detection (13.2.1) necessary but insufficient -- behavioral detection (13.2.4) and session analysis (13.2.8) are critical complementary layers.
- **Multi-turn Crescendo Attacks** -- Attackers gradually shift model behavior across conversation turns, with no single turn being overtly malicious. Detection requires session-level trajectory analysis (13.2.8) using sliding window analysis over conversation embeddings or cumulative risk scoring.
- **Indirect Prompt Injection via Tool Use** -- As AI agents gain tool-calling capabilities, adversarial content embedded in retrieved documents, web pages, or API responses can hijack agent behavior. Detection must extend beyond user-submitted prompts to cover all inputs the model processes.
- **LLM-as-C2-Channel** -- No longer theoretical. LAMEHUG (identified by CERT-UA in July 2025) uses the Hugging Face API with Qwen 2.5-Coder-32B-Instruct to dynamically generate reconnaissance and data theft commands. The SesameOp backdoor (Microsoft DART, July 2025) abuses OpenAI's Assistant API feature as a covert C2 server, avoiding attacker-hosted infrastructure entirely. Akamai Hunt identified a separate malware strain sending Base64-encoded C2 commands to /v1/chat/completions endpoints. Splunk has published 14 specific detection rules for LAMEHUG indicators. Detection should monitor for unauthorized outbound calls to AI API endpoints, DNS queries to AI service domains from unexpected processes, and entropy anomalies in request/response payloads.
- **AI Agent Context Poisoning & Memory Manipulation** -- Added to MITRE ATLAS in October 2025 (collaboration with Zenity Labs), these techniques target agentic AI systems. Context poisoning manipulates an agent's LLM context to persistently influence responses or actions. Memory manipulation alters long-term memory to ensure malicious changes persist across sessions. As of February 2026, Microsoft documented real-world memory poisoning via "Summarize with AI" buttons, identifying over 50 unique injection prompts from 31 companies embedding hidden instructions into AI assistant memory via URL prompt parameters.
- **TokenBreak Attacks** -- A newer class of adversarial attack targeting the tokenization layer rather than the prompt itself, manipulating how input text is broken into tokens to bypass content classifiers. This technique exploits an often-overlooked component in NLP pipelines and requires detection at the pre-tokenization stage.
- **Policy Simulation Attacks** -- Trick the model into believing unsafe outputs are allowed under updated or alternate policies, using prompts that mimic policy file structures (XML, JSON, INI). Roleplay-variant prompt injections achieved the highest attack success rate (89.6%) in 2025 red-team evaluations by deflecting responsibility away from the model.
- **Autonomous Reasoning-Model Jailbreaks** -- Published in Nature Communications (2026), research by Hagendorff et al. demonstrates that large reasoning models (DeepSeek-R1, Gemini 2.5 Flash, Grok 3 Mini, Qwen3 235B) can autonomously plan and execute multi-turn jailbreaks against other models with a 97.14% success rate, requiring no human guidance beyond an initial system prompt. This creates an alignment regression feedback loop: as reasoning models improve, they become more effective at subverting alignment in target models, making automated detection (13.2.4, 13.2.8) essential.
- **Unicode and Encoding Evasion** -- An empirical study (arxiv:2504.11168, 2025) tested character injection attacks against major guardrail products (Azure Prompt Shield, Meta Prompt Guard, Nvidia NeMo Guard, ProtectAI). Emoji smuggling achieved 100% evasion success for both jailbreaks and prompt injections; bidirectional text reached 99.23%. These techniques are black-box, requiring no knowledge of the guardrail internals, and highlight the need for input normalization before classification.
- **Indirect Injection as the Dominant Enterprise Threat** -- In the February 2026 Claude Opus 4.6 system card, Anthropic dropped the direct prompt injection metric entirely, arguing that indirect injection is now the relevant enterprise threat. Published numbers: a constrained coding environment held at 0% attack success across 200 attempts, while a GUI-based agent with extended thinking reached 17.8% at 1 attempt, climbing to 78.6% at 200 attempts without safeguards and 57.1% with safeguards. The shift matters for detection program design -- investments in per-message direct-injection classifiers yield diminishing returns compared to content scanning on untrusted inputs (emails, documents, web pages, tool output) before they enter the model's context window.
- **MCP Channel Compromise** -- January 2026 disclosed three CVEs (CVE-2025-68143/68144/68145) in Anthropic's official MCP Git server: two path traversal and one argument injection flaws exploitable via prompt injection in README files and poisoned issue descriptions. Separately, a public GitHub issue was shown to hijack a coding agent, pull data from private repos via an over-privileged PAT, and exfiltrate it into a public pull request. A systematic analysis of 78 studies (Jan 2026) reported adaptive attack success rates exceeding 85% against Claude Code, GitHub Copilot, and Cursor. Detection for 13.2.6 and 13.2.10 should treat MCP tool invocations as first-class telemetry, not sidecar traffic.

### Detection Architecture

A layered detection architecture addresses the full spectrum of requirements:

1. **Layer 1 -- Signature-Based (13.2.1):** Pattern matching against known jailbreak databases (JailbreakBench, PromptInject). Fast, low false-positive, but easily evaded by novel attacks. Signature databases require continuous updates.
2. **Layer 2 -- Classifier-Based (13.2.4):** ML classifiers trained on adversarial input datasets detect prompt injection and jailbreaks that evade signatures. Azure Prompt Shields and similar products operate at this layer.
3. **Layer 3 -- Behavioral Anomaly (13.2.4, 13.2.9):** Statistical anomaly detection on session-level features (retry rates, token consumption patterns, input diversity scores, query timing). Catches automated attack tools and probing behavior.
4. **Layer 4 -- Session Trajectory (13.2.8):** Multi-turn conversation analysis using stateful architectures. DeepContext (Feb 2026) demonstrated that RNN-based hidden state propagation over turn embeddings achieves 0.84 F1 at sub-20ms latency, outperforming stateless approaches like Llama-Prompt-Guard-2 (0.67 F1). Detects crescendo attacks, progressive manipulation, and autonomous reasoning-model-driven jailbreaks.
5. **Layer 5 -- Cross-Session Correlation (13.2.6):** SIEM rules correlating events across users, IPs, and time windows to detect coordinated campaigns and model extraction attempts.

### SIEM Integration Patterns

For requirement 13.2.2, the most maintainable integration approach in 2025-2026 uses OpenTelemetry pipelines:

- **Collection:** OpenTelemetry SDK emits GenAI spans with security event attributes
- **Processing:** OpenTelemetry Collector or Vector/Fluentd normalizes and enriches events
- **Routing:** OTLP export to SIEM (Splunk, Elastic, Sentinel) alongside observability backends
- **Detection:** Custom SIEM rules augmented with AI-specific detection logic (13.2.6)

As of March 2026, OCSF v1.8.0 ships native AI support: the `ai_operation` profile with `ai_model` and `message_context` objects provides vendor-neutral schema coverage for AI workloads including token usage metrics and role-based interaction tracking. OCSF is now a Linux Foundation project targeted for international standard ratification by June 2026. CrowdStrike Falcon AIDR natively streams AI detection findings to Falcon Next-Gen SIEM; Splunk has published detection rules for LLM-abusing malware. Microsoft Incident Response published structured playbooks for prompt abuse detection and response in March 2026, emphasizing defense-in-depth combining monitoring, analysis, and documented response protocols. Organizations adopting OCSF-aligned pipelines can reduce custom parsing overhead significantly.

### MITRE ATLAS Coverage (Updated v5.1.0 November 2025 / v5.3.0 January 2026)

MITRE ATLAS v5.1.0 (November 2025) expanded the framework to 16 tactics, 84 techniques, 56 sub-techniques, 32 mitigations, and 42 real-world case studies. The January 2026 v5.3.0 release added three new case studies centered on Model Context Protocol (MCP) abuse: MCP server compromise, indirect prompt injection via MCP channels, and malicious AI agent deployment. The October 2025 expansion (in collaboration with Zenity Labs) introduced 14 agent-focused techniques. Key additions relevant to abuse detection:

- **AI Agent Context Poisoning (AML.T0057)** -- Manipulating context to persistently influence agent responses or actions
- **Memory Manipulation (AML.T0058)** -- Altering long-term LLM memory to persist malicious changes across sessions
- **Thread Injection** -- Introducing malicious instructions into a specific chat thread to change behavior for the duration of that conversation
- **Modify AI Agent Configuration** -- Persisting adversary-controlled settings in agent configuration stores so that malicious behavior survives restart
- **AI Service API (AML.T0096)** -- Using legitimate AI service APIs (OpenAI Assistant API, Hugging Face Inference API, MCP endpoints) as covert C2 channels, demonstrated in the wild by LAMEHUG and SesameOp

Detection rules (13.2.6) should map to ATLAS techniques, and analysts should use ATLAS indicators to draft SOC detection hypotheses that SIEM/XDR platforms can exercise; published mappings show roughly 70% of ATLAS mitigations line up with existing enterprise security controls, so much of the work is instrumenting and labeling rather than inventing from scratch. The SAFE-AI framework maps ATLAS threats to NIST SP 800-53 controls at each system element (Environment, AI Platform/Tools, AI Models, AI Data), providing a structured approach to control selection. NIST IR 8596 (Cybersecurity Framework Profile for AI, 2025) provides complementary guidance on managing cybersecurity challenges specific to AI systems.

### Benchmarks for Measuring Detection Effectiveness

Abuse detection claims should be backed by repeatable benchmark runs, not vendor marketing. Useful public evaluation suites as of April 2026:

- **AgentDojo (NeurIPS 2024, active through 2026)** -- 97 realistic agent tasks (email client, e-banking, travel booking) with 629 security test cases and attack/defense paradigms from the literature. Baseline GPT-4o achieves 69% benign utility, dropping to 45% under attack; targeted "Important message" attacks succeed 53.1% of the time on an undefended baseline. Prompt sandwiching improves utility (65.7%) but leaves attack success at 30.8%; tool filtering drops attack success to 7.5% at the cost of utility (53.3%). US and UK AISI used AgentDojo to document Claude 3.5 Sonnet (new) vulnerabilities. Use this suite to validate 13.2.6 detection rules and 13.2.8 session trajectory analysis.
- **JailbreakBench, PromptInject** -- Still the standard corpora for 13.2.1 signature-based testing; require refresh against BoN and Unicode-evasion variants to stay relevant.
- **Comparative Evaluation of LLM Security Detection Systems (arxiv:2506.19109, Jun 2025)** -- Published FPR/recall numbers for Vigil, Rebuff, and LLM Guard; use as a baseline when procuring or swapping detection tooling.

Publish benchmark deltas alongside each SIEM rule rollout so detection coverage is measurable rather than asserted.

## Open Research Questions

- How effective are embedding-based similarity detectors at catching novel jailbreaks not in signature databases? (Initial data from June 2025 benchmarks suggests vector database scanners degrade 10-40% against obfuscated attacks.)
- What is the optimal window size for multi-turn trajectory analysis (13.2.8) that balances detection latency with false positive rates?
- Can LLM-as-judge approaches reliably detect adversarial conversations without introducing unacceptable latency? RLM-JB research shows GPT-5.2 achieving 98% recall with zero false positives, but production latency characteristics remain understudied.
- What entropy thresholds reliably distinguish C2 traffic from legitimate Base64 usage (e.g., image uploads) in LLM API traffic? The LAMEHUG and SesameOp incidents in 2025 provide real-world data points that should inform threshold calibration.
- How should detection rules be shared across organizations without revealing proprietary model vulnerabilities?
- How do AI firewall products (Azure Prompt Shields, Cloudflare, Akamai, CrowdStrike Falcon AIDR) compare in detection rates for novel vs. known jailbreak techniques, and what are the latency/accuracy tradeoffs of edge-layer vs. application-layer vs. SDK-based detection?
- What is the false positive rate of behavioral anomaly detection systems in high-volume production environments, and how should thresholds adapt to different user populations?
- How should organizations detect and prevent AI memory poisoning attacks at scale, given that Microsoft identified 50+ unique injection prompts from 31 companies as of February 2026?
- With OCSF v1.8.0 now providing a native `ai_operation` profile, how complete is its coverage for adversarial AI security events (jailbreaks, prompt injection, model extraction) vs. operational AI telemetry? Are extensions needed for attack-specific event classes?
- How should detection architectures adapt to autonomous reasoning-model jailbreaks (97.14% success rate), where the attacker is itself an LLM capable of real-time strategy adaptation?
- Can DeepContext-style stateful RNN detectors (0.84 F1) be combined with per-turn classifiers to achieve higher recall without unacceptable false positive rates in production?
- What input normalization strategies effectively counter Unicode/encoding evasion (emoji smuggling, bidirectional text) without degrading legitimate multilingual input handling?
- Now that Anthropic has dropped its direct prompt injection metric in favor of indirect injection (Feb 2026 system card), how should SOC detection programs rebalance investment between per-message classifiers and upstream content-scanning on untrusted inputs (emails, retrieved documents, tool outputs)?
- With the Jan 2026 MCP Git server CVEs (CVE-2025-68143/68144/68145) and the 78-study systematic finding of >85% adaptive attack success on Claude Code, Copilot, and Cursor, what is the minimum viable telemetry schema for MCP tool invocations that SIEM rules (13.2.6) can act on across vendors?
- How should benchmark results (AgentDojo utility/attack-success trade-offs) be surfaced to SOC leadership as ongoing KPIs for 13.2.4 and 13.2.8 coverage rather than one-time procurement data points?
- As the Check Point + Lakera, Meta LlamaFirewall, Portkey, and CrowdStrike Falcon AIDR ecosystems converge on similar claims (98%+ detection, sub-50ms latency), what independent evaluation methodology can buyers use to distinguish marketing from measurable detection coverage?
