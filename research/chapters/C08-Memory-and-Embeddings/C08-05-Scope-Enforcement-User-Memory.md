# C8.5: Scope Enforcement for User-Specific Memory

> [Back to C08 Index](C08-Memory-and-Embeddings.md)
> **Last Researched:** 2026-06-09

## Purpose

Prevent cross-tenant and cross-user leakage in retrieval, memory lookup, semantic caching, and prompt assembly. In multi-tenant RAG systems, similarity search is not an authorization decision: a chunk can be the nearest neighbor and still be illegal for the caller to see. Scope therefore has to be resolved from the authenticated session, pushed into the vector or search engine query, verified again after retrieval, and carried through any response cache or agent memory layer.

As of May 2026, the defensive baseline is better documented than it was a year ago, but the gap between documentation and field deployment is also wider. OWASP LLM08:2025 explicitly calls for permission-aware vector stores, strict logical partitioning, and immutable retrieval logs. Pinecone has shifted its multi-tenancy guidance toward namespace-per-tenant as the recommended pattern, made namespace creation generally available on March 26, 2026, brought CMEK and audit logs to GA in the same release, opened a HIPAA add-on for the Standard plan in February 2026, and documents BYOC as a public-preview zero-access data-plane across AWS, GCP, and Azure rather than a general-availability replacement for tenant authorization. Weaviate stores each tenant on a separate shard once multi-tenancy is enabled and supports tenant offloading to object storage. Qdrant recommends a single collection per embedding model with indexed payload partitioning for most large deployments, while noting that separate collections may be appropriate when the tenant count is small and isolation is more important than operational overhead. The HONEYBEE work from Georgia Tech, Duke, VMware Research, and Microsoft Research (SIGMOD 2026) reframes RBAC-aware vector partitioning as a joint authorization and performance problem, reporting up to 13.5x latency reduction versus row-level security at only 1.24x memory overhead and matching dedicated per-role indexes with about 90 percent less memory.

The threat model now clearly extends beyond the vector database. EchoLeak (CVE-2025-32711, disclosed June 2025 with CVSS 9.3) showed a zero-click prompt-injection chain inside Microsoft 365 Copilot where a crafted email sat in the inbox, slipped past XPIA classifiers, and used Copilot's normal RAG retrieval to exfiltrate emails, OneDrive files, SharePoint content, and Teams chats. The March 2026 CERT/CC CrewAI advisory (VU#221883, CVE-2026-2275, CVE-2026-2285, CVE-2026-2286, CVE-2026-2287) showed how direct or indirect prompt injection against an agent with code or RAG tools can chain into local file read, SSRF, and code execution. Microsoft's May 2026 disclosure on Semantic Kernel (CVE-2026-26030 and CVE-2026-25592, fixed in Python 1.39.4 and .NET 1.71.0) shows the same shape inside an In-Memory Vector Store filter: an unsafe `lambda` string built from search parameters allowed code injection through retrieval calls, and an early AST blocklist had bypasses. The August 2024 Slack AI prompt-injection case remains the canonical collaboration-layer pattern: retrieved content from one scope influenced answers shown in another scope when the application treated indexed workspace content as uniformly trusted. Scope enforcement for user memory therefore needs to include tools, caches, logs, internal admin paths, and agent-to-agent memory sharing, not only the happy-path vector query.

As of June 2026, the memory side of this control has its own named risk class and its first dedicated tooling. The OWASP Top 10 for Agentic Applications 2026 codifies memory and context poisoning as ASI06, and OWASP's May 13, 2026 essay on the topic (by ASI core-team co-lead Idan Habler) makes the C8.5-relevant point directly: once attacker content reaches a trusted surface like memory, hooks, or configuration, it shapes reasoning across future sessions, not just one response. The essay's headline example, Cisco's MemoryTrap research on Claude Code, showed routine repository cloning and dependency approval planting persistent prompt injection in user memory and global hooks; Anthropic's remediation in Claude Code v2.1.50 removed user memories from system prompts, fixing one path but not the architectural pattern. On the tooling side, OWASP launched Agent Memory Guard (announced June 1, 2026) as the reference implementation for ASI06 — a middleware layer that screens every memory read and write through a detector pipeline and declarative YAML policy, tags writes with source-channel provenance, keeps SHA-256 integrity baselines, and supports snapshot and rollback — and Microsoft open-sourced the Agent Governance Toolkit (April 2, 2026), which intercepts agent actions including memory operations with sub-millisecond policy enforcement mapped to all ten agentic risks. Both are young (Agent Memory Guard is an incubator project targeting v1.0 in Q4 2026), but they are the first off-the-shelf enforcement points for the memory-write gating this section has been recommending. On the platform side, HiddenLayer's ChromaToast disclosure (CVE-2026-45829, May 18, 2026, CVSS 10.0) is a blunt reminder that scope enforcement collapses entirely when the vector store itself is compromised: an unauthenticated `create_collection` call in every ChromaDB Python FastAPI server since 1.0.0 executes attacker-supplied code before authentication fires, no patch exists, and roughly 73 percent of internet-accessible deployments were estimated exploitable.

As of February 2026, persistent memory writes need the same attention as retrieval reads. Microsoft Defender Security Research documented AI Recommendation Poisoning attempts where hidden "Summarize with AI" links and pre-filled `?q=` or `?prompt=` parameters tried to make assistants remember a site as a trusted source. Microsoft mapped the pattern to MITRE ATLAS AML.T0080 (AI Agent Context Poisoning: Memory) and AML.T0080.000, and reported more than 50 observed prompt-based attempts from 31 companies across 14 industries. Palo Alto Unit 42's October 2025 demonstration on Amazon Bedrock Agent showed that a single poisoned memory entry from a malicious webpage was enough for the agent to poison roughly 87 percent of downstream decisions within four hours, with the payload surviving session boundaries. Tenable's HackedGPT research and the Check Point ChatGPT DNS-channel disclosure (patched February 20, 2026) extended that pattern to ChatGPT memory exfiltration via covert side channels. For C8.5, that means user memory, citation preferences, source rankings, semantic caches, conversation summaries, and long-term recommendation state all need tenant, user, origin, and policy checks before anything is persisted, and that retrieved content should never be allowed to silently mutate that state.

| **8.5.1** | **Verify that** every retrieval operation enforces scope constraints (tenant/user/classification) in the vector engine query and verifies them again before prompt assembly (post-filter). | 1 | Cross-tenant data leakage where one missing metadata filter, unsafe retriever helper, or prompt-influenced query path lets semantically similar but unauthorized chunks reach the model. OWASP LLM08:2025 treats permission-aware vector stores, strict logical partitioning, and retrieval monitoring as baseline controls. The Slack AI prompt-injection case, the EchoLeak (CVE-2025-32711) zero-click chain inside Microsoft 365 Copilot, and the March 2026 CrewAI advisory all show how retrieved or tool-fed content can cross intended boundaries when indexed content is treated as trusted. | Trace a request end-to-end. Deliberately remove or tamper with the query filter, then confirm the post-filter blocks leakage and emits a security event with the policy version and denied candidate scope. | Keep pre-filter and post-filter policy definitions in one source of truth. Pinecone namespaces, Weaviate tenants, Qdrant payload partitioning, and Azure permission metadata are strong pre-filter foundations only when the application refuses raw unscoped search calls. Framework wrappers such as LangChain and LlamaIndex should expose tenant-aware retrievers rather than generic `similarity_search` entry points, and any custom filter language must avoid `eval`-style interpreters of caller input. Post-filtering should be treated as a hard fail-closed gate, not a best-effort sanitizer after the model has already seen context. |
| **8.5.2** | **Verify that** vector identifiers, namespaces, and metadata indexing prevent cross-scope collisions and enforce uniqueness per tenant. | 1 | Vector ID collision, namespace traversal, and control-plane enumeration. If vector IDs are globally predictable or not tenant-scoped, an attacker may overwrite another tenant's vectors, infer document existence through fetch/list behavior, or poison a shared semantic cache. If namespace and collection controls are broader than the data-plane query path, a service account for Tenant A may still list, describe, snapshot, export, or delete Tenant B resources. The February 2026 AnythingLLM/Pinecone incident, where an unauthenticated endpoint exposed Pinecone API keys with full read/write/delete on enterprise embeddings, is a reminder that key scope and control-plane scope matter as much as namespace design. ChromaToast (CVE-2026-45829, disclosed May 18, 2026 by HiddenLayer, CVSS 10.0) is the extreme case: a pre-authentication code-execution flaw in ChromaDB's `create_collection` handler that runs attacker-supplied HuggingFace model code with `trust_remote_code` before any auth check, voiding every namespace and metadata boundary in the deployment at once. | Review ID and namespace design. Attempt cross-tenant upsert, update, delete, fetch, and list operations using another tenant's vector ID, namespace, collection, shard, payload key, cache key, and backup identifier. Confirm IDs are unique within each tenant boundary and that cross-scope collisions are either impossible by construction or rejected by server-side authorization. Probe control-plane APIs separately from query APIs: index listing, namespace stats, tenant management, backup/offload, snapshot, import, and export need the same scope model. Confirm CMEK or BYOC configurations do not leak key references across tenants and that audit logs (Pinecone audit logs, Weaviate role-based logs, Qdrant access logs, or Azure AI Search activity logs) capture cross-tenant attempts with caller, tenant, namespace, vector ID, operation, and outcome. | Pinecone serverless namespaces isolate records and require every data-plane operation to target one namespace; namespace creation API went GA in March 2026 alongside CMEK and audit logs, and BYOC remains public preview across AWS, GCP, and Azure but does not remove the need for tenant authorization. Weaviate tenants live on separate shards, but auto-tenant creation can turn typos into unintended tenants and offloaded tenants need cloud-storage bucket policy review. Qdrant recommends payload-based multitenancy for scale and separate collections for smaller high-isolation deployments. For role-heavy enterprise search, HONEYBEE's SIGMOD 2026 work is a useful design reference because it treats RBAC-aware vector partitioning as a performance and recall problem, not only an authorization wrapper. |
| **8.5.3** | **Verify that** retrieval results that match similarity criteria but fail scope checks are discarded. | 1 | Scope bypass through similarity, cache bleed, side-channel leakage, and persistent memory poisoning. In a shared vector space, a nearest neighbor can belong to another tenant, another user, a revoked document, or a higher classification. If that candidate reaches the prompt, response cache, trace, or agent memory summary, the leak has already happened. Embedding caches and semantic response caches create a parallel path: two tenants can ask semantically similar questions and receive cached context or answers unless tenant and user identity are part of the cache key. Memory-write paths add another failure mode: Palo Alto Unit 42's October 2025 work on Amazon Bedrock Agent showed a single poisoned memory entry could influence around 87 percent of downstream decisions within four hours, and Microsoft's February 2026 AI Recommendation Poisoning report (mapped to MITRE ATLAS AML.T0080 and AML.T0080.000) documented 31 companies using "Summarize with AI" parameters to try to mark themselves as trusted sources. | Execute tenant canary tests. Seed each tenant with distinctive sensitive documents, query from another tenant with semantically close prompts, and inspect raw candidates, post-filtered candidates, assembled prompt context, response cache entries, traces, saved memories, and final output. Verify every out-of-scope match is dropped before prompt construction and that discarded candidate IDs, scopes, and policy versions are logged without storing document bodies in general-purpose logs. Test revoked documents, users who recently changed teams, and poisoned retrieved documents that attempt to create or update long-term memory. Replay the EchoLeak (CVE-2025-32711) email pattern and the "Summarize with AI" link pattern against your own retriever to confirm classifiers, markdown link sanitization, and memory-write gates fail closed. | Dropping unauthorized matches must not create a visible oracle. Avoid returning exact denied counts, cross-tenant similarity scores, or latency patterns that reveal another tenant's corpus shape. A common production pattern is over-fetching within the authorized partition, then returning the top-k authorized results while logging denied candidates privately. Response caches, semantic caches, conversation summaries, and agent memory stores should key on tenant, user, role, data classification, policy version, embedding model, and retrieval corpus version, not only prompt text or embedding similarity. Treat memory writes as privileged side effects: retrieved text should not be able to create durable user preferences, trusted-source rankings, or citation instructions without explicit policy approval and user-visible provenance. Mem0, Letta, and similar memory layers ship policies and review hooks for exactly this — the burden is on the caller to actually enable them. OWASP Agent Memory Guard (June 2026, incubator) now offers a dedicated enforcement point: drop-in middleware for LangChain, LlamaIndex, and CrewAI that screens memory reads and writes against a YAML policy, tags source-channel provenance, and supports snapshot/rollback — early-stage, but the first OWASP-blessed implementation of exactly this gate. |
| **8.5.4** | **Verify that** multi-tenant tests simulate adversarial retrieval attempts (prompt-based and query-based) and show that no out-of-scope documents appear in prompts or outputs. | 2 | Regression in tenant isolation after framework upgrades, connector changes, index rebuilds, cache tuning, model changes, memory feature changes, or new agent tools. Prompt-based attacks try to convince the retriever or agent to widen scope. Query-based attacks tamper with metadata filter JSON, type handling, boolean logic, namespace names, or collection IDs (CVE-2026-26030 is a real example of a filter language that compiled attacker-supplied strings into Python `lambda`s). Poisoned-document attacks plant indirect instructions in indexed content so the retrieved chunk tries to weaken scope checks later, in the pattern Unit 42 demonstrated on Bedrock Agent and the MemoryGraft research on poisoned experience retrieval. The May 2026 MemMorph paper sharpens this threat for tool-enabled agents: three crafted records disguised as technical facts or operational policies in long-term memory hijacked the agent's tool selection with up to an 85.9 percent success rate across 10 agent backbones and 3 memory-module implementations, without touching any tool metadata that an audit would inspect; companion work on "sleeper" memory poisoning shows planted memories can stay dormant and trigger sessions later. Context-window overflow can push policy or source constraints out of the final prompt, and agent-tool vulnerabilities (CrewAI VU#221883, Semantic Kernel CVE-2026-26030/25592) can turn a retrieval failure into SSRF, file read, code execution, or persistent memory poisoning. | Review the adversarial test suite for direct cross-tenant similarity queries, wrong-tenant vector IDs, metadata filter injection, namespace traversal, control-plane enumeration, poisoned embedded chunks, indirect prompt injection, context-window overflow, cache-key collision, revoked-access replay, multi-agent memory sharing, and memory-poisoning links that use `?q=` or `?prompt=` parameters with "remember", "trusted source", "authoritative", "future", or "citation" language. Use canary documents with unique phrases in every tenant and assert they never appear in another tenant's raw prompt, model output, trace, cache, saved memory, or memory summary. Run these tests in CI/CD and after data migrations, vector DB upgrades, retrieval-library upgrades, cache configuration changes, and memory-store schema changes. Map findings to MITRE ATLAS (AML.T0070 RAG Poisoning, AML.T0071 False RAG Entry Injection, AML.T0080 Memory Poisoning), OWASP LLM08:2025, and OWASP Agentic ASI06 (Memory & Context Poisoning) so coverage gaps are visible at review time. Include MemMorph-style cases: poisoned records that masquerade as legitimate operational policies and bias tool selection rather than directly exfiltrating data. | This requirement is verification-focused. Promptfoo's RAG red-team guidance covers prompt injection, context injection, RAG poisoning, data exfiltration, source-attribution fabrication, and context-window overflow, and its `rbac`, `bola`, `bfla`, `pii`, and `policy` plugins map well to multi-tenant RAG. Microsoft PyRIT and NVIDIA Garak remain useful for broader generative-system testing, but multi-tenant RAG still needs custom harnesses that can inspect raw candidates, prompt context, and saved memory state, not only final answers. Property-based tests with Hypothesis-style generation are valuable for filter tampering because the dangerous cases often come from type confusion and unusual combinations rather than natural-language prompts. Newer entrants such as Pipelock (agent egress and DLP firewall for MCP) and SentinelOne's 2026 memory integrity verification (reported to cut mean time to detect persistent memory poisoning from about 72 hours to under 15 minutes) are early in maturity but worth tracking for coverage of memory-side regressions. OWASP Agent Memory Guard's detector pipeline (injection attempts, leakage, unauthorized key modification, anomalous write size and rate) doubles as a test oracle for memory-side assertions, and Microsoft's Agent Governance Toolkit (April 2026) adds a runtime policy interception layer with cross-model majority voting against memory poisoning — both too new for track records, but they give CI suites concrete enforcement points to test against rather than bespoke asserts alone. |

## Notable Incidents

- **EchoLeak / CVE-2025-32711 (June 2025).** Aim Labs disclosed a zero-click prompt-injection chain in Microsoft 365 Copilot. A crafted external email reached the user's inbox, bypassed Microsoft's XPIA classifier, exploited reference-style Markdown and auto-fetched images, and abused a Teams proxy permitted by the content security policy. Copilot's own RAG retrieval then pulled the email into context and exfiltrated chats, OneDrive files, SharePoint content, and Teams messages. CVSS 9.3. Patched in the June 2025 Patch Tuesday. Reinforces the lesson that retrieval boundaries must be enforced even when the attacker only controls indexable content. [HackTheBox writeup](https://www.hackthebox.com/blog/cve-2025-32711-echoleak-copilot-vulnerability) | [arXiv preprint](https://arxiv.org/abs/2509.10540)
- **Slack AI indirect prompt injection (August 2024).** PromptArmor showed that a public-channel message could coerce Slack AI into exfiltrating private-channel content within the same workspace. Slack patched the issue server-side. Useful baseline test case for any workspace-scoped retrieval system. [PromptArmor advisory](https://www.promptarmor.com/resources/data-exfiltration-from-slack-ai-via-indirect-prompt-injection)
- **CrewAI / VU#221883 (March 2026).** CERT/CC documented CVE-2026-2275, CVE-2026-2285, CVE-2026-2286, and CVE-2026-2287: prompt-injection paths into a tool-enabled CrewAI agent chained into SSRF, local file read, and code execution. Shows how a memory-scope or retrieval-scope failure becomes a full system compromise once tools are in scope. [CERT/CC VU#221883](https://kb.cert.org/vuls/id/221883)
- **Microsoft Semantic Kernel / CVE-2026-26030 and CVE-2026-25592 (May 2026).** Microsoft Security disclosed that the In-Memory Vector Store filter compiled attacker-influenced strings into Python `lambda`s; an early AST allowlist had bypasses. CVE-2026-25592 covered an arbitrary file write through the SessionsPythonPlugin. Fixed in Semantic Kernel Python 1.39.4 and .NET 1.71.0. Direct example of CVE-class filter-injection inside a retrieval engine. Mapped to MITRE ATLAS AML.T0080 and AML.T0080.000. [Microsoft Security Blog](https://www.microsoft.com/en-us/security/blog/2026/02/10/ai-recommendation-poisoning/)
- **Unit 42 persistent memory poisoning (October 2025).** Palo Alto Networks demonstrated that a single poisoned memory entry from a malicious webpage on Amazon Bedrock Agent corrupted approximately 87 percent of downstream decisions within four hours, with the payload surviving session boundaries. [Unit 42 research](https://unit42.paloaltonetworks.com/indirect-prompt-injection-poisons-ai-longterm-memory/)
- **AnythingLLM/Pinecone API key exposure (February 2026).** An unauthenticated endpoint in AnythingLLM exposed Pinecone API keys, providing full read/write/delete access to enterprise embeddings. A reminder that key scope is part of the multi-tenancy story. [Blockchain Council overview](https://www.blockchain-council.org/ai/securing-and-governing-vector-databases-privacy-prompt-injection-multi-tenant-access-control/)
- **ChromaToast / CVE-2026-45829 (May 2026).** HiddenLayer disclosed a CVSS 10.0 pre-authentication remote code execution flaw affecting every ChromaDB Python FastAPI server since 1.0.0: the `create_collection` handler accepts a client-supplied embedding model identifier and executes code from external HuggingFace repositories with `trust_remote_code` enabled before authentication runs. Anthropic's fix in Claude Code v2.1.50 removed user memories from system prompts. ## Implementation Maturity

- **Vector database multitenancy primitives.** Mature. Pinecone namespaces (GA), Weaviate tenants, Qdrant payload partitioning, and Azure AI Search Entra-based document-level ACLs all ship with documented patterns. Most production gaps are in how applications use these primitives, not in the primitives themselves.
- **Encryption and audit controls on managed vector stores.** Maturing. Pinecone reached CMEK GA and audit logs GA on March 26, 2026, with a HIPAA add-on for Standard plans in February 2026 and BYOC in public preview across the three major clouds. Weaviate Cloud and Qdrant Cloud have parallel offerings. Default-off configurations remain common in field deployments.
- **Document-level access in enterprise retrievers.** Maturing. Azure AI Search supports security filters (GA), POSIX-like ACL / RBAC scopes (preview through the 2025-11-01-preview REST API), Microsoft Purview sensitivity labels (preview), and SharePoint ACL preservation (preview). Preview-only API surface limits adoption.
- **RBAC-aware vector partitioning research.** Emerging. HONEYBEE (SIGMOD 2026) and SD-RAG (2026) show practical RBAC-aware partitioning and selective disclosure but are not yet shipped as turnkey products.
- **Memory-write guardrails.** Emerging, with a reference implementation as of June 2026. OWASP Agent Memory Guard (incubator, v1.0 targeted for Q4 2026) gives ASI06 a concrete enforcement layer — detector pipeline, YAML policy, source-channel provenance, SHA-256 baselines, snapshot/rollback — as drop-in middleware for LangChain, LlamaIndex, and CrewAI. Microsoft's Agent Governance Toolkit (April 2026, MIT-licensed) intercepts agent actions including memory operations at sub-millisecond latency and maps to all ten OWASP agentic risks. SentinelOne memory integrity verification (2026), Mem0 policy hooks, Letta governance, and the open-source Pipelock agent firewall round out the field, but production teams still rarely gate retrieved-content-to-memory writes through formal policies, classifiers, and user-visible provenance.
- **Semantic cache scope enforcement.** Emerging. AWS Bedrock prompt caching, Azure OpenAI semantic cache, and third-party caches like GPTCache and Redis Vector Cache are widely used, but default configurations often omit tenant, user, classification, and policy-version in the cache key. Academic work (Semantic Chameleon, Semantic Cache Poisoning) shows realistic poisoning and bleed attacks.
- **Multi-tenant RAG red teaming.** Emerging. Promptfoo has the most coverage today through RAG-poisoning, RBAC, BOLA, BFLA, PII, and policy plugins. PyRIT and Garak help with broader generative testing, and Pipelock contributes egress-side guardrail tests. Multi-tenant memory-write testing still relies on bespoke harnesses.

## Implementation Notes

**Tenant context is not user input.** Build the retrieval scope from the authenticated session, source-system ACLs, policy engine, and data-classification labels. Do not let the user, the model, or a retrieved document provide `tenant_id`, namespace, collection, classification, or cache-key fields.

**Filter languages must not compile attacker input.** CVE-2026-26030 in Semantic Kernel's In-Memory Vector Store is a reminder that any filter language that compiles caller-influenced text into `lambda`, `eval`, or AST-evaluated expressions is one prompt away from code execution. Prefer typed filter primitives in the vector engine (Pinecone metadata expressions, Weaviate `where` clauses, Qdrant payload conditions, Azure AI Search OData) over Python-string filters built by the retriever.

**Treat caches and logs as retrieval systems.** A response cache, semantic cache, trace store, evaluation dataset, or conversation memory can reintroduce cross-tenant leakage even when the vector database is configured correctly. Every secondary store should carry the same tenant, user, classification, policy-version, and revocation semantics as the original retrieval path.

**Memory writes are privileged side effects.** Do not let retrieved text, external pages, emails, documents, or tool results write facts, preferences, trusted-source rankings, citation preferences, or long-term memories without a trusted caller, user-visible review, and scope labels. Log proposed memory writes with the origin URL or document, tenant, user, policy version, and approval result. Verify periodically that the memory store contains no instructions sourced from retrieved-only content (the Unit 42 and Microsoft Defender patterns above).

**Review the secondary stores.** In implementation reviews, confirm:

- Every memory store carries tenant, user, role, classification, and policy-version fields.
- Every cache key includes the same scope inputs as the retriever.
- Every memory-write API rejects instructions sourced from retrieved content or tool output by default.
- Admin APIs for list, export, snapshot, restore, BYOC peering, and CMEK key rotation enforce the same scope model as query APIs.
- Incident logs capture denied candidate IDs plus blocked memory-write attempts without copying sensitive document bodies into broad log sinks.

**Separate query authority from tool authority.** Agent frameworks often combine retrieval, browsing, file loading, and code execution in one runtime. The CrewAI advisory and the Semantic Kernel CVEs above are reminders that a prompt-injection path into a tool-enabled agent can turn a memory-scope bug into SSRF, local file read, or code execution. Scope checks should apply before tool selection and again before any retrieved result is passed to a tool.
