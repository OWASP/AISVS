# C10: Model Context Protocol (MCP) Security

> **Source:** [`1.0/en/0x10-C10-MCP-Security.md`](https://github.com/OWASP/AISVS/blob/main/1.0/en/0x10-C10-MCP-Security.md)
> **Requirements:** 28 | **Sections:** 6

## Control Objective

Ensure secure discovery, authentication, authorization, transport, and use of MCP-based tool and resource integrations to prevent context confusion, unauthorized tool invocation, or cross-tenant data exposure.

> **2025-2026 Highlights:** The MCP spec matured significantly with OAuth 2.1 authorization and streamable HTTP transport replacing legacy SSE. Tool poisoning and rug-pull attacks were publicly disclosed, driving requirements for component allowlisting (C10.1) and tool definition change detection (C10.4.5). April 2026 brought two more CVEs into the ecosystem — CVE-2026-39313 (mcp-framework DoS via unbounded `readRequestBody`, disclosed April 16) and CVE-2026-39884 (Flux159 mcp-server-kubernetes argument injection in `port_forward` via string-split kubectl construction, CVSS 8.3) — while CVE-2026-33032 ("MCPwn," CVSS 9.8) in Nginx UI's `/mcp_message` endpoint is now under active exploitation with ~2,600 exposed instances globally. The cumulative picture: 50+ CVEs tracked in the [Vulnerable MCP Project](https://vulnerablemcp.info/) (13 critical), a scan of 5,618 servers that found only 2.5% verified safe, and the first malicious MCP server (postmark-mcp) proving real-world supply chain compromise. Newer research catalogued elsewhere includes MCP config-file RCE in developer tools (CVE-2025-59536, CVSS 8.7 — malicious `.mcp.json` in repos triggers shell commands on tool init), SSRF bypasses (CVE-2026-31945, LibreChat), and KQL injection via MCP tool parameters (CVE-2026-33980, Azure Data Explorer MCP Server). On the defensive side, April 2026 saw Snyk Agent Scan v0.4.13, MCP Gateway v1.1.0 (proactive reputation analysis and tool-description scanning), Golf Scanner (20 automated checks across seven IDEs), MintMCP Gateway (first SOC 2 Type II certified MCP platform), IBM ContextForge (open-source multi-transport gateway), AWS IAM context keys for differentiating agent vs. human actions, and the publicly released DVMCP training environment (28 tools, 38 challenges). The OWASP Cheat Sheet Series maintains a dedicated MCP Security Cheat Sheet, the MSSS certification standard (v0.1.0, 24 controls) was published in January 2026, and DPoP/Workload Identity proposals (SEP-1932, SEP-1933) remain in "On the Horizon" status on the 2026 roadmap — community-formed working groups only, no core maintainer drive this cycle.

## Threat Landscape

Known attacks, real-world incidents, and threat vectors relevant to this chapter:

- **Tool poisoning at scale** — MCPTox benchmark (August 2025) tested 20 LLM agents across 45+ real-world MCP servers with 353 tools. o1-mini achieved a 72.8% attack success rate; Claude 3.7 Sonnet had the highest refusal rate but still under 3%. More capable models are often *more* susceptible because attacks exploit superior instruction-following. Between Jan-Feb 2026, 30+ CVEs were filed targeting MCP infrastructure — 43% were exec/shell injection.
- **Rug-pull attacks** — MCP servers change tool definitions after initial trust establishment. Users are never re-prompted for consent. Invariant Labs demonstrated SSH key exfiltration via Cursor IDE where the confirmation dialog hid the malicious payload entirely.
- **DNS rebinding** — CVE-2025-66414 (CVSS 8.1) affected MCP TypeScript SDK <1.24.0 with no DNS rebinding protection by default. Docker MCP Gateway and Microsoft Playwright MCP Server were also vulnerable. Remote attackers invoke any tool on localhost MCP servers from a malicious webpage.
- **OAuth consent bypass and token relay** — Obsidian Security (January 2026) found three attack vectors: shared client ID enables consent caching bypass, URL capture bypasses MCP-layer consent entirely, cookie injection forces state validation under attacker control. Square's MCP server exposed merchant data, transactions, and banking details. Token passthrough (now explicitly forbidden in spec) creates confused deputy vulnerabilities.
- **Cross-tool contamination** — Microsoft Developer Blog (April 2025) documented how compromised MCP servers influence legitimate tools through shared LLM context. Tool shadowing via crafted descriptions intercepts calls intended for legitimate servers.
- **MCP sampling exploitation** — Unit 42 (December 2025) demonstrated resource theft (hidden billing), conversation hijacking (persistent session compromise), and covert tool invocation through MCP's sampling capability.
- **Copilot RCE via MCP** — CVE-2025-53773 (CVSS 7.8) — prompt injection in source files manipulates GitHub Copilot into enabling "YOLO mode" (auto-approve all tools), achieving full RCE including MCP server injection. Wormable via infected repos.
- **Shadow MCP servers** — 41% of servers in the official MCP registry have zero authentication as of February 2026 (518 servers scanned). OWASP MCP Top 10 lists Shadow MCP Servers (MCP09) as a top risk.
- **First malicious MCP server in the wild** — The `postmark-mcp` npm package (September 2025) impersonated the legitimate Postmark email service across 15 versions to build trust, then introduced a backdoor in v1.0.16 that silently BCC'd all emails to an attacker-controlled address (`phan@giftshop[.]club`). Downloaded 1,643 times before removal. The attack demonstrated how MCP servers operate with broad permissions — exposed data included password resets, invoices, and internal communications.
- **Azure MCP SSRF** — CVE-2026-26118 (CVSS 8.8, patched March 10, 2026) — SSRF in Azure MCP Server Tools allowed attackers to redirect outbound requests to internal endpoints and exfiltrate the server's managed identity token. Successful exploitation granted access to entire ML pipelines including training data and model repositories. Root cause: improper input validation on tool parameters accepting user-provided URLs.
- **SDK cross-client data leak** — CVE-2026-25536 (CVSS 7.1, February 2026) affected TypeScript SDK v1.10.0–1.25.3. When a single `McpServer` instance with `StreamableHTTPServerTransport` served multiple clients, responses could route to wrong clients — a common production deployment pattern. Fixed in v1.26.0.
- **MCP Inspector RCE** — CVE-2026-23744 (CVSS 9.8, January 2026) — MCPJam Inspector versions through 1.4.2 listened on 0.0.0.0 by default instead of 127.0.0.1, allowing remote attackers to trigger MCP server installation via a crafted HTTP request. The most critical MCP CVE to date. Fixed in v1.4.3.
- **MCP Memory Service CORS bypass** — CVE-2026-33010 (CVSS 8.1, March 2026) — the `mcp-memory-service` package configured CORS with `allow_origins=['*']` and `allow_credentials=True` when HTTP mode was enabled with anonymous access. Any malicious website could silently read, modify, and delete all stored agent memories without authentication. Fixed in v10.25.1.
- **Git-MCP-Server command injection** — CVE-2026-4496 (CVSS 5.3, March 2026) — OS command injection in `child_process.exec` within `src/gitUtils.ts` affecting merge diff and file diff operations. Exploit publicly disclosed; vendor unresponsive as of disclosure date.
- **Claude Code MCP/hooks RCE** — CVE-2025-59536 (CVSS 8.7) and CVE-2026-21852 (CVSS 5.3) — Check Point Research demonstrated that malicious `.mcp.json` config files in untrusted repositories could execute arbitrary shell commands on tool initialization and exfiltrate Anthropic API keys before any user trust decision. MCP servers configured in repo-level config run automatically when Claude Code opens the directory. Patched by Anthropic prior to disclosure (February 2026).
- **LibreChat MCP SSRF bypass** — CVE-2026-31945 (CVSS 7.7, disclosed March 27, 2026) — LibreChat v0.8.2-rc2 through v0.8.2 hostname validation for agent actions and MCP did not verify whether DNS resolution pointed to private IP addresses. Attackers could access internal RAG APIs and cloud metadata endpoints. Fixed in v0.8.3-rc1.
- **Ecosystem-wide vulnerability survey** — As of March 2026, a scan of 5,618 MCP servers found only 2.5% (143) verified safe, with 90.2% needing security review. Across 2,614 implementations surveyed separately, 82% had file operations vulnerable to path traversal, 67% had code injection risks, and 36.7% had SSRF exposure. The [Vulnerable MCP Project](https://vulnerablemcp.info/) now tracks 50 CVEs (13 critical) across the MCP ecosystem.
- **Nginx UI "MCPwn" active exploitation** — CVE-2026-33032 (CVSS 9.8, disclosed March 24, 2026) exposed the `/mcp_message` endpoint in Nginx UI without authentication middleware. Any unauthenticated network attacker can open an SSE connection, reuse the returned session ID, and invoke every MCP tool — restart nginx, read/modify/delete configuration files, add malicious server blocks, trigger reloads. [Recorded Future's CVE Landscape](https://thehackernews.com/2026/04/critical-nginx-ui-vulnerability-cve.html) confirmed active exploitation in the wild; [Pluto Security](https://pluto.security/blog/mcp-bug-nginx-security-vulnerability-cvss-9-8/) identified ~2,600 publicly exposed instances concentrated in China, the US, Indonesia, Germany, and Hong Kong. Patched in Nginx UI 2.3.6. First case of confirmed mass exploitation of an MCP-layer flaw against production infrastructure.
- **mcp-framework HTTP DoS** — CVE-2026-39313 (disclosed April 16, 2026) in [mcp-framework](https://www.thehackerwire.com/vulnerability/CVE-2026-39313/) ≤0.2.21. The `readRequestBody()` helper in the HTTP transport concatenates body chunks without enforcing the configured `maxMessageSize`, so a single large POST to `/mcp` exhausts memory and crashes the server. Unauthenticated, trivially scriptable. Fixed in v0.2.22. Illustrates that size-limit configuration values are worthless without enforcement paths that actually reference them.
- **Kubernetes MCP argument injection** — CVE-2026-39884 (CVSS 8.3, April 2026) in Flux159 `mcp-server-kubernetes` ≤3.4.0. The `port_forward` tool concatenates user-controlled `namespace`, `resourceType`, `resourceName`, `localPort`, and `targetPort` fields into a kubectl command string, then splits on spaces before passing to `spawn()` — other tools in the same server use the safer `execFileSync` array form. Attackers inject additional kubectl flags (e.g., `--address=0.0.0.0`, repeated `-n` flags) to expose internal services, pivot across namespaces, or chain with prompt injection against a connected agent. Fixed in v3.5.0. Classic CWE-88 in an MCP wrapper.
- **Azure Data Explorer MCP KQL injection** — CVE-2026-33980 in `adx-mcp-server` let MCP clients inject arbitrary Kusto Query Language through multiple tool parameters, effectively bypassing any agent-layer policy enforcement. Data plane access through an untrusted agent channel — the kind of boundary failure the MSSS "Input Validation" domain is explicitly designed to catch.
- **Historical incident trail** — [Authzed's MCP breach timeline](https://authzed.com/blog/timeline-mcp-breaches) consolidates cases missed by the CVE system: WhatsApp MCP chat history exfiltration (April 2025), GitHub MCP overscoped PAT leakage to public repos (May 2025), Asana MCP cross-tenant visibility (June 2025), Anthropic Filesystem MCP sandbox escapes via symlinks (CVE-2025-53109 / CVE-2025-53110), Smithery MCP Registry path traversal exposing 3,000+ hosted servers (October 2025), and Figma/Framelink command injection (CVE-2025-53967). Five patterns repeat: localhost-as-trusted, over-privileged tokens, tool-poisoning supply chain, prompt injection as exfil vector, and concentration risk in hosted registries.

## Tooling & Implementation

Current tools, frameworks, and libraries that help implement these controls:

- **MCP SDKs:** [TypeScript SDK v1.27.1](https://github.com/modelcontextprotocol/typescript-sdk) (Zod v4 for schema validation, host header validation middleware, OAuth 2.1 auth helpers, 161 contributors), [Python SDK](https://github.com/modelcontextprotocol/python-sdk) (Pydantic-based validation, `mcp.server.auth` for OAuth 2.1 RS, supports stdio/SSE/Streamable HTTP), [C# SDK v1.0](https://www.infoq.com/news/2026/03/mcp-csharp-v1/) (Microsoft, March 2026 — full 2025-11-25 spec support, OAuth 2.1 discovery with three Protected Resource Metadata methods, incremental scope consent, URL mode elicitation, tool calling in sampling, Tasks support). V2 pre-alpha in development for TS and Python.
- **MCP security gateways:** [Lasso MCP Gateway v1.1.0](https://github.com/lasso-security/mcp-gateway/releases) (open-source, April 2026 — new proactive Security Scanner subsystem with Reputation Analysis, Tool Description Scanning, Automatic Blocking, and Configuration Updates layered on top of the original prompt-injection and Presidio DLP engines), [MintMCP Gateway](https://www.integrate.io/blog/best-mcp-gateways-and-ai-agent-security-tools/) (April 2026 — first SOC 2 Type II certified MCP platform, role-based endpoints, OAuth-protected access), [IBM ContextForge](https://www.integrate.io/blog/best-mcp-gateways-and-ai-agent-security-tools/) (open-source multi-transport gateway covering HTTP(S), WebSocket, SSE, and stdio — suitable for heterogeneous MCP estates), [TrueFoundry MCP Gateway](https://www.integrate.io/blog/best-mcp-gateways-and-ai-agent-security-tools/) (performance-oriented, 3–4ms latency at 350+ RPS), [Operant AI MCP Gateway](https://www.integrate.io/blog/best-mcp-gateways-and-ai-agent-security-tools/) (security-focused with Shadow Escape attack detection and AI-DR capabilities), [Traefik Hub MCP Gateway](https://www.integrate.io/blog/best-mcp-gateways-and-ai-agent-security-tools/) (triple-layer AI/MCP/API protection), [Enkrypt AI Secure MCP Gateway](https://www.enkryptai.com/mcp-scan) (commercial automated static analysis), [Proofpoint Secure Agent Gateway](https://www.proofpoint.com/us/blog/corporate-news/stop-ai-agents-losing-data) (Q1 2026 — MCP-native DLP for agentic workflows), [Salesforce Agentforce MCP Gateway](https://www.salesforce.com/agentforce/mcp-support/) (beta — trusted gateway with allowlisting and tool poisoning defense), and [Peta Agent Vault](https://www.integrate.io/blog/best-mcp-gateways-and-ai-agent-security-tools/) (zero-trust scoped, time-limited token issuance specifically for agents).
- **MCP security scanners:** [Snyk Agent Scan v0.4.13](https://github.com/snyk/agent-scan) (April 2026 — rebranded from MCP-Scan post-acquisition; auto-discovers MCP configs across Claude/Cursor/Windsurf/VS Code, 15+ security risks including agent skills), [Golf Scanner](https://github.com/golf-mcp/golf-scanner) (April 2026 — Go CLI that discovers MCP configs across seven popular IDEs and runs 20 automated security checks; aimed at security teams sweeping developer laptops), [mcp-sec-audit toolkit](https://arxiv.org/html/2603.21641v1) (combined static + dynamic analysis; reports 100% detection on the MCPTox benchmark for overly privileged MCP capabilities), [mcpscan.ai](https://mcpscan.ai/) (SaaS scanner — reports command injection in 23% of scanned servers, tool poisoning, rug pulls, confused deputy), [Cisco MCP Scanner](https://github.com/cisco-ai-defense/mcp-scanner) (open-source), [Proximity](https://github.com/fr0gger/nova-proximity) (NOVA rule engine for first-pass assessment), [Invariant mcp-scan](https://invariantlabs.ai/blog/introducing-mcp-scan) (dedicated MCP vulnerability scanning).
- **Training & red-team environments:** [Damn Vulnerable MCP Server (DVMCP)](https://github.com/Kyze-Labs/damn-vulnerable-MCP-Server) (April 2026 — Docker-ready deliberately vulnerable MCP server with 28 tools and 38 challenges across 19 vulnerability categories for hands-on training), [TIP Framework](https://arxiv.org/html/2603.24203v1) (tree-based adaptive search demonstrating 95% attack success against stealthy MCP injection payloads across LM Studio and VS Code + GPT-4o).
- **MCP Inspector:** [Official tool](https://github.com/modelcontextprotocol/inspector) (`npx @modelcontextprotocol/inspector`) — browser-based debugging revealing all tools/resources/prompts a server exposes. `--cli` flag for CI integration with exit codes.
- **Auth:** MCP Authorization Specification evolved through [March](https://modelcontextprotocol.io/specification/2025-03-26/basic/authorization/), [June](https://modelcontextprotocol.io/specification/2025-06-18/basic/authorization/), and [November 2025](https://modelcontextprotocol.io/specification/2025-11-25) versions. OAuth 2.1 + PKCE (S256) mandatory, RFC 8707 Resource Indicators required, step-up authorization, token passthrough explicitly forbidden. Active security enhancement proposals: [SEP-1932 (DPoP)](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/1932) for sender-constrained access tokens preventing token replay, and SEP-1933 (Workload Identity Federation) for machine-to-machine identity attestation via Kubernetes ServiceAccounts, cloud instance identity, and GitHub OIDC.
- **Vulnerability tracking:** [Vulnerable MCP Project](https://vulnerablemcp.info/) — comprehensive CVE database tracking 50 MCP vulnerabilities (13 critical) with taxonomy across input validation, prompt injection, network security, integrity, and trust model categories. 32 contributing researchers.
- **Secret scanning:** [GitHub MCP Server secret scanning](https://github.blog/changelog/2026-03-17-secret-scanning-in-ai-coding-agents-via-the-github-mcp-server/) (March 2026, public preview) — detects exposed credentials in code changes before commit via MCP-compatible IDEs and AI coding agents. Requires GitHub Secret Protection enabled.
- **Cloud provider primitives:** [AWS IAM context keys for managed MCP servers](https://aws.amazon.com/blogs/security/understanding-iam-for-managed-aws-mcp-servers/) (April 2026) — standardized IAM context keys that let administrators differentiate AI-agent-initiated actions from human-initiated actions in policy evaluation across AWS-managed MCP deployments. First major cloud provider to ship native MCP-aware IAM primitives, addressing OWASP MCP02 (Scope Creep) at the cloud control plane. Complements Azure's earlier managed-identity posture fixes for its own MCP server (see CVE-2026-26118).
- **Security standards:** [MCP Server Security Standard (MSSS)](https://github.com/mcp-security-standard/mcp-server-security-standard) v0.1.0 (January 2026, Community Review Draft) — 24 controls across 8 domains (Filesystem, Execution, Network, Authorization, Input Validation, Logging, Supply Chain, Deployment), 4 compliance levels (L1-Essential through L4-Maximum Assurance) inspired by NIST CSF, OWASP ASVS, and CIS Controls. v1.0 targeted for Q4 2026.
- **Registries:** Official MCP registry (518 servers, 41% with zero auth). [Kong MCP Registry](https://konghq.com/) (enterprise, AI Alliance compliant). Gap: no standardized allowlisting mechanism in the protocol itself.
- **Validation:** Zod v4 (TS) / Pydantic (Python) for strict tool argument validation. JSON-RPC 2.0 with discriminated unions for automatic routing. [OWASP MCP Cheat Sheet](https://genai.owasp.org/resource/cheatsheet-a-practical-guide-for-securely-using-third-party-mcp-servers-1-0/) provides implementation guidance.

## Open Research Questions

- [x] **How should MCP servers handle token exchange?** — The November 2025 spec explicitly forbids token passthrough and mandates separate tokens for downstream APIs. On-behalf-of (OBO) or client credentials flows are the documented patterns. But Obsidian Security's Square analysis shows real implementations still get this wrong.
- [ ] **How do you prevent tool description manipulation when descriptions are LLM-consumed?** — MCPTox showed 72.8% attack success even on capable models. The fundamental tension: tool descriptions must be informative enough for the LLM to use tools correctly, but this makes them a first-class prompt injection surface. Lasso Gateway scans descriptions, but no solution is comprehensive.
- [ ] **Can rug-pull attacks be reliably detected?** — Tool definitions changing between `tools/list` calls exploit cached trust. Invariant Labs' `mcp-scan` provides detection, but real-time monitoring of definition changes during active sessions has no standard solution.
- [ ] **Will MCP registry security improve?** — 41% of official registry servers have zero auth. A March 2026 scan of 5,618 servers found only 2.5% verified safe. The postmark-mcp incident proved supply chain compromise is practical, not theoretical. The Vulnerable MCP Project tracks 50 CVEs. Kong MCP Registry adds enterprise governance, but the open ecosystem still lacks signing, reputation scoring, or mandatory security baselines.
- [ ] **How should the auth spec evolve for regulated environments?** — AAIF's financial sector emphasis suggests compliance-specific extensions are coming. The 2026 roadmap shifted to Working Group-driven priorities (transport evolution, agent communication, governance maturation, enterprise readiness) rather than release milestones. Two active proposals address key gaps: SEP-1932 (DPoP) adds sender-constrained access tokens to prevent token replay, and SEP-1933 (Workload Identity Federation) enables machine-to-machine identity via Kubernetes ServiceAccounts, cloud instance identity, and GitHub OIDC. Token Exchange (RFC 8693) is now recommended for Streamable HTTP deployments needing downstream OAuth tokens.
- [ ] **Is MCP's security model adequate for multi-tenant deployments?** — Cross-tool contamination through shared LLM context, namespace collisions, and tool shadowing all exploit multi-server scenarios. CVE-2026-25536 demonstrated that even the SDK itself can misroute responses between clients. The spec doesn't address namespace isolation or tool provenance verification at the protocol level.
- [ ] **How should MCP-enabled developer tools handle untrusted project configurations?** — CVE-2025-59536 (CVSS 8.7) showed that `.mcp.json` files in cloned repositories can trigger shell commands and exfiltrate API keys before any user trust decision. This is a new attack surface: MCP config-as-code in untrusted repos. The MSSS standard (v0.1.0) includes execution controls, but no consensus exists on safe defaults for repo-level MCP configuration discovery.
- [ ] **How should the ecosystem track and respond to MCP-specific CVEs?** — The Vulnerable MCP Project (50 CVEs, 13 critical as of March 2026) provides community tracking, but there's no coordinated disclosure process specific to MCP. The 30+ CVEs filed in 60 days (Jan-Feb 2026) overwhelmed existing response mechanisms. GitHub's MCP Server secret scanning (March 2026) is a step forward for credential leaks, but broader vulnerability management across 5,600+ servers remains unsolved.
- [ ] **How should operators respond now that MCP-layer flaws are under mass exploitation?** — CVE-2026-33032 (Nginx UI "MCPwn") confirmed by Recorded Future as actively exploited in April 2026 with ~2,600 exposed instances is the first documented case of an MCP-specific vulnerability being weaponized at scale against production infrastructure. Prior MCP CVEs were largely proof-of-concept. The implication: MCP endpoints must now be treated as internet-facing administrative surfaces with the same hardening rigor as Kubernetes API servers or CI/CD control planes. No standard patch SLA, exposure discovery, or emergency-response playbook exists specifically for MCP yet.
- [ ] **Are schema-strict wrapper tools enough when the wrapped CLI is the attack surface?** — CVE-2026-39884 (mcp-server-kubernetes `port_forward`) and CVE-2026-33980 (Azure Data Explorer MCP KQL injection) both passed schema validation at the MCP boundary and then reintroduced classical injection flaws below it: kubectl argv construction via `String.split()`, or direct KQL concatenation. The pattern generalizes — every MCP tool that ultimately shells out to a CLI, SQL engine, graph query language, or interpreter inherits that target's injection taxonomy. The community currently has no catalog of "MCP wrappers to CLI/DSL targets" with per-target hardening guidance. MSSS "Input Validation" and "Execution" domains imply this, but don't specify.
