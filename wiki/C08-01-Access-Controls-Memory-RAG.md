# C8.1: Access Controls on Memory & RAG Indices

> [Back to C08 Index](C08-Memory-and-Embeddings.md)
> **Last Researched:** 2026-03-29

## Purpose

Enforce fine-grained access controls and query-time scope enforcement for every vector collection. Without proper access controls on vector stores, similarity queries can cross tenant boundaries, unauthorized users can retrieve or modify embeddings, and poisoned documents cannot be traced back to their source. As of early 2026, major managed vector databases (Pinecone, Weaviate, Qdrant) now offer RBAC, JWT authentication, namespaces, and compliance certifications (SOC 2, HIPAA, ISO 27001), but anomaly detection for retrieval poisoning remains an emerging area with limited native tooling.

| **8.1.1** | **Verify that** vector insert, update, delete, and query operations are enforced with namespace/collection/document-tag scope controls (e.g., tenant ID, user ID, data classification labels) with default-deny. | 1 | Cross-tenant data leakage via unrestricted vector queries; unauthorized document retrieval where Tenant A's similarity search returns Tenant B's chunks. Directly maps to OWASP LLM08:2025. OWASP formally recognizes that inadequate access controls on vector stores can lead to unauthorized retrieval of sensitive embeddings. | Review vector DB configuration for namespace isolation. Attempt cross-namespace queries with test credentials. Confirm default-deny by issuing queries without explicit scope and verifying rejection. Inspect middleware/SDK wrappers that append scope filters. | Pinecone supports namespaces natively and added per-index RBAC with project-scoped API keys; Weaviate has multi-tenancy since v1.20 with tenant-aware classes and lifecycle endpoints; Qdrant now offers first-class multitenancy with named collections/tenants, quota controls, sharding, and tenant-level lifecycle APIs (enterprise tier). Milvus supports partition keys and RBAC (note: Milvus fixed an ETCD prefix matching bug that could cause data leakage across RBAC boundaries). HoneyBee (2025) demonstrated that dynamic RBAC partitioning for vector databases achieves up to 13.5x lower query latency than row-level security with only 1.24x memory overhead, making per-role access control practical at scale. Many teams still rely on application-layer enforcement only, which is fragile. In BYOC mode (GA 2024), Pinecone clusters run inside the customer's own AWS/Azure/GCP account for hard isolation. |
| **8.1.2** | **Verify that** API credentials used for vector operations carry **scoped claims** (e.g., permitted collections, allowed verbs, tenant binding). | 1 | Credential misuse or lateral movement — a compromised API key with full read/write across all collections enables mass data exfiltration or poisoning of the entire knowledge base. PoisonedRAG (Zou et al., USENIX Security 2025) showed that just 5 carefully crafted documents can manipulate AI responses 90% of the time, making write-scoped credential control critical. | Inspect API key/token scopes in the vector DB management console. Verify JWT claims or API key metadata restrict operations to specific collections and verbs. Test that a scoped key cannot perform out-of-scope operations. | Pinecone supports project-scoped API keys and per-index RBAC. Qdrant now offers JWT-based authentication with granular claim scoping and read-only API keys. Weaviate Cloud supports ACLs per tenant-aware class. Milvus added RBAC with partition keys in v2.3+. Self-hosted Chroma still often relies on a single master key — compensating controls at the API gateway or service mesh layer are needed. Compliance certifications: Pinecone (SOC 2, HIPAA, ISO 27001, GDPR), Weaviate Enterprise Cloud (HIPAA on AWS since 2025, SOC 2 Type II), Qdrant Cloud (SOC 2 Type II, HIPAA-ready for enterprise). |
| **8.1.3** | **Verify that** cross-scope access attempts (e.g., cross-tenant similarity queries, namespace traversal, tag bypass) are detected and rejected. | 2 | Namespace traversal attacks where an attacker crafts queries or metadata filters to escape their tenant boundary. Similar to SQL injection but targeting vector query DSLs and metadata filter syntax. Christian Schneider's 2025 research on "RAG security: the forgotten attack surface" highlights metadata filter injection as an underappreciated vector for cross-tenant leakage. | Perform penetration testing with crafted metadata filter payloads attempting to bypass tenant scoping (e.g., injecting `OR tenant_id != 'attacker'` in filter expressions). Review logs for cross-scope query attempts. Validate that alerting fires on detection. Use Garak or PyRIT for automated prompt-based retrieval manipulation testing. | Vector query languages vary widely — Weaviate uses GraphQL filters, Pinecone uses JSON filter syntax, Qdrant uses its own filter DSL. Each needs separate injection testing. Filter injection is an emerging attack class with limited tooling for automated testing. Promptfoo now supports RAG-specific red-teaming scenarios including metadata filter injection tests. |
| **8.1.4** | **Verify that** every ingested document is tagged at write time with source, writer identity (authenticated user or system principal), timestamp, batch ID, and embedding model version. | 2 | Data provenance attacks — without lineage metadata, poisoned vectors cannot be traced back to their source for quarantine or removal. Also prevents "ghost documents" inserted by compromised ingestion pipelines. Maps to MITRE ATLAS AML.T0071 (False RAG Entry Injection). Prompt Security's 2025 RAG_Poisoning_POC demonstrated that a single poisoned embedding can alter system behavior across multiple unrelated queries — provenance tagging is essential for rapid quarantine. | Inspect ingestion pipeline code for metadata attachment at write time. Query stored vectors and confirm all required metadata fields are present. Attempt to update immutable metadata fields via API and verify rejection. Audit batch ingestion logs for completeness. | Immutability enforcement varies: Pinecone metadata can be overwritten via upsert; Weaviate properties can be updated unless schema-level restrictions are set. True immutability may require append-only storage patterns or write-once object storage for metadata alongside the vector index. The four-layer defense architecture (Schneider 2025) recommends provenance tagging as part of Layer 2 (memory sanitization), including creation time, session context, and trust scores for each stored entry. |
| **8.1.5** | **Verify that** RAG pipeline retrieval events log the query issued, the documents or chunks retrieved, similarity scores, the knowledge source, and whether retrieved content passed prompt injection scanning before being incorporated into model context. | 2 | Undetected RAG poisoning — without retrieval logging, poisoned documents that influence model outputs cannot be identified forensically. Also enables detection of indirect prompt injection via retrieved content (OWASP LLM01:2025). Prompt Security's 2025 research showed 80% attack success rate across multiple queries from a single poisoned embedding, and retrieval monitoring (logging document access patterns) is cited as a primary defense — repeated hits from a single source indicate poisoning. | Review logging pipeline configuration. Issue test queries and confirm logs contain all required fields (query text, retrieved doc IDs, similarity scores, source, injection scan result). Verify logs are shipped to a SIEM or centralized logging system. Test log completeness under high-throughput conditions. | Logging similarity scores and retrieved chunks at scale generates significant data volume. Teams need to balance completeness with storage costs. LlamaIndex and LangChain both support callback-based logging but require custom integration for injection scan results. Consider sampling strategies for Level 2 compliance. Schneider's four-layer defense (2025) recommends Layer 4 behavioral monitoring with immutable logging and circuit breakers for rapid isolation when compromise is detected. Qdrant now integrates with Prometheus/Grafana for operational monitoring. |
| **8.1.6** | **Verify that** retrieval anomaly detection identifies embedding density outliers, repeated dominance of specific documents in similarity results, and sudden shifts in retrieval bias distribution that may indicate vector database poisoning. | 3 | Sophisticated vector DB poisoning where an attacker injects many semantically similar vectors to dominate retrieval results for target queries, steering model outputs. Maps to MITRE ATLAS AML.T0070 (RAG Poisoning). PoisonedRAG (Zou et al., USENIX Security 2025) demonstrated that just 5 poisoned documents achieve 90% attack success — the attacker creates a "gravity well" in embedding space. The RSB benchmark (Zhang et al., 2025) evaluated 13 poisoning attack methods and found that all 7 defense mechanisms tested failed to provide robust protection, underscoring the difficulty of detection. Future variants could evolve into "vector worms" that instruct models to re-embed and reintroduce poisoned data elsewhere, creating cascading contamination (Prompt Security, 2025). | Review anomaly detection implementation. Verify it monitors: (1) document retrieval frequency distributions, (2) embedding space density metrics per namespace, (3) temporal shifts in which documents are retrieved for stable query sets. Test with synthetic poisoning scenarios and confirm alerts fire. Behavioral drift detection using runtime analytics to identify sudden tone shifts or stylistic anomalies is a recommended complementary approach. Consider hybrid BM25 + vector retrieval as an architectural defense — research shows it reduces gradient-guided poisoning success from 38% to 0%, though joint sparse+dense optimization can partially circumvent it (20–44% success). | This is an emerging area with limited production tooling but two notable frameworks appeared recently. **RAGuard** (NeurIPS 2025) provides a layered two-step defense: adversarial retriever training using synthetic poisoned documents, plus a zero-knowledge inference patch that identifies and filters suspicious documents based on causal influence on QA correctness — together they reduce ASR while maintaining retrieval quality without requiring poison labels. Custom implementations typically use statistical process control on retrieval frequency histograms or embedding space density estimation (e.g., k-distance plots). No major vector DB offers anomaly detection natively. CorruptRAG (Jan 2026) showed that single-document attacks can dominate retrieval for targeted queries, making per-query retrieval frequency monitoring essential — if one document consistently appears across unrelated queries, that is a strong poisoning signal. SDAG (Feb 2026) provides an inference-time defense by preventing cross-document interaction via sparse attention. Schneider's four-layer defense (2025) recommends trust-weighted ranking adjusted by provenance metadata and temporal decay at the retrieval layer. The RSB benchmark (13 attacks across 5 QA datasets) provides a standardized evaluation framework for testing defense effectiveness. |
