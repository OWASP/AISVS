# C5.3: Query-Time Authorization

> [Back to C05 Index](C05-Access-Control.md)
> **Last Researched:** 2026-06-09 | **Updated:** 2026-06-09 (rev 3)

> **What's new (June 2026):** The headline change since May is ChromaToast — CVE-2026-45829 (CVSS 10.0), HiddenLayer's 18 May 2026 disclosure of a pre-authentication remote-code-execution flaw in ChromaDB's Python FastAPI server. The `create_collection` handler resolves client-supplied embedding-model references, including loading external Hugging Face code with `trust_remote_code` enabled, *before* the authentication check runs; every Python-server release from 1.0.0 through at least 1.5.9 is affected, no patch exists after roughly six months of attempted coordinated disclosure (first report November 2025), and HiddenLayer estimates ~73% of internet-accessible deployments are exploitable. The Rust server implementation does not share the vulnerable code path. That lands on top of May's Open WebUI advisories — CVE-2026-44560 (unauthorized RAG vector-search access through three unguarded paths in `get_sources_from_items`) and CVE-2026-44554 (knowledge-base overwrite and poisoning), both fixed in 0.9.0 — and the earlier AnythingLLM (CVE-2026-24477) and Milvus (CVE-2026-26190, fixed in 2.5.27/2.6.10) exposures. On the enforcement side, Microsoft's query-time ACL/RBAC documentation (refreshed 17 May 2026) pins down behaviour worth testing for: ACL-evaluation failures return 5xx rather than a partially filtered result set (genuinely fail-closed), ACL filters apply even to service-API-key queries since the November 2025 behaviour change, an `x-ms-enable-elevated-read` debug header is gated by a dedicated `contentSecurity/elevatedOperations/read` data-plane permission, SharePoint site groups now resolve at query time with `spg:`-prefixed group IDs in the 2026-05-01-preview API — and per-document ACL entry limits (32 on ADLS Gen2, 1,000 on SharePoint) silently stop enforcing entries beyond the cap. Oso's pgvector/SQLAlchemy integration rounds out the engine options by compiling Polar policies into SQL filters applied through `.authorized()` on the same query that runs vector similarity. On regulation, the Digital Omnibus deferral of EU AI Act high-risk obligations (Article 12 logging included) from 2 August 2026 to 2 December 2027 reached provisional agreement on 6–7 May 2026 and was confirmed by member states on 13 May, but it only takes legal effect on publication in the Official Journal — expected before the original deadline — and the supporting logging standards (prEN 18229-1, ISO/IEC DIS 24970) are still drafts.

## Purpose

Enforce authorization at the data access layer to prevent unauthorized data retrieval through AI queries. Vector similarity search introduces a distinct class of access control challenge: unlike traditional SQL queries with explicit WHERE clauses, vector searches return results based on mathematical similarity, which can cross tenant and classification boundaries if security filters are not mandatory at the data access layer. Collection identifiers, file IDs, metadata filters, row-level security, field-level masking, and re-authorization on retries all need explicit tests because RAG pipelines often have several retrieval paths that look equivalent to users but bypass different code.

> **Scope note:** Controls in this section govern data-layer query enforcement -- mandatory security filters, fail-closed behavior, row/field masking, and retry re-authorization. Authorization policy definition is in C5.2; output-level filtering is in C5.4; tenant isolation at the infrastructure level is in C5.6.

## Verification Notes

- Seed each connected source with matched canary records that share the same semantic content but differ by tenant, user, group, document ACL, classification, purpose, and revocation state. This catches systems that rely on semantic similarity first and only filter after the wrong chunks have already been retrieved.
- Exercise every retrieval entry point, not only the normal chat UI. Include attached files, knowledge-base IDs, direct collection names, agent tools, API batch endpoints, streaming paths, retry handlers, cache reads, full-context modes, non-full-context modes, and fallback retrievers.
- Check both application filters and vector-store enforcement. A passing UI test is not enough if the vector database API key can read the same collection directly, if a management port exposes unauthenticated REST APIs, or if the service account can query every tenant's index. ChromaToast makes this concrete: a network-reachable ChromaDB Python server is exploitable pre-authentication regardless of application-layer filter quality, so the network exposure review is part of the query-time authorization audit, not a separate infrastructure concern.
- Audit the escape hatches and the limits, not just the happy path. Where the platform offers an elevated or unfiltered read mode (Azure AI Search's `x-ms-enable-elevated-read` header, vector-store admin credentials, debug endpoints), enumerate who holds the gating permission and confirm its use is logged and alerting. Where the platform caps permission metadata (32 ACL entries per ADLS Gen2 file, 1,000 per SharePoint item), test a deliberately over-shared document to see whether out-of-limit entries fail open or fail closed.
- Treat generated or implicit filters as relevance helpers unless they are constrained by trusted authorization data. Filters inferred from user prompts can narrow search results, but they should not decide whether a caller is entitled to a source document.
- Validate the retrieval audit trail end-to-end. For each query, the trace should record principal, generated authorization filter, retrieved document IDs (including any rejected by post-filter), the policy decision, the ACL version of the source system, and the vector-store credential used. Under EU AI Act Article 12 these logs need to be automatic, tamper-evident, and retained for at least six months for high-risk systems.
- Anchor the regression suite to a real authorization engine, not a mock. The Docker Labs "RAG Permission Testing with Testcontainers & SpiceDB" walkthrough (15 January 2026) shows a reproducible pattern: spin up `authzed/spicedb:v1.47.1` through the Testcontainers `spicedbcontainer.Run` helper, load a fixed ReBAC schema and tuples per test, then assert that `CheckPermission` denials really do drop documents before the LLM call. The Cerbos `latest` documentation publishes a parallel recipe that maps a single PDP query-plan into a Chroma `where` clause, which can be exercised the same way with `chromadb`'s in-memory client. Run both during CI on every change to retrieval, prompt assembly, or policy schemas.
- For Microsoft 365 Copilot tenants, treat the Purview "Risky AI usage" Insider Risk Management policy template as a required detective control alongside the preventive ACL/label filters; it captures prompt-injection attempts and unauthorised-material access in the same audit pipeline DSPM for AI consumes, which is what links a retrieval-side denial to a tenant-level investigation.

## Notable Incidents

- **ChromaDB "ChromaToast" CVE-2026-45829 (May 2026)** -- HiddenLayer disclosed a CVSS 10.0 pre-authentication remote-code-execution flaw in ChromaDB's Python FastAPI server on 18 May 2026 after roughly six months without maintainer response (first researcher report November 2025, HiddenLayer contact from 17 February 2026). The `create_collection` handler parses client-supplied embedding-model configuration — including loading external Hugging Face repository code with `trust_remote_code` — before the authentication check executes, yielding code execution with access to API keys, environment variables, and mounted secrets. All Python-server versions from 1.0.0 through at least 1.5.9 are affected, no patch is available, and an estimated 73% of internet-accessible deployments are exploitable; the Rust server does not share the code path. See https://www.hiddenlayer.com/research/chromatoast-served-pre-auth and https://www.securityweek.com/unpatched-chromadb-vulnerability-can-lead-to-server-takeover/.
- **Open WebUI CVE-2026-44560 (May 2026)** -- Three RAG source-resolution paths in `backend/open_webui/retrieval/utils.py` (`get_sources_from_items`) accepted user-supplied collection names and file references and queried the vector store without an authorization check. The advisory text emphasises that access revocation is ineffective for already-indexed RAG content until upgrading to 0.9.0. CVSS 6.5. See https://github.com/open-webui/open-webui/security/advisories/GHSA-h36f-rqpx-j5wx.
- **Open WebUI CVE-2026-44554 (May 2026)** -- `POST /api/v1/retrieval/process/web` accepted an attacker-controlled `collection_name` plus an `overwrite=True` default that allowed any authenticated user to destroy or poison another user's knowledge base after guessing the UUID, the integrity-side counterpart to the read failure above. See https://advisories.gitlab.com/pypi/open-webui/CVE-2026-44554/.
- **AnythingLLM CVE-2026-24477 (Feb 2026)** -- An unauthenticated `setup-complete` endpoint exposed Qdrant API keys, giving any unauthenticated attacker direct read/write access to the vector store and bypassing every application-layer ACL or tenant filter. See https://nvd.nist.gov/vuln/detail/CVE-2026-24477.
- **Milvus CVE-2026-26190** -- Milvus deployments exposed management and REST APIs on TCP 9091 without authentication, allowing direct collection enumeration, insertion, and deletion regardless of any RAG-layer authorization. Fixed in Milvus 2.5.27 and 2.6.10. See https://nvd.nist.gov/vuln/detail/CVE-2026-26190.
- **Microsoft 365 Copilot oversharing audits (2026)** -- EPC Group's 2026 audit of 47 enterprise Copilot deployments since the GA release found that roughly 80% of tenants had material oversharing exposure, almost entirely traceable to pre-existing SharePoint/OneDrive permission breadth that became newly reachable once Copilot started reasoning across user-accessible content. The Microsoft Learn "blueprint for oversharing" and "secure-govern Copilot" pages now codify the remediation pattern: tighten source ACLs first, then layer Restricted Access Control, sensitivity labels, and SharePoint Advanced Management governance on top. See https://learn.microsoft.com/en-us/copilot/microsoft-365/microsoft-365-copilot-blueprint-oversharing.

## Implementation Maturity

- **Mature** -- Document-level ACL enforcement at query time exists in Azure AI Search (with Microsoft Purview sensitivity labels, though the permission-filter features remain on the 2026-05-01-preview API as of June 2026), Amazon Bedrock Knowledge Bases metadata filtering, OpenSearch DLS/FLS, Qdrant payload filters, and Pinecone namespaces with API-key roles. Authorization engines such as SpiceDB, OpenFGA, OPA, Cerbos, Oso, and Permit.io are well-established for the policy layer, and Oso's SQLAlchemy/pgvector integration covers the Postgres-native retrieval path.
- **Emerging** -- Library-level glue that translates an authorization decision into a native vector-store filter is now usable but still young. Authzed's `langchain-spicedb` v0.1.1 (Jan 2026) ships pre-filter `LookupResources` and post-filter `CheckBulkPermissionsRequest` paths for LangChain/LangGraph, `ragguard` provides multi-engine pre-filter translation across 12+ vector stores, and Cerbos publishes a query-plan recipe that compiles ABAC/RBAC/ReBAC policies into Chroma `where` filters in a single PDP call -- a useful counterweight to per-result post-filter checks. Docker Labs' Testcontainers + SpiceDB pattern (15 Jan 2026, pinned to `authzed/spicedb:v1.47.1`) is now the lowest-friction way to run these checks in CI. Glean's Purview integration for OneDrive/SharePoint sources is still beta. Expect breaking API changes and lean on canary regression tests.
- **Immature** -- Vector-store-native enforcement of source-system ACLs (where the database itself understands the upstream permission model) is still rare; Databricks' May 2026 docs continue to say row and column level permissions are not supported and must be implemented in the application's filter calls (Unity Catalog ABAC with governed tags is the recommended workaround, but the policy still runs in Unity Catalog, not in Mosaic AI Vector Search). Post-filter result-count side channels, retry-loop fail-open behaviour, and stale tuple/cache eviction under permission revocation remain weakly covered by off-the-shelf tooling. Standardised agent-identity primitives for RAG retrieval are still in draft: the NIST NCCoE concept paper closed comments on 2 April 2026, but normative guidance and a reference architecture have not shipped yet, so teams that want to bind retrieval traces to an agent identity have to roll their own SPIFFE/JWT plumbing on top of the engines above. The same is true of the Article 12 logging standards the EU AI Act will eventually lean on -- prEN 18229-1 (AI logging and human oversight) and ISO/IEC DIS 24970 (AI system logging) are both unfinished drafts as of June 2026. And the vector stores themselves remain the weakest link: ChromaToast shows a top-five vector database shipping a pre-auth RCE for over six months without a patch, so "the store enforces nothing on its own" is still the safe planning assumption.
