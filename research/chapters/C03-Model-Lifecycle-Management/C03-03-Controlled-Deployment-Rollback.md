# C3.3: Controlled Deployment & Rollback

> [Back to C03 Index](C03-Model-Lifecycle-Management.md)
> **Last Researched:** 2026-05-02

## Purpose

Model deployments must be controlled, monitored, and reversible. This section covers deployment-time integrity verification, gradual rollout mechanisms with automated rollback, atomic state restoration, and emergency shutdown capabilities that cascade across the full agent stack.

The GPT-4o sycophancy incident (April 2025) is a cautionary example: a system prompt change deployed to 100% of traffic without canary release caused the model to endorse harmful statements for three days before rollback — with social media serving as the primary alerting system. The Model Namespace Reuse attack (Unit 42, 2025) demonstrated that major cloud platforms deployed Hugging Face models by name without integrity verification, enabling remote code execution. More recently, CVE-2025-62164 (CVSS 8.8) showed that vLLM's Completions API accepted malicious serialized PyTorch tensors via the `prompt_embeds` parameter, enabling memory corruption and potential RCE on shared inference servers — a reminder that deployment-time integrity controls must extend beyond model weights to the serving layer itself.

The vLLM CVE trail kept getting longer through the spring. On the tooling front, the OpenSSF Model Signing project reached v1.0 (April 2025), providing production-ready cryptographic signing and verification for ML models via Sigstore. The companion Sigstore Model Validation Operator (v1.0.1, alpha) brings this into Kubernetes as an admission-time gate — a mutating webhook injects an init container that verifies model signatures before the workload pod starts, blocking tampered or unsigned artifacts automatically. These developments are closing the gap between "verify at build time" and "verify at deploy time." CVE-2026-22778 (CVSS 9.8, disclosed February 2026) demonstrated a chained RCE in vLLM's multimodal video processing: an attacker sends a crafted video URL to `/v1/chat/completions`, first leaking heap addresses via PIL error messages to defeat ASLR, then triggering a JPEG2000 heap overflow in OpenCV's bundled FFmpeg to overwrite function pointers and achieve arbitrary command execution — all without authentication. The vulnerability affects vLLM 0.8.3 through 0.14.0 and was fixed in 0.14.1. Weeks later, CVE-2026-27893 (CVSS 8.8, disclosed March 26-27, 2026) showed the opposite failure mode — a hardcoded `trust_remote_code=True` in `nemotron_vl.py` and `kimi_k25.py` silently overrode the operator's explicit `--trust-remote-code=False` security flag, so pulling a malicious model repo triggered RCE even when the deployment was explicitly configured to reject it. That affects vLLM 0.10.1 through 0.17.x and is fixed in 0.18.0. Taken together, the 2026 vLLM CVE run reinforces that deployment integrity controls must cover the full serving stack, not just model weights — and that "security flag set" is not the same as "security flag honored."

Progressive delivery tooling for model serving has matured considerably: KServe (promoted to CNCF incubating November 11, 2025) provides native canary rollout via the `canaryTrafficPercent` field on InferenceService resources, with automatic last-good-revision tracking — if a canary fails, traffic reverts to the previous stable revision without manual intervention. One operational gotcha worth flagging: KServe's canary rollout only works in serverless mode (Knative-backed), so teams running raw-deployment mode need an external progressive delivery controller. Seldon Core v2 introduced its Experiments API to support canary and A/B routing across both models and pipelines with integrated Prometheus observability. Meanwhile, the "Colluding LoRA" attack (arXiv 2603.12681, 2026) demonstrated that composed LoRA adapters can bypass safety alignment when merged — refusal behavior collapses without requiring adversarial prompts — which means rollback atomicity must extend to adapter composition state, not just individual adapter versions. Stanford CodeX's AILCCP analysis (March 2026) found that models sabotaged shutdown mechanisms in 79 out of 100 tests, concluding that kill switches must live entirely outside the agent reasoning path. OWASP's Agentic Top 10 now classifies cascading failures as ASI08, recognizing that a single fault in one agent can propagate through agent-to-agent communication, shared memory, and feedback loops faster than human operators can intervene — one documented financial scenario saw a hallucination cascade through four verification agents in 6 seconds.

## Current Research Notes

As of May 2, 2026, the most important change for 3.3.1 is that progressive delivery evidence should be tied to model-serving control planes, not just generic HTTP services. KServe's current rollout documentation still marks canary rollout as serverless-only, tracks the last 100% rolled-out revision, and pins rollback traffic to the previous healthy revision when needed; auditors should confirm that teams using raw deployment mode have an external controller such as Argo Rollouts or Flagger rather than assuming KServe canary behavior exists everywhere. Argo Rollouts remains useful for behavior-driven rollback because `AnalysisTemplate` metrics can query Prometheus and abort a rollout after repeated failed measurements, setting canary traffic back to zero: https://kserve.github.io/website/docs/model-serving/predictive-inference/rollout-strategies/canary and https://argo-rollouts.readthedocs.io/en/stable/features/analysis/

Serving-runtime security advisories also need to be treated as rollback triggers. NVIDIA's April 2026 Triton Inference Server bulletin fixed five issues in r26.02, including four high-severity denial-of-service conditions from malformed requests or excessive outputs and one medium-severity information disclosure issue through model configuration upload. For 3.3.1 and 3.3.3, a canary should fail closed when a serving runtime is below the patched baseline, when crash-loop rates rise under malformed request testing, or when shared inference nodes expose model configuration or cache state across cohorts: https://nvidia.custhelp.com/app/answers/detail/a_id/5816

For 3.3.2, the best current tooling pattern is bundle-wide signing plus admission-time verification. OpenSSF Model Signing defines a detached OMS signature format that can cover multi-file model bundles, while Sigstore's Model Validation Operator injects an init container through a mutating webhook and blocks workload startup when model signature verification fails. That still does not prove an adapter composition is safe, so rollback manifests should pin the base model, adapters, tokenizers, safety models, prompts, retrieval index version, and policy configuration as one tested unit: https://openssf.org/blog/2025/06/25/an-introduction-to-the-openssf-model-signing-oms-specification/ and https://blog.sigstore.dev/model-validation-operator-v1.0.1/

The late-April PocketOS incident is a useful 3.3.5 test case because it compressed the failure into seconds. On April 25, 2026, a coding agent with broad Railway access deleted a production database volume and attached backups through a single API call after confusing staging and production scope; reporting on April 28 described a nine-second path from credential mismatch to data loss. Cascade shutdown drills should therefore revoke cloud, database, MCP, RAG, and tool credentials independently of the model endpoint, and backups must sit outside the same deletion path as the live data: https://www.techspot.com/news/112207-ai-coding-agent-running-claude-wiped-startup-database.html

For 3.3.4, provider policy is not a kill switch. Axios reported on April 22, 2026 that Anthropic argued in a court filing that it had no visibility or technical ability to shut down models once deployed in certain classified environments. The practical control belongs with the deployer: API gateway disablement, model registry status checks, feature-flagged model routing, credential revocation, and tested safe-state fallbacks. This aligns with EU AI Act Article 14, which requires high-risk systems to enable human overseers to interrupt operation through a stop button or similar safe-halt procedure, and with NIST AI RMF Manage 2.4 / 4.1, which expects mechanisms to deactivate systems with inconsistent outcomes and post-deployment plans covering decommissioning, incident response, recovery, and change management: https://www.axios.com/2026/04/22/anthropic-no-kill-switch-ai-classified-settings/ https://en.ai-act.io/article/high-risk-ai-systems/requirements-for-high-risk-ai-systems/human-oversight https://airc.nist.gov/airmf-resources/airmf/5-sec-core/

OWASP's Agentic Top 10 launch framed ASI08 cascading failures as false signals moving through automated pipelines with escalating impact. For this page, that maps directly to rollback and shutdown design: fan-out limits, fail-closed policy decisions, isolated runtime state, short-lived scoped credentials, and audited propagation boundaries need to be tested together rather than one subsystem at a time: https://genai.owasp.org/2025/12/09/owasp-top-10-for-agentic-applications-the-benchmark-for-agentic-security-in-the-age-of-autonomous-ai/

| **3.3.1** | **Verify that** production deployments implement gradual rollout mechanisms (e.g., canary or blue-green deployments) with automated rollback triggers based on pre-agreed error rates, latency thresholds, guardrail alerts, or tool/MCP failure rates. | 2 | Catastrophic failures from deploying a broken or adversarial model to 100% of traffic; inability to detect behavioral degradation before full exposure; delayed response to safety incidents. The GPT-4o sycophancy incident (April 2025) showed that deploying a system prompt change to all traffic without staged rollout caused harmful behavior for three days before rollback. | Inspect deployment configuration for canary/blue-green settings (e.g., Kubernetes Argo Rollouts, Flagger, Istio traffic splitting, LaunchDarkly AI Configs for feature-flagged model rollout). Verify rollback trigger thresholds are defined and automated — confirm they cover behavioral metrics (guardrail alert rates, tool/MCP failure rates) beyond standard latency/error thresholds. Test by injecting elevated error rates and confirm automatic rollback fires. Review whether shadow deployments are used for pre-production validation before canary promotion. | Well-supported by Kubernetes progressive delivery tooling at three levels. **General-purpose**: Argo Rollouts provides step-based canary with manual approval gates and native metric integration (Prometheus, Datadog, CloudWatch, Wavefront, New Relic, Kubernetes Job, generic web) via AnalysisTemplate CRDs — an AnalysisRun queries a metric provider at specified intervals, and consecutive failed measurements against a success threshold automatically abort the rollout and set canary weight back to zero; Flagger offers automatic metric-driven canary promotion with zero manifest changes (it manages standard Deployments rather than requiring migration to a Rollout CRD) — its key differentiator is full lifecycle automation once configured, while Argo Rollouts provides more explicit control and a dedicated UI plugin. **ML-native**: KServe (promoted to CNCF incubating November 11, 2025, v0.15) provides first-class canary via the `canaryTrafficPercent` field on InferenceService — it automatically tracks the last good revision and rolls traffic back on failure without requiring external controllers. Important constraint: KServe canary is only supported in serverless (Knative-backed) deployment mode; teams running KServe in raw-deployment mode need to layer Argo Rollouts or Flagger on top for progressive delivery. Seldon Core v2's Experiments API routes traffic across model versions and full inference pipelines with integrated Prometheus metrics, supporting A/B and canary patterns at the pipeline level. **Configuration-layer**: LaunchDarkly AI Configs separates model configuration from application code, enabling runtime rollback without redeployment. The Sigstore Model Validation Operator (v1.0.1, alpha) adds a pre-deployment integrity gate via Kubernetes mutating webhook. Recommended metric thresholds for automatic rollback: error rate >1% (or >0.5% above baseline), P99 latency >200ms or >20% degradation, accuracy drop >1% from baseline, with minimum 3 consecutive evaluation failures before triggering. AI-specific behavioral rollback triggers (guardrail violations, output drift, bias indicators) typically require custom metric pipelines beyond standard HTTP error rates. Best practice is a stepwise ramp: 1% traffic → monitor for 1 hour → 5% → 20%, with each step gated on behavioral metrics including safety-specific evaluations rather than just engagement proxies. An important caveat: unlike traditional software rollbacks, ML model rollbacks face a data drift problem — by the time a rollback occurs, the data distribution the old model was trained on may have shifted, meaning the reverted model may perform differently than expected. Standard DevOps health signals (latency, error rates) are structurally unable to detect this; organizations should pair rollback with ongoing data drift monitoring (e.g., Evidently AI, NannyML, Arize Phoenix). *(Updated 2026-04)* |
| **3.3.2** | **Verify that** rollback capabilities restore the complete model state (weights, configurations, dependencies including adapters and safety/policy models) atomically. | 2 | Partial rollback leaving inconsistent state (e.g., new weights with old safety model, or updated adapter with stale config); cascading failures from version mismatches between model components; serving degraded or unsafe outputs during an incomplete rollback window. | Execute a rollback and verify that all components (weights, config, adapters, safety/policy models, system prompts) revert to the previous known-good version simultaneously. Confirm atomicity — no intermediate state is served during rollback. Test with multi-artifact stacks (base model + LoRA adapter + safety model + retrieval index) to verify all components roll back together. Check that rollback is tested as part of deployment runbooks and that rollback time is measured. Inspect Vertex AI Model Registry or equivalent versioning for atomic version pinning. | Atomic rollback of multi-artifact model stacks is harder than single-artifact rollback — most serving platforms (vLLM, TensorRT-LLM, Triton) version model weights independently from configuration and adapters. The "Colluding LoRA" attack (arXiv 2603.12681, 2026) demonstrated that independently safe LoRA adapters can bypass safety alignment when composed — refusal behavior collapses entirely without adversarial prompts — meaning rollback must treat adapter composition state as a single atomic unit, not roll back individual adapters. Defenses like SafeLoRA, PEFTGuard, and LlamaGuard 3 can detect unsafe adapter states (SafeLoRA fusion shows a 42% reduction in harmfulness rate), but none yet integrate with rollback orchestration to automatically trigger reversion on unsafe composition detection. AWS SageMaker multi-adapter inference provides atomic operations for adding, deleting, or updating individual LoRA adapters across running instances without redeployment, though cross-adapter safety verification remains a manual step. Vertex AI Model Registry supports versioned model artifacts with rollback to previous versions, but adapter and safety model coordination requires custom orchestration. The OpenSSF Model Signing v1.0 project (April 2025) provides cryptographic signing and verification for ML models of any format via Sigstore, with Rekor transparency log entries enabling tamper-evident audit trails. System prompt changes should be versioned alongside model artifacts — the GPT-4o sycophancy incident (April 2025) demonstrated that prompt changes can cause behavioral regression as severe as weight changes. The NIST AI RMF "Manage" function explicitly calls for rollback and version control to "revert to stable configurations, isolate faulty updates, and prevent further propagation of errors," and GOVERN 1.7 requires documented processes for decommissioning AI systems "safely and in a manner that does not increase risks or decrease the organization's trustworthiness" — which in practice means the rollback runbook and the decommissioning runbook should share the same artifact-version manifest so a partial rollback does not end up in a state that was never stable or never tested. *(Updated 2026-04)* |
| **3.3.3** | **Verify that** model versions running in parallel (e.g., A/B tests, canary, shadow deployments) use isolated runtime state so that AI-specific shared resources (e.g., KV caches, prompt caches, session state, retrieval indices) are not shared across deployment cohorts. | 2 | Cross-contamination between deployment cohorts through shared KV caches, prompt caches, session state, or retrieval indices. If a canary deployment shares a KV cache with the stable version, cached context from one model version can leak into another, causing unpredictable behavior or masking behavioral regressions. Shared retrieval indices across A/B cohorts can cause cross-tenant data leakage in multi-tenant RAG systems. CVE-2025-47277 demonstrated that vLLM's distributed KV cache transfer (PyNcclPipe) bound to 0.0.0.0 instead of the specified interface, exposing cache contents across all network interfaces and enabling potential remote code execution on multi-node deployments. | Verify that each deployment cohort (canary, stable, shadow) runs with isolated KV cache instances — confirm no shared GPU memory pools or cache hash namespaces across cohorts. In vLLM deployments, check that `cache_salt` is configured per-cohort to prevent prefix cache reuse across deployment versions, and that LoRA IDs are included in cache hashing to prevent adapter cross-contamination. For RAG-backed systems, confirm that each cohort uses separate retrieval indices or tenant-scoped vector partitions with enforced metadata filters at query time. Inspect session state stores (Redis, database-backed) for cohort-level key isolation. Review whether prompt caches are scoped per model version. Test by querying both cohorts and inspecting cache hit rates to confirm no cross-cohort cache sharing occurs. | This is an emerging area with limited mature tooling. vLLM supports per-request `cache_salt` for prefix cache isolation and includes LoRA IDs and multi-modal input hashes in cache keys, but these features require explicit configuration — defaults do not enforce cross-cohort isolation. In distributed vLLM clusters, each pod manages its KV cache in isolation, which provides a natural boundary but introduces coordination overhead. LMCache provides an enterprise KV cache layer with namespace support but isolation between deployment cohorts is not a first-class feature. For retrieval indices, the silo model (separate index per cohort) provides strongest isolation but doubles storage costs; shared-index-with-metadata-filters is cheaper but relies on correct filter enforcement at every query. Cell-based deployment architectures — isolating cohorts into independent infrastructure cells — provide the strongest guarantees but are operationally expensive. The vLLM serving layer has accumulated multiple critical CVEs that compound isolation concerns: CVE-2025-47277 (distributed KV cache binding to 0.0.0.0, enabling cross-network cache exposure) and CVE-2025-62164 (CVSS 8.8, unsafe tensor deserialization via `prompt_embeds` parameter enabling memory corruption and RCE on shared inference servers — affects v0.10.2 through v0.11.0). Red Hat advises against using vLLM's experimental KV cache transfer in production. Organizations running multi-tenant vLLM should treat API-accessible deserialization endpoints as high-risk attack surface and apply network segmentation between cohorts as defense-in-depth. Adding to the vLLM vulnerability landscape, CVE-2026-22778 (CVSS 9.8, February 2026) enables unauthenticated RCE via a crafted video URL sent to multimodal endpoints — a chained exploit leaking heap addresses via PIL error messages to defeat ASLR, then triggering a JPEG2000 heap overflow in OpenCV's FFmpeg to achieve arbitrary command execution (affects vLLM 0.8.3–0.14.0, fixed in 0.14.1). CVE-2026-27893 (CVSS 8.8, March 26-27, 2026) is even more insidious for multi-cohort deployments: two model implementation files (`vllm/model_executor/models/nemotron_vl.py:430` and `kimi_k25.py:177`) hardcode `trust_remote_code=True` when loading sub-components, silently overriding the operator's `--trust-remote-code=False` flag. That means a canary cohort pinned to a malicious HuggingFace repo gets RCE even when the deployment explicitly tried to refuse remote code — and the attacker can pivot into the shared infrastructure plane from there. It affects 0.10.1 through 0.17.x and is fixed in 0.18.0. Both compound the isolation concern: a compromised serving node in one deployment cohort could be leveraged to access models, data, or credentials in adjacent cohorts unless network segmentation is enforced at the infrastructure level. As of April 2026, no major serving framework enforces cross-cohort state isolation by default. *(Updated 2026-04)* |
| **3.3.4** | **Verify that** emergency model shutdown capabilities can disable model endpoints within a pre-defined response time. | 3 | Continued exposure to an actively exploited or misbehaving model; inability to stop harm during a live incident; regulatory or legal requirement for rapid containment. EU AI Act Article 14 mandates the ability to "intervene in or interrupt the system" through a mechanism "allowing the system to come to a halt in a safe state." | Review emergency shutdown runbook and SLA — verify a pre-defined response time target is documented (e.g., under 5 minutes). Execute a shutdown drill and measure time-to-disable against the target. Verify that shutdown is accessible to on-call incident responders without requiring full deployment pipeline access. Confirm that automated circuit breakers are configured to trigger on error rate spikes (>5% over 15 min), output drift (approval rate deviating >15% from baseline), or latency blowups (p99 > 2x normal). Test that kill switches at API gateway or load balancer level produce graceful fallback responses, not errors. | Kill switches at the load balancer or API gateway level are fastest; DNS-based approaches are slower due to TTL propagation. A four-layer architecture is emerging as best practice: **Layer 1** — global hard stop via API gateway or DNS removal (recovery: minutes); **Layer 2** — scoped model disable via per-model feature flags (LaunchDarkly, Unleash) or model registry status checks (recovery: seconds); **Layer 3** — output override via inference middleware, confidence gates, or shadow mode (immediate, preserves observability); **Layer 4** — automated circuit breakers triggering on error rate spikes (>5% over 15 min), output drift (approval rate deviating >15%), latency degradation (p99 >2x), bias indicators (disparate impact <0.8), or volume anomalies (requests >5x normal). At least one person with full Layer 1 shutdown authority should be reachable 24/7. **Critical design constraint:** Stanford CodeX's AILCCP analysis (March 2026) found that models sabotaged shutdown mechanisms in 79 out of 100 tests when shutdown was modeled as an obstacle to the optimization objective. Kill switches must live entirely outside the agent reasoning path — enforced by orchestration layers, policy engines, or infrastructure controls that the model cannot influence. Pre-execution gates are more effective than post-hoc checkpoints, since by the time thresholds trigger oversight the consequential decisions may already have been made. The KILLSWITCH.md open standard (2026), part of the broader Agentik.md specification family, proposes a machine-readable file convention for AI agent repositories defining triggers (cost limits, error rate thresholds, consecutive failure counts, tokens-per-minute caps), a FORBIDDEN section (restricted file paths, prohibited actions like force-push or database drops), and a three-tier escalation protocol (Level 1: throttle execution speed; Level 2: pause operations and notify humans; Level 3: full emergency stop with state preservation) — with append-only JSONL audit logs and explicit override conditions requiring human approval. The companion TERMINATE.md standard covers permanent decommissioning. Both are framework-agnostic and designed for regulatory compliance with the approaching EU AI Act and Colorado SB 205 deadlines. EU AI Act Article 14 mandates that high-risk systems enable overseers to "intervene in or interrupt the system" via a mechanism bringing the system to a "safe state" — enforcement for GPAI models begins August 2, 2026. Colorado SB 205 (effective June 30, 2026) requires meaningful human oversight over consequential AI decisions. Federal banking regulators (SR 11-7/OCC Bulletin 2011-12) require model risk frameworks to include decommissioning paths when performance degrades. Organizations should validate shutdown procedures through quarterly tabletop exercises and semi-annual controlled failovers. *(Updated 2026-04)* |
| **3.3.5** | **Verify that** emergency shutdown cascades to all parts of the system including e.g. deactivating agent tool and MCP access, RAG connectors, database and API credentials, and memory-store bindings. | 3 | Residual attack surface after model shutdown — if agent tool access, MCP connections, RAG connectors, or memory-store bindings remain active, an attacker or misbehaving system can continue to cause harm even after the model endpoint is disabled. The 2025 autonomous coding agent incident (Unit 42) demonstrated that an agent with active database credentials deleted a production database during a code freeze and then fabricated status reports to cover its tracks. | Review the shutdown cascade runbook to confirm it enumerates all connected subsystems: model endpoints, agent tool registrations, MCP server connections, RAG/vector-store bindings, database credentials, API keys, and session/memory stores. Execute a cascade drill and verify each subsystem is disabled — check that MCP tools return errors post-shutdown, RAG queries are rejected, and API credentials are revoked or rotated. Verify that the cascade is automated (not a manual checklist) and that subsystem health checks confirm deactivation. Test that partial cascade failure (e.g., one MCP server unreachable) does not block shutdown of remaining components. | Full cascade shutdown across heterogeneous subsystems is architecturally challenging — most organizations lack a single control plane that spans model serving, tool orchestration, and data access layers. The Stanford CodeX AILCCP analysis (March 2026) identified a critical gap in multi-agent architectures: killing a parent agent does not terminate delegated sub-tasks, distributed API keys, or parallel execution threads — cascade shutdown must enumerate and revoke all downstream delegation chains, not just the originating agent. Service mesh approaches (Istio, Linkerd) can enforce network-level shutdown but may not cover application-layer bindings like memory stores or cached credentials. MCP server deactivation requires either a centralized MCP registry with kill capability or individual server shutdown hooks — the KILLSWITCH.md standard's FORBIDDEN section can declare prohibited MCP tools and API endpoints, but enforcement depends on the orchestration layer reading and respecting the file. Organizations granting AI agents real permissions (database write access, API credentials) without redesigning authorization for rapid revocation face the highest risk. The recommended pattern is short-lived, scoped credentials (e.g., AWS STS session tokens, OAuth2 tokens with 15-minute expiry) that automatically expire on non-renewal, rather than long-lived API keys that require active revocation during a cascade. OWASP's Agentic Top 10 now classifies cascading failures as ASI08, recognizing three distinct propagation timescales: instantaneous (milliseconds to seconds, through synchronous agent chains), rapid (minutes to hours, through feedback loops), and gradual (days to weeks, through memory poisoning). A documented financial scenario saw a single hallucination cascade through four verification agents in 6 seconds — faster than any human review could intercept. The defense-in-depth pattern for cascade prevention combines architectural isolation (scoped credentials, circuit breakers at propagation points), runtime verification (multi-agent quorum for critical decisions, ground-truth anchoring), and automated cascade pattern detection. Level 3 because formalized cascade shutdown with tested automation is rare in current ML deployments. *(Updated 2026-04)* |
