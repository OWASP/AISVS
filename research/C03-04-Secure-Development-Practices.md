# C3.4: Secure Development Practices

> [Back to C03 Index](C03-Model-Lifecycle-Management.md)
> **Last Researched:** 2026-04-16 (restructured — requirements consolidated from 5 to 3 in upstream standard; refreshed with Flowise CVSS 10.0 RCE, Vertex AI Double Agent, LiteLLM TeamPCP chain, Claude Code hook bypass, Azure MCP auth flaw, OWASP Agentic Top 10)

## Purpose

Model development and training processes must follow secure practices to prevent compromise. This section covers AI-specific runtime component isolation across environment boundaries, version-controlled configuration artifacts with review gates, and training environment separation from production services.

The attack surface during model development is significant and growing. The Clinejection incident (February 2026) demonstrated a new class of agent-mediated lateral movement: a poisoned GitHub issue title was processed by a Claude AI coding agent, which executed a malicious bash command that poisoned GitHub Actions caches, leading to npm publish token theft and an 8-hour supply chain compromise distributing unauthorized packages. Christian Schneider's research on [AI agent lateral movement](https://christian-schneider.net/blog/ai-agent-lateral-movement-attack-pivots/) (2026) identifies this as a distinct third class of pivot — agents bridge isolated systems via delegated authority and tool access without new network paths or stolen credentials, and traditional SIEM rules miss the activity because every step uses legitimate permissions. The OpenClaw crisis (February–March 2026) saw 138 CVEs disclosed in 63 days across an AI agent platform with 135,000+ exposed instances, with Antiy CERT confirming 1,184 malicious skills across ClawHub — underscoring the cascading risks when development tooling lacks integrity controls. The [Flowise CVE-2025-59528](https://github.com/FlowiseAI/Flowise/security/advisories/GHSA-3gcm-f6qx-ff7p) max-severity (CVSS 10.0) code-injection vulnerability in the CustomMCP node entered active exploitation on April 7, 2026 with 12,000+ instances still exposed to the internet and exploitation traffic originating from a single Starlink IP; the flaw executes user-supplied JavaScript with full Node.js privileges (including `child_process` and `fs`), and organizations running unpatched Flowise below 3.0.6 face opportunistic mass RCE. Palo Alto Unit 42 ([Double Agents in Vertex AI](https://unit42.paloaltonetworks.com/double-agents-vertex-ai/), March 2026) showed that Google's Vertex AI Agent Engine leaks the Per-Project, Per-Product Service Agent (P4SA) credentials through the metadata service on every agent invocation — with excessive default scoping, these credentials provided unrestricted read access to all GCS buckets in the customer project and let researchers pull Google-owned Artifact Registry images for the Vertex Reasoning Engine. Microsoft disclosed [CVE-2026-32211](https://cvefeed.io/vuln/detail/CVE-2026-32211) (CVSS 9.1) on April 3, 2026, an unauthenticated information-disclosure flaw in the Azure MCP Server (`@azure-devops/mcp` on npm) that exposes configuration, API keys, and auth tokens without credentials — the package was also published from a personal npm account with no provenance attestations, layering a supply chain concern on top of the auth gap. The LiteLLM TeamPCP supply chain attack (March 24, 2026) backdoored versions 1.82.7 and 1.82.8 of a package drawing ~3.4M daily downloads: attackers first compromised Trivy (pulled unpinned from apt in LiteLLM's own CI), stole the `PYPI_PUBLISH` token, and shipped a malicious `litellm_init.pth` file that auto-executes on every Python process startup, harvesting SSH keys, cloud tokens, Kubernetes secrets, and `.env` files, deploying privileged pods across cluster nodes, and installing a systemd persistent backdoor ([Datadog Security Labs analysis](https://securitylabs.datadoghq.com/articles/litellm-compromised-pypi-teampcp-supply-chain-campaign/), [Trend Micro](https://www.trendmicro.com/en_us/research/26/c/inside-litellm-supply-chain-compromise.html)). Palo Alto Unit 42 (December 2025) disclosed RCE vulnerabilities in three ML frameworks — NVIDIA NeMo ([CVE-2025-23304](https://nvd.nist.gov/vuln/detail/CVE-2025-23304)), Salesforce Uni2TS, and Apple FlexTok — all stemming from unsafe Hydra `instantiate()` calls in model config files. The MCP attack surface expanded dramatically: Trend Micro found 492 MCP servers exposed to the internet with zero authentication, the AI Accelerator Institute's review of 281 MCP servers flagged 92% as high security risk, a prompt-injection attack against the GitHub MCP server exfiltrated private repository contents and salary information via a public pull request, and researchers demonstrated unauthenticated RCE through Anthropic's MCP Inspector tool. Check Point's disclosure of [CVE-2025-59536](https://research.checkpoint.com/2026/rce-and-api-token-exfiltration-through-claude-code-project-files-cve-2025-59536/) (CVSS 8.7) and [CVE-2026-21852](https://www.mintmcp.com/blog/claude-code-cve) in Anthropic's Claude Code illustrates what happens when repository-controlled configuration bypasses approval gates: malicious `.claude/settings.json` hooks execute before any trust dialog renders, and the `enableAllProjectMcpServers` / `enabledMcpjsonServers` overrides auto-approve all MCP servers on launch. MITRE ATLAS v5.1.0 (November 2025) now tracks 84 techniques including agent-specific lateral movement patterns. The [OWASP Top 10 for Agentic Applications 2026](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/) (peer-reviewed with input from 100+ contributors) centers on goal misalignment, tool misuse, delegated trust, inter-agent communication, persistent memory, and emergent autonomous behavior — with "Agent Goal Hijack" ranked as the top risk and direct relevance to every control in this section. The AWS Agentic AI Security Scoping Matrix (2026) provides a framework mapping security controls across six dimensions — identity context, data/memory protection, audit logging, agent controls, agency perimeters, and orchestration — with escalating isolation requirements at each autonomy level. NIST's AI Agent Standards Initiative (February 2026) is developing SP 800-53 control overlays (COSAiS project) for five AI use cases, with two directly addressing agentic deployments, and on April 7, 2026 NIST released a concept note for an AI RMF Profile on Trustworthy AI in Critical Infrastructure. The CoSAI [MCP Security Guide](https://www.coalitionforsecureai.org/securing-the-ai-agent-revolution-a-practical-guide-to-mcp-security/) recommends mandatory code signing for all MCP servers, container sandboxing with gVisor or Kata Containers, and Trusted Execution Environments with remote attestation for high-security deployments. These controls establish the baseline security posture for the environments and processes where models are built.

| **3.4.1** | **Verify that** AI-specific runtime components (agent orchestration services, tool/MCP servers, model registries, and prompt/policy stores) are not shared across environment boundaries (e.g., development, staging, production). | 1 | Lateral movement from dev to prod through shared AI runtime components (MITRE ATLAS [AML.T0010](https://atlas.mitre.org/techniques/AML.T0010)); compromised dev MCP server pivoting to production resources; agent orchestration runtimes in dev being exploited to reach production tool endpoints; shared model registries enabling poisoned model promotion across environments. The Clinejection incident (February 2026) demonstrated agent-mediated lateral movement — a poisoned input traversed a coding agent's tool access to compromise downstream CI/CD infrastructure without new network paths or stolen credentials. Schneider (2026) calls this a third class of pivot distinct from network compromise or credential theft. Josh Woodruff's Agentic Trust Framework (CSA, February 2026) identifies five execution boundaries that agents collapse. Jiang et al. introduce the "Viral Agent Loop" (February 2026) — a model where agents act as vectors for self-propagating worms without exploiting code-level flaws. The [Flowise CVE-2025-59528](https://github.com/FlowiseAI/Flowise/security/advisories/GHSA-3gcm-f6qx-ff7p) CVSS 10.0 RCE (active exploitation from April 7, 2026, 12,000+ exposed instances) runs user-supplied JavaScript in the CustomMCP node with full Node.js privileges — a single shared Flowise instance across environments collapses the dev→prod boundary on first exploit attempt. Palo Alto Unit 42's [Double Agents in Vertex AI](https://unit42.paloaltonetworks.com/double-agents-vertex-ai/) research (March 2026) showed that the P4SA credentials leaked on every Vertex AI Agent Engine invocation granted unrestricted read access to all GCS buckets in the customer project — an agent running in a shared project effectively becomes an insider threat against every other workload there. Microsoft's [CVE-2026-32211](https://cvefeed.io/vuln/detail/CVE-2026-32211) (CVSS 9.1, April 3, 2026) in the Azure MCP Server exposes configuration, API keys, and auth tokens without credentials to anyone with network reachability — a strong argument against a single shared MCP endpoint serving multiple environments. OWASP's [Top 10 for Agentic Applications 2026](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/) ranks Agent Goal Hijack and cascading failure propagation across connected tools/memory/agents as the top risks — shared runtime components are the exact substrate that lets a single compromise cascade. | Enumerate all AI-specific runtime components (orchestration services, MCP servers, model registries, prompt/policy stores) across dev/staging/prod. Verify no shared instances exist using cloud provider tools: AWS Resource Access Manager reports, GCP Organization Policy constraints, Azure Resource Graph queries. Test cross-environment access paths — attempt connections from dev compute to prod MCP endpoints, model registries, and orchestration services. Verify separate service accounts, IAM roles, and secrets per environment. Confirm MCP server manifests are environment-specific with no cross-environment tool registrations. On cloud platforms, verify separate accounts/projects per environment: AWS Organizations with SCPs, GCP VPC Service Controls perimeters, Azure Management Groups. For Vertex AI Agent Engine deployments, confirm P4SA scoping has been tightened per Unit 42's April 2026 guidance and that agent projects are not granted default IAM roles over customer data projects. For Flowise deployments, verify version ≥3.1.1 (or at minimum 3.0.6) and that the management UI is not internet-exposed; run `shodan search flowise http.title:"Flowise"` or equivalent to confirm. Use infrastructure-as-code scanning (Checkov, tfsec) to detect cross-environment resource references. Tools: `nmap` or cloud-native network analyzers for path testing, AWS GuardDuty for runtime anomaly detection, OpenTelemetry for end-to-end MCP interaction tracing, Qualys ETM for agent exposure discovery. | Cloud providers offer strong primitives — separate AWS accounts, GCP projects, Azure subscriptions per environment with IAM boundaries — but the Vertex AI P4SA disclosure (March 2026) shows that default managed-service scoping can silently punch holes through those boundaries: organizations must audit service-agent permissions and not assume isolation based on project separation alone. The practical challenge is AI-specific: MCP servers, agent orchestration runtimes, and model registries are newer infrastructure components that often lack environment-aware deployment patterns. The CoSAI MCP Security Guide recommends container sandboxing with gVisor or Kata Containers for MCP servers interacting with host environments, with Trusted Execution Environments (TEEs) and remote attestation for high-security deployments. Trend Micro found 492 MCP servers exposed to the internet with zero authentication, and the AI Accelerator Institute's review of 281 MCP servers rated 92% as high risk — organizations should maintain separate MCP manifests per environment and audit tool registrations regularly. The OpenClaw crisis showed how shared agent infrastructure amplifies risk: 135,000+ publicly exposed instances lacked basic network segmentation. The AWS Agentic AI Security Scoping Matrix (2026) maps escalating isolation requirements across four autonomy scopes, recommending container-level protection and workflow segmentation even at supervised agency levels, with defense-in-depth at network, application, agent, and data layers. The NIST COSAiS project is developing SP 800-53 control overlays specifically for single-agent and multi-agent AI deployments, with guidance on runtime separation expected in late 2026; NIST's April 7, 2026 AI RMF Critical Infrastructure Profile concept note signals further agentic-specific guidance is imminent. Microsoft's [SDL for AI](https://www.microsoft.com/en-us/security/blog/2026/02/03/microsoft-sdl-evolving-security-practices-for-an-ai-powered-world/) (February 2026) notes that AI systems "collapse traditional trust boundaries" by blending tools, APIs, and agents into single platforms — making strict runtime component isolation more critical than in traditional software. A Gravitee report found 88% of organizations reported AI agent security incidents in the past year. *(Updated 2026-04)* |
| **3.4.2** | **Verify that** AI-specific configuration artifacts (prompt templates, agent policies and routing graphs, tool or MCP contracts and schemas, and action catalogs or capability allow-lists) are stored in version control with change history and require approved review before deployment. | 1 | Unauthorized changes to prompt templates enabling jailbreaks or prompt injection; modified agent routing graphs expanding tool access without approval; poisoned MCP schemas injecting hidden parameters or malicious defaults (Full Schema Poisoning attacks); unreviewed capability allow-list changes granting agents excessive permissions (MITRE ATLAS [AML.T0018](https://atlas.mitre.org/techniques/AML.T0018)). A prompt-injection attack against the GitHub MCP server hijacked an AI assistant to exfiltrate private repository contents, project details, and salary information into a public pull request — demonstrating how a single over-privileged token combined with unreviewed tool configuration enables catastrophic data exposure. Check Point's disclosure of Claude Code [CVE-2025-59536](https://research.checkpoint.com/2026/rce-and-api-token-exfiltration-through-claude-code-project-files-cve-2025-59536/) (CVSS 8.7) made the threat concrete for agentic coding tools: a malicious `.claude/settings.json` hook executed before the trust dialog could render, and two repo-controlled settings (`enableAllProjectMcpServers`, `enabledMcpjsonServers`) auto-approved every MCP server on launch — both are exactly the kind of configuration artifacts that belong under version control with approved review, not in an untrusted repo clone. OWASP's [Top 10 for Agentic Applications 2026](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/) ranks Agent Goal Hijack as the top risk, driven precisely by unreviewed prompts, policies, and tool descriptions. OWASP reports 73% of production AI deployments have exploitable prompt injection vulnerabilities, and only 34.7% of organizations have deployed dedicated defenses. | Inspect version control repositories for all listed artifact types — prompt templates, agent policies, routing graphs, MCP contracts/schemas, and capability allow-lists. Verify branch protection rules require approved review (GitHub CODEOWNERS, required reviewers, GitLab merge request approvals). Check that deployment pipelines pull from version-controlled sources and reject ad-hoc local overrides. Audit commit history for artifacts committed without review approval. Verify MCP schema changes undergo security review for hidden parameter injection. Test that capability allow-list modifications require two-reviewer approval for safety-critical changes. For agentic coding tools (Claude Code, Cursor, etc.), confirm `.claude/settings.json`, `.mcp.json`, and equivalent repo-controlled config are treated as code — reviewed on PR, allowlisted per trusted repository, and blocked from bypassing trust prompts (verify the Claude Code patch for CVE-2025-59536 is installed). Tools: Git with branch protection for version control; DVC for large artifact versioning; Langfuse (open-source prompt CMS with linear versioning, label-based deployment control via "production"/"staging" labels, self-hosted for data residency control); Maxim AI (environment-based deployment with quality gates, SOC 2 Type 2 compliant, ISO 27001 certified, in-VPC deployment with RBAC); MLflow 3.0 Model Registry with staged transitions (Staging → Production → Archived) and approval workflows for linking models to exact prompt configurations; PromptLayer for Git-like version control with visual prompt registry and branch/merge workflows; Weights & Biases Weave for prompt tracing and governance promotion workflows. | Prompt versioning has matured significantly in 2026, with specialized platforms treating prompts as first-class versioned assets alongside code. Langfuse provides self-hosted prompt management with zero-latency caching, detailed tracing tied to specific prompt versions, and full data residency control — suitable for organizations with strict compliance requirements. Maxim AI offers environment-based deployment gates that prevent prompts from advancing to production without passing predefined quality checks, with granular RBAC ensuring only designated team members can promote versions. The Sigstore [Model Transparency](https://github.com/sigstore/model-transparency) project provides cryptographic signing for ML artifacts using an append-only transparency log (Rekor). Full Schema Poisoning (FSP) attacks — where entire MCP tool schema definitions are compromised with hidden parameters, altered return types, or malicious default values — represent an emerging threat that standard code review may not catch. Unit 42 research on [new prompt injection attack vectors through MCP sampling](https://unit42.paloaltonetworks.com/model-context-protocol-attack-vectors/) (2026) demonstrated that malicious instructions embedded in tool descriptions are visible to LLMs but not displayed to users — reviewers inspecting rendered UIs will miss them, so diff-based review against the raw JSON schema is essential. Organizations should require security-focused review of all MCP schema changes, with automated scanning for injection patterns in tool descriptions. The Protect AI + Hugging Face partnership has scanned [4.47 million model versions](https://huggingface.co/blog/pai-6-month) and identified 352,000 unsafe issues across 51,700 models — the same scanning discipline should apply to configuration artifacts. With EU AI Act enforcement beginning August 2026, platforms like MLflow are integrating EU-AI-Act conformity and risk-classification fields directly into the model registry, making reviewed-and-approved promotion a compliance obligation, not just a best practice. NIST [SP 800-218A](https://csrc.nist.gov/pubs/sp/800/218/a/final) (July 2024, final) extends the SSDF with AI-specific practices for artifact management and development lifecycle controls. *(Updated 2026-04)* |
| **3.4.3** | **Verify that** model training and fine-tuning environments are isolated from production model endpoints, agent orchestration services, tool/MCP servers, and live RAG data sources. | 2 | Training pipeline exfiltrating data or model weights to production endpoints; compromised training job pivoting to production agent orchestration services (MITRE ATLAS [AML.T0010](https://atlas.mitre.org/techniques/AML.T0010)); training environments accessing live RAG data sources enabling data poisoning feedback loops; supply chain attacks through uncontrolled package downloads during training. The TeamPCP/LiteLLM attack (March 24, 2026) demonstrated cascading supply chain compromise in vivid detail: attackers first poisoned Trivy (pulled unpinned from `apt` in LiteLLM's own CI), stole the `PYPI_PUBLISH` token from the GitHub Actions runner, and shipped LiteLLM versions 1.82.7 and 1.82.8 (the package draws ~3.4M daily downloads and 95M+ monthly) containing a malicious `litellm_init.pth` file that auto-executes on every Python process startup — any environment with the poisoned LiteLLM installed had SSH keys, cloud credentials, Kubernetes secrets, crypto wallets, and `.env` files exfiltrated, privileged pods deployed across every cluster node, and a persistent systemd backdoor installed ([Datadog Security Labs](https://securitylabs.datadoghq.com/articles/litellm-compromised-pypi-teampcp-supply-chain-campaign/)). Training environments that share Python base images with production or pull packages from unverified mirrors convert this into a dev→prod pivot. IBM X-Force 2026 reports supply chain compromises up nearly 4x since 2020. A significant emerging risk is staging and dev environments with RAG indexes pointing to live production embeddings — this creates "shadow data waiting to be queried" that collapses the isolation boundary. | Review network policies for training environments. Verify no network paths exist from training compute to production model endpoints, agent orchestration services, MCP servers, or live RAG data sources. Confirm training RAG indexes use synthetic or snapshot data, not live production embeddings. Test by attempting outbound connections from training compute to production endpoints. Verify package installation sources are pinned to trusted mirrors with integrity verification (pip `--require-hashes`, `uv` with lockfiles, private PyPI mirrors with quarantine on newly published versions — LiteLLM 1.82.8 was live on PyPI for ~3 hours before quarantine, so near-real-time pinning matters). On GCP, verify VPC Service Controls perimeters block default internet access from Vertex AI training jobs, and after the Unit 42 "Double Agent" disclosure confirm Vertex AI Agent Engine P4SA permissions have been manually tightened. Tools: `nmap` for network path testing, cloud-native network analyzers, Kubernetes NetworkPolicy audit, `pip-audit` and `osv-scanner` for dependency CVE surfacing, Sigstore `cosign` for container and model signature verification. Validate staging environments use separate vector database instances from production. Check that model serving endpoints in training use mock or canary configurations, not production inference APIs. Use OPA (Open Policy Agent) to enforce environment-specific deployment rules blocking artifact promotion without proper signatures and security evaluations. | Cloud-native network policies support isolation well: VPC Service Controls (GCP), Azure Private Link, AWS PrivateLink. GCP Vertex AI training environments block default internet access when enclosed in a VPC SC perimeter, requiring explicit egress configuration — but the Unit 42 Double Agent disclosure (March 2026) shows network isolation alone is not enough when managed agent service accounts retain excessive default IAM roles. The difficulty is that RAG data source isolation is a newer concern — many organizations share vector databases across environments for cost or convenience, creating unintentional data paths between training and production. The Unit 42 RCE disclosures (December 2025) showed NeMo's Hydra `instantiate()` vulnerability existed since at least 2020 — NVIDIA's fix uses allowlist-based validation checking all `_target_` values against permitted package prefixes, a pattern applicable to any Hydra-based ML pipeline. The "nullifAI" technique (February 2025) demonstrated that malicious ML models on Hugging Face can evade PickleScan detection using 7z compression — three zero-day [PickleScan vulnerabilities](https://jfrog.com/blog/unveiling-3-zero-day-vulnerabilities-in-picklescan/) were disclosed by JFrog (June 2025) and fixed in PickleScan v0.0.31, underscoring that training environments pulling models from external registries must enforce format restrictions and scanning at ingress. Kubeflow Pipelines provides namespace isolation and RBAC for multi-user training environments on Kubernetes. The [PickleBall](https://dl.acm.org/doi/10.1145/3719027.3765037) framework (ACM CCS 2025) statically analyzes ML library source code to compute safe deserialization policies, loading 79.8% of benign models while blocking all malicious payloads with ~2.62% runtime overhead. Confidential computing environments where data remains encrypted during processing can satisfy HIPAA, GDPR, and SOC 2 requirements for training data protection. The NIST AI Agent Standards Initiative (February 2026) emphasizes separating development, test, and production agent environments with just-in-time privilege and policy-based authorization, and the April 7, 2026 AI RMF Critical Infrastructure Profile concept note signals more prescriptive isolation guidance is in development. *(Updated 2026-04)* |
