# C8.1: Access Controls on Memory & RAG Indices

> [Back to C08 Index](C08-Memory-and-Embeddings.md)
> **Last Researched:** 2026-06-20

## Purpose

Enforce fine-grained access controls and query-time scope enforcement for every memory store, vector collection, retrieval index, and semantic cache. A RAG system should never retrieve content merely because it is semantically close to the question; it should retrieve content only when the caller, agent, service principal, and current task are authorized to see it.

The recurring failure mode is permission stripping during ingestion. Connectors for SharePoint, Confluence, file shares, ticket systems, and object stores often convert source documents into chunks and embeddings while dropping the source ACL, writer identity, sensitivity label, or revocation state. Once that happens, the vector store has no reliable basis for enforcing access at query time, and the application is left to guess or filter after retrieval. That is why C8.1 treats provenance, immutable metadata, retrieval logging, canary records, and anomaly detection as part of access control rather than as separate observability concerns.

As of May 2026, the practical implementation pattern is clearer than it was a year ago: store authorization metadata with every chunk, inject the caller's scope server-side, push filters into the vector/search engine whenever possible, and keep post-retrieval checks as a backup rather than the primary control. Azure AI Search now documents native document-level access control patterns using Microsoft Entra tokens, POSIX-like ACL/RBAC scopes, SharePoint ACLs, and Purview sensitivity labels. The 2025-05-01-preview REST API introduced native ACL/RBAC enforcement, and 2025-11-01-preview added the explicit `x-ms-elevated-read` header so that troubleshooting paths must be opted into instead of accidentally bypassing ACLs. Databricks Mosaic AI Vector Search is governed through Unity Catalog and explicit vector-index privileges, but native row-level and column-level permissions are still not supported on vector indexes, so multi-tenant deployments have to enforce per-row scope through the filter API in the application path. Pinecone namespaces, Weaviate tenants, and Qdrant payload indexes are still useful isolation primitives, but they need server-side enforcement and regression tests so a missing filter cannot silently become a cross-tenant leak.

The threat landscape also keeps pushing access control toward provenance and monitoring. OWASP LLM08:2025 calls out unauthorized vector access, cross-context leakage, poisoning, and weak retrieval monitoring. MITRE ATLAS (latest release 2026.05, published 2026-05-27) separates RAG Poisoning (AML.T0070) from RAG database collection (AML.T0085.000), and the discrete False RAG Entry Injection technique (AML.T0071) is a useful reminder that adversaries deliberately forge the title, author, and creation-date metadata of injected entries so they read as trusted — which is exactly why 8.1.2's immutable-metadata requirement is an access-control concern, not just a hygiene one. Defenders need both write-path integrity controls and query-path data leakage detection. PoisonedRAG showed that five malicious texts can drive a 90% attack success rate in large corpora; the May 2025 One-Shot Dominance / AuthChain work pushed that down to a single document with high stealth against existing RAG defenses. The April 2026 Black-Hole Attack showed centroid-positioned vectors appearing in up to 99.85% of top-10 results. Semantic caches add another retrieval surface: the January 2026 Key Collision Attack showed that embedding-based cache keys behave like fuzzy hashes and can be collided for response hijacking. Access controls therefore need to cover insert, update, delete, query, retrieval logging, cache hit paths, restricted-index canaries, and incident replay. The detection side finally has packaged tooling: in February 2026 Cisco's AI Defense team released HubScan, an open-source scanner that finds adversarial hubs — vectors engineered to appear in top-k results for a disproportionate share of queries — across FAISS, Pinecone, Qdrant, and Weaviate indexes.

The connector and copilot incidents over the past year keep validating these priorities. Microsoft's January-February 2026 Copilot Chat bug (CW1226324) processed emails marked `Confidential` and `Highly Confidential` from Sent Items and Drafts despite Purview sensitivity labels and DLP policies being correctly configured. It was the second sensitivity-label bypass in eight months, after the June 2025 EchoLeak (CVE-2025-32711) zero-click exfiltration. Concentric AI's 2026 data risk reporting still shows roughly 16% of business-critical data overshared, averaging around 800,000 at-risk files per organization, which makes connector-stage ACL propagation and document-level filters the most leveraged controls in any enterprise RAG rollout.

## Implementation Notes

**Default-deny retrieval path.** Treat vector retrieval as a privileged database read. The caller's tenant, role, data classification clearance, source-system permissions, and task scope should be computed server-side and injected into the query. The client should never be allowed to choose its own authorization filter.

**Source ACL propagation.** For connector-based ingestion, preserve the source object's stable ID, ACL or policy reference, sensitivity label, owner, and revocation state before chunking. If the connector cannot provide reliable ACLs, route that content to a separate low-trust index rather than mixing it into the enterprise retrieval pool.

**Pre-filter and post-filter tradeoff.** Pre-filtering keeps unauthorized content out of vector ranking and model context, but it depends on accurate metadata and filter support. Post-filtering is useful as a defense-in-depth check, but over-fetching can still reveal side channels through counts, scores, latency, and empty-answer patterns. Log both allowed and denied candidate counts.

**Restricted-index canaries.** Place synthetic canary chunks in every restricted retrieval scope and treat any retrieval, citation, cache hit, or prompt-context inclusion as a high-confidence incident. The canary should identify its owning scope through security metadata, not through real secrets in the text. Exercise canaries from denied users, overbroad service principals, and cache-hit paths so the alert path is tested as often as the authorization path.

**Regression test set.** Maintain tenant canary documents, revoked documents, high-sensitivity documents, poisoned test chunks, semantic-cache collision probes, and stable benchmark queries. Run the set after connector changes, vector DB upgrades, embedding model changes, policy changes, and cache tuning.

**Incident response.** Quarantine must operate by source ID, batch ID, embedding model version, and content hash, not only by vector ID. During a poisoning incident, responders need to remove or suppress every chunk derived from the bad source and then replay retrieval logs to identify affected users and responses.

## Notable Incidents and Research

- January-February 2026: Microsoft's [Copilot Chat CW1226324 advisory](https://venturebeat.com/security/microsoft-copilot-ignoring-sensitivity-labels-dlp-cant-stop-ai-trust-failures) covered a retrieval-path bug that pulled emails from Sent Items and Drafts into the assistant's context even when the messages carried Purview `Confidential` or `Highly Confidential` labels and matching DLP policies. It was the second sensitivity-label bypass in eight months, after June 2025 [EchoLeak (CVE-2025-32711)](https://msrc.microsoft.com/update-guide/vulnerability/CVE-2025-32711), and confirms that document-level ACLs and DLP have to be enforced by trusted server code at every retrieval entrypoint, not relied on as a single perimeter.
- February 2026: [Milvus CVE-2026-26190](https://github.com/milvus-io/milvus/security/advisories) (critical, fixed in 2.5.27+ / 2.6.10+) and the earlier [CVE-2025-64513](https://nvd.nist.gov/vuln/detail/CVE-2025-64513) proxy-path bypass are reminders that vector-database control planes are themselves an authorization surface. Operators should pin patched versions, monitor advisories, and audit metrics/REST exposure on internal networks.
- May-June 2026: two new vector-database control-plane bypasses landed close together. [ChromaDB CVE-2026-45829 ("ChromaToast")](https://www.securityweek.com/unpatched-chromadb-vulnerability-can-lead-to-server-takeover/) is a pre-authentication RCE — the server downloads and executes a client-supplied HuggingFace model identifier before checking credentials, affecting every release from 1.0.0 and still unpatched as of 1.5.8, with roughly 73% of internet-exposed instances vulnerable. [Weaviate CVE-2026-11500](https://vulnerability.circl.lu/vuln/cve-2026-11500) (affects 1.37.0-1.37.7, fixed in 1.38.0-rc.0) bypasses the static API-key authorization handler through a manipulated `StaticApiKey` argument. Both are reminders that a takeover of the vector store collapses every tenant and namespace boundary built on top of it, so the control plane needs its own patch cadence and network isolation.
- May-June 2026: poisoning research moved toward pipeline-realistic and metadata-only attacks. ["When Poison Fails After Retrieval"](https://arxiv.org/abs/2606.11265) (June 2026) showed that earlier corpus-poisoning attacks degrade sharply once realistic chunking and reranking are in the loop, and introduced a consistency-optimized framework (CRCP) that keeps poison effective end-to-end — so verification fixtures should include a reranker, not just a bare retriever. [SilentRetrieval](https://arxiv.org/abs/2605.28074) (May 2026) keeps poisoned documents fluent and low-perplexity via coordinated beam search before embedding attacker-chosen triggers, and ["A Wolf in Sheep's Clothing"](https://arxiv.org/abs/2605.28112) (May 2026) hijacks federated-RAG routing by forging semantic profiles to attract queries a malicious client has no legitimate data for — a cross-client scope failure relevant to 8.1.1.
- February 2026: Cisco AI Defense released [HubScan](https://github.com/cisco-ai-defense/adversarial-hubness-detector), an Apache-2.0 scanner for adversarial hubs in RAG vector indexes, with the accompanying [hubness-poisoning paper](https://arxiv.org/abs/2602.22427) reporting 90% recall at a 0.2% alert budget and full recovery of domain-targeted attacks that escaped global detection across Food-101, MS-COCO, FiQA, and 1M MS MARCO web documents. First packaged tooling for the C8.1.7 control.
- March 2026: [Semantic Chameleon](https://arxiv.org/abs/2603.18034) demonstrated corpus-dependent poisoning crafted to match the statistical profile of the target corpus, directly targeting the perplexity, duplication, and outlier signals most anomaly-filtering defenses rely on. A reminder that 8.1.7 detectors need adversarial re-testing, not one-time validation.
- April 2026: [RAGShield](https://arxiv.org/abs/2604.00387) proposed a five-layer defense-in-depth framework that treats RAG knowledge bases as a software supply chain: C2PA-style cryptographic document attestation, trust-weighted retrieval, a formal taint lattice with contradiction detection, provenance-aware generation with auditable citations, and a NIST SP 800-53 mapping across 15 control families. Reported 0% attack-success rate against tested poisoning tiers (including adaptive attacks) on the evaluated benchmark.
- April 2026: the [Securing RAG taxonomy survey](https://arxiv.org/abs/2604.08304) organized known attacks and defenses across six workflow stages and three trust boundaries, and concluded that current defenses are "largely reactive and fragmented." Useful as a structured checklist when reviewing existing pipelines for missing controls at ingestion, retrieval, generation, and post-retrieval stages.
- December 2025: [RAGPart and RAGMask](https://arxiv.org/abs/2512.24268) introduced retrieval-stage defenses that fragment documents and aggregate across multiple retrievers (RAGPart) or use targeted token masking to flag suspicious similarity shifts (RAGMask). Both operate on the retriever, are computationally lightweight, and consistently reduce attack-success rates across four poisoning strategies and four retrievers without modifying the generation model.
- September 2025 (NeurIPS 2025): [ReliabilityRAG](https://arxiv.org/abs/2509.23519) introduced a graph-theoretic defense that runs maximum-independent-set analysis over a contradiction graph of retrieved documents, with provable robustness guarantees under bounded adversarial corruption — one of the few defenses with formal guarantees rather than benchmark-only results.
- May 2025: [One-Shot Dominance (AuthChain)](https://arxiv.org/abs/2505.11548) demonstrated that a single poisoned document can dominate retrieval and bypass common RAG defenses across six LLMs, including multi-hop queries. Lifts the urgency of single-document poisoning tests in any C8.1 verification fixture.
- 2026 enterprise data: Concentric AI's [2026 Data Risk Report](https://concentric.ai/too-much-access-microsoft-copilot-data-risks-explained/) and the [EPC Group Copilot oversharing guidance](https://www.epcgroup.net/copilot-sharepoint-permissions-oversharing-fix-2026) report roughly 16% of business-critical data overshared and around 800,000 at-risk files per organization. Connector-stage ACL propagation, SharePoint site cleanup, and sensitivity-label remediation should run before any Copilot or RAG production cutover.
- Earlier 2025-2026 research that remains directly relevant: [PoisonedRAG](https://www.usenix.org/conference/usenixsecurity25/presentation/zou-poisonedrag) (five documents, 90% ASR), [Black-Hole Attack](https://arxiv.org/abs/2604.05480) (centroid-positioned vectors in up to 99.85% of top-10 results), the [Key Collision Attack on semantic caching](https://arxiv.org/abs/2601.23088), [RevPRAG](https://aclanthology.org/2025.findings-emnlp.698/) for activation-based poisoned-response detection, and [RAGForensics](https://doi.org/10.1145/3696410.3714756) for traceback.

## Implementation Maturity

- Mature: tenant isolation at the namespace, collection, or sharded-tenant level is well supported by Pinecone, Weaviate, Qdrant, Milvus, and pgvector. The hard part is the application contract that injects scope filters and prevents client-supplied authorization.
- Mature, with one preview caveat: query-time document-level ACL enforcement on managed search platforms. Elastic's document-level security and OpenSearch fine-grained access control are production-ready. Azure AI Search's native ACL/RBAC enforcement is heavily documented and shipping, but as of the `2026-05-01-preview` REST API it is still preview-only (no GA/stable API version exists yet), so production teams should track the GA timeline and validate the `x-ms-enable-elevated-read` break-glass path in audit. The 2026-05-01-preview release extended enforcement to SharePoint site-group memberships (Owners/Members/Visitors and custom groups, carried as `spg:`-prefixed values in `groupIds`) at query time.
- Developing: vector-database governance with row- and column-level permissions. Databricks Mosaic AI Vector Search inherits Unity Catalog object-level permissions but still requires application-level filter expressions for per-document scope; the documentation still states plainly (as of its June 2026 revision) that row- and column-level permissions are not supported and that teams must implement their own application-level ACLs through the filter API. Weaviate's role-based access control, by contrast, moved out of technical preview to GA in the v1.29 line with further granularity in v1.30, so native RBAC plus per-tenant shards is now a supported combination there.
- Developing: provenance and immutable metadata. Append-only metadata stores, signed manifests, content hashes, and per-chunk source-ACL snapshots are achievable today, but cryptographic document attestation (RAGShield, C2PA-style) is still research-grade for general RAG.
- Developing: relationship-based authorization for retrieval. SpiceDB (Authzed), OpenFGA, and Cerbos have documented LangChain/Pinecone integration paths and increasing production deployment. Authzed's `langchain-spicedb` library (released March 2026) packages this further, shipping a `SpiceDBAuthFilter` for post-retrieval permission checks in LCEL chains, a LangGraph authorization node, and a `SpiceDBPermissionTool` for agents, using bulk permission APIs so a 100-document batch costs a single network call. It is vector-store agnostic, but the per-document scope-to-filter mapping still has to be designed deliberately rather than assumed.
- Early: retrieval-anomaly detection for poisoning. HubScan (February 2026) is the first packaged open-source scanner and covers FAISS, Pinecone, Qdrant, and Weaviate, but it is an offline/scheduled scan rather than a runtime control. RAGPart, RAGMask, ProGRank, ReliabilityRAG, RevPRAG, and RAGForensics round out retrieval- and generation-stage defenses, yet all remain benchmark-validated rather than turnkey, and adaptive corpus-matched attacks already target their detection signals.
- Early: canary-hit alerting. No vector database emits native canary alerts; Canarytokens and canari-llm cover document/URL and prompt-injection canaries, and Splunk's MITRE ATLAS app provides ATLAS-mapped SIEM rules, but the similarity-match alert path still has to be hand-built into every retrieval entrypoint.
- Early: connector ACL propagation for SharePoint, Confluence, Drive, ticket systems, and email. The 2026 Microsoft Copilot incidents show that even well-resourced vendors still ship retrieval paths that bypass labels, so verification has to test each connector independently rather than relying on platform claims.
