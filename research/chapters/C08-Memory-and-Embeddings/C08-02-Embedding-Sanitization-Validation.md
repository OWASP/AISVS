# C8.2: Embedding Sanitization & Validation

> [Back to C08 Index](C08-Memory-and-Embeddings.md)
> **Last Researched:** 2026-05-14

## Purpose

Pre-screen content before vectorization; treat memory writes as untrusted inputs; prevent ingestion of unsafe payloads. Embedding ingestion is a durable trust boundary: once text, metadata, or generated memory is converted into vectors, it can influence many future retrievals and is difficult to redact with confidence. As of May 2026, the practical attack surface has expanded from generic corpus poisoning into loader-stage obfuscation, retrieval-optimized indirect prompt injection, and persistent memory poisoning. [OWASP LLM08:2025](https://genai.owasp.org/llmrisk/llm082025-vector-and-embedding-weaknesses/) calls out vector data leakage, cross-context knowledge conflict, embedding inversion, and data poisoning as core risks. Recent work reinforces the point: [PoisonedRAG](https://www.usenix.org/conference/usenixsecurity25/presentation/zou-poisonedrag) showed high-success knowledge corruption from a few injected texts; [The Hidden Threat in Plain Text](https://arxiv.org/abs/2507.05093) showed DOCX, HTML, and PDF loader manipulations surviving into RAG pipelines; [Overcoming the Retrieval Barrier](https://arxiv.org/abs/2601.07072) showed retrieval-optimized indirect prompt injection under natural queries; and Microsoft's February 2026 [AI Recommendation Poisoning](https://www.microsoft.com/en-us/security/blog/2026/02/10/ai-recommendation-poisoning/) disclosure showed real-world attempts to plant persistent recommendations through hidden prompt parameters. The April 2026 [Cloud Security Alliance research note](https://labs.cloudsecurityalliance.org/research/csa-research-note-indirect-prompt-injection-in-the-wild-2026/) and [OWASP GenAI Q1 2026 exploit round-up](https://genai.owasp.org/2026/04/14/owasp-genai-exploit-round-up-report-q1-2026/) show the ingestion layer is now an operational target, with hidden web content, log entries, and tool inputs used for exfiltration and unsafe actions. Mature controls exist for PII detection and basic content filtering, but outlier detection, contradiction review, and memory-write authorization still need corpus-specific baselines, provenance, quarantine workflows, and human review for high-impact stores. As of May 2026, [OWASP Agent Memory Guard](https://owasp.org/www-project-agent-memory-guard/) is useful to track as an incubator implementation for central memory read/write gates: it validates integrity with hashes, detects injection and sensitive-data leakage, enforces YAML policies, snapshots memory, and supports rollback, while its own roadmap still treats vector-store protection and broader framework coverage as developing work.

| **8.2.2** | **Verify that** content intended to poison retrieval (e.g., text crafted to project to attacker-chosen embedding neighborhoods, hidden instructions intended for downstream model context, or steganographic payloads in non-text inputs) is detected and rejected or quarantined before vectorization. | 1 | RAG poisoning, indirect prompt injection, and loader-stage content manipulation. Attackers can hide instructions in HTML, PDF, DOCX, comments, CSS, white text, Unicode confusables, OCR layers, metadata, or crafted text that is optimized to be retrieved for target queries. [The Hidden Threat in Plain Text](https://arxiv.org/abs/2507.05093) reported a 74.4% attack success rate across 357 loader attack scenarios using content obfuscation and content injection against common document loaders. [Hidden-in-Plain-Text](https://arxiv.org/abs/2601.10923) introduced OpenRAG-Soc for web-native RAG attacks that survive HTML, Markdown, Unicode, and social-content ingestion paths. [Overcoming the Retrieval Barrier](https://arxiv.org/abs/2601.07072) showed that trigger fragments can make malicious content reliably retrieved under natural queries, and [Machine Against the RAG](https://www.usenix.org/conference/usenixsecurity25/presentation/shafran) showed that a single blocker document can suppress answers without relying on explicit instruction injection. April 2026 operational reporting from the [Cloud Security Alliance](https://labs.cloudsecurityalliance.org/research/csa-research-note-indirect-prompt-injection-in-the-wild-2026/) connects the same hidden-content patterns to live payloads against browsing agents, coding assistants, and enterprise copilots. MITRE ATLAS tracks the related families as [AML.T0051 Prompt Injection](https://atlas.mitre.org/techniques/AML.T0051), [AML.T0070 RAG Poisoning](https://atlas.mitre.org/techniques/AML.T0070), and [AML.T0080 AI Agent Context Poisoning](https://atlas.mitre.org/techniques/AML.T0080). | Build an intake test suite for every accepted format. Compare raw text extraction, rendered view, OCR output, metadata, links, alt text, comments, and hidden layers before embedding. Normalize Unicode, strip or flag zero-width characters, reject invisible or off-canvas text, limit embedded links and executable content, and quarantine chunks containing instruction-like text aimed at the downstream model. Test with fixture documents that include hidden prompts, white-on-white text, tiny fonts, HTML comments, PDF annotation layers, DOCX comments, homoglyphs, base64-like payloads, suspicious URL prompt parameters, adversarial repetitions of target-query terms, and OpenRAG-Soc-style web markup carriers. For application code, verify that LangChain, LlamaIndex, Haystack, or custom loaders pass through a security transformation before vector upsert. Use [LangChain's indirect prompt injection guidance](https://docs.langchain.com/oss/python/langchain/rag), [NVIDIA NeMo Guardrails retrieval rails](https://docs.nvidia.com/nemo/guardrails/latest/about/rail-types.html), custom prompt-injection classifiers, static document scanners, and manual review queues for high-impact collections. | No parser-level defense is complete because attackers can target differences between raw extraction, rendered display, OCR, and model interpretation. Defensive prompts and delimiters help at generation time but do not sanitize a poisoned vector store. Treat sanitization as a gate: untrusted documents should not be embedded until the original file, extracted text, metadata, and source provenance have passed policy checks. Live web payloads increasingly use ordinary fields such as comments, attributes, CSS-suppressed text, and encoded fragments, so tests need to cover real loader behavior rather than only clean plaintext chunks. Mature commercial tooling for document-intake poisoning is still emerging, so teams should keep adversarial loader fixtures in CI and require human review for quarantine releases. |
| **8.2.3** | **Verify that** vectors that fall outside normal clustering patterns are flagged and quarantined before entering production indices. | 2 | Adversarial embedding injection, vector collision, and high-centrality poisoning. A poisoned chunk may be crafted to land near attacker-chosen query neighborhoods or to be retrieved broadly across unrelated questions. [PoisonedRAG](https://www.usenix.org/conference/usenixsecurity25/presentation/zou-poisonedrag) achieved strong results by injecting a small number of malicious texts into large knowledge bases, while [CorruptRAG](https://arxiv.org/abs/2504.03957) focuses on practical single-text poisoning. [Hidden in the Metadata](https://arxiv.org/abs/2603.00172) extends the risk to multimodal RAG, showing metadata-only poisoning can steer retrieval even when the visible image content is unchanged. The April 2026 [Black-Hole Attack](https://arxiv.org/abs/2604.05480) adds a hubness-specific failure mode: vectors placed near the embedding-space centroid can become top-k neighbors for unrelated queries, so a poison record may look like an ordinary central vector rather than an outlier. Retrieval-optimized attacks mean the vector may look linguistically or visually plausible while still being geometrically or metadata-wise suspicious. | Establish baseline distributions per collection, tenant, source type, language, media type, metadata schema, and embedding model version. Before production upsert, score each candidate chunk against nearest-neighbor distance, local density, centroid distance, duplicate similarity, high-degree retrieval frequency, reverse-nearest-neighbor count, source reputation, metadata drift, and caption/OCR consistency. Test with synthetic poisoned documents targeting known canary queries and confirm quarantine triggers before the production index changes. For multimodal collections, compare extracted text, captions, alt text, EXIF-like metadata, OCR output, and source-system fields before embedding. Useful methods include isolation forest, DBSCAN or HDBSCAN, robust z-score on cosine distance, cleanlab-style out-of-distribution scoring, cross-encoder reranking disagreement, hubness scoring, and similarity search against a malicious-pattern fixture set. Verify that vector stores such as Qdrant, Milvus, Weaviate, Pinecone, Vespa, or OpenSearch can support pre-commit staging collections and audit logs for rejected vectors. | Outlier detection catches geometric anomalies but misses semantic poisoning that sits inside a normal cluster. Centroid-near vectors can be dangerous specifically because they are central rather than distant, so quarantine rules should not rely only on far-out distance thresholds. Metadata-only attacks may leave the user-visible artifact clean, so reviewers need to inspect the fields that actually feed the embedding and retriever. New product documentation, legal updates, incident reports, and rare languages may also look anomalous. Baselines must be segmented and versioned; a single global threshold usually creates both false positives and blind spots. Emerging defenses such as [RevPRAG](https://aclanthology.org/2025.findings-emnlp.698/), [Sparse Document Attention RAG](https://arxiv.org/abs/2602.04711), and [RAGForensics](https://arxiv.org/abs/2504.21668) are useful complementary signals, but they do not replace pre-ingestion quarantine and source review. |
| **8.2.4** | **Verify that** an agent's own outputs are not automatically written back into its trusted memory without explicit validation (such as content-origin checks or write-authorization controls that verify the content's source before committing writes). | 2 | Persistent memory poisoning and self-reinforcing corruption. If generated summaries, tool results, or model conclusions are written into trusted memory without validation, one successful injection can become durable context for later sessions. Microsoft's February 2026 [AI Recommendation Poisoning](https://www.microsoft.com/en-us/security/blog/2026/02/10/ai-recommendation-poisoning/) research observed more than 50 prompt-based attempts from 31 companies across 14 industries that tried to plant trusted-source recommendations through prefilled assistant URLs, mapped to MITRE ATLAS [AML.T0051](https://atlas.mitre.org/techniques/AML.T0051) and [AML.T0080.000](https://atlas.mitre.org/techniques/AML.T0080.000). [ER-MIA](https://arxiv.org/abs/2602.15344) frames similarity-based long-term memory retrieval itself as a system-level vulnerability, with content-based and question-targeted attacks that persist across later interactions. [A-MemGuard](https://arxiv.org/abs/2510.02373) describes why isolated memory-entry review is weak: malicious effects may activate only in a later context and then reinforce themselves through future writes. | Enumerate every path that can write to long-term memory, vector stores, profiles, summaries, preference stores, and shared agent state. Confirm each write records origin, author, source URI, source trust tier, extraction method, review status, timestamp, tenant, and approving principal. Block auto-promotion of generated content into trusted memory unless a separate validator checks source provenance, policy compliance, contradiction risk, and write authorization. Test with prompts and retrieved documents that ask the system to remember a vendor, override a policy, promote a source as authoritative, persist a hidden instruction, or insert a memory that only activates under a later question. Include content-based and question-targeted memory injection fixtures, then verify they remain quarantined across session restart, compaction, summarization, and retrieval. Review logs for memory writes containing terms such as "remember", "trusted source", "from now on", "future conversations", "always cite", and "authoritative". Require human approval or policy-engine approval for high-impact memory writes, and evaluate memory middleware such as [OWASP Agent Memory Guard](https://owasp.org/www-project-agent-memory-guard/) for policy enforcement, snapshotting, rollback, and protected-key tamper detection. | Product memory features often blur preferences, summaries, user-authored facts, and model-inferred facts. That makes origin tagging and trust tiers more important than the model's confidence. Validation should run outside the same prompt context that produced the candidate memory. A-MemGuard-style consensus validation, dual-memory lessons, and Agent Memory Guard-style read/write interception are promising, but production teams still need explicit write permissions, reviewable memory ledgers, rollback, and per-user or per-tenant scoping. |
| **8.2.5** | **Verify that** tool outputs and orchestration results are not automatically written to agent memory without the same source validation and write-authorization controls applied to the agent's own outputs. | 2 | Tool-result poisoning, orchestration-result poisoning, and durable contamination of agent memory. A compromised MCP server, browser page, search result, code-analysis output, email, ticket, or nested agent result can carry hidden instructions or false claims that look like normal operational data. If that output is summarized into memory, promoted into a vector store, or saved as a "successful" trajectory, the compromise can survive the original run. [OWASP MCP Tool Poisoning](https://owasp.org/www-community/attacks/MCP_Tool_Poisoning) describes this runtime trust gap for tool responses; the latest [MCP tool specification](https://modelcontextprotocol.io/specification/2025-11-25/server/tools) requires servers to sanitize tool outputs, supports output schemas for structured results, and recommends that clients validate tool results before passing them to the model. The same pattern maps to MITRE ATLAS [AML.T0051 Prompt Injection](https://atlas.mitre.org/techniques/AML.T0051), [AML.T0080 AI Agent Context Poisoning](https://atlas.mitre.org/techniques/AML.T0080), and [AML.T0080.000 Memory](https://atlas.mitre.org/techniques/AML.T0080.000). Recent memory work makes the persistence risk concrete: [MemoryGraft](https://arxiv.org/abs/2512.16962) implants malicious successful experiences into long-term memory, and April 2026 [eTAMP](https://arxiv.org/abs/2604.02623) showed cross-session web-agent compromise from a single contaminated environmental observation. | Inventory every memory write path that can receive tool or orchestration output: MCP tools, browser/search connectors, email and ticketing tools, code execution, retrieval pipelines, delegated agents, workflow summaries, and background compaction jobs. For each path, verify that writes are denied by default unless the candidate memory has source URI, tool/server identity, tool version or descriptor hash, invocation ID, user and tenant, timestamp, content hash, trust tier, sensitivity classification, and approving principal. Test with malicious tool fixtures that return valid-looking JSON plus hidden prompt text, poisoned webpage observations, forged "trusted source" claims, and nested-agent summaries that ask to be remembered. Confirm schema validation rejects unexpected fields, structured MCP outputs conform to the advertised output schema, free-text directives are quarantined, and high-impact writes pause for approval outside the model context. Useful implementation hooks include [OpenAI Agents SDK tool guardrails](https://openai.github.io/openai-agents-python/guardrails/), [OpenAI hosted MCP approval flows](https://openai.github.io/openai-agents-python/mcp/), [LangGraph long-term stores](https://docs.langchain.com/oss/python/langchain/long-term-memory) wrapped by application authorization, LangGraph persistence and interrupts for human review, and [NVIDIA NeMo Guardrails execution rails](https://docs.nvidia.com/nemo/guardrails/0.19.0/user-guides/configuration-guide/guardrails-configuration.html). For OpenAI Agents SDK deployments, verify tool guardrails cover custom function tools and separately gate hosted tools, MCP tools, handoffs, shell or browser tools, and any background memory writer that does not pass through the custom function-tool guardrail pipeline. Review logs to ensure rejected write attempts preserve the raw tool result, normalized candidate memory, validation decision, and reviewer action. | Tool outputs are often designed to be consumed by the model, not stored as durable facts, so memory-write policy has to be stricter than ordinary context-injection policy. Schema validation helps but does not prove a claim is true, and a valid tool response may still contain adversarial natural language in a legitimate field. Hosted tools, handoffs, browser observations, and background summarizers may bypass the same guardrail hooks used for custom function tools, so teams need a central memory-write service rather than scattered `store.put()` calls. [NIST AI 600-1](https://www.nist.gov/publications/artificial-intelligence-risk-management-framework-generative-artificial-intelligence) reinforces the need for provenance tracking, third-party monitoring, and incident response for GAI value-chain components; in practice, this means reviewable memory ledgers, rollback by batch or source, per-tenant scoping, and regular red-team tests for tool-to-memory poisoning. |
| **8.2.6** | **Verify that** new content written to memory is checked for contradictions with what is already stored and that conflicts trigger alerts. | 3 | Knowledge-base corruption, slow-burn poisoning, and cross-context knowledge conflict. Attackers may introduce plausible but conflicting facts over time so future retrieval selects the poisoned version. OWASP LLM08 identifies cross-context information leaks and federation knowledge conflict as vector and embedding risks, and Microsoft's AI Recommendation Poisoning disclosure shows that persistent "trusted source" claims can bias later recommendations. Contradiction checks help detect poisoned updates before they become accepted memory. | For each new memory write or knowledge-base update, extract claims and compare them with authoritative sources, prior approved memories, policy documents, and recent conflicting chunks. Use natural-language inference models, semantic similarity plus polarity checks, knowledge-graph triples, source-of-truth precedence rules, and reviewer workflows. Test by ingesting documents that contradict known policies, dates, owners, entitlements, prices, safety instructions, or source trust rankings. Verify that conflicts create alerts, quarantine the new content, preserve both versions for investigation, and avoid silent merging. Confirm that rollback can remove a poisoned batch from the vector index, document store, caches, and generated summaries. | Automated contradiction detection remains an incomplete NLP control. It works best for explicit factual conflicts and performs poorly on subtle framing, omissions, stale-but-once-true facts, and domain-specific policies. Full pairwise comparison does not scale, so practical systems compare against curated golden sources, high-trust memories, and topically nearest approved chunks. Alerts should include provenance and retrieval impact, not just a model-generated contradiction label. |

## Notable Incidents and Research

- February 2026: Microsoft reported AI Recommendation Poisoning attempts where public "Summarize with AI" links used prompt parameters to plant persistent trusted-source recommendations in assistant memory, including more than 50 observed prompt patterns from 31 companies across 14 industries. Security teams should hunt for assistant URLs with `q` or `prompt` parameters containing memory terms such as "remember", "trusted", "authoritative", "future", "citation", or "cite". [Microsoft Security](https://www.microsoft.com/en-us/security/blog/2026/02/10/ai-recommendation-poisoning/)
- March 2026: MM-MEPA showed that multimodal RAG poisoning can live entirely in image-text metadata while leaving visible image content unchanged, which makes metadata provenance and caption/OCR consistency checks part of embedding sanitization rather than a separate multimedia concern. [Hidden in the Metadata](https://arxiv.org/abs/2603.00172)
- April 2026: eTAMP showed that a single contaminated web observation can poison a browsing agent's long-term memory and later activate across different sites and sessions. C8.2.4 and C8.2.5 tests should therefore include environmental observations, not only direct memory-write APIs. [Poison Once, Exploit Forever](https://arxiv.org/abs/2604.02623)
- April 2026: the Black-Hole Attack showed that vectors injected near the embedding-space centroid can become frequent top-k neighbors across broad query sets, with reported malicious appearances in up to 99.85% of top-10 retrievals. C8.2.3 audits should therefore include hubness, reverse-nearest-neighbor, centroid-distance, and query-saturation metrics, not only ordinary outlier scores. [Black-Hole Attack](https://arxiv.org/abs/2604.05480)

## Implementation Maturity

- Mature: regulated-data detection before embedding is well supported for common identifiers through DLP services, Presidio-style recognizers, checksum validators, and custom enterprise dictionaries.
- Developing: loader-stage poisoning defenses are practical when teams maintain adversarial fixtures for each accepted format, but coverage still depends heavily on the exact parser, renderer, OCR engine, and metadata fields used in production.
- Developing: vector quarantine can be implemented with staging collections, source-trust labels, tenant partitions, and audit logs, while robust detection of high-centrality or semantically normal poisoned vectors remains research-heavy.
- Early: agent memory write authorization is still mostly application-defined. Strong implementations use a central memory-write service with provenance, trust tiering, reviewer identity, rollback, and policy checks outside the same prompt context that produced the candidate memory. OWASP Agent Memory Guard is a useful reference point for read/write interception, policy checks, snapshots, and rollback, but vector-store protection and multi-framework coverage are still maturing.
- Early: contradiction detection helps with explicit factual conflicts but still needs curated golden sources, source precedence rules, and reviewer workflows for subtle framing, stale facts, and domain-specific policy conflicts.

## Audit Checks to Add to Test Fixtures

- Loader-stage poisoning fixtures should place the same instruction in visible text, CSS-hidden HTML, DOCX comments, PDF annotations, OCR layers, metadata fields, and Unicode-obfuscated text, then compare what reviewers, scanners, retrievers, and model context each see before vectorization.
- Vector-stage fixtures should include centroid-near "black-hole" candidates, duplicate near-neighbor clusters, metadata-only multimodal poison records, and query-saturation probes so quarantine logic measures hubness, reverse-nearest-neighbor count, source trust, and retrieval dominance.
- Memory-stage fixtures should use an environmental observation, generated summary, MCP tool result, and delegated-agent result that each tries to persist a vendor, source, policy, or future instruction as trusted memory; passing tests should keep the write quarantined across restart, summarization, and retrieval.
- Contradiction fixtures should introduce lower-trust content that conflicts with a golden policy, owner, entitlement, deadline, or approved source ranking, then verify the alert records provenance, affected chunks, retrieval impact, reviewer decision, and rollback batch.

## Community Notes

- As of May 2026, loader-stage validation should diff the raw file, rendered view, extracted text, OCR text, metadata, and embedded resource list before any vector upsert. The practical test is whether the same hidden instruction is visible to reviewers, scanners, and the retriever; if not, quarantine the file and record which extractor produced each chunk.
- For memory validation, research prototypes are starting to converge on explicit trust scoring rather than prompt-only filtering. [SuperLocalMemory](https://arxiv.org/abs/2603.02240) is useful as an audit pattern because it separates behavioral data, records per-agent provenance, and evaluates trust degradation against sleeper-style memory poisoning; production systems still need the same ideas implemented in their central memory-write service.
- Memory poisoning research is moving from prompt-like persistence into action-plan poisoning. A March 2026 ScienceDirect paper on [Deceptive Semantic Reasoning Manipulation](https://www.sciencedirect.com/science/article/pii/S0952197626002496) shows forged historical knowledge can steer agent decisions and tool selection, so memory-write review should inspect proposed actions, not only factual claims.
- Vector-store staging is easier when quarantine, tenant, and source-trust labels are first-class metadata. [Qdrant multitenancy](https://qdrant.tech/documentation/manage-data/multitenancy/) and [Milvus multi-tenancy](https://milvus.io/docs/multi_tenancy.md) document native partitioning patterns, but the application still has to enforce source validation before promotion from staging to production collections.
- For tool-to-memory writes, schema validation is necessary but not enough. The [MCP tools specification](https://modelcontextprotocol.io/specification/2025-11-25/server/tools) explicitly calls for output sanitization and client-side result validation, and its optional output schema support gives auditors a concrete control to test. Memory writers should treat unstructured tool text, annotations, resource links, and fallback serialized JSON as untrusted even when the tool call itself was authorized.
- For poisoned-index investigations, keep enough provenance to support traceback after detection. [RAGForensics](https://arxiv.org/abs/2504.21668) is a useful audit pattern because it searches for the texts responsible for poisoned outputs, but it depends on preserved source IDs, batch IDs, retriever scores, and prompt-time evidence.
