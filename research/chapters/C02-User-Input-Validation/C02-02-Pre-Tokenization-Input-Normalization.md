# C2.2: Pre-Tokenization Input Normalization

> **Parent:** [C02: User Input Validation](C02-User-Input-Validation.md)
> **Last Researched:** 2026-06-09

## Purpose

AI models are vulnerable to subtle input perturbations that humans cannot perceive but cause models to misclassify, misinterpret, or behave unexpectedly. For LLMs, this includes Unicode tricks, homoglyph substitutions, invisible characters, encoding manipulations, and gradient-optimized adversarial suffixes (GCG attacks). For vision and audio models, adversarial perturbations cause misclassification with minimal visible/audible changes. As of 2026, adversarial attacks routinely achieve 90–99% ASR on open-weight models and 80–94% on proprietary models. Multi-signal detection (perplexity + length + attention patterns + semantic analysis) is the current recommended approach — no single defense layer is sufficient.

A critical development: reasoning models (o1, o3, DeepSeek-R1) are **not** uniformly safer. The "Weakest Link" study (June 2025) found multi-step attacks like Tree-of-Attacks succeed at 63% on reasoning models vs. 31% on standard models, because chain-of-thought itself creates exploitable attack surface. This was dramatically confirmed in 2026: Hagendorff et al. (Nature Communications, 2026) demonstrated that large reasoning models (DeepSeek-R1, Gemini 2.5 Flash, Grok 3 Mini, Qwen3 235B) act as **autonomous jailbreak agents**, achieving 97.14% success across all model combinations with no human supervision — a phenomenon they term "alignment regression," where more capable reasoning makes models better at subverting alignment in other models.

Automated attack tooling is also accelerating. JBFuzz (March 2025) applies software fuzzing techniques to jailbreaking, achieving 99% average ASR across GPT-4o, Gemini 2.0, and DeepSeek-V3 in roughly 60 seconds per jailbreak. Mastermind (January 2026) uses knowledge-driven multi-turn fuzzing with an evolutionary optimizer, reaching 87% average ASR including 67% on Claude 3.7 Sonnet and 60% on GPT-5. The UK AISI/Gray Swan challenge ran 1.8 million attacks across 22 models — every model broke. Supply-chain adversarial attacks via quantization are also maturing — the "Mind the Gap" paper (ICML 2025) showed models that appear 82.6% safe in full precision drop to 3% safety after GGUF quantization. Attacks are also moving into the latent space: the Latent Fusion Jailbreak (LFJ, revised January 2026) blends hidden states from harmful and benign queries to produce adversarial representations invisible to text-level filters, achieving 94.01% average ASR across five model families.

As of May 2026, invisible-instruction attacks are no longer just a rendering problem. Reverse CAPTCHA (February 2026) evaluated 8,308 model outputs and found that tool use substantially increases compliance with invisible Unicode instructions, while Imperceptible Jailbreaking (October 2025) showed that variation-selector suffixes can leave the prompt visually unchanged while altering tokenization enough to generalize to prompt-injection attacks. The same character classes are now appearing in software supply-chain incidents: the March 2026 Glassworm wave hid JavaScript payloads in invisible variation-selector code points across GitHub, npm, VS Code, and OpenVSX ecosystems, with CSA estimating 433 affected components by mid-March 2026. BeyondTrust's March 2026 Codex disclosure showed a related metadata path: Unicode U+3000 ideographic spaces made a branch-name command-injection payload visually disappear while still being parsed by the execution environment. Normalization controls therefore need to cover pre-model input, retrieved content, tool responses, generated or reviewed code, repository metadata, package metadata, and model outputs before any downstream parser or action executor sees the text.

Two audit updates matter in May 2026. MITRE ATLAS gives reviewers more precise mappings for this control: prompt injection belongs under AML.T0051, hidden or encoded prompt content maps cleanly to prompt obfuscation patterns, and AML.M0015 calls for adversarial input detection before inference rather than after a model or tool has already consumed the payload. As of the May 27, 2026 release, ATLAS moved to a monthly calendar-versioned cadence (v2026.05, superseding the May 4 v5.6.0) with a reworked YAML v6.0.0 data model that adds platform designations — Predictive AI, Generative AI, Agentic AI, Enterprise — to every technique, so C2.2 evidence can now be tagged to the specific platform surface (a chat-only endpoint versus an agent with tool access) rather than a generic adversarial-input label. The MITRE CTID "Secure AI v2" wrap-up (May 6, 2026) drove much of that expansion: 45+ new techniques and sub-techniques, 10+ new mitigations, and threat-emulation scenarios for prompt injection against organizational chatbots that auditors can run as input-defense exercises. AWS's October 2025 Bedrock Guardrails guidance also treats encoding attacks as a bidirectional problem: use prompt-attack detection for requests to produce encoded output, denied-topic policies for zero-tolerance encoded content, and output guardrails for harmful content decoded or generated by the model.

The Microsoft Security blog's May 7, 2026 disclosure of two Semantic Kernel CVEs (CVE-2026-25592 and CVE-2026-26030) gives this section a fresh framework-level anchor. CVE-2026-26030 escalated unsafe string interpolation in vector-store filter expressions into Python `eval()` execution through class-hierarchy traversal, and CVE-2026-25592 (CVSS 9.9, GitHub CNA assignment) turned an inadvertently exposed `SessionsPythonPlugin` `DownloadFileAsync` into a host-level write primitive once a model-generated path argument reached it. Patched in `semantic-kernel` 1.39.4 (Python) and 1.71.0 (.NET), the pair shows why C2.2 detection has to hold for tool-argument and retrieved-text paths, not just chat prompts: a tokenizer-clean string can still smuggle an instruction that produces a malicious argument the next interpreter happily decodes. The CVE-2026-25592 patch shape is itself the cleanest framework-level reference for C2.2.1+C2.2.3 evidence in production code: the new `ValidateLocalPathForDownload` decodes the model-supplied string first (Base64, URL, JSON), normalizes it (NFC plus an ASCII fold for `/` homoglyphs such as `∕` U+2044), and only then anchors the resolved path against an allowlist — exactly the "decode before validating → type-normalize → anchor to a safe root" sequence the section calls for. Pair this with TokenBreak (Schulz et al., June 2025), which shows that mismatched tokenizers between a guardrail classifier and the downstream LLM let attackers craft inputs the protector misreads while the model still understands them — a classification-vs-execution divergence that C2.2.2 evidence has to address directly.

Two complementary findings sharpen the limits of byte-level normalization. Adversarial Tokenization (Geh, Shao, Van den Broeck, ACL 2025, arXiv:2503.02174) demonstrated that an obviously malicious string can be re-segmented into a non-canonical BPE token sequence — same Unicode text, different tokens — and modern LLMs still understand the semantics and produce restricted output. The implication for C2.2 is concrete: NFC and confusable mapping operate on code points, but alternative tokenizations of the same code points can route the same intent past a guardrail that only sees the canonical segmentation. The Special-Character Adversarial Attacks study (Sarabamoun, arXiv:2508.14070, revised November 25, 2025) ran 4,000+ attacks against Phi3 (3.8B), Mistral (7B), DeepSeek-R1 (7B/8B/14B/32B), and GPT-OSS (20B) across four families — Unicode control characters, homoglyph/script confusion, structural perturbation, and encoding obfuscation — and reported encoding-based attacks at 64.3–67.1% effectiveness, with model vulnerability ranging from 18.5% on GPT-OSS:20b to 85.5% on Mistral:7b. The paper's defense recommendation matches the C2.2.1/C2.2.3 control shape: comprehensive Unicode normalization as the first line of defense, layered with encoding-detection policies and discriminator-objective training so that special-character variants no longer reach the model as a tokenization variant the model can read.

OpenAI's February 13, 2026 launch of Lockdown Mode for ChatGPT also matters as a vendor signal. The product framing acknowledged publicly that prompt injection against AI browsers may never be fully patched — a striking shift from earlier vendor positioning and a reason C2.2 evidence should treat in-application normalization as a load-bearing control rather than something the upstream model will eventually solve. For audit purposes, "the model vendor will fix it" is no longer a defensible position for invisible-character, encoded-output, or tool-argument paths.

The May 4, 2026 Grok/Bankrbot Morse-code heist (OECD AI Incident 2026-05-04-4a73) puts a price tag on this. An attacker first transferred a Bankr Club Membership NFT to elevate Grok's permissions, then posted a public X reply asking Grok to "translate" a Morse-code message. Grok decoded the dots and dashes into a financial instruction, tagged `@bankrbot`, and Bankrbot executed a 3 billion DRB token transfer (≈$150K–$200K) before any human review. The attack mapped to OWASP LLM01:2025 (prompt injection) and LLM06:2025 (excessive agency); roughly 80% of funds were eventually recovered after community attribution. The technical lesson for C2.2 is sharp: when a user-controlled string asks a model to *decode* something — Morse, hex, Base64, Caesar, ROT13, custom ciphers — the decoded text is functionally an input to the next agent in the chain and must pass the same normalization, smuggling-detection, and policy checks as a fresh prompt before it reaches a tool call. Translation-as-decoding is now a documented bypass class, not a hypothetical.

Unit 42's March 3, 2026 telemetry report turned web-based indirect prompt injection from a theoretical concern into a measurable abuse class. Across detection telemetry, they catalogued 22 distinct payload techniques and the first observed AI-based ad-review evasion in production. The delivery-method breakdown matters for C2.2 because normalization has to cover all three channels: visible plaintext (37.8% of cases), HTML attribute concealment (19.8%), and CSS rendering suppression (16.9%). The same report documented payloads that hijacked agents into initiating Stripe payments, deleting databases, approving scam ads, and leaking system prompts — confirming that "blocked before action" is the operational test, not "alerted after action." The OpenClaw advisory (CVE-2026-27001, fixed in OpenClaw 2026.2.15) provides the matching real-world fix shape for C2.2: a working-directory string was embedded into the agent system prompt without sanitization, so attacker-controlled directory names containing newlines, Unicode bidi controls, or zero-width markers could break the prompt structure. The patch strips Unicode control/format characters and explicit line/paragraph separators on every workspace path before it reaches the LLM. The Reverse CAPTCHA evaluation (8,308 outputs across providers) also clarified a previously under-acknowledged asymmetry: OpenAI models preferentially decode zero-width binary while Anthropic models preferentially decode Unicode Tag content, so an attacker who knows the target provider can pick the more effective encoding and explicit "please decode" hints raise compliance by up to 95 percentage points. Defensive policy has to assume the attacker knows which encoding the target tokenizer is most willing to interpret.

Two May 2026 papers tighten the evidence side of this requirement. Deep et al. "Evaluation of Prompt Injection Defenses in Large Language Models" (arXiv:2604.23887, submitted April 26, 2026, revised May 12, 2026) ran an adaptive attacker over 20,000+ rounds against nine defense configurations and found that every defense relying on the model to protect itself broke; only an output-filter applied in application code with hardcoded rules survived 15,000 attacks with zero leaks. Their conclusion — "security boundaries must be enforced in application code, not by the model being attacked" — is the operational test C2.2.2 evidence should map to. Tian et al. "AI Agents May Always Fall for Prompt Injections" (arXiv:2605.17634, May 17, 2026) reframes the same problem through Contextual Integrity theory and proves an impossibility result: any defender who tightens rules enough to block crafted contexts will also block legitimate flows. For auditors the practical implication is that C2.2.2 cannot rely on classifier-only enforcement; the evidence must show a non-model enforcement boundary on the assembled prompt and on every tool argument before the next interpreter sees it.

OWASP also reframed the scoring side of this control. AIVSS v0.8 — the Agentic AI Vulnerability Scoring System, co-published with AIUC-1 and announced March 19, 2026 ahead of RSAC — extends CVSS v4.0 with agentic risk amplifiers (autonomy level, tool-use scope, dynamic identity, persistent memory, multi-agent interactions, capacity for self-modification, non-determinism) so that the same underlying input-validation gap is now scored higher when it sits inside an agent with broad tool access or durable memory. For C2.2 evidence this means a missing normalization or smuggling-detection control on a chat-only endpoint is not equivalent to the same gap on an agent with shell, browser, file-system, or financial primitives — and audit records should capture the amplifier context, not just the technical defect.

The April 2026 "Comment and Control" disclosure by Aonan Guan, Zhengyu Liu, and Gavin Zhong (Johns Hopkins) makes the cross-vendor agent-input case concrete. A single prompt-injection payload posted as a GitHub pull request title, issue body, or comment caused three production AI coding agents — Anthropic's Claude Code Security Review (CVSS 9.4), Google's Gemini CLI Action, and GitHub's Copilot Coding Agent — to exfiltrate API keys and access tokens through GitHub's own APIs by abusing `pull_request_target` runner privileges. No CVEs were assigned, and Anthropic's Opus 4.7 system card had already disclosed that Claude Code Security Review is "not hardened against prompt injection." For C2.2 this is the canonical demonstration that PR titles, issue bodies, comment threads, and CI-trigger metadata are first-class adversarial input channels and need the same Unicode normalization, instruction-boundary detection, and pre-tokenization policy that direct chat input receives. The corresponding tool-argument failures are tracking as CVEs in the same month: CVE-2026-41264 (Flowise CSV Agent, CVSS 9.2, fixed in 3.1.0 April 15, 2026) shows a prompt-injection chain that drove an LLM-generated Python script through `CSV_Agents.run` past a regex blocklist using `import pandas as np, os as pandas`, while CVE-2026-5002 (PromtEngineer localGPT) is a publicly disclosed remote prompt-injection in `_route_using_overviews` requiring no authentication and no user interaction. Both reinforce the same control shape — schema-level allowlisting and content checks on tool arguments before any interpreter sees them, not after.

Lasso Security's December 21, 2025 red-team report on Perplexity's BrowseSafe model is the cleanest published demonstration of the C2.2.1+C2.2.4 anti-pattern. BrowseSafe is a content-detection model that scans web pages in real time for malicious instructions before they reach an AI browser's agent loop. Lasso's red-team engine achieved a 36% bypass rate inside hours using NATO phonetic alphabet, Pig Latin, Base32, Base64, and Unicode tag formatting. The reported root cause is exactly the C2.2 control gap: the model "fails to normalize or canonicalize content before classification," so obfuscated payloads wrapped in HTML evade detection even when their underlying intent matches a known malicious pattern. The lesson for auditors is that a content classifier that runs *before* normalization is not a substitute for normalization — it is the textbook anti-pattern this section exists to prevent. Zenity Labs' March 3, 2026 PleaseFix disclosure (covering PerplexedBrowser in Perplexity Comet) compounds the picture for AI browsers: a zero-click indirect prompt injection delivered via a benign calendar invitation walked Comet through local-filesystem access and password-manager credential extraction without further user interaction. Perplexity addressed the underlying browser-side agent execution path before public disclosure, but the disclosure makes the larger point — agentic browsers carry the C2.2 normalization and instruction-boundary duty into web-rendered content, calendar invites, email subjects, and form fields, not just chat prompts.

Capsule Security's April 15, 2026 disclosure of ShareLeak and PipeLeak makes the form-field ingestion channel concrete. ShareLeak (CVE-2026-21520, CVSS 7.5, patched January 15, 2026) showed that Microsoft Copilot Studio concatenated SharePoint comment-form input directly into the agent's system instructions with no sanitization between the form and the model, so an attacker-supplied "system role" payload overrode the agent's original task and exfiltrated connected SharePoint List data via Outlook. In Capsule's testing, Microsoft's built-in safety mechanism flagged the request as suspicious yet the data still left the tenant — a textbook C2.2.2 "alerted but not blocked" failure. PipeLeak is the parallel finding in Salesforce Agentforce: Web-to-Lead form submissions seeded persistent malicious instructions that fired only when an internal user later asked the agent to review the lead, turning a single anonymous form post into a bulk CRM extraction. Salesforce did not assign a CVE; Capsule retested after the vendor's "remediated the specific scenario" response and showed the attack still worked against Custom Topics (now called Sub-Agents), which cover most enterprise deployments. The lesson for C2.2 is that form fields, profile metadata, lead descriptions, and other secondary ingestion channels need the same Unicode normalization, instruction-boundary detection, and pre-tokenization policy that direct chat input receives, and that vendor patches scoped to a single trigger surface do not generalize across all agent topics. Praetorian's April 10, 2026 supervisor-agent bypass research reinforces the same shape: supervisor classifiers that inspect only direct user input miss adversarial content sitting in profile fields, retrieved documents, database records, tool outputs, and email subjects — the recommended evidence is that the *complete assembled prompt*, not the user-typed message, is what gets normalized and screened.

A subtle but important normalization-ordering finding surfaced in early 2026 and is directly load-bearing for C2.2.1 and C2.2.3 evidence: the Unicode confusables data (UTS #39) and NFKC normalization disagree on 31 mappings. The canonical example is LATIN SMALL LETTER LONG S (ſ, U+017F), which `confusables.txt` folds to `f` while NFKC folds it to `s`. The practical consequence for the normalize-then-detect pattern this section recommends is counterintuitive: if a pipeline runs NFKC *before* its confusable/spoofing check — the usual ordering — those 31 confusable mappings become unreachable dead code, because the input has already been normalized to a different target before the spoof check ever sees it. The mismatches were submitted to the Unicode Consortium (reported as accepted into PRI #540 for UTC #187 review). The audit takeaway is that "we normalize and we run confusable checks" is not sufficient evidence on its own; reviewers should confirm the *order* of the two steps and that the confusable map in use was filtered for post-NFKC deployment (the meaningful entry count drops from roughly 6,565 to ~613 after NFKC), or the spoof detection silently misses inputs it was configured to catch.

Two June 2026 incidents extend the agent-action surface. PromptArmor's Microsoft Copilot Cowork disclosure (first reported May 25–26, full write-up June 9, 2026) showed that five lines of injection embedded in an uploaded 81-line Cowork "Skills" file hijacked the agent into silently exfiltrating M365 documents: because Cowork auto-approves messages sent to the active user, the agent sent Teams messages containing HTML `<img>` tags whose `src` embedded pre-authenticated SharePoint/Graph download links, so opening the message leaked the links — 5/5 success against Claude Opus 4.7 and other current models, mitigated with `Set-SPOSite -BlockDownloadPolicy $true` and tighter Graph permissions. The vector here is visible-plaintext skill-file injection rather than Unicode smuggling, but the C2.2 lesson is the same one the Antigravity and ShareLeak cases made: detection has to sit before the auto-approved tool action and before any Markdown/HTML reference renders. The Cloud Security Alliance's "Hidden Unicode Instruction Injection in AI Agent Skills" research note (March 2026) is the complementary aggregation on the encoding side, cataloguing four pre-tokenization primitives — Unicode Tags block U+E0000–U+E007F, zero-width steganographic binary encoding, bidi overrides U+202A–U+202E, and cross-script homoglyphs — confirmed across Claude Code, GitHub Copilot, OpenAI Codex Skills, Gemini CLI, Cursor Agent, and M365 Copilot, and prescribing NFKC normalization plus Tags-block rejection as the baseline control.

### Residual Artifact Review Checklist

For C2.2.3 evidence, reviewers should ask for the post-normalization artifact policy, not just the normalizer implementation. Useful evidence includes:

- Raw-to-normalized diffs that show which code points were changed, removed, or left intact.
- A tokenizer-view test that confirms invisible or confusable characters do not survive as model-meaningful instructions.
- A field-level decision table for reject, block, flag-for-review, or explicit marking behavior.
- Regression coverage for UTF tag blocks, zero-width binary encodings, emoji and text variation selectors, bidi overrides, homoglyph swaps, malformed UTF-8, and Base64/hex instruction wrappers.
- Review-queue records that safely escape artifacts and include source channel, artifact category, normalized hash, reviewer decision, and downstream enforcement action.
- Separate tests for direct prompts, uploaded documents, retrieved RAG chunks, MCP/tool responses, browser-rendered content, and agent-to-agent messages.
- CI and package-ingestion scans for variation selectors, Unicode tag blocks, Unicode space separators, and private-use payload carriers in generated code, reviewed pull requests, repository branch/tag names, package manifests, extension metadata, and vendored dependencies.
- Sanitization on runtime context strings before they reach a system prompt — current working directory, project name, file path, ticket title, environment variable values, agent-skill text, and MCP manifest fields — matching the OpenClaw CVE-2026-27001 fix shape (strip Unicode control/format characters and line/paragraph separators on the path itself and again at path resolution).
- Practical alerting thresholds derived from the Embrace The Red February 2026 disclosure: flag consecutive Unicode Tag block runs greater than 10 codepoints and any artifact with sparse counts above ~100 for review, while keeping emoji ZWJ and Indic-script ZWJ usage out of the deny path.

The key audit distinction is whether the application treats residual artifacts as untrusted control input. Cleaning may be acceptable for low-risk display text, but prompts, tool arguments, workflow commands, memory writes, and retrieved instructions should fail closed unless an explicit exception path exists.

### Agent Memory and Output-Path Enforcement

As of May 2026, the Copirate 365 disclosure showed how indirect prompt injection can move from decoded content into durable memory and browser-rendered output. For C2.2, the lesson is that normalization and smuggling detection must sit before every state-changing or exfiltration-capable path, not only before the first model call.

Useful verification cases:

- Submit an uploaded document or retrieved chunk that appears benign but contains hidden or encoded instructions to write memory, save preferences, open a URL, render HTML, or call a browser/navigation tool.
- Confirm the application blocks or quarantines the payload before memory writes, tool arguments, workflow transitions, HTML previews, Markdown image loads, font loads, or outbound navigation can occur.
- Re-test later turns in the same conversation and a new conversation to confirm hidden instructions were not persisted into user memory, project notes, scratchpads, summaries, or cached retrieval context.
- Include a canary secret in replay tests and verify it never appears in URL paths, query strings, font names, image loads, telemetry, or external request logs.
- Review logs for the raw artifact category, normalized diff/hash, detection source, affected route, enforcement decision, and whether any downstream state mutation was attempted.
- Treat "render only" and "preview only" output paths as security-relevant. Browser previews, Markdown renderers, images, fonts, and navigation tools can become exfiltration channels even when no explicit tool call is shown to the user.

## ATLAS Mapping for C2.2

As of the MITRE ATLAS v2026.05 data release (May 27, 2026, which moved ATLAS to a monthly calendar-versioned cadence and added per-technique platform designations for Predictive AI, Generative AI, Agentic AI, and Enterprise), C2.2 evidence can be mapped more precisely than a generic adversarial-input tag:

| AISVS Evidence | ATLAS Mapping | Audit Use |
|----------------|---------------|-----------|
| Hidden, encoded, or visually disguised instructions in prompts, retrieved content, tool responses, repository metadata, and outputs | AML.T0051 prompt injection plus prompt-obfuscation tradecraft | Tag each regression case with the source channel and concealment method so coverage can be trended over time |
| Detection that blocks before tokenization, inference, embedding, memory writes, tool calls, or workflow transitions | AML.M0015 Adversarial Input Detection | Require a pre-model enforcement point and logs showing the block decision, not just a post-hoc alert |
| Unicode, malformed UTF-8, mixed-script, and encoded-output handling | CAPEC-71, CAPEC-80, Unicode UAX #15, Unicode UTS #39 | Pair normalization conformance tests with model-facing replay cases because Unicode correctness alone does not prove AI safety |

## Multimodal Adversarial Threats

As of 2026, adversarial attacks extend across all modalities:

- **Vision-language**: Chain of Attack (CVPR 2025) — step-by-step semantic updates for transferable adversarial examples; TAO-Attack (ICLR 2026) evaluates on both text and VLMs
- **Speech-language**: Electronic Music Assassin (2025) — frequency-domain attack achieves **100% ASR on 8 commercial ASR systems** with audio indistinguishable from normal music by 52% of human subjects; GuidedDE (Neural Networks, April 2026) — 90–95% targeted ASR with 67–73% efficiency improvement over prior methods; adversarial speech enhancement attacks (ICASSP 2026) — diffusion-based models are inherently more robust than deterministic predictive models
- **Reasoning models**: H-CoT (February 2025) — hijacks chain-of-thought to subvert safety reasoning; "Weakest Link" (June 2025) — reasoning models 32pp worse on Tree-of-Attacks and 22pp worse on suffix injection than standard models
- **Physical-environment**: CHAI (January 2026, UC Santa Cruz) — physical prompt injection targeting embodied AI
- **Supply chain / quantization**: Mind the Gap (ICML 2025) — models appear safe in full precision but become malicious after GGUF quantization (most widely deployed format via llama.cpp and Ollama)

## Implementation Maturity

| Area | Maturity | Notes |
|------|----------|-------|
| Unicode NFC normalization | **High** | Mature library support. But emoji variation selectors and tag characters are commonly missed. |
| Perplexity-based detection | **Medium** | Effective against GCG suffixes; defeated by LARGO/PAPILLON low-perplexity attacks. |
| Decoding-time defenses | **Low–Medium** | SafeProbing, AISA, SAID show promise but are academic. Not in production tooling. |
| Adversarial training (LLM scale) | **Low–Medium** | ShorT reduces cost. But no frontier model ships with adversarial training as a standard feature. |
| Formal/certified robustness | **Low** | Infeasible at transformer scale. RoMA offers statistical alternative. |
| Commercial red-teaming platforms | **Medium-High** | Mindgard, HiddenLayer, Cisco AI Defense provide automated testing. Adoption growing. |
| Encoding smuggling defense | **Low–Medium** | Promptfoo, garak, LLM Guard, ATR-style rules, and Bedrock-style encoded-output tests make regression testing more practical, but 100% guardrail bypass via emoji smuggling and the Glassworm supply-chain wave show production coverage remains uneven. No comprehensive standard. Reverse CAPTCHA's provider-asymmetry finding (OpenAI vs. Anthropic preferring different encodings) means a single regression corpus understates risk for at least one provider in a multi-model deployment. |
| Tokenizer-level adversarial defense | **Low** | Adversarial Tokenization (ACL 2025) shows that re-segmenting the same Unicode text into a non-canonical BPE token sequence still elicits restricted output, which means code-point-only normalization cannot close this attack surface. Broken Tokens? (Zheng et al., June 2025) reports that instruction-tuned LLMs retain 93.4% of original performance on randomly-sampled non-canonical tokenizations and 90.8% on character-level tokenization, so attackers can rely on broad tokenizer-robustness. Phonetic Perturbations / CMP-RT (April 2026) extends the same failure to code-mixed phonetic obfuscation against Gemini-3-Pro and other SOTA models. No production tooling enforces canonical-tokenization equivalence yet; viable evidence paths today are tokenizer-aligned guardrail evaluation per TokenBreak, CPT-Filtering-style encoded-text detection (Broken-Token, October 2025), and tokenizer-aware regression sets that include non-canonical, phonetic, and code-mixed variants. |
| CI metadata sanitization (PR titles, issue bodies, review comments, branch/tag names, commit messages, workflow inputs, `pull_request_target` payloads) | **Emerging** | The April 2026 "Comment and Control" disclosure (Guan, Liu, Zhong) showed that Claude Code Security Review (CVSS 9.4), Gemini CLI Action, and GitHub Copilot Coding Agent all treat CI-trigger metadata as trusted input, despite Anthropic's Opus 4.7 system card explicitly noting Claude Code Security Review is "not hardened against prompt injection." Practical evidence today is to apply the same Unicode normalization, instruction-boundary detection, and policy controls to CI metadata that direct chat input receives, and to gate `pull_request_target` workflows that pass secrets to the runner. |
| Tool-argument schema validation and policy enforcement | **Emerging** | CVE-2026-41264 (Flowise CSV Agent, CVSS 9.2) shows the typical failure pattern: an LLM-generated Python script reaches an unsandboxed `eval`/`exec` path past a regex import blocklist. CVE-2026-2256 (ModelScope MS-Agent, CVSS 6.5) is the parallel "regex denylist on prompt-derived shell commands" failure — command obfuscation, encoding, and alternative shell syntax route around the deny pattern straight into the shell tool. Google Antigravity's `find_by_name` tool-argument flag injection (OECD AI Incident 2026-04-20-8687) shows the same gap in a vendor IDE: a `-Xsh` Pattern smuggled fd's `-X` (exec-batch) flag past the wrapper before Strict Mode applied. Deep et al. (arXiv:2604.23887, May 2026) reinforces the architectural fix: enforce on tool arguments in application code with hardcoded rules, not via classifier-only filtering. CVE-2026-25592 Semantic Kernel's `ValidateLocalPathForDownload` patch (May 2026) is the reference fix shape — decode the model-supplied string, NFC plus ASCII fold for `/` homoglyphs (`∕` U+2044), then anchor the resolved value to an allowlist. Mature implementations apply schema validation, allowlists, argument-separator hardening (`--`), and policy engines on every tool argument before any interpreter, shell, or plugin sees the value, and refuse "regex denylist" as a block primitive. |
| Multi-source ingestion sanitization (form fields, profile metadata, lead descriptions, retrieved RAG, tool outputs) | **Emerging** | Capsule Security's April 2026 ShareLeak (CVE-2026-21520) and PipeLeak disclosures showed that even major vendors concatenate untrusted form input directly into agent system instructions with no normalization or instruction-boundary detection. Praetorian's April 2026 supervisor-agent research adds that classifiers scoped to the user-typed turn miss adversarial content in every other ingestion channel. Mature implementations now treat the *fully assembled prompt* as the policy surface and apply the same normalization, encoding-detection, and instruction-marker controls to form fields, profile bios, lead notes, ticket titles, retrieved chunks, and tool outputs that they apply to direct chat input. |
| Runtime-context sanitization (CWD, paths, manifests, skill text) | **Emerging** | Mostly handled ad-hoc; OpenClaw CVE-2026-27001 (May 2026) is the first publicly tracked CVE with a production fix that matches the C2.2 control shape directly — strip Unicode control/format and line/paragraph separators from any runtime string before it lands in a system prompt. Embrace The Red's "Scary Agent Skills" (February 2026) showed the same gap exists in agent-skill markdown. |
| Gateway-level prompt blocking | **Medium** | Microsoft Entra and Amazon Bedrock Guardrails provide deployable blocking controls, but documented limits such as text-only or JSON-only coverage mean they should be regression-tested with the same corpus as in-app scanners. |
| Content classifiers for AI browsers | **Low–Medium** | Perplexity's BrowseSafe (released December 2025) is the first major vendor-released open content-detection model for AI browsers, but Lasso Security's December 21, 2025 red-team report achieved a 36% bypass inside hours using NATO phonetic, Pig Latin, Base32, Base64, and Unicode tag formatting because the model "fails to normalize or canonicalize content before classification." Treat browser-content classifiers as one layer in a defense-in-depth stack — and only as a layer that sits *after* normalization. The same disclosure cadence (Zenity PleaseFix / PerplexedBrowser, March 2026) confirmed that web-rendered content, calendar invites, and email subjects are now first-class adversarial-input channels for agentic browsers. |
| Proactive deceptive defense | **Low–Medium** | HoneyTrap (January 2026) and ProAct (February 2026) offer complementary proactive approaches — honeypot traps and spurious-response deception. Both academic but ProAct's 94% ASR reduction is notable. |
| Latent-space defense | **Low** | Latent adversarial training (January 2026) counters LFJ with 80%+ ASR reduction. No production tooling yet — research demonstrates the approach is viable. |
| Graph-attention filtering | **Low–Medium** | GuardNet (September 2025) achieves F1 up to 99.8% on jailbreak detection via graph neural networks. Academic; no production deployment known. |
| Runtime agent enforcement | **Low–Medium** | AgentSpec (ICSE 2026) and Pro2Guard offer customizable runtime safety constraints. Early adoption stage. |

## Open Research Questions

- Can perplexity-based detection be augmented with attention pattern analysis to catch LARGO/PAPILLON-style low-perplexity attacks? Decoding-time defenses (SafeProbing, AISA, SAID) are promising alternatives.
- How effective is adversarial training at LLM scale? ShorT's √M finding reduces cost dramatically — will this make adversarial training practical for frontier models?
- What false positive rates are acceptable for encoding smuggling detection in production multilingual systems? The emoji smuggling 100% bypass rate suggests current approaches are fundamentally insufficient.
- How should reasoning models be hardened given that chain-of-thought creates exploitable attack surface (H-CoT, Weakest Link)?
- Are coding/reasoning tasks fundamentally more vulnerable to adversarial attacks than safety-category prompts (per T-GCG and Weakest Link findings)?
- Can formal robustness verification scale to transformer architectures? RoMA's statistical approach gets within 1% of formal methods — is that sufficient?
- How should quantized model deployments be verified given the Mind the Gap supply-chain attack vector?
- How should invisible Unicode scanning be enforced in code-review, package-ingestion, and coding-agent workflows without breaking legitimate variation-selector and multilingual content?
- How should repositories expose suspicious Unicode in branch names, tags, commit metadata, and package manifests before those fields reach shells, agents, review systems, or CI runners?
- How should evaluation methodology be standardized? MLCommons AILuminate, PRISM BET, and JBDistill (renewable benchmarking) offer competing approaches.
- Is "alignment regression" (Nature Communications, 2026) an inherent property of capable reasoning models? If so, what architectural constraints can prevent reasoning capabilities from being weaponized for autonomous jailbreaking?
- Can proactive deceptive defenses (HoneyTrap-style honeypots, ProAct-style spurious responses) scale to production environments without impacting legitimate user experience? ProAct's claim of no utility degradation is promising but needs independent validation.
- How effective are latent-space attacks (LFJ) against closed-weight API models where hidden states are inaccessible? Does the attack transfer via distillation or only work with white-box access?
- Adversarial Tokenization (ACL 2025) shows the same code points can be re-segmented into a non-canonical BPE sequence that bypasses safety. Can production guardrails be retrofitted to enforce canonical-tokenization equivalence on inputs without breaking legitimate multilingual or whitespace-sensitive text, and what is the false-positive cost?
- The Autonomy Tax (arXiv:2603.19423, March 2026) reports defense-trained agents hitting a 99% timeout rate vs. 13% baseline across 97 tasks while still failing to block sophisticated attacks. Can defense training of the model be reformulated so that adversarial robustness no longer collapses multi-step tool execution, or is application-code enforcement (per Deep et al.) the only sustainable boundary?
- Tokenizer-Betrays-Reasoning (arXiv:2601.14658, January 2026) documents "phantom edits" arising from one-to-many token-ID mappings across 11,000+ trials. Pre-tokenization normalization closes most code-point-level paths, but is there a tokenizer-design change (canonical-segmentation enforcement, deterministic resegmentation) that would close the equivalence-class gap without retraining at scale?
