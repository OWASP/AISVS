# C8.2: Embedding Sanitization & Validation

> [Back to C08 Index](C08-Memory-and-Embeddings.md)
> **Last Researched:** 2026-05-02

## Purpose

Pre-screen content before vectorization; treat memory writes as untrusted inputs; prevent ingestion of unsafe payloads. Embedding ingestion is a durable trust boundary: once text, metadata, or generated memory is converted into vectors, it can influence many future retrievals and is difficult to redact with confidence. As of May 2026, the practical attack surface has expanded from generic corpus poisoning into loader-stage obfuscation, retrieval-optimized indirect prompt injection, and persistent memory poisoning. [OWASP LLM08:2025](https://genai.owasp.org/llmrisk/llm082025-vector-and-embedding-weaknesses/) calls out vector data leakage, cross-context knowledge conflict, embedding inversion, and data poisoning as core risks. Recent work reinforces the point: [PoisonedRAG](https://www.usenix.org/conference/usenixsecurity25/presentation/zou-poisonedrag) showed high-success knowledge corruption from a few injected texts; [The Hidden Threat in Plain Text](https://arxiv.org/abs/2507.05093) showed DOCX, HTML, and PDF loader manipulations surviving into RAG pipelines; [Overcoming the Retrieval Barrier](https://arxiv.org/abs/2601.07072) showed retrieval-optimized indirect prompt injection under natural queries; and Microsoft's February 2026 [AI Recommendation Poisoning](https://www.microsoft.com/en-us/security/blog/2026/02/10/ai-recommendation-poisoning/) disclosure showed real-world attempts to plant persistent recommendations through hidden prompt parameters. Mature controls exist for PII detection and basic content filtering, but outlier detection, contradiction review, and memory-write authorization still need corpus-specific baselines, provenance, quarantine workflows, and human review for high-impact stores.

## Requirements

| # | Requirement | Level | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|-----------------|----------------------|--------------|
| **8.2.1** | **Verify that** regulated data and sensitive fields are detected prior to embedding and are masked, tokenized, transformed, or dropped based on policy, recognizing that data once embedded cannot be reliably redacted from the resulting index. | 1 | Sensitive data exposure through retained chunks, metadata, and vectors. This includes regulated PII, PHI, PCI data, credentials, customer secrets, and sensitive business identifiers that become searchable through the vector store. It maps to [OWASP LLM02:2025 Sensitive Information Disclosure](https://genai.owasp.org/llmrisk/llm02-sensitive-information-disclosure/) and [OWASP LLM08:2025 Vector and Embedding Weaknesses](https://genai.owasp.org/llmrisk/llm082025-vector-and-embedding-weaknesses/), which explicitly includes unauthorized access, embedding inversion, and data leakage risks. | Review the ingestion path before chunking and embedding: source connector, parser, text normalization, DLP scan, masking/tokenization step, metadata enrichment, embedding call, vector upsert, and audit log. Test with synthetic records containing SSNs, payment cards, email addresses, addresses, MRNs, API keys, access tokens, and internal project identifiers. Confirm detections are applied to extracted text, document metadata, table cells, image OCR output, and attachment names. Use [Microsoft Presidio](https://microsoft.github.io/presidio/text_anonymization/) analyzer and anonymizer evaluations, cloud DLP services such as Google Cloud Sensitive Data Protection, AWS Comprehend, or Microsoft Purview, and labeled false-negative review sets. Verify that masked text and metadata, not the original sensitive strings, are what reach the embedding provider and vector store. | PII tooling is mature for common identifiers but weaker for domain-specific secrets, internal codenames, and context-dependent sensitive facts. Presidio supports regex, deny lists, checksum, rule-based recognizers, NER, and context-aware detection, but custom recognizers are usually needed for enterprise data. Masking can reduce retrieval quality, so teams need policy-specific choices between redaction, tokenization, deterministic hashing, surrogate replacement, and dropping the chunk entirely. Redaction must happen before embedding; deleting the source document later does not prove the semantic signal disappeared from derived vectors, replicas, caches, or backups. |
| **8.2.2** | **Verify that** content intended to poison retrieval (e.g., text crafted to project to attacker-chosen embedding neighborhoods, hidden instructions intended for downstream model context, or steganographic payloads in non-text inputs) is detected and rejected or quarantined before vectorization. | 1 | RAG poisoning, indirect prompt injection, and loader-stage content manipulation. Attackers can hide instructions in HTML, PDF, DOCX, comments, CSS, white text, Unicode confusables, OCR layers, metadata, or crafted text that is optimized to be retrieved for target queries. [The Hidden Threat in Plain Text](https://arxiv.org/abs/2507.05093) reported a 74.4% attack success rate across 357 loader attack scenarios using content obfuscation and content injection against common document loaders. [Overcoming the Retrieval Barrier](https://arxiv.org/abs/2601.07072) showed that trigger fragments can make malicious content reliably retrieved under natural queries, and [Machine Against the RAG](https://www.usenix.org/conference/usenixsecurity25/presentation/shafran) showed that a single blocker document can suppress answers without relying on explicit instruction injection. MITRE ATLAS tracks the related families as [AML.T0051 Prompt Injection](https://atlas.mitre.org/techniques/AML.T0051), [AML.T0070 RAG Poisoning](https://atlas.mitre.org/techniques/AML.T0070), and [AML.T0080 AI Agent Context Poisoning](https://atlas.mitre.org/techniques/AML.T0080). | Build an intake test suite for every accepted format. Compare raw text extraction, rendered view, OCR output, metadata, links, alt text, comments, and hidden layers before embedding. Normalize Unicode, strip or flag zero-width characters, reject invisible or off-canvas text, limit embedded links and executable content, and quarantine chunks containing instruction-like text aimed at the downstream model. Test with fixture documents that include hidden prompts, white-on-white text, tiny fonts, HTML comments, PDF annotation layers, DOCX comments, homoglyphs, base64-like payloads, suspicious URL prompt parameters, and adversarial repetitions of target-query terms. For application code, verify that LangChain, LlamaIndex, Haystack, or custom loaders pass through a security transformation before vector upsert. Use [LangChain's indirect prompt injection guidance](https://docs.langchain.com/oss/python/langchain/rag), [NVIDIA NeMo Guardrails retrieval rails](https://docs.nvidia.com/nemo/guardrails/latest/about/rail-types.html), custom prompt-injection classifiers, static document scanners, and manual review queues for high-impact collections. | No parser-level defense is complete because attackers can target differences between raw extraction, rendered display, OCR, and model interpretation. Defensive prompts and delimiters help at generation time but do not sanitize a poisoned vector store. Treat sanitization as a gate: untrusted documents should not be embedded until the original file, extracted text, metadata, and source provenance have passed policy checks. Mature commercial tooling for document-intake poisoning is still emerging, so teams should keep adversarial loader fixtures in CI and require human review for quarantine releases. |
| **8.2.3** | **Verify that** vectors that fall outside normal clustering patterns are flagged and quarantined before entering production indices. | 2 | Adversarial embedding injection, vector collision, and high-centrality poisoning. A poisoned chunk may be crafted to land near attacker-chosen query neighborhoods or to be retrieved broadly across unrelated questions. [PoisonedRAG](https://www.usenix.org/conference/usenixsecurity25/presentation/zou-poisonedrag) achieved strong results by injecting a small number of malicious texts into large knowledge bases, while [CorruptRAG](https://arxiv.org/abs/2504.03957) focuses on practical single-text poisoning. Retrieval-optimized attacks mean the vector may look linguistically plausible while still being geometrically suspicious. | Establish baseline distributions per collection, tenant, source type, language, and embedding model version. Before production upsert, score each candidate chunk against nearest-neighbor distance, local density, centroid distance, duplicate similarity, high-degree retrieval frequency, and source reputation. Test with synthetic poisoned documents targeting known canary queries and confirm quarantine triggers before the production index changes. Useful methods include isolation forest, DBSCAN or HDBSCAN, robust z-score on cosine distance, cleanlab-style out-of-distribution scoring, cross-encoder reranking disagreement, and similarity search against a malicious-pattern fixture set. Verify that vector stores such as Qdrant, Milvus, Weaviate, Pinecone, Vespa, or OpenSearch can support pre-commit staging collections and audit logs for rejected vectors. | Outlier detection catches geometric anomalies but misses semantic poisoning that sits inside a normal cluster. New product documentation, legal updates, incident reports, and rare languages may also look anomalous. Baselines must be segmented and versioned; a single global threshold usually creates both false positives and blind spots. Emerging defenses such as [RevPRAG](https://aclanthology.org/2025.findings-emnlp.698/) and [Sparse Document Attention RAG](https://arxiv.org/abs/2602.04711) are useful complementary signals, but they do not replace pre-ingestion quarantine and source review. |
| **8.2.4** | **Verify that** an agent's own outputs are not automatically written back into its trusted memory without explicit validation (such as content-origin checks or write-authorization controls that verify the content's source before committing writes). | 2 | Persistent memory poisoning and self-reinforcing corruption. If generated summaries, tool results, or model conclusions are written into trusted memory without validation, one successful injection can become durable context for later sessions. Microsoft's February 2026 [AI Recommendation Poisoning](https://www.microsoft.com/en-us/security/blog/2026/02/10/ai-recommendation-poisoning/) research observed more than 50 prompt-based attempts from 31 companies across 14 industries that tried to plant trusted-source recommendations through prefilled assistant URLs, mapped to MITRE ATLAS [AML.T0051](https://atlas.mitre.org/techniques/AML.T0051) and [AML.T0080.000](https://atlas.mitre.org/techniques/AML.T0080.000). [A-MemGuard](https://arxiv.org/abs/2510.02373) describes why isolated memory-entry review is weak: malicious effects may activate only in a later context and then reinforce themselves through future writes. | Enumerate every path that can write to long-term memory, vector stores, profiles, summaries, preference stores, and shared agent state. Confirm each write records origin, author, source URI, source trust tier, extraction method, review status, timestamp, tenant, and approving principal. Block auto-promotion of generated content into trusted memory unless a separate validator checks source provenance, policy compliance, contradiction risk, and write authorization. Test with prompts and retrieved documents that ask the system to remember a vendor, override a policy, promote a source as authoritative, or persist a hidden instruction. Review logs for memory writes containing terms such as "remember", "trusted source", "from now on", "future conversations", "always cite", and "authoritative". Require human approval or policy-engine approval for high-impact memory writes. | Product memory features often blur preferences, summaries, user-authored facts, and model-inferred facts. That makes origin tagging and trust tiers more important than the model's confidence. Validation should run outside the same prompt context that produced the candidate memory. A-MemGuard-style consensus validation and dual-memory lessons are promising, but production teams still need explicit write permissions, reviewable memory ledgers, rollback, and per-user or per-tenant scoping. |
| **8.2.5** | **Verify that** new content written to memory is checked for contradictions with what is already stored and that conflicts trigger alerts. | 3 | Knowledge-base corruption, slow-burn poisoning, and cross-context knowledge conflict. Attackers may introduce plausible but conflicting facts over time so future retrieval selects the poisoned version. OWASP LLM08 identifies cross-context information leaks and federation knowledge conflict as vector and embedding risks, and Microsoft's AI Recommendation Poisoning disclosure shows that persistent "trusted source" claims can bias later recommendations. Contradiction checks help detect poisoned updates before they become accepted memory. | For each new memory write or knowledge-base update, extract claims and compare them with authoritative sources, prior approved memories, policy documents, and recent conflicting chunks. Use natural-language inference models, semantic similarity plus polarity checks, knowledge-graph triples, source-of-truth precedence rules, and reviewer workflows. Test by ingesting documents that contradict known policies, dates, owners, entitlements, prices, safety instructions, or source trust rankings. Verify that conflicts create alerts, quarantine the new content, preserve both versions for investigation, and avoid silent merging. Confirm that rollback can remove a poisoned batch from the vector index, document store, caches, and generated summaries. | Automated contradiction detection remains an incomplete NLP control. It works best for explicit factual conflicts and performs poorly on subtle framing, omissions, stale-but-once-true facts, and domain-specific policies. Full pairwise comparison does not scale, so practical systems compare against curated golden sources, high-trust memories, and topically nearest approved chunks. Alerts should include provenance and retrieval impact, not just a model-generated contradiction label. |
