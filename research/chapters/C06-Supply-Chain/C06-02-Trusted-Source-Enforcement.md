# C6.2: Trusted Source Enforcement for AI Artifacts

> **Parent:** [C06: Supply Chain Security](C06-Supply-Chain.md) | **Requirements:** 2 | **IDs:** 6.2.1–6.2.2

## Purpose

AI pipelines download large artifacts — model weights (often multi-GB), datasets, container images, and specialized libraries — from a variety of external sources. Without strict source enforcement, any network-accessible endpoint becomes a potential supply-chain injection point. This section ensures that all artifact sources are explicitly approved, cryptographically verified, and network-enforced, reducing the attack surface to known, trusted origins.

---

## 2025-2026 Landscape Update

### Malicious Models at Scale: The Numbers Keep Growing

The partnership between Protect AI and Hugging Face, now over a year into operation, has scanned 4.47 million unique model versions across 1.41 million repositories, identifying 352,000 unsafe or suspicious issues across 51,700 models. JFrog's independent analysis flagged 25 models as zero-day malicious — models not detected by any other scanner. As of 2025, JFrog's Software Supply Chain Report documented a **6.5-fold increase** in malicious models on Hugging Face compared to the prior year, with over 1 million new models hitting the platform annually. Malicious package uploads to open-source repositories overall jumped 156% year-over-year.

Common attack patterns on Hugging Face include:

- **Pickle deserialization attacks**: Malicious Python code embedded in `.pt` and `.pkl` model files that executes during model loading. Protect AI and Cisco's ClamAV integration now scan for these at upload time.
- **Library-dependent attack chains**: Exploiting functions from common ML environment libraries (PyTorch, NumPy, Pandas) to stage payloads that appear benign individually but chain into full compromise.
- **Payload obfuscation**: Using compression, encoding, and nested container formats (Keras `.h5`, NeMo `.nemo`) to hide malicious code from single-layer scanners.
- **Framework extensibility exploits**: Abusing custom layer definitions and configuration-based loading, including CVE-2025-1550 in Keras which enabled arbitrary code execution through custom layers.
- **NullifAI evasion techniques**: In early 2025, ReversingLabs discovered malicious models using "nullifAI" evasion that bypassed Picklescan detection entirely. Hugging Face responded within 24 hours, removing the models and updating Picklescan, but the episode demonstrated that scanner evasion is an active arms race.

As of March 2026, model scanning is expanding beyond Protect AI and JFrog. [Socket.dev announced experimental malware scanning for Hugging Face](https://socket.dev/blog/announcing-experimental-malware-scanning-for-hugging-face), bringing supply chain protection to AI model artifacts. The Socket Model Scanner inspects PyTorch, TensorFlow, Keras, TFLite/LiteRT, GGUF, and Llamafile formats for hidden malware and backdoors, while the Socket Package Scanner examines metadata and associated code for malware injection, typosquatting, and credential leakage. Developers can access the scanning through a PURL API or by uploading an AI-BOM, which integrates naturally with the AI-BOM practices described in C6.7.

### Fake Privacy-Filter Model on Hugging Face (May 2026)

On May 7, 2026, [HiddenLayer reported](https://www.hiddenlayer.com/research/malware-found-in-trending-hugging-face-repository-open-oss-privacy-filter) that a Hugging Face repository named `Open-OSS/privacy-filter` had typosquatted OpenAI's legitimate Privacy Filter release, copied the model card nearly verbatim, and reached the platform's trending list with more than 200,000 downloads before removal. The repository's `loader.py` displayed plausible model-loading behavior, then silently disabled SSL verification, fetched a command from a JSON paste service, and launched hidden PowerShell on Windows to retrieve a second-stage batch file and a Rust-based infostealer. HiddenLayer also tied the same loader infrastructure to six other Hugging Face repositories under an `anthfu` account, all uploaded on April 24, 2026.

This is a useful audit pattern because the artifact passed the first social-trust checks that many teams still rely on: a familiar brand name, copied documentation, trending placement, and inflated likes. Trusted-source policies should therefore require namespace ownership validation, repository-age and engagement-anomaly review, mandatory offline or mirrored loading for approved models, and scanner coverage for helper scripts such as `loader.py`, `start.bat`, notebooks, and install instructions — not just model weight files.

### The LiteLLM Supply Chain Cascade (March 2026)

On March 24, 2026, LiteLLM — an open-source Python library used to route API calls across hundreds of LLMs, averaging 95 million monthly downloads — was [compromised in a supply chain attack by threat actor TeamPCP](https://securitylabs.datadoghq.com/articles/litellm-compromised-pypi-teampcp-supply-chain-campaign/). The attacker published backdoored versions (1.82.7 and 1.82.8) to PyPI after stealing CI/CD credentials through a chain of upstream compromises. LiteLLM serves as a foundational dependency for frameworks like CrewAI, DSPy, and MLflow, and organizations including Stripe, Netflix, and Google.

The attack timeline illustrates how a single credential compromise cascades across ecosystems:

1. **March 19**: Trivy GitHub Action credentials compromised (related to the earlier Trivy VS Code incident)
2. **March 20–22**: Self-propagating npm worm deployed across 44+ packages
3. **March 23**: Checkmarx and OpenVSX repositories targeted
4. **March 24**: LiteLLM PyPI packages poisoned; versions 1.82.7 and 1.82.8 published
5. **March 27**: Telnyx packages (4.87.1, 4.87.2) also backdoored

The payload in version 1.82.8 was particularly insidious — a malicious `.pth` file (`litellm_init.pth`) that executes automatically on *every* Python process startup when litellm is installed, not just when the library is imported. The three-stage payload harvested credentials (SSH keys, cloud tokens, Kubernetes secrets, `.env` files), attempted lateral movement by deploying privileged pods to every Kubernetes node, and installed a persistent systemd backdoor polling for additional payloads. Exfiltration used AES-256/RSA-4096 encryption to `models.litellm[.]cloud`.

The malicious versions were available for approximately three hours before PyPI quarantined the package. Detection indicators include the `litellm_init.pth` filesystem artifact, DNS queries to exfiltration domains, and anomalous Kubernetes secret access patterns. [Sonatype](https://www.sonatype.com/blog/compromised-litellm-pypi-package-delivers-multi-stage-credential-stealer) and [Snyk](https://snyk.io/articles/poisoned-security-scanner-backdooring-litellm/) published independent analyses confirming the attack chain.

This incident reinforces that trusted source enforcement must cover the entire dependency graph — a model serving library (not just model weights) can be the vector. Organizations should treat any system that installed affected versions as a full credential exposure event and rotate all reachable secrets. In the aftermath, LiteLLM [opened an issue to migrate to PyPI Trusted Publishers](https://github.com/BerriAI/litellm/issues/24542) (OIDC-based publishing that eliminates long-lived API tokens), a remediation pattern that other AI library maintainers should follow.

### The LiteLLM Cascade: From Library Compromise to AI Training Data Exposure

The LiteLLM supply chain attack had consequences well beyond the initial three-hour package exposure. On March 31, 2026, [Mercor](https://techcrunch.com/2026/03/31/mercor-says-it-was-hit-by-cyberattack-tied-to-compromise-of-open-source-litellm-project/) — a $10-billion AI recruiting startup that handles custom datasets and training methodologies for OpenAI, Anthropic, and Meta — confirmed it was among thousands of organizations affected by the LiteLLM compromise. Attackers (with Lapsus$ claiming responsibility) obtained an estimated 4TB of stolen data including candidate profiles, PII, employer data, source code, and API keys. By April 9, 2026, [Meta had indefinitely paused contracts with Mercor](https://techcrunch.com/2026/04/09/after-data-breach-10b-valued-startup-mercor-is-having-a-month/) due to concerns about exposure of AI training secrets, and six class-action lawsuits had been filed by affected workers.

This cascade — from a 40-minute window of poisoned PyPI packages to a multi-billion-dollar AI training data exposure — is the clearest illustration to date of why trusted source enforcement must cover the entire dependency graph, not just model weights. A compromised model-serving library reached AI training infrastructure at multiple frontier labs.

### Unit 42 TeamPCP Analysis: Advanced Supply Chain Tradecraft

Palo Alto Unit 42 published a [detailed analysis of TeamPCP's supply chain campaign](https://unit42.paloaltonetworks.com/teampcp-supply-chain-attacks/) revealing attack techniques that went beyond simple credential theft. Key findings:

- **Scale**: 300 GB of data and approximately 500,000 credentials exfiltrated across at least 16 publicly disclosed victim organizations, with 48+ additional npm packages compromised via harvested tokens.
- **CanisterWorm**: A self-replicating malware with wiper capabilities that targets Kubernetes clusters, masquerading as legitimate services (systemd, PostgreSQL) to evade detection.
- **WAV steganography**: Encrypted payloads hidden inside WAV audio files to bypass content-based detection.
- **Memory reading via `/proc/<pid>/mem`**: Bypasses GitHub Actions secret masking by reading process memory directly, extracting secrets that would otherwise be redacted in logs.
- **Decentralized C2**: Command-and-control infrastructure hosted on Internet Computer Protocol (ICP) canisters, making takedown significantly harder than traditional domain-based C2.

These techniques represent a significant escalation in supply chain attack sophistication. The `/proc/mem` secret extraction technique in particular means that CI/CD secrets are not safe even with standard masking — organizations must assume that any secret accessible to a compromised GitHub Action is fully exposed.

### CI/CD Pipeline Vulnerabilities in ML Projects

A March 2026 analysis by [Mitiga Labs](https://www.mitiga.io/blog/inside-the-ai-supply-chain-security-lessons-from-10-000-open-source-ml-projects) examined 10,000 open-source AI/ML repositories and found that **70% contained at least one critical or high-severity GitHub Actions vulnerability**. The top vulnerability classes by prevalence:

| Vulnerability Class | Prevalence | Risk to AI Supply Chain |
|---|---|---|
| Unpinned third-party Actions (mutable tags like `@v4`) | 68.4% | Compromised action repository enables model poisoning, credential theft |
| Overprivileged `GITHUB_TOKEN` | 42.7% | Enables repo takeover, model card poisoning, malicious release publishing |
| Script/command injection in workflow steps | 34.1% | PR titles or issue bodies can exfiltrate tokens (HF_TOKEN, WANDB_API_KEY) |
| Unsafe workflow triggers (`pull_request_target`) | 27.2% | Single malicious PR achieves RCE with repository write access |
| Hard-coded or leaked secrets | 22.8% | Credentials for private registries or GPU cloud accounts exposed in logs |

These findings are directly relevant to trusted source enforcement — even when organizations mandate approved registries, compromised CI/CD pipelines can bypass those controls by modifying build scripts or exfiltrating registry credentials. Key mitigations include pinning Actions to full 40-character commit SHAs, enforcing least-privilege `GITHUB_TOKEN` permissions per workflow, adopting OIDC for cloud authentication instead of long-lived secrets, and eliminating string interpolation in `run:` steps.

### Model Namespace Reuse and Model Confusion Attacks

Palo Alto Unit 42 documented a "model namespace reuse" attack in early 2025 where cloud provider model catalogs retrieve models by name from Hugging Face. When a model author deletes or transfers their namespace, an attacker can re-register the abandoned namespace and publish a malicious model under the original path. Any pipeline that references the model by name (without hash pinning) would silently fetch the attacker-controlled version.

In January 2026, Checkmarx published research on a related technique they call **"AI Model Confusion"** — essentially dependency confusion adapted for model registries. Code that loads models using relative paths (e.g., `checkpoints/my-model`) can be tricked into fetching a remote model from Hugging Face if no local copy exists, because the Hugging Face client library falls back to remote resolution by default. Mitigations include setting `HF_HUB_OFFLINE=1`, using `local_files_only=True`, or specifying absolute paths. For enterprise environments, Hugging Face recommends subscribing to Enterprise Hub with an allowlist of approved models.

As of April 2026, the platform response to namespace-reuse has diverged. Google's Vertex AI Model Garden [now performs daily scans to identify orphaned namespaces](https://unit42.paloaltonetworks.com/model-namespace-reuse/) and automatically marks them as "verification unsuccessful," blocking deployment to Vertex endpoints. Hugging Face's mitigation remains more ad-hoc — a manually maintained retirement blocklist that expanded only after researchers reached out for specific popular namespaces — leaving the long tail of less-popular names still exploitable. Microsoft Azure AI has not publicly disclosed a programmatic mitigation. Trusted-source policies should not assume uniform platform hygiene: teams pulling models from Hugging Face by name should still pin by `revision` (commit hash) or clone to an internal mirror regardless of upstream platform defenses, because detection lag between namespace abandonment and blocklist update creates an exploitation window that neither daily scans nor human-curated lists fully close.

### Model Signing: From Nascent to v1.0

Model signing has progressed significantly. In April 2025, Google's Open Source Security Team, OpenSSF, NVIDIA, and HiddenLayer released **model-signing v1.0** — the first stable release of the [OpenSSF Model Signing (OMS)](https://github.com/sigstore/model-transparency) specification. Key capabilities as of April 2026:

- **Sigstore-based keyless signing**: Short-lived certificates bound to OpenID Connect tokens, with all signing events recorded in a transparency log for auditability.
- **Platform integration**: Kaggle and Hugging Face support automatic signing during model upload; TensorFlow and PyTorch frameworks support signing via `model.save()` hooks.
- **`pip install model-signing`**: CLI and library for signing and verifying models locally. The subsequent **v1.1.x** releases ([PyPI](https://pypi.org/project/model-signing/), currently 1.1.1) broadened cryptographic flexibility with **BLAKE3 hashing** support for faster multi-GB model hashing, **PKCS #11 HSM integration** for hardware-backed signing keys, and a `--trust_config` option for pointing signing operations at private Sigstore instances (useful for air-gapped or regulated environments that cannot use the public good instance).
- **Kubernetes enforcement**: The [Sigstore Model Validation Operator](https://blog.sigstore.dev/model-validation-operator-v1.0.1/) (v1.0.1) uses mutating webhooks to inject init containers that verify model signatures before workload pods start. If verification fails, the pod is blocked. Currently supports models on PVs, S3/MinIO, and shared volumes.
- **OCI artifact packaging**: Work is underway to package ML models as OCI artifacts, enabling lightweight verification via signed manifests without downloading full model files.

Google now uses OMS across its internal model infrastructure, with Kaggle and NVIDIA NGC both supporting model signing as of mid-2025. Sigstore-signed attestations have also been adopted by Homebrew, PyPI, and Maven Central, establishing a pattern that ML model hubs are following.

For raw model weight files (safetensors, ONNX, GGUF) outside platform-managed hubs, organizations should still implement internal signing-upon-import as a compensating control until OMS adoption broadens further.

### SafeTensors Joins PyTorch Foundation (April 2026)

On April 8, 2026, the [PyTorch Foundation announced](https://www.linuxfoundation.org/press/pytorch-foundation-announces-safetensors-as-newest-contributed-project-to-secure-ai-model-execution) that Hugging Face's SafeTensors format has joined as an officially hosted project under the Linux Foundation, alongside DeepSpeed, Helion, PyTorch, Ray, and vLLM. SafeTensors eliminates the arbitrary code execution risk inherent in pickle-based model serialization — a vulnerability class responsible for the majority of malicious models found on Hugging Face. The Linux Foundation's CTO described SafeTensors as "a de facto standard for open-weight model distribution." With vendor-neutral governance, formalized maintainer structure, and a roadmap targeting device-aware loading, parallel loading for distributed deployments, and quantization support, SafeTensors is now positioned as critical supply chain infrastructure rather than a single-vendor project. Organizations implementing trusted source policies should mandate SafeTensors (or ONNX) for all model artifacts where format choice is possible, reducing the attack surface from deserialization exploits to weight-level trojans only.

### NSA/Five Eyes AI Supply Chain Guidance (March 2026)

In March 2026, the NSA together with Five Eyes partners (ASD, CCCS, GCHQ, NCSC-NZ) released [CSI: AI/ML Supply Chain Risks and Mitigations](https://www.cyber.gov.au/business-government/secure-design/artificial-intelligence/artificial-intelligence-and-machine-learning-supply-chain-risks-and-mitigations), covering six supply chain components: training data, models, software, infrastructure, hardware, and third-party services. Key recommendations relevant to trusted source enforcement:

- Establish **verified model registries** with cryptographic signing across the model lifecycle.
- Require an **AI Bill of Materials (AI-BOM)** identifying models, datasets, libraries, and external services alongside a traditional SBOM.
- Deploy integrity checks and malware scanning **before** operational deployment.
- Define role-based access to models, training data, and configurations with segregation of duties.
- Evaluate external AI providers for inherited risks from their own sub-suppliers.

This guidance provides strong institutional backing for the controls in this section, particularly 6.4.1 (approved sources), 6.4.2 (cryptographic verification), and 6.4.4 (periodic review).

### Private Model Registry Options

Organizations implementing trusted source policies have several options for internal model registries:

- **Hugging Face Enterprise Hub**: Private instance with access controls, audit logging, and model allowlisting. Now recommended by Hugging Face's own security team for large organizations as a mitigation against model confusion attacks.
- **NVIDIA NGC Private Registry**: For NVIDIA GPU-optimized models with enterprise security features; now supports Sigstore-based model signing as of mid-2025.
- **JFrog Artifactory ML**: General-purpose artifact management with ML model support; JFrog and Hugging Face announced a partnership in 2025 to improve ML security and transparency.
- **AWS SageMaker Model Registry / Azure ML Model Registry / GCP Vertex AI Model Registry**: Cloud-native options with IAM integration.
- **Self-hosted solutions**: MinIO or S3-compatible stores with custom admission policies.

### VirusTotal Integration with Hugging Face (October 2025)

In October 2025, [Hugging Face and VirusTotal announced a collaboration](https://huggingface.co/blog/virustotal) to continuously scan all 2.2 million+ public model and dataset repositories on the Hub. The integration compares file hashes against VirusTotal's threat-intelligence database — covering 70+ antivirus engines — without sharing raw file contents. Results surface automatically on repository pages, giving developers immediate visibility into whether any files have been flagged across the VirusTotal ecosystem. This complements the existing Protect AI/Guardian scanning (format-level pickle/custom-layer detection) and Cisco ClamAV integration (deserialization risk scanning) with a broader malware intelligence layer. Organizations can integrate VirusTotal check results into CI/CD workflows for automated gating.

### Weight-Level Backdoor Detection: Moving Beyond Format Scanning

Traditional model scanning focuses on unsafe serialization formats (pickle, custom layers), but a new category of tooling targets backdoors embedded directly in learned weights — so-called "sleeper agents" that behave normally until a specific trigger is encountered. In February 2026, Microsoft Research published a [practical scanner for detecting backdoored language models at scale](https://www.microsoft.com/en-us/security/blog/2026/02/04/detecting-backdoored-language-models-at-scale/). The scanner uses three behavioral signals: trigger attention spikes (abnormal attention patterns on specific tokens), memorized poisoning data leaks (coaxing models into regurgitating fragments of the data used to implant the backdoor), and fuzzy trigger activation (testing near-trigger inputs to detect activation gradients). It works on models from 270M to 14B parameters using forward passes only — no retraining required — with a low false-positive rate. The scanner requires access to model weights and tokenizer, making it applicable to open-weight models but not black-box APIs.

This represents an important shift: organizations can no longer rely solely on format-level scanning (Picklescan, ClamAV) and must also consider weight-level behavioral audits for high-risk models. Safer serialization formats like SafeTensors and ONNX separate weights from executable logic, eliminating deserialization-based attacks, but do not prevent weight-level trojans embedded during training or fine-tuning.

### The Trivy VS Code Incident: Insider Threats in Developer Tools (February 2026)

In late February 2026, a former employee with retained publishing credentials uploaded [malicious versions of the Aqua Trivy VS Code extension](https://www.netcrook.com/trivy-vs-code-ai-supply-chain-attack-2026/) to the OpenVSX marketplace. The compromised versions (1.8.12 and 1.8.13) contained crafted prompts that hijacked installed AI coding assistants — Copilot, Claude, Gemini, Codex, and Kiro — directing them to act as "forensic agents" that scanned the developer's system for credentials and financial records, then exfiltrated findings via email, Slack, or GitHub. The malicious versions were detected and removed within 36 hours, but thousands of developers were potentially exposed.

This incident illustrates that trusted source enforcement must extend beyond model artifacts to the entire development toolchain. Even trusted security tools (Trivy is one of the most widely deployed vulnerability scanners) can become supply chain vectors when publishing credentials are not properly revoked. It also highlights a new attack surface: AI coding assistants themselves can be weaponized through prompt injection embedded in trusted extensions.

### The OpenClaw/ClawHub ClawHavoc Campaign: Supply Chain Attacks Hit AI Agent Skill Registries

In early February 2026, Koi Security researchers disclosed that [ClawHub — the package registry for the OpenClaw AI agent framework — contained 341 malicious skills](https://www.esecurityplanet.com/threats/hundreds-of-malicious-skills-found-in-openclaws-clawhub/) in a coordinated campaign dubbed "ClawHavoc." Subsequent scans by Antiy CERT expanded the count to 1,184 malicious skills — approximately one in five packages in the ecosystem at its peak. The campaign represents the largest confirmed supply chain attack targeting AI agent infrastructure to date.

Attack techniques included typosquatting legitimate skill names, social engineering via fake "prerequisite" installation steps, reverse shell backdoors embedded in functional code, credential exfiltration via webhook services, and password-protected ZIP archives to bypass antivirus scanning. The primary payload was [Atomic macOS Stealer (AMOS)](https://www.trendmicro.com/en_us/research/26/b/openclaw-skills-used-to-distribute-atomic-macos-stealer.html), which harvests browser credentials, keychain passwords, crypto wallets, SSH keys, and API tokens — particularly dangerous on always-on machines hosting AI agents.

Compounding the skill registry problem, [CVE-2026-25253](https://socradar.io/blog/cve-2026-25253-rce-openclaw-auth-token/) (CVSS 8.8) disclosed a one-click RCE chain in OpenClaw itself: the Control UI blindly trusted a `gatewayUrl` query parameter, automatically connecting to attacker-controlled WebSocket servers and leaking authentication tokens. Combined with missing Origin header validation on the WebSocket server, this enabled full remote code execution — even against localhost-bound instances. Over 40,000 internet-exposed OpenClaw instances were discovered, with 63% assessed as vulnerable. The flaw was patched in v2026.1.29.

This incident extends the trusted source enforcement perimeter beyond model artifacts, libraries, and MCP connectors to **AI agent skill registries**. Organizations deploying agentic AI frameworks should include skill/plugin registries in their approved-source policies, apply allowlisting and signature verification to agent skills, and enforce egress controls on agent runtime environments. The recommended mitigations — skill allowlisting, sandboxed execution, least-privilege access, and outbound network monitoring — parallel the controls already established for model artifacts and library dependencies.

### Haldir AI: Emerging Skill-Level Signing for Agent Ecosystems

Directly in response to the ClawHavoc incident, [Haldir AI](https://github.com/Haldir-AI) has published an open cryptographic signing standard — the **Agent Skill Attestation Format (ASAF)** — purpose-built for AI agent skills and MCP server packages. The project explicitly targets the gap exposed by ClawHavoc (where zero major agent registries shipped cryptographic signing) and independent research showing prompt-injection vulnerabilities in roughly 36% of skills across major registries. Technically, ASAF reuses the same primitives that protect npm, PyPI, and container images: DSSE v1.0.0 envelopes, Ed25519 signatures, SHA-256 integrity allowlists, and signed revocation statements with fail-closed install and fail-open runtime semantics. Haldir maintains Sigstore compatibility for keyless signing and transparency-log publication, so organizations can layer identity-pinning policies on top of ASAF attestations without running a separate trust root. As of April 2026 Haldir ships a published specification, a TypeScript reference implementation, and a GitHub Action for CI-time signing; adoption across major agent frameworks (OpenClaw, CrewAI, AutoGen, LangGraph) is still early. Organizations building custom agent skill libraries should track ASAF alongside the OpenSSF Model Signing spec — the two cover complementary parts of the agent supply chain.

### "Two Kinds of Attestation": Avoiding Term Collision Under EU AI Act

In February 2026, Andrew Nesbitt's ["Two Kinds of Attestation"](https://nesbitt.io/2026/02/25/two-kinds-of-attestation.html) flagged a regulatory risk that is directly relevant to trusted source enforcement: the word *attestation* is being used for two very different things. The first is **cryptographic build provenance** — the Sigstore-backed attestations npm and PyPI now emit by default for trusted publishers, which bind an artifact to its source repo, commit, and build workflow and land in a public transparency log. The second is a **human-evaluated security checklist** emerging from the EU Cyber Resilience Act and the EU AI Act implementation work, where "open-source stewards" self-attest (point-in-time) to hygiene indicators such as test coverage, vulnerability-reporting contacts, and lifecycle policies. The CRA Article 25 delegated act governing steward attestations is still unwritten as of early 2026, but once both meanings enter regulations, downstream verification tooling will inherit the ambiguity. Trusted-source policies should explicitly distinguish the two: cryptographic provenance is machine-verifiable and continuous; steward attestations are human judgment at a moment in time and must not be treated as a substitute for signature verification. Organizations writing approved-source policies for the updated EU AI Act timetable should state clearly which form of attestation is being required at each gate (publisher identity binding vs. project hygiene claim).

### Rekor v2 GA and Transparency Log Monitoring

Sigstore's Rekor transparency log reached [v2 General Availability in October 2025](https://blog.sigstore.dev/rekor-v2-ga/), bringing a redesigned tile-backed implementation that simplifies maintenance, lowers storage costs through CDN-cacheable reads, and supports higher query throughput via request batching. The new architecture reduces entry types to just `hashedrekord` and `dsse`, eliminating legacy types. Client support is available in Cosign v2.6.0+, v3.0.1+, and the Go, Python, and Java Sigstore clients. Sigstore plans to deploy a 2026 Rekor v2 instance and eventually sunset the 2025 shard.

In parallel, Trail of Bits (funded by OpenSSF) has been [preparing rekor-monitor for production use](https://openssf.org/blog/2025/12/19/catching-malicious-package-releases-using-a-transparency-log/) — enabling package maintainers who publish Sigstore-signed attestations (as supported by PyPI and npm) to receive alerts when their identity is used for unauthorized releases. The tool monitors the append-only log for unexpected signing events, providing early detection of compromised release processes. A planned hosted service would allow maintainers to subscribe for notifications via a web form without self-hosting monitoring infrastructure. This proactive approach would have shortened the detection window in incidents like the Axios and LiteLLM compromises, where attackers published under hijacked identities.

### Cosign v3: Verification Defaults Tighten

In April 2026, [Cosign v3](https://blog.sigstore.dev/cosign-3-0-available/) shipped as a stable release, flipping a cluster of previously opt-in features to on-by-default — the most operationally significant change since keyless signing itself. For trusted-source enforcement, three defaults matter:

- **New Sigstore bundle format is now required on the verify path.** The `--bundle` flag moved from optional to required, and bundles are the unit that carries verification material (certificate, signature, log inclusion proof) for offline verification. Pipelines still emitting legacy signature+payload pairs must migrate before upgrading to v3.
- **Trusted Root is the default trust material.** Instead of clients bundling hardcoded Sigstore public keys, Cosign v3 pulls a signed "trusted root" document that lists current Rekor, Fulcio, and CT log keys. This decouples key rotation from client upgrades — a direct response to the Rekor v2 migration and the need to rotate transparency-log keys without re-releasing every verification client.
- **Signing Config support is on by default**, allowing transparency-log shards to rotate independently of client versions.

The Sigstore team has signaled that v3 is a short-lived bridge: "there probably won't be many releases of Cosign v3, and soon we will start removing the old functionality for the initial release of Cosign v4," which will drop roughly half of the current flags. Organizations that pinned Cosign verification policies to v2 behavior should plan the v2 to v3 migration now and budget a second migration when v4 lands. For ML workflows specifically, the defaults mean that signed model bundles produced by `model-signing` v1.1.x verify cleanly against Cosign v3 without interoperability flags — closing a gap that previously required coordinating client versions across signing and verification systems.

### Trusted-Publisher Adoption: The Positive Side of the Attestation Picture

While the [2026 npm audit](https://dev.to/thecryptodonkey/i-audited-the-top-50-npm-packages-almost-none-ship-with-supply-chain-attestations-3ki8) found only 12% of the top 50 packages shipping SLSA provenance, the PyPI trajectory tells a more encouraging story. Per [Socket's adoption report](https://socket.dev/blog/adoption-of-trusted-publishers-growing-among-open-source-package-repositories) and PyPI's own statistics, more than **50,000 PyPI projects** had adopted Trusted Publishers by late 2025, accounting for **roughly 20% of all file uploads** to the index over the prior year — over one million files published via OIDC-based trusted publishing since the feature launched in April 2023. RubyGems, Dart pub.dev, and crates.io (Rust) followed PyPI's design. npm's Trusted Publishing feature, launched in 2025, [added CircleCI support alongside GitHub Actions in April 2026](https://docs.npmjs.com/trusted-publishers/), broadening the CI/CD providers that can publish without long-lived tokens. Critically, the per-package enrollment friction that previously throttled adoption for maintainers with large portfolios has been addressed in npm CLI v11.10.0 (GA February 18, 2026): the new [`npm trust` command](https://docs.npmjs.com/cli/v11/commands/npm-trust/) now allows bulk configuration of trusted publishing across many packages in a single operation. In practice, a maintainer can script a loop over all their packages and configure approximately 80 packages within the 5-minute 2FA skip window, enabling wholesale migration off long-lived classic tokens. For AI/ML library supply chains, this is the inflection point where approved-source policies can begin to treat OIDC-based publishing as table-stakes for any actively maintained upstream — the remaining holdouts are no longer excused by tooling friction. Organizations should check trusted-publisher status per library and, as adoption accelerates post-Axios, raise the risk threshold for libraries still shipping via long-lived tokens.

### npm `minimumReleaseAge` and Script Security: Ecosystem-Wide Release Cooldowns (February 2026)

The Axios compromise's three-to-four-hour exposure window triggered rapid adoption of time-based release gating across the JavaScript ecosystem. By April 2026, all major Node.js package managers support some form of release-age gating — pnpm added `minimumReleaseAge` in v10.16 (September 2025), Yarn added `npmMinimalAgeGate` in v4.10.0 (September 2025), Bun added `minimumReleaseAge` in v1.3 (October 2025), and [npm itself shipped `minimumReleaseAge` in CLI v11.10.0](https://socket.dev/blog/npm-introduces-minimumreleaseage-and-bulk-oidc-configuration) on February 18, 2026. Recommended values are 24–72 hours: long enough for upstream detection and takedown of poisoned versions, short enough to not block legitimate security patches. For AI/ML CI/CD workflows specifically, this gates the exact attack pattern the Axios attackers depended on — rapid distribution of a malicious version before detection. An AI coding agent running `npm install` behind a 72-hour release-age gate would have been immune to the 39-minute Axios window that cascaded into the OpenAI macOS signing pipeline. Organizations should configure `minimumReleaseAge` (or its ecosystem equivalents) in approved-source policies and grant exceptions only for internally maintained packages via explicit allowlists.

The same npm v11.10.0 release also introduced the `--allow-git` flag, addressing a separate attack vector: git dependencies can ship `.npmrc` files that override the `git` executable path, enabling code execution during install even with `--ignore-scripts` enabled. The recommended setting is `--allow-git=none`, which is [expected to become the default in npm CLI v12](https://github.blog/changelog/2026-02-18-npm-bulk-trusted-publishing-config-and-script-security-now-generally-available/). ML repositories that pull Python training code with git-based Node tooling (or that vendor JavaScript tools alongside model serving) should flip this setting now rather than wait for the default change.

### The Axios npm Compromise: Nation-State Supply Chain Attack (March 2026)

On March 31, 2026, the [axios npm package](https://thehackernews.com/2026/03/axios-supply-chain-attack-pushes-cross.html) — an HTTP client with over 70 million weekly downloads — was compromised when a North Korean state actor (tracked as Sapphire Sleet by Microsoft, UNC1069 by Google) hijacked maintainer "jasonsaayman"'s npm account and changed its registered email to a Proton Mail address under attacker control. Backdoored versions 1.14.1 and 0.30.4 were published with a malicious dependency that deployed cross-platform RATs (macOS, Windows, Linux) within two seconds of `npm install` — before npm had even finished resolving dependencies. Both release branches were hit within 39 minutes, and the malicious dependency had been staged 18 hours in advance.

This incident is directly relevant to trusted source enforcement for several reasons:

- **Identity substitution without key pinning**: The attacker changed the account's registered email and published under the original identity. Any verification system not pinned to a specific OIDC identity or cryptographic key would accept the malicious packages as legitimate.
- **AI coding agent amplification**: As [multiple researchers noted](https://www.stepsecurity.io/blog/axios-compromised-on-npm-malicious-versions-drop-remote-access-trojan), AI coding agents executing `npm install` in the background eliminate the human friction that previously provided a detection window for supply chain attacks. A compromised package can infect a system in a 179-minute window even if `npm audit` later returns clean.
- **Credential rotation imperative**: [Microsoft's mitigation guidance](https://www.microsoft.com/en-us/security/blog/2026/04/01/mitigating-the-axios-npm-supply-chain-compromise/) recommended treating any system that installed affected versions as a full credential exposure event — the same guidance given for the LiteLLM incident days earlier.

### OpenAI Certificate Rotation: When Supply Chain Attacks Hit Code Signing (April 2026)

The Axios compromise cascaded further than initially apparent. On April 11, 2026, [OpenAI disclosed](https://socket.dev/blog/axios-supply-chain-attack-reaches-openai-macos-signing-pipeline-forces-certificate-rotation) that a GitHub Actions workflow used in its macOS app-signing process had downloaded and executed malicious Axios version 1.14.1 on March 31. The workflow had access to certificates and notarization material used for signing ChatGPT Desktop, Codex, Codex CLI, and Atlas. OpenAI's [official response](https://openai.com/index/axios-developer-tool-compromise/) identified the root cause as a GitHub Actions workflow misconfiguration — specifically, the Action was pinned to a floating tag rather than a specific commit SHA, and the workflow had no configured `minimumReleaseAge` gate for new packages. Both preventative controls (pinning and release-age gating) would have blocked the compromised axios versions from entering the signing environment.

OpenAI assessed that the signing certificate was likely not successfully exfiltrated due to timing and sequencing factors, but treated it as compromised — revoking and rotating all affected certificates. Older macOS app versions signed with the previous certificate will be blocked by macOS security protections after May 8, 2026, affecting [ChatGPT Desktop, Codex, Codex CLI, and Atlas](https://openai.com/index/axios-developer-tool-compromise/) — users running older builds must upgrade or lose auto-update and launch capability. OpenAI also engaged a [third-party digital forensics and incident response firm](https://thehackernews.com/2026/04/openai-revokes-macos-app-certificate.html) and publicly committed to a full third-party library audit, immediate invalidation of all access tokens, real-time supply chain scanners, and enhanced internal authentication. The April 12, 2026 disclosure confirmed no user data, systems, intellectual property, or software was altered. This incident demonstrates that supply chain attacks on common libraries can reach code-signing infrastructure at major AI companies, and that CI/CD hygiene (SHA-pinned dependencies, release age gates, `--allow-git=none`) is not optional even for well-resourced organizations. The companion takeaway: a dependency-based supply chain attack now produces end-user consequences (forced app updates, revoked trust chains) visible at the OS level — not just backend compromise.

### Axios Post-Mortem: Maintainer Device as the Attack Surface (April 2026)

The [public axios post-mortem (GitHub issue #10636)](https://github.com/axios/axios/issues/10636) published after the compromise clarifies how the attack actually started — and the lessons are blunt for approved-source policies. The initial access vector was not an npm infrastructure compromise or a public code vulnerability: the attacker ran a targeted social-engineering campaign against the lead maintainer's personal computer, installed RAT malware on that device roughly two weeks before the package publish, and used the already-authenticated npm session to push the backdoored versions. The post-mortem also discloses that prior to the incident, axios had no automated monitoring for unauthorized publish events, so detection relied entirely on community reporters filing a GitHub advisory — in this case Elastic Security Labs at 01:50 UTC on March 31.

The remediation commitments axios has published are the most concrete template to date for a post-compromise open-source project:

- **OIDC-based publishing** (npm Trusted Publishers) to eliminate long-lived tokens on personal machines.
- **Immutable release process** run from isolated, hardened CI rather than maintainer devices.
- **Automated publish monitoring** that alerts on any release not originating from the approved CI pipeline.
- **No publishing from personal accounts** — releases flow through a project-owned publisher identity.

For AI/ML library consumers writing approved-source policies, the axios post-mortem supports three concrete policy statements: (1) trust claims about an upstream project should include whether that project publishes via OIDC Trusted Publishers (verifiable from the registry, not self-attested), (2) any project still accepting publishes from maintainer laptops is a higher-risk source and should either be pinned by commit SHA with `minimumReleaseAge` or vendored, and (3) internal mirrors of critical AI libraries should capture a known-good baseline so that rollback to pre-compromise versions is a documented one-command operation rather than forensic cleanup during an active incident.

### npm Classic Token Decommissioning and the Forced Migration (December 2025 – April 2026)

The Axios and LiteLLM compromises made the case that long-lived classic tokens are unfixable in principle, and the ecosystem response accelerated. [npm classic tokens were revoked on December 9, 2025](https://philna.sh/blog/2026/01/28/trusted-publishing-npm/), forcing a migration to either Trusted Publishers (OIDC-based) or granular access tokens. This is the first ecosystem-wide forced decommission of long-lived publisher credentials by a major package registry and is a useful precedent for AI model registry policy. Combined with the npm CLI v11.10.0 `npm trust` bulk configuration command (February 2026) and CircleCI joining GitHub Actions as a supported npm Trusted Publisher provider (April 2026), the practical friction for maintainers moving to OIDC is now low enough that approved-source policies can reasonably treat continued classic-token usage as a red flag rather than an acceptable legacy pattern. Organizations should query the npm registry API to confirm that actively maintained AI/ML dependencies in their approved list publish via Trusted Publishers and flag any that have not migrated ten months after the classic-token revocation.

### Google Vertex AI "Double Agents": Supply Chain Blast Radius from the Agent Layer (March 2026)

### MCP Connector Poisoning: Extending Trust to AI Tool Integrations (April 2026)

### CVE-2026-1839: Checkpoint-Based RCE in HuggingFace Transformers

In early 2026, [CVE-2026-1839](https://advisories.gitlab.com/pkg/pypi/transformers/CVE-2026-1839/) disclosed a remote code execution vulnerability in the HuggingFace Transformers `Trainer` class (CVSS 6.5). The `_load_rng_state()` method at line 3059 of `trainer.py` calls `torch.load()` without `weights_only=True`, making it susceptible to pickle deserialization when processing untrusted checkpoint files (e.g., `rng_state.pth`). The vulnerability affects all Transformers versions supporting `torch>=2.2` when paired with PyTorch versions below 2.6. An attacker can exploit this by supplying a malicious checkpoint file during what appears to be a normal training resume — a realistic scenario in collaborative ML workflows where teams share checkpoints across environments. The fix landed in Transformers v5.0.0rc3; upgrading PyTorch to 2.6+ also mitigates the issue independently. This CVE underscores that even well-audited libraries can carry unsafe deserialization patterns in less-obvious code paths like RNG state restoration.

### MLflow Registry CVEs: Model Registry Itself as a Supply Chain Weak Point (April 2026)

On April 7, 2026, CERT Polska disclosed two MLflow vulnerabilities affecting all versions through 3.10.1 — [CVE-2026-33865](https://cert.pl/en/posts/2026/04/CVE-2026-33865/) (stored XSS via unsafe YAML parsing of `MLmodel` artifacts) and CVE-2026-33866 (authorization bypass on the AJAX endpoint that downloads saved model artifacts). Critically, the authorization bypass is unauthenticated — a user without permissions to an experiment can directly query the download endpoint and retrieve model artifacts they were never granted access to. The XSS path enables an authenticated uploader to plant payloads in `MLmodel` files that execute when teammates view the artifact in the UI, enabling session hijacking against privileged users who approve or promote models.

This is the fourth auth-related MLflow flaw in roughly six weeks, with three critical disclosures since February 20, 2026. The pattern matters for trusted source enforcement because MLflow is often treated as "the internal registry" — the place approved-source policies point to — yet it has repeatedly shipped with integrity-breaking bugs in its own artifact-handling code. Organizations cannot treat model-registry software as a black box: the registry software itself needs CVE tracking, upgrade SLAs, and network isolation equivalent to what they apply to model artifacts. Recommended mitigations: upgrade MLflow immediately once patched releases ship, audit artifact-access logs for direct-endpoint download patterns that bypassed the UI, restrict MLflow to authenticated networks only, and add WAF rules scoped to the AJAX download path. This episode also argues for keeping model artifact integrity verification (Sigstore/OMS signatures) outside the registry trust boundary, so that a compromised registry cannot silently swap artifacts downstream consumers have already accepted.

### Python ML Library Metadata Poisoning (January 2026)

A distinct attack class disclosed in January 2026 by security researchers affects ML Python libraries that use [Meta's Hydra configuration framework](https://www.theregister.com/2026/01/13/ai_python_library_bugs_allow/) — namely NVIDIA's NeMo, Salesforce's Uni2TS, and the Apple/EPFL FlexTok project. Malicious code hidden inside model *metadata* (not weights) executes automatically when a file containing poisoned configuration is loaded, bypassing format-level scanners that only inspect weight contents. Hugging Face does not currently flag files using the NeMo file format (`.nemo`) as potentially unsafe in the same way it flags pickle files, and the metadata contents are not as easily inspectable in the UI as other file types. For trusted source enforcement, this means format-allowlisting alone (mandate SafeTensors, ONNX) is insufficient when configuration-management libraries like Hydra can execute arbitrary code from config fields. Approved-source policies should include a static-analysis pass over any ML library that processes configuration files (Hydra, OmegaConf, custom YAML loaders) as part of vendor vetting, and runtime sandboxing for model-loading code paths where config-driven code execution is architecturally possible.

### OCI Artifacts for ML Models: KAITO's Production Implementation

The abstract claim that "OCI artifact packaging is coming for ML models" now has a concrete reference implementation. Cisco's State of AI Security 2026 found that only 29% of organizations feel ready to secure their AI deployments despite 83% planning agentic AI. Protect AI Guardian now scans 35+ model formats including Joblib, TensorFlow SavedModel, and Llamafile. ## Open Research Questions

- How should organizations handle trust decisions for models accessed only via API (e.g., OpenAI, Anthropic, Google) where weights are never downloaded? The NSA guidance flags third-party AI service providers as a distinct supply chain component requiring their own risk assessment.
- What is the right model for "trust delegation" when Model A is fine-tuned from Model B — does trust in B's publisher extend to A? GUAC 1.0's graph model can support supply chain graph queries for transitive trust, but AI-BOM-specific integration is still maturing.
- With model-signing v1.0, GUAC 1.0 (now merged with Trustify), and SafeTensors under PyTorch Foundation governance, what is the adoption trajectory for end-to-end supply chain attestation? Will OCI artifact packaging for ML models reach production readiness in 2026, enabling signature verification without downloading multi-GB weight files?
- Given both namespace reuse and model confusion attack vectors, should all model references include cryptographic hashes rather than human-readable names, similar to container digest pinning? What tooling can enforce this in Jupyter notebook workflows?
- How should organizations assess "attack vector chaining" risk when scanning models, given that individual components may appear benign but combine into exploits (as seen with library-dependent attack chains on Hugging Face)?
- With the EU AI Act high-risk timetable now shifting to December 2027 for standalone systems and August 2028 for regulated-product systems, while NIST AI RMF and ISO 42001 continue to mature, how will organizations reconcile overlapping supply chain transparency requirements across jurisdictions without slowing down already-needed technical controls?
- Microsoft's backdoor scanner targets open-weight models up to 14B parameters. How will weight-level behavioral auditing scale to frontier models (100B+ parameters), and what is the path toward standardized backdoor detection benchmarks?
- The Trivy VS Code incident showed that AI coding assistants can be weaponized through prompt injection in trusted extensions. Should trusted source enforcement policies extend to AI tool configurations and extension marketplaces, not just model artifacts?
- The TeamPCP campaign demonstrated cross-ecosystem cascade (GitHub Actions → npm → PyPI) with a single initial credential compromise. What detection architecture can identify multi-hop supply chain attacks in real time, rather than relying on post-compromise forensics? Can GUAC's graph model or Datadog's execution context grouping provide transitive compromise tracing across package ecosystems?
- Mitiga found that 70% of ML repos have critical CI/CD vulnerabilities. Should trusted source enforcement policies require CI/CD security posture assessments for upstream model and library providers, not just the artifacts themselves?
- The Axios compromise showed that nation-state actors can hijack maintainer accounts and publish backdoored packages that pass identity checks. Does Sigstore's keyless model (identity-pinning via OIDC issuer + subject) provide sufficient protection against account takeover, given that Sigstore cannot verify whether signer accounts have 2FA enabled? What complementary controls are needed?
- MCP connector poisoning represents a new supply chain attack surface where tool descriptions — invisible to users — can instruct AI agents to exfiltrate data. Should MCP connector packages be subject to the same cryptographic signing and approved-source policies as model artifacts, and what tooling can enforce this in practice?
- The OpenAI certificate rotation incident showed that supply chain attacks on common libraries can reach code-signing infrastructure. Should organizations isolate code-signing workflows in air-gapped or hardware-backed environments, and what is the operational cost of doing so at scale for AI applications with frequent release cycles?
- The Mercor breach demonstrated that supply chain attacks on AI libraries can reach AI training data and model development secrets at frontier labs. What contractual and technical controls should organizations require of data intermediaries who handle training datasets for multiple AI providers? Should trusted source policies extend to the vetting of data pipeline partners, not just software dependencies?
- TeamPCP's use of `/proc/mem` reading to bypass GitHub Actions secret masking, WAV steganography for payload delivery, and decentralized ICP canisters for C2 represents a significant tradecraft escalation. What detection architectures can identify these techniques in CI/CD environments, and should organizations assume all CI/CD secrets are exposed when any upstream action is compromised?
