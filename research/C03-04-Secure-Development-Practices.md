# C3.4: Secure Development Practices

> [Back to C03 Index](C03-Model-Lifecycle-Management.md)
> **Last Researched:** 2026-04-15 (refreshed)

## Purpose

Model development and training processes must follow secure practices to prevent compromise. This section covers environment separation, version control for development artifacts, training environment isolation with network controls, and training data integrity validation.

The attack surface during model development is significant and growing. The TeamPCP campaign (March 2026) demonstrated that even security tooling (Trivy) installed in training environments can become a vector for credential theft and lateral movement. IBM X-Force 2026 reports supply chain compromises up nearly 4x since 2020, with AI-powered coding tools introducing unvetted code into pipelines. Palo Alto Unit 42 (December 2025) disclosed RCE vulnerabilities in three widely-used ML frameworks — NVIDIA NeMo ([CVE-2025-23304](https://nvd.nist.gov/vuln/detail/CVE-2025-23304)), Salesforce Uni2TS ([CVE-2026-22584](https://nvd.nist.gov/vuln/detail/CVE-2026-22584)), and Apple FlexTok — all stemming from unsafe use of Hydra's `instantiate()` function in model config files, allowing arbitrary code execution when loading models from untrusted sources like HuggingFace. CSA research (September 2025) highlights cross-tool permission mismatches as a key vector — a user with limited access in one MLOps platform but elevated permissions in another can indirectly manipulate models. The OpenClaw crisis (February–March 2026) saw 138 CVEs disclosed in 63 days across an AI agent platform with 135,000+ exposed instances, with 12% of its skill marketplace found to contain malicious packages — underscoring the cascading risks when development tooling lacks integrity controls. MITRE ATLAS v5.1.0 (November 2025) now tracks 84 techniques including agent-specific attack patterns like context poisoning and memory manipulation. Microsoft's updated SDL for AI (February 2026) now explicitly treats model weights and training data as equivalent to source code for security purposes, requiring environment separation, agent identity controls, and formal validation gates before model publishing. NIST SP 800-218A (July 2024, final) extends the Secure Software Development Framework (SSDF v1.1) with AI-specific practices covering training data security, model artifact integrity, and development environment controls. On the artifact integrity front, the Protect AI + Hugging Face partnership has now scanned 4.47 million model versions and flagged 352,000 unsafe issues across 51,700 models — with pickle deserialization remaining the dominant attack vector (44.9% of popular models still use pickle format, accounting for 400M+ monthly downloads). The PickleBall framework (ACM CCS 2025) offers a practical mitigation, statically analyzing ML library source code to compute safe deserialization policies with only ~2.62% runtime overhead. These controls establish the baseline security posture for the environments and processes where models are built.

## Requirements

| # | Requirement | Level | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|-----------------|----------------------|--------------|
| **3.4.1** | **Verify that** model development, testing, and production environments are logically or physically separated with distinct access controls. | 1 | Lateral movement from dev to prod (MITRE ATLAS [AML.T0010](https://atlas.mitre.org/techniques/AML.T0010)); accidental deployment of experimental models; data leakage between environments; compromised dev environment providing prod access. NIST's Cyber AI Profile (December 2025 draft) maps environment separation to the Protect function of the Cybersecurity Framework, reinforcing this as a foundational AI security control. | Review network architecture diagrams and IAM policies for environment separation. Verify no shared service accounts, databases, or compute resources across dev/test/prod. Attempt cross-environment access and confirm it is denied. Check that MCP servers and tool endpoints are environment-specific. On cloud platforms, verify separate accounts/projects per environment: AWS Organizations with SCPs, GCP Organization Policy constraints, Azure Management Groups. Use infrastructure-as-code scanning (Terraform plan review, Checkov, tfsec) to detect cross-environment resource references. | Standard practice in software engineering but often neglected in ML workflows where data scientists may have broad access. Cloud providers offer strong primitives — separate AWS accounts, GCP projects, Azure subscriptions per environment with IAM boundaries. The NIST Cyber AI Profile (NIST IR 8596, December 2025 draft) explicitly addresses environment separation under its Protect function, with the COSAiS project developing SP 800-53 control overlays for AI use cases. The Unit 42 RCE findings (December 2025) reinforce this: Hydra-based config deserialization attacks in NeMo, Uni2TS, and FlexTok mean a compromised model artifact in dev could execute arbitrary code if loaded in a shared prod environment — environment separation limits the blast radius of such attacks. MCP server isolation is a newer concern — most MCP deployments lack environment-aware routing, and organizations should maintain separate MCP manifests per environment. NIST's AI Agent Standards Initiative (February 2026) is developing additional guidance for agent infrastructure separation. Microsoft's [SDL for AI](https://www.microsoft.com/en-us/security/blog/2026/02/03/microsoft-sdl-evolving-security-practices-for-an-ai-powered-world/) (February 2026) frames environment separation as foundational, noting that AI systems "collapse traditional trust boundaries" by blending structured data, tools, APIs, and agents into single platforms — making strict environment isolation even more critical than in traditional software. The Cycode State of Product Security 2026 report found 81% of organizations have no visibility into how AI is used across their development lifecycle, suggesting many lack the basic inventory needed to enforce environment boundaries. *(Updated 2026-04)* |
| **3.4.2** | **Verify that** separated environments do not share infrastructure, data stores, agent orchestration runtimes, or tool/MCP servers across environment boundaries. | 1 | Shared infrastructure enabling lateral movement between dev and prod (MITRE ATLAS [AML.T0010](https://atlas.mitre.org/techniques/AML.T0010)); cross-environment data leakage through shared data stores; compromised dev MCP server pivoting to prod resources; agent orchestration runtimes in dev being exploited to reach production tool endpoints. CSA (September 2025) identifies cross-tool permission mismatches as a key vector for indirect model manipulation. | Enumerate all infrastructure components (compute, storage, databases, message queues, MCP servers, agent runtimes) across dev/test/prod. Verify no shared resources exist using cloud provider tools: AWS Resource Access Manager reports, GCP Organization Policy constraints, Azure Resource Graph queries. Test cross-environment network paths — attempt connections from dev compute to prod data stores and MCP endpoints. Verify separate service accounts, IAM roles, and secrets per environment. Use tools like `nmap` or cloud-native network analyzers to confirm isolation. | Cloud providers offer strong primitives — separate AWS accounts/GCP projects per environment, VPC Service Controls perimeters (Google Cloud), Azure Private Link boundaries. The practical challenge is maintaining strict separation as ML teams scale: shared model registries, centralized experiment tracking (MLflow), and unified feature stores often bridge environment boundaries by design. The OpenClaw crisis (February–March 2026) illustrated how shared agent infrastructure amplifies risk: 12% of the ClawHub marketplace was compromised with malicious skills distributing keyloggers and Atomic Stealer malware, and 135,000+ publicly exposed instances lacked basic network segmentation. The MCPTox benchmark (2025) tested attacks across 45 real MCP servers and achieved 72% success rates on some agents, demonstrating that shared MCP infrastructure is an active attack surface. Organizations should audit tool registrations per environment and ensure MCP manifests are environment-specific. The DevSecMLOps reference architecture (2026) recommends using Open Policy Agent (OPA) to enforce environment-specific deployment rules — policy gates can block artifact promotion across environment boundaries when signatures, SBOMs, or security evaluations are missing. As of October 2025, Hugging Face scans all 2.2M+ public model and dataset repositories continuously with [VirusTotal](https://huggingface.co/blog/virustotal), but organizations hosting internal model registries need equivalent scanning per environment. *(Updated 2026-04)* |
| **3.4.3** | **Verify that** model development artifacts (such as hyperparameters, training scripts, configuration files, prompt templates, agent policies/routing graphs, tool or MCP contracts/schemas, and action catalogs or capability allow-lists) are stored in version control and require peer review approval before use in training. | 1 | Unauthorized changes to training configuration introducing backdoors (MITRE ATLAS [AML.T0018](https://atlas.mitre.org/techniques/AML.T0018)) or degrading safety alignment; unreviewed prompt template changes enabling jailbreaks; modified agent routing graphs expanding tool access without approval; inability to reproduce training runs due to untracked artifact drift. | Inspect version control repository for all listed artifact types — code, configs, hyperparameters, prompt templates, agent policies, MCP schemas, and capability allow-lists. Verify branch protection rules require peer review (GitHub CODEOWNERS, required reviewers, GitLab merge request approvals). Check that training pipelines pull from version-controlled sources and reject ad-hoc local files. Audit commit history for artifacts committed without review approval. Tools: DVC for large data/model versioning alongside Git; MLflow 3.0 Model Registry for linking models to exact code versions and prompt configurations; Git-LFS for binary artifacts. | Well-supported by Git-based workflows. DVC and Git-LFS handle large artifacts effectively. MLflow 3.0 (2025) extended its model registry to track generative AI applications including prompt configs and evaluation runs. The Sigstore [Model Transparency](https://github.com/sigstore/model-transparency) project provides cryptographic signing and verification for ML model artifacts using Sigstore's infrastructure — signing events are recorded to an append-only transparency log (Rekor), enabling anyone to verify a model's provenance and detect tampering. The Unit 42 Hydra `instantiate()` RCE findings (December 2025) highlight why peer review of model configs is critical: malicious `_target_` values in YAML configs can embed arbitrary code execution, making config files as dangerous as executable code. Organizations should require at least one peer reviewer for all artifact changes, with two reviewers for safety-critical components like capability allow-lists and agent policies. Google Cloud Artifact Registry provides secure storage with IAM-based access control and VPC Service Controls integration for ML artifacts. NIST [SP 800-218A](https://csrc.nist.gov/pubs/sp/800/218/a/final) (July 2024, final) extends the Secure Software Development Framework (SSDF v1.1) with AI-specific practices for model artifact management, training data security, and development lifecycle controls — organizations subject to federal requirements or using the SSDF should map these practices to their artifact review workflows. The Protect AI + Hugging Face partnership has now scanned [4.47 million model versions](https://huggingface.co/blog/pai-6-month) and identified 352,000 unsafe issues across 51,700 models, with detection modules covering archive slip vulnerabilities (PAIT-ARV-100), Joblib code execution (PAIT-JOBLIB-101), TensorFlow SavedModel backdoors (PAIT-TF-200), and Llamafile inference-time execution (PAIT-LMAFL-300). The [PickleBall](https://dl.acm.org/doi/10.1145/3719027.3765037) framework (ACM CCS 2025) provides a practical safe-loading solution: it statically analyzes ML library source code to compute custom deserialization policies, loads 79.8% of benign models while blocking all malicious payloads, with only ~2.62% runtime overhead — a strong complement to peer review for catching malicious artifacts that pass visual inspection. *(Updated 2026-04)* |
| **3.4.4** | **Verify that** model training and fine-tuning occur in isolated environments with controlled network access using egress allow-lists and no access to production tools or MCP resources. | 2 | Training pipeline exfiltrating data or model weights; compromised training job pivoting to production systems (MITRE ATLAS [AML.T0010](https://atlas.mitre.org/techniques/AML.T0010)); supply chain attacks through uncontrolled package downloads during training. The TeamPCP/LiteLLM attack (March 2026) demonstrated cascading supply chain compromise: malicious PyPI packages installed in training/serving environments exfiltrated API keys, SSH keys, cloud configs, and Kubernetes tokens to attacker-controlled C2 servers. | Review network policies for training environments. Verify egress rules are allow-list based (not deny-list). Confirm no network paths exist from training compute to production endpoints, MCP servers, or tool registries. Test by attempting outbound connections to unauthorized destinations from training compute. Verify package installation sources are pinned to trusted mirrors with integrity verification. Tools: `nmap` for network path testing, cloud-native network analyzers, Kubernetes NetworkPolicy audit. On GCP, verify VPC Service Controls perimeters block default internet access from Vertex AI training jobs — explicit secure egress paths are required. | Cloud-native network policies support this well: VPC Service Controls (GCP), Azure Private Link, AWS PrivateLink. GCP's Vertex AI training environments block default internet access when enclosed in a VPC SC perimeter, requiring explicit egress configuration. The difficulty is maintaining egress allow-lists as training dependencies change — package mirrors and approved registries help. The TeamPCP campaign (March 2026) showed that even security tooling (Trivy) can become an attack vector, underscoring that egress controls must apply to all software in the training environment, not just ML-specific packages. The Unit 42 RCE disclosures (December 2025) revealed that NeMo's vulnerability existed since at least 2020 — NVIDIA's fix uses allowlist-based validation that recursively checks all `_target_` values against permitted package prefixes, a pattern that should be applied to any Hydra-based ML pipeline. Kubeflow Pipelines provides namespace isolation and RBAC for multi-user training environments on Kubernetes, with KServe abstracting model serving with environment-specific routing. IBM X-Force 2026 reports supply chain compromises up nearly 4x since 2020. Private Service Connect interfaces on GCP provide granular control over which Google Cloud services training jobs can reach. The "nullifAI" technique (February 2025) demonstrated that malicious ML models on Hugging Face can evade PickleScan detection by using 7z compression instead of ZIP format — three zero-day [PickleScan vulnerabilities](https://jfrog.com/blog/unveiling-3-zero-day-vulnerabilities-in-picklescan/) were disclosed by JFrog (June 2025) and fixed in PickleScan v0.0.31 (September 2025), but the incident underscores that training environments pulling models from external registries must enforce format restrictions and scanning at ingress. DevSecMLOps pipelines should enforce policy-as-code gates (e.g., OPA + Cosign) that block unsigned or unscanned model artifacts from entering training environments, treating model files with the same rigor as executable code. *(Updated 2026-04)* |
| **3.4.5** | **Verify that** training data sources are validated through integrity checks and authenticated via trusted sources with documented chain of custody before use in model development, including RAG indexes, tool logs, and agent-generated data used for fine-tuning. | 2 | Training data poisoning (MITRE ATLAS [AML.T0020](https://atlas.mitre.org/techniques/AML.T0020)); fine-tuning on manipulated agent logs or RAG content leading to backdoored models; using unvetted third-party datasets containing adversarial examples; supply chain injection of poisoned data via compromised data pipelines. Check Point's 2026 Tech Tsunami report calls data poisoning the "new zero-day" threat. Distributed attack models coordinate poisoning across multiple stages of the ML lifecycle for amplified impact. | Inspect data ingestion pipelines for integrity checks: verify SHA-256 checksums on all dataset files, validate schemas before ingestion, confirm data source authentication (signed datasets, authenticated API sources). Review chain of custody documentation for training datasets. Check that agent-generated data and RAG content used for fine-tuning pass quality and integrity gates before inclusion. Verify dataset provenance records are maintained and auditable. Tools: Great Expectations (GX) for data validation rules, Soda Core + SodaGPT for automated data quality checks, OpenSSF Model Signing (`pip install model-signing`) for cryptographic verification of datasets alongside model artifacts, DVC for dataset version tracking with Git integration. For EU AI Act compliance, verify the GPAI training data summary template is maintained per the July 2025 Commission template. | Data provenance tooling is maturing but still behind code provenance. The OpenSSF Model Signing (OMS) specification (June 2025) supports signing datasets alongside model weights in a single verifiable bundle using Sigstore, though the dataset-specific signing roadmap is still evolving — the Coalition for Secure AI is working on embedding training data source metadata into OMS signatures. The Sigstore [Model Transparency](https://github.com/sigstore/model-transparency) project records all signing events to Rekor (an append-only transparency log), giving verifiers a public record to audit model and dataset provenance. NVIDIA has signed all NGC Catalog models with OMS since March 2025, setting an industry precedent. PoisonBench (ICML 2025) is the first benchmark for evaluating data poisoning in LLMs during preference learning — it found that larger models do not automatically resist poisoning better and that poisoned behaviors can generalize to triggers never seen in training, necessitating adaptive detection beyond pattern matching. MCPTox (2025) systematically tested poisoning attacks across 45 real MCP servers with over 1,300 malicious cases and achieved 72% success rates on some agents, highlighting how agent-generated data used for fine-tuning can carry poisoned instructions. The EU AI Act mandates training data provenance documentation for GPAI models, with the Commission's training data summary template effective August 2025 for new models (transitional deadline August 2027 for existing models). Documentation must be retained for 10 years. The NSA/CISA AI Data Security advisory (May 2025) mandates provenance checks, digital signatures, and source authentication for all third-party datasets. Chain of custody for dynamically generated data (agent logs, RAG content, synthetic data) remains particularly challenging — Lakera Guard provides runtime detection of poisoned patterns and output drift monitoring, but continuous provenance tracking of streaming data used in fine-tuning remains an open problem. OWASP ML Top 10 identifies data poisoning as a top-tier threat. [CVE-2025-1550](https://nvd.nist.gov/vuln/detail/CVE-2025-1550) demonstrated that Keras custom layers can execute arbitrary code during model loading — the Protect AI Guardian scanner detected this in production before public disclosure, but organizations loading third-party models for fine-tuning transfer learning must treat model files as untrusted input. AI security incidents increased 56.4% from 2023 to 2024, and the average cost of a data breach involving AI now stands at $4.88 million (IBM Cost of a Data Breach 2025). Microsoft's SDL for AI (February 2026) now explicitly requires that training data receive "protection equivalent to source code," establishing a clear industry benchmark for data integrity controls. *(As of April 2026)* |
