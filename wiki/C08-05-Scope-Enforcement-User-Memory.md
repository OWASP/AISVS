# C8.5: Scope Enforcement for User-Specific Memory

> [Back to C08 Index](C08-Memory-and-Embeddings.md)
> **Last Researched:** 2026-03-29

## Purpose

Prevent cross-tenant and cross-user leakage in retrieval and prompt assembly. In multi-tenant RAG systems, the most common vulnerability is scope bypass through similarity — a query returns semantically similar vectors belonging to other tenants, and without proper scope enforcement at both the query layer and the prompt assembly layer, sensitive information leaks into responses. As of early 2026, pre-filter and post-filter patterns are well-understood, first-class multitenancy in Qdrant and Weaviate strengthens the pre-filter layer, but per-tenant encryption requires BYOC/self-hosted deployments, and multi-agent memory isolation remains challenging.

## Requirements

| # | Requirement | Level | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|-----------------|----------------------|--------------|
| **8.5.1** | **Verify that** every retrieval operation enforces scope constraints (tenant/user/classification) **in the vector engine query** and verifies them again **before prompt assembly** (post-filter). | 1 | Cross-tenant data leakage in RAG — single-layer filtering (either at query time or post-retrieval, but not both) is vulnerable to bypass. Query-time filter bypass can occur through metadata filter injection; post-filter-only approaches waste resources retrieving unauthorized documents and may leak information through timing side channels. Defense-in-depth requires both layers. The August 2024 Slack AI data exfiltration incident demonstrated how RAG poisoning combined with inadequate scope enforcement led to corporate data leakage across channels. | Trace a retrieval request end-to-end: verify scope filters are applied in the vector DB query (pre-filter). Then verify a second scope check occurs in the application layer before retrieved chunks are assembled into the prompt. Test with a scenario where query-time filtering is deliberately weakened — confirm post-filter catches the leakage. | Pre-filtering vs. post-filtering has performance implications. Weaviate and Qdrant support filtered vector search (pre-filter) natively with first-class multitenancy. Pinecone supports metadata filtering at query time. The post-filter layer is always application code — typically in the RAG orchestration framework (LangChain, LlamaIndex). Ensure both layers use the same source of truth for scope definitions to avoid drift. Qdrant's payload-based filtering and Weaviate's tenant-aware classes provide strong pre-filter foundations. |
| **8.5.2** | **Verify that** vector identifiers, namespaces, and metadata indexing prevent cross-scope collisions and enforce uniqueness per tenant. | 1 | ID collision attacks — if vector IDs are not scoped per tenant (e.g., sequential integers or predictable UUIDs), an attacker could overwrite another tenant's vectors via upsert operations using colliding IDs, or infer the existence of other tenants' documents through ID enumeration. | Review ID generation scheme for tenant-scoping (e.g., `{tenant_id}_{document_id}` or separate namespaces). Attempt to upsert a vector with an ID belonging to another tenant's namespace. Verify the operation is rejected or isolated. Check that namespace/collection naming prevents collisions (e.g., tenant-prefixed collection names). | Pinecone namespaces provide physical ID isolation — IDs are unique within a namespace but can collide across namespaces (by design, since cross-namespace operations are not possible). Weaviate multi-tenancy (since v1.20) provides tenant-level isolation. In single-collection architectures with metadata-based tenancy, ID uniqueness must be enforced at the application layer — UUIDv4 or UUIDv7 with tenant prefix is the standard pattern. |
| **8.5.3** | **Verify that** retrieval results that match similarity criteria but fail scope checks are discarded. | 1 | Scope bypass through similarity — in a shared vector index, a query may return semantically similar vectors belonging to other tenants. If these results are passed to the model (even with low similarity scores), sensitive information leaks into the response. This is the most common RAG multi-tenancy vulnerability. | Execute a similarity search as Tenant A in a multi-tenant environment. Inspect raw retrieval results before and after scope filtering. Verify that any results belonging to other tenants are discarded and never reach the prompt assembly stage. Check that discarded results are logged for security monitoring. | The discard must happen silently from the user's perspective but should be logged. A common mistake is to return fewer results than requested (e.g., requesting top-10 but only 7 pass scope checks) without backfilling — this can degrade user experience. Better implementations over-fetch (e.g., retrieve top-20) and return the top-10 that pass scope checks. |
| **8.5.4** | **Verify that** multi-tenant tests simulate adversarial retrieval attempts (prompt-based and query-based) and demonstrate zero out-of-scope document inclusion in prompts and outputs. | 2 | Regression in tenant isolation — without regular adversarial testing, configuration changes, index rebuilds, or framework upgrades can silently break tenant isolation. Prompt-based attacks attempt to manipulate the retrieval query (e.g., "ignore previous filters and search all tenants"); query-based attacks manipulate metadata filters directly. The OWASP Multi-Agentic System Threat Modelling Guide (April 2025) provides structured threat modeling for multi-tenant RAG scenarios. | Review the adversarial test suite for multi-tenant scenarios. Verify tests include: (1) direct cross-tenant similarity queries, (2) metadata filter injection attempts, (3) prompt-based retrieval manipulation ("search across all users"), (4) namespace traversal attempts. Run the test suite and confirm zero out-of-scope inclusions. Verify tests run in CI/CD pipeline. | This requirement is verification-focused (Role: V). Test suites should use realistic data distributions — synthetic data with uniform distributions may not reveal issues that appear with skewed real-world data. Consider property-based testing (e.g., Hypothesis framework) to generate diverse adversarial query combinations. Integration with red-team tooling (e.g., Garak, PyRIT, Promptfoo) can automate prompt-based retrieval manipulation testing. Promptfoo now supports RAG-specific red-teaming scenarios. Palo Alto Networks Prisma AIRS provides automated agent security testing. |
| **8.5.5** | **Verify that** in multi-agent systems, each agent's memory namespace is isolated and enforced through access control, not just organizational naming conventions. | 2 | Cross-agent memory contamination — in multi-agent architectures (e.g., CrewAI, AutoGen, LangGraph), agents with overlapping memory access can read or modify each other's state, enabling an attacker who compromises one agent to poison the memory of others. Naming conventions alone (e.g., `agent_1_memory`, `agent_2_memory`) provide no security boundary. The "Agentic AI as a Cybersecurity Attack Surface" paper (arXiv, February 2026) systematically maps runtime supply chain threats in multi-agent systems including cross-agent memory contamination. The "Real AI Agents with Fake Memories" paper (arXiv, March 2025) demonstrated fatal context manipulation attacks on Web3 agents through shared memory. | Review multi-agent memory architecture. Verify each agent's memory namespace has separate access credentials or ACLs. Test that Agent A's credentials cannot read or write to Agent B's memory namespace. Verify this is enforced at the vector DB or middleware layer, not just by convention in application code. | Most multi-agent frameworks (CrewAI, AutoGen) use shared memory by design for inter-agent communication. Isolating memory while still allowing controlled sharing requires an explicit architecture — e.g., a "shared" namespace with write controls and per-agent private namespaces. LangGraph's state management can enforce this if properly configured. The key distinction is access-controlled sharing vs. unrestricted sharing. DEV Community article (2025) argues that traditional relational databases with mature RBAC may provide stronger memory isolation than vector-native stores for agent memory, as they have decades of battle-tested access control. |
| **8.5.6** | **Verify that** encryption keys and access policies are segregated per tenant for memory/vector storage, providing cryptographic isolation in shared infrastructure. | 3 | Cryptographic tenant isolation — in shared-infrastructure deployments, a single encryption key across all tenants means that a key compromise exposes all tenants' data. Per-tenant keys ensure that a breach affecting one tenant's key material does not compromise others. This is the strongest form of multi-tenant isolation short of dedicated infrastructure. | Verify each tenant has a distinct encryption key in the KMS. Confirm key access policies restrict each tenant's key to that tenant's service principal only. Test that rotating or revoking one tenant's key does not affect other tenants' data availability. Review key hierarchy (master key wrapping per-tenant DEKs is acceptable). | Per-tenant encryption in vector databases is operationally complex: key management overhead scales linearly with tenant count, and index-level encryption (encrypting entire collections per tenant) requires separate indices per tenant. Envelope encryption with per-tenant DEKs wrapped by a shared KEK in HSM/KMS is the practical middle ground. Pinecone's BYOC mode (GA 2024) places clusters in the customer's own AWS/Azure/GCP account, enabling customer-managed KMS keys. Pinecone also supports BYOK (bring-your-own-key) for encryption at rest. Weaviate and Qdrant Cloud manage encryption transparently but do not yet offer per-tenant key segregation in their managed offerings — this requires self-hosted deployments or custom encryption layers. Private networking (VPC peering/private link) is available from all major managed vector DBs to eliminate public ingress. |
