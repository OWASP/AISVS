# C5.5: Multi-Tenant Isolation

> [Back to C05 Index](C05-Access-Control.md)
> **Last Researched:** 2026-03-29

## Purpose

Ensure logical and cryptographic isolation between tenants in shared AI infrastructure. Multi-tenant AI deployments share GPU clusters, model-serving endpoints, vector databases, and inference caches across tenants. Without rigorous isolation, cross-tenant data leakage can occur through shared KV-caches, embedding stores, network segments, and even GPU memory residuals. Recent research has demonstrated that timing side channels in shared inference infrastructure can reconstruct other tenants' prompts with 95-99% accuracy, making KV-cache isolation a critical and rapidly evolving concern.

> **Scope note:** Controls in this section govern infrastructure-level tenant isolation -- network policies, namespace segregation, per-tenant encryption, and KV-cache partitioning. Query-time tenant filtering is in C5.3; authorization policy is in C5.2.

## Requirements

| # | Requirement | Level | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|-----------------|----------------------|--------------|
| 5.5.1 | **Verify that** network policies implement default-deny rules for cross-tenant communication. | 2 | Lateral movement between tenant environments; cross-tenant network-level attacks exploiting shared infrastructure; data exfiltration through unrestricted network paths. | Review Kubernetes NetworkPolicies or cloud security group rules; attempt cross-tenant network connections and confirm they are blocked; verify default-deny is the baseline with explicit allow rules for required communication. | Standard multi-tenant security practice. In AI contexts, shared GPU clusters and model-serving infrastructure often share network segments. Kubernetes NetworkPolicies, Calico, or Cilium can enforce tenant isolation at the network layer. |
| 5.5.2 | **Verify that** every API request includes an authenticated tenant identifier that is cryptographically validated against session context and user entitlements. | 1 | Tenant ID spoofing allowing cross-tenant data access; confused deputy attacks where a shared service accesses the wrong tenant's data; IDOR vulnerabilities in multi-tenant AI APIs. | Attempt API calls with mismatched tenant IDs and session tokens; confirm rejection; test tenant ID tampering in request headers; verify tenant ID is derived from authenticated session rather than client-supplied parameters. | Tenant ID must be extracted from the authenticated token (JWT claim, session attribute) rather than accepted as a client-supplied parameter. This prevents trivial tenant spoofing. Shared inference endpoints are particularly vulnerable if tenant context is passed as a query parameter. **[2026-03]** The 2025 Salesloft-Drift OAuth token breach demonstrates the cascading risk: compromised third-party app tokens granted attackers access to hundreds of downstream tenant environments. When a single API key is leaked in multi-tenant AI systems, all tenants relying on that key are affected and every permission attached becomes accessible. |
| 5.5.3 | **Verify that** memory spaces, embedding stores, cache entries (e.g., result caches, embedding caches), and temporary files are namespace-segregated per tenant with secure purging on tenant deletion or session termination. | 2 | Cross-tenant data leakage through shared caches, GPU memory, or temporary files; residual tenant data persisting after tenant offboarding; embedding store contamination between tenants. | Verify namespace segregation in embedding stores, caches, and temp directories; trigger tenant deletion and confirm all associated data is purged; check for residual data in GPU memory, swap files, and temporary storage. | AI-specific concern: GPU memory is shared across inference requests and may retain activations from previous tenants. KV-cache in transformer inference, embedding store collections, and prompt caches all need tenant-scoped namespacing. Secure purging of GPU memory is not natively supported by most frameworks. **[2026-03]** Pinecone namespaces physically partition records so queries are limited to one namespace. Weaviate collections ensure embeddings are never co-mingled across tenants. The "Curator" research (arxiv 2401.07119) addresses efficient indexing for multi-tenant vector databases with tenant-aware partitioning strategies. |
| 5.5.4 | **Verify that** encryption keys are unique per tenant with customer-managed key (CMK) support and cryptographic isolation between tenant data stores. | 3 | Compromise of a single encryption key exposing all tenants' data; inability to perform cryptographic tenant offboarding (key destruction); regulatory requirements for customer-controlled encryption. | Verify unique key ARNs/IDs per tenant; confirm CMK integration (AWS KMS, Azure Key Vault, GCP Cloud KMS); test that tenant A's key cannot decrypt tenant B's data; verify key destruction on tenant offboarding removes access to all associated data. | Per-tenant keys add key management complexity but enable cryptographic shredding on tenant offboarding. For AI workloads, this extends to encrypted model artifacts, training data snapshots, and embedding stores. HSM-backed key management is recommended for high-sensitivity deployments. |
| 5.5.5 | **Verify that** inference-time KV-cache entries are partitioned by authenticated session or tenant identity and that automatic prefix caching does not share cached prefixes across distinct security principals, to prevent timing-based prompt reconstruction attacks. | 2 | Timing-based prompt reconstruction via KV-cache sharing side channels; the PROMPTPEEK attack (NDSS 2025) achieves 95-99% accuracy reconstructing other tenants' prompts by probing cache hit/miss latency differences in vLLM and SGLang. Direct KV-cache content extraction enables collision and injection attacks that reverse-engineer user inputs from plaintext cache entries (Shadow in the Cache, arxiv 2508.09442). Cross-tenant prefix reuse leaks query patterns even when outputs are properly isolated. | Confirm inference framework uses per-tenant or per-session cache salting (e.g., vLLM `cache_salt` parameter, merged May 2025); measure response latency variance across probe sequences to detect cache-hit timing leakage; verify automatic prefix caching does not share prefixes across tenant boundaries; inspect KV-cache storage to confirm entries are not persisted in plaintext across sessions; test that expired or terminated sessions have their cache entries evicted. | **[2026-03]** Active research area with rapidly maturing defenses. vLLM merged cache salting (RFC #16016, May 2025) for single-barrier tenant isolation; hierarchical multi-barrier design (per-message salts) is in active development as of Mar 2026. SafeKV (arxiv 2508.08438, Feb 2026) integrates ChunkGuard for real-time sensitivity classification with selective cache sharing -- mitigates 94-97% of timing side-channels while reducing TTFT overhead by up to 40% vs. full isolation and raising throughput up to 2.66x. KV-Cloak (arxiv 2508.09442) provides reversible matrix obfuscation of KV-cache contents with ~5% latency overhead and negligible accuracy impact. No production framework yet combines all three approaches (salting + selective sharing + content obfuscation); organizations should evaluate their threat model to choose the right tradeoff between isolation strength and performance. |
