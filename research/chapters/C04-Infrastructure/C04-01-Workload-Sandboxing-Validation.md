# C4.1 AI Workload Sandboxing & Validation

[Back to C04 Index](C04-Infrastructure.md)

## Purpose

Isolate untrusted AI models in secure sandboxes and protect sensitive AI workloads using trusted execution environments (TEEs) and confidential computing technologies. AI systems increasingly execute models from external sources (open-source model hubs, third-party vendors, user-uploaded models). These models can contain arbitrary code (via pickle deserialization, custom operators, or embedded scripts) and must be treated as untrusted. This section addresses both defensive sandboxing of untrusted models and proactive protection of high-value models and data using confidential computing.

## Implementation Guidance

### Sandbox Technology Selection (2025-2026 State of Practice)

Choosing the right sandboxing technology depends on workload sensitivity, performance requirements, and operational maturity. The three dominant approaches have distinct security/performance tradeoffs:

**Firecracker MicroVMs** provide hardware-level isolation via KVM, with each workload running a dedicated Linux kernel completely separated from the host. Firecracker boots in approximately 125ms with less than 5 MiB of memory overhead per VM, supporting up to 150 VMs per second per host. This is widely regarded as the gold standard for running untrusted AI-generated code in production. AWS Lambda and Fly.io use Firecracker in production at massive scale. Even Firecracker is not immune to vulnerabilities: CVE-2026-5747 (CVSS 8.7) exposed an OOB write in the opt-in virtio-pci transport allowing guest-to-host escalation (fixed in 1.14.4+ / 1.15.1+), and CVE-2026-1386 revealed a symlink race in the jailer initialization that allowed arbitrary host file overwrites (fixed in 1.13.2+ / 1.14.1+). The default MMIO transport is unaffected by the PCI vulnerability, but organizations using PCI for I/O performance should prioritize the upgrade.

**gVisor** intercepts system calls in user space via its Sentry process (implemented in memory-safe Go) before they reach the host kernel, reimplementing ~200 Linux syscalls while exposing only ~70 syscalls to the host kernel -- down from the ~340 that standard containers expose. This dramatically reduces the kernel attack surface and has historically protected GKE workloads from kernel CVEs that affected unprotected containers (e.g., CVE-2020-14386). gVisor imposes 10-30% overhead on I/O-heavy workloads but has fast startup. It is best suited for compute-heavy AI workloads (such as inference) where full VM isolation is not required. Google uses gVisor extensively across GKE and Cloud Run. Important GPU caveat: gVisor's `nvproxy` GPU support only allows CUDA compute and utility capabilities by default, and does not introduce hardware-level GPU isolation beyond what the NVIDIA kernel-mode driver configures -- it intercepts `ioctl(2)` calls against an allowlist but cannot protect against all NVIDIA driver vulnerabilities. For full GPU passthrough isolation, Kata Containers (one GPU per pod) or Firecracker with VFIO are required.

**Kata Containers** orchestrate multiple VMMs (Firecracker, Cloud Hypervisor, QEMU) through standard container APIs with native Kubernetes integration. Boot time is approximately 200ms with minimal memory overhead. Kata Containers are ideal for regulated industries requiring zero-trust environments with Kubernetes-native orchestration. Treat the runtime/VMM combination as patch-managed infrastructure, not a static security boundary: CVE-2026-24054 in versions before 3.26.0 can let malformed no-layer images hotplug a host block device into the guest and force host filesystem failure; CVE-2026-24834 in versions before 3.27.0 affects Kata with Cloud Hypervisor and can let a container user modify the guest microVM filesystem and gain root inside that VM. Verification should include the Kata release, selected VMM, snapshotter/rootfs driver, runtime class, and whether privileged containers, host devices, or `CAP_MKNOD` are permitted.

**Google Agent Sandbox**, launched at KubeCon NA 2025 as a CNCF project under Kubernetes SIG Apps, provides a declarative API for managing isolated, stateful sandbox pods on Kubernetes clusters specifically for AI agent workloads.

**Docker Sandboxes**, launched in March 2026 as an experimental feature, runs AI coding agents inside dedicated microVMs with their own Linux kernel, Docker daemon, filesystem, and network stack. It supports Claude Code, Gemini CLI, Codex, Copilot, and other AI agent tools out of the box. Each sandbox provides hardware-level isolation -- a kernel exploit inside the sandbox compromises only that sandbox's kernel, not the host. The network model is especially relevant to requirement 4.1.2: raw TCP, UDP, ICMP, private IP ranges, loopback, and link-local access are blocked; DNS and HTTP(S) egress are mediated by policy-aware host proxies; and model-provider credentials are injected by the host proxy rather than stored inside the VM. Because Docker still labels the feature Early Access / Experimental, production use should require independent compensating controls and audit evidence rather than relying on the product label alone.

**NVIDIA OpenShell** (March 2026, open source) is a security runtime purpose-built for autonomous AI agents. Rather than replacing a VM or container boundary, OpenShell wraps agents with out-of-process policy enforcement across filesystem, network, and process actions. Every action can be evaluated at the binary, destination, method, and path level, with default-deny semantics -- anything not explicitly allowed is blocked. Popular coding agents can run unmodified inside the runtime, but auditors should review the actual policy loaded for the agent: NVIDIA's default policy reference currently lists full out-of-the-box coverage for Claude Code, partial coverage for OpenCode, and no default coverage for Codex without a custom policy. OpenShell is complementary to microVM isolation: it provides fine-grained runtime policy enforcement within an already-isolated environment, addressing the gap between coarse hardware boundaries and application-level permission scoping.

**Microsoft LiteBox** (February 2026, open source) is an experimental Rust-based library OS that rethinks sandboxing for modern workloads. Instead of intercepting syscalls (gVisor) or running full VMs (Firecracker), LiteBox bundles minimal OS functionality as libraries alongside each application, providing a narrowly scoped kernel interface tailored to the workload. LiteBox supports AMD SEV-SNP confidential computing, enabling workloads to execute inside encrypted memory regions. While still experimental with no stable releases, the library OS approach represents an emerging fourth isolation paradigm alongside containers, microVMs, and user-space kernels. Microsoft has not confirmed whether LiteBox will integrate into Azure or WSL.

As of April 2026, at least twelve major platforms have converged on microVM isolation for AI agent workloads (E2B, Fly.io Sprites, Docker Sandboxes, Vercel Sandbox, AWS Bedrock AgentCore, Northflank, and others), establishing microVMs as the de facto standard for untrusted AI code execution. Firecracker's codebase is approximately 83,000 lines of Rust versus QEMU's 1.7 million lines of C, representing a dramatically reduced attack surface. Snapshot-restore enables 28ms boot times for pre-warmed environments.

For production deployments executing untrusted models or AI-generated code, Firecracker microVMs or Kata Containers are recommended. The hardware boundary they provide prevents entire classes of kernel-based attacks that user-space isolation (containers, gVisor) cannot fully address.

### WebAssembly (Wasm) for AI Workload Sandboxing

WebAssembly has emerged as a lightweight sandboxing option for AI workloads, particularly at the edge and for agentic AI workflows. Key developments in 2025-2026:

- **NVIDIA published guidance** on using Wasm (via Pyodide, a CPython port compiled to WebAssembly) to sandbox LLM-generated Python code execution in agentic workflows. The approach shifts code execution from application servers to browser sandboxes, providing two protective layers: execution failure in the restricted runtime and browser-level isolation of any successful exploit.
- **Microsoft Wassette** (August 2025) is a security-oriented runtime that executes WebAssembly Components via Model Context Protocol (MCP), allowing AI agents to autonomously fetch Wasm Components from OCI registries and execute them securely using the Wasmtime runtime.
- **WASI AI** enables Wasm modules to interact with ML models and hardware accelerators, supporting inference at the edge with minimal latency. AWS Lambda now supports Wasm functions as a first-class runtime with 10-40x cold start improvements over containers.
- **wasmCloud** supports distributed ML and AI workloads with Wasm-based isolation.

However, Wasm sandboxing is not without risk. Documented Wasm runtime CVEs include: CVE-2024-30266 (Wasmtime type confusion in externref handling causing host-side memory disclosure), CVE-2023-51661 (Wasmer WASI filesystem restriction bypass enabling unauthorized file access), and CVE-2023-2033 (V8 type-confusion exploit achieving arbitrary write primitives on the host, exploited in the wild). As of April 2026, CVE-2026-34971 (CVSS 9.0) exposed a critical sandbox escape in Wasmtime's Cranelift compiler backend on AArch64: a logic error in ISLE instruction selection rules caused bounds-check calculations and actual memory loads to diverge, giving attackers arbitrary read/write across the host process address space. The vulnerability requires 64-bit Wasm linear memory with Spectre mitigations and signals-based traps both disabled -- a configuration some edge AI deployments use for performance. Fixed in Wasmtime 36.0.7, 42.0.2, and 43.0.1. Attack vectors include linear memory overflows in modules compiled from C/C++, type confusion in indirect function tables allowing function call redirection outside the sandbox, and JIT-compiler bounds check elision creating time-of-check/time-of-use windows. Malicious AI model files could trigger specific JIT-compiler bugs during weight compilation, compromising inference servers. Wasm should be treated as a defense-in-depth layer, not the sole isolation mechanism for high-risk workloads.

### Model Format Validation and Pre-Deployment Scanning

The pickle deserialization threat remains one of the most critical attack vectors in the AI supply chain. Research from 2025-2026 underscores that model scanning tools are necessary but insufficient on their own:

**PickleScan Bypass Vulnerabilities (2025):** JFrog Security Research discovered three critical zero-day vulnerabilities (CVE-2025-10155, CVE-2025-10156, CVE-2025-10157, all CVSS 9.3) in PickleScan, the most widely deployed model scanning tool used by platforms including Hugging Face:

- **CVE-2025-10155 (File Extension Bypass):** PickleScan prioritizes file extension checks over content analysis. Renaming malicious pickle files with PyTorch extensions (.bin, .pt) causes PickleScan to attempt PyTorch-specific parsing, fail silently, and return no error -- while PyTorch identifies file types by content and loads the malicious payload successfully.
- **CVE-2025-10156 (CRC Bypass):** Attackers introduce intentional CRC errors in ZIP-archived models. PickleScan (using Python's zipfile module) raises exceptions and fails, while PyTorch's model loader bypasses CRC validation and loads the content.
- **CVE-2025-10157 (Unsafe Globals via Subclasses):** PickleScan checks exact module names against its blocklist, but attackers use subclasses of dangerous imports (e.g., internal asyncio classes). PickleScan marks these as "Suspicious" rather than "Dangerous," failing to block them.

These vulnerabilities were fixed in PickleScan 0.0.31+ (September 2025), but they expose a systemic problem: PickleScan and PyTorch handle files differently, creating exploitable semantic gaps.

**Sonatype PickleScan Vulnerabilities (2025):** Independently, Sonatype discovered four additional CVEs in PickleScan, all fixed in version 0.0.23:

- **CVE-2025-1716:** Unsafe deserialization bypasses static analysis, enabling arbitrary code execution via malicious pip package installation during model loading.
- **CVE-2025-1889:** Hidden pickle files embedded in PyTorch archives with non-standard naming conventions evade detection but are loaded by `torch.load()`.
- **CVE-2025-1944:** ZIP filename tampering causes PickleScan to crash with BadZipFile errors, while PyTorch's lenient ZIP handling loads the malicious payload.
- **CVE-2025-1945:** Modified ZIP flag bits prevent detection of malicious pickle files, though PyTorch deserializes them successfully.

Between the JFrog and Sonatype disclosures, a total of seven distinct PickleScan bypass techniques were documented in 2025 alone, underscoring that blocklist-based scanning is fundamentally fragile against determined adversaries.

**nullifAI Attack Technique (February 2025):** ReversingLabs discovered two malicious models on Hugging Face that evaded the platform's security scanning by using 7z compression instead of the ZIP format PyTorch traditionally uses. Hugging Face's PickleScan-based scanner did not process 7z-compressed archives, while PyTorch's model loader handled them without issue, creating yet another format-handling semantic gap. The malicious payloads contained reverse shells connecting to hardcoded attacker IPs. This technique is distinct from the JFrog and Sonatype bypasses and represents an eighth documented evasion vector against model scanning tools.

**PickleBall (CCS 2025):** Academic research from Brown University introduced PickleBall, a secure deserialization framework that performs fine-grained analysis of pickle bytecode rather than relying on blocklist matching. Evaluated model scanners fail to identify known malicious models, while PickleBall achieves higher detection rates with fewer false positives on benign models.

**SafePickle (February 2026):** A newer approach using robust ML-based detection of malicious pickle models, addressing the limitations of signature-based scanning.

**Hydra Configuration RCE (Unit42, January 2026):** Palo Alto Unit42 discovered a class of RCE vulnerabilities affecting AI libraries that use Hydra's `instantiate()` function for model configuration. Attackers embed arbitrary Python callables (e.g., `builtins.exec()`) as `_target_` values in model metadata files, achieving code execution when the model is loaded. Affected libraries include NVIDIA NeMo (CVE-2025-23304, fixed in v2.3.2), Salesforce Uni2TS (CVE-2026-22584, fixed July 2025), and Apple/EPFL ml-flextok (patched June 2025). Over 700 HuggingFace models use NeMo format, and approximately 100 Python libraries on HuggingFace use Hydra for configuration. This vulnerability class operates independently of pickle -- even SafeTensors-based models can be exploited if the library unsafely instantiates configuration metadata.

**Recommended defense-in-depth for model validation:**
1. **Prefer safe serialization formats.** Use SafeTensors for model weights wherever possible. SafeTensors stores only raw tensor data with JSON-based headers and eliminates the possibility of arbitrary code execution by design.
2. **Use PyTorch safe-loading controls, but do not over-trust them.** PyTorch 2.6+ defaults `torch.load()` to `weights_only=True` when no custom pickle module is passed, and `TORCH_FORCE_WEIGHTS_ONLY_LOAD=1` can force that behavior across call sites. PyTorch's own documentation is explicit that this narrows remote-code-execution risk but does not prevent denial-of-service or all memory-corruption cases, so it is a compatibility backstop rather than a primary trust boundary.
3. **Scan all model files in CI/CD.** All model binary files (.bin, .pkl, .pt, .pth, .ckpt) must be scanned using tools like modelscan, picklescan (0.0.31+), or PickleBall. If a scan returns "dangerous," the pipeline must block deployment.
4. **Fail closed.** If an automated model scan fails for any reason (timeout, parsing error, tool crash), immediately quarantine the model and prevent distribution.
5. **Sandbox model loading.** Even scanned models should be loaded inside sandboxed environments, because scanning tools and safe-loading modes have known bypass or residual-risk classes. Scanning reduces risk but does not eliminate it.
6. **Maintain an approved model registry.** Curate an internal registry of validated models with cryptographic signatures. Sandbox anything not in the approved registry.

Approximately 44.9% of model repositories on public hubs still contain pickle-format models, making this a persistent and widespread attack surface. In February 2025, researchers from Reverse Engineering Labs uncovered malicious ML models on Hugging Face that exploited "broken" pickle files to evade platform detection mechanisms and execute reverse shells.

**SafeTensors conversion pipeline as an attack surface:** Even the migration to SafeTensors introduces its own risks. HiddenLayer (February 2024) demonstrated "Silent Sabotage" -- an attack against Hugging Face's automated SafeTensors conversion service (SFConvertbot). A crafted malicious PyTorch binary triggers arbitrary code execution during `torch.load()` in the conversion pipeline, enabling theft of the SFConvertbot authentication token. With that token, an attacker could impersonate the official bot and submit poisoned pull requests to any model repository, creating a supply chain attack vector that scales to models with millions of monthly downloads. This underscores that format-level safety only applies at the loading boundary -- any process that ingests pickle-format models as input (conversion services, scanning pipelines, evaluation frameworks) must itself run in a sandboxed environment.

### Container and Sandbox Escape: Recent Incidents and Research (2025-2026)

The AI workload sandboxing threat landscape has shifted significantly, with several high-profile escapes demonstrating that isolation boundaries require constant vigilance:

**NVIDIAScape (CVE-2025-23266, CVSS 9.0, July 2025):** Wiz Research discovered that a three-line Dockerfile could exploit an OCI hook misconfiguration in the NVIDIA Container Toolkit to achieve full root access on the host. The `createContainer` hook trusted unfiltered environment variables, allowing `LD_PRELOAD` injection to load attacker-controlled libraries. Approximately 37% of cloud environments were susceptible. Fixed in NVIDIA Container Toolkit v1.17.8+ and GPU Operator v25.3.1+. This is particularly relevant because virtually all GPU-accelerated AI workloads use this toolkit.

**runC Triple Vulnerability (November 2025):** Three vulnerabilities in runC created container breakout vectors across the entire container ecosystem:
- CVE-2025-31133: Symlink replacement targeting procfs files to bypass maskedPaths protections
- CVE-2025-52565: Timing-based attacks during container initialization bypassing security restrictions
- CVE-2025-52881: Write redirection to critical system files enabling host crashes and full breakouts

These affected Docker, Kubernetes, containerd, and CRI-O. Patched in runC 1.2.8+, 1.3.3+, or 1.4.0-rc.3+.

**N8scape (CVE-2025-68668, CVSS 9.9):** A Pyodide sandbox escape in n8n's Python Code Node allowed authenticated users to execute arbitrary commands on the host. The root cause was a blocklist-based sandbox that could not enumerate every dangerous capability -- attackers used `_pyodide._base.eval_code()` to bypass restrictions. This demonstrates why blocklist-based sandboxing approaches are fundamentally weaker than capability-based or hardware-enforced isolation for AI code execution.

**Agenta-API Sandbox Escape (CVE-2026-27952, CVSS 8.8):** A Python sandbox escape in Agenta-API's custom code evaluator (versions prior to 0.48.1) allowed authenticated users to gain arbitrary code execution, further illustrating the fragility of application-level sandboxing in AI evaluation platforms.

**OpenClaw Docker Config Escape (CVE-2026-27002):** A configuration injection vulnerability in OpenClaw's Docker tool sandbox allows attackers who control sandbox configuration to specify dangerous Docker options, achieving container escape and host compromise. This highlights that sandboxing implementations must validate and restrict configuration parameters, not just runtime behavior.

**AWS Bedrock AgentCore Sandbox Bypass (Unit42, April 2026):** Palo Alto Unit42 published a two-part series ("Cracks in the Bedrock") demonstrating that AgentCore's Code Interpreter sandbox -- which claims "complete isolation" -- can be bypassed via DNS tunneling for data exfiltration and C2. The second part, "Agent God Mode," showed that the AgentCore starter toolkit generates overly permissive IAM roles granting wildcard access across the entire AWS account. A compromised agent can exfiltrate proprietary ECR images, access other agents' memories and conversation histories, invoke all code interpreters in the account, and extract sensitive configuration. AWS responded by setting MMDSv2 as the default (February 2026) and updating documentation (April 2026). The research demonstrates that cloud-managed AI sandbox services require the same scrutiny as self-hosted isolation -- "managed" does not mean "secure by default."

**MaxKB AI Assistant Sandbox Escape (CVE-2026-39421, CVSS 6.3):** The ToolExecutor sandbox in MaxKB (an open-source enterprise AI assistant) versions 2.7.1 and below used LD_PRELOAD to intercept dangerous libc calls (execve, system, connect, mprotect). However, Python's `ctypes` library can invoke kernel syscalls directly, bypassing the interception entirely. Attackers used `pkey_mprotect` (which was not hooked) to allocate executable memory, achieving arbitrary code execution with network exfiltration capability. Fixed in MaxKB 2.8.0. This is a textbook example of why LD_PRELOAD/blocklist sandboxes are insufficient -- Python's dynamic nature provides too many paths to kernel access for interception-based approaches to cover.

**vm2 Node.js Sandbox Escape (CVE-2026-22709, CVSS 9.8, January 2026):** A critical sandbox escape in vm2, the most widely used Node.js sandboxing library, allows arbitrary code execution on the host via unsanitized Promise callback handlers. Async functions return a global Promise whose `.then()` and `.catch()` callbacks bypass vm2's internal sanitization, enabling prototype chain traversal to the host `Function` constructor. Fixed in vm2 3.10.2. This is particularly relevant because vm2 is used in AI tool execution environments, workflow automation (n8n, Langchain tool nodes), and code evaluation sandboxes. The vm2 library has accumulated over 20 known sandbox escape CVEs since 2022, demonstrating that JavaScript-based sandboxing is fundamentally fragile -- the language's dynamic prototype system provides too many escape paths for blocklist approaches to cover.

**LLM Sandbox Escape Research (March 2026):** The UK AI Security Institute published SandboxEscapeBench, evaluating frontier LLMs' ability to escape Docker containers across 18 scenarios. Key findings:
- Frontier models achieved ~40-50% success rates on moderate-complexity scenarios (runtime vulnerabilities, multi-step exploitation chains)
- Performance degraded to near-zero on kernel-level exploits requiring deep system reasoning
- All successful escapes exploited known misconfigurations or publicly disclosed CVEs -- no novel vulnerabilities were discovered
- Individual escape attempts cost approximately $1 at current inference pricing, making brute-force economically feasible
- The researchers concluded that "plain Docker isolation" is insufficient for frontier models and recommended hardware-enforced boundaries (microVMs)

These incidents collectively reinforce that for AI workloads executing untrusted code or running untrusted models, microVM-level isolation (Firecracker, Kata Containers) provides a fundamentally stronger security boundary than container-level isolation alone.

### Confidential Computing Performance (2025-2026 Benchmarks)

Recent benchmarking studies have refined performance expectations for confidential AI inference:

- **NVIDIA H100 Confidential Computing** overhead for typical LLM inference queries remains below 5%, with larger models and longer sequences experiencing nearly zero computational overhead within the GPU. The primary performance penalty comes from encrypted data transfers between CPU and GPU via bounce buffers, not from computation itself.
- **NVIDIA Vera Rubin NVL72** (announced CES 2026, partner availability H2 2026) encrypts every bus across 72 Rubin GPUs, 36 Vera CPUs, and the entire NVLink fabric, creating the first rack-scale unified trust domain for confidential computing. It integrates with NVIDIA Remote Attestation Services (NRAS) to provide cryptographic proof that CPUs, GPUs, NICs, firmware, drivers, and the running workload all match known-good reference measurements -- achieving zero-trust architecture at rack scale and directly addressing the multi-node TEE orchestration gap identified in requirement 4.1.5.
- **Red Hat and NVIDIA** have demonstrated a proof-of-concept for running AI workloads in Confidential Containers (CoCo) with NVIDIA GPU acceleration, providing Kubernetes-native TEE orchestration with GPU support. As of October 2025, Red Hat OpenShift confidential containers with NVIDIA confidential GPUs supports end-to-end attestation covering both CPU TEEs (Intel TDX or AMD SEV-SNP) and NVIDIA Blackwell/Hopper GPUs, creating a cryptographically provable chain of trust for AI inference workloads.
- **NVIDIA Zero-Trust Confidential AI Factories (March 2026):** NVIDIA published a reference architecture for zero-trust confidential AI factories that eliminates implicit trust in infrastructure. The architecture combines six pillars: hardware root of trust, Kata Containers runtime, minimal guest OS, attestation services (NRAS + Intel Trust Authority), secure workload lifecycle management, and native Kubernetes integration. CoCo remains in Early Access for GPU workloads -- not yet considered production-grade -- but the reference architecture establishes the target design for organizations deploying proprietary models on infrastructure they do not fully trust.
- **Composite attestation limits:** NVIDIA's confidential AI factory reference architecture uses a RATS-style key-release flow where a guest Attestation Agent gathers CPU/GPU evidence, a Key Broker Service delegates validation to an Attestation Service, and secrets are released only after measurements match policy. That workflow protects model keys and plaintext data from host inspection, but it does not address application vulnerabilities, availability attacks, or application-layer network/storage security. Auditors should therefore verify both the attestation chain and ordinary application controls such as authenticated service channels, egress policy, and patch management.
- The NVIDIA confidential computing architecture now covers **Rubin, Blackwell, and Hopper GPU families**, with hardware root of trust anchored on-die for attestation.
- **NVIDIA Blackwell B200 production deployment (March 2026):** Corvex achieved the first verified production deployment of confidential computing on HGX B200 systems. Blackwell is the first TEE-I/O capable GPU, supporting encrypted NVLink/NVSwitch communication between GPUs without bounce buffer overhead for intra-CVM traffic. Joint CPU-GPU attestation via Intel Trust Authority enables cryptographic verification of the full execution stack. Multi-GPU configurations (1-8 GPUs) are supported in TEE mode, a significant expansion over H100-generation systems.

### Defense-in-Depth for AI Workload Isolation

Production AI workload sandboxing requires layered defenses beyond compute isolation alone:

1. **Isolation boundaries:** Firecracker microVMs or Kata Containers for untrusted workloads; gVisor for lower-risk inference.
2. **Resource limits:** CPU, memory, GPU, and disk quotas to prevent resource exhaustion and noisy-neighbor attacks.
3. **Network controls:** Default-deny egress with proxy-based domain allowlists. Pre-stage all model dependencies before cutting network access.
4. **Permission scoping:** Short-lived credentials, tool-specific access grants, and minimal capability sets.
5. **Model validation:** SafeTensors preference, multi-tool scanning, fail-closed policies.
6. **Continuous monitoring:** Execution tracking, syscall auditing, and anomaly detection within sandboxed environments.

### Verification Checklist Updates (May 2026)

- Treat DNS as real egress. The AgentCore DNS-tunneling research and AWS's updated documentation show that a sandbox can block direct internet traffic while still allowing recursive DNS and selected service traffic. Verification should include DNS queries to attacker-controlled domains, direct TCP/UDP attempts, metadata-service access, and S3/object-store upload attempts.
- Verify the exact isolation layer, not just the product name. For Docker Sandboxes, confirm the workload is actually inside the microVM path and that host Docker socket access is absent. For Kata, record the VMM (`firecracker`, `cloud-hypervisor`, or `qemu`), Kata version, runtime class, and rootfs driver. For OpenShell, review the loaded policy and deny logs for the specific agent binary.
- Validate model loading in a negative test harness. Submit pickle, renamed pickle (`.bin` / `.pt`), malformed ZIP, model config with unsafe Hydra `_target_`, and SafeTensors-only models. The expected result is reject-before-load for unsafe formats and successful load only for approved formats from an approved registry.
- Tie attestation evidence to release policy. For TEE workloads, confirm the relying party checks freshness, hardware evidence, firmware/driver versions, image measurements, GPU attestation where applicable, and reference-value policy before releasing model decryption keys.
- Separate confidentiality from containment. Confidential computing protects data in use from the infrastructure operator; it does not make untrusted model code safe. High-value confidential workloads still need sandboxing, egress control, vulnerability management, and model artifact allowlisting.

- [ ] What is the practical performance overhead of confidential GPU inference (NVIDIA H100 CC mode) for large language models across different batch sizes? *Partial answer: benchmarks show sub-5% overhead for typical LLM queries, with overhead concentrated in CPU-GPU data transfer rather than computation.*

- [ ] Can homomorphic encryption (HE) complement or replace SMPC for specific inference workloads, and what are the size/latency tradeoffs? *A 2025 SoK paper found that only three FHE methods achieve computational generality for AI (bit-wise TFHE, Scheme Switching, Encoding Switching), all with orders-of-magnitude overhead -- multiplying two encrypted 16-bit integers in TFHE takes ~30 seconds. - [ ] What is the security boundary of WASI AI when used for edge inference -- can malicious model files exploit JIT-compiler bugs to escape the Wasm sandbox?
- [ ] As frontier LLMs demonstrate increasing container escape capabilities (SandboxEscapeBench, March 2026), at what capability threshold should organizations mandate microVM isolation for all agentic AI workloads?
- [ ] How should organizations handle the Azure Confidential Containers AKS preview sunset (March 2026) -- what are the production-ready alternatives for Kubernetes-native confidential AI?
