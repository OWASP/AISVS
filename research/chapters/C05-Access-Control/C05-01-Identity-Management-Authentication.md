# C5.1: Authentication

> [Back to C05 Index](C05-Access-Control.md)
> **Last Researched:** 2026-06-20

## Purpose

Establish verified identities for all entities interacting with AI systems, with authentication strength appropriate to the risk level. AI systems introduce identity challenges beyond traditional web applications: model-serving endpoints, inference APIs, model registries, and agent tool gateways often bypass centralized identity providers in favor of standalone API keys. In multi-agent and federated deployments, agent identity itself is now a first-order control: short-lived, cryptographically signed credentials must bind the issuing system, workload key, audience, and user delegation path so compromised agents cannot replay broad bearer tokens across systems. The 2026 Verizon DBIR makes the same point bluntly — "we should pay special attention to service and machine accounts, as those will likely be the ones leveraged in our potential agentic AI future" — and pairs it with a 60% YoY rise in third-party breaches frequently rooted in compromised OAuth tokens.

> **Scope note:** Controls in this section govern the authentication layer for humans, services, and AI agents accessing AI system resources. Resource authorization, query-time enforcement, output entitlement enforcement, and PDP isolation are addressed in C5.2; infrastructure-level multi-tenant isolation is in C5.3. General identity management, federation, and MFA patterns are covered by ASVS v5 V6; this chapter addresses AI-specific concerns that ASVS does not cover.

## Implementation Notes

**High-risk operation inventory.** Maintain a machine-readable list of operations that always require step-up: model-weight export, production model deployment, training-pipeline write access, dataset export, model-registry admin changes, inference endpoint key creation, and production agent tool permission changes. The list should be owned jointly by security and the model platform team, not buried in application code.

**Device-code flow boundary.** Device-code authentication is useful for constrained devices, but it is a poor fit for high-risk AI administration because the user authorizes an attacker-controlled session on a legitimate identity-provider page. Where the IdP supports it, block device-code flow globally and allow it only for documented device classes, managed networks, and non-administrative resources.

**Non-human identity lifecycle.** Every agent, model-serving workload, CI/CD job, and MCP server should have an owner, an issuance path, a rotation schedule, and a revocation path. Prefer workload identity, token exchange, and dynamic secrets over API keys stored in environment variables or local agent configuration files. Treat third-party AI SaaS integrations (chat-assistant connectors, IDE plugins, agent gateways) as first-class identities with their own rotation cadence — the Salesloft Drift and Context.ai incidents both started by stealing an integration's refresh token.

**Authenticate the MCP boundary before tuning token quality.** The 2026 exposure measurements (12,520 internet-reachable MCP services per Censys; ~40% of remote MCP servers with no authentication) show the dominant real-world failure is not weak token binding but the complete absence of caller authentication on agent tool endpoints. Inventory every MCP server reachable from outside a trust boundary, confirm it rejects anonymous `tools/call` requests, and only then assess token lifetime, audience, and sender-constraining. Treat database-backed MCP servers (Pinot, Doris, RDS-style connectors) as production database surfaces, not developer conveniences.

**Issuer-key and tenant binding.** Treat "signed JWT" as necessary but insufficient. Verifiers should pin accepted issuers, fetch keys from the issuer's JWKS or validate the issuer's certificate chain, enforce narrow `aud`/`resource` values, check `iss`/`tid` against the *target* tenant for every request (not just the caller's tenant), and require DPoP, mTLS, WIMSE HTTP signatures, or an equivalent proof-of-possession signal before accepting agent-to-agent credentials for high-impact actions.

**Delegation chain integrity.** When an agent acts on behalf of a user, the verifier should be able to reconstruct the full chain: which human authorized the action, which agent or workload is executing it, which model/code version produced the request, and which intermediary forwarded it. RFC 8693 token exchange with `act` (actor) claims, OAuth On-Behalf-Of flows, and CoSAI's code-bound credential pattern give workable building blocks; gateways and PDPs should refuse requests whose delegation chain cannot be replayed from audit data.

**Audit evidence.** For humans, preserve the auth context / ACR value, authenticator class, step-up timestamp, operation name, and session expiry. For agents, preserve issuer, audience, subject, proof-of-possession status, delegated user or actor chain, token exchange event ID, model/code version (where the agent identity is code-bound), and downstream credential lease ID where available.

**Treat the agent's execution path as a token-exfiltration sink.** The Codex Unicode-branch-name CVE showed that short-lived OAuth tokens do not help when the agent itself runs attacker-controlled strings through a shell. Strip or strictly validate any external input that flows into shell, container, or subprocess invocations the agent makes on its own behalf — Git refs, repository names, tool arguments, user-supplied URLs, branch parameters, and MCP-tool arguments — *before* tokens are mounted into the environment. Where feasible, scope the agent's token to a separate ephemeral identity that has no read access to the repos or APIs the user owns, and rotate it per task.

## Notable Incidents

- **Klue / Battlecards OAuth supply-chain breach (June 2026).** A threat actor tracked as "Icarus" compromised legacy service-account credentials, pivoted into the infrastructure of competitive-intelligence vendor Klue, deployed token-theft code, and reused harvested OAuth tokens to enumerate and begin exfiltrating customer Salesforce CRM records via automated REST API calls. Salesforce disabled the Klue "Battlecards" connected app on 2026-06-11 to contain it. The pattern mirrors Salesloft Drift almost exactly — a long-lived third-party integration token defeats end-user MFA entirely — and reinforces that AI/SaaS connectors holding standing OAuth grants must be treated as production identities with short-lived, audience-bound credentials and a fast revocation path. ([ThreatLocker writeup](https://www.threatlocker.com/blog/klue-saas-supply-chain-compromise-through-long-lived-oauth-tokens))
- **MCP Go SDK cross-site tool execution (CVE-2026-33252, June 2026).** The official Model Context Protocol Go SDK's Streamable HTTP transport accepted browser-generated cross-site `POST` requests without validating the `Origin` header or requiring `Content-Type: application/json`. On a stateless/sessionless local MCP server with no `Authorization`, any web page the developer happened to visit could drive `tools/call` — a CSRF (CWE-352) path to tool execution rated CVSS 7.1, fixed in go-sdk 1.4.1. A reminder that "local and unauthenticated" is not a safe default for agent tool endpoints. ([CVE-2026-33252 lookup](https://vulnerability.circl.lu/vuln/cve-2026-33252))
- **Unauthenticated MCP server exposure wave (May–June 2026).** Akamai disclosed three MCP database-server flaws on 2026-05-12: an authentication-validation bypass in Apache Pinot's MCP server enabling SQL injection (StarTree added OAuth in response), Alibaba's RDS MCP server invoking its RAG tool without authenticating the caller (Alibaba declined to patch), and SQL injection in Apache Doris MCP (CVE-2025-66335). The same window produced measurement data: Censys counted 12,520 internet-reachable MCP services, an academic study found ~40% of remote MCP servers expose tools with no authentication and traced nine CVEs to broken OAuth flows, and Trend Micro found CVSS 9.8 command-injection flaws in unofficial AWS/Azure MCP servers. The NSA published MCP security design-considerations guidance in June 2026. Related: CVE-2026-32211 (Azure MCP Server missing authentication, CVSS 9.1, April 2026) and CVE-2026-20205 (Splunk MCP Server app logging session/authorization tokens in cleartext, fixed in 1.0.3). ([Akamai research](https://www.akamai.com/blog/security-research/one-fluke-3-pattern-mcp-back-end-vulnerabilities), [measurement study](https://arxiv.org/abs/2605.22333), [Trend Micro](https://www.trendmicro.com/vinfo/us/security/news/vulnerabilities-and-exploits/update-on-exposed-mcp-servers-the-threat-widens-to-the-cloud), [Splunk advisory](https://advisory.splunk.com/advisories/SVD-2026-0407))
- **Red Hat npm packages trojanized via GitHub Actions OIDC (2026-06-01).** A credential-stealing worm published 96 malicious versions across 32 packages in the `@redhat-cloud-services` npm scope (~117k weekly downloads). The packages were published through GitHub Actions OIDC federation rather than a stolen static npm token — a reminder that short-lived federated tokens inherit the integrity of the pipeline that attests them, and that publishing pipelines are high-risk operations deserving step-up or human approval. ([Aikido writeup](https://www.aikido.dev/blog/red-hat-npm-packages-compromised-credential-stealing-worm))
- **FBI advisory on device-code phishing kits; "ConsentFix" legacy-scope abuse (May 2026).** An FBI PSA named the Telegram-distributed Kali365 (aka CLURE) kit, which combines device-code phishing with adversary-in-the-middle capture, as a chief enabler of OAuth-token theft. Push Security tracked a 37.5x jump in device-code phishing pages across 14+ kits, and the "ConsentFix" campaign abuses undocumented legacy OAuth scopes that are not logged by default — making post-compromise detection harder. Passkeys do not stop the technique, because the bypass is at the consent/token-issuance layer. ([Push Security analysis](https://pushsecurity.com/blog/device-code-phishing), [Infosecurity Magazine](https://www.infosecurity-magazine.com/news/oauth-phishing-campaigns/))
- **TeamPCP supply-chain campaign hits LiteLLM (March 2026).** A four-wave campaign (2026-03-19 to 03-24) compromised the Trivy project, then reused tokens stolen in that wave five days later to compromise PyPI publishing for LiteLLM — a widely deployed AI gateway — with a memory-scraping credential stealer running on GitHub Actions runners. AI gateways concentrate provider API keys and agent credentials, making their publishing identities high-value targets. ([The Register](https://www.theregister.com/security/2026/03/24/litellm-infected-with-credential-stealing-code-via-trivy/5223422))
- **CVE-2026-32173 — Azure SRE Agent improper authentication (disclosed 2026-04-03, CVSS 8.6).** A multi-tenant design oversight let any holder of a free Entra ID account silently subscribe to another tenant's SRE Agent SignalR channel and watch live AI command streams, internal reasoning, tool calls, and embedded credentials. Microsoft fixed it server-side; the lesson is that signed-token validation without a tenant-scope check is not authentication. ([CVE advisory](https://github.com/advisories/GHSA-85hw-hqj5-m956), [BankInfoSecurity coverage](https://www.bankinfosecurity.com/token-flaw-turned-azures-ai-agent-into-spy-a-31462))

Attackers hid a payload behind 94 ideographic spaces (U+3000) and `${IFS}`-substituted whitespace so the branch appeared as `main` in the Codex UI; on clone, bash executed inside the container, exfiltrating the agent's short-lived GitHub OAuth token (and GitHub Installation Access tokens when invoked via `@codex` comments). - **EvilTokens phishing-as-a-service / OAuth-consent phishing (Feb 2026).** Within 5 weeks of launch, EvilTokens compromised 340+ Microsoft 365 organizations across 5 countries by impersonating device-login pages so victims clicked through what looked like an MFA prompt and instead approved an OAuth grant. Direct implication for AI step-up: never let an OAuth consent grant alone clear a high-risk AI operation. ([The Hacker News coverage](https://thehackernews.com/2026/05/the-new-phishing-click-how-oauth.html))
- **Meta rogue AI agent Sev-1 (March 2026).** An internal agent with legitimate forum-posting privileges and clean identity checks gave confident but wrong guidance that an engineer executed, broadening data visibility internally for ~2 hours. ([Kiteworks writeup](https://www.kiteworks.com/cybersecurity-risk-management/meta-rogue-ai-agent-data-exposure-governance/), [VentureBeat analysis](https://venturebeat.com/security/meta-rogue-ai-agent-confused-deputy-iam-identity-governance-matrix))
- **Okta "Phishing the agent" / OpenClaw token exfiltration (April–May 2026).** Okta Threat Intelligence showed an OpenClaw + Claude Sonnet 4.6 deployment handing over an OAuth token after a guardrail reset — the agent forgot it had printed the token, was asked to screenshot the terminal, and forwarded the image to Telegram. ([CSO Online coverage](https://www.csoonline.com/article/4166133/ai-agents-can-bypass-guardrails-and-put-credentials-at-risk-okta-study-finds.html))
- **Vercel / Context.ai compromise (April 2026).** A third-party AI tool used by a Vercel employee was compromised, exposing OAuth tokens and Vercel environment variables. ## Implementation Maturity

- **Phishing-resistant human auth (5.1.1, NIST AAL3) — mature.** FIDO2, device-bound passkeys, and Conditional Access / Identity Engine policies are well documented; the hard part is *enumerating* AI-specific high-risk operations and wiring them to step-up. NIST SP 800-63-4 (final, July 2025) is now the authoritative reference.
- **Sender-constrained OAuth (DPoP, mTLS) — emerging-to-mature.** Stable RFCs (RFC 9449, RFC 8705) and production support in Keycloak 26.4 (DPoP GA), Auth0 (mTLS GA, DPoP EA), and Microsoft Entra. Less mature for non-OAuth agent-to-agent traffic.
- **SPIFFE / SPIRE workload identity — production-grade.** CNCF-graduated, integrated with HashiCorp Vault 1.21 (SPIFFE auth) and Vault 2.0 (SPIFFE secrets engine). Best fit for service identity in Kubernetes / multi-cluster deployments. Still workload-coarse — does not by itself solve per-agent granularity inside a shared runtime.
- **WIMSE workload identity standards — early/active.** Architecture, workload credentials (WIT/WIC with required proof-of-possession), HTTP signature profile, Workload Proof Token, and mutual-TLS drafts are all moving through IETF in 2025–2026; the identity-practices draft passed WG last call in April 2026 and is awaiting write-up. Implementations exist as reference code (SPIRE 1.15 now issues WIT-SVIDs with `iss` claims), but production deployments are still rare.
- **MCP authorization — emerging, spec maturing faster than deployments.** The 2025-11-25 spec (CIMD client registration, OIDC discovery, mandatory PKCE verification, Cross App Access extension) supersedes 2025-06-18, and the 2026-07-28 release candidate adds RFC 9207 `iss` validation and issuer-bound client credentials. Adherence in the wild is poor: 2026 measurements found ~40% of remote MCP servers with no authentication at all, the June 2026 VIPER-MCP study confirmed 106 exploitable zero-days (67 CVEs) across ~40k MCP server repos, and many implementations still use bearer API keys, ship CSRF-able transports (CVE-2026-33252), or pass tokens through to downstream APIs.
- **Code-bound agent credentials / delegation-chain enforcement — emerging.** CoSAI's Agentic IAM framework (April 2026) and a growing set of IETF individual drafts describe the model: `draft-klrc-aiagent-auth-02` (2026-06-01) — co-authored across Defakto, AWS, Zscaler, Ping Identity, OpenAI, and Okta — composes WIMSE/SPIFFE identity, proof-of-possession, mTLS, and OAuth delegation grants instead of minting a new primitive, and `draft-ni-wimse-ai-agent-identity-02` (Huawei) proposes WIMSE dual-identity credentials that bind agent and owner together. None has WG adoption yet, and production tooling outside of Auth0, Okta, and a handful of gateway vendors is thin. The Meta Sev-1 incident is the warning shot that "valid token" is not the same as "authorized intent."
- **Vendor agent-identity platforms (RSAC 2026) — newly shipping.** Five vendors announced agent-identity stacks during RSAC 2026 (late April / early May): Cisco Duo Agentic Identity (agents as distinct identity objects mapped to human owners, every tool call through an MCP gateway in Secure Access SSE) plus Identity Intelligence agent discovery and the open-source DefenseClaw framework (GitHub, 2026-03-27); Microsoft's MCP gateway; Palo Alto Networks Prisma AIRS 3.0 with an agentic identity provider for agent discovery and credential validation; CrowdStrike Charlotte AI AgentWorks for custom security agents on Falcon; and Cato Networks' agent identity additions. The common gap reviewers flagged at RSAC: every framework verifies *who* the agent is but few track *what* the agent actually did once authenticated, leaving delegation-chain and action-governance evidence weak. ([VentureBeat — five frameworks, three gaps](https://venturebeat.com/security/rsac-2026-agent-identity-frameworks-three-gaps), [VentureBeat — Cisco/CrowdStrike/RSAC maturity model](https://venturebeat.com/security/cisco-crowdstrike-rsac-2026-agent-identity-iam-gap-maturity-model), [Constellation Research](https://www.constellationr.com/insights/news/rsac-2026-everyone-trying-secure-ai-agents-various-claws))
- **Cloud / IdP agent identity — GA wave (April–May 2026).** Microsoft Entra Agent ID is GA with agent-specific Conditional Access templates ("Block access for high-risk agent identities", autonomous-agent and on-behalf-of policies), downstream token-validation guidance, and federation patterns for non-Microsoft agent platforms; Okta for AI Agents went GA 2026-04-30 with agents as first-class Universal Directory identities, Cross App Access (ID-JAG), shadow-agent discovery, and Universal Logout for instant agent-token revocation; Google Cloud Agent Identity is GA for Agent Runtime with SPIFFE-based principals, auto-rotated 24-hour X.509 certificates, and access tokens cryptographically bound to the agent certificate; AWS Bedrock AgentCore Identity offers workload identities with SigV4/JWT inbound auth and brokered outbound OAuth; Aembit's IAM for Agentic AI (GA April 2026) evaluates agent and human operator together per request and mints per-session tokens so agents never hold standing credentials. Issuance is now well served; independent verification and delegation-chain audit tooling remain the thin layer. ([Entra Agent ID what's new](https://learn.microsoft.com/en-us/entra/agent-id/whats-new-agent-id), [Okta Showcase 2026](https://www.okta.com/newsroom/press-releases/showcase-2026/), [Google Cloud Agent Identity](https://docs.cloud.google.com/iam/docs/agent-identity-overview), [Aembit GA](https://aembit.io/blog/aembit-iam-for-agentic-ai-is-now-generally-available/))
