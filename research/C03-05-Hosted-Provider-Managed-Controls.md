# C3.5: Hosted and Provider-Managed Model Controls

> [Back to C03 Index](C03-Model-Lifecycle-Management.md)
> **Last Researched:** 2026-04-17

## Purpose

Hosted and provider-managed models may change behavior without notice. These controls help ensure visibility, reassessment, and safe operation when the organization does not control the model weights.

This is an increasingly critical control area as most organizations consume AI capabilities through hosted APIs rather than self-hosted models. Provider transparency remains inconsistent: Azure AI Foundry offers 60-day GA retirement notices, OpenAI returns exact dated snapshot IDs in API responses, but alias resolution changes remain opaque and silent updates are common. The GPT-4o sycophancy incident (April 2025) demonstrated that even system prompt changes -- not model weight updates -- can cause severe behavioral regressions with no advance consumer notification. The pattern repeated in 2026: on March 3, 2026, Anthropic reduced Claude Code's default "effort" level from high to medium (also adjusting default adaptive thinking behavior) to economize on compute. Anthropic later confirmed it "made real configuration changes that objectively reduced default thinking depth across all surfaces" even though the underlying model weights did not change -- customers had no programmatic signal of the behavior shift until developer communities surfaced complaints. MIT research found 91% of ML models degrade over time, and no standard protocol exists for provider-to-consumer change notifications. With EU AI Act GPAI enforcement powers (including fines up to 3% of global annual turnover or EUR 15 million) activating August 2, 2026, the regulatory cost of treating hosted models as opaque dependencies is rising sharply.

## Requirements

| # | Requirement | Level | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|-----------------|----------------------|--------------|
| **3.5.1** | **Verify that** hosted model dependencies are inventoried with provider, endpoint, provider-exposed model identifier, version or release identifier when available, and fallback or routing relationships. | 1 | Shadow AI usage via untracked hosted model endpoints; inability to assess blast radius when a provider deprecates or silently updates a model; compliance gaps from unaudited third-party AI dependencies; fallback routing to untested models during outages. Maps to MITRE ATLAS AML.T0010 (ML Supply Chain Compromise) and OWASP LLM Top 10 2025 LLM03 (Supply Chain Vulnerabilities). | Inspect hosted model inventory for completeness: verify each entry includes provider name, endpoint URL, model identifier (e.g., `gpt-4o-2024-08-06`, `claude-sonnet-4-20250514`), version or release identifier where exposed, and documented fallback/routing relationships. Cross-reference against actual API calls in production logs. Confirm the inventory updates when new hosted models are added or routing changes. Tools: MLflow AI Gateway (`mlflow.litellm.autolog()`), LiteLLM proxy (1,600+ models, unified logging across OpenAI/Anthropic/Bedrock/Vertex/Azure), Portkey (250+ models, edge-worker gateway with metadata tracking), Helicone gateway, PromptLayer. | Major providers expose model identifiers at varying granularity: OpenAI returns the exact snapshot ID in the response `model` field (e.g., dated snapshot identifiers); Anthropic returns the model identifier and exposes `request-id` and `anthropic-organization-id` headers; Google Vertex AI and AWS Bedrock include model identifiers in response metadata. However, some providers alias model names (e.g., `gpt-4o` resolves to a dated snapshot that can change), making inventory accuracy dependent on logging the actual resolved identifier, not just the alias. The deprecations.info service tracks deprecation timelines across 7+ providers (OpenAI, Anthropic, Google AI/Gemini, Vertex AI, AWS Bedrock, Cohere, xAI) via RSS (`v1/feed.xml`), JSON Feed (`v1/feed.json`), and raw JSON (`v1/deprecations.json`), updated daily. As of April 2026, no industry-standard schema exists for hosted model inventory records, though CycloneDX ML BOM (v1.5+) can represent third-party model dependencies. EU AI Act GPAI obligations (effective Aug 2025, enforcement Aug 2, 2026) require technical documentation of third-party model dependencies retained for at least ten years after a model is placed on the market. *(Updated 2026-04)* |
| **3.5.2** | **Verify that** provider model, version, or routing changes trigger security re-evaluation before continued use in high-risk workflows. | 2 | Silent provider model updates changing safety behavior, output quality, or compliance posture without consumer awareness; prompts degrading or producing unsafe outputs after provider-side model swaps; regression in adversarial robustness from provider version changes; compliance violations when updated models no longer meet previously validated thresholds. The GPT-4o sycophancy incident (April 29, 2025) is a concrete example: a system prompt change caused the model to endorse harmful and delusional statements (including praising a "shit on a stick" business plan, endorsing a user's decision to stop medication, and reportedly supporting plans to commit terrorism), and consumers relying on GPT-4o in high-risk workflows had no advance warning or re-evaluation opportunity. The pattern repeated in 2026: Anthropic's March 3, 2026 reduction of Claude Code default effort from high to medium (and default adaptive-thinking behavior change) reduced task computational investment without customer notification, prompting a wave of developer complaints before Anthropic publicly acknowledged the configuration change in April 2026. | Review change detection mechanisms: verify the system monitors provider model identifiers (via API response fields, deprecation feeds, or provider changelogs) for changes. Confirm that a detected model change triggers a re-evaluation workflow before the new model version is used in high-risk paths. Test by simulating a model identifier change and verifying the re-evaluation gate fires. Inspect re-evaluation criteria: safety test suite re-run, output quality benchmarks, compliance checklist. Tools: deprecations.info (RSS/JSON feeds for 7+ providers, updated daily), Azure Service Health alerts (auto-notification to owner/contributor/reader/monitoring roles), Portkey observability (model name/provider/version metadata for variance and regression detection after upgrades), Braintrust Gateway (evaluations against production traffic to catch regressions in CI/CD), custom API response monitoring. | Provider change notification remains inconsistent. Azure AI Foundry formalized lifecycle stages in the March 2026 docs refresh (Preview, GA, Legacy, Deprecated, Retired): models stay in Legacy for at least 30 days, then Deprecated for at least 90 days before retirement; Azure OpenAI specifically provides at least 60 days notice for GA retirements and at least 30 days notice before preview model version upgrades, with a "not sooner than" retirement date (typically one year out) assigned at launch. OpenAI publishes deprecation timelines (e.g., March 24, 2026 notice of Videos API + Sora 2 removal on September 24, 2026), but aliases can resolve to new snapshots without notice. Anthropic publishes model lifecycle information but silent routing and effort-level changes within model families are possible, as the March 2026 Claude Code effort reduction demonstrated. MIT research found 91% of ML models degrade over time. EU AI Act GPAI enforcement powers (including the authority to impose fines up to 3% of global annual turnover or EUR 15 million) activate August 2, 2026, and providers must retain technical documentation for at least ten years. NIST released a concept note for an AI RMF Profile on Trustworthy AI in Critical Infrastructure on April 7, 2026. No standard protocol exists for provider-to-consumer change notifications. *(Updated 2026-04)* |
| **3.5.3** | **Verify that** logs record the exact hosted model identifier returned by the provider, or explicitly record that no such identifier was exposed. | 2 | Inability to reconstruct which model version produced a specific output during incident investigation; audit trail gaps when provider aliases resolve to different snapshots over time; compliance failures requiring per-request model traceability; forensic ambiguity when a provider silently swaps underlying models. Particularly acute for post-incident analysis when a behavior regression (e.g., the GPT-4o sycophancy rollback or the Claude Code effort reduction) surfaces weeks later and investigators must reconstruct which exact model/config produced earlier outputs. | Inspect application logs for hosted model API calls. Verify each log entry captures the model identifier from the API response (e.g., OpenAI `model` field in response body, Anthropic `model` field, Bedrock model ID in response metadata). For providers that do not expose a model identifier, confirm the log explicitly records the absence. Test by querying logs for a specific time window and verifying model identifiers are present and granular (dated snapshots, not just aliases). Tools: PromptLayer, Langfuse, MLflow AI Gateway autologging (`mlflow.litellm.autolog()`), LiteLLM callbacks (two-line integration with PromptLayer, Lunary, Langfuse, Helicone, Traceloop), Helicone request-level tracing, Portkey metadata capture, custom middleware. | As of April 2026, major providers expose model identifiers at different granularity: OpenAI returns the exact dated snapshot in the response `model` field (e.g., `gpt-4o-2024-08-06`); Anthropic returns the requested model identifier and provides `request-id` headers for traceability; Google Vertex AI includes model version in response metadata; AWS Bedrock includes the model ARN. However, when using provider aliases (e.g., `gpt-4o` without a date suffix), the resolved snapshot can change without the consumer's knowledge -- logging the response-side identifier rather than just the request-side alias is critical. Note that API-exposed identifiers do not always capture configuration-level changes: the March 2026 Claude Code effort reduction changed default `effort` from high to medium without a model identifier change, so teams that logged only model IDs were unable to distinguish pre-change from post-change outputs. Practitioners increasingly log the full request configuration (effort, temperature, system prompt hash, tools list) alongside the model identifier. Some proxy/gateway tools (LiteLLM, Portkey) normalize model identifiers across providers but may obscure provider-side resolution details. The requirement to log the *absence* of a model identifier is equally important for audit completeness. *(Updated 2026-04)* |
| **3.5.4** | **Verify that** high-assurance deployments fail closed or require explicit approval when the provider does not expose sufficient model identity or change notification information for verification. | 3 | Operating high-risk AI workflows on opaque provider models whose behavior, version, or safety posture cannot be independently verified; regulatory non-compliance when audit evidence of model identity is unavailable; undetected provider-side model swaps undermining previously validated safety thresholds; liability exposure from deploying unverifiable models in regulated domains (healthcare, finance, legal). Silent configuration drift (e.g., the March 2026 Claude Code effort-level reduction) can cause high-assurance workloads to silently consume less computational effort than validated, without changing model identifiers. | Define minimum model identity requirements for high-assurance tiers (e.g., dated snapshot ID, version hash, change notification feed subscription, documented configuration parameters including effort/reasoning mode). Test by configuring a provider endpoint that returns insufficient identity information and verify the system either blocks the request or routes it through an explicit approval workflow. Confirm approval records capture the risk acceptance decision, approver identity, and business justification. Review fallback behavior: does the system degrade gracefully (e.g., route to a verified model) or halt? Run shadow evaluations via Braintrust/Portkey against production traffic to detect behavioral regression that identifier-only monitoring would miss. | This is a defense-in-depth control for scenarios where 3.5.3 logging reveals persistent identity gaps. As of April 2026, no provider guarantees immutable model identifiers -- even dated snapshots could theoretically be repointed, and configuration-level changes can occur without any identifier change. The fail-closed posture is most relevant for regulated industries where the EU AI Act GPAI obligations became effective August 2, 2025 and enforcement powers (fines up to 3% of global annual turnover or EUR 15 million) activate August 2, 2026. GPAI providers making "significant modifications" are treated as new providers under the Act, but the line between "significant" and "minor" remains unsettled and is interpreted by the AI Office on a case-by-case basis. MIT found 95% of corporate AI initiatives struggle with governance; fail-closed defaults help enforce discipline. Implementation requires defining a "sufficiency threshold" for model identity -- a policy decision that varies by risk appetite and regulatory context. Level 3 reflects the organizational maturity required: most teams lack formal policies for hosted model identity sufficiency. *(Updated 2026-04)* |
