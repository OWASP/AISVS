# C8.1: Access Controls on Memory & RAG Indices

> [Back to C08 Index](C08-Memory-and-Embeddings.md)
> **Last Researched:** 2026-04-15

## Purpose

Enforce fine-grained access controls and query-time scope enforcement for every vector collection. Without proper access controls on vector stores, similarity queries can cross tenant boundaries, unauthorized users can retrieve or modify embeddings, and poisoned documents cannot be traced back to their source. As of April 2026, major managed vector databases (Pinecone, Weaviate, Qdrant) offer RBAC, JWT authentication, namespaces, and compliance certifications (SOC 2, HIPAA, ISO 27001). Milvus 2.6, now GA on Zilliz Cloud as of early 2026, adds hot/cold tiering and marks a shift from "vector search + glue code" to a full retrieval engine. However, UpGuard's April 2025 scan found 1,170 internet-exposed Chroma instances, with roughly a third allowing anonymous access — a reminder that defaults matter. On the provenance front, RAGShield (April 2026) introduced C2PA-inspired cryptographic document attestation for RAG ingestion pipelines, and organizations are extending CycloneDX ML-BOM to include integrity attestation records for retrieval sources. The attack landscape continues to expand: the Black-Hole Attack (Li et al., April 2026) demonstrated that injecting vectors near the geometric centroid of embedding space achieves 99.85% top-10 contamination at just 1% poisoning rates, while the Retrieval Pivot Attack (Thornton, February 2026) showed that hybrid vector+knowledge-graph RAG systems amplify leakage 160–194x compared to vector-only retrieval. RevPRAG (EMNLP 2025) now provides a detection pipeline achieving 98% TPR with 1% FPR by analyzing LLM activation patterns during generation. Anomaly detection is maturing — practical thresholds of 0.85 cosine similarity for individual documents and 0.90 for batch clustering have been validated as effective baselines — but no major vector database offers native anomaly detection yet.

| **8.1.3** | **Verify that** cross-scope access attempts (e.g., cross-tenant similarity queries, namespace traversal, tag bypass) are detected and rejected. | 2 | Namespace traversal attacks where an attacker crafts queries or metadata filters to escape their tenant boundary. | **8.1.4** | **Verify that** every ingested document is tagged at write time with source, writer identity (authenticated user or system principal), timestamp, batch ID, and embedding model version. | 2 | Data provenance attacks — without lineage metadata, poisoned vectors cannot be traced back to their source for quarantine or removal. Also prevents "ghost documents" inserted by compromised ingestion pipelines. Maps to MITRE ATLAS AML.T0071 (False RAG Entry Injection). Prompt Security's 2025 RAG_Poisoning_POC demonstrated that a single poisoned embedding can alter system behavior across multiple unrelated queries — provenance tagging is essential for rapid quarantine. RAGForensics (ACM Web Conference 2025) — the first traceback system for RAG poisoning — relies on provenance metadata to iteratively identify and remove poisoned texts, achieving >97.4% detection accuracy with <2.7% false positive rate. Without provenance tags, traceback is impossible. | Inspect ingestion pipeline code for metadata attachment at write time. Query stored vectors and confirm all required metadata fields are present. Attempt to update immutable metadata fields via API and verify rejection. Audit batch ingestion logs for completeness. Verify that signed ingestion jobs are accepted only from allow-listed sources, with review workflows for high-risk collections. | Immutability enforcement varies: Pinecone metadata can be overwritten via upsert; Weaviate properties can be updated unless schema-level restrictions are set. True immutability may require append-only storage patterns or write-once object storage for metadata alongside the vector index. Schneider's defense architecture (2025) recommends provenance tagging as part of Layer 2 (memory sanitization), including creation time, session context, and trust scores for each stored entry. RAGShield (April 2026) extends this with C2PA-inspired cryptographic document attestation — documents are signed at ingestion and unsigned/forged documents are rejected before vectorization. RAGShield also introduces a formal taint lattice with cross-source contradiction detection and NIST SP 800-53 compliance mapping across 15 control families. On 430 real IRS document attacks, RAGShield achieved 0.0% ASR while embedding-based defenses missed 79–90% of attacks. EU AI Act high-risk obligations (enforceable August 2026) require data governance and record-keeping for AI systems — provenance metadata on ingested documents directly supports these compliance requirements. The GAO documented 89 RAG-based citizen-facing services across 28 U.S. federal agencies as of FY2025, highlighting the scale at which provenance tracking matters for government deployments. Organizations are now extending CycloneDX ML-BOM to include integrity attestation records for retrieval sources alongside training corpora, documenting curation method and contributor vetting as first-class provenance artifacts. On April 7, 2026, NIST released a concept note for an AI RMF Profile on Trustworthy AI in Critical Infrastructure, which will guide operators toward specific risk management practices — provenance metadata on ingested documents directly supports the "Map" and "Manage" functions of NIST AI RMF 1.0. |
| **8.1.7** | **Verify that** document metadata tags applied at ingestion are immutable after initial write and cannot be modified by subsequent pipeline stages or user operations. | 2 | Metadata tampering post-ingestion — if an attacker or compromised pipeline stage can modify provenance tags (source, writer identity, timestamp, batch ID), poisoned documents can be re-attributed to trusted sources, undermining quarantine and forensic workflows. Maps to MITRE ATLAS AML.T0071 (False RAG Entry Injection). | Attempt to overwrite metadata on existing vectors via the update/upsert API — verify rejection or version-controlled append. Inspect database schema or configuration for write-once or append-only metadata policies. Review ingestion pipeline code for metadata modification after initial write. Test that bulk re-ingestion preserves original metadata rather than overwriting it. Verify that audit logs capture any metadata change attempts. | Most vector databases do not enforce metadata immutability natively. Pinecone metadata can be overwritten via upsert — the API treats metadata as mutable by design. Weaviate properties can be updated unless cross-references or schema-level restrictions prevent it. Qdrant allows metadata (payload) updates on existing points. Milvus supports field-level updates in v2.3+. Compensating controls: (1) use lakeFS to version embeddings and metadata as immutable commits, enabling rollback and audit trails at the storage layer; (2) RAGShield's C2PA-inspired cryptographic attestation (April 2026) signs documents at ingestion so any post-write metadata change invalidates the signature; (3) write metadata to append-only storage (e.g., WORM-configured S3, Azure immutable blob) alongside the vector index, treating the object store as the authoritative provenance record; (4) implement an application-layer middleware that hashes metadata at write time and verifies the hash before any retrieval, rejecting tampered entries. The MIND metadata-driven ingestion pattern (2025) uses Delta Lake for ACID-guaranteed metadata repositories with version control built in. As of April 2026, no major vector database offers a native "freeze metadata after write" toggle — this remains an application-architecture concern. |
| **8.1.5** | **Verify that** RAG pipeline retrieval events log the query issued, the documents or chunks retrieved, similarity scores, the knowledge source, and whether retrieved content passed prompt injection scanning before being incorporated into model context. | 2 | Undetected RAG poisoning — without retrieval logging, poisoned documents that influence model outputs cannot be identified forensically. Also enables detection of indirect prompt injection via retrieved content (OWASP LLM01:2025). Prompt Security's 2025 research showed 80% attack success rate across multiple queries from a single poisoned embedding, and retrieval monitoring (logging document access patterns) is cited as a primary defense — repeated hits from a single source indicate poisoning. RAGForensics (ACM Web Conference 2025) demonstrated that retrieval logs are the foundation for traceback — the system iteratively retrieves candidate texts and uses LLM-guided evaluation to identify poisoned documents, achieving >97.4% detection accuracy with <3.2% false negative rate. Without comprehensive retrieval logs, this traceback capability is lost. | Review logging pipeline configuration. Issue test queries and confirm logs contain all required fields (query text, retrieved doc IDs, similarity scores, source, injection scan result). Verify logs are shipped to a SIEM or centralized logging system. Test log completeness under high-throughput conditions. Validate that logs avoid storing raw prompts or full retrieved chunks in cleartext — use tamper-protected logging with restricted access controls per Cisco's vector DB security guidance. | Logging similarity scores and retrieved chunks at scale generates significant data volume. Teams need to balance completeness with storage costs. LlamaIndex and LangChain both support callback-based logging but require custom integration for injection scan results. Consider sampling strategies for Level 2 compliance. RevPRAG (EMNLP 2025) provides a complementary detection layer by analyzing LLM activation patterns during generation — distinct activation signatures emerge when the model processes poisoned vs. clean retrievals, achieving 98% true positive rate with only 1% false positives across multiple benchmark datasets. Schneider's defense architecture (2025) recommends behavioral monitoring with immutable logging and circuit breakers for rapid isolation when compromise is detected. ProvenanceGuard (IEEE TIFS 2026) provides a 120,000-document evaluation dataset across medical (PubMed), legal (CourtListener), and financial (SEC EDGAR) domains for benchmarking retrieval poisoning defenses — useful for validating that logging pipelines actually detect poisoned retrievals. | **8.1.6** | **Verify that** retrieval anomaly detection identifies embedding density outliers, repeated dominance of specific documents in similarity results, and sudden shifts in retrieval bias distribution that may indicate vector database poisoning. | 3 | Sophisticated vector DB poisoning where an attacker injects many semantically similar vectors to dominate retrieval results for target queries, steering model outputs. Maps to MITRE ATLAS AML.T0070 (RAG Poisoning). PoisonedRAG (Zou et al., USENIX Security 2025) demonstrated that just 5 poisoned documents achieve 90% attack success — the attacker creates a "gravity well" in embedding space. The RSB benchmark (Zhang et al., 2025) evaluated 13 poisoning attack methods and found that all 7 defense mechanisms tested failed to provide robust protection, underscoring the difficulty of detection. Future variants could evolve into "vector worms" that instruct models to re-embed and reintroduce poisoned data elsewhere, creating cascading contamination (Prompt Security, 2025). | Review anomaly detection implementation. Verify it monitors: (1) document retrieval frequency distributions, (2) embedding space density metrics per namespace, (3) temporal shifts in which documents are retrieved for stable query sets. Test with synthetic poisoning scenarios — including centroid-targeted injection (Black-Hole Attack pattern) — and confirm alerts fire. The Black-Hole Attack (Li et al., April 2026) showed that vectors injected near the geometric centroid of embedding space exploit "centrality-driven hubness" to appear in up to 99.85% of top-10 results at just 1% poisoning rates; probe-vector detection (querying with random vectors and flagging anomalously frequent neighbors) removes suspicious vectors with minimal legitimate data loss. Behavioral drift detection using runtime analytics to identify sudden tone shifts or stylistic anomalies is a recommended complementary approach. Practical embedding anomaly thresholds validated in testing: 0.85 cosine similarity flags individual document overrides, 0.90 flags batch clustering anomalies — embedding anomaly detection alone reduced knowledge poisoning success from 95% to 20%. Consider hybrid BM25 + vector retrieval as an architectural defense — research shows it reduces gradient-guided poisoning success from 38% to 0%, though joint sparse+dense optimization can partially circumvent it (20–44% success). Evaluate the Normalized Passage Attention Score (NPAS) and Attention-Variance Filter (AV Filter) as lightweight runtime detectors — research shows up to 20% higher accuracy than baseline defenses at flagging anomalous passages. RevPRAG (EMNLP 2025) offers a generation-time complement: analyzing LLM activation patterns during response generation achieves 98% TPR at 1% FPR for detecting poisoned retrievals via a Siamese network trained on final-token activations across all LLM layers. | This is an emerging area with limited production tooling but several notable frameworks have appeared. **RAGuard** (2025) uses a three-component strategy: (1) retrieval expansion (top-N where N > k) to dilute poisoned content, (2) chunk-wise perplexity filtering using non-parametric percentile thresholds to detect unnatural linguistic patterns, and (3) text similarity filtering to flag documents with unusually high query similarity. RAGuard achieves 92–99% detection accuracy across four attack types (prompt injection, general trigger, jamming, PoisonedRAG) with FPR below 6%, and remains robust against GPT-4-paraphrased adaptive attacks (DACC >93%). **RAGForensics** (ACM Web Conference 2025) provides post-incident traceback — iteratively retrieving candidate texts and using LLM-guided evaluation to identify poisoned documents with >97.4% DACC and <2.7% FPR, even against adaptive evasion attempts. Custom implementations typically use statistical process control on retrieval frequency histograms or embedding space density estimation (e.g., k-distance plots). No major vector DB offers anomaly detection natively. CorruptRAG (Jan 2026) showed that single-document attacks can dominate retrieval for targeted queries, making per-query retrieval frequency monitoring essential — if one document consistently appears across unrelated queries, that is a strong poisoning signal. SDAG (Feb 2026) provides an inference-time defense by preventing cross-document interaction via sparse attention. Schneider's four-layer defense (2025) recommends trust-weighted ranking adjusted by provenance metadata and temporal decay at the retrieval layer. The RSB benchmark (13 attacks across 5 QA datasets) provides a standardized evaluation framework for testing defense effectiveness. MM-MEPA (February 2026) showed that metadata-only poisoning in multimodal RAG achieves up to 91% ASR while leaving visual content unaltered — existing defenses including query paraphrasing and image-metadata consistency checks were largely ineffective, highlighting that anomaly detection must monitor metadata distributions alongside embedding space density. ProvenanceGuard (IEEE TIFS 2026) with 120,000 documents across medical, legal, and financial domains and 12,000 adversarial passages covering four attack strategies (targeted factual poisoning, semantic camouflage, ranking manipulation, multi-passage coordinated poisoning) provides the most comprehensive evaluation dataset available for benchmarking anomaly detection pipelines. The Black-Hole Attack (Li et al., April 2026) adds a new attack class: centroid-targeted injection exploiting hubness, where defense options include hubness mitigation (CL2/Z-score normalization, TCPR, noHub — though these sacrifice retrieval quality) and probe-vector detection (querying with synthetic probe vectors and flagging anomalously frequent neighbors). The Key Collision Attack (January 2026, arxiv 2601.23088) targets semantic caching layers by crafting inputs that collide with cached query embeddings, causing stale or poisoned responses to be served without hitting the RAG pipeline at all — teams deploying semantic caching should validate cache key entropy and set TTLs aligned with content freshness requirements. |
