# C2.4 Schema, Type & Length Validation

> **Parent:** [C02: User Input Validation](C02-User-Input-Validation)

## Purpose

AI systems increasingly expose structured interfaces — APIs, tool calls, MCP endpoints, and function-calling interfaces — where malformed or oversized inputs can cause parsing errors, prompt spillage across fields, type confusion, and resource exhaustion. Strict schema enforcement ensures that inputs conform to expected structure before they reach the model or trigger tool execution. This is especially critical for agentic systems where tool calls are generated by the model and must be validated before execution. Schema validation is a deterministic defense layer that does not depend on model behavior.

As of March 2026, structured output and tool-calling have become the dominant interface pattern for production AI systems, making schema validation a frontline control rather than a nice-to-have. The Constrained Decoding Attack (CDA) research published in March 2025 demonstrated that schema constraints themselves can become attack surfaces — `enum` fields in structured generation APIs achieved a 96.2% attack success rate across GPT-4o, Gemini, and open-source models by hiding malicious intent in decoding constraints while keeping visible prompts benign.

The urgency of schema enforcement was underscored in early 2026 when over 30 CVEs were filed against MCP servers in a 60-day window (January–February 2026), with 43% of them being command injection vulnerabilities rooted in missing input validation. A survey of 2,614 MCP implementations found that 82% had file operations vulnerable to path traversal and 67% had some form of code injection risk — confirming that the gap between schema *definition* and schema *enforcement* remains the ecosystem's most exploitable weakness. The OWASP MCP Top 10 (published 2025) formalized this, with MCP03 (Tool Poisoning) specifically calling out schema poisoning as a first-class threat.

As of March 2026, the industry is converging on stricter validation paradigms. OpenAI's Strict Mode (`type: "json_schema"` with `strict: true`) is now the production default for structured output and agent tool calls — the older JSON Mode (which only guaranteed valid JSON syntax, not schema adherence) is considered legacy. GPT-5.2 uses a Context-Free Grammar (CFG) engine to mask invalid tokens before generation, making it structurally impossible to produce a non-conforming response. However, syntactic schema compliance does not guarantee semantic correctness — application-layer validation of values remains essential. Meanwhile, the broader agent ecosystem continues to demonstrate that validation bypass is the dominant attack pattern: CVE-2026-2256 (ModelScope MS-Agent) showed that regex-based command blacklisting is trivially bypassable through shell parsing semantics, and CVE-2026-22708 (Cursor AI) proved that even allowlist-based validation fails when shell built-in commands are implicitly trusted and can poison the execution environment.

## Implementation Maturity

| Area | Maturity | Notes |
|------|----------|-------|
| JSON Schema validation for REST APIs | **High** | Well-established libraries (Pydantic v2, Zod v4, ajv) with broad adoption. |
| Schema enforcement for LLM tool calls | **Medium–High** | Improving rapidly as of March 2026. OpenAI's Strict Mode with CFG token masking guarantees syntactic conformance for structured outputs and tool calls. Anthropic and Google provide similar server-side schema enforcement. However, runtime validation of *semantics* (value ranges, business logic) remains the application's responsibility, and many frameworks still skip it. |
| MCP input schema enforcement | **Low** | Despite the MCP spec (2025-11-25) defining `inputSchema`, enforcement remains critically absent in practice. A 2026 survey of 2,614 MCP implementations found 82% vulnerable to path traversal and 67% with code injection risk. | Token/byte limit enforcement | **Medium** | tiktoken (v0.12.0) and SentencePiece are mature for counting, but consistent enforcement across multimodal inputs remains challenging. |
| MIME type validation by content inspection | **High** | python-magic and file-type are stable and widely used. |
| ReDoS-safe validation | **Medium** | Google RE2 provides guaranteed linear-time matching, but adoption requires rewriting regex patterns that use backreferences or lookaheads. regexploit enables proactive scanning. | Structured validation logging | **Low–Medium** | Most AI frameworks still lack built-in structured validation failure logging, but the landscape is improving. | Agent command validation | **Low** | CVE-2026-2256 and CVE-2026-22708 show that regex-based blacklists and static allowlists are both insufficient for agent shell command validation. Parameterized invocation and sandboxed execution are the emerging standard, but adoption is early. |

## Open Research Questions

- How should schema validation be applied to free-form text inputs that inherently lack structure? The OWASP five-layer defense model treats schema validation as layer 1 but acknowledges that unstructured text fundamentally resists schema-based controls.
- The Constrained Decoding Attack (March 2025) shows that output schema constraints can bypass safety alignment — should schema definitions themselves be treated as security-sensitive configuration that requires review and access control?
- Should validation schemas be shared between the model's tool definition and the runtime validation layer, or should they be independent? Shared schemas reduce drift but create a single point of failure; independent schemas add maintenance burden but provide defense-in-depth.
- How do multimodal schemas (combining text, image, and audio constraints in a single request) evolve as models gain more modalities? Token-equivalent calculations vary significantly across providers and modalities.
- What verification approaches can detect tool shadowing attacks where cross-tool reasoning manipulation bypasses per-tool schema validation?
- LLM-generated regexes frequently contain ReDoS vulnerabilities (ICPC 2024 research) — should AI-assisted development workflows include mandatory static regex analysis as a CI/CD gate?
- The 2026 MCP CVE wave showed that allowlist/blocklist approaches to argument validation are consistently bypassable (CVE-2026-30861). Should the security community adopt a "no string interpolation into shell/SQL/command contexts from MCP tool arguments" principle as an absolute rule, analogous to parameterized queries for SQL injection?
- JSON Schema's remote `$ref` resolution created an entire vulnerability class (IDEsaster, CVE-2025-49150). Should AI-facing JSON Schema implementations disable remote schema fetching by default and require explicit opt-in for trusted registries?
- OpenAI's Strict Mode guarantees syntactic schema conformance via CFG token masking, but semantic validation remains an application responsibility. Should LLM providers expose a semantic validation callback layer alongside structured outputs, or does that create new attack surface?
- CVE-2026-22708 showed that shell built-in commands are invisible to allowlist-based validation in AI coding agents. Should the industry converge on a "no direct shell access" principle for AI agents, requiring all system interaction through parameterized APIs with explicit permission grants?
- With 36.7% of MCP servers vulnerable to SSRF through URL-type string arguments, should the MCP specification define a `format: "safe-uri"` type that mandates runtime SSRF protections (scheme allowlisting, private IP blocking, DNS rebinding defense) rather than leaving this to individual implementations?
