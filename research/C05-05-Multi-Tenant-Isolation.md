# C5.5: Multi-Tenant Isolation

> [Back to C05 Index](C05-Access-Control.md)
> **Last Researched:** 2026-04-15

## Purpose

Ensure logical and cryptographic isolation between tenants in shared AI infrastructure. Multi-tenant AI deployments share GPU clusters, model-serving endpoints, vector databases, and inference caches across tenants. Without rigorous isolation, cross-tenant data leakage can occur through shared KV-caches, embedding stores, network segments, and even GPU memory residuals. Recent research has demonstrated that timing side channels in shared inference infrastructure can reconstruct other tenants' prompts with 95-99% accuracy, making KV-cache isolation a critical and rapidly evolving concern. As of April 2026, new attack techniques like RL-enhanced prompt leakage (RL2eak, OptiLeak) have further reduced the cost and effort of exploiting shared caches, container escape vulnerabilities in NVIDIA Container Toolkit (CVE-2024-0132, CVE-2025-23266) have underscored the systemic fragility of GPU isolation in shared environments, and CPU-level cache side channels (Spill The Beans, arxiv 2505.00817) have shown that token-level leakage is possible even without GPU-side cache sharing. On the defensive side, NVIDIA Blackwell B200 confidential computing now provides encrypted NVLink/NVSwitch GPU-to-GPU communication with near-native inference performance, and multiple new KV-cache defense frameworks (entropy-based obfuscation, CacheSolidarity, KV-Cloak) offer layered protection against timing side channels. At KubeCon Europe 2026, NVIDIA announced deep Kata Containers integration with DRA (Dynamic Resource Allocation) and CDI for GPU-aware confidential containers, providing a "glide path" from VM-level isolation to full confidential computing with encrypted and measured VM memory. Enterprise KV-cache management is also maturing: LMCache has emerged as a widely-adopted open-source KV caching layer (integrated with vLLM, SGLang, Red Hat, CoreWeave, Google Cloud) that supports cache offloading and prefill-decode disaggregation at scale -- though its tenant isolation story still depends on the underlying engine's salt-based partitioning.

> **Scope note:** Controls in this section govern infrastructure-level tenant isolation -- network policies, namespace segregation, per-tenant encryption, and KV-cache partitioning. Query-time tenant filtering is in C5.3; authorization policy is in C5.2.

For AI agent workloads specifically, MicroVM-based isolation (Firecracker, Kata Containers) ensures each tenant's agent runs in its own kernel with hardware virtualization (Intel VT-x, AMD-V), closing the shared-kernel attack surface that container-based multi-tenancy leaves open. |

TEE-I/O extends hardware trust boundaries beyond the CPU, eliminating PCIe bottlenecks that constrained earlier platforms. Weaviate collections ensure embeddings are never co-mingled across tenants. Qdrant offers three levels of isolation: payload-based filtering within a single collection, custom sharding per tenant, or dedicated collections per tenant. Milvus supports database-level, collection-level, and partition-level multi-tenancy strategies. Turbopuffer provides CMEK and HIPAA BAA even on non-enterprise plans. For Kubernetes deployments, the GPU Operator now supports RuntimeClass selection per pod -- operators can transparently choose bare-metal containers, Kata Containers (VM-level isolation), or full Confidential Containers depending on tenant sensitivity tier. GPU topology handling in Kata preserves NVLink peer-to-peer transfers and NUMA affinity even inside VM boundaries, so multi-GPU training and inference workloads maintain near-native performance. Weaviate's multi-tenant architecture now features automatic tenant activity tracking and resource offloading for inactive tenants (hot/warm/cold/frozen states), reducing the attack surface of dormant tenant data while cutting storage costs. Qdrant's custom sharding with dedicated shard-per-tenant provides the strongest isolation short of separate collections, with independent scaling and no cross-tenant query interference. |

HSM-backed key management is recommended for high-sensitivity deployments. **[2026-04]** Cryptographic erasure is increasingly recognized as the primary tenant offboarding mechanism for cloud AI workloads. GCP transitioned to cryptographic-erasure-only media sanitization in November 2025, acknowledging that traditional overwriting is impractical at modern storage scale. The EU Data Protection Board (formerly Article 29 Working Party) has acknowledged cryptographic erasure as a valid deletion method under GDPR. For AI-specific data, per-tenant encryption must extend to embeddings, vector indices, KV-cache snapshots, and fine-tuned model weights. NVIDIA Blackwell B200 confidential computing provides hardware-anchored AES-GCM 256 encryption for GPU memory with purpose-built encryption engines (replacing repurposed general-purpose logic from prior generations) and near-zero performance overhead at production scale. Pinecone's SOC 2, HIPAA, ISO 27001, and GDPR compliance certifications now cover its namespace-level encryption, while Qdrant Cloud offers SOC 2 and HIPAA compliance with per-collection encryption. | 5.5.5 | **Verify that** inference-time KV-cache entries are partitioned by authenticated session or tenant identity and that automatic prefix caching does not share cached prefixes across distinct security principals, to prevent timing-based prompt reconstruction attacks. | 2 | Timing-based prompt reconstruction via KV-cache sharing side channels; the PROMPTPEEK attack (NDSS 2025) achieves 95-99% accuracy reconstructing other tenants' prompts by probing cache hit/miss latency differences in vLLM and SGLang. Direct KV-cache content extraction enables collision and injection attacks that reverse-engineer user inputs from plaintext cache entries (Shadow in the Cache, arxiv 2508.09442). Cross-tenant prefix reuse leaks query patterns even when outputs are properly isolated. **[2026-04]** RL-enhanced attacks now lower the cost of exploitation: RL2eak uses reinforcement learning with DPO to identify "hard tokens" (sensitive domain-specific terms), achieving 12.48x reduction in token requests needed for prompt reconstruction on domain-specific datasets. OptiLeak (arxiv 2602.20595, Feb 2026) similarly uses RL for efficient prompt reconstruction. | Confirm inference framework uses per-tenant or per-session cache salting (e.g., vLLM `cache_salt` parameter, merged May 2025); measure response latency variance across probe sequences to detect cache-hit timing leakage; verify automatic prefix caching does not share prefixes across tenant boundaries; inspect KV-cache storage to confirm entries are not persisted in plaintext across sessions; test that expired or terminated sessions have their cache entries evicted. | **[2026-04]** Active research area with rapidly maturing defenses. vLLM merged cache salting (RFC #16016, May 2025) for single-barrier tenant isolation; hierarchical multi-barrier design (per-message salts) is in active development. SafeKV (arxiv 2508.08438) integrates ChunkGuard for real-time sensitivity classification with selective cache sharing -- mitigates 94-97% of timing side-channels while reducing TTFT overhead by up to 40% vs. full isolation and raising throughput up to 2.66x. KV-Cloak (arxiv 2508.09442) provides reversible matrix obfuscation of KV-cache contents with ~5% latency overhead and negligible accuracy impact. CacheSolidarity (arxiv 2603.10726, March 2026) takes a selective isolation approach: monitors cross-user cache reuse patterns, flags anomalous sharing, and selectively isolates prefixes only when threats are detected -- achieves 70% higher cache reuse and 30% lower latency vs. full user isolation while blocking side-channel attacks. Entropy-based obfuscation (IJISRT, September 2025) injects controlled randomness into cached attention key-value states using Shannon and Rényi entropy as guiding metrics, dynamically modulating perturbation levels to balance privacy and performance -- effective at reducing prompt leakage with relatively small computational cost. Beyond GPU-level caches, Spill The Beans (arxiv 2505.00817, May 2025) demonstrates that CPU cache side channels can leak individual tokens from LLM inference even without shared GPU memory, expanding the attack surface to include the host processor. No production framework yet combines all five defensive approaches (salting + selective sharing + content obfuscation + anomaly-based isolation + entropy perturbation); organizations should evaluate their threat model to choose the right tradeoff between isolation strength and performance. **[2026-04]** Shadow in the Cache (NDSS 2026, February 2026) presented the first comprehensive taxonomy of KV-cache attack vectors -- direct inversion attacks that reconstruct prompts from cache entries, collision attacks that identify matching prefixes, and semantic injection attacks that manipulate cached context. KV-Cloak's reversible matrix obfuscation with operator fusion was formally validated at the same venue. LMCache has become the de facto open-source KV caching layer for enterprise inference (integrated with vLLM, SGLang, Google Cloud, CoreWeave, Redis, PliOps), delivering up to 15x throughput improvement and 2x latency reduction via cache offloading and prefill-decode disaggregation. However, LMCache's tenant isolation depends entirely on the underlying engine's cache_salt implementation -- it does not add its own isolation layer, so organizations must verify that salt-based partitioning is active end-to-end when deploying LMCache in multi-tenant configurations. FlexServe (arxiv 2603.09046, March 2026) introduces TrustZone-based Flexible Resource Isolation for on-device inference, creating Flexible Secure Memory (Flex-Mem) to protect model weights, KV caches, and intermediate tensors with a lightweight secure hypervisor, achieving 10x TTFT improvement over naive TEE approaches -- relevant for edge multi-tenant scenarios where GPU TEEs are unavailable. |
| **5.5.6** | **Verify that** per-tenant memory spaces, embedding stores, cache entries, and temporary files are securely purged on tenant deletion or session termination. | 2 | Residual tenant data persisting in GPU memory, KV-caches, embedding stores, vector indices, and temporary processing artifacts after tenant offboarding or session termination. Embeddings and vector representations derived from sensitive data may retain recoverable information even after source data deletion. Fine-tuned model weights can memorize training data, creating a persistent leakage vector if model artifacts are not purged per-tenant. | Trigger tenant deletion and verify complete purge across: GPU memory, KV-cache entries, embedding store namespaces, vector indices, temporary files, and model artifacts. Confirm cryptographic key destruction via KMS audit logs. Verify tombstone-based deletion workflow (immediate access block followed by background purge). Check that derived data (embeddings, cached inference results) is linked to source record lifecycle and deleted accordingly. Rebuild vector indices post-deletion and verify document counts match expectations. | **[2026-04]** Secure tenant purging in AI systems requires coordinated deletion across multiple data stores that traditional approaches do not cover. CSA's "Data Security within AI Environments" (2025) recommends DSP-02 (secure disposal) and DSP-16 (retention and deletion) controls, with time-bound deletion (TTL) policies for temporary artifacts and session-scoped tokens that expire automatically. Cryptographic erasure via per-tenant key destruction (AWS KMS, Azure Key Vault, GCP Cloud KMS, HashiCorp Vault) is the most practical approach for bulk tenant data removal -- destroying the encryption key renders all protected data irrecoverable. For embeddings specifically, prefer tokenizing PII before creating embeddings so vector stores never memorize sensitive data; use standardized placeholders that do not distort similarity search. Vector database purging should include full namespace deletion (Pinecone) or collection removal (Weaviate) followed by index rebuild and checksum verification. GPU memory zeroing between sessions is a baseline but does not cover persistent caches; organizations should implement explicit KV-cache eviction on session termination. No unified tooling exists for cross-stack tenant purging across GPU memory, vector databases, model registries, and object storage simultaneously -- this remains a manual orchestration challenge in most deployments. Qdrant supports tenant-scoped deletion via collection removal or payload-filtered point deletion with custom shard cleanup. Milvus partition-level multi-tenancy enables targeted partition drops for clean tenant offboarding without affecting other tenants' data. For defense-in-depth, combine cryptographic erasure (per-tenant key destruction) with explicit data deletion across all stores, and verify with post-deletion index rebuilds and document count reconciliation. **[2026-04]** Weaviate's tenant lifecycle management now supports automatic state transitions (hot → warm → cold → frozen) that progressively offload inactive tenant data to cheaper storage tiers and reduce the live attack surface. When a tenant is deleted, Weaviate removes the entire tenant shard including all vectors, inverted indices, and metadata. For Kata Containers / Confidential Containers deployments, tenant session termination destroys the entire lightweight VM, taking its kernel, memory pages, and any ephemeral filesystems with it -- a stronger purge guarantee than container cleanup alone. In LMCache-backed deployments, KV-cache entries persisted to external storage (Redis, Ceph, PliOps) must be explicitly evicted on tenant offboarding; relying on TTL-based expiration alone is insufficient because cached prefixes may contain reconstructible prompt fragments (per Shadow in the Cache findings). Organizations should implement tombstone-based deletion workflows: immediately block access on tenant deletion request, then asynchronously purge across all cache layers, vector stores, model registries, and object storage, with post-purge verification via index rebuilds and record count reconciliation. |
