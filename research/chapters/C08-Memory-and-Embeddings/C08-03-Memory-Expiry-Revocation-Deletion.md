# C8.3: Memory Expiry, Revocation & Deletion

> [Back to C08 Index](C08-Memory-and-Embeddings.md)
> **Last Researched:** 2026-06-10

## Purpose

Retention and revocation for memory and RAG systems must be measurable operations, not policy statements. A delete, expiry, quarantine, or reset request has only taken effect when the content is excluded from every retrieval path that can influence a response: dense vectors, sparse or keyword indexes, rerankers, graph memories, tool caches, application caches, backups, and restored snapshots. For systems with long-term or cross-session memory, deletion is also a security boundary because poisoned entries can persist after the initiating interaction has ended.

As of 10 June 2026, the practical control pattern is query-time exclusion first, physical reclamation second, and audit evidence throughout. Pinecone documents log sequence numbers for checking whether deletes are reflected in serverless query responses, and the namespace creation API went generally available on 26 March 2026, giving teams a stable primitive for per-tenant isolation and bulk deletion: https://docs.pinecone.io/guides/index-data/check-data-freshness and https://docs.pinecone.io/release-notes/2026 Milvus TTL excludes expired entities from query results before later compaction, Qdrant exposes `wait` and `ordering` controls on point deletion, and Weaviate v1.36 promoted object TTL to general availability with the optional `filterExpiredObjects` query exclusion setting. These vendor semantics are useful, but they do not replace application-level deletion orchestration across hybrid search, caches, evaluation copies, exported files, and backup media.

Regulatory pressure is also concrete. The EDPB's 18 February 2026 right-to-erasure report found recurring problems with internal procedures, ineffective anonymisation used instead of deletion, uncertain retention periods, and backup deletion limits, drawing on 764 controllers across 32 DPAs: https://www.edpb.europa.eu/system/files/2026-02/edpb_cef-report_2025_right-to-erasure_en.pdf The EU AI Act requires high-risk systems to support automatic event logging under Article 12 and requires providers to retain covered logs for at least six months under Article 19. The Council and Parliament reached provisional agreement on the Digital Omnibus on 7 May 2026, deferring stand-alone Annex III high-risk obligations to 2 December 2027 and Annex I embedded-product obligations to 2 August 2028, with formal adoption expected before the original 2 August 2026 deadline — so teams need a written conflict-resolution model for evidence retention, incident response, and privacy erasure rather than ad hoc deletion exceptions: https://www.gibsondunn.com/eu-ai-act-omnibus-agreement-postponed-high-risk-deadlines-and-other-key-changes/ and https://www.helpnetsecurity.com/2026/04/16/eu-ai-act-logging-requirements/

The OWASP Top 10 for Agentic Applications 2026, published 9 December 2025, formalises Memory & Context Poisoning as `ASI06`, recommending memory segmentation per tenant, expiry of unverified data, and provenance tracking as core mitigations. This raises the bar for evidence: an auditor reviewing a memory deletion path expects logs that prove which path was excluded, when, by whom, and that the change survived restart, replica failover, and backup restore: https://genai.owasp.org/2025/12/09/owasp-top-10-for-agentic-applications-the-benchmark-for-agentic-security-in-the-age-of-autonomous-ai/

## Recent Research Notes

- Hidden in Memory (submitted 14 May 2026) demonstrates sleeper memory poisoning: corrupted external content (documents, webpages, repositories) plants false memories in stateful assistants that stay dormant and re-emerge in later conversations. Reported memory-injection success reached up to 99.8% on one tested frontier model, and successfully retrieved poisoned memories triggered attacker-intended actions in 60-89% of cases — a strong argument for treating memory reset as a rehearsed incident-response operation rather than a rare manual cleanup: https://arxiv.org/abs/2605.15338
- From Untrusted Input to Trusted Memory (3 June 2026) systematises the field: four memory write channels, nine structural vulnerabilities, a six-class attack taxonomy, and the MPBench benchmark. Two findings matter for this section — agents tuned for aggressive memory writing and retrieval are measurably more vulnerable, and existing prompt-injection defenses do not cover memory poisoning, so write-path screening and deletion/reset controls have to be separate investments: https://arxiv.org/abs/2606.04329
- Spring AI disclosed five CVEs on 27 April 2026 that land directly on deletion and quarantine plumbing. CVE-2026-40967 (CVSS 8.6) covers `FilterExpressionConverter` injection — PgVectorStore, OracleVectorStore, and CouchbaseSearchVectorStore concatenated filter output into SQL without parameterization, enabling tenant-isolation bypass and deletion of arbitrary rows — and CVE-2026-40978 is SQL injection in `CosmosDBVectorStore.doDelete()`. Any quarantine design built on metadata filters inherits the injection posture of the filter layer itself: https://spring.io/security/cve-2026-40967/ and https://spring.io/security/cve-2026-40978/
- Microsoft reported AI Recommendation Poisoning on 10 February 2026: more than 50 prompt-based attempts from 31 companies tried to plant persistent recommendations through pre-filled AI-assistant URLs. Microsoft maps the activity to MITRE ATLAS `AML.T0051` and `AML.T0080.000`, making memory reset and review a practical incident-response requirement, not only a privacy feature: https://www.microsoft.com/en-us/security/blog/2026/02/10/ai-recommendation-poisoning/
- MINJA, revised 12 February 2026, shows query-only memory injection against LLM agents without direct access to the memory bank; the January 2026 follow-up found that realistic memory state and trust-aware sanitization reduce attack effectiveness but require careful threshold tuning: https://arxiv.org/abs/2503.03704 and https://arxiv.org/abs/2601.05504
- InjecMEM, submitted to ICLR 2026 and modified 11 February 2026, shows that a single interaction can plant a targeted memory record that later steers related queries, including an indirect path where a compromised tool writes the poisoned memory: https://openreview.net/forum?id=QVX6hcJ2um
- Zombie Agents, revised 5 March 2026, shows self-evolving agents can store attacker-controlled web content in long-term memory and later trigger unauthorized tool behavior; the paper is a useful reminder that per-session prompt filtering does not clear persistent memory state: https://arxiv.org/abs/2602.15654
- Poison Once, Exploit Forever, revised 7 April 2026, introduces eTAMP, where ordinary web observations poison an agent's trajectory memory and activate on a different site or task without direct memory-store access. The reported experiments reached 32.5% attack success on one tested model and showed environmental stress can increase susceptibility: https://arxiv.org/abs/2604.02623
- MemoryGraft, published 18 December 2025, demonstrates persistent compromise through poisoned experience retrieval where small sets of malicious procedure memories can influence later tasks through combined lexical and embedding retrieval: https://arxiv.org/abs/2512.16962
- Semantic Chameleon, submitted 10 March 2026 and scheduled for AISec '26, shows gradient-guided dual-document poisoning (sleeper plus trigger) and demonstrates that hybrid retrieval can collapse one tested attack from 38% success to 0% on a Security Stack Exchange corpus without retraining the LLM or retriever — a useful complement to deletion controls because hybrid retrieval also limits how long a missed poisoned chunk can keep influencing answers: https://arxiv.org/abs/2603.18034
- RAGForensics, published at The Web Conference 2025, gives responders a practical reason to retain quarantined material under restricted access: traceback needs the suspect chunks to remain available for analysis while they are excluded from production retrieval: https://papersgraph.com/research-papers/traceback-of-poisoning-attacks-to-retrieval-augmented-generation
- OWASP LLM08:2025 treats vector and embedding weaknesses as a RAG security risk class covering unauthorized access, cross-context leakage, embedding inversion, poisoning, and retrieval behavior alteration. The same page recommends fine-grained access controls, data validation, classification, and immutable retrieval logs: https://genai.owasp.org/llmrisk/llm082025-vector-and-embedding-weaknesses/
- OWASP Top 10 for Agentic Applications 2026 added `ASI06: Memory & Context Poisoning`, and OWASP Agent Memory Guard has moved quickly from incubator stub to usable middleware: v0.3.0 (10 June 2026) ships an MCP server, CLI scanner, ML-based detection, and a GitHub Action alongside the existing SHA-256 integrity baselines, declarative YAML policies (allow/redact/quarantine/block), and snapshot capture with rollback. Integrations now cover LangChain, the OpenAI Agents SDK, AutoGen, and mem0, with vector-store protection slated for v0.4.0 and a v1.0.0 target in Q4 2026. Help Net Security's 1 June 2026 write-up reports 92.5% recall and 100% precision at 59-microsecond median latency across the project's 55-case test suite — promising numbers, though from the project's own benchmark: https://github.com/OWASP/www-project-agent-memory-guard and https://www.helpnetsecurity.com/2026/06/01/owasp-agent-memory-guard/
- Weaviate TTL is useful but easy to misconfigure: expired objects are removed by background processing, `OBJECTS_TTL_DELETE_SCHEDULE` must be set, and `filterExpiredObjects` should be enabled when expired-but-not-yet-deleted objects must be hidden from queries. In multi-tenant collections, TTL deletion is skipped for inactive or offloaded tenants and only runs when the tenant becomes active again, so quarantine workflows that suspend tenants must plan for delayed cleanup: https://docs.weaviate.io/weaviate/manage-collections/time-to-live and https://docs.weaviate.io/weaviate/manage-collections/multi-tenancy
- Physical deletion evidence is storage-engine specific. OpenSearch marks deleted documents until segment merge expunges them, and on native k-NN engines the vector data structures are rebuilt rather than merged when segments are combined, so a delete-heavy index needs `_forcemerge` with `only_expunge_deletes=true` in a non-write window to actually reclaim vector memory: https://docs.opensearch.org/latest/api-reference/index-apis/force-merge/ pgvector deployments should pair delete-heavy HNSW tables with `REINDEX INDEX CONCURRENTLY` before `VACUUM` because the three-pass HNSW vacuum (RemoveHeapTids, RepairGraph, MarkDeleted) leaves dead-but-traversable nodes that slow search until the graph is rebuilt: https://github.com/pgvector/pgvector and https://deepwiki.com/pgvector/pgvector/7.2-index-maintenance-and-vacuum
- NIST AI 600-1 connects memory-retention engineering to governance: keep TEVV and transparency history, plan decommissioning around data retention and leakage, and define collection/retention/minimum-quality policies for sensitive data: https://nvlpubs.nist.gov/nistpubs/ai/NIST.AI.600-1.pdf
- The EDPB's 2026 right-to-erasure findings and the EU AI Act's official Articles 12 and 19 text should be read together when setting memory retention policy: erasure must work in practice, but security and high-risk-system logs may have minimum retention duties: https://www.edpb.europa.eu/our-work-tools/our-documents/other/coordinated-enforcement-action-implementation-right-erasure_en and https://eur-lex.europa.eu/eli/reg/2024/1689/oj?locale=en
- The technical standard that will define what "adequate" Article 12 logging looks like is close but not finished: ISO/IEC 24970 (AI system logging) reached FDIS stage as FprEN ISO/IEC FDIS 24970, while the European prEN 18229-1 (logging and human oversight under JTC 21) is at Enquiry stage. Teams designing quarantine and deletion audit logs now should track both drafts rather than invent a third schema: https://www.iso.org/standard/88723.html and https://adamleonsmith.substack.com/p/two-standards-one-architecture-fpren
- Litigation hold is a third force alongside erasure and AI-Act retention: the May 2025 US federal preservation order forced OpenAI to retain deleted and temporary ChatGPT conversations for ordinary consumer tiers until it was terminated on 9 October 2025. Memory deletion runbooks should state explicitly how a legal hold suspends expiry, purge, and reset operations — and how held material is segregated from production retrieval in the meantime: https://openai.com/index/response-to-nyt-data-demands/

## Notable Incidents

- **Slack AI RAG exfiltration (August 2024, catalogued as MITRE ATLAS `AML.CS0035`)**: Researchers showed that hidden prompts in Slack messages could cause Slack AI to weave attacker-controlled links into answers and exfiltrate private-channel content. The case study remains a canonical reference for why retrieval indexes need both query-time filtering and deletion paths that cover archived support material and pasted secrets: https://www.startupdefense.io/mitre-atlas-case-studies/aml-cs0035-data-exfiltration-from-slack-ai-via-indirect-prompt-injection
- **Gemini long-term memory poisoning (Embrace The Red, 2025; thehackernews follow-up January 2026)**: Indirect prompt injection through a summarised document plus delayed tool invocation persisted attacker content into Gemini's long-term memory, surviving sessions. The 2026 calendar-invite variant showed the same memory pathway can be reached through pre-filled invite text, which is the same pattern Microsoft later documented as AI Recommendation Poisoning: https://embracethered.com/blog/posts/2025/gemini-memory-persistence-prompt-injection/ and https://thehackernews.com/2026/01/google-gemini-prompt-injection-flaw.html
- **AI Recommendation Poisoning (Microsoft, 10 February 2026)**: 50+ prompt-based attempts from 31 companies used pre-filled assistant URLs to plant persistent "trusted source" memories, mapped to `AML.T0051` and `AML.T0080.000` and tied to a recommended hunt for prompts containing `remember`, `trusted source`, `authoritative`, and `future conversations`: https://www.microsoft.com/en-us/security/blog/2026/02/10/ai-recommendation-poisoning/

## Implementation Maturity

- **Logical exclusion via vendor delete and TTL APIs**: *Mature.* Pinecone, Milvus, Qdrant, Weaviate, OpenSearch, and pgvector all provide explicit delete, TTL, or filter primitives, and they document how to verify them (LSN headers, `wait`/`ordering`, TTL exclusion flags, `docs.deleted`, `REINDEX`/`VACUUM`).
- **Cross-path quarantine and reset state machines**: *Emerging.* Most engines provide deletes and filters, but a complete quarantine→purge→reset state machine that spans dense, sparse, reranker, cache, agent-memory, and backup paths is still application code. OWASP Agent Memory Guard reached v0.3.0 on 10 June 2026 (MCP server, CLI scanner, ML detection, GitHub Action, quarantine policy action) and covers agent memory well, but vector-store protection is not due until v0.4.0 and v1.0.0 remains a Q4 2026 target.
- **Memory-poisoning forensics**: *Emerging.* RAGForensics-style traceback is being published in academic venues, OWASP Agent Memory Guard adds SHA-256 integrity and snapshot rollback, MPBench (June 2026) gives responders a shared attack taxonomy and benchmark, and Microsoft has published one practical hunt query template — but there is no widely adopted SIEM content pack for memory-poisoning indicators.
- **Backup and restore hygiene for vector data**: *Emerging.* Qdrant snapshots, Milvus backups, Weaviate backups, OpenSearch snapshots, and pgvector pg_dump all need explicit checks to confirm that restoring a snapshot does not silently rejoin deleted or quarantined content; this remains team-by-team operational work.
- **Regulatory reconciliation between erasure and AI-Act log retention**: *Emerging.* The EDPB's February 2026 enforcement report shows recurring failures in real deletion, and the EU AI Act's August 2026 enforcement deadline is forcing teams to write down the conflict resolution model rather than improvise it.

## Audit Checks to Add to Test Fixtures

- Expiry fixtures should create a canary document, vector, lexical record, reranker output, semantic-cache entry, and application-cache entry, then prove each path returns zero results inside the stated propagation SLO.
- Reset fixtures should seed a benign memory and a poisoned memory, exercise quarantine, selective purge, and full reset independently, and confirm the poisoned trigger fails after session restart and memory compaction.
- Quarantine fixtures should keep the suspect content visible only to a restricted forensic interface while proving dense, lexical, hybrid, cached, and agent-memory retrieval paths all enforce the same exclusion state.
- Filter-integrity fixtures should fuzz the quarantine and deletion filter layer with injection payloads in metadata keys, values, and document IDs — the April 2026 Spring AI CVEs showed filter-expression and delete-path injection reaching SQL in PgVector, Oracle, Couchbase, and Cosmos DB backends — and confirm a crafted filter can neither reveal quarantined content nor delete records outside its tenant scope.
- Storage fixtures should record the vendor-specific evidence used to distinguish logical exclusion from physical reclamation, such as Pinecone LSNs and per-namespace counts, Qdrant `wait` behavior, Milvus TTL/compaction timing (controlled by `dataCoord.compaction.expiry.tolerance`), Weaviate TTL cleanup with `filterExpiredObjects` and per-tenant activation, OpenSearch deleted-document counts with `only_expunge_deletes` force merges, and pgvector `REINDEX`/`VACUUM` maintenance.
- Backup fixtures should restore a recent snapshot into an isolated test environment and verify that expired, deleted, or quarantined records do not silently rejoin production retrieval when the restored index is promoted.
- Governance fixtures should document the reviewer, legal basis, retention exception, and purge decision whenever security quarantine conflicts with privacy erasure or high-risk-system log-retention duties.
- Memory-poisoning hunt fixtures should include the Microsoft and Christian Schneider indicator set (assistant-URL prefills containing `remember`, `trusted source`, `authoritative`, `future conversations`, `cite`) and replay the indirect-tool path that InjecMEM and Zombie Agents describe, then confirm the reset operation prevents the trigger query from retrieving the poisoned record.
