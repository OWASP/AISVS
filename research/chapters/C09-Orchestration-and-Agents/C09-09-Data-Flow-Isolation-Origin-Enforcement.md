# C9.9: Architectural Data-Flow Isolation and Origin Enforcement

[Back to C09 Index](C09-Orchestration-and-Agents.md)

## Purpose

Prevent data-flow attacks that exploit agentic tool-calling pipelines by manipulating tool arguments without altering the control flow, through architectural separation of planning from untrusted data processing and origin-aware policy enforcement at the tool-call boundary. These controls complement C2.1 (input-level filtering) and C9.7 (intent verification gates), which do not address attacks where the correct tools are called in the correct sequence but with attacker-controlled arguments derived from untrusted data. Any isolation mechanism that achieves the stated outcome satisfies the requirement.

---

## 2024-2026 Research Context

Indirect prompt injection is no longer only a prompt-filtering problem. The hard case for agent systems is that untrusted content can influence a later tool argument while the overall plan still looks legitimate. OWASP LLM01:2025 calls out external-content prompt injection, and OWASP LLM06:2025 ties the impact to excessive agency: functions, permissions, and autonomy exposed to manipulated model outputs. MITRE ATLAS techniques relevant to this section include AML.T0051.001 (indirect prompt injection), AML.T0053 (AI agent tool invocation), and AML.T0086 (exfiltration via AI agent tool invocation).

The most directly relevant academic work is [AgentDojo](https://arxiv.org/abs/2406.13352), which evaluates agents operating over untrusted data and external tools, and [CaMeL](https://arxiv.org/abs/2503.18813), which defends by extracting control flow from the trusted user request, treating retrieved or tool-returned content as data, and enforcing capabilities on data flows before tool calls. CaMeL is a research artifact rather than a drop-in production control, but it is useful as a reference architecture: do not let untrusted data decide which tools run, and do not let data with the wrong provenance cross into sensitive tool arguments.

Several 2026 papers are pushing this from prompt filtering toward runtime evidence. [AdapTools](https://arxiv.org/abs/2602.20720) shows why static prompt-injection test strings are insufficient: adaptive attacks can choose stealthier tools and vary payloads to evade task-relevance checks. [AttriGuard](https://arxiv.org/abs/2603.10749) frames the key question as causal attribution: would the same tool call still be justified if the untrusted observation were control-attenuated? [ClawGuard](https://arxiv.org/abs/2604.11790), last revised on May 11, 2026, takes a boundary-enforcement angle by deriving task-specific constraints before external tool use and checking later tool calls against that rule set. [LogJack](https://arxiv.org/abs/2604.15368) shows why this matters operationally: cloud logs can become the indirect prompt-injection channel for debugging agents that later execute remediation commands. These are not mature production standards, but they give auditors concrete test patterns for "same plan, unsafe arguments" failures.

On April 30, 2026, the joint NSA/CISA/Five Eyes guidance [Careful Adoption of Agentic AI Services](https://media.defense.gov/2026/Apr/30/2003922823/-1/-1/0/CAREFUL%20ADOPTION%20OF%20AGENTIC%20AI%20SERVICES_FINAL.PDF) reinforced the same architectural theme from an operational risk angle. The guidance highlights privilege, design/configuration, structural, behavior, and accountability risks in agentic systems, including stale authorization checks, weak segmentation, excessive privileges, and audit logs that appear legitimate after a trusted agent is misused. For C9.9, the practical takeaway is to evaluate origin-aware authorization at each tool invocation, bind agent identities and credentials tightly to intended duties, and preserve enough dependency evidence to explain why a tool call happened after the fact.

The December 2025 [OWASP Top 10 for Agentic Applications 2026](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/) and OWASP's April 2026 exploit round-up make the same point in incident language: agent goal hijack, tool misuse, identity abuse, and human-agent trust exploitation often show up as ordinary-looking actions with bad data provenance. C9.9 therefore treats policy evaluation as a per-invocation data-flow decision, not only a prompt-safety or workflow-design review.

Production implementations usually need several layers working together: explicit orchestration roles for planners versus extractors, policy-as-code at the tool boundary, origin metadata on every value that can reach a tool argument, and durable telemetry that lets reviewers reconstruct which data sources influenced each invocation. As of May 2026, frameworks such as LangChain provide useful tool approval middleware, Pydantic AI supports deferred approval/external-execution patterns, the OpenAI Agents SDK exposes tool guardrails and tracing hooks for function-tool calls, and policy engines such as OPA/Rego or Cedar can enforce machine-readable decisions. Complete per-value provenance and taint-aware tool execution still usually require custom wrapper code around the agent runtime.

MCP deployments make the same data-flow issue concrete. OWASP's [MCP Tool Poisoning](https://owasp.org/www-community/attacks/MCP_Tool_Poisoning) write-up describes a runtime trust gap where a tool response enters the model context as trusted material, and [MCPTox](https://arxiv.org/abs/2508.14925) measured tool-metadata poisoning across real MCP servers and agents. AgentSeal's March 2026 scan of 5,125 MCP servers found [toxic data flows](https://agentseal.org/blog/toxic-data-flows-mcp-servers) where individually plausible tools combined into exfiltration, destructive-action, or privilege-escalation paths. On May 4, 2026, the Cloud Security Alliance's [MCP Security Crisis](https://labs.cloudsecurityalliance.org/research/csa-research-note-mcp-security-crisis-20260504-csa-styled/) note tied these composition risks to STDIO command execution, optional authorization, rug-pull changes, and cross-server tool shadowing. The November 2025 MCP authorization specification adds better OAuth resource binding, scope challenge, and confused-deputy guidance for HTTP transports, but it also leaves authorization optional and STDIO credentials environment-dependent. For C9.9, these findings reinforce that tool review must cover compositions, configuration provenance, argument origins, and token audience boundaries, not just whether each tool looks safe in isolation.

## Implementation Guidance

### Implementation Maturity

Emerging to early mature. Policy engines, approval middleware, tracing, and tool-call guardrails are available, but few platforms provide complete per-value provenance, taint-aware argument construction, cross-tool composition analysis, and tool-definition integrity out of the box. Treat vendor guardrails as useful enforcement plumbing only after confirming where they run, which tool types they cover, whether they see structured origin labels, and whether they fail closed before side effects.

### Notable Incidents and Case Studies

- **October 3, 2025: Amp DNS exfiltration via prompt injection.** MCPSec demonstrated that untrusted repository content could steer an agent into running a DNS command that encoded an API key in a subdomain without asking for approval. The useful C9.9 test case is not the exact command; it is the untrusted-file-to-shell-argument path and whether network-capable commands are denied when their arguments derive from project content.
- **November 19, 2025: ServiceNow Now Assist second-order prompt injection.** AppOmni reported that agent discovery and team-based collaboration settings could let a low-privilege, data-reading agent recruit more powerful agents to perform sensitive actions. This is a cross-agent version of the same origin problem: the task looks like normal delegation unless reviewers can see which untrusted field influenced the later action.
- **March 20, 2026: MCP toxic data-flow scan.** AgentSeal reported that 555 of 5,125 scanned MCP servers had tool combinations that could create exfiltration, destructive-action, privilege-escalation, or untrusted-relay paths. Treat this as a review pattern for installed tool catalogs: identify source tools, sink tools, and the policy that prevents untrusted data from connecting them.
- **April 14, 2026: OWASP GenAI exploit round-up.** OWASP's Q1 2026 report mapped recent agentic incidents to ASI01 Agent Goal Hijack, ASI02 Tool Misuse, and ASI09 Human-Agent Trust Exploitation, and noted that many events are architectural or configuration failures rather than CVE-style bugs. For C9.9 reviews, that is a reminder to preserve provenance and policy evidence even when no single vulnerable package explains the misuse.
- **May 4, 2026: CSA MCP systemic design-risk note.** The Cloud Security Alliance summarized MCP attack patterns spanning STDIO command execution, optional authorization, rug-pull tool changes, cross-server tool shadowing, and unauthenticated deployments. For C9.9, the practical review point is that data-flow isolation has to include tool configuration provenance, re-approval on definition changes, per-server session isolation, and behavioral monitoring of MCP server processes.

### Origin-Aware Tool Boundary

Treat the tool dispatcher as a policy enforcement point. Before any tool call runs, the dispatcher should receive a structured request containing the tool name, argument object, user or delegated agent identity, side-effect classification, destination, origin metadata for each argument, and the policy version to evaluate. The dispatcher should return an allow, deny, or review decision with a durable decision ID. The model can propose a call, but it should not be the component that authorizes the call.

### MCP Authorization Boundary

For MCP HTTP transports, verify OAuth resource indicators, audience validation, and scope challenges before treating a server as an approved tool source. These checks reduce token confusion and overbroad grants, but they answer "may this client call this server?" rather than "may this untrusted value populate this argument?" Keep the value-origin decision at the dispatcher or tool wrapper even when MCP transport authorization is configured correctly.

For STDIO MCP servers, treat the process launch configuration, inherited environment, command path, package version, and server manifest as part of the trust boundary. The November 2025 authorization specification explicitly leaves STDIO outside the HTTP OAuth flow, so auditors should look for separate controls: pinned binaries, minimal environment variables, per-server working directories, no shared high-value credentials, and re-approval when tool definitions or command lines change.

### Runtime Enforcement Checks

At a minimum, the pre-dispatch gate should validate three things before a tool executes: the proposed call is still attributable to trusted user intent, each protected argument field has an allowed origin, and the current policy version permits the destination and side effect. If any of those checks cannot be evaluated because origin metadata is missing or malformed, the safer default is deny or human review, not silent downgrade to prompt-only judgment.

For testing, keep a regression pack that mutates arguments without changing the tool sequence. Useful cases include hostile web text changing an email recipient, log text changing a remediation shell command, retrieved documents changing a database predicate, tool output changing a webhook URL, and mixed-origin text flowing into a public upload. These tests catch the failure mode where an agent appears to follow the approved plan while attacker-controlled data rewrites the consequential fields.

### Evidence Auditors Should Request

- Architecture diagrams showing the separation between planner, untrusted-data processors, policy engine, and tool executor.
- Policy-as-code repositories or managed policy snapshots with review history and tests.
- Tool-call traces showing argument origins, policy inputs, policy decisions, trace/span IDs, approvals, and execution outcomes.
- Negative-test results for indirect prompt injection, origin laundering, and same-plan argument manipulation.
- MCP or plugin inventory showing approved server versions, configuration provenance, tool-definition hashes, OAuth resource indicators, scope grants, token-audience validation, and re-approval records after manifest changes.
- Incident response queries that reconstruct which sources influenced a specific tool invocation.
- API and tool manifests that declare agent trust requirements, such as OpenAPI `x-agent-trust` metadata or MCP authorization scopes, mapped to runtime checks rather than accepted as documentation-only controls.

### Practical Starting Patterns

- Use a no-tool or read-only extractor for untrusted documents, web pages, emails, tickets, and tool outputs.
- Pass extracted values as typed data objects with source IDs and content hashes, not as unstructured text pasted into the planner context.
- Keep sensitive destinations, recipients, paths, SQL predicates, and mutating parameters under trusted-origin or explicit-approval rules.
- Route policy decisions through OPA/Rego, Cedar, or equivalent application code that produces replayable decision logs.
- Validate policy schemas so origin, trust class, side-effect level, destination, and agent identity are required fields for protected tools.
- Use runtime middleware and guardrails as enforcement plumbing, but treat them as incomplete unless they receive structured origin labels and fail closed before tool side effects.
- For MCP and large tool catalogs, review toxic combinations as a system: untrusted input tools, private-data tools, public sinks, destructive tools, privileged execution tools, and any server that combines several of those roles.
- Pin MCP server packages and tool definitions; require re-approval when descriptions, schemas, command lines, scopes, or server origins change.
- Use OpenTelemetry traces for execution causality and a separate provenance store for per-value or per-object data dependencies.
- Redact or hash sensitive argument values in logs, but keep enough metadata to support incident reconstruction.
