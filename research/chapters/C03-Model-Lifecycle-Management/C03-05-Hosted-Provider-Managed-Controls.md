# C3.5: Hosted and Provider-Managed Model Controls

> [Back to C03 Index](C03-Model-Lifecycle-Management.md)
> **Last Researched:** 2026-05-02

## Purpose

Hosted and provider-managed models may change behavior without notice. These controls help ensure visibility, reassessment, and safe operation when the organization does not control the model weights.

As of May 2026, the practical risk is no longer hypothetical. OpenAI's GPT-4o sycophancy rollback showed that an incremental model-behavior update can materially change safety posture even when the product name remains familiar; OpenAI's follow-up postmortem also said subtle model changes need more proactive communication and should treat behavior issues as launch-blocking concerns. Anthropic's Claude Code changelog records a March 4, 2026 configuration change where Opus 4.6 defaulted to `medium` effort for Max and Team subscribers, illustrating that hosted-model behavior can shift through configuration as well as model weights. Provider lifecycle transparency has improved in some places: Microsoft Foundry publishes lifecycle stages and retirement tables, OpenAI and Anthropic expose model identifiers in API responses, and observability tools increasingly support OpenTelemetry GenAI attributes. The remaining control gap is that no cross-provider protocol guarantees immutable model identity, routing transparency, or advance notification for configuration-level changes.

Verify each entry captures the requested model/deployment, provider-returned model identifier, provider name, endpoint/region, request ID, response ID where available, token usage, safety decision, gateway route, fallback target, and full safety-relevant request configuration (`temperature`, `top_p`, tools list, system prompt hash, service tier, effort/reasoning mode, and any provider-specific backend fingerprint). For providers that do not expose a model identifier, confirm the log contains an explicit `model_identifier_exposed=false` style field rather than silently omitting the value. Validate logs against the OpenTelemetry GenAI conventions where possible, and confirm gateway tools do not normalize away provider-native identifiers. | Model ID logging is necessary but not sufficient. OpenAI exposes response-side model and backend fingerprint data in Chat Completions; Anthropic exposes the response model and request ID; other providers expose model identity through deployment metadata, ARNs, or response headers. None of these fields reliably capture every configuration-level change, such as default effort, safety policy, system prompt, routing, or service-tier behavior. | **3.5.4** | **Verify that** high-assurance deployments fail closed or require explicit approval when the provider does not expose sufficient model identity or change notification information for verification. | 3 | High-risk workflows running on opaque hosted models whose identity, version, routing, or change history cannot be independently verified; regulated deployments lacking evidence for model provenance or lifecycle control; silent fallback to unapproved models during provider outages; risk acceptance decisions being made implicitly by gateway defaults rather than by accountable owners. | Define a minimum hosted-model identity policy for high-assurance tiers: provider-returned model ID, stable deployment or snapshot identifier, request ID, documented lifecycle/retirement channel, notification subscription, gateway route history, and logged configuration fingerprint. Test a provider or route that omits required identity data and verify the system blocks the request, routes to an approved fallback, or requires explicit approval with approver, justification, expiry, and compensating controls. Review exception records and confirm they are time-bounded. Run shadow evaluations against production traffic before approving a provider or model whose identity signals are weaker than policy requires. | This is a defense-in-depth control for cases where 3.5.1-3.5.3 reveal persistent identity gaps. It is most relevant for healthcare, finance, legal, public-sector, and critical-infrastructure workflows where auditability matters as much as model quality. ## Verification Notes

For audits, the strongest evidence is a joined trail across four sources: a hosted-model inventory, production inference logs, gateway route history, and the evaluation record that approved the model/configuration for the workflow. The auditor should be able to pick one high-risk output and answer: which provider handled it, which model identifier the provider returned, which route or fallback policy selected it, which request configuration was active, which eval bundle approved that configuration, and what would have happened if the provider failed to expose enough identity data.

Requested model names are not reliable evidence by themselves. Treat aliases, deployment names, and gateway route labels as local pointers that must resolve to provider-returned identity, provider request IDs, and routing history at the time of the request.
