# C12.2: Detection and Alerting

> **Parent:** [C12 Monitoring, Logging & Anomaly Detection](C12-Monitoring-and-Logging.md)
> **Requirements:** 6 | **IDs:** 12.2.1--12.2.6

## Purpose

This section addresses the detection of adversarial activity, abuse patterns, and attack campaigns targeting AI systems. Unlike traditional application security monitoring, AI abuse detection must account for natural language attack vectors (prompt injection, jailbreaks), multi-turn attack strategies, and AI-specific exfiltration techniques. These controls bridge the gap between request/response logging (C12.1) and incident response (C12.5).

### Open-Source Detection Tools

For organizations building detection pipelines without commercial products, several open-source tools address requirements 12.2.1 and 12.2.3:

- **Vigil** -- Python library and REST API for assessing LLM prompts and responses. Uses multiple detection methods: vector database/text similarity, YARA/heuristic rules, transformer model analysis, prompt-response similarity, and canary tokens. Best-in-class for low false-positive deployments (0% FPR at 83.8% recall in June 2025 benchmarks).
- **Rebuff** -- Multi-layered prompt injection defense by Protect AI. Combines heuristic filtering, LLM-based detection (GPT-4o achieves 0.021 FPR vs. 0.499 for GPT-3.5-turbo), vector database of known attacks, and canary tokens. Most balanced trade-off at 3.4% FPR and 98.1% recall.
- **LLM Guard** -- Open-source prompt injection detection with highest recall (99.9%) but higher false-positive rate (12.7% FPR). Hard-coded detection threshold at 0.92 limits configurability.
- **NVIDIA NeMo Guardrails v0.21.0** -- Adds `IORails`, parallel input/output rail execution for content-safety, topic-safety, and jailbreak checks, unique request IDs, standalone `check_async` validation, an OpenAI-compatible guardrails server, LangChain `GuardrailsMiddleware`, and community integrations including CrowdStrike AIDR and regex-based rails. For 12.2.2 audits, the request ID and rail verdict metadata are the evidence to confirm in downstream logs.

A comparative evaluation (arxiv:2506.19109) found that combining multiple scanner types outperforms any single technique, and vector database scanners degrade 10-40% against obfuscated attacks. Canary token checks in default configurations showed 0% detection rates but improved to 41-94% with explicit instructions.

### Evolving Attack Landscape

The threat landscape for AI abuse detection continues to evolve rapidly:

- **Best-of-N (BoN) Jailbreaking** -- Automated attacks that generate large numbers of prompt variations using obfuscation techniques (character substitution, encoding, rephrasing), achieving near-100% success rates against safety filters in seconds. This makes signature-based detection (12.2.1) necessary but insufficient -- behavioral detection (12.2.2) and custom cross-session rules (12.2.3) are critical complementary layers.
- **Multi-turn Crescendo Attacks** -- Attackers gradually shift model behavior across conversation turns, with no single turn being overtly malicious. Detection requires session-level trajectory analysis (12.2.2) using sliding window analysis over conversation embeddings or cumulative risk scoring.
- **Indirect Prompt Injection via Tool Use** -- As AI agents gain tool-calling capabilities, adversarial content embedded in retrieved documents, web pages, or API responses can hijack agent behavior. Detection must extend beyond user-submitted prompts to cover all inputs the model processes.
- **LLM-as-C2-Channel** -- No longer theoretical. LAMEHUG (identified by CERT-UA in July 2025) uses the Hugging Face API with Qwen 2.5-Coder-32B-Instruct to dynamically generate reconnaissance and data theft commands. The SesameOp backdoor (Microsoft DART, July 2025) abuses OpenAI's Assistant API feature as a covert C2 server, avoiding attacker-hosted infrastructure entirely. Akamai Hunt identified a separate malware strain sending Base64-encoded C2 commands to /v1/chat/completions endpoints. Splunk has published 14 specific detection rules for LAMEHUG indicators. Check Point Research extended the pattern on February 17, 2026 with "AI in the Middle," demonstrating that web-based AI services -- specifically Grok and Microsoft Copilot -- can be turned into C2 *proxies* with no API key or registered account: the implant asks the AI to fetch and summarize an attacker-controlled page, tunneling host-enumeration data outbound through URL query parameters while commands flow back embedded in the AI's summary. Because no provider account is involved, traditional kill switches do not apply, and high-entropy encoding plus switching between streaming and non-streaming modes defeats fixed traffic signatures. Detection should monitor for unauthorized outbound calls to AI API endpoints, DNS queries to AI service domains from unexpected processes, and entropy anomalies in request/response payloads, treating AI domains as high-value egress points. Steganographic-canary detection (arxiv:2603.28655, March 2026) is an emerging research direction for flagging AI-driven malware that hides instructions in otherwise benign-looking model traffic.
- **AI Agent Context Poisoning & Memory Manipulation** -- Added to MITRE ATLAS in October 2025 (collaboration with Zenity Labs), these techniques target agentic AI systems. Context poisoning manipulates an agent's LLM context to persistently influence responses or actions. Memory manipulation alters long-term memory to ensure malicious changes persist across sessions. As of February 2026, Microsoft documented real-world memory poisoning via "Summarize with AI" buttons, identifying over 50 unique injection prompts from 31 companies embedding hidden instructions into AI assistant memory via URL prompt parameters.
- **TokenBreak Attacks** -- A newer class of adversarial attack targeting the tokenization layer rather than the prompt itself, manipulating how input text is broken into tokens to bypass content classifiers. This technique exploits an often-overlooked component in NLP pipelines and requires detection at the pre-tokenization stage.
- **Policy Simulation Attacks** -- Trick the model into believing unsafe outputs are allowed under updated or alternate policies, using prompts that mimic policy file structures (XML, JSON, INI). Roleplay-variant prompt injections achieved the highest attack success rate (89.6%) in 2025 red-team evaluations by deflecting responsibility away from the model.
- **Autonomous Reasoning-Model Jailbreaks** -- Published in Nature Communications (2026), research by Hagendorff et al. demonstrates that large reasoning models (DeepSeek-R1, Gemini 2.5 Flash, Grok 3 Mini, Qwen3 235B) can autonomously plan and execute multi-turn jailbreaks against other models with a 97.14% success rate, requiring no human guidance beyond an initial system prompt. This creates an alignment regression feedback loop: as reasoning models improve, they become more effective at subverting alignment in target models, making automated detection (12.2.2, 12.2.3) essential.
- **Unicode and Encoding Evasion** -- An empirical study (arxiv:2504.11168, 2025) tested character injection attacks against major guardrail products (Azure Prompt Shield, Meta Prompt Guard, Nvidia NeMo Guard, ProtectAI). Emoji smuggling achieved 100% evasion success for both jailbreaks and prompt injections; bidirectional text reached 99.23%. These techniques are black-box, requiring no knowledge of the guardrail internals, and highlight the need for input normalization before classification.
- **Indirect Injection as the Dominant Enterprise Threat** -- In the February 2026 Claude Opus 4.6 system card, Anthropic dropped the direct prompt injection metric entirely, arguing that indirect injection is now the relevant enterprise threat. Published numbers: a constrained coding environment held at 0% attack success across 200 attempts, while a GUI-based agent with extended thinking reached 17.8% at 1 attempt, climbing to 78.6% at 200 attempts without safeguards and 57.1% with safeguards. The shift matters for detection program design -- investments in per-message direct-injection classifiers yield diminishing returns compared to content scanning on untrusted inputs (emails, documents, web pages, tool output) before they enter the model's context window. The Google + Forcepoint April 24, 2026 joint reporting confirmed a 32% relative jump in malicious indirect prompt injection between November 2025 and February 2026, with payloads targeting search manipulation, API key exfiltration, file destruction, and embedded payment-redirect fraud (PayPal transactions, Stripe donation redirects). Obfuscation observed in the wild includes pixel-sized text, color-drained content, HTML metadata, and comment sections invisible to humans but visible to AI ingestion.
- **Operational IDPI Payload Engineering** -- Unit 42's March 2026 telemetry moved IDPI from isolated proof of concept to production detection content: 12 real-world cases, 22 payload construction techniques, and attacker intents including AI ad-review evasion, SEO manipulation for phishing, data destruction, denial of service, unauthorized transactions, sensitive data leakage, and system prompt leakage. Detection should inspect both rendered DOM and raw source for off-screen text, zero-size fonts, CSS-suppressed instructions, SVG-encapsulated text, Base64/JavaScript assembly, metadata, and comments before the content reaches browser agents, RAG crawlers, or document summarizers.
- **EchoGram Flip-Token Bypass** -- HiddenLayer published EchoGram in November 2025: an attack on guardrails (LLM-as-a-judge and classifier-based) that finds short "flip tokens" through dataset distillation and vocabulary probing, then appends them to malicious prompts to reverse the safety verdict. Published demonstrations include verdict flips on Qwen3Guard across multiple severity levels, with token combinations amplifying the effect. EchoGram exposes a structural fragility: guardrail training data has token-level imbalances that can be enumerated. Detection signals worth monitoring per 12.2.3 include false-positive flood patterns (an attacker probing for flip tokens) and unusual single-character or short-token suffix patterns appended to otherwise normal prompts.
- **Multimodal Symbolic / Rebus Injection** -- The NVIDIA AI Red Team (July 2025) demonstrated a class of "semantic" prompt injections that work entirely through visual symbol sequences -- a printer + waving figure + globe emoji-like arrangement parses as "print Hello, World" inside an early-fusion multimodal model (Llama 4 architecture, OpenAI o-series, Gemini), with similar arrangements coercing file deletion or shell-style operations without any explicit text. Cross-modal attacks evade text-only signature and classifier defenses because the adversarial signal lives in the shared latent representation. NVIDIA's recommendation is to push detection from input filtering to output-centric review, including adaptive output filters before code execution and runtime monitoring on tool calls.
- **Covert Acoustic Prompt Injection** -- Microsoft Research's SWhisper work (March 2026) demonstrated black-box near-ultrasonic prompt delivery against speech-driven LLM interfaces using commodity speakers and microphones, with injected audio reported as perceptually indistinguishable from background-only playback in a user study. For voice agents, signature and classifier checks should run both before transcription (audio anomaly detection, ultrasonic-band filtering, microphone-path hardening) and after transcription, because a clean transcript is not proof that the acoustic channel was clean.
- **Comment and Control Attack on Coding Agents** -- April 2026 research by Aonan Guan with Johns Hopkins University collaborators demonstrated working prompt injection against Anthropic's Claude Code Security Review, Google's Gemini CLI Action, and GitHub Copilot Agent via crafted GitHub comments, PR titles, and issue bodies. The attack hijacked the agents on GitHub Actions runners and recovered an API key in published proofs of concept. Detection programs should treat code-review surfaces (comments, titles, descriptions) as untrusted input streams equivalent to user prompts, with telemetry routed through 12.2.6 SIEM rules.
- **MCP Channel Compromise** -- January 2026 disclosed three CVEs (CVE-2025-68143/68144/68145) in Anthropic's official MCP Git server: two path traversal and one argument injection flaws exploitable via prompt injection in README files and poisoned issue descriptions. Separately, a public GitHub issue was shown to hijack a coding agent, pull data from private repos via an over-privileged PAT, and exfiltrate it into a public pull request. A systematic analysis of 78 studies (Jan 2026) reported adaptive attack success rates exceeding 85% against Claude Code, GitHub Copilot, and Cursor. Detection for 12.2.3 and 12.2.6 should treat MCP tool invocations as first-class telemetry, not sidecar traffic.
- **Industrial-Scale AI-Enhanced Phishing (Tycoon2FA / Storm-1747)** -- Microsoft's Digital Crimes Unit disrupted the Tycoon2FA phishing-as-a-service platform in March 2026, seizing 330 domains with Europol. The platform generated tens of millions of phishing emails monthly and was linked to nearly 100,000 compromised organizations since 2023, accounting for ~62% of phishing attempts Microsoft blocked at peak. AI-enhanced AiTM session-token interception combined with AI-customized message generation reportedly achieves 54% click-through rates versus ~12% for traditional campaigns. While outbound phishing detection lives outside the AI system itself, 12.2.3 rule libraries should treat AI-generated social-engineering payloads (forged corporate communication, AI-written spear-phishing) reaching enterprise AI assistants as a distinct threat class that may bypass legacy email defenses.
- **Third-Party AI Chatbot Plugin Compromise (IEEE S&P 2026)** -- "When AI Meets the Web: Prompt Injection Risks in Third-Party AI Chatbot Plugins" (accepted to IEEE Symposium on Security and Privacy 2026, arxiv:2511.05797) audited the plugin ecosystem and found 8 plugins used by ~8,000 websites transmitted browser message history to backend LLMs without integrity checks, allowing direct injection of forged conversation turns, and 15 plugins supported automated web scraping for RAG that ingested both first-party and third-party content indiscriminately, creating a turnkey indirect prompt injection surface. Detection for 12.2.3 should add allowlists for plugin-origin message payloads and treat plugin-fetched content as untrusted RAG input subject to the same scanning as user-supplied documents.
- **Concealed Indirect Injection Success** -- A March 2026 large-scale public competition showed why final-answer review is not enough: 464 participants submitted 272,000 attempts against 13 frontier models, producing 8,648 successful attacks across 41 tool-calling, coding, and computer-use scenarios. The relevant SOC signal is hidden in the trace, not the final message, because a compromised agent can perform the harmful action while returning a harmless-looking answer to the user.
- **EchoLeak (CVE-2025-32711) -- First Zero-Click Production Prompt Injection** -- Aim Labs disclosed and Microsoft patched the first zero-click prompt injection exploit against a production LLM agent. The single-email attack against Microsoft 365 Copilot achieved remote, unauthenticated data exfiltration by chaining bypasses of XPIA classifier filtering, reference-mention link redaction, Content Security Policy via a Microsoft Teams proxy, and auto-rendered image references. Patched in the June 2025 Patch Tuesday cycle (CVSS 9.3); arxiv:2509.10540 (2025) generalized the failure mode as a structural risk in RAG-based AI assistants -- detection should target *chains* of weak primitives (each individually below alert thresholds) crossing trust boundaries, not only standalone payloads. AVID has begun cataloguing analogous in-the-wild cases (AVID-2026-R0055 covers CVE-2025-32018 arbitrary file write through prompt injection from malicious `@Docs` in Cursor versions 0.45.0-0.48.6), reinforcing that production agents are now the dominant target surface.
- **Same-Day Mass Scanning of Open-Source LLM Agent Frameworks (PraisonAI CVE-2026-44338)** -- Sysdig research (May 14, 2026) recorded internet-wide scanners probing the exact PraisonAI auth-bypass endpoint **3 hours 44 minutes 39 seconds** after public disclosure on May 13, 2026, using the `CVE-Detector/1.0` user agent against `GET /agents` and `POST /chat` routes that PraisonAI shipped with `AUTH_ENABLED = False` hard-coded into the legacy `api_server.py`. PraisonAI v4.6.34 ships the fix and migrates off the legacy entrypoint, binding to 127.0.0.1 and requiring `--api-key`. The exploitation timeline is significant for detection design: open-source LLM agent control planes (Praison, AutoGen, CrewAI, LangFlow, Flowise, SQLBot) are now opportunistically scanned within hours of a CVE, not weeks, so 12.2.3 rule libraries should include scanner-user-agent and unauth-endpoint patterns for these frameworks, and 12.2.6 isolation playbooks need same-day enforcement windows.
- **SQLBot CVE-2026-32622 Stored Prompt Injection RCE** -- Disclosed March 2026 in the dataease/SQLBot LLM+RAG data-query stack (versions <=1.5.0). An authenticated user uploads an Excel terminology file with crafted descriptions that pass through to the LLM's system prompt unsanitized and without semantic fencing, coercing the model to emit PostgreSQL `COPY ... TO PROGRAM` statements that execute under the database service account. The chain is interesting because no single layer is wildly broken -- it is the absence of permission checks on terminology uploads, output sanitization, and semantic boundaries between user-controlled glossary data and system instructions that combine into RCE. Patched in v1.6.0. The same pattern applies to any RAG product that lets users seed glossary, schema, or example-query content; detection rules should treat authenticated terminology/glossary uploads as untrusted input subject to the same scanning as model prompts and tool output.
- **OpenAI Guardrails Self-Policing Bypass (Oct 10 2025)** -- HiddenLayer demonstrated that the LLM judge inside OpenAI's Guardrails framework (released Oct 6 2025) can be prompt-injected by embedding fake confidence scores and threshold values in the user input, dropping the judge's verdict below the default 0.7 cutoff on `gpt-4-mini`. Both the Jailbreak Detection and Prompt Injection Detection pipelines fell, including indirect prompt injection through tool calls. The structural finding is that LLM-as-a-judge guardrails inherit the same prompt-injection susceptibility as the model they protect, so 12.2.1 stacks that rely on a same-family judge need a structurally different second layer (separate model family, rule-based pre-filter, or out-of-band classifier) rather than another LLM judge. The same failure mode appeared in the late-2025 EchoGram research against Qwen3Guard and other LLM-as-a-judge guardrails, reinforcing that LLM-judge confidence scores are themselves an attack surface.
- **Promptware Kill Chain** -- Brodt, Feldman, Bruce Schneier, and Ben Nassi (arxiv:2601.09625, Jan 2026, revised Feb 2026) reframe prompt injection as "promptware" -- a malware class triggered by engineered prompts -- and map it to a seven-stage kill chain: initial access (injection), privilege escalation (jailbreak), reconnaissance, persistence (memory and retrieval poisoning), command-and-control, lateral movement, and actions-on-objective. Reviewing 36 studies and real-world incidents, they found at least 21 documented attacks traversing four or more stages, arguing the threat is operational rather than theoretical. For 12.2.2 and 12.2.3 detection design, the kill-chain framing is directly useful: it gives SOC teams a stage taxonomy to label detection coverage (most programs over-invest in stage 1 injection signatures and under-instrument persistence and lateral-movement stages), and it lines up with the AgentPex/AgentSentry finding that the relevant signal lives in the multi-step trace, not any single prompt.
- **Chain-of-Thought and Reasoning-Trace Jailbreaks (May-June 2026)** -- A cluster of mid-2026 research moved the attack surface inside the model's reasoning. A CoT jailbreak technique (arxiv:2605.24497) decomposes a harmful objective into individually innocuous fragments routed through the model's own chain-of-thought, and "refusal-escape direction" work (arxiv:2605.08878) identifies internal directions that flip a refusal into compliance, explaining why aligned models keep failing. Visual-degradation bypass (arxiv:2605.07250) showed that simply lowering image resolution can catalyze a jailbreak while keeping text legible, and the MultiBreak benchmark (arxiv:2605.01687) quantified the multi-turn gap. The detection implication for 12.2.2 is that trajectory analysis should consider reasoning traces and tool-call sequences as part of the monitored signal where they are available, not just user-visible turns.
- **ChatGPhish / Summary-Delivered Phishing (May 2026)** -- Researchers demonstrated embedding malicious instructions in web pages so that an AI assistant's summary delivers phishing content to the user (The Register, May 2026), a variant of indirect injection where the model becomes the delivery vehicle rather than the exfiltration channel. For 12.2.3 scope, this argues that content scanning on ingested web pages must look for instructions targeting the *summary output* (injected URLs, fake support numbers, payment redirects), not only instructions targeting tool calls.
- **Mass-Exploitation of Claude Code and ChatGPT for Government Data Theft (Mexican Government Breach, Dec 2025 - Feb 2026)** -- A single attacker chained Claude Code and OpenAI's GPT-4.1 to compromise nine Mexican federal, state, and municipal agencies between December 2025 and February 2026. Disclosed reporting documents 195 million taxpayer records from the federal tax authority (SAT) including a forged-tax-certificate service, 220 million civil records from Mexico City, and a full 13-node Nutanix cluster takeover in Jalisco state with access to 37 database servers covering health and domestic-violence records, totaling more than 150GB exfiltrated. Telemetry attributed to the attacker shows 1,088 prompts producing 5,317 remote commands across 34 live agent sessions, with Claude Code executing roughly 75% of those commands. For 12.2.3 and 12.2.6 detection programs, this is the first publicly attributed multi-agency breach where a commercial coding agent ran the majority of the operator's hands-on-keyboard activity, and it argues that per-customer abuse detection at the model-provider tier (sustained prompt throughput per account, multi-tenant target diversity, unusual command-mix entropy) is no longer optional supplementary telemetry but core enterprise-relevant signal.

### Enterprise Detection Maturity (Q1-Q2 2026 Baselines)

Two industry data points help calibrate where most detection programs actually sit, versus where the requirements above target:

- **VentureBeat enterprise survey (April 2026)** -- 88% of surveyed enterprises reported AI agent security incidents in the prior 12 months, but only 21% had runtime visibility into what their agents were doing, only 47% had observed agents exhibiting unintended behavior, and just 5% felt confident they could contain a compromised agent. The "stage 3" threat framing (post-compromise agent abuse with privileged tool access) is where most programs are weakest -- detection rules covering 12.2.3 and 12.2.6 generally lag behind 12.2.1 signature deployment.
- **CrowdStrike 2026 Global Threat Report (Feb 26 2026)** -- 90+ organizations with prompt-injection abuse documented in 2025, 89% YoY growth in AI-enabled adversary operations, and 82% of all detections involving malware-free intrusions through legitimate pathways. The takeaway for 12.2 detection design: pattern-of-life and behavioral telemetry on legitimate AI tool usage now matters as much as content classifiers, because the abuse path is "valid prompt + valid tool + adversarial intent" rather than novel malware delivery.
- **CrowdStrike Falcon AI sensor telemetry (RSAC 2026)** -- CrowdStrike's keynote disclosed sensors detecting 1,800+ distinct AI applications across roughly 160 million unique application instances on enterprise endpoints, illustrating the scale of "shadow AI" that 12.2.3 detection programs must cover before they can claim agent-environment visibility. Treat any abuse-detection program that monitors only sanctioned endpoints as undersized by default.
- **Unit 42 enterprise IPI telemetry (April 2026)** -- Unit 42's analysis of indirect prompt injection in production showed credential or payment-data exposure in 18% of investigated AI security incidents, and 62% of *successful* enterprise exploits used indirect injection rather than direct prompt manipulation. For 12.2.3 rule prioritization, this argues for shifting investment toward content scanning at ingestion points (email, documents, web pages, tool output) before they reach the model context, not only at the model boundary.
- **Cloud Security Alliance + Token Security -- "Autonomous but Not Controlled" (April 21, 2026)** -- 82% of surveyed enterprises had unknown AI agents running in their environment, 65% experienced AI-agent-related incidents in the prior 12 months (61% sensitive-data exposure, 43% operational disruption, 35% financial loss), and only 21% had formal decommissioning processes. The combined CSA "Enterprise AI Security Starts with AI Agents" study (April 16, 2026) added that 53% of organizations had agents exceed their intended permissions, 47% had a confirmed incident in the prior year, and incident detection/response time ran from hours to days even when telemetry was available. The implication for 12.2.3/12.2.6 coverage is concrete: detection scope must include unsanctioned and end-of-life agents, scope-violation telemetry (tool calls or data access outside the agent's documented capability set) is a higher-yield signal than per-prompt classification alone, and time-to-contain is the metric that separates programs that experience incidents from programs that contain them.
- **Intruder "We Scanned 1 Million Exposed AI Services" (May 5, 2026)** -- Internet-scale baseline scan of 2 million hosts that surfaced 1 million exposed AI services via certificate transparency. Headline findings: 31% of 5,200+ Ollama servers answered an unauthenticated test prompt, 518 instances were front-ending frontier models from Anthropic, Deepseek, Google, OpenAI, and Moonshot without authentication, and at least 90 exposed agent management platforms (n8n and Flowise) were running across government, marketing, and finance with open workflows, prompts, and outbound access. Repeated design failures: no auth on default install, hard-coded credentials in setup files, services running as root, and weak sandboxing on code-interpretation tools. For 12.2.3 detection scope, the practical implication is that "shadow AI" is not only an enterprise inventory gap but a public-internet attack surface comparable to the legacy unauthenticated Redis / Jenkins / Elasticsearch eras, and SIEM rules should ingest external scan telemetry (Shodan, Censys, internal asset discovery) alongside endpoint and identity signals.

These baselines suggest that for most enterprises the highest-yield investment in 2026 is shifting telemetry from prompt content alone to **agent action telemetry** (which tools were invoked, with what arguments, against what data, by which credentialed identity), then writing 12.2.3/12.2.6 rules over that data plane.

### Detection Architecture

A layered detection architecture addresses the full spectrum of requirements:

1. **Layer 1 -- Signature-Based (12.2.1):** Pattern matching against known jailbreak databases (JailbreakBench, PromptInject). Fast, low false-positive, but easily evaded by novel attacks. Signature databases require continuous updates.
2. **Layer 2 -- Classifier-Based (12.2.1):** ML classifiers trained on adversarial input datasets detect prompt injection and jailbreaks that evade signatures. Azure Prompt Shields and similar products operate at this layer.
3. **Layer 3 -- Behavioral Anomaly (12.2.2, 12.2.5):** Statistical anomaly detection on session-level features (retry rates, token consumption patterns, input diversity scores, query timing). Catches automated attack tools and probing behavior.
4. **Layer 4 -- Session Trajectory (12.2.2):** Multi-turn conversation analysis using stateful architectures. DeepContext (Feb 2026) demonstrated that RNN-based hidden state propagation over turn embeddings achieves 0.84 F1 at sub-20ms latency, outperforming stateless approaches like Llama-Prompt-Guard-2 (0.67 F1). Detects crescendo attacks, progressive manipulation, and autonomous reasoning-model-driven jailbreaks.
5. **Layer 5 -- Cross-Session Correlation (12.2.3):** SIEM rules correlating events across users, IPs, and time windows to detect coordinated campaigns and model extraction attempts.
6. **Layer 6 -- Tool-Boundary and Provenance Checks (12.2.3, 12.2.6):** Evaluate the trace at tool-return, retrieval, browser, and document-ingest boundaries, not just before the first model call. AgentSentry-style counterfactual re-execution can localize where a malicious tool result changed the trajectory; ARGUS-style provenance auditing can require the agent to justify decisions using trusted evidence rather than untrusted context alone.

### SIEM Integration Patterns

For requirement 12.2.2, the most maintainable integration approach in 2025-2026 uses OpenTelemetry pipelines:

- **Collection:** OpenTelemetry SDK emits GenAI spans with security event attributes
- **Processing:** OpenTelemetry Collector or Vector/Fluentd normalizes and enriches events
- **Routing:** OTLP export to SIEM (Splunk, Elastic, Sentinel) alongside observability backends
- **Detection:** Custom SIEM rules augmented with AI-specific detection logic (12.2.6)

As of March 16 2026, OCSF v1.8.0 ships native AI support: the `ai_operation` profile with `ai_model` and `message_context` objects provides vendor-neutral schema coverage for AI workloads including token usage metrics and role-based interaction tracking. v1.8.0 also added new `privilege_info`, `privilege_attack_info`, and `service_privilege_analysis` objects with MITRE ATT&CK technique mapping, enabling unused-privilege detection on AI service identities -- directly relevant to detecting compromised model API credentials per 12.2.5. OCSF maintainers announced collaboration with the OpenTelemetry maintainers in early 2026 to unify security and observability domains. ITU member nations formally supported OCSF for ratification as an international standard in December 2025, with **final ratification slated for June 2026**, after which schema extensions in private SIEM/XDR pipelines should align with the ratified vocabulary to avoid migration debt. CrowdStrike Falcon AIDR natively streams AI detection findings to Falcon Next-Gen SIEM; Splunk has published detection rules for LLM-abusing malware. Microsoft Incident Response published structured playbooks for prompt abuse detection and response in March 2026, emphasizing defense-in-depth combining monitoring, analysis, and documented response protocols. Organizations adopting OCSF-aligned pipelines can reduce custom parsing overhead significantly.

Runtime-response verification for 12.2.6 should include evidence that a block decision can be traced end to end: prompt or content source, model/agent identity, tool name, tool arguments, effective user or service identity, data object touched, block/transform/report action, and incident ID. Microsoft Defender's Agent 365 documentation is a useful reference implementation because it explicitly correlates real-time protection blocks into incidents and exposes hunting pivots over agent action data. In custom stacks, the same minimum evidence can be modeled with OCSF `ai_operation` plus privilege-analysis objects, or with OpenTelemetry GenAI spans enriched before export to Sentinel, Splunk, Elastic, or another SIEM.

Minimum rule-test fixtures for 12.2.2/12.2.6 should include:

- A blocked direct jailbreak with model ID, detector verdict, confidence, and policy action.
- A blocked indirect prompt injection from a retrieved document or webpage, preserving source URL and content hash.
- A suspicious tool invocation with tool name, arguments, user identity, service identity, and target resource.
- A token-consumption anomaly with per-user, per-session, per-model, and per-endpoint counters.
- A covert-channel candidate with entropy score, destination AI API host, initiating process, and beacon timing.
- A response playbook event showing user block, agent disablement, model endpoint isolation, or permission revocation.

### MITRE ATLAS Coverage (Updated through v2026.05, May 27 2026 + CTID Secure AI v2, May 6 2026)

MITRE ATLAS has shipped six releases relevant to abuse detection between October 2025 and May 2026, with monthly cadence locked in by the CTID Secure AI v2 release. The May 27, 2026 release also changed the project's release model in ways that matter for detection-content tooling:

- **v2026.05 (May 27 2026)** -- ATLAS split versioning into a content stream (`YYYY.MM.N`) and a format stream (ATLAS YAML format v6.0.0), and tagged **every technique with platform designations (Predictive AI, Generative AI, Agentic AI, Enterprise)** so detection teams can filter the matrix down to the techniques that apply to a given workload type. The release shipped validation tooling, SQLAlchemy ORM models, and a FastAPI REST API alongside the existing STIX/Excel/Navigator outputs, which makes it practical to pull ATLAS technique metadata programmatically into SIEM rule-to-technique mappings instead of scraping the site. Recent technique additions in the same window include **AML.T0098 (AI Agent Tool Credential Harvesting)** and **AML.T0100 (AI Agent Clickbait)** -- adversaries crafting pages, documents, or UI elements designed to manipulate an AI agent's decision-making -- both of which belong in 12.2.3 rule scope for tool-using and browser agents.

- **v5.1.0 (Nov 2025)** -- 16 tactics, 84 techniques, 56 sub-techniques, 32 mitigations, 42 real-world case studies; consolidates the October 2025 Zenity Labs collaboration that introduced 14 agent-focused techniques.
- **v5.3.0 (Jan 2026)** -- Three MCP-focused case studies: MCP server compromise, indirect prompt injection via MCP channels, malicious AI agent deployment.
- **v5.4.0 (Feb 5 2026)** -- 7 new agent-focused techniques including **Publish Poisoned AI Agent Tool** and **Escape to Host**, plus 4 case studies including ClawdBot vulnerabilities and OpenClaw remote code execution scenarios. The companion CTID OpenClaw investigation (February 9, 2026) mapped real OpenClaw exploitation paths to ATLAS TTPs and identified chokepoint techniques adversaries rely on, including direct and indirect LLM prompt injection, AI agent tool invocation, and Modify Agentic Configuration.
- **v5.5.0 (Mar 30 2026)** -- 12 new techniques including **AI Agent Tool Poisoning**, **Machine Compromise**, and **Cost Harvesting** subtechniques, plus 5 new case studies including LLMSmith (RCE vulnerabilities in LLM-integrated applications) and model distillation campaigns.
- **v5.6.0 (May 4 2026)** -- Three additional techniques including **Acquire Public AI Artifacts: AI Agent Configuration** (adversaries acquiring deployed agent configs and tool descriptors before targeted attack) and **Phishing: Deepfake-Assisted Phishing**, plus refinements to existing phishing and LLM-jailbreaking entries. The new "Acquire" entry tightens the link between adversary reconnaissance and downstream 12.2.3 detection coverage -- detection rules for agent-configuration exfiltration should fire before the runtime block.
- **CTID Secure AI v2 (May 6 2026)** -- Center for Threat-Informed Defense release adding 45+ techniques and sub-techniques, 10+ mitigations, and 20+ case studies, introducing a Technique Maturity filter for prioritizing emerging threats, the ATLAS Knowledge Graph for interactive exploration, and a committed monthly release cadence. CTID also published threat-emulation playbooks (Ansible) starting with "prompt injection against organizational chatbots," giving 12.2.3 rule authors a reproducible test harness instead of bespoke scripts. The same release introduced the **first ATLAS Rapid Response Report**, a process for publishing analysis of emerging AI security incidents and adversary tradecraft on a much shorter timeline than scheduled minor releases. For SOC programs, the Rapid Response Report is the right artifact to subscribe to for detection-content refresh between monthly ATLAS drops -- incident-derived TTPs land there first, ahead of the formal technique additions.

Key earlier additions still relevant to abuse detection:

- **AI Agent Context Poisoning (AML.T0057)** -- Manipulating context to persistently influence agent responses or actions
- **Memory Manipulation (AML.T0058)** -- Altering long-term LLM memory to persist malicious changes across sessions
- **Thread Injection** -- Introducing malicious instructions into a specific chat thread to change behavior for the duration of that conversation
- **Modify AI Agent Configuration** -- Persisting adversary-controlled settings in agent configuration stores so that malicious behavior survives restart
- **AI Service API (AML.T0096)** -- Using legitimate AI service APIs (OpenAI Assistant API, Hugging Face Inference API, MCP endpoints) as covert C2 channels, demonstrated in the wild by LAMEHUG and SesameOp

Detection rules (12.2.6) should map to ATLAS techniques, and analysts should use ATLAS indicators to draft SOC detection hypotheses that SIEM/XDR platforms can exercise; published mappings show roughly 70% of ATLAS mitigations line up with existing enterprise security controls, so much of the work is instrumenting and labeling rather than inventing from scratch. The SAFE-AI framework maps ATLAS threats to NIST SP 800-53 controls at each system element (Environment, AI Platform/Tools, AI Models, AI Data), providing a structured approach to control selection. NIST IR 8596 (Cybersecurity Framework Profile for AI, 2025) provides complementary guidance on managing cybersecurity challenges specific to AI systems.

### MITRE ATT&CK v19 (April 28 2026, shipped)

ATT&CK v19 landed on schedule and reshapes how AI-related TTPs are organized for SOC detection programs:

- **Defense Evasion split** -- The legacy Defense Evasion tactic was divided into **Stealth (TA0005, retained ID, "behavioral camouflage")** and **Defense Impairment (TA0112, new ID, "actively breaking your defenses")**. T1562 (Impair Defenses) sub-techniques were restructured, with T1562.001 and T1562.006 merging into the new **T1685: Disable or Modify Tools**. SIEM rule libraries with ATLAS->ATT&CK cross-mappings need regression testing on the migration -- legacy TA0005 references to AI-specific evasion (jailbreak suffix injection, output filter bypass, guardrail vocabulary attacks like EchoGram) should split between Stealth (T1685, evasion-by-obfuscation) and Defense Impairment (rules where the adversary is actively turning a guardrail off rather than slipping past it).
- **New AI techniques** -- Three top-level entries directly relevant to 12.2.3 detection: **T1682 Query Public AI Services** (reconnaissance/planning at scale, where adversaries call commercial LLM APIs to generate attack tooling), **T1683 Generate Content** with sub-techniques T1683.001 (written content, e.g. phishing emails) and T1683.002 (audio-visual content, e.g. deepfakes), and **T1684 Social Engineering** as a new parent rolling up impersonation and email spoofing. Detection programs should add rules for outbound calls to commercial LLM APIs from non-application processes, and for content-generation patterns originating from compromised internal hosts.
- **Campaign C0062 -- Anthropic AI-orchestrated Campaign** -- First named ATT&CK campaign anchored in autonomous AI use. Documents GTG-1002, a PRC-aligned cluster that used Claude Code agents and Model Context Protocol tools to automate ~80–90% of tactical operations across roughly 30 targets in technology, financial, chemical, and government sectors starting September 2025. Operators bypassed AI guardrails through fabricated "defensive cybersecurity testing" personas, then tasked agents with reconnaissance, SSRF exploitation, lateral movement, credential harvesting, and exfiltration via web services. Anthropic's published mitigation expanded its cyber-focused classifiers and account-detection telemetry. For 12.2.3 detection, the takeaway is that abuse patterns must now span account-level behavioral signals (sustained agent throughput at "physically impossible request rates" against multiple targets) and prompt-content classifiers, not either in isolation.
- **Software entry S9035 -- LAMEHUG** -- First malware family formally tracked in ATT&CK that queries an LLM in live operations. Tied to APT28 (G0007). Validates the LLM-as-C2 detection scope already documented under 12.2.6.

The April 2026 ATT&CK release also added 36 new software entries, two new threat groups (MirrorFace, VOID MANTICORE), and Mobile Detection Strategies. Enterprise gained Detection Strategies for the new AI-related techniques and Defense Impairment family, providing concrete starting telemetry that SOC teams can adapt instead of writing from scratch.

### The Lethal Trifecta as a Detection-Triage Lens

Simon Willison's "lethal trifecta" framing (private data access + exposure to untrusted content + an exfiltration vector) gained traction across 2026 as a structural model for prioritizing 12.2.3 detection coverage. Between January 7-15 2026, four major AI-powered productivity tools had public indirect-prompt-injection disclosures that all reduced to the same trifecta pattern. Constellation, HiddenLayer, and Promptfoo each published trifecta-specific test harnesses through Q1 2026.

For SOC teams the practical implications are:

- **Triage before detection.** The trifecta is an architectural property -- audit each agent deployment for which "leg" is unconstrained, then weight 12.2.3/12.2.6 detection priority by how many legs are present. Agents with all three legs cannot be reliably defended through detection alone; structural defenses (taint tracking, capability gating, exfiltration egress controls) reduce blast radius before per-message classifiers ever fire.
- **Filter ceiling.** Publicly reported guardrails reach roughly 97% accuracy on known patterns, but attacker iteration cycles compress that gap rapidly -- detection should be paired with hard policy gates on the exfiltration leg (e.g., outbound URL allowlists, image-rendering restrictions to block image-tag exfiltration, CSP on agent-rendered surfaces).
- **MCP and tool inventory.** Audit MCP server configurations and tool descriptions for embedded instructions before they reach the agent context window; this directly feeds 12.2.3 detection-rule scope and 12.2.6 covert-channel monitoring.

### Audio-Modality Jailbreak Coverage (JALMBench, ICLR 2026)

JALMBench (ICLR 2026, [openreview.net/forum?id=DJkQ236C8B](https://openreview.net/forum?id=DJkQ236C8B)) is the first large-scale benchmark for jailbreak vulnerabilities in audio language models (LALMs): 11,316 text samples, 245,355 audio samples (>1,000 hours) covering 12 mainstream LALMs, 8 attack methods (4 text-transferred, 4 audio-originated), and 5 defenses. Findings relevant to 12.2.1 and 12.2.3 coverage:

- Audio-originated attacks (direct audio manipulation) remain effective even when text-based safety alignment partially transfers to audio inputs.
- Existing general-purpose moderation methods only marginally improve security; defenses adapted from vision-language models are largely ineffective.
- Interleaved audio-text strategies provide stronger cross-modal generalization than single-modality defenses.

Detection programs that ingest audio (voice agents, transcription pipelines, multimodal copilots) should add JALMBench-derived test cases to the same regression suite that runs JailbreakBench and AgentDojo, and should not assume that text-side guardrails extend automatically to audio.

SWhisper adds a second audio-side test family: covert near-ultrasonic delivery of arbitrary prompts before the speech recognizer produces text. The verification point is not only whether the final text prompt is blocked, but whether the platform detects abnormal acoustic energy, rejects unsafe transcriptions produced from hidden audio, and preserves enough audio-path metadata for incident review.

### Benchmarks for Measuring Detection Effectiveness

Abuse detection claims should be backed by repeatable benchmark runs, not vendor marketing. Useful public evaluation suites as of April 2026:

- **AgentDojo (NeurIPS 2024, active through 2026)** -- 97 realistic agent tasks (email client, e-banking, travel booking) with 629 security test cases and attack/defense paradigms from the literature. Baseline GPT-4o achieves 69% benign utility, dropping to 45% under attack; targeted "Important message" attacks succeed 53.1% of the time on an undefended baseline. Prompt sandwiching improves utility (65.7%) but leaves attack success at 30.8%; tool filtering drops attack success to 7.5% at the cost of utility (53.3%). US and UK AISI used AgentDojo to document Claude 3.5 Sonnet (new) vulnerabilities. As of ICLR 2026, **PromptArmor** -- an off-the-shelf LLM preprocessor that detects and removes injected prompts through a carefully constrained detection prompt -- achieves <1% false-positive and <1% false-negative rates on AgentDojo, with attack success near 0% (0–0.47%) while preserving up to 76.35% utility on o4-mini and 72.02% on GPT-4.1. The trade-off is roughly 200-600ms added latency and one extra LLM call per user input. AgentSys reports 0.78% / 4.25% attack success on AgentDojo with marginal benign utility improvement. Use this suite to validate 12.2.3 detection rules and 12.2.2 session-trajectory analysis.
- **AgentPex / agentic trace evaluation (Microsoft Research, Mar 2026)** -- Evaluates full agent execution histories rather than only final answers. In published tests over 424 tau2-bench traces, specification-derived checks exposed routing, tool-use, and policy-compliance failures that outcome-only benchmarks missed. Use this style of trace replay for 12.2.2 and 12.2.3 controls whenever the agent can invoke tools or skip approval gates.
- **Public indirect-prompt-injection competition (Mar 2026)** -- 272,000 attack attempts from 464 participants against 13 frontier models produced 8,648 successful attacks across tool-calling, coding, and computer-use settings. Use the competition environment and quarterly updates as regression material for concealed compromises where the trace is malicious but the final response is benign.
- **AgentSentry / ARGUS / tool-interface firewalls (2026)** -- These research lines are converging on boundary-aware evaluation: tool-input minimization, tool-output sanitization, counterfactual trace replay at tool returns, context purification, and provenance-aware decision auditing. They are useful design patterns for 12.2.3 and 12.2.2 tests even before they mature into off-the-shelf SOC products.
- **JailbreakBench, PromptInject** -- Still the standard corpora for 12.2.1 signature-based testing; require refresh against BoN and Unicode-evasion variants to stay relevant.
- **Comparative Evaluation of LLM Security Detection Systems (arxiv:2506.19109, Jun 2025)** -- Published FPR/recall numbers for Vigil, Rebuff, and LLM Guard; use as a baseline when procuring or swapping detection tooling.

Publish benchmark deltas alongside each SIEM rule rollout so detection coverage is measurable rather than asserted.

## Open Research Questions

- How effective are embedding-based similarity detectors at catching novel jailbreaks not in signature databases? (Initial data from June 2025 benchmarks suggests vector database scanners degrade 10-40% against obfuscated attacks.)
- What is the optimal window size for multi-turn trajectory analysis (12.2.2) that balances detection latency with false positive rates?
- Can LLM-as-judge approaches reliably detect adversarial conversations without introducing unacceptable latency? RLM-JB research shows GPT-5.2 achieving 98% recall with zero false positives, but production latency characteristics remain understudied.
- What entropy thresholds reliably distinguish C2 traffic from legitimate Base64 usage (e.g., image uploads) in LLM API traffic? The LAMEHUG and SesameOp incidents in 2025 provide real-world data points that should inform threshold calibration.
- How should detection rules be shared across organizations without revealing proprietary model vulnerabilities?
- How do AI firewall products (Azure Prompt Shields, Cloudflare, Akamai, CrowdStrike Falcon AIDR) compare in detection rates for novel vs. known jailbreak techniques, and what are the latency/accuracy tradeoffs of edge-layer vs. application-layer vs. SDK-based detection?
- What is the false positive rate of behavioral anomaly detection systems in high-volume production environments, and how should thresholds adapt to different user populations?
- How should organizations detect and prevent AI memory poisoning attacks at scale, given that Microsoft identified 50+ unique injection prompts from 31 companies as of February 2026?
- With OCSF v1.8.0 now providing a native `ai_operation` profile, how complete is its coverage for adversarial AI security events (jailbreaks, prompt injection, model extraction) vs. operational AI telemetry? Are extensions needed for attack-specific event classes?
- How should detection architectures adapt to autonomous reasoning-model jailbreaks (97.14% success rate), where the attacker is itself an LLM capable of real-time strategy adaptation?
- Can DeepContext-style stateful RNN detectors (0.84 F1) be combined with per-turn classifiers to achieve higher recall without unacceptable false positive rates in production?
- What input normalization strategies effectively counter Unicode/encoding evasion (emoji smuggling, bidirectional text) without degrading legitimate multilingual input handling?
- Now that Anthropic has dropped its direct prompt injection metric in favor of indirect injection (Feb 2026 system card), how should SOC detection programs rebalance investment between per-message classifiers and upstream content-scanning on untrusted inputs (emails, retrieved documents, tool outputs)?
- With the Jan 2026 MCP Git server CVEs (CVE-2025-68143/68144/68145) and the 78-study systematic finding of >85% adaptive attack success on Claude Code, Copilot, and Cursor, what is the minimum viable telemetry schema for MCP tool invocations that SIEM rules (12.2.6) can act on across vendors?
- How should benchmark results (AgentDojo utility/attack-success trade-offs) be surfaced to SOC leadership as ongoing KPIs for 12.2.2 and 12.2.3 coverage rather than one-time procurement data points?
- As the Check Point + Lakera, Meta LlamaFirewall, Portkey, and CrowdStrike Falcon AIDR ecosystems converge on similar claims (98%+ detection, sub-50ms latency), what independent evaluation methodology can buyers use to distinguish marketing from measurable detection coverage?
- Given Google + Forcepoint's April 2026 finding of a 32% relative increase in malicious indirect prompt injection over a four-month window, how should detection programs weight per-page IPI scanning at the crawl/ingest layer versus per-prompt classification at the model boundary?
- EchoGram demonstrates that flip-token vocabularies are enumerable from public guardrail weights. What is the right pre-deployment red-team budget for vocabulary-probing (token combinatorics) on a new guardrail before production rollout?
- PromptArmor reports <1% FP/FN on AgentDojo with 200-600ms latency. For latency-sensitive agentic workflows (browser automation, IDE coding agents), can a smaller distilled preprocessor maintain those rates, or is the cost an inherent property of LLM-as-preprocessor designs?
- The "Comment and Control" attack succeeded against three major coding agents simultaneously by treating PR comments as trusted text. What minimum-viable detection schema for code-review surfaces (titles, descriptions, comments, review threads) should ship with every CI/CD-integrated AI agent by default?
- With OCSF v1.8.0 adding `privilege_info` and `service_privilege_analysis` objects mapped to MITRE ATT&CK, can SOC teams reliably detect compromised model-API service accounts using the same unused-privilege analysis already deployed for human identities, or do AI-specific privilege baselines need to be built from scratch?
- MITRE ATT&CK v19 shipped on April 28 2026 with the Defense Evasion split (Stealth TA0005 / Defense Impairment TA0112) and new AI techniques T1682/T1683/T1684. With ATT&CK v19 in production, what is the right regression-testing cadence to keep SIEM rule libraries with ATLAS->ATT&CK cross-mappings aligned, and how should detection coverage for adversary "intent" (Stealth vs. Impairment) be scored separately for AI-specific evasion (jailbreak suffix injection vs. guardrail-disable techniques)?
- ATT&CK Campaign C0062 (GTG-1002 / Anthropic AI-orchestrated Campaign) demonstrates that account-level behavioral telemetry (sustained agent throughput, multi-target coordination, persona-bypass prompts) is necessary alongside content classifiers. What public detection schema -- agent provider, customer SOC, or shared community ruleset -- best captures the cross-account signals that surfaced GTG-1002, and how do organizations that operate inside a model provider's tenant (rather than running their own) gain the equivalent visibility?
- The "lethal trifecta" framing (private data + untrusted content + exfiltration vector) treats detection as a downstream control behind structural defenses. For agent deployments where all three legs are unavoidable (e.g., enterprise email assistants), what minimum-viable trifecta-aware detection-rule pack should ship with the agent platform, and how should success be measured beyond per-message classifier accuracy?
- JALMBench (ICLR 2026) shows audio-originated jailbreaks evade defenses adapted from vision-language models. For voice agents, multimodal copilots, and transcription pipelines, what detection layer is the right one to add audio-side classifiers -- pre-transcription on raw audio, post-transcription on text, or both -- and what latency budget is realistic for a sub-second voice interaction?
- The VentureBeat April 2026 enterprise survey shows only 21% of enterprises have runtime visibility into AI agents and only 5% are confident they can contain a compromised one. What is the minimum-viable runtime telemetry surface (tool calls, arguments, identity, data egress) that must ship with every enterprise AI agent platform by default, and how should regulators or insurers measure runtime-visibility maturity rather than self-reported control attestation?
- CrowdStrike's 2026 Global Threat Report shows 90+ organizations with prompt-injection abuse but does not publish per-organization attack-pattern signatures. With Microsoft Agent 365 Runtime Protection (Apr 30 2026) shipping with behavioral baselines rather than static rules, can detection vendors converge on a shared, anonymized incident-pattern feed (analogous to OSINT IOC sharing for traditional malware) so that 90+ org compromises become reusable detection content rather than per-vendor proprietary insight?
- Anthropic classified Comment and Control as CVSS 9.4 Critical while Google and GitHub paid sub-$2k bounties. With cross-vendor severity scoring this divergent for the same root-cause class (untrusted text into agent prompt without sanitization), what shared scoring rubric should the AI security community adopt so that 12.2.3 detection prioritization can be benchmarked across products?
- The May 2026 Semantic Kernel CVEs (CVE-2026-26030 Python `eval()` bypass and CVE-2026-25592 .NET `DownloadFileAsync` sandbox escape) close the loop from prompt to host RCE through framework-supplied tools rather than user-installed plugins. What is the right baseline telemetry surface for framework-shipped tool surfaces so that SIEM rules can flag dangerous patterns (unexpected `os`-module loads, `__subclasses__` traversal, arbitrary file writes from agent hosts) even before a specific CVE is published?
- "The Attacker Moves Second" (arxiv:2510.09023) showed adaptive attacks bypass 12 published prompt-injection defenses at >90% success. If single-layer detection is structurally unreliable, how should detection programs measure marginal detection value across stacked layers (signature + classifier + behavioral + session trajectory) instead of standalone per-tool accuracy that vendors typically report?
- Meta's October 2025 "Agents Rule of Two" frames safe agent sessions as having at most two of {untrusted input, sensitive data, state change/external comms}. How should 12.2.3 detection coverage be weighted differently for agents that satisfy all three legs versus agents that satisfy two, and can the rule-of-two scoring be operationalized as a runtime guardrail rather than only a deployment-time review check?
- VentureBeat's April 2026 analysis of CrowdStrike, Cisco, and Palo Alto Networks RSAC 2026 announcements found that none shipped an agent behavioral baseline. If three top SOC platforms cannot supply per-tenant behavioral baselines for 12.2.3/12.2.5/12.2.2 detection out of the box, what minimum-viable agent-baseline schema (sustained throughput, tool-call frequency by category, identity reuse across sessions, data-egress fanout) should the AISVS community publish so detection programs can build comparable baselines on top of these platforms in 2026?
- EchoLeak (CVE-2025-32711) demonstrated that a chain of below-threshold primitives can defeat every individual production guardrail in Microsoft 365 Copilot. What detection-rule schema captures *chains* of weakly-flagged events crossing trust boundaries (classifier hits + link-redaction near-misses + auto-render + outbound proxy) without exploding false-positive volume, and how should that schema be shared across vendors so the next chained zero-click is not rediscovered case-by-case?
- Sysdig's measurement that PraisonAI CVE-2026-44338 was scanned within 3h44m of disclosure compresses the patch-versus-detect window to a few hours for open-source LLM agent frameworks. What is the minimum-viable rapid-detection content pack (scanner user-agent IOCs, default-route signatures, agent-control-plane unauth indicators) that should ship with every open-source agent framework release, and who -- framework maintainers, CSP marketplaces, or the AISVS community -- should own the cross-vendor IOC feed?
- Vassilev's June 2026 NIST proof establishes that no finite guardrail set is bypass-proof, which makes "mean time to detect a novel bypass" and "mean time to harden after discovery" the metrics that actually bound 12.2.1 risk. What instrumentation lets a detection program measure those two intervals in production rather than asserting a static detection rate, and what is a defensible target for each given attacker iteration speeds documented in "The Attacker Moves Second"?
- The Promptware Kill Chain shows most documented attacks traverse four or more stages, yet most detection investment concentrates on stage 1 (injection signatures). What is the right way to score 12.2.2/12.2.3 coverage *per kill-chain stage* (initial access, privilege escalation, reconnaissance, persistence, C2, lateral movement, actions-on-objective) so that under-instrumented stages like memory/retrieval persistence and agent lateral movement become visible as coverage gaps rather than silent blind spots?
- CSA's April 2026 finding that 82% of enterprises have unknown AI agents but only 21% have formal decommissioning processes is a structural mismatch for 12.2.3 detection scope. What discovery techniques (network telemetry, identity-provider analysis, endpoint sensor signatures, egress-traffic clustering) most reliably surface shadow agents at the speed at which they are being deployed, and how should detection programs measure shadow-agent dwell time as a leading indicator rather than only a backward-looking statistic?
