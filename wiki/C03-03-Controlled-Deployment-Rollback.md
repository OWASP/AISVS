# C3.3: Controlled Deployment & Rollback

> [Back to C03 Index](C03-Model-Lifecycle-Management.md)
> **Last Researched:** 2026-03-29

## Purpose

Model deployments must be controlled, monitored, and reversible. This section covers deployment-time integrity verification, gradual rollout mechanisms with automated rollback, atomic state restoration, and emergency shutdown capabilities that cascade across the full agent stack.

The GPT-4o sycophancy incident (April 2025) is a cautionary example: a system prompt change deployed to 100% of traffic without canary release caused the model to endorse harmful statements for three days before rollback -- with social media serving as the primary alerting system. The Model Namespace Reuse attack (Unit 42, 2025) demonstrated that major cloud platforms deployed Hugging Face models by name without integrity verification, enabling remote code execution. These incidents underscore why deployment-time signature checks, staged rollouts, and rapid shutdown capabilities are essential controls.

| **3.3.3** | **Verify that** emergency model shutdown capabilities can disable model endpoints within a pre-defined response time. | 3 | Partial rollback leaving inconsistent state (e.g., new weights with old safety model); cascading failures from version mismatches between model components. | Execute a rollback and verify that all components (weights, config, adapters, safety models) revert to the previous known-good version. Check that rollback is tested as part of deployment runbooks. | Atomic rollback of multi-artifact model stacks (base model + adapter + safety model + config) is harder than single-artifact rollback. Model serving platforms vary in support--some require custom orchestration. |
