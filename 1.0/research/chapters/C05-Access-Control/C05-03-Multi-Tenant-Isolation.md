# C5.3: Multi-Tenant Isolation

> [Back to C05 Index](C05-Access-Control.md)
> **Last Researched:** 2026-06-20

## Purpose

Ensure logical and cryptographic isolation between tenants in shared AI infrastructure. Multi-tenant AI deployments share GPU clusters, model-serving endpoints, vector databases, and inference caches across tenants. Without rigorous isolation, cross-tenant data leakage can occur through shared KV-caches, embedding stores, network segments, and even GPU memory residuals. Recent research has demonstrated that timing side channels in shared inference infrastructure can reconstruct other tenants' prompts with 95-99% accuracy, making KV-cache isolation a critical and rapidly evolving concern. As of April 2026, new attack techniques like RL-enhanced prompt leakage (RL2eak, OptiLeak) have further reduced the cost and effort of exploiting shared caches, container escape vulnerabilities in NVIDIA Container Toolkit (CVE-2024-0132, CVE-2025-23266) have underscored the systemic fragility of GPU isolation in shared environments, and CPU-level cache side channels (Spill The Beans, arxiv 2505.00817) have shown that token-level leakage is possible even without GPU-side cache sharing. Cloud-management-plane vulnerabilities remain just as impactful as data-plane attacks: CVE-2026-20965 (Cymulate Research Labs, reported Aug 2025, patched in Windows Admin Center Azure Extension v0.70.00 on January 13, 2026) let a local admin on any Azure VM or Arc-joined machine bypass Azure AD SSO token validation and pivot to every other WAC-installed machine in the same tenant — a textbook demonstration that tenant isolation failures in shared management tooling have the same blast radius as GPU-layer escapes, and the April 2026 Patch Tuesday added CVE-2026-32196 (Windows Admin Center spoofing) to the same attack surface. The same Patch Tuesday (163–168 CVEs depending on count, including two actively-exploited 0-days CVE-2026-32201 SharePoint spoofing and CVE-2026-33825 Defender) also included **CVE-2026-27928**, a Windows Hello security-feature bypass caused by improper input validation; NVD lists it as CVSS 8.7 with Scope:Changed (`S:C`), so it is still a useful identity-control example even though it is not an Azure tenant-isolation CVE. The `S:C` scope-changed classification is the one operators should watch closely on shared-infrastructure procurement checklists: it signals that exploitation can affect a security authority beyond the initially vulnerable component, the same kind of boundary-crossing pattern that NVIDIAScape and the WAC CVE instantiated in different layers of the stack. On the threat-taxonomy side, MITRE ATLAS has continued expanding through v5.6.0 (April 30, 2026); the v5.1.0 changelog baseline documented **16 tactics, 84 techniques, 56 sub-techniques, 32 mitigations, and 42 case studies**, with later 2026 releases adding agent supply-chain, cost-harvesting, AI-service-proxy, and deepfake-assisted phishing coverage — "Exfiltration via AI Inference API" and "LLM Data Leakage" techniques both apply directly to shared-tenancy inference surfaces. The OWASP LLM Top 10 2025 explicitly calls out multi-tenant isolation gaps under LLM02 (Sensitive Information Disclosure) and LLM04 (Data and Model Poisoning) -- the reference mitigation for a shared vector database with only application-layer tenant filtering is hard cryptographic namespace segregation per tenant at the database level, because a crafted embedding payload can otherwise pull another tenant's chunks into the retrieval context. On the defensive side, NVIDIA Blackwell B200 confidential computing now provides encrypted NVLink/NVSwitch GPU-to-GPU communication with near-native inference performance, and multiple new KV-cache defense frameworks (entropy-based obfuscation, CacheSolidarity, KV-Cloak) offer layered protection against timing side channels. At KubeCon Europe 2026, NVIDIA announced deep Kata Containers integration with DRA (Dynamic Resource Allocation) and CDI for GPU-aware confidential containers, providing a "glide path" from VM-level isolation to full confidential computing with encrypted and measured VM memory. Enterprise KV-cache management is also maturing: LMCache has emerged as a widely-adopted open-source KV caching layer (integrated with vLLM, SGLang, Red Hat, CoreWeave, Google Cloud) that supports cache offloading and prefill-decode disaggregation at scale -- though its tenant isolation story still depends on the underlying engine's salt-based partitioning. On the network isolation front, NVIDIA's BlueField Astra architecture (announced for Vera Rubin NVL72) anchors security enforcement in the BlueField-4 DPU rather than the host OS, extending a unified trust boundary across both front-end (North-South) and AI fabric (East-West) networking domains -- tenant workloads use SuperNIC for data movement but have zero access to management functions, closing a longstanding gap where host-OS compromise could bypass network isolation. Regulated workloads push the isolation bar further still: DoD Impact Level 5 requires physical separation from non-DoD tenants on top of FedRAMP High, and in March 2026 Oracle announced DISA IL5 and FedRAMP High authorization for OCI Generative AI and Exadata Cloud@Customer, making GenAI inference with enforced physical tenant separation a concrete procurement option rather than a paper requirement. On March 3, 2026 Corvex announced the first verified production deployment of NVIDIA HGX B200 confidential computing for multi-tenant AI, combining Intel TDX on the CPU side with NVIDIA Confidential Computing on the GPU side, encrypted NVLink/NVSwitch fabric, and CPU+GPU remote attestation through Intel Trust Authority -- this flips GPU confidential computing from a lab demo into a purchasable production primitive. Nine days later (March 12, 2026) Corvex followed up with Secure Model Weights, which keeps model weights encrypted until they are decrypted inside the GPU TEE using post-quantum Kyber-768 key exchange and open-source Confidential Containers, addressing the long-standing concern that sensitive model IP is exposed to the cloud host during load. NVIDIAScape (CVE-2025-23266, CVSS 9.0) remains the canonical example of why host-layer isolation alone is insufficient: it affects roughly 37% of cloud environments running GPU workloads and allows a single container breakout to reach all co-tenants' data and model weights, which is now driving enterprise adoption of purpose-built hardened runtimes such as Edera's zone-based isolation (announced KVM support and the `ocirender` Rust library at KubeCon Europe 2026) alongside Kata Containers with NVIDIA DRA/CDI integration. The inference layer itself continues to produce tenant-isolation breaks at the application level: CVE-2026-22778 (disclosed February 2, 2026, CVSS 9.8) chains a PIL exception heap-address leak with a JPEG2000 heap overflow in the FFmpeg/OpenCV path used by vLLM's multimodal endpoints, allowing an unauthenticated attacker to send a crafted video URL to `/v1/chat/completions` or `/v1/invocations` and achieve arbitrary code execution on the serving host -- affecting vLLM 0.8.3 through 0.14.0 inclusive and patched in v0.14.1, with the practical consequence that a single malicious tenant on a shared vLLM deployment can reach every other tenant's prompts, KV cache, and model weights without touching the GPU isolation layer at all; CVE-2026-22807 and CVE-2026-27893 extend the same trust boundary problem, where vLLM's loader honored `trust_remote_code` paths from Hugging Face `auto_map` dynamic modules even when operators had explicitly opted out, turning any tenant-uploadable model repository into an RCE primitive (CVE-2026-27893 was the specific incomplete-fix bypass — `nemotron_vl.py` and `kimi_k25.py` both retained hardcoded `trust_remote_code=True` paths that overrode the global `--trust-remote-code=False` flag, CVSS 8.8, patched in vLLM 0.18.0 on March 26, 2026). The problem class is broader than multimodal: **CVE-2025-62164** (CVSS 8.8, patched in vLLM 0.11.1) exposes the Completions API `prompt_embeds` parameter to a `torch.load()` deserialization flaw where PyTorch 2.8.0+ disabled sparse-tensor integrity checks by default, letting any API-key-holding tenant write out of bounds and achieve RCE on the serving host by uploading a crafted serialized tensor; **CVE-2026-24779** (MediaConnector SSRF, patched 0.15.1) and **CVE-2026-25960** (CVSS 5.4, patched 0.17.0) show the same URL-parsing inconsistency pattern — `urllib3.util.parse_url()` validates hostnames while `aiohttp`/`yarl` executes the request, and backslashes are treated differently by each parser so `https[:]//allowed.example\@attacker.tld/` passes validation but resolves to the attacker's host, letting any tenant pivot to internal services and metadata endpoints. On April 3, 2026 vLLM published three more advisories in a single batch: **GHSA-pq5c-rjhq-qp7p** (Moderate, DoS via `VideoMediaIO.load_base64()` parsing thousands of comma-separated base64 JPEG frames), **GHSA-pf3h-qjgv-vcpr** (Moderate, SSRF via `download_bytes_from_url` processing batch-input JSON with user-controlled `file_url`), and **GHSA-3mwp-wvh9-7528** (Moderate, OOM DoS via the unbounded `n` parameter in the OpenAI-compatible server) — the cumulative pattern across the last quarter is that media/URL-parsing and deserialization paths are the dominant inference-layer attack surface for multi-tenant serving stacks, not novel model-layer attacks. Outside vLLM, **CVE-2026-27940** (llama.cpp integer overflow in `gguf_init_from_file_impl()` in `gguf.cpp`, CVSS 7.8, published March 13, 2026, patched in commit b8146) is an incomplete-fix bypass of CVE-2025-53630: the size calculation still wraps under certain GGUF tensor headers, producing an undersized heap allocation that the subsequent `fread()` overflows by 528+ attacker-controlled bytes, which matters for any shared-inference platform that accepts tenant-uploaded GGUF weights (on-prem llama.cpp-based private-cloud offerings and mobile-edge inference wrappers in particular). On the standards side, **NIST CAISI** (Center for AI Standards and Innovation) launched the **AI Agent Standards Initiative** on February 17, 2026, organized around three pillars (industry-led standards via ISO/IEC JTC 1, community-led open-source agent protocols co-invested with NSF, and fundamental research in AI agent security and identity infrastructure) -- April 2026 virtual workshops are soliciting sector-specific barriers to multi-tenant agentic AI adoption in healthcare, finance, and education, and are explicitly formalizing authentication, authorization, and governance expectations for enterprise agent deployments. In parallel, **NIST COSAiS** (Control Overlays for Securing AI Systems, building on SP 800-53 plus SP 800-218A, Draft NIST AI 800-1, and NIST AI 100-2e2025) published its first discussion-draft annotated outline on January 8, 2026 covering Predictive AI adaptation and fine-tuning; the comment window closed February 13, 2026, and the single-agent and multi-agent overlays -- which directly address multi-tenant agent cooperation and per-tenant authorization for autonomous planning systems -- are on track for mid-to-late 2026 discussion-draft release, with agencies unable to wait for final publication being offered interim compensating-control guidance. NVIDIA's developer-blog guidance "Structuring Applications to Secure the KV Cache" (April 29, 2025) is now the most practical per-request mitigation pattern operators can adopt today without reworking the serving stack: order dynamically constructed prompts as **system prompt → non-guessable, periodically rotated per-session or per-tenant identifier → tool/plugin augmentation context → validated user prompt**, set maximum length limits, avoid repeating user input across prompt sections, and pair prefix caching with monitoring for enumeration-style query patterns (repeated queries with minor variations) combined with rate limiting -- the same blog explicitly recommends considering "isolating KV caches across tenants entirely" via session-ID cache keys or disabling prefix caching in high-risk contexts, which slots in cleanly above vLLM `cache_salt` as a prompt-layer defense-in-depth.

> **Scope note:** Controls in this section govern infrastructure-level tenant isolation -- network policies, namespace segregation, per-tenant encryption, and KV-cache partitioning. Query-time tenant filtering for retrieval lives in [C08-01 Access Controls for Memory and RAG](../C08-Memory-and-Embeddings/C08-01-Access-Controls-Memory-RAG.md); per-tenant authorization policy is in [C5.2 AI Resource Authorization](C05-02-AI-Resource-Authorization-Classification.md).

## Current Research Notes

- **[2026-05] KV-cache partitioning:** vLLM v0.20.0 documents cache salts as part of the prefix-cache block hash material for multi-tenant isolation and makes `sha256` the default hash algorithm as of v0.11; the same guidance warns that non-cryptographic `xxhash` can theoretically cause undefined behavior or leak private information in multi-tenant deployments.
  For 5.3.1, auditors should verify `cache_salt` is set from authenticated tenant or session identity, `--prefix-caching-hash-algo` is `sha256` or `sha256_cbor`, and high-sensitivity tenants either use per-tenant cache keys or have prefix caching disabled.
  See [vLLM prefix caching](https://docs.vllm.ai/en/v0.20.0/design/prefix_caching/).
- **[2026-05] Serving-stack patch floor:** vLLM's practical multi-tenant baseline has moved to v0.20.0 or later. v0.14.1 patched the critical video-processing RCE and MediaConnector SSRF, v0.17.0 patched the backslash-parser SSRF bypass, v0.18.0 patched CVE-2026-27893 hardcoded `trust_remote_code=True`, v0.19.0 patched the April 3 DoS/SSRF/OOM advisory batch, and v0.20.0 patched GHSA-83vm-p52w-f9pw where one request with a penalty parameter can crash `extract_hidden_states` speculative decoding.
  While rollout lags, gateways should reject or tightly constrain tenant-controlled `video_url`, batch-input `file_url`, `prompt_embeds`, oversized `n`, and speculative-decoding penalty combinations.
  See [vLLM security advisories](https://github.com/vllm-project/vllm/security/advisories) and [GHSA-83vm-p52w-f9pw](https://github.com/vllm-project/vllm/security/advisories/GHSA-83vm-p52w-f9pw).
- **[2026-05] Cache sharing scale vs. isolation:** LMCache's January 2026 P2P CPU-memory sharing work shows why cache reuse is operationally attractive: in its two-instance demonstration, peer retrieval cut mean TTFT from 2.028s to 0.490s and total round time from 38.323s to 7.964s.
  That same architecture expands the isolation boundary to worker registration, heartbeat, controller metadata, peer KV transfer paths, and cache-key derivation.
  For 5.3.1, treat those as tenant-scoped security surfaces, not just performance plumbing. See [LMCache P2P sharing](https://blog.lmcache.ai/en/2026/01/21/p2p-1/) and [LMCache integration](https://docs.lmcache.ai/developer_guide/integration.html).
- **[2026-05] Threat taxonomy and standards:** MITRE ATLAS v5.6.0 shipped on April 30, 2026, and the March v5.5.0 release added agent supply-chain and cost-harvesting techniques that matter for shared agent platforms.
  NIST CAISI's AI Agent Standards Initiative now explicitly tracks agent authentication and identity infrastructure for secure human-agent and multi-agent interactions, while COSAiS is turning SP 800-53, SP 800-218A, Draft NIST AI 800-1, and NIST AI 100-2e2025 into implementation-focused overlays.
  For this page, the practical mapping is straightforward: tenant isolation evidence should include identity binding, authorization scope, telemetry, segmentation, artifact-signing, and cache/model-state lifecycle controls. See [MITRE ATLAS changelog](https://github.com/mitre-atlas/atlas-data/blob/main/CHANGELOG.md), [NIST AI Agent Standards Initiative](https://www.nist.gov/artificial-intelligence/ai-agent-standards-initiative), [NIST COSAiS](https://csrc.nist.gov/Projects/cosais), and [NIST SP 800-218A](https://csrc.nist.gov/pubs/sp/800/218/a/final).
- **[2026-05] CVE-2026-27928 handling:** NVD and MSRC describe CVE-2026-27928 as a Windows Hello security-feature bypass caused by improper input validation, with CVSS v3.1 scope changed (`S:C`).
  It remains useful as a changed-scope identity-control example, but it should not be treated as an Azure tenant-isolation CVE.
  See [NVD CVE-2026-27928](https://nvd.nist.gov/vuln/detail/CVE-2026-27928) and [MSRC CVE-2026-27928](https://msrc.microsoft.com/update-guide/vulnerability/CVE-2026-27928).
- **[2026-05] vLLM deployment hardening:** vLLM's security guidance now gives auditors concrete multi-tenant checks beyond framework version: inter-node traffic for PyTorch Distributed, KV-cache transfer, and tensor/pipeline/data parallelism is insecure by default unless isolated on trusted networks; firewalls should expose only the API listener and keep internal communication ports tenant-private.
  The same guidance warns that `--api-key` protects the OpenAI-compatible `/v1` endpoints only, while endpoints such as `/invocations`, `/pause`, `/resume`, `/tokenize`, `/detokenize`, and development-only cache reset/RPC endpoints need reverse-proxy and network controls.
  For 5.3.1 evidence, ask for listener bindings, firewall rules, endpoint exposure tests, and proof that tenant-facing deployments do not expose internal control endpoints. See [vLLM Security](https://docs.vllm.ai/en/v0.20.0/usage/security/).
- **[2026-05] Dynamic LoRA loading boundary:** vLLM's LoRA documentation says runtime resolver plugins can load adapters from local files or Hugging Face Hub, but enabling remote downloads is not intended for production.
  In shared serving clusters, treat `VLLM_ALLOW_RUNTIME_LORA_UPDATING`, `VLLM_PLUGINS`, resolver cache directories, and S3/Hugging Face repository allowlists as tenant-bound security controls rather than convenience settings.
  A practical 5.3.1 test is to attempt cross-tenant adapter selection, adapter replacement through `load_inplace`, and path traversal or repository-confusion cases while confirming every adapter is signed and mapped to the authenticated tenant. See [vLLM LoRA adapters](https://docs.vllm.ai/en/v0.20.0/features/lora/).
- **[2026-05] Selective prefix-cache isolation:** CacheSolidarity (arxiv 2603.10726, March 2026) proposes a middle ground between disabling automatic prefix caching and leaving cross-user cache reuse open: track owner metadata on cache entries, flag suspicious cross-user reuse, and selectively isolate prefixes when hit/miss timing becomes distinguishable.
  This is not yet a mainstream serving-engine control, so auditors should not accept it as deployed unless there is implementation evidence, but it is useful as a design pattern for vendors that need 5.3.1 assurance without sacrificing all cache efficiency.
  See [CacheSolidarity](https://arxiv.org/abs/2603.10726) and [vLLM prefix caching](https://docs.vllm.ai/en/v0.20.0/design/prefix_caching/).
- **[2026-05] ATLAS and agent standards drift:** MITRE ATLAS v5.6.1 was a small schema-fix release on May 5, 2026, while v5.6.0 on April 30 added `Acquire Public AI Artifacts: AI Agent Configuration`, `Search Open Websites/Domains: Code Repositories`, and `Phishing: Deepfake-Assisted Phishing`; the March v5.5.0 release added AI agent tool poisoning, AI service proxy, and cost-harvesting techniques plus updates to AI telemetry logging and component segmentation mitigations.
  The ATLAS program also moved to a monthly release cadence in 2026, so auditors evaluating multi-tenant platforms should expect agent-supply-chain and AI-service-proxy techniques to keep evolving and should refresh ATLAS-mapped tenant-isolation evidence at least quarterly. For multi-tenant platforms, the important control mapping is identity-bound agent/tool execution, tenant-scoped telemetry, signed agent/tool artifacts, and segmentation of agent components from shared model-serving and cache planes.
  See [MITRE ATLAS changelog](https://github.com/mitre-atlas/atlas-data/blob/main/CHANGELOG.md), [MITRE ATLAS monthly cadence announcement](https://ctid.mitre.org/blog/2026/05/06/secure-ai-v2-release), and [NIST AI Agent Standards Initiative](https://www.nist.gov/artificial-intelligence/ai-agent-standards-initiative).
- **[2026-05] vLLM v0.21.0 patch floor and the post-0.20 baseline:** vLLM v0.21.0 shipped on May 15, 2026 with KV-offload integration into the Hybrid Memory Allocator, FlashInfer top-k/top-p sampling on by default, and a TOKENSPEED_MLA backend for Blackwell GPUs (DeepSeek-R1, Kimi-K25), and it deprecates Transformers v4.
  The release itself did not bundle new tenant-isolation CVE fixes, but operators using KV offload to host CPU memory or NVMe should now treat the offload tier as part of the tenant cache boundary — the same `cache_salt`, prefix-cache hash, and per-tenant key derivation requirements apply to spilled blocks as to in-GPU blocks, and any external KV store (LMCache, NVMe pool, Redis-backed CPU cache) added underneath needs its own access control and tenant scoping.
  Two earlier 2026 advisories are now worth re-checking on this baseline: **CVE-2026-22773 / GHSA-grg2-63fw-f2qr** (Moderate, patched in 0.12.0) crashed the Idefics3 multimodal path with a 1x1-pixel image and is fixed under any 0.20+ baseline but illustrates that even non-RCE multimodal-ingestion bugs are full-tenant DoS primitives in a shared deployment, and **CVE-2025-66448 / GHSA-8fr4-5q9j-m8gm** (High, CVSS 7.1, patched in 0.11.1, December 1, 2025) is the earlier ancestor of the `auto_map` trust-remote-code family already cited on this page: it specifically resolves the `Nemotron_Nano_VL_Config` class via `get_class_from_dynamic_module(...)` even with `trust_remote_code=False`, so any tenant-uploadable model whose `config.json` references a remote `auto_map` could already trigger host RCE in vLLM ≤ 0.11.0 — auditors confirming patch posture should check both this CVE and the later CVE-2026-22807 / CVE-2026-27893 incomplete-fix chain.
  See [vLLM v0.21.0 release notes](https://github.com/vllm-project/vllm/releases) and [vLLM security advisories](https://github.com/vllm-project/vllm/security/advisories).
- **[2026-05] SGLang unpatched pickle-RCE pattern:** SGLang now carries **CVE-2026-3059** and **CVE-2026-3060** (both CVSS 9.8, unauthenticated RCE via `pickle.loads()` on ZMQ broker traffic bound to all interfaces in multimodal and prefill-decode disaggregation modes) and **CVE-2026-3989** (CVSS 7.8, RCE through the `replay_request_dump.py` analysis script), disclosed publicly on March 12, 2026 with CERT/CC coordination tracked as VU#665416 — and as of May 2026 the SGLang project has not shipped an official patch.
  In multi-tenant deployments this is a critical 5.3.1 finding: any operator running SGLang with multimodal or disaggregated serving enabled on a shared cluster is one network reachability check away from a co-tenant RCE pivot, and the only mitigations are to disable the affected modules, bind the ZMQ broker to loopback only, or move sensitive tenants off SGLang entirely until the project patches the deserialization paths.
  See [CERT/CC VU#665416](https://kb.cert.org/vuls/id/665416), [SGLang RCE writeup](https://chocapikk.com/posts/2026/sglang-pickle-rce/), and [SGLang AI infrastructure RCE analysis](https://www.orca.security/resources/blog/sglang-llm-framework-rce-vulnerabilities/).
- **[2026-05] PyTorch `weights_only=True` no longer a hard safety boundary:** **CVE-2026-24747 / GHSA-63cw-57p8-fm3p** documents memory corruption and potential RCE in PyTorch's `torch.load(..., weights_only=True)` unpickler, caused by missing validation of pickle `SETITEM`/`SETITEMS` opcodes on non-dictionary types and a storage-size mismatch between declared element count and archive contents; the fix landed in PyTorch 2.10.0.
  Practical impact for shared inference is significant: many operators have treated `weights_only=True` as the safe loader for tenant-supplied checkpoints, and this CVE means that assumption no longer holds on PyTorch < 2.10. For 5.3.1, every shared serving stack that accepts tenant `.pt`/`.pth`/`.ckpt` uploads (HuggingFace bridges, customer fine-tuning pipelines, third-party adapter marketplaces, async policy servers) needs to either upgrade PyTorch to ≥ 2.10.0 or sandbox the loader off the production serving host.
  See [PyTorch GHSA-63cw-57p8-fm3p](https://github.com/pytorch/pytorch/security/advisories/GHSA-63cw-57p8-fm3p) and [NVD CVE-2026-24747](https://nvd.nist.gov/vuln/detail/CVE-2026-24747).
- **[2026-05] ZMQ/WebSocket pickle RCE is the new shared-host pattern:** Alongside SGLang, **LightLLM CVE-2026-26220** (CVSS 9.3) calls `pickle.loads()` on raw WebSocket frames in its prefill-decode disaggregation system with no authentication, and **ktransformers CVE-2026-26210** binds a ZMQ ROUTER socket on all interfaces and feeds attacker-controlled bytes into `pickle.loads()` on worker threads — both yield unauthenticated co-tenant RCE on the serving host.
  **LeRobot CVE-2026-25874** is the same class of bug in Hugging Face's async-inference PolicyServer (gRPC + pickle). The composite signal for 5.3.1 reviewers is that any tenant-isolation evidence for "we deserialize on this network plane" must enumerate every pickle/ZMQ/WebSocket/gRPC entrypoint, document its auth posture, and either bind it to loopback or wrap it in mutual-auth — not just check the OpenAI-compatible `/v1` listener.
  See [LightLLM RCE writeup](https://chocapikk.com/posts/2026/lightllm-pickle-rce/), [ktransformers RCE writeup](https://chocapikk.com/posts/2026/ktransformers-pickle-rce/), and [LeRobot CVE-2026-25874 advisory](https://www.resecurity.com/blog/article/cve-2026-25874-hugging-face-lerobot-unauthenticated-rce-via-pickle-deserialization).
- **[2026-05] LMCache AMD MI300X benchmarking and cache-key consistency:** LMCache's May 12, 2026 multi-turn agentic-workload benchmark on 2× AMD MI300X (MiniMax-M2.5, 230 GB FP8 MoE, 739 anonymized Claude Code traces) shows that at high concurrency (32 users, ~100k context) CPU-DRAM offload via LMCache delivers ~3.0x lower mean TTFT, ~2.1x lower p95, and ~2.3x more sustained requests vs. HBM-only prefix caching, while smaller working sets still favor HBM caching.
  Two operational notes matter directly for 5.3.1: (1) the post explicitly calls out `PYTHONHASHSEED=0` as **mandatory** for tensor-parallel cache-key consistency — without it, identical prompts hash differently across TP workers, which both kills cache hit rate and means any per-tenant or per-session salting scheme that relies on those Python hashes will silently produce inconsistent block keys; (2) AMD ROCm and CPU-DRAM tiering shift the cache trust boundary onto host memory and the NUMA topology, so DRAM zeroing, NUMA-bound tenants, and CPU-side side-channel hardening (Spill The Beans, USENIX Security 2025) become part of the audit scope.
  See [LMCache MI300X benchmark](https://blog.lmcache.ai/en/2026/05/12/benchmarking-lmcache-for-multi-turn-agentic-workloads-on-amd-mi300x/) and [LMCache technical report](https://lmcache.ai/tech_report.pdf).
- **[2026-06] 5.3.2 co-located-execution side channels — the hardware-isolation gap is now well-documented:** The research record makes 5.3.2 the hardest requirement on this page to satisfy with anything short of physical separation. MIG does not partition the last-level GPU TLB or PCIe bandwidth (TunneLs for Bootlegging, CCS 2023 — 31 kbps cross-instance covert channel at 99.8%); NVLink leaks through contention timing and performance counters even across VMs (NVBleed, arxiv 2503.17847 — >70 kbps, and >83% F1 from timing alone after counter lockdown); and GDDR6 Rowhammer can corrupt GPU page tables for arbitrary read/write and a host root shell with the IOMMU enabled (GPUBreach, IEEE S&P 2026, disclosed to NVIDIA Nov 11 2025; concurrent GDDRHammer and GeForge). MPS provides no security isolation and should never co-locate distinct tenants.
  For 5.3.2 evidence, auditors should confirm the scheduler never packs two tenants on one physical GPU for high-sensitivity workloads, enable NVIDIA Confidential Computing with composite CPU+GPU attestation (Intel TDX / AMD SEV-SNP / AWS Nitro Enclaves on the CPU side) gating key release, turn on ECC per NVIDIA's Rowhammer security notice, patch the NVIDIA driver against the GPUBreach chain, and run a TLB/NVLink-contention canary to measure residual cross-instance bandwidth. The honest conclusion is that Confidential Computing closes memory confidentiality but not interconnect timing side channels, so dedicated per-tenant GPUs remain the only fully robust posture.
  See [TunneLs for Bootlegging](https://dl.acm.org/doi/10.1145/3576915.3616672), [NVBleed](https://arxiv.org/abs/2503.17847), [GPUBreach](https://gpubreach.ca/), and [gpu-tlb probe](https://github.com/0x5ec1ab/gpu-tlb).
- **[2026-06] SGLang deserialization cluster widens and stays unpatched:** A second SGLang RCE, **CVE-2026-7304** (CERT/CC VU#777338, disclosed May 18, 2026), passes attacker-controlled bytes straight into `dill.loads()` from the `CustomLogitProcessor.from_str()` path — an unauthenticated POST to `/generate` with a crafted `custom_logit_processor` field runs arbitrary code as the inference-process user, with access to loaded model weights, before any other application logic. Combined with the earlier `pickle.loads()` ZMQ bugs (CVE-2026-3059/-3060, CVSS 9.8), this means SGLang now carries multiple unauthenticated co-tenant RCE primitives that remain present on `main` across every release from v0.5.11 through v0.5.12.post1; CERT/CC still lists the vendor status as "no statement received," and a proposed loopback-bind + msgpack patch has not been merged. One important scoping refinement for 5.3.1 audits: CVE-2026-3059/-3060 only fire when the multimodal-generation or prefill-decode disaggregation modules are explicitly enabled, so a default text-only SGLang deployment is not exposed to those two — but CVE-2026-7304 needs only the custom-logit-processor feature, which is a more commonly enabled path. The practical 5.3.1 finding for any shared cluster running SGLang is unchanged: disable the affected modules, bind ZMQ/control sockets to loopback, or move sensitive tenants off SGLang until upstream patches land.
  See [CERT/CC VU#777338](https://kb.cert.org/vuls/id/777338), [SGLang CVE-2026-7304 analysis](https://blog.securelayer7.net/cve-2026-7304-sglang-unauthenticated-rce/), and [Orca "Pickle in the Pipeline"](https://orca.security/resources/blog/sglang-llm-framework-rce-vulnerabilities/).
- **[2026-06] Standards currency:** NIST COSAiS single-agent and multi-agent SP 800-53 control overlays — the ones that bear directly on per-tenant authorization and multi-agent cooperation in shared platforms — remain pre-discussion-draft as of June 2026 (annotated outline January 2026, Predictive-AI-adaptation draft only), so 5.3.1/5.3.2 evidence should still map to the published SP 800-53, SP 800-218A, and NIST AI 100-2e2025 baselines rather than waiting on the agentic overlays. NVIDIA's latest Secure-AI guidance and the April 2026 Confidential Computing deployment guide (DU-12302-001 v7.1, Intel TDX) read as a quiet de-emphasis of single-GPU multi-tenant confidential computing in favor of per-VM and dedicated-GPU postures, which reinforces the 5.3.2 conclusion.
  See [NIST COSAiS use cases](https://csrc.nist.gov/Projects/cosais/use-cases) and [NVIDIA CC-for-TDX deployment guide](https://docs.nvidia.com/cc-deployment-guide-tdx.pdf).

## Notable Incidents and Research Signals

- **PROMPTPEEK, NDSS 2025:** The first focused public analysis of prompt leakage through shared KV-cache reuse showed that timing differences in multi-tenant vLLM/SGLang-style serving can let an attacker reconstruct another user's prompt under realistic prior-knowledge assumptions. This remains the core test case for 5.3.1 because it turns a performance optimization into a confidentiality boundary. See [PROMPTPEEK](https://www.ndss-symposium.org/ndss-paper/i-know-what-you-asked-prompt-leakage-via-kv-cache-sharing-in-multi-tenant-llm-serving/).
- **vLLM prompt embedding deserialization, November 2025:** GHSA-mrw7-hf4f-83pf / CVE-2025-62164 shows how a tenant-facing model-server feature can become a shared-host compromise path: user-supplied `prompt_embeds` were loaded with `torch.load()` and could trigger memory corruption through sparse-tensor handling. Even when direct cache isolation is sound, 5.3.1 evidence has to include admission controls for tenant-controlled embeddings, model files, media URLs, and adapter artifacts. See [vLLM advisory GHSA-mrw7-hf4f-83pf](https://github.com/vllm-project/vllm/security/advisories/GHSA-mrw7-hf4f-83pf).
- **Agent/tool supply-chain expansion, March-May 2026:** ATLAS v5.5.0 and v5.6.0 added agent tool poisoning, AI service proxy, cost-harvesting, public AI artifact acquisition, and code-repository discovery techniques. These are not GPU attacks, but they matter for multi-tenant isolation because a poisoned tenant-scoped agent tool or proxy can bridge the same boundaries that model-serving controls try to preserve.
- **Pickle-in-the-pipeline disclosure cluster, March-May 2026:** SGLang's multimodal-generation and prefill-decode disaggregation modules (CVE-2026-3059, CVE-2026-3060 at CVSS 9.8 and CVE-2026-3989 at CVSS 7.8, coordinated through [CERT/CC VU#665416](https://kb.cert.org/vuls/id/665416), public disclosure March 12, 2026), LightLLM's WebSocket disaggregation path (CVE-2026-26220, CVSS 9.3), ktransformers' ZMQ ROUTER (CVE-2026-26210), and Hugging Face LeRobot's async PolicyServer (CVE-2026-25874) all call `pickle.loads()` on attacker-controlled bytes from network surfaces that were never properly authenticated. SGLang remains unpatched as of May 2026. For 5.3.1 reviewers this is the most important multi-tenant cluster of the quarter: any shared deployment that exposes one of these stacks' non-`/v1` ports to tenant networks is a confirmed cross-tenant RCE pivot, not a theoretical one.
- **PyTorch `weights_only=True` failure, May 2026:** [CVE-2026-24747 / GHSA-63cw-57p8-fm3p](https://github.com/pytorch/pytorch/security/advisories/GHSA-63cw-57p8-fm3p) shows that `torch.load(..., weights_only=True)` is no longer a hard safety boundary for tenant-supplied checkpoints — the unpickler missed validation on `SETITEM`/`SETITEMS` opcodes for non-dictionary types and trusted archive-declared storage sizes, allowing memory corruption and potential RCE. Fixed in PyTorch 2.10.0. Every shared-inference platform that previously waved through tenant `.pt`/`.pth`/`.ckpt` uploads on the strength of this flag needs to either upgrade or sandbox the loader.
- **GPUBreach, IEEE S&P 2026 (disclosed to NVIDIA Nov 11, 2025):** University of Toronto researchers used GDDR6 Rowhammer to corrupt GPU page tables and obtain arbitrary GPU memory read/write, then chained NVIDIA-driver memory-safety bugs into a host root shell — and it works **with the IOMMU enabled**, which prior GPU Rowhammer work assumed would block it. Two concurrent S&P 2026 papers, GDDRHammer and GeForge, independently demonstrate GPU page-table corruption via GDDR6 Rowhammer. This is the strongest evidence yet that 5.3.2's "no software-only mitigation" framing is correct: shared GPUs require physical separation or confidential computing plus ECC, not just driver hardening. See [GPUBreach](https://gpubreach.ca/).
- **MIG isolation is incomplete (TunneLs for Bootlegging, CCS 2023):** Reverse-engineering of NVIDIA MIG showed the last-level GPU TLB is shared across all compute units and **not** partitioned by MIG, yielding a 31 kbps cross-instance covert channel at 99.8% accuracy without root privileges; PCIe bandwidth is likewise statically shared. This is the canonical 5.3.2 test case for why MIG alone does not satisfy side-channel isolation. See [TunneLs for Bootlegging](https://dl.acm.org/doi/10.1145/3576915.3616672) and the public [gpu-tlb](https://github.com/0x5ec1ab/gpu-tlb) probe.
- **NVBleed, March 2025:** NVLink contention timing plus accessible performance counters leak across GPUs and across VM instances in the cloud — >70 kbps covert channel and application/3D-character fingerprinting — and the timing-only variant survives counter lockdown at >83% F1, showing that disabling the perf-counter API is not a complete defense for interconnect side channels. See [NVBleed (arXiv 2503.17847)](https://arxiv.org/abs/2503.17847).

## Implementation Maturity

- **5.3.1 KV-cache isolation:** Improving. vLLM exposes `cache_salt`, SHA256 is the default prefix-cache hash as of v0.11, and prompt-structure mitigations are practical today. Selective isolation designs such as CacheSolidarity are promising, but should be treated as research unless the serving provider can show implementation and test evidence.
- **5.3.2 co-located-execution side channels:** Immature and unlikely to mature in software. There is no software-only mitigation in current shared inference servers, and every hardware "isolation" mechanism short of physical separation has a documented microarchitectural leak: MIG leaves the last-level TLB and PCIe bandwidth shared (TunneLs for Bootlegging), MPS provides no security isolation, NVLink leaks through contention timing (NVBleed), and GDDR6 is exposed to Rowhammer page-table corruption (GPUBreach/GDDRHammer/GeForge at IEEE S&P 2026). Confidential Computing (H100/B200 TEE-I/O) closes memory confidentiality and is now the strongest available control, but it does not eliminate contention/timing interconnect side channels and requires composite CPU+GPU attestation plus ECC. The only fully robust posture is dedicated per-tenant GPUs, which sacrifices the utilization economics of shared inference — most operators accept some residual exposure, which should be documented against tenant sensitivity. Verification tooling is largely academic (TLB/contention probes, fingerprinting harnesses) rather than productized.
- **5.3.1 shared model-serving isolation:** Mixed and worsening at the framework layer. Kubernetes namespaces, network policy, reverse proxies, signed artifacts, and patch management are mature controls, but runtime LoRA loading, adapter attestation, cross-tenant batching kernels, and GPU/TEE attestation are still uneven across open-source serving stacks, and the March-May 2026 pickle-deserialization cluster (SGLang, LightLLM, ktransformers, LeRobot) plus PyTorch's `weights_only=True` failure have moved several frameworks from "tenant-isolated by configuration" to "unpatched cross-tenant RCE primitives" in a single quarter. The strongest current posture layers tenant-bound adapters, disabled remote runtime loading, isolated inter-node networks, hardened container or VM runtimes, PyTorch ≥ 2.10.0, and explicit tests for unprotected pickle/ZMQ/WebSocket/gRPC control endpoints — and, for SGLang specifically, either disabling the multimodal and disaggregation modules or moving sensitive tenants off the stack until upstream patches land.
