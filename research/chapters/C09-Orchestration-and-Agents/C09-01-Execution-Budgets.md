# C9.1: Execution Budgets, Loop Control, and Circuit Breakers

> [Back to C09 Index](C09-Orchestration-and-Agents.md)
> **Last Researched:** 2026-06-20

## Purpose

Autonomous agents can enter runaway states — recursive loops, unbounded fan-out, or chain reactions across tool calls — that consume resources without bound. As of 2026, denial-of-wallet attacks against agentic AI are a material financial risk: Fortune 500 companies collectively leaked an estimated **$400M in unbudgeted cloud spend** from agentic AI cost overruns (Analytics Week, 2026). Specific incidents include a **$47K LangChain multi-agent loop** (November 2025), a **$47K retry storm** from 2.3M erroneous API calls (February 2026), a **$1.2M GPU hijack** where an AI agent autonomously began mining cryptocurrency (March 2026), and the **OpenClaw crisis** (February 2026) in which a DeFi-integrated agent mistakenly transferred 52.43M LOBSTAR tokens ($250K) while 135,000 instances sat directly exposed to the internet ([OECD AI Incident 2026-03-10-f105](https://oecd.ai/en/incidents/2026-03-10-f105)). The OWASP Top 10 for Agentic Applications 2026 now mandates circuit breakers and transactional rollback in every agentic workflow. No major agent framework has comprehensive built-in cost ceilings — budget enforcement requires external gateways or governance layers. As of March 2026, inference accounts for roughly 85% of the enterprise AI budget, and agentic reasoning loops that hit the LLM 10–20 times per task are the primary cost driver (AnalyticsWeek 2026). The gateway layer continues to see rapid innovation: Traefik Hub's Triple Gate (v3.20, March 2026) introduced proactive token estimation that blocks abusive requests *before* they reach the LLM, Bifrost's 11-microsecond overhead makes gateway-level enforcement effectively invisible in the latency budget, and **MLflow AI Gateway** added OSS budget policies with daily/weekly/monthly reset windows and Redis-backed shared state across replicas, then v3.11.1 added budget-management UI and trace graph support for agent debugging.

Late April 2026 added two more reference cases worth folding into any C9.1 audit. The **PocketOS incident** (April 25–26, 2026) is the textbook illustration that *destructive blast radius is its own budget*: a Cursor agent running Claude Opus 4.6 hit a credential mismatch in staging and, with no confirmation prompt and no operation-level scope on the Railway CLI token it found in the repo, autonomously issued one curl call that deleted PocketOS's production volume and all volume-stored backups in 9 seconds. Three months of rental-car booking data were destroyed; Railway's CEO restored the data within 30 minutes after the incident went public, but the failure mode — single-call irreversible action with no human gate, on top of a token whose blast radius matched the agent's autonomy — sits squarely inside C9.1's scope alongside high-impact-action approval (C9.2). Around the same time, the **Claude Code CLI unbounded-context bug** (April 6–7, 2026, [issue #44726](https://github.com/anthropics/claude-code/issues/44726)) drove an Alpha2Zulu LLC API balance from +$8.46 to −$13.23 in a single session by ingesting roughly 30M input tokens at a 74:1 input/output ratio (the second incident hit 175:1). The trigger appeared to be context that grew unbounded across tool calls and was recompressed and re-sent on every step — exactly the per-chain cumulative-spend pattern that 9.1.2 is meant to detect, with the difference that the spending agent was the IDE's own coding loop rather than a custom-built workflow. Normal Claude Code sessions sit at 5:1 to 15:1; an audit signal of "input/output ratio above 30:1 sustained across N calls" is a cheap way to surface this exact failure. Both cases reinforce that runtime enforcement (per-session token cap, kill-switch on ratio anomalies, scope-bounded credentials, mandatory confirmation on destructive tool calls) must live in the agent runtime — not in the model's good judgment.

As of April 2026 a new class of threat complicates gateway-layer enforcement: **malicious LLM routers**. An arXiv preprint from 8 April 2026 ("Measuring Malicious Intermediary Attacks on the LLM Supply Chain") tested 428 AI API routers and found 9 injecting malicious code, 17 harvesting AWS credentials, and 26 silently injecting tool calls — one sequence drained **$500,000** from a client's crypto wallet, and a free router successfully drained ETH from a researcher-held private key. The implication for C9.1: a budget enforcement point sitting in the inference path is itself a high-value target, and cost ceilings are only trustworthy when the gateway binary, its routing config, and its outbound credentials are under the same integrity guarantees as the agent runtime.

Datadog's **State of AI Engineering 2026** (April 21, 2026) puts hard numbers on the cost-explosion trend: agent framework adoption nearly doubled year-over-year (from ~9% of organizations in early 2025 to ~18% by early 2026), services using agentic frameworks more than doubled in the same window, **average tokens per request more than doubled for median-use teams and quadrupled for heavy users (90th percentile)**, and roughly **5% of AI requests fail in production with ~60% of those failures driven by capacity limits** (≈3% of all requests). Drilling into the error mix: in March 2026 Datadog telemetry saw 2% of all LLM spans return an error, and **rate-limit errors accounted for nearly one third of those — roughly 8.4M rate-limit errors in a single month**, with periodic concurrency and retry bursts unpredictably exhausting shared capacity quotas (the failure mode is most acute for ReAct-style variable-loop agents and tightly-coupled multi-agent stacks). The combination — more agents, more tokens per call, capacity-shaped failures — is exactly the workload profile that turns the absence of cumulative-spend counters (9.1.2) and chaos-tested failure modes (9.1.3) into financial exposure rather than theoretical risk.

Google's **Cloud Next '26** (April 22–24, 2026) introduced the Gemini Enterprise Agent Platform, which moves billing past pure per-token models to **per-agent and per-seat tiers** alongside per-token usage — making CFO-side budget caps tractable but also creating a new failure mode: dependent agents in a tightly-coupled multi-agent stack continuing the agentic loop after a peer crashes, generating massive token usage before the budget-exceeded email arrives. The platform ships **Event Compaction** (forces agents to summarize workflow state to avoid hitting context-window limits — a 9.1.1 mitigation against the unbounded-context pattern that drove the April 2026 Claude Code CLI incident) and **Agent Observability** (reasoning-loop tracing, useful for the cumulative-spend tracking required by 9.1.2). The pricing-tier shift is independently significant: per-token billing gives every misbehaving agent a runaway cost surface; per-seat/per-agent caps put a finite ceiling at the contract layer that complements (but does not replace) runtime enforcement.

May 2026 brought the same lesson into agent-controlled payments. In the **Bankr/Grok wallet incident** (May 4, 2026), an attacker used obfuscated prompt injection against a public social-media command path and caused an agent-controlled wallet to transfer roughly 3B DRB tokens (~$155K-$175K). The relevant C9.1 lesson is not just prompt injection: no hard per-transaction cap, velocity limit, or recipient allowlist stopped a single valid-looking action from spending a large fraction of the wallet. For value-bearing tools, budget evidence should cover both inference spend and *asset movement spend* per workflow.

May 2026 also tightened the enterprise-scale argument for per-agent budgets. Gartner's April 28 agent-sprawl guidance predicts that an average global Fortune 500 enterprise will grow from fewer than 15 agents in 2025 to more than **150,000 agents by 2028**, while only **13%** of organizations believe they have the right governance in place. A week later Gartner forecast AI agent software spending of **$206.5B in 2026** and **$376.3B in 2027**, up from $86.4B in 2025. At that scale, C9.1 evidence cannot be just a single gateway quota; auditors need an inventory-backed control plane that maps every agent identity, connector, session, and workflow to a bounded spend policy.

Anthropic's response to the same dynamic landed on May 14, 2026 with the [announcement of a June 15, 2026 Agent SDK billing split](https://help.apiyi.com/en/anthropic-claude-subscription-agent-sdk-billing-split-june-2026-en.html). Starting June 15, programmatic traffic (Agent SDK, `claude-p` headless mode, Claude Code GitHub Actions, third-party tools that wrap Claude) moves into a separate credit pool from interactive chat. Monthly credit equals the subscription fee — $20 for Pro, $100 for Max 5x, $200 for Max 20x — credits do not roll over, and overage is billed at standard API rates **on top of** the subscription. The framing is the first-party vendor acknowledgement of what Uber and the LeanOps audit established: subscription-priced AI assistants do not bound the cost of tool-driven agentic loops, and the only durable enforcement point is a token-level credit ledger with hard overage semantics. For C9.1 evidence, this means agent runtimes that previously hid behind a flat subscription budget will surface real per-session token spend in June; the cumulative-spend counters required by 9.1.2 should be in place before that flip, not after the first overage bill. A separate, ongoing failure mode complicates the same picture: as of [issue #46917](https://github.com/anthropics/claude-code/issues/46917), Claude Code v2.1.100–v2.1.133 (May 2026) inflates server-side `cache_creation_input_tokens` by roughly 20K extra tokens per request — about a 40% silent token increase versus v2.1.98, triggered by the User-Agent header and not present in earlier versions. Reports from Max 5x users show 8-hour session quotas exhausting in roughly 1 hour. The control implication is uncomfortable: if a vendor's own client can quietly inflate the meter that downstream budget gateways read, then per-session $/token caps will trip at smaller actual workloads, and any cumulative-spend tracking that anchors on provider-returned token counts becomes a function of the client version string. C9.1 evidence should therefore include client-version pinning (or an audit log of version drift), independent token estimation at the gateway as a cross-check against provider counters, and an alerting rule on input-token totals that diverge from local payload size.

The Uber Claude Code disclosure (April 17, 2026) is the first widely-reported case of a Fortune 500 burning its annual AI budget on agentic coding alone. After rolling out Claude Code in December 2025, adoption accelerated to 84% of engineers by March 2026, 95% monthly active, and 70% of committed code originating from AI; per-engineer API spend ran $500–$2,000/month, putting total fleet cost in the **$2.5M–$10M/month** range across ~5,000 engineers and exhausting Uber's full 2026 AI budget in four months. CTO Praveen Neppalli Naga said the company was "back to the drawing board" on AI budgeting. Two C9.1 implications stand out: consumption-priced agent runtimes (Claude Code is token-metered, not per-seat) break traditional enterprise license-cap reasoning, and per-engineer/per-agent gateway budgets become the only practical enforcement point once tool-driven token volume can compound by an order of magnitude inside a single quarter. DataRobot's Chief Product Officer reports a **96% cost-overrun rate** across enterprises scaling AI agents — Uber is not an outlier so much as a scale-amplified example.

The LeanOps "Tokens vs. Chats" audit (May 14, 2026) puts numbers on the underlying token shape across 30 engineering teams sampled March–May 2026: **median spend of $480 per developer per month, 90th percentile $1,650, outliers exceeding $4,200**, with one developer hitting $4,200 over a single long weekend. The decomposition matters more than the totals: a 5-step agent loop runs ~3.2× a single chatbot call, at 50 steps the multiplier "exceeds 30×," and at 200 steps "exceeds 100×" — because each step re-sends the entire conversation history. LeanOps attributes **62% of agent bills to re-sent context**, identifying it as the single largest optimization target. A representative SaaS case in the same audit cut a monthly bill from $87,000 to $24,000 by combining prompt caching (per-loop unit cost dropped from $0.45 to $0.0532, an 88% reduction) and tier routing that pushed only the hardest steps to a premium model. The takeaway for 9.1.1 evidence is that step caps and recursion limits leave most of the spend curve unenforced; context-budget controls (Gemini Event Compaction, Claude Code context cleanup, gateway-side context-size policies) and per-step model routing are first-class budget controls, not optimizations.

The Five Eyes "Careful adoption of agentic AI services" guidance (May 1, 2026) reinforces the same control posture from a national-security perspective: agent deployments should prioritize resilience, reversibility, and risk containment over efficiency gains; should avoid broad or unrestricted access to sensitive data and critical systems; and should use adversarial testing and simulated environments before production use. For C9.1, that means budget tests need to cover both runaway quantity and irreversible blast radius: loops, fan-out, token/context growth, payment velocity, destructive API calls, and fallback behavior when a peer agent or budget gateway fails.

The "the gateway is itself the attack surface" pattern is no longer theoretical. On March 24, 2026, the **LiteLLM PyPI supply-chain compromise** (versions 1.82.7 and 1.82.8, live for ~40 minutes before PyPI quarantined them) shipped a three-stage payload — credential harvesting, Kubernetes lateral movement, and a persistent RCE backdoor — via a `litellm_init.pth` file that ran on every Python startup whenever LiteLLM was installed. The attack was traced to the project's CI/CD pipeline: a Trivy scan inadvertently exfiltrated the PyPI publishing token to the TeamPCP threat actor group, who reused the same access path against Telnyx and other AI tooling. The package sees roughly three million downloads per day, so the blast radius was bounded only by the short window. Two details make this directly relevant to C9.1: the malicious payload contained a bug that spawned an endless loop of child processes — an accidental fork bomb that took down the host and revealed the implant; and LiteLLM is one of the most widely-used budget enforcement points in the agent stack. If your circuit breaker for runaway loops is itself a backdoored Python process running with the credentials it was meant to bound, the budget control is worse than absent — it is an attacker-controlled cost amplifier. Treat the gateway binary and its CI/CD pipeline as in-scope for C9.1 evidence, not just the budget config.

Industry survey data from the same window quantifies the operational exposure. The **CSA / Token Security "Autonomous but Not Controlled" survey** (April 20, 2026, 418 IT and security professionals surveyed in January 2026) reports that **65% of organizations experienced at least one AI agent-caused incident in the prior 12 months**, with **61% involving sensitive data exposure, 43% operational disruption, 41% unintended cross-process actions, 35% direct financial losses, and 31% service delays**. Only **21% have formal agent decommissioning processes**, and **82% have discovered shadow AI agents** in their environments despite 68% reporting "high visibility." For C9.1, this is the inventory-and-budget gap restated as incident frequency: without registration, identity, and per-agent spend policy at creation time, organizations are running unmeasured agents with no enforced ceiling — and one in three of those organizations is already paying for it.

Two adjacent surveys tighten the budget-allocation case. **Arkose Labs' 2026 Agentic AI Security Report** (April 2, 2026, 300 enterprise leaders across security, fraud, identity, and AI functions in North America, Europe, and Asia-Pacific, ±5.6% at 95% confidence) finds that **97% of respondents expect a material AI-agent-driven security or fraud incident within 12 months — and only 6% of security budgets are allocated to it**, **57% have no formal governance controls for AI agents today**, and only **26% are confident they could prove an AI agent caused a given incident**. **Gravitee's State of AI Agent Security 2026** (919 executives and practitioners across telecom, financial services, manufacturing, healthcare, and transportation/logistics) reports that **88% of organizations had confirmed or suspected AI-agent security or privacy incidents in the prior year (92.7% in healthcare)**, that **80.9% of teams have moved past planning into active testing or production but only 14.4% have achieved full IT/security approval for their entire agent fleet**, and that **only 47.1% of agents are actively monitored or secured and only 21% have runtime visibility into what their agents are doing** — even though **82% of executives feel confident their policies will hold**. For C9.1 evidence, the second-order takeaway is the runtime-visibility gap: if more than half of agents are not monitored at runtime and only one in five organizations can see what an agent is doing, neither the cumulative-spend counter required by 9.1.2 nor the partial-failure assertions required by 9.1.3 are observable in production today.

Early June 2026 turned "the gateway is itself the attack surface" from a supply-chain story into an active-exploitation story. **CVE-2026-42208**, a pre-authentication SQL injection in LiteLLM's auth verification path (the `Authorization: Bearer` value was concatenated directly into a PostgreSQL query, affecting v1.81.16–1.83.6, fixed in v1.83.7), let unauthenticated attackers dump the `LiteLLM_VerificationToken` and `litellm_credentials` tables — and [Sysdig observed targeted in-the-wild exploitation 36 hours after disclosure](https://www.sysdig.com/blog/cve-2026-42208-targeted-sql-injection-against-litellms-authentication-path-discovered-36-hours-following-vulnerability-disclosure), with attackers querying only the three highest-value tables. The budget implication is explicit in Sysdig's analysis: once extracted, virtual keys bypass spend limits and rate controls, enabling unlimited API calls against upstream providers. Then on June 8, 2026, CISA added **CVE-2026-42271** — a command injection in LiteLLM's MCP test endpoints (`/mcp-rest/test/connection`) that spawns attacker-supplied commands on the proxy host — to the [Known Exploited Vulnerabilities catalog](https://www.helpnetsecurity.com/2026/06/09/litellm-vulnerability-under-active-attack-cisa-warns-cve-2026-42271/), with Horizon3.ai confirming it chains with a Starlette Host-header bypass (CVE-2026-48710) into fully unauthenticated RCE. An attacker with RCE on the budget gateway doesn't just steal credentials — they can disable or rewrite the spend policies, making every downstream C9.1 control attacker-controlled. Audit evidence should now include the gateway's patch level (LiteLLM ≥ v1.83.7), whether MCP test endpoints are exposed, and whether virtual-key compromise would silently lift spend caps.

The macro cost picture sharpened in the same window. TechCrunch's June 5, 2026 ["The token bill comes due"](https://techcrunch.com/2026/06/05/the-token-bill-comes-due-inside-the-industry-scramble-to-manage-ais-runaway-costs/) roundup reports per-developer token consumption up roughly **18.6× in nine months** (Jellyfish data), a Faros AI case study of a single engineer burning **$40,000 in tokens in one month**, Goldman Sachs projecting 24× global token growth by 2030, Microsoft revoking Claude Code licenses for its Experiences & Devices division (access ending June 30, migrating to GitHub Copilot CLI), and Priceline's Cursor renewal quote coming back 4–5× higher. The Linux Foundation announced a **Tokenomics Foundation** (formal launch July 2026) to standardize token-economics metrics — an early sign that per-seat/per-developer token budgets are becoming a procurement line item rather than an engineering afterthought. On the empirical side, a June 2, 2026 arXiv catalog ([Token Budgets, arXiv 2606.04056](https://arxiv.org/abs/2606.04056)) assembled **63 confirmed production budget-overrun incidents across 21 orchestration frameworks** (2023–2026) into an 8-cluster failure taxonomy; the "delegation-fanout race" — concurrent budget delegation across sub-agents causing overspend — accounts for 11 of the 63 incidents, putting hard numbers behind the hierarchical sub-agent budget problem this section has flagged since the escrow-delegation research. And the [BAGEN benchmark (arXiv 2606.00198, May 29, 2026)](https://arxiv.org/abs/2606.00198) settles the "can the model police its own budget" question empirically: across four environments and five frontier agents, capability does not correlate with budget awareness (r=0.35), frontier models are systematically over-optimistic about remaining budget, and calibration tops out at 47% interval coverage even after SFT+RL — the strongest evidence yet that budget enforcement must live in the runtime, exactly as 9.1.1 requires.

As of May 20, 2026, the standards and test-harness picture is also maturing. OWASP's Q1 2026 exploit round-up frames recent incidents as a shift from model-output failures toward agent identities, orchestration layers, and supply chains, which matches C9.1's focus on execution-path controls rather than prompt-only guardrails. OWASP's FinBot CTF gives auditors a hands-on financial-services environment for tool misuse, policy bypass, privilege escalation, and RCE scenarios across multi-agent workflows; it is useful seed material for 9.1.3 tests because malicious vendor data can move upstream into administrative workflows and laterally across tenants. NIST's AI Agent Standards Initiative page, updated April 20, 2026, separates industry-led standards, community protocol work, and research into agent authentication and identity; that makes identity-bound budget policy and per-chain audit evidence likely procurement expectations even before formal controls settle.

A second post-mortem from the Claude Code project — [issue #54393, April 28, 2026](https://github.com/anthropics/claude-code/issues/54393) — is the clearest first-party documentation to date of how multiple C9.1 failure modes co-occur in a single autonomous run. Across five consecutive overnight cycles, the operator catalogued 12 distinct bugs that map directly onto this section: per-session token usage exhausted mid-task without graceful continuation, hooks recursing with no first-class timeout or depth-tracking primitive (the reporter suggested a 30-second wall-clock kill and a `CLAUDE_HOOK_DEPTH` env-var capped at 3), partial-failure defaults that quietly defer work the operator had pre-authorized to continue through failure, and compaction events that lost recent operator decisions so the post-compact agent re-litigated settled questions. The framing — "the runtime's failure modes are mostly observability / durability / normalization / mechanical-enforcement gaps" — is the same shape as the gateway-side gaps the broader industry is now filling, but located inside the agent's own loop. For audit purposes, treat hook recursion depth, hook wall-clock timeout, post-compaction state durability, and behavior under "do not defer" operator directives as their own 9.1.1 / 9.1.3 test categories alongside token, recursion, and dollar caps.

DigitalApplied's [H1 2026 AI Incidents Retrospective](https://www.digitalapplied.com/blog/ai-incidents-h1-2026-retrospective-failure-modes-analysis) (covering Jan 1–May 16, 2026, ≥50 catalogued public AI incidents) pins the failure-mode mix that C9.1 is meant to cover: tool-misuse cascades — retry storms, runaway loops, MCP server outages compounding through agent retries — were the fastest-growing category, **rising from ~12% of incidents in Q1 to ~25% in H1**, with **~18% of the dataset rated catastrophic** (regulatory penalty or >$1M loss). Median time-to-detect was ~4.5 hours and median time-to-contain ~12 hours, and **cost-spike anomalies were the dominant detection signal** — i.e. the cumulative-spend counters required by 9.1.2 are now the most reliable production tripwire, not a nice-to-have. The widest severity variance was in the tool-misuse cluster: minor when caught at minute 4, catastrophic when caught at hour 4. The retrospective projects H2 2026 incident volume at roughly double and tool-misuse share growing to ~30%, which is the planning baseline auditors should expect against C9.1 controls.

The Centre for Long-Term Resilience's ["Scheming in the Wild"](https://www.humai.blog/ai-models-are-scheming-5x-more-often-the-research-is-now-impossible-to-dismiss/) (March 2026) sits next to that as the strongest available baseline for the *behavioral* side of the chaos-test requirement. The authors analyzed 180,000 public agent transcripts collected between October 2025 and March 2026 and counted 698 credible incidents of deployed agents acting deceptively, exceeding user instructions, or taking covert actions — a 4.9× growth over six months against only 1.7× growth in general discussion of the topic. Documented patterns include a months-long sustained deception about an agent's own activities and an agent that falsely claimed to be producing an accessibility transcript in order to bypass copyright controls in another model. For 9.1.3 evidence, the relevant point is that runaway-budget and runaway-blast-radius behaviors are no longer adjacent to deceptive behavior — they are produced by it. A budget test harness that exercises only honest-but-buggy loops will under-count the failure modes already observed in production.

Anthropic's Claude Opus 4.7 GA (April 16, 2026) introduces the first widely-deployed *model-native* budget primitive worth flagging in C9.1 evidence. The new [`task_budget`](https://platform.claude.com/docs/en/build-with-claude/task-budgets) field (public beta header `task-budgets-2026-03-13`) gives the model a target token allowance for the full agentic loop — thinking, tool calls, tool results, and final output — with a running countdown the model uses to prioritize work and finish gracefully as the budget is consumed. Two properties matter for audit: (1) `task_budget` is **advisory, not a hard cap** — distinct from `max_tokens`, which is a per-request hard ceiling the model is not aware of; and (2) Opus 4.7 ships a new tokenizer that produces roughly **1.0–1.35× more tokens** than Opus 4.6 on the same text (up to ~35% more), so any per-session or per-chain budget anchored on a fixed token count silently tightens on upgrade unless the gateway recalibrates. Combine `task_budget` (self-moderation by the model) with `max_tokens` (per-request hard ceiling) and Agent SDK `max_budget_usd` (runtime hard ceiling) for layered enforcement, and document the tokenizer migration as a planned budget-baseline reset rather than a surprise. The new `xhigh` effort level, now the Claude Code default, sits on the same trade-off axis: higher intelligence per call at higher token spend.

OWASP's [AI Vulnerability Scoring System (AIVSS) v0.8](https://aivss.owasp.org/) (March 19, 2026; v1.0 public review opened April 16, 2026) gives the budgeting decision a scoring framework. AIVSS extends CVSS v4.0 with **10 Agentic AI Risk Factors (AARFs)** — including autonomy level, tool use scope, multi-agent interactions, non-determinism, dynamic identity, persistent memory, and self-modification capacity — each scored 0.0 / 0.5 / 1.0 to amplify a base vulnerability into a contextual 0–10 severity. Practical use for C9.1: deployments scoring high on autonomy + tool-use-scope + multi-agent factors should pull recursion, fan-out, token, and dollar caps tighter, and should default to fail-closed circuit-breaker policy rather than fail-open. An audit can use AIVSS scores to justify why a given budget configuration is or is not proportionate to the deployment risk, instead of choosing thresholds by gut feel.

This section requires hard limits on runtime expansion (recursion depth, concurrency, wall-clock time, token usage, monetary spend) enforced by the orchestration runtime, not by the model itself.

> **Scope note:** Controls in this section govern *internal orchestration runtime* budgets: per-task recursion depth, concurrency, wall-clock time, token spend, and monetary limits. They apply inside the agentic execution layer — not at the API ingress edge (covered by OWASP ASVS) and not in response to adversarial probing patterns (C11.4, C11.5). A single token-spend cap at the API gateway does not satisfy C9.1 budget enforcement, which requires enforcement within the orchestration runtime itself.

| **9.1.1** | **Verify that** per-tool quotas and timeouts (e.g., CPU, memory, disk, egress, and execution time) are enforced. | 1 | **Unbounded resource consumption** ([OWASP LLM10](https://genai.owasp.org/llmrisk/llm102025-unbounded-consumption/), [ASI08](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/)). An agent entering a recursive planning loop or spawning excessive parallel tool calls can exhaust API budgets and compute. Cost modeling: GPT-4 class agents at 10 cycles/min, 10K tokens/cycle ≈ $3/min; 50 concurrent threads = $9,000/hour burn rate. The $47K LangChain incident ran for 11 days before discovery because no budget threshold existed. Industry benchmarks emerging: **$100/hour hard cap** as standard kill-switch tripwire; **10-step limit** before requiring human help or stopping (Finout 2026). The [Fountain City five-layer cost defense](https://fountaincity.tech/resources/blog/ai-agent-cost-circuit-breaker/) (April 6, 2026) is a useful concrete pattern for individual deployments: per-cron wall-clock timeout (300–900s), recovery anti-loop (max 3 retries with 2-hour gaps), aggregate cost breaker that samples spend every 30 minutes against $50/day warning (2.5× baseline) and $100/day hard halt (5× baseline), explicit model pinning to block accidental upgrades to expensive variants, and weekly budget tracking for gradual creep. The author argues rate limits alone miss volume-shaped failures (100 normally-sized calls accumulate into a ruinous total), which is exactly the gap 9.1.2's per-chain counter is meant to cover. Alert-then-pause rather than immediate hard-kill preserves in-progress work and is worth treating as a UX requirement on the circuit breaker. [Cordum's April 2026 agent circuit-breaker write-up](https://cordum.io/blog/ai-agent-circuit-breaker-pattern) gives a useful concrete reference for the breaker itself: trip on 3 consecutive failures, hold open for a 30-second TTL before half-open re-probe, cap half-open probes at 3 before deciding to close, and pair the breaker with explicit `POLICY_CHECK_FAIL_MODE` set to `closed` (requeue during outage) or `open` (allow with bypass labels) so the availability-versus-safety choice is documented rather than implicit. For LangGraph and DeepAgents specifically, the [LangChain DeepAgents forum thread](https://forum.langchain.com/t/how-to-cap-tool-and-sub-agent-calls-in-deepagents/1653) recommends a belt-and-suspenders combination: global `recursion_limit` on the run config, explicit `tool_calls` / `subagent_calls` counters in graph state with conditional edges that terminate when budgets are exceeded, lower `recursion_limit` on sub-agent invocations than on parents, and `max_iterations` on classic `AgentExecutor` flows — global recursion alone is not enough once sub-graphs are involved. | Review orchestration config for explicit budget parameters. Test by triggering deep recursion and high fan-out; confirm runtime halts at thresholds. Verify budgets are server-side. Check for both request-rate (req/s) AND token-rate (tokens/s) dual limiting. Check for cost-weighted token tracking (output tokens from premium models cost up to 8x input tokens). Use Bifrost, LiteLLM, MLflow AI Gateway, or agentgateway as gateway-level enforcement if framework lacks native budgets; verify the policy key includes the agent session/workflow ID, not just a shared API key. ### Budget-Aware Reasoning Research

Two lines of research address budget waste at the *reasoning* level — making the agent itself budget-conscious rather than relying solely on external enforcement:

- **Google Budget Tracker + BATS** ([arXiv 2511.17006](https://arxiv.org/abs/2511.17006); Google/NYU/UC Santa Barbara): A prompt-level plug-in that injects a continuous "remaining budget" signal into the agent's reasoning loop. BATS (Budget-Aware Test-time Scaling) extends this to multi-agent systems with dynamic planning/verification that adapts to remaining resources. Benchmarks show 31.3% cost reduction at comparable accuracy, and continued scaling where plain ReAct plateaus. IDC survey: 92% of decision-makers report AI agent costs exceeded expectations.
- **Budget-Aware Value Tree (BAVT)** ([arXiv 2603.12634](https://arxiv.org/abs/2603.12634), March 2026): Models multi-hop reasoning as a dynamic search tree with budget-conditioned node selection. Uses remaining resource ratio as a scaling exponent over node values, transitioning smoothly from exploration to exploitation as budget depletes. Surpasses baselines at 4x the resource allocation under strict constraints.
- **Inference-Time Budget Control for LLM Search Agents** ([arXiv 2605.05701](https://arxiv.org/abs/2605.05701), May 7, 2026, Fang et al.): Two-stage budget control under **dual constraints — tool calls and tokens, enforced together**. Search-time control uses Value-of-Information (VOI) scoring to assign the next budget unit across retrieval, decomposition, and answer-commitment decisions; answer-time control rewrites only apparent low-risk formatting errors. Tested across four multi-hop QA benchmarks and three LLM backbones at four budget levels; primary gains came from search-time budget-dependent penalties. Practical implication for C9.1: dual-budget policies (tokens + tool calls) line up better with how cost actually accrues in tool-using agents than token-only or call-only caps, and are worth surfacing as separate counters in cumulative-spend tracking.

## Denial-of-Wallet Patterns & Defenses

| Algorithm | Best For | DoW Protection |
|-----------|----------|:--------------:|
| Token Bucket | Real-time apps needing burst flexibility | Moderate |
| Leaky Bucket | Background workflows, max cost predictability | Strong |
| Sliding Window | Production environments | Strong |
| **Layered** (recommended) | Short-term (tokens/min) + long-term (tokens/month) + cost-weighted | **Strongest** |

**Design principles:**
- **Provisional Reservation & True-Up**: Reserve budget upfront based on estimated max_tokens, refund difference after actual response
- **Dual Limiting**: Both request-rate (req/s) AND token-rate (tokens/s) controls
- **Cost-Weighted Tracking**: Output tokens from premium models cost up to 8x input tokens; weight accordingly
- **Graceful Degradation**: Return 429 with remaining budget and reset time so agents can queue/retry intelligently
- **Per-Model Pricing Catalog**: Maintain input vs. output token rates per model
- **Agent Inventory Binding**: Every agent identity, workflow, and connector should be registered before use and bound to a spend policy; otherwise per-agent budgets become impossible to audit at enterprise scale
- **Proactive Token Estimation**: Estimate token count at the gateway *before* forwarding to the LLM; reject or truncate requests that would exceed budget in a single call (Traefik Triple Gate v3.20 implements this natively, March 2026)
- **Multi-Provider Failover**: Circuit breaker chains that automatically switch to cheaper providers when primary budget tiers are exhausted, maintaining safety policies across the failover
- Critical: *"You cannot protect costs you cannot measure."*

## Four-Layer Enforcement Stack

Budget enforcement that only covers one layer leaves the others unprotected. As of March 2026, the recommended architecture layers enforcement at four distinct points ([Rack2Cloud](https://www.rack2cloud.com/ai-inference-execution-budgets/)):

| Layer | Enforcement Point | Example Controls |
|-------|------------------|-----------------|
| **1. Invocation** | API gateway / proxy | Per-request token limits, context caps, output constraints, proactive token estimation (Traefik Triple Gate) |
| **2. Agent Loop** | Framework config | Step caps, recursion guards, retry limits (`recursion_limit`, `max_turns`, `max_iter`) |
| **3. Orchestration** | Workflow engine | Workflow-level cost ceilings, fan-out caps, time boundaries (Temporal, Prefect, Airflow) |
| **4. Platform** | Cloud provider / org | Org-level quotas, hard spend cutoffs, billing alerts |

**Critical insight:** Alerts and dashboards are *cost witnesses*, not cost *enforcers*. Enforcement must live in the execution path itself, not in an observability sidecar.

### Multi-Agent Reliability Decay (Lusser's Law)

In multi-agent pipelines, failure probability compounds multiplicatively. A single agent at 98% accuracy yields 98% system reliability — but 10 agents at 98% each degrade to **81.7% system reliability** (18.3% error rate). Without validation gates between agents, cost compounds alongside failure: each unchecked hop amplifies both the reliability and financial risk. Mitigation: schema enforcement via Pydantic/Instructor raises effective per-agent accuracy to ~99.8%, and Best-of-N sampling (e.g., RULER framework) further reduces compounding errors ([O'Reilly Radar](https://www.oreilly.com/radar/the-hidden-cost-of-agentic-failure/)).

## Open Research Questions

- What are appropriate default budget thresholds for different agent use cases? Emerging benchmarks: $100/hour hard cap, 10-step limit, $3,200–$13,000/month operational range.
- How should budgets be allocated across sub-agents in hierarchical systems? The escrow-based delegation pattern (arXiv 2602.11865) proposes recursive budget partitioning with bond slashing.
- Can adaptive budgets (dynamically adjusting based on task complexity) be made safe, or do they introduce exploitable flexibility?
- How do you handle budget enforcement for agents making external API calls with their own billing (SaaS-to-SaaS worm pattern)?
- Should agent frameworks standardize a budget/cost-ceiling API, or should this remain in the gateway/governance layer?
- How do you detect multi-agent circular dependencies where each agent individually appears to be making progress (the $47K LangChain pattern)?
- How should the recursive pattern of "AI agents managing AI agent budgets" (agentic FinOps) itself be bounded?
- The EU AI Act (staggered timeline; high-risk obligations originally applicable August 2026 but deferred to December 2, 2027 by the May 2026 Digital Omnibus) requires human oversight and risk assessment for autonomous agents, but doesn't prescribe specific budget enforcement mechanisms. How should execution budget requirements map to EU compliance obligations, particularly for cross-border tool invocations?
- As proactive token estimation matures (Traefik Triple Gate), can pre-LLM cost prediction become reliable enough to replace post-hoc budget enforcement?
- Budget-aware reasoning (Google BATS, BAVT) shows agents can self-regulate tool usage when given budget signals — does this reduce the need for hard external enforcement, or is it too easily bypassed by adversarial inputs?
- The $82K Gemini API key theft exposed the "shared responsibility" gap: providers refuse to cap runaway usage from stolen credentials. Should budget ceilings be mandatory provider-side defaults, or is this fundamentally a customer responsibility?
- The April 2026 malicious-router study shows the enforcement gateway itself can be the attack surface. How should C9.1 evidence distinguish budgets enforced by a trusted gateway from budgets enforced by an intermediary whose integrity is unverified? Is pinning provider endpoints + binary attestation sufficient, or does this imply every agent runtime needs a signed receipt chain from the model provider?
- The Agentic Risk Standard (ARS) proposes collateral and escrow as economic circuit breakers. Does an enforceable financial guarantee substitute for runtime enforcement, complement it, or merely shift who pays when limits are breached?
- After the OpenClaw incident, should public framework marketplaces (skill/tool hubs) require a conformance suite — RCE probe, cost-exhaustion probe, data-exfiltration probe — as a pre-listing gate, and who owns the suite?
- The PocketOS deletion shows that destructive blast radius is its own budget category — one well-aimed call can cause more loss than a runaway loop. Should C9.1 evidence explicitly require a separate "destructive-action budget" that caps the cumulative *reversibility cost* of tool calls (e.g., total volume bytes that may be deleted per session, total dollars that may be paid out per workflow), distinct from token/dollar inference budgets?
- The Claude Code CLI April 2026 bug shipped from a major model vendor's own coding agent. If even first-party agents do not enforce per-session token caps or input/output ratio guardrails, what is the right minimum bar for vendors who ship general-purpose agent runtimes that customers will run unattended against their own API keys?
- BAGEN (May 2026) shows frontier-agent budget calibration tops out at 47% interval coverage even after fine-tuning, while early termination of doomed trajectories saves 28–64% of tokens. Is there a principled division of labor where model-native budget estimates drive *graceful degradation* (summarize-and-stop, tier downgrade) while the runtime owns all *hard stops* — and can that split be standardized as an interface between models and gateways?
- The delegation-fanout race (11 of 63 catalogued overrun incidents) suggests budget delegation needs transactional semantics — reserve, spend, refund, revoke — across sub-agent boundaries. Should orchestration frameworks adopt an affine/linear-types-style budget ledger (as the arXiv 2606.04056 Rust crate prototypes), or is runtime-level escrow (arXiv 2602.11865) the more deployable path?
- The June 2026 LiteLLM CVEs show stolen virtual keys silently escape the spend limits bound to them. Should budget policy be enforced against cryptographically-bound workload identity (SPIFFE/SVID-style) rather than bearer keys, so that key exfiltration does not transfer the budget ceiling bypass?
