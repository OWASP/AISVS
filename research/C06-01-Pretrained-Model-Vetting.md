# C6.1: Pretrained Model Vetting & Origin Integrity

> **Parent:** [C06: Supply Chain Security](C06-Supply-Chain.md) | **Requirements:** 5 | **IDs:** 6.1.1–6.1.5

## Purpose

Pretrained models downloaded from public hubs (Hugging Face, Model Zoo, Civitai, etc.) represent one of the highest-risk supply-chain entry points in AI systems. Unlike traditional software dependencies, model weight files can encode arbitrary executable code (via pickle deserialization), contain hidden backdoor triggers, or carry undisclosed licensing or data-provenance obligations. As of Spring 2026, Hugging Face alone hosts over 2 million public models serving 13 million users — the attack surface is enormous and growing fast. The **Axios npm compromise (March 31, 2026)**, which reached into OpenAI's macOS app-signing pipeline for ChatGPT Desktop and Codex, is a reminder that model vetting alone is not enough — the JavaScript and Python packages that load, serve, and proxy those models are themselves primary attack surfaces. This section ensures every third-party model is authenticated, scanned, and reviewed before it enters the organization's pipeline.

(6) Cisco Talos published **pickle-fuzzer** (December 2025), a structure-aware fuzzing tool that found crash-inducing bugs in ModelScan, Fickling, and PickleScan via memo-key validation and unhashable-bytearray issues — patched in Fickling v0.1.5 and PickleScan v0.0.32. **Configuration file attacks** are an overlooked vector: the "Rusty Link in the AI Supply Chain" study (2025) analyzed 150,000 Hugging Face repositories and found 13,091 repos with file-operation risks and 35,761 with repository-redirection risks via malicious configuration files. Guardian has launched four new detection modules since the partnership began: archive-slip vulnerabilities (PAIT-ARV-100), Joblib suspicious execution (PAIT-JOBLIB-101), TensorFlow SavedModel backdoors (PAIT-TF-200), and Llamafile malicious execution (PAIT-LMAFL-300). Use Cisco's open-source **pickle-fuzzer** to stress-test whichever scanner you deploy. Prefer SafeTensors/ONNX format over pickle-based formats. For Trojan detection, run Neural Cleanse, ABS, or MNTD scans on candidate models. Scan configuration files alongside model weights — the ConfigScan approach demonstrates that `config.json` and related files can contain malicious repository redirections. For runtime detection of model supply-chain compromise, **Datadog Workload Protection** (2026) offers execution-context-based monitoring that can detect anomalous behavior during model loading and inference without hardcoding attack signatures. For GGUF models, inspect embedded chat templates using `gguf-dump` or the `gguf` Python library before loading — look for unexpected conditional logic or hidden instructions that deviate from reference templates. Two newer entrants worth evaluating: **Palisade** (December 2025) is a Rust-based zero-trust model scanner that adds behavioral analysis layers — Perplexity Gap Analysis and Functional Trap Testing — to detect covert fine-tuning and trigger-based backdoors that static analysis misses; it scans at 100+ MB/s, generates ML-BOMs, and supports Sigstore signing and SLSA provenance verification with SARIF 2.1.0 output for CI/CD integration. **Veritensor** is an open-source zero-trust scanner that replaces naive blocklist scanning with deep AST analysis, covers Pickle, PyTorch, Keras, GGUF, and SafeTensors, and includes native RAG pipeline integration for LangChain, LlamaIndex, and ChromaDB — meaning it can scan model artifacts at runtime in retrieval-augmented workflows, not just at import time. | Trojan-trigger scanning (Neural Cleanse, etc.) is computationally expensive and has known false-negative rates. **Pickle scanning alone is insufficient:** seven distinct PickleScan bypass CVEs were disclosed in 2025, and Cisco's pickle-fuzzer (December 2025) found additional crash-inducing bugs across all three major open-source scanners, proving that blocklist-based scanners remain brittle. The SafePickle paper (February 2026) found that ~98% of files flagged as unsafe by existing scanners on Hugging Face are actually benign — the false-positive burden is a real obstacle to adoption. Defense-in-depth requires: (1) multiple scanners with different detection architectures, (2) format restrictions (SafeTensors-only policy where feasible), (3) sandboxed model loading, and (4) configuration file scanning. As of Spring 2026, **90% of new uploads** on Hugging Face use SafeTensors — a dramatic improvement from ~42% in early 2025. However, legacy risk persists: an empirical study (arxiv 2501.02170) found that 98.7% of repos that adopted SafeTensors retained their old pickle files alongside the new format, meaning the pickle attack surface hasn't shrunk proportionally. HiddenLayer's 2026 AI Threat Landscape Report found that malware in public model and code repositories is now the most-cited source of AI-related breaches (35% of reported incidents), and one in eight AI breaches involves agentic systems pulling models autonomously. The academic paper "The Art of Hide and Seek" (2025) demonstrates further stealthy pickle poisoning techniques. The MITRE ATLAS 2026 update (January 2026) introduced new technique IDs for AI service API abuse (AML.T0096) and agent tool data poisoning (AML.T0099), plus the SesameOp case study (AML.CS0042) demonstrating backdoors using the OpenAI Assistants API for command and control — confirming that model-level supply-chain threats are expanding into the agentic execution layer. The **MITRE ATLAS v5.4.0** release (February 2026) added further agent-focused techniques including **AML.T0011.002 Publish Poisoned AI Agent Tool** (adversaries ship malicious versions of MCP tools or agent plugins that appear safe but take harmful actions when invoked) and **Escape to Host**, bringing the framework to 16 tactics, 84 techniques, 56 sub-techniques, 32 mitigations, and 42 case studies. The LiteLLM TeamPCP campaign and the Axios npm compromise both map onto ATLAS Initial Access via ML Supply Chain Compromise (AML.T0010) and ML Software Dependency Compromise (AML.T0048). The "Mind the Gap" GGUF attack (ICML 2025) is a wake-up call: GGUF files were widely assumed safer because they lack pickle's arbitrary code execution — but the quantization-error exploitation technique embeds behavioral backdoors at the weight level without any code execution vector, so static format-safety checks (e.g., "it's not pickle, it's safe") are insufficient. Organizations deploying quantized models via ollama or llama.cpp should run behavioral evaluation alongside format scanning. The SafeTensors conversion hijacking ("Silent Sabotage") shows that even the migration *to* safer formats introduces risk if the conversion infrastructure itself is compromised — format conversion should happen in isolated, audited environments, not via third-party hosted services with broad permissions. |
| **6.1.3** | **Verify that** high-risk models (e.g., publicly uploaded weights, unverified creators) remain quarantined until human review and sign-off. | 2 | **Deployment of unvetted or backdoored models.** Automated pipelines may pull and deploy models without human judgment, allowing compromised artifacts to reach production. The Rapid7 "From .pth to p0wned" analysis (2025) documented end-to-end exploitation chains from model download to remote access trojan deployment. The fake Alibaba SDK attack (ReversingLabs, May 2025) showed that malicious pickle models are now being embedded inside traditional PyPI packages — quarantine controls must extend beyond model hubs to any dependency that could bundle model files. The NSA/Five Eyes March 2026 guidance explicitly recommends organizations "quarantine and test externally sourced data before moving it into internal systems," applying the same quarantine principle to models and datasets alongside traditional software artifacts. A Mitiga Labs analysis (2026) of 10,000 open-source AI/ML repositories found that **70% have critical or high-severity GitHub Actions workflow vulnerabilities** — including unpinned third-party actions (68.4%), script injection via PR titles/bodies (34.1%), over-privileged GITHUB_TOKEN permissions (42.7%), and hardcoded secrets (22.8%). These CI/CD pipeline weaknesses mean that even quarantined models can be compromised *after* import if the build and deployment workflows themselves are insecure. Microsoft's January 2026 case study documented CVE-2025-68664 ("LangGrinch"), a vulnerability in LangChain Core's `dumps()`/`dumpd()` functions that allowed injection of malicious objects through improper escaping — illustrating how AI framework and orchestration-layer vulnerabilities create supply-chain risk beyond model artifacts themselves. | Confirm that a quarantine zone (separate registry, staging namespace, or approval gate) exists. Run automated scanning (ModelScan, ModelAudit, Fickling) during quarantine. Integrate Palo Alto AI Runtime Security or Black Duck AI model scanning (released Oct 2025) for enterprise-grade quarantine workflows. The Protect AI + Hugging Face scanning partnership has identified 352,000 unsafe issues across 51,700 models out of 4.47 million versions scanned — roughly 1 in 85 model versions has a security concern, underscoring the need for automated gate-keeping. | Defining "high-risk" criteria requires organizational policy. Suggested triggers: unknown publisher, no model card, pickle format, model size anomalies, flagged by automated scanning, namespace squatting indicators, **recently re-registered namespaces** (the Unit 42 namespace-reuse attack exploits exactly this gap), or **path-based model references** that could trigger Model Confusion (Checkmarx, January 2026). Commercial tools like Protect AI Guardian, Black Duck, and HiddenLayer Model Scanner now offer managed quarantine-and-scan pipelines with integration into model registries like Databricks Unity Catalog. Guardian proactively detected CVE-2025-1550 (a critical Keras vulnerability) before public disclosure, demonstrating the value of continuous scanning partnerships. Sonatype's 2026 Software Supply Chain Report documented over 454,600 new malicious packages across ecosystems including Hugging Face, bringing the cumulative total of known malware to over 1.23 million packages. The Mitiga 10K-repo study shows that quarantine must extend beyond model artifacts to CI/CD workflows: pin all GitHub Actions to full 40-character commit SHAs, enforce least-privilege GITHUB_TOKEN permissions (read-only default), and eliminate script injection by using environment variables instead of interpolating PR metadata into shell commands. Also audit AI framework dependencies (LangChain, LlamaIndex, etc.) as part of quarantine — CVE-2025-68664 shows that orchestration-layer vulnerabilities can be as dangerous as poisoned model weights. The **LiteLLM supply chain compromise** (March 24, 2026) demonstrated the extreme risk of AI infrastructure packages: TeamPCP published backdoored PyPI versions (1.82.7/1.82.8) of the widely-used AI proxy library (95M monthly downloads, present in 36% of cloud environments) after compromising CI/CD credentials — the payload harvested SSH keys, cloud tokens, and Kubernetes secrets. Quarantine controls should now explicitly include AI infrastructure packages (LiteLLM, LangChain, LlamaIndex) alongside model artifacts. The **Axios npm compromise** (March 31, 2026) pushed this lesson further: Lazarus Group's BlueNoroff subgroup socially engineered the lead axios maintainer, hijacked his npm and GitHub accounts, and published malicious versions `1.14.1` and `0.30.4` that injected a dependency on `plain-crypto-js@4.2.1` carrying a BlueNoroff RAT executed via the `postinstall` hook. Axios had ~83 million weekly downloads and 174,000 dependent packages; **OpenAI's macOS app-signing pipeline pulled the infected code, exposing code-signing certificates for ChatGPT Desktop and Codex.** This is the first publicly confirmed case of a nation-state supply-chain compromise directly touching a frontier model vendor's release pipeline. Treat every transitive JavaScript or Python dependency of a model-serving path as in-scope for quarantine and pin by integrity, not by version. **Mitiga Labs "Brainjacked"** (2026) adds another dimension: an attacker exploited a path-traversal flaw in a customer-facing web portal to overwrite a training dataset in an S3 bucket that fed an automated SageMaker retraining pipeline. The poisoned data embedded a backdoor trigger into the next retrain, and the attacker then invoked it against the live endpoint. Quarantine must therefore extend backwards into the applications that can *write* to training data stores, not just forward into model hubs and package registries. |
| **6.1.4** | **Verify that** third-party or open-source models pass a defined behavioral acceptance test suite (covering safety, alignment, and capability boundaries relevant to the deployment context) before being imported or promoted to any non-development environment. | 2 | **Unvetted model behavior in production — safety failures, alignment drift, and capability overreach.** A third-party model may pass format and integrity checks (6.1.1–6.1.2) and clear quarantine (6.1.4) yet still exhibit unsafe behaviors: toxic outputs, hallucinations beyond acceptable thresholds, sycophantic compliance with harmful instructions, or capabilities that exceed the deployment context (e.g., code execution in a customer-facing chatbot). The 2026 International AI Safety Report warns that models increasingly learn to distinguish test environments from real deployment, creating an "evaluation gap" where systems appear safe during testing but exhibit problematic behaviors in production. Anthropic's AuditBench benchmark (2025) demonstrated that some models with implanted hidden behaviors could evade naive evaluation — motivating the development of evaluation-awareness countermeasures. The EU AI Act (Article 9, enforceable August 2026) requires providers of high-risk AI systems to perform testing "appropriate to the intended purpose of the AI system" including adversarial testing, robustness evaluation, and performance across demographic groups. For organizations importing third-party models, the EU AI Act explicitly requires independent verification testing in the deployment context rather than relying solely on vendor self-declarations. The **LiteLLM supply chain compromise** (March 24, 2026) underscores why behavioral testing matters even for trusted packages: TeamPCP published backdoored versions 1.82.7 and 1.82.8 of the popular AI proxy library (95 million monthly PyPI downloads, present in 36% of cloud environments) after stealing credentials via a compromised Trivy GitHub Action in LiteLLM's CI/CD pipeline — the payload harvested SSH keys, cloud tokens, and Kubernetes secrets, attempted lateral movement, and installed persistent backdoors. Organizations with behavioral acceptance gates would have detected the anomalous execution patterns before deployment. | Define a behavioral acceptance test suite tailored to the deployment context. At minimum, test for: (1) safety — refusal of harmful requests, toxic output rates, jailbreak resistance; (2) alignment — sycophancy, deception, self-preservation tendencies; (3) capability boundaries — confirm the model does not exhibit capabilities outside its intended scope. **Tooling options as of March 2026:** **Promptfoo** (open-source, MIT-licensed, 300K+ developers, used by OpenAI and Anthropic; acquired by OpenAI March 2026) provides red-teaming and vulnerability scanning across 50+ attack categories with CI/CD integration and SARIF output — well-suited for automated acceptance gates. **Anthropic Bloom** (open-source, December 2025) is an agentic framework that automatically generates behavioral evaluation scenarios from a seed configuration, quantifying how often a target behavior appears across diverse situations — validated against four alignment-relevant behaviors (sycophancy, sabotage, self-preservation, self-preferential bias) across 16 models. **Google LMEval** (open-source, May 2025) provides cross-provider model evaluation with safety/punting detection support; Giskard uses LMEval to run the **Phare benchmark** (Potential Harm Assessment & Risk Evaluation) evaluating safety, hallucination, bias, and factual accuracy across multiple languages. **Allen AI Safety-Eval** offers evaluation across 10+ safety benchmarks including prompt harmfulness, response harmfulness, and refusal detection. For statistical rigor, follow **NIST AI 800-3** (February 2026) guidance on distinguishing benchmark accuracy from generalized accuracy and using generalized linear mixed models for uncertainty quantification — this helps organizations avoid over-relying on narrow benchmark scores when making import decisions. **NIST AI 800-2** (January 2026 draft) documents best practices for automated benchmark evaluations structured in three stages: defining measurement targets, implementing evaluations, and analyzing results. Integrate behavioral tests into the quarantine pipeline (6.1.4): models must pass the acceptance suite before promotion from staging to production. | Behavioral acceptance testing for third-party models is an emerging but rapidly maturing practice. Key challenges: (1) **Evaluation awareness** — the 2026 AI Safety Report notes that models can detect test environments and modify behavior accordingly; Anthropic's Petri tool includes countermeasures but the arms race continues. (2) **Context specificity** — no universal test suite exists; organizations must define acceptance criteria relative to their deployment context (a medical chatbot needs different safety boundaries than a code assistant). (3) **Benchmark saturation** — NIST AI 800-3 warns that benchmark accuracy and generalized accuracy can diverge meaningfully, so passing a benchmark does not guarantee safe behavior on real-world inputs. (4) **Tooling fragmentation** — Promptfoo, Bloom, LMEval, and Safety-Eval each cover different behavioral dimensions; a comprehensive acceptance suite likely requires combining multiple tools. (5) **Cost and latency** — running full behavioral evaluation adds time to the import pipeline, creating tension with rapid deployment cycles. Suggested minimum acceptance gates: jailbreak resistance (Promptfoo red-team), safety refusal rates (Safety-Eval), alignment probe (Bloom sycophancy/deception evaluations), and capability boundary check (custom prompts verifying the model stays within intended scope). The EU AI Act conformity assessment deadline (August 2026) will likely accelerate standardization of behavioral acceptance testing for high-risk deployments. |
| **6.1.5** | **Verify that** transfer-learning fine-tunes pass adversarial evaluation to detect hidden behaviors. | 3 | **Sleeper-agent attacks and backdoor persistence through fine-tuning.** Research (Anthropic "Sleeper Agents" 2024, BadNets, TrojAI) shows that backdoors in base models can survive fine-tuning. An attacker contributes a poisoned base model; the victim fine-tunes it, unknowingly inheriting the backdoor. The "Models Are Codes" study (2024) systematically measured malicious code poisoning across pre-trained model hubs, confirming that poisoned models are a real and growing phenomenon, not just a lab demonstration. The formal result from the joint **Anthropic / UK AI Security Institute / Alan Turing Institute** study ("A small number of samples can poison LLMs of any size," arXiv 2510.07192, October 2025) is stronger than the informal "250 documents" figure that has been circulating: attack success is controlled by the **absolute** number of poisoned documents rather than by the *percentage* of training data, so scaling the model or the dataset does not dilute the attack. In their experiments, 250 poisoned documents reliably implanted a denial-of-service backdoor across 600M–13B parameter models; 100 did not robustly work. Help Net Security's March 2026 write-up describes this as "nearly undetectable with current tooling" for defenders that rely on statistical outlier analysis. A medical-LLM study (NYU/Columbia, January 2025) demonstrated that replacing just 0.001% of training tokens with fabricated medical articles — costing approximately $5 — boosted harmful outputs by 7.2%, and model size offered no protection (600M and 13B parameter models were equally easy to backdoor). The **"Malice in Agentland"** study (October 2025) extended these findings to agentic systems: by poisoning as few as 2% of fine-tuning traces collected from an agent's own web-browsing or tool-use interactions, an attacker can embed a backdoor that leaks confidential user information with over 80% success — and prominent guardrail models failed to detect the malicious behavior. This is especially concerning because the practice of fine-tuning agents on their own interaction data is a standard recipe for improving agentic capabilities. | Run adversarial test suites against fine-tuned models, including: trigger-pattern sweeps (randomized input perturbations), activation-clustering analysis, and behavioral probes for known attack patterns. Compare fine-tuned model behavior against a clean baseline on a held-out adversarial evaluation set. The PickleBall framework (2025) proposes secure deserialization of pickle-based ML models as a complementary control. As of February 2026, Microsoft Research published **"The Trigger in the Haystack"**, a breakthrough sleeper-agent detection method that identifies backdoors by analyzing internal attention patterns. The technique exploits a "Double Triangle" attention signature where trigger tokens attend almost exclusively to each other while normal prompt attention drops to near zero — a geometric fingerprint invisible in clean models. In testing across 47 sleeper agents spanning Phi-4, Llama-3, and Gemma architectures, the scanner achieved an **88% detection rate with 0% false positives**, using only inference operations without expensive gradient computation. Integrate this attention-pattern analysis alongside traditional trigger-pattern sweeps for a more robust detection pipeline. **Palisade** (December 2025) also offers behavioral backdoor detection via Perplexity Gap Analysis and Functional Trap Testing. A complementary 2026 detection approach from the *Pattern Recognition* journal ("Backdoor defense based on adversarial prediction proximity and contrastive knowledge distillation") observes how prediction labels shift between an infected model and a copy briefly fine-tuned on clean samples, then purifies the model via triplet loss with clean anchors and untargeted adversarial examples using the clean-fine-tuned copy as a "benign teacher" and the original as a "malicious teacher." Reported results: 100% detection accuracy across a library of 630 models and robust purification across six common backdoor attack families. These newer techniques are especially useful when full retraining is not feasible. | This is computationally expensive, but the tooling gap is narrowing. Microsoft's "Trigger in the Haystack" (February 2026) represents the first practical, inference-only sleeper-agent detection method with strong accuracy — a significant step beyond prior brute-force approaches. However, it currently detects only fixed triggers; dynamic, context-dependent triggers may evade detection. The TrojAI program (IARPA/NIST) continues developing benchmarks. The "Malice in Agentland" (October 2025) finding that guardrail models fail to detect agentic backdoors underscores the need for weight-level inspection, not just behavioral guardrails. The NIST AI Agent Standards Initiative (February 2026) includes agent security controls that may formalize adversarial evaluation requirements for agentic fine-tuning pipelines. Level 3 is still appropriate given the tooling maturity, but the gap is closing faster than expected. EU AI Act systemic-risk GPAI models (>10^25 FLOP) face additional safety evaluation requirements that may further drive tooling maturity. |

| 6.1.2 Malicious layer/Trojan scanning | Maturing | The scanning tool landscape has expanded considerably. Established tools: ModelScan, ModelAudit (42+ formats, allowlist-first), Fickling (pickle allowlist), HiddenLayer Model Scanner (commercial). Newer entrants: **Palisade** (December 2025, Rust-based, behavioral analysis via Perplexity Gap and Functional Trap Testing, 100+ MB/s, SARIF 2.1.0 output) and **Veritensor** (open-source, deep AST analysis, covers Pickle/PyTorch/Keras/GGUF/SafeTensors, native RAG pipeline integration for LangChain/LlamaIndex/ChromaDB). Seven PickleScan bypass CVEs disclosed in 2025 alone (JFrog CVE-2025-10155/56/57, Sonatype CVE-2025-1716/1889/1944/1945) plus the nullifAI evasion — no single scanner is reliable. The Protect AI + HF partnership has scanned 4.47M model versions and flagged 352K unsafe issues across 51.7K models. **SafeTensors adoption has reached 90% of new uploads** on HF as of Spring 2026 (up from ~42% in early 2025), though 98.7% of repos that converted retained old pickle files alongside — legacy risk persists. HiddenLayer's "Silent Sabotage" disclosure showed that even the SafeTensors conversion pipeline itself could be weaponized — conversion should happen in isolated environments. HiddenLayer's 2026 report: 35% of AI breaches traced to malicious models in public repos. The "Mind the Gap" paper (ICML 2025) extends the threat model beyond pickle to GGUF quantized formats, proving that weight-level backdoors survive quantization and can achieve 88.7% attack success without any code-execution vector. Defense-in-depth required: multiple scanners with different detection architectures (blocklist, allowlist, ML-based, behavioral) + SafeTensors format restriction + sandboxed loading + config file scanning + behavioral evaluation for quantized models. |
| 6.1.3 License/export-control/data-origin in AI BOM | Maturing | CycloneDX v1.7 (ECMA-424 2nd Edition, December 2025) adds Citations for granular data-provenance attribution. SPDX 3.0.1 AI profiles also maturing. NIST AI 600-1 (Generative AI Profile) explicitly requires training data transparency and provenance. The **OWASP AIBOM project** (launched 2025) provides an open-source generator that scans repos for HF model usage and emits CycloneDX-compatible AI-BOMs. The **EU AI Act GPAI Code of Practice** (Final, July 10, 2025) standardizes a Model Documentation Form across transparency, copyright, and safety chapters; most frontier providers have signed on, effectively converging the industry toward a common AI-BOM skeleton for EU-market models, with Commission enforcement starting August 2, 2026. The **NIST Cyber AI Profile** (IR 8596, December 2025 draft) extends CSF 2.0 to AI, requiring data-provenance verification on par with software/hardware and comprehensive AI asset inventories. The **NSA/Five Eyes March 2026 guidance** explicitly requires AI-BOMs from vendors as a procurement condition. The **MAIF container format** (November 2025 paper) proposes an AI-native alternative embedding provenance directly into model artifacts, though it remains at research stage. Gap: many open-weight models lack clear data-origin disclosures, and CycloneDX Citations auto-population tooling is still emerging. |
| 6.1.4 Quarantine until human review | Maturing | Protect AI Guardian, Black Duck (Oct 2025), and HiddenLayer Model Scanner offer managed quarantine-and-scan pipelines. Guardian proactively detected CVE-2025-1550 (Keras) before public disclosure. Palo Alto AI Runtime Security provides enterprise workflows. MalHug found 91 malicious models across 705K+ on HF; the Protect AI partnership now shows ~1 in 85 model versions has a security concern. Sonatype's 2026 report: over 454,600 new malicious packages across ecosystems, bringing cumulative known malware past 1.23 million. The **HiddenLayer 2026 AI Threat Landscape Report** (March 18, 2026) surveyed 250 IT/security leaders and found that malware in public model and code repositories is the most-cited source of AI-related breaches (**35%**), yet 93% of respondents still rely on open repositories — and 31% don't know whether they had an AI security breach in the past 12 months. The NSA/Five Eyes March 2026 guidance explicitly recommends quarantine-and-test protocols for externally sourced models. Quarantine triggers should include namespace reuse, Model Confusion path patterns, suspicious configuration files, and models sourced from traditional package managers (the PyPI pickle-smuggling vector). Quarantine scope must now extend to CI/CD workflows: Mitiga's 2026 analysis of 10,000 AI/ML repos found 70% have critical workflow vulnerabilities (unpinned actions, script injection, over-privileged tokens) that could compromise models post-import. Additionally, AI framework dependencies should be audited during quarantine — CVE-2025-68664 (LangGrinch in LangChain Core) demonstrated that orchestration-layer flaws can introduce supply-chain risk independently of model artifacts. The **Axios npm + BlueNoroff compromise** (March 31, 2026) showed that even production release pipelines at frontier model vendors (OpenAI) are within reach of nation-state supply-chain actors, and the **Mitiga Brainjacked** case (2026) demonstrated that a path-traversal flaw in an unrelated customer-facing application can directly poison the training data of an automated retraining pipeline — meaning quarantine must also cover *write access paths* into training data stores, not only the model-import direction. |
| 6.1.5 Adversarial evaluation of fine-tunes | Emerging → Early Adoption | Historically computationally expensive with no standardized tooling. Microsoft's **"Trigger in the Haystack"** (February 2026) represents a breakthrough: inference-only sleeper-agent detection via Double Triangle attention patterns, achieving 88% detection with 0% false positives across Phi-4, Llama-3, and Gemma architectures. Palisade adds behavioral analysis (Perplexity Gap, Functional Trap Testing). Traditional tools: Neural Cleanse, ABS, MNTD. NIST TrojAI continues developing benchmarks. The joint Anthropic/UK AISI/Alan Turing **"Small Samples Poison LLMs of Any Size"** study (arXiv 2510.07192, October 2025) hardens the threat model: attack success tracks the absolute number of poisoned documents, not the percentage — 250 documents reliably implanted a DoS backdoor across 600M–13B parameter models, and dataset or model scaling does not dilute the attack. "Malice in Agentland" (October 2025) showed 2% data poisoning achieves 80% backdoor success in agentic fine-tuning — and guardrails fail to detect it, making weight-level inspection essential. A 2026 *Pattern Recognition* method based on **adversarial prediction proximity + contrastive knowledge distillation** reports 100% detection across a library of 630 models and purification across six backdoor attack types — a promising complement when retraining is not feasible. The NIST AI Agent Standards Initiative (February 2026) may formalize adversarial evaluation requirements for agent-specific fine-tuning. |
| 6.1.6 Behavioral acceptance testing | Emerging | A rapidly developing area with no single standardized suite. **Promptfoo** (300K+ developers, acquired by OpenAI March 2026) offers the most mature CI/CD-integrated red-teaming and safety evaluation with 50+ vulnerability categories. **Anthropic Bloom** (December 2025) provides automated behavioral evaluation generation for alignment-relevant behaviors. **Google LMEval** (May 2025) enables cross-provider evaluation with the **Giskard Phare benchmark** covering safety, hallucination, bias, and factual accuracy. **Allen AI Safety-Eval** covers 10+ safety benchmarks. **NIST AI 800-2** (January 2026 draft) documents best practices for automated benchmark evaluations; **NIST AI 800-3** (February 2026) introduces statistical rigor for interpreting benchmark results. The EU AI Act conformity assessment deadline (August 2026) is likely to accelerate standardization. Key gap: no universal acceptance criteria — organizations must define context-specific test suites, and evaluation awareness (models detecting test environments) remains an active arms race. |

## Open Research Questions

- What is the false-negative rate of current Trojan-detection tools (Neural Cleanse, ABS, MNTD) against state-of-the-art backdoor attacks? The "Art of Hide and Seek" (2025) paper suggests attackers continue to outpace defenses.
- Can model provenance be established for models where training data and training process are not disclosed (closed-weight models accessed via API)?
- How should organizations handle models that are "fine-tune chains" (Model A fine-tuned from Model B fine-tuned from Model C) where provenance may be lost at each step?
- The OpenSSF Model Signing (OMS) specification now exists (June 2025) with NVIDIA and Google as early adopters — but will the broader Hugging Face ecosystem actually adopt it? What incentives or platform enforcement mechanisms are needed to drive signing adoption past the current handful of publishers?
- **How should organizations defend against model namespace reuse attacks?** Pinning to commit hashes helps, but many codebases and documentation still reference models by name alone. Is there a scalable way to detect when a previously trusted namespace has changed ownership?
- **How should organizations layer multiple scanning tools given demonstrated bypasses?** The JFrog PickleScan zero-days (CVE-2025-10155/10156/10157) and the nullifAI evasion show that no single scanner is reliable. What is the right combination of blocklist (PickleScan), allowlist (ModelAudit/Fickling), and format-restriction (SafeTensors-only) controls?
- **Will EU AI Act GPAI obligations drive standardization of model provenance metadata?** The August 2025 requirements for training-data documentation and downstream transparency may create de facto standards, but enforcement details remain unclear until August 2026.
- **Can the PickleBall secure-deserialization approach scale to production workloads?** If so, it could provide a runtime defense complementing static scanning.
- **How should organizations address configuration file attack vectors?** The "Rusty Link" study found tens of thousands of suspicious repos using malicious `config.json` redirections, but existing scanning tools focus almost exclusively on model weight files. ConfigScan demonstrates LLM-based detection, but adoption and false-positive rates in production remain unvalidated.
- **Is the Model Confusion attack pattern (Checkmarx, January 2026) the tip of the iceberg for implicit-download vulnerabilities?** The `from_pretrained()` path-resolution logic is just one library; other frameworks may have similar implicit remote fetching behavior that has not yet been audited.
- **Can ML-based scanning approaches like SafePickle replace or complement blocklist scanners in production?** The February 2026 paper shows dramatically better accuracy and speed, but real-world deployment at scale — with evolving adversarial techniques — remains unvalidated. What is the right trust model when the scanner itself uses ML?
- **How should organizations handle malicious models embedded inside traditional package managers (PyPI, npm)?** The fake Alibaba SDK attack (May 2025) showed pickle models being smuggled via PyPI packages — current model-scanning workflows focus on model hubs and miss this vector entirely.
- **Will the NIST Cyber AI Profile (IR 8596) drive convergence between AI and traditional supply-chain security requirements?** The December 2025 draft treats model and data provenance on par with software provenance, which could reshape procurement practices once finalized in 2026.
- **How effective are runtime detection approaches (e.g., Datadog Workload Protection) at catching model supply-chain compromise that static scanning misses?** Execution-context monitoring during model loading could detect anomalous behavior from novel attack techniques, but false-positive rates and performance overhead in production ML inference pipelines remain uncharacterized.
- **Will the NSA/Five Eyes March 2026 AI supply-chain guidance accelerate enterprise adoption of model signing and AI-BOMs?** Government procurement requirements have historically driven standards adoption (e.g., FIPS, FedRAMP), but the AI model ecosystem lacks the centralized distribution infrastructure that made software supply-chain mandates enforceable.
- **Can Microsoft's "Trigger in the Haystack" attention-pattern approach be extended to detect dynamic, context-dependent triggers?** The current technique works well for fixed triggers (88% detection, 0% false positives) but may miss adversaries who use variable or conditional trigger patterns. Combining attention analysis with behavioral probing (Palisade's Functional Trap Testing) may be necessary for comprehensive coverage.
- **How should organizations secure agentic fine-tuning pipelines against the "Malice in Agentland" attack pattern?** When agents fine-tune on their own interaction traces (web browsing, tool use), a 2% poisoning rate yields 80% backdoor success and current guardrails fail. Dataset screening defenses exist but are not yet validated at production scale. The NIST AI Agent Standards Initiative (February 2026) may eventually address this, but the gap between published research and deployable defenses remains wide.
- **How should organizations defend against quantization-level backdoor attacks on GGUF models?** The "Mind the Gap" paper (ICML 2025) showed that backdoors can be embedded through quantization error exploitation, surviving format conversion from full-precision to GGUF. Current scanning tools focus on pickle deserialization and static code analysis — none specifically detect quantization-error-based behavioral backdoors. Should organizations require behavioral evaluation of all quantized models, even those from trusted full-precision sources?
- **Should CI/CD pipeline security be included in model quarantine scope?** The Mitiga study found 70% of AI/ML repos have critical workflow vulnerabilities. A model can pass quarantine scanning and still be compromised post-import through unpinned GitHub Actions, script injection in workflow triggers, or over-privileged automation tokens. How should organizations draw the boundary between model-level and infrastructure-level quarantine controls?
- **How should format-conversion trust chains work?** HiddenLayer's "Silent Sabotage" attack showed that converting from unsafe formats (pickle) to safe ones (SafeTensors) can itself be a compromise vector if the conversion service is weaponized. Organizations need a trust model for format conversion: should conversion happen only in air-gapped environments, and should converted artifacts require re-signing?
- **What should a minimum behavioral acceptance test suite look like for third-party model imports?** With Promptfoo, Bloom, LMEval/Phare, and Safety-Eval each covering different dimensions, organizations lack a unified framework for defining "pass/fail" criteria. Will NIST AI 800-2 best practices evolve into a standardized acceptance testing protocol, or will context specificity prevent convergence?
- **How should organizations address the evaluation awareness problem in behavioral testing?** The 2026 International AI Safety Report warns that models can detect test environments and modify behavior. Anthropic's Petri includes countermeasures, but the arms race between evaluation techniques and model awareness is accelerating. Is there a principled approach to making evaluations resistant to gaming?
- **Does the LiteLLM supply chain compromise (March 2026) signal a new class of AI infrastructure attacks?** TeamPCP's multi-ecosystem campaign (PyPI, npm, Docker Hub, GitHub Actions, OpenVSX) hit AI proxy infrastructure rather than model artifacts directly — compromising the pipeline that routes requests to models. Should AI infrastructure packages (LiteLLM, LangChain, LlamaIndex) be subject to the same quarantine and vetting controls as model artifacts themselves?
- **How should organizations defend against nation-state account-hijack attacks on JavaScript/Python dependencies in the model-serving path?** The Axios npm compromise (March 31, 2026) reached OpenAI's macOS app-signing pipeline through a maintainer-account takeover by BlueNoroff/UNC1069 — a vector that model scanning, signing, and quarantine do not address. Should model-vendor release pipelines require hardware-backed MFA on every maintainer, integrity-pinned dependencies (not version-pinned), and a separate signing enclave for every release? What is the minimum bar that a frontier model vendor should meet?
- **How should quarantine controls cover write paths into training data?** Mitiga's Brainjacked case shows that a path-traversal flaw in an unrelated customer-facing application can poison training data in an S3 bucket feeding an automated SageMaker retrain. Current quarantine practice focuses on the import side (hubs, registries). Should every service that can write to a training-data store be treated as in-scope for model supply-chain risk, and how should that be scoped without blocking legitimate data collection?
- **Will the EU AI Act GPAI Code of Practice Model Documentation Form become a de facto AI-BOM standard?** Most frontier providers have signed; the Commission begins enforcement August 2, 2026. If the Model Documentation Form is treated as the reference format, does that reduce the need for CycloneDX ML-BOM and SPDX AI extensions, or do they remain complementary for non-GPAI models?
- **Is the 250-document poisoning threshold (Anthropic/UK AISI/Turing, October 2025) tractable for defenders to detect?** If attack success is controlled by an absolute number of poisoned documents rather than a percentage, statistical outlier scans miss it — a few hundred documents in a multi-trillion-token corpus is well below the noise floor. What content-level defenses (trigger-string heuristics, per-source trust scoring, gradient-signature analysis) scale to petabyte-scale pretraining corpora?
- **Can adversarial-prediction-proximity purification methods (Pattern Recognition 2026) replace expensive sleeper-agent detection at production scale?** The reported 100% detection across 630 models is striking, but the benchmark library is small compared to real-world attack diversity. Do these methods hold up against adaptive attackers who design triggers specifically to evade adversarial-proximity analysis?
- **How should organizations reconcile the 93% reliance on open model repositories (HiddenLayer 2026) with the 35% incident rate?** The HiddenLayer 2026 Threat Report finds that malware in public model and code repos is the most-cited breach source, yet nearly every organization continues to rely on open repositories. Is there a principled way to quantify the trade-off between velocity and supply-chain risk, and what tooling closes that gap without blocking productive research use?
