# C5.1: Authentication

> [Back to C05 Index](C05-Access-Control.md)
> **Last Researched:** 2026-04-16

## Purpose

Establish verified identities for all entities interacting with AI systems, with authentication strength appropriate to the risk level. AI systems introduce identity challenges beyond traditional web applications: model-serving endpoints, inference APIs, and model registries often bypass centralized identity providers in favor of standalone API keys. In multi-agent and federated deployments, agent identity itself is an emerging concern — short-lived, cryptographically signed tokens are needed to prevent impersonation and replay attacks across agent boundaries.

> **Scope note:** Controls in this section govern the authentication layer for humans, services, and AI agents accessing AI system resources. Resource authorization, query-time enforcement, output entitlement enforcement, PDP isolation, and multi-tenant isolation are addressed in C5.2–C5.6. General identity management, federation, and MFA patterns are covered by ASVS v5 V6; this chapter addresses AI-specific concerns that ASVS does not cover.

## Requirements

| # | Requirement | Level | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|-----------------|----------------------|--------------|
| **5.1.1** | **Verify that** high-risk AI operations (model deployment, weight export, training data access, production configuration changes) require step-up authentication with session re-validation. | 2 | Stolen session tokens used to exfiltrate model weights or poison training data; insider threats performing high-impact changes with passive credentials; session hijack via infostealer-captured cookies (observed throughout 2025 as MFA-bypass vector); compromised CI/CD pipelines issuing unauthorized production changes. MITRE ATLAS AML.T0024 (Exfiltration via ML Inference API), AML.T0010 (ML Supply Chain Compromise). | Trigger each high-risk operation (model registry push, weight download, training-data bucket read, production config change) and confirm a fresh MFA challenge is presented; verify session re-validation timestamps in audit logs and that elevated sessions carry a shorter TTL than base sessions; test that expired step-up sessions are denied without silent downgrade; confirm phishing-resistant authenticator is enforced (FIDO2 security key or device-bound passkey) rather than SMS/TOTP; inspect IdP policy using tools like Okta Identity Engine, Auth0 Adaptive MFA, Microsoft Entra Conditional Access, or Ping DaVinci to confirm re-authentication frequency and authenticator-class constraints. | Defining "high-risk" remains organization-specific. Model weight export and training-data access are AI-specific high-value targets not typically covered by general MFA policies — they need to be explicitly enumerated in Conditional Access / policy-as-code. **[2026-04]** NIST SP 800-63-4 (final, July 2025) now **requires** phishing-resistant authenticators at AAL3 with non-exportable private keys; syncable passkeys are explicitly disallowed at AAL3. AAL3 is the baseline recommendation for model-weight export and training-pipeline write access. Microsoft's April 2026 threat report documented a large-scale device-code phishing campaign that bypassed MFA entirely by coaxing users through the OAuth device authorization grant — a reminder that step-up flows must avoid device-code grants on high-risk paths. The Jul 2025 xAI/Grok incident (DOGE staffer committed API key to public GitHub, exposing 52 LLMs) and the 2024 Hugging Face 1,500+ exposed-token incident show how a single compromised static credential without step-up auth can expose entire model inventories. Truffle Security's 2025 Common Crawl analysis surfaced 12,000+ active API keys and passwords embedded in LLM training data, providing attackers with credentials that entirely bypassed MFA for downstream systems. The Mercor/LiteLLM compromise (March 2025) showed that step-up auth must extend to CI/CD pipeline changes — TeamPCP used a compromised open-source dependency as the entry point before pivoting into AI training infrastructure. |
| **5.1.2** | **Verify that** AI agents in federated or multi-system deployments authenticate via short-lived, cryptographically signed authentication tokens (e.g., signed JWT assertions) with a maximum lifetime appropriate to the risk level and including cryptographic proof of origin. | 3 | Agent impersonation in multi-agent orchestration; token replay attacks enabling unauthorized agent actions; long-lived credentials being exfiltrated from agent memory or prompt logs; confused-deputy attacks where a compromised MCP proxy reuses a broadly-scoped static client credential; token-passthrough attacks where an MCP server forwards a token it was never issued. MITRE ATLAS AML.T0051 (LLM Prompt Injection) chained with agent-identity abuse; OWASP LLM09 (Excessive Agency). | Inspect token lifetimes and signing algorithms (reject `alg: none`, prefer EdDSA/ES256 over RS256 where feasible); verify token rotation occurs well before expiry; test that expired or tampered tokens are rejected with a clock-skew tolerance no wider than 60s; confirm proof-of-origin claims (SPIFFE IDs, `iss`/`aud` pair, x5t#S256 binding) are validated on every hop; check that `act` claims document the full delegation chain per the OAuth on-behalf-of draft; audit MCP servers to ensure they **refuse** tokens not explicitly issued with the MCP server in the `aud` claim (token passthrough is forbidden by the MCP spec); run `step certificate inspect` against SPIRE-issued SVIDs to confirm lifetime ≤ a few minutes for JWT-SVIDs and ≤ a few hours for X.509-SVIDs; verify incremental/progressive scope elevation rather than omnibus scopes. | Multi-agent identity is an emerging but rapidly maturing area. SPIFFE/SPIRE provides workload identity with well-defined short-lived JWT-SVID and X.509-SVID formats that auto-rotate; IETF WIMSE (`draft-ietf-wimse-arch-07`, March 2026) is now formalizing multi-system workload identity and binding JWTs to cryptographic keys via WIT (workload identity token) credentials. **[2026-04]** NIST NCCoE released concept paper (5 Feb 2026, public comment closed 2 Apr 2026) "Accelerating the Adoption of Software and AI Agent Identity and Authorization" naming MCP, OAuth 2.0/2.1, OpenID Connect, SPIFFE/SPIRE, SCIM, and NGAC as the target standards set, mapped against SP 800-207 Zero Trust and the new SP 800-63-4. The IETF draft `draft-oauth-ai-agents-on-behalf-of-user-02` (August 2025; -01 May 2025) introduces `requested_actor` (authorization request) and `actor_token` (token request) parameters so the authorization server issues a composite token with an `act` claim documenting `user → client → agent` delegation, enabling auditable chains. Auth0 Token Vault (GA Jan 2026) implements RFC 8693 token exchange for AI agents with 30+ pre-integrated providers, keeping long-lived provider credentials off agent infrastructure and issuing short-lived scoped tokens per transaction; HashiCorp Vault's dynamic-secrets pattern for AI agents (2026 validated pattern) offers an on-prem alternative. The ARIA protocol (filed with NIST docket 2025-0035, 9 Mar 2026) models delegation as a cryptographically verifiable graph of W3C Verifiable Credentials with L0–L3 verification levels, making delegation paths surgically revocable. The 2026 MCP authorization specification mandates OAuth 2.1, PKCE, resource-indicator binding (RFC 8707), per-client consent to prevent confused-deputy attacks against proxy servers, and incremental scope consent; token passthrough is explicitly forbidden. Gaps remain: agent-level granularity inside a single workload is still workload-coarse in SPIFFE; OIDC-A (OpenID Connect for Agents) is still a proposal; and tooling to verify `act`-chain claims end-to-end in production authorization servers is scarce outside Auth0, Okta Auth for GenAI, and custom Keycloak/Ory deployments. |
