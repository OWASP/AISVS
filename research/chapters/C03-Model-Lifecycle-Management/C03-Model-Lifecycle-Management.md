# C03: Model Lifecycle Management & Change Control

> **Source:** [`1.0/en/0x10-C03-Model-Lifecycle-Management.md`](https://github.com/OWASP/AISVS/blob/main/1.0/en/0x10-C03-Model-Lifecycle-Management.md)
> **Requirements:** 27 | **Sections:** 6

## Control Objective

AI systems must implement change control processes that prevent unauthorized or unsafe model modifications from reaching production. Only authorized, validated models reach production by employing controlled processes that maintain integrity, traceability, and recoverability.

> **2025-2026 Highlights:** The AI model supply chain became a top-tier attack surface in 2024-2026, with JFrog documenting a 6.5x increase in malicious models on Hugging Face and Protect AI finding 352K unsafe issues across 51.7K models. The TeamPCP campaign (March 2026) showed how one compromised security tool can cascade into AI infrastructure: Trivy, npm packages, Checkmarx KICS, LiteLLM 1.82.7/1.82.8, and Telnyx 4.87.1/4.87.2 were all affected within days. May 2026 added two more lifecycle warnings: a fake OpenAI Privacy Filter repository reached Hugging Face's trending list before HiddenLayer flagged its infostealer, and the Mini Shai-Hulud wave compromised npm/PyPI packages while abusing trusted publishing paths. OpenSSF Model Signing (OMS) v1.0 reached production readiness in April 2025, NVIDIA has signed all NVIDIA-published NGC Catalog models since 2025, and Cohere signed all Command models hosted on Hugging Face in October 2025. The March 2026 eight-nation AI/ML supply chain guidance reinforced C3 controls by recommending digital signatures, checksums, version-specific model registries, and testing before approval. The GPT-4o sycophancy rollback (April 2025) remains a useful lifecycle lesson: behavior and system-prompt changes need safety thresholds, pre-deployment feedback, and rollback evidence, not just functional tests. The current C3 source has 27 requirements: 3.2 now stops at 3.2.7 after removing the prior validation-override item, while 3.2.4 remains the explicit fail-closed deployment gate.

## Threat Landscape

Known attacks, real-world incidents, and threat vectors relevant to this chapter:

- Model tampering during training or fine-tuning pipeline
- Unauthorized model deployment bypassing validation gates, especially where failed safety evaluations only warn instead of blocking promotion
- Model supply chain attacks (compromised checkpoints on Hugging Face, etc.) -- 6.5x increase in malicious models in 2024 (JFrog); 352K unsafe issues across 51.7K models (Protect AI, 2025)
- Model namespace reuse/hijacking: attackers reclaim deleted namespaces to distribute malicious models via trusted names (Unit 42, 2025)
- Picklescan evasion via nullifAI techniques, bypassing format-based defenses (ReversingLabs, 2025)
- Stale or vulnerable models remaining in production without rotation -- 91% of ML models degrade over time (MIT)
- Silent hosted model updates changing behavior without consumer awareness -- prompts degrade after provider updates
- Provider model deprecations with limited notice windows (60 days for Azure GA models; varies widely across providers)
- Incomplete decommissioning leaving model artifacts accessible to attackers
- Weaponized open-source AI repositories (NullBulge campaign, 2024: ransomware delivery via Hugging Face/GitHub)
- Post-training compression (quantization, pruning, distillation) degrading safety alignment and adversarial robustness -- research shows extreme quantization introduces unpredictable representational harm and pruning reduces model robustness to adversarial inputs
- Eight-nation AI/ML supply chain guidance (March 2026) reframes AI as a layered supply chain spanning data, models, software, infrastructure, hardware, and third-party services; recommends cryptographic signing, verified model registries, and provenance checks across the lifecycle
- AI infrastructure supply chain attacks escalating: TeamPCP campaign (March 2026) compromised Trivy, npm packages, Checkmarx KICS, LiteLLM, and Telnyx in a five-day chain; affected hosts needed credential rotation, persistence hunting, and Kubernetes review, not just package rollback
- Hosted model behavioral regressions from system-prompt and feedback-weighting changes -- GPT-4o sycophancy incident (April 2025) required rollback after the update over-weighted short-term feedback signals
- Parallel-deployment cross-contamination: in canary, blue/green, shadow, and A/B cohorts, AI-specific shared state (KV caches, prompt caches, session state, retrieval indices) can leak context from one model version into another and confound rollback metrics; addressed by AISVS requirement 3.3.3
- IBM X-Force 2026 Threat Index: 300K+ ChatGPT credentials on dark web marketplaces in 2025; supply chain compromises up nearly 4x since 2020; 44% increase in exploitation of public-facing applications
- Model-hub trust-signal abuse: HiddenLayer found the fake Open-OSS/privacy-filter repository on Hugging Face on May 7, 2026; it copied a legitimate model card, used inflated engagement to reach the trending list, and shipped a loader that executed Windows infostealer malware
- Trusted-publishing abuse in package ecosystems: the May 2026 Mini Shai-Hulud wave compromised TanStack, Mistral AI, UiPath, OpenSearch, Guardrails AI, and other packages; the TanStack path combined `pull_request_target`, cache poisoning, and OIDC token extraction, showing that provenance attestations do not help if the trusted build identity is compromised

## Tooling & Implementation

Current tools, frameworks, and libraries that help implement these controls:

- **MLOps platforms:** MLflow, Weights & Biases, Kubeflow, SageMaker, Vertex AI Model Registry, Azure AI Foundry
- **Model signing:** [OpenSSF Model Signing (OMS) v1.0](https://openssf.org/projects/model-signing/) and the [sigstore/model-transparency](https://github.com/sigstore/model-transparency) library -- production-ready Sigstore-based library and CLI for signing any model format (April 2025); NVIDIA signs all NGC Catalog models with OMS since March 2025; Google's Kaggle and NVIDIA's NGC actively adopting OMS as the industry standard; Cohere (2025) signs every Command model hosted on Hugging Face; also cosign, in-toto attestations, TUF (The Update Framework). [CoSAI](https://www.coalitionforsecureai.org/) (40+ members) released model signing and incident response frameworks in 2025-2026. Sigstore transparency logs provide verifiable proof-of-inclusion so any model verifier can confirm a signature appeared in the append-only log at a specific time
- **SBOM/ML-BOM:** CycloneDX ML-BOM for model and dataset transparency, provenance, training methods, framework configuration, and dependencies; SPDX AI/ML profiles; Syft for container SBOM generation. Current C3.1.1 only requires deployed-model inventory, but registry records should still link to C6 AI-BOM evidence when available.
- **Deployment:** Seldon Core, BentoML, TorchServe, Triton Inference Server, vLLM with canary/blue-green strategies via Argo Rollouts or Flagger; OPA Gatekeeper or Kyverno policies can enforce signature, checksum, evaluation-artifact, and model-owner admission checks before promotion. The Mini Shai-Hulud wave is a reminder to pair provenance checks with hardened CI identity, cache isolation, protected release workflows, and post-build malware scanning because valid trusted-publishing metadata can still describe malicious output.
- **Testing:** [OWASP AI Testing Guide v1](https://owasp.org/www-project-ai-testing-guide/) for repeatable trustworthiness testing; Giskard, Deepchecks, Great Expectations (for data validation), custom red-team harnesses, and CI evaluation jobs that fail closed when C3.2 safety thresholds are missed.
- **Data versioning:** DVC, LakeFS, Delta Lake
- **Audit logging:** Cloud-native (CloudTrail, Cloud Audit Logs, Azure Monitor), append-only log stores
- **Hosted model monitoring:** [deprecations.info](https://deprecations.info/) (tracks deprecations across OpenAI, Anthropic, Google, AWS Bedrock, Cohere, xAI via JSON/RSS), [llm-stats.com](https://llm-stats.com/llm-updates) (model release tracking)
- **Supply chain scanning:** Protect AI model scanning (352K issues found across 51.7K models in 2025), Hugging Face Picklescan (updated after nullifAI evasion), safetensors format, and static/dynamic inspection of repository helper scripts (`loader.py`, `start.bat`, notebooks, install hooks). For public model hubs, popularity, likes, and trending rank are weak evidence; C3 gates should prefer signed provenance, verified publisher identity, controlled sandbox loading, and reproducible evaluation records.
- **Compression safety testing:** SafetyBench, TruthfulQA for benchmarking quantized/pruned model safety; quantization-aware training to preserve robustness during compression
- **Hosted model logging:** [PromptLayer](https://docs.promptlayer.com/), [Langfuse](https://langfuse.com/), [LiteLLM](https://docs.litellm.ai/) (unified proxy with model identifier logging across providers), MLflow AI Gateway
- **RLHF safety:** [Safe-RLHF](https://github.com/PKU-Alignment/safe-rlhf) (PKU, constrained value alignment), SafeMoE (ICLR 2025, safety routing for MoE LLMs), KL penalty monitoring (standard PPO), Adversarial Reward Auditing (ARA, Feb 2026, latent-space detection), InfoRM (information-theoretic reward modeling with MOP metric)
- **Standards/Frameworks:** NIST AI RMF 1.0 (January 2023) plus the Generative AI Profile NIST-AI-600-1 (July 2024) and the Trustworthy AI in Critical Infrastructure Profile concept note (April 7, 2026) -- a full 1.1 version has not been released, and guidance is rolling out via profiles and addenda through 2026; ISO/IEC 42001 (AI management systems); EU AI Act Chapter V GPAI provider obligations in force since August 2, 2025, including technical documentation, copyright policy, training-content summary, systemic-risk assessment, incident reporting, and cybersecurity obligations; GPAI Code of Practice finalized as a voluntary compliance route; SLSA for ML (emerging); eight-nation AI/ML Supply Chain Guidance (March 2026); DevSecMLOps (OpenSSF MLSecOps whitepaper, 2025); OCC Bulletin 2025-26 (October 2025) clarified model-risk-management expectations for community banks and announced a broader OCC review of model-risk practices at banks of all sizes, the first place many US banks are formalizing LLM/GenAI governance under the existing OCC Bulletin 2011-12 model-risk framework

### Implementation Maturity

| Control Area | Tooling Maturity | Notes |
|--------------|:---:|-------|
| C3.1 Model Authorization & Integrity | Medium-High | Model registries are mature. OpenSSF Model Signing (OMS) v1.0 reached production readiness in April 2025; NVIDIA signs all NGC Catalog models with OMS, Google's Kaggle integrated the sigstore/model-transparency library, and Cohere (2025) signs all Command models on Hugging Face. CoSAI (40+ orgs) released model signing frameworks. The March 2026 eight-nation AI/ML supply chain guidance recommends verified registries and cryptographic signing. CycloneDX ML-BOM supports model, dataset, framework, provenance, and training-method metadata, but the current AISVS source trims 3.1.1 to model-registry inventory only; AI-BOM detail now belongs mainly in C6 and should be linked from the registry rather than duplicated. The May 2026 Mini Shai-Hulud wave shows the practical gap: signatures and SLSA-style provenance need protected CI identities, cache isolation, and release-workflow review, or the registry may faithfully record a malicious but validly published artifact. *(Updated 2026-05)* |
| C3.2 Model Validation & Testing | Medium | Safety testing tools exist but standardized pass/fail thresholds still vary by domain. OWASP AI Testing Guide v1 (November 2025) improves the evidence base with repeatable trustworthiness tests across application, model, infrastructure, and data layers. The current source keeps separate checks for input validation (3.2.1), output sanitization (3.2.2), safety evaluations with pass/fail thresholds (3.2.3), an explicit deployment block when safety thresholds fail (3.2.4), versioned security testing (3.2.5), audit records (3.2.6), and post-compression retesting (3.2.7). The prior validation-override item was removed from C3.2, so auditors should expect fail-closed evidence here and look to governance/oversight controls for exception handling. SafetyBench and TruthfulQA can benchmark quantized models, but edge-specific safety benchmarks remain a gap. *(Updated 2026-05)* |
| C3.3 Controlled Deployment & Rollback | High | Canary/blue-green well-supported in Kubernetes ecosystem; atomic multi-artifact rollback less mature. Model Namespace Reuse attack (2025) demonstrated need for deployment-time integrity checks at cloud platform level. GPT-4o sycophancy incident (April 2025) highlighted that behavioral metrics must be part of staged evaluation -- functional metrics can pass while safety regresses. The latest C3 source drops the emergency shutdown and cascade requirements from this section, leaving kill-switch governance to C14; C3.3 now focuses on gradual rollout, complete-state rollback, and AI-specific cohort isolation. Requirement 3.3.3 addresses model versions running in parallel (A/B, canary, shadow): they must not share KV caches, prompt caches, session state, or retrieval indices. This remains a gap in many inference stacks because throughput-oriented cache sharing requires explicit per-deployment routing or per-tenant cache partitioning to comply. *(Updated 2026-05)* |
| C3.4 Secure Development Practices | High | Version control and environment isolation are solved problems; cultural adoption in ML teams varies. Data provenance remains a gap--digital signatures cannot adequately cover dynamic training data. The April 2026 source refactor narrowed C3.4 from four requirements to three and scoped them explicitly to AI-specific runtime components (agent orchestration services, tool/MCP servers, model registries, prompt/policy stores): the broad environment-separation and data-integrity requirements moved into C4/C6, leaving C3.4 focused on AI-pipeline isolation and AI-specific configuration artifacts. The TeamPCP/LiteLLM attack (March 2026) demonstrated that even security tooling in training environments can become an attack vector -- egress controls and package integrity verification are essential. IBM X-Force 2026 reports supply chain compromises up nearly 4x since 2020, with AI-powered coding tools introducing unvetted code into pipelines. *(Updated 2026-04)* |
| C3.5 Hosted and Provider-Managed Model Controls | Low-Medium | Provider transparency improving: deprecation tracking services (deprecations.info) now cover 7+ providers via JSON/RSS. Azure AI Foundry offers 60-day GA retirement notices. OpenAI returns exact dated snapshot IDs in API responses. However, alias resolution changes remain opaque, silent updates are common, and 91% of models degrade over time (MIT). The GPT-4o sycophancy incident (April 2025) demonstrated that even system prompt changes -- not model weight updates -- can cause severe behavioral regressions with no advance consumer notification. EU AI Act Chapter V GPAI provider obligations have been in force since August 2, 2025 and the European Commission's full enforcement and fining powers start August 2, 2026, which should drive more consistent downstream-provider transparency including mandatory technical documentation retention for 10 years after market release. NIST has not released a 1.1 version of AI RMF; the April 2026 Trustworthy AI in Critical Infrastructure Profile concept note is the most recent RMF-linked guidance. No standard protocol for provider-to-consumer change notifications. *(Updated 2026-04)* |
| C3.6 Fine-Tuning Pipeline Authorization & Reward Model Integrity | Low | Section renumbered from C3.7 to C3.6 in the April 2026 source refactor, and the current source now has five requirements. Reward model signing and verification moved to 3.6.1, separation of duties is 3.6.2, stage-output integrity verification is 3.6.4, and intermediate checkpoint registration with version or digest identifiers is now separate 3.6.5. Reward model integrity can leverage OMS v1.0 signing. Reward hacking detection is an active research area--InfoRM (information-theoretic reward modeling with CSI indicator), ODIN (disentangled reward), ARA (Feb 2026 latent-space detection), and classifier-based anti-reward-hacking rewards (Anthropic) offer promising approaches but none are production-hardened as drop-in defenses. Multi-stage checkpoint integrity verification and per-stage rollback still require custom pipeline logic in most MLOps stacks. *(Updated 2026-05)* |

## Open Research Questions

- [ ] What's the right cadence for model re-validation in production (drift vs. cost)? (MIT finding: 91% of models degrade; Gartner: 67% within 12 months)
- [ ] How do hosted model providers (OpenAI, Anthropic, Google) handle versioning transparency? (Partially answered: deprecations.info tracks 7 providers; Azure offers 60-day notices; OpenAI provides model IDs but aliases them; transparency remains inconsistent)
- [ ] What constitutes 'adequate' pre-deployment security testing for fine-tuned models now that C3.2.4 requires deployment to fail closed when safety thresholds are missed? (OWASP AI Testing Guide v1 gives a structure, but domain-specific thresholds still need calibration.)
- [ ] How should organizations manage rollback for models with stateful fine-tuning?
- [ ] Can SLSA (Supply-chain Levels for Software Artifacts) be adapted for ML model provenance? (ReversingLabs 2025 proposes extending SLSA provenance files with ML-specific metadata; not yet standardized)
- [ ] What lifecycle evidence should C3 retain when C14 kill-switch controls are activated against a deployed model, agent, or provider dependency?
- [ ] How should organizations handle provider auto-upgrade behavior (e.g., Azure's upgrade-on-expiry for pinned model deployments)?
- [ ] What is the minimum viable model identity information that providers should expose for audit compliance under the EU AI Act?
- [ ] How should a C3.1 model registry prove completeness while avoiding duplication with C6 AI-BOM and source-vetting records?
- [ ] What safety thresholds should apply to post-compression artifacts? Research shows quantization and pruning degrade alignment and robustness unpredictably -- standardized benchmarks for compressed model safety are still lacking.
- [ ] How quickly will OpenSSF Model Signing (OMS) achieve broad adoption beyond NVIDIA NGC? CoSAI is driving standardization but hub-level adoption on Hugging Face, Kaggle, etc. is not yet universal.
- [ ] Which reward hacking detection approaches (ARA, InfoRM, BNRM, KL penalty) will prove most practical for production RLHF pipelines? Current research shows promising AUC scores but no production-hardened implementations exist as of early 2026.
- [ ] Should hosted model providers be required to expose immutable model identifiers (not just aliases) in API responses? Current practice varies widely and creates audit gaps for consumers in regulated industries.
- [ ] How should organizations defend against cascading supply chain campaigns like TeamPCP (March 2026) where a single compromised security tool (Trivy) led to backdoored AI infrastructure packages (LiteLLM, 97M monthly downloads) within five days?
- [ ] What extra evidence should model registries require when a package or model has valid provenance but the trusted release workflow may have been compromised, as in the Mini Shai-Hulud wave against TanStack and Mistral AI packages in May 2026?
- [ ] Should system prompt changes be subject to the same staged deployment discipline (canary, blue-green) as model weight updates? The GPT-4o sycophancy incident (April 2025) suggests yes, but no framework currently mandates this.
- [ ] How can production inference servers (vLLM, Triton, TGI, SGLang) provide per-cohort isolation of KV caches, prompt caches, session state, and retrieval indices without sacrificing the throughput benefits that shared caches provide? AISVS requirement 3.3.3 mandates this isolation, but mainstream inference stacks default to shared caches across requests.
- [ ] How will the OCC's broader review of model-risk-management guidance (announced alongside OCC Bulletin 2025-26 in October 2025) treat LLMs and generative AI, and will other sector regulators (SEC, FINRA, NYDFS) follow with AI-specific MRM guidance?
- [ ] Will NIST publish a full AI RMF 1.1 version or continue releasing guidance through profiles (GenAI Profile 2024, Critical Infrastructure Profile concept note April 2026) and addenda? Federal contractors need clarity on which baseline to target.
- [ ] How will the August 2, 2026 activation of European Commission enforcement powers against GPAI providers -- including fines up to 3% of global turnover or EUR 15M -- change provider behavior around model identity disclosure, version tracking, and downstream documentation?
