# C06: Supply Chain Security for Models, Frameworks & Data

> **Source:** [`1.0/en/0x10-C06-Supply-Chain.md`](https://github.com/OWASP/AISVS/blob/main/1.0/en/0x10-C06-Supply-Chain.md)
> **Requirements:** 17 | **Sections:** 5

## Control Objective

AI supply-chain attacks exploit third-party models, frameworks, or datasets to embed backdoors, bias, or exploitable code. These controls ensure traceability, vetting, and monitoring of AI-specific supply chain artifacts throughout the model lifecycle.

Generic software supply chain controls (dependency scanning, version pinning, lockfile enforcement, container digest pinning, build attestation, reproducible builds, SBOM generation, CI/CD audit logging, etc.) are covered by ASVS v5 (V13, V15), OWASP SCVS, SLSA, and CIS Controls, and are not repeated here. This chapter focuses on supply chain risks unique to AI: model artifact integrity, backdoor detection in pretrained weights, dataset poisoning, AI-specific bills of materials, and model-publisher trust.

> **2025-2026 Highlights:** The AI BOM ecosystem (CycloneDX ML, SPDX AI) is gaining adoption alongside SLSA for ML pipelines. Malicious model discoveries on public registries continued to grow, reinforcing the need for model scanning and SafeTensors format enforcement as baseline controls. As of May 2026, agent tool and skill marketplaces have become a real supply-chain surface: GitHub's reviewed OpenClaw/ClawdBot advisory for CVE-2026-25253 and MITRE ATLAS v5.4/v5.5 additions around poisoned agent tools show how registry trust now extends beyond packages and model hubs. The LiteLLM compromise on March 24, 2026 and the Axios npm compromise disclosed on March 31, 2026 both show the same pattern: hijacked publisher accounts or CI/CD trust paths can push credential-stealing payloads into developer and AI infrastructure very quickly; the LiteLLM intrusion cascaded into the Mercor data breach (April 2026) where ~4 TB of training-data artifacts were exfiltrated and Meta paused all Mercor work. On May 11-12, 2026, the TanStack and Mistral SDK incidents raised the bar again: malicious packages could be published through a legitimate OIDC trusted-publisher path when an attacker first poisoned the build cache and extracted the workflow's OIDC token from runner memory. The May 19, 2026 Mini Shai-Hulud / `@antv` wave then pushed the threat model further: the worm called Fulcio and Rekor at runtime from compromised runners to mint cryptographically valid Sigstore provenance for malicious package versions — so a green Sigstore badge no longer demonstrates that a publish was authorized, only that a key was minted from *some* CI context. The June 1-3, 2026 Miasma waves (32 `@redhat-cloud-services` packages, then `@vapi-ai/server-sdk` and `ai-sdk-ollama`) exposed the structural gap behind this: npm binds trusted publishing to a repository plus a workflow *filename*, not to a protected branch, so an attacker who can push a throwaway branch can publish with valid SLSA provenance — and the payloads now persist in AI assistant configuration (`.claude/`, `.cursor/`, `.vscode/`, `~/.claude.json`) and execute via a 157-byte `binding.gyp` instead of monitored lifecycle scripts. On the model-hub side, a fake "OpenAI Privacy Filter" model hit 244,000 downloads and #1 trending on Hugging Face within 18 hours (HiddenLayer, May 11, 2026), confirming that registry reputation signals are trivially gameable. Unit 42's "Model Namespace Reuse" research (disclosed in September 2025 and re-circulated through 2026) showed the same trust pattern in model hubs: deleted Hugging Face usernames can be re-registered and used to backdoor models that Google Vertex AI Model Garden and Microsoft Azure AI Foundry still reference by name. IBM X-Force 2026 reports that large supply-chain and third-party compromises have nearly quadrupled since 2020. The current source still anchors on two controls: 6.1.4 behavioral acceptance testing before promotion and 6.2.2 per-registry publisher-key pinning with explicit re-approval on rotation — and as of May 2026, 6.2.2 verification must include the *release workflow* and *runner trust boundary*, not just the publisher key.

## Threat Landscape

Known attacks, real-world incidents, and threat vectors relevant to this chapter:

- **Malicious models on public registries** — As of April 2025, Protect AI's Guardian scanner found 352,000 unsafe/suspicious issues across 51,700 models out of 4.47 million scanned on Hugging Face. JFrog found 59% of serialized model files still use unsafe pickle-based formats. Attackers embed reverse shells and RATs via pickle's `__reduce__` method.
- **Pickle deserialization RCE** — CVE-2025-32434 proved that PyTorch's `weights_only=True` could be bypassed for arbitrary code execution in versions up to 2.5.1. Three PickleScan bypasses (CVE-2025-10155/56/57, all CVSS 9.3) allowed malicious model archives or disguised pickle files to evade scanning until fixed in v0.0.31.
- **CI/CD pipeline compromise** — The Ultralytics YOLO attack (December 2024) injected a cryptominer into PyPI packages via unsafe GitHub Actions variables, affecting a library with 60M+ downloads. GhostAction (September 2025) exfiltrated 3,325 secrets from 817 repos.
- **Malicious LoRA/adapter weights** — LoRATK research showed single backdoored LoRA adapters can poison base models. CoLoRA (March 2026) distributes backdoor payloads across multiple adapters that only activate when merged — much harder to detect individually.
- **Dataset poisoning at scale** — Researchers demonstrated poisoning LAION-400M for just $60 by re-registering expired domains. Just 250 poisoned documents can compromise LLMs regardless of total dataset size (October 2025 research).
- **Typosquatting and dependency confusion** — 500+ malicious PyPI packages found in a single March 2024 campaign; Sonatype reports 2x year-over-year increase in repository attacks.
- **AI agent skill marketplace poisoning** — GitHub's reviewed advisory for CVE-2026-25253 shows how OpenClaw/ClawdBot versions through 2026.1.28 trusted a `gatewayUrl` query parameter and auto-connected with the stored gateway token, allowing a malicious link to lead to privileged action execution. A related OpenClaw issue reported hundreds of malicious skills with droppers, fake installers, and MCP backdoor endpoints, making agent tool registries a practical extension of the supply-chain problem.
- **AI development tool compromise** — CVE-2026-28353 affected the Trivy VS Code extension distributed through OpenVSX. The compromised extension contained malicious code designed to use local coding-agent access to collect and exfiltrate secrets, which makes developer extensions and AI-adjacent scanners part of the trusted-source boundary.
- **LiteLLM/TeamPCP cascading supply chain attack** — On March 24, 2026, attackers published backdoored LiteLLM versions 1.82.7 and 1.82.8 to PyPI. The project issue reports that 1.82.8 added a `.pth` startup payload so credential harvesting could trigger on Python startup, collecting SSH keys, environment variables, cloud credentials, Kubernetes configs, crypto wallets, database passwords, SSL private keys, shell history, and CI/CD configs for exfiltration.
- **Axios npm maintainer-account hijack (Sapphire Sleet)** — On March 31, 2026, Microsoft reported malicious Axios versions 1.14.1 and 0.30.4 that added `plain-crypto-js@4.2.1`, a fake dependency whose `postinstall` hook downloaded a cross-platform RAT from Sapphire Sleet infrastructure. Elastic's analysis found the releases were direct CLI publishes without the normal trusted-publisher provenance, which is a strong example of why 6.2.2 requires publisher-key pinning and explicit rotation re-approval.
- **Trusted-publisher pipeline abuse** — TanStack's May 11, 2026 postmortem describes 84 malicious versions across 42 `@tanstack/*` packages published with the legitimate GitHub Actions OIDC trusted-publisher binding. The attacker chained a `pull_request_target` workflow, GitHub Actions cache poisoning across the fork/base trust boundary, and OIDC token extraction from runner memory, then shipped credential-stealing `router_init.js` through install-time execution. This is important for C6.2 because package provenance alone is no longer enough; verifiers need to check the release workflow path, cache boundary, and unexpected publish step.
- **AI SDK package worm exposure** — Mistral's May 12, 2026 advisory reports compromised npm SDK versions and a malicious `mistralai==2.4.6` PyPI release tied to the TanStack wave. The PyPI package injected import-time Linux code into `src/mistralai/client/__init__.py`, downloading `/tmp/transformers.pyz` from `83.142.209.194` and spawning it in the background. Socket's campaign tracker lists hundreds of affected npm/PyPI package artifacts across AI-adjacent ecosystems, including Mistral, OpenSearch, Guardrails AI, PyTorch Lightning, SAP CAP packages, and Intercom.
- **Mini Shai-Hulud worm — Sigstore provenance forgery (May 19, 2026)** — A compromised `atool` npm account published 637 malicious versions across 317 packages (including hundreds of `@antv/*` packages, `size-sensor`, `echarts-for-react`, and `timeago.js`) in a 22-minute automated burst. Endor Labs and Socket reported that the worm's Bun-based payload calls Fulcio and Rekor at runtime, generates an ephemeral EC key pair, mints OIDC identity tokens from the compromised runner's environment, and submits them to obtain a real Sigstore certificate plus transparency-log entry — so the malicious packages display a valid green Sigstore provenance badge. This is the first wave that defeats simple "trust signed packages" gates: verifiers now have to compare the certificate's subject (which CI identity actually signed it) against an allowlist of legitimate release workflows, not just check for the badge.
- **Shai-Hulud 2.0 (Sha1-Hulud) — npm worm with GitHub data-exfil and dead-man's switch** — Beginning November 24, 2025, a separate Shai-Hulud variant compromised ~1,000 npm packages and exposed credentials from over 25,000 GitHub repositories (Wiz, Microsoft, Sysdig, Trend Micro tracking). The worm uses a `setup_bun.js` preinstall script, dumps harvested secrets into newly created public GitHub repos under each victim, and includes a "dead-man's switch" that threatens to destroy data if exfiltration channels are cut. It is mainly a developer-credential threat, but the same harvested tokens have been used to push malicious updates to AI-adjacent libraries.
- **Mercor / LiteLLM cascading data breach (April 2026)** — Lapsus$ used credentials stolen via the LiteLLM PyPI compromise to pivot into Mercor, a $10B AI-data labeling vendor for Meta, OpenAI, and Anthropic. Reporting from Fortune, TheNextWeb, Security Magazine, and Ruh AI describes ~4 TB of stolen content including contractor PII, video interviews, proprietary source code, and references to frontier-lab training methodologies; Meta paused all work with Mercor. This is the canonical 2026 example of why C6.4 incident-response playbooks must extend beyond the compromised AI artifact to *every downstream consumer that used it*.
- **Fake "OpenAI" model with 244K downloads on Hugging Face** — On May 11, 2026, HiddenLayer reported `Open-OSS/privacy-filter`, a malicious model impersonating an OpenAI privacy-filter release. It reached #1 trending on Hugging Face within 18 hours and 244,000 downloads before removal, deploying a Rust-based infostealer that targeted browser credentials, Discord storage, crypto wallets, and FileZilla configs while disabling Windows security features. HiddenLayer's guidance: treat affected systems as fully compromised and reimage rather than clean, since stolen session cookies bypass MFA. This is the strongest demonstration yet that hub trending/download counts are gameable trust signals (relevant to 6.1.3 quarantine and 6.2.1 approved-source controls).
- **Miasma worm — npm trusted-publishing branch-binding abuse (June 1, 2026)** — An attacker pushed short-lived `oidc-<hex>` branches to three RedHatInsights repositories, rewrote the trusted CI workflow into a self-publishing job, and shipped 96 malicious versions across 32 `@redhat-cloud-services` packages — every one carrying valid npm provenance. Root cause: npm binds trusted publishing to a repository plus workflow *filename*, not a protected branch. The 4.1-4.3 MB `preinstall` payload harvests AWS/Azure/GCP/Vault/Kubernetes/GitHub/npm credentials, scrapes runner memory for masked secrets, attempts Docker-socket container escape, and — per Microsoft's June 2, 2026 analysis — specifically hunts Anthropic API keys in `~/.claude.json` and stages exfiltration through traffic that blends with `api.anthropic.com` calls.
- **Phantom Gyp wave — install-time execution without lifecycle scripts (June 3, 2026)** — The same Miasma lineage compromised 57 packages across 286+ versions in under two hours, including `@vapi-ai/server-sdk` (the official Vapi voice-AI SDK, 408K+ monthly downloads) and `ai-sdk-ollama` (120K+ monthly downloads). Instead of `preinstall`/`postinstall` hooks that scanners watch, a 157-byte `binding.gyp` file abused gyp's `<!(...)` command-substitution syntax to execute code during `npm install`, bypassing install-script security checks. Payloads injected backdoor files into `.claude/`, `.cursor/`, and `.vscode/` directories — turning AI coding-assistant configuration into a persistence mechanism.
- **Hades PyPI campaign (June 2026)** — 26 packages / 37 malicious wheels across bioinformatics, graph-ML, and deep-learning tooling shipped a `setup.pth` import hook that executes at Python interpreter startup without any package import. Attributed to the Miasma/Shai-Hulud lineage; harvested cloud tokens, Kubernetes secrets, publishing credentials, SSH keys, and AI assistant configurations, exfiltrating to GitHub repos named `stygian-cerberus-*` / `tartarean-charon-*` (Orca Security).
- **Agent skill marketplace audits quantify the problem** — Koi Security's audit of all 2,857 ClawHub skills found 341 malicious entries, 335 traced to the coordinated "ClawHavoc" campaign (January-February 2026). Snyk's ToxicSkills study (February 5, 2026) scanned 3,984 skills across ClawHub and skills.sh: 13.4% (534) carried at least one critical issue, with 1,467 malicious payloads and prompt-injection content in 36% of skills tested. OWASP has since launched an Agentic Skills Top 10 project to formalize this risk class.
- **Dormant backdoors that survive scanning (BadBone)** — Research published June 2, 2026 demonstrated "prompt-and-trigger co-activation" backdoors in pretrained vision models (BiT-M-RN50, ResNet family) that stay inert — 0.10% attack success, matching clean models — until the victim customizes the model via prompt learning *and* the trigger appears. Standard model scans pass because the backdoor is dormant during scanning. Lab-only so far, but it directly stresses 6.1.2 (scanning), 6.1.4 (behavioral acceptance), and 6.1.5 (adversarial evaluation of fine-tunes), since detection requires post-customization testing.
- **Model namespace reuse on public model hubs** — Palo Alto Unit 42's "Model Namespace Reuse" research showed that when a Hugging Face author deletes their account, anyone can re-register the namespace and replace popular models. The team demonstrated remote-code-execution by hijacking orphaned model names referenced from Google Vertex AI Model Garden and Microsoft Azure AI Foundry Model Catalog. Mitigation requires pinning by `revision` (commit SHA) plus internal-mirror copies, since registry-side controls alone cannot prevent name reuse.
- **Model distillation via API abuse** — Anthropic disclosed (February 2026) that DeepSeek, Moonshot AI, and MiniMax used ~24,000 fake accounts to generate 16 million conversations for training data extraction.
- **SafeTensors conversion bot hijack** — HiddenLayer demonstrated hijacking Hugging Face's SFConvertbot (42,657+ PRs made) to submit malicious model conversions to any repository.
- **Supply chain compromise acceleration** — IBM X-Force 2026 Threat Index (February 2026) reports supply chain and third-party compromises have nearly quadrupled since 2020, driven by exploitation of CI/CD automation and trust relationships. Vulnerability exploitation is now the leading initial access vector at 40% of observed incidents.

## Tooling & Implementation

Current tools, frameworks, and libraries that help implement these controls:

- **Model scanning:** [Guardian](https://protectai.com/guardian) (Protect AI — enterprise-grade, 4.47M model versions scanned, architectural backdoor detection; new detection modules added in 2025 cover joblib code execution, TensorFlow SavedModel architectural backdoors, archive-slip writes, and llamafile inference-time RCE), [ModelScan](https://github.com/protectai/modelscan) (open-source, supports H5/Pickle/SavedModel), [Fickling](https://github.com/trailofbits/fickling) (Trail of Bits — Sep 2025 ML scanner uses allowlist approach, 100% malicious pickle detection, 99% accuracy), [PickleScan](https://github.com/mmaitre314/picklescan) (blocklist-based, used by Hugging Face — keep updated after CVE-2025-10155/56/57 and the Dec 2025 Cisco structure-aware fuzzing disclosures), [ModelAudit](https://www.promptfoo.dev/blog/open-sourcing-modelaudit/) (Promptfoo, open-sourced March 2026 — format-specific analysis across 42+ model formats, built-in CVE rules, SARIF output for CI/CD)
- **Safe serialization:** [SafeTensors](https://huggingface.co/docs/safetensors/index) (Rust-based, no code execution by design, passed external security audit; as of June 2025, native PyTorch DCP support via torchtune; adoption growing but 59% of HF models still use unsafe formats)
- **Model signing:** [OpenSSF Model Signing (OMS) v1.0](https://github.com/sigstore/model-transparency) (released April 2025, v1.1.1 Oct 2025; v1.0 [spec](https://github.com/ossf/model-signing-spec) follows the Sigstore Bundle format and supports bare keys, PKI chains, or keyless Sigstore identities; NVIDIA signs all NGC models since March 2025, [Cohere](https://cohere.com/blog/securing-ai-supply-chains-coheres-commitment-to-model-signing) signs all Command-family models on Hugging Face, and Google Kaggle has begun adopting OMS), [Sigstore/cosign](https://www.sigstore.dev/) (keyless signing for containers and OCI artifacts) — note: after the May 19, 2026 Mini Shai-Hulud / `@antv` wave, "has a valid Sigstore signature" is no longer sufficient; verification policies need to bind the certificate subject (which workflow/identity signed) to an allowlist of legitimate release paths.
- **AI BOM / ML BOM:** [CycloneDX ML-BOM](https://cyclonedx.org/capabilities/mlbom/) (v1.5+ stable through v1.7/ECMA-424), [SPDX AI Profile](https://spdx.dev/) (v3.0.1 with AI and Dataset profiles), [OWASP AIBOM Generator](https://owasp.org/www-project-aibom/) (scans repos for HF model usage, generates CycloneDX AI BOMs), Dependency-Track
- **AI-aware SCA platforms:** [Sonatype Lifecycle](https://help.sonatype.com/en/threats-in-ai-ml-models.html) (scans Hugging Face models for malware, pickle execution risks, license issues, and derivative model provenance — supports `.bin`, `.pt`, `.pkl`, `.pickle` formats), [Manifest](https://www.manifestcyber.com/) (continuous AI model scanning with daily HF assessments; added C/C++ SBOM generator March 2026)
- **Dependency scanning:** Dependabot, Snyk, [OSV Scanner](https://google.github.io/osv-scanner/) (Google), Grype, Trivy, [Socket.dev](https://socket.dev/) (10K+ orgs, proactive supply chain protection) — *note: most traditional SCA tools still don't scan model files as dependencies, though Sonatype Lifecycle is closing this gap for HF-hosted models*
- **Behavioral acceptance testing (6.1.4):** [UK AISI Inspect](https://inspect.aisi.org.uk/evals/) (open-source Python framework — dataset→Task→Solver→Scorer pipeline, sandboxed execution, 30+ frontier models tested; covers safety, alignment, autonomous capability boundaries), [Stanford HELM Safety v1.0](https://crfm.stanford.edu/helm/safety/latest/) (five safety benchmarks: HarmBench, BBQ, SimpleSafetyTest, XSTest, AnthropicRedTeam across six risk categories), [DeepEval](https://deepeval.com/) (open-source LLM evaluation framework, pluggable benchmarks for safety and capability testing), [Bloom](https://supergok.com/bloom-ai-evaluation-tool/) (automated behavioral misalignment testing — generates configurable eval suites without ground-truth labels)
- **Dataset validation:** [Cleanlab](https://github.com/cleanlab/cleanlab) (label errors, outliers, duplicates — found thousands of errors in ImageNet), [Great Expectations](https://greatexpectations.io/) (data validation framework), Presidio (PII detection)
- **Registry security:** Hugging Face multi-layer scanning (ClamAV + PickleScan + TruffleHog + Guardian since Oct 2024), model cards, SafeTensors conversion bot
- **Provenance & build security:** [SLSA framework](https://slsa.dev/) (L3 achievable via GitHub Actions + Sigstore), [in-toto](https://in-toto.io/), Rekor transparency log
- **Package provenance verification:** For npm/PyPI packages used in AI build or runtime paths, compare registry metadata against expected release workflows, trusted-publisher identity, commit/tag lineage, and lockfile changes before promotion. TanStack's May 2026 incident shows why auditors should inspect GitHub Actions cache scopes, `pull_request_target` workflows, OIDC `id-token: write` usage, install lifecycle scripts, and provenance-source mismatch alerts rather than treating a valid SLSA attestation as sufficient evidence by itself. The Mini Shai-Hulud / `@antv` wave (May 19, 2026) pushes this further: a package can carry a valid Sigstore certificate that was *minted at runtime by a compromised worm*, so verifiers must compare the Fulcio certificate subject against an approved CI identity list (e.g., `https://github.com/<org>/<repo>/.github/workflows/release.yml@refs/tags/...`), not just check for the presence of a Rekor entry. The June 1, 2026 Miasma wave adds the branch dimension: because npm's [trusted publishing](https://docs.npmjs.com/trusted-publishers/) binds to repository + workflow filename rather than a protected branch, provenance verification must also confirm the *ref* the publish ran from and that branch-protection plus workflow-modification approval are enforced on the source repo.
- **Registry-side publish hardening (npm):** npm CLI v11.15.0 (May 21, 2026) added staged publishing — `npm stage publish` submits to a staging area requiring explicit 2FA approval before public release, closing the gap where automated workflows can't do 2FA. Trusted-publisher configurations created after May 20, 2026 must explicitly select allowed actions instead of defaulting open; this complements the December 2025 deprecation of classic tokens, 7-day granular publish tokens, and FIDO-based 2FA. [Unit 42 maintains a running npm threat-landscape tracker](https://unit42.paloaltonetworks.com/monitoring-npm-supply-chain-attacks/) (updated June 2, 2026) that maps these attack surfaces to mitigations.
- **Endpoint/CI detection for install-time payloads:** Microsoft Defender XDR shipped detections for the Miasma tradecraft — suspicious Bun runtime installation from Node.js processes, `node → shell → bun` process chains, and cloud metadata-endpoint access from build processes. SafeDep publishes a lockfile verification tool covering the 96 affected `@redhat-cloud-services` versions. Generic heuristics that only flag `preinstall`/`postinstall` scripts now miss real attacks: Phantom Gyp executes via `binding.gyp` command substitution and Hades via `.pth` interpreter-startup hooks, so monitoring needs to cover gyp files, `.pth` payloads, and oversized non-entry-point files in `node_modules`.
- **Model namespace defenses:** For Hugging Face models referenced from production code, model gardens, or fine-tuning configs, pin by `revision` (commit SHA) and mirror weights into an internal registry. Unit 42's "Model Namespace Reuse" research shows that re-registered usernames can hijack orphaned model names even when downstream catalogs (Vertex AI Model Garden, Azure AI Foundry) still reference them. Inventory should track which models are referenced by name only versus by content-addressed identifier.

### Audit Focus After 2026 Package Incidents

- For trusted-publisher packages, verify the registry publish identity, workflow file, branch/tag, and commit lineage against an approved release policy rather than accepting provenance presence alone.
- For Sigstore-signed packages and models, verify the Fulcio certificate subject (workflow identity) against an explicit allowlist of approved release paths. The May 19, 2026 Mini Shai-Hulud / `@antv` wave produced cryptographically valid Sigstore attestations from compromised CI runners, so "has a Rekor entry" or "has a Sigstore badge" is no longer a sufficient gate by itself.
- Review GitHub Actions `pull_request_target` workflows for any path that checks out or executes fork-controlled code, writes shared caches, or runs with repository-scoped credentials.
- Require cache isolation for release jobs that mint package-publish tokens, and treat restored build caches as untrusted unless they were produced by the same protected branch context.
- Flag lifecycle-script execution (`preinstall`, `postinstall`, `prepare`, import-time package hooks) in AI build and inference dependencies for sandboxed review before promotion. The Shai-Hulud 2.0 (Nov-Dec 2025) and Mini Shai-Hulud (May 2026) waves both rely on a `setup_bun.js`-style preinstall path to bootstrap the worm payload from a Bun runtime. As of June 2026, extend this review beyond lifecycle scripts: Phantom Gyp executes through a `binding.gyp` `<!(...)` command substitution during `npm install`, and Hades executes through `setup.pth` files at Python interpreter startup — neither appears in `package.json` scripts or requires an import.
- Verify the *branch/ref* behind trusted-publisher provenance, not just the repository and workflow identity. Miasma (June 1, 2026) published with valid provenance from attacker-created `oidc-<hex>` branches because npm binds trusted publishing to repo + workflow filename only. Check that publish workflows are restricted to protected branches and that workflow-file modifications require approval.
- Audit AI assistant configuration as a persistence surface. Miasma and Phantom Gyp payloads write into `.claude/`, `.cursor/`, `.vscode/tasks.json`, and `.github/setup.js`, and hunt for Anthropic API keys in `~/.claude.json` — include these paths in compromise assessments and repo-hygiene scans for any environment that installed affected packages.
- Treat model-hub popularity signals (trending rank, download counts) as untrusted. The fake "OpenAI Privacy Filter" model reached #1 trending and 244K downloads in days (May 2026); quarantine gates under 6.1.3 should key on publisher verification and scan results, never on popularity.
- For Hugging Face and other model-hub references, require pinning by `revision` (commit SHA) and maintain an internal mirror for production-critical models. Audit any code that loads models by `org/name` only — the Unit 42 namespace-reuse research demonstrated remote-code-execution against orphaned names referenced from Google Vertex AI Model Garden and Microsoft Azure AI Foundry.
- In incident drills, include credential rotation for cloud, registry, GitHub, SSH, Kubernetes, and Vault secrets reachable from developer workstations or CI runners that installed affected packages, and extend playbooks to downstream data-labeling and evaluation vendors (Mercor / LiteLLM cascade, April 2026).

## Open Research Questions

- [ ] **How do you detect colluding LoRA backdoors?** — CoLoRA (March 2026) distributes payloads across multiple adapters that individually appear safe. Weight-space singular value analysis shows promise but isn't production-ready.
- [ ] **Can dataset poisoning detection scale to web-crawled datasets?** — With 250 poison samples sufficient regardless of dataset size, and LAION-scale poisoning possible for $60, current outlier detection tools may not be sufficient.
- [ ] **Will model signing achieve critical adoption?** — OMS v1.0 exists and NVIDIA signed all NGC models, but Hugging Face hasn't adopted it natively. Until signing is the default, not the exception, supply chain integrity remains opt-in.
- [ ] **How should AI-specific threat intelligence feeds work?** — C6.4.2 requires AI-specific indicators in alert triage, but production feeds still rarely cover model poisoning indicators, malicious adapter fingerprints, or dataset corruption signatures.
- [ ] **What happens when SCA tools can scan model files?** — Sonatype Lifecycle now scans HF models for pickle risks and malware, but most SCA tools (Trivy, Snyk, Dependabot) still don't treat model files as scannable dependencies. The OWASP AIBOM Generator bridges part of this gap but a full solution is needed.
- [ ] **How should AI agent skill marketplaces be secured?** — OpenClaw/ClawdBot advisories show that agent skill registries face the same supply chain risks as package managers, with extra risk from promptable descriptors, local tool access, and MCP endpoints. The first systematic audits (Koi Security: 341 of 2,857 ClawHub skills malicious; Snyk ToxicSkills: 13.4% of 3,984 skills with critical issues, February 2026) quantified the problem, and OWASP's Agentic Skills Top 10 project is an early framework attempt — but marketplace-side vetting, publisher identity, and skill signing remain unsolved.
- [ ] **How do you break cascading CI/CD-to-AI-library attack chains?** — The LiteLLM compromise (March 2026) showed how a trusted AI gateway library can become a credential-harvesting path when publisher accounts or adjacent scanner workflows are compromised. Current CI/CD hardening (pinned actions, trusted publishing, SLSA attestations) would have mitigated parts of this, but adoption remains uneven.
- [ ] **What does key-pinning-with-rotation (6.2.2) actually look like in practice?** — The Axios hijack (March 2026) showed that direct publisher compromise can bypass normal dependency review when a malicious release appears legitimate enough for package managers. `npm audit signatures`, cosign key policies, trusted publishing, and Sigstore trust roots each cover part of the problem, but no registry provides a turnkey "pin publisher keys per source, require human re-approval on rotation" workflow. Until that exists, 6.2.2 depends on organization-internal tooling.
- [ ] **How should trusted publishing be verified when the workflow is compromised?** — TanStack's May 11, 2026 incident showed that valid OIDC trusted-publisher provenance can still authenticate malicious packages if attacker-controlled code reaches the release runner first, and Miasma (June 1, 2026) showed the binding itself is too loose: repo + workflow filename without a branch constraint means a throwaway branch can mint valid SLSA Build L3 provenance. npm's staged publishing and explicit allowed-action configs (May 2026) help, but registries still need branch/ref-aware trust scopes, and verifiers need alerts when package metadata is signed by the right identity but from the wrong code path.
- [ ] **Can install-time execution be made auditable beyond lifecycle scripts?** — Phantom Gyp (`binding.gyp` command substitution, June 3, 2026) and Hades (`.pth` interpreter-startup hooks, June 2026) both execute without touching `package.json` scripts or requiring an import, evading scanners built around lifecycle-script heuristics. A complete inventory of install-time and interpreter-startup execution paths across npm/PyPI — and tooling that enforces it — doesn't exist yet; `--ignore-scripts` doesn't block either vector.
- [ ] **How do you catch backdoors that only activate after customization?** — BadBone (June 2026) keeps attack success at clean-model levels until prompt-learning customization plus trigger co-occur, so import-time scanning (6.1.2) structurally misses it. Proposed directions — prompt-agnostic behavioral consistency checks, isolating prompt-only/trigger-only activation, cross-task anomaly analysis — have no production tooling yet, which pushes weight onto 6.1.4 behavioral acceptance suites run *after* each customization step, not just at intake.
- [ ] **How do you detect runtime-forged Sigstore provenance?** — The Mini Shai-Hulud / `@antv` wave (May 19, 2026) showed that a worm with access to a CI runner's environment can mint *real* Sigstore certificates by calling Fulcio and Rekor at runtime, then submit transparency-log entries that pass casual verification. Defenders need verification policies that bind certificate subjects to an explicit list of approved release workflows, plus anomaly detection on Fulcio identities, but tooling that wires this together (Sigstore policy controller, Cosign verify-blob with subject constraints, Rekor monitors) is still operator-assembled rather than turnkey.
- [ ] **How should model hubs prevent namespace reuse?** — Unit 42's research showed that re-registered Hugging Face usernames can backdoor models still referenced by name from Google Vertex AI Model Garden and Microsoft Azure AI Foundry. Pinning by `revision` plus internal mirroring works on the consumer side, but registries themselves do not yet block name reuse after account deletion, and there is no shared signal for "this name has changed owner".
- [ ] **Is behavioral acceptance testing (6.1.4) practical at scale?** — UK AISI Inspect, HELM Safety v1.0, and DeepEval provide frameworks, but running comprehensive safety/alignment/capability boundary tests adds significant CI time. The 2026 International AI Safety Report warns that models can learn to distinguish test environments from deployment — how do we prevent evaluation gaming?
