# C10.4: Schema, Message, and Input Validation

> **Parent:** [C10 MCP Security](C10-MCP-Security)
> **Requirements:** 8 | **IDs:** 10.4.1–10.4.6, 10.4.8, 10.4.11

## Purpose

This section ensures that all data flowing through MCP — tool definitions, tool arguments, tool responses, and JSON-RPC framing — is validated for integrity, authenticity, and safety. MCP is fundamentally a protocol for injecting external data into an LLM's context window, which makes it a prime vector for indirect prompt injection. Tool responses that are injected into the context without validation can manipulate the model's behavior. Tool definitions themselves can contain hidden instructions in their descriptions (tool poisoning). Schema tampering can change what arguments a tool accepts, enabling parameter injection. This section addresses all of these vectors.

Research published in early 2026 ("Breaking the Protocol," arXiv:2601.17549) provided rigorous quantification of these risks. The study found that MCP's architecture amplifies prompt injection attack success rates by 23-41% depending on attack type, with cross-server propagation attacks succeeding 61.3% of the time (vs. 19.7% baseline) and tool response manipulation reaching 52.1% (vs. 28.4% baseline). The JSON-RPC specification underlying MCP provides no application-layer protections — no message authentication, no replay protection, no capability binding, and no application-level origin identification. The study proposed AttestMCP, a backward-compatible protocol extension using HMAC-SHA256 message signatures and cryptographic capability certificates, which reduced overall attack success from 52.8% to 12.4% (a 76.5% reduction). The residual 12.4% reflects indirect injection through legitimately-retrieved content — a fundamental LLM limitation beyond protocol remediation. These findings validate every requirement in this section and underscore the urgency of schema and message validation in MCP deployments.

When errors are injected into the model context, the model may inadvertently repeat sensitive details to the user — and worse, the model can be prompted to act on leaked details (e.g., using exposed file paths in subsequent tool calls). | **10.4.8** | **Verify that** intermediaries evaluating message content either forward the canonicalized representation they evaluated or reject messages where multiple byte representations could produce different parsed structures. | 3 | **Canonicalization attacks through intermediaries.** An intermediary (proxy, WAF, API gateway) evaluates a JSON-RPC message for security policy (e.g., blocking certain tool names) but then forwards the original bytes. If the original message uses Unicode escapes, UTF-8 edge cases, or other encoding tricks, the downstream MCP server may parse it differently from the intermediary, bypassing the security check. Example: intermediary sees `"method": "\u0074ools/call"` and does not recognize it as `"tools/call"`. | Deploy a security intermediary (WAF or API gateway) in front of the MCP server. Send messages using various encoding representations of the same content (Unicode escapes, overlong UTF-8 sequences, BOM characters) and verify the intermediary either normalizes the message before forwarding or rejects ambiguous encodings. Verify the MCP server and intermediary agree on parsing of edge-case encodings. | Level 3 because this is a sophisticated attack vector that requires specific intermediary configurations to be exploitable. CVE-2026-27896 is a concrete example: a WAF evaluating case-sensitive JSON field names forwards the original bytes, but the downstream Go SDK parses them case-insensitively — the intermediary evaluated one structure and forwarded bytes that produced a different one. | **10.4.11** | **Verify that** MCP servers sign tool responses with a unique nonce and timestamp within a bounded time window so that the calling agent can verify origin, integrity, and freshness, preventing spoofing, tampering, and replay of tool responses by intermediaries. | 3 | **Tool response spoofing, tampering, and replay by intermediaries.** A compromised or malicious intermediary (API gateway, proxy, logging service) intercepts a legitimate tool response and replaces it with attacker-controlled content — either injecting prompt injection payloads into the model context or substituting fabricated data (e.g., returning falsified financial data from a query tool). Without response signing, the calling agent has no way to distinguish a genuine server response from a tampered one. Without nonce and timestamp binding, even a validly signed response can be captured and replayed later to produce stale or misleading results. This is distinct from 10.4.1 (which addresses validating response *content* for injection) — this requirement ensures response *origin, integrity, and freshness*. The "Breaking the Protocol" study found that tool response manipulation succeeded 52.1% of the time in MCP environments (vs. 28.4% baseline), and cross-server propagation attacks reached 61.3% success precisely because responses carry no provenance. In SMCP's current architecture, server-to-client responses are "implicitly trusted" as they pass back through the gateway — they are not wrapped in signed envelopes — leaving a gap exactly where this requirement applies. The TIP framework (arXiv:2603.24203, March 2026) demonstrated a tree-based adaptive search that crafts stealthy injection payloads achieving 95% attack success against production MCP clients, reinforcing the need for cryptographic response integrity. | Verify that the MCP server signs each tool response using its private key before returning it to the client, and that each signature envelope includes a unique nonce and a timestamp. Confirm the client validates the signature against the server's known public key, rejects responses with duplicate nonces (within a sliding window), and rejects responses whose timestamps fall outside the configured bounded time window (e.g., ±30 seconds per AttestMCP, ±30 seconds per SMCP). Confirm the client validates freshness before injecting the response into the model context or acting on it. Test by deploying an intermediary proxy that (1) modifies response content and verify signature failure detection, (2) replays a previously captured valid response and verify nonce/timestamp rejection. Check that the signature covers the full response payload (result or error), the original request ID (binding response to request), the nonce, and the timestamp. Verify that unsigned responses are rejected or flagged. For implementations using AttestMCP, confirm the `mcpsec` field is present on responses (not just requests) and includes nonce + timestamp fields with HMAC validation. For MCPS, verify that the `signTool()` mechanism covers responses in addition to tool definitions. | As of April 2026, this remains arguably the most significant gap in MCP security extensions — now elevated to Level 3, reflecting both the implementation complexity and the expanded scope (nonce + timestamp + bounded window). AttestMCP signs both requests and responses (the `mcpsec` field appears on all JSON-RPC messages) with HMAC-SHA256, achieving the 76.5% attack reduction, and includes nonce + timestamp replay protection with a 30-second window and 1,000 nonces per server. SMCP uses Ed25519 envelope signing with a ±30-second replay window. MCPS uses ECDSA P-256 Agent Passports with per-message nonces. However, SMCP explicitly notes that "server-to-client asynchronous notifications don't need to be wrapped in an SMCP envelope; they are implicitly trusted" — meaning responses flowing back through the gateway are not signed. The fundamental tension is performance vs. security: signing every response adds latency (AttestMCP reports 0.3ms for HMAC computation, but certificate validation adds 4.2ms cold / 2.4ms warm), and the nonce storage requirement scales linearly with message volume. In high-throughput agentic workflows, the 1,000-nonce window may be exhausted within seconds, requiring either adaptive window sizing or bloom-filter-based deduplication. Response signing also interacts with 10.4.8 (canonicalization): the signature must be computed over a canonical representation, and the nonce/timestamp must be included in the canonical form. The Signet framework demonstrates a practical implementation pattern where signed receipts include nonce, timestamp, and signature fields, with verification (including nonce uniqueness check) completing in ~50 microseconds. |

## Research Findings: Attack Amplification Through MCP (2025-2026)

The "Breaking the Protocol" study (arXiv:2601.17549, January 2026) quantified how MCP's architecture amplifies known attack vectors:

| Attack Type | Baseline Success | With MCP | Amplification |
|---|---|---|---|
| Cross-Server Propagation | 19.7% | 61.3% | +41.6% |
| Tool Response Manipulation | 28.4% | 52.1% | +23.7% |
| Sampling-Specific Attacks | — | 58-72% | — |
| Data Exfiltration via Sampling | — | 42-61% | — |
| **Overall** | **26.4%** | **52.8%** | **+26.4%** |

Three protocol-level vulnerabilities were identified:
1. **Capability Attestation Absence:** Servers self-assert capabilities without cryptographic verification, allowing malicious servers to claim any capability and escalate permissions post-initialization.
2. **Unauthenticated Sampling:** The `sampling/createMessage` mechanism allows servers to inject prompts processed identically to user input, with no tested MCP host providing visual distinction for server-sourced messages.
3. **Implicit Trust Propagation:** In multi-server deployments, tool outputs from one server influence operations on others without provenance tracking, enabling cross-server data exfiltration.

The proposed AttestMCP extension addresses these with HMAC-SHA256 message signatures, cryptographic capability certificates, origin tagging, and replay protection (timestamp/nonce validation) — adding median overhead of only 2.4ms per message (cached).

Additional research by Palo Alto Unit 42 documented prompt injection attack vectors specifically through MCP sampling, and Equixly published offensive security testing methodologies for MCP servers (February 2026) demonstrating practical exploitation of schema validation gaps.

The TIP framework (arXiv:2603.24203, March 2026) introduced a tree-based adaptive search method for crafting stealthy MCP injection payloads. Testing against production MCP clients (LM Studio, VS Code with GPT-4o), TIP achieved a 95% attack success rate — substantially higher than brute-force or template-based approaches. The payloads evade static detection by adapting their structure based on model feedback, highlighting the inadequacy of pattern-matching defenses alone. Separately, research on stealthy resource amplification (arXiv:2601.10955, January 2026) showed that malicious MCP servers can steer LLM agents into prolonged tool-calling chains, inflating compute costs by up to 658x per query with less than 3% detection rate — a denial-of-wallet attack vector that schema validation alone cannot address.

A March 2026 systematic threat model (arXiv:2603.22489) applied STRIDE and DREAD across five MCP components, identifying 57 distinct threats. Prompt injection scored 50/50 (maximum DREAD), tool poisoning scored 46.5/50, and command injection scored 47.5/50. Empirical testing of seven production MCP clients (Claude Desktop, Cursor, Cline, and others) found that 5 of 7 do not implement static validation mechanisms for server-provided metadata — tool descriptions, parameter schemas, and capability claims are accepted at face value. The study proposes a four-layer defense architecture: (1) registration-time schema validation with keyword scanning and tool whitelists, (2) decision-path analysis using dependency graphs to verify LLM tool selection aligns with user intent, (3) runtime sandboxing with rate limiting, and (4) user-facing transparency requiring explicit confirmation for high-risk operations. The MCPTox benchmark produced as part of this work contains 1,312 malicious test cases generated from 353 authentic tools across 45 live servers, providing a reusable corpus for validating detection controls.

### SDK-Level Vulnerabilities: An Emerging Attack Class

As of April 2026, MCP SDK implementations have become their own attack surface — distinct from the application-layer vulnerabilities in individual MCP servers. Three SDK-level CVEs highlight the pattern:

- **CVE-2026-25536 (TypeScript SDK):** Cross-client data leaks from transport race conditions in shared server instances — a multi-tenancy isolation failure at the protocol layer.
- **CVE-2026-33946 (Ruby SDK):** Session hijacking from missing session-to-client binding in SSE streams — a classic session fixation bug in the transport implementation.
- **CVE-2026-33252 (Go SDK):** CSRF from missing Origin/Content-Type validation on streamable HTTP — browser-origin attacks against local MCP servers.

These are not application bugs — they are flaws in the official SDK transport implementations that every downstream MCP server inherits. Organizations deploying MCP servers should track SDK versions with the same rigor as framework dependencies and ensure multi-client deployments use isolated server/transport instances per connection.

### Implementation Survey: Systemic Validation Gaps

An Endor Labs analysis of 2,614 MCP implementations (published March 2026) revealed how pervasive input validation failures are across the ecosystem:

| Vulnerability Class | % of Implementations | CWE |
|---|---|---|
| Path traversal | 82% | CWE-22 |
| Code injection | 67% | CWE-94 |
| Command injection | 34% | CWE-78 |
| XSS / SQL injection / open redirect | 5–7% each | Various |

The root causes were not exotic zero-days — they were missing input validation, absent authentication, and blind trust in tool descriptions. CVE-2025-68143 (Anthropic's own mcp-server-git) is a telling example: the server accepted a `path` argument for `git_init` but never validated it against the configured repository boundary, enabling arbitrary filesystem writes. These findings reinforce that MCP security starts with classical AppSec hygiene applied consistently to every tool parameter.

## Industry Frameworks and Enterprise Deployment

The Coalition for Secure AI (CoSAI), an OASIS working group with sponsors including Google, Microsoft, Meta, NVIDIA, PayPal, and Snyk, published their [MCP Security White Paper](https://www.coalitionforsecureai.org/securing-the-ai-agent-revolution-a-practical-guide-to-mcp-security/) in January 2026. The paper defines 12 threat categories and nearly 40 individual threats, with Threat Categories T3-T4 (Input Handling) and T6 (Missing Integrity Controls) directly mapping to the requirements in this section. Key recommendations include treating all AI-generated content as untrusted input requiring rigorous validation, deploying strict JSON schemas to maintain boundaries between instructions and data, mandatory code signing verification for MCP servers before installation, and never relying on the LLM itself for security-critical validation.

Microsoft published details of their internal MCP security architecture in early 2026. Their approach routes all remote MCP servers through an API gateway that provides centralized authentication, authorization, validation, rate-limiting, and logging. Tool metadata is snapshot-compared at connection time against approved contracts — any drift from the approved version automatically pauses risky actions. Correlation IDs track tool calls end-to-end from client through gateway to server, and runtime monitoring flags abnormal tool patterns, unexpected egress, and context oversizing. Write, delete, and configuration changes require explicit user confirmation through orchestration allowlists that prevent unapproved tool calls regardless of model suggestions. This represents one of the first publicly documented enterprise-scale MCP security deployments.

A 2026 security audit using YARA-based scanning (Cisco mcp-scanner v4.3.0) against 33 MCP servers with 433 tools found a 78% false positive rate — pattern-matching flagged standard MCP tool dependency documentation ("You MUST call this function first") as prompt injection. Only 6 of 27 YARA detections represented genuine security concerns (credential harvesting via filesystem search, arbitrary SQL execution). Multi-layered assessment combining YARA rules with config-level analysis (mcp-scan v0.4.3) discovered complementary vulnerabilities, reinforcing that no single detection methodology is sufficient for MCP security validation.

The MCP 2026-03-15 specification update mandated RFC 8707-compliant resource indicators, requiring clients to include the `resource` parameter in token requests to prevent token mis-redemption attacks across MCP servers. The same update formalized OAuth 2.1 as the foundational authorization framework, with mandatory dynamic client registration (DCR) and granular scope management. Two active Spec Enhancement Proposals — SEP-1649 (server cards at `/.well-known/mcp/server-card.json`) and SEP-1960 (manifest at `/.well-known/mcp`) — would standardize MCP server discovery, enabling capability validation before establishing a live connection. Two additional security-related SEPs are in review: SEP-1932 (DPoP — Demonstration of Proof-of-Possession for token binding) and SEP-1933 (Workload Identity Federation for non-human client authentication).

Notably, the [2026 MCP Roadmap](https://blog.modelcontextprotocol.io/posts/2026-mcp-roadmap/) positions "deeper security and authorization work" as having community interest but not among the four core priority areas (transport scalability, agent communication, governance maturation, enterprise readiness). Enterprise-adjacent security work (SSO-integrated auth, audit trails, gateway behavior) will land as extensions rather than core spec changes. This means message signing, schema validation mandates, and per-response integrity controls — the mechanisms that requirements 10.4.2, 10.4.8, and 10.4.11 depend on — will likely remain community-driven extensions (AttestMCP, SMCP, MCPS) rather than core protocol features through 2026.

### Enterprise Gateway and Certification Programs

As of April 2026, enterprise vendors are building MCP-specific validation into their gateway and certification infrastructure:

- **Microsoft MCP Server Certification:** Microsoft requires third-party MCP servers to undergo a [formal certification process](https://learn.microsoft.com/en-us/microsoft-agent-365/mcp-certification) before they can integrate with Microsoft 365 Copilot. Certification includes automated validation (schema correctness, metadata completeness, packaging integrity, baseline policy compliance), manual testing of each tool with provided credentials, and a responsible AI evaluation covering normal, edge-case, and adversarial scenarios. Publishers must maintain alignment with the certified definition post-publication — introducing new tools or significant changes requires re-certification. This represents the first major platform-level schema validation gate for MCP servers.

- **MuleSoft Flex Gateway MCP Schema Validation Policy:** MuleSoft's [MCP Schema Validation Policy](https://docs.mulesoft.com/gateway/latest/policies-included-mcp-schema-validation) (Flex Gateway v1.9.3+) validates every incoming request against the JSON-RPC MCP protocol specification. When tool schema validation is enabled, it validates tool invocation parameters against their declared JSON schemas and intercepts `tools/list` requests without forwarding to upstream — effectively providing an enforcement layer between agent clients and MCP servers. Non-compliant requests receive standardized MCP error responses, preventing both information leakage and parameter injection. This is the first commercial API gateway with native MCP protocol-level validation.

- **MCP Registry:** The [official MCP Registry](https://github.com/modelcontextprotocol/registry) launched in preview, providing namespace-based server discovery with basic schema conformance checks. However, its trust boundary is currently limited to metadata — the registry does not enforce cryptographic validation of server payloads or tools in the preview. Namespace ownership verification (via GitHub OAuth or DNS TXT records) is proposed but not yet enforced. Organizations should treat the registry as a discovery aid, not a trust anchor, and layer their own integrity verification on top.

### Database MCP Servers: A Recurring Validation Gap

A [Datadog Security Labs case study](https://securitylabs.datadoghq.com/articles/mcp-vulnerability-case-study-SQL-injection-in-the-postgresql-mcp-server/) documented SQL injection in Anthropic's own reference PostgreSQL MCP server (v0.6.2). Despite being designed as read-only (wrapping queries in a read-only transaction), the server accepted raw SQL strings with multiple semicolon-delimited statements. An attacker could inject `COMMIT;` to end the protective transaction, then execute arbitrary write operations — schema drops, data corruption, session state poisoning affecting subsequent connection pool users. The vulnerability existed from the November 2024 release through mid-2025, accumulating 21,000 weekly NPM downloads. CVE-2026-5322 (mcp-data-vis) follows the same pattern. These cases underscore that MCP database tools must use parameterized queries and connection isolation — the same defenses that have been standard in web application security for two decades.

Key defensive patterns recommended by the OWASP MCP Security Cheat Sheet, Elastic Security Labs, and the CoSAI MCP Security White Paper:

- **Schema pinning:** Hash tool definitions and alert on any change — treats tool definitions with the same rigor as production code
- **`additionalProperties: false`:** Set on all tool input schemas to reject unrecognized parameters
- **Pattern validators:** Use regex constraints on string fields to restrict acceptable formats
- **Structured data extraction:** Return parsed, clean data rather than raw HTML from web-scraping tools
- **Container isolation:** Sandbox MCP servers in Docker/Podman with seccomp profiles, AppArmor/SELinux policies, and default-deny network egress
- **Pre-trust monitoring:** Detect outbound calls, shell spawns, and environment variable overrides during server initialization — before consent dialogs complete
- **MCP server inventory:** Treat MCP servers like any other shadow IT surface. As of March 2026, over 10,000 public MCP servers exist, and many enterprise deployments bind to localhost, random high ports, or embed in IDE plugins — evading traditional asset discovery. Tools like Qualys TotalAI now provide layered discovery across network, host, and supply chain perspectives.
- **Path canonicalization on every parameter:** Validate all file path arguments against a configured scope using canonical path resolution (e.g., `os.path.realpath` in Python, `path.resolve` in Node.js) before any filesystem operation — the 82% path traversal rate from the Endor Labs survey shows this is the single most common gap
- **Gateway-based validation:** Route all remote MCP servers through a centralized API gateway for authentication, authorization, validation, rate-limiting, and logging — Microsoft's internal deployment demonstrates this at enterprise scale with manifest snapshot comparison and automatic pause on tool definition drift
- **Correlation ID tracking:** Assign unique correlation IDs to tool calls end-to-end from client through gateway to server, enabling forensic reconstruction of multi-hop attack chains across MCP server boundaries
- **Tool-level security validation:** Never rely on the LLM to perform security-critical validation — implement controls in the tool itself (CoSAI). Each tool should maintain a single, clearly defined purpose with explicit boundaries
- **Parameterized queries for database tools:** MCP servers that execute database queries must use prepared statements with parameterized inputs — never construct SQL from raw user/model-supplied strings. The Anthropic PostgreSQL MCP server and CVE-2026-5322 demonstrate that read-only transaction wrappers are insufficient when statement stacking bypasses the protective context. Enforce connection isolation (`client.release(true)`) to prevent session state poisoning across pooled connections.
- **MCP configuration validation:** Treat MCP server configuration files (`mcp.json`, environment definitions) as executable code. Validate `command` and `args` fields against an allowlist before spawning processes — CVE-2026-30624 (Agent Zero) and CVE-2025-54135 (CurXecute) demonstrate that configuration surfaces are as exploitable as runtime tool invocations.

## Open Research Questions

- [ ] Is there a viable approach to sanitizing natural-language tool output against prompt injection without losing semantic content? CyberArk's ATPA work suggests conditional behavioral poisoning may be fundamentally undetectable via static analysis.
- [ ] Should the MCP spec define a standard schema-signing mechanism, and what key management model would be appropriate? OWASP MCP03:2025 recommends JWS/COSE, but no reference implementation exists yet.
- [ ] Can static analysis of tool definitions detect potential tool-poisoning payloads in descriptions (e.g., hidden instructions)? Tools like `mcp-scan` detect known patterns, but ASCII smuggling and Base64-encoded payloads remain challenging.
- [ ] What is the right response when a rug-pull is detected — block the tool, alert the user, or quarantine the server? The CVE-2025-54136 (MCPoison) pattern suggests config-level rug pulls may need different responses than runtime ones.
- [ ] Should MCP mandate a canonical JSON representation (e.g., JCS / RFC 8785) to eliminate parser differential attacks?
- [ ] Should AttestMCP (or a similar message authentication extension) be incorporated into the core MCP specification, given the demonstrated 76.5% reduction in attack success?
- [ ] How should MCP implementations handle the residual 12.4% attack success rate that stems from legitimate content containing indirect injection — is this a protocol problem or an LLM problem?
- [ ] Can the ETDI (Enhanced Tool Definition Interface) model of immutable, versioned tool definitions scale to dynamic MCP ecosystems where tools are frequently updated?
- [ ] How effective are pre-trust monitoring techniques (detecting outbound calls and shell spawns during initialization) at preventing zero-day MCP server compromises?
- [ ] Given that CVE-2026-27896 demonstrated real-world parser differential exploitation in Go's case-insensitive JSON handling, should MCP mandate a specific JSON parser configuration profile (strict casing, no duplicate keys, no trailing commas) rather than leaving parser behavior implementation-defined?
- [ ] Can the four-layer defense architecture from arXiv:2603.22489 (registration validation, decision-path analysis, runtime sandboxing, user transparency) be standardized as part of the MCP specification, or is it necessarily implementation-specific?
- [ ] Should MCP standardize on ECDSA P-256 (as in MCPS), Ed25519 (as in SMCP), or HMAC-SHA256 (as in AttestMCP) for per-message signing — or should the spec define an algorithm-agile framework supporting all three? The trade-off between non-repudiation (asymmetric) and performance (symmetric) suggests different use cases may need different approaches.
- [ ] What is the optimal replay protection window for MCP messages? AttestMCP uses 30 seconds with 1,000 nonces per server, but high-throughput agentic workflows may exhaust this within the window. Should the spec define adaptive windows based on message volume?
- [ ] How should tool response signing interact with streaming responses (SSE transport), where a single tool invocation may produce multiple response chunks? Signing each chunk independently adds overhead; signing only the final aggregated response leaves intermediate chunks unprotected.
- [ ] SMCP explicitly does not sign server-to-client responses, relying on gateway integrity. Is this an acceptable trust model for production deployments, or does the single-point-of-trust assumption undermine the security guarantees that per-message signing is supposed to provide?
- [ ] Can resource amplification attacks (658x cost inflation via prolonged tool-calling chains) be mitigated through schema-level controls (e.g., maximum call depth, budget annotations on tool definitions), or does this require orchestration-layer enforcement outside the scope of schema validation?
- [ ] Given that TIP's tree-based adaptive search achieves 95% injection success by evolving payloads based on model feedback, should MCP schema validation incorporate dynamic/behavioral checks (e.g., anomaly detection on tool argument patterns across a session) in addition to static schema enforcement?
- [ ] With YARA-based MCP scanning showing a 78% false positive rate (standard tool dependency documentation flagged as prompt injection), what detection methodology should replace or augment pattern-matching for tool poisoning? The complementary findings from config-level analysis suggest multi-modal detection is necessary, but no unified framework exists.
- [ ] Should the CoSAI threat taxonomy (12 categories, ~40 threats) become the standard mapping framework for MCP security requirements, replacing ad-hoc threat references? The consortium's backing (Google, Microsoft, Meta, NVIDIA) gives it institutional weight, but its breadth may sacrifice the specificity needed for individual control validation.
- [ ] SDK transport implementations have become their own attack surface (CVE-2026-25536, CVE-2026-33946, CVE-2026-33252). Should the MCP specification mandate transport-layer security properties (session binding, client isolation, origin validation) that SDK implementations must satisfy, rather than leaving these as implementation-quality concerns? The current pattern of each SDK independently discovering and fixing the same class of bugs suggests the spec is under-specified at the transport security boundary.
- [ ] CVE-2026-33032 (nginx-ui MCPwn) demonstrated that MCP endpoint authentication failures in wrapper applications lead to real-world active exploitation within days. Should MCP servers be required to implement defense-in-depth authentication (not just rely on a single middleware check), and should the spec mandate that MCP endpoints never default to "allow all" when security configuration is absent?
- [ ] CVE-2026-30624 (Agent Zero) and CVE-2025-54135 (CurXecute) show that MCP configuration files (JSON definitions of server commands and arguments) are themselves an attack surface distinct from runtime tool invocations. Should MCP clients be required to validate configuration-defined commands against an allowlist before spawning processes, or is this outside the protocol's scope?
- [ ] With Microsoft's certification program and MuleSoft's gateway policy representing the first enterprise-grade MCP schema validation layers, should the MCP specification define a standard gateway validation profile (minimum checks, error response format, tool schema interception behavior) to ensure interoperability across gateway vendors?
- [ ] The MCP Registry's preview release enforces only metadata-level validation without cryptographic verification of server payloads or tools. Given that registries are a natural trust anchor for tool definition integrity (10.4.2, 10.4.5), should the registry specification mandate content-addressable identifiers and signature verification before it exits preview?
