# C3.5: Hosted and Provider-Managed Model Controls

> [Back to C03 Index](C03-Model-Lifecycle-Management.md)
> **Last Researched:** 2026-06-10

## Purpose

Hosted and provider-managed models may change behavior without notice. These controls help ensure visibility, reassessment, and safe operation when the organization does not control the model weights.

As of May 2026, hosted-model risk is not hypothetical. OpenAI's GPT-4o sycophancy rollback (April 2025) showed that an incremental model-behavior update can materially change safety posture even when the product name remains familiar; OpenAI's follow-up postmortem treated sycophancy as launch-blocking, and in February 2026 OpenAI retired five legacy ChatGPT models — including the sycophancy-prone GPT-4o build at the center of self-harm and AI-psychosis lawsuits. GPT-5.5 followed on April 23, 2026, introduced as a default ChatGPT model and a separate API model line, which means many production consumers crossed a behavior boundary inside the same product surface. Anthropic's Claude Code changelog records a March 4, 2026 configuration change where Opus 4.6 defaulted to `medium` effort for Max and Team subscribers, and on April 14, 2026 Anthropic announced retirement of `claude-sonnet-4-20250514` and `claude-opus-4-20250514` on June 15, 2026 with recommended migration to Sonnet 4.6 and Opus 4.7 — illustrating that hosted-model behavior can shift through configuration, model retirement, or default routing. Vertex AI followed its own schedule: Anthropic's Claude 3.7 Sonnet was shut down on May 11, 2026 with Claude 3.5 Haiku scheduled to follow on July 5, 2026, confirming that partner-operated platforms run retirement calendars independent of the upstream provider. xAI then introduced a more aggressive failure mode on May 15, 2026: eight Grok model slugs (`grok-4-1-fast-reasoning`, `grok-4-1-fast-non-reasoning`, `grok-4-fast-reasoning`, `grok-4-fast-non-reasoning`, `grok-4-0709`, `grok-code-fast-1`, `grok-3`, and `grok-imagine-image-pro`) were retired but the slugs continue to resolve, with traffic silently redirected to `grok-4.3` at flagship pricing of $1.25/$2.50 per million input/output tokens and a default reasoning effort of `low` or `none` depending on the requested slug — meaning availability monitoring stays green while billing, latency, output quality, and even the underlying image generator change underneath consumers who did not migrate.

The hosting topology itself moved in late April and May 2026. On April 28-29, 2026, AWS and OpenAI announced GPT-5.5, GPT-5.4, Codex, and a new category of Bedrock Managed Agents on Amazon Bedrock in limited preview (us-east-1 and us-west-2 first), exposed under the `arn:aws:bedrock:<region>::foundation-model/openai.gpt-5-...` ARN pattern and reached through the standard `InvokeModel`/`Converse` APIs — the first time OpenAI's frontier models are available outside Microsoft Azure since the 2019 partnership, which means the same `gpt-5.5` name now resolves on at least three control planes (OpenAI direct, Azure Foundry, and Amazon Bedrock) with different IAM, lifecycle, regional, and routing semantics. On May 11, 2026, Anthropic followed with the general availability of [Claude Platform on AWS](https://aws.amazon.com/about-aws/whats-new/2026/05/claude-platform-aws/) across 18 regions, a hosting path that is *operated by Anthropic* rather than AWS — customer data is processed outside the AWS security boundary, AWS provides only SigV4/IAM authentication, AWS Marketplace billing, and CloudTrail audit logging — distinct from Claude on Amazon Bedrock where AWS operates the inference stack and data stays inside the AWS security boundary. Anthropic's [model-deprecations page](https://platform.claude.com/docs/en/about-claude/model-deprecations) now makes the operator split explicit: the deprecation dates apply to *Anthropic-operated platforms* (the Claude API, Claude Platform on AWS, and Microsoft Foundry), while Amazon Bedrock and Vertex AI run partner-operated retirement schedules. Inventory and re-evaluation cannot rely on a model alias alone — they must capture the operator, control plane, regional footprint, and lifecycle channel for every hosted-model call path.

Provider lifecycle transparency has improved in places, but is still uneven. Microsoft Foundry publishes lifecycle stages and retirement tables and began auto-upgrading retired Azure OpenAI Standard SKU `gpt-4o` (2024-05-13 and 2024-08-06) deployments to `gpt-5.1` on March 9, 2026 with the cutover completing on March 31, 2026 — newly placing `gpt-5.1` Standard in eight US/Sweden regions where it had no prior Standard footprint — while Provisioned, Global Standard, and Data Zone Standard tiers were left on a 2026-10-01 retirement; Amazon Bedrock exposes `modelLifecycle` through `GetFoundationModel` and `ListFoundationModels`, and its cross-Region inference profiles can re-route a request to a different commercial region based on real-time capacity, with global profiles such as `global.anthropic.claude-sonnet-4-5-20250929-v1:0` causing `aws:RequestedRegion` to be set to `unspecified` (a behavior change that quietly breaks region-deny SCPs written against named regions); Google Vertex AI publishes stable-model retirement dates (Gemini 2.5 Pro, Flash, and Flash-Lite currently set to 2026-10-16, blocking new access one month earlier) and auto-updates Gemini aliases to newer stable versions; OpenAI exposes a `system_fingerprint` plus response `model` and `service_tier`; Anthropic exposes the response `model` plus a `request-id` header; OpenTelemetry GenAI conventions now treat `gen_ai.system` as deprecated in favor of `gen_ai.provider.name`, with the full attribute set still marked Development. The remaining control gap is that no cross-provider protocol guarantees immutable model identity, routing transparency, or advance notification for configuration-level changes, and the LiteLLM PyPI compromise on March 24, 2026 (CVE-2026-33634) followed by the LiteLLM SQL injection (CVE-2026-42208, CVSS 9.3, exploited in the wild within 26 hours of disclosure on April 19, 2026, added to the CISA Known Exploited Vulnerabilities catalog on May 8, 2026 with a federal civilian remediation deadline of June 5, 2026 after CISA recorded exploitation against U.S. financial-services and healthcare critical-infrastructure sectors) showed that the gateways consumers use to inventory hosted models are themselves a hostile target, and that both PyPI publishing and the gateway data plane have been weaponised within a single quarter. By June 2026 the gateway problem had escalated again: a third LiteLLM flaw in the same release window — CVE-2026-42271, a command injection in the MCP stdio test endpoints — was chained by Horizon3.ai researchers on June 1, 2026 with the Starlette "BadHost" header-validation bypass (CVE-2026-48710) into unauthenticated remote code execution scored CVSS 10.0, and CISA added CVE-2026-42271 to the Known Exploited Vulnerabilities catalog on June 8, 2026 with a federal remediation deadline of June 22, 2026 — making it the second LiteLLM KEV entry in a single month. The re-evaluation toolchain is also now a lifecycle risk in its own right: on June 3, 2026, OpenAI announced deprecation of its Evals platform (read-only October 31, 2026; dashboard and API shut down November 30, 2026) and Agent Builder, three months after acquiring promptfoo — so teams whose hosted-model regression gates run on provider-owned evaluation tooling have to track retirement calendars for the gates themselves, not just the models they gate.

Re-evaluation tooling itself now carries lifecycle risk: OpenAI announced on June 3, 2026 that its Evals platform goes read-only on October 31, 2026 and shuts down November 30, 2026, that Agent Builder retires after November 30, 2026, and that the `v1/prompts` reusable-prompt API shuts down the same day — alongside model retirement notices moving `gpt-5.2-chat-latest`/`gpt-5.3-chat-latest` to `gpt-5.5` (August 10, 2026), `computer-use-preview` to `gpt-5.4-mini` and `o3-deep-research` to `gpt-5.5-pro` (July 23, 2026), and GPT-4.5 out of service on June 27, 2026. The Digital Omnibus provisional political agreement of May 7, 2026 does not relax any of this for hosted-model consumers: it postpones *high-risk system* obligations (Annex III stand-alone systems to December 2, 2027; Annex I embedded systems to August 2, 2028) while leaving GPAI obligations, the August 2, 2026 enforcement window, and the Article 50 transparency duties (applying August 2, 2026, with a watermarking grace period to December 2, 2026) in place, and it hands the AI Office exclusive supervisory competence over AI systems built on a GPAI model by the same provider — formal adoption was still pending Official Journal publication as of early June 2026. NIST's April 7, 2026 AI RMF Profile on Trustworthy AI in Critical Infrastructure reorients the four AI RMF functions around the operator-vendor relationship and pushes hosted-model trustworthiness requirements down the supply chain. ISO/IEC 42001 frames traceability, transparency, reliability, and risk treatment as management-system concerns. Most teams still lack a crisp definition of "sufficient model identity"; Level 3 reflects the policy, procurement, and operational maturity required. *(Updated 2026-06)* |

## Notable Incidents

- **LiteLLM CVE-2026-42271 exploited in the wild and KEV-listed (June 1-9, 2026).** The third LiteLLM vulnerability of the quarter — an authenticated command injection in the MCP stdio test endpoints `POST /mcp-rest/test/connection` and `POST /mcp-rest/test/tools/list` (CVSS 8.7, affects `>=1.74.2, <1.83.7`, fixed in 1.83.7-stable by restricting the endpoints to the `PROXY_ADMIN` role) — escalated from "patch eventually" to "patch now" in nine days. The endpoints accepted full server configurations including `command`, `args`, and `env` fields, so any holder of a low-privilege internal-user key could run arbitrary commands with the proxy process's privileges and reach the stored upstream credentials. ## Implementation Maturity

- **Hosted-model inventory and identity logging:** *Emerging.* MLflow AI Gateway, LiteLLM (only on `1.83.7-stable` or later after the CVE-2026-33634 PyPI compromise, the CVE-2026-42208 SQL injection, and the KEV-listed CVE-2026-42271 MCP command injection), Langfuse, Braintrust, Portkey, and Helicone all capture provider/model identity to varying degrees, and OpenTelemetry GenAI conventions standardize the attribute names (with `gen_ai.system` now deprecated in favor of `gen_ai.provider.name`, per-provider convention pages for OpenAI/Anthropic/Bedrock, and native Datadog ingestion). Gateway *discovery* is improving too — runZero's June 2026 LiteLLM fingerprints make shadow-proxy sweeps practical. Coverage of Bedrock cross-Region inference profile destinations — particularly `global.*` profiles whose `aws:RequestedRegion` collapses to `unspecified` — Vertex routing, and provider-side configuration fingerprints is still uneven.
- **Provider lifecycle signals:** *Mature on the major hyperscalers, weak elsewhere.* Azure Foundry, AWS Bedrock, Google Vertex AI, OpenAI, and Anthropic now publish retirement schedules and at least partial behavior-change notices; Bedrock formalized a richer state machine in April 2026 (Active → Legacy → public extended access period → EOL, with a 15-day-inactivity access loss inside Legacy that inventory tooling should treat as its own alert condition); Anthropic now distinguishes Anthropic-operated platforms (Claude API, Claude Platform on AWS, Microsoft Foundry) from partner-operated ones (Bedrock, Vertex AI) on its deprecation page so consumers can tell which calendar applies to a given control plane. Smaller hosted-model providers and self-hosted gateways frequently expose no comparable feed.
- **Cross-platform model identity:** *Newly load-bearing.* As of April-May 2026 the same model alias can resolve on multiple operator boundaries — `gpt-5.5` on OpenAI direct, Azure Foundry, and Amazon Bedrock; `claude-opus-4-7` on the Claude API, Microsoft Foundry, Claude Platform on AWS, Bedrock, and Vertex AI — each with different IAM, retirement calendar, data-residency posture, and managed-agent features. Existing inventory and re-eval tooling treats the operator/control-plane as metadata rather than as part of the identity key, so most teams are one Bedrock-vs-direct re-route away from a silent compliance change.
- **Change-driven re-evaluation gates:** *Emerging, with new concentration risk.* Braintrust, MLflow, DeepEval, promptfoo (under OpenAI ownership since March 2026), Portkey, and Helicone provide the technical hooks; the gap is organizational — most teams do not have a documented gating policy keyed off model-identifier changes, fingerprint changes, or retirement notices. The June 3, 2026 deprecation of OpenAI's Evals platform (full shutdown November 30, 2026) adds a second-order problem: re-eval gates built on provider-owned evaluation products inherit those products' retirement calendars, so the gate infrastructure itself needs an entry in the hosted-dependency inventory.
- **Fail-closed/high-assurance policies:** *Emerging.* The EU AI Act GPAI enforcement window opening August 2, 2026 (left intact by the May 7, 2026 Digital Omnibus provisional agreement, which postponed only the high-risk system deadlines), NIST's April 2026 critical-infrastructure profile, and back-to-back CISA KEV listings of LiteLLM CVE-2026-42208 (May 8, due June 5) and CVE-2026-42271 (June 8, due June 22) are accelerating the move from informal acceptance to documented exception workflows, but most organizations still rely on implicit defaults.

## Verification Notes

For audits, the strongest evidence is a joined trail across five sources: a hosted-model inventory, production inference logs, gateway route history, the evaluation record that approved the model/configuration for the workflow, and the dependency manifest plus signing attestations for the gateway itself. The auditor should be able to pick one high-risk output and answer: which provider handled it, which model identifier the provider returned, which route or fallback policy selected it, which request configuration was active, which eval bundle approved that configuration, what would have happened if the provider failed to expose enough identity data, and whether the gateway version handling that request was on the known-good list at the time.

Requested model names are not reliable evidence by themselves. Treat aliases, deployment names, and gateway route labels as local pointers that must resolve to provider-returned identity, provider request IDs, and routing history at the time of the request.

## Provider Signal Checklist

For each production hosted-model call path, retain enough evidence to reconstruct these fields:

- Requested provider, endpoint, region, tenant, and local deployment or gateway route.
- Requested model alias and provider-returned model, deployment, or version identifier.
- Provider request ID, response ID where available, and backend fingerprint or service tier where exposed.
- Lifecycle state, retirement date, replacement model, and notification source for hosted catalog entries.
- Gateway fallback decision, policy rule, and approved fallback target when routing changes.
- Cross-Region inference profile destination or routed region when the provider abstracts location (Bedrock geographic/global profiles, Vertex global endpoints).
- Safety-relevant request configuration, including tools, system prompt hash, effort/reasoning mode, temperature, and top-p.
- Evaluation bundle, approval record, and exception expiry tied to the exact hosted-model configuration.
- Gateway/proxy version, integrity attestation, and dependency manifest hash for every component touching the request.
