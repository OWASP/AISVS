# C8.5: Scope Enforcement for User-Specific Memory

> [Back to C08 Index](C08-Memory-and-Embeddings.md)
> **Last Researched:** 2026-05-02

## Purpose

Prevent cross-tenant and cross-user leakage in retrieval, memory lookup, semantic caching, and prompt assembly. In multi-tenant RAG systems, similarity search is not an authorization decision: a chunk can be the nearest neighbor and still be illegal for the caller to see. Scope therefore has to be resolved from the authenticated session, pushed into the vector or search engine query, verified again after retrieval, and carried through any response cache or agent memory layer.

As of May 2026, the defensive baseline is better documented than it was a year ago. OWASP LLM08:2025 explicitly calls for permission-aware vector stores, strict logical partitioning, and immutable retrieval logs. Pinecone recommends one namespace per tenant for serverless multitenancy and documents BYOC as a public-preview zero-access data-plane option, not a general-availability replacement for tenant authorization. Weaviate stores each tenant on a separate shard once multi-tenancy is enabled. Qdrant recommends a single collection per embedding model with indexed payload partitioning for most large deployments, while noting that separate collections may be appropriate when the tenant count is small and isolation is more important than operational overhead.

The threat model now extends beyond the vector database. The March 2026 CERT/CC CrewAI advisory shows how direct or indirect prompt injection against an agent with code or RAG tools can chain into local file read, SSRF, and code execution across four CVEs: CVE-2026-2275, CVE-2026-2285, CVE-2026-2286, and CVE-2026-2287. The August 2024 Slack AI prompt-injection case showed the same pattern at the collaboration layer: retrieved content from one scope can influence answers shown in another scope when the application treats indexed workspace content as uniformly trusted. Scope enforcement for user memory therefore needs to include tools, caches, logs, internal admin paths, and agent-to-agent memory sharing, not only the happy-path vector query.

Use canary documents with unique phrases in every tenant and assert they never appear in another tenant's raw prompt, model output, trace, cache, or memory summary. Property-based tests with Hypothesis-style generation are valuable for filter tampering because the dangerous cases often come from type confusion and unusual combinations rather than natural-language prompts. |

## Implementation Notes

**Tenant context is not user input.** Build the retrieval scope from the authenticated session, source-system ACLs, policy engine, and data-classification labels. Do not let the user, the model, or a retrieved document provide `tenant_id`, namespace, collection, classification, or cache-key fields.

**Treat caches and logs as retrieval systems.** A response cache, semantic cache, trace store, evaluation dataset, or conversation memory can reintroduce cross-tenant leakage even when the vector database is configured correctly. Every secondary store should carry the same tenant, user, classification, policy-version, and revocation semantics as the original retrieval path.

**Separate query authority from tool authority.** Agent frameworks often combine retrieval, browsing, file loading, and code execution in one runtime. The CrewAI advisory is a reminder that a prompt-injection path into a tool-enabled agent can turn a memory-scope bug into SSRF, local file read, or code execution. Scope checks should apply before tool selection and again before any retrieved result is passed to a tool.
