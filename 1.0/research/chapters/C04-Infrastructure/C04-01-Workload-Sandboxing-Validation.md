# C4.1: AI Workload Sandboxing & Validation

[Back to C04 Index](C04-Infrastructure.md)

## Purpose

Isolate untrusted AI models in secure sandboxes and protect sensitive AI workloads using trusted execution environments (TEEs) and confidential computing technologies. AI systems increasingly execute models from external sources (open-source model hubs, third-party vendors, user-uploaded models). These models can contain arbitrary code (via pickle deserialization, custom operators, or embedded scripts) and must be treated as untrusted. This section addresses both defensive sandboxing of untrusted models and proactive protection of high-value models and data using confidential computing.

## Implementation Guidance

### Sandbox Technology Selection (2025-2026 State of Practice)

Choosing the right sandboxing technology depends on workload sensitivity, performance requirements, and operational maturity. The three dominant approaches have distinct security/performance tradeoffs:

**Firecracker MicroVMs** provide hardware-level isolation via KVM, with each workload running a dedicated Linux kernel completely separated from the host. Firecracker boots in approximately 125ms with less than 5 MiB of memory overhead per VM, supporting up to 150 VMs per second per host. This is widely regarded as the gold standard for running untrusted generated code in production. AWS Lambda and Fly.io use Firecracker in production at massive scale. Even Firecracker is not immune to vulnerabilities: CVE-2026-5747 (CVSS 8.7) exposed an OOB write in the opt-in virtio-pci transport allowing guest-to-host escalation (fixed in 1.14.4+ / 1.15.1+), and CVE-2026-1386 revealed a symlink race in the jailer initialization that allowed arbitrary host file overwrites (fixed in 1.13.2+ / 1.14.1+). The default MMIO transport is unaffected by the PCI vulnerability, but organizations using PCI for I/O performance should prioritize the upgrade.

**gVisor** intercepts system calls in user space via its Sentry process (implemented in memory-safe Go) before they reach the host kernel, reimplementing ~200 Linux syscalls while exposing only ~70 syscalls to the host kernel -- down from the ~340 that standard containers expose. This dramatically reduces the kernel attack surface and has historically protected GKE workloads from kernel CVEs that affected unprotected containers (e.g., CVE-2020-14386). gVisor imposes 10-30% overhead on I/O-heavy workloads but has fast startup. It is best suited for compute-heavy AI workloads (such as inference) where full VM isolation is not required. Google uses gVisor extensively across GKE and Cloud Run. Important GPU caveat: gVisor's `nvproxy` GPU support only allows CUDA compute and utility capabilities by default, and does not introduce hardware-level GPU isolation beyond what the NVIDIA kernel-mode driver configures -- it intercepts `ioctl(2)` calls against an allowlist but cannot protect against all NVIDIA driver vulnerabilities. For full GPU passthrough isolation, Kata Containers (one GPU per pod) or Firecracker with VFIO are required.

**Kata Containers** orchestrate multiple VMMs (Firecracker, Cloud Hypervisor, QEMU) through standard container APIs with native Kubernetes integration. Boot time is approximately 200ms with minimal memory overhead. Kata Containers are ideal for regulated industries requiring zero-trust environments with Kubernetes-native orchestration. Treat the runtime/VMM combination as patch-managed infrastructure, not a static security boundary: CVE-2026-24054 in versions before 3.26.0 can let malformed no-layer images hotplug a host block device into the guest and force host filesystem failure; CVE-2026-24834 in versions before 3.27.0 affects Kata with Cloud Hypervisor and can let a container user modify the guest microVM filesystem and gain root inside that VM. Verification should include the Kata release, selected VMM, snapshotter/rootfs driver, runtime class, and whether privileged containers, host devices, or `CAP_MKNOD` are permitted.

**Google Agent Sandbox**, launched at KubeCon NA 2025 as a CNCF project under Kubernetes SIG Apps, provides a declarative API for managing isolated, stateful sandbox pods on Kubernetes clusters specifically for AI agent workloads.

**Docker Sandboxes**, launched in March 2026 as an experimental feature, runs AI coding agents inside dedicated microVMs with their own Linux kernel, Docker daemon, filesystem, and network stack. It supports Claude Code, Gemini CLI, Codex, Copilot, and other AI agent tools out of the box. Each sandbox provides hardware-level isolation -- a kernel exploit inside the sandbox compromises only that sandbox's kernel, not the host. The network model is especially relevant to requirement 4.1.1: raw TCP, UDP, ICMP, private IP ranges, loopback, and link-local access are blocked; DNS and HTTP(S) egress are mediated by policy-aware host proxies; and model-provider credentials are injected by the host proxy rather than stored inside the VM. Because Docker still labels the feature Early Access / Experimental, production use should require independent compensating controls and audit evidence rather than relying on the product label alone.

**NVIDIA OpenShell** (March 2026, open source) is a security runtime purpose-built for autonomous AI agents. Rather than replacing a VM or container boundary, OpenShell wraps agents with out-of-process policy enforcement across filesystem, network, and process actions. Every action can be evaluated at the binary, destination, method, and path level, with default-deny semantics -- anything not explicitly allowed is blocked. Popular coding agents can run unmodified inside the runtime, but auditors should review the actual policy loaded for the agent: NVIDIA's default policy reference currently lists full out-of-the-box coverage for Claude Code, partial coverage for OpenCode, and no default coverage for Codex without a custom policy. OpenShell is complementary to microVM isolation: it provides fine-grained runtime policy enforcement within an already-isolated environment, addressing the gap between coarse hardware boundaries and application-level permission scoping.

**Microsoft LiteBox** (February 2026, open source) is an experimental Rust-based library OS that rethinks sandboxing for modern workloads. Instead of intercepting syscalls (gVisor) or running full VMs (Firecracker), LiteBox bundles minimal OS functionality as libraries alongside each application, providing a narrowly scoped kernel interface tailored to the workload. LiteBox supports AMD SEV-SNP confidential computing, enabling workloads to execute inside encrypted memory regions. While still experimental with no stable releases, the library OS approach represents an emerging fourth isolation paradigm alongside containers, microVMs, and user-space kernels. Microsoft has not confirmed whether LiteBox will integrate into Azure or WSL.

As of April 2026, at least twelve major platforms have converged on microVM isolation for AI agent workloads (E2B, Fly.io Sprites, Docker Sandboxes, Vercel Sandbox, AWS Bedrock AgentCore, Northflank, and others), establishing microVMs as the de facto standard for untrusted AI code execution. Firecracker's codebase is approximately 83,000 lines of Rust versus QEMU's 1.7 million lines of C, representing a dramatically reduced attack surface. Snapshot-restore enables 28ms boot times for pre-warmed environments.

For production deployments executing untrusted models or generated code, Firecracker microVMs or Kata Containers are recommended. The hardware boundary they provide prevents entire classes of kernel-based attacks that user-space isolation (containers, gVisor) cannot fully address.

### WebAssembly (Wasm) for AI Workload Sandboxing

WebAssembly has emerged as a lightweight sandboxing option for AI workloads, particularly at the edge and for agentic AI workflows. Key developments in 2025-2026:

- **NVIDIA published guidance** on using Wasm (via Pyodide, a CPython port compiled to WebAssembly) to sandbox LLM-generated Python code execution in agentic workflows. The approach shifts code execution from application servers to browser sandboxes, providing two protective layers: execution failure in the restricted runtime and browser-level isolation of any successful exploit.
- **Microsoft Wassette** (August 2025) is a security-oriented runtime that executes WebAssembly Components via Model Context Protocol (MCP), allowing AI agents to autonomously fetch Wasm Components from OCI registries and execute them securely using the Wasmtime runtime.
- **WASI AI** enables Wasm modules to interact with ML models and hardware accelerators, supporting inference at the edge with minimal latency. AWS Lambda now supports Wasm functions as a first-class runtime with 10-40x cold start improvements over containers.
- **wasmCloud** supports distributed ML and AI workloads with Wasm-based isolation.

However, Wasm sandboxing is not without risk. Documented Wasm runtime CVEs include: CVE-2024-30266 (Wasmtime type confusion in externref handling causing host-side memory disclosure), CVE-2023-51661 (Wasmer WASI filesystem restriction bypass enabling unauthorized file access), and CVE-2023-2033 (V8 type-confusion exploit achieving arbitrary write primitives on the host, exploited in the wild). As of April 2026, CVE-2026-34971 (CVSS 9.0) exposed a critical sandbox escape in Wasmtime's Cranelift compiler backend on AArch64: a logic error in ISLE instruction selection rules caused bounds-check calculations and actual memory loads to diverge, giving attackers arbitrary read/write across the host process address space. The vulnerability requires 64-bit Wasm linear memory with Spectre mitigations and signals-based traps both disabled -- a configuration some edge AI deployments use for performance. Fixed in Wasmtime 36.0.7, 42.0.2, and 43.0.1. Attack vectors include linear memory overflows in modules compiled from C/C++, type confusion in indirect function tables allowing function call redirection outside the sandbox, and JIT-compiler bounds check elision creating time-of-check/time-of-use windows. Malicious AI model files could trigger specific JIT-compiler bugs during weight compilation, compromising inference servers. Wasm should be treated as a defense-in-depth layer, not the sole isolation mechanism for high-risk workloads.

### Model Format Validation and Pre-Deployment Scanning

The pickle deserialization threat remains one of the most critical attack vectors in the AI supply chain. Research from 2025-2026 underscores that model scanning tools are necessary but insufficient on their own:

**PickleScan Bypass Vulnerabilities (2025):** JFrog Security Research discovered three critical zero-day vulnerabilities (CVE-2025-10155, CVE-2025-10156, CVE-2025-10157, all CVSS 9.3) in PickleScan, the most widely deployed model scanning tool used by platforms including Hugging Face:

- **CVE-2025-10155 (File Extension Bypass):** PickleScan prioritizes file extension checks over content analysis. Renaming malicious pickle files with PyTorch extensions (.bin, .pt) causes PickleScan to attempt PyTorch-specific parsing, fail silently, and return no error -- while PyTorch identifies file types by content and loads the malicious payload successfully.
- **CVE-2025-10156 (CRC Bypass):** Attackers introduce intentional CRC errors in ZIP-archived models. PickleScan (using Python's zipfile module) raises exceptions and fails, while PyTorch's model loader bypasses CRC validation and loads the content.
- **CVE-2025-10157 (Unsafe Globals via Subclasses):** PickleScan checks exact module names against its blocklist, but attackers use subclasses of dangerous imports (e.g., internal asyncio classes). PickleScan marks these as "Suspicious" rather than "Dangerous," failing to block them.

These vulnerabilities were fixed in PickleScan 0.0.31+ (September 2025), but they expose a systemic problem: PickleScan and PyTorch handle files differently, creating exploitable semantic gaps.

**Sonatype PickleScan Vulnerabilities (2025):** Independently, Sonatype discovered four additional CVEs in PickleScan, all fixed in version 0.0.23:

- **CVE-2025-1716:** Unsafe deserialization bypasses static analysis, enabling arbitrary code execution via malicious pip package installation during model loading.
- **CVE-2025-1889:** Hidden pickle files embedded in PyTorch archives with non-standard naming conventions evade detection but are loaded by `torch.load()`.
- **CVE-2025-1944:** ZIP filename tampering causes PickleScan to crash with BadZipFile errors, while PyTorch's lenient ZIP handling loads the malicious payload.
- **CVE-2025-1945:** Modified ZIP flag bits prevent detection of malicious pickle files, though PyTorch deserializes them successfully.

Between the JFrog and Sonatype disclosures, a total of seven distinct PickleScan bypass techniques were documented in 2025 alone, underscoring that blocklist-based scanning is fundamentally fragile against determined adversaries.

**nullifAI Attack Technique (February 2025):** ReversingLabs discovered two malicious models on Hugging Face that evaded the platform's security scanning by using 7z compression instead of the ZIP format PyTorch traditionally uses. Hugging Face's PickleScan-based scanner did not process 7z-compressed archives, while PyTorch's model loader handled them without issue, creating yet another format-handling semantic gap. The malicious payloads contained reverse shells connecting to hardcoded attacker IPs. This technique is distinct from the JFrog and Sonatype bypasses and represents an eighth documented evasion vector against model scanning tools.

**PickleBall (CCS 2025):** Academic research from Brown University introduced PickleBall, a secure deserialization framework that performs fine-grained analysis of pickle bytecode rather than relying on blocklist matching. Evaluated model scanners fail to identify known malicious models, while PickleBall achieves higher detection rates with fewer false positives on benign models.

**SafePickle (February 2026):** A newer approach using robust ML-based detection of malicious pickle models, addressing the limitations of signature-based scanning.

**Hydra Configuration RCE (Unit42, January 2026):** Palo Alto Unit42 discovered a class of RCE vulnerabilities affecting AI libraries that use Hydra's `instantiate()` function for model configuration. Attackers embed arbitrary Python callables (e.g., `builtins.exec()`) as `_target_` values in model metadata files, achieving code execution when the model is loaded. Affected libraries include NVIDIA NeMo (CVE-2025-23304, fixed in v2.3.2), Salesforce Uni2TS (CVE-2026-22584, fixed July 2025), and Apple/EPFL ml-flextok (patched June 2025). Over 700 HuggingFace models use NeMo format, and approximately 100 Python libraries on HuggingFace use Hydra for configuration. This vulnerability class operates independently of pickle -- even SafeTensors-based models can be exploited if the library unsafely instantiates configuration metadata.

**Recommended defense-in-depth for model validation:**
1. **Prefer safe serialization formats.** Use SafeTensors for model weights wherever possible. SafeTensors stores only raw tensor data with JSON-based headers and eliminates the possibility of arbitrary code execution by design.
2. **Use PyTorch safe-loading controls, but do not over-trust them.** PyTorch 2.6+ defaults `torch.load()` to `weights_only=True` when no custom pickle module is passed, and `TORCH_FORCE_WEIGHTS_ONLY_LOAD=1` can force that behavior across call sites. PyTorch's own documentation is explicit that this narrows remote-code-execution risk but does not prevent denial-of-service or all memory-corruption cases, so it is a compatibility backstop rather than a primary trust boundary. CVE-2026-24747 (CVSS 8.8, fixed in PyTorch 2.10.0) drove this point home by making the `weights_only` unpickler *itself* exploitable: malformed pickle opcodes and storage metadata in a `.pth` file corrupt heap memory and can reach arbitrary code execution even when `weights_only=True` is set, on every release through 2.9.1. Pin to 2.10.0+ and still sandbox untrusted checkpoints -- the "safe" loader is only as safe as its own parser.
3. **Scan all model files in CI/CD.** All model binary files (.bin, .pkl, .pt, .pth, .ckpt) must be scanned using tools like modelscan, picklescan (0.0.31+), or PickleBall. If a scan returns "dangerous," the pipeline must block deployment.
4. **Fail closed.** If an automated model scan fails for any reason (timeout, parsing error, tool crash), immediately quarantine the model and prevent distribution.
5. **Sandbox model loading.** Even scanned models should be loaded inside sandboxed environments, because scanning tools and safe-loading modes have known bypass or residual-risk classes. Scanning reduces risk but does not eliminate it.
6. **Maintain an approved model registry.** Curate an internal registry of validated models with cryptographic signatures. Sandbox anything not in the approved registry.

Approximately 44.9% of model repositories on public hubs still contain pickle-format models, making this a persistent and widespread attack surface. In February 2025, researchers from Reverse Engineering Labs uncovered malicious ML models on Hugging Face that exploited "broken" pickle files to evade platform detection mechanisms and execute reverse shells.

**GGUF chat-template SSTI: the "format-safety" gap (CVE-2024-34359 -> CVE-2026-5760):** GGUF stores not just weights but model-supplied metadata, including a `tokenizer.chat_template` field that inference servers render with Jinja2 at every prompt. When that rendering uses the default `jinja2.Environment()` instead of `ImmutableSandboxedEnvironment`, the field becomes an arbitrary-code-execution sink even though the GGUF file itself contains no Python pickle. Checkmarx's CVE-2024-34359 ("Llama Drama", CVSS 9.6) disclosed this pattern in llama-cpp-python in May 2024, with HuggingFace inventorying roughly 6,000 affected models at the time; the maintainer shipped sandboxing in v0.2.72. Two years later, the same class of bug re-appeared in SGLang as CVE-2026-5760 (CERT/CC VU#915947, April 20 2026, CVSS 9.8) -- a malicious GGUF model that hits the `/v1/rerank` endpoint triggers SSTI via the unsandboxed template environment, achieving full RCE in the inference server's process. As of May 2026, SGLang has not responded to CERT/CC coordination, no patch is available, and a public proof-of-concept against SGLang 0.5.5-0.5.9+ is published. Auditors should grep inference-server code for `jinja2.Environment(` and `Template(` outside `jinja2.sandbox`, and prefer `ImmutableSandboxedEnvironment` for any path that renders model-supplied templates. HuggingFace publishes `gguf-jinja-analysis`, a Rust + dynamic/static analysis tool that fetches GGUF headers via HTTP RANGE requests and tests chat templates inside `SandboxedEnvironment`; an internal sweep of 116,000+ models found only one confirmed dangerous template, which makes it practical to wire into model-intake pipelines. Related precedent: CVE-2025-61620 in vLLM's OpenAI-compatible server lets user-supplied Jinja2 templates exhaust CPU and memory through nested-loop DoS even when the sandbox prevents code execution, so template-rendering boundaries also need resource limits, not just sandboxing.

**SafeTensors conversion pipeline as an attack surface:** Even the migration to SafeTensors introduces its own risks. HiddenLayer (February 2024) demonstrated "Silent Sabotage" -- an attack against Hugging Face's automated SafeTensors conversion service (SFConvertbot). A crafted malicious PyTorch binary triggers arbitrary code execution during `torch.load()` in the conversion pipeline, enabling theft of the SFConvertbot authentication token. With that token, an attacker could impersonate the official bot and submit poisoned pull requests to any model repository, creating a supply chain attack vector that scales to models with millions of monthly downloads. This underscores that format-level safety only applies at the loading boundary -- any process that ingests pickle-format models as input (conversion services, scanning pipelines, evaluation frameworks) must itself run in a sandboxed environment.

### Container and Sandbox Escape: Recent Incidents and Research (2025-2026)

The AI workload sandboxing threat landscape has shifted significantly, with several high-profile escapes demonstrating that isolation boundaries require constant vigilance:

**NVIDIAScape (CVE-2025-23266, CVSS 9.0, July 2025):** Wiz Research discovered that a three-line Dockerfile could exploit an OCI hook misconfiguration in the NVIDIA Container Toolkit to achieve full root access on the host. The `createContainer` hook trusted unfiltered environment variables, allowing `LD_PRELOAD` injection to load attacker-controlled libraries. Approximately 37% of cloud environments were susceptible. NVIDIA's November 2025 bulletin lists fixed versions as Container Toolkit 1.17.8, Kubernetes Device Plugin 0.17.3, MIG Manager 0.12.2, and GPU Operator 25.3.2. This is particularly relevant because virtually all GPU-accelerated AI workloads use this toolkit.

**runC Triple Vulnerability (November 2025):** Three vulnerabilities in runC created container breakout vectors across the entire container ecosystem:
- CVE-2025-31133: Symlink replacement targeting procfs files to bypass maskedPaths protections
- CVE-2025-52565: Timing-based attacks during container initialization bypassing security restrictions
- CVE-2025-52881: Write redirection to critical system files enabling host crashes and full breakouts

These affected Docker, Kubernetes, containerd, and CRI-O. Patched in runC 1.2.8+, 1.3.3+, or 1.4.0-rc.3+.

**N8scape (CVE-2025-68668, CVSS 9.9):** A Pyodide sandbox escape in n8n's Python Code Node allowed authenticated users to execute arbitrary commands on the host. The root cause was a blocklist-based sandbox that could not enumerate every dangerous capability -- attackers used `_pyodide._base.eval_code()` to bypass restrictions. This demonstrates why blocklist-based sandboxing approaches are fundamentally weaker than capability-based or hardware-enforced isolation for AI code execution.

**Agenta-API Sandbox Escape (CVE-2026-27952, CVSS 8.8):** A Python sandbox escape in Agenta-API's custom code evaluator (versions prior to 0.48.1) allowed authenticated users to gain arbitrary code execution, further illustrating the fragility of application-level sandboxing in AI evaluation platforms.

**OpenClaw Docker Config Escape (CVE-2026-27002):** A configuration injection vulnerability in OpenClaw's Docker tool sandbox allows attackers who control sandbox configuration to specify dangerous Docker options, achieving container escape and host compromise. This highlights that sandboxing implementations must validate and restrict configuration parameters, not just runtime behavior.

**AWS Bedrock AgentCore Sandbox Bypass (Unit42 and BeyondTrust, March-April 2026):** Unit42 and BeyondTrust each showed that AgentCore's Code Interpreter Sandbox mode allowed DNS tunneling for data exfiltration and C2 while blocking broader outbound traffic. BeyondTrust updated its report on April 22, 2026 to state that AWS had remediated DNS-based exfiltration, and AWS documentation now describes Sandbox mode as "limited external network access" with DNS resolution and S3 data operations. Unit42's second part, "Agent God Mode," showed that overly permissive IAM roles can let a compromised agent exfiltrate proprietary ECR images, access other agents' memories and conversation histories, invoke all code interpreters in the account, and extract sensitive configuration. The research demonstrates that cloud-managed AI sandbox services require the same scrutiny as self-hosted isolation -- "managed" does not mean "secure by default."

**MaxKB AI Assistant Sandbox Escape (CVE-2026-39421, CVSS 6.3):** The ToolExecutor sandbox in MaxKB (an open-source enterprise AI assistant) versions 2.7.1 and below used LD_PRELOAD to intercept dangerous libc calls (execve, system, connect, mprotect). However, Python's `ctypes` library can invoke kernel syscalls directly, bypassing the interception entirely. Attackers used `pkey_mprotect` (which was not hooked) to allocate executable memory, achieving arbitrary code execution with network exfiltration capability. Fixed in MaxKB 2.8.0. This is a textbook example of why LD_PRELOAD/blocklist sandboxes are insufficient -- Python's dynamic nature provides too many paths to kernel access for interception-based approaches to cover.

**vm2 Node.js Sandbox Escape (CVE-2026-22709, CVSS 9.8, January 2026):** A critical sandbox escape in vm2, the most widely used Node.js sandboxing library, allows arbitrary code execution on the host via unsanitized Promise callback handlers. Async functions return a global Promise whose `.then()` and `.catch()` callbacks bypass vm2's internal sanitization, enabling prototype chain traversal to the host `Function` constructor. Fixed in vm2 3.10.2. This is particularly relevant because vm2 is used in AI tool execution environments, workflow platforms (n8n, Langchain tool nodes), and code evaluation sandboxes. The vm2 library has accumulated over 20 known sandbox escape CVEs since 2022, demonstrating that JavaScript-based sandboxing is fundamentally fragile -- the language's dynamic prototype system provides too many escape paths for blocklist approaches to cover.

**vm2 May 2026 CVE Cluster (May 4-5, 2026):** A second wave of vm2 escapes added more than ten further CVE identifiers across two days, including CVE-2026-24118, CVE-2026-24120, CVE-2026-24781, CVE-2026-26332, CVE-2026-26956, CVE-2026-43997, CVE-2026-43999, and CVE-2026-44005 through CVE-2026-44009. All scored between CVSS 9.0 and 10.0. Each entry exploits a different isolation weakness -- prototype pollution variants, exception-handler escape paths, async/promise primitives, and proxy manipulation -- and the cluster reinforces that vm2 should be treated as deprecated for new AI agent deployments. AI agent frameworks that still embed vm2 for "safe" JavaScript tool execution should plan migration to microVM-backed sandboxes (Cloudflare Workers, isolated-vm, Firecracker-based execution environments) and, in the interim, monitor for anomalous child-process creation, unexpected network egress from Node.js runtimes, and suspicious filesystem reads.

**Cohere Terrarium / Pyodide Sandbox Escape (CVE-2026-5752, CVSS 9.3, April 2026):** Cohere's Terrarium project -- a Docker-deployed Pyodide-based Python sandbox used to execute LLM-generated code -- carried a JavaScript prototype-chain traversal flaw in the underlying Pyodide WebAssembly environment. Sandboxed Python code could reference and manipulate the host Node.js global, walk up to the `Function` constructor, and reach `require()`, achieving root-level command execution and file reads such as `/etc/passwd`. Terrarium v1.0.1 (April 22, 2026) ships configuration mitigations, but CERT/CC's coordination note (VU#414811) flagged that the underlying Pyodide repository was archived in 2025 with no maintainer, so the root-cause class is structurally unpatched. Sister disclosures include CVE-2026-24002 ("Cellbreak") against Grist's formula engine (January 2026, CVSS 9.1) and the earlier CVE-2025-68668 ("N8scape") against n8n's Python Code Node. Together they reinforce that Pyodide/WASM in-browser sandboxes are not a hardened isolation boundary for AI tool execution -- any project using Pyodide for untrusted code should wrap it inside a microVM or Firecracker boundary and treat the WASM layer as defense-in-depth, not the primary control.

**Claude Code Symlink Sandbox Escape (CVE-2026-39861, CVSS 10.0, May 2026):** Anthropic's Claude Code agent ran sandboxed tool commands but did not stop a sandboxed process from creating symlinks pointing outside the workspace; when the separate unsandboxed application process later wrote to a path traversing such a symlink, it followed the link and wrote arbitrary files outside the sandbox boundary without prompting (CWE-22 path traversal). Neither the sandboxed command nor the unsandboxed writer could escape alone -- the breakout came from composing the two trust domains, and reliable exploitation required indirect prompt injection to steer the agent into first planting the symlink and then writing through it. Fixed in Claude Code 2.1.64; all earlier versions are affected. This is a clean example of a coupling bug specific to agentic sandboxes: the isolation review must cover the interaction between the sandboxed execution surface and the privileged orchestration process that consumes its outputs (file writes, generated paths, tool results), not just the inner sandbox in isolation. Pair the symlink lesson with requirement 4.1.1's negative tests by attempting symlink/`..` traversal from inside the sandbox and confirming the host-side writer canonicalizes and rejects out-of-workspace targets.

**ShadowMQ: Pickle-over-ZeroMQ Code Reuse Across AI Inference Servers (Oligo Security, November 2025 -- May 2026):** Oligo Security catalogued a class of pre-auth RCE vulnerabilities propagated by line-for-line code reuse across major inference frameworks. Each instance has the same shape: a ZeroMQ ROUTER/DEALER (or similar) socket bound to a publicly reachable interface, calling `recv_pyobj()` (which internally calls `pickle.loads()`) on untrusted bytes without authentication. Confirmed cases include Meta Llama Stack (CVE-2024-50050, patched October 2024 with a JSON serializer), NVIDIA TensorRT-LLM (CVE-2025-23254, patched May 2025), vLLM (CVE-2025-30165 patched May 2025 and CVE-2025-32444 in the Mooncake KV-transfer integration at CVSS 10.0, patched in vLLM 0.8.5), Modular Max Server (CVE-2025-60455, patched June 2025 with msgpack), Microsoft Sarathi-Serve (still vulnerable as of May 2026), and SGLang. SGLang has now drawn at least two distinct CERT/CC coordination notes: VU#665416 covering CVE-2026-3059 and CVE-2026-3060 (disaggregated-serving ZMQ `pickle.loads()` in `encode_receiver.py`, CVSS 9.8, reported February 11 2026 and assigned March 12 2026, affecting versions 0.5.5 through 0.5.9+) together with the lower-severity CVE-2026-3989 (insecure pickle in the crash-dump replay utility, CVSS 7.8); and VU#777338 covering CVE-2026-7301 ROUTER pickle, CVE-2026-7302 path traversal, and CVE-2026-7304 dill-deserialization via custom logit processor, disclosed May 18 2026. All five CVEs remain unpatched as of May 2026, and CERT/CC's recorded mitigation -- binding ZMQ sockets to localhost and migrating from `recv_pyobj()` to msgpack with optional pickle fallback -- exists only as an unmerged community proposal. Two additional disclosures broaden the surface beyond ShadowMQ's original disclosure: CVE-2026-26210 (ktransformers `balance_serve` backend, April 23, 2026, CVSS 9.8) carries the identical ZMQ + `pickle.loads()` pattern and ships Docker manifests with `--network=host`, eliminating container isolation; and CVE-2026-26220 (LightLLM prefill-decode disaggregation, February 2026) accepts pickle-serialized WebSocket frames with a nonce field that defaults to the empty string, making the authentication check vacuous. The pattern keeps spreading into mid-2026: CVE-2026-49121 (AMD ROCm AI Tensor Engine, "AITER," through 0.1.14, CVSS 9.2, disclosed June 1 2026) carries the same `MessageQueue.recv()` -> `pickle.loads()` shape over an unauthenticated ZMQ SUB/XPUB socket in `shm_broadcast.py` -- notable because it extends ShadowMQ-class deserialization beyond the CUDA inference stack into AMD's ROCm tooling, with no released fix as of June 2026 (tracked in upstream issue #3076 and PR #3170). CVE-2026-25873 (BAAI OmniGen2 reward server, CVSS 9.3, disclosed March 18 2026) is the HTTP variant: `reward_proxy.py` and `reward_server.py` call `pickle.loads()` directly on POST bodies while binding to `0.0.0.0` on ports 23456 and 18888+, with no maintainer response and a community fix PR (#139) proposing JSON plus shared-secret headers. On the remediation side, SGLang finally has movement: pull request #27196 ("pickle to msgpack migration") is in progress on `main` and wraps objects that cannot serialize cleanly with a `PickleWrapperIpc` shim, but as of June 2026 it is unmerged, so deployments still need compensating network isolation rather than a released patch. Oligo's internet scans found thousands of these ZMQ sockets exposed publicly, including production inference services. Defensive guidance: replace `recv_pyobj()` with JSON or msgpack on every network channel, restrict scheduler sockets to localhost or mTLS-protected meshes, never run AI inference containers with `--network=host`, and refuse to deploy frameworks whose maintainers have not patched known ShadowMQ-class issues without external compensating controls (network segmentation, mesh-enforced identity, allowlisted ingress).

**Hugging Face LeRobot Pre-Auth RCE (CVE-2026-25874, CVSS 9.3, April 2026):** Independent researchers reported in December 2025 that LeRobot 0.4.3's `PolicyServer` and `RobotClient` deserialized attacker-supplied data via `pickle.loads()` over gRPC channels created with `add_insecure_port()` -- no TLS, no authentication. Vulnerable RPCs include `SendPolicyInstructions`, `SendObservations`, and `GetActions`, and successful exploitation yields unauthenticated remote code execution, theft of API keys/SSH credentials/model files, lateral movement into connected robot fleets, and potential physical safety harm. A permanent fix replacing pickle with SafeTensors and JSON is planned for LeRobot 0.6.0; in the interim, operators should restrict the gRPC port to localhost or mTLS-protected meshes and block the listener from any untrusted network. The incident underscores that even teams who pioneered SafeTensors can ship runtime control planes that re-introduce pickle on the network boundary -- format allowlists must cover gRPC/ZeroMQ/WebSocket handlers, not just on-disk model files.

**LLM Sandbox Escape Research (March 2026):** The UK AI Security Institute published SandboxEscapeBench, evaluating frontier LLMs' ability to escape Docker containers across 18 scenarios. Key findings:
- Frontier models achieved ~40-50% success rates on moderate-complexity scenarios (runtime vulnerabilities, multi-step exploitation chains)
- Performance degraded to near-zero on kernel-level exploits requiring deep system reasoning
- All successful escapes exploited known misconfigurations or publicly disclosed CVEs -- no novel vulnerabilities were discovered
- Individual escape attempts cost approximately $1 at current inference pricing, making brute-force economically feasible
- The researchers concluded that "plain Docker isolation" is insufficient for frontier models and recommended hardware-enforced boundaries (microVMs)

These incidents collectively reinforce that for AI workloads executing untrusted code or running untrusted models, microVM-level isolation (Firecracker, Kata Containers) provides a fundamentally stronger security boundary than container-level isolation alone.

### Confidential Computing Performance (2025-2026 Benchmarks)

Recent benchmarking studies have refined performance expectations for confidential AI inference:

- **NVIDIA H100 Confidential Computing** overhead for typical LLM inference queries remains below 5%, with larger models and longer sequences experiencing nearly zero computational overhead within the GPU. The primary performance penalty comes from encrypted data transfers between CPU and GPU via bounce buffers, not from computation itself.
- **NVIDIA Vera Rubin NVL72** (announced CES 2026, partner availability Q3 2026) encrypts every bus across 72 Rubin GPUs, 36 Vera CPUs, and the entire NVLink fabric, creating the first rack-scale unified trust domain for confidential computing. It integrates with NVIDIA Remote Attestation Services (NRAS) to provide cryptographic proof that CPUs, GPUs, NICs, firmware, drivers, and the running workload all match known-good reference measurements -- achieving zero-trust architecture at rack scale and directly addressing the multi-node TEE orchestration gap relevant to requirement 4.1.4. As of May 2026, AWS, Google Cloud, Microsoft Azure, OCI, CoreWeave, Lambda, Nebius, and Nscale are publicly named as first-wave Vera Rubin host providers, so auditors planning multi-cloud confidential AI deployments now have a concrete provider matrix to compare against.
- **Red Hat and NVIDIA** have demonstrated a proof-of-concept for running AI workloads in Confidential Containers (CoCo) with NVIDIA GPU acceleration, providing Kubernetes-native TEE orchestration with GPU support. As of October 2025, Red Hat OpenShift confidential containers with NVIDIA confidential GPUs supports end-to-end attestation covering both CPU TEEs (Intel TDX or AMD SEV-SNP) and NVIDIA Blackwell/Hopper GPUs, creating a cryptographically provable chain of trust for AI inference workloads.
- **NVIDIA Zero-Trust Confidential AI Factories (March 2026):** NVIDIA published a reference architecture for zero-trust confidential AI factories that eliminates implicit trust in infrastructure. The architecture combines six pillars: hardware root of trust, Kata Containers runtime, minimal guest OS, attestation services (NRAS + Intel Trust Authority), secure workload lifecycle management, and native Kubernetes integration. CoCo remains in Early Access for GPU workloads -- not yet considered production-grade -- but the reference architecture establishes the target design for organizations deploying proprietary models on infrastructure they do not fully trust.
- **Composite attestation limits:** NVIDIA's confidential AI factory reference architecture uses a RATS-style key-release flow where a guest Attestation Agent gathers CPU/GPU evidence, a Key Broker Service delegates validation to an Attestation Service, and secrets are released only after measurements match policy. That workflow protects model keys and plaintext data from host inspection, but it does not address application vulnerabilities, availability attacks, or application-layer network/storage security. Auditors should therefore verify both the attestation chain and ordinary application controls such as authenticated service channels, egress policy, and patch management.
- The NVIDIA confidential computing architecture now covers **Rubin, Blackwell, and Hopper GPU families**, with hardware root of trust anchored on-die for attestation.
- **NVIDIA Blackwell B200 production deployment (March 2026):** Corvex achieved the first verified production deployment of confidential computing on HGX B200 systems. Blackwell is the first TEE-I/O capable GPU, supporting encrypted NVLink/NVSwitch communication between GPUs without bounce buffer overhead for intra-CVM traffic. Joint CPU-GPU attestation via Intel Trust Authority enables cryptographic verification of the full execution stack. Multi-GPU configurations (1-8 GPUs) are supported in TEE mode, a significant expansion over H100-generation systems.
- **CoCo + Kyverno admission policy (May 2026):** A CNCF blog walk-through demonstrates using Kyverno as a Policy-as-Code engine to enforce CoCo configuration at pod admission: required `runtimeClass`, complete `initdata` (attestation server endpoints, image policy, kata-agent policy), and CoCo-specific annotations are validated before scheduling, while remote attestation still gates secret release at runtime. This closes a long-standing operational gap where CoCo's runtime guarantees depended on every application team configuring pods correctly. Auditors should treat the Kyverno (or equivalent OPA/Gatekeeper) policy bundle as part of the assessed CoCo trust boundary, alongside the Attestation Service and KBS.

### Defense-in-Depth for AI Workload Isolation

Production AI workload sandboxing requires layered defenses beyond compute isolation alone:

1. **Isolation boundaries:** Firecracker microVMs or Kata Containers for untrusted workloads; gVisor for lower-risk inference.
2. **Resource limits:** CPU, memory, GPU, and disk quotas to prevent resource exhaustion and noisy-neighbor attacks.
3. **Network controls:** Default-deny egress with proxy-based domain allowlists. Pre-stage all model dependencies before cutting network access.
4. **Permission scoping:** Short-lived credentials, tool-specific access grants, and minimal capability sets.
5. **Model validation:** SafeTensors preference, multi-tool scanning, fail-closed policies.
6. **Continuous monitoring:** Execution tracking, syscall auditing, and anomaly detection within sandboxed environments.

### Verification Checklist Updates (May 2026)

- Treat DNS as real egress. The AgentCore DNS-tunneling research and AWS's updated documentation show that a sandbox can block direct internet traffic while still allowing recursive DNS and selected service traffic. Verification should include DNS queries to attacker-controlled domains, direct TCP/UDP attempts, metadata-service access, and S3/object-store upload attempts.
- Verify the exact isolation layer, not just the product name. For Docker Sandboxes, confirm the workload is actually inside the microVM path and that host Docker socket access is absent. For Kata, record the VMM (`firecracker`, `cloud-hypervisor`, or `qemu`), Kata version, runtime class, and rootfs driver. For OpenShell, review the loaded policy and deny logs for the specific agent binary.
- Validate model loading in a negative test harness. Submit pickle, renamed pickle (`.bin` / `.pt`), malformed ZIP, model config with unsafe Hydra `_target_`, and SafeTensors-only models. The expected result is reject-before-load for unsafe formats and successful load only for approved formats from an approved registry.
- Inventory every networked deserialization sink. After CVE-2026-25874 (LeRobot), the ShadowMQ family (vLLM, NVIDIA TensorRT-LLM, Modular Max Server, SGLang, ktransformers, LightLLM, Microsoft Sarathi-Serve, AMD ROCm AITER via CVE-2026-49121, and BAAI OmniGen2 via CVE-2026-25873), and CVE-2026-26215 (manga-image-translator FastAPI), treat any gRPC, ZeroMQ, WebSocket, FastAPI/HTTP, or message-queue handler that calls `pickle.loads`, `cloudpickle.loads`, `joblib.load`, `dill.loads`, or `recv_pyobj` on remote data as a pre-auth RCE risk, even on "internal" networks. Note that this is no longer a CUDA-only concern -- AITER moved the identical `MessageQueue.recv()`/`pickle.loads()` shape into AMD's ROCm tooling -- so the grep should run across every accelerator backend, not just NVIDIA inference stacks. Where a fix is mid-flight rather than released (for example SGLang's unmerged pickle-to-msgpack migration PR #27196), treat the framework as unpatched and keep network-level compensating controls in place. Require schema-validated JSON, protobuf, or SafeTensors at every such boundary. Confirm vLLM is at 0.8.5+, NeMo at 2.3.2+, NVIDIA TensorRT-LLM at the May 2025 advisory release or later, Modular Max Server at the June 2025 fix or later, and that any deployment of SGLang, ktransformers, LightLLM, manga-image-translator, or Microsoft Sarathi-Serve has network-level compensating controls until upstream patches land. Also grep deployment configuration for `nonce=""` or `token=""` defaults and missing `MT_WEB_NONCE`-style environment variables -- the empty-string-nonce pattern silently disabled auth in two independent 2026 CVEs.
- Cover native media-decoder and auth-bypass paths, not just deserialization (June 2026). Multimodal inference is a fresh untrusted-input surface: CVE-2026-22778 (vLLM, fixed 0.14.1) reached pre-auth RCE through FFmpeg's JPEG2000 decoder via a video URL, and CVE-2026-24207 (NVIDIA Triton / DALI backend, fixed r26.03) was an auth bypass that can lead to code execution. For any media-accepting endpoint, confirm versions are patched, `--allowed-media-domains` (or equivalent) constrains outbound fetches, and image/video decoding runs inside the same isolation boundary as model loading; treat the inference server's authentication layer as part of the assessed sandbox perimeter.
- Inspect JavaScript and Python sandbox library versions. For any AI agent platform that runs untrusted code, fail the assessment if vm2 or Pyodide is the sole isolation boundary. Require an outer microVM and capture the inner sandbox library version, advisory inventory (vm2 May 2026 cluster, CVE-2026-5752 Terrarium/Pyodide), and migration plan.
- Tie attestation evidence to release policy. For TEE workloads, confirm the relying party checks freshness, hardware evidence, firmware/driver versions, image measurements, GPU attestation where applicable, and reference-value policy before releasing model decryption keys. For Kubernetes-native CoCo deployments, also verify that admission-time policies (Kyverno, OPA/Gatekeeper) enforce `runtimeClass`, `initdata`, and CoCo annotations before scheduling.
- Separate confidentiality from containment. Confidential computing protects data in use from the infrastructure operator; it does not make untrusted model code safe. High-value confidential workloads still need sandboxing, egress control, vulnerability management, and model artifact allowlisting.

### Implementation Maturity (May 2026)

- **Mature:** CPU-only untrusted-code isolation using Firecracker, Kata Containers, gVisor, strict seccomp/AppArmor profiles, and default-deny egress can be verified with repeatable negative tests. The main failure mode is operational drift: exposed Docker sockets, privileged pods, hostPath mounts, unpatched GPU runtimes, or network exceptions added after deployment. As of May 2026, microVM-per-agent platforms (E2B, Fly.io Sprites, Northflank, Docker Sandboxes, Vercel Sandbox, AWS Bedrock AgentCore, Cloudflare Workers Sandboxes, Modal, Ramp) have converged on Firecracker or Cloud Hypervisor as the de facto isolation layer for untrusted AI code.
- **Advancing:** GPU-aware confidential workloads are moving from proof-of-concept to early production. Blackwell/Hopper confidential GPU support, Intel Trust Authority composite attestation, and Confidential Containers with Trustee provide concrete evidence chains, but policy lifecycle, firmware reference values, and multi-node orchestration still need careful engineering. Vera Rubin NVL72 (partner availability Q3 2026) targets the multi-node and rack-scale gaps directly, and admission-time CoCo automation via Kyverno (May 2026) closes the configuration-drift gap for Kubernetes-native deployments.
- **Emerging:** General-purpose SMPC and FHE for large-model training remain specialized. They are practical for narrow analytics, model evaluation, or small collaborative learning workflows, while most high-throughput inference deployments still rely on TEEs plus conventional network, identity, and monitoring controls. AI-agent-specific runtime policy engines (NVIDIA OpenShell, Microsoft LiteBox library OS) are still experimental and should be layered on top of microVM isolation rather than replacing it.

### Notable Incidents and Signals

- **NVIDIAScape (July 2025):** CVE-2025-23266 showed that GPU container tooling can become the host escape path even when the application container itself is otherwise ordinary. Auditors should record NVIDIA Container Toolkit, GPU Operator, Device Plugin, and MIG Manager versions as first-class sandbox evidence.
- **AgentCore DNS tunneling (March-April 2026):** The BeyondTrust and Unit42 disclosures showed that "no internet access" claims need protocol-level testing. Recursive DNS, metadata services, and service-specific exceptions should be tested separately from generic TCP egress.
- **SafeTensors operationalization (April 2026):** SafeTensors joining the PyTorch Foundation strengthens the ecosystem path away from pickle, but conversion, scanning, and exception workflows still need their own sandbox boundary because they often ingest legacy unsafe artifacts.
- **LeRobot pickle-over-gRPC (April 2026):** CVE-2026-25874 demonstrated that pickle-over-network channels remain a live attack surface even in projects affiliated with safer-format ecosystems. Sandbox reviews should now explicitly enumerate every networked deserialization sink (gRPC, ZeroMQ, WebSockets, message queues) and verify the format is SafeTensors/JSON/protobuf with schema validation.
- **vLLM media-decoder pre-auth RCE (CVE-2026-22778, February 2026):** A CVSS 9.8 unauthenticated RCE in vLLM 0.8.3 through 0.14.0 (fixed in 0.14.1) chained a heap-address information leak with a heap buffer overflow in FFmpeg's JPEG2000 decoder, reachable by submitting a malicious video URL to `POST /v1/chat/completions` or `POST /v1/invocations`. It illustrates a class of sandbox-relevant bugs distinct from pickle: untrusted multimodal input handed to native media decoders (FFmpeg, OpenCV). Sandbox reviews of multimodal inference services should enumerate media-fetch and decode paths, set `--allowed-media-domains` to constrain SSRF, and run the decoder inside the same isolation boundary as model loading.
- **NVIDIA Triton authentication bypass (CVE-2026-24207, May 2026):** A CVSS 9.8 authentication bypass in all Linux versions of Triton Inference Server and the DALI backend prior to r26.03 that NVIDIA states can lead to code execution, privilege escalation, data tampering, DoS, or information disclosure. The inference server's own auth layer is part of the sandbox perimeter -- "managed" or "internal" model-serving endpoints still need network segmentation and patch tracking.
- **ESA AnomalyMatch torch.load RCE (CVE-2026-38950, June 2026):** A CVSS 7.8 arbitrary-code-execution bug (fixed in 1.3.1) where the application calls `torch.load()` on checkpoint files from session directories without restricting deserialization. Even though PyTorch 2.6 defaults to `weights_only=True`, applications that do not rely on the default re-open the pickle RCE path, so loader-level allowlisting and sandboxing must not be skipped on the assumption that the runtime default protects every call site.
- **PyTorch `weights_only` unpickler RCE (CVE-2026-24747, June 2026):** A CVSS 8.8 flaw (GHSA-63cw-57p8-fm3p, published January 27 2026, fixed in PyTorch 2.10.0) in the restricted unpickler that `torch.load(..., weights_only=True)` relies on. The unpickler did not validate pickle opcodes and storage metadata, so a crafted `.pth` checkpoint triggers heap memory corruption (via `SETITEM`/`SETITEMS` opcodes on non-dictionary types and a declared-vs-actual storage-size mismatch) and can escalate to arbitrary code execution on all versions through 2.9.1 -- even when the caller did everything right and set `weights_only=True`. This matters for every page section that treats the PyTorch 2.6 safe default as a backstop: the safe mode is only as safe as its own parser, so pin to 2.10.0+, prefer SafeTensors, and keep untrusted checkpoint loading inside a sandbox.
- **Antigravity Secure Mode bypass via native tool injection (Pillar Security, April 2026):** Google's Antigravity agentic IDE dispatched its built-in `find_by_name` tool before Secure Mode's command-level restrictions were evaluated, and the tool's `Pattern` parameter flowed unsanitized into the `fd` binary. An indirect prompt injection that smuggled `fd`'s `-X`/`--exec-batch` flag through that parameter achieved arbitrary command execution while bypassing the product's most restrictive sandbox profile (reported January 7 2026, fixed February 28 2026, disclosed April 20 2026). The lesson mirrors the Claude Code symlink case: agentic sandbox reviews must cover native/built-in tool dispatch and argument handling, not just the explicit shell tool, and must confirm restricted modes gate the earliest execution path.
- **ShadowMQ code-reuse pattern (Oligo Security, November 2025 -- June 2026):** Pickle-over-ZeroMQ/HTTP deserialization was copied across Meta Llama Stack, NVIDIA TensorRT-LLM, vLLM, SGLang, Modular Max Server, Microsoft Sarathi-Serve, ktransformers, and LightLLM, and by mid-2026 the same shape reached AMD's ROCm stack (CVE-2026-49121, AITER through 0.1.14, CVSS 9.2, June 1 2026, `MessageQueue.recv()` over an unauthenticated ZMQ SUB socket, unpatched) and a multimodal generation project (CVE-2026-25873, BAAI OmniGen2 reward server, CVSS 9.3, March 18 2026, `pickle.loads()` on HTTP POST bodies bound to `0.0.0.0`, no maintainer response). CERT/CC VU#777338 (May 18, 2026) covers three unpatched SGLang issues (CVE-2026-7301, CVE-2026-7302, CVE-2026-7304) with no vendor response, though SGLang now has an in-progress (still unmerged as of June 2026) pickle-to-msgpack migration in PR #27196. Auditors should grep for `recv_pyobj`, `pickle.loads`, and `dill.loads` in scheduler, KV-transfer, and reward/eval code paths across both CUDA and ROCm backends, treat `--network=host` Docker manifests as findings, and require msgpack/JSON/protobuf for every cross-process control channel.
- **Claude Code symlink sandbox escape (CVE-2026-39861, May 2026):** A CVSS 10.0 path-traversal escape (fixed in Claude Code 2.1.64) where a sandboxed process plants a symlink pointing outside the workspace and the separate unsandboxed application process follows it on a later write, yielding arbitrary out-of-workspace file writes and, via indirect prompt injection, potential host code execution. The lesson for agentic sandbox reviews: the privileged orchestration process that consumes sandbox outputs (paths, file writes, tool results) is inside the trust boundary and must canonicalize and re-validate every path, because neither half of the system is exploitable alone.
- **Empty-string nonce anti-pattern (February 2026):** CVE-2026-26215 (manga-image-translator FastAPI shared-API mode, CVSS 9.3) and CVE-2026-26220 (LightLLM prefill-decode WebSocket) both default their authentication nonce to `""`, which Python evaluates as falsy, so the `if self.nonce:` guard silently disables auth on every default install. The manga-image-translator case is particularly notable because Tencent YunDing Security Lab filed the same RCE in issue #946 in May 2025 and the report sat unanswered for six months before the stale bot auto-closed it -- CVE assignment only happened after a second independent disclosure (VulnCheck, February 11, 2026). Reviews of any AI service that ships with a "nonce" or "token" parameter should grep the source for empty-string defaults, missing environment variables in Docker/Compose files, and `if self.nonce:` checks; the safe pattern is `nonce=None` with mandatory generation at startup and an explicit opt-out flag for unauthenticated mode.
- **vm2 May 2026 CVE wave (May 4-5, 2026):** Ten-plus new vm2 escape CVEs in two days pushed the cumulative count above thirty and effectively confirmed that JavaScript-based sandboxing libraries cannot be repaired piecemeal. AI agent frameworks should treat vm2 as deprecated and migrate to microVM-backed JavaScript execution paths.
- **Cohere Terrarium / Pyodide (April 2026):** CVE-2026-5752 plus the archival of the Pyodide repository show that browser-style WASM Python sandboxes will not receive structural fixes. Pyodide should be paired with an outer microVM and treated as defense-in-depth.
- **GGUF chat-template SSTI class (May 2024 -> April 2026):** CVE-2024-34359 in llama-cpp-python and CVE-2026-5760 in SGLang demonstrate the same attack twice in two years: malicious `tokenizer.chat_template` metadata in a GGUF file achieves pre-auth RCE when an inference server renders it with un-sandboxed `jinja2.Environment()`. Format-safety claims about GGUF only hold for weights; metadata fields rendered by a templating engine are part of the attack surface. Audit any GGUF-aware service for the literal token `jinja2.Environment(` outside `jinja2.sandbox`, prefer `ImmutableSandboxedEnvironment`, and scan candidate models with HuggingFace's `gguf-jinja-analysis` tool before allowlisting.
- **MITRE ATLAS v2 (May 2026):** The Center for Threat-Informed Defense expanded ATLAS to 16 tactics, 84 techniques, and 56 sub-techniques (up from 15/66 in October 2025), added a Technique Maturity filter, and is integrating emulations into Caldera. AI Supply Chain Compromise (AML.T0010) sub-techniques covering ML software (T0010.001), models (T0010.003), and GPU/data supply chain remain the most directly tied to this section, and new sub-techniques should be cross-walked into the 4.1.1 threat narrative as ATLAS publishes them.
- **NIST AI RMF developments (Q1-Q2 2026):** NIST published an April 7, 2026 concept note for an AI RMF Critical Infrastructure Profile and announced the AI Agent Standards Initiative in February 2026, with an AI Agent Interoperability Profile slated for Q4 2026. Both should be tracked because they will likely formalize attestation, sandbox, and identity controls that sit on top of this section's requirements.
- **EU AI Act milestones (May-August 2026):** The Digital Omnibus on AI reached provisional political agreement May 6, 2026 (confirmed by the Council May 13), deferring stand-alone Annex III high-risk obligations to December 2, 2027 (AI in Annex I regulated products: August 2, 2028) and national regulatory sandboxes to August 2, 2027, while Article 50 transparency rules still apply from August 2, 2026. Confidential computing and TEE deployments increasingly serve as evidence for the cybersecurity and robustness obligations EU operators must demonstrate for high-risk AI systems.

## Open Research Questions

- [ ] What is the practical performance overhead of confidential GPU inference (NVIDIA H100 CC mode) for large language models across different batch sizes? *Partial answer: benchmarks show sub-5% overhead for typical LLM queries, with overhead concentrated in CPU-GPU data transfer rather than computation.*
- [ ] How can model partitioning strategies enable confidential inference for models that exceed single-TEE memory limits? *NVIDIA Vera Rubin NVL72 rack-scale confidential computing may address this for future deployments.*
- [ ] When does SMPC become practical for production ML training -- what model sizes and dataset sizes are feasible today?
- [ ] How should attestation policies handle TEE firmware updates that change expected measurement values without disrupting production inference? *Attestation policy management at scale (hundreds of TEE instances with frequent firmware updates) remains operationally challenging -- CoCo's "lazy attestation" model helps but does not solve the firmware versioning problem.*
- [ ] Can homomorphic encryption (HE) complement or replace SMPC for specific inference workloads, and what are the size/latency tradeoffs? *A 2025 SoK paper found that only three FHE methods achieve computational generality for AI (bit-wise TFHE, Scheme Switching, Encoding Switching), all with orders-of-magnitude overhead -- multiplying two encrypted 16-bit integers in TFHE takes ~30 seconds. GPU-accelerated approaches (Falcon on A100, FHEFusion) have reduced overhead by 21-30% but FHE inference remains impractical for production LLMs. Hybrid FHE+TEE approaches show more promise for the near term.*
- [ ] How reliable are model scanning tools (PickleScan, modelscan) against adversarial evasion? *As of 2026, at least 37 distinct scanner evasion techniques are documented: 7 PickleScan CVEs (JFrog + Sonatype), 1 nullifAI 7z bypass, and 29 modelscan bypasses across five categories. Blocklist-based scanning is fundamentally limited because Python provides hundreds of indirect code execution paths that cannot be exhaustively enumerated. Allowlist-based approaches (PickleBall, SafePickle) and format-level safety (SafeTensors) remain the stronger alternatives.*
- [ ] Can WebAssembly-based sandboxing achieve sufficient isolation guarantees for high-risk AI workloads, or does it remain limited to defense-in-depth supplementation? *CVE-2026-34971 (Wasmtime AArch64 sandbox escape) demonstrates that even well-engineered Wasm runtimes have architecture-specific JIT compiler vulnerabilities.*
- [ ] Will Microsoft LiteBox's library OS approach (minimal per-workload kernel interfaces in Rust) prove viable for production AI workloads, and how does its isolation boundary compare to microVMs and user-space kernels under adversarial conditions?
- [ ] What is the security boundary of WASI AI when used for edge inference -- can malicious model files exploit JIT-compiler bugs to escape the Wasm sandbox?
- [ ] As frontier LLMs demonstrate increasing container escape capabilities (SandboxEscapeBench, March 2026), at what capability threshold should organizations mandate microVM isolation for all agentic AI workloads?
- [ ] How should organizations handle the Azure Confidential Containers AKS preview sunset (March 2026) -- what are the production-ready alternatives for Kubernetes-native confidential AI?
- [ ] Given Unit42's discovery that ~100 HuggingFace libraries use Hydra's `instantiate()` for model configuration, how broadly does the configuration-metadata RCE attack surface extend beyond the three libraries with assigned CVEs? Are other configuration frameworks (OmegaConf, Pydantic-AI) similarly exploitable?
- [ ] Will attestation-bound TLS (draft-fossati-tls-attestation) be revived or superseded by a different IETF standard, and how will production CoCo deployments handle the gap between platform-specific attestation integrations and a universal attestation-channel protocol?
- [ ] As Docker Sandboxes and similar microVM platforms become the default isolation layer for AI agents, what is the residual attack surface via VMM escape? CVE-2026-5747 (Firecracker virtio-pci OOB write) and CVE-2026-1386 (jailer symlink race) demonstrate that even Rust-based VMMs with 83K lines of code are not immune to exploitable bugs.
- [ ] How does NVIDIA OpenShell's YAML-based default-deny policy model compare in practice to traditional capability-based sandboxing (seccomp-bpf, AppArmor) for AI agent workloads -- does the per-action enforcement granularity introduce measurable latency at high throughput?
- [ ] Given Unit42's "Cracks in the Bedrock" findings (April 2026), how should organizations verify that cloud-managed AI sandbox services actually enforce the network isolation they claim? DNS tunneling bypassed AgentCore's "complete isolation" -- are similar covert channels present in other managed AI sandbox offerings (Azure AI, Google Vertex AI, etc.)?
- [ ] CVE-2026-34971 demonstrated that Wasm JIT compiler bugs on specific architectures (AArch64) can enable full sandbox escape. As WASI AI pushes model inference to edge devices running ARM processors, how should organizations assess and mitigate architecture-specific JIT compiler risks in their Wasm-based AI deployments?
- [ ] With CVE-2026-25874 (LeRobot) demonstrating pre-auth RCE via pickle over gRPC, how broad is the pickle-over-network attack surface across other AI/robotics frameworks (ROS bridges, federated learning clients, observability collectors)? What is the right minimum control: mTLS plus pickle, JSON/protobuf at the network boundary with pickle internal only, or full elimination of pickle from deployed services?
- [ ] After the May 2026 vm2 cluster (10+ CVEs in two days) and the unmaintained status of Pyodide, what is the realistic migration timeline for AI agent platforms that still rely on JavaScript or Pyodide-based "in-process" sandboxes? Are isolated-v8 forks, Cloudflare Workers-style isolates, or microVM-per-call architectures the most viable replacements at production cost?
- [ ] Does admission-time CoCo policy enforcement via Kyverno (May 2026) materially reduce the configuration drift that historically undermined Confidential Containers, or does it create a new soft target (the admission controller and its policy bundle)?
- [ ] As Vera Rubin NVL72 enters Q3 2026 partner availability across AWS, GCP, Azure, OCI, CoreWeave, Lambda, Nebius, and Nscale, how comparable will composite CPU+GPU+NIC+NVLink attestation be across providers? Will a portable RATS profile emerge or will each provider expose a divergent attestation surface that complicates multi-cloud confidential AI?
- [ ] How will the EU AI Act obligations — Article 50 transparency from August 2026, and the high-risk duties deferred to December 2, 2027 by the May 2026 Digital Omnibus (Annex I regulated products: August 2, 2028) — interact with TEE-based confidential AI evidence: will documented attestation chains and sandbox controls count toward Article 15 cybersecurity requirements, or will operators need separate compensating documentation?
- [ ] The ShadowMQ disclosures show that a single insecure deserialization pattern, copied between AI inference projects through informal code reuse, can produce a new pre-auth RCE every few months. What governance, secure-by-default templates, or AI-BOM signals would let downstream operators detect "inherited" insecure deserialization code before it ships? With SGLang still unpatched in May 2026 despite CERT/CC coordination, what is the realistic upper bound on the lifetime of these patterns in actively maintained projects?
