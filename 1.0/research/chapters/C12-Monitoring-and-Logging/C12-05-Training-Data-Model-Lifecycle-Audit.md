# C12.5: Training Data & Model Lifecycle Audit

> **Parent:** [C12 Monitoring, Logging & Anomaly Detection](C12-Monitoring-and-Logging.md)
> **Requirements:** 4 | **IDs:** 12.5.1--12.5.4

## Purpose

This section covers the provenance and change history of the three asset classes that ultimately determine model behavior: training data, model artifacts, and the knowledge sources (documents, embeddings, RAG corpora) that feed retrieval at runtime. The control objective is reconstructability -- given a model in production, an auditor should be able to answer "which datasets, transformations, labels, and code produced this exact artifact, who changed it and when, and how do I roll back to a known-good state?"

That objective matters because the most damaging 2024--2026 AI incidents share a root cause: the chain from raw data to deployed model was not auditable, so poisoning, tampering, and silent regressions went undetected until after harm. Web-scale poisoning research showed that controlling **0.01%** of a crawl is enough to corrupt a foundation model for roughly $60 (Carlini, Tramèr et al.); malicious models on public hubs shipped reverse shells through pickle deserialization; and knowledge-base poisoning of RAG systems demonstrated that a handful of untagged ingested documents can dominate retrieval. None of these are detectable without lineage, labeling logs, immutable model-change records, write-time document tagging, and uniquely identified, versioned datasets -- the evidence covered by the four requirements below.

> **Scope note (from AISVS source):** C12.5 is the *audit and provenance* control. It establishes that the records exist, are complete, and are tamper-evident. The integrity of the data itself (poisoning detection, sanitization) lives in **C1 (Training Data)**; model authorization and validation live in **C3 (Model Lifecycle Management)**; supply-chain artifact signing lives in **C6 (Supply Chain)**; and the memory/RAG access controls that consume the tags from 12.5.4 live in **C8 (Memory & Embeddings)**. Verification of 12.5.x should focus on whether the audit trail can reconstruct the lifecycle, not on whether each individual control upstream is correctly configured.

## Implementation Guidance

### Minimum Audit Evidence and Negative Tests

An assessor should be able to correlate the following records without relying on names or mutable registry aliases alone:

- A lineage event that identifies input and output dataset versions, transformation or augmentation code revision, parameters, execution identity, and event time.
- A labeling event that identifies the item, prior and new label, annotator or labeling service, guideline version, and event time.
- A model-change event that identifies the before/after state, actor, artifact digest, deployment or registry target, change parameters, and event time.
- An ingestion event whose source, writer identity, and timestamp match the metadata stored with the document and every derived chunk or embedding.
- Independent retention evidence showing who can alter or delete the records, how alteration is detected, and whether the retention period covers the system's investigation and regulatory needs.

Negative tests should exercise omission and tampering, not merely confirm that a dashboard displays successful runs:

1. Bypass one declared dataset transform or change its parameters; the resulting artifact should either be rejected or appear as a new, attributable lineage edge.
2. Relabel a sampled item; both values, the responsible identity, and the governing guideline version should remain reconstructable.
3. Change a model alias, deployment configuration, system prompt, or guardrail threshold; an independent audit record should capture the exact change and artifact/configuration identity.
4. Ingest a document without one required tag or with a caller-supplied false writer identity; the write should fail or the trusted ingestion service should overwrite the untrusted fields.
5. Attempt to alter or delete a retained record using a privileged application account; immutable storage or an independently monitored integrity mechanism should prevent or detect the action.

### Threat Landscape and Notable Incidents (as of July 2026)

The case for auditable data and model lifecycle is best made by the incidents where its absence was the decisive failure.

- **Poisoning Web-Scale Training Datasets Is Practical (Carlini, Tramèr et al., arXiv 2302.10149; v2 May 6, 2024).** The paper introduced **split-view poisoning** -- mutable web content means the snapshot an annotator vetted can differ from what later downloaders fetch -- and **frontrunning poisoning**, injecting malicious content into Wikipedia/crowd-sourced snapshots during the crawl window. The authors showed that poisoning just **0.01%** of LAION-400M or COYO-700M was achievable for roughly **$60**. The defensive implication is squarely 12.5.1: without recorded lineage and content-addressed dataset versions, you cannot prove the bytes you trained on are the bytes that were reviewed. See [arXiv 2302.10149](https://arxiv.org/abs/2302.10149).
- **Nightshade public release (January 18, 2024).** The University of Chicago tool perturbs image pixels so models learn corrupted concept associations (e.g., "dog" pixels that teach "cat"); it saw more than 250,000 downloads in its first week. Concept poisoning is invisible unless labeling/transformation steps are logged (12.5.1, 12.5.2). See [MIT Technology Review coverage](https://www.technologyreview.com/2023/10/23/1082189/data-poisoning-artists-fight-generative-ai/).
- **Malicious Hugging Face models with silent backdoors (JFrog, February 27, 2024).** Roughly 100 models carried real payloads abusing Python pickle `__reduce__` to open reverse shells on load (for example `baller423/goober2` connecting to `210.117.212.93:4242`). This is the canonical 12.5.3/12.5.4 case: an untrusted model artifact ingested and deployed without provenance verification or an immutable record of where it came from. Tracked in MITRE ATLAS as a malicious-model case study. See the [JFrog write-up](https://jfrog.com/blog/data-scientists-targeted-by-malicious-hugging-face-ml-models-with-silent-backdoor/).
- **Large-scale malicious-model scanning (2025--2026).** Model-scanning vendors report hundreds of thousands of unsafe or suspicious findings across tens of thousands of public models -- pickle code execution, namespace hijacking, and "nullifAI"-style evasion of scanners. The volume reinforces that artifact ingestion without verified writer identity (12.5.4) and immutable deployment records (12.5.3) is the dominant supply-chain exposure. See [NSFOCUS analysis](https://nsfocusglobal.com/ai-supply-chain-security-hugging-face-malicious-ml-models/).
- **PoisonedRAG (USENIX Security 2025) and the RAG traceback problem.** PoisonedRAG was the first knowledge-corruption attack on retrieval-augmented generation: injecting a few crafted documents into the knowledge base lets an attacker control the model's answers, including through editable sources like Wikipedia. Follow-on defense work (RAGForensics, *Traceback of Poisoning Attacks to RAG*, ACM Web Conference 2025) confirms that **write-time provenance tagging is the enabling control for traceback** -- exactly requirement 12.5.4. The 2026 research wave has pushed past anomaly detection toward provenance-and-claim verification: **RAGShield** (arXiv 2604.00387, April 2026) targets *numerical-claim manipulation* in government RAG systems -- where editing a tax figure by $50,000 yields cosine similarity 0.9998 and is invisible to every embedding-based detector -- and catches all 430 test attacks by cross-checking extracted claims against a corpus-derived registry and flagging value changes outside known update schedules, while embedding defenses missed 79-90%. The takeaway for 12.5.4 is that tags alone are not enough; the source/writer/timestamp metadata only becomes actionable when retrieval-time logic cross-verifies claims against trusted provenance. See the [PoisonedRAG paper](https://www.usenix.org/system/files/usenixsecurity25-zou-poisonedrag.pdf), [RAGForensics](https://dl.acm.org/doi/10.1145/3696410.3714756), and [RAGShield](https://arxiv.org/abs/2604.00387).
- **MLflow auth-bypass-to-RCE chain (CVE-2026-2635 + CVE-2026-2033; disclosed February 2026).** A hard-coded default credential in `basic_auth.ini` (CVE-2026-2635, CVSS 9.8) and an artifact-handler directory traversal (CVE-2026-2033) independently allowed unauthenticated compromise on versions before **3.8.0rc0**. MLflow's stable 3.8.0 changelog is dated **December 19, 2025**. An attacker at that level can tamper with model versions and rewrite registry history -- the precise threat 12.5.3 immutable audit records exist to make detectable. This was not an isolated flaw: through the first half of 2026 further MLflow advisories covered unauthenticated arbitrary-file read in `_create_model_version()` (CVE-2026-2614), unauthenticated job access leading to RCE (CVE-2026-0545), and unauthorized access to multipart-upload endpoints under `--serve-artifacts` (CVE-2026-2651). The broader pattern is documented in JFrog's "MLOps to MLOops" attack-surface research. See the [MLflow 3.8.0 changelog](https://github.com/mlflow/mlflow/blob/master/CHANGELOG.md#380-2025-12-19), [CVE-2026-2635 GitHub advisory](https://github.com/advisories/GHSA-gq3w-7jj3-x7gr), and [CVE-2026-2033 NVD record](https://nvd.nist.gov/vuln/detail/CVE-2026-2033).

The connective tissue across these cases: every one was a provenance or integrity failure that the data/model audit trail is designed to surface. Poisoning that controls 0.01% of a corpus is undetectable without component-level lineage; a backdoored model is just another version without immutable deployment records; a poisoned RAG document is anonymous without write-time tagging.

### Defensive Tooling Landscape

The tooling splits cleanly along the four requirements. Maturity varies widely -- versioning and lineage are well served; write-time document tagging and immutable model-change records are not.

**Dataset versioning and reproducible lineage (12.5.1).** Content-addressed, immutable dataset snapshots are what let a lineage record be *proven* rather than asserted -- they bind each transformation/merge in the graph to an exact, reproducible byte set.
- **DVC** -- Git-extension that keeps large data/models in object storage while committing lightweight pointer files; `dvc checkout` restores prior versions and `dvc dag` renders pipeline lineage. See [DVC docs](https://doc.dvc.org/use-cases/versioning-data-and-models).
- **lakeFS** -- Git-style branch/commit/merge/tag layer over object storage; each commit is an immutable snapshot enabling instant rollback and reproducible experiments. See [lakeFS docs](https://docs.lakefs.io/).
- **Pachyderm** -- Kubernetes-native pipelines with automatic data versioning and *global provenance*: every output commit traces to the exact input commits and pipeline version. See [Pachyderm docs](https://docs.pachyderm.com/).
- **Lakehouse table formats** -- **Delta Lake** (`RESTORE` + time travel by version/timestamp), **Apache Iceberg** (manifest-based snapshots), and **Apache Hudi** (timeline versioning) give point-in-time reproducibility and rollback. See [Delta Lake time travel](https://docs.delta.io/latest/delta-batch.html#-deltatimetravel) and [Iceberg docs](https://iceberg.apache.org/docs/latest/).
- **Hugging Face dataset revisions** -- every dataset version is a Git commit/branch/tag, loadable via `load_dataset(..., revision=...)`. See [HF dataset docs](https://huggingface.co/docs/hub/datasets-adding).

**Lineage and metadata (12.5.1).**
- **OpenLineage + Marquez** -- an open standard (LF AI & Data) emitting run/job/dataset events as a pipeline runs, aggregated by Marquez into a lineage graph. See [OpenLineage](https://openlineage.io/getting-started/) and [Marquez](https://github.com/MarquezProject/marquez).
- **ML Metadata (MLMD)** -- TFX library recording artifacts, executions, and context into a queryable lineage graph. See [MLMD guide](https://www.tensorflow.org/tfx/guide/mlmd).
- **DataHub** and **OpenMetadata** -- open-source catalogs with table- and column-level lineage. See [DataHub](https://datahubproject.io/docs/) and [OpenMetadata](https://docs.open-metadata.org/).
- **Weights & Biases Artifacts** -- versioned, hash-deduplicated artifacts with an automatic lineage DAG connecting runs to the artifacts they consumed/produced. See [W&B Artifacts](https://docs.wandb.ai/guides/artifacts/).

**Model lifecycle / registry audit (12.5.3).**
- **MLflow Model Registry** -- versioned model store with aliases, tags, stage transitions, and each version linked to its producing run. See [MLflow Model Registry](https://mlflow.org/docs/latest/ml/model-registry/).
- **Amazon SageMaker Model Registry + Model Cards** -- model packages with approval status, plus captured model lineage (datasets, training jobs, endpoints). See [SageMaker Model Registry](https://docs.aws.amazon.com/sagemaker/latest/dg/model-registry.html).
- **Vertex AI Model Registry** -- central GCP repository integrated with Vertex ML Metadata for lineage. See [Vertex Model Registry](https://cloud.google.com/vertex-ai/docs/model-registry/introduction).

**Immutable / tamper-evident audit logging (12.5.3).**
- **AWS CloudTrail log-file validation + S3 Object Lock (WORM)** -- hash-chained digest files plus write-once storage make deletion or alteration detectable. See [CloudTrail log-file validation](https://docs.aws.amazon.com/awscloudtrail/latest/userguide/cloudtrail-log-file-validation-intro.html) and [S3 Object Lock](https://docs.aws.amazon.com/AmazonS3/latest/userguide/object-lock.html).
- **Sigstore `model-transparency` (model-signing v1.0)** -- the OpenSSF/Google/NVIDIA/HiddenLayer library hashes a model's files and metadata into an in-toto Statement, signs it (keyless via Fulcio or with keys), and records it in the **Rekor** immutable transparency log; supports incremental re-signing for multi-GB models. The companion **Sigstore Model Validation Operator** (v1.0.1) enforces signature verification at the Kubernetes admission boundary, so an unsigned or tampered model artifact is rejected before it can be served -- turning the transparency record into an enforcement gate, not just an after-the-fact log. See [sigstore/model-transparency](https://github.com/sigstore/model-transparency) and the [Model Validation Operator announcement](https://blog.sigstore.dev/model-validation-operator-v1.0.1/).
- **in-toto attestations + SLSA provenance** -- capture builder identity, build parameters, environment, and dependency digests for ML artifacts. See [SLSA provenance](https://slsa.dev/spec/v1.0/provenance) and [in-toto attestation](https://github.com/in-toto/attestation).

**Document provenance and labeling logs (12.5.4, 12.5.2).**
- **C2PA / Content Credentials (spec 2.x)** -- cryptographically signed manifests binding source, creator/tool, timestamp, and edit history to an asset; any modification breaks the signature. See the [C2PA explainer](https://c2pa.org/specifications/specifications/2.4/explainer/Explainer.html).
- **Label Studio** -- provides per-annotation history for imports, submissions, updates, and reviews through an [Enterprise API and history panel](https://api.labelstud.io/api-reference/api-reference/annotation-history/list); Community Edition does not expose that endpoint.
- **Amazon SageMaker Ground Truth** -- logs worker events to the `/aws/sagemaker/groundtruth/WorkerActivity` CloudWatch log group and records label metadata in output manifests. AWS has announced that new customer access closes on July 30, 2026, although existing customers can continue using the service. See [worker activity tracking](https://docs.aws.amazon.com/sagemaker/latest/dg/workteam-private-tracking.html) and [Ground Truth input/output data](https://docs.aws.amazon.com/sagemaker/latest/dg/sms-data.html).

Key anchors for auditors:
- **EU AI Act Art. 10** requires data governance covering "data collection processes and the **origin of data**" and "data-preparation operations including **annotation, labelling, cleaning, updating, enrichment and aggregation**" -- a near-verbatim map to 12.5.1, 12.5.2, and 12.5.4. **Annex IV §2(d)** requires datasheets describing data provenance, how data was obtained/selected, and labelling procedures. **Art. 12** requires automatic event logging over the system lifetime, and **Art. 19** requires those logs be retained for **at least six months** -- the forensic floor for the immutable model-change records of 12.5.3. The **Digital Omnibus** political agreement of **7 May 2026** moves the standalone Annex III high-risk obligations to **2 December 2027** and Annex I embedded high-risk systems to **2 August 2028**. See the [official AI Act text on EUR-Lex](https://eur-lex.europa.eu/eli/reg/2024/1689/oj?locale=en) and the [European Commission announcement](https://digital-strategy.ec.europa.eu/en/news/eu-agrees-simplify-ai-rules-boost-innovation-and-ban-nudification-apps-protect-citizens).
- **NIST AI 600-1 (Generative AI Profile, July 2024)** names **Data Provenance** and **Information Integrity** as risk categories and recommends logging, metadata recording, watermarking, fingerprinting, and source attribution to track data origin and authenticity -- the clearest federal mapping for 12.5.1 and 12.5.4. See [NIST AI 600-1](https://doi.org/10.6028/NIST.AI.600-1).
- **NIST SP 800-53 Rev 5 AU family** is the concrete control catalog for 12.5.3: **AU-3** (audit record content including subject identity), **AU-9(1)** (write-once/WORM protection of audit data), and **AU-12** (audit generation). See [SP 800-53 AU controls](https://csf.tools/reference/nist-sp-800-53/r5/au/).
- **ISO/IEC 42001:2023** Annex **A.7.5 (Data Provenance)** requires recording where data originates, how it was processed, and by whom, with audit trails; **A.6.2.8** covers event-log recording; Clause 8 governs lifecycle changes and retirement. **ISO/IEC 5259-3:2024** defines a Data Quality Management System across the ML lifecycle. See [ISO/IEC 42001](https://www.iso.org/standard/81230.html) and the [Annex A.7 control guide](https://www.isms.online/iso-42001/annex-a-controls/a-7-data-for-ai-systems/).
- **MITRE ATLAS mitigations** give the threat-informed mapping: **AML.M0025** (Maintain AI Dataset Provenance), **AML.M0023** (AI Bill of Materials), **AML.M0014** (Verify ML Artifacts), and **AML.M0007** (Sanitize Training Data). Relevant techniques include **AML.T0020** (Poison Training Data), **AML.T0019** (Publish Poisoned Datasets), **AML.T0010** (AI Supply Chain Compromise), and **AML.T0058** (Publish Poisoned Models). See the [ATLAS matrix](https://atlas.mitre.org/) and the [ATLAS data repository](https://github.com/mitre-atlas/atlas-data).

### Implementation Maturity

| Requirement | Maturity | Assessment |
|---|---|---|
| 12.5.1 lineage | **Moderate** | Strong for structured pipelines (OpenLineage, MLMD, Pachyderm, W&B); weak for ad-hoc notebook transforms and web-scale crawled corpora where only dataset-level, not component-level, lineage is typically recorded. |
| 12.5.2 labeling logs | **Moderate** | Available in managed platforms, but Label Studio's annotation-history API is Enterprise-only and SageMaker Ground Truth closes to new customers after July 30, 2026. Spreadsheet and in-house workflows frequently omit activity history; logs are often mutable, and guideline-version context is rarely captured. |
| 12.5.3 immutable model-change records | **Emerging** | Registries log to mutable stores by default; true immutability (WORM, Rekor transparency logs, signed attestations) is available but not yet default. Sigstore model-signing v1.0 and its Kubernetes admission-time Model Validation Operator are pushing this toward enforcement, but adoption is early. Inference-time config changes routinely escape the audit trail, and as the 2026 MLflow CVE run shows, a registry's audit log is only as trustworthy as the registry host. |
| 12.5.4 write-time document tagging | **Immature** | The weakest area. C2PA covers media provenance; RAG-chunk tagging is bespoke pipeline code with no enforced, cryptographically bound standard. Forged `source` metadata is largely undetectable. |

## Open Research Questions

- **Component-level lineage at web scale.** No practical method exists to record per-record provenance for billion-item crawled corpora; teams settle for dataset-level lineage, leaving split-view/frontrunning poisoning hard to trace.
- **Cryptographically bound RAG tagging.** Write-time `source`/`writer`/`timestamp` tags on ingested documents are unforgeable only if cryptographically bound to the stored embedding -- no widely adopted standard does this today.
- **Immutable-by-default model registries.** Mainstream registries still write to mutable stores; integrating transparency logs (Rekor) or WORM storage as a default, low-friction path is unsolved.
- **Config-change audit coverage.** Inference parameters, system prompts, and guardrail thresholds change outside the model registry and routinely escape the audit trail; there is no accepted boundary for what counts as an auditable "model change."
- **Distinguishing legitimate relabeling from tampering.** Label edit logs rarely capture the guideline version that justified a change, making malicious label-flipping difficult to separate from sanctioned correction.
