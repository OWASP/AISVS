# C5.3: Query-Time Security Enforcement

> [Back to C05 Index](C05-Access-Control.md)
> **Last Researched:** 2026-04-15 | **Updated:** 2026-04-15

## Purpose

Enforce authorization at the data access layer to prevent unauthorized data retrieval through AI queries. Vector similarity search introduces a distinct class of access control challenge: unlike traditional SQL queries with explicit WHERE clauses, vector searches return results based on mathematical similarity, which can cross tenant and classification boundaries if security filters are not mandatory at the data access layer. Row-level security, field-level masking, and re-authorization on retries are all needed to prevent cross-tenant leakage and TOCTOU vulnerabilities in AI query pipelines.

> **Scope note:** Controls in this section govern data-layer query enforcement -- mandatory security filters, fail-closed behavior, row/field masking, and retry re-authorization. Authorization policy definition is in C5.2; output-level filtering is in C5.4; tenant isolation at the infrastructure level is in C5.5.

| 5.3.1 | **Verify that** all data store queries (e.g., vector databases, SQL databases, search indices) include mandatory security filters (tenant ID, sensitivity labels, user scope) enforced at the data access layer. | 1 | Cross-tenant data leakage via RAG retrieval (OWASP LLM08:2025 — Vector and Embedding Weaknesses); unauthorized access to sensitive documents through vector similarity search bypassing traditional access controls; MITRE ATLAS AML.T0024 (Exfiltration via AI Inference API); retrieval pivot attacks in hybrid RAG where vector seeds bridge into unauthorized graph nodes across tenant boundaries. | Issue queries without required security filters and confirm they are rejected; test with filters for Tenant A and verify no Tenant B data is returned; inspect query construction in application code; run cross-tenant retrieval tests using tools like promptfoo or mcp-attack-labs RAG security lab; for hybrid RAG systems, test graph expansion paths for cross-tenant entity linking (retrieval pivot risk). | This is a critical AI-specific concern. Vector similarity search can return semantically similar content across tenant boundaries if filters are not enforced. Testing in 2025 demonstrated 100% cross-tenant leakage success rates against ChromaDB + LangChain RAG stacks when metadata filters were absent — 20/20 queries leaked data with zero technical sophistication required. **[2026-04]** Pinecone supports up to 100K namespaces per index for tenant isolation with queries limited to one namespace at a time. Weaviate enforces tenant isolation via collections/namespaces where embeddings are never co-mingled. Qdrant supports payload-based filtering with indexed metadata fields for tenant scoping plus first-class multitenancy through named collections with quota controls and tenant-level lifecycle APIs. CVE-2025-64513 (CVSS 9.3) demonstrated a critical authentication bypass in the Milvus Proxy component — unauthenticated attackers could forge a `sourceID` header containing a hardcoded constant (`@@milvus-member@@`) to gain full admin access, including reading all embeddings and metadata, across Milvus versions 2.4.0–2.4.23, 2.5.0–2.5.20, and 2.6.0–2.6.4 (fixed in 2.4.24, 2.5.21, 2.6.5). This reinforces that query-layer filters alone are insufficient if the database authentication layer itself is compromised. Research on retrieval pivot attacks (arxiv 2602.08668, 2026) found that hybrid RAG systems combining vector search with knowledge graph expansion exhibit retrieval pivot risk (RPR) of 0.95 in undefended configurations — 95.4% of benign queries triggered cross-tenant leakage through naturally shared entities at exactly 2 hops, with 16–19 unauthorized items per query and amplification factors of 160–194x compared to vector-only baselines. Per-hop authorization at the vector-to-graph boundary eliminates all leakage (RPR→0.0) with negligible latency overhead. CanaryRAG (arxiv 2604.10717, April 2026) proposes embedding canary tokens into retrieved chunks as a plug-and-play runtime detection mechanism for RAG extraction attacks, modeled as a dual-path integrity game. **[2026-04]** Oracle Deep Data Security, introduced in Oracle AI Database 26ai (April 2026), is the first commercial implementation of database-kernel-level query-time authorization for vector similarity searches — every `VECTOR_DISTANCE()` query, metadata lookup, and chunk retrieval is filtered automatically below the SQL execution layer based on end-user identity propagated via application context (`DBMS_SESSION.SET_CONTEXT`). It enforces row-level, column-level, and cell-level masking declaratively in SQL, making prompt injection and unexpected query construction irrelevant to data access. Supabase launched Vector Buckets in 2026 with S3-backed vector storage supporting up to 50 million vectors per index and multi-index buckets for multi-tenant applications, with PostgreSQL RLS applying to vector tables the same as any relation. |
| 5.3.2 | **Verify that** failed authorization evaluations immediately abort queries and return explicit authorization error codes. | 1 | Partial query execution leaking data before authorization failure is detected; ambiguous error responses masking authorization failures as system errors; MITRE ATLAS AML.T0057 (LLM Data Leakage) where partial results expose sensitive context. | Submit queries with invalid or insufficient credentials; confirm immediate abort (no partial results returned); verify error codes distinguish authorization failures from other error types; test that streaming responses are terminated immediately on auth failure (not just flagged post-generation); inspect RAG pipeline code for try/catch blocks that swallow auth exceptions and return fallback content; verify no fail-open paths exist per EU AI Act Article 15(4) requirements. | Fail-closed behavior is essential. Some AI frameworks return partial results or default outputs on authorization failure rather than explicit errors. In streaming inference pipelines, tokens may be emitted before authorization is fully evaluated — the query engine must buffer or gate output until authorization completes. Error codes should be specific enough for debugging but not leak information about the authorization policy or the existence of protected resources. **[2026-04]** The EU AI Act Article 15(4), enforceable from August 2026, explicitly requires fail-safe design for high-risk AI systems — all requests must be blocked on scan errors, timeouts, or parse failures, with no fail-open paths permitted. Article 12 further mandates structured automatic event logging with agent identification, timestamps, and scan results for regulatory review. Organizations deploying RAG systems in the EU should treat fail-closed query abort as a compliance obligation, not just a security best practice. **[2026-04]** Oracle Deep Data Security in Oracle AI Database 26ai provides a structural solution to fail-closed enforcement: because authorization is enforced at the database kernel below the SQL execution layer, unauthorized rows, columns, and cells are never visible to the query engine — there is no partial result path to fail open on, since the data is filtered before it reaches the application. This eliminates the class of bugs where application-layer try/catch blocks swallow auth exceptions and return fallback content. For systems that cannot use kernel-level enforcement, the OWASP LLM Top 10 2025 (LLM08) specifically recommends permission-aware vector databases that restrict access at the retrieval layer rather than relying on post-retrieval filtering. |
| 5.3.3 | **Verify that** row-level security policies are enabled for all data stores containing sensitive data used by AI systems. | 2 | Sensitive fields (SSN, medical records) exposed through AI queries despite user lacking access; fine-grained data leakage when coarse access controls allow record-level access but not field-level restriction; membership inference attacks (MIAs) against RAG databases revealing whether specific documents exist in the knowledge base. | Query sensitive tables with users at different privilege levels; confirm row filtering and field masking are applied; verify masking persists through joins and derived queries; test that AI pipelines respect RLS policies; run membership inference probes (S2MIA, MBA, interrogation attack) to verify RLS prevents document existence disclosure; verify vector database RBAC is configured per vendor best practices (Qdrant JWT + RBAC, Pinecone per-index RBAC, Milvus RBAC roles). | PostgreSQL, Snowflake, BigQuery support native RLS. Vector databases generally lack native RLS; enforcement must be layered via application-level filtering. Field-level masking for embeddings is conceptually challenging since the embedding encodes the masked content. **[2026-04]** HoneyBee (SIGMOD 2026, arxiv 2505.01538) introduces a dynamic partitioning framework for RBAC in vector databases that achieves 13.5x lower query latency than row-level security filtering with only 1.24x memory overhead, and 90.4% less memory than per-role dedicated indexes. This is the first academic work to co-design access control with vector index partitioning. Recent membership inference research (Li et al., 2025) shows attackers can determine document presence in RAG databases through output analysis — RLS must extend to preventing inference-based disclosure, not just direct retrieval. On the vector database side, as of 2026: Qdrant offers JWT-based RBAC with SSO, TLS, and HIPAA/SOC 2 compliance in its managed cloud; Pinecone provides project-scoped API keys, per-index RBAC, and BYOC mode (GA since 2024) for hard isolation in the customer's own cloud account; Weaviate Enterprise Cloud gained HIPAA compliance on AWS in 2025 with SOC 2 Type II and tenant-aware classes with lifecycle endpoints; Milvus supports RBAC roles but CVE-2025-64513 (CVSS 9.3) exposed that its Proxy authentication could be fully bypassed via a forged header, underscoring that RBAC configuration alone is insufficient without robust authentication at the transport layer. pgvector CVE-2026-3172 (CVSS 8.1, fixed in pgvector 0.8.2, February 2026) demonstrated a buffer overflow via integer wraparound in parallel HNSW index builds that could leak data from other relations or crash the server — affected versions 0.6.0 through 0.8.1. This reinforces that RLS alone is insufficient if the index layer itself has memory-safety vulnerabilities that cross relation boundaries. PostgreSQL 18 (released 2026) continues to strengthen RLS with asynchronous I/O and OAuth authentication improvements, and co-locating embeddings with business data in PostgreSQL via pgvector allows unified SQL governance with RLS enforced at the vector table level — the same RLS policies that protect relational data protect similarity search results. Oracle Deep Data Security in Oracle AI Database 26ai (April 2026) offers an alternative approach: row-level, column-level, and cell-level enforcement at the database kernel below the SQL execution layer, applying transparently to both relational and vector queries without application code changes. |
| 5.3.4 | **Verify that** query retry mechanisms re-evaluate authorization policies to account for dynamic permission changes within active sessions. | 3 | Revoked permissions still honored during retry loops; time-of-check-to-time-of-use (TOCTOU) vulnerabilities where permissions change between initial authorization and retry execution; stale authorization tokens in long-running agentic workflows; retrieval pivot attacks where retry with graph expansion crosses authorization boundaries at hop 2+. | Revoke a user's permission mid-session; trigger a query retry and confirm the retry is denied; verify retry logic calls the PDP rather than relying on cached authorization results; test with LangChain RetryWithErrorOutputParser and similar retry wrappers to confirm they don't carry stale auth context; verify token expiry is shorter than maximum retry window; for hybrid RAG, confirm per-hop re-authorization is enforced on graph expansion during retries. | Retry logic in AI pipelines (e.g., LangChain retry chains, embedding batch retries) often caches context including authorization state. Agentic orchestration frameworks like LangGraph and CrewAI may execute multi-step retry sequences spanning minutes — authorization tokens must be re-validated on each attempt. Requires explicit re-authorization hooks in retry middleware. No widely adopted framework currently enforces re-authorization on retries by default; this must be implemented as custom middleware or interceptors. **[2026-04]** Research on retrieval pivot attacks (arxiv 2602.08668) provides a concrete model for why per-hop re-authorization matters: in hybrid RAG pipelines, each graph expansion step can cross tenant boundaries through naturally shared entities, and the only defense that achieved RPR→0.0 was per-hop authorization checks at every expansion step — cached authorization from the initial vector retrieval is insufficient once the query enters the graph layer. This maps directly to the TOCTOU concern: authorization state valid for the vector retrieval step may not be valid for subsequent graph hops, especially during retries where the expansion path may differ from the original query. |
| **5.3.5** | **Verify that** field-level masking is applied to sensitive fields in all data stores used by AI systems, restricting access to only authorized principals. | 2 | Sensitive metadata (PII, classification labels, source provenance) exposed alongside embeddings in vector databases; embedding inversion attacks recovering 50–70% of original input words from unprotected vectors (OWASP LLM08:2025); MITRE ATLAS AML.T0024 exfiltration via inference API exposing unmasked fields in query responses. | Verify that sensitive metadata fields are encrypted or masked before storage; query the vector database as an unprivileged user and confirm masked fields return redacted values; test that embedding inversion on stored vectors does not recover sensitive source text; inspect application code for per-field encryption configuration (e.g., IronCore Labs Cloaked AI field-level encryption modes); evaluate concept-aware privacy mechanisms (SPARSE) for selective embedding dimension protection. | Traditional field-level masking in relational databases (PostgreSQL dynamic masking, Snowflake column masking policies, BigQuery column-level security) is well-supported. Vector databases present a harder problem: the embedding itself encodes masked content, so field-level masking of metadata alone is insufficient if the embedding is invertible. **[2026-04]** IronCore Labs Cloaked AI provides application-layer field-level encryption for vector database metadata with three modes: random encryption (store-only), deterministic encryption (exact-match filterable), and pass-through. It supports Pinecone, Weaviate, Qdrant, Chroma, Milvus, pgvector, OpenSearch, Elasticsearch, and others. For the embeddings themselves, Cloaked AI uses distance-preserving encryption that maintains nearest-neighbor search accuracy while preventing inversion attacks. Eguard (Liu et al., 2024) is an academic alternative using transformer-based projection networks to protect embeddings while retaining utility — it protects over 95% of tokens from inversion while maintaining downstream task performance. SPARSE (ICLR 2026, arxiv 2602.07090) introduces concept-aware privacy for text embeddings: it uses differentiable mask learning to identify which embedding dimensions encode user-defined sensitive concepts, then applies anisotropic Mahalanobis noise calibrated to dimension sensitivity, achieving 60%+ privacy leakage reduction compared to generalized Laplace mechanisms while maintaining ε-local differential privacy guarantees. SPARSE works across GTR-base, Sentence-T5, and SBERT embedding models. The limitation is that it requires predefining sensitive information categories — unspecified attributes receive less protection. As of April 2026, Oracle Deep Data Security in Oracle AI Database 26ai provides the first database-kernel-level field masking for vector workloads: column-level hiding (entire columns vanish from result sets based on agent identity — the agent cannot detect the column's existence) and cell-level masking (individual cell values return NULL based on clearance level). This is the closest any commercial system has come to native field-level masking for vector queries. For other vector databases, application-layer enforcement remains the only viable approach, though the tooling landscape is maturing rapidly. |
| **5.3.6** | **Verify that** row-level security and field-level masking policies are inherited by derived or replicated data stores and are not bypassed by replication or ETL processes. | 2 | RLS bypass via materialized views exposing all tenant data without policy enforcement; ETL processes running under elevated service accounts extracting unfiltered data from RLS-protected tables; replication pipelines copying sensitive data to less-protected downstream systems; SECURITY DEFINER views in PostgreSQL executing with creator privileges and bypassing RLS entirely; hybrid RAG graph expansion inheriting embeddings from vector stores without carrying forward tenant-scoped filters. | Verify that materialized views derived from RLS-protected tables carry their own access policies; test ETL exports to confirm tenant filtering is applied during extraction (not just at source); audit replication targets for equivalent RLS/masking policies; for PostgreSQL, verify views use `security_invoker = true` (available in PostgreSQL 15+); for Snowflake, confirm row access policies replicate via replication groups; for hybrid RAG, verify graph expansion nodes inherit the authorization context of the originating vector query. | This is a well-documented gap across all major platforms. **[2026-04]** In PostgreSQL, materialized views bypass source table RLS entirely — data copied to materialized views is unfiltered. Prior to PostgreSQL 15, SECURITY DEFINER views (the default when created by superusers) also bypassed RLS; PostgreSQL 15+ supports `security_invoker = true` to enforce caller-context RLS on views. In Snowflake, row access policies can be set on materialized views but not if the underlying table already has one (mutual exclusion constraint). Snowflake supports replicating row access policies via database replication groups. BigQuery maintains RLS enforcement through materialized views but with performance limitations (no BI Engine acceleration). For vector databases, there is no native RLS inheritance — any ETL pipeline that copies embeddings to a secondary store (e.g., for analytics or backup) must explicitly re-apply tenant filters and metadata encryption. ETL tools like dbt, Fivetran, and Airbyte do not natively enforce source RLS during extraction; explicit WHERE clauses or pre-filtered views must be used as extraction sources. Research on retrieval pivot attacks (arxiv 2602.08668, 2026) extends this concern to hybrid RAG architectures: when vector search results feed into a knowledge graph expansion layer, the authorization context from the vector query does not automatically propagate — graph traversal can bridge across tenant boundaries through shared entities, with amplification factors of 160–194x over vector-only baselines. The paper found 95.4% of benign queries triggered cross-tenant leakage through naturally shared entities at exactly 2 hops, demonstrating that policy inheritance across data layers is not just an ETL concern but an active runtime risk in modern RAG pipelines. **[2026-04]** Oracle Deep Data Security addresses this gap by enforcing access policies at the database kernel level — policies apply consistently across relational tables, materialized views, and vector similarity searches without requiring separate enforcement per data layer, and they cannot be bypassed by ETL processes or replication since the kernel filters all query execution paths. For PostgreSQL-based stacks, pgvector 0.8.2 (February 2026) fixed CVE-2026-3172 (CVSS 8.1), a buffer overflow in parallel HNSW index builds that could leak data across relations — organizations replicating pgvector indexes to secondary stores should ensure all replicas run patched versions to prevent index-layer leakage that circumvents RLS entirely. |
