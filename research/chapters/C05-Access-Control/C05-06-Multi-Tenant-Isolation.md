# C5.6: Multi-Tenant Isolation

> [Back to C05 Index](C05-Access-Control.md)
> **Last Researched:** 2026-05-13 (refresh: vLLM v0.20.0 prefix-cache salt and SHA256 guidance, vLLM deployment-hardening and API-key limitation guidance, dynamic LoRA resolver production warning, CacheSolidarity selective prefix-cache isolation research, LMCache P2P KV-sharing isolation considerations, MITRE ATLAS v5.6.1/v5.6.0, NIST CAISI agent identity/authorization work, NIST COSAiS overlays)

## Purpose

Ensure logical and cryptographic isolation between tenants in shared AI infrastructure. Multi-tenant AI deployments share GPU clusters, model-serving endpoints, vector databases, and inference caches across tenants. Without rigorous isolation, cross-tenant data leakage can occur through shared KV-caches, embedding stores, network segments, and even GPU memory residuals. Recent research has demonstrated that timing side channels in shared inference infrastructure can reconstruct other tenants' prompts with 95-99% accuracy, making KV-cache isolation a critical and rapidly evolving concern. As of April 2026, new attack techniques like RL-enhanced prompt leakage (RL2eak, OptiLeak) have further reduced the cost and effort of exploiting shared caches, container escape vulnerabilities in NVIDIA Container Toolkit (CVE-2024-0132, CVE-2025-23266) have underscored the systemic fragility of GPU isolation in shared environments, and CPU-level cache side channels (Spill The Beans, arxiv 2505.00817) have shown that token-level leakage is possible even without GPU-side cache sharing. Cloud-management-plane vulnerabilities remain just as impactful as data-plane attacks: CVE-2026-20965 (Cymulate Research Labs, reported Aug 2025, patched in Windows Admin Center Azure Extension v0.70.00 on January 13, 2026) let a local admin on any Azure VM or Arc-joined machine bypass Azure AD SSO token validation and pivot to every other WAC-installed machine in the same tenant — a textbook demonstration that tenant isolation failures in shared management tooling have the same blast radius as GPU-layer escapes, and the April 2026 Patch Tuesday added CVE-2026-32196 (Windows Admin Center spoofing) to the same attack surface. The same Patch Tuesday (163–168 CVEs depending on count, including two actively-exploited 0-days CVE-2026-32201 SharePoint spoofing and CVE-2026-33825 Defender) also included **CVE-2026-27928**, a Windows Hello security-feature bypass caused by improper input validation; NVD lists it as CVSS 8.7 with Scope:Changed (`S:C`), so it is still a useful identity-control example even though it is not an Azure tenant-isolation CVE. The `S:C` scope-changed classification is the one operators should watch closely on shared-infrastructure procurement checklists: it signals that exploitation can affect a security authority beyond the initially vulnerable component, the same kind of boundary-crossing pattern that NVIDIAScape and the WAC CVE instantiated in different layers of the stack. On the threat-taxonomy side, MITRE ATLAS has continued expanding through v5.6.0 (April 30, 2026); the v5.1.0 changelog baseline documented **16 tactics, 84 techniques, 56 sub-techniques, 32 mitigations, and 42 case studies**, with later 2026 releases adding agent supply-chain, cost-harvesting, AI-service-proxy, and deepfake-assisted phishing coverage — "Exfiltration via AI Inference API" and "LLM Data Leakage" techniques both apply directly to shared-tenancy inference surfaces. The OWASP LLM Top 10 2025 explicitly calls out multi-tenant isolation gaps under LLM02 (Sensitive Information Disclosure) and LLM04 (Data and Model Poisoning) -- the reference mitigation for a shared vector database with only application-layer tenant filtering is hard cryptographic namespace segregation per tenant at the database level, because a crafted embedding payload can otherwise pull another tenant's chunks into the retrieval context. On the defensive side, NVIDIA Blackwell B200 confidential computing now provides encrypted NVLink/NVSwitch GPU-to-GPU communication with near-native inference performance, and multiple new KV-cache defense frameworks (entropy-based obfuscation, CacheSolidarity, KV-Cloak) offer layered protection against timing side channels. At KubeCon Europe 2026, NVIDIA announced deep Kata Containers integration with DRA (Dynamic Resource Allocation) and CDI for GPU-aware confidential containers, providing a "glide path" from VM-level isolation to full confidential computing with encrypted and measured VM memory. Enterprise KV-cache management is also maturing: LMCache has emerged as a widely-adopted open-source KV caching layer (integrated with vLLM, SGLang, Red Hat, CoreWeave, Google Cloud) that supports cache offloading and prefill-decode disaggregation at scale -- though its tenant isolation story still depends on the underlying engine's salt-based partitioning. On the network isolation front, NVIDIA's BlueField Astra architecture (announced for Vera Rubin NVL72) anchors security enforcement in the BlueField-4 DPU rather than the host OS, extending a unified trust boundary across both front-end (North-South) and AI fabric (East-West) networking domains -- tenant workloads use SuperNIC for data movement but have zero access to management functions, closing a longstanding gap where host-OS compromise could bypass network isolation. Regulated workloads push the isolation bar further still: DoD Impact Level 5 requires physical separation from non-DoD tenants on top of FedRAMP High, and in March 2026 Oracle announced DISA IL5 and FedRAMP High authorization for OCI Generative AI and Exadata Cloud@Customer, making GenAI inference with enforced physical tenant separation a concrete procurement option rather than a paper requirement. On March 3, 2026 Corvex announced the first verified production deployment of NVIDIA HGX B200 confidential computing for multi-tenant AI, combining Intel TDX on the CPU side with NVIDIA Confidential Computing on the GPU side, encrypted NVLink/NVSwitch fabric, and CPU+GPU remote attestation through Intel Trust Authority -- this flips GPU confidential computing from a lab demo into a purchasable production primitive. Nine days later (March 12, 2026) Corvex followed up with Secure Model Weights, which keeps model weights encrypted until they are decrypted inside the GPU TEE using post-quantum Kyber-768 key exchange and open-source Confidential Containers, addressing the long-standing concern that sensitive model IP is exposed to the cloud host during load. NVIDIAScape (CVE-2025-23266, CVSS 9.0) remains the canonical example of why host-layer isolation alone is insufficient: it affects roughly 37% of cloud environments running GPU workloads and allows a single container breakout to reach all co-tenants' data and model weights, which is now driving enterprise adoption of purpose-built hardened runtimes such as Edera's zone-based isolation (announced KVM support and the `ocirender` Rust library at KubeCon Europe 2026) alongside Kata Containers with NVIDIA DRA/CDI integration. The inference layer itself continues to produce tenant-isolation breaks at the application level: CVE-2026-22778 (disclosed February 2, 2026, CVSS 9.8) chains a PIL exception heap-address leak with a JPEG2000 heap overflow in the FFmpeg/OpenCV path used by vLLM's multimodal endpoints, allowing an unauthenticated attacker to send a crafted video URL to `/v1/chat/completions` or `/v1/invocations` and achieve arbitrary code execution on the serving host -- affecting vLLM 0.8.3 through 0.14.0 inclusive and patched in v0.14.1, with the practical consequence that a single malicious tenant on a shared vLLM deployment can reach every other tenant's prompts, KV cache, and model weights without touching the GPU isolation layer at all; CVE-2026-22807 and CVE-2026-27893 extend the same trust boundary problem, where vLLM's loader honored `trust_remote_code` paths from Hugging Face `auto_map` dynamic modules even when operators had explicitly opted out, turning any tenant-uploadable model repository into an RCE primitive (CVE-2026-27893 was the specific incomplete-fix bypass — `nemotron_vl.py` and `kimi_k25.py` both retained hardcoded `trust_remote_code=True` paths that overrode the global `--trust-remote-code=False` flag, CVSS 8.8, patched in vLLM 0.18.0 on March 26, 2026). The problem class is broader than multimodal: **CVE-2025-62164** (CVSS 8.8, patched in vLLM 0.11.1) exposes the Completions API `prompt_embeds` parameter to a `torch.load()` deserialization flaw where PyTorch 2.8.0+ disabled sparse-tensor integrity checks by default, letting any API-key-holding tenant write out of bounds and achieve RCE on the serving host by uploading a crafted serialized tensor; **CVE-2026-24779** (MediaConnector SSRF, patched 0.15.1) and **CVE-2026-25960** (CVSS 5.4, patched 0.17.0) show the same URL-parsing inconsistency pattern — `urllib3.util.parse_url()` validates hostnames while `aiohttp`/`yarl` executes the request, and backslashes are treated differently by each parser so `https[:]//allowed.example\@attacker.tld/` passes validation but resolves to the attacker's host, letting any tenant pivot to internal services and metadata endpoints. On April 3, 2026 vLLM published three more advisories in a single batch: **GHSA-pq5c-rjhq-qp7p** (Moderate, DoS via `VideoMediaIO.load_base64()` parsing thousands of comma-separated base64 JPEG frames), **GHSA-pf3h-qjgv-vcpr** (Moderate, SSRF via `download_bytes_from_url` processing batch-input JSON with user-controlled `file_url`), and **GHSA-3mwp-wvh9-7528** (Moderate, OOM DoS via the unbounded `n` parameter in the OpenAI-compatible server) — the cumulative pattern across the last quarter is that media/URL-parsing and deserialization paths are the dominant inference-layer attack surface for multi-tenant serving stacks, not novel model-layer attacks. Outside vLLM, **CVE-2026-27940** (llama.cpp integer overflow in `gguf_init_from_file_impl()` in `gguf.cpp`, CVSS 7.8, published March 13, 2026, patched in commit b8146) is an incomplete-fix bypass of CVE-2025-53630: the size calculation still wraps under certain GGUF tensor headers, producing an undersized heap allocation that the subsequent `fread()` overflows by 528+ attacker-controlled bytes, which matters for any shared-inference platform that accepts tenant-uploaded GGUF weights (on-prem llama.cpp-based private-cloud offerings and mobile-edge inference wrappers in particular). On the standards side, **NIST CAISI** (Center for AI Standards and Innovation) launched the **AI Agent Standards Initiative** on February 17, 2026, organized around three pillars (industry-led standards via ISO/IEC JTC 1, community-led open-source agent protocols co-invested with NSF, and fundamental research in AI agent security and identity infrastructure) -- April 2026 virtual workshops are soliciting sector-specific barriers to multi-tenant agentic AI adoption in healthcare, finance, and education, and are explicitly formalizing authentication, authorization, and governance expectations for enterprise agent deployments. In parallel, **NIST COSAiS** (Control Overlays for Securing AI Systems, building on SP 800-53 plus SP 800-218A, Draft NIST AI 800-1, and NIST AI 100-2e2025) published its first discussion-draft annotated outline on January 8, 2026 covering Predictive AI adaptation and fine-tuning; the comment window closed February 13, 2026, and the single-agent and multi-agent overlays -- which directly address multi-tenant agent cooperation and per-tenant authorization for autonomous planning systems -- are on track for mid-to-late 2026 discussion-draft release, with agencies unable to wait for final publication being offered interim compensating-control guidance. NVIDIA's developer-blog guidance "Structuring Applications to Secure the KV Cache" (April 29, 2025) is now the most practical per-request mitigation pattern operators can adopt today without reworking the serving stack: order dynamically constructed prompts as **system prompt → non-guessable, periodically rotated per-session or per-tenant identifier → tool/plugin augmentation context → validated user prompt**, set maximum length limits, avoid repeating user input across prompt sections, and pair prefix caching with monitoring for enumeration-style query patterns (repeated queries with minor variations) combined with rate limiting -- the same blog explicitly recommends considering "isolating KV caches across tenants entirely" via session-ID cache keys or disabling prefix caching in high-risk contexts, which slots in cleanly above vLLM `cache_salt` as a prompt-layer defense-in-depth.

> **Scope note:** Controls in this section govern infrastructure-level tenant isolation -- network policies, namespace segregation, per-tenant encryption, and KV-cache partitioning. Query-time tenant filtering is in C5.3; authorization policy is in C5.2.

## Current Research Notes

- **[2026-05] KV-cache partitioning:** vLLM v0.20.0 documents cache salts as part of the prefix-cache block hash material for multi-tenant isolation and makes `sha256` the default hash algorithm as of v0.11; the same guidance warns that non-cryptographic `xxhash` can theoretically cause undefined behavior or leak private information in multi-tenant deployments.
  For 5.6.1, auditors should verify `cache_salt` is set from authenticated tenant or session identity, `--prefix-caching-hash-algo` is `sha256` or `sha256_cbor`, and high-sensitivity tenants either use per-tenant cache keys or have prefix caching disabled.
  See [vLLM prefix caching](https://docs.vllm.ai/en/v0.20.0/design/prefix_caching/).
- **[2026-05] Serving-stack patch floor:** vLLM's practical multi-tenant baseline has moved to v0.20.0 or later. v0.14.1 patched the critical video-processing RCE and MediaConnector SSRF, v0.17.0 patched the backslash-parser SSRF bypass, v0.18.0 patched CVE-2026-27893 hardcoded `trust_remote_code=True`, v0.19.0 patched the April 3 DoS/SSRF/OOM advisory batch, and v0.20.0 patched GHSA-83vm-p52w-f9pw where one request with a penalty parameter can crash `extract_hidden_states` speculative decoding.
  While rollout lags, gateways should reject or tightly constrain tenant-controlled `video_url`, batch-input `file_url`, `prompt_embeds`, oversized `n`, and speculative-decoding penalty combinations.
  See [vLLM security advisories](https://github.com/vllm-project/vllm/security/advisories) and [GHSA-83vm-p52w-f9pw](https://github.com/vllm-project/vllm/security/advisories/GHSA-83vm-p52w-f9pw).
- **[2026-05] Cache sharing scale vs. isolation:** LMCache's January 2026 P2P CPU-memory sharing work shows why cache reuse is operationally attractive: in its two-instance demonstration, peer retrieval cut mean TTFT from 2.028s to 0.490s and total round time from 38.323s to 7.964s.
  That same architecture expands the isolation boundary to worker registration, heartbeat, controller metadata, peer KV transfer paths, and cache-key derivation.
  For 5.6.1 and 5.6.2, treat those as tenant-scoped security surfaces, not just performance plumbing. See [LMCache P2P sharing](https://blog.lmcache.ai/en/2026/01/21/p2p-1/) and [LMCache integration](https://docs.lmcache.ai/developer_guide/integration.html).
- **[2026-05] Threat taxonomy and standards:** MITRE ATLAS v5.6.0 shipped on April 30, 2026, and the March v5.5.0 release added agent supply-chain and cost-harvesting techniques that matter for shared agent platforms.
  NIST CAISI's AI Agent Standards Initiative now explicitly tracks agent authentication and identity infrastructure for secure human-agent and multi-agent interactions, while COSAiS is turning SP 800-53, SP 800-218A, Draft NIST AI 800-1, and NIST AI 100-2e2025 into implementation-focused overlays.
  For this page, the practical mapping is straightforward: tenant isolation evidence should include identity binding, authorization scope, telemetry, segmentation, artifact-signing, and cache/model-state lifecycle controls. See [MITRE ATLAS changelog](https://github.com/mitre-atlas/atlas-data/blob/main/CHANGELOG.md), [NIST AI Agent Standards Initiative](https://www.nist.gov/artificial-intelligence/ai-agent-standards-initiative), [NIST COSAiS](https://csrc.nist.gov/Projects/cosais), and [NIST SP 800-218A](https://csrc.nist.gov/pubs/sp/800/218/a/final).
- **[2026-05] CVE-2026-27928 handling:** NVD and MSRC describe CVE-2026-27928 as a Windows Hello security-feature bypass caused by improper input validation, with CVSS v3.1 scope changed (`S:C`).
  It remains useful as a changed-scope identity-control example, but it should not be treated as an Azure tenant-isolation CVE.
  See [NVD CVE-2026-27928](https://nvd.nist.gov/vuln/detail/CVE-2026-27928) and [MSRC CVE-2026-27928](https://msrc.microsoft.com/update-guide/vulnerability/CVE-2026-27928).
- **[2026-05] vLLM deployment hardening:** vLLM's security guidance now gives auditors concrete multi-tenant checks beyond framework version: inter-node traffic for PyTorch Distributed, KV-cache transfer, and tensor/pipeline/data parallelism is insecure by default unless isolated on trusted networks; firewalls should expose only the API listener and keep internal communication ports tenant-private.
  The same guidance warns that `--api-key` protects the OpenAI-compatible `/v1` endpoints only, while endpoints such as `/invocations`, `/pause`, `/resume`, `/tokenize`, `/detokenize`, and development-only cache reset/RPC endpoints need reverse-proxy and network controls.
  For 5.6.2 evidence, ask for listener bindings, firewall rules, endpoint exposure tests, and proof that tenant-facing deployments do not expose internal control endpoints. See [vLLM Security](https://docs.vllm.ai/en/v0.20.0/usage/security/).
- **[2026-05] Dynamic LoRA loading boundary:** vLLM's LoRA documentation says runtime resolver plugins can load adapters from local files or Hugging Face Hub, but enabling remote downloads is not intended for production.
  In shared serving clusters, treat `VLLM_ALLOW_RUNTIME_LORA_UPDATING`, `VLLM_PLUGINS`, resolver cache directories, and S3/Hugging Face repository allowlists as tenant-bound security controls rather than convenience settings.
  A practical 5.6.2 test is to attempt cross-tenant adapter selection, adapter replacement through `load_inplace`, and path traversal or repository-confusion cases while confirming every adapter is signed and mapped to the authenticated tenant. See [vLLM LoRA adapters](https://docs.vllm.ai/en/v0.20.0/features/lora/).
- **[2026-05] Selective prefix-cache isolation:** CacheSolidarity (arxiv 2603.10726, March 2026) proposes a middle ground between disabling automatic prefix caching and leaving cross-user cache reuse open: track owner metadata on cache entries, flag suspicious cross-user reuse, and selectively isolate prefixes when hit/miss timing becomes distinguishable.
  This is not yet a mainstream serving-engine control, so auditors should not accept it as deployed unless there is implementation evidence, but it is useful as a design pattern for vendors that need 5.6.1 assurance without sacrificing all cache efficiency.
  See [CacheSolidarity](https://arxiv.org/abs/2603.10726) and [vLLM prefix caching](https://docs.vllm.ai/en/v0.20.0/design/prefix_caching/).
- **[2026-05] ATLAS and agent standards drift:** MITRE ATLAS v5.6.1 was a small schema-fix release on May 5, 2026, while v5.6.0 on April 30 added `Acquire Public AI Artifacts: AI Agent Configuration`, `Search Open Websites/Domains: Code Repositories`, and `Phishing: Deepfake-Assisted Phishing`; the March v5.5.0 release added AI agent tool poisoning, AI service proxy, and cost-harvesting techniques plus updates to AI telemetry logging and component segmentation mitigations.
  For multi-tenant platforms, the important control mapping is identity-bound agent/tool execution, tenant-scoped telemetry, signed agent/tool artifacts, and segmentation of agent components from shared model-serving and cache planes.
  See [MITRE ATLAS changelog](https://github.com/mitre-atlas/atlas-data/blob/main/CHANGELOG.md) and [NIST AI Agent Standards Initiative](https://www.nist.gov/artificial-intelligence/ai-agent-standards-initiative).

| **5.6.1** | **Verify that** inference-time KV-cache entries are partitioned by authenticated session or tenant identity and that automatic prefix caching does not share cached prefixes across distinct security principals, to prevent timing-based prompt reconstruction attacks. | 2 | Timing-based prompt reconstruction via KV-cache sharing side channels; the PROMPTPEEK attack (NDSS 2025) achieves 95-99% accuracy reconstructing other tenants' prompts by probing cache hit/miss latency differences in vLLM and SGLang. Direct KV-cache content extraction enables collision and injection attacks that reverse-engineer user inputs from plaintext cache entries (Shadow in the Cache, arxiv 2508.09442). Cross-tenant prefix reuse leaks query patterns even when outputs are properly isolated. **[2026-04]** RL-enhanced attacks now lower the cost of exploitation: RL2eak uses reinforcement learning with DPO to identify "hard tokens" (sensitive domain-specific terms), achieving 12.48x reduction in token requests needed for prompt reconstruction on domain-specific datasets. OptiLeak (arxiv 2602.20595, Feb 2026) similarly uses RL for efficient prompt reconstruction. The Shadow-in-the-Cache / KV-Cloak line of work at **NDSS 2026** ("Unveiling and Mitigating Privacy Risks of KV-cache in LLM", arxiv 2508.09442) is the first comprehensive academic taxonomy of KV-cache attacks as a distinct class with three branches: (i) **KV-cache Inversion** — reverse-calculate the input prompt directly from cached K/V tensors using the model's own known weight matrices; (ii) **KV-cache Collision** — frame prompt reconstruction as a matching problem by running candidate inputs through a local copy of the same model and comparing the produced KV-caches against the intercepted target; and (iii) **KV-cache Injection** — semantically manipulate other tenants' cached context rather than just reading it. Operators evaluating a multi-tenant inference stack should treat "is the KV cache readable outside its originating session?" and "is the KV cache writable by anyone other than the originating session?" as two separate questions — pure read-path defenses (salting, selective sharing) do not mitigate the injection branch. | Confirm inference framework uses per-tenant or per-session cache salting (e.g., vLLM `cache_salt` parameter, merged May 2025); measure response latency variance across probe sequences to detect cache-hit timing leakage; verify automatic prefix caching does not share prefixes across tenant boundaries; inspect KV-cache storage to confirm entries are not persisted in plaintext across sessions; test that expired or terminated sessions have their cache entries evicted. | **[2026-04]** Active research area with rapidly maturing defenses. vLLM merged cache salting (RFC #16016, May 2025) for single-barrier tenant isolation; hierarchical multi-barrier design (per-message salts) is in active development. The official vLLM docs explicitly warn that the underlying default hash function is not cryptographically secure, so a determined attacker with control over prompt content could in principle force salt collisions -- this is a known tradeoff against the 14-24x throughput gain from PagedAttention and the ~10x cost reduction on cached tokens, and should be explicitly weighed against workload sensitivity rather than silently enabled. **[2026-04]** vLLM v0.8.3+ now supports SHA256 as the prefix-cache hash function (PR #15297) to close the collision-forcing gap, and in later builds SHA256 has been promoted to the default for new deployments; operators running older builds should set `--prefix-caching-hash-algo sha256` explicitly when `enable_prefix_caching=True` in multi-tenant configurations. Service providers can also combine a per-organization default `cache_salt` with per-request overrides so cache reuse is scoped to a trust group rather than the entire serving instance. SafeKV (arxiv 2508.08438) integrates ChunkGuard for real-time sensitivity classification with selective cache sharing -- mitigates 94-97% of timing side-channels while reducing TTFT overhead by up to 40% vs. full isolation and raising throughput up to 2.66x. KV-Cloak (arxiv 2508.09442) provides reversible matrix obfuscation of KV-cache contents with ~5% latency overhead and negligible accuracy impact. CacheSolidarity (arxiv 2603.10726, March 2026) takes a selective isolation approach: extends each KV-cache entry with minimal metadata to track user-prompt interactions, flags anomalous cross-user reuse, and selectively isolates prefixes only when threats are detected. The paper reports CacheSolidarity stays within a 6% performance margin of unrestricted Automatic Prefix Caching while blocking timing side-channels -- if a prefix is flagged it stops reuse at that token and recomputes the remainder, producing a partial hit rather than forcing full isolation. This "detect-and-isolate" posture is currently the closest thing to a drop-in defense that does not require application changes. Entropy-based obfuscation (IJISRT, September 2025) injects controlled randomness into cached attention key-value states using Shannon and Rényi entropy as guiding metrics, dynamically modulating perturbation levels to balance privacy and performance -- effective at reducing prompt leakage with relatively small computational cost. Beyond GPU-level caches, Spill The Beans (arxiv 2505.00817, May 2025) demonstrates that CPU cache side channels can leak individual tokens from LLM inference even without shared GPU memory, expanding the attack surface to include the host processor. No production framework yet combines all five defensive approaches (salting + selective sharing + content obfuscation + anomaly-based isolation + entropy perturbation); organizations should evaluate their threat model to choose the right tradeoff between isolation strength and performance. **[2026-04]** Shadow in the Cache (NDSS 2026, February 2026) presented the first comprehensive taxonomy of KV-cache attack vectors -- direct inversion attacks that reconstruct prompts from cache entries, collision attacks that identify matching prefixes, and semantic injection attacks that manipulate cached context. KV-Cloak's reversible matrix obfuscation with operator fusion was formally validated at the same venue. LMCache has become the de facto open-source KV caching layer for enterprise inference (integrated with vLLM, SGLang, Google Cloud, CoreWeave, Redis, PliOps), delivering up to 15x throughput improvement and 2x latency reduction via cache offloading and prefill-decode disaggregation. As of LMCache v0.11 (January 2026 P2P CPU memory sharing release cycle), the default hashing algorithm is SHA256, which closes the collision-forcing gap that prior hash choices left open in multi-tenant deployments — operators running older LMCache builds should upgrade or explicitly pin `sha256` on the underlying engine. However, LMCache's tenant isolation still depends entirely on the underlying engine's cache_salt implementation — it does not add its own isolation layer, so organizations must verify that salt-based partitioning is active end-to-end when deploying LMCache in multi-tenant configurations, especially when the P2P CPU memory sharing feature is enabled across nodes. FlexServe (arxiv 2603.09046, March 2026) introduces TrustZone-based Flexible Resource Isolation for on-device inference, creating Flexible Secure Memory (Flex-Mem) to protect model weights, KV caches, and intermediate tensors with a lightweight secure hypervisor, achieving 10x TTFT improvement over naive TEE approaches -- relevant for edge multi-tenant scenarios where GPU TEEs are unavailable. TZ-LLM (EuroSys '26, arxiv 2511.13717, April 2026) complements FlexServe with a pipelined-restoration scheme that prefetches encrypted model parameters into Arm TrustZone memory on demand and a co-driver design that keeps only a minimal NPU data-plane driver inside the TEE — reducing TCB size, shrinking world-switch overhead, and reporting up to 90.9% TTFT reduction vs. a strawman TEE baseline on OpenHarmony / llama.cpp. Together FlexServe and TZ-LLM are pushing on-device tenant isolation (per-app secure inference on shared mobile NPUs) from "research curiosity" toward something a mobile OS vendor can ship. **[2026-04]** For operators who cannot yet adopt cryptographic cache salting or selective-isolation frameworks, NVIDIA's developer-blog guidance "Structuring Applications to Secure the KV Cache" (April 29, 2025) documents a request-composition pattern that meaningfully reduces cross-tenant prefix collision risk without touching the serving stack: order every prompt as **system prompt → non-guessable rotating per-session or per-tenant identifier → tool/plugin augmentation context → sanitized and length-capped user prompt**, and monitor for enumeration-style probe patterns (repeated queries with minor variations) alongside rate limits -- the same note explicitly advises disabling prefix caching entirely in high-risk contexts, which for regulated tenants is often the right default until hardened multi-tenant cache frameworks (SafeKV, CacheSolidarity, KV-Cloak) ship in mainstream inference engines. |

The April 3, 2026 vLLM batch (GHSA-pq5c-rjhq-qp7p unbounded base64 frame DoS in `VideoMediaIO.load_base64()`, GHSA-pf3h-qjgv-vcpr SSRF via `download_bytes_from_url` batch-input JSON, GHSA-3mwp-wvh9-7528 OOM DoS via the unbounded OpenAI-compatible `n` parameter) confirms the pattern: media ingestion and URL-parsing paths are the chronic multi-tenant weakness, not novel model-layer attacks. **CVE-2026-27940** (llama.cpp GGUF integer overflow in `gguf_init_from_file_impl()`, CVSS 7.8, fixed in commit b8146) extends the pattern to any shared llama.cpp-based tenant-model-upload service, especially edge and mobile deployments. **[2026-05]** Microsoft's April Patch Tuesday also added **CVE-2026-27928**, a Windows Hello security-feature bypass caused by improper input validation with CVSS Scope:Changed semantics. Treat it as an identity/control-plane boundary warning, not as an Azure tenant-isolation CVE. Batched heterogeneous LoRA execution (Punica SGMV, S-LoRA unified paging) can leak adapter activation patterns across tenants when kernels don't strictly bound memory regions, and careless adapter caching (e.g., LoRAX tenant routing) allows one tenant's adapter to be served to another if API-key-to-adapter binding is misconfigured. Fine-tuning-time poisoning where a tenant's malicious training job corrupts shared base-model state or shared optimizer buffers (MITRE ATLAS T0020 Poison Training Data, T0031 Erode ML Model Integrity — ATLAS v5.1.0 documented 16 tactics, 84 techniques, 56 sub-techniques, 32 mitigations, and 42 case studies, and v5.5.0/v5.6.0 added agent supply-chain, cost-harvesting, AI-service-proxy, and deepfake-assisted phishing coverage that operators should cross-check against their multi-tenant threat model). Cloud-management-plane pivots: CVE-2026-20965 (Windows Admin Center Azure Extension, Cymulate Research Labs) let a local admin on any Azure VM or Arc-joined machine bypass Azure AD SSO and gain tenant-wide RCE on every other WAC-installed machine — a reminder that tenant isolation in AI-adjacent management tooling (orchestrators, notebook hosts, training dashboards) is just as important as the GPU and network layers, and April 2026's follow-on CVE-2026-32196 (WAC spoofing) widened the same attack surface. Embedding-service cross-tenant probing: Privacy Side Channels in ML Systems (USENIX 2024) showed that attackers can probe system-level components to leak more about training and query sets than pure model-API attacks. Whisper Leak (arxiv 2511.03675, Nov 2025) showed that even encrypted streaming inference responses leak topic and content across tenants via packet timing. **[2026-04]** Spill The Beans (arxiv 2505.00817) and Unveiling Hardware Cache Side-Channels in Local LLMs (USENIX Security 2025) confirmed that cache timing in embedding layers can recover other tenants' tokens — including high-entropy credentials — even without shared GPU memory. | Review multi-LoRA serving configuration: confirm vLLM, LoRAX, SageMaker, Punica, or S-LoRA are **not** running with `--enable-lora --allow-runtime-lora-adapter-updates` or equivalent dynamic-load flags in production (vLLM docs explicitly state dynamic LoRA loading "should not be used in production unless it is an isolated, fully trusted environment"). Verify every adapter loaded at runtime carries a cryptographic signature bound to the tenant and that the API gateway enforces an allowlist mapping `api_key → {allowed_adapter_ids}`; attempt to invoke another tenant's adapter and confirm 403. Inspect GPU partitioning: for NVIDIA H100/B200 confirm MIG is enabled with dedicated memory and compute slices per tenant, with separate isolated paths through the entire memory system; for Blackwell B200 confirm Confidential Computing with TEE-I/O is enabled so model weights and activations are encrypted end-to-end (<5% overhead for typical LLM queries). Check Kubernetes isolation: each tenant workload runs in its own namespace with NetworkPolicy / Cilium L7 rules, and GPU-accelerated pods use Kata Containers with GPU pass-through (announced at KubeCon Europe 2026 via NVIDIA DRA/CDI integration) for hardware-backed VM-level isolation rather than a shared container runtime. Confirm GPU memory zeroing between jobs (vcluster / Run:AI lifecycle hooks, or `nvidia-smi --gpu-reset`); run a canary job that writes a known pattern to VRAM and a follow-up job that scans for residuals. Test that fine-tuning jobs run in isolated per-tenant LoRA workspaces and cannot write to shared base weights or shared gradient/optimizer state. Validate embedding-service request batching: confirm that in a single batch, per-tensor masks prevent one tenant's inputs from attending to another's, and that attention/softmax kernels are validated against segmented-batch reference implementations. LoRAX supports per-request tenant isolation and API-key-to-adapter binding, but enforcement is application-layer; there is no cryptographic adapter-to-tenant attestation in any mainstream open-source multi-LoRA server. Punica and S-LoRA papers focus on throughput/latency — neither has a published formal security analysis for cross-tenant leakage in their heterogeneous batching kernels, and the newer dLoRA and LoRAServe (2025-2026) inherit the same gap. SageMaker multi-tenant LoRA (AWS ML blog, 2025) documents tenant isolation at the control-plane level but relies on IAM and account-level separation rather than GPU-level isolation. NVIDIA Blackwell B200 Confidential Computing with TEE-I/O is the first production option for cryptographic memory + NVLink isolation at near-native inference performance, and Vera Rubin NVL72 extends confidential computing to rack scale with unified CPU-GPU-NVLink trust domains — but both require application code to opt into attestation, and adoption across major clouds (Azure confidential VMs with H100 are GA; B200 confidential compute is rolling out through 2026) is still partial. **[2026-04]** Corvex's March 3, 2026 deployment is the first publicly verified production reference for HGX B200 confidential computing outside hyperscaler internal trials, pairing Intel TDX, NVIDIA CC, encrypted NVLink, and CPU+GPU remote attestation via Intel Trust Authority — this takes B200 CC from roadmap item to purchasable primitive. For operators who cannot yet adopt B200 CC, Edera for GPUs (hardened Xen-based runtime, announced KVM support and the `ocirender` Rust image pipeline at KubeCon Europe 2026) provides a Kata-class alternative specifically positioned against the NVIDIAScape (CVE-2025-23266) class of container-escape vulnerabilities that affect roughly 37% of cloud environments running GPU workloads; independent benchmarks (arxiv 2501.04580) show Edera matches Kata on virtualized hosts and outperforms gVisor across metrics, closing a real gap in Kubernetes GPU isolation that MIG alone does not cover. Memory hygiene — zeroing GPU memory between sessions — is not default in most Kubernetes GPU operators and must be enabled explicitly. For regulated workloads, DoD IL5 and FedRAMP High now require physical tenant separation on top of logical isolation; Oracle OCI Generative AI (IL5 + FedRAMP High, March 2026) and Azure Government's H100 confidential VM offering are the primary procurement paths. NIST SP 800-207 Zero Trust tenets apply directly: treat every tenant, adapter, and inference request as an untrusted resource subject to per-session authorization. **[2026-04]** Standards work is actively maturing: NIST CAISI (Center for AI Standards and Innovation) launched the AI Agent Standards Initiative on February 17, 2026 with April 2026 sector workshops specifically addressing multi-tenant agentic deployment barriers in healthcare, finance, and education; NIST COSAiS (Control Overlays for Securing AI Systems, built on SP 800-53 plus SP 800-218A, Draft NIST AI 800-1, and NIST AI 100-2e2025) released its first discussion-draft annotated outline for Predictive AI on January 8, 2026, with single-agent and multi-agent overlays -- the documents most directly relevant to cooperative multi-tenant agent systems -- expected in discussion-draft form through mid-to-late 2026, and interim compensating-control guidance available for agencies that cannot wait. Operators running multi-tenant agent platforms today should map their existing controls to the NIST AI RMF plus SP 800-53 AC/SC families now, so that the COSAiS overlay crosswalk is a gap analysis rather than a fresh compliance build. No production framework yet combines (a) adapter signing + attestation, (b) GPU TEE-I/O, (c) DPU-enforced network isolation, and (d) side-channel-resistant kernels — the best current posture is layered defenses: MIG + confidential VM + signed adapters + Kata runtime + network policy, weighed against the 20-40% cost premium and operational complexity vs. plain multi-LoRA serving. |

## Notable Incidents and Research Signals

- **PROMPTPEEK, NDSS 2025:** The first focused public analysis of prompt leakage through shared KV-cache reuse showed that timing differences in multi-tenant vLLM/SGLang-style serving can let an attacker reconstruct another user's prompt under realistic prior-knowledge assumptions. This remains the core test case for 5.6.1 because it turns a performance optimization into a confidentiality boundary. See [PROMPTPEEK](https://www.ndss-symposium.org/ndss-paper/i-know-what-you-asked-prompt-leakage-via-kv-cache-sharing-in-multi-tenant-llm-serving/).
- **vLLM prompt embedding deserialization, November 2025:** GHSA-mrw7-hf4f-83pf / CVE-2025-62164 shows how a tenant-facing model-server feature can become a shared-host compromise path: user-supplied `prompt_embeds` were loaded with `torch.load()` and could trigger memory corruption through sparse-tensor handling. Even when direct cache isolation is sound, 5.6.2 evidence has to include admission controls for tenant-controlled embeddings, model files, media URLs, and adapter artifacts. See [vLLM advisory GHSA-mrw7-hf4f-83pf](https://github.com/vllm-project/vllm/security/advisories/GHSA-mrw7-hf4f-83pf).
- **Agent/tool supply-chain expansion, March-May 2026:** ATLAS v5.5.0 and v5.6.0 added agent tool poisoning, AI service proxy, cost-harvesting, public AI artifact acquisition, and code-repository discovery techniques. These are not GPU attacks, but they matter for multi-tenant isolation because a poisoned tenant-scoped agent tool or proxy can bridge the same boundaries that model-serving controls try to preserve.

## Implementation Maturity

- **5.6.1 KV-cache isolation:** Improving. vLLM exposes `cache_salt`, SHA256 is the default prefix-cache hash as of v0.11, and prompt-structure mitigations are practical today. Selective isolation designs such as CacheSolidarity are promising, but should be treated as research unless the serving provider can show implementation and test evidence.
- **5.6.2 shared model-serving isolation:** Mixed. Kubernetes namespaces, network policy, reverse proxies, signed artifacts, and patch management are mature controls, but runtime LoRA loading, adapter attestation, cross-tenant batching kernels, and GPU/TEE attestation are still uneven across open-source serving stacks. The strongest current posture layers tenant-bound adapters, disabled remote runtime loading, isolated inter-node networks, hardened container or VM runtimes, and explicit tests for unprotected control endpoints.
