# C3.3: Controlled Deployment & Rollback

> [Back to C03 Index](C03-Model-Lifecycle-Management.md)
> **Last Researched:** 2026-06-20

## Purpose

Model deployments must be controlled, monitored, reversible, and isolated across rollout cohorts. This section covers gradual rollout mechanisms with automated rollback, atomic state restoration, and AI-specific runtime-state isolation for parallel model versions.

The GPT-4o sycophancy incident ([OpenAI, April-May 2025](https://openai.com/index/expanding-on-sycophancy/)) is a cautionary example: a system prompt change reached full ChatGPT rollout, behavior evaluations missed the regression, and the full rollback took about 24 hours because stability had to be managed during reversal. The [Model Namespace Reuse attack](https://unit42.paloaltonetworks.com/model-namespace-reuse/) (Unit 42, 2025) demonstrated that major cloud platforms deployed Hugging Face models by name without integrity verification, enabling remote code execution. More recently, CVE-2025-62164 (CVSS 8.8) showed that vLLM's Completions API accepted malicious serialized PyTorch tensors via the `prompt_embeds` parameter, enabling memory corruption and potential RCE on shared inference servers — a reminder that deployment-time integrity controls must extend beyond model weights to the serving layer itself.

The vLLM CVE trail kept getting longer through the spring. On the tooling front, the OpenSSF Model Signing project reached v1.0 (April 2025), providing production-ready cryptographic signing and verification for ML models via Sigstore. The companion Sigstore Model Validation Operator (v1.0.1, alpha) brings this into Kubernetes as an admission-time gate — a mutating webhook injects an init container that verifies model signatures before the workload pod starts, blocking tampered or unsigned artifacts automatically. These developments are closing the gap between "verify at build time" and "verify at deploy time." CVE-2026-22778 (CVSS 9.8, disclosed February 2026) demonstrated a chained RCE in vLLM's multimodal video processing: an attacker sends a crafted video URL to `/v1/chat/completions`, first leaking heap addresses via PIL error messages to defeat ASLR, then triggering a JPEG2000 heap overflow in OpenCV's bundled FFmpeg to overwrite function pointers and achieve arbitrary command execution — all without authentication. The vulnerability affects vLLM 0.8.3 through 0.14.0 and was fixed in 0.14.1. Weeks later, CVE-2026-27893 (CVSS 8.8, disclosed March 26-27, 2026) showed the opposite failure mode — a hardcoded `trust_remote_code=True` in `nemotron_vl.py` and `kimi_k25.py` silently overrode the operator's explicit `--trust-remote-code=False` security flag, so pulling a malicious model repo triggered RCE even when the deployment was explicitly configured to reject it. That affects vLLM 0.10.1 through 0.17.x and is fixed in 0.18.0. April 2026 then added a fresh batch: CVE-2026-44223 / GHSA-83vm-p52w-f9pw (CVSS 6.5, disclosed April 28, 2026) is a single-request server crash in `extract_hidden_states` speculative decoding — any API call carrying `repetition_penalty`, `frequency_penalty`, or `presence_penalty` hits a tensor-shape mismatch and kills the EngineCore process; it affects vLLM 0.18.0–0.19.1 and is fixed in 0.20.0. CVE-2026-34753 / GHSA-pf3h-qjgv-vcpr (CVSS 5.4, disclosed April 3, 2026) is a parallel SSRF in the batch runner's `download_bytes_from_url` that performs no hostname, IP, port, or redirect validation, letting batch-input JSON pivot to cloud metadata endpoints from inside the serving cohort (fixed in 0.19.0). Two same-day siblings, CVE-2026-34755 / GHSA-pq5c-rjhq-qp7p (unbounded frame count in video/jpeg base64 processing) and CVE-2026-34756 / GHSA-3mwp-wvh9-7528 (OOM via unbounded `n` parameter in the OpenAI-compatible API), round out a quarter where vLLM's request surface kept finding new ways to take a serving pod down. Taken together, the 2026 vLLM CVE run reinforces that deployment integrity controls must cover the full serving stack, not just model weights — that "security flag set" is not the same as "security flag honored," and that a canary fielded on a vulnerable serving version is its own incident waiting to happen.

Progressive delivery tooling for model serving has matured considerably: KServe (promoted to CNCF incubating November 11, 2025) provides native canary rollout via the `canaryTrafficPercent` field on InferenceService resources, with automatic last-good-revision tracking — if a canary fails, traffic reverts to the previous stable revision without manual intervention. One operational gotcha worth flagging: KServe's canary rollout only works in serverless mode (Knative-backed), so teams running raw-deployment mode need an external progressive delivery controller. Seldon Core v2 introduced its Experiments API to support canary and A/B routing across both models and pipelines with integrated Prometheus observability. Meanwhile, the "Colluding LoRA" attack (arXiv 2603.12681, 2026) demonstrated that composed LoRA adapters can bypass safety alignment when merged — refusal behavior collapses without requiring adversarial prompts — which means rollback atomicity must extend to adapter composition state, not just individual adapter versions. Stanford CodeX's AILCCP analysis (March 2026) found that models sabotaged shutdown mechanisms in 79 out of 100 tests, concluding that kill switches must live entirely outside the agent reasoning path. OWASP's Agentic Top 10 now classifies cascading failures as ASI08, recognizing that a single fault in one agent can propagate through agent-to-agent communication, shared memory, and feedback loops faster than human operators can intervene — one documented financial scenario saw a hallucination cascade through four verification agents in 6 seconds.

## Current Research Notes

As of May 2, 2026, the most important change for 3.3.1 is that progressive delivery evidence should be tied to model-serving control planes, not just generic HTTP services. KServe's current rollout documentation still marks canary rollout as serverless-only, tracks the last 100% rolled-out revision, and pins rollback traffic to the previous healthy revision when needed; auditors should confirm that teams using raw deployment mode have an external controller such as Argo Rollouts or Flagger rather than assuming KServe canary behavior exists everywhere. Argo Rollouts remains useful for behavior-driven rollback because `AnalysisTemplate` metrics can query Prometheus and abort a rollout after repeated failed measurements, setting canary traffic back to zero: https://kserve.github.io/website/docs/model-serving/predictive-inference/rollout-strategies/canary and https://argo-rollouts.readthedocs.io/en/stable/features/analysis/

Serving-runtime security advisories also need to be treated as rollback triggers. NVIDIA's April 2026 Triton Inference Server bulletin fixed five issues in r26.02, including four high-severity denial-of-service conditions from malformed requests or excessive outputs and one medium-severity information disclosure issue through model configuration upload. For 3.3.1 and 3.3.3, a canary should fail closed when a serving runtime is below the patched baseline, when crash-loop rates rise under malformed request testing, or when shared inference nodes expose model configuration or cache state across cohorts: https://nvidia.custhelp.com/app/answers/detail/a_id/5816

For 3.3.2, the best current tooling pattern is bundle-wide signing plus admission-time verification. OpenSSF Model Signing defines a detached OMS signature format that can cover multi-file model bundles, while Sigstore's Model Validation Operator injects an init container through a mutating webhook and blocks workload startup when model signature verification fails. That still does not prove an adapter composition is safe, so rollback manifests should pin the base model, adapters, tokenizers, safety models, prompts, retrieval index version, and policy configuration as one tested unit: https://openssf.org/blog/2025/06/25/an-introduction-to-the-openssf-model-signing-oms-specification/ and https://blog.sigstore.dev/model-validation-operator-v1.0.1/

OWASP's Agentic Top 10 launch framed ASI08 cascading failures as false signals moving through automated pipelines with escalating impact. For this page, that maps directly to rollback and cohort-isolation design: fan-out limits, fail-closed policy decisions, isolated runtime state, short-lived scoped credentials, and audited propagation boundaries need to be tested together rather than one subsystem at a time: https://genai.owasp.org/2025/12/09/owasp-top-10-for-agentic-applications-the-benchmark-for-agentic-security-in-the-age-of-autonomous-ai/

As of May 13, 2026, cohort-routing evidence should include state isolation proof, not just traffic percentages. Seldon Core 2 Experiments can split traffic between models or pipelines, mirror a candidate, and return an `x-seldon-route` sticky-session header, but its documentation warns that sticky sessions do not necessarily route stateful models to the same replica. For 3.3.3, that means route-stickiness screenshots are not enough; auditors should see separate KV-cache namespaces, Redis/session keys, prompt-cache salts, retrieval-index partitions, and service-account scopes for each rollout cohort: https://docs.seldon.ai/seldon-core-2/user-guide/experiment

The official vLLM advisories now make several concrete rollback gates easy to test. GHSA-4r2x-xpjr-7cvv / CVE-2026-22778 affects vLLM 0.8.3 through versions before 0.14.1 when video-capable multimodal endpoints process malicious remote video URLs, and GHSA-7972-pg2x-xr59 / CVE-2026-27893 affects vLLM 0.10.1 through versions before 0.18.0 when hardcoded `trust_remote_code=True` bypasses an operator's explicit opt-out. The April 2026 batch adds three more pre-promotion gates worth wiring into a canary `AnalysisRun`: GHSA-83vm-p52w-f9pw / CVE-2026-44223 (vLLM 0.18.0–0.19.1, fixed in 0.20.0) crashes the server on any request carrying a penalty parameter, so a synthetic probe with `repetition_penalty=1.1` against the canary will trip the crash-loop detector in seconds; GHSA-pf3h-qjgv-vcpr / CVE-2026-34753 (≥0.16.0, fixed in 0.19.0) is an SSRF in the batch runner's `download_bytes_from_url` that ignores hostname, IP, port, and redirect validation, so a canary running the batch endpoints must be on a network egress policy that blocks cloud metadata addresses; GHSA-3mwp-wvh9-7528 / CVE-2026-34756 lets a single OpenAI-API call with an unbounded `n` parameter OOM the pod, which is a clean DoS probe to keep in the smoke-test set. For 3.3.1, an advisory or SBOM check should halt or roll back a canary before traffic promotion; for 3.3.3, a vulnerable canary must not share nodes, credentials, model-cache volumes, or egress paths with the stable cohort: https://github.com/vllm-project/vllm/security/advisories/GHSA-4r2x-xpjr-7cvv, https://github.com/vllm-project/vllm/security/advisories/GHSA-7972-pg2x-xr59, https://github.com/vllm-project/vllm/security/advisories/GHSA-83vm-p52w-f9pw, https://github.com/vllm-project/vllm/security/advisories/GHSA-pf3h-qjgv-vcpr and https://github.com/vllm-project/vllm/security/advisories/GHSA-3mwp-wvh9-7528

As of May 19, 2026, three more deployment-tooling shifts are worth pulling into 3.3.1 and 3.3.3 evidence. Argo Rollouts v1.9.0 (released March 20, 2026) fixed a blue-green analysis bug where new ReplicaSets reported premature success when they became unsaturated, and tightened Istio DestinationRule weight management so traffic actually returns to the previous stable weight during rollback rather than getting stuck mid-shift — both are direct rollback-correctness improvements that auditors should confirm are picked up before relying on rollback evidence from older releases. The same release line ships a `Metric AI` plugin in Argo Project Labs that lets an `AnalysisTemplate` call out to an AI agent which fetches stable and canary pod logs, runs `kubectl` queries, and votes on rollback — useful as a defense-in-depth analysis source, but the agent itself becomes part of the rollback control plane and needs the same scoped-credential, fail-closed treatment as any other automated decision-maker: https://github.com/argoproj/argo-rollouts/releases. KServe issue #5335 (opened April 2, 2026, still open as of May 19) confirms that `canaryTrafficPercent` is silently ignored in raw-deployment mode — a `RawHTTPRouteReconciler` extension is proposed to honor weighted backendRefs through Gateway API, but until that lands, teams running KServe outside Knative still need Argo Rollouts or Flagger as the actual progressive-delivery controller and should not rely on the InferenceService field at all: https://github.com/kserve/kserve/issues/5335. For 3.3.3, the CacheSolidarity research (arXiv 2603.10726, March 11, 2026) proposes a system-level defense for prefix-cache side channels in multi-tenant LLM serving: continuously monitor prefix reuse across users, flag prefixes that look suspiciously cross-tenant, and refuse further reuse — the authors report up to 70% higher cache reuse and 30% lower latency than the naive per-user isolation defenses, and the technique is complementary to vLLM's per-request `cache_salt`. Treat it as the current state of the art for keeping canary and stable cohorts from leaking through cache timing while preserving most of the throughput win: https://arxiv.org/abs/2603.10726. NVIDIA's May 2026 Triton bulletin (r26.03) is also out and should be folded into the same patched-baseline policy that already gates the April r26.02 bulletin: https://nvidia.custhelp.com/app/answers/detail/a_id/5828

As of June 10, 2026, the patched-baseline policy that already gates vLLM and Triton needs to widen to every serving runtime in the fleet. SGLang CVE-2026-5760 (CVSS 9.8, disclosed April 20, 2026) renders a GGUF model's `tokenizer.chat_template` through an unsandboxed `jinja2.Environment()` when the `/v1/rerank` endpoint is used, so loading a malicious model is arbitrary Python execution — no patch existed at disclosure, and the practical mitigation is switching template rendering to Jinja2's `ImmutableSandboxedEnvironment`. Ollama's "Bleeding Llama" (CVE-2026-7482, published May 5, 2026, fixed in 0.17.1) is a heap out-of-bounds read in the GGUF model loader that leaked other users' prompts, environment secrets, and API keys from process memory, with roughly 300,000 internet-exposed servers in scope; the same week, CVE-2026-42248 and CVE-2026-42249 in Ollama's Windows auto-updater allowed a persistent attacker-controlled executable to be planted through the update channel itself. For 3.3.2 that means deployment gates have to treat model-file metadata (GGUF headers, chat templates) as executable code and verify it the same way they verify weights — and the rollback/update channel itself is part of the integrity boundary. For 3.3.3, Bleeding Llama is the clearest live demonstration yet of cross-request state leakage inside a single serving process: https://thehackernews.com/2026/04/sglang-cve-2026-5760-cvss-98-enables.html, https://www.securityweek.com/critical-bug-could-expose-300000-ollama-deployments-to-information-theft/ and https://www.helpnetsecurity.com/2026/05/05/ollama-windows-vulnerabilities-cve-2026-42248-cve-2026-42249/

Adapter integrity gates also got more practical this spring. "Token-Level Generalization in LoRA Adapter Backdoors" (arXiv 2605.30189, May 28, 2026) showed a LoRA adapter backdoored through training-data poisoning that preserves clean-task accuracy while the trigger generalizes at the token-feature level — and, usefully, demonstrated both behavioral-probe and weight-level detection with clean separation between poisoned and clean adapters. Combined with earlier weight-space detection work (arXiv 2602.15195, February 2026), pre-promotion adapter scanning is now feasible and belongs in the same gate that verifies bundle signatures for 3.3.2: https://arxiv.org/abs/2605.30189 and https://arxiv.org/abs/2602.15195

On the tooling side, vLLM v0.21/v0.22 (May–early June 2026) shipped sleep-mode weight reload — in-place model weight swaps without a pod restart — plus a multi-tier KV-cache offloading framework with a filesystem tier. In-place reload makes rollback faster, but it bypasses admission-time controls like the Sigstore Model Validation Operator's init-container signature check, so for 3.3.2 auditors should confirm signature verification and rollback-manifest pinning run on the reload path, not only at pod admission; for 3.3.3 the new offload tier is another shared resource cohort-isolation policy must cover: https://github.com/vllm-project/vllm/releases. Flagger v1.43.0 (April 21, 2026) added a Kubernetes External Metrics provider — letting canary analysis consume external or AI-behavioral metrics as rollback triggers without a custom webhook — a `primarySessionAffinityCookie` field that pins users to one cohort across the canary window, and a fix for CanaryWeight resetting to zero during promotion: https://github.com/fluxcd/flagger/blob/main/CHANGELOG.md. On KServe, note the nuance between two raw-deployment stories: v0.16.0 (November 2025) did ship progressive-rollout support for raw deployments (PR #4623) and v0.18 (April 2026) added namespace-scoped ModelCache, but issue #5335 — `canaryTrafficPercent` silently ignored in raw mode — remains open as of June 2026, so the audit question is which mechanism actually mediates the traffic split: https://github.com/kserve/kserve/releases/tag/v0.16.0. LMCache v0.4.4–v0.4.6 (April–May 2026) added per-request cache-hit-rate metrics, which make a decent cross-cohort cache-leak canary signal even though cohort isolation is still not a first-class LMCache feature: https://github.com/LMCache/LMCache/releases

Standards and incident context worth folding into evidence packets: MITRE CTID's Secure AI v2 release (May 6, 2026) pushed 45+ techniques and 20+ case studies into ATLAS, moved ATLAS to a monthly release cadence, and introduced Rapid Response Reports for AI security incidents — a natural upstream feed for advisory-driven rollback triggers: https://ctid.mitre.org/blog/2026/05/06/secure-ai-v2-release. NIST has said its Cyber AI Framework Profile is due summer 2026 with COSAiS control overlays following (predictive-AI draft summer, agentic late summer/fall): https://csrc.nist.gov/projects/cosais. The EU Commission's GPAI enforcement powers — including the power to order model recalls — take effect August 2, 2026, which makes a regulator-forced rollback a scenario worth drilling: https://artificialintelligenceact.eu/implementation-timeline/. Two May 2026 incidents sharpen the evidence bar: the Gemini coding-assistant incident (May 21, 2026), where the model deleted roughly 28,700 lines across 340 files, caused a 33-minute production outage, and then produced a fictitious post-mortem — recovery evidence must come from systems of record, never from the model being rolled back: https://www.theregister.com/ai-ml/2026/05/21/gemini-accused-of-30000-line-code-purge-and-fake-recovery-report/5244219 — and Azure OpenAI's May 29, 2026 multi-region degradation (tracking ID LYXT-C1Z; a retry storm without rate limiting plus backend memory exhaustion), which was contained by failing over to an isolated backend cluster rather than a version rollback — cohort/cluster isolation working as the containment control 3.3.3 intends: https://azure.status.microsoft/en-us/status/history/. PROMPTPEEK (NDSS 2025) remains the strongest published evidence that shared KV caches are an information channel — up to 99% prompt reconstruction via cache timing in SGLang-style serving: https://www.ndss-symposium.org/wp-content/uploads/2025-1772-paper.pdf

Human oversight and deactivation evidence belongs in the rollback packet, not only in a governance policy. EU AI Act Article 14 requires high-risk systems to let assigned people monitor, override, reverse, intervene, or interrupt operation into a safe state, while NIST AI RMF Manage 2.3 and Manage 4.1 call for response, recovery, deactivation, decommissioning, monitoring, appeal/override, incident response, and change-management procedures. For this control, the strongest evidence is a rollback drill where an authorized person can stop promotion, force traffic back to the last-good revision, and put the system into a safe state without relying on the model or agent being rolled back: https://ai-act-service-desk.ec.europa.eu/en/ai-act/article-14 and https://airc.nist.gov/airmf-resources/playbook/manage/

As of June 20, 2026, vLLM published a fresh cluster of advisories (June 16–17) that map cleanly onto all three requirements and sharpen the point that "patched-baseline" is a continuously moving gate. For 3.3.1, two are clean pre-promotion canary probes: CVE-2026-54235 / GHSA-7h4p-rffg-7823 (CVSS 6.5) lets `temperature=NaN` or `+Infinity` slip past every comparison-based validation gate and reach the GPU sampling kernels, crashing the inference worker; CVE-2026-54233 / GHSA-6pr9-rp53-2pmc (CVSS 6.5, affects all versions up to 0.23.0, fixed in 0.23.1rc0) is an audio decompression bomb where a 25MB OPUS upload expands to roughly 14.9GB of float32 PCM on `/v1/audio/transcriptions`, OOM-killing the pod. Both are single-request kills, so a canary on a vulnerable image vanishes on the first synthetic probe and the rollout system must score the disappearance as a failed canary, not a vacuous green. For 3.3.2 and the integrity boundary, CVE-2026-41523 / GHSA-q8gq-377p-jq3r (CVSS 7.5, fixed 0.22.0) is the sharpest reminder yet that the same serving image can be safe or exploitable depending on how it is launched: an assert-based security check in activation-function loading is stripped when vLLM runs under `python -O` or `PYTHONOPTIMIZE=1`, so a malicious Hugging Face model achieves unauthenticated RCE precisely on the performance-optimized images teams favor in production — a deployment gate has to pin the interpreter flags, not just the version string. CVE-2026-48746 / GHSA-94f4-hr76-p5j6 (CVSS 9.1, affects 0.3.0–0.21.x, fixed 0.22.0) is an OpenAI-API authentication bypass through ASGI/Starlette request-trust handling that lets callers skip the configured `--api-key` entirely, so for 3.3.3 a vulnerable canary sharing an ingress or trust boundary with stable becomes an unauthenticated entry point into the serving plane. The strongest new cross-cohort leak is CVE-2026-53923 / GHSA-5jv2-g5wq-cmr4 (CVSS 5.4, affects 0.5.5–0.23.0, no fix at disclosure): integer truncation in the GGUF dequantize kernels allocates a full output buffer via `torch::empty` while the CUDA kernel fills only a truncated element count, so the uninitialized tail of the tensor leaks residual GPU memory — tensor data from other tenants' inference requests — across cohort boundaries on shared GPUs. With no patch to roll forward to, per-cohort GPU pools and network segmentation are the only mitigations, which is the clearest argument yet for not co-scheduling canary and stable on the same device: https://advisories.gitlab.com/pypi/vllm/CVE-2026-54235/, https://advisories.gitlab.com/pypi/vllm/CVE-2026-54233/, https://advisories.gitlab.com/pypi/vllm/CVE-2026-41523/, https://advisories.gitlab.com/pypi/vllm/CVE-2026-48746/ and https://advisories.gitlab.com/pypi/vllm/CVE-2026-53923/

Two June 2026 incidents reinforce that the slowest part of a rollback is usually the shared layer, not the model. Microsoft 365 Copilot suffered a roughly seven-hour global outage on June 11, 2026 when a faulty deployment broke the federated authentication layer connecting Copilot's front end to Microsoft Graph and Azure OpenAI: the change began rejecting valid credentials and triggered retry storms that overloaded the remaining healthy nodes. Microsoft's monitoring flagged it within minutes, but recovery still took until 18:00 UTC because the rollback had to propagate across all global Azure regions and wait for cached authentication tokens to expire — a textbook case of rollback latency being dominated by a shared, stateful dependency rather than by swapping a model version, and the second such Copilot disruption in eleven days after the June 1 routing incident. For 3.3.1 and 3.3.2, the lesson is that rollback-time SLAs must be measured against the slowest shared component in the dependency graph (auth, routing, token caches), and that "caught in minutes" and "recovered in minutes" are very different numbers worth drilling separately: https://www.techtimes.com/articles/318290/20260612/microsoft-copilot-fails-twice-june-enterprise-it-has-no-sla-protection-ai-downtime.htm

For this page, the highest-yield verifier checks are:
- Trigger a failed Argo Rollouts `AnalysisRun` or KServe canary health condition and confirm traffic returns to the previous healthy revision; for KServe, additionally confirm the cluster is in serverless mode or has Argo Rollouts/Flagger layered in front, since RawDeployment silently ignores `canaryTrafficPercent` (KServe #5335, open as of May 2026).
- Block deployment of an unsigned or tampered model bundle through OMS/Sigstore verification before the serving pod starts.
- Prove the rollback manifest pins base model, adapters, tokenizer, prompt templates, safety/policy models, retrieval index, and runtime image as one tested unit.
- Run a vLLM or Triton patched-baseline policy check and confirm vulnerable serving images cannot receive canary traffic; include the April 2026 vLLM advisories (penalty-parameter crash, batch-runner SSRF, unbounded-frame DoS, unbounded-`n` OOM), the June 2026 vLLM advisories (temperature NaN/Infinity worker crash, audio decompression-bomb OOM, assert-bypass RCE under `python -O`, OpenAI-API auth bypass, GGUF dequantize GPU-memory disclosure), and the May 2026 Triton r26.03 bulletin in the policy.
- Add `temperature=NaN`/`temperature=Infinity` and an audio decompression-bomb upload against `/v1/audio/transcriptions` to the synthetic canary probe set; both are single-request worker kills (CVE-2026-54235, CVE-2026-54233), so confirm the rollout system scores a disappeared canary as a failed canary rather than promoting on a vacuous green signal.
- Confirm the deployment gate pins the serving image's interpreter launch flags, not just its version: an image running under `python -O` or `PYTHONOPTIMIZE=1` strips assert-based security checks and is exploitable for unauthenticated RCE from a malicious model even on a "patched" version (CVE-2026-41523, fixed 0.22.0).
- Exercise a canary and stable cohort with identical prompts and confirm KV-cache hits, prompt-cache hits, Redis keys, and vector queries cannot cross cohorts; add an egress probe to confirm cohort network policy blocks cloud metadata addresses even when the application is tricked into requesting them.
- Test a malicious Hugging Face namespace or unpinned model revision and confirm the deployment gate rejects name-only model references.
- Run a human rollback drill and retain evidence that the operator path can stop promotion and reach a safe state without model cooperation.
- Extend the patched-baseline gate to every serving runtime in the fleet — SGLang and Ollama, not just vLLM and Triton — and confirm GGUF metadata and chat templates are sandboxed or scanned before a model loads (CVE-2026-5760 makes a model's chat template arbitrary code).
- If in-place weight reload (vLLM sleep mode) is enabled, confirm signature verification and rollback-manifest pinning run on the reload path, not only at pod admission time.
- Scan LoRA adapters with behavioral probes and weight-space backdoor detection as a pre-promotion gate alongside signature checks (arXiv 2605.30189 / 2602.15195).
- Confirm post-rollback recovery evidence is generated from systems of record (deployment logs, git history, registry manifests), never from the model or agent that was rolled back.

The v1.9.0 release explicitly fixed a blue-green analysis bug where new ReplicaSets reported premature success when they became unsaturated, and tightened Istio DestinationRule weight management during rollbacks — both are direct rollback-correctness improvements, and teams pinned to older releases should plan an upgrade before quoting Argo Rollouts as their rollback control. The v1.8.0 line (released alongside Kubernetes 1.33 and Prometheus 3.1) added native histogram support and shared informer cache optimizations that cut controller memory by ~30%, enabling two-second canary aborts with sub-second metric lag. Argo Project Labs also ships a `Metric AI` plugin that lets an `AnalysisTemplate` delegate the pass/fail decision to an AI agent — useful as defense-in-depth analysis, but the agent itself joins the rollback control plane and needs scoped credentials and fail-closed behavior. Flagger offers automatic metric-driven canary promotion with zero manifest changes (it manages standard Deployments rather than requiring migration to a Rollout CRD) — its key differentiator is full lifecycle automation once configured, while Argo Rollouts provides more explicit control and a dedicated UI plugin. **ML-native**: KServe (promoted to CNCF incubating November 11, 2025, v0.15) provides first-class canary via the `canaryTrafficPercent` field on InferenceService — it automatically tracks the last good revision and rolls traffic back on failure without requiring external controllers. Important constraint that auditors keep catching: KServe canary is only supported in serverless (Knative-backed) deployment mode. KServe issue #5335 (April 2026, still open as of May 19, 2026) confirms that in raw-deployment mode the `canaryTrafficPercent` field is silently ignored — users get a full rollout while believing they're getting a weighted split. A `RawHTTPRouteReconciler` Gateway API extension has been proposed but is not yet merged, so teams running raw deployment still need to layer Argo Rollouts or Flagger on top and should not rely on the InferenceService field at all. Seldon Core v2's Experiments API routes traffic across model versions and full inference pipelines with integrated Prometheus metrics, supporting A/B and canary patterns at the pipeline level. **Configuration-layer**: LaunchDarkly AI Configs separates model configuration from application code, enabling runtime rollback without redeployment. The Sigstore Model Validation Operator (v1.0.1, alpha) adds a pre-deployment integrity gate via Kubernetes mutating webhook. Recommended metric thresholds for automatic rollback: error rate >1% (or >0.5% above baseline), P99 latency >200ms or >20% degradation, accuracy drop >1% from baseline, with minimum 3 consecutive evaluation failures before triggering. AI-specific behavioral rollback triggers (guardrail violations, output drift, bias indicators) typically require custom metric pipelines beyond standard HTTP error rates. Best practice is a stepwise ramp: 1% traffic → monitor for 1 hour → 5% → 20%, with each step gated on behavioral metrics including safety-specific evaluations rather than just engagement proxies. An important caveat: unlike traditional software rollbacks, ML model rollbacks face a data drift problem — by the time a rollback occurs, the data distribution the old model was trained on may have shifted, meaning the reverted model may perform differently than expected. Standard DevOps health signals (latency, error rates) are structurally unable to detect this; organizations should pair rollback with ongoing data drift monitoring (e.g., Evidently AI, NannyML, Arize Phoenix). June 2026 additions: Flagger v1.43.0 (April 21, 2026) added a Kubernetes External Metrics provider, which lets canary analysis consume external or AI-behavioral metrics as rollback triggers without writing a custom webhook, plus a `primarySessionAffinityCookie` field that keeps each user pinned to one cohort for the whole canary window, and fixed a bug where CanaryWeight reset to zero during promotion. On KServe, distinguish two raw-deployment stories: v0.16.0 (November 2025) shipped progressive-rollout support for raw deployments (PR #4623), but the `canaryTrafficPercent` field remains silently ignored in raw mode (issue #5335, still open as of June 2026) — ask which mechanism actually mediates the traffic split rather than accepting the field's presence as evidence. MITRE CTID's Secure AI v2 release (May 6, 2026) introduced ATLAS Rapid Response Reports for AI security incidents, a useful upstream feed for advisory-driven rollback triggers. And a base rate worth quoting when someone argues rollback automation is over-engineering: Sinch's 2026 research found 74% of organizations have shut down or rolled back a live AI customer-communications agent after launch — rollback is the expected path, not the exception. A June 2026 data point on rollback latency: the Microsoft 365 Copilot outage of June 11, 2026 (a faulty deployment that broke the shared Microsoft Graph / Azure OpenAI auth layer, ~7 hours) was caught by monitoring within minutes but took until 18:00 UTC to recover because the rollback had to propagate across all Azure regions and wait for cached auth tokens to expire — rollback-time SLAs should be measured against the slowest shared dependency, not the model swap, and "caught in minutes" should be drilled separately from "recovered in minutes." *(Updated 2026-06-20)* |
| **3.3.2** | **Verify that** rollback capabilities restore the complete model state. | 2 | Partial rollback leaving inconsistent state (e.g., new weights with old safety model, or updated adapter with stale config); cascading failures from version mismatches between model components; serving degraded or unsafe outputs during an incomplete rollback window. | Execute a rollback and verify that all components (weights, config, adapters, safety/policy models, system prompts) revert to the previous known-good version simultaneously. Confirm atomicity — no intermediate state is served during rollback. Test with multi-artifact stacks (base model + LoRA adapter + safety model + retrieval index) to verify all components roll back together. Check that rollback is tested as part of deployment runbooks and that rollback time is measured. Inspect Vertex AI Model Registry or equivalent versioning for atomic version pinning. | Atomic rollback of multi-artifact model stacks is harder than single-artifact rollback — most serving platforms (vLLM, TensorRT-LLM, Triton) version model weights independently from configuration and adapters. The "Colluding LoRA" attack (arXiv 2603.12681, 2026) demonstrated that independently safe LoRA adapters can bypass safety alignment when composed — refusal behavior collapses entirely without adversarial prompts — meaning rollback must treat adapter composition state as a single atomic unit, not roll back individual adapters. Defenses like SafeLoRA, PEFTGuard, and LlamaGuard 3 can detect unsafe adapter states (SafeLoRA fusion shows a 42% reduction in harmfulness rate), but none yet integrate with rollback orchestration to automatically trigger reversion on unsafe composition detection. AWS SageMaker multi-adapter inference provides atomic operations for adding, deleting, or updating individual LoRA adapters across running instances without redeployment, though cross-adapter safety verification remains a manual step. Vertex AI Model Registry supports versioned model artifacts with rollback to previous versions, but adapter and safety model coordination requires custom orchestration. The OpenSSF Model Signing v1.0 project (April 2025) provides cryptographic signing and verification for ML models of any format via Sigstore, with Rekor transparency log entries enabling tamper-evident audit trails. System prompt changes should be versioned alongside model artifacts — the GPT-4o sycophancy incident (April 2025) demonstrated that prompt changes can cause behavioral regression as severe as weight changes. vLLM's CVE-2026-44223 (April 28, 2026) is a smaller-scale reminder of the same lesson at the serving layer: a refactor changed the return tensor shape inside `extract_hidden_states` speculative decoding from `(batch_size, 1)` to `(batch_size, 2)` after the first decode step, and any request with a penalty parameter crashes the server. Atomic rollback has to cover the serving runtime, not just the model — and a runtime version pinned in one place while the rest of the stack rolls forward is exactly the kind of split state that turns a regression into an outage. The NIST AI RMF "Manage" function explicitly calls for rollback and version control to "revert to stable configurations, isolate faulty updates, and prevent further propagation of errors," and GOVERN 1.7 requires documented processes for decommissioning AI systems "safely and in a manner that does not increase risks or decrease the organization's trustworthiness" — which in practice means the rollback runbook and the decommissioning runbook should share the same artifact-version manifest so a partial rollback does not end up in a state that was never stable or never tested. The spring 2026 advisory wave broadened the "complete model state" boundary in three directions. First, model metadata is code: SGLang CVE-2026-5760 (CVSS 9.8, disclosed April 20, 2026) renders a GGUF model's `tokenizer.chat_template` through an unsandboxed `jinja2.Environment()` on the `/v1/rerank` path, so loading a malicious model yields arbitrary Python execution — deployment gates must scan or sandbox template metadata, not just hash weights. Second, the update channel itself is in scope: Ollama's Windows auto-updater chain (CVE-2026-42248/CVE-2026-42249, May 5, 2026) let attackers plant a persistent executable through the update path. Third, adapters can now be scanned before promotion: arXiv 2605.30189 (May 28, 2026) demonstrated LoRA backdoors that preserve clean-task accuracy while generalizing at the token-feature level, and showed behavioral-probe and weight-level detection that cleanly separates poisoned from clean adapters (see also arXiv 2602.15195) — wire that scan into the same gate that verifies bundle signatures. Operationally, vLLM v0.21/v0.22 sleep-mode weight reload (May–June 2026) enables in-place weight swaps without pod restart — faster rollback, but it bypasses admission-time gates like the Sigstore Model Validation Operator init container, so signature verification must also run on the reload path. Two governance anchors: the EU Commission's GPAI enforcement powers including model recalls take effect August 2, 2026, making regulator-forced rollback a drillable scenario; and the May 21, 2026 Gemini coding-assistant incident (~28,700 lines deleted across 340 files, then a fictitious model-generated post-mortem) is the standing reminder that rollback evidence comes from systems of record, never from the model being rolled back. The June 2026 vLLM advisories widen the integrity boundary again: CVE-2026-41523 / GHSA-q8gq-377p-jq3r (CVSS 7.5, fixed 0.22.0) strips an assert-based security check when the serving process runs under `python -O` or `PYTHONOPTIMIZE=1`, yielding unauthenticated RCE from a malicious Hugging Face model — so the rolled-back/pinned state must include the serving image's interpreter launch flags, since the same version is safe or exploitable depending on how it is started — and CVE-2026-48746 / GHSA-94f4-hr76-p5j6 (OpenAI-API auth bypass via ASGI/Starlette trust handling, CVSS 9.1, affects 0.3.0–0.21.x, fixed 0.22.0) shows the serving runtime's own authentication surface is part of the state a "complete" rollback has to restore. *(Updated 2026-06-20)* |
| **3.3.3** | **Verify that** model versions running in parallel use isolated runtime state so that AI-specific shared resources are not shared across deployments. | 2 | Cross-contamination between deployment cohorts through shared KV caches, prompt caches, session state, or retrieval indices. If a canary deployment shares a KV cache with the stable version, cached context from one model version can leak into another, causing unpredictable behavior or masking behavioral regressions. Shared retrieval indices across A/B cohorts can cause cross-tenant data leakage in multi-tenant RAG systems. CVE-2025-47277 demonstrated that vLLM's distributed KV cache transfer (PyNcclPipe) bound to 0.0.0.0 instead of the specified interface, exposing cache contents across all network interfaces and enabling potential remote code execution on multi-node deployments. CVE-2026-34753 / GHSA-pf3h-qjgv-vcpr (April 2026) is the same shape of risk one layer up: vLLM's batch runner `download_bytes_from_url` accepted any user-controlled URL with no hostname, IP, port, or redirect validation, so a canary cohort running batch endpoints could reach cloud metadata addresses and pivot into the surrounding infrastructure plane — meaning egress segmentation, not just cache namespacing, is part of cohort isolation. The June 2026 vLLM GGUF dequantize bug (CVE-2026-53923) is the same hazard at the hardware layer — uninitialized GPU memory leaking other tenants' tensors across cohorts on a shared device. | Verify that each deployment cohort (canary, stable, shadow) runs with isolated KV cache instances — confirm no shared GPU memory pools or cache hash namespaces across cohorts. In vLLM deployments, check that `cache_salt` is configured per-cohort to prevent prefix cache reuse across deployment versions, and that LoRA IDs are included in cache hashing to prevent adapter cross-contamination. For RAG-backed systems, confirm that each cohort uses separate retrieval indices or tenant-scoped vector partitions with enforced metadata filters at query time. Inspect session state stores (Redis, database-backed) for cohort-level key isolation. Review whether prompt caches are scoped per model version. Test by querying both cohorts and inspecting cache hit rates to confirm no cross-cohort cache sharing occurs. Add an egress test: send a batch-runner request from each cohort with a URL pointing at a known internal metadata address and confirm the network policy rejects it (not just the application code), to cover the CVE-2026-34753 class of issue. | This is an emerging area with limited mature tooling. vLLM supports per-request `cache_salt` for prefix cache isolation and includes LoRA IDs and multi-modal input hashes in cache keys, but these features require explicit configuration — defaults do not enforce cross-cohort isolation. The CacheSolidarity paper (arXiv 2603.10726, March 2026) proposes a system-level alternative: monitor prefix reuse across users, flag prefixes that look suspiciously cross-tenant, and refuse further reuse on flagged prefixes — the authors report up to 70% higher cache reuse and 30% lower latency than naive per-user isolation defenses. Treat it as the current state of the art for keeping canary and stable cohorts from leaking through prefix-cache timing while preserving most of the throughput win; it is complementary to `cache_salt` rather than a replacement, and no production-grade reference implementation is shipped with vLLM yet. In distributed vLLM clusters, each pod manages its KV cache in isolation, which provides a natural boundary but introduces coordination overhead. LMCache provides an enterprise KV cache layer with namespace support but isolation between deployment cohorts is not a first-class feature. For retrieval indices, the silo model (separate index per cohort) provides strongest isolation but doubles storage costs; shared-index-with-metadata-filters is cheaper but relies on correct filter enforcement at every query. Cell-based deployment architectures — isolating cohorts into independent infrastructure cells — provide the strongest guarantees but are operationally expensive. The vLLM serving layer has accumulated multiple critical CVEs that compound isolation concerns: CVE-2025-47277 (distributed KV cache binding to 0.0.0.0, enabling cross-network cache exposure) and CVE-2025-62164 (CVSS 8.8, unsafe tensor deserialization via `prompt_embeds` parameter enabling memory corruption and RCE on shared inference servers — affects v0.10.2 through v0.11.0). Red Hat advises against using vLLM's experimental KV cache transfer in production. Organizations running multi-tenant vLLM should treat API-accessible deserialization endpoints as high-risk attack surface and apply network segmentation between cohorts as defense-in-depth. Adding to the vLLM vulnerability landscape, CVE-2026-22778 (CVSS 9.8, February 2026) enables unauthenticated RCE via a crafted video URL sent to multimodal endpoints — a chained exploit leaking heap addresses via PIL error messages to defeat ASLR, then triggering a JPEG2000 heap overflow in OpenCV's FFmpeg to achieve arbitrary command execution (affects vLLM 0.8.3–0.14.0, fixed in 0.14.1). CVE-2026-27893 (CVSS 8.8, March 26-27, 2026) is even more insidious for multi-cohort deployments: two model implementation files (`vllm/model_executor/models/nemotron_vl.py:430` and `kimi_k25.py:177`) hardcode `trust_remote_code=True` when loading sub-components, silently overriding the operator's `--trust-remote-code=False` flag. That means a canary cohort pinned to a malicious HuggingFace repo gets RCE even when the deployment explicitly tried to refuse remote code — and the attacker can pivot into the shared infrastructure plane from there. It affects 0.10.1 through 0.17.x and is fixed in 0.18.0. The April 2026 batch (CVE-2026-44223 penalty-parameter crash, CVE-2026-34753 batch-runner SSRF, CVE-2026-34755 unbounded-frame DoS, CVE-2026-34756 unbounded-`n` OOM) extends the same pattern across the request surface, which is why cohort isolation now has to cover not just cache state but also egress, file-descriptor budget, and GPU memory budget so a single noisy cohort cannot starve its neighbors. As of May 2026, no major serving framework enforces cross-cohort state isolation by default. June 2026 additions: Ollama's "Bleeding Llama" (CVE-2026-7482, published May 5, 2026, fixed in 0.17.1) is the clearest cross-tenant state-leak CVE yet at the serving layer — a heap out-of-bounds read in the GGUF model loader leaked other users' prompts, environment secrets, and API keys from process memory, with roughly 300,000 internet-exposed servers in scope. PROMPTPEEK (NDSS 2025) showed up to 99% prompt reconstruction via shared KV-cache timing in SGLang-style serving, the strongest published evidence that a shared cache across cohorts is an information channel, not just a correctness hazard. LMCache v0.4.4–v0.4.6 (April–May 2026) added per-request cache-hit-rate metrics — a usable cross-cohort cache-leak canary signal — though cohort isolation is still not a first-class LMCache feature, and vLLM v0.22's multi-tier KV-cache offloading adds a filesystem tier that cohort-isolation policies must now also cover. An operational data point in favor of cell-based isolation: Azure OpenAI's May 29, 2026 multi-region degradation (tracking ID LYXT-C1Z; an unthrottled retry storm plus backend memory exhaustion) was contained by failing over to an isolated backend cluster rather than by version rollback — cohort isolation working as the containment control this requirement intends. June 2026 sharpened the shared-GPU-memory hazard directly: vLLM CVE-2026-53923 / GHSA-5jv2-g5wq-cmr4 (CVSS 5.4, affects 0.5.5–0.23.0, no fix at disclosure) is an integer-truncation bug in the GGUF dequantize kernels that allocates a full output buffer via `torch::empty` while the CUDA kernel fills only a truncated element count, so the uninitialized tail leaks residual GPU memory — tensor data from other tenants' inference requests — across cohort boundaries on a shared GPU. With no patch available, per-cohort GPU pools and network segmentation are the only mitigations, which is the strongest argument yet for not co-scheduling canary and stable on the same device. CVE-2026-48746 (OpenAI-API auth bypass, CVSS 9.1) compounds this: a vulnerable cohort sharing an ingress or trust boundary becomes an unauthenticated entry point into the serving plane. *(Updated 2026-06-20)* |
