# C7.1: Output Format Enforcement

[Back to C07 Index](C07-Model-Behavior.md)

## Purpose

Output format enforcement ensures that model-generated content conforms to expected structures before it reaches downstream systems. Treating model output as untrusted input is the single most important principle in this section. Without strict schema validation and safe handling, model outputs can introduce injection vulnerabilities (XSS, SQLi, command injection), cause deserialization attacks, or corrupt data pipelines. These controls apply the same defense-in-depth approach used for any untrusted data source.

The "Comment and Control" disclosures highlight a governance gap separate from the technical one: the fact that Anthropic, Google, and Microsoft each paid bug bounties for a critical, externally exploitable agent vulnerability and then patched without issuing CVEs or GHSA advisories means downstream consumers had no way to detect or correlate the fix — an emerging concern for organizations that need a defensible record of agent runtime patches for SOC2, ISO 42001, or EU AI Act conformity assessments. CEN-CENELEC JTC 21, the standards body responsible for EU AI Act harmonized standards, has not yet finalized comprehensive standards covering Articles 9–15 as of early 2026, leaving providers to assemble their own evidence packs against the high-level Article 15 requirements. |

## Implementation Guidance

### Constrained Decoding vs. Post-hoc Validation

There are two complementary strategies for enforcing output format, and production systems should use both:

**Constrained decoding** inserts a logit processor between the model's raw output logits and the sampling step. The processor tracks the current position within a target grammar (JSON Schema, regex, or CFG) and masks out tokens that would violate the grammar. This guarantees syntactic conformance at generation time.

- **XGrammar** splits vocabulary tokens into context-independent tokens (~99% of vocabulary, precomputed into bitmask tables) and context-dependent tokens (~1% requiring runtime stack inspection), achieving up to 100x speedup over earlier grammar-constrained methods.
- **Microsoft llguidance** uses a derivative-based regex engine and optimized Earley parser for CFG rules, running in Rust with ~50 microseconds CPU time per token.
- **DOMINO** (ICML 2024, ETH Zurich) enforces constraints in a fully subword-aligned fashion, solving the token misalignment problem where BPE tokenization clashes with grammar boundaries. By leveraging pre-computation and speculative decoding, DOMINO achieves zero overhead and in some workloads nearly 2x speedup over unconstrained generation — making it the first constrained decoding method that can actually accelerate inference.
- **Pre3** (ACL 2025) uses deterministic pushdown automata with precomputed prefix-conditioned edges, improving time per output token by up to 40% and throughput by up to 36% over existing constrained decoding approaches.
- **Outlines** (dottxt) and **LMQL** provide Python-native constrained decoding for open-weight models.
- **Provider-side structured output**: OpenAI Structured Outputs (which credited llguidance as foundational work in May 2025), Anthropic tool-use schemas, and Google Gemini Structured Outputs all offer API-level format enforcement. OpenAI and Gemini both document this as schema-subset enforcement: useful for syntactic/type conformance, but still requiring application-side semantic checks, refusal handling, and business-rule validation before the output reaches an interpreter or workflow.
- **CRANE** (ICML 2025, UIUC) — reasoning-augmented constrained decoding that augments output grammars with rules permitting intermediate reasoning steps. Rather than suppressing the model's chain-of-thought, CRANE alternates between unconstrained reasoning and constrained output generation, achieving up to 10 percentage points accuracy improvement on symbolic reasoning benchmarks (GSM-symbolic, FOLIO) while maintaining structural correctness.
- **In-Writing** (2025) — a unified hybrid decoding framework that defers constraint enforcement entirely until the model emits a trigger token (e.g., `{`), then switches to grammar-guided generation. This separates the reasoning phase from the formatting phase, gaining up to 27% accuracy on classification tasks with only 10–20 extra tokens of overhead. Particularly relevant for smaller models (1.7B parameters) deployed at the edge where reasoning capacity is limited.
- **DCCD (Draft-Conditioned Constrained Decoding)** (March 2026, arXiv 2603.03305) — a training-free two-step procedure that fully decouples semantic planning from structural enforcement. The model first generates an unconstrained draft, then re-generates under grammar constraints conditioned on that draft. This reduces the cumulative "projection tax" (analyzed via a KL-projection framework) that standard token-by-token masking imposes. Results are striking: on GSM8K under strict JSON constraints, DCCD improves a 1B model from 15.2% to 39.0% accuracy (+24pp), and a 1.5B model from 49.4% to 73.9% (+24.5pp). Smaller DCCD-equipped models match or exceed much larger constrained baselines, making this particularly relevant for cost-constrained deployments. An optional best-of-K draft selection further improves results.

### Serving Framework Integration

Production deployment of constrained decoding depends heavily on the serving framework. As of 2026, XGrammar is the default structured-generation backend for vLLM, SGLang, and TensorRT-LLM, achieving under 40 microseconds CPU per token with near-zero overhead for JSON generation. Both vLLM and SGLang integrate XGrammar and llguidance as pluggable grammar backends:

- **SGLang** uses XGrammar as its default backend and overlaps CPU-intensive grammar mask generation with GPU inference, effectively hiding grammar processing latency. This architectural choice makes SGLang particularly efficient for structured output workloads — independent benchmarks in 2026 consistently show SGLang sustaining throughput close to its unconstrained baseline even at batch size 8 and above.
- **vLLM** supports both backends via its guided decoding API, with llguidance available as an alternative to XGrammar. As of 2026, vLLM still shows significant throughput degradation with guided decoding enabled at higher batch sizes due to non-overlapped, sequential mask generation. Teams optimizing for high-throughput JSON-mode agent pipelines often see better results with SGLang, while vLLM remains competitive at low concurrency.
- **Workload-dependent selection**: SqueezeBits benchmarks (September 2025, using vLLM v0.10 / SGLang 0.5) found that XGrammar outperforms llguidance on repetitive schemas (same schema reused across requests) due to pre-computation caching, while llguidance excels on dynamic schemas where each request introduces a unique structure. XGrammar handles more schemas initially but can timeout on highly complex ones; llguidance has zero timeouts but slightly more compilation failures. Invalid JSON rates drop to 2.21% (XGrammar) and 0.12% (llguidance) with guided decoding enabled. JSONSchemaBench (10K real-world schemas) is now the de facto evaluation harness — it tests Guidance, Outlines, Llamacpp, XGrammar, OpenAI, and Gemini across efficiency, coverage, and quality dimensions, and surfaces an important nuance: FSM-based engines like Outlines reject or flatten recursive schemas, so workloads with self-referential JSON Schema must use a CFG-based engine (XGrammar or llguidance).
- **Overhead**: Typical constraint overhead ranges from 5–15% latency increase for simple schemas to 30–60% for complex schemas, with conformance rates of 95–99% for well-supported engines.
- **vLLM API migration**: Starting with vLLM v0.12.0, the `guided_*` request parameters (`guided_json`, `guided_regex`, `guided_grammar`, etc.) have been removed in favor of a unified `structured_outputs` field. The `guided_decoding_backend` field is also being deprecated; the equivalent runtime configuration is now `--structured-outputs-config.backend` on `vllm serve`. Teams migrating from older configurations should update both request payloads and server flags. SGLang continues to use simpler parameter names (`choice`, `regex`, `ebnf`) rather than vLLM's `guided_*`/`structured_outputs` naming, so cross-framework client code typically needs a thin adapter layer.
- **Serving framework security**: The infrastructure that runs constrained decoding itself carries risk. CVE-2026-27893 (vLLM, CVSS 8.8, March 2026) revealed that two model implementation files hardcoded `trust_remote_code=True` when loading sub-components, overriding the user's explicit `--trust-remote-code=False` flag. A malicious model repository could achieve RCE on the inference server — the same server enforcing output format constraints. Affected versions: vLLM 0.10.1 through 0.17.x; fixed in 0.18.0. Similarly, CVE-2026-22778 (vLLM, critical) demonstrated RCE via crafted video URLs processed during multimodal inference. Teams relying on vLLM or SGLang for structured output enforcement should treat serving framework patching as a security-critical dependency.

**Post-hoc validation** applies schema checks after generation completes. This catches semantic issues that grammar-level enforcement cannot (e.g., a field is valid JSON but contains a script payload in a string value). Tools include:

- **Guardrails AI** — open-source framework with a hub of pre-built validators for output format, content safety, PII detection, and custom rules. Supports Python and JavaScript. The 2026 release line cut baseline overhead by roughly 40% versus the 2025 engine, making it more viable to run multiple validators (PII, toxicity, regex, competitor-mention) on every response without breaking latency budgets. A common 2026 production pattern stacks Instructor + Pydantic for structural validation, Guardrails AI for content validation, and LLM Guard or Lakera Guard for security scanning, with NeMo Guardrails managing conversation flow on top.
- **NVIDIA NeMo Guardrails** — programmable rails using Colang (a domain-specific language) for defining conversational safety policies. Complements Guardrails AI's validation approach with state-machine-based flow control. The March 12, 2026 v0.21.0 release introduced the **IORails engine**, which runs content-safety, topic-safety, and jailbreak detection rails in parallel (reducing serial latency) and assigns unique request IDs for correlating guardrail decisions with downstream logs. A new `check_async` method lets applications validate input or output against rails without standing up a full conversation flow — useful for batch jobs, test harnesses, or integrating guardrails into existing pipelines. v0.21.0 also added OpenAI API compatibility (including a `v1/models` endpoint), a `GuardrailsMiddleware` for LangChain agents, and lazy initialization of embedding indexes to improve cold-start latency.
- **Instructor** — lightweight library (Python/TypeScript) that wraps LLM API calls with Pydantic model validation, automatic retries on schema violations, and structured extraction patterns.

Additional post-hoc tools worth evaluating:

- **LLM Guard** — open-source, dependency-light scanner that runs entirely locally without requiring an LLM call. Provides 20 output scanners covering toxicity, bias, code injection, sensitive data, and format violations. Sub-10ms per check makes it suitable for high-throughput pipelines. MIT licensed.
- **Galileo Luna-2** — enterprise guardrails platform using purpose-built small language models for real-time detection of hallucinations, PII leakage, and toxicity at 0.95 F1 accuracy. Supports configurable output policies (block, redact, override) and integrates evaluation-to-production workflows.

Both layers together provide defense-in-depth: constrained decoding prevents malformed structure, and post-hoc validation catches dangerous content within well-formed output.

### Output Injection Attack Patterns

LLM output can serve as an injection vector in any downstream execution context. Key patterns to defend against:

| Attack Pattern | Mechanism | Defense |
|---------------|-----------|---------|
| **XSS via LLM output** | Model generates `<script>` tags or event handlers in output rendered as HTML. Common in chat UIs using markdown renderers that pass through raw HTML. | Context-appropriate output encoding, DOMPurify, CSP headers. Verify all rendering paths — streaming, cached, and historical — use the same sanitization pipeline. |
| **Markdown renderer XSS** | AI coding agents and chat interfaces render LLM markdown output containing embedded HTML/JS without sanitization, exploiting markdown-to-HTML conversion | Use markdown renderers with HTML sanitization enabled by default; add CSP headers blocking inline scripts; never use `dangerouslySetInnerHTML` on LLM output without DOMPurify |
| **Custom markdown extension bypass** | Custom tag components (iframe renderers, chart renderers, math renderers) consume LLM output but skip the sanitizer applied to standard markdown — as in MaxKB's `MdRenderer.vue` (`<iframe_render>` tag) and `EchartsRender` components | Audit every custom markdown extension for independent sanitization; never combine iframe `allow-scripts` with `allow-same-origin` on LLM-controlled content; treat custom tag inputs as untrusted even when the base markdown is sanitized |
| **Tool widget / chart renderer XSS** | Agent tools generate auxiliary UI content such as chart captions, tables, cards, and widget labels that bypass the main chat sanitizer. CVE-2026-41318 in AnythingLLM showed the chart caption path calling `renderMarkdown(content.caption)` without DOMPurify even though most chat render paths sanitized the same renderer output. | Route every tool-specific widget, caption, and card through the same sanitizer and output encoder as the main chat renderer; add regression tests for streaming, historical, cached, and tool-result render paths; encode attribute values before interpolation into custom markdown renderers. |
| **SQL injection via text-to-SQL** | Natural language translated to SQL by LLM; malicious prompt causes query manipulation | Parameterized query templates; LLM fills only data values, never query structure |
| **Agent tool-argument SQL injection** | Function-calling arguments generated by the model are treated as trusted identifiers or SQL fragments. CVE-2026-32628 in AnythingLLM's SQL Agent passed the LLM-generated `table_name` argument into MySQL, PostgreSQL, and MSSQL query strings via template literals, so prompt injection could turn a schema lookup tool into arbitrary SQL execution. | Validate tool arguments against server-side schemas and allowlists; parameterize values; escape identifiers with database-driver primitives; run agent database connections as read-only roles; enforce read-only query policy on the server rather than relying on natural-language tool descriptions. |
| **Command injection** | LLM output interpolated into shell commands via `exec`, `eval`, or `os.system` | Never pass LLM output to shell interpreters; use structured API calls instead |
| **SSRF via LLM output** | Model generates URLs that are fetched server-side without validation | URL allowlisting; validate scheme, host, and port before any server-side fetch |
| **Path traversal** | LLM-generated file paths contain `../` sequences | Canonicalize paths; validate against allowed directory prefixes |
| **Deserialization attacks** | LLM output passed to unsafe deserializers (`pickle`, `eval`, `yaml.load`) | Use safe parsers only (`json.loads`, `yaml.safe_load`); never deserialize LLM output with code-execution-capable formats |
| **Serialization injection** | LLM-influenced data containing framework-reserved keys (e.g., LangChain's `lc` marker) triggers arbitrary object instantiation during deserialization | Sanitize LLM output before passing to any framework serialization layer; strip or escape reserved keys; pin framework versions |
| **Checkpoint/query metadata injection** | LLM- or user-influenced metadata keys are interpolated into SQL, vector-store filters, or checkpoint queries while only values are parameterized; LangGraph's SQLite checkpointer CVE-2025-67644 is the clean example | Treat metadata keys as schema-bound identifiers, not free-form strings; allowlist key names before query construction; add tests where model output controls both keys and values in checkpoint, cache, vector-store, and analytics queries |
| **Server-side template injection (SSTI)** | LLM output or LLM-controlled template variables are passed to a template engine (Jinja2, Mustache, etc.) without sandboxing, enabling arbitrary code execution on the server | Use sandboxed template environments (e.g., Jinja2 `SandboxedEnvironment`); never allow LLM output to define template structure; treat template fields as data, not code |
| **Mermaid diagram renderer XSS → RCE** | AI chat apps pass LLM-generated Mermaid diagram syntax unsanitized to the Mermaid library, which supports HTML in node labels; an attacker crafts diagram content with `<img onerror=...>` tags that execute JavaScript; in Electron desktop apps, the payload calls the exposed IPC bridge for OS-level RCE (CVE-2026-23733, CVE-2025-58768, CVE-2026-40322 — three products, same chain) | Configure Mermaid with `securityLevel: 'strict'` to disable HTML in labels; enable Electron `contextIsolation` and never expose raw IPC to the renderer; treat all LLM-generated diagram syntax as untrusted input |
| **Marketplace README XSS → RCE** | AI tools that ship a community marketplace render package READMEs (markdown that may include LLM-related content like prompts, plugin descriptions, or model packs) as HTML; missing sanitizer flags let raw HTML through, and Electron `nodeIntegration: true` / `contextIsolation: false` escalates to RCE. CVE-2026-33066 (SiYuan Bazaar, May 2026) used `lute.New()` without `SetSanitize(true)`; the patch in 3.6.1 enabled the sanitizer but left iframe + `srcdoc` unfiltered, producing CVE-2026-40922 (versions 3.6.1–3.6.3) before the proper fix in 3.6.4 | Enable HTML sanitization explicitly in the markdown renderer (e.g., `SetSanitize(true)` for Lute, DOMPurify on the frontend); add iframe to sanitizer skip-content sets or strip `srcdoc` recursively; disable `nodeIntegration`, enable `contextIsolation` and `sandbox: true` in Electron `webPreferences`; treat marketplace authors as untrusted publishers |
| **Unicode-smuggled LLM output** | LLM emits output containing invisible Unicode Variation Selectors, Tags-block code points, or zero-width characters that encode hidden instructions or payloads; downstream agents, audit logs, and human reviewers see only the visible glyph (e.g., a smiley face) while the underlying byte sequence carries malicious content. CVE-2026-27966 (Langflow CSV Agent, CVSS 9.8) hardcoded `allow_dangerous_code=True`; CVE-2026-41264 (Flowise CSV Agent, CVSS 8.1, April 2026) bypassed a regex blocklist added for CVE-2026-41137 | Disable `allow_dangerous_code` by default in any LangChain-style agent; replace regex blocklists with positive AST-level allowlists; route LLM-generated code through an isolated execution sandbox (e.g., gVisor, Firecracker, ephemeral container) with no host network, no host filesystem, and per-call CPU/memory limits; never expose unauthenticated chatflow endpoints that wrap a code-executing tool; treat regex pattern blocklists on LLM output as defense-in-depth, never as the sole control |
| **Agent shell command emission via PR/comment injection** | An AI coding agent (Claude Code Security Review, Gemini CLI Action, GitHub Copilot Agent) consumes untrusted PR titles, issue bodies, or HTML-comment-hidden text. Prompt injection causes the LLM to emit shell commands like `cat $GITHUB_TOKEN` or `echo $GEMINI_API_KEY` posted as a public issue comment; the agent runtime executes those commands with full CI privileges. Traditional WAFs often miss these attacks because the injection payload originates from the model's generation, not from raw user input. Defense must be applied at the point where LLM output enters a downstream system.

### Real-World Incident Case Studies

These incidents illustrate why output format enforcement is not optional:

**CVE-2026-32626 — AnythingLLM Streaming XSS to RCE (March 2026).** AnythingLLM Desktop v1.11.1 and earlier had a critical flaw (CVSS 9.6) in its streaming chat renderer. The `PromptReply` component used `dangerouslySetInnerHTML` without DOMPurify sanitization on streamed tokens, while the `HistoricalMessage` component correctly applied `DOMPurify.sanitize()`. An attacker could embed malicious payloads in RAG documents or control a malicious LLM endpoint; when the model generated output containing the payload, JavaScript executed in the Electron renderer context, escalating to full RCE on the host OS. This perfectly demonstrates why sanitization must be applied uniformly across all rendering paths — streaming and historical — not just on saved content.

**CVE-2026-41318 and CVE-2026-32628 — AnythingLLM Tool-Output Sink Failures (March-April 2026).** Two later AnythingLLM advisories show that output handling has to cover tool-specific UI and tool-call arguments, not just the primary chat renderer. CVE-2026-41318 affected versions up to 1.12.0: chart captions generated around the `create-chart` tool were rendered through the same custom markdown image rule that interpolated `alt` text into HTML attributes, but the `Chartable` component skipped the DOMPurify wrapper used elsewhere. An attacker who influenced a chart caption through indirect prompt injection or a stored chart record could trigger stored DOM XSS when another workspace user opened the conversation. CVE-2026-32628 affected the built-in SQL Agent in versions up to 1.11.1: the LLM-generated `table_name` argument for `sql-get-table-schema` flowed directly into MySQL, PostgreSQL, and MSSQL query strings. The advisory also noted that the `sql-query` tool relied on a natural-language "read-only" instruction rather than server-side enforcement. Together, these cases give auditors two concrete tests: every custom tool widget must share the main sanitization pipeline, and every model-generated tool argument must be validated, parameterized, escaped as an identifier where appropriate, and constrained by least-privilege backend credentials.

**CVE-2025-68664 — LangChain Core "LangGrinch" (December 2025).** LangChain Core's `dumps()` and `dumpd()` functions failed to escape dictionaries containing the reserved `lc` key (CVSS 9.3). An attacker who influenced LLM output (e.g., via prompt injection populating `additional_kwargs` or `response_metadata`) could inject structures that were treated as legitimate LangChain objects during deserialization. This enabled secret extraction via `secrets_from_env`, arbitrary class instantiation across multiple LangChain provider packages, and code execution through Jinja2 template rendering in `PromptTemplate`. The vulnerability affected 12 common flows including event streaming, message history, and caching — impacting hundreds of millions of installations.

The companion LangChain JS advisory, GHSA-r399-636x-v7f6 / CVE-2025-68665, shows the same output-handling mistake in another runtime: `Serializable.toJSON()` failed to escape user-controlled objects with `lc` keys before later `load()` deserialization. The affected fields (`additional_kwargs`, `metadata`, and `response_metadata`) are exactly the kind of data that can be influenced by prompt injection or model output. The Python fix changed `secrets_from_env` to default `False`, added allowlist enforcement through `allowed_objects`, and blocked Jinja2 templates by default; the JavaScript fix changed `secretsFromEnv` to default `false`, escaped reserved keys, and added a `maxDepth` guard. For auditors, this is a concrete test case: feed reserved framework keys through every model-output metadata path and confirm they remain plain data after serialization, streaming, checkpointing, caching, and replay.

**CVE-2026-34070 and CVE-2025-67644 — LangChain Prompt Loader Path Traversal and LangGraph Checkpoint SQL Injection (December 2025–March 2026).** Two adjacent advisories show why output handling reviews need to include framework plumbing, not only the final chat renderer. CVE-2026-34070 affected LangChain Core versions before 1.2.22: legacy `load_prompt()` and `load_prompt_from_config()` APIs read file paths embedded in deserialized prompt configuration dictionaries without rejecting absolute paths or traversal sequences, so any application that let LLM- or user-influenced prompt configs reach those loaders could expose host files. The fix deprecated the legacy file-reading APIs and steered users toward the allowlist-based `dumpd` / `dumps` / `load` / `loads` serialization path. CVE-2025-67644 affected `langgraph-checkpoint-sqlite` before 3.0.1: checkpoint filter values were parameterized, but metadata filter keys were interpolated into `json_extract(...)` SQL fragments with f-strings. For C7.1 auditors, the test is straightforward: identify every place model output can shape configuration keys, file paths, metadata filter names, or checkpoint queries, and confirm those identifiers are selected from a fixed schema rather than accepted as arbitrary strings.

**CVE-2025-25362 — spacy-llm Server-Side Template Injection (April 2025).** The spacy-llm library (v0.7.2 and earlier) used Jinja2's default `Environment` — without sandboxing — to process prompt templates. An attacker who could influence the template field (e.g., through user-supplied configuration or injection into template parameters) could execute arbitrary code on the server via Python's object introspection chain. The fix in v0.7.3 switched to `SandboxedEnvironment`. This is a textbook example of an output-adjacent vulnerability: the injection occurs not in the model's response but in the template layer that constructs prompts and processes results, reinforcing that every component in the LLM pipeline — not just the model output itself — must treat dynamic content as untrusted.

**CVE-2026-25802 — Prompt Injection to Stored XSS in new-api (2026).** The new-api project (QuantumNous, versions before 0.10.8-alpha.9) unsafely rendered LLM output using React's `dangerouslySetInnerHTML` without sanitization (CVSS 7.6). A custom JSON syntax highlighter also failed to escape HTML entities in token values, allowing attackers to break out of `<span>` tags and inject event handlers. Since chat histories persist in the database, this became a stored XSS vulnerability — any administrator reviewing compromised chat logs would trigger the payload. The fix introduced iframe sandboxing (without `allow-scripts`) and proper HTML entity escaping for all syntax-highlighted tokens.

**CVE-2026-22813 — OpenCode AI Coding Agent XSS (2026).** OpenCode (versions prior to 1.1.10) contained a critical XSS vulnerability (CVSS 9.4) in its web interface's markdown renderer. The renderer inserted LLM response content directly into the DOM without DOMPurify sanitization, and the application lacked Content Security Policy headers entirely. An attacker who could influence LLM responses — through prompt injection, model compromise, or man-in-the-middle on the localhost:4096 interface — could execute arbitrary JavaScript within chat sessions, potentially achieving session hijacking and data exfiltration. This reinforces a growing pattern: AI coding tools that render model output as rich markdown are particularly vulnerable when their rendering pipelines skip sanitization.

**CVE-2026-23733, CVE-2025-58768, CVE-2026-40322 — Mermaid Diagram Renderer XSS-to-RCE Cluster (2025–2026).** A consistent exploit chain has been demonstrated across three AI desktop applications: LobeChat (CVE-2026-23733, CVSS 9.7, January 2026), DeepChat (CVE-2025-58768, CVSS 9.7, September 2025), and SiYuan (CVE-2026-40322, CVSS 9, April 2026). In each case, LLM-generated content is passed directly to the Mermaid diagram rendering library without sanitization. Mermaid supports HTML in node labels by default (`securityLevel: 'loose'`), so a payload like a Mermaid graph definition where a node label contains `<img onerror="window.electronAPI.invoke('runCommand','open -a Calculator')">` renders the image tag in the DOM, triggers the error handler, and executes arbitrary JavaScript. In the Electron desktop build, the script then calls the exposed `electronAPI` IPC bridge — accessible directly from the renderer process because `contextIsolation` is disabled or the preload script exposes privileged methods without validation — to run arbitrary OS commands. The fix in all three products involved both sanitizing Mermaid input and configuring `securityLevel: 'strict'`. The pattern highlights that Mermaid is a systematically underdefended rendering surface in AI applications: it is widely used for displaying LLM-generated diagrams, its default configuration executes HTML, and Electron's IPC bridge provides a reliable escalation path when `contextIsolation` is absent.

**CVE-2026-33066 and CVE-2026-40922 — SiYuan Bazaar Marketplace XSS-to-RCE Pair (May 2026).** SiYuan's community marketplace (Bazaar) renders package README content as HTML so users can browse plugins, themes, and AI-related content packs. CVE-2026-33066 (CVSS critical) revealed that the backend `renderREADME` function instantiated the Lute markdown engine without `SetSanitize(true)`, letting raw HTML pass through unmodified, and the frontend then assigned the rendered HTML to `innerHTML` without any additional sanitizer. SiYuan's Electron build shipped with `nodeIntegration: true` and `contextIsolation: false`, so any XSS payload escalated immediately to full RCE — capable of stealing API tokens, SSH keys, browser credentials, and installing persistent backdoors across Windows, macOS, and Linux. A malicious package author could weaponize the official marketplace itself, turning supply-chain trust into a remote-code-execution channel. The initial fix in 3.6.1 enabled the Lute sanitizer but missed two gaps that produced CVE-2026-40922 (affecting 3.6.1–3.6.3): the sanitizer did not block `<iframe>` tags (iframe was explicitly commented out of `setOfElementsToSkipContent`), and its URL-prefix blocklist applied to attributes like `src` did not filter the `srcdoc` attribute, which contains raw HTML rather than a URL. The proper fix landed in 3.6.4. The pair is instructive on three fronts: (1) sanitizer defaults matter — opting in to sanitization must be explicit, never assumed; (2) URL-based filters cannot defend attributes that carry raw HTML; (3) AI-adjacent marketplaces that ship LLM-relevant content should treat package metadata as fully untrusted input. The same lesson generalizes to any AI tool with a model pack hub, prompt library, or plugin store.

**Unicode and Emoji Smuggling in LLM Output (FireTail, January 2026).** FireTail published research demonstrating that modern LLMs reliably emit and round-trip Unicode-smuggled payloads — instructions or data hidden inside Variation Selectors attached to a single visible glyph (an emoji or letter). To a human reviewer, the output is a smiley face. To a tokenizer, a downstream agent, or another LLM that ingests the output, the underlying byte sequence carries an arbitrary payload. AWS's 2026 security blog post on Unicode character smuggling reinforced the defense pattern: normalize output with NFKC, allowlist Unicode categories, and explicitly reject Tags-block code points (U+E0000–U+E007F) and invisible Variation Selectors unless a specific use case requires them. Audit logs that record only the rendered glyph silently lose the smuggled payload, leaving incident response without the data needed to explain why an LLM agent took an apparently benign output and executed a malicious tool call. This is a structural problem for output handling: traditional XSS defenses (DOMPurify, output encoding) catch HTML and JavaScript, but they do not catch Unicode payloads embedded in legitimate-looking text. Output validation pipelines need a Unicode normalization step distinct from the existing HTML sanitizer.

**CVE-2026-39426 and CVE-2026-39423 — MaxKB Custom Markdown Extension XSS (April 2026).** MaxKB, an open-source enterprise AI assistant, shipped two stored-XSS vulnerabilities in the same release family (versions 2.7.1 and earlier). CVE-2026-39426 (CVSS 5.1, published April 14, 2026) exploited the `MdRenderer.vue` component's handling of a custom `<iframe_render>` tag parsed from LLM responses and Application Prologue configuration. The custom tag bypassed the standard markdown sanitization pipeline, and the iframe it produced was configured with `sandbox="allow-scripts allow-same-origin"` — a combination that is effectively no sandbox at all, because injected scripts can simply reach through `window.parent` to execute in the parent document context. CVE-2026-39423 (CVSS 6.9) hit the `EchartsRender` component with an eval injection flaw in the markdown rendering engine: any authenticated user interacting with the AI chat interface could emit markdown that triggered arbitrary JavaScript execution in other users' browsers, including administrators. Both were fixed in v2.8.0. The key lesson is that custom markdown extensions — iframe renderers, chart renderers, math renderers, custom tag components — frequently bypass the sanitization carefully applied to the base markdown renderer. Every such extension needs its own sanitization audit, and iframe sandbox attributes must never combine `allow-scripts` with `allow-same-origin` for LLM-controlled content.

**CVE-2025-66562 — Tuui ECharts Markdown RCE Chain (December 5, 2025).** Tuui, an Electron-based AI chat client, contained a markdown-to-RCE chain (CVSS 8.9) in `MarkdownCard.vue` that depended on three independent misconfigurations. First, the markdown renderer evaluated code blocks labelled `echarts` directly without sanitizing the JavaScript inside, granting the block access to the global `window` object. Second, the application's HTML CSP allowed `unsafe-eval`, so arbitrary string-to-code conversion was permitted at runtime. Third, the Electron preload script exposed `ipcRenderer.invoke`, and the main process used that bridge — via the MCP servers initialization handler — to spawn child processes from renderer-supplied arguments. An attacker who could influence an LLM response (via prompt injection, RAG poisoning, or a malicious MCP server) emitted markdown containing an `echarts` code block, the renderer eval'd it, the script reached through the IPC bridge, and the main process spawned a child process with attacker-controlled command-line arguments. The fix shipped in v1.3.4. The case is illustrative on three fronts: (1) `unsafe-eval` in a CSP is not a "modest weakening" — it neutralizes most XSS sandbox guarantees; (2) any markdown code-block language that triggers JavaScript execution (echarts, vega, vega-lite, plotly) needs the same sanitization treatment as `<script>` itself; (3) Electron preload scripts that expose generic IPC primitives like `invoke` to the renderer create a guaranteed escalation path the moment any DOM injection is found.

**CVE-2026-27966 and CVE-2026-41264 — CSV Agent Prompt-to-RCE Cluster (February–April 2026).** Two CSV Agent vulnerabilities in low-code AI builders demonstrated the same anti-pattern: routing LLM output directly into a non-sandboxed Python interpreter. CVE-2026-27966 (Langflow, CVSS 9.8, February 26, 2026) hardcoded `allow_dangerous_code=True` in the CSV Agent node, which silently exposed LangChain's `python_repl_ast` tool to any chatflow request. Prompt injection induced the LLM to emit Python that the server immediately executed as an unauthenticated user. The fix in Langflow 1.8.0 added a `BoolInput` configuration that defaults to `False`. CVE-2026-41264 (Flowise, CVSS 8.1, published April 21, 2026) was a bypass of the input validation that had been introduced for the earlier CVE-2026-41137: the `validatePythonCodeForDataFrame()` function used regex patterns to block dangerous imports, but the patterns failed against aliased imports like `import pandas as np, os as pandas` followed by `pandas.system(...)`. The LLM-generated script was still evaluated in a non-sandboxed pyodide environment, producing RCE. The fix in Flowise 3.1.0 disallowed all imports in the CSV Agent rather than continuing the regex-arms-race. Both CVEs underscore that regex blocklists on LLM-generated code are unreliable; the only durable defense is positive AST-level allowlists combined with execution sandboxing (gVisor, Firecracker, ephemeral containers). The cluster is also a reminder that the broader Flowise ecosystem has been under active exploitation: CVE-2025-59528 (Flowise CustomMCP node, CVSS 10.0) was disclosed in September 2025 and saw active exploitation in April 2026 from a single Starlink IP, with over 12,000 internet-facing Flowise instances vulnerable. Six months elapsed between disclosure and exploitation — defenders had time, but the exposed surface area was overwhelming.

**Comment and Control — Coordinated Prompt Injection Across Three AI Coding Agents (early 2026).** Researcher Aonan Guan disclosed a single technique that compromised three production AI coding agents through GitHub PR titles, issue bodies, and HTML-comment-hidden instructions. Against Anthropic's Claude Code Security Review, the unsanitized PR title was injected directly into the agent's prompt; the model emitted shell commands that exfiltrated environment variables containing the GitHub token and AWS credentials. Anthropic awarded CVSS 9.4 (Critical) and a $100 bounty. Against Google's Gemini CLI Action, a fake "Trusted Content Section" appended after legitimate content overrode Gemini's safety instructions, causing it to post the `GEMINI_API_KEY` as a public issue comment; Google paid a $1,337 bounty. Against GitHub Copilot Agent, the injection was hidden inside an HTML comment in a Markdown body — invisible in the rendered view but present in the prompt context — and bypassed the agent's environment-variable filtering, secret scanning, and network restrictions in sequence; GitHub paid $500 through the Copilot Bounty Program. None of the three vendors assigned a CVE or published a security advisory. All three patched silently. For C7.1.3, the case demonstrates that an AI agent's "output" is not just text shown to a user — it is also the sequence of shell commands, tool calls, and API requests the runtime executes on the agent's behalf. The trust boundary the agent crosses is the line between "characters streamed to a screen" and "system call issued with the runtime's privileges," and that boundary needs the same parameterized handling as any other interpreter boundary. Anthropic's own Claude Opus 4.7 system card (April 2026) acknowledges that Claude Code Security Review "is not hardened against prompt injection," and Anthropic's November 2025 prompt injection defense paper reports that Claude Opus 4.5 still has a 1% attack success rate against an internal adaptive Best-of-N attacker — meaningful enough that high-privilege agent contexts cannot rely on model-level robustness alone.

### Agent Runtime Output Enforcement

The CSV Agent cluster (CVE-2026-27966, CVE-2026-41264) and the Comment and Control disclosures show that for agentic systems, output format enforcement is no longer just about validating returned text — it is about constraining what the runtime is allowed to do with model output. Three patterns are emerging in 2026:

- **Sandboxed code execution as default.** Agent runtimes that route LLM output to a code interpreter should default to per-session sandboxes (gVisor, Firecracker, ephemeral containers) with no host network, no host filesystem outside an explicit workspace, and per-call CPU/memory budgets. Cloudflare Dynamic Workers, Northflank's per-session sandboxes, and similar offerings now provide V8 isolates or microVMs that spin up in tens of milliseconds, making "sandbox per LLM-emitted code block" practical rather than aspirational.
- **Runtime enforcement DSLs.** AgentSpec (ICSE 2026, NUS) is the first systematic framework for declaring runtime safety constraints on LLM agents. It expresses rules as a triggering event (e.g., "agent is about to execute a financial transaction"), predicate conditions (e.g., "amount exceeds $1,000"), and an enforcement action (e.g., "require human confirmation"). The framework lets organizations encode policy in a centralized, auditable layer rather than scattering it through prompt templates and ad-hoc code.
- **Tool-output classification.** Anthropic's November 2025 prompt injection defense work introduces a server-side prompt-injection probe that scans tool outputs (file reads, web fetches, shell output, MCP server responses) before they enter the agent's context. When content looks like an attempt to hijack behavior, the probe annotates the agent's context with a warning, telling the model to treat the result as suspect. A separate transcript classifier evaluates each proposed action against a policy before it executes, acting as an automated substitute for human approval. Anthropic published quantified metrics: Claude Opus 4.5 attack success against an internal Best-of-N attacker dropped to 1%, but they explicitly call this "meaningful risk" — agent runtimes should not rely on model-level robustness alone.

The common thread across all three is that the agent runtime — not the model — is the right layer to enforce output handling controls. Treating LLM output as untrusted input must extend to every system call, every tool invocation, and every credential the runtime can access on the agent's behalf.

### Streaming Output Validation

When using streaming responses (SSE/WebSocket), full schema validation cannot be applied until generation completes. CVE-2026-32626 demonstrated the real danger: AnythingLLM sanitized historical messages but not streaming tokens, creating an exploitable gap. Recommended approaches:

1. **Buffer and validate**: Accumulate streamed tokens, validate the complete output before forwarding to downstream systems. Display partial output to users but gate any programmatic consumption on full validation. Apply the same sanitization (DOMPurify, output encoding) to streaming renderers as to historical/saved content — never assume streaming output is safe because it will be "re-sanitized later."
2. **Incremental parsing**: Use incremental JSON parsers that can detect structural violations mid-stream and abort generation early. CRANE (2025) introduces reasoning-aware constrained generation that can validate semantic constraints incrementally during streaming.
3. **Constrained decoding**: When available, constrained decoding at the inference layer eliminates the need for post-hoc streaming validation since every emitted token is guaranteed to be grammar-conformant.
4. **Uniform sanitization paths**: As a design principle, ensure every code path that renders or processes LLM output — streaming, cached, historical, or real-time — uses the same sanitization pipeline. The streaming/historical renderer divergence in CVE-2026-32626 is a common anti-pattern.

## Open Research Questions

- How should applications handle partial schema conformance (e.g., output is valid JSON but missing optional fields)? Should this be a hard reject or a soft warning?
- What is the performance impact of constrained decoding vs. post-hoc validation, and when is each approach preferable? Pre3 (ACL 2025) and llguidance show near-zero overhead, but the "Thinking Before Constraining" framework (2026) demonstrates that constrained decoding can degrade semantic correctness on reasoning-intensive tasks — suggesting the optimal strategy depends on task complexity, not just latency.
- How should streaming outputs be validated when the full schema cannot be checked until generation is complete? CVE-2026-32626 demonstrated real exploitation of the streaming/historical sanitization gap. Incremental parsing and buffered validation are emerging but not yet standardized.
- Can constrained decoding prevent semantic injection (e.g., a valid JSON string field containing a script payload)? Grammar-level enforcement guarantees structure but not content safety — the gap between syntactic and semantic validation remains an open problem. CRANE (ICML 2025) and In-Writing (2025) both explore hybrid approaches that preserve reasoning while enforcing structure, but neither addresses content-level safety within well-formed outputs.
- How effective are backdoor attacks against fine-tuned text-to-SQL models, and what defenses beyond parameterized queries can mitigate model-level compromise?
- How should frameworks handle LLM output that contains reserved internal markers (like LangChain's `lc` key)? CVE-2025-68664 showed that framework-level serialization assumptions create novel attack surfaces when LLM output flows through framework internals. Should frameworks adopt namespace isolation for serialized objects?
- What is the right boundary between constrained decoding approaches? DOMINO (ICML 2024) achieves zero overhead with speculative decoding but requires pre-computation; Pre3 (ACL 2025) uses pushdown automata for throughput gains; XGrammar optimizes context-independent token masking. How should teams choose between these for different schema complexity profiles?
- How prevalent are SSTI vulnerabilities in LLM tooling? CVE-2025-25362 (spacy-llm) showed that template engines used for prompt construction can be exploited when user-controlled data reaches them unsandboxed. Are other LLM frameworks using default Jinja2, Mustache, or Handlebars environments without sandboxing?
- The proliferation of AI coding agents with rich markdown rendering (CVE-2026-22813 in OpenCode, CVE-2026-32626 in AnythingLLM) suggests that markdown-to-HTML conversion is a systematically underdefended attack surface. Should AI tools adopt a "safe-by-default" markdown renderer that strips all raw HTML, or would that break legitimate use cases?
- With the EU AI Act's high-risk obligations enforcing from August 2026, how should organizations document and demonstrate output handling controls for conformity assessments? Is there an emerging consensus on what constitutes adequate output validation evidence? The European Commission's support instruments for conformity assessment are expected in Q2 2026, which may clarify output validation documentation requirements.
- How should teams manage the security of the serving infrastructure that enforces output constraints? CVE-2026-27893 and CVE-2026-22778 (both vLLM, March 2026) demonstrated that the inference server itself can be compromised via malicious model repositories or crafted inputs, undermining any output format guarantees. Should constrained decoding enforcement run in a separate trust boundary from model inference?
- Custom markdown extensions — iframe renderers, chart renderers, math renderers — are emerging as a consistent bypass vector for LLM output sanitization. The April 2026 MaxKB CVEs (CVE-2026-39426 in `MdRenderer.vue`, CVE-2026-39423 in `EchartsRender`) both exploited custom tags that skipped the sanitizer applied to the base markdown. Should AI UI toolkits publish a standardized threat model for custom markdown components, including mandatory iframe sandbox policies (never `allow-scripts` + `allow-same-origin` together on LLM-controlled content) and sanitizer pass-through requirements?
- The Mermaid renderer cluster (CVE-2026-23733, CVE-2025-58768, CVE-2026-40322 across LobeChat, DeepChat, and SiYuan) demonstrates that diagram renderers accepting LLM output are a systemic attack surface in Electron-based AI apps. Should AI desktop application frameworks enforce `contextIsolation: true` and a strict preload API surface as baseline security requirements, rather than leaving Electron IPC exposure to individual developers? Is Mermaid's `securityLevel: 'strict'` mode sufficient, or should applications strip HTML from diagram content before it reaches Mermaid at all?
- The SiYuan Bazaar pair (CVE-2026-33066, CVE-2026-40922) extends the output-handling problem into AI tool ecosystems with marketplaces or model-pack hubs. When an AI application renders community-contributed content — package READMEs, prompt libraries, plugin descriptions — every author becomes a potential XSS-to-RCE vector. Should AI tool ecosystems publish a baseline marketplace security model that defaults to strict markdown sanitization, blocks iframes with `srcdoc` attributes, and assumes Electron desktop builds will not have `nodeIntegration` available?
- Unicode smuggling (FireTail, January 2026; AWS guidance 2026) demonstrates that LLM output can carry payloads invisible to human reviewers, downstream tokenizers, and conventional HTML sanitizers. Should output validation pipelines treat Unicode normalization (NFKC) and Variation-Selector / Tags-block filtering as a default first-pass step before any other sanitization or logging, on equal footing with HTML escaping? How should agentic systems handle the case where one LLM emits Unicode-smuggled content that is then consumed by another LLM as a tool result?
- With XGrammar now the default backend across vLLM, SGLang, and TensorRT-LLM, and SGLang's overlapped mask generation visibly outperforming vLLM at high batch sizes, is structured-output throughput becoming a deciding factor in inference-engine selection? When a workload depends on JSON-mode for downstream parsing, should constrained-decoding throughput be evaluated alongside raw token generation throughput in capacity planning, rather than treated as a separate concern?
- The CSV Agent cluster (Langflow CVE-2026-27966, Flowise CVE-2026-41264, Flowise CVE-2025-59528) shows that low-code AI agent builders repeatedly route LLM-generated code into non-sandboxed Python interpreters and rely on regex blocklists that are then bypassed. Should the broader agent-builder ecosystem adopt a baseline standard requiring (a) sandboxed execution by default for any node that consumes LLM output as code, (b) positive AST-level allowlists rather than regex blocklists, and (c) explicit per-node opt-in for any "dangerous code" mode? Is there a role for an OWASP profile specifically for agent-builder platforms similar to the Top 10 for LLM Applications?
- Anthropic's, Google's, and Microsoft's silent patches for the Comment and Control disclosures highlight a governance gap: critical agent vulnerabilities were paid as bug bounties but never assigned CVEs or GHSAs. How should organizations track and document agent runtime patches when the vendor does not publish advisories? Should procurement language for AI coding agents require GHSA-equivalent disclosure as a contractual obligation, especially for organizations preparing EU AI Act conformity assessments or ISO 42001 audits?
- Anthropic's published metric (1% attack success rate for Claude Opus 4.5 against an internal Best-of-N attacker) is the first time a frontier vendor has quantified prompt injection robustness. Should industry consensus form around a shared adversarial benchmark — analogous to JSONSchemaBench for constrained decoding — so that prompt injection robustness is comparable across models and over time? UK AISI's Inspect framework and the SandboxEscapeBench results suggest a path, but no single benchmark is yet treated as authoritative.
- The Tuui chain (CVE-2025-66562) showed that an `unsafe-eval` CSP combined with an unsanitized markdown code-block language and an exposed Electron IPC primitive produces RCE from a single LLM-emitted block. Should AI desktop framework templates ship with a default-deny CSP (no `unsafe-eval`, no `unsafe-inline`), an allowlist of safe markdown code-block languages, and a guarantee that the preload script never exposes generic `invoke`-style primitives to the renderer? Is there appetite for an "AI Electron baseline" similar to Electron's own existing security checklist but tuned for LLM-output rendering?
