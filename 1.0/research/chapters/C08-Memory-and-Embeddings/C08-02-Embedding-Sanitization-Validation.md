# C8.2: Embedding Sanitization & Validation

> [Back to C08 Index](C08-Memory-and-Embeddings.md)
> **Last Researched:** 2026-07-13

## Purpose

Pre-screen content before vectorization; treat memory writes as untrusted inputs; prevent ingestion of unsafe payloads. Embedding ingestion is a durable trust boundary: once text, metadata, or generated memory is converted into vectors, it can influence many future retrievals and is difficult to redact with confidence. As of June 2026, the practical attack surface has expanded from generic corpus poisoning into loader-stage obfuscation, retrieval-optimized indirect prompt injection, persistent memory poisoning, vector-store service vulnerabilities that allow writing around the ingestion pipeline, and AI-gateway supply-chain compromise that can silently rewrite what reaches the embedding step. [OWASP LLM08:2025](https://genai.owasp.org/llmrisk/llm082025-vector-and-embedding-weaknesses/) calls out vector data leakage, cross-context knowledge conflict, embedding inversion, and data poisoning as core risks, and the [OWASP Top 10 for Agentic Applications 2026](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/) elevates the same problem to **ASI06: Memory & Context Poisoning** for autonomous agents. Recent work reinforces the point: [PoisonedRAG](https://www.usenix.org/conference/usenixsecurity25/presentation/zou-poisonedrag) showed high-success knowledge corruption from a few injected texts; [The Hidden Threat in Plain Text](https://arxiv.org/abs/2507.05093) showed DOCX, HTML, and PDF loader manipulations surviving into RAG pipelines; [Overcoming the Retrieval Barrier](https://arxiv.org/abs/2601.07072) showed retrieval-optimized indirect prompt injection under natural queries; the May 2026 [Hidden in Memory](https://arxiv.org/abs/2605.15338) paper measured sleeper-style memory poisoning that stored false memories in 99.8% of GPT-5.5 tests and influenced harmful agentic actions in 60-89% of successful retrievals; and Microsoft's February 2026 [AI Recommendation Poisoning](https://www.microsoft.com/en-us/security/blog/2026/02/10/ai-recommendation-poisoning/) disclosure showed real-world attempts to plant persistent recommendations through hidden prompt parameters. The April 2026 [Cloud Security Alliance research note](https://labs.cloudsecurityalliance.org/research/csa-research-note-indirect-prompt-injection-in-the-wild-2026/) and [OWASP GenAI Q1 2026 exploit round-up](https://genai.owasp.org/2026/04/14/owasp-genai-exploit-round-up-report-q1-2026/) show the ingestion layer is now an operational target, with hidden web content, log entries, tool inputs, and compromised AI gateways used for exfiltration and unsafe actions. Mature controls exist for PII detection and basic content filtering, but outlier detection, contradiction review, and memory-write authorization still need corpus-specific baselines, provenance, quarantine workflows, and human review for high-impact stores. As of June 2026, [OWASP Agent Memory Guard](https://owasp.org/www-project-agent-memory-guard/) is the most concrete reference implementation for central memory read/write gates: it validates integrity with hashes, detects injection and sensitive-data leakage, enforces YAML policies, snapshots memory, and supports rollback. Its [v0.3.0 release](https://github.com/OWASP/www-project-agent-memory-guard/releases/tag/v0.3.0) (June 2026) shipped the promised LlamaIndex and CrewAI adapters plus several items pulled forward from the roadmap: an MCP server, a CLI scanner for offline audits, ML-based detection, a GitHub Action for CI scanning of agent memory, provenance-based memory classes (external_tool, user_input, agent_authored, system) with promotion rules, and a SelfReinforcementDetector. The promised Redis/PostgreSQL backends slipped out of v0.3.0, and vector-store protection remains Q3 2026 v0.4.0 work, so production teams still need wrapper code, central memory-write services, and human approval for the gaps it does not yet cover.

| **8.2.1** | **Verify that** sensitive fields are detected before embedding and are masked, tokenized, or dropped. | 1 | Sensitive data exposure through retained chunks, metadata, and vectors. This includes regulated PII, PHI, PCI data, credentials, customer secrets, and sensitive business identifiers that become searchable through the vector store. It maps to [OWASP LLM02:2025 Sensitive Information Disclosure](https://genai.owasp.org/llmrisk/llm02-sensitive-information-disclosure/) and [OWASP LLM08:2025 Vector and Embedding Weaknesses](https://genai.owasp.org/llmrisk/llm082025-vector-and-embedding-weaknesses/), which explicitly includes unauthorized access, embedding inversion, and data leakage risks. MITRE ATLAS tracks the credential-harvesting variant as `RAG Credential Harvesting`, where adversaries query the RAG store for tokens that leaked into ingested documents. Embedding inversion has also moved past the lab assumption that attackers need leaked (text, embedding) training pairs: the February 2026 [Zero2Text](https://arxiv.org/abs/2602.01757) attack reconstructs source text from stored vectors with no attacker training, combining LLM priors with iterative ridge regression against OpenAI embeddings, and its authors report that differential-privacy-style noise defenses fail against the adaptive variant; companion work on [conditional masked diffusion inversion](https://arxiv.org/abs/2602.11047) improves reconstruction fidelity over autoregressive inverters. A compromised or oversharded vector store should therefore be treated as recoverable plaintext for whatever sensitive data reached the embedding step. The May 14, 2026 [VectorSmuggle](https://www.helpnetsecurity.com/2026/05/14/vectorsmuggle-vector-embedding-security/) disclosure adds an exfiltration angle: a research framework from ThirdKey catalogs six steganographic techniques (noise, scaling, offset, rotation, and others) that hide data inside embeddings while preserving normal retrieval behavior, with rotation-based encoding carrying roughly 1,920 bytes per vector at zero error rate and evading off-the-shelf anomaly detectors. Sensitive data can therefore leave the network not only as raw text but as covert payload inside an ordinary embedding upsert. | Review the ingestion path before chunking and embedding: source connector, parser, text normalization, DLP scan, masking/tokenization step, metadata enrichment, embedding call, vector upsert, and audit log. Test with synthetic records containing SSNs, payment cards, email addresses, addresses, MRNs, API keys, access tokens, and internal project identifiers. Confirm detections are applied to extracted text, document metadata, table cells, image OCR output, and attachment names. Use [Microsoft Presidio](https://microsoft.github.io/presidio/text_anonymization/) analyzer and anonymizer evaluations, cloud DLP services such as Google Cloud Sensitive Data Protection, AWS Comprehend, or Microsoft Purview, and labeled false-negative review sets. Verify that masked text and metadata, not the original sensitive strings, are what reach the embedding provider and vector store. For exfiltration risk, add egress visibility for embedding traffic: log per-tenant vector upsert volumes, validate vectors against expected distribution baselines, and pilot cryptographic embedding signing such as the [VectorPin](https://www.helpnetsecurity.com/2026/05/14/vectorsmuggle-vector-embedding-security/) reference design (Python and Rust implementations) so that tampered or steganographically encoded vectors fail verification before reaching production indices. | PII tooling is mature for common identifiers but weaker for domain-specific secrets, internal codenames, and context-dependent sensitive facts. Presidio supports regex, deny lists, checksum, rule-based recognizers, NER, and context-aware detection, but custom recognizers are usually needed for enterprise data. The Spring 2026 Presidio releases added a [MedicalNERRecognizer](https://github.com/microsoft/presidio) for clinical entity detection, GPU acceleration through GLiNER/Transformers/Stanza engines, ONNX Runtime support, REST batch processing, and an Azure Health Data Services surrogate operator for realistic PHI replacement; the v2.2.362 release (March 2026, still current as of June 2026) also broadened multi-jurisdiction coverage with checksum-validated recognizers for Canadian SINs, German tax/health/passport identifiers, Korean RRNs, Swedish personnummer, and Thai and Turkish national IDs, plus a locale-scoped `countries` registry filter. These reduce false-negative rates for regulated ingestion but still need labeled enterprise evaluation sets. Masking can reduce retrieval quality, so teams need policy-specific choices between redaction, tokenization, deterministic hashing, surrogate replacement, and dropping the chunk entirely. Redaction must happen before embedding; deleting the source document later does not prove the semantic signal disappeared from derived vectors, replicas, caches, or backups. Existing DLP and CASB stacks do not currently inspect embedding payloads, so VectorSmuggle-style covert channels remain largely invisible to network defenders; cryptographic signing, per-tenant upsert telemetry, and inspection of the embedding pipeline as a regulated egress channel are still emerging controls rather than standard enterprise practice. Because blanket differential-privacy noise degrades retrieval utility and still falls to adaptive inverters like Zero2Text, two early-2026 defenses target the leakage more surgically and are worth piloting for high-sensitivity collections: [SPARSE](https://arxiv.org/abs/2602.07090) (Tsai et al., February 2026) identifies privacy-sensitive embedding dimensions for user-defined concepts and applies calibrated elliptical noise through a Mahalanobis mechanism, reporting lower privacy leakage with better downstream performance than standard DP across six datasets and three embedding models; [Detecting and Perturbing Privacy-Sensitive Neurons](https://openreview.net/forum?id=DF5TVzpTW0) (DPPN) learns a differentiable neuron mask and perturbs only the small set of privacy-carrying dimensions, cutting measured sensitive-information leakage to roughly 17% versus 43% for baseline noise while improving downstream task performance. Both remain research prototypes without production embedding-provider integration, so encryption-at-rest, strict store access control, and treating any breached vector store as recoverable plaintext stay the load-bearing controls. |
| **8.2.2** | **Verify that** vectors that fall outside normal clustering patterns are flagged and quarantined before entering production indices. | 2 | Adversarial embedding injection, vector collision, and high-centrality poisoning. A poisoned chunk may be crafted to land near attacker-chosen query neighborhoods or to be retrieved broadly across unrelated questions. [PoisonedRAG](https://www.usenix.org/conference/usenixsecurity25/presentation/zou-poisonedrag) achieved strong results by injecting a small number of malicious texts into large knowledge bases, while [CorruptRAG](https://arxiv.org/abs/2504.03957) focuses on practical single-text poisoning. [Hidden in the Metadata](https://arxiv.org/abs/2603.00172) extends the risk to multimodal RAG, showing metadata-only poisoning can steer retrieval even when the visible image content is unchanged. The April 2026 [Black-Hole Attack](https://arxiv.org/abs/2604.05480) adds a hubness-specific failure mode: vectors placed near the embedding-space centroid can become top-k neighbors for unrelated queries, so a poison record may look like an ordinary central vector rather than an outlier. Retrieval-optimized attacks mean the vector may look linguistically or visually plausible while still being geometrically or metadata-wise suspicious. | Establish baseline distributions per collection, tenant, source type, language, media type, metadata schema, and embedding model version. Before production upsert, score each candidate chunk against nearest-neighbor distance, local density, centroid distance, duplicate similarity, high-degree retrieval frequency, reverse-nearest-neighbor count, source reputation, metadata drift, and caption/OCR consistency. Test with synthetic poisoned documents targeting known canary queries and confirm quarantine triggers before the production index changes. For multimodal collections, compare extracted text, captions, alt text, EXIF-like metadata, OCR output, and source-system fields before embedding. Useful methods include isolation forest, DBSCAN or HDBSCAN, robust z-score on cosine distance, cleanlab-style out-of-distribution scoring, cross-encoder reranking disagreement, hubness scoring, and similarity search against a malicious-pattern fixture set; for the hubness family specifically, the February 2026 [adversarial hubness detection](https://arxiv.org/abs/2602.22427) work (open-source `hubscan`) statistically flags vectors that appear in top-k results for disproportionately many unrelated queries, directly operationalizing Black-Hole-style poison screening. Verify that vector stores such as Qdrant, Milvus, Weaviate, Pinecone, Vespa, or OpenSearch can support pre-commit staging collections and audit logs for rejected vectors; native support is improving — [Weaviate v1.38.0](https://github.com/weaviate/weaviate/releases/tag/v1.38.0) (June 2026) added preview namespaces with per-namespace RBAC, limits, and audit-log integration that work as a quarantine/staging primitive, and [Qdrant v1.18.0](https://github.com/qdrant/qdrant/releases/tag/v1.18.0) (May 2026) added per-event API method paths and tracing IDs to its audit log plus strict-mode write rejection. | Outlier detection catches geometric anomalies but misses semantic poisoning that sits inside a normal cluster. Centroid-near vectors can be dangerous specifically because they are central rather than distant, so quarantine rules should not rely only on far-out distance thresholds. Metadata-only attacks may leave the user-visible artifact clean, so reviewers need to inspect the fields that actually feed the embedding and retriever. Quarantine logic also assumes the production index can only be written through the sanctioned ingestion pipeline; 2026 vector-store vulnerabilities broke that assumption — [Milvus CVE-2026-26190](https://nvd.nist.gov/vuln/detail/CVE-2026-26190) (CVSS 9.8, fixed in 2.5.27/2.6.10) exposed the full REST API unauthenticated on the default metrics port, allowing silent corpus writes, and the unpatched ChromaDB "ChromaToast" pre-auth RCE ([CVE-2026-45829](https://www.securityweek.com/unpatched-chromadb-vulnerability-can-lead-to-server-takeover/)) allows full server takeover — so audits should confirm the store itself cannot be written around the staging path. Graph-based RAG raises a parallel concern: the May 2026 [A Few Words Can Distort Graphs](https://arxiv.org/abs/2508.04276) paper reports a targeted attack with 93.1% success rate and a universal variant that drops QA accuracy from 95% to 50% while modifying fewer than 0.05% of source text, and notes that current SOTA defenses miss these attacks because they live in entity/relation extraction rather than in vector geometry. Quarantine logic for graph-backed corpora therefore needs to inspect proposed triples, entity changes, and structural deltas before commit, not just vector neighborhoods. The May 2026 [ShadowMerge](https://arxiv.org/abs/2605.09033) attack sharpens this for graph-based agent memory specifically: it injects poisoned relations that share the same activation anchors and canonicalized relation channels as legitimate edges but carry conflicting values, so the memory system extracts, merges, and retrieves them as ordinary interactions (93.8% average attack success on Mem0 across PubMedQA, WebShop, and ToolEmu, +50.3 points over the best baseline), and the authors show representative input-side defenses are insufficient — meaning graph-memory quarantine must compare merged relation values against existing high-trust edges at commit time, not only validate each write in isolation. New product documentation, legal updates, incident reports, and rare languages may also look anomalous. Baselines must be segmented and versioned; a single global threshold usually creates both false positives and blind spots. Emerging defenses such as [RevPRAG](https://aclanthology.org/2025.findings-emnlp.698/), [Sparse Document Attention RAG](https://arxiv.org/abs/2602.04711), [RAGForensics](https://arxiv.org/abs/2504.21668), the August 2025 [Defending Against Knowledge Poisoning Attacks During RAG](https://arxiv.org/abs/2508.02835) paper, the December 2025 [RAGPart and RAGMask](https://arxiv.org/abs/2512.24268) retrieval-stage defenses (document partitioning with mean-pooled fragment embeddings, plus token-masking similarity shifts), the March 2026 [Semantic Chameleon](https://arxiv.org/abs/2603.18034) hybrid-retrieval defense (BM25 plus vector similarity reduced gradient-guided corpus poisoning from 38% to 0% on the Security Stack Exchange corpus without modifying the model), the [RAGuard non-parametric detection framework](https://arxiv.org/abs/2510.25025) (IEEE BigData 2025) which combines expanded retrieval scope, chunk-wise perplexity filtering, and similarity filtering to flag poisoned passages without retraining the retriever or generator, [ReliabilityRAG](https://arxiv.org/abs/2509.23519) (Shen et al.), which uses randomized aggregation over retrieval results to give a *certified* bound on how much injected content can move the final answer regardless of poisoning volume — useful when web-search retrieval makes per-document trust labels impractical — and [SeCon-RAG](https://arxiv.org/abs/2510.09710) (Si et al., NeurIPS 2025), whose first stage uses entity-intent-relation extraction plus semantic and K-means cluster-based filtering to flag poison documents that cluster too tightly around centroids — the same hubness signal Black-Hole exploits — while its second conflict-aware stage screens retrieved candidates for contradictions before generation, so it doubles as a bridge between this requirement and the 8.2.5 contradiction control, are useful complementary signals, but they do not replace pre-ingestion quarantine, structural review for graph stores, and per-tenant source review. |
| **8.2.3** | **Verify that** agent outputs and tool outputs are not automatically written to trusted agent memory without explicit source validation. | 2 | Persistent memory poisoning and self-reinforcing corruption. If generated summaries, tool results, or model conclusions are written into trusted memory without validation, one successful injection can become durable context for later sessions. This is the [OWASP Top 10 for Agentic Applications 2026 ASI06: Memory & Context Poisoning](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/) risk, and it maps to MITRE ATLAS [AML.T0051](https://atlas.mitre.org/) and [AML.T0080.000](https://atlas.mitre.org/). Microsoft's February 2026 [AI Recommendation Poisoning](https://www.microsoft.com/en-us/security/blog/2026/02/10/ai-recommendation-poisoning/) research observed more than 50 prompt-based attempts from 31 companies across 14 industries that tried to plant trusted-source recommendations across Copilot, ChatGPT, Claude, Perplexity, and Grok through prefilled assistant URLs. The May 2026 [Hidden in Memory: Sleeper Memory Poisoning](https://arxiv.org/abs/2605.15338) paper measured 99.8% storage success on GPT-5.5 and 95% on Kimi-K2.6, with 60-89% of successful retrievals influencing harmful agentic actions, demonstrating that poisoned memories can remain dormant and activate across later conversations. [MINJA](https://arxiv.org/abs/2503.03704) used query-only interaction to plant bridging steps in the memory bank with a 98.2% injection success rate and 76.8% attack success rate, while evading Llama Guard input/output moderation. The May 2026 [MemMorph](https://arxiv.org/abs/2605.26154) attack showed that as few as three injected memory records disguised as technical facts, incident reports, or operational policies can bias which tools an agent selects (up to 85.9% attack success across ten agent architectures and three memory implementations), and that it survives representative defenses because the poison reshapes contextual understanding rather than tool metadata — so tool-description audits alone miss it. [ER-MIA](https://arxiv.org/abs/2602.15344) frames similarity-based long-term memory retrieval itself as a system-level vulnerability, [A-MemGuard](https://arxiv.org/abs/2510.02373) describes why isolated memory-entry review is weak when malicious effects only activate in a later context, the April 2026 [Long-Term Memory in LLM Agents survey](https://arxiv.org/abs/2604.16548) catalogs the broader attack surface across ingestion, storage, retrieval, and behavior monitoring, and the March 2026 [Taming OpenClaw](https://arxiv.org/abs/2603.11619) paper from Ant Group and Tsinghua proposes a five-layer lifecycle framework (initialization, input, inference, decision, execution) for autonomous LLM agents, treating memory poisoning, indirect prompt injection, skill supply-chain contamination, and intent drift as compound threats that point-based defenses miss. The June 2026 [From Untrusted Input to Trusted Memory](https://arxiv.org/abs/2606.04329) study (Dash et al., arXiv:2606.04329) systematizes this surface, mapping four memory write channels and nine structural vulnerabilities into a taxonomy of six memory-poisoning attack classes, releasing the MPBench benchmark, and confirming both that agents tuned to write and retrieve memory more aggressively are more exploitable and that existing prompt-injection defenses do not cover memory poisoning — a direct argument for treating memory writes as their own trust boundary with dedicated validation. Two May 2026 attacks show why screening only the *stored* payload is insufficient: [Hijacking Agent Memory](https://arxiv.org/abs/2605.29960) (Wang et al., arXiv:2605.29960; the MemPoison attack) plants triggerable backdoors through ordinary dialogue that survive the selective extraction and rewriting stages of modern memory pipelines, and [Trojan Hippo](https://arxiv.org/abs/2605.01970) (Das et al., arXiv:2605.01970) plants a dormant payload via a single untrusted interaction (e.g., a crafted email reaching memory through a tool call) that stays inert until the user later discusses finance, health, or identity, then exfiltrates personal data — reaching 85-100% attack success against current OpenAI and Google frontier models and still firing after 100 benign sessions across explicit-tool, agentic, RAG, and sliding-window memory designs. The same paper reports that four memory defenses cut success to 0-5% but at substantial utility cost, underscoring that write-time provenance and authorization, not post-hoc filtering, are the load-bearing controls. | Enumerate every path that can write to long-term memory, vector stores, profiles, summaries, preference stores, and shared agent state. Confirm each write records origin, author, source URI, source trust tier, extraction method, review status, timestamp, tenant, and approving principal. Block auto-promotion of generated content into trusted memory unless a separate validator checks source provenance, policy compliance, contradiction risk, and write authorization. Test with prompts and retrieved documents that ask the system to remember a vendor, override a policy, promote a source as authoritative, persist a hidden instruction, or insert a memory that only activates under a later question. Include content-based, question-targeted, MINJA-style bridging-step, and sleeper memory fixtures, then verify they remain quarantined across session restart, compaction, summarization, and retrieval. Review logs for memory writes containing terms such as "remember", "trusted source", "from now on", "future conversations", "always cite", and "authoritative". Require human approval or policy-engine approval for high-impact memory writes, and evaluate memory middleware such as [OWASP Agent Memory Guard](https://owasp.org/www-project-agent-memory-guard/) for policy enforcement, snapshotting, rollback, and protected-key tamper detection — its June 2026 [v0.3.0 release](https://github.com/OWASP/www-project-agent-memory-guard/releases/tag/v0.3.0) added provenance-based memory classes with promotion rules, ML-based detection, a CLI scanner, an MCP server, and a GitHub Action so memory snapshots can be scanned in CI. For managed stacks, [Policy in Amazon Bedrock AgentCore](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/policy.html) (GA March 2026) attaches Cedar policy engines to AgentCore gateways that deterministically evaluate every agent-to-tool call — including memory tools — against principal identity and tool input parameters before execution, with decisions logged to CloudWatch. For consumer assistants, also verify the platform mitigations OpenAI and Google shipped through 2025-2026, such as memory-write classifiers, confirmation prompts for sensitive memory operations, tool-output sandboxing, and Model Armor inspection. | Product memory features often blur preferences, summaries, user-authored facts, and model-inferred facts. That makes origin tagging and trust tiers more important than the model's confidence. Validation should run outside the same prompt context that produced the candidate memory. A-MemGuard-style consensus validation, dual-memory lessons, and Agent Memory Guard-style read/write interception are promising, but MINJA and Sleeper Memory Poisoning both bypass single-pass moderation, so production teams still need explicit write permissions, reviewable memory ledgers, rollback, and per-user or per-tenant scoping. Composite trust scoring across multiple orthogonal signals and temporal-decay sanitization, as proposed in the January 2026 [Memory Poisoning Attack and Defense](https://arxiv.org/abs/2601.05504) paper, are useful templates but still require curated baselines per deployment. |
| **8.2.4** | **Verify that** content crafted to manipulate retrieval results is detected and rejected or quarantined before vectorization. | 3 | RAG poisoning, indirect prompt injection, and loader-stage content manipulation. Attackers can hide instructions in HTML, PDF, DOCX, comments, CSS, white text, Unicode confusables, OCR layers, metadata, or crafted text that is optimized to be retrieved for target queries. [The Hidden Threat in Plain Text](https://arxiv.org/abs/2507.05093) reported a 74.4% attack success rate across 357 loader attack scenarios using content obfuscation and content injection against common document loaders. [Hidden-in-Plain-Text](https://arxiv.org/abs/2601.10923) introduced OpenRAG-Soc for web-native RAG attacks that survive HTML, Markdown, Unicode, and social-content ingestion paths. [Overcoming the Retrieval Barrier](https://arxiv.org/abs/2601.07072) showed that trigger fragments can make malicious content reliably retrieved under natural queries, and [Machine Against the RAG](https://www.usenix.org/conference/usenixsecurity25/presentation/shafran) showed that a single blocker document can suppress answers without relying on explicit instruction injection. [One Shot Dominance](https://arxiv.org/abs/2505.11548) demonstrated that a single stylistically legitimate poisoned document per target can steer answers even on multi-hop questions, defeating injection-rate anomaly thresholds, and [Chain-of-Thought Poisoning](https://arxiv.org/abs/2505.16367) showed that corpus documents carrying fabricated reasoning chains can hijack reasoning-model RAG pipelines through the model's chain-of-thought rather than surface factual claims — so intake screening needs to look for embedded reasoning templates, not only falsehoods and hidden instructions. The May 2026 [M3Att](https://arxiv.org/abs/2605.10253) attack extended query-agnostic poisoning to medical multimodal knowledge bases, aligning PGD-perturbed images to knowledge-base cluster centers and pairing them with clinically plausible text corruption that evades common distributional-filtering defenses. April 2026 operational reporting from the [Cloud Security Alliance](https://labs.cloudsecurityalliance.org/research/csa-research-note-indirect-prompt-injection-in-the-wild-2026/) connects the same hidden-content patterns to live payloads against browsing agents, coding assistants, and enterprise copilots. A harder-to-screen variant is content that is *factually accurate* yet crafted to skew retrieval: [Epistemic Bias Injection](https://arxiv.org/abs/2512.00804) (Wu & Saxena, arXiv:2512.00804, revised March 2026) injects truthful passages that systematically crowd out alternative viewpoints on multi-perspective questions, achieving significant perspective shifts while bypassing retrieval-based sanitization that looks for falsehoods or hidden instructions — a direct reason this requirement screens for retrieval manipulation generally, not only for malicious-looking text, and part of why it now sits at Level 3. MITRE ATLAS tracks the related families as [AML.T0051 Prompt Injection](https://atlas.mitre.org/), [AML.T0070 RAG Poisoning](https://atlas.mitre.org/), and [AML.T0080 AI Agent Context Poisoning](https://atlas.mitre.org/). | Build an intake test suite for every accepted format. Compare raw text extraction, rendered view, OCR output, metadata, links, alt text, comments, and hidden layers before embedding. Normalize Unicode, strip or flag zero-width characters, reject invisible or off-canvas text, limit embedded links and executable content, and quarantine chunks containing instruction-like text aimed at the downstream model. Test with fixture documents that include hidden prompts, white-on-white text, tiny fonts, HTML comments, PDF annotation layers, DOCX comments, homoglyphs, base64-like payloads, suspicious URL prompt parameters, adversarial repetitions of target-query terms, and OpenRAG-Soc-style web markup carriers. For application code, verify that LangChain, LlamaIndex, Haystack, or custom loaders pass through a security transformation before vector upsert. Use [LangChain's indirect prompt injection guidance](https://docs.langchain.com/oss/python/langchain/rag), [NVIDIA NeMo Guardrails retrieval rails](https://docs.nvidia.com/nemo/guardrails/latest/about/rail-types.html), custom prompt-injection classifiers, static document scanners, and manual review queues for high-impact collections. NeMo Guardrails [v0.22.0](https://github.com/NVIDIA-NeMo/Guardrails/releases/tag/v0.22.0) (May 2026) is worth re-evaluating for ingestion pipelines specifically: its standalone `check_async` input/output validation and parallel NemoGuard rails let teams run injection and content-safety screening on document chunks pre-vectorization without standing up a full conversation flow. For the epistemic-bias variant that survives content filters, pair pre-ingestion screening with the post-retrieval [BiasDef](https://arxiv.org/abs/2512.00804) filter, which scores candidate passages against a geometric bias metric computed directly on text embeddings and removes those that skew a multi-viewpoint result before generation. | No parser-level defense is complete because attackers can target differences between raw extraction, rendered display, OCR, and model interpretation. Defensive prompts and delimiters help at generation time but do not sanitize a poisoned vector store. Treat sanitization as a gate: untrusted documents should not be embedded until the original file, extracted text, metadata, and source provenance have passed policy checks. Live web payloads increasingly use ordinary fields such as comments, attributes, CSS-suppressed text, and encoded fragments, so tests need to cover real loader behavior rather than only clean plaintext chunks. Mature commercial tooling for document-intake poisoning is still emerging, so teams should keep adversarial loader fixtures in CI and require human review for quarantine releases. |
| **8.2.5** | **Verify that** new content written to memory is checked for contradictions with what is already stored and that conflicts trigger alerts. | 3 | Silent override of stored truth: a poisoned, stale, or forged claim that contradicts an existing golden record can quietly displace it during summarization, compaction, or memory promotion unless conflicts are detected and surfaced. The contradictory content usually arrives through tool-result poisoning, orchestration-result poisoning, AI-gateway compromise, and durable contamination of agent memory. A compromised MCP server, AI gateway, browser page, search result, code-analysis output, email, ticket, or nested agent result can carry hidden instructions, forged tool responses, or false claims that look like normal operational data. If that output is summarized into memory, promoted into a vector store, or saved as a "successful" trajectory, the compromise can survive the original run. The March 2026 [TeamPCP LiteLLM supply-chain compromise](https://securitylabs.datadoghq.com/articles/litellm-compromised-pypi-teampcp-supply-chain-campaign/) showed that malicious PyPI releases of an AI gateway can harvest credentials and pivot to Kubernetes clusters from inside the proxy that mediates retrieval and tool calls, and the April 2026 [LiteLLM CVE-2026-42208 SQL injection](https://docs.litellm.ai/blog/cve-2026-42208-litellm-proxy-sql-injection) (added to the CISA Known Exploited Vulnerabilities catalog on May 8, 2026) showed that an unauthenticated header can reach the gateway's authentication database, exposing provider keys and configuration that govern downstream memory writes. The April-May 2026 [MCP stdio command-injection disclosure (CVE-2026-30623)](https://docs.litellm.ai/blog/mcp-stdio-command-injection-april-2026) widened the same surface: OX Security showed that the official MCP SDKs' stdio transport executes attacker-influenced server commands across an ecosystem of 150M+ downloads, with demonstrated exploitation in LiteLLM, LangChain, and LangFlow, and Anthropic treating the behavior as by-design — pushing sanitization responsibility onto every gateway and agent-stack integrator. [OWASP MCP Tool Poisoning](https://owasp.org/www-community/attacks/MCP_Tool_Poisoning) describes the runtime trust gap for tool responses; the latest [MCP tool specification](https://modelcontextprotocol.io/specification/2025-11-25/server/tools) requires servers to sanitize tool outputs, supports output schemas for structured results, and recommends that clients validate tool results before passing them to the model. For each path, verify that writes are denied by default unless the candidate memory has source URI, tool/server identity, tool version or descriptor hash, invocation ID, user and tenant, timestamp, content hash, trust tier, sensitivity classification, and approving principal. Pin AI-gateway versions, monitor PyPI/npm advisories for proxy components (LiteLLM, Telnyx, similar packages), and treat any embedding upserts or memory writes that touched a compromised gateway window as quarantine candidates rather than trusted facts. Test with malicious tool fixtures that return valid-looking JSON plus hidden prompt text, poisoned webpage observations, forged "trusted source" claims, and nested-agent summaries that ask to be remembered. Confirm schema validation rejects unexpected fields, structured MCP outputs conform to the advertised output schema, free-text directives are quarantined, and high-impact writes pause for approval outside the model context. Useful implementation hooks include [OpenAI Agents SDK tool guardrails](https://openai.github.io/openai-agents-python/guardrails/), [OpenAI hosted MCP approval flows](https://openai.github.io/openai-agents-python/mcp/), [LangGraph long-term stores](https://docs.langchain.com/oss/python/langchain/long-term-memory) wrapped by application authorization, LangGraph persistence and interrupts for human review, [NVIDIA NeMo Guardrails execution rails](https://docs.nvidia.com/nemo/guardrails/latest/configure-guardrails/yaml-schema/guardrails-configuration), and the [Microsoft Agent Governance Toolkit](https://github.com/microsoft/agent-governance-toolkit), which ships an `McpSecurityScanner` for tool-poisoning, typosquatting, and hidden-instruction detection, an `McpResponseSanitizer` that strips prompt-injection patterns, credentials, and exfiltration URLs from tool output before re-entry into the model, episodic memory integrity checks, and a YAML-policy `GovernanceKernel` with audit and OpenTelemetry integration that sits between MCP clients and tool servers and evaluates each tool call before execution; its June 2026 [v4.0.0/v4.1.0 releases](https://github.com/microsoft/agent-governance-toolkit/releases/tag/v4.1.0) added Ed25519-verified agent registration, escrow-signed proof-of-outcome, dynamic Cedar policy conditions with time- and cost-aware rules, expanded credential redaction across four languages, and a LangGraph v1.0 governance adapter. Tool outputs are often designed to be consumed by the model, not stored as durable facts, so memory-write policy has to be stricter than ordinary context-injection policy. Schema validation helps but does not prove a claim is true, and a valid tool response may still contain adversarial natural language in a legitimate field. Hosted tools, handoffs, browser observations, and background summarizers may bypass the same guardrail hooks used for custom function tools, so teams need a central memory-write service rather than scattered `store.put()` calls. AI gateways that mediate embedding and memory traffic are themselves a supply-chain risk, so signed releases, pinned versions, integrity checks on the proxy binary, and rollback procedures for embeddings written during a compromise window need to be part of the program. [NIST AI 600-1](https://www.nist.gov/publications/artificial-intelligence-risk-management-framework-generative-artificial-intelligence) reinforces the need for provenance tracking, third-party monitoring, and incident response for GAI value-chain components; in practice, this means reviewable memory ledgers, rollback by batch or source, per-tenant scoping, and regular red-team tests for tool-to-memory poisoning. |

## Notable Incidents and Research

- February 2026: Microsoft reported AI Recommendation Poisoning attempts where public "Summarize with AI" links used prompt parameters to plant persistent trusted-source recommendations in assistant memory, including more than 50 observed prompt patterns from 31 companies across 14 industries. Security teams should hunt for assistant URLs with `q` or `prompt` parameters containing memory terms such as "remember", "trusted", "authoritative", "future", "citation", or "cite". [Microsoft Security](https://www.microsoft.com/en-us/security/blog/2026/02/10/ai-recommendation-poisoning/)
- March 2026: MM-MEPA showed that multimodal RAG poisoning can live entirely in image-text metadata while leaving visible image content unchanged, which makes metadata provenance and caption/OCR consistency checks part of embedding sanitization rather than a separate multimedia concern. [Hidden in the Metadata](https://arxiv.org/abs/2603.00172)
- April 2026: eTAMP showed that a single contaminated web observation can poison a browsing agent's long-term memory and later activate across different sites and sessions. C8.2.3 and C8.2.5 tests should therefore include environmental observations, not only direct memory-write APIs. [Poison Once, Exploit Forever](https://arxiv.org/abs/2604.02623)
- April 2026: the Black-Hole Attack showed that vectors injected near the embedding-space centroid can become frequent top-k neighbors across broad query sets, with reported malicious appearances in up to 99.85% of top-10 retrievals. C8.2.2 audits should therefore include hubness, reverse-nearest-neighbor, centroid-distance, and query-saturation metrics, not only ordinary outlier scores. [Black-Hole Attack](https://arxiv.org/abs/2604.05480)
- March-May 2026: the [TeamPCP LiteLLM supply-chain compromise](https://securitylabs.datadoghq.com/articles/litellm-compromised-pypi-teampcp-supply-chain-campaign/) (March 24, 2026 PyPI 1.82.7/1.82.8 with credential stealer and Kubernetes lateral movement) and the subsequent [LiteLLM CVE-2026-42208 SQL injection](https://docs.litellm.ai/blog/cve-2026-42208-litellm-proxy-sql-injection) (CVSS 9.3, CISA KEV May 8 2026, exploited within 36 hours of disclosure) demonstrated that the AI gateway between application code and embedding/memory APIs is itself a poisoning surface. C8.2.5 audits should pin gateway versions, validate release signatures, monitor PyPI/npm/Docker advisories for proxy components, and treat embedding upserts during a known compromise window as quarantine candidates rather than trusted records.
- May 2026: [Hidden in Memory: Sleeper Memory Poisoning](https://arxiv.org/abs/2605.15338) reported 99.8% memory-storage success on GPT-5.5 and 60-89% influence over harmful agentic actions in successful retrievals, confirming that sleeper-style memory poisoning bypasses single-pass moderation. C8.2.3 tests should include dormant-then-active fixtures that only manifest under specific later contexts, not only first-turn injection attempts.
- March 2026: [Semantic Chameleon](https://arxiv.org/abs/2603.18034) showed that gradient-guided dual-document (sleeper + trigger) corpus poisoning succeeds 38% of the time with vector-only retrieval but drops to 0% across 50 attacks when retrieval combines BM25 and vector similarity, with no model retraining required. Adaptive attackers who jointly optimize for sparse and dense signals still reach 20-44%, so hybrid retrieval should be paired with pre-ingestion quarantine and per-tenant source review rather than treated as a standalone control.
- April 2026: Microsoft released the open-source [Agent Governance Toolkit](https://github.com/microsoft/agent-governance-toolkit) and a [.NET integration walkthrough](https://devblogs.microsoft.com/dotnet/governing-mcp-tool-calls-in-dotnet-with-the-agent-governance-toolkit/) that provides MCP tool-poisoning scanning, response sanitization, memory integrity checks, and YAML-policy enforcement across LangChain, AutoGen, CrewAI, Semantic Kernel, OpenAI Agents SDK, and Google ADK. It is one of the first runtime control-plane references that maps explicitly to all ten OWASP Top 10 for Agentic Applications categories, including ASI06 Memory and Context Poisoning, and is useful as a verification baseline for C8.2.3 and C8.2.5.
- February-May 2026: two vector-store vulnerabilities showed that production indices can be poisoned by writing around the ingestion pipeline entirely. [Milvus CVE-2026-26190](https://nvd.nist.gov/vuln/detail/CVE-2026-26190) (CVSS 9.8, fixed in 2.5.27/2.6.10) exposed the full REST API without authentication on the default metrics port 9091, allowing arbitrary vector data manipulation; the ChromaDB "ChromaToast" pre-auth RCE ([CVE-2026-45829](https://www.securityweek.com/unpatched-chromadb-vulnerability-can-lead-to-server-takeover/), disclosed May 2026) lets unauthenticated attackers load a malicious model identifier during collection creation and take over the server, and remained unpatched as of 1.5.8 with roughly 73% of internet-exposed deployments vulnerable. C8.2.2 quarantine logic should assume the store itself is an attack surface and verify network exposure, authentication on every port, and write-path allowlisting.
- May 13, 2026: the OWASP GenAI Security Project published [Memory Is a Feature. It Is Also an Attack Surface](https://genai.owasp.org/2026/05/13/memory-is-a-feature-it-is-also-an-attack-surface/), detailing Cisco's "MemoryTrap" finding in Claude Code v2.1.50 where payloads in routine developer workflows reached persistent memory, global hooks configuration, and system prompts — a one-time action shaping behavior across sessions and reboots. It is a concrete reminder that C8.2.3 write validation must cover configuration files and hooks, not just designated memory APIs.
- May 2026: [MemMorph](https://arxiv.org/abs/2605.26154) demonstrated tool hijacking via memory poisoning — three injected records disguised as ordinary technical facts or incident reports biased agent tool selection with up to 85.9% success across ten agent architectures, surviving defenses that audit tool descriptions because the poison lives in contextual memory instead. C8.2.3 fixtures should include memories that change which tool gets selected, not only memories that change factual answers.
- May 14, 2026: ThirdKey published the [VectorSmuggle](https://www.helpnetsecurity.com/2026/05/14/vectorsmuggle-vector-embedding-security/) research framework (Apache 2.0), which catalogs six embedding-steganography techniques (noise, scaling, offset, rotation, and others) that hide exfiltrated data inside ordinary embeddings shipped to embedding providers and vector databases. Rotation-based encoding moved roughly 1,920 bytes per vector with zero retrieval error and bypassed off-the-shelf anomaly detectors. The companion [VectorPin](https://www.helpnetsecurity.com/2026/05/14/vectorsmuggle-vector-embedding-security/) reference cryptographically signs each embedding at creation time so that any later perturbation breaks the signature; Python and Rust implementations exist. C8.2.1 audits should add embedding egress telemetry, per-tenant upsert distribution baselines, and a pilot signing path for high-trust collections.
- May 2026: [Trojan Hippo](https://arxiv.org/abs/2605.01970) (Das et al., UC Berkeley/ETH Zurich) weaponized agent memory for targeted data exfiltration — a single untrusted interaction plants a dormant payload that stays inert until the user discusses finance, health, or identity, then exfiltrates personal data, reaching 85-100% attack success against current OpenAI and Google frontier models and surviving 100 benign sessions across explicit-tool, agentic, RAG, and sliding-window memory designs. Four memory defenses cut success to 0-5% but with significant utility tradeoffs, so C8.2.3 write-validation fixtures should include topic-triggered dormant payloads and measure the utility cost of any defense actually enabled in production.
- May 2026: [ShadowMerge](https://arxiv.org/abs/2605.09033) (Luo et al.) attacked graph-based agent memory through relation-channel conflicts — poisoned relations that share activation anchors and canonicalized channels with legitimate edges but carry conflicting values — reaching 93.8% average attack success on Mem0 (PubMedQA, WebShop, ToolEmu) and defeating representative input-side defenses. C8.2.2 quarantine for graph-backed stores should compare merged relation values against existing high-trust edges at commit time, not validate each write in isolation.
- June 2026: [From Untrusted Input to Trusted Memory](https://arxiv.org/abs/2606.04329) (Dash et al.) published the first systematic taxonomy of memory poisoning — four write channels, nine structural vulnerabilities, six attack classes — plus the MPBench benchmark, confirming that agents tuned to write/retrieve memory more aggressively are more exploitable and that existing prompt-injection defenses do not cover memory poisoning. It is a useful baseline for scoping C8.2.3/C8.2.5 test coverage against named write channels rather than ad hoc fixtures.
- June 2026: [SMSR](https://arxiv.org/abs/2606.12703) evaluated signed memory plus randomized retrieval ablation across 15 enterprise scenarios. HMAC-SHA256 provenance blocked unsigned inserts, but the authors also tested the harder case where the agent itself wrote a poisoned trace through the authorized signing path; combined retrieval defenses reduced that query-only attack from 65.3% to 5.3% in 150 trials. The result is a useful C8.2.3 test pattern, not production validation: signatures establish which write path produced a memory, not whether an authorized writer stored trustworthy content.
- July 2026: [Knowledge Base Poisoning Attacks and Defense for Policy-Aware LLM-RAG](https://arxiv.org/abs/2607.04379) showed a query-agnostic poison rule placed near several policy-category neighborhoods. One injected rule in a 60-rule IoBT corpus appeared in retrieved context for 17 of 20 prompts; DBSCAN and LOF treated the poison as an inlier, while a category-spread signal complemented One-Class SVM. The experiment is a narrow preprint and its CLD-KB defense filters after retrieval rather than before vectorization, but its failure mode belongs in C8.2.2 staging tests: centroid distance and generic outlier scores need a domain-taxonomy or source-consistency signal before promotion to production.
- July 2026: [GhostWriter](https://arxiv.org/abs/2607.06595) poisoned five long-term-memory agent architectures through ordinary email or calendar inputs, averaging about 98% memory admission and 60% later activation across four model families. Its AM-Sentry prototype scores origin, trust, memory type, utility, contradiction, and verifiability before admission, then screens again at retrieval. For C8.2.3, the important regression is two-session and tool-mediated: an untrusted message is summarized or transformed into memory, then a separate benign request retrieves it and attempts an unauthorized action.
- July 2026: [CVE-2026-49471](https://nvd.nist.gov/vuln/detail/CVE-2026-49471) turned an exposed local dashboard into a direct persistent-memory write path in Serena versions before 1.5.2. The [vendor advisory](https://github.com/oraios/serena/security/advisories/GHSA-37h2-6p4f-mp3q) documents how DNS rebinding could reach an unauthenticated `/save_memory` endpoint and plant content that the coding agent later consumed; chaining that memory to the default shell tool produced remote code execution. This is a concrete C8.2.3 case where content screening alone is irrelevant because the attacker bypasses the intended writer. Audits should authenticate every memory-management surface, validate `Host` and origin boundaries for local web services, disable unused dashboards, and prove that only the central admission service can create trusted records.

## Implementation Maturity

- Mature: regulated-data detection before embedding is well supported for common identifiers through DLP services, Presidio-style recognizers, checksum validators, and custom enterprise dictionaries.
- Early: egress visibility for embedding traffic and cryptographic signing of upserts. VectorSmuggle showed that embedding providers and vector databases sit outside the inspection scope of existing DLP, CASB, and proxy stacks; VectorPin-style signing, per-tenant upsert telemetry, and embedding-distribution baselines are reference implementations rather than enterprise defaults.
- Developing: loader-stage poisoning defenses are practical when teams maintain adversarial fixtures for each accepted format, but coverage still depends heavily on the exact parser, renderer, OCR engine, and metadata fields used in production.
- Developing: vector quarantine can be implemented with staging collections, source-trust labels, tenant partitions, and audit logs, while robust detection of high-centrality or semantically normal poisoned vectors remains research-heavy.
- Developing: the [OWASP RAG Security Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/RAG_Security_Cheat_Sheet.html) now supplies a practical C8.2.2/C8.2.4 baseline: hash and provenance every document at ingestion, monitor embedding distributions and model-version drift, compare rankings across embedding models for high-security collections, restrict index writes to the ingestion pipeline, and retain snapshots and modification logs. It correctly treats cross-model validation as an advanced control, but does not provide thresholds or benchmark data; teams still need corpus-specific acceptance bands and false-positive measurements before using rank disagreement to quarantine records.
- Developing: agent memory write authorization is moving from purely application-defined toward reusable middleware. Strong implementations use a central memory-write service with provenance, trust tiering, reviewer identity, rollback, and policy checks outside the same prompt context that produced the candidate memory. [OWASP Agent Memory Guard v0.3.0](https://github.com/OWASP/www-project-agent-memory-guard/releases/tag/v0.3.0) (June 2026) now ships LangChain, LlamaIndex, and CrewAI adapters, provenance-based memory classes with promotion rules, ML-based detection, a SelfReinforcementDetector, a CLI scanner, an MCP server, and a CI GitHub Action — independent benchmark coverage reported 92.5% recall at 100% precision on project test sets — while Redis/PostgreSQL backends slipped from the release and vector-store protection remains Q3 2026 v0.4.0 work. Cloud-managed alternatives like Cedar-based [Policy in Amazon Bedrock AgentCore](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/policy.html) cover the gateway-enforced slice. Multi-framework fleets and vector-store coverage still need additional wrappers and manual approval.
- Experimental: signed-memory schemes and retrieval-time randomized ablation provide a stronger test target than content filters alone. [SMSR](https://arxiv.org/abs/2606.12703) demonstrates why auditors should test both unsigned store tampering and poisoned content written by an authorized, signing agent; only the first is stopped by provenance verification, while the second still needs admission policy, trust-scoped retrieval, and impact-limiting aggregation. The published results use one prototype and synthetic enterprise scenarios, so key rotation, replay protection, multi-tenant key separation, latency, and recovery behavior remain deployment-specific gaps.
- Developing: category-aware geometric checks can catch poison that ordinary far-outlier detectors miss. The July 2026 [CLD-KB study](https://arxiv.org/abs/2607.04379) combined a global boundary detector with distance spread across known policy categories, but evaluated only a 60-rule, three-category IoBT corpus and applied filtering after retrieval. Production C8.2.2 controls need to move equivalent signals into the staging gate, preserve source and category labels, and measure false positives on legitimate records that naturally span several categories.
- Early: contradiction detection helps with explicit factual conflicts but still needs curated golden sources, source precedence rules, and reviewer workflows for subtle framing, stale facts, and domain-specific policy conflicts.
- Early: shared-memory contradiction handling also has pipeline-ordering hazards. The June 2026 [Governed Shared Memory](https://arxiv.org/abs/2606.24535) evaluation found that a synchronous near-duplicate gate could reject a conflicting candidate before an asynchronous contradiction detector examined it. C8.2.5 implementations should make every admitted or rejected write observable to contradiction analysis, preserve provenance through temporal supersession, and test direct get-by-ID paths as well as normal scoped retrieval; the paper reports that tenant isolation initially held while an agent-scoped credential could still bypass sub-tenant scope through direct lookup.

## Audit Checks to Add to Test Fixtures

- Loader-stage poisoning fixtures should place the same instruction in visible text, CSS-hidden HTML, DOCX comments, PDF annotations, OCR layers, metadata fields, and Unicode-obfuscated text, then compare what reviewers, scanners, retrievers, and model context each see before vectorization.
- Vector-stage fixtures should include centroid-near "black-hole" candidates, duplicate near-neighbor clusters, metadata-only multimodal poison records, and query-saturation probes so quarantine logic measures hubness, reverse-nearest-neighbor count, source trust, and retrieval dominance.
- Add an embedding-drift fixture based on the [OWASP RAG Security Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/RAG_Security_Cheat_Sheet.html): record the model version and baseline neighborhood for clean, poisoned, and legitimate cross-domain documents; re-embed them with a second model and after a model upgrade; then verify that abnormal rank or neighborhood shifts are reviewed in staging rather than silently promoted. Preserve the compared vectors, thresholds, decision, and false-positive result so an auditor can reproduce the quarantine outcome.
- Memory-stage fixtures should use an environmental observation, generated summary, MCP tool result, and delegated-agent result that each tries to persist a vendor, source, policy, or future instruction as trusted memory; passing tests should keep the write quarantined across restart, summarization, and retrieval.
- Add an indirect two-session memory fixture modeled on GhostWriter: deliver an untrusted email or calendar item containing a hidden workflow, contact, or exfiltration directive; verify that summarization preserves its low-trust provenance, and then issue a separate benign task that would activate it. Passing systems should deny trusted admission or block retrieval and record the source, transformed memory, policy decision, and attempted action.
- Exercise signed-memory controls twice: first insert an unsigned record directly into the store, then cause the authorized agent to write a poisoned record through its normal signed path. The first must fail integrity validation; the second must still face source validation and retrieval screening, proving that a valid signature is not treated as evidence that the content is safe.
- Add cross-category semantic-bridge vectors that sit near several legitimate category neighborhoods, alongside benign records that genuinely span those categories. Before production upsert, compare generic distance and density detectors with category-spread, source-consistency, and retrieval-frequency signals, and retain both poison-detection and false-positive measurements.
- Contradiction fixtures should introduce lower-trust content that conflicts with a golden policy, owner, entitlement, deadline, or approved source ranking, then verify the alert records provenance, affected chunks, retrieval impact, reviewer decision, and rollback batch.
- Run contradiction fixtures through each earlier admission branch, including duplicate suppression, schema rejection, rate limiting, and quarantine. A passing system either evaluates the candidate for conflict before rejection or emits an equivalent conflict event; it must not let an earlier short-circuit hide an attempted override from C8.2.5 monitoring.

## Community Notes

- As of May 2026, loader-stage validation should diff the raw file, rendered view, extracted text, OCR text, metadata, and embedded resource list before any vector upsert. The practical test is whether the same hidden instruction is visible to reviewers, scanners, and the retriever; if not, quarantine the file and record which extractor produced each chunk.
- For memory validation, research prototypes are starting to converge on explicit trust scoring rather than prompt-only filtering. [SuperLocalMemory](https://arxiv.org/abs/2603.02240) is useful as an audit pattern because it separates behavioral data, records per-agent provenance, and evaluates trust degradation against sleeper-style memory poisoning; production systems still need the same ideas implemented in their central memory-write service.
- Memory poisoning research is moving from prompt-like persistence into action-plan poisoning. A March 2026 ScienceDirect paper on [Deceptive Semantic Reasoning Manipulation](https://www.sciencedirect.com/science/article/pii/S0952197626002496) shows forged historical knowledge can steer agent decisions and tool selection, so memory-write review should inspect proposed actions, not only factual claims.
- Vector-store staging is easier when quarantine, tenant, and source-trust labels are first-class metadata. [Qdrant multitenancy](https://qdrant.tech/documentation/manage-data/multitenancy/) and [Milvus multi-tenancy](https://milvus.io/docs/multi_tenancy.md) document native partitioning patterns, but the application still has to enforce source validation before promotion from staging to production collections.
- For tool-to-memory writes, schema validation is necessary but not enough. The [MCP tools specification](https://modelcontextprotocol.io/specification/2025-11-25/server/tools) explicitly calls for output sanitization and client-side result validation, and its optional output schema support gives auditors a concrete control to test. Memory writers should treat unstructured tool text, annotations, resource links, and fallback serialized JSON as untrusted even when the tool call itself was authorized.
- For poisoned-index investigations, keep enough provenance to support traceback after detection. [RAGForensics](https://arxiv.org/abs/2504.21668) is a useful audit pattern because it searches for the texts responsible for poisoned outputs, but it depends on preserved source IDs, batch IDs, retriever scores, and prompt-time evidence.
- For knowledge-graph-backed RAG, embedding sanitization needs to extend into graph construction. The May 2026 [A Few Words Can Distort Graphs](https://arxiv.org/abs/2508.04276) paper showed that <0.05% text edits can drop QA accuracy from 95% to 50%, so review queues should include proposed entities, relations, and triple deltas before commit, and structural diffs against a high-trust baseline should be alertable instead of only vector-distance signals.
- For AI-gateway and proxy components, treat the supply chain itself as part of the embedding-sanitization control. After the March 2026 LiteLLM PyPI compromise and the April 2026 LiteLLM CVE-2026-42208 SQL injection, defenders need pinned versions, signed releases, integrity checks on the proxy binary, runtime alerts for unexpected outbound connections from the gateway, and rollback procedures for embeddings or memory writes performed during a compromise window.
- For MCP-mediated memory writes, the April 2026 [Microsoft Agent Governance Toolkit](https://microsoft.github.io/agent-governance-toolkit/) is useful as a concrete reference for a control plane that sits between MCP clients and tool servers, rather than scattering custom sanitization into each agent. Its `McpSecurityScanner` and `McpResponseSanitizer` are aimed precisely at C8.2.5 verification (tool-poisoning, hidden instructions, exfiltration URLs, credential leakage), and the YAML-policy `GovernanceKernel` plus audit/OpenTelemetry hooks give auditors a reviewable trail of allowed and blocked tool calls. Production teams should still wire its outputs into a central memory-write service so that ingestion, tool-output capture, and durable memory writes share one policy surface.
- For EU customer data, provenance and content marking are becoming regulated controls on a firm timeline: Commission enforcement powers over GPAI providers and the [Article 50 transparency rules](https://artificialintelligenceact.eu/transparency-rules-article-50/) requiring machine-readable marking of AI-generated content take effect August 2, 2026, and the [Code of Practice on AI-Generated Content Transparency](https://artificialintelligenceact.eu/code-of-practice-overview/) reached a second draft in March 2026 with final text expected mid-2026 ahead of the December 2, 2026 transparency deadline. Embedding pipelines should record source provenance, transformations applied between ingestion and vector, and the trust tier or marking carried over from the source — including whether ingested content is itself machine-marked as AI-generated — so generated outputs can preserve or surface that signal.
- Standards mapping is firming up as of mid-2026. [MITRE ATLAS v2026.05](https://github.com/mitre-atlas/atlas-data/releases/tag/v2026.05) (May 27, 2026) moved to calendar versioning and a normalized v6.0.0 data format with typed relationships and a per-technique `platforms` field, so the AML.T0080 context-poisoning family is now explicitly scoped to Agentic AI — useful for filtering detections by deployment type, though no new memory/RAG techniques were added. [ISO/IEC 27090](https://www.iso.org/standard/56581.html) (guidance for addressing security threats to AI systems, covering data poisoning and adversarial ML) reached FDIS in March 2026 with publication expected later in 2026, and NIST's [COSAiS control overlays](https://csrc.nist.gov/Projects/cosais) for SP 800-53 are drafting single-agent and multi-agent overlays that should give ingestion provenance and memory integrity their first canonical 800-53 mappings. The OWASP GenAI [AI Security Solutions Landscape for Agentic AI Q2 2026](https://genai.owasp.org/resource/ai-security-solutions-landscape-for-agentic-ai-q2-2026/) maps vendor categories to ASI06, including memory-write scanning, memory segmentation, and provenance/trust scoring with decay of low-trust entries.
- The April 2026 [Securing Retrieval-Augmented Generation: A Taxonomy of Attacks, Defenses, and Future Directions](https://arxiv.org/abs/2604.08304) survey organizes RAG security across six workflow stages and three trust boundaries, identifying four primary security surfaces: pre-retrieval knowledge corruption, retrieval-time access manipulation, downstream context exploitation, and knowledge exfiltration. Its conclusion that "current defenses remain largely reactive and fragmented" and the call for layered, boundary-aware protection across the entire knowledge-access lifecycle aligns with the C8.2 stance that pre-ingestion quarantine, vector staging, memory-write authorization, and contradiction review must work as a single program rather than as point fixes.
- For embedding egress, treat the embedding pipeline as a regulated data-exit surface. After the May 2026 VectorSmuggle disclosure, monitoring should include per-tenant upsert volumes, vector-distribution drift, distance-to-baseline checks, and cryptographic signing for embeddings written from high-trust sources; existing DLP, CASB, and proxy stacks cannot inspect embedding payloads today, so signing, ingress/egress allowlists for embedding providers, and provenance-bound retention are the controls that catch covert-channel exfiltration before vectors land in a third-party store.
