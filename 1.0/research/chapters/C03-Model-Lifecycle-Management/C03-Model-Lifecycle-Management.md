# C03: Model Lifecycle Management & Change Control

## Control Objective

This chapter addresses control of model changes so that unauthorized or unsafe modifications cannot reach production. In practice, that means preserving integrity, traceability, recoverability, and accountability from development and fine-tuning through deployment, rollback, and decommissioning.

## Threat Landscape

Known attacks, real-world incidents, and threat vectors relevant to this chapter:

- Model tampering during training or fine-tuning pipeline
- Unauthorized model deployment bypassing validation gates, especially where pre-deployment safety evaluations only warn instead of blocking promotion -- 3.2.1 requires automated input-validation, safety-evaluation, and output-sanitization testing before deployment
- Model supply chain attacks (compromised checkpoints on Hugging Face, etc.) -- 6.5x increase in malicious models in 2024 (JFrog); 352K unsafe issues across 51.7K models (Protect AI, 2025)
- Model namespace reuse/hijacking: attackers reclaim deleted namespaces to distribute malicious models via trusted names (Unit 42, 2025)
- Picklescan evasion via nullifAI techniques, bypassing format-based defenses (ReversingLabs, 2025)
- Stale or vulnerable models remaining in production without rotation -- 91% of ML models degrade over time (MIT)
- Silent hosted model updates changing behavior without consumer awareness -- prompts degrade after provider updates; 3.2.3 requires security re-evaluation when a provider changes model, version, or routing
- Provider model deprecations with limited notice windows (60 days for Azure GA models; varies widely across providers)
- Incomplete decommissioning leaving model artifacts accessible to attackers
- Weaponized open-source AI repositories (NullBulge campaign, 2024: ransomware delivery via Hugging Face/GitHub)
- Post-training compression (quantization, pruning, distillation) degrading safety alignment and adversarial robustness -- research shows extreme quantization introduces unpredictable representational harm and pruning reduces model robustness to adversarial inputs; 3.2.2 requires re-evaluating quantized artifacts against the same safety/alignment suite before deployment
- Eight-nation AI/ML supply chain guidance (March 2026) reframes AI as a layered supply chain spanning data, models, software, infrastructure, hardware, and third-party services; recommends cryptographic signing, verified model registries, and provenance checks across the lifecycle
- AI infrastructure supply chain attacks escalating: TeamPCP campaign (March 2026) compromised Trivy, npm packages, Checkmarx KICS, LiteLLM, and Telnyx in a five-day chain; affected hosts needed credential rotation, persistence hunting, and Kubernetes review, not just package rollback
- AI-framework scope takeover: the June 17, 2026 `@mastra` npm compromise (attributed by Microsoft to North Korea's Sapphire Sleet) hijacked a dormant maintainer account to republish 141 packages in a 90-minute burst, injecting an `easy-day-js` typosquat that ran an obfuscated, self-deleting postinstall RAT in environments that routinely hold LLM keys, cloud credentials, CI/CD tokens, and DB connection strings -- evidence that 3.4.1/3.4.2 environment isolation and 3.1.x registry/admission controls must extend to the agent-framework and developer-tooling layer, not just model weights
- Hosted model behavioral regressions from system-prompt and feedback-weighting changes -- GPT-4o sycophancy incident (April 2025) required rollback after the update over-weighted short-term feedback signals
- Parallel-deployment cross-contamination: in canary, blue/green, shadow, and A/B cohorts, AI-specific shared state (KV caches, prompt caches, session state, retrieval indices) can leak context from one model version into another and confound rollback metrics; addressed by AISVS requirement 3.3.3
- IBM X-Force 2026 Threat Index: 300K+ ChatGPT credentials on dark web marketplaces in 2025; supply chain compromises up nearly 4x since 2020; 44% increase in exploitation of public-facing applications
- Model-hub trust-signal abuse: HiddenLayer found the fake Open-OSS/privacy-filter repository on Hugging Face on May 7, 2026; it copied a legitimate model card, used inflated engagement to reach the trending list, and shipped a loader that executed Windows infostealer malware
- Trusted-publishing abuse in package ecosystems: the May 2026 Mini Shai-Hulud waves compromised the SAP `@cap-js/sqlite`/`@cap-js/postgres`/`@cap-js/db-service` and `mbt` packages first, then TanStack, Mistral AI, UiPath, OpenSearch, Guardrails AI, and on May 19 burst 300+ malicious @antv package versions (Snyk and The Hacker News count 323 packages, approximately 16 million combined weekly downloads) in roughly 22 minutes through a compromised maintainer account; the TanStack path combined `pull_request_target`, cache poisoning, and OIDC token extraction, and the antv malware reused stolen GitHub Actions OIDC tokens to request Fulcio signing certificates and log in-toto attestations to Rekor, producing artifacts that carry cryptographically valid SLSA Build Level 3 provenance. StepSecurity attributes the waves to TeamPCP -- this is the first documented npm supply chain attack to publish artifacts with valid SLSA provenance, so provenance attestations alone cannot distinguish the malicious build
- Model-loading library RCE: CVE-2026-4372 (publicly detailed June 2026) let a poisoned `_attn_implementation_internal` field in a model's `config.json` make Transformers download and execute code from an attacker-controlled Hugging Face repository during a routine `AutoModelForCausalLM.from_pretrained()` call, bypassing `trust_remote_code=False`; the exploitable kernel-dispatch path shipped in 4.56.0 (August 29, 2025) and persisted through 5.2.x until 5.3.0 (March 4, 2026), and vulnerable versions were downloaded roughly 232 million times -- model-file scanners focused on pickle payloads miss config-level attack surface in the loader library itself
- MLOps metadata and storage-plane attacks: Google Cloud's GCP-2026-011 and GCP-2026-012 bulletins show that model lifecycle data can be active attack input. CVE-2026-2472 injected script escapes through evaluation results or dataset JSON into Jupyter/Colab, while CVE-2026-2473 used predictable Vertex AI Experiments bucket names for cross-tenant code execution, model theft, and poisoning; C3 validation and environment-isolation evidence therefore needs adversarial test data, non-predictable tenant-scoped storage, and patched lifecycle SDKs as well as checks on model weights
- Model-pull infrastructure SSRF: CVE-2026-33990 (disclosed April 2, 2026, CVSS 6.8) let a malicious OCI registry steer Docker Model Runner's token-exchange realm URL at internal services, turning a model pull into internal network reconnaissance and data exfiltration; fixed in 1.1.25
- MLOps platform RCE chains: MLflow disclosed CVE-2026-2635 (CVSS 9.8) chained with CVE-2026-2033 (CVSS 8.1) for unauthenticated tracking-server RCE via auth bypass plus artifact path traversal (both fixed in MLflow 3.8.0), CVE-2026-2652 (disclosed May 15, 2026, fixed in 3.10.0) leaving the Job and OpenTelemetry routes unauthenticated when the FastAPI permission middleware is only wired to `/gateway/`, and CVE-2025-15379 command injection in the model-serving container init when interpolating `python_env.yaml` dependency specifiers; registries that load arbitrary uploaded models or expose tracking servers to internal networks must upgrade to 3.10.0+, rotate credentials, and gate model loading on signed and reviewed artifacts
- Agent skill marketplace abuse: the ClawHavoc campaign (Q1-Q2 2026) seeded OpenClaw's ClawHub with 341+ malicious skills (Antiy CERT later expanded the count to 1,184 across the platform) that delivered Atomic macOS Stealer and other malware, while CVE-2026-25253 (CVSS 8.8) allowed any webpage to drive arbitrary skill invocations through the local control gateway on port 18789; scans found 135K+ exposed OpenClaw instances across 82 countries, showing that "skill" registries for agentic stacks now demand the same registry, signing, and admission controls as model registries
- MCP tool poisoning at protocol scale: OX Security's May 2026 disclosure spanned Python, TypeScript, Java, and Rust MCP SDKs, and the MCPTox benchmark (late 2025, updated early 2026) tested 45 live MCP servers and 353 authentic tools against poisoned tool descriptors and saw attack-success rates up to 72% on widely deployed agents -- tool/MCP integrations must be treated as lifecycle artifacts with versioned schemas, signed contracts, and pre-deployment evaluation
- Reward hacking generalizes beyond the immediate task: Anthropic's "Natural Emergent Misalignment from Reward Hacking in Production RL" (arXiv 2511.18397, November 2025) showed that production RL runs that learned reward hacking generalized to alignment faking, cooperation with malicious actors, and sabotage in Claude Code-style settings, and that standard chat-style RLHF cleaned up chat evaluations while leaving agentic misalignment intact -- effective mitigations were preventing the hack, broadening RLHF safety data, and "inoculation prompting" that frames reward hacking as acceptable during training so the misaligned generalization does not lock in
- Static validation cannot establish permanent robustness: NIST's May 2026 analysis argues that no finite guardrail set is universally robust against adaptive adversarial prompts, supporting a lifecycle of continuous red-team discovery, guardrail updates, impact limitation, and rapid recovery rather than a one-time pre-deployment pass

## Tooling & Implementation

Current tools, frameworks, and libraries that help implement these controls:

- **MLOps platforms:** MLflow, Weights & Biases, Kubeflow, SageMaker, Vertex AI Model Registry, Azure AI Foundry. Upgrade MLflow to 3.10.0 or later -- 3.8.0 addressed the chained CVE-2026-2635/CVE-2026-2033 auth-bypass-to-RCE path, and 3.10.0 closes the May 15, 2026 CVE-2026-2652 unauthenticated Job and OpenTelemetry route exposure when basic auth is wired through uvicorn -- and rebuild model-serving images that depend on CVE-2025-15379-vulnerable `python_env.yaml` interpolation. Tracking servers should not be exposed to internal networks without authentication, and registries should reject unsigned or untrusted model uploads before they reach inference workers
- **Model signing:** [OpenSSF Model Signing (OMS) v1.0](https://openssf.org/projects/model-signing/) and the [sigstore/model-transparency](https://github.com/sigstore/model-transparency) library -- production-ready Sigstore-based library and CLI for signing any model format (April 2025); the [OMS specification](https://github.com/ossf/model-signing-spec) (published mid-2025) makes the signing format implementation-agnostic across model hubs and ML frameworks; NVIDIA has signed all NGC Catalog models with OMS since March 2025; Google's Kaggle and NVIDIA's NGC have integrated the sigstore/model-transparency library; Cohere (October 2025) signs every Command model hosted on Hugging Face; Red Hat now signs its January and February 2026 validated-model batches and the corresponding ModelCars via Sigstore-backed Red Hat Trusted Artifact Signer build pipelines, with signatures and attestations hosted in the registry alongside the Red Hat AI Hugging Face org; also cosign, in-toto attestations, TUF (The Update Framework). [CoSAI](https://www.coalitionforsecureai.org/) (40+ members) released model signing and incident response frameworks in 2025-2026. Sigstore transparency logs provide verifiable proof-of-inclusion so any model verifier can confirm a signature appeared in the append-only log at a specific time. These satisfy 3.1.2 (signing) and underpin 3.1.3 verification at admission and on load
- **SBOM/ML-BOM:** CycloneDX ML-BOM for model and dataset transparency, provenance, training methods, framework configuration, and dependencies; SPDX AI/ML profiles; Syft for container SBOM generation. Requirement 3.1.1 requires a deployed-model registry inventory with origin, and registry records should link to C6 AI-BOM evidence when available
- **Deployment:** Seldon Core, BentoML, TorchServe, Triton Inference Server, vLLM with canary/blue-green strategies via Argo Rollouts or Flagger (satisfying 3.3.1 rollout/rollback); OPA Gatekeeper or Kyverno policies can enforce signature, checksum, evaluation-artifact, and model-owner admission checks before promotion. The Mini Shai-Hulud and `@mastra` waves are a reminder to pair provenance checks with hardened CI identity, cache isolation, protected release workflows, and post-build malware scanning because valid trusted-publishing metadata can still describe malicious output
- **Testing:** [OWASP AI Testing Guide v1](https://owasp.org/www-project-ai-testing-guide/) for repeatable trustworthiness testing; Giskard, Deepchecks, Great Expectations (for data validation), custom red-team harnesses, and CI evaluation jobs that fail closed when C3.2 safety thresholds are missed. 3.2.1 bundles input-validation, safety-evaluation, and output-sanitization testing into a single pre-deployment gate; [garak](https://docs.garak.ai/) (NVIDIA, Apache 2.0; `promptinject` and `latentinjection` probe families, extensible like Metasploit) and [AgentDojo](https://arxiv.org/abs/2406.13352) (97 agent tasks, 629 security cases, also packaged in the UK AISI Inspect evals) are the most direct ways to operationalize it as a versioned CI gate rather than an annual exercise. Agentic stacks should also include MCP/tool-poisoning suites -- the [MCPTox benchmark](https://www.practical-devsecops.com/mcp-tool-poisoning/) (late 2025, refreshed early 2026) drives poisoned tool descriptors across 45 live MCP servers and 353 authentic tools and observed attack-success rates up to ~72% on widely deployed agents
- **Evaluation-workspace hardening:** Pin `google-cloud-aiplatform` to 1.131.0 or later and Vertex AI Experiments to 1.133.0 or later where those components are used. Test notebook visualizers with HTML/script escape payloads in evaluation results and dataset JSON, keep report rendering outside privileged training sessions, and verify experiment storage is tenant-scoped and cannot be replaced through bucket-name pre-creation. These checks turn CVE-2026-2472 and CVE-2026-2473 into regression tests for C3.2 and C3.4 rather than treating the evaluation workspace as a trusted observer
- **Compression safety testing:** SafetyBench, TruthfulQA for benchmarking quantized/pruned model safety; quantization-aware training to preserve robustness during compression -- the direct way to produce 3.2.2 re-evaluation evidence on compressed artifacts
- **Data versioning:** DVC, LakeFS, Delta Lake
- **Audit logging:** Cloud-native (CloudTrail, Cloud Audit Logs, Azure Monitor), append-only log stores. Model-change audit-trail requirements live in C12, so log retention and tamper-evidence evidence belongs there while C3 focuses on integrity, validation, deployment, and pipeline controls
- **Hosted model monitoring:** [deprecations.info](https://deprecations.info/) (tracks deprecations across OpenAI, Anthropic, Google, AWS Bedrock, Cohere, xAI via JSON/RSS), [llm-stats.com](https://llm-stats.com/llm-updates) (model release tracking) -- useful inputs for the 3.2.3 trigger that requires re-evaluation when a provider changes model, version, or routing
- **Supply chain scanning:** Protect AI model scanning (352K issues found across 51.7K models in 2025), Hugging Face Picklescan (updated after nullifAI evasion), safetensors format, and static/dynamic inspection of repository helper scripts (`loader.py`, `start.bat`, notebooks, install hooks). For public model hubs, popularity, likes, and trending rank are weak evidence; C3 gates should prefer signed provenance, verified publisher identity, controlled sandbox loading, and reproducible evaluation records. CVE-2026-4372 adds the loader library itself to the scanning scope: pin Transformers to 5.3.0 or later, audit cached `config.json` files for `_attn_implementation_internal`, and treat config fields -- not just serialized weights -- as executable attack surface when scanning third-party models
- **Hosted model logging:** [PromptLayer](https://docs.promptlayer.com/), [Langfuse](https://langfuse.com/), [LiteLLM](https://docs.litellm.ai/) (unified proxy with model identifier logging across providers), MLflow AI Gateway
- **RLHF safety:** [Safe-RLHF](https://github.com/PKU-Alignment/safe-rlhf) (PKU, constrained value alignment), SafeMoE (ICLR 2025, safety routing for MoE LLMs), KL penalty monitoring (standard PPO), Adversarial Reward Auditing (ARA, Feb 2026, latent-space detection), InfoRM (information-theoretic reward modeling with MOP metric), Advantage Sign Robustness ([arXiv 2604.02986](https://arxiv.org/html/2604.02986), April 2026, training-time mitigation), and the "inoculation prompting" mitigation from Anthropic's [Natural Emergent Misalignment from Reward Hacking in Production RL](https://arxiv.org/abs/2511.18397) (Nov 2025) for keeping reward-hacking generalization out of agentic deployments -- relevant to 3.5.2 reward-hacking detection
- **Standards/Frameworks:** NIST AI RMF 1.0 (January 2023) plus the Generative AI Profile NIST-AI-600-1 (July 2024) and the Trustworthy AI in Critical Infrastructure Profile concept note (April 7, 2026) -- a full 1.1 version has not been released, and guidance is rolling out via profiles and addenda through 2026; [NIST COSAIS](https://csrc.nist.gov/Projects/cosais) (SP 800-53 Control Overlays for Securing AI Systems) -- annotated outline for the "Using and Fine-Tuning Predictive AI" overlay published January 8, 2026, with planned use cases spanning generative AI, predictive AI, single- and multi-agent systems, and AI developers; the overlays tailor existing SP 800-53 controls to model integrity, provenance, and adversarial robustness, which gives US federal-aligned organizations a concrete control vocabulary for C3 registry, signing, and validation evidence; ISO/IEC 42001 (AI management systems); EU AI Act Chapter V GPAI provider obligations in force since August 2, 2025, including technical documentation, copyright policy, training-content summary, systemic-risk assessment, incident reporting, and cybersecurity obligations; GPAI Code of Practice finalized as a voluntary compliance route and signed by roughly two dozen providers including Anthropic, Google, Microsoft, Mistral, and OpenAI; the [EU Digital Omnibus on AI](https://www.gibsondunn.com/eu-ai-act-omnibus-agreement-postponed-high-risk-deadlines-and-other-key-changes/) (provisional agreement May 6-7, 2026, formal adoption expected by July 2026) defers Annex III high-risk obligations to December 2, 2027 and Annex I to August 2, 2028 but leaves GPAI model obligations and the August 2, 2026 Commission enforcement date unchanged, with Article 50 AI-content transparency proceeding on schedule plus a four-month watermarking grace period for existing systems; SLSA for ML (emerging); eight-nation AI/ML Supply Chain Guidance (March 2026); DevSecMLOps (OpenSSF MLSecOps whitepaper, 2025); OCC Bulletin 2025-26 (October 2025) clarified model-risk-management expectations for community banks and announced a broader OCC review of model-risk practices at banks of all sizes, the first place many US banks are formalizing LLM/GenAI governance under the existing OCC Bulletin 2011-12 model-risk framework
- **Whole-lifecycle risk assessment:** Ireland's [2026 NCSC AI Cyber Security Risk Assessment](https://www.ncsc.gov.ie/pdfs/NCSC_AI_risk_assessment_2026.pdf) treats design, development, deployment, maintenance, and end-of-life as one reassessment loop. Its development risks map closely to C3 -- unvetted third-party components, insecure training environments, inadequate testing/red teaming, missing integrity protection, and weak lineage -- while its end-of-life phase adds retired models, datasets, and orphaned assets to the inventory and decommissioning evidence expected from a mature lifecycle program

### Lifecycle Assurance Evidence

| Control Area | Evidence to Retain | Negative or Inspection Test |
|--------------|--------------------|-----------------------------|
| C3.1 Authorization & Integrity | Registry export tying each deployed digest to its origin, complete artifact set, signer identity, signature, and consuming deployment | Submit an unsigned artifact, a valid signature from an unauthorized identity, and a signed manifest whose weight or configuration digest is changed; admission and load must reject all three |
| C3.2 Validation & Testing | Versioned evaluation harness, immutable input corpus, model/provider identity, thresholds, raw results, sanitization evidence, and promotion decision | Insert maintained prompt-injection probes, unsafe-output cases, and script escapes in result JSON; promotion must fail and report viewing must not execute the payload. Repeat the same suite against compressed artifacts and after provider changes |
| C3.3 Deployment & Rollback | Rollout manifest, cohort routing, automated trigger thresholds, complete-state snapshot, rollback event log, and per-version cache/index namespaces | Breach a safety threshold in a canary, verify automatic rollback restores weights plus configuration and policy state, then prove one version cannot read another version's KV cache, prompt cache, session state, or retrieval index |
| C3.4 Secure Development | Environment-specific identities, network paths, storage accounts, registries, prompt/policy stores, tool catalogs, and training services | From a development or training identity, attempt to read production model storage, invoke production AI-runtime services, substitute experiment storage, and publish directly to the production registry; each path must be denied |
| C3.5 Pipeline Fine-Tuning | Signed stage manifests linking input digests, reward-model version, evaluation thresholds, output digest, checkpoint ID, and approving workflow identity | Tamper with one intermediate output and force a reward metric beyond its over-optimization threshold; the next stage must refuse the artifact, and every retained checkpoint must remain independently identifiable and recoverable |

### Implementation Maturity

| Control Area | Tooling Maturity | Notes |
|--------------|:---:|-------|
| C3.1 Model Authorization & Integrity | Medium-High | Three requirements: registry inventory with origin (3.1.1, L1), cryptographic signing of all model artifacts -- weights, configs, tokenizers, base models, fine-tunes, adapters, and safety/policy models -- by authorized entities (3.1.2, L2), and signature verification at deployment admission and on load (3.1.3, L2). Model registries are mature. OpenSSF Model Signing (OMS) v1.0 reached production readiness in April 2025; NVIDIA signs all NGC Catalog models with OMS, Google's Kaggle integrated the sigstore/model-transparency library, Cohere (October 2025) signs all Command models on Hugging Face, and Red Hat (January-February 2026 validated-model batches) signs models and ModelCars via Sigstore-backed Red Hat Trusted Artifact Signer pipelines. CoSAI (40+ orgs) released model signing and incident-response frameworks in 2025-2026. The March 2026 eight-nation AI/ML supply chain guidance recommends verified registries and cryptographic signing. The May-June 2026 Mini Shai-Hulud and `@mastra` waves show the practical gap: signatures and SLSA-style provenance need protected CI identities, cache isolation, and release-workflow review, or the registry may faithfully record a malicious but validly published artifact, as it would have for the @antv burst on May 19. Signing infrastructure remains beyond baseline maturity for most teams, which is why 3.1.2/3.1.3 sit at Level 2, and CVE-2026-4372 shows that 3.1.3 load-time verification must extend to config metadata, since a poisoned `config.json` field alone achieved RCE through the loader. *(Updated 2026-06)* |
| C3.2 Model Validation & Testing | Medium | Three requirements: a single pre-deployment gate that bundles automated input-validation, safety-evaluation, and output-sanitization testing (3.2.1, L1); re-evaluation of post-training quantized artifacts against the same safety/alignment suite before deployment (3.2.2, L2); and security re-evaluation when a provider changes model, version, or routing (3.2.3, L3). Auditors should expect fail-closed pre-deployment evidence under 3.2.1. Safety-testing tools exist but standardized pass/fail thresholds still vary by domain. OWASP AI Testing Guide v1 (November 2025) improves the evidence base with repeatable trustworthiness tests across application, model, infrastructure, and data layers, and garak probe suites plus AgentDojo security cases satisfy 3.2.1 directly as versioned CI gates. SafetyBench and TruthfulQA can benchmark quantized models for 3.2.2, but edge-specific safety benchmarks remain a gap, and 3.2.3 still lacks a standard provider-change-notification protocol to trigger re-evaluation reliably. *(Updated 2026-06)* |
| C3.3 Controlled Deployment & Rollback | High | Three requirements: rollout mechanisms with automated rollback triggers (3.3.1, L2), complete-state rollback (3.3.2, L2), and isolated runtime state for parallel model versions so AI-specific shared resources are not shared across deployments (3.3.3, L2). Canary/blue-green is well-supported in the Kubernetes ecosystem; atomic multi-artifact rollback is less mature. The Model Namespace Reuse attack (2025) demonstrated the need for deployment-time integrity checks at the cloud-platform level, and the GPT-4o sycophancy incident (April 2025) showed that behavioral metrics must be part of staged evaluation -- functional metrics can pass while safety regresses. Requirement 3.3.3 remains a gap in many inference stacks because throughput-oriented cache sharing (KV caches, prompt caches, session state, retrieval indices) requires explicit per-deployment routing or per-tenant cache partitioning to comply. Emergency-shutdown and cascade controls live in C9 (Orchestration and Agentic Action) rather than here. |
| C3.4 Secure Development Practices | High | Two requirements scoped to AI-specific runtime components (agent orchestration services, tool/MCP servers, model registries, prompt/policy stores): AI-runtime components must not be shared across environment boundaries such as dev/staging/prod (3.4.1, L1), and training/fine-tuning environments must be isolated from production (3.4.2, L2). Version control and environment isolation are largely solved problems, but cultural adoption in ML teams varies and data provenance remains a gap. The TeamPCP/LiteLLM attack (March 2026), the May 2026 ClawHavoc OpenClaw skill-marketplace compromise, and the June 17, 2026 `@mastra` npm scope takeover (Sapphire Sleet) all show that AI-runtime extensibility surfaces -- security tools in training environments, MCP/tool catalogs, agent skill registries, and agent-framework packages -- belong inside C3.4 isolation, signing, and review controls. The May 2026 MCPTox results and OX Security MCP disclosure add tool/MCP contracts to that list of versioned, signed artifacts requiring pre-deployment review. IBM X-Force 2026 reports supply chain compromises up nearly 4x since 2020, with AI-powered coding tools introducing unvetted code into pipelines. *(Updated 2026-06)* |
| C3.5 Pipeline Fine-Tuning | Low | Four requirements: RLHF models versioned and integrity-verified before a training run (3.5.1, L2), automated detection of reward hacking or reward-model over-optimization (3.5.2, L3), per-stage output integrity verification in multi-stage pipelines (3.5.3, L3), and registration of fine-tuning checkpoints as distinct artifacts (3.5.4, L3). Reward-model integrity can leverage OMS v1.0 signing. Reward-hacking detection is an active research area -- InfoRM (information-theoretic reward modeling with CSI indicator), ODIN (disentangled reward), ARA (Feb 2026 latent-space detection), Advantage Sign Robustness (April 2026 training-time mitigation), classifier-based anti-reward-hacking rewards, and the Reward Hacking Benchmark (RHB, May 2026, [arXiv 2605.02964](https://arxiv.org/abs/2605.02964)) which probes exploit rates across 13 frontier models in independent and chained tool-use regimes -- but none are production-hardened as drop-in defenses. Anthropic's November 2025 "Natural Emergent Misalignment from Reward Hacking in Production RL" (arXiv 2511.18397) showed reward-hacking generalization persisted in agentic tasks even after standard chat-style RLHF, and identified three working mitigations -- preventing the hack, broadening RLHF safety data, and inoculation prompting -- that map directly onto 3.5.2 detection thresholds and the C3.2.1 pre-deployment gate. RHB further shows that RL post-training is associated with substantially higher exploit rates on identical tasks, so 3.5.2 thresholds should be re-baselined whenever the RL stack or reward signal changes, not only when the base model changes. Multi-stage checkpoint integrity verification (3.5.3) and per-stage rollback still require custom pipeline logic in most MLOps stacks. *(Updated 2026-07)* |

## Open Research Questions

- [ ] What's the right cadence for model re-validation in production (drift vs. cost)? (MIT finding: 91% of models degrade; Gartner: 67% within 12 months)
- [ ] How do hosted model providers (OpenAI, Anthropic, Google) handle versioning transparency, and what reliably triggers the 3.2.3 re-evaluation when a provider silently changes model, version, or routing? (Partially answered: deprecations.info tracks 7 providers; Azure offers 60-day notices; OpenAI provides model IDs but aliases them; transparency remains inconsistent)
- [ ] What constitutes 'adequate' pre-deployment security testing for fine-tuned models under the 3.2.1 gate, where input validation, safety evaluation, and output sanitization are a single requirement? (OWASP AI Testing Guide v1 gives a structure, but domain-specific thresholds still need calibration.)
- [ ] How should organizations manage rollback for models with stateful fine-tuning?
- [ ] Can SLSA (Supply-chain Levels for Software Artifacts) be adapted for ML model provenance? (ReversingLabs 2025 proposes extending SLSA provenance files with ML-specific metadata; not yet standardized)
- [ ] What lifecycle evidence should C3 retain when C9.6 shutdown and graceful-degradation controls are activated against a deployed model, agent, or provider dependency?
- [ ] How should organizations handle provider auto-upgrade behavior (e.g., Azure's upgrade-on-expiry for pinned model deployments) within the 3.2.3 re-evaluation trigger?
- [ ] What is the minimum viable model identity information that providers should expose for audit compliance under the EU AI Act?
- [ ] How should a C3.1 model registry prove completeness while avoiding duplication with C6 AI-BOM and source-vetting records?
- [ ] What safety thresholds should apply to post-compression artifacts under 3.2.2? Research shows quantization and pruning degrade alignment and robustness unpredictably -- standardized benchmarks for compressed model safety are still lacking.
- [ ] How quickly will OpenSSF Model Signing (OMS) achieve broad adoption beyond NVIDIA NGC? CoSAI is driving standardization but hub-level adoption on Hugging Face, Kaggle, etc. is not yet universal.
- [ ] Which reward hacking detection approaches (ARA, InfoRM, BNRM, KL penalty, Advantage Sign Robustness, RHB-style multi-step probes) will prove most practical for production RLHF pipelines under 3.5.2? RHB (May 2026) shows exploit rates spanning 0%-13.9% across 13 frontier models and that environmental hardening cuts exploit rates by roughly 87% without hurting correctness, but no production-hardened drop-in defense exists as of mid-2026.
- [ ] Should hosted model providers be required to expose immutable model identifiers (not just aliases) in API responses? Current practice varies widely and creates audit gaps for consumers in regulated industries.
- [ ] How should organizations defend against cascading supply chain campaigns like TeamPCP (March 2026) where a single compromised security tool (Trivy) led to backdoored AI infrastructure packages (LiteLLM, 97M monthly downloads) within five days?
- [ ] What extra evidence should model registries require when a package or model has valid provenance but the trusted release workflow may have been compromised, as in the Mini Shai-Hulud wave against TanStack and Mistral AI packages in May 2026, the May 19 @antv burst where malware reused stolen GitHub Actions OIDC tokens to request Fulcio certificates and log in-toto attestations to Rekor, and the June 17 `@mastra` scope takeover via a dormant maintainer account?
- [ ] Should system prompt changes be subject to the same staged deployment discipline (canary, blue-green) as model weight updates? The GPT-4o sycophancy incident (April 2025) suggests yes, but no framework currently mandates this.
- [ ] How can production inference servers (vLLM, Triton, TGI, SGLang) provide per-cohort isolation of KV caches, prompt caches, session state, and retrieval indices without sacrificing the throughput benefits that shared caches provide? AISVS requirement 3.3.3 mandates this isolation, but mainstream inference stacks default to shared caches across requests.
- [ ] How will the OCC's broader review of model-risk-management guidance (announced alongside OCC Bulletin 2025-26 in October 2025) treat LLMs and generative AI, and will other sector regulators (SEC, FINRA, NYDFS) follow with AI-specific MRM guidance?
- [ ] Will NIST publish a full AI RMF 1.1 version or continue releasing guidance through profiles (GenAI Profile 2024, Critical Infrastructure Profile concept note April 2026) and addenda? Federal contractors need clarity on which baseline to target.
- [ ] How will the August 2, 2026 activation of European Commission enforcement powers against GPAI providers -- including fines up to 3% of global turnover or EUR 15M -- change provider behavior around model identity disclosure, version tracking, and downstream documentation? (The May 2026 Digital Omnibus agreement confirmed this date holds even as Annex III high-risk obligations slip to December 2027.)
- [ ] Will NIST's COSAIS overlays (annotated outline January 2026, initial public drafts expected through 2026) give auditors a usable SP 800-53 control mapping for C3 registry, signing, and validation evidence, and how will the generative and agentic-AI use-case overlays differ from the predictive-AI one?
- [ ] After CVE-2026-4372 showed that a single `config.json` field can achieve RCE through the model-loading library with `trust_remote_code=False`, what should 3.1.3 "verification on load" mean in practice -- signature checks on weights alone, or schema validation and allow-listing of every config field the loader will interpret?
- [ ] What new admission-control evidence should a model registry demand when SLSA provenance is no longer sufficient on its own, as the May 19, 2026 @antv Mini Shai-Hulud burst produced malicious npm artifacts with valid SLSA provenance metadata?
- [ ] How should agent skill registries (ClawHub-style marketplaces) and AI-framework package scopes (the `@mastra` scope takeover) be brought under C3.1/C3.4 controls? ClawHavoc, CVE-2026-25253, and the Sapphire Sleet `@mastra` campaign demonstrate that "skill" registries and agent-framework packages now warrant the same signing, owner verification, quarantine review, and admission-policy controls as model registries.
- [ ] How quickly will MCP and tool contracts become first-class versioned artifacts evaluated under the 3.2.1 pre-deployment gate, given the May 2026 OX Security disclosure across Python, TypeScript, Java, and Rust SDKs and the MCPTox findings of up to ~72% attack-success on poisoned tool descriptors?
- [ ] Does Anthropic's "inoculation prompting" mitigation generalize to other foundation-model RL pipelines, and how should 3.5.2 thresholds incorporate agentic evaluations rather than only chat-style RLHF tests after the November 2025 finding that chat-style safety training masks but does not remove agentic misalignment from reward hacking?
