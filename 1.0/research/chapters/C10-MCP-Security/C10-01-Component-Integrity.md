# C10.1: Component Integrity & Supply Chain Hygiene

> **Parent:** [C10 MCP Security](C10-MCP-Security.md)
> **Requirements:** 3 | **IDs:** 10.1.1--10.1.3

## Purpose

This section ensures that MCP server and client software is obtained from trusted sources and cryptographically verified, that only approved MCP servers are permitted, and that locally launched servers run within least-privilege sandboxes. MCP components are typically distributed as npm packages, Python packages, Docker images, or standalone binaries — all of which are susceptible to supply-chain attacks including typosquatting, dependency confusion, and build-time tampering. Because MCP servers execute arbitrary code with access to sensitive resources (databases, APIs, file systems), compromised components have outsized blast radius compared to typical library dependencies.

The severity of these risks was demonstrated in real-world incidents throughout 2025-2026, and the late-April 2026 wave made the threat model explicit: supply-chain compromises now deliberately target MCP infrastructure as a persistence and exfiltration channel rather than treating MCP exposure as collateral damage. The Mini Shai-Hulud campaign (April 29-30, 2026) compromised SAP Cloud Application Programming packages (`mbt`, `@cap-js/sqlite`, `@cap-js/postgres`, `@cap-js/db-service` — ~572K weekly downloads), the `intercom-client` npm SDK (v7.0.4 and v7.0.5, April 30, 2026), and PyTorch Lightning 2.6.2/2.6.3 in a coordinated wave. The payload writes `.claude/settings.json` containing a `SessionStart` hook that re-executes the credential stealer every time the developer launches Claude Code, drops `.vscode/tasks.json` with `"runOn": "folderOpen"` for VS Code re-execution on project open, and harvests AI-tooling-specific paths including `~/.claude.json`, `~/.claude/mcp.json`, and `~/.kiro/settings/mcp.json` for MCP server credentials, API keys, and database connection strings. SecurityWeek tracked roughly 1,800 victim repositories and reported credential exfiltration to public GitHub repos named with Dune-universe terminology — converting MCP configuration files from passive credential stores into both the harvest target and the persistence mechanism. The Postmark MCP supply chain breach (2025) saw attackers backdoor an npm package used in AI automation pipelines, inserting a single line of malicious code that directed compromised MCP servers to BCC every outgoing email to an attacker-controlled address. The Smithery supply chain attack (October 2025) affected 3,000+ hosted applications and their API tokens. Research using the MCPTox benchmark showed that tool poisoning attacks — where malicious instructions are embedded in tool metadata invisible to human users but processed by AI models — pass seamlessly into agent contexts with alarming frequency. A 2026 survey of 2,614 MCP implementations found 82% use file operations vulnerable to path traversal, two-thirds have code injection risks, and over a third are susceptible to command injection. AgentSeal's April 2026 scan of 1,808 MCP servers (sourced from GitHub repositories, npm and PyPI packages, and public registries including Smithery and MCP.run) found 66% had at least one security finding — 43% involved shell or command injection, 20% targeted tooling infrastructure, 13% were authentication bypasses, and 10% exploited path traversal. These findings underscore why component integrity and supply-chain hygiene are foundational to MCP security. The ecosystem's governance is also maturing: the Linux Foundation launched the Agentic AI Foundation in December 2025 to provide open governance over MCP standard development, and the MCP specification's November 2025 revision (2025-11-25) introduced formal governance structures including working groups and SDK tiering — though as of April 2026 the spec still stops short of mandating integrity verification or supply-chain controls for servers, and Ox Security's April 15, 2026 disclosure of a systemic STDIO-execution flaw in the official MCP SDKs (Python, TypeScript, Java, Rust — covering roughly 150 million cumulative downloads and an estimated 200,000 vulnerable instances) demonstrated that architectural assumptions baked into the reference SDKs can inherit through to every downstream server, with Anthropic declining to patch on the grounds that STDIO is a "secure default" and sanitization is the developer's responsibility. The downstream consequence arrived almost immediately: LiteLLM tracked the issue as [CVE-2026-30623](https://docs.litellm.ai/blog/mcp-stdio-command-injection-april-2026) and shipped a fix in [PR #25343](https://github.com/BerriAI/litellm/pull/25343) (merged April 8, 2026, stable in v1.83.7) introducing an `MCP_STDIO_ALLOWED_COMMANDS` constant that restricts STDIO launches to a seven-binary allowlist (`npx`, `uvx`, `python`, `python3`, `node`, `docker`, `deno`) with optional extension via `LITELLM_MCP_STDIO_EXTRA_COMMANDS`. The pattern — a small, hardcoded launcher allowlist enforced both at request validation and at client instantiation — is now the practical reference design for any MCP host that cannot rely on the SDK to sanitize STDIO commands, and the same design has been independently proposed for OpenAI Codex via [issue #15814](https://github.com/openai/codex/issues/15814) (March 2026) which adds an optional `sha256` field to the `identity` section of `requirements.toml` so that allowlist entries pin a verified binary hash before execution rather than relying on command-name matching alone. Late April 2026 reinforced why pre-install gating beats post-facto scanning: between April 21 and 30, four independent supply-chain campaigns hit the AI/MCP-adjacent ecosystem within ten days — the [CanisterSprawl](https://socket.dev/blog/namastex-npm-packages-compromised-canisterworm) self-propagating npm worm (16+ malicious package versions across `@automagik/genie`, `pgserve`, `@fairwords`, `@openwebconcept`, harvesting ~40 credential categories and using an Internet Computer Protocol canister as decentralized C2), the [xinference PyPI](https://blog.gitguardian.com/three-supply-chain-campaigns-hit-npm-pypi-and-docker-hub-in-48-hours/) compromise (three consecutive credential-stealing releases on April 22), the [Bitwarden CLI v2026.4.0 cascade](https://blog.ibvl.in/index.php/2026/04/27/teampcp-supply-chain-campaign-update-008-26-day-pause-ends-with-three-concurrent-compromises-checkmarx-kics-bitwarden-cli-cascade-xinference-pypi-canistersprawl-npm-worm-identified-and-tier-1/) (poisoned via Dependabot pulling a trojanized Checkmarx KICS image, the first documented cascade compromise through trusted automation in the TeamPCP campaign), and the [PyTorch Lightning 2.6.2/2.6.3](https://socket.dev/blog/lightning-pypi-package-compromised) compromise (April 30, 11 MB Bun-runtime payload that runs a daemon thread on import, harvesting GitHub/npm/AWS/Azure/GCP credentials and 20+ CI environment variables, with on-import execution making it especially dangerous in agentic workflows where Claude Code or Cursor run `pip install` autonomously). Socket detected the Lightning payload eighteen minutes after publication — useful but too late for any developer who installed during that window — which is why the conversation has decisively shifted from "scan after install" to "verify before install."

As of May 2026, component integrity also needs to account for approved-but-vulnerable MCP servers, not only malicious packages. Akamai's May 12 analysis of Apache Doris MCP, Apache Pinot MCP, and Alibaba RDS MCP showed that SQL injection, missing authentication, and unauthenticated data exposure can exist inside recognizable upstream projects; MCP-BiFlow's May 8 study of 15,452 MCP repositories similarly found 118 confirmed unsafe data-flow paths across 87 servers. For C10.1, the practical takeaway is that allowlists should bind server identity to patched version floors, transport mode, deployment profile, dependency/SBOM evidence, and runtime scanner results rather than treating registry presence as a complete trust decision.

Mid-to-late May 2026 hardened that posture into a non-negotiable. The Mini Shai-Hulud worm returned on May 11-12 in a coordinated cross-ecosystem wave hitting 172 packages across 403 malicious versions on npm and PyPI — `@tanstack/*` (42 packages, 84 versions published between 19:20 and 19:26 UTC on May 11), the `@mistralai` SDK suite (npm and `mistralai@2.4.6` on PyPI), `@uipath/*` (65 packages), `@opensearch-project/*` (1.3M weekly downloads), and `guardrails-ai@0.10.1` (PyPI, an AI guardrails library frequently bundled with agent and MCP stacks) — combined cumulative downloads above 518 million. The worm hijacks GitHub Actions workflows from compromised maintainer machines to forge valid publish tokens and then harvests AWS IAM keys, GitHub PATs, HashiCorp Vault tokens, and Kubernetes secrets, exfiltrating through the same Dune-themed public-repo pattern as the late-April wave. The [TanStack chain](https://www.aikido.dev/blog/mini-shai-hulud-is-back-tanstack-compromised) is especially important for any team relying on OIDC trusted publishing: the attackers used a `pull_request_target`-triggered workflow to poison the GitHub Actions cache with a malicious pnpm store, then extracted the workflow's short-lived OIDC publish token directly from GitHub Actions runner process memory and used it to publish via the legitimate trusted-publishing path — meaning OIDC removed the long-lived token without removing the in-workflow code-execution surface that produces a valid publish credential. A week later, on May 19, the worm hit the @antv ecosystem in a [22-minute automated burst](https://socket.dev/blog/antv-packages-compromised) between 01:56 and 02:56 UTC, pushing 637 malicious versions across 317 unique packages from the hijacked `atool` (`[email protected]`) npm account — major libraries included `size-sensor` (~4.2M monthly downloads), `echarts-for-react` (~3.8M), `@antv/scale` (~2.2M), `timeago.js` (~1.15M), and the @antv visualization suite (`@antv/g2`, `@antv/g6`, `@antv/x6`, `@antv/l7`, `@antv/s2`, `@antv/f2`, `@antv/g2plot`, `@antv/graphin`, `@antv/data-set`). The ~499 KB obfuscated Bun payload used 1,732 Base64-encoded strings in a rotated array plus PBKDF2/SHA-256 runtime decryption for its C2 configuration, executed during `preinstall`, scraped GitHub Actions runner process memory to bypass secret masking, harvested credentials from 100+ file paths (AWS via env vars/config/EC2 IMDS/ECS metadata/Secrets Manager, HashiCorp Vault across 12+ paths, npm with OIDC exchange, Kubernetes service accounts, 1Password with 2FA bypass attempts), and exfiltrated through a three-tier channel: HTTPS to `t[.]m-kosche[.]com/api/public/otel/v1/traces` disguised as OpenTelemetry traces, GitHub Git Data API blobs/commits in victim repositories as fallback, and 2,200+ public repos under victim accounts with reversed Dune-themed descriptions (`niagA oG eW ereH :duluH-iahS`) as tertiary. Later the same day, TeamPCP pushed three malicious versions of Microsoft's official `durabletask` Python client (v1.4.1, v1.4.2, v1.4.3 — ~400K monthly downloads) to PyPI within a 35-minute window between 15:08 and 15:16 UTC, with the infection chain traced back to the earlier `@antv` (May 19) and `guardrails-ai` (May 11) waves; the 28 KB `rope.pyz` dropper harvests AWS/Azure/GCP/Kubernetes credentials and 90+ developer-tool configurations and then propagates laterally through cloud infrastructure. The same week, GitHub confirmed that an employee device compromise (via a poisoned VS Code extension) led to exfiltration of roughly 3,800 internal repositories — the public leaked listing includes `github-mcp-server` and `github-mcp-server-remote`, meaning the internal source code for GitHub's own MCP server family is now in adversary hands, raising the bar for what "trusted source" means even for first-party Microsoft/GitHub MCP infrastructure. Five CVEs disclosed against the official MCP Registry in the same May window made the registry layer itself part of the supply-chain attack surface: [CVE-2026-44427](https://advisories.gitlab.com/golang/github.com/modelcontextprotocol/registry/CVE-2026-44427/) (CVSS 3.1, CWE-601, fixed in v1.7.5) is an open redirect in `TrailingSlashMiddleware` — a protocol-relative path such as `//evil.com/` survives slash stripping and lands in the `Location` header so browsers treat it as an absolute redirect to an attacker domain, a classic phishing primitive against developers who follow links from registry-hosted listings; [CVE-2026-44428](https://advisories.gitlab.com/golang/github.com/modelcontextprotocol/registry/CVE-2026-44428/) (CVSS 4.7, fixed in v1.7.6) is a cross-deployment OIDC token replay — the GitHub Actions ID token is minted with a single global audience string (`mcp-registry`) and the registry's exchange endpoint derives publish permissions directly from `repository_owner`, so a token legitimately issued for one registry deployment is accepted by every other deployment sharing the same code and inherits publish rights for `io.github.<owner>/*` namespaces; [CVE-2026-44429](https://radar.offseq.com/threat/cve-2026-44429-cwe-79-improper-neutralization-of-i-5df190d3) (CVSS 4.0 5.1, CWE-79, fixed in v1.7.7) is a stored XSS in the catalogue UI — `server.websiteUrl` validation accepted quote characters and the `escapeHtml` helper only encoded `&`, `<`, `>`, and `U+00A0`, leaving `"` and `'` unescaped, so a publisher with a valid publish token could break out of the `href=""` attribute and append arbitrary `on*` event handlers that fire on any visitor's registry homepage; [CVE-2026-44430](https://advisories.gitlab.com/golang/github.com/modelcontextprotocol/registry/CVE-2026-44430/) (CVSS 6.3, fixed in v1.7.7) is an unauthenticated SSRF in the registry's HTTP-based namespace verification — the Go stdlib `IsPrivate`/`IsLoopback`/`IsLinkLocalUnicast`/`IsMulticast`/`IsUnspecified` blocklist misses IPv6 6to4 (`2002::/16`), NAT64 (`64:ff9b::/96` and `64:ff9b:1::/48` per RFC 8215), and deprecated site-local (`fec0::/10`), all of which encode arbitrary IPv4 and tunnel to RFC1918 or cloud-metadata services on dual-stack and NAT64-enabled hosts; the endpoint is the login flow itself, so attack complexity is trivial wherever the host's egress routing reaches a metadata service; and [CVE-2026-45781](https://radar.offseq.com/threat/cve-2026-45781-cwe-636-not-failing-securely-failin-6930022f) (CWE-636 "Not Failing Securely", fixed in v1.7.9) is an OCI ownership-validation fail-open — `ValidateOCI` in `internal/validators/registries/oci.go` returns success when the upstream OCI registry replies with HTTP 429, skipping the label-match check that proves the publisher controls the namespace, while every other validator in the same file (`npm`, `PyPI`, `NuGet`, `MCPB`) treats non-200 as a hard error, so an authenticated publisher can bind `io.github.<owner>/*` namespaces to OCI container images they do not own by waiting for or inducing rate-limiting on the upstream fetch. Read together, the five CVEs say that the registry can be coerced into redirecting users to attacker domains (44427), accepting cross-deployment publish tokens for victim namespaces (44428), executing script in maintainers' browsers under the registry's own origin (44429), reading internal services on the operator's behalf (44430), and admitting attacker-controlled OCI artifacts under victim namespaces (45781) — so any allowlist that anchors trust to "the server appears in the official registry" inherits whatever the registry's patch state is, and the registry-patch floor for a production admission policy is now v1.7.9, not v1.7.7. [CVE-2026-33224](https://www.ox.security/blog/mcp-supply-chain-advisory-rce-vulnerabilities-across-the-ai-ecosystem/) hit Bisheng's MCP server: the "Add MCP Server" interface routed user-supplied stdio command/args through `StdioServerParameters` without sanitization or an allowlist, granting authenticated callers RCE on the host — and Bisheng allows open user registration, so authentication was a formality. OX Security's broader registry-poisoning trial balloon (disclosed alongside the April STDIO advisory and revisited in May coverage) landed a malicious test entry in 9 of 11 MCP registries sampled and demonstrated command execution against six live production platforms (LiteLLM, LangChain, IBM LangFlow among them) — confirming that the same Anthropic-SDK STDIO flaw that motivated the April LiteLLM patch is now being exploited not just through compromised servers but through the registries that gate which servers a host will trust. Together, these three lines of evidence — worm-class supply chain hits on AI-adjacent packages, source-code exfiltration of trusted first-party MCP servers, and SSRF/RCE in the registry-and-launcher layer — make it impossible to treat "the server is in an approved registry" as a credible Level 1 admission decision; component integrity in May 2026 has to assume the registry itself, the launcher path, and the package's upstream maintainer accounts can all be hostile within the same week.

Early June 2026 added two lessons that the May waves had only implied. First, the Miasma campaign (June 1–7, 2026, attributed to the same TeamPCP-linked actor behind the May 19 `durabletask` push) demonstrated that worm-class attackers now pivot delivery mechanisms faster than defenders close them: Wave 1 (June 1) compromised 32 `@redhat-cloud-services` npm packages (~117K weekly downloads) via classic `preinstall` hooks; Wave 2 (June 4) hit 57 npm packages (~647K monthly downloads) using `binding.gyp` build-file execution — which fires during native compilation and therefore bypasses `ignore-scripts` exceptions granted for legitimate native builds; the June 5 phase abandoned package managers entirely, using a previously compromised contributor account to push commit `5f456b8` into `Azure/durabletask` planting `.claude/settings.json`, `.gemini/settings.json`, `.cursor/rules/setup.mdc`, and `.vscode/tasks.json` — all pointing at a 4.6 MB obfuscated `.github/setup.js` that executes when a developer simply opens the repository in an AI coding tool; and the Hades phase (June 7–8) published 37 malicious wheels across 19+ PyPI packages (notably `dynamo-release@1.5.4`, `spateo-release@1.1.2`, `ensmallen@0.8.101`, `embiggen@0.11.97`) using Python `.pth` startup hooks that execute on *any* interpreter launch — `python`, `pip`, `pytest` — with no package import required, defeating both `--ignore-scripts` and import-time scanning. GitHub's automated response disabled 73 repositories across four Microsoft organizations in 105 seconds, taking down `Azure/functions-action@v1` and breaking CI/CD pipelines globally — collateral damage from remediation is now part of the threat model. Second, the Hades payload carries "AI analyst misdirection": prompt-injection blocks at the head of the payload designed to hijack LLM-based security scanners into emitting false negatives — a direct attack on the LLM-as-judge analysis engines that several MCP scanners (Cisco MCP Scanner, mcpscan.ai) rely on, and the first documented case of malware defending itself against AI-powered review. Its C2 traffic camouflages as HTTPS to a non-existent `api.anthropic.com/v1/api` route, blending exfiltration into legitimate AI-developer traffic. Across all 448 Miasma/Hades artifacts, zero CVEs were assigned during active exploitation — CVE-feed-driven admission gating sees nothing — while CVE-2026-45321, assigned to the May TanStack CI/CD chain, confirmed that even packages carrying valid SLSA Build Level 3 provenance can be attacker-published. The governance layer is responding: the NSA's Artificial Intelligence Security Center published its first MCP Cybersecurity Information Sheet on May 20, 2026 ("Model Context Protocol: Security Design Considerations for AI-Driven Automation," U/OO/6030316-26), recommending component zoning, pre-deployment server vetting with scanner tooling, comprehensive invocation logging with cryptographic hashes of results, and filtering egress proxies — the first national-security-agency guidance to treat MCP supply-chain hygiene as a named risk area.

Generate SBOMs for MCP server dependencies using tools like Syft or Trivy (CycloneDX/SPDX formats) — traditional manifest parsing often misses build-time dependencies that execute during installation. Run [Cisco MCP Scanner](https://github.com/cisco-ai-defense/mcp-scanner) (open-source, October 2025) for pre-deployment vulnerability assessment — it employs three independent scanning engines (YARA signature-based detection, LLM-as-judge semantic analysis, and Cisco AI Defense integration) to identify tool poisoning, rug-pull attacks, and over-privileged permissions before MCP servers are integrated into agent environments. Evaluate [ToolTrust](https://glama.ai/mcp/servers/AgentSafe-AI/tooltrust-scanner) (AgentSafe) for CI/CD-integrated MCP scanning — it detects prompt injection, data exfiltration, privilege escalation, and known-compromised packages (including TeamPCP-affected LiteLLM/Trivy versions) via OSV CVE integration and an offline blacklist for zero-latency detection; deployable as an MCP server, CLI tool, GitHub Action, or Git pre-commit hook. Use [mcpscan.ai](https://mcpscan.ai/) for repository-level MCP security scanning with LLM-based tool poisoning classification, covering command/code injection, path traversal, authentication deficiencies, and IDOR vulnerabilities. For STDIO-launched MCP servers, mirror the [LiteLLM CVE-2026-30623 mitigation pattern](https://github.com/BerriAI/litellm/pull/25343): enforce a hardcoded launcher allowlist (e.g., `MCP_STDIO_ALLOWED_COMMANDS` restricted to `npx`, `uvx`, `python`, `python3`, `node`, `docker`, `deno`) at both request-validation time and client-instantiation time, and confine any test/preview endpoints to admin roles only — this closes the SDK-level RCE gap that Anthropic declined to patch upstream. Treat command-name allowlisting as necessary but insufficient: the [Flowise CVE-2026-40933 and Upsonic CVE-2026-30625 advisories](https://www.ox.security/blog/flowise-cve-2026-40933-upsonic-cve-2026-30625-what-to-do-when-best-practice-isnt-enough/) demonstrated that the allowlisted launchers themselves accept argument flags that turn them into arbitrary command interpreters (`npx -c <cmd>`, `python -c <cmd>`, `node -e <cmd>`), so production hosts must also strip or reject the eval-like flags (`-c`, `-e`, `--eval`, `--exec`), reject shell-style argv strings, pin the required argv shape per server, and confine the launcher to a sandbox with no developer secrets in its environment. Pair this with package-manager publish-cooldown gates ([pnpm 11 `minimumReleaseAge: 1440`](https://lilting.ch/en/articles/pnpm-11-minimum-release-age-default-shai-hulud-defense) by default, Yarn 4.10 `npmMinimalAgeGate: "3d"`, npm v11.10 `min-release-age=7`) on every developer machine and CI runner that installs MCP server packages — the cooldown defeats the rapid-burst worm pattern (TanStack's six-minute window, @antv's 22-minute burst, durabletask's 35-minute push) because malicious versions are quarantined and yanked before the gate releases them. Audit the `onlyBuiltDependencies` / `npmPreapprovedPackages` / explicit `npm rebuild --ignore-scripts=false` allowlist alongside the cooldown, since every entry that re-enables postinstall scripts restores the supply-chain attack surface that `ignore-scripts=true` would otherwise close. For pre-install binary verification, follow the [OpenAI Codex `requirements.toml` SHA-256 proposal](https://github.com/openai/codex/issues/15814) (March 2026): augment allowlist entries with an optional `sha256` field so the host computes and verifies the binary hash before spawning the server process, rejecting any binary whose hash does not match the pinned value — this catches PATH-based typosquatting, post-installation tampering, and unauthorized auto-updates that command-name matching misses. After the Mini Shai-Hulud campaign demonstrated agent-config injection at scale, monitor the developer's MCP configuration paths for unauthorized writes as a first-class control: file-integrity monitoring on `~/.claude.json`, `~/.claude/settings.json`, `~/.claude/mcp.json`, `~/.kiro/settings/mcp.json`, `.vscode/tasks.json`, and the equivalent paths for Cursor, Continue, and Windsurf — alert on any addition of `SessionStart` hooks, `runOn: folderOpen` tasks, or new MCP server entries. Pair with auditd/Falco on Linux, EndpointSecurity framework on macOS, or Sysmon on Windows to catch the underlying file-write events; commit `.claude/` and `.vscode/` directories to version control so unauthorized modifications surface as unexpected diffs in `git status`. After Miasma/Hades (June 2026) extended config injection to `.gemini/settings.json`, `.cursor/rules/*.mdc`, `.cursorrules`, and GitHub Copilot instruction files — and delivered some of it via `git pull` from a poisoned upstream rather than any package install — broaden the FIM scope to every AI assistant's config surface (14+ assistants were targeted) and treat config files arriving in upstream commits as untrusted until reviewed; tools like [Watchdog](https://www.yuyjo.com/archives/62856) now monitor these paths specifically, using SHA-256 snapshots with HMAC verification to catch hook injection, shell-alias injection, and MCP configuration tampering with diff/revert/approve workflows. Extend installation audits beyond lifecycle scripts: enumerate `site-packages` for unexpected `.pth` files (the Hades execution vector — any line starting with `import` in a `.pth` file runs at every interpreter startup) and flag npm packages whose `binding.gyp` is not justified by a genuine native build, since both vectors execute outside the `postinstall` surface that `ignore-scripts` controls. Use the [NSA AISC MCP Cybersecurity Information Sheet](https://media.defense.gov/2026/Jun/02/2003943289/-1/-1/0/CSI_MCP_SECURITY.PDF) (May 20, 2026) as an audit baseline — it names pre-deployment server vetting with scanner tooling (MCP Scanner, Ramparts, CyberMCP, Proximity), invocation logging with cryptographic result hashes, and component zoning as expected controls for MCP deployments. Update incident-response runbooks for the destructive tripwire confirmed in the May 19 @antv wave: the worm's `gh-token-monitor` daemon (macOS LaunchAgent / Linux systemd) polls GitHub every 60 seconds and runs `rm -rf ~/; rm -rf ~/Documents` on a 40x token-revocation response, and JFrog's analysis identified a second `kitty-monitor` daemon performing GitHub commit-search C2 (`firedalazer` keyword) that survives token rotation. The correct response order is now (1) network-isolate, (2) stop and remove the persistence daemons, (3) only then rotate npm/GitHub/cloud tokens; running [Intrudify's mini-shai-hulud-scanner](https://github.com/Intrudify/mini-shai-hulud-scanner) before any token rotation surfaces the persistence artefacts that need to be removed first. Evaluate gateway-class tools that sit between agents and MCP servers: [MCP Manager](https://mcpmanager.ai/) provides RBAC, SSO, SCIM, and end-to-end logging across MCP traffic; [Golf.dev](https://golf.dev/) centralizes MCP server deployment behind an authentication-gated master server with prompt-injection mitigation and MCP-specialized rate limiting; [Obot MCP Gateway](https://obot.ai/) ships project sharing, power-user roles, and governance upgrades for enterprise MCP fleets; [TrueFoundry MCP Gateway](https://www.truefoundry.com/mcp-gateway) layers federated login (Okta, Azure AD), RBAC, and OAuth 2.0 token brokering; [MintMCP](https://www.mintmcp.com/) provides a SOC 2 Type II certified gateway with malicious-command blocking, file-access restrictions, and LLM monitoring. | As of March 2026, the MCPS project offers the first MCP-specific signing layer (five trust levels L0–L4, from unsigned through audited), but adoption is early-stage. General package manager features (npm provenance, PyPI attestations, Sigstore) remain the primary verification mechanism, with [npm Trusted Publishing](https://docs.npmjs.com/trusted-publishers/) (OIDC) now strongly preferred over long-lived publish tokens following the May 19 npm mass-invalidation of write-enabled tokens with 2FA bypass — though Trusted Publishing alone is not sufficient, since the May 11 TanStack compromise [showed](https://snyk.io/blog/tanstack-npm-packages-compromised/) that a `pull_request_target` workflow with cache poisoning can still extract the short-lived OIDC publish token from GitHub Actions runner memory, so verification must also pin workflow definitions, restrict `id-token: write` scope, and avoid `pull_request_target` triggers on fork-supplied content. A harder result followed on May 19-20: [Endor Labs](https://www.endorlabs.com/learn/mini-shai-hulud-returns-42-malicious-npm-packages-fake-sigstore-badges-in-antv-ecosystem-attack) documented that the Mini Shai-Hulud Wave 3 worm now **forges cryptographically valid Sigstore provenance attestations at runtime** — it mints fresh OIDC tokens from the compromised runner's environment, submits ephemeral P-256 keys to Fulcio, logs entries to Rekor, and emits complete DSSE bundles that pass `npm install --enforce-attestations` and every standard verification tool, so the "green provenance badge" can no longer be treated as proof that the build was authorized. The MCP spec itself does not enforce audit, sandboxing, or verification — enforcement is left to implementers. A scan of 39 agent frameworks found 13 failed baseline security tests. Research by Invariant Labs found 5.5% of MCP servers exhibit tool poisoning characteristics — malicious instructions hidden in tool descriptions that are invisible to users but processed by AI models. Chained vulnerabilities in Anthropic's own `mcp-server-git` (CVE-2025-68145, CVE-2025-68143, CVE-2025-68144) demonstrated that even vendor-maintained components lack hardening, achieving full RCE via malicious `.git/config` files. Tool shadowing — where a malicious server registers a tool with the same name as a trusted one, exploiting naming collisions in the absence of namespace isolation — is an emerging class of supply-chain attack; CVE-2026-30856 in the Tencent WeKnora framework demonstrated this concretely, enabling tool execution hijacking through identifier collisions until patched with UUID-based tool identifiers in v0.3.0. Academic work on Secure Tool Manifests (2026) proposes ECDSA-signed manifests with Merkle-tree transparency logs for tamper detection, demonstrating near-linear scalability across 50,000 manifest instances with under 5% cryptographic overhead — though this remains a research prototype rather than a deployable standard. The "NeighborJack" exposure pattern (2026) revealed hundreds of MCP servers defaulting to `0.0.0.0` binding, making them reachable from adjacent network hosts and exposing command injection paths — a supply-chain risk even for locally-installed components. The MCP spec's 2025-11-25 revision formalized governance (working groups, SDK tiering, formal governance structure) but still does not mandate integrity verification, leaving enforcement entirely to implementers and gateway providers. As of March 2026, lockfile-based integrity verification is emerging as a practical intermediate layer: MCPShield's SHA-512-hashed `mcp.lock.json` and MCPTrust's capability-pinned lockfiles provide deterministic snapshots that CI gates can enforce, catching drift before production — though neither is a substitute for cryptographic attestation of the server binary itself. The TeamPCP supply chain campaign (March 19–24, 2026) demonstrated cascading compromise across five ecosystems: attackers weaponized the Trivy security scanner via stolen CI/CD secrets, then pivoted to npm packages, Docker Hub (44 Aqua Security repositories defaced), Checkmarx GitHub Actions, and LiteLLM on PyPI (95 million monthly downloads, present in 36% of monitored cloud environments). LiteLLM is a common transitive dependency in MCP servers and agent frameworks, meaning compromised versions propagated silently into environments that never explicitly installed it. Mandiant estimated 1,000+ SaaS environments impacted, with projections reaching 5,000–10,000. The attack underscores that MCP supply chain integrity must extend beyond direct dependencies to transitive dependency chains — particularly AI infrastructure libraries like LiteLLM that store API keys for 100+ LLM providers. A March 2026 audit of 194 MCP server packages (via agentaudit.dev) found 118 security findings: 5 critical (4.2%), 9 high (7.6%), 63 medium (53.4%), with unsanitized shell command inputs and environment variable leakage as the most prevalent vulnerability patterns. AgentSeal's April 2026 scan of 1,808 MCP servers found 66% had at least one finding — 43% shell/command injection, 20% tooling infrastructure, 13% authentication bypass, 10% path traversal — confirming that the underlying vulnerability distribution has not shifted materially since the early-2026 CVE surge despite a growing scanner ecosystem. The MCP security scanning landscape is rapidly expanding — Cisco MCP Scanner, ToolTrust, MCPShield, mcpscan.ai, Snyk agent-scan, and AgentSeal's [awesome-mcp-security](https://github.com/AgentSeal/awesome-mcp-security) (daily-updated scores for 800+ servers via nine analyzers) now provide overlapping but complementary coverage — though no single tool covers the full attack surface from source through runtime. The most sobering development is Ox Security's April 15, 2026 disclosure of a systemic flaw in the official MCP SDKs (Python, TypeScript, Java, Rust): the STDIO transport executes launch commands even when the child process fails to start, with no toolchain-level sanitization warning, enabling arbitrary command execution on any system that connects to a malicious or compromised MCP server configuration. Ox estimates 150 million cumulative downloads, 7,000+ publicly accessible servers, and up to 200,000 vulnerable instances; Anthropic declined to patch, stating the STDIO model is a "secure default" and input sanitization is the developer's responsibility. The implication for supply-chain verification is that integrity attestation of the server binary is necessary but not sufficient — even a cryptographically verified server inherits this transport-level exposure from the SDK it was built with, and allowlisting must therefore pin SDK versions as well as server versions. The downstream-mitigation pattern is now visible in production code: LiteLLM's CVE-2026-30623 patch (v1.83.7-stable, April 21, 2026) introduces an `MCP_STDIO_ALLOWED_COMMANDS` constant restricted to seven launcher binaries with extension via `LITELLM_MCP_STDIO_EXTRA_COMMANDS`, defended at both request validation and client instantiation, and locks test/preview endpoints behind a `PROXY_ADMIN` role — confirming that the workable defense at the host layer is a hardcoded launcher allowlist rather than runtime sanitization, because attacker-supplied command strings cannot reliably be made safe by escaping. The same pattern is being formalized at the configuration layer: the OpenAI Codex SHA-256 proposal (issue #15814, March 26, 2026) adds an optional `sha256` field to `requirements.toml` allowlist entries so the host computes and verifies binary hashes before process spawn, with OpenTelemetry audit logging of verification results — closing the gap between command-name allowlisting and verified binary identity. As of May 2026, neither pattern is mandated by the MCP spec, but both are reasonable verification expectations for any production deployment. The late-April 2026 supply-chain wave reinforced this — CanisterSprawl, xinference, the Bitwarden CLI cascade, and PyTorch Lightning 2.6.2/2.6.3 (the last with on-import payload execution that runs even before any user-visible code path) collectively demonstrated that auto-updating dependencies, postinstall hooks, and import-time side effects make scan-after-install structurally insufficient; pre-install hash pinning, command-launcher allowlisting, and SDK-version pinning together form the minimum credible posture. The Mini Shai-Hulud campaign that closed out the month (April 29-30, 2026, hitting SAP CAP packages and `intercom-client` ~7.0.4/7.0.5) crossed a final architectural line: rather than merely stealing MCP credentials, the payload writes `.claude/settings.json` with a `SessionStart` hook and `.vscode/tasks.json` with `"runOn": "folderOpen"` so that future developer sessions silently re-execute attacker code. This collapses the historically separate categories of "supply-chain compromise" and "MCP configuration tampering" into a single attack chain — meaning component-integrity verification must now extend to monitoring the agent's own configuration directory for unauthorized writes, not just to verifying the integrity of installed packages. Microsoft's vscode issue [#309406](https://github.com/microsoft/vscode/issues/309406) (filed sixteen days before the Mini Shai-Hulud campaign) flagged exactly this `tasks.json` `folderOpen` trigger as an unaddressed attack surface, and the campaign exploited it within the disclosure window — illustrating that for IDE-resident MCP infrastructure, even pre-disclosed editor-level vectors can be operationalized faster than mitigations land. The May 11-12 Wave 2 re-run drove the same point home from the package-registry side: TanStack's compromise window was six minutes (19:20 to 19:26 UTC), Mistral's SDK landed on both npm and PyPI before either registry's defenses fired, and guardrails-ai@0.10.1 — a library deliberately chosen by teams trying to *improve* AI safety — became a credential-stealer delivery vehicle. The May 19 TeamPCP `durabletask` push narrowed that window further to a 35-minute three-version burst on Microsoft's own official PyPI listing. As long as registry-side defenses operate on hours-to-minutes detection latency while attacker publishing windows are minutes, post-publication scanning cannot be the integrity boundary; only pre-install hash pinning, OIDC/trusted-publishing provenance, and host-side launcher allowlists keep up. OX Security's broader registry-poisoning trial balloon — disclosed alongside the April STDIO advisory and revisited in May coverage — landed a malicious test entry in 9 of 11 sampled MCP registries and demonstrated command execution against six live production platforms (LiteLLM, LangChain, IBM LangFlow); the registries themselves are now part of the attack surface that allowlist policies have to model, not a trust foundation that policies can lean on. Two June 2026 developments cut at the remaining verification layers: [CVE-2026-45321](https://www.tenable.com/blog/mini-shai-hulud-frequently-asked-questions) formalized the TanStack chain as a CI/CD-configuration weakness whose output was malicious packages carrying *valid SLSA Build Level 3 provenance* — confirming at the CVE level what Endor Labs documented for the @antv wave — and the Hades payload's "AI analyst misdirection" embeds prompt-injection blocks aimed at LLM-based security scanners, meaning the LLM-as-judge engines used by Cisco MCP Scanner, mcpscan.ai, and similar tools are themselves now an adversarial target whose verdicts need corroboration from non-LLM analyzers (YARA, static taint analysis, hash pinning) before they gate an admission decision. |

## Real-World Incidents & Research (2025-2026)

- **WhatsApp MCP Tool Poisoning (April 2025):** A malicious MCP server silently exfiltrated a user's entire WhatsApp chat history through tool poisoning, exposing personal conversations and business data. One of the earliest documented cases of MCP-specific data exfiltration.
- **GitHub MCP Prompt Injection (May 2025):** A prompt-injection attack against a GitHub MCP integration enabled theft of private repository contents, internal project details, and personal financial information. Overly broad API tokens amplified the blast radius.
- **Asana MCP Cross-Tenant Leak (June 2025):** An access control flaw in an Asana MCP integration permitted cross-tenant data visibility between different customer organizations.
- **Anthropic MCP Inspector RCE (June 2025):** An unauthenticated remote code execution vulnerability in the MCP Inspector development tool exposed developer workstations' filesystems, API keys, and environment secrets.
- **CVE-2025-6514 — mcp-remote Command Injection (July 2025):** Critical command-injection bug (CVSS 10.0) in the `mcp-remote` npm package affected 437,000+ downloads. The package's OAuth proxy passed malicious endpoints directly to system shell handlers (PowerShell on Windows, shell commands on Unix), enabling credential theft, Git repository exfiltration, and full system compromise. The first documented case of full system compromise achieved through MCP infrastructure.
- **Anthropic Filesystem MCP "EscapeRoute" Sandbox Escape ([CVE-2025-53109 / CVE-2025-53110](https://cymulate.com/blog/cve-2025-53109-53110-escaperoute-anthropic/), disclosed March 2025, patched July 1, 2025):** The official filesystem MCP server restricted operations to configured "allowed directories" using naive prefix matching, so a path such as `/tmp/allow_dir_sensitive` passed validation against an `/tmp/allow_dir` allowlist (CVE-2025-53110, CVSS 7.3). A second flaw validated a symlink's parent directory rather than its real target, letting an attacker symlink to files like `/etc/sudoers` and write a malicious LaunchAgent `.plist` for code execution (CVE-2025-53109, CVSS 8.4). Both were fixed in `@modelcontextprotocol/server-filesystem` 2025.7.1 — the canonical demonstration that an in-process path allowlist is not a substitute for an OS-enforced least-privilege sandbox.
- **Postmark MCP Supply Chain Breach (September 2025):** Attackers injected a single line of malicious code into an npm package used for sending transactional emails through Postmark in AI automation pipelines. Compromised MCP servers silently BCC'd every outgoing email to the attackers.
- **Smithery Registry Path Traversal (October 2025):** A path-traversal bug in the Smithery.ai build configuration exposed Docker credentials controlling 3,000+ hosted MCP servers and their API tokens.
- **Figma MCP Command Injection (October 2025):** A command-injection flaw via unsanitized input in the Figma/Framelink MCP integration enabled shell execution and design data access.
- **Supabase Cursor Agent Exploit (mid-2025):** Supabase's Cursor agent, running with privileged service-role access, processed support tickets containing user-supplied input as commands. Attackers embedded SQL instructions to exfiltrate sensitive integration tokens by leaking them into a public support thread.
- **CVE-2026-21852 — Claude Code Project File RCE (January 2026):** Critical vulnerability allowing remote code execution and API credential theft through malicious project configuration files.

For C10.1, the implications are concrete: the registry-patch floor for any production admission policy is now v1.7.9 (not v1.7.7 as the earlier four-CVE wave suggested), OCI-backed allowlist entries need host-side independent verification of image digest plus label set rather than implicit trust in the registry's ownership decision, and registry operators must treat "upstream OCI 429 immediately followed by a successful publish" as a single combined signal worth alerting on.

It warns that MCP adoption has outpaced safeguards, highlights the protocol's inversion of the client-server trust pattern, and makes nine recommendations including component zoning with scoped data-source access, pre-deployment server vetting with scanner tooling (naming MCP Scanner, Ramparts, CyberMCP, and Proximity), logging of all tool and model invocations with parameters, identities, and cryptographic hashes of results, and filtering egress proxies or DLP for external MCP connections. [PipeLab's analysis](https://pipelab.org/blog/nsa-mcp-security-guidance/) notes the CSI acknowledges gaps the guidance cannot close from outside — protocol-native message signing, registry trust, and first-lookup malicious-resolution attacks "need their own moves" at the spec level. For C10.1, the CSI gives auditors a citable government baseline for requiring supply-chain vetting evidence at admission time.
- **Miasma Supply-Chain Worm — Four Delivery Pivots in Seven Days (June 1–7, 2026):** A TeamPCP-derived campaign ([Phoenix Security](https://phoenix.security/miasma-azure-hades-pypi-supply-chain-worm-2026/), [Upwind](https://www.upwind.io/feed/large-scale-npm-pypi-supply-chain-campaign-suspected-across-multiple-popular-packages), [NHS England cyber alert CC-4781](https://digital.nhs.uk/cyber-alerts/2026/cc-4781)) that compromised 448 total artifacts while pivoting delivery mechanisms every 48–72 hours, each pivot aimed at the detection blind spot left by the previous wave's disclosures. Wave 1 (June 1): 32 `@redhat-cloud-services` npm packages (~117K weekly downloads) via `preinstall` hooks, bypassing code review. Wave 2 (June 4): 57 npm packages (~647K monthly downloads) via `binding.gyp` execution — payload runs during native compilation, so packages allowlisted for legitimate native builds re-open the attack surface that `ignore-scripts` closed. June 5: repository injection — a previously compromised contributor account (linked to the May 19 `durabletask` attack) pushed commit `5f456b8` to `Azure/durabletask`, planting `.claude/settings.json` (Claude Code `SessionStart`), `.gemini/settings.json` (Gemini CLI), `.cursor/rules/setup.mdc` (Cursor), and `.vscode/tasks.json` (VS Code auto-run) — all pointing at a 4.6 MB obfuscated `.github/setup.js` that executes when a developer opens the repo in an AI coding tool, with no package installation involved. GitHub's automated response disabled 73 repositories across the Azure, microsoft, Azure-Samples, and MicrosoftDocs organizations in 105 seconds (16:00:50–16:02:35 UTC), taking `Azure/functions-action@v1` offline and breaking deployment pipelines globally — the first case where remediation automation caused a major availability incident in its own right. Zero CVEs were assigned to any Miasma artifact during active exploitation. Credential sweep covered GitHub PATs, AWS/GCP/Azure secrets managers, Kubernetes service-account tokens, npm/PyPI tokens, SSH keys, and Docker configs, exfiltrating via GitHub dead-drop repositories with Hades-themed naming.
- **Hades PyPI Wave — `.pth` Startup Hooks and AI Analyst Misdirection (June 7–8, 2026):** The Miasma campaign's PyPI phase ([StepSecurity](https://www.stepsecurity.io/blog/the-hades-campaign-pypi-packages), [Phoenix Security](https://phoenix.security/miasma-azure-hades-pypi-supply-chain-worm-2026/)) published 37 malicious wheels across 19+ packages in the computational-biology and agent-framework ecosystems — `dynamo-release@1.5.4`, `spateo-release@1.1.2`, `coolbox@0.4.1–0.4.2`, `ensmallen@0.8.101`, `embiggen@0.11.97`, `gpsea@0.9.14`, `napari-ufish`, and others, all sharing a single `.pth` payload SHA-256 (`c539766062555d47716f8432e73adbe3a0c0c954a0b6c4005017a668975e275c`). The `.pth` mechanism executes `import`-prefixed lines at every Python interpreter startup — `python`, `pip`, even `pytest` — requiring no package import and defeating both `--ignore-scripts` and import-time scanning. Three escalations matter for MCP component integrity: (1) "AI analyst misdirection" — prompt-injection blocks at the head of the payload designed to hijack LLM-based security scanners into producing false negatives, the first documented malware defense aimed at LLM-as-judge analysis engines; (2) cross-platform memory scrapers (Linux `/proc/{pid}/mem`, macOS Mach kernel APIs via ctypes, Windows dynamically compiled C# calling `ReadProcessMemory`) targeting CI runner process memory; and (3) C2 camouflage as HTTPS to a non-existent `api.anthropic.com/v1/api` route, blending exfiltration into legitimate AI-developer traffic. The payload plants backdoors in 14+ AI assistant config surfaces (`.cursorrules`, `.claude/settings.json`, Copilot instruction files) and carries the same `gh-token-monitor` wiper-deterrent as the May @antv wave. Detection artifacts: `/tmp/.bun_ran` lock files, `~/.config/systemd/user/` and `~/Library/LaunchAgents/` persistence, and exfil repos named `stygian-cerberus-*` with the description "Hades – The End for the Damned."
- **CVE-2026-45321 — TanStack CI/CD Chain Formalized (May–June 2026):** [Tenable's FAQ](https://www.tenable.com/blog/mini-shai-hulud-frequently-asked-questions) documents the CVE assigned to the chained exploitation of three weaknesses in TanStack's GitHub Actions configuration during Mini Shai-Hulud Wave 2. The decisive detail: because the attacker hijacked the legitimate build pipeline rather than publishing from an unauthorized account, the 84 malicious versions carried valid SLSA Build Level 3 provenance attestations. Together with Endor Labs's @antv runtime-Sigstore-forging analysis, the CVE cements the "valid-but-unauthorized" provenance class as a recognized vulnerability pattern rather than a one-off — provenance verification proves build origin, not build authorization.
- **Trustworthy MCP Registry Architectural Blueprint (May 2026):** A peer-reviewed proposal ([Future Internet 18(5):243](https://www.mdpi.com/1999-5903/18/5/243)) for a three-layer trustworthy MCP registry: RFC 8615 `.well-known` decentralized discovery, Sigstore OIDC-backed provenance for registry entries, and JCS/JWS signing of runtime MCP messages. Evaluation reported ~0.61 ms per signing operation and rejection of all 100 simulated rug-pull attempts. The first academic design connecting registry-layer provenance to message-layer integrity — directly responsive to the "unverified pointer architecture" criticism of the current official registry — though no production registry has implemented it.

### June–July 2026 verification updates

- **AutoJack local MCP process-spawn chain (June 18, 2026):** [Microsoft Defender Security Research](https://www.microsoft.com/en-us/security/blog/2026/06/18/autojack-single-page-rce-host-running-ai-agent/) showed how a page rendered by AutoGen Studio's browsing agent could reach an unauthenticated loopback MCP WebSocket and pass attacker-controlled `StdioServerParams` directly to the local process launcher. The affected development-only surface was hardened in upstream commit `b047730` and never shipped in a PyPI release, so this is a design case study rather than evidence of exploitation. It still gives C10.1.2 and C10.1.3 a precise negative test: from untrusted rendered content, attempt to reach every loopback MCP control endpoint and request an unapproved executable; the endpoint must require authentication, reject the executable and arguments against the server allowlist, and keep any spawned process inside a low-privilege profile or container.
- **Jenkins MCP Server Plugin permission gap (June 24, 2026):** The [Jenkins security advisory](https://www.jenkins.io/security/advisory/2026-06-24/#SECURITY-3759) assigned CVE-2026-57300 to a missing permission check in `getReplayScripts`: versions through `0.177.v629fdb_2557fe` let callers with `Item/Read` retrieve Pipeline replay scripts, while `0.178.vffe5a_e770f3b_` requires `Item/Extended Read`. This is not a sandbox escape, but it demonstrates why a production allowlist cannot stop at a server name. Record the approved version floor and expected permission model, then test with an `Item/Read`-only principal and require denial when it calls `getReplayScripts`.
- **Agentic supply-chain scope now includes tool metadata:** Microsoft's [updated agentic failure-mode taxonomy](https://www.microsoft.com/en-us/security/blog/2026/06/04/updating-taxonomy-failure-modes-agentic-ai-systems-year-red-teaming-taught-us/) treats natural-language tool definitions from MCP servers and registries as supply-chain inputs even when no binary changes. Its operational mitigations are directly testable under C10.1.1 and C10.1.2: generate an SBOM that includes tool dependencies, verify signatures and provenance before installation, pin server and tool-definition versions, monitor description hashes for drift, and force review before a changed definition re-enters the allowlist.
- **Official MCP Registry v1.8.0 identity hardening (July 13, 2026):** The [v1.8.0 release](https://github.com/modelcontextprotocol/registry/releases/tag/v1.8.0) added crates.io/Cargo packages and tightened several trust-boundary checks: on the PAT/OAuth (`github-at`) exchange path, organization namespaces are now granted only to organization owners rather than every member, PyPI and NuGet `mcp-name` matching is anchored and safe across README comment forms, Cargo README retrieval and status handling fail more defensively, and the bundled `mcp-publisher` CLI rejects non-UTF-8 or unpaired-surrogate metadata during `publish` and `validate`. For C10.1.2, deployments should use the v1.8.0 Registry and publisher tooling or carry equivalent backports. On the PAT/OAuth exchange path, test that a non-owner organization member cannot publish under the organization namespace; reject partial or longer-name matches and glued metadata, while accepting valid HTML-comment `mcp-name` tokens (including no-space forms) only when the token matches exactly; separately confirm that `mcp-publisher publish` and `validate` reject non-UTF-8 and unpaired-surrogate metadata; then independently verify the listed artifact because these checks establish publisher authority, not code safety. The [organization-namespace fix](https://github.com/modelcontextprotocol/registry/pull/1383) deliberately covers the PAT/OAuth exchange path only: GitHub Actions OIDC remains organization-wide, and an owner-scoped credential can still overwrite another server in the same namespace until the Registry binds updates to a per-server publisher identity.
- **`2026-07-28` specification release candidate changes revocation timing (May 21, 2026):** The [official release-candidate notice](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/) introduces `ttlMs` and `cacheScope` on list and resource-read results, including `tools/list`, so clients can cache a server's exposed catalog for a declared period. Admission tests should cap accepted TTLs at or below the organization's allowlist-revocation objective, invalidate cached catalogs when a server is removed or its identity tuple changes, and prove that a cached tool entry cannot be invoked after the underlying server is denied. The underlying [SEP-2549](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2549) treats an omitted `ttlMs` as `0`, making the cached result immediately stale; clients may still apply local caching heuristics or serve stale data after a refetch error, so verify the actual client's behavior. The candidate contains breaking changes and is scheduled to become final on July 28, so this behavior should be tested against the exact client and server build rather than assumed from the draft.

These updates sharpen the admission test into three separate decisions: verify the component artifact, authorize the exact server version and metadata, and confine local execution. Passing one decision does not compensate for failure in either of the others.

## Open Research Questions

- [ ] Should the MCP ecosystem adopt a standardized package-signing mechanism? MCPS proposes ECDSA P-256 "Agent Passports" with five trust levels, but adoption is early-stage — is this the right foundation, or should the MCP spec itself incorporate signing requirements?
- [ ] How should MCP configuration files handle secret references in a cross-platform, cross-SDK way? Only ~8.5% of MCP implementations currently use OAuth 2.1 — what would it take to make secretless access the default?
- [ ] Is there a viable model for an MCP server registry with enforced integrity metadata? GitHub, AWS Q Developer, and Kiro now offer registry-based enforcement, but all rely on name-based matching rather than cryptographic attestation — can the ecosystem converge on a standard that ties server identifiers to verified binaries?
- [ ] How do you verify integrity of MCP servers that are dynamically discovered at runtime rather than pre-configured? MCPS's signed tool definitions are one approach, but runtime discovery introduces trust bootstrapping challenges.
- [ ] Can MCPTox-style automated benchmarking be integrated into CI/CD pipelines to detect tool poisoning before deployment?
- [ ] The MCP gateway debate: centralized gateways offer unified enforcement but create single points of failure. Should per-server enforcement (signed envelopes, per-connection authentication) replace or complement gateway architectures?
- [ ] How should organizations handle the "contextual secret leakage" problem, where credentials persist in agent context memory across sessions?
- [ ] Should AI-BOM (AI Bill of Materials) standards extend to MCP server dependencies? Traditional SBOMs miss model versions, training data lineage, and agentic dependency chains — is CycloneDX/SPDX sufficient, or does the MCP ecosystem need purpose-built transparency formats?
- [ ] How should organizations detect and govern "shadow MCP" deployments where developers connect servers to IDEs without IT oversight, bypassing security monitoring? Qualys TotalAI's three-layer detection (network/host/supply-chain) is one approach — are there lighter-weight alternatives?
- [ ] MCP sampling creates a bidirectional control reversal where servers can request LLM completions — should the protocol require explicit user consent for sampling requests, or can capability declarations and prompt sandboxing suffice?
- [ ] Tool shadowing via namespace collisions (demonstrated by CVE-2026-30856) is a gap in the MCP spec. Should tool identifiers be globally unique (UUID-based) by default, or is registry-backed namespace isolation sufficient?
- [ ] NIST's AI Agent Standards Initiative (February 2026) is collecting input on agent security and identity — should the MCP community align with NIST's forthcoming framework for AI agent identity and authorization, or develop MCP-specific standards independently?
- [ ] Enterprise platforms (GitHub Copilot, AWS Q, Kiro, Agentforce, Microsoft) each implement MCP governance differently. Can the ecosystem converge on a standard registration and verification workflow — identity verification, automated scanning, human review, cryptographic signing — or will fragmentation persist?
- [ ] ToolHive demonstrates container-based provenance verification using Sigstore, but most MCP servers are distributed as npm/PyPI packages rather than container images. How should provenance verification extend to non-containerized MCP components?
- [ ] The MCP gateway market is fragmenting (MintMCP, Kong, Bifrost, Lunar.dev, Docker, Lasso) with different enforcement models. Should the MCP spec define a standard gateway API or enforcement interface, or will market consolidation handle this?
- [ ] Lockfile-based integrity verification (MCPShield's SHA-512 `mcp.lock.json`, MCPTrust's capability-pinned lockfiles) provides deterministic snapshots enforceable in CI — but lockfiles verify capability contracts, not binary integrity. Can lockfile verification be composed with cryptographic attestation to close the gap, or are these fundamentally different trust layers?
- [ ] The Claude DXT architecture runs extensions unsandboxed with full system privileges, and Anthropic's position that this "falls outside our current threat model" leaves users exposed. Should MCP hosts be required to sandbox MCP server extensions, or is this an acceptable tradeoff for local development tooling? How should this be reflected in verification standards?
- [ ] DPoP (SEP-1932) and Workload Identity Federation (SEP-1933) are under review for the MCP spec. If ratified, these could eliminate static client secrets and bind tokens to specific client instances — but adoption requires coordinated changes across all MCP SDKs. What is the realistic timeline for this?
- [ ] The Proofpoint Agent Integrity Framework proposes a five-phase maturity model for agent verification. Should MCP server registries require attestation of a minimum maturity level before listing, similar to how app stores require security reviews?
- [ ] The TeamPCP campaign demonstrated that transitive dependencies (e.g., LiteLLM as an AI gateway library) can propagate supply chain compromises silently into MCP servers and agent frameworks. Should MCP integrity verification require recursive dependency scanning, and how deep should the scan go? Current SBOM tooling captures direct dependencies but often misses build-time and runtime transitive chains.
- [ ] The MCP security scanning landscape now includes at least five major tools (Cisco MCP Scanner, ToolTrust, MCPShield, mcpscan.ai, Snyk agent-scan) with overlapping but non-identical coverage. Should the ecosystem converge on a common vulnerability taxonomy and reporting format for MCP-specific findings, analogous to CWE for traditional software?
- [ ] The Ox Security April 2026 STDIO disclosure demonstrates that vulnerabilities baked into the official MCP SDKs propagate to every downstream server built on them. Should allowlisting and component-integrity verification require SDK version pinning as a first-class check alongside server name and version — and should MCP registries track SDK-level CVE exposure per listed server, analogous to how container registries track base-image CVEs?
- [ ] When a vendor declines to patch a disclosed architectural vulnerability (as Anthropic did for both Claude DXT and the STDIO STDIO-execution flaw), what is the correct posture for a verification standard? Should AISVS treat vendor-affirmed architectural risks as implementer-level requirements (e.g., mandate sandboxing at the host layer), accept them as residual risk disclosed in threat models, or recommend replacement of the affected component entirely?
- [ ] The Axios compromise affected transitive dependencies that most MCP server authors never explicitly declared. Traditional SBOM and provenance checks capture direct dependencies well, but transitive chains — particularly for autonomous agent workflows that run `npm install` without human review — remain a blind spot. Is pre-install provenance gating (SLSA Level 3+ verification at install time, not scan time) a realistic requirement for MCP fleets, or is the tooling not yet mature enough to enforce it without breaking developer workflows?
- [ ] The LiteLLM CVE-2026-30623 patch demonstrates that downstream hosts can credibly mitigate the Anthropic STDIO flaw through a hardcoded launcher allowlist (`MCP_STDIO_ALLOWED_COMMANDS`). Should AISVS treat this pattern as a normative requirement for STDIO-launching MCP hosts, or only as guidance until the MCP spec catches up? And if normative, what is the right minimum allowlist — should it be defined per-language-runtime (Node, Python, container runtime) or per-deployment-context (developer machine, CI, production)?
- [ ] The OpenAI Codex SHA-256 allowlist proposal (issue #15814) extends `requirements.toml` from command-name matching to verified binary hash pinning. If this becomes a de-facto standard across IDEs and agent runtimes, how should hash rotation be managed when servers update? Lockfile-style locked-version-with-hash, signed update channels, or operator-mediated rotation? Each has different ergonomics for autonomous agent workflows where updates may happen without human review.
- [ ] The PyTorch Lightning April 2026 compromise executed its payload **on import**, not via postinstall — bypassing the `--ignore-scripts` mitigation that has been the default defensive posture for npm. Should MCP component integrity verification require evidence that a Python package's import path performs no network or filesystem side effects, and how should this be tested at scale across PyPI's hundreds of thousands of packages? Existing static analysis tools struggle with dynamic import-time behavior, particularly when payloads are obfuscated or fetched at runtime.
- [ ] The Mini Shai-Hulud campaign (April 29-30, 2026) collapsed the historically separate categories of "supply-chain compromise" and "MCP configuration tampering" into one attack chain by writing `.claude/settings.json` `SessionStart` hooks and `.vscode/tasks.json` `folderOpen` tasks that re-execute attacker code on every developer session. Should AISVS C10.1 require file-integrity monitoring on the agent's configuration directory (`~/.claude/`, `~/.kiro/`, `~/.cursor/`, project-local `.vscode/`, `.claude/`) as a normative control alongside package-level integrity verification, or treat it as part of C12 monitoring? And should IDE vendors be expected to require explicit user approval before honoring newly-introduced `folderOpen` or `SessionStart` hooks that arrive via `git pull` rather than user action?
- [ ] The official MCP Registry now provides namespace and ownership checks, immutable version metadata, and MCPB `fileSha256` metadata, but artifact hash validation and source-to-build attestation remain client- or downstream-registry responsibilities. Should AISVS require production MCP allowlists to include registry identity plus independent provenance evidence (SLSA/Sigstore/npm provenance/PyPI attestations/OCI digests), or is registry ownership verification enough for Level 1 deployments?
- [ ] CVE-2026-44430 demonstrated that the official MCP Registry itself can carry an unauthenticated SSRF in its namespace-verification path — an IP blocklist that misses IPv6 6to4, NAT64, and site-local prefixes. Should AISVS treat the patch state and SBOM of the registry server as a first-class entry in the allowlist trust chain, alongside the listed servers? And should registry operators be required to enforce network-layer egress controls that block IPv6 transition prefixes and cloud-metadata endpoints rather than relying on application-level blocklists?
- [ ] The SmartLoader Oura MCP campaign (active since February 2026) and the OX Security registry trial balloon (9 of 11 registries successfully poisoned, six live platforms popped) show that registry-side admission cannot be the trust boundary for MCP allowlists. What is the right combination of (a) registry-side namespace ownership verification, (b) host-side launcher allowlist, (c) signed-artifact hash pinning, and (d) maintainer-identity binding for a Level 1 admission decision? And how should a verification standard handle cases where the same display name is held by multiple publishers across different registries?
- [ ] The Mini Shai-Hulud Wave 2 campaign (May 11-12, 2026) targeted `guardrails-ai` — a library deliberately chosen by teams trying to improve AI safety — and the May 19 `durabletask` push compromised a Microsoft-published Python client used in agent workflow orchestration. As AI-safety and AI-orchestration libraries become deliberate supply-chain targets, should MCP allowlisting policies treat any package whose stated purpose is "AI safety" or "agent orchestration" as a higher-trust target requiring stricter provenance evidence (SLSA Level 3+, signed build attestations, and reproducible builds) than general-purpose dependencies?
- [ ] The May 2026 GitHub internal repository exfiltration includes `github-mcp-server` and `github-mcp-server-remote`. When the source code of a trusted first-party MCP server is in adversary hands but no compromise of the published binary has been confirmed, what is the correct allowlist posture? Should the affected server be moved to a heightened-scrutiny tier until a clean rebuild from new infrastructure can be verified, or treated as residual risk pending evidence of exploitation?
- [ ] The Hades campaign's "AI analyst misdirection" embeds prompt-injection blocks designed to make LLM-based security scanners emit false negatives — and several MCP scanners (Cisco MCP Scanner's LLM-as-judge engine, mcpscan.ai's tool-poisoning classifier) depend on exactly that analysis mode. Should verification standards require that any LLM-derived security verdict be corroborated by a non-LLM analyzer before it gates an admission decision, and how should scanner vendors harden their analysis prompts against payloads that are written to be read by the scanner itself?
- [ ] Zero CVEs were assigned across all 448 Miasma/Hades artifacts during active exploitation, and the campaign pivoted delivery mechanisms (preinstall → `binding.gyp` → repository config injection → `.pth` startup hooks) every 48–72 hours — faster than CVE assignment, scanner signature updates, or advisory publication cycles. If CVE-feed-driven gating is structurally blind to worm-class supply-chain campaigns, what should the admission-time signal be instead: publish-cooldown age, maintainer-identity continuity, behavioral sandbox detonation, or something the ecosystem has not built yet? And does GitHub's 105-second mass-disable response (which broke `Azure/functions-action@v1` globally) mean availability impact from automated remediation now belongs in the same risk model?
