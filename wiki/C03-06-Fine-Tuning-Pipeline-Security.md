# C3.6: Fine-Tuning Pipeline Authorization & Reward Model Integrity

> [Back to C03 Index](C03-Model-Lifecycle-Management.md)
> **Last Researched:** 2026-03-29

## Purpose

Fine-tuning pipelines are high-privilege operations that can alter deployed model behavior at scale. Multi-stage pipelines compound this risk because a compromise at any intermediate stage produces a subtly altered artifact that subsequent stages accept. Reward models used in RLHF are ML artifacts subject to tampering yet often treated as static infrastructure rather than versioned, validated components.

This section addresses the separation of duties for fine-tuning authorization, the integrity of reward models used in RLHF, automated detection of reward hacking and over-optimization, and per-stage integrity verification in multi-stage pipelines. Research has demonstrated that fine-tuning can remove RLHF safety protections, and the Checkpoint-GCG attack (2025) showed that intermediate fine-tuning checkpoints can be exploited as stepping stones for adversarial attacks.

## Requirements

|---|-------------|:-----:|-----------------|----------------------|--------------|
| **3.6.1** | **Verify that** initiating a fine-tuning or retraining run requires authorization from a person who did not request the run (separation of duties). | 3 | Insider threat: a single compromised or malicious actor launching a poisoned fine-tuning run that alters deployed model behavior at scale; unauthorized fine-tuning removing RLHF safety alignment (demonstrated via OpenAI fine-tuning API where few-shot examples bypassed safety protections); lack of accountability for training decisions; regulatory non-compliance with dual-control requirements. | Review fine-tuning pipeline access controls for separation of duties enforcement. Verify that the person approving a run is distinct from the requester (e.g., GitHub CODEOWNERS requiring a different reviewer, CI/CD approval gates with identity checks, ServiceNow dual-approval workflows). Test by having the same user attempt to both request and approve a run--confirm rejection. Inspect audit logs for requester and approver identities on past training runs. | NIST AI RMF (March 2025 update) explicitly identifies separation of those building models from those validating them as a governance best practice under the GOVERN function. Research has demonstrated that fine-tuning can remove RLHF protections--PKU's Safe-RLHF framework and SafeMoE (ICLR 2025) address this by integrating safety constraints directly into fine-tuning, but they do not replace organizational separation of duties. Most ML teams lack formal dual-control processes for training runs--CI/CD-based approval gates (GitHub Actions required reviewers, GitLab approval rules) are the most practical implementation path. Level 3 reflects the organizational maturity gap: few organizations enforce dual authorization for ML training operations. *(Added 2026-03)* |
| **3.6.2** | **Verify that** reward models used in RLHF fine-tuning are versioned, cryptographically signed, and integrity-verified before use in a training run. | 2 | Reward model tampering producing a subtly corrupted reward signal that steers the policy model toward unsafe or adversarial behavior; substitution of a backdoored reward model enabling targeted manipulation of RLHF alignment outcomes; use of stale or unvalidated reward models introducing drift in safety alignment; supply chain attack where a compromised reward model artifact poisons downstream policy training. | Inspect the RLHF pipeline for reward model handling: verify reward model artifacts are stored in a versioned registry with cryptographic signatures (OpenSSF Model Signing / Sigstore cosign). Confirm the training pipeline verifies the signature and computes an integrity checksum before loading the reward model. Test by substituting a modified reward model artifact and confirming the pipeline rejects it. Check that reward model version is recorded in the training run metadata alongside the policy model version. | Reward models are often treated as static infrastructure rather than versioned ML artifacts--this is a significant gap. OpenSSF Model Signing (OMS) v1.0 can sign reward model artifacts just like any other model format. The challenge is organizational: reward models may be trained by a different team or sourced externally, making provenance tracking harder. Anthropic's research on reward model integrity includes classifiers for detecting reward hacking, suggesting reward models themselves need integrity guarantees. Most MLOps platforms (MLflow, W&B) can version reward models, but few organizations explicitly include them in their signing and verification workflows. The NSA/CISA advisory (March 2026) recommends cryptographic signing across the model lifecycle, which applies to reward models as first-class artifacts. *(Added 2026-03)* |
| **3.6.3** | **Verify that** RLHF training stages include automated detection of reward hacking or reward model over-optimization (e.g., held-out human-preference probe sets, divergence thresholds, or KL penalty monitoring), with the run blocked from promotion if detection thresholds are exceeded. | 3 | Reward hacking: the policy model exploits spurious correlations in the reward model (length bias, sycophancy, code gaming) to achieve high reward scores while violating genuine human preferences; over-optimization causing Goodhart's Law failures where the model optimizes the proxy reward at the expense of actual alignment; subtly misaligned models passing superficial evaluations but behaving unsafely in production edge cases. MITRE ATLAS [AML.T0043](https://atlas.mitre.org/techniques/AML.T0043) (Craft Adversarial Data). | Inspect the RLHF training pipeline for reward hacking detection mechanisms. Verify at least one detection approach is implemented: held-out human-preference probe sets evaluated periodically during training, KL divergence penalty monitoring against the reference policy, reward score distribution analysis for outlier detection, or dedicated detection frameworks. Confirm that exceeding detection thresholds automatically blocks the run from promotion to production. Test by injecting a training run with known reward-hacking characteristics and verifying the detection gate fires. Review historical training logs for detection metric trends. | Active research area with several promising approaches as of early 2026: Adversarial Reward Auditing (ARA, Feb 2026) trains an Auditor network on reward model latent-space activations to distinguish genuine alignment from exploitation, achieving 0.82 detection AUC with cross-domain transfer (0.75-0.77 for sycophancy/length bias). InfoRM (information-theoretic reward modeling) uses a variational information bottleneck to filter spurious reward signals and introduces Mahalanobis Outlier Probability (MOP) as a quantitative metric for reward hacking severity. Bayesian Non-Negative Reward Modeling (BNRM, Feb 2026) uses sparse non-negative latent factors to create disentangled reward representations. Standard KL penalty monitoring (PPO with KL constraint) remains the most widely deployed approach but is insufficient alone. Level 3 reflects the research maturity gap--detection frameworks exist in academic literature but few are production-hardened. *(Added 2026-03)* |
| **3.6.4** | **Verify that** in multi-stage fine-tuning pipelines, each stage's output is integrity-verified before the next stage consumes it, and intermediate checkpoints are registered as distinct artifacts enabling per-stage rollback. | 3 | Compromise at any intermediate fine-tuning stage producing a subtly altered artifact that subsequent stages accept as legitimate; inability to isolate which pipeline stage introduced a safety regression; Checkpoint-GCG attack (2025) demonstrated that intermediate fine-tuning checkpoints can be exploited as stepping stones for adversarial attacks; cascading corruption across pipeline stages without detection; inability to perform per-stage rollback when the final artifact fails validation. | Inspect multi-stage fine-tuning pipeline for checkpoint handling: verify each stage's output is registered as a distinct artifact in the model registry with a unique version/digest. Confirm integrity verification (checksum or cryptographic signature check) occurs before a downstream stage consumes the checkpoint. Test by corrupting an intermediate checkpoint and verifying the next stage rejects it. Verify per-stage rollback capability by rolling back to a specific intermediate checkpoint and confirming the pipeline can resume from that point. Review artifact lineage to confirm each stage's input and output are linked. | DevSecMLOps frameworks (OpenSSF MLSecOps, 2025) recommend integrating security checks directly within ML pipeline stages rather than only at pipeline boundaries. Recovery procedures should validate checkpoint integrity before restoration by verifying checksums, comparing timestamps, and validating compatibility with the current pipeline version. AWS SageMaker and similar platforms support checkpointing and model parallelism with fine-tuning, but checkpoint integrity verification between pipeline stages is typically custom logic. The main challenge is that multi-stage pipelines (e.g., SFT → reward model training → RLHF → DPO refinement) may span different compute environments and teams, making consistent integrity verification across stages operationally complex. OpenSSF Model Signing (OMS) v1.0 can sign intermediate checkpoints as distinct artifacts. Level 3 reflects the tooling and organizational maturity required for consistent cross-stage integrity verification. *(Added 2026-03)* |
