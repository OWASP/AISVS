# C8.1: Access Controls on Memory & RAG Indices

> [Back to C08 Index](C08-Memory-and-Embeddings.md)
> **Last Researched:** 2026-04-16

## Purpose

Enforce fine-grained access controls and query-time scope enforcement for every vector collection. Without proper access controls on vector stores, similarity queries can cross tenant boundaries, unauthorized users can retrieve or modify embeddings, and poisoned documents cannot be traced back to their source. As of April 2026, major managed vector databases (Pinecone, Weaviate, Qdrant) offer RBAC, JWT authentication, namespaces, and compliance certifications (SOC 2, HIPAA, ISO 27001). Milvus 2.6, now GA on Zilliz Cloud as of early 2026, adds hot/cold tiering and row-level RBAC using bitmap-indexed permission arrays, moving beyond collection-scoped RBAC to true per-row authorization. However, UpGuard's April 2025 scan found 1,170 internet-exposed Chroma instances, with roughly a third allowing anonymous access — a reminder that defaults matter. The fundamental architectural gap that keeps surfacing in incident reports is **permission stripping at ingestion**: when a RAG connector converts a SharePoint page, Confluence article, or Notion doc into embeddings, the source ACL is dropped unless the pipeline explicitly preserves it, leaving the vector store to either re-enforce permissions from scratch or fall back to application-layer checks. Microsoft Copilot deployments have made this risk acute in 2026 — pre-existing SharePoint oversharing that was merely sloppy before is now actively dangerous because Copilot surfaces any content that is technically accessible, even when the access was unintended. Zanzibar-style authorization engines have emerged as the dominant answer: SpiceDB (used by OpenAI, Workday, Netflix, Turo) ships official RAG integrations with Pinecone and LangChain/LangGraph, OpenFGA is GA as a CNCF sandbox project with Couchbase Vector Search integration for agentic pipelines, and Cerbos query-plan mode rewrites authorization policies into filter expressions that get pushed directly into the vector query. On the provenance front, RAGShield (April 2026) introduced C2PA-inspired cryptographic document attestation for RAG ingestion pipelines, and organizations are extending CycloneDX ML-BOM to include integrity attestation records for retrieval sources. The attack landscape continues to expand: the Black-Hole Attack (Li et al., April 2026) demonstrated that injecting vectors near the geometric centroid of embedding space achieves 99.85% top-10 contamination at just 1% poisoning rates, while the Retrieval Pivot Attack (Thornton, February 2026) showed that hybrid vector+knowledge-graph RAG systems amplify leakage 160–194x compared to vector-only retrieval. HijackRAG (observed 97% success rate at manipulating retrieval outcomes) joined PoisonedRAG as a standard reference attack in 2026 pentest playbooks. RevPRAG (EMNLP 2025) now provides a detection pipeline achieving 98% TPR with 1% FPR by analyzing LLM activation patterns during generation. Anomaly detection is maturing — practical thresholds of 0.85 cosine similarity for individual documents and 0.90 for batch clustering have been validated as effective baselines — but no major vector database offers native anomaly detection yet. On the cryptographic front, homomorphic encryption for vector search is moving from research to early production: CryptoLab's enVector product uses the CKKS scheme to enable similarity search over encrypted vectors without decryption, and the Hermes system (updated March 2026) demonstrates FHE-native vector queries inside MySQL via OpenFHE, achieving 3,400x encryption throughput improvement over scalar FHE baselines. Azure AI Search now offers native Entra-based document-level ACLs with POSIX-style access control lists (REST API 2025-05-01-preview), enabling query-time token evaluation against sensitivity labels — a significant step toward database-enforced rather than application-layer access control for RAG. Databricks Mosaic AI Vector Search (GA) inherits Unity Catalog ACLs end-to-end, so the same table-level governance that protects analytical workloads applies to vector retrieval without additional policy duplication. On the supply-chain front, Milvus shipped 2.6.5 in early 2026 to patch **CVE-2025-64513** (critical) alongside CVE-2025-63811 and bundled Go-toolchain fixes for CVE-2025-68121, CVE-2026-27142, and CVE-2026-25679 — a reminder that vector database patching discipline is now an explicit operational concern, not a research edge case.

| **8.1.1** | **Verify that** vector insert, update, delete, and query operations are enforced with namespace/collection/document-tag scope controls (e.g., tenant ID, user ID, data classification labels) with default-deny. | 1 | Cross-tenant data leakage via unrestricted vector queries; unauthorized document retrieval where Tenant A's similarity search returns Tenant B's chunks. Verify ABAC policies evaluate device posture, network location, and data classification at query time — not just static RBAC roles. | Pinecone supports namespaces natively and added per-index RBAC with project-scoped API keys; Weaviate has multi-tenancy since v1.20 with tenant-aware classes and lifecycle endpoints; Qdrant now offers first-class multitenancy with named collections/tenants, quota controls, sharding, and tenant-level lifecycle APIs (enterprise tier). HoneyBee (2025) demonstrated that dynamic RBAC partitioning for vector databases achieves up to 13.5x lower query latency than row-level security with only 1.24x memory overhead, making per-role access control practical at scale. Many teams still rely on application-layer enforcement only, which is fragile — UpGuard's April 2025 scan found 1,170 internet-exposed Chroma instances, 34% with anonymous access and exposed PII including customer support logs and Canva creator data. In BYOC mode (GA 2024), Pinecone clusters run inside the customer's own AWS/Azure/GCP account for hard isolation. Weaviate's native multi-tenancy physically isolates tenant data at the index level rather than relying on namespace filters alone. Milvus 2.6 (GA early 2026, Zilliz Cloud) adds hot/cold tiering and partition-key-based RBAC refinements. Cisco's 2025 vector database security whitepaper recommends layering ABAC on top of RBAC — evaluating device posture, network location, data classification, and temporal factors per query — and mandating MFA for administrative and bulk-export operations. The Retrieval Pivot Attack (Thornton, February 2026) demonstrated that hybrid vector+knowledge-graph RAG systems amplify cross-tenant leakage 160–194x through 2-hop graph traversals — per-hop authorization that re-checks tenant and sensitivity labels during graph expansion reduces RPR to 0.0 across all tested corpora. For cryptographic isolation, searchable encryption (SE) using KMIP-compliant key providers (AWS KMS, Google Cloud KMS, Azure Key Vault) allows querying encrypted vectors without decryption; Microsoft SEAL, HElib, and Pyfhel provide open-source homomorphic encryption primitives, though fully homomorphic encryption remains computationally expensive for production similarity search. The Hermes system (March 2026) narrows this gap with SIMD-aware FHE packing that achieves 3,400x encryption throughput and 4,000x insertion speedup over scalar FHE via OpenFHE v1.5.0 loadable functions in MySQL. CryptoLab's enVector uses CKKS (fourth-generation HE) to support encrypted vector search in production. Enterprise multi-tenant RAG architectures should follow the silo/pool/bridge isolation model: silo (dedicated index per tenant) for regulated data, pool (shared index with mandatory tenant_id filters) for cost efficiency, and bridge (hybrid routing) for tiered deployments — application-layer filter-only isolation risks "semantic bleed" where a dropped WHERE clause silently exposes cross-tenant content that looks legitimate because vector similarity surfaces semantically related documents. Azure AI Search (REST API 2025-05-01-preview) now provides native Entra-based document-level ACLs with POSIX-style access control lists evaluated at query time via `x-ms-query-source-authorization` tokens, moving access enforcement from application middleware into the search engine itself. Databricks Mosaic AI Vector Search (GA on AWS/Azure/GCP) inherits Unity Catalog governance — Unity Catalog ACLs on the underlying Delta table propagate automatically to the vector index, so role-based access that already governs lakehouse queries also governs retrieval, eliminating the duplicate-policy problem common in bolt-on vector DB deployments. For teams that need Zanzibar-style relationship-based authorization, **SpiceDB** (open-source, used by OpenAI, Workday, Netflix, Turo) now ships official RAG integrations — SpiceDB's query plan generates a pre-filter that is applied to the vector store query, enforcing authorization at retrieval rather than after the LLM response. Authzed publishes deployment guides for SpiceDB + Pinecone and SpiceDB + LangChain/LangGraph, with Testcontainers-based permission testing patterns for CI pipelines. **OpenFGA** (CNCF sandbox project, Auth0 origin) is now GA with Couchbase Vector Search integration for agentic pipelines, performing authorization checks during retrieval so the LLM only sees content the user is allowed to view. Milvus 2.6 introduced **row-level RBAC using bitmap-indexed permission arrays** — each row stores an array of role IDs that can access it, and a BITMAP index makes per-query permission evaluation efficient at scale (Python and Java SDKs only, as of April 2026). Seoul National University's "Permission-Aware RAG" paper (IEEE Access, 2025) formalized IAM-based access filtering across multi-resource environments, defining a reference architecture for unified identity across vector, graph, and relational retrieval. |
| **8.1.2** | **Verify that** API credentials used for vector operations carry **scoped claims** (e.g., permitted collections, allowed verbs, tenant binding). | 1 | Credential misuse or lateral movement — a compromised API key with full read/write across all collections enables mass data exfiltration or poisoning of the entire knowledge base. PoisonedRAG (Zou et al., USENIX Security 2025) showed that just 5 carefully crafted documents can manipulate AI responses 90% of the time, making write-scoped credential control critical. | Inspect API key/token scopes in the vector DB management console. Verify JWT claims or API key metadata restrict operations to specific collections and verbs. Test that a scoped key cannot perform out-of-scope operations. Confirm short-lived credential rotation schedules are enforced and that ingestion-writer, index-maintainer, read-only-RAG, and security-auditor roles are separated. | Pinecone supports project-scoped API keys and per-index RBAC. Qdrant now offers JWT-based authentication with granular claim scoping, read-only API keys, and as of 2025 supports OAuth2/OIDC synchronization for enterprise directory integration. Weaviate Cloud supports ACLs per tenant-aware class. Milvus added RBAC with partition keys in v2.3+. Self-hosted Chroma still ships with authentication disabled by default — UpGuard found a third of internet-facing instances wide open (April 2025). Chroma's Rust-core rewrite (2025) added performance improvements and Chroma v1.5.0 (February 2026) introduced array metadata operators for more precise filtering, but self-hosted authentication remains an external concern (e.g., Cerbos for RBAC/ABAC on RAG pipelines). Cisco's vector DB security whitepaper recommends separating roles by function — ingestion writer, index maintainer, read-only RAG service, security auditor — with short-lived API credentials on defined rotation schedules. Per-tenant encryption keys should be managed separately, with tenant-specific key retrieval on encryption/decryption operations — this ensures that a compromised credential for one tenant cannot decrypt another tenant's vectors even if filter isolation fails. For enterprise deployments, demand contractual guarantees on vector isolation model specifics, namespace enforcement mechanism (session token vs. user parameter), and tenant-scoped audit logging before vendor selection. Compliance certifications: Pinecone (SOC 2, HIPAA, ISO 27001, GDPR), Weaviate Enterprise Cloud (HIPAA on AWS since 2025, SOC 2 Type II), Qdrant Cloud (SOC 2 Type II, HIPAA-ready for enterprise). **Connector permission propagation is the dominant operational risk in 2026** — when RAG pipelines ingest from SharePoint, Confluence, Notion, Google Drive, or Slack, the source ACL is typically stripped at vectorization time and replaced (if at all) by application-layer checks. Glean, Paragon, and LlamaCloud publish permission-aware connector patterns that persist source ACLs as vector metadata and re-check permissions at query time; the alternative is to maintain a centralized permissions service that reconciles upstream ACLs nightly, though stale permissions (team moves, access revocations) remain a weak spot between syncs. Microsoft Copilot deployments have made this risk acute: oversharing that was merely sloppy before is now actively dangerous because Copilot surfaces any content the user technically has access to, even when that access was unintended. Teams deploying enterprise RAG should (1) require credential scoping per upstream system (one set of keys per Confluence space, not a single admin token), (2) propagate source ACLs into vector metadata and enforce at query time, (3) validate permission freshness with scheduled recertification sweeps, and (4) log all access-denied events for audit. Milvus CVE-2025-64513 (patched in 2.6.5, early 2026) is a concrete reminder that credential scope alone is insufficient when the vector store itself has auth bypass flaws — pin minor versions, subscribe to vendor security feeds, and treat vector DB patching as a first-class operational task. |
| **8.1.3** | **Verify that** cross-scope access attempts (e.g., cross-tenant similarity queries, namespace traversal, tag bypass) are detected and rejected. | 2 | Namespace traversal attacks where an attacker crafts queries or metadata filters to escape their tenant boundary. Similar to SQL injection but targeting vector query DSLs and metadata filter syntax. Christian Schneider's 2025 research on "RAG security: the forgotten attack surface" highlights metadata filter injection as an underappreciated vector for cross-tenant leakage. The Phantom attack (late 2024) demonstrated that a single malicious document can remain dormant during normal queries while selectively activating on specific trigger keywords, making static filter checks insufficient without behavioral monitoring. | Perform penetration testing with crafted metadata filter payloads attempting to bypass tenant scoping (e.g., injecting `OR tenant_id != 'attacker'` in filter expressions). Review logs for cross-scope query attempts. Validate that alerting fires on detection. Use Garak or PyRIT for automated prompt-based retrieval manipulation testing. Enforce strict schema validation on ingestion — reject unrecognized metadata keys and unexpected data types to reduce hidden payloads. | Vector query languages vary widely — Weaviate uses GraphQL filters, Pinecone uses JSON filter syntax, Qdrant uses its own filter DSL. Each needs separate injection testing. Filter injection is an emerging attack class with limited tooling for automated testing. Promptfoo now supports RAG-specific red-teaming scenarios including metadata filter injection tests. Chroma v1.5.0 (February 2026) added array metadata operators and regex search (mid-2025), which expand filter expressiveness but also expand the injection surface — test these new operators explicitly. Network-level controls provide defense-in-depth: private endpoints (AWS PrivateLink), IP allow-listing for ingestion and admin consoles, and rate limiting mitigate scraping and brute-force attempts when credential compromise occurs. For hybrid RAG architectures combining vector retrieval with knowledge graphs, the Retrieval Pivot Attack (Thornton, February 2026) showed that shared entity nodes create 2-hop traversal paths leaking 16–19 unauthorized items per query — per-hop authorization is the only defense that reduced leakage to zero across all tested corpora and attack variants with <1ms latency overhead. **HijackRAG** (documented throughout 2026 pentest reports with ~97% success rate at manipulating retrieval outcomes) joined PoisonedRAG as a standard reference attack — red teams should run HijackRAG-style payloads as part of release gating, not just academic benchmarks. For teams running Mosaic AI Vector Search, Unity Catalog's row-level filters propagate automatically into vector queries, meaning cross-tenant filter bypass requires compromising UC governance itself rather than individual application code — a significantly harder target than application-layer filters. OpenSearch and Elasticsearch document-level security (DLS) extend the same read-side guarantees to hybrid BM25+vector deployments via DSL-based role filters; note that DLS only restricts *read* operations — write-side controls still need separate role scoping, and the OpenSearch security docs explicitly warn that a role with write access can still modify documents hidden by DLS. |
| **8.1.4** | **Verify that** every ingested document is tagged at write time with source, writer identity (authenticated user or system principal), timestamp, batch ID, and embedding model version. | 2 | Data provenance attacks — without lineage metadata, poisoned vectors cannot be traced back to their source for quarantine or removal. Also prevents "ghost documents" inserted by compromised ingestion pipelines. Maps to MITRE ATLAS AML.T0071 (False RAG Entry Injection). Prompt Security's 2025 RAG_Poisoning_POC demonstrated that a single poisoned embedding can alter system behavior across multiple unrelated queries — provenance tagging is essential for rapid quarantine. RAGForensics (ACM Web Conference 2025) — the first traceback system for RAG poisoning — relies on provenance metadata to iteratively identify and remove poisoned texts, achieving >97.4% detection accuracy with <2.7% false positive rate. Without provenance tags, traceback is impossible. | Inspect ingestion pipeline code for metadata attachment at write time. Query stored vectors and confirm all required metadata fields are present. Attempt to update immutable metadata fields via API and verify rejection. Audit batch ingestion logs for completeness. Verify that signed ingestion jobs are accepted only from allow-listed sources, with review workflows for high-risk collections. | Immutability enforcement varies: Pinecone metadata can be overwritten via upsert; Weaviate properties can be updated unless schema-level restrictions are set. True immutability may require append-only storage patterns or write-once object storage for metadata alongside the vector index. Schneider's defense architecture (2025) recommends provenance tagging as part of Layer 2 (memory sanitization), including creation time, session context, and trust scores for each stored entry. RAGShield (April 2026) extends this with C2PA-inspired cryptographic document attestation — documents are signed at ingestion and unsigned/forged documents are rejected before vectorization. RAGShield also introduces a formal taint lattice with cross-source contradiction detection and NIST SP 800-53 compliance mapping across 15 control families. On 430 real IRS document attacks, RAGShield achieved 0.0% ASR while embedding-based defenses missed 79–90% of attacks. EU AI Act high-risk obligations (enforceable August 2026) require data governance and record-keeping for AI systems — provenance metadata on ingested documents directly supports these compliance requirements. The GAO documented 89 RAG-based citizen-facing services across 28 U.S. federal agencies as of FY2025, highlighting the scale at which provenance tracking matters for government deployments. Organizations are now extending CycloneDX ML-BOM to include integrity attestation records for retrieval sources alongside training corpora, documenting curation method and contributor vetting as first-class provenance artifacts. On April 7, 2026, NIST released a concept note for an AI RMF Profile on Trustworthy AI in Critical Infrastructure, which will guide operators toward specific risk management practices — provenance metadata on ingested documents directly supports the "Map" and "Manage" functions of NIST AI RMF 1.0. |
| **8.1.7** | **Verify that** document metadata tags applied at ingestion are immutable after initial write and cannot be modified by subsequent pipeline stages or user operations. | 2 | Metadata tampering post-ingestion — if an attacker or compromised pipeline stage can modify provenance tags (source, writer identity, timestamp, batch ID), poisoned documents can be re-attributed to trusted sources, undermining quarantine and forensic workflows. Verify that audit logs capture any metadata change attempts. | Most vector databases do not enforce metadata immutability natively. Pinecone metadata can be overwritten via upsert — the API treats metadata as mutable by design. Weaviate properties can be updated unless cross-references or schema-level restrictions prevent it. Qdrant allows metadata (payload) updates on existing points. Milvus supports field-level updates in v2.3+. Compensating controls: (1) use lakeFS to version embeddings and metadata as immutable commits, enabling rollback and audit trails at the storage layer; (2) RAGShield's C2PA-inspired cryptographic attestation (April 2026) signs documents at ingestion so any post-write metadata change invalidates the signature; (3) write metadata to append-only storage (e.g., WORM-configured S3, Azure immutable blob) alongside the vector index, treating the object store as the authoritative provenance record; (4) implement an application-layer middleware that hashes metadata at write time and verifies the hash before any retrieval, rejecting tampered entries. The MIND metadata-driven ingestion pattern (2025) uses Delta Lake for ACID-guaranteed metadata repositories with version control built in. As of April 2026, no major vector database offers a native "freeze metadata after write" toggle — this remains an application-architecture concern. The Hermes FHE system (March 2026) introduces data-oblivious homomorphic algorithms using slot masking and shifting that make in-place record modifications cryptographically controlled — while designed for general FHE database operations, the approach could be adapted to enforce metadata immutability by encrypting provenance fields with keys unavailable to the ingestion pipeline after initial write. For compliance-driven deployments, pair metadata immutability controls with continuous compliance monitoring platforms (e.g., Vanta, Drata) that validate metadata integrity as part of automated SOC 2 / HIPAA evidence collection rather than relying on point-in-time audits. |
| **8.1.5** | **Verify that** RAG pipeline retrieval events log the query issued, the documents or chunks retrieved, similarity scores, the knowledge source, and whether retrieved content passed prompt injection scanning before being incorporated into model context. | 2 | Undetected RAG poisoning — without retrieval logging, poisoned documents that influence model outputs cannot be identified forensically. Also enables detection of indirect prompt injection via retrieved content (OWASP LLM01:2025). Prompt Security's 2025 research showed 80% attack success rate across multiple queries from a single poisoned embedding, and retrieval monitoring (logging document access patterns) is cited as a primary defense — repeated hits from a single source indicate poisoning. RAGForensics (ACM Web Conference 2025) demonstrated that retrieval logs are the foundation for traceback — the system iteratively retrieves candidate texts and uses LLM-guided evaluation to identify poisoned documents, achieving >97.4% detection accuracy with <3.2% false negative rate. Without comprehensive retrieval logs, this traceback capability is lost. | Review logging pipeline configuration. Issue test queries and confirm logs contain all required fields (query text, retrieved doc IDs, similarity scores, source, injection scan result). Verify logs are shipped to a SIEM or centralized logging system. Test log completeness under high-throughput conditions. Validate that logs avoid storing raw prompts or full retrieved chunks in cleartext — use tamper-protected logging with restricted access controls per Cisco's vector DB security guidance. | Logging similarity scores and retrieved chunks at scale generates significant data volume. Teams need to balance completeness with storage costs. LlamaIndex and LangChain both support callback-based logging but require custom integration for injection scan results. Consider sampling strategies for Level 2 compliance. NVIDIA NeMo Guardrails (open-source) provides programmable input/output guardrails that can intercept retrieval events and enforce fact-checking, content policy screening, and prompt injection detection at the retrieval stage — integrating guardrail scan results into retrieval logs is a practical path to satisfying the "injection scan" requirement. For sensitive deployments, ephemeral RAG processing with zero-retention enterprise APIs ensures that intermediate representations (query embeddings, retrieved chunks, assembled prompts) are purged after request completion — verify these guarantees are backed by SOC 2 Type II attestation and contractual commitments, not just marketing claims. RevPRAG (EMNLP 2025) provides a complementary detection layer by analyzing LLM activation patterns during generation — distinct activation signatures emerge when the model processes poisoned vs. clean retrievals, achieving 98% true positive rate with only 1% false positives across multiple benchmark datasets. Schneider's defense architecture (2025) recommends behavioral monitoring with immutable logging and circuit breakers for rapid isolation when compromise is detected. ProvenanceGuard (IEEE TIFS 2026) provides a 120,000-document evaluation dataset across medical (PubMed), legal (CourtListener), and financial (SEC EDGAR) domains for benchmarking retrieval poisoning defenses — useful for validating that logging pipelines actually detect poisoned retrievals. Qdrant now integrates with Prometheus/Grafana for operational monitoring. The SIEM market is projected to reach $11.3B by 2026 at 14.5% CAGR, with AI-driven correlation reducing false positives by an estimated 40% — meaning retrieval anomaly alerts from vector DB logging can be prioritized more effectively when shipped to modern SIEM platforms. |
| **8.1.6** | **Verify that** retrieval anomaly detection identifies embedding density outliers, repeated dominance of specific documents in similarity results, and sudden shifts in retrieval bias distribution that may indicate vector database poisoning. | 3 | Sophisticated vector DB poisoning where an attacker injects many semantically similar vectors to dominate retrieval results for target queries, steering model outputs. Maps to MITRE ATLAS AML.T0070 (RAG Poisoning). PoisonedRAG (Zou et al., USENIX Security 2025) demonstrated that just 5 poisoned documents achieve 90% attack success — the attacker creates a "gravity well" in embedding space. The RSB benchmark (Zhang et al., 2025) evaluated 13 poisoning attack methods and found that all 7 defense mechanisms tested failed to provide robust protection, underscoring the difficulty of detection. Future variants could evolve into "vector worms" that instruct models to re-embed and reintroduce poisoned data elsewhere, creating cascading contamination (Prompt Security, 2025). | Review anomaly detection implementation. Verify it monitors: (1) document retrieval frequency distributions, (2) embedding space density metrics per namespace, (3) temporal shifts in which documents are retrieved for stable query sets. Test with synthetic poisoning scenarios — including centroid-targeted injection (Black-Hole Attack pattern) — and confirm alerts fire. The Black-Hole Attack (Li et al., April 2026) showed that vectors injected near the geometric centroid of embedding space exploit "centrality-driven hubness" to appear in up to 99.85% of top-10 results at just 1% poisoning rates; probe-vector detection (querying with random vectors and flagging anomalously frequent neighbors) removes suspicious vectors with minimal legitimate data loss. Behavioral drift detection using runtime analytics to identify sudden tone shifts or stylistic anomalies is a recommended complementary approach. Practical embedding anomaly thresholds validated in testing: 0.85 cosine similarity flags individual document overrides, 0.90 flags batch clustering anomalies — embedding anomaly detection alone reduced knowledge poisoning success from 95% to 20%. Consider hybrid BM25 + vector retrieval as an architectural defense — research shows it reduces gradient-guided poisoning success from 38% to 0%, though joint sparse+dense optimization can partially circumvent it (20–44% success). Evaluate the Normalized Passage Attention Score (NPAS) and Attention-Variance Filter (AV Filter) as lightweight runtime detectors — research shows up to 20% higher accuracy than baseline defenses at flagging anomalous passages. RevPRAG (EMNLP 2025) offers a generation-time complement: analyzing LLM activation patterns during response generation achieves 98% TPR at 1% FPR for detecting poisoned retrievals via a Siamese network trained on final-token activations across all LLM layers. | This is an emerging area with limited production tooling but several notable frameworks have appeared. **RAGuard** (2025) uses a three-component strategy: (1) retrieval expansion (top-N where N > k) to dilute poisoned content, (2) chunk-wise perplexity filtering using non-parametric percentile thresholds to detect unnatural linguistic patterns, and (3) text similarity filtering to flag documents with unusually high query similarity. RAGuard achieves 92–99% detection accuracy across four attack types (prompt injection, general trigger, jamming, PoisonedRAG) with FPR below 6%, and remains robust against GPT-4-paraphrased adaptive attacks (DACC >93%). **RAGForensics** (ACM Web Conference 2025) provides post-incident traceback — iteratively retrieving candidate texts and using LLM-guided evaluation to identify poisoned documents with >97.4% DACC and <2.7% FPR, even against adaptive evasion attempts. Custom implementations typically use statistical process control on retrieval frequency histograms or embedding space density estimation (e.g., k-distance plots). No major vector DB offers anomaly detection natively. CorruptRAG (Jan 2026) showed that single-document attacks can dominate retrieval for targeted queries, making per-query retrieval frequency monitoring essential — if one document consistently appears across unrelated queries, that is a strong poisoning signal. SDAG (Feb 2026) provides an inference-time defense by preventing cross-document interaction via sparse attention. Schneider's four-layer defense (2025) recommends trust-weighted ranking adjusted by provenance metadata and temporal decay at the retrieval layer. The RSB benchmark (13 attacks across 5 QA datasets) provides a standardized evaluation framework for testing defense effectiveness. MM-MEPA (February 2026) showed that metadata-only poisoning in multimodal RAG achieves up to 91% ASR while leaving visual content unaltered — existing defenses including query paraphrasing and image-metadata consistency checks were largely ineffective, highlighting that anomaly detection must monitor metadata distributions alongside embedding space density. ProvenanceGuard (IEEE TIFS 2026) with 120,000 documents across medical, legal, and financial domains and 12,000 adversarial passages covering four attack strategies (targeted factual poisoning, semantic camouflage, ranking manipulation, multi-passage coordinated poisoning) provides the most comprehensive evaluation dataset available for benchmarking anomaly detection pipelines. The Black-Hole Attack (Li et al., April 2026) adds a new attack class: centroid-targeted injection exploiting hubness, where defense options include hubness mitigation (CL2/Z-score normalization, TCPR, noHub — though these sacrifice retrieval quality) and probe-vector detection (querying with synthetic probe vectors and flagging anomalously frequent neighbors). The Key Collision Attack (January 2026, arxiv 2601.23088) targets semantic caching layers by crafting inputs that collide with cached query embeddings, causing stale or poisoned responses to be served without hitting the RAG pipeline at all — teams deploying semantic caching should validate cache key entropy and set TTLs aligned with content freshness requirements. |
