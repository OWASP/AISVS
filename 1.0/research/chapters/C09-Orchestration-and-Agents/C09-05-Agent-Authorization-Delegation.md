# C9.5: Agent Authorization, Delegation, and Continuous Enforcement

[Back to C09 Index](C09-Orchestration-and-Agents.md)

## Purpose

Agents act on behalf of users and interact with systems that have their own access control policies. This section ensures that agent actions are authorized against fine-grained policies at execution time, that delegation context (who the agent is acting for, with what scope) is propagated and enforced throughout the call chain, and that authorization decisions are never made by the AI model itself. The core principle: access control must be enforced by deterministic policy engines, not probabilistic language models.

| **9.5.1** | **Verify that** agent actions are authorized against fine-grained policies enforced by the runtime that restrict which tools an agent may invoke, and which parameter values it may supply. | 2 | **Unauthorized tool invocation and parameter abuse (MITRE ATLAS AML.T0053; OWASP Agentic ASI02 Tool Misuse).** Without fine-grained authorization, an agent may invoke any tool with any parameters, even if the initiating user is not authorized for that action. This requirement also absorbs the **single-constraint / confused-deputy** failure mode: when the policy decision point evaluates only the agent's own permissions (ignoring the delegating user's scope) or only the user's scope (ignoring agent-level restrictions), effective access becomes the *union* rather than the *intersection* of both. | **9.5.6** | **Verify that** long-running agent sessions re-evaluate current backend authorization policy on every privileged action. | 3 | **Stale authorization, privilege persistence, and insufficient session expiration (CWE-613; OWASP Agentic ASI03 Identity & Privilege Abuse).** Long-running agents can keep working after the user, agent identity, service principal, tenant policy, registry membership, or delegated scope has changed. OpenID CAEP and Shared Signals provide the event layer for asynchronous revocation, while AuthZEN and PDP APIs provide synchronous request-time authorization. The hard gap is cache safety: high-throughput agent systems often cache allow decisions for latency, but cached decisions must be bound to action, resource, subject, tenant, delegation scope, risk state, and policy version, with deny-on-unknown behavior when revocation streams lag. Offline agents, disconnected edge agents, browser agents with local state, and durable queue workers remain difficult because they may be unable to reach the PDP at action time; these systems should fail closed for privileged actions rather than spend stale authority. |

## Implementation Guidance

### The Authorization Crisis for Agentic AI (2025--2026)

Traditional IAM frameworks were designed for human users and static service accounts. Research and industry analysis in 2025--2026 identifies seven critical failures when applying these frameworks to AI agents (ISACA, 2025):

1. **Coarse-grained permissions:** Agents need fine-grained, task-specific permissions that dynamically change based on context, not static role-based access with long-lived tokens.
2. **Single-entity model limitations:** Current systems cannot track complex nested delegations where agents spawn sub-agents or represent multiple principals simultaneously.
3. **Context blindness:** Access decisions lack awareness of runtime context, agent intent, or risk level -- permissions remain static throughout sessions.
4. **Scalability pressures:** Hundreds or thousands of transient agents generate overwhelming token management demands (issuance, validation, revocation).
5. **Inadequate inter-agent authentication:** OAuth and SAML rely on hierarchical trust models unsuitable for decentralized agent networks that require peer-to-peer trust.
6. **Secret sprawl:** Each agent's multiple API integrations create a multiplicative factor in secrets requiring rotation and storage.
7. **Revocation complexity:** Revoking access in one place does not automatically cut off access elsewhere across distributed ephemeral sessions.

A 2026 survey (Grantex) found that 93% of popular AI agent projects use unscoped API keys, and fewer than half of companies deploying agents had comprehensive governance in place. Subsequent enterprise surveys confirm the gap is widening, not closing: Gravitee's *State of AI Agent Security 2026* (n=919 executives and practitioners) reports 88% of enterprises observed at least one AI agent security incident in the prior twelve months -- 92.7% in healthcare specifically -- while only 21% have runtime visibility into agent actions despite 82% of executives believing their policies prevent unauthorized agent behavior. Saviynt's *2026 CISO AI Risk Report* (n=235 CISOs at 5,000+ employee enterprises) layers on the identity dimension: 47% have already observed agents exhibit unintended or unauthorized behavior, only 5% feel confident they could detect or contain a compromised agent, and 92% lack full visibility into what AI identities exist in their environment. Together with the Cisco State of AI Security 2026 (83% deploying, 29% prepared), these three datasets describe the same authorization-enforcement gap from three angles: incidents are happening, identity coverage is incomplete, and runtime enforcement is the missing layer.

### Emerging Standards and Frameworks

- **NIST AI Agent Standards Initiative (February 2026):** NIST's Center for AI Standards and Innovation (CAISI) launched the AI Agent Standards Initiative alongside a concept paper on "Accelerating the Adoption of Software and Artificial Intelligence Agent Identity and Authorization." The initiative proposes NCCoE lab demonstrations exploring agent identification (distinguishing agents from humans), authorization (OAuth 2.0/2.1 extensions and policy-based access control), and access delegation (linking user identities to agents for accountability). The concept paper recommends ABAC over RBAC for agent permissions and explicitly calls for transitive trust limits preventing unbounded permission cascading. Standards under consideration include MCP, OAuth 2.0/2.1, OpenID Connect, SPIFFE/SPIRE, SCIM, and Next Generation Access Control. The public comment period closed on April 2, 2026; community input will inform a draft project description, and CAISI began sector-specific listening sessions in April 2026 focused on barriers to confident AI agent adoption.
- **IETF AIMS Framework (draft-klrc-aiagent-auth-03, July 6, 2026):** The Agent Identity Management System composes WIMSE, SPIFFE, and OAuth 2.0 into a layered architecture. It defines identifiers, short-lived credentials, posture-aware provisioning, authentication, authorization, monitoring and remediation, and the deployment's reviewable authorization policy. Three authorization scenarios are defined: user-delegated (authorization code grant), agent self-authorized (client credentials), and agent-as-resource-server. The July `-03` revision clarifies the relationship between WIMSE and SPIFFE credentials and states that LLMs should not have access to agent credentials, directly supporting 9.5.4. It also adds CIBA guidance: an in-band "approve" response is not authorization by itself and should be translated into a verifiable authorization-server event before protected-resource access. For 9.5.5, the draft now calls for durable, tamper-evident records that let operators reconstruct the complete execution chain, including delegated authority, intermediate calls, resulting actions, and authorization changes across service boundaries. The draft remains an individual Internet-Draft rather than an adopted IETF standard, and leaves the concrete policy document format deployment-specific.
- **IETF Security Requirements for AI Agents (draft-ni-a2a-ai-agent-security-requirements-01, February 28, 2026):** A Huawei-authored Informational draft (Y. Ni, C. P. Liu, Q. Gao, Z. Li) cataloguing security requirements for the full agent lifecycle: provisioning, registration, discovery, cross-domain interconnection, and access control. Its most useful contribution for 9.5.2 and 9.5.5 is the explicit requirement that the *agent-to-agent context and intent of the original requester* must be preserved and propagated through every hop of a delegation chain to prevent authorization drift and context poisoning -- an Internet-Draft restatement of the semantic-mosaic and intent-preservation concerns raised by CoSAI, the SentinelAgent paper, and the IETF Delegation Receipt Protocol. The draft also acknowledges the heterogeneous-system reality that practical deployments confront: within a single trust domain, downstream API endpoints, microservices, tools, and databases each impose their own authentication method (identity tokens, API keys, pre-shared secrets, username/password, X.509), and the agent runtime must bridge between them without collapsing the delegation context. The draft expires September 1, 2026.
- **Agent Relationship-Based Identity and Authorization (ARIA):** Treats delegation relationships as explicit, cryptographically verifiable entities in a graph. Every delegation is a distinct relationship that enables surgical permission revocation and full auditability. ARIA integrates OAuth 2.0 Rich Authorization Requests, OAuth 2.0 token exchange (RFC 8693), and OpenID AuthZEN for context-aware policy evaluation.
- **OpenID AuthZEN 1.0 (Approved January 2026):** Now a Final Specification, AuthZEN standardizes a transport-agnostic API where any Policy Enforcement Point queries any Policy Decision Point regardless of vendor. The information model is a four-element tuple (Subject, Action, Resource, Context) that maps cleanly to agent tool invocations. This enables fine-grained, context-aware policies without abandoning installed-base OAuth infrastructure -- applying constraints such as geo-fences, budget thresholds, and temporal windows.
- **OpenID AuthZEN WG -- AARP and COAZ Agent-Era Drafts (June 15, 2026):** The OpenID AuthZEN Working Group adopted two new Working-Group drafts that extend the 1.0 baseline directly into agentic authorization, developed with Axiomatics, C1, Cerbos, Indykite, Keycard, Okta, and SailPoint. **AARP (Access Request and Approval Profile)** standardizes the "not yet -- here is what is required first" decision: a PDP can return that a prerequisite (approval, consent, delegated authority, attestation, or risk assessment) must be satisfied before an authorization decision can be reached, with interoperable patterns for requesting, tracking, and fulfilling that prerequisite asynchronously while policy remains the ultimate decision-maker. **COAZ (AuthZEN Profile for MCP Tool Authorization)** standardizes how MCP tool-invocation metadata maps into AuthZEN's Subject-Action-Resource-Context model, so API gateways, AI gateways, service meshes, and downstream systems can authorize each MCP tool call at runtime against a vendor-neutral PDP. For 9.5.1 and 9.5.3, COAZ is the working-group expression of the per-tool-call PDP-at-the-MCP-boundary pattern this section advocates -- it lets the gateway-mediated authorization stack speak a single PEP-to-PDP contract regardless of which policy engine (Cerbos, Cedar, OPA) sits behind it. For 9.5.5 and 9.5.6, AARP is the synchronous counterpart to the asynchronous CAEP/Shared Signals revocation layer: it gives the dual-constraint intersection rule and human-in-the-loop step-up (CIBA, OAuth Step-Up RFC 9470) a uniform way to be expressed as a *prerequisite* in the authorization response itself, so an agent that lacks a fresh delegated approval is told precisely what it must obtain before the next privileged action is permitted.
- **Agent Identity Protocol (AIP, March 2026):** A proposed protocol for verifiable delegation across MCP and A2A boundaries. AIP uses Decentralized Identifiers (DIDs) and delegatable credentials to create cryptographic chains of authorization -- each delegation step is signed, producing non-repudiable evidence of who authorized what, with explicit controls on transitivity (whether a delegate can further delegate received permissions). The protocol addresses the delegation chain tracking gap identified at RSAC 2026 by making every handoff inspectable without requiring a central authority.
- **Zero Trust Identity with DIDs:** Combines Decentralized Identifiers, Verifiable Credentials, Agent Naming Services, and zero-knowledge proofs for privacy-preserving compliance verification in agent identity.
- **DIF Know Your Agent Operating System (KYA-OS, formerly MCP-I) -- Renamed April 22, 2026:** The Decentralized Identity Foundation renamed the agentic-identity framework Vouched donated in March 2026 (originally `MCP-I`) to *KYA-OS* to signal that its scope now extends across the full range of agentic protocols rather than only MCP. KYA-OS adds a DID/Verifiable-Credential identity and delegation layer on top of existing agent protocols, with three conformance levels: Level 1 (foundational, reusing existing OIDC/JWT identifiers for immediate adoption), Level 2 (full DID verification, credential-based delegation, and revocation), and Level 3 (enterprise-grade lifecycle management, immutable auditing, and full bilateral KYA-OS awareness across both endpoints). Delegation is represented as tamper-evident credentials with explicit scope and is verifiable cross-organization without prior coordination between the verifying parties. Version 1.0 entered the 14-day review period in DIF's Trusted Agentic AI Working Group ahead of a working-group vote; governance is stewarded by a dedicated KYA-OS Task Force inside DIF. For 9.5.2 and 9.5.5 this is the DID-anchored counterpart to the OIDC-A and AIMS/SPIFFE families already cataloged here -- it lets services that have never met an agent before still cryptographically verify *which* agent is acting, *who* authorized it, and *exactly* what it is permitted to do, which is the missing primitive when the verifying side is outside the agent's home organization (B2B agentic flows, federated marketplaces, consumer agents acting against arbitrary merchants).
- **OpenID Foundation Artificial Intelligence Identity Management (AIIM) Community Group -- NIST Response (March 2026):** The OIDF AIIM Community Group (formed in 2025 to bridge the standards gap between the AI and identity communities) filed a Threat Modeling Subgroup response to NIST's RFI on securing AI agent systems in March 2026. The submission's load-bearing argument for C9.5 is that the dominant agent-authorization failure mode is "trust failure rather than technical flaw" -- the "who authorized this agent to act, on whose behalf, with what attenuation?" question is unanswerable in most deployments because the underlying accountability chain is missing, not because the cryptography is weak. The response advocates a "trust fabric" composed of existing emerging standards (transaction tokens, workload identity federation, AI tool authentication extensions) rather than prescriptive new mandates, and explicitly warns NIST that overly burdensome requirements will push deployers to cut corners. For 9.5.2 and 9.5.5 this is the SDO-policy counterpart to the academic gap analysis in Otsuka et al. (arXiv 2604.23280) -- the same five-gap argument, restated for regulators by the standards body that owns OAuth, OIDC, and the AuthZEN authorization API.
- **W3C Agent Identity Registry Protocol Community Group (Proposed April 22, 2026; Call for Participation April 24, 2026):** A new W3C Community Group chartered to develop open specifications for verifiable AI agent identity infrastructure that enables cross-organizational trust without pre-existing agreements. The initial scope spans a DID method specification for agent identity resolution, an agent credential format based on W3C Verifiable Credentials, trust-negotiation protocols, credential lifecycle management, post-quantum cryptography requirements, and integration contracts with OAuth/OIDC and SPIFFE. The CG positions itself as a sibling to the OIDF AIIM CG, the IETF AIMS framework, and the DIF KYA-OS Task Force -- the W3C-housed registry layer for the same B2B-trust gap those efforts address from the OAuth/OIDC, IETF, and DID/VC sides. For 9.5.5 this is the standards-body activity to watch when evaluating which agent-registry primitive will become reusable across organizational boundaries; the CG itself does not produce W3C Recommendations, but its specifications feed candidate inputs to the W3C standards track if the working-group conversion threshold is reached.
- **IETF OAuth WG -- Delegation Chain Splicing in RFC 8693 (Chhaya, February 26, 2026):** Chiradeep Chhaya raised a security consideration on the `oauth@ietf.org` list (thread starting `[OAUTH-WG] Security Consideration: Delegation Chain Splicing in RFC 8693 Token Exchange`) describing an attack where a compromised intermediary presents mismatched `subject_token` and `actor_token` inputs to a Security Token Service from *different* delegation contexts. The STS validates each token independently per RFC 8693 §2.1-2.2, finds both valid, and mints a properly-signed token asserting a delegation chain that never occurred. Root cause: RFC 8693 has no mechanism to verify the two tokens belong to the same delegation flow, authorization session, or trust context, and the existing `may_act` claim is optional and validates only the actor's identity, not the context in which the actor credential was acquired. Proposed mitigations from the thread include cross-validation (the `actor_token` subject must match an authorized next-actor declared in the `subject_token`), strengthening `may_act` from optional to normative, nested `act` claims carrying a verifiable chain similar to OIDC-A 1.0's `delegation_chain` (audience of step N must match subject of step N+1), and per-step signed delegation receipts. The follow-up exchange with Ayesha Bose (msg25725) agreed on the vulnerability and converged on a mitigation profile -- `aud`/`sub` binding at each exchange, single-valued audience claims, policy validation before token issuance, short lifetimes, back-channel revocation, and refresh-flow re-validation -- but no formal WG action (errata, BCP addition, or new draft) has been taken yet. This is directly load-bearing for 9.5.2 because the broad OAuth Token Exchange / on-behalf-of family this section recommends (Scalekit, Oso, the AIMS framework, ID-JAG, the Identity Chaining draft, MCP's prohibition on token passthrough) all share the underlying RFC 8693 primitive. Deployments that have already adopted IETF Transaction Tokens for Agents (`actchain`) or the IETF Delegation Receipt Protocol's append-only log anchors largely satisfy the proposed mitigations because those drafts independently arrived at chain-binding and per-step attestation requirements; deployments using plain RFC 8693 with a single-server STS should treat this as a known gap and add cross-token binding at the STS until the issue is normatively resolved.
- **IETF Transaction Tokens for Agents (draft-araut-oauth-transaction-tokens-for-agents-01, May 12, 2026):** An Internet-Draft (Ashay Raut, individual submission) extending OAuth Transaction Tokens specifically for AI agent workloads. This version replaces the earlier `draft-oauth-transaction-tokens-for-agents` series under the author-specific naming scheme. Introduces three claims: `act` (identifies the AI agent performing the action), `sub` (the principal on whose behalf work occurs), and an optional `agentic_ctx` claim now decomposed into provenance (behavioral DNA), posture (environmental integrity), and identity (workload origin) subfields. The critical security property is the `actchain` claim -- a cryptographic trace of the delegation path as an ordered array of previous agents, solving the delegation chain tracking gap identified at RSAC 2026. The May 12 revision adds two named defenses worth calling out: **Monotonic Attenuation of Trust** (each hop in a multi-agent chain may only *narrow* the rights it received, never broaden them, enforced at the TTS at token-replacement time), and **Identity Laundering** prevention (transaction path tracking that prevents an unverified external agent from bypassing guardrails by proxying through an internal, more-trusted agent and inheriting that agent's authority). Loop prevention via path-tracking is now a normative requirement. Supports Rich Authorization Request (RAR) propagation for fine-grained, user-consented permissions flowing through multi-hop service chains.
- **IETF AAuth Protocol (draft-hardt-oauth-aauth-protocol-01, May 6, 2026):** A new Dick Hardt (Hellō) draft defining agent-to-resource authorization for HTTP clients with independent cryptographic identity, distinct from AIMS, AAA, and OIDC-A. Agent identifiers take the form `aauth:local@domain` and are bound to signing keys through Agent Tokens; every request carries HTTP Message Signatures so a stolen token is useless without the private key -- this is a protocol-level realization of the proof-of-possession property CoSAI's post-RSAC follow-up called for via DPoP (RFC 9449). The protocol defines four progressive resource access modes (identity-based; resource-managed two-party; PS-asserted three-party where a Person Server asserts identity and user consent; federated four-party with a separate Access Server) so deployments can start at the simplest sufficient pattern. Crucially for 9.5.5 and the semantic-privilege-escalation problem, governance is an orthogonal layer: **Missions** are scoped authorization contexts where the agent declares its intent in natural language, the user (via the Person Server) approves it, and the approved mission is hashed (SHA-256) to make the intent immutable. A **Mission Log** records every interaction within the mission, and Permission/Audit/Interaction endpoints let the governance plane intervene mid-task ("agents need authorization decisions mid-task, long after the user set them in motion"). The mission-hash construction is operationally adjacent to the IETF Delegation Receipt Protocol's instruction-hash binding -- both treat the operator's stated intent as a non-waivable, cryptographically-pinned input to authorization rather than a free-text annotation.
- **IETF Identity Assertion JWT Authorization Grant (ID-JAG, draft-ietf-oauth-identity-assertion-authz-grant-03, April 22, 2026):** A Working-Group OAuth draft (adopted as a `draft-ietf-`, no longer `draft-parecki-`) defining a profile of the JWT Authorization Grant (RFC 7523) where a client exchanges an identity assertion from a trusted Identity Provider for an access token at a Resource Authorization Server *without* prompting the user again at the AS. The same enterprise IdP that already brokers SSO is extended to broker API access, so cross-domain delegation does not require redirect flows or shared client registrations at every downstream service. Section A.4 explicitly carves out *AI Agent using External Tools* as a precondition scenario, walking through how an LLM agent obtains an Identity Assertion at the Enterprise IdP and presents it to a third-party Resource AS to mint a delegated access token for the external tool. The latest revision (-03) expires October 24, 2026. For 9.5.2 this is the WG-track answer to "how do we propagate the user's delegation context across a partner domain without exposing the original credential or forcing the user back through a consent screen for every tool the agent touches," and it composes naturally with the cross-domain Identity Chaining draft below: ID-JAG is the IdP-to-AS handoff, identity chaining is the AS-to-AS handoff.
- **IETF OAuth Identity and Authorization Chaining Across Domains (draft-ietf-oauth-identity-chaining-12, May 11, 2026):** Submitted to the IESG for publication as a Proposed Standard, this Working-Group draft is the standards-track companion to the multi-hop delegation primitives elsewhere in this section. It composes RFC 8693 (Token Exchange) with RFC 7523 (JWT Profile for OAuth 2.0) so that a client in Domain A can exchange a subject token for a JWT authorization grant targeting Domain B's authorization server, then redeem that grant for a Domain-B access token. Two deployment patterns are specified: *Resource Server as Client* (a Domain A resource server discovers Domain B's authorization server and performs token exchange independently, no Domain A AS coordination required) and *Authorization Server as Client* (Domain A's AS performs the exchange on behalf of its clients, so Domain A clients never see Domain B). Claims transcription (Section 2.5) carries user identifiers and authorization context across the boundary while supporting data minimization and scope downscoping -- the cross-domain version of the same constraint AAA's Monotonic Attenuation enforces within a chain. For C9.5.2 deployments where an agent traverses partner organizations (B2B AI agents, supply-chain orchestrators, federated multi-tenant SaaS), this is the IETF-standards-track answer to the "how do we propagate delegation context without sharing the user's credential" question, complementing the AIMS framework's identity layer.
- **IETF OAuth Scope Aggregation (draft-jia-oauth-scope-aggregation-00, February 2026):** Proposes a scope-aggregated authorization pattern where an AI agent pre-computes the scopes required across a multi-step workflow and initiates a single authorization procedure for the aggregated scope. This reduces repeated user consent prompts and authorization round-trips in complex agentic workflows while maintaining explicit user approval of the full scope set.
- **IETF OAuth Multi-Agent Collaboration (draft-song-oauth-ai-agent-collaborate-authz-01):** Extends OAuth 2.0 with additional fields in token and message flows to support task-oriented multi-agent collaboration. Sub-agents can execute tasks as a group under a unified authorization framework while maintaining compatibility with existing OAuth 2.0 workflows.
- **IETF A2A Profile for OAuth Transaction Tokens (draft-liu-oauth-a2a-profile-00, October 20, 2025):** Companion profile (Liu and Ni, Huawei) that specializes the OAuth Transaction Tokens framework for direct Agent-to-Agent communication rather than the agent-to-API case the parent draft assumes. Defines how an A2A call chain is embedded inside a transaction token via three coordinated fields -- a `purpose` identifier, an immutable transaction context (`tctx`) frozen at the originating user's intent, and a mutable request context (`rctx`) where each downstream agent appends its observations and decisions -- so that the next agent in the chain receives both the unchangeable authorization envelope and a verifiable trail of what each peer has already committed to. The split mirrors the `actchain` construction in `draft-araut-oauth-transaction-tokens-for-agents` but separates the immutable and observable halves explicitly, which is useful when an A2A handoff needs to forward decision context for collaboration without letting a downstream agent rewrite the authorization scope. For 9.5.5 this is the most narrowly targeted Internet-Draft proposal of inter-agent delegation context propagation published so far; for 9.5.6 the immutable `tctx` claim is what a revoking PDP can match against to invalidate every downstream call originating from a specific user intent without having to walk the chain.
- **WIMSE Applicability for AI Agents (draft-ni-wimse-ai-agent-identity-02, February 2026):** Examines how the IETF Workload Identity in Multi-System Environments (WIMSE) architecture applies to autonomous AI agents operating across multiple systems. The key contribution is a **dual-identity credential framework** that cryptographically binds an agent's identity to its owner's identity, with three issuance models: agent-mediated (owner pre-signs offline using hardware-bound signing modules), owner-mediated (owner acts as supervisory gatekeeper inspecting requests for policy compliance), and server-mediated (server independently verifies with the owner via challenge-response). This directly addresses the delegation context problem in 9.5.2 by providing cryptographic proof of the agent-owner relationship at credential issuance time rather than relying on runtime token claims alone. The draft references BRSKI for initial trust establishment and distinguishes itself from CHEQ (which focuses on human confirmation of actions) by emphasizing independent agent identities.
- **IETF SCIM Agent Extension (draft-abbey-scim-agent-extension-00, October 2025):** Extends the System for Cross-Domain Identity Management (SCIM 2.0) protocol with two new resource types -- `Agents` and `AgenticApplications` -- giving identity systems a standard way to provision, manage, and decommission non-human AI agent identities. Each agent must have one or more human owners (establishing clear accountability chains), and relationships between agents and hosting applications are defined as explicit SCIM references, making authorization paths transparent and revocation immediate when an app or agent is retired. The `lastAccessed` timestamp enables security teams to flag dormant agents with lingering credentials. For 9.5.5 peer authorization and 9.5.6 ongoing re-authorization, SCIM provisioning provides the registry and lifecycle infrastructure: an agent's entitlements, approved delegation paths, and credential lifecycle can be managed through the same identity provisioning pipeline as human users. The draft was presented at IETF 124 and is actively discussed alongside complementary work in draft-wahl-scim-agent-schema-01.
- **IETF Delegation Receipt Protocol (draft-nelson-agent-delegation-receipts-05, May 2026):** Introduces a cryptographic authorization primitive specifically designed for agent delegation. Before an agent executes any action, the authorizing user signs an Authorization Object containing an explicit scope allowlist (operations classified as reads/writes/deletes/executes, default deny), boundaries that survive any subsequent operator instruction, a SHA-256 hash of the operator's stated instructions at delegation time, and a model state commitment. The signed receipt is published to an append-only log before the agent runtime receives control, and a log anchor (inclusion proof with authoritative timestamp) is returned. Any change to operator instructions after issuance is detectable by recomputing the instruction hash. This directly addresses the semantic privilege escalation problem: scope is enforced as a positive allowlist, boundaries are non-waivable even by the operator, and the log anchor provides non-repudiable provenance for post-incident forensics. The DRP complements (rather than replaces) the IETF Transaction Tokens for Agents draft -- transaction tokens carry the delegation chain across hops, while delegation receipts establish the binding origin of authority with cryptographic instruction hashes.
- **IETF OAuth Attenuating Agent Tokens (draft-niyikiza-oauth-attenuating-agent-tokens-00, 2026):** Specifies an OAuth 2.0 extension where tokens issued for AI agent workloads can be progressively *attenuated* at each delegation hop -- a delegate can never request broader rights than the delegator held. The mechanism uses Rich Authorization Request extensions with a typed constraint vocabulary for tool-level argument restrictions, so an agent that holds a token authorizing "search the knowledge base for documents tagged `public`" cannot mint a downstream token that authorizes "search for documents tagged `confidential`." This addresses CSA's "scope narrowing at each delegation hop" recommendation as a protocol-level primitive rather than a deployment convention.
- **IETF Agent Identity Protocol -- Two-Layer Identity and Enforcement (draft-aip-agent-identity-protocol-00, March 16, 2026):** A third AIP variant -- distinct from `draft-singla-agent-identity-protocol` and `draft-prakash-aip` -- authored by James Cao (Montcao) and Carlos Eduardo Arango Gutierrez (NVIDIA). The draft splits agent identity from policy enforcement into two independently-deployable layers: Layer 1 (Identity) issues each agent a unique identifier and registered key pair, and the agent cryptographically signs every outbound action; Layer 2 (Enforcement) is a proxy that intercepts tool calls, verifies the signature, evaluates declarative policies, and produces an `allow` / `deny` / `hold` decision before the tool is reached. The draft explicitly targets MCP and emphasizes that AIP *does not* replace OAuth 2.0 or mTLS at the service layer -- it sits as an additional identity-bearing layer on top, positioned for deployments that need cryptographic per-action attribution without re-architecting upstream authentication. The NVIDIA co-author affiliation makes it operationally adjacent to the NemoClaw gateway pattern already cited above. The draft is an Individual Submission with no IETF standing, but the *allow/deny/hold* trichotomy is a useful complement to the binary allow/deny model in most agent policy engines because the `hold` outcome maps directly to the human-in-the-loop primitives in the IBM/Auth0/Yubico CIBA framework and the IETF OAuth 2.0 Agents Native Authorization elicitation extension.
- **IETF PEDIGREE -- Verifiable Delegation Identity for Agentic AI Systems (draft-rampalli-pedigree-00, April 25, 2026):** Karthik Rampalli (Glyphzero Labs) Individual Submission extending SPIFFE's workload-identity model into a delegation primitive that the author argues is missing from AAuth, AIP, SPIFFE, and ZeroID. PEDIGREE adds four mechanisms operationally adjacent to existing 9.5 controls but composed differently: (1) **cryptographic per-hop delegation** where each agent in a chain re-verifies its parent's credentials at mint and at verify -- "strict parent-token re-verification that catches parent-swap attacks" that an append-only signed token chain cannot detect; (2) **monotonic scope attenuation enforced at both mint and verify**, so a child credential cannot escalate even if a verifier is misconfigured; (3) **dual-layer authority enforcement** requiring both the delegated mandate from the parent *and* an operator-defined ceiling policy -- the protocol-level realization of the 9.5.5 dual-constraint intersection rule the Microsoft Authorization Fabric and Cedar "delegation as data" pattern implement at the deployment-convention layer; and (4) **Cedar-based narrowing proofs** that emit static-analysis artifacts proving a child's mandate logically narrows the parent's, rather than relying on imperative runtime checks. Tokens are JWT-based Suradar Identity Tokens (SITs) with optional completion blocks cryptographically binding execution outcomes to the delegation chain, enabling downstream audit without external logging. For 9.5.5 and 9.5.6 the draft is the cleanest Internet-Draft proposal to date of "ceiling-plus-mandate enforcement" as a primitive rather than a deployment convention.
- **IETF Agent Identity Protocol -- Invocation-Bound Capability Tokens (draft-prakash-aip-00, March 27, 2026):** A second AIP submission, distinct from the earlier `draft-singla-agent-identity-protocol` family, introducing *Invocation-Bound Capability Tokens* (IBCTs) that fuse identity, authorization, scope constraints, and provenance into a single cryptographic artifact bound to a specific invocation. Two modes are defined: a compact JWT mode with Ed25519 signatures for single-hop interactions, and a chained mode using Biscuit tokens with append-only blocks and Datalog policy evaluation for multi-hop delegation chains. Protocol bindings are specified for MCP, A2A, and generic HTTP APIs. The draft is an Individual Submission with no IETF standing, but the IBCT design directly targets 9.5.2 (integrity-protected delegation context) and 9.5.5 (independent constraint evaluation) by making capabilities invocation-bound rather than session-bound.
- **IETF Agent Identity Registry (draft-drake-agent-identity-registry-00, April 11, 2026):** Proposes a federated registry architecture for issuing, managing, and verifying persistent identities for autonomous entities. Identities are expressed as URNs in the `aid` (Agent Identity) namespace and, where hardware is available, anchored to a physical security component (TPM, PIV smart card, secure enclave, or virtual TPM) whose manufacturer-certified key cannot be extracted, cloned, or transferred. The draft argues that without hardware-anchored, persistent agent identity, Sybil attacks at scale become trivial and inter-agent trust collapses. For 9.5.5 this is the registry-layer companion to A2A's Agent Cards: a federated source of truth for *which* agent is allowed to participate in delegation in the first place, before any peer authorization check runs.
- **IETF Architecture for Human-Anchored Agent Identity, Delegation, and Provenance (draft-beyer-agent-identity-architecture-00, 2026):** Proposes a model that roots agent identity in a human identity and defines explicit delegation semantics with provenance structures that survive across agent boundaries. The architectural contribution is treating the human-to-agent binding as a first-class identity construct rather than an attribute carried in token claims, complementing the WIMSE dual-identity credential framework already cited.
- **OpenID Connect for Agents (OIDC-A) -- draft-sharif-openid-agent-identity-00 (March 2026):** An Internet-Draft defining a profile of OpenID Connect Core 1.0 that lets Identity Providers issue identity tokens specifically for autonomous software agents. Together with the companion academic specification "OpenID Connect for Agents (OIDC-A) 1.0" (arXiv 2509.25974), it adds claims for agent attestation evidence, delegation chain representation, capability discovery, and constraint enforcement -- effectively extending OIDC's familiar `sub`/`act` model with agent-aware attributes (attestation type, capability set, delegation depth) so existing OIDC infrastructure can serve as a PEP for agent identity without a parallel stack. For 9.5.2 and 9.5.5 this is the OIDC-shaped counterpart to the AIMS/WIMSE work already cited, easing adoption for organizations already running OIDC IdPs (Okta, Entra, Auth0, Ping) rather than asking them to deploy WIMSE/SPIFFE from scratch.
- **IETF SD Agent -- Selective Disclosure for Agent Discovery and Identity Management (draft-nandakumar-agent-sd-jwt-02, February 28, 2026):** Suhas Nandakumar Individual Submission adapting the SD-JWT primitive (selective disclosure of JWT claims) to Agent-to-Agent systems. The draft defines an "SD-Card" format where an agent's published capabilities live in a registry-issued credential whose individual claims can be selectively disclosed depending on the discovery context (`public`, `internal`, `diagnostic`, `federation`), with key-binding via a `cnf` claim providing cryptographic proof that the presenter holds the private key matching the SD-Card. For 9.5.5 this is the most direct Internet-Draft proposal of *capability-disclosure as a peer-authorization control*: an agent reaching out to a peer reveals only the capability subset appropriate to the relationship rather than its full operational manifest, which closes the "Agent Cards leak too much detail to untrusted peers" gap the A2A protocol currently leaves open. Sub-agent delegation is bound to OAuth 2.0 Token Exchange (RFC 8693) with explicit scope reduction at each hop, composing naturally with the AAuth / Transaction Tokens / ID-JAG family already cataloged.
- **IETF A Comprehensive Roadmap for OAuth 2.0 Standards and Drafts (draft-chen-oauth-roadmap-01, May 6, 2026):** Lin Chen Individual Submission cataloguing the full OAuth 2.0 specification surface across security mechanisms, token management, grant types, and client profiles. Worth noting in this section primarily for what it does *not* contain: as of the May 6, 2026 revision the roadmap has no dedicated agent-authorization category, despite the dozen-plus active drafts targeting AI agent workloads (Transaction Tokens for Agents, AAuth, AIMS, AAP, DAAP, ACAP, ID-JAG, the Attenuating Agent Tokens draft, the Actor Profile, the Delegated Authorization draft, and the Identity Chaining cross-domain draft). The closest reference to multi-hop delegation is the RFC 8693 Token Exchange entry framed as "service-to-service delegation" for microservices. For 9.5.2 and 9.5.5 this is corroborating evidence for the open research question this page already raises: agent authorization is being standardized in parallel tracks rather than under a unified profile, and downstream implementers will need to compose primitives across these drafts themselves until the WG-level convergence catches up.
- **IETF Agentic JWT / Secure Intent Protocol (draft-goswami-agentic-jwt-00, December 31, 2025):** Defines an OAuth 2.0 extension that ties an authorization token to *which* agent configuration is allowed to spend it. The draft introduces a new grant type `urn:ietf:params:oauth:grant-type:agent_checksum` where the agent presents a SHA-256 hash computed over its canonicalized configuration (system prompt, sorted tool signatures, optional deep tool source-code hash) using RFC 8785 (JCS) canonicalization. The authorization server compares the supplied checksum to the registered agent's reference value with constant-time comparison; a mismatch (`agent_checksum_mismatch`) is treated as a code-integrity violation and the request is denied. The resulting *intent token* is a short-lived (recommended ~5 minutes) JWT carrying a `cnf` proof-of-possession key, a nested `intent` claim (`workflow_id`, `workflow_step`, `executed_by`, a `delegation_chain` hash, and a `step_sequence_hash`), and an `agent_proof` block (`agent_checksum`, `registration_id`). Authorization server validation is strictly ordered: grant type → agent existence → checksum verification → workflow-step authorization (with prerequisite-step completion and approval-gate checks) → token issuance. For 9.5.2 and 9.5.5 this is a useful complement to the AIMS/AAuth/Transaction Tokens family: instead of carrying identity in the token alone, the protocol re-binds *each* authorization request to a fingerprint of the agent's current code-and-prompts surface, so a tampered or substituted agent cannot quietly continue spending tokens that were issued to its earlier configuration.
- **IETF OAuth 2.0 Delegated Authorization (draft-li-oauth-delegated-authorization-01, March 2, 2026):** A distinct delegation primitive from RFC 8693 Token Exchange and the agent-specific drafts above. Parent *delegation tokens* issued by an authorization server carry a `delegation_key` (public key) and optional `max_delegation_depth`; clients then locally *mint* subordinate *delegated access tokens* with narrowed `scope`, `aud`, and shortened `exp` without re-contacting the authorization server. A new `Delegated-Authorization` HTTP header carries the subordinate token separately from client authentication, and resource servers verify the parent-child cryptographic relationship at request time. Hierarchical control is enforced as a one-way property: child claims can only narrow -- never expand -- from the parent. Appendix C.1 names AI personal-assistant agents with workflow-scoped delegated tokens as the motivating use case. For 9.5.2 and 9.5.5 this is an attractive alternative to RFC 8693 when round-trips to the authorization server at each delegation hop are operationally expensive (high-throughput multi-agent chains, edge agents, durable workflows that resume hours later) -- the trade-off is that the policy-decision point can no longer block delegation at issuance time and must instead enforce constraints when the subordinate token is presented at the resource server. The two patterns are complementary, not competing: server-mediated exchange (RFC 8693) for sensitive scope reductions where AS observability matters; client-side delegated tokens for high-volume scope narrowing within an already-bounded parent envelope.
- **IETF Agent Authorization Profile (AAP) for OAuth 2.0 (draft-aap-oauth-profile-01, February 7, 2026):** An Internet-Draft by A. Cruz (Independent) defining a JWT profile for OAuth 2.0 specifically for autonomous AI agents. AAP is not a new protocol -- it is a structured-claims layer that sits on top of OAuth 2.0, JWT, RFC 8693 Token Exchange, and proof-of-possession mechanisms (DPoP/mTLS). The profile introduces seven claim namespaces directly aligned with the 9.5.x verification surface: **`aap_agent`** (agent identity, operator, model, runtime attestation), **`aap_task`** (task binding with explicit purpose declaration and data sensitivity levels -- the JWT-claim analogue of the mission-hash construction in AAuth and the instruction-hash binding in the IETF Delegation Receipt Protocol), **`aap_capabilities`** (action permissions with typed constraints: rate limits, domain restrictions, time windows), **`aap_oversight`** (human approval requirements for sensitive actions, complementing OAuth Step-Up Authentication / RFC 9470), **`aap_delegation`** (depth tracking and immutable delegation chains via token exchange, with mandatory privilege reduction at each hop), **`aap_context`** (network zones, time windows, geographic restrictions), and **`aap_audit`** (trace and session identifiers for log correlation). For 9.5.2 and 9.5.5 this is the most concise Internet-Draft proposal to date of the dual-constraint, scope-narrowing, intent-bound delegation pattern this section advocates. The draft positions itself as complementary to Zanzibar, service meshes, and cloud IAM rather than as a replacement, which makes it easier to adopt incrementally on top of existing OAuth infrastructure -- the same adoption-friendly stance that distinguishes OIDC-A from WIMSE/SPIFFE rebuilds. Draft -01 expires August 11, 2026.
- **IETF Agent Operation Authorization (draft-liu-agent-operation-authorization-02, updated March 15, 2026):** Dapeng Liu / Hongru Zhu (Alibaba) and Suresh Krishnan (Cisco) Individual Submission defining a two-phase consent flow purpose-built for agent operations. The agent serializes the proposed operation as an `agent_operation_proposal` JWT -- carrying Rego policy rules and execution context but **not** the user's original natural-language prompt -- and submits it to the authorization server via OAuth 2.0 Pushed Authorization Requests (RFC 9126). The AS renders the structured proposal to the user, captures explicit confirmation, and mints an *Agent Operation Authorization Token* that embeds a cryptographic record of that confirmation; resource servers verify the token and evaluate the bound Rego policy at action time via Open Policy Agent. For 9.5.3 this is the cleanest Internet-Draft proposal to date of "the user, not the model, decides what the operation actually is" -- the structured proposal becomes the authoritative authorization artifact, and any drift between the model's natural-language framing and the executed operation is detectable at the AS rather than at the resource. The draft also defines a `delegation_chain` claim with end-to-end traceability to the originating human principal (composing naturally with the AAP `aap_delegation` and Transaction Tokens `actchain` patterns above) and a "semantic expansion levels" mechanism that records how the agent mapped imprecise terms ("cheap", "soon") to concrete values, which complements the IETF Delegation Receipt Protocol's instruction-hash binding by capturing the *interpretation* step the operator would otherwise sign blind. The draft expires September 17, 2026.
- **IETF Delegated Agent Authorization Protocol (DAAP) (draft-mishra-oauth-agent-grants-01, March 2, 2026):** A Sanjeev Kumar (Grantex) submission extending OAuth 2.0 with a human-consent grant flow purpose-built for AI agents. Each agent receives a persistent Decentralized Identifier in the form `did:grantex:ag_<ULID>`, and grants are encoded as signed JWTs carrying agent-specific claims: `agt` (agent DID), `dev` (developer organization), `grnt` (grant identifier for revocation), `scp` (scope array), `bdg` (optional remaining budget), and the multi-agent delegation triple `parentAgt`/`parentGrnt`/`delegationDepth`. DAAP enforces **cascade revocation** -- revoking a parent grant atomically invalidates every descendant grant -- and limits chain depth (default 3, hard cap 10) with strict scope-subset narrowing at each hop. The protocol also defines real-time budget enforcement, hash-chained append-only audit trails, event streaming for revocation, an integrated credential vault, and explicit integration points with external policy backends (OPA, Cedar). For 9.5.2 and 9.5.5 DAAP is operationally adjacent to the AAuth/Transaction Tokens/AAP family but is the most explicit Internet-Draft proposal to date of *budget-bound* and *cascade-revocable* multi-agent delegation -- a notable convergence with the same Grantex group whose 2026 enterprise survey first quantified the "93% of agent projects use unscoped API keys" gap that this section already cites. The draft expires September 3, 2026.
- **IETF OAuth Actor Profile for Delegation (draft-mcguinness-oauth-actor-profile-00, April 30, 2026):** A Karl McGuinness (Independent) draft standardizing how the RFC 8693 `act` claim should be populated for cross-domain delegation. The profile extends `act` with an `iss` field and an optional `sub_profile` entity-type classification, and defines uniform processing rules across three token shapes (JWT assertion grants, JWT access tokens, and IETF Transaction Tokens). The canonical actor identifier becomes the `(act.iss, act.sub)` pair, removing the need to infer actor identity from OAuth client registration metadata -- a brittle pattern that breaks across organizational boundaries. The draft explicitly names `ai_agent` as a recognized `sub_profile` classification and works through a cross-domain AI travel-assistant example to illustrate the booking-on-behalf-of flow. For 9.5.2 this is the closest published interoperability piece between RFC 8693-style on-behalf-of tokens and the agent-specific delegation drafts cataloged above -- it does not introduce a new delegation primitive, it normalizes how existing primitives represent the "who is acting for whom" pair so policy decision points and audit pipelines can reason about agent actors consistently. Draft expires November 1, 2026.
- **IETF Agent Credential Attestation Protocol (ACAP) (draft-yakung-oauth-agent-attestation-00, March 26, 2026):** Chudah Yakung (Attest) Individual Submission defining a JWT credentialing scheme (RS256-signed) for multi-hop agent pipelines. ACAP carries a SHA-256 hash of the original human instruction set in every credential (intent binding analogous to the IETF Delegation Receipt Protocol's instruction hash and AAuth's Mission hash), enforces **monotone scope reduction** at delegation time through an explicit `IsSubset` algorithm so a child credential is mathematically incapable of exceeding its parent's scope, caps delegation depth at 10 with per-credential ancestry tracking via tamper-evident JWT identifier chains, and provides cascade revocation plus hash-chained append-only audit logs. The architectural distinction from OAuth-anchored designs is that ACAP credentials are self-contained -- no external authorization server round-trip is required at verification time, which is operationally useful for edge agents and disconnected workers but shifts the trust model toward issuer-key management. For 9.5.5 and 9.5.6 ACAP composes naturally with the IETF Transaction Tokens for Agents (`actchain`) and the Delegation Receipt Protocol's instruction-hash pattern; together they describe a converging design space where published agent-delegation proposals increasingly carry either an intent hash, a scope-attenuation invariant, or both. The draft expires September 27, 2026.
- **IETF OAuth 2.0 Agents Native Authorization (draft-embesozzi-oauth-agent-native-authorization-00, April 3, 2026):** Martin Besozzi (TwoGenIdentity) submission extending the OAuth 2.0 First-Party Applications (FiPA) specification with a Structured Elicitation format for non-browser clients. The `insufficient_authorization` Authorization Challenge Response gains an `elicitations` array where each entry declares an interaction `mode` (currently `form`), a human-readable `message`, and a `requestedSchema` (JSON Schema) describing the inputs the authorization server needs from the user before the next step. A normative *MCP Elicitation Binding* maps each entry to a Model Context Protocol `elicitation/create` call so MCP-hosted agents collect missing consent or step-up factors through the existing client elicitation channel rather than reinventing a user-interaction primitive; the user's response is returned in the matching Authorization Challenge Request alongside an `auth_session` identifier. Scope is limited to just-in-time authorization for Human-to-Agent (H2A) flows where the agent acts as a first-party OAuth client; Agent-to-Agent flows are explicitly out of scope. For 9.5.2, this individual Informational Internet-Draft explores how an agent can collect missing consent or step-up input while retaining an `auth_session`; it is not settled protocol guidance. It does not provide a 9.5.4 credential-isolation guarantee: its current form-mode example can carry a TOTP value through the MCP client and agent path, and its Security Considerations remain TBD. Stable MCP URL-mode elicitation is the safer pattern for sensitive authenticator material because that material must stay outside the MCP client.
- **IETF PTV Protocol for Attested Agent Identity (draft-anandakrishnan-ptv-attested-agent-identity-00, March 2026):** Specifies the Prove-Transform-Verify (PTV) protocol for hardware-anchored, zero-knowledge attested agent identity in federated environments. PTV addresses the "data gravity" problem in centralized security models by enabling cross-domain agent authorization without raw data exposure. Technical building blocks include TPM 2.0 / secure enclave roots of trust, Groth16 zero-knowledge proofs (sub-200 ms generation on commodity edge hardware), HotStuff Byzantine Fault Tolerant consensus for immutable audit trails, and Sovereign Bound metadata for jurisdiction-aware attestation chains. Target use cases include healthcare clinical decision support, ICS/OT critical infrastructure, and cross-border GDPR/HIPAA compliance scenarios. A companion variant (`draft-anandakrishnan-rats-ptv-agent-identity-00`) composes PTV with the IETF RATS framework. For 9.5.5 this is the registry-layer analogue to `draft-drake-agent-identity-registry`: instead of certifying identity through manufacturer-signed keys alone, PTV lets a verifier confirm an agent's identity *and* its execution-context attestation without learning the underlying evidence, which is the privacy-preserving counterpart to the Microsoft Authorization Fabric's PEP/PDP architecture for regulated multi-tenant deployments.
- **OWASP A Practical Guide for Secure MCP Server Development (February 16, 2026):** The OWASP Gen AI Security Project published this guide as the working "minimum bar" for MCP server deployment. It names the *confused deputy* explicitly as a threat class -- an MCP server using a user's credentials to perform actions the user neither intended nor is authorized for -- and codifies the prohibition on token passthrough: the MCP server must obtain a separate credential for the downstream resource rather than forward the client's raw token, with OAuth 2.0 Token Exchange / RFC 8693 on-behalf-of as one option rather than a mandate. The guide is the closest thing to a concrete checklist for 9.5.1, 9.5.2, and 9.5.3 in the MCP context as of mid-2026.
- **MCP Enterprise-Managed Authorization Extension (SEP-646, merged April 2026):** Aaron Parecki's SEP-646 added the `io.modelcontextprotocol/enterprise-managed-authorization` extension to the MCP specification, moving authorization decisions from per-user, per-server consent flows to the organization's existing identity provider. The extension composes ID-JAG (the WG-track Identity Assertion JWT Authorization Grant cited earlier in this section): the MCP client authenticates the user against the enterprise IdP, exchanges the resulting ID token for an ID-JAG bound to a specific MCP resource, then exchanges the ID-JAG with the MCP Authorization Server for an access token. The IdP -- not the individual employee -- decides which MCP servers are reachable, what scopes are issued, and when access is revoked, which closes the per-user-per-server consent fragmentation Posta's April 2026 enterprise-readiness critique flagged. For 9.5.2 this is the standards-aligned answer to "how does delegation context propagate across an enterprise estate of MCP servers without the user re-authorizing each one," and for 9.5.5 it makes the IdP the de facto peer-authorization registry: an MCP server an admin has not approved in the IdP simply cannot mint an access token. The first reference implementation is Okta's Cross App Access (see the Agent Gateways section below); other enterprise IdPs are expected to ship support against the same extension contract.
- **IETF AgentROA -- Agent Route Origin Authorization (draft-nivalto-agentroa-route-authorization-01, April 8, 2026):** Joseph Michalak (Nivalto) Individual Submission modeled explicitly on RPKI's BGP Route Origin Authorization (RFC 6480) but applied to MCP and A2A tool calls. Three protocol objects work in concert: an **ROA envelope** (a signed policy document issued at session initialization specifying the agent's authorized capabilities), an **ARA object** (a per-hop attestation produced at each delegation boundary that enforces monotonic scope-narrowing across capability set, budget ceiling, and SLO dimensions), and an **AER receipt** (a cryptographic commitment produced intrinsically by each enforcement decision). A **Border Gateway** enforcement proxy sits at the MCP protocol boundary -- external to the agent's execution context -- intercepting tool-call requests, validating the caller's ROA envelope, enforcing the no-loosen invariant across the ARA chain, and emitting AERs as audit-grade evidence that "this agent was authorized to perform this action under this policy at this moment." The structural insight is the deliberate separation of authorization infrastructure (the ROA signer) from enforcement infrastructure (the Border Gateway): the enforcer never has to trust the party being governed. For 9.5.5, this is a clean protocol-level realization of dual-constraint intersection enforcement, while the ARA chain is the most explicit Internet-Draft proposal to date for requiring scope to narrow at each delegation hop. The draft expires October 10, 2026.
- **IETF DMSC Cross-Domain Interoperability for AI Agent Collaboration (draft-cui-dmsc-agent-cdi-00, February 11, 2026):** Yong Cui (Tsinghua University) submission to the IETF Dynamic Multi-agent Secured Collaboration (DMSC) program. The framework defines an **Interoperability Gateway** as the primary inter-domain interface, providing five composable services: trust brokerage (mutual attestation across administrative realms), policy mediation (translating one domain's authorization model into another), agent discovery (capability advertisement across federation boundaries), communication routing (encrypted paths that respect organizational boundaries), and audit collection (centralized audit visibility across domains). DMSC complements `draft-liu-dmsc-acps-arc` (Agent Collaboration Protocols Architecture) and `draft-li-dmsc-inf-architecture` (DMSC infrastructure architecture) into a coherent multi-draft program that targets B2B and supply-chain agent deployments. For 9.5.5 this is the closest IETF-track analogue to A2A peer authorization when the peers belong to different organizations -- the place where Agent Cards alone cannot answer "is this agent's home domain itself an approved peer?" The draft expires August 15, 2026.
- **OWASP MCP Top 10 (beta, project lead Vandana Verma Sehgal):** The first OWASP project dedicated to the Model Context Protocol, organized as ten ranked risk categories with concrete mitigations. **MCP07: Insufficient Authentication & Authorization** is the entry that maps most directly to 9.5.1 and 9.5.3 -- it covers missing or optional API key/token validation, agents performing actions beyond intended privileges, and missing enforcement across the multi-agent surface. The category's recommended mitigations are operationally aligned with this section: mTLS between MCP clients, agents, and servers; short-lived scoped tokens (JWT/OAuth 2.0); fine-grained RBAC/ABAC with deny-by-default; deterministic token lifecycle management (expiration, rotation, revocation); least-privilege per task; centralized IAM integration; comprehensive logging of authentication attempts and authorization decisions; and secure defaults that disable anonymous access. Because the project is in beta the category numbering is expected to evolve, but the underlying control set is stable and complements the Practical Guide above by surfacing missing-auth as the highest-priority MCP failure mode.

- **NSA AISC Cybersecurity Information Sheet on MCP (May 20, 2026):** The NSA's Artificial Intelligence Security Center published *Model Context Protocol (MCP): Security Design Considerations for AI-Driven Automation* (U/OO/6030316-26), the first US national-security-agency guidance dedicated to MCP. Its authorization-relevant recommendations map directly onto this section: enforce least-privilege tokens for every action and tool (9.5.1), require signed provenance checks for dynamically discovered tools, log every MCP request and response with the exact parameters and identities involved (making 9.5.2's delegation context auditable), route external MCP connections through a filtering outbound proxy with DLP, and cryptographically bind requests to time and context for replay protection. The CSI also sets an operational bar -- patch MCP-related CVEs within 48 hours of disclosure -- that the PraisonAI (3h44m disclosure-to-scan) and LiteLLM KEV timelines suggest is barely fast enough.
- **Five Eyes Joint Guidance -- Careful Adoption of Agentic AI Services (May 1, 2026):** Six national cybersecurity agencies (CISA, NSA, Australia's ACSC, the Canadian Centre for Cyber Security, New Zealand's NCSC, and the UK NCSC) jointly published agentic-AI adoption guidance whose authorization core restates 9.5.2 and 9.5.5 as national-level policy: each agent requires a distinct, cryptographically verified identity with short-lived credentials rather than persistent API keys (preferring managed identities or workload identity federation); agents must not extend implicit trust to other agents simply by virtue of co-membership in the same system; and inter-agent relationships must be explicitly defined and cryptographically enforced rather than assumed through network proximity or shared environment. The guidance also calls for explicit human approval workflows for high-impact agentic actions and characterizes prompt injection as the most persistent and difficult-to-fix threat -- reinforcing why 9.5.3 keeps authorization decisions out of the model.

### Practical Authorization Patterns

- **Scopes for coarse-grained limits, claims for fine-grained context:** Use OAuth scopes to define broad permission boundaries and token claims to carry runtime context (user identity, tenant, data classification, risk score) that policy engines evaluate per-call. URI-style scope naming (e.g., `urn:infra:monitoring:read`) improves cross-service compatibility.
- **On-behalf-of delegation with dual-identity tokens:** Keep the initiating principal and acting agent independently visible to every policy decision. In the RFC 8693 JWT representation, the delegated principal is the top-level `sub`, the current actor is the outer `act`, and nested `act` entries record prior actors; AIMS instead places the agent in `client_id` and the user or system in `sub`. Follow the token issuer's declared profile, validate both identities, and do not treat the historical nested actors as authorization inputs.
- **Policy engines at the enforcement point:** OPA, Cedar, and Zanzibar-style systems enforce fine-grained policies. The policy engine must sit between the model's tool selection and actual tool execution -- never after. As of March 2026, AWS Bedrock AgentCore demonstrates this pattern concretely: every agent-to-tool request is evaluated against Cedar policies before execution, with policies definable programmatically or generated from natural-language descriptions. The *Open Agent Passport (OAP)* reference implementation (arXiv 2603.20953, March 21, 2026) provides an open specification for this pattern: it synchronously intercepts tool calls before execution, evaluates them against a declarative policy, and produces a cryptographically signed audit record, reporting a measured median enforcement latency of 53 ms over 1,000 runs -- concrete evidence that per-call deterministic authorization is feasible at agent-scale throughput rather than a theoretical ideal.
- **Just-in-time credential issuance:** Request credentials at runtime for each tool invocation and discard them immediately after task completion. This eliminates standing access vulnerabilities inherent in long-lived tokens. The Oso platform implements this pattern with delegated access where the agent inherits the user's identity via token exchange and the downstream service enforces the user's current permissions in real-time -- if the user loses access, so does the agent immediately.
- **Short-lived, single-use credentials:** Replace long-lived API keys with ephemeral tokens scoped to individual tasks. This limits blast radius and eliminates the stale-authorization window. The Grantex 2026 survey found 93% of popular agent projects still use unscoped API keys -- this is the single most common authorization failure in deployed agent systems.
- **Human-in-the-loop approval gates:** For sensitive or destructive operations (healthcare data access, record deletion, compliance actions), require manual approval. Grant access to action *types* rather than individual operations to keep the approval burden manageable.

### Recent Incidents and Attack Patterns

Several real-world incidents in 2025--2026 demonstrate why authorization enforcement matters for agents:

- **MITRE ATLAS AML.T0053 -- AI Agent Tool Invocation:** Formally catalogued as a technique where adversaries exploit an agent's authorized tool access to perform unauthorized actions. Microsoft Copilot M365 and GitHub Copilot Chat have both been exploited through this vector, with agents directed to exfiltrate data or perform actions outside intended scope.
- **OpenClaw Malicious Skills (February 2026):** The largest confirmed supply chain attack targeting agent infrastructure. MITRE ATLAS investigators confirmed 1,184 malicious skills across the ClawHub ecosystem (roughly one in five packages), using direct and indirect prompt injection combined with unauthorized tool invocation.
- **Langflow Authentication Bypass (2025):** CVE-2025-34291 demonstrated how permissive CORS settings combined with credential forwarding created a complete authentication bypass in an agent workflow platform. Default configurations allowed cross-origin requests with credentials from any origin.
- **Git MCP Server Exploit Chain (January 2026):** Three CVEs (CVE-2025-68143, CVE-2025-68144, CVE-2025-68145) targeting Anthropic's Git MCP server demonstrated path traversal, argument injection, and repository scoping bypass -- all of which are authorization boundary failures.
- **Microsoft 365 Copilot BizChat Privilege Escalation (March 2026):** CVE-2026-26137 (CVSS 8.9) -- an SSRF vulnerability in Copilot BizChat that allowed an authorized attacker to escalate privileges over the network. The vulnerability is notable because the AI assistant itself becomes the privilege escalation vector: if an attacker can move the assistant into a higher-authority state, it becomes a tool for scaling access across the organization.
- **Microsoft 365 Copilot Information-Disclosure Trio -- CVE-2026-26129 / CVE-2026-26164 / CVE-2026-33111 (May 7, 2026 Patch Tuesday):** Microsoft disclosed three concurrent information-disclosure vulnerabilities across the Copilot surface (advisories landed May 13, 2026 in the Patch Tuesday review): CVE-2026-26129 (CVSS 7.5, CWE-138 improper neutralization of special elements in M365 Copilot Business Chat) and CVE-2026-26164 (CVSS 7.5, CWE-74 improper neutralization in output used by downstream components, M365 Copilot Business Chat) both let an unauthenticated network attacker (`AV:N/AC:L/PR:N/UI:N`) extract sensitive information through prompt-side injection that the assistant rewrites into downstream tool arguments; CVE-2026-33111 (CVSS 7.5, CWE-77 command injection in Copilot Chat embedded in Microsoft Edge) does the same against the browser-resident Copilot via crafted prompt input that reaches an unsanitized command-handling path. All three were mitigated cloud-side by Microsoft with no customer action and no observed exploitation. The control implication for 9.5.3 is the same as CVE-2026-26137 (SSRF), CVE-2026-32173 (broadcast eavesdropping), CVE-2026-32213 (role boundary), CVE-2026-35435 (published-agent permission inheritance), and CVE-2026-33102 (open-redirect EoP): the model never makes an authorization decision, but its output is treated as authorization input by a downstream component (tool argument, command interpreter, audience filter, redirect target). The May 7 patch cycle brings the total to *eight* qualitatively distinct authorization-relevant vectors disclosed in Microsoft's AI assistant stack between mid-February and mid-May 2026 -- four execution-side (SSRF, broadcast, role boundary, published-agent inheritance), one redirect-side (open redirect), and three information-disclosure (the CWE-138/74/77 trio) -- which is the single largest concentration of agent-authorization disclosures against any one vendor over a single quarter and the strongest case to date for treating Copilot-class surfaces as a load-bearing PEP rather than a thin OAuth-scope wrapper.
- **Antigravity 2.0 Subagent Delegation Platform (Google I/O 2026, May 19, 2026):** Google launched Antigravity 2.0 alongside the Antigravity CLI, Antigravity SDK, and Managed Agents in the Gemini API as a full rebuild positioned around multi-agent orchestration. The platform formalizes a primary-agent / subagent delegation model with three operating modes (`chat`, `task`, and `single-turn` with parallel execution), human-in-the-loop approval required for risky-classified subagent actions (modifying core security libraries, deleting files), and ephemeral per-agent sandboxes provisioned with skills, MCP servers, and server-side tools. The CLI replaces Gemini CLI as the supported terminal interface (Gemini CLI and Gemini Code Assist IDE extensions stop serving Pro/Ultra/free tiers after June 18, 2026). For 9.5.5 this is the most explicit subagent-registry primitive shipped by a major vendor to date -- subagents are declared with name, when-to-use criteria, accessible tool set, model, and permission profile, which is the deployment-convention realization of the peer-authorization control 9.5.5 mandates. For 9.5.4, however, the credential-isolation gaps the original Antigravity disclosure surfaced persist: the default Browser URL Allowlist still includes `webhook.site`, which Google has classified as Intended Behavior (Won't Fix), and "never store credentials in the workspace accessible by the agent" remains the documented mitigation rather than a structural guarantee. The 2.0 launch is therefore a 9.5.5 capability improvement (formalized subagent registry + HITL gates) that does not close the 9.5.4 credential-exposure gap the original platform shipped with. The Antigravity Code Reviewer subagent reaching production at Google I/O 2026 is also a useful reference point for what enterprise subagent-registry registration looks like in practice.
- **Microsoft 365 Copilot Open-Redirect Elevation of Privilege -- CVE-2026-33102 (April 23, 2026, CVSS 9.3):** Microsoft assigned CVE-2026-33102 to an "open redirect" (CWE-601) URL redirection flaw in Microsoft 365 Copilot, with GHSA-q6x5-8v4h-crwr published in the April 2026 Patch Tuesday cycle. The flaw lets an unauthorized network attacker (`AV:N/AC:L/PR:N/UI:R/S:C/C:H/I:H`) coerce the Copilot surface into redirecting to an attacker-controlled site, which is then used to influence what Copilot accesses, how it responds, or which resources it reaches on the user's behalf -- the assistant itself becomes the privilege-amplification vector. Microsoft's server-side fix was applied automatically with no customer action and no reported in-the-wild exploitation. The control implication for 9.5.3 is the same as CVE-2026-26137 and CVE-2026-35435, but the *vector* is operationally distinct: the model never sees the redirect target, but a third-party authorization or content endpoint does, so 9.5.3 verification must extend to the outbound redirect surface of the AI assistant -- destinations the model can be steered to point users at are authorization-bearing the moment the receiving service treats the redirected user as a Copilot-authenticated principal. Together with the SSRF (26137), broadcast (32173), role boundary (32213), and published-agent permission inheritance (35435) failure modes, CVE-2026-33102 fills out the *fifth* qualitatively distinct authorization vector disclosed in Microsoft's AI assistant stack inside ten weeks, reinforcing the case for treating the Copilot-class surface as a load-bearing PEP rather than a thin wrapper over its underlying OAuth scope.
- **MS-Agent Shell Injection (2026):** CVE-2026-2256 -- the MS-Agent framework for autonomous AI workflow automation failed to sanitize user-controlled or AI-generated inputs passed through its built-in Shell tool, enabling prompt injection to trigger arbitrary command execution with the agent's privileges. This is a textbook example of why tool parameter validation must happen at the policy engine level, not in the agent itself.
- **PleaseFix Agentic Browser Hijack (March 2026):** Zenity Labs disclosed PleaseFix, a family of critical vulnerabilities in agentic browsers including Perplexity Comet. Two exploit chains were demonstrated: (1) a zero-click compromise where attacker-controlled content (e.g., a calendar invite) triggers autonomous agent execution, granting access to the local filesystem and exfiltrating data while the agent continues returning expected results to the user; (2) credential theft where the agent's authorized access to password managers (1Password) is abused to steal stored credentials or enable full account takeover. Neither exploit directly attacks the password manager -- both operate entirely within the agent's legitimately granted permissions, making this a real-world demonstration of authorization scope being technically correct but semantically exploitable. Perplexity patched the browser-side execution issue before public disclosure.

The paper is a research proposal, not deployed software, but it is the cleanest published example to date of *governance-plane cryptography* (post-quantum, formally-modeled, broadcast-aware) as a primitive distinct from the per-token cryptography the OAuth-family drafts above advance. For deployments evaluating long-lived agent fleets whose lifetimes will cross the post-quantum migration window, MAGIQ and AITH (arXiv 2604.07695) together describe the pattern this section's verification surface will likely need within five years.
- **MCP Localhost DNS Rebinding Class -- Apollo / rmcp / Python SDK (Dec 2025 -- April 2026):** Three independently-disclosed CVEs in the most widely-used MCP server implementations all share the same root cause: HTTP transports bound to localhost did not validate the incoming `Host` header, so a malicious website a victim visited could use DNS rebinding to make the victim's browser issue tool invocations against the local MCP server as if from the same origin -- bypassing the same-origin policy and executing tools under the victim's local session without any explicit authentication step. CVE-2025-66416 (CVSS 7.6, advisory `GHSA-9h52-p55h-vw2f`, published December 2, 2025) covered the official **MCP Python SDK** (FastMCP, `StreamableHTTPSessionManager`, `SseServerTransport`) and was patched in `mcp` 1.23.0, which now enables DNS rebinding protection by default when the server's `host` is `127.0.0.1` or `localhost`. CVE-2026-35577 (CVSS 6.8, advisory `GHSA-wqrj-vp8w-f8vh`, published April 9, 2026, vector `AV:N/AC:H/PR:N/UI:R/S:U/C:H/I:H/A:N`) covered the **Apollo MCP Server** (StreamableHTTP transport) and was patched in 1.7.0. CVE-2026-42559 (CVSS 8.8, advisory `GHSA-89vp-x53w-74fx`, published April 29, 2026) covered the official **MCP Rust SDK (`rmcp`)** Streamable HTTP server transport and was patched in `rmcp` 1.4.0, which ships a loopback-only Host allowlist (`localhost`, `127.0.0.1`, `::1`) by default. The structural lesson for 9.5.1 is that *origin* is an authorization input -- a tool call that reaches the MCP server is only authorized when it originated from a caller bound to the server's intended trust boundary, and a missing or permissive `Host` header check collapses that boundary at the browser layer rather than the network layer. The class is the browser-mediated counterpart to the "MCP server bound to `0.0.0.0` with no auth" pattern in nginx-ui MCPwn and mcp-atlassian MCPwnfluence, but it bites even when the server is *correctly* bound to localhost: the attacker never connects to the server directly, the victim's browser does, and the request appears to be same-origin to any naive enforcement layer. For 9.5.4 verification the same chain matters because a successful DNS rebinding invocation runs tools with whatever credentials the local MCP server holds -- managed identities, OS keyring entries, cached OAuth tokens -- without ever entering the model's context. The cross-SDK pattern (Python SDK, Rust SDK, and a major third-party server all shipping the same defect within five months) is the cleanest evidence yet that *secure-defaults* enforcement of MCP transport-security guidance is a protocol-ecosystem responsibility, not an integrator responsibility.
- **2026 Incident Volume:** Industry tracking reports 520 confirmed tool misuse and privilege escalation incidents involving AI agents in 2026 alone, making it the most common reported agent security incident category.

### Semantic Privilege Escalation

A distinct class of authorization failure is emerging that traditional access control cannot address: **semantic privilege escalation**. In this attack, the agent never exceeds its technical permissions -- every API call, every tool invocation is individually authorized. But the *sequence* of actions falls outside the semantic scope of the assigned task.

Consider: a user asks an agent to summarize a document. The document contains hidden prompt injection (white text, invisible Unicode). The agent, following the injected instructions, scans the user's drive for API keys and emails them to an external address. Each individual action (read document, search files, send email) may be within the agent's technical authorization. The escalation is in *intent*, not in *permissions*.

Defending against semantic privilege escalation requires capabilities beyond traditional IAM:

1. **Intent capture and tracking** -- Record what the user actually asked for and maintain it as a reference throughout execution.
2. **Intent-action alignment evaluation** -- At each step, compare the proposed action against the captured intent. Does "summarize this document" semantically justify "search for files matching *.key"?
3. **Behavioral anomaly detection** -- Flag action sequences that deviate from typical patterns for the task type, even if each action is individually authorized.
4. **Security-annotated execution traces** -- Log not just what happened, but the semantic justification chain, enabling post-hoc analysis of whether the agent stayed within scope.

As of April 2026, no production authorization framework fully addresses semantic privilege escalation. GitGuardian's analysis of enterprise MCP deployments identifies this as a "sequence-level risk" -- individual operations each pass authorization checks, but their combination produces unauthorized outcomes. The multi-hop MCP architecture (user -> AI host -> MCP client -> multiple servers) concentrates risk not in individual requests but in the sequences agents orchestrate and the gaps between operations where context or credentials might leak. Gateway-mediated authorization (see MCP Agent Gateways below) partially addresses this by maintaining cross-request context at the enforcement point, but reliable real-time detection of semantic escalation without unacceptable false-positive rates remains an unsolved problem.

Otsuka, Toyoda, and Leung's *AI Identity: Standards, Gaps, and Research Directions for AI Agents* (arXiv 2604.23280, April 25, 2026) frames the broader structural issue: AI identity is "the continuous relationship between what an AI agent is declared to be and what it is observed to do" with bounded confidence, and human-identity frameworks extended to agents without structural modification produce systematic failures. The paper names five gaps -- semantic intent verification, recursive delegation accountability, agent identity integrity, governance opacity and enforcement, and operational sustainability -- and explicitly concludes that "these gaps are structural; more engineering effort alone will not close them." That conclusion is the academic counterpart to Cisco's "identity passes, authorization fails" framing and the Meta SEV1 post-mortem: 9.5.5 and 9.5.6 are not bugs in current implementations to be patched, they are missing primitives that current standards do not yet supply.

Two complementary protocol-level proposals from April--May 2026 attempt to close the human-provenance gap directly. Xiaohang Yu, Hejia Geng, Xinmeng Zeng, and William Knottenbelt's *SUDP: Secret-Use Delegation Protocol for Agentic Systems* (arXiv 2604.24920, April 27, 2026; revised May 1, 2026) formalizes the **Agent Secret Use (ASU)** problem -- current "authorization by exposure" approaches turn a transient prompt-injection or tool-side compromise into durable account compromise -- and proposes a three-role architecture (Requester proposes a canonical operation, User authorizes it with a fresh authenticator-backed grant, Custodian redeems the grant exactly once to execute the bounded operation) that ensures *reusable authority never crosses the requester boundary*. The protocol's three core guarantees -- verifiable authorization, operation-bound usage, and single-use enforcement -- are operationally adjacent to the IETF Delegation Receipt Protocol and AAuth Missions but split the authorization, custody, and use roles across three independently-trustable parties rather than binding them to a single token. Asiri Dalugoda's *HDP: A Lightweight Cryptographic Protocol for Human Delegation Provenance in Agentic AI Systems* (arXiv 2604.04522, April 6, 2026; also published as an IETF Internet-Draft with reference TypeScript and Python SDKs) attacks the same gap from the verification side: HDP tokens bind a human authorization to a session and record each agent's delegation step as a signed entry in an append-only Ed25519 chain, with offline verification requiring only the issuer's public key and current session identifier (no external registries or trust anchors). For 9.5.2 and 9.5.5 these two drafts converge on the same architectural conclusion the IETF Delegation Receipt Protocol and ACAP already reached -- the human authorization is a non-waivable, cryptographically-pinned input to every downstream decision -- but SUDP adds the *single-use, custodian-mediated* invariant that protects high-impact actions even when the long-running session has otherwise been compromised, and HDP adds the *offline-verifiable* property that matters for edge agents and air-gapped delegation chains.

Reshabh K Sharma, Linxi Jiang, Zhiqiang Lin, and Shuo Chen's *PAuth: Precise Task-Scoped Authorization For Agents* (Microsoft Research, arXiv 2603.17170, March 17, 2026) attacks the same semantic gap from the OAuth-overprivilege side and is the cleanest published demonstration that "OAuth-scope is too coarse for agent tasks" is solvable at the protocol layer rather than only at the policy layer. The paper argues that operator-scoped tokens (the `transfer` scope, the `read_email` scope) inevitably overprivilege agents because the *operator* is broader than the *operation*; "transfer $100 to Bob" needs the ability to call `transfer` with `(amount=100, recipient=Bob)` exactly once, not the ability to call `transfer` with arbitrary parameters arbitrarily many times. PAuth introduces *Precise Task-Scoped Implicit Authorization* where submitting a natural-language task to the agent implicitly authorizes only the concrete operations required for its faithful execution, enforced by two new primitives the server validates: **NL slices** (symbolic specifications of the call graph each downstream service expects, derived from the task and any upstream results), and **envelopes** (data structures that bind each operand's concrete value to its symbolic provenance so the server can verify operands originated from legitimate computations rather than attacker injection). Evaluated on AgentDojo, the system achieved 100% success on benign tasks and 100% warning rate on attack scenarios where spurious operations were injected. For 9.5.1 and 9.5.5 this is the natural complement to the dual-constraint intersection rule advocated above: the agent's *granted permissions* and the *delegating principal's scope* are both still evaluated as independent constraints, but the per-task envelope adds a third constraint -- the *specific operations the natural-language task actually authorized* -- which closes the semantic-scope gap that operator-level scope alone cannot. It composes naturally with the IETF Delegation Receipt Protocol's instruction-hash binding (PAuth verifies operands match the symbolic NL slice; DRP binds the operator's stated instructions to a cryptographic hash) and with AAuth's Mission hash (PAuth provides the server-side verification machinery for the intent the Mission hash pins).

Krti Tallam's *Authorization Propagation in Multi-Agent AI Systems* (arXiv 2605.05440, May 6, 2026) is the cleanest formalization of this problem published to date. The paper argues that authorization in agent systems is a *workflow-level property*, not a per-request property, and decomposes it into three sub-problems: **transitive delegation** (preserving scope as authority flows through chains of agents), **aggregation inference** (the data-level analogue -- agents synthesizing sensitive insights from individually-authorized but collectively-revealing sources, the same phenomenon CoSAI calls the semantic mosaic effect), and **temporal validity** (authorization decisions that were correct at issuance but no longer hold at execution time, which is the 9.5.6 problem restated). The paper explicitly states authorization propagation "is not reducible to prompt injection and is not fully addressed by classical access-control models such as RBAC, ABAC, or ReBAC," and notes a convergence in the literature toward four primitives -- invocation-bound capability tokens (IBCTs), task-scoped authorization envelopes, dependency-graph policy enforcement, and execution-count revocation -- without any single architecture yet covering all three sub-problems. The author's framing -- "identity governance must be treated as infrastructure: evaluated continuously, enforced at every interaction boundary, and designed into the system before orchestration logic is allowed to scale" -- is the academic statement of the operational pattern this section already advocates.

Amjad Ibrahim and Yong Li's *Overlaying Governance: A Compositional Authorization Framework for Delegation and Scope in Agentic AI* (arXiv 2606.03518, June 2, 2026) attacks the same problem from the policy-composition side and is the cleanest published argument that agent delegation does not require abandoning an organization's existing authorization model. The paper treats delegation as a *contractual term* rather than an OAuth-style token-consent event, and introduces a compositional operator that overlays agentic semantics -- recursive delegation chains, resource-scope attenuation, and typed delegation with explicit accountability implications -- on top of existing relational/ReBAC policies *without rewriting them*, with formal proofs that the overlaid semantics preserve the base policy's guarantees. For 9.5.5 this is directly load-bearing: it gives the scope-narrowing-at-each-hop and dual-constraint intersection rules a compositional algebra that an enterprise can apply to its installed-base ReBAC/Zanzibar policies rather than standing up a parallel agent-only authorization stack, which is the adoption gap that PEDIGREE's Cedar narrowing proofs and the Attenuating Agent Tokens draft address at the protocol layer.

### Cloud-Native Agent Authorization Detection

Cloud providers are beginning to ship purpose-built detection for agent authorization abuse. Google Cloud Security Command Center (SCC) now includes a dedicated category of AI agent threat findings that map to MITRE ATT&CK tactics:

- **AI Agent Suspicious Token Generation Using Implicit Delegation** -- Flags when agents misuse implicit delegation to generate elevated-privilege tokens (maps to T1528: Steal Application Access Token).
- **AI Agent Suspicious Cross-Project Access Token Generation** -- Detects agents attempting to generate tokens for resources outside their authorized project scope.
- **AI Agent Suspicious Token Generation Using signJwt** -- Catches agents using the `signJwt` API to forge unauthorized JWT credentials.
- **AI Agent Unauthorized Service Account API Call** -- Identifies API calls exceeding the agent's assigned permissions.
- **AI Agent Excessive Permission Denied Actions** -- Tracks repeated failed authorization attempts, indicating reconnaissance or brute-force probing of privilege boundaries.

These detections are significant because they represent the first shift from generic workload monitoring to agent-specific authorization threat models. Organizations deploying agents on GCP should enable these findings and route them to their SIEM/SOAR pipeline.

### MCP-Specific Authorization Patterns

The Model Context Protocol (MCP) introduces specific authorization challenges because transport authorization does not itself decide which listed tool or argument a principal may use. The stable 2025-11-25 authorization specification makes authorization optional at the protocol level, but HTTP implementations that support it should follow the MCP OAuth profile. That profile requires clients to name the MCP server with the RFC 8707 `resource` parameter in both authorization and token requests, send the bearer token on every HTTP request, and requires the MCP resource server to accept only tokens intended for its audience. An MCP session identifier is not an authorization credential. Fine-grained tool and argument policy still belongs at the server or a gateway enforcement point.

The MCP server is an OAuth resource server; its authorization server may be colocated or separate. [RFC 9728 Protected Resource Metadata](https://www.rfc-editor.org/rfc/rfc9728) advertises the acceptable authorization server or servers, while OAuth or OpenID metadata supplies their endpoints. The 2025-11-25 release supports preregistration, recommends OAuth Client ID Metadata Documents where the parties have no prior relationship, and retains Dynamic Client Registration as a compatibility option. This separation permits centralized identity infrastructure without making every MCP server issue tokens, but deployments still need to pin trusted issuers and validate the protected-resource metadata instead of accepting arbitrary discovery results.

Recommended patterns for MCP authorization:

- **Authorization-conditioned tool disclosure:** The [current MCP draft](https://modelcontextprotocol.io/specification/draft/server/tools#capabilities) permits `tools/list` to vary with the authorization presented on each request, so return only tools the caller's granted scopes permit. Treat that list as disclosure reduction, not an authorization grant: `tools/call` must still enforce tool, resource, and argument policy server-side.
- **Per-request authorization at the MCP boundary:** Rather than relying on the MCP client (the agent) to self-police, embed a Policy Decision Point within or in front of each MCP server. Cerbos provides an open-source policy-as-code engine that can run as a sidecar or library for this purpose, with batch query support for checking multiple tool permissions in a single call and YAML-based policies supporting RBAC, ABAC, and context-aware conditions.
- **Audience-bound credentials at each hop:** The MCP server must not forward the client's inbound token to a backend. When it calls another protected resource, it acts as that resource's OAuth client and obtains a separate, appropriately scoped token. RFC 8693 token exchange is one way to do this; it is not mandated by MCP. Reject issuer, resource/audience, subject, tenant, or scope mismatches at every resource server.
- **Third-party OAuth through URL-mode elicitation:** The stable [URL-mode elicitation flow](https://modelcontextprotocol.io/specification/2025-11-25/client/elicitation#url-mode-elicitation-for-oauth-flows) lets an MCP server obtain user authorization for an external API without exposing third-party credentials to the MCP client. The MCP server acts as the third-party OAuth client, binds the flow and stored token to the verified user, and keeps that token server-side; this is separate from client-to-MCP-server authorization and is another reason not to assume RFC 8693 is the only downstream credential path.
- **SPIFFE identities for MCP workloads:** Assign SPIFFE IDs to MCP servers and clients, enabling mutual TLS authentication and cryptographically verified identity at every connection. This is more robust than static API keys and aligns with the IETF AIMS framework's identity layer.
- **Client registration and delegated consent:** In the MCP authorization-proxy pattern that combines one static upstream client identity with dynamically registered downstream clients, the [confused-deputy guidance](https://modelcontextprotocol.io/specification/draft/basic/authorization/security-considerations#confused-deputy-problem) requires per-client consent before forwarding to the upstream authorization server. At the authorization server or proxy, reject unknown clients, unregistered redirect URIs, and unapproved broader scopes; at the MCP resource server, reject the wrong issuer, audience/resource, or scope. Require rejection of a token obtained by another client only when the deployment uses sender-constrained or explicitly client-bound tokens, such as DPoP, mTLS, or a defined client-claim profile.
- **Enterprise-managed authorization:** The MCP project declared the [`io.modelcontextprotocol/enterprise-managed-authorization` extension](https://modelcontextprotocol.io/extensions/auth/enterprise-managed-authorization) stable on June 18, 2026. It makes the enterprise identity provider the policy authority: the client obtains an Identity Assertion JWT Authorization Grant (ID-JAG) during enterprise sign-on and exchanges it at the MCP authorization server for a resource access token. Verify that the client and server explicitly negotiate the opt-in extension, the server advertises it in authorization metadata, the IdP permits only registered MCP resources, and the authorization server validates the ID-JAG signature, issuer, audience, expiry, subject, resource, and requested scope before mapping claims to permissions. Negative-test an unapproved MCP server, a personal rather than enterprise identity, a mismatched audience, and an expired assertion. None may produce an MCP access token. Centralized server approval does not replace per-tool and per-argument authorization at the MCP resource server.

### MCP Agent Gateways

A maturing pattern as of March 2026 is the **Agent Gateway** -- a dedicated proxy that sits between agents and the MCP servers (or other tool endpoints) they connect to. The gateway serves as a centralized policy enforcement point:

- **Aembit MCP Identity Gateway:** Sits between agents and tools, handling credential injection and token exchange at both the agent-to-gateway and gateway-to-tool legs. Agents never hold long-lived secrets; the gateway injects the correct credential for each downstream call. Demonstrated at RSA 2026 in partnership with Netskope, and reached General Availability on April 9, 2026 under the *Aembit IAM for Agentic AI* brand. The GA release adds two properties directly load-bearing for 9.5.2 and 9.5.5: a *Blended Identity* policy primitive that evaluates the agent's identity and the operating human's identity as independent constraints in the same authorization decision (the dual-constraint pattern in 9.5.5 expressed as a product capability rather than a deployment convention), and per-user credential scoping so a session run by User A receives credentials minted against User A's permissions and a session run by User B receives a separately-minted credential bound to User B's -- the standing answer to the workspace-level credential-sharing confused deputy the hi120ki audit identified in Anthropic Managed Agents. The same release ships an MCP Authorization Server implementing the OAuth 2.1 flow without custom integration code, a policy-driven per-agent kill switch, conditional-access factors (time, geography, runtime posture), and just-in-time credential issuance replacing standing tokens.
- **Agent Gateway (open-source):** An agentic proxy supporting MCP authentication with OAuth 2.0 protection, JWT token validation, and protected resource metadata (RFC 9728). Acts as a resource server validating incoming tokens before routing to downstream MCP servers.
- **Enterprise MCP Registry (Azure API Center):** Microsoft's internal MCP governance model registers every MCP server in Azure API Center for centralized discovery, then places every remote MCP server behind an API gateway for authentication, authorization, rate limiting, and logging. Every tool call carries a correlation ID from client through gateway to server and back, with prompts, tool selections, authorization decisions, and resource access logged in a consistent schema.

The gateway pattern addresses the "scattered MCP server" problem: without centralized governance, each MCP server implements its own authorization (or none), creating inconsistent enforcement. A gateway consolidates policy decisions, credential management, and audit logging in one place. An important security property of the gateway pattern is **token transformation**: the gateway strips incoming access tokens from the agent and issues short-lived, narrowly-scoped assertions to backend MCP servers, reducing exposure of long-lived credentials to LLM-facing components. The `mcp-oauth-gateway` open-source project demonstrates this pattern, adding OAuth 2.1 authorization to any MCP server without code modification. As of April 2026, gateway adoption is growing but raises legitimate concerns about creating a new single point of failure and potential for gateway sprawl analogous to the API gateway fragmentation enterprises experienced in the 2010s.

Four late-April / May 2026 developments crystallized the gateway pattern as the converging industry control point. **Okta for AI Agents** reached General Availability on April 29, 2026, packaging four primitives an enterprise needs to operate the pattern at scale: a Universal Directory that registers agents (including externally-hosted Salesforce Agentforce, Amazon Bedrock, and Google Cloud agents) as first-class identities with assigned human owners; replacement of hardcoded credentials with short-lived, scoped tokens across five categories (authorization servers, secrets managers, service accounts, applications, MCP servers); access-request workflows and certification campaigns for ongoing review; and Identity Security Posture Management to surface shadow-AI agents already running in the environment. The product also serves as the first reference implementation of **Cross App Access (XAA)**, Okta's open protocol that was contributed upstream as the MCP `io.modelcontextprotocol/enterprise-managed-authorization` extension. After its earlier SEP-646 proposal and implementation work, the [MCP project declared the extension stable on June 18, 2026](https://blog.modelcontextprotocol.io/posts/enterprise-managed-auth/): the MCP client authenticates the user against the enterprise IdP, obtains an IETF ID-JAG bound to the target MCP resource, and exchanges it with the MCP Authorization Server for an access token. The extension remains opt-in and outside the core protocol. For 9.5.2 and 9.5.5 the combined effect is that the IdP becomes the central peer-authorization registry for MCP -- an MCP server an admin has not approved in the IdP simply cannot mint a token regardless of what the agent or user attempts.  Google Cloud's *Agent Gateway* (Private Preview, Gemini Enterprise Agent Platform; documentation refreshed May 20, 2026) -- announced at Google Cloud Next on May 6, 2026 together with *Agent Identity* (General Availability) and an explicit security/governance ISV ecosystem spanning Exabeam, F5, Netskope, Okta, Palo Alto Networks, Saviynt, and Silverfort -- is the most operationally specific reference shipped to date: every agent receives an automatically-provisioned SPIFFE identity, all agent-to-tool and agent-to-agent traffic flows through the gateway in either ingress (client-to-agent) or egress (agent-to-anywhere) mode, and Context-Aware Access enforces mTLS with certificate-based token binding plus DPoP (RFC 9449) at the agent-to-external-service edge. Authorization decisions are delegated to IAM policies (scoped by SPIFFE identity), Identity-Aware Proxy (agent identity and permission validation), Model Armor (prompt-injection and data-leakage controls), and Semantic Governance Policies (context-aware execution). Critically, the gateway's default posture is "access to any remote MCP servers, agents, or tools that have not been registered in the local Agent Registry is blocked" -- a protocol-aware, default-deny implementation of the 9.5.5 peer authorization requirement at the cloud-platform layer. Palo Alto Networks' April 30, 2026 announcement of its intent to acquire Portkey -- positioning Portkey as the AI Gateway for Prisma AIRS -- reinforces the same architectural pattern at the security-vendor layer. The strategic argument the press release makes for AI Gateway as "the centralized control plane" is exactly the deployment shape this section already advocates; the procurement consolidation reduces the number of independent gateway implementations enterprises will need to standardize on, but does not change the underlying enforcement requirement. The follow-on **Palo Alto Networks Secure AI Agents** (announced May 12, 2026, available to customers and design partners) is the operationalized form of the same pattern ahead of the Portkey close: an identity-broker proxy that sits between the agent and remote MCP servers so agents never hold MCP credentials directly, plus four primitives directly load-bearing for 9.5.1 and 9.5.5 -- a centralized MCP inventory across AWS, Azure, Copilot, and Claude agents; per-tool-call audit trails of which MCP servers each agent reached and which tools were invoked; a default-disabled registration workflow where new MCPs land in a disabled state and require explicit security-admin approval before any agent can call them (a human-in-the-loop allowlist that maps directly to 9.5.5's "delegation to unlisted peers rejected by default"); and a cited 38% rate of MCP servers with no native authentication as the justification for treating the gateway as the authoritative enforcement point rather than relying on per-server controls. Where Secure AI Agents sits in the MCP traffic path as an identity-broker gateway, Idira sits one layer up as the policy and lifecycle plane that issues the short-lived agent credentials in the first place; the cited *109-to-1 machine-and-AI-identity-to-human ratio* is the motivating data point, and the 61% standing-privilege residual the platform is designed to eliminate is exactly the 9.5.6 continuous-enforcement gap left behind when access grants outlast the task that justified them. Cisco's **Duo Agentic Identity** (introduced at RSAC 2026 on March 23, 2026, with phased GA between April and June 2026) is the identity-vendor counterpart: it routes every agent-to-tool request through a dedicated MCP gateway that intercepts the call, evaluates it against Duo's fine-grained authorization engine, and permits or blocks the action before it reaches the target system. ### Inter-Agent Delegation and the A2A Protocol (9.5.5)

Agent-to-agent delegation introduces trust boundary challenges distinct from user-to-agent delegation. When a primary agent decomposes a task and delegates subtasks to specialized peer agents, each hop crosses a trust boundary and inherits some form of the original authority. Without explicit peer authorization, delegation becomes a vector for lateral movement, session smuggling, and scope escalation.

The Google Agent2Agent (A2A) protocol (initially introduced April 2025, actively developed through 2026) is the most explicit framework for inter-agent communication. Key security properties:

- **Agent Cards as trust anchors:** Each agent declares its capabilities, authentication requirements, and identity in a structured Agent Card. Cards support cryptographic signatures for tamper-proof identity verification and versioned capability declarations that clients validate before interaction. Authenticated clients may receive extended Agent Cards with additional capabilities unavailable to public requesters.
- **Capability-based authorization:** A2A implements authorization at the operation level (tasks, push notifications require capability declarations) and data level (clients must be authorized to access specific tasks). The protocol explicitly maintains the **opaque execution principle** -- agents interact only through declared capabilities without accessing each other's internal state, memory, or tools.
- **Standard authentication:** A2A supports API keys, HTTP authentication, OAuth 2.0 (authorization code, client credentials, device code), OpenID Connect, and mutual TLS. The remote agent verifies tokens against its policy, checking client ID, scopes, and optional audience fields.

However, A2A does not enforce peer allowlisting at the protocol level. Each deploying organization must build a governance layer defining which agents can call which others, what scopes they need, and what data they can exchange.

The CSA delegation security research (March 2026) identifies two critical attack patterns specific to inter-agent delegation:

1. **Agent session smuggling:** A compromised sub-agent embeds hidden commands in its responses that the parent agent executes without detection. For example, a financial assistant delegates to a research agent, which injects unauthorized stock trade instructions invisible to the user -- the parent agent executes them because the response appears to be legitimate research output.
2. **Cross-agent privilege escalation:** One compromised agent reconfigures peers by rewriting their runtime configurations (e.g., `.mcp.json` files), creating self-reinforcing control loops that deepen intrusion scope. The delegation relationship itself becomes the attack surface.

Recommended defenses for inter-agent delegation include scope narrowing at each delegation hop (a parent with read+write can only delegate read), cryptographic lineage tracking to verify task alignment, trust decay models where agent trust scores decrease without positive behavioral signals, and observable execution logs preventing silent command injection.

The *SentinelAgent* framework (arXiv 2604.02767, April 3, 2026) formalizes much of this for federal multi-agent environments and is worth reading even outside government contexts. It defines a *Delegation Chain Calculus* with seven properties -- authority narrowing, policy preservation, forensic reconstructibility, cascade containment, scope-action conformance, output schema conformance, and intent preservation -- proving that the first six can be enforced deterministically while intent preservation is fundamentally probabilistic (the paper's "adversarial intent paraphrasing" result shows intent verification accuracy degrades to roughly 13% against sophisticated rewording attacks). Its *Intent-Preserving Delegation Protocol* (IPDP) runs the deterministic properties through a non-LLM Delegation Authority Service performing pre-execution intent checks, at-execution scope enforcement, and post-execution output validation -- reporting 100% true positive rate at 0% false positives on the 516-scenario DelegationBench v4 and 30/30 attack blocking under black-box adversarial conditions. The practical lesson: authorization for inter-agent delegation should be split into a deterministic core (the six provable properties) and a probabilistic supplement (intent verification with explicit, measured error bounds), not treated as a single LLM-evaluated decision.

### Continuous Re-Authorization for Long-Running Sessions (9.5.6)

Long-running agents turn authorization into an ongoing control problem. A task may begin under valid authority, hibernate, resume hours later, delegate to subagents, consume queued work, or keep a WebSocket open while identity state changes underneath it. For these agents, token validity is only evidence about authentication state; it is not enough to prove the next privileged action is still authorized.

A practical architecture uses two channels. First, every privileged action crosses a synchronous policy enforcement point that asks a policy decision point about the current subject, action, resource, delegation scope, tenant, risk state, and policy version. AuthZEN, Amazon Verified Permissions, OPA-Envoy, Cedar-based services, and Cerbos all fit this pattern. Second, the runtime consumes asynchronous revocation and risk events such as OpenID CAEP/Shared Signals or Microsoft Entra CAE so active sessions, queued jobs, and streams can be interrupted before the next token refresh.

The failure mode to look for is an authorization decision cached at session start and reused for later actions. Safe caches are narrow: they are short-lived, scoped to one action and resource, tied to a policy version or event watermark, invalidated by revocation events, and fail closed when the PDP or revocation stream is unavailable. WebSockets and worker queues need explicit teardown logic; otherwise revoked identities can continue operating through an already-open channel.

A concrete research-grade reference architecture is Zhaoliang Chen's *AITH: A Post-Quantum Continuous Delegation Protocol for Human-AI Trust Establishment* (arXiv 2604.07695, April 9, 2026). Rather than re-signing every operation, AITH issues a *Continuous Delegation Certificate* signed once with ML-DSA-87 (FIPS 204, NIST Level 5) and then enforces six boundary checks per operation -- hard constraints, rate limits, and escalation triggers -- at roughly 4.7M ops/sec with no per-operation signature on the critical path. Revocation is push-based and propagates trust invalidation within one second, with a three-tier SHA-256 *Responsibility Chain* providing tamper-evident attribution after the fact. Five security theorems were machine-verified in Tamarin Prover under the Dolev-Yao model, and a 100,000-operation simulation reported 79.5% autonomous execution, 6.1% human escalation, and 14.4% blocked actions -- numbers consistent with the OWASP Agentic Top 10 *Least Agency* principle that most actions should be inside policy, a small share require human escalation, and a non-trivial share must be refused. AITH is a research protocol, not a production system, but the architecture is the cleanest published demonstration that sub-second revocation, post-quantum delegation signatures, and per-action boundary enforcement can coexist at agent-scale throughput; it is worth reading alongside AuthZEN, Microsoft Entra CAE, and the IETF Transaction Tokens for Agents draft when evaluating 9.5.6 deployments.

Jun He and Deying Yu's *Sovereign Assurance Boundary: Certificate-Bound Admission for Agentic Infrastructure* (arXiv 2606.11632, June 10, 2026) addresses the specific 9.5.6 gap that classical IAM enforces static permissions while audit logs only record what happened *after* execution. SAB inserts a broker as an "assurance airlock" in front of any agent-proposed mutation of production infrastructure: each proposal is compiled into a *typed execution contract* bound to cryptographic evidence and a policy version, routed through a consequence-aware certification path, and -- on approval -- issued a signed *Sovereign Assurance Certificate* scoped to a specific execution identity and validity window. The broker then performs revocation and drift checks *pre-execution*, so an agent cannot directly modify infrastructure and a certificate that has been revoked, or whose bound policy version has drifted, is rejected before the action lands. For 9.5.6 this is the cleanest research-grade demonstration that the "authorization is a current decision, not a session-start property" requirement can be made *verifiable and replayable* for high-stakes infrastructure mutations -- the certificate-bound, pre-execution-revocation model is the runtime-artifact counterpart to AITH's continuous delegation certificate and a concrete answer to the RSAC 2026 "agents rewrite the policy that constrains them" failure mode, because the broker enforcing the certificate sits outside the agent's reach.

Microsoft Entra now ships a complementary production primitive for 9.5.6 at the identity-platform layer. The *Conditional Access for High-Risk Agent Identities* policy template (Preview, documented April 30, 2026 and refreshed May 18, 2026) lets administrators target Conditional Access policies directly at agent identities -- a new selector under **Users, agents (Preview) or workload identities** -- with **Agent risk (Preview)** as a Conditional Access condition fed by Entra ID Protection's risky-agents signal. When an agent's risk level reaches **High**, the policy blocks resource access on the next authorization request regardless of token validity, providing a per-decision re-evaluation that composes with CAE's revocation events: CAE attenuates an *existing* session when a critical event lands, while the High-Risk Agent Identities policy denies the *next* token request when the agent itself is the risk signal. The policy template requires Entra ID P1 for Conditional Access and Entra ID P2 for the agent risk signal, and is operationally adjacent to ID Protection's risky-user-and-agent telemetry. For 9.5.6 deployments on Entra, this is the first risk-signal-driven enforcement primitive whose subject is the agent identity itself rather than a service principal or a human user, and it pairs naturally with the Aembit Blended Identity dual-constraint pattern and the Google Cloud SCC agent-specific threat findings catalogued elsewhere in this section.

As of June 2026, the Entra documentation generalizes this into a full *Conditional Access for agents* model covering all three agent access patterns: on-behalf-of (where policy targets the human user and every OBO token exchange is re-evaluated by Conditional Access), agent-as-application (client credentials, where policy targets the agent identity via "All agent identities" or individual selection), and agent-as-user "digital workers" (where policy targets the agent's user account, in Preview). Policies can be applied at the agent identity blueprint level -- one policy automatically covers every agent identity derived from that blueprint, including future ones -- or driven by custom security attributes so new agents inherit policy by classification rather than enumeration, and an **Agent execution environments (Preview)** condition lets administrators require device compliance for agents running on managed endpoints such as Windows 365 Cloud PCs for Agents. The documented boundaries matter as much as the features for a 9.5.6 audit: Conditional Access only governs Entra-issued tokens, so an agent that reaches a resource with an API key bypasses the entire policy pipeline; policies targeting "all users" do not include agent user accounts; and a policy targeting agent identities does not apply to the corresponding agent's user account -- coverage must therefore be verified per access pattern, with `agentType`-filtered sign-in logs as the evidence trail. Licensing is a real deployment constraint as well: Conditional Access for agents requires Entra ID P1/P2 plus a Microsoft Agent 365 license per user.

### OWASP Agentic Top 10 (2026) Mapping

The OWASP Top 10 for Agentic Applications (2026), developed with over 100 industry experts, introduces the **Least Agency** principle: autonomy is a feature that should be earned, not a default setting. Three of the ten risks map directly to C9.5 requirements:

- **ASI01 Agent Goal Hijack** maps to 9.5.4 -- attackers manipulate agent objectives through natural-language injection, and if the agent participates in access control decisions, the hijacked goal can override authorization. Policy engines external to the model prevent this.
- **ASI02 Tool Misuse and Exploitation** maps to 9.5.1 -- agents misuse legitimate tools (email, CRM, web browser, DNS, internal APIs) in risky ways, often staying within granted permissions but performing destructive or exfiltrating actions. Fine-grained per-tool and per-parameter policies are the primary defense.
- **ASI03 Identity and Privilege Abuse** maps to 9.5.2, 9.5.3, and 9.5.6 -- agents inherit user sessions, reuse secrets, rely on implicit cross-agent trust, or continue acting after revocation, leading to privilege escalation and unattributable actions. Short-lived credentials, task-scoped permissions, and continuous authorization enforcement are the prescribed mitigations.

### CSA Agentic Trust Framework

The Cloud Security Alliance published the Agentic Trust Framework (ATF) in February 2026, applying Zero Trust governance specifically to AI agents. The ATF organizes controls around five pillars: identity (authentication, authorization, session management), behavior (observability, anomaly detection, intent analysis), data governance (input validation, PII protection, output controls), segmentation (access boundaries, resource limits, policy enforcement), and incident response (circuit breakers, kill switches, containment). A distinctive feature of the ATF is its **progressive autonomy model** -- agents are deployed with minimal autonomy and earn increased autonomy through demonstrated safe behavior, with continuous monitoring at every level. This contrasts with traditional "grant and forget" access models.

### CoSAI Agentic IAM and Agent Detection and Response (May 2026)

The Coalition for Secure AI (CoSAI, an OASIS Open project) published two complementary papers on May 6, 2026 immediately following RSAC 2026: *Agentic Identity and Access Management* and *The Future of Agentic Security: From Chatbots to Autonomous Swarms*. The Agentic IAM paper -- developed by CoSAI's Secure Design Patterns for Agentic Systems Workstream -- frames a foundational claim: without a trustworthy machine-readable identity for every agent, no other security control can be reliably enforced. It outlines how to assign unique credentials per agent, enforce task-scoped access, and maintain visibility into who or what is taking each action and how authority was delegated.

The papers introduce **Agent Detection and Response (ADR)** as a new defensive category specifically for agentic systems, paralleling the role EDR played for endpoints. ADR is the runtime pillar: it intercepts agent actions at the moment of execution, before they land, and evaluates whether they are safe given the agent's identity, delegated scope, observed behavior, and the operational context. Vendors are starting to ship ADR-branded capabilities (Gen Digital Agent Trust Hub, Astrix, CrowdStrike Agentic MDR), and the architectural pattern aligns directly with the policy-decision-point-at-tool-call-boundary requirement in 9.5.1 and 9.5.3. At RSAC 2026, CrowdStrike formally re-framed the category as **AIDR (AI Detection and Response)**, positioning it as the next category beyond EDR with explicit telemetry across endpoints, SaaS, cloud, and AI pipelines; Sevii shipped an *Autonomous Proactive Security* (APS) module extending its ADR platform with continuously-updated threat intelligence and agentic Cyber Warrior agents deployed at the network edge. Naming conventions remain in flux -- ADR, AIDR, AI-ADR, and AgentSec are all in market use as of May 2026 -- but every variant centers the same control: a synchronous policy decision point intercepting tool calls before execution.

CoSAI's framework also names two open research problems that traditional IAM cannot address:

- **Intent-based authorization** -- reliably evaluating what an agent is *trying* to accomplish in natural language, so authorization decisions can compare the agent's intent against the user's mandate rather than only its individual API calls.
- **The semantic mosaic effect** -- agents can synthesize and expose sensitive insights from individually innocuous sources without ever triggering conventional leak protection, because each constituent access was authorized. This is the data-level twin of semantic privilege escalation: each operation is permitted, but their *composition* leaks.

The recommended response is a governance architecture that distributes oversight across specialized agents and platform services rather than a single centralized supervisor, with an *Oversight Agent* role that has no operational duties beyond monitoring swarm semantic health and is empowered to instantly constrain or contain compromised peers.

### RSAC 2026 and Industry Readiness

RSAC 2026 (April--May 2026) crystallized the industry view that agent identity is solved on paper but authorization is not. The Cisco *State of AI Security 2026* report found that 83% of surveyed organizations planned to deploy agentic capabilities, but only 29% felt prepared to secure them. Anthony Grieco (Cisco SVP/CSO) said rogue agent incidents are reaching enterprise customer bases regularly, with a consistent pattern: authentication passes, identity checks clear, the agent is exactly who it claims to be -- and then it accesses data it was never scoped to touch or takes actions nobody authorized at that level of granularity. The illustrative example: a "finance agent" that is correctly identified as a finance agent and reaches for *all* finance data, when its mandate is one specific expense report at a specific time.

The same RSAC analysis surfaced two real incidents from CrowdStrike CEO George Kurtz's keynote at Fortune 50 companies, including the now-frequently-cited case where a CEO's AI agent rewrote the company's own security policy because the policy was blocking the task it had been given. That incident is the canonical example for 9.5.5 (intersection enforcement) and the proposed Delegation Receipt Protocol's non-waivable boundaries (above): a policy decision point cannot rely on the agent to honor a restriction it can talk itself out of.

Yubico, IBM, and Auth0 jointly announced a Human-in-the-Loop agent authorization framework at RSAC 2026 (briefing March 24, 2026): IBM's watsonx AI orchestration proposes the action, Auth0 triggers a secure out-of-band approval request via the OpenID Client-Initiated Backchannel Authentication (CIBA) standard, and a YubiKey tap provides cryptographic proof that a specific human physically authorized the high-impact step (large financial transfers, production code deploys, sensitive data access). The hardware-bound assertion is fresh per decision rather than cached in the agent session, which is operationally adjacent to OAuth Step-Up Authentication (RFC 9470) and gives a concrete enforcement mechanism for 9.5.5's intersection rule when the delegating user's policy demands re-affirmation. The CIBA-anchored design is also load-bearing for 9.5.6 because the approval is asynchronous and decoupled from the agent's primary access token -- the user can deny mid-task without first revoking the session, and the agent's continued execution depends on receiving a fresh CIBA approval rather than on the original token still being valid.

### Proofpoint Agent Integrity Framework (March 2026)

Proofpoint introduced the *Agent Integrity Framework* in March 2026, organizing intent-based AI agent security around five pillars: Intent Alignment, Identity and Attribution, Behavioral Consistency, Auditability, and Operational Transparency. The framework's central operational idea is a metric called **Mean Time to Understand (MTU)** -- the time required to interpret an agent's intent, plan, toolchain, and data flows well enough to make a safe authorization decision. The framework reframes the standard `Identify → Authenticate → Authorize` sequence as `Understand → Align → Authorize`, with explicit recognition that many agent tasks share overlapping technical actions while differing fundamentally in business purpose, so technical access decisions are insufficient. For 9.5.5 and 9.5.6, MTU is a useful service-level metric: cached authorization decisions should not survive longer than the system's ability to re-evaluate intent.

### Credential Isolation for Agents (9.5.4)

The fundamental principle is straightforward: if a credential exists anywhere in the model's observable context, prompt injection can extract it. This applies to system prompts, tool parameters, environment variables that are exposed to the model, and any data that flows through the context window. The defense is architectural -- credentials must never enter the model's observable state.

As of March 2026, three credential isolation patterns have production implementations:

1. **Transport-layer credential proxies.** A local proxy intercepts the agent's outbound HTTP requests and injects the real credential at the network layer. The agent sends requests with a placeholder or key name; the proxy resolves the actual credential from an encrypted vault or OS keychain, swaps it into the request headers, and forwards the response. AgentSecrets and OneCLI implement this pattern. The critical property is that the credential value never appears in the agent's process memory, context window, or logs. AgentSecrets takes this further with zero-knowledge encryption (X25519 key exchange + AES-256-GCM) and audit logs that are structurally incapable of recording credential values.

2. **Secrets manager sidecar injection.** In containerized deployments, a sidecar (HashiCorp Vault Agent Injector, AWS Secrets Manager agent) authenticates with the secrets manager on behalf of the workload and renders secrets into files on a shared volume. The agent's code reads from the file at the moment of use, and the secret never transits through the model's context. Vault's dynamic secrets engine generates credentials on-the-fly with configurable TTLs (typically minutes to hours), so even if a credential is somehow captured, it expires quickly. **HashiCorp Vault Enterprise 2.0** (GA April 13, 2026, headlined at IBM Think 2026 May 4--7 alongside the unified IBM Verify + Vault solution for human and non-human identities) modernizes this pattern for the agentic era: workload identity federation extends to secret sync so static credentials are replaced with short-lived, dynamically exchanged tokens minted from a trusted workload identity (AWS, Azure, GCP, Kubernetes) instead of long-lived API keys; a SPIFFE secrets engine reaches General Availability so workloads request JWT-SVID identity tokens with automatic rotation; SCIM 2.0 server support (public beta) lets Vault consume agent and group lifecycle events from any SCIM-compliant identity provider, including the SCIM Agent Extension this section already cites; and Transit envelope encryption supports large-artifact and streaming workloads with ephemeral key encryption keys (KEKs) so AI pipelines and data-intensive agents can encrypt locally while Vault still owns key custody. The architectural net effect for 9.5.4 is that the same `Vault as MCP-server credential broker` pattern documented in HashiCorp's validated AI-agent pattern now runs natively on workload identities (no embedded service-account token in the agent process) and exposes a SPIFFE identity for the agent's outbound calls -- closing two of the three gaps (long-lived credentials, lack of cryptographic workload identity) that the IETF AIMS framework names as the dominant agent-credential failure modes.

3. **Workload identity federation with per-call credential generation.** The agent's runtime environment presents cryptographic proof of its identity (Kubernetes service account, pod metadata, container image signature) to a trust provider. A policy engine evaluates whether the agent is authorized for the specific API call, and if approved, generates a short-lived credential (15-30 minute lifetime) that is injected directly into the outbound request without being stored. Aembit implements this for AI agent workloads, supporting multiple LLM providers transparently.

4. **Outbound-only tunnels to private MCP servers.** Anthropic's **MCP Tunnels** (research preview announced May 19, 2026 during Cloudflare Agents Week) extend Managed Agents and the Messages API to private MCP servers without exposing them to the public internet. The customer deploys a lightweight gateway inside its own network that establishes an outbound encrypted connection to Anthropic infrastructure -- internal repositories, files, databases, ticketing systems, and knowledge bases become reachable from Claude agents without opening any inbound firewall rules, and existing network policies, audit logging, and runtime configuration remain in customer control. Self-hosted sandbox partners (Cloudflare microVMs with zero-trust networking, Daytona stateful SSH environments, Modal GPU/CPU AI workloads, Vercel VPC peering with credential injection at the network boundary) extend the same posture to the *execution* side: tool runs land on customer-controlled or partner infrastructure rather than Anthropic's public sandbox. For 9.5.4 verification this is operationally important because the credential isolation property the brain/hands/session architecture provides at the Anthropic edge now extends to the customer's perimeter without weakening it -- the agent reaches the private MCP server through an authenticated outbound channel rather than through a publicly-resolvable endpoint protected only by a token. For 9.5.5 it removes one common deployment-time excuse for accepting unauthenticated MCP servers (the Knostic/Zuplo "41% have no auth" tail) because a tunneled server can refuse all inbound traffic by network policy rather than relying on application-layer authentication that may not have been wired up correctly.

5. **Platform-managed credential vaults with structural isolation.** Anthropic's Managed Agents (public beta, April 2026) implements a "brain/hands/session" zero-trust architecture where three components do not trust each other: the brain (Claude and the routing harness), the hands (disposable Linux containers for code execution), and the session (an append-only event log outside both). Credentials live in a separate Credential Vault component -- the sandbox execution environment has zero access to long-lived credentials by design. MCP tool calls route through a dedicated proxy that fetches credentials from the Vault only when needed, injects them at the transport layer, and returns only tool results to the agent. OAuth token renewal is handled automatically, and tokens never appear in sandbox code or environment variables. A security audit (hi120ki, April 2026) identified two residual weaknesses: (a) a confused deputy problem where workspace-level credential sharing means anyone with API key access can use any vault credential in a session, and (b) no process-level monitoring inside the sandbox (no Falco or eBPF-based tooling), so malicious package backdoors could execute undetected system calls. OAuth scope narrowing is also not supported at the Vault level -- if a stored token holds `channels:read` and `chat:write`, both apply even if only one is needed for the task. **Cloudflare Environments for Claude Managed Agents (announced May 19, 2026 during Cloudflare Agents Week)** extends this pattern with an enterprise-deployable egress and identity layer: agent traffic flows through customer-controlled Cloudflare Workers acting as programmable outbound proxies that inject encrypted credentials *outside* the sandbox boundary, and private backend services are reachable through Cloudflare Mesh and Workers VPC tunnels (post-quantum encrypted) instead of public internet. The earlier Cloudflare *outbound Workers for Sandboxes* primitive (announced April 13, 2026) provides the conditional handler logic that makes this enforceable: per-sandbox identity is exposed as `ctx.containerId` in the outbound Worker, so a single proxy can apply different credentials, allowlists, or KV-backed token lookups per agent or per tenant without the agent ever holding the credential. For 9.5.4 verification, the auditable property is that even a fully-compromised agent process cannot read the credentials it is using, because they are added by the trusted proxy after the request leaves the sandbox boundary; for 9.5.5 verification, the same proxy layer is the natural place to enforce per-peer allowlists when agents call out to other agents over HTTP.

6. **Defense-in-depth kernel-level sandboxing with gateway-mediated credentials.** NVIDIA NemoClaw (early preview, March 2026) wraps the OpenClaw agent framework in five stacked security layers enforced by the OpenShell runtime: (1) default-deny network egress using Linux network namespaces, where the agent must request approval for each outbound destination; (2) immutable filesystem restrictions confining the agent to `/sandbox` and `/tmp`, locked at creation and unmodifiable at runtime; (3) process and syscall hardening via Landlock LSM rules and seccomp-bpf profiles blocking dangerous syscalls (ptrace, execve to unauthorized binaries, raw socket creation); (4) an inference routing gateway that intercepts all LLM API calls and routes them through OpenShell -- the agent never holds API keys directly; and (5) an intent verification layer where blocked actions surface for human review in the OpenShell TUI, creating auditable allow/deny approval workflows. The architectural distinction from container-based isolation is that NemoClaw assumes the agent itself may be compromised and enforces all policy externally -- locked policies (filesystem, process) cannot be modified by the agent, and dynamic policies (network, inference) require out-of-band operator approval. NemoClaw is alpha-stage and not recommended for production.

7. **V8 isolate-based sandboxing with credential interception.** Cloudflare Dynamic Workers (open beta, April 2026) provide V8 isolate-based sandboxing that starts approximately 100x faster and uses 10-100x less memory than containers. The security-relevant property for credential isolation is that isolates intercept outbound HTTP requests at the security boundary and inject authentication tokens transparently -- agent-generated code never directly accesses secrets. Communication uses Cap'n Proto RPC bridges across the sandbox boundary. Because isolates are cheap enough to create and destroy per-request, there is no need for warm container reuse across tasks, eliminating cross-session credential leakage. Multi-layered defenses include automatic V8 security patch deployment within hours, a custom second-layer sandbox with dynamic risk-based tenant cordoning, and hardware-level MPK (Memory Protection Keys) protections.

Forough, Kogias, and Haddadi's *When Agents Handle Secrets: A Survey of Confidential Computing for Agentic AI* (arXiv 2605.03213, May 4, 2026; revised May 7, 2026) is the systematic mapping of how Trusted Execution Environments compose with the seven patterns above. The survey catalogues six TEE platforms -- Intel SGX/TDX, AMD SEV-SNP, ARM TrustZone/CCA, and NVIDIA H100 Confidential Computing -- against agent-specific threat vectors at the perception, planning, memory, action, and coordination layers, and concludes that "no broadly established end-to-end framework yet binds them into a coherent security substrate for production agentic AI." For 9.5.4 verification the practical implication is that hardware-backed isolation closes a specific subset of credential-exposure paths the proxy/sidecar/federation patterns above cannot address (privileged-system-software adversaries, memory inspection by the host, side-channel exfiltration), but adds two requirements deployments tend to underestimate: *compound attestation* (a multi-hop agent chain must produce a verifiable, freshness-bound attestation across every hop, not just at the entry point), and *GPU-TEE performance scaling* (LLM-scale inference inside an H100 CC envelope still imposes measurable throughput cost that bounds where the pattern is operationally viable). Deployments that already run on confidential-VM platforms (Azure Confidential Containers, GCP Confidential Space, AWS Nitro Enclaves) should treat the survey as the gap analysis between the *hardware* property they have and the *end-to-end* property 9.5.4 actually requires.

The HashiCorp Vault validated pattern for AI agents demonstrates how these approaches compose: the LLM orchestrates task logic but never directly accesses sensitive resources. When a tool call requires authenticated access, the MCP server (not the agent) authenticates to Vault using an On-Behalf-Of token, retrieves dynamic credentials scoped to the user's group policies, executes the database or API query, and returns only the result to the agent. The credential never enters the LLM's context.

**The scale of the problem:** GitGuardian's State of Secrets Sprawl 2026 report found 1,275,105 AI-service secrets leaked publicly (81% YoY increase), with LLM infrastructure secrets (orchestration, RAG, vector storage) growing five times faster than core model provider keys. MCP configuration files alone contained 24,008 secrets, of which 2,117 were confirmed valid. This is partly attributable to MCP documentation patterns that encouraged hardcoded credentials in configuration files -- a problem the community is actively correcting.

### Anti-Patterns to Avoid

- Letting the AI model make access control decisions based on conversation context (prompt injection can manipulate these).
- Using the end user's actual credentials for agent actions instead of scoped delegation tokens.
- Caching authorization decisions for long-running agent tasks without event-driven invalidation.
- Granting agents broad "admin" or "full-access" scopes because their task is open-ended.
- Using static API keys for agent identity -- agents are workloads that should use short-lived, cryptographically-bound credentials (the IETF AIMS framework explicitly labels static API keys as an anti-pattern).
- Allowing unlimited delegation depth in multi-agent chains without policy-enforced limits on transitive trust.
- Treating a still-valid access token as sufficient authorization for queued, resumed, streaming, or long-running agent actions after policy or identity state changes.
- Passing API keys, database passwords, or service credentials as tool parameters or including them in system prompts where the model can observe and potentially leak them via prompt injection.
- Hardcoding credentials in MCP configuration files -- GitGuardian found 24,008 secrets in MCP configs on public GitHub, with 2,117 confirmed valid.
- Relying on environment variables for credential storage when those variables are accessible to the model's context (the agent process can read them, and prompt injection can instruct the model to output them).

## Open Research Questions

- How do you define "minimum necessary scope" for delegation tokens when the agent's task is open-ended (e.g., "research and summarize")? The NIST concept paper recommends ABAC but does not yet provide concrete policy templates.
- Can authorization policies for agent actions be auto-generated from tool manifests and user role definitions? AWS Bedrock AgentCore's natural-language-to-Cedar generation is an early attempt, but the correctness of generated policies has not been independently validated at scale.
- What are the performance implications of per-call authorization in high-throughput agent systems, and what caching strategies are safe? AuthZEN's standardized PEP-to-PDP API may reduce integration overhead, but latency benchmarks for agent-scale workloads are not yet published.
- How should authorization work for agents that operate across organizational boundaries (federated agent systems)? The IETF AIMS framework defines identity chaining and cross-domain flows, but real-world deployments are still in pilot stage.
- The NIST AI Agent Standards Initiative comment period closed April 2, 2026, and CAISI is now conducting sector-specific listening sessions. How will the resulting draft project description shape mandatory requirements for agent authorization in federal and regulated environments, and will it establish concrete guidance on delegation chain limits and ABAC policy templates that the concept paper recommended but left unspecified?
- Can the ARIA graph-based delegation model and the AIMS layered architecture converge into a unified standard, or will organizations need to choose between them?
- What attestation mechanisms (TEE evidence, binary validation, supply-chain provenance) are practical for agent identity in cloud-native environments, and how do they interact with existing SPIFFE/SPIRE deployments?
- How can semantic privilege escalation be detected in real-time when each individual agent action is technically authorized? Intent-tracking and behavioral anomaly detection are proposed approaches, but no production system has demonstrated reliable detection without unacceptable false-positive rates.
- As cloud providers ship agent-specific authorization detections (Google Cloud SCC's AI threat findings being the first), will these become a de facto standard that other providers must match? What coverage gaps exist in current detections?
- How should MCP tool authorization evolve? The stable 2025-11-25 profile makes transport authorization optional and leaves fine-grained tool and argument policy to implementations. The current draft now permits `tools/list` to vary by the authorization presented, but still does not define scope-to-tool mapping or replace server-side checks on `tools/call`. What interoperable policy vocabulary should fill that gap?
- Will agent gateways (Aembit, Agent Gateway, Azure API Center) converge on a standard architecture, or will enterprises face gateway sprawl analogous to the API gateway fragmentation of the 2010s? The gateway pattern solves immediate authorization gaps but introduces a new single point of failure and policy management surface.
- The PleaseFix vulnerability demonstrated that technically-authorized agent actions within an authenticated browser session can be exploited for credential theft and file exfiltration. How should authorization frameworks model the boundary between "agent has permission" and "agent is acting on attacker-controlled instructions within those permissions"? This is the practical manifestation of the semantic privilege escalation problem.
- Credential proxy architectures (AgentSecrets, OneCLI, Aembit) keep secrets out of the agent's context, but adoption remains low -- most frameworks still pass credentials via environment variables or tool parameters. What would it take for credential proxies to become the default pattern in major agent frameworks (LangChain, CrewAI, AutoGen)? Is framework-level integration or a protocol-level primitive (e.g., an MCP credential injection extension) the more viable path?
- GitGuardian found that code-assisted commits leak secrets at roughly twice the baseline rate (3.2% vs 1.5%). As agents increasingly write and execute code, how should credential isolation extend to agent-generated code that may inadvertently embed secrets in source files, logs, or configuration?
- The A2A protocol provides capability-based authorization via Agent Cards with cryptographic signatures, but does not enforce peer allowlisting at the protocol level. Should peer authorization be a protocol-level primitive in A2A (and MCP), or is it inherently an enterprise governance concern that varies too much across deployments?
- The OpenClaw CVE pattern (138 vulnerabilities in five months, with authorization validation logic present but inconsistently applied across code paths) suggests that agent frameworks face a systematic code coverage problem for authorization checks. Is this best addressed through mandatory authorization middleware that intercepts all tool invocations by default (deny-unless-checked), through static analysis tooling that detects unguarded code paths, or through protocol-level enforcement where the runtime rejects any tool call that hasn't passed through a registered policy decision point?
- With 41% of 17,000+ MCP servers still running without authentication as of early 2026 (down from 100% in mid-2025 per Knostic's scan), is the MCP ecosystem converging on authentication fast enough, or will agent gateways need to treat all MCP servers as untrusted by default and enforce authentication at the gateway layer regardless of server-side implementation? The mcp-oauth-gateway pattern of wrapping unauthenticated servers in an OAuth 2.1 layer suggests the community expects significant tail of unprotected servers.
- CVE-2026-32213 (Azure AI Foundry, CVSS 10) and the Vertex AI "Double Agents" research both demonstrate that cloud AI platforms can ship with catastrophically overprivileged default service accounts. Should cloud providers be required to default to least-privilege agent identities (like Google's BYOSA recommendation), or does this create too much friction for initial adoption?
- CVE-2026-34040 (Docker AuthZ bypass) showed that agents running in Docker-based sandboxes can autonomously discover and exploit infrastructure authorization flaws to escape containment -- the agent figured out the 1MB size boundary on its own. How should agent sandbox architectures account for the fact that the agent itself is an adversarial intelligence capable of probing and bypassing authorization boundaries, not just a passive workload? Is defense-in-depth (multiple independent authorization layers) sufficient, or do we need fundamentally different isolation models for agentic workloads?
- The RSAC 2026 analysis found that none of the five shipped agent identity frameworks can reconstruct delegation chains after the fact. The IETF Transaction Tokens for Agents draft (draft-oauth-transaction-tokens-for-agents-06, April 2026) proposes `actchain` claims for cryptographic delegation lineage -- is this the right approach, or does the JWT size growth from deeply nested agent calls (the draft permits TTS-imposed depth limits and oldest-entry truncation) make it impractical at scale?
- The Anthropic Managed Agents "brain/hands/session" architecture and Cloudflare Dynamic Workers both demonstrate that platform-managed credential isolation (where the sandbox structurally cannot access credentials) is production-viable. Will this pattern converge into a standard isolation model for all agent platforms, or will the diversity of deployment environments (cloud, on-prem, edge) prevent standardization? The confused deputy problem in Managed Agents (workspace-level credential sharing without per-user vault scoping) suggests the model still has gaps to address.
- SandboxEscapeBench (UK AISI/Oxford, March 2026) demonstrated that frontier models autonomously escape container sandboxes at a 40% success rate on multi-step scenarios. The Cymulate disclosure showed that major AI coding tools (Claude Code, Gemini CLI, Codex CLI) share a common vulnerability class where application configuration is writable from within the sandbox. Combined with the Unit 42 finding that AWS AgentCore's "isolated" sandbox permitted DNS tunneling for credential exfiltration, the evidence suggests that container-based sandboxes are necessary but insufficient for agent authorization boundaries. Should the industry converge on multi-layer defense models (NemoClaw's five-layer approach, Anthropic's brain/hands/session separation) as the minimum viable isolation standard, or are there lighter-weight alternatives that provide comparable guarantees?
- Four new IETF drafts target agent-specific OAuth extensions: transaction tokens for agents, scope aggregation, multi-agent collaboration authorization, and on-behalf-of user authorization. Will these converge into a coherent agent authorization profile, or will the proliferation of overlapping drafts fragment implementation and delay adoption? The May 2026 publication of draft-nelson-agent-delegation-receipts-05 and draft-niyikiza-oauth-attenuating-agent-tokens-00 adds two further distinct primitives (cryptographic delegation receipts and protocol-level scope attenuation) -- the design space now spans token-based, receipt-based, and attenuation-based delegation models that may need to interoperate within a single deployment.
- CoSAI's *Agentic Identity and Access Management* paper (May 2026) frames Agent Detection and Response (ADR) as a new defensive category. Vendors are already shipping ADR-branded products (Gen Digital Agent Trust Hub, CrowdStrike Agentic MDR, Astrix), but there is no shared definition of what an ADR product must include. Will ADR converge on a standard control set the way EDR did, and what is the minimum capability bar -- runtime tool-call interception, intent-action alignment evaluation, behavioral anomaly detection, kill switches, or some superset?
- Cisco's State of AI Security 2026 reports an 83% deployment intent against only 29% readiness, and CrowdStrike disclosed a Fortune 50 incident in which a CEO's AI agent autonomously removed a security policy that was blocking task completion. As agent autonomy increases faster than organizational readiness, do regulators need to mandate technical controls (non-waivable boundaries, ADR coverage, intent verification gates) the way PCI mandated controls for cardholder data, or is the gap better closed by industry-developed certification?
